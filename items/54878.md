## Summary

Keep open. Current `main` still appears to have the reported bug: `createSessionVisibilityGuard` computes spawned-child keys for `visibility === "tree"`, but it rejects any cross-agent target before consulting that spawned lineage, so an ACP child like `agent:codex:acp:<uuid>` spawned by `agent:communicator:main` would still be denied by `sessions_history`. This also conflicts with the shipped docs/config help, which still describe `tools.sessions.visibility=tree` as including sessions spawned by the current session.

## Evidence

- **Docs still promise tree includes spawned children:** The config docs say the default `tools.sessions.visibility` is `tree` and define it as `current session + sessions spawned by the current session (subagents)`.
  - file: [docs/gateway/config-tools.md:257](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/docs/gateway/config-tools.md#L257)

- **Generated config help says the same thing:** The schema help text still says `"tree" default = current session + spawned subagent sessions`, so current documented behavior matches the issue reporter's expectation.
  - file: [src/config/schema.help.ts:705](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/config/schema.help.ts#L705)

The later `tree` branch that consults `spawnedKeys` only runs for same-agent targets.
  - file: [src/agents/tools/sessions-access.ts:201](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/agents/tools/sessions-access.ts#L201)

- **ACP spawns still record parent lineage, so the denied case is structurally real:** ACP spawn writes `spawnedBy: requesterInternalKey` onto the child session record, which means cross-agent ACP children do carry parent lineage even though the visibility guard will still reject them before using it.
  - file: [src/agents/acp-spawn.ts:1174](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/agents/acp-spawn.ts#L1174)
