## Summary

Keep open. Current main has targeted safety nets for config and secrets, but it does not implement the broader automatic rollback requested for self-modification/high-risk startup-affecting changes. The existing update path still relies on build/doctor checks plus manual rollback if an update goes bad.

## Evidence

- **Gateway config already has a narrow last-known-good restore path:** Docs say invalid `openclaw.json` is preserved as `.clobbered.*` and the gateway restores the last-known-good copy after successful startup, which covers config corruption but not the broader self-modification scope requested in the issue.
  - file: [docs/gateway/configuration.md:87](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/docs/gateway/configuration.md#L87)
  - command: `nl -ba docs/gateway/configuration.md | sed -n '80,110p'`
  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **Secrets apply snapshots and restores only its own touched files:** `secrets apply` captures per-file snapshots and best-effort restores them on write failure. This is a transactional safeguard for one command path, not a general rollback layer for risky self-edits to startup/runtime artifacts.
  - file: [src/secrets/apply.ts:798](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/secrets/apply.ts#L798)
  - command: `nl -ba src/secrets/apply.ts | sed -n '780,850p'`
  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **Update flow validates after update but does not revert automatically:** The git update runner performs `openclaw doctor --non-interactive --fix` after install/build, then reports failure. There is no rollback/recheckout step in this path when health checks fail.
  - file: [src/infra/update-runner.ts:1184](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/infra/update-runner.ts#L1184)

  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **User-facing rollback guidance is still manual:** The updating docs describe rollback as manually pinning an npm version or checking out an older git commit, which is the opposite of the requested built-in automatic rollback after failed self-modification/update health checks.
  - file: [docs/install/updating.md:138](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/docs/install/updating.md#L138)

## Risks / Open Questions

- Current main already has partial recovery mechanisms, so any future implementation should define scope carefully to avoid duplicating config/secrets protections while still covering update or other startup-critical edit paths.
