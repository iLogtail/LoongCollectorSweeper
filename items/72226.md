## Summary

Current main still has a plausible, source-backed path for this bug. Main-session cron systemEvents are delivered through the heartbeat embedded-agent path, but heartbeat runs deliberately use an empty transcript prompt. That makes the systemEvent become runtime/system context while the submitted prompt is empty, so an Anthropic replay whose stored session already ends on an assistant message is not forced to end with a new user turn before provider submission.

## Best Possible Solution

Keep #72226 open and fix the OpenClaw embedded-runner path so main-session cron/systemEvent heartbeat runs always submit an Anthropic-valid user turn when there is runtime event content, especially when the stored conversation tail is assistant. A focused fix likely belongs around `resolveRuntimeContextPromptParts`/prompt submission or the heartbeat systemEvent prompt assembly, with a regression test covering `sessionTarget: "main"`, `Provider: "cron-event"`, Anthropic replay policy, empty heartbeat transcript prompt, and prior assistant-tail history.
