## Summary

Keep open. Current `main` does not already contain this PR's mutex-based serialization, and the proposed patch still has credible regressions against current contracts: it would make guarded `appendMessage` effectively async while current codepaths still consume synchronous `messageId` results, and it would move pending tool-call registration ahead of write acceptance. `main` already carries a narrower cleanup-side mitigation (`waitForIdle` before flush), so this PR is neither implemented on `main` nor safe to auto-close from the available evidence.

## Evidence

- **Current guard tracks pending tool calls only after an accepted write:** `installSessionToolResultGuard` applies `beforeMessageWriteHook`, returns early when blocked, calls `originalAppend`, emits transcript update, and only then calls `pendingState.trackToolCalls(toolCalls)`. That ordering avoids phantom pending IDs for dropped/failed assistant writes.
