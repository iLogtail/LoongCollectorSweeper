## Summary

Keep #38439 open. Current main still has the source-level Windows namespace path-containment gap described in the issue discussion, and the related fix PR #38923 is still identified in the provided GitHub context as open/unmerged. The route and IDENTITY.md fallback exist, but valid Windows local avatars can still resolve to `none/outside_workspace`, which makes `/avatar/<agentId>?meta=1` return `avatarUrl: null` and `/avatar/<agentId>` return 404.

## Best Possible Solution

Keep #38439 open until #38923 or an equivalent maintainer-reviewed fix lands. The fix should normalize Windows namespaced and drive-letter paths before avatar containment checks, preserve the existing workspace-boundary and symlink protections, keep `too_large` metadata behavior, and add regression coverage for a non-default Windows-style agent resolving a workspace-relative IDENTITY.md avatar where `/avatar/<agentId>?meta=1` returns a local avatar URL and `/avatar/<agentId>` serves the image.
