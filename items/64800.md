## Summary

Keep PR #64800 open. Current main still exposes a single-provider `wrapProviderStreamFn` path: it resolves one hook plugin and invokes only that plugin's `wrapStreamFn`, so the reported primary-provider plus hookAlias-plugin composition gap is not implemented on main. The PR is also not low-signal: its discussion shows the bot-raised P1 concerns were addressed in the latest pushed diff with provider filtering and ID-based dedupe, though the patch now needs rebasing onto the current `provider-hook-runtime.ts` location.

## Best Possible Solution

Keep the PR open and ask for a rebase onto current main. The best implementation is a narrow change in `src/plugins/provider-hook-runtime.ts`: compose the primary matched provider's `wrapStreamFn` first, then iterate other providers whose id/aliases/hookAliases match the requested provider, dedupe by stable plugin/provider identity, pass the already-wrapped stream as `ctx.streamFn`, and preserve a targeted regression test covering primary plus hookAlias chaining and unrelated-provider skipping.
