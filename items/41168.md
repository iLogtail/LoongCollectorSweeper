## Summary

Keep open. Current `main` still initializes the TUI to the main session key when no `--session` is provided, and the TUI session refresh path only hydrates metadata for the already-selected key instead of switching to the most recently active session. The storage-bloat part is also still plausible on `main`: new sessions persist a `skillsSnapshot.prompt` string into each session entry.

## Evidence

- **Blank TUI startup still defaults to main session:** `resolveTuiSessionKey()` returns the agent main session when `raw` is empty for non-global scope, so the default startup path is still `agent:<agentId>:<mainKey>` rather than a latest-active lookup.
  - file: [src/tui/tui.ts:138](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/tui/tui.ts#L138)
  - command: `nl -ba src/tui/tui.ts | sed -n '132,160p'`
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **runTui applies that default immediately:** `runTui()` trims `opts.session` into `initialSessionInput` and then sets `currentSessionKey = resolveSessionKey(initialSessionInput)`. With no explicit session argument, startup still lands on the main session key.
  - file: [src/tui/tui.ts:289](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/tui/tui.ts#L289)

- **Post-connect agent refresh does not choose latest active session:** After agents are loaded, `applyAgentsResult()` reuses `resolveSessionKey(initialSessionInput)` and marks `initialSessionApplied`; it does not inspect recent-session ordering or switch away from the main session automatically.
  - file: [src/tui/tui-session-actions.ts:83](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/tui/tui-session-actions.ts#L83)
