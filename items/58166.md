## Summary

Current `main` does not contain this fix. `repairToolUseResultPairing()` still builds its pairing set from raw extracted tool calls and will synthesize missing `toolResult` messages with `toolCallId: call.id`, while `extractToolCallsFromAssistant()` only rejects empty strings, not whitespace-only IDs. The PR also adds regression coverage that current `main` lacks, so there is not strong evidence to close this PR conservatively.

## Evidence

- **Current main still synthesizes missing results from unfiltered tool call IDs:** `repairToolUseResultPairing()` reads `toolCalls = extractToolCallsFromAssistant(assistant)`, builds `toolCallIds` directly from those IDs, and later calls `makeMissingToolResult({ toolCallId: call.id, ... })` for every missing result. There is no trim/blank-ID filter in the current implementation.
