## Summary

Keep #39476 open. Current main still exposes requester session identity to A2A target runs, still runs the bounded reply-back plus final announce lifecycle for normal peer sessions, and still lacks a per-run guard that removes or blocks `sessions_send` from A2A target/reply/announce turns. The recent issue discussion also reports the behavior still reproducing, while the related fix PRs were closed unmerged and #44309 is a broader dispatch-mode request rather than an equivalent duplicate-delivery guard.

## Best Possible Solution

Keep the issue open and fix the core `sessions_send` A2A path. The likely maintainer path is a framework-level per-run guard that prevents A2A target/reply/announce turns from calling `sessions_send` back to the requester, plus narrowing or removing requester-session prompt exposure where feasible. Add regression coverage for duplicate visible delivery and keep #44309 separate as the broader dispatch-only/handoff semantics request.
