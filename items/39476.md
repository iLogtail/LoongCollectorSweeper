## Summary

Keep #39476 open. Current main still exposes requester session context to A2A target/reply/announce prompts, still runs the normal reply-back ping-pong plus announce flow for peer sessions, and still has no per-run hard guard that removes or denies `sessions_send` during those nested A2A turns. The existing parent-owned ACP-child skip is narrower than this issue, and #44309 tracks broader dispatch-only semantics rather than superseding the reverse-call duplicate bug.

## Best Possible Solution

Keep the issue open and fix the core A2A `sessions_send` path. Add a framework-level guard that prevents A2A target/reply/announce turns from calling `sessions_send` back to the requester, narrow or remove requester-session prompt exposure where feasible, preserve the existing parent-owned ACP-child skip, keep #44309 as the broader dispatch-only semantics discussion, and add regression coverage for normal peer A2A duplicate visible delivery.
