## Summary

Keep #39476 open. Current main still exposes requester-session context to A2A target/reply/announce prompts, still runs the normal requester-target ping-pong plus target announce flow, and still has no per-run tool policy that denies `sessions_send` during nested A2A turns. The linked one-way dispatch issue #44309 is adjacent but broader and does not supersede this reverse-call duplicate-delivery bug.

## Best Possible Solution

Keep the issue open and fix the core A2A `sessions_send` path. Add a framework-level guard that prevents target/reply/announce A2A turns from calling `sessions_send` back to the requester, remove or narrow requester-session prompt exposure where feasible, preserve the existing owned ACP-child skip, keep #44309 as the broader dispatch-only semantics discussion, and add regression coverage for peer A2A duplicate visible delivery.
