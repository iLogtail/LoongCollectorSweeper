## Summary

Current main still lacks the requested Gateway receive timestamp in inbound metadata. The shared inbound prompt metadata only emits the source-channel `timestamp`, the core message context and hook event types only expose `Timestamp`/`timestamp`, and channel-local receive timestamps such as Telegram `receivedAtMs` or WhatsApp `inboundAt` are used for debug/status paths rather than propagated to agent-visible metadata.

## Best Possible Solution

Keep the issue open. The best implementation is an additive, cross-channel Gateway receive timestamp, likely as `receivedAt`/`gatewayReceivedAt` on `MsgContext`, canonical inbound hook context, `message_received` metadata, and structured inbound user metadata, with channel ingress paths threading their receive time into the shared finalization path. Add focused tests for the prompt metadata and hook event mapping, plus docs for plugin authors.
