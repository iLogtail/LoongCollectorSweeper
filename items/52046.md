## Summary

Keep #52046 open. Current main has mitigations for busy queues and expensive heartbeat runs, but it still routes heartbeat agent turns through the same reply pipeline/global main lane rather than a separate non-blocking worker or dedicated heartbeat execution lane. The issue discussion also frames this as a valid feature request for an isolated heartbeat turn model, not something already fixed on main.

## Best Possible Solution

Keep the issue open for maintainer product/design work. The likely fix is a dedicated heartbeat execution boundary: a heartbeat-specific global lane or worker-like async path, plus explicit isolation rules for session state, delivery ordering, tool side effects, cancellation, and wake/retry behavior. The implementation would likely touch src/infra/heartbeat-runner.ts and the embedded agent lane plumbing, with regression tests proving an inbound user turn can start promptly while a heartbeat is already running without corrupting the shared session or delivery state.
