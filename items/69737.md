## Summary

Keep open. Current main partially improves the raw-error case by redacting/truncating lifecycle error text, but it still falls back to `lastAssistant.errorMessage` for unmatched provider errors instead of suppressing or replacing it with a generic user-facing message. The multi-text behavior also remains: the delivery path still extracts all visible assistant text blocks with newline joining and sends that combined text as the block reply.

## Best Possible Solution

Keep the issue open and split implementation review by behavior: first change the lifecycle delivery fallback so unmatched `stopReason=error` messages do not use `lastAssistant.errorMessage` as user-facing text, with regression coverage for generic OpenAI 5xx/request-id text; then handle multi-text assistant delivery as a deliberate policy change, preferably in the delivery layer or behind a scoped option so shared transcript/TUI extractors are not unintentionally changed.
