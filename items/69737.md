## Summary

Keep open. Current main has partial lifecycle error redaction, but the reported delivery problems are not fully resolved: unknown assistant `errorMessage` text can still be selected for user-facing lifecycle error emission, and message-end delivery still builds one visible reply by joining all matching assistant text blocks rather than choosing one. The issue comment’s split-blast-radius analysis is consistent with current code, so this should remain actionable or be split by a maintainer rather than auto-closed.

## Best Possible Solution

Keep #69737 open, preferably split into two maintainer-scoped fixes: first, make Slack/Telegram lifecycle delivery avoid raw unknown `lastAssistant.errorMessage` by using a generic/classified fallback while preserving raw details only in redacted logs; second, decide a delivery-specific policy for multiple assistant text blocks, such as last non-empty or configurable behavior, without unintentionally changing TUI/gateway/chat-history semantics. Add focused regression tests for a plain OpenAI 500 with request ID/help-link text and for multi-text assistant content in the delivery path.
