## Summary

Keep open. Current main still does not satisfy the requested delivery-layer behavior: unrecognized short assistant error text can still flow into user-facing error payloads/lifecycle events, and assistant visible text extraction still joins multiple matching text blocks instead of selecting one. The issue comment's recommendation to split Bug 1 from the broader multi-text policy question remains accurate; related #42243 is adjacent cron/announce delivery work, not a canonical duplicate for this Slack/Telegram assistant-content report.

## Best Possible Solution

Keep the issue open. Implement a delivery-level `stopReason=error` sanitizer that emits a generic fallback or suppresses delivery for unknown raw provider errors, then handle multi-text assistant messages with a scoped outbound policy such as last non-empty text for Slack/Telegram delivery. Add regression coverage for both the raw OpenAI-style 500 prose case and multi-text assistant content; consider splitting the multi-text policy into a narrower follow-up if maintainers want separate review ownership.
