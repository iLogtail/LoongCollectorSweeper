## Summary

Keep open. Current main still matches both reported behaviors: unclassified assistant errors can flow into delivery as formatted error payloads that fall back to raw text, and assistant visible-text extraction still joins multiple text content blocks rather than selecting one. The existing discussion correctly notes that Bug 1 is a narrow error-copy path while Bug 2 is a wider extractor policy question, so the best maintainer path is probably to split the fixes, not close this issue.

## Best Possible Solution

Keep the issue open, but have maintainers split or scope it clearly: first patch the delivery error path so unrecognized stopReason=error messages use a generic user-facing fallback and add a regression test for an OpenAI-style 500 with request ID/help URL; then decide the multi-text policy separately, preferably at the delivery boundary or behind a narrow option so shared transcript/TUI/session-dump extractors are not unintentionally changed.
