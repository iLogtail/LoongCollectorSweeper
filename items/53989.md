## Summary

Keep open. #53989 is recent, specific, and reports a real supported surface: Slack Socket Mode multi-account inbound routing with account-scoped agent bindings. Current main has code paths that should route named Slack accounts independently and skip unconfigured accounts independently, and current docs point to a likely config pitfall around Slack channel-name allowlists, but I did not find issue-specific confirmation, a linked fixing PR, or an exact regression test proving this reported 2026.3.23-2 config-taint scenario is resolved.

## Best Possible Solution

Keep the issue open and narrow it with maintainer evidence: request the sanitized current `channels.slack.channels`, `channels.slack.accounts`, `channels.slack.defaultAccount`, and `bindings` values, plus verbose routing/drop logs on current main or latest release. Add a focused Slack inbound unit/contract test for two configured named accounts with an unconfigured/disabled `default` account, account-scoped bindings, an allowed channel ID, and an @mention dispatching to the bound agent. If the reporter was using channel names, resolve with the stable-ID docs and the #46327 workaround pattern.
