## Summary

Keep open. Current main has narrow startup recovery and a few targeted startup writes, but it does not auto-run the full `doctor --fix` repair sequence on gateway start or provide a `doctor.autoFix`/protected-value mechanism. The issue's later comment also adds useful regression data showing why unconditional startup `doctor --fix` would need careful audit logging and protection semantics rather than cleanup closure.

## Best Possible Solution

Keep this open for maintainer product judgment. The best path is not a blanket startup `doctor --fix`; it should define a narrow, auditable startup repair contract, probably opt-in, with explicit logs/diffs and a way to skip or protect intentional config paths such as sandbox mode and channel/plugin enablement. If maintainers only want the already-shipped narrow legacy/startup recovery behavior, the issue should be explicitly reframed or closed by a maintainer rather than auto-cleaned.

## Risks / Open Questions

- Unconditionally running the full doctor repair sequence during gateway startup could silently rewrite intentional operator choices, matching the regression data in the issue comments.
- A safe implementation likely needs explicit opt-in, structured repair summaries, protected-path semantics, and tests around channel enablement, sandbox mode, plugin auto-enable, and last-known-good recovery interactions.
- Current main already performs some startup writes, so any implementation must avoid duplicating existing plugin auto-enable, auth-token persistence, control-UI origin seeding, and last-known-good recovery behavior.
