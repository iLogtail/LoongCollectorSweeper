## Summary

Keep open. Current main still lacks a narrow modifying hook between model reply generation and TTS directive parsing/synthesis. Existing hooks can observe model output, claim the whole dispatch path, or rewrite outbound delivery content after TTS has already run, but none provide the requested post-LLM/pre-TTS text transform for dynamic `[[tts:...]]` injection or per-message TTS skipping.

## Best Possible Solution

Keep #69307 open as an additive Plugin SDK design request. The likely path is a narrow modifying hook such as `before_tts` or a generalized reply-payload transform, added to `src/plugins/hook-types.ts` and `src/plugins/hooks.ts`, invoked from `src/auto-reply/reply/dispatch-from-config.ts` immediately before each `maybeApplyTtsToReplyPayload()` call, with docs and tests covering directive injection, skip-TTS behavior, routing/account/session fields, and conversation-access policy.

## Risks / Open Questions

- The hook contract needs maintainer API design: final replies only versus final/tool/block/synthetic TTS-only payloads, and whether the result should support `skip` or a more general payload transform.
- Because this exposes raw post-LLM assistant text to plugins, it should align with existing conversation-access policy and documentation.
