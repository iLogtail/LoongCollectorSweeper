## Summary

Keep #69307 open. Current main still lacks a modifying hook between final LLM text production and TTS directive parsing/synthesis. Existing hooks either observe only, run before model dispatch, claim the whole reply dispatch path, or run at delivery time after TTS has already been applied. Current TTS personas/config help with global, per-agent, channel, account, and host-local voice selection, but they do not cover the requested per-message/per-chat plugin routing seam.

## Best Possible Solution

Keep the issue open for maintainer/product design. The likely best path is a documented plugin API extension: a sequential modifying hook immediately before maybeApplyTtsToReplyPayload parses directives, with typed channel/session/account/target context, optional text rewrite, optional TTS skip, tests covering final/block/tool dispatch paths, and docs under the plugin hook and TTS pages. Current ClawHub plugins could then implement per-chat voice routing on top of that seam.

## Risks / Open Questions

- A new hook would need a precise ordering contract across final, block, tool, and synthetic TTS-only reply paths.
- Because the hook would expose and mutate final assistant text, maintainers should decide whether it falls under the existing conversation-access policy for non-bundled plugins.
- The proposed skip semantics need to be reconciled with existing TTS auto modes, per-session /tts chat state, and directive text cleanup.
