## Summary

Keep #69307 open. Current main still lacks a supported modifying hook between natural reply generation and TTS directive parsing. The dispatch path applies TTS before final delivery/dispatcher hooks, `llm_output` remains observation-only, `message_sending` is too late for voice selection, and `reply_dispatch` requires taking over dispatch. This is a missing public plugin/API seam rather than a ClawHub-only plugin request.

## Best Possible Solution

Keep the issue open for maintainer design. The likely fix belongs in the public plugin hook surface and dispatch pipeline: add a documented, additive modifying hook or reply-transform seam that runs after reply text is available and before `parseTtsDirectives()`/`maybeApplyTtsToPayload()`, with clear behavior for final, block, tool, synthetic, and skip-TTS cases plus SDK docs and tests.

## Risks / Open Questions

- The best API shape may be broader than the proposed `before_tts` name, for example a reply-transform hook that covers final/tool/block payloads consistently.
- Adding this seam would touch public plugin SDK contracts and the reply/TTS dispatch order, so it needs maintainer API design rather than a community plugin alone.
