## Summary

Keep open. Current main has partial session management via Gateway RPC, Control UI delete/reset/compaction, chat `/stop`, and `openclaw sessions cleanup`, but it still does not provide the requested CLI `sessions stop/kill/pause/resume` lifecycle commands or a general pause/resume/resource-limit control model.

## Best Possible Solution

Keep this open as the broader CLI and Control UI session lifecycle tracker. The likely maintainer path is to add narrow CLI commands on top of existing Gateway methods first, such as `openclaw sessions stop <key>` using `sessions.abort` for active runs and a clearly named delete/archive command using `sessions.delete`; then decide separately whether real pause/resume needs new protocol/runtime support. UI session cards should expose only controls backed by actual Gateway behavior, and resource-limit asks should either integrate with existing `session.maintenance`/agent concurrency settings or split into a separate scoped issue.

## Risks / Open Questions

- A future implementation should reuse the existing Gateway RPC/session cleanup paths instead of adding a parallel session state machine.
- Pause/resume may require provider or harness-specific runtime semantics; it should not be treated as a simple store flag unless active runtimes can actually pause work.
