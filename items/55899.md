## Close Comment

Thanks for the write-up. After a Codex review of current `main`, this looks better scoped as plugin/ClawHub work rather than an OpenClaw core issue. `VISION.md:52` says optional capability should usually ship as plugins and that memory is a special plugin slot. Current main already treats this area that way: `docs/concepts/memory.md:31` documents pluggable memory search and multiple memory backends, `docs/plugins/memory-wiki.md:10` documents a companion knowledge-layer plugin with bridge/search/apply flows, and `docs/plugins/sdk-overview.md:102` exposes memory prompt/corpus extension seams for additive experience systems. There is also an existing third-party path for automatic cross-session learning behavior in `docs/concepts/memory-honcho.md:9`. Given that scope, I’m closing this core feature request in favor of the plugin/ClawHub route. If you want to pursue it, the next actionable step would be a concrete plugin proposal describing the memory backend, retrieval flow, sharing model, and quality controls on top of those existing seams.
