## Summary

Keep open. Current `main` has overlapping WebUI model-ref refactors, but it does not implement this PR’s core raw-alias preservation for ambiguous bare server models; the code still falls back to provider-qualifying those values, and current tests codify that fallback.

## Evidence

- **Bare server models are still provider-qualified on current main:** `ui/src/ui/chat-model-ref.ts` still routes plain `model + provider` pairs through `buildQualifiedChatModelValue(...)`. For a bare alias like `glm-5` with a present provider, that path still produces `provider/model` unless the model already starts with that provider prefix.
  - file: [ui/src/ui/chat-model-ref.ts:49](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/ui/src/ui/chat-model-ref.ts#L49)
  - command: `nl -ba ui/src/ui/chat-model-ref.ts | sed -n '1,80p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Ambiguous-or-missing catalog cases still fall back to the qualified server value:** `resolvePreferredServerChatModelValue(...)` only keeps a raw model when catalog normalization succeeds uniquely. When that normalization does not change the bare id, it explicitly falls back to `resolveServerChatModelValue(...)`, so an ambiguous bare alias still gets rewritten with `modelProvider` on current `main`.
  - file: [ui/src/ui/chat-model-ref.ts:125](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/ui/src/ui/chat-model-ref.ts#L125)

- **Current UI tests expect qualified fallback when lookup fails:** `ui/src/ui/chat-model-select-state.test.ts` currently asserts that when catalog lookup fails, the active-session model resolves to `openai/gpt-5-mini`, not a preserved raw alias. That is the opposite of this PR’s `glm-5` raw-alias behavior.
