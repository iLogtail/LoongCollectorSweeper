## Summary

Keep #52442 open. Current main still closes the long-lived WhatsApp inbound listener through `closeInboundMonitorSocket()`, which calls `sock.ws?.close?.()` rather than Baileys `sock.end(...)`; related tests currently assert the raw WebSocket close path. Baileys v7.0.0-rc.9 exposes `end` on the returned socket and its teardown clears timers, removes WebSocket listeners, emits `connection.update: close`, and removes connection listeners, so the core reported cleanup gap remains. The linked fix PR #52443 was closed unmerged.

## Best Possible Solution

Keep the issue open and implement a Baileys-aware WhatsApp socket shutdown helper. Prefer `sock.end(new Error("OpenClaw WhatsApp socket close"))` when available, fall back to `sock.ws?.close?.()` for compatibility, and update inbound monitor plus connection-controller/login tests so the long-lived listener teardown asserts the normal Baileys shutdown path rather than only raw WebSocket close.
