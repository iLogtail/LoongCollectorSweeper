## Summary

Current main has the replay/liveness metadata from the parity work, but #64887's remaining user-facing payload request is not implemented. `livenessState` is still only in `meta` and lifecycle events; final reply payloads delivered to channels do not carry a structured state, and the abandoned path still sends generic text without the `abandoned` state.

## Best Possible Solution

Keep #64887 open. The narrow fix should surface terminal `blocked`/`abandoned` state through the user-facing reply path, either by adding a structured `livenessState`/status field to the shared reply payload contract and channel renderers, or by appending a deterministic final-payload footer as the issue suggests. Add regression coverage for at least strict-agentic `blocked` and incomplete-turn `abandoned`, and verify one real delivery surface such as CLI or a channel adapter renders the signal.
