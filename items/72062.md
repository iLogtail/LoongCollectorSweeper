## Summary

Keep #72062 open. Current main still has only tool-name allow/deny policy plus exec allowlist/approval policy; there is no schema, type, docs, or runtime hook for a per-agent deny list that matches resolved shell wrapper script paths before approval. Existing exec allowlist logic can positively match shell wrapper script operands, but it does not provide the requested deny-wins wrapper path policy, and a broad `sh`/`bash` allowlist entry can still satisfy the exec allowlist before script-specific intent is considered.

## Best Possible Solution

Keep the issue open and treat it as a focused core exec-policy feature request. The best implementation path is to add a documented deny-wins wrapper/script path policy, likely under the exec policy surface such as `tools.exec.denyWrappers` or `agents.list[].tools.exec.denyWrappers`, validate it in the strict config schema/types, resolve shell/interpreter script operands with the existing wrapper parsing helpers, apply it before approval prompts and before allowlist satisfaction, and cover gateway plus node host paths with config/schema/runtime/docs tests. Related #69570 and #69386 can continue tracking approval/stop cascade behavior; they do not supersede this wrapper-path policy gap.
