## Summary

Keep #72062 open. Current main has per-agent tool-name allow/deny and exec allowlist controls, and it now has some shell-script path matching for positive allowlist entries, but it does not implement a documented per-agent negative policy for exec wrapper/script paths. The requested `agents.list[].tools.deny_wrappers` field is absent from the strict config schema/types, and the existing tool policy still filters only `tool.name`, so `tools.deny: ["github__*"]` cannot inspect `exec` command arguments.

## Best Possible Solution

Keep the issue open and implement this as a core exec policy gap, preferably by adding a documented, typed per-agent config surface under `agents.list[].tools` or `agents.list[].tools.exec`. Reuse the existing shell wrapper/script candidate resolution in `src/infra/exec-approvals-allowlist.ts`, apply deny-before-approval/allowlist at exec dispatch for gateway and node paths, make deny win over allowlist, and add schema, docs, and regression tests for `sh|bash <script>` plus interpreter script operands.
