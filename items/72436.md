## Summary

Keep PR #72436 open. Current main has adjacent session persistence, manual compaction, outbound delivery diagnostics, and a write-ahead outbound delivery queue, but it does not implement the PR's central feature: an append-only `recovery-events.jsonl` sidecar with sanitized session lifecycle breadcrumbs for inbound metadata, route resolution, manual `/compact`, and outbound success/failure. The related session-recovery PRs in the provided context are adjacent but do not supersede this narrower recovery-event log.

## Best Possible Solution

Keep the PR open for maintainer review. The likely best path is a focused core implementation of a best-effort, append-only, redacted session recovery event log in `src/config/sessions`, with narrow call sites in inbound session metadata, route resolution, manual compaction, and outbound delivery, plus tests for redaction, permissions, non-blocking failures, and no message-body logging. Adjacent recovery queue and passive checkpoint work should remain separate unless maintainers choose a unified recovery ledger design.
