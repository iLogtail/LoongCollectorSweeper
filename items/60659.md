## Summary

Keep #60659 open. Current main still implements session reset policy at global, chat-type, and channel scope only. Agent and binding config schemas do not accept a reset policy, and the inbound session initialization path only resolves a channel override before applying type/global reset policy. ACP `oneshot` mode is a separate ACP-specific runtime mode, not a generic per-agent or per-binding `session.reset` policy.

## Best Possible Solution

Keep the issue open. Add an explicit, additive per-agent and/or per-binding reset policy surface, wire it into session reset resolution for inbound messages and other freshness checks, define precedence against `session.resetByChannel` and `session.resetByType`, and cover the new behavior with strict schema, generated config metadata/help, docs, and focused tests.

## Risks / Open Questions

- ACP `oneshot` mode may be a partial workaround for ACP-bound workflows, but it does not cover normal route bindings or embedded agents and should not be treated as implementing this request.
- A correct implementation needs maintainer-approved precedence, likely binding > agent > channel > type > global, plus updates to config docs, generated schema/help, and all freshness call sites.
