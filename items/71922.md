## Summary

Keep open. Current main still collapses the reported empty/no-details/catchall model fallback paths to `unknown`, and the failover reason unions, cooldown policy, and tests have not been updated for `empty_response`, `no_error_details`, or `unclassified`. The provided related item #49190 is about cron delivery status and does not supersede this model-fallback issue.

## Best Possible Solution

Keep the issue open and implement the split in the model fallback classification path: add the three labels to the relevant failover/runtime reason contracts, map empty/no-status to `empty_response`, exact no-details payloads to `no_error_details`, and unmatched message-bearing failures to `unclassified`; preserve raw message previews for unclassified errors; extend cooldown probing for the transient subtypes; and add focused regression coverage in the classifier, cooldown policy, model fallback attempt logging, and fallback notice/observability surfaces.

## Risks / Open Questions

- Changing these labels touches a public runtime/fallback contract, so the implementation should update `FailoverReason`, `AgentRuntimeFailoverReason`, tests, observability expectations, and any plugin/provider docs that expose failover reasons.
- The maintainers should decide whether the new labels are user-facing `FailoverReason` values, provider-runtime-only diagnostic kinds, or both; the issue asks about fallback events and cooldown behavior, so a provider-runtime-only change would likely be incomplete.
