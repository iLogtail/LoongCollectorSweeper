## Summary

Current main does not show a maintainer-grade fix for #40179. Host exec still launches the user shell via `SHELL -c`, passes inherited env into that shell, and documents `OPENCLAW_SHELL=exec` as a hook for shell/profile rules; that means startup files can still rewrite inherited variables before the command runs. I found PATH-merge and marker coverage, but no code path or regression test that proves daemon env wins over shell startup files for exec.

## Evidence

- **Host exec still uses the configured shell with `-c`:** `getShellConfig()` reads `process.env.SHELL` and returns that shell with `["-c"]` on non-Windows hosts, rather than a no-startup-files mode.
  - file: [src/agents/shell-utils.ts:67](https://github.com/openclaw/openclaw/blob/b98a6a46fb54145a33e753101c02f0d7d7e0bc3b/src/agents/shell-utils.ts#L67)

That preserves the inherited env, but still lets the shell's own startup behavior run first.
