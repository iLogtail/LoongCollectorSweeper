## Summary

Current `main` still appears to have the reported Google Chat bug. The channel creates a visible `_... is typing..._` placeholder before reply dispatch, but it only clears or repurposes that placeholder inside the outbound delivery path. Exact `NO_REPLY` final payloads are normalized away before delivery, so a reaction-only turn that ends with `NO_REPLY` never reaches the Google Chat cleanup path.

## Evidence

- **Typing placeholder is created before reply dispatch:** `processMessageWithPipeline()` sets `typingIndicator` to message mode, sends `_... is typing..._`, and stores the returned `typingMessageName` before calling reply dispatch.
  - file: [extensions/googlechat/src/monitor.ts:250](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/extensions/googlechat/src/monitor.ts#L250)
  - command: `nl -ba extensions/googlechat/src/monitor.ts | sed -n '247,314p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Placeholder cleanup only happens from delivery:** The dispatch `deliver` callback passes `typingMessageName` into `deliverGoogleChatReply()`, and that function only deletes/updates the placeholder when it is actively delivering reply text or media.
  - file: [extensions/googlechat/src/monitor.ts:291](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/extensions/googlechat/src/monitor.ts#L291)
  - command: `nl -ba extensions/googlechat/src/monitor.ts | sed -n '247,440p'`

- **Exact NO_REPLY payloads are dropped before `deliver` runs:** `normalizeReplyPayload()` returns `null` for an exact silent token with no other content, and `createReplyDispatcher()` returns early when normalization yields `null`, so no channel delivery callback runs for that final payload.

  - file: [src/auto-reply/reply/reply-dispatcher.ts:178](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/auto-reply/reply/reply-dispatcher.ts#L178)
