## Summary

Do not close #50154. The report is recent, has a concrete foreground-vs-LaunchAgent reproduction, and the only issue comment usefully identifies a likely LaunchAgent environment gap. Current main has adjacent daemon environment hardening, especially deterministic PATH construction and service env filtering/preservation, but I did not find source, tests, docs, or a canonical related issue proving the LAN GitLab SSH failure is fixed or superseded.

## Best Possible Solution

Keep the issue open. The best maintainer path is to reproduce on macOS with a LaunchAgent gateway and a LAN SSH remote, compare foreground vs launchd `PATH`, `HOME`, `SSH_AUTH_SOCK`, `ssh -G`, and `git ls-remote`, then either add a safe macOS service-env/SSH-agent resolution mechanism or document a supported LaunchAgent setup. A focused regression test should cover the install-plan/service-env behavior once the intended contract is chosen.

## Evidence

- **Concrete open bug report:** Issue #50154 reports that macOS LaunchAgent-managed gateway host exec cannot run `git ls-remote origin` / `git push` to a LAN GitLab over SSH, while the same Telegram-triggered workflow succeeds after stopping the service and running the gateway in the foreground. The sole comment identifies LaunchAgent PATH or SSH_AUTH_SOCK differences as the likely root cause.
