## Summary

Keep this issue open. The reported failure is a concrete Linux/pnpm global startup regression, and current main still has the same eager core logging dependency on bare `tslog` during CLI startup. The latest published release tarball still contains built startup chunks importing `Logger` from `tslog`, so there is not enough evidence that current main or the latest release has fixed the observable `openclaw onboard --install-daemon` failure.

## Best Possible Solution

Keep this issue open and verify it with a packaged pnpm global install on Node 24. If it reproduces, fix the CLI startup/logging packaging path so `openclaw onboard --install-daemon` does not depend on a fragile bare `tslog` resolution before subcommand dispatch, or bundle/pin/import the logger dependency in a resolver-safe way. Add a packaged install smoke or release-check coverage for the pnpm global startup path.
