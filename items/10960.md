## Summary

Keep #10960 open. Current main still does not provide the requested mid-stream soft-steer behavior for inbound messaging-channel queue mode: steer injection is gated on streaming state, non-streaming active runs fall back to follow-up queuing, and the public docs still describe next-boundary/fallback semantics rather than clean pause/abort/resume with injected context. The related implementation PR #64758 is still open and unmerged in the provided GitHub context, so the issue remains a valid tracker.

## Best Possible Solution

Keep #10960 open as the canonical product/runtime tracker. A complete fix should update the inbound auto-reply queue and embedded/reply-run control path so steer-mode input during active non-streaming generation or tool waits can cleanly pause or abort active work, preserve usable partial context, append the steering message, and resume in the same session. Regression coverage should distinguish streaming steer success, non-streaming/tool-wait fallback, steer-backlog/followup behavior, explicit gateway sessions.steer, and failure handling when a backend cannot accept same-turn steering.
