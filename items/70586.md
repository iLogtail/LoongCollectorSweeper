## Summary

Keep PR #70586 open. Current main still has a real narrow gap for Slack message-event paths that return before `createSlackMessageHandler`, but the PR as written should not be closed as implemented or merged unchanged because normal message/app_mention flows already call `trackEvent` in the shared message handler and would be double-counted.

## Best Possible Solution

Keep the PR open and ask for a surgical revision: preserve exactly-once tracking for normal `message` and channel `app_mention` flows, while adding tracking only for message registrar branches that return before `createSlackMessageHandler` can run, such as authorized subtype system events and DM/mpim app_mention skips. Update tests to cover both the new early-return tracking and the existing no-double-count behavior.
