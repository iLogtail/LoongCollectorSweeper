## Summary

Keep PR #70586 open. Current main still has the narrow Slack message-event status gap the PR is trying to address, but the submitted implementation is not safe to close or merge as-is: normal message and app_mention flows already call `trackEvent` through `createSlackMessageHandler`, so the PR's unconditional registrar-level callback would double-update liveness/status for handled messages. Also, the socket-health stale rationale is now superseded by #69833 because stale-socket health uses provider-proven transport activity, not app-event freshness.

## Best Possible Solution

Keep the PR open and request a narrower revision: keep `createSlackMessageHandler` as the single tracking point for normal handled messages, thread a registrar-level callback only for message-event branches that return before that handler, and add tests proving normal message/app_mention flows track once, early-return accepted branches track once, and mismatched or unauthorized events do not track. The PR description should also be reframed away from stale-socket health, because current main's stale-socket policy is already transport-activity based after #69833.
