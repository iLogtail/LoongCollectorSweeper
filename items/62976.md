## Summary

Keep open. Current main has partial mitigation: plugin runtime/register failures are downgraded to plugin load errors, and startup can restore a last-known-good config snapshot. But invalid third-party plugin config still makes the config snapshot invalid, gateway startup still aborts on that invalid snapshot, and `openclaw doctor --fix` does not quarantine or remove the bad plugin config.

## Evidence

- **Gateway startup still hard-fails on an invalid config snapshot:** Startup first tries last-known-good / prefix recovery, then unconditionally throws if the config snapshot is still invalid, with only a doctor hint. That means a bad plugin config still blocks boot when no valid recovery snapshot exists.
  - file: [src/gateway/server-startup-config.ts:72](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/gateway/server-startup-config.ts#L72)
  - command: `nl -ba src/gateway/server-startup-config.ts | sed -n '60,113p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Plugin config schema errors remain fatal config-validation issues:** `validateConfigObjectWithPlugins` pushes plugin config schema failures into `issues`, not warnings. The dedicated test still expects invalid third-party plugin config to make validation fail.
  - file: [src/config/validation.ts:1153](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/config/validation.ts#L1153)

  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Doctor repair only removes stale missing plugin refs, not bad plugin config:** The repair sequence includes `maybeRepairStalePluginConfig`, and that helper only deletes stale `plugins.allow` / `plugins.entries` ids after manifest discovery. It does not disable or quarantine entries whose plugin exists but whose config is invalid.
  - file: [src/commands/doctor/repair-sequencing.ts:58](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/commands/doctor/repair-sequencing.ts#L58)

  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Doctor intentionally preserves invalid config and only reports it at the end:** The doctor flow has an explicit test asserting invalid config is preserved for repairs, and the final doctor contribution only prints invalid-config issues rather than auto-removing bad plugin config.
  - file: [src/commands/doctor-config-flow.test.ts:1345](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/commands/doctor-config-flow.test.ts#L1345)

Gateway startup uses the normal gateway plugin load path without setting that flag, so runtime init failures no longer imply the same hard boot failure as config-validation failures.
