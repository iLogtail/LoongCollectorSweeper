## Summary

#8130 should remain open. Current main still lacks proactive active-channel retry/failover notifications and any `agents.defaults.model` notification config. The code now tracks fallback state, emits lifecycle/status/UI events, and can prepend a final `Model Fallback:` notice, but that user-visible notice is still gated behind verbose mode and does not cover the silent retry/failover window requested by the issue.

## Best Possible Solution

Keep #8130 open as the canonical core gateway/reply-runner UX gap. Implement configurable proactive retry/failover notices near `runWithModelFallback`, likely by wiring sanitized candidate-failure and successful-fallback events through the active-channel block reply delivery path after an optional delay threshold, with tests proving visible retry/failover notices work when verbose mode is off and do not duplicate existing verbose/status/UI notices.
