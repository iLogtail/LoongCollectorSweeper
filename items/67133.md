## Summary

Keep open. Current `main` shows that cron jobs are supposed to persist `lastRunAtMs`, but the issue also references hook names and files that do not exist in the OpenClaw tree, so there is not high-confidence evidence that the whole report is already fixed, obsolete, or clearly out of core scope. The item is recent and still plausibly describes a real cron-state or custom-hook integration problem that needs maintainer triage rather than auto-close.

## Evidence

- **Cron completion records `lastRunAtMs` on current main:** The cron service writes `job.state.lastRunAtMs = result.startedAt` when applying a finished run result, so the intended current-main behavior is to persist last-run history for completed jobs.
  - file: [src/cron/service/timer.ts:417](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/cron/service/timer.ts#L417)

- **Current tests assert `lastRunAtMs` is preserved:** Regression tests explicitly expect `lastRunAtMs` to be set after both successful and failed runs, which supports that missing history is not an intentional behavior on `main`.
  - file: [src/cron/service/timer.regression.test.ts:1060](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/cron/service/timer.regression.test.ts#L1060)
