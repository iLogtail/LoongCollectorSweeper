## Summary

Keep open. Current main has relevant safeguards, including global lane concurrency caps, per-session serialization, heartbeat wake coalescing/retry behavior, and lower default main/subagent concurrency than the reporter cited. However, the central report is still a plausible core/gateway performance bug: heartbeat turns can still spawn background subagents onto the shared `subagent` lane, there is no heartbeat-origin-specific subagent throttling or CPU/resource isolation, and the docs still describe subagents as sharing the same gateway process resources. A related optional inference-guard PR was closed unmerged and is not present on main, so it does not supersede the issue.

## Best Possible Solution

Keep this issue open and treat it as a core gateway/agent scheduling performance bug. The best path is to add or validate a heartbeat-origin resource policy: cap or serialize subagent spawns initiated by heartbeat turns, expose/measure lane and subagent backlog state, and document/configure safe defaults for multi-agent heartbeat deployments. A focused regression or stress test should cover several heartbeat-enabled agents spawning background subagents while Feishu or another channel remains responsive.
