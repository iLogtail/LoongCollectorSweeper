## Summary

Keep open. Current `main` has queue caps and heartbeat skip guards, but nothing in the inspected code or docs shows a shipped fix for the reported gateway CPU saturation under concurrent heartbeat/subagent load. The report is recent, specific, and still plausible against the current architecture.

## Evidence

- **Issue is recent and concrete:** Issue #63149 was opened on 2026-04-08 with a specific multi-agent heartbeat reproduction, concrete timeout symptoms, version `2026.4.5`, and channel details (`Feishu WebSocket channel x 5`). That is enough detail to keep investigating rather than close as stale or incoherent.

- **Current main still relies on global queue caps, not CPU/resource isolation:** Gateway lane concurrency is still limited only by configured queue caps: cron, main, and subagent lanes are set from config at startup. This shows queue shaping exists, but not a clear fix for single-process CPU saturation under bursty background work.
