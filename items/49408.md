## Summary

Keep open. Current main partly addresses the commenter’s low-level `Dirent.isDirectory()` concern by considering directory symlinks during skill-root child discovery, but it still does not implement the reported workflow: a symlink under `~/.openclaw/skills/<name>` pointing to an arbitrary external skill directory is realpath-checked and skipped because it resolves outside the managed skills root. The supported current workaround is `skills.load.extraDirs`, but that is not the same as resolving external symlinked children under the managed skills directory. The related #60144/#60155/#60513/#60145 context concerns audit-boundary alignment for workspace skill symlink escapes and does not supersede this managed-skill loading request.

## Best Possible Solution

Keep the issue open for maintainer product/security judgment. The likely path is either to document external managed-skill symlink escapes as unsupported and point users to `skills.load.extraDirs`, or to explicitly add a trusted symlink/linked-root contract for managed skills with clear containment rules, audit coverage, and regression tests for `~/.openclaw/skills` symlinks.

## Evidence

- **Issue reproduction and discussion:** #49408 reports `ln -s /Users/ray/my-custom-skill ~/.openclaw/skills/my-custom-skill` not appearing during session initialization. The first comment correctly calls out the `fs.readdir(..., { withFileTypes: true })` symlink-directory pitfall, and the reporter agrees that following symlink directories would address the intended workflow.

This means the raw readdir/Dirent omission is no longer the only blocker.
