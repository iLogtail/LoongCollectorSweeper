## Summary

Current main still matches the reported BlueBubbles bug: the plugin-auth webhook route authenticates inbound webhook requests against `account.config.password`, the same field used for BlueBubbles API/server auth, and the public docs still instruct operators to put that password in the webhook URL. The linked PR #70757 is open/unmerged, so the fix is not shipped on main.

## Best Possible Solution

Keep #70755 open and review or land PR #70757, or an equivalent focused fix, that adds a dedicated BlueBubbles inbound webhook secret, authenticates webhooks only against that secret, updates schema/setup/secret metadata/docs, and adds regression tests for rejecting the inherited server-password path.
