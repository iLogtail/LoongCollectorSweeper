## Summary

Current main still reuses `channels.bluebubbles.password` as the inbound BlueBubbles webhook credential, and the docs still instruct operators to publish that password in the webhook URL. The linked fix PR #70757 is open and unmerged, so #70755 remains actionable and should stay open.

## Best Possible Solution

Keep #70755 open. Review and land PR #70757 or a narrower equivalent that adds a distinct BlueBubbles webhook secret surface, authenticates inbound webhooks only against that secret, keeps `channels.bluebubbles.password` for BlueBubbles API access, and updates setup, docs, secret metadata, config validation, and regression tests consistently.
