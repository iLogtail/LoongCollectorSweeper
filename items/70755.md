## Summary

Keep #70755 open. Current main still authenticates BlueBubbles webhooks against `channels.bluebubbles.password`, the BlueBubbles config/schema only exposes `password` plus `webhookPath`, and public docs still tell operators to publish `?password=<password>` on the webhook URL. The linked fix PR #70757 is open and unmerged, so it is useful related work but does not supersede or ship the fix yet.

## Best Possible Solution

Keep the issue open until a reviewed fix lands on main. The likely product path is to add a dedicated BlueBubbles `webhookSecret` configuration and secret registration, authenticate inbound webhook requests only against that value, update setup/docs/generated config metadata, and keep regression tests proving the BlueBubbles server API password is no longer accepted as the public webhook credential.
