## Summary

Keep open. Current `main` still applies `messages.responsePrefix` on the normal reply path, but the shared message-tool send path forwards raw outbound text without any corresponding prefix resolution or normalization step.

## Evidence

- **Direct replies still prepend responsePrefix:** `routeReply()` resolves `responsePrefix` and passes it into `normalizeReplyPayload()`, which is the normal auto-reply path.
  - file: [src/auto-reply/reply/route-reply.ts:112](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/auto-reply/reply/route-reply.ts#L112)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **normalizeReplyPayload is where prefixes are actually added:** `normalizeReplyPayload()` prepends the effective prefix when text does not already start with it.
  - file: [src/auto-reply/reply/normalize-reply.ts:112](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/auto-reply/reply/normalize-reply.ts#L112)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Message tool send path does not resolve responsePrefix:** `handleSendAction()` reads `message`, applies cross-context markers, then calls `executeSendAction()`; there is no `resolveEffectiveMessagesConfig()`, `createReplyPrefixContext()`, or `normalizeReplyPayload()` on this path.
  - file: [src/infra/outbound/message-action-runner.ts:491](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/infra/outbound/message-action-runner.ts#L491)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Shared outbound send forwards raw content unchanged:** `sendMessage()` builds the outbound payload directly from `params.content` and sends it through direct delivery or gateway send without any response-prefix step.
  - file: [src/infra/outbound/message.ts:240](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/infra/outbound/message.ts#L240)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Config/docs still describe responsePrefix as outbound-wide:** The public config type says `responsePrefix` is a prefix auto-added to all outbound replies, and the docs describe it as an outbound prefix cascade.
  - file: [src/config/types.messages.ts:89](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/config/types.messages.ts#L89)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)

## Risks / Open Questions

- A channel plugin could add its own prefixing, but the shared core message-tool send path on `main` does not.
