## Summary

Keep PR #45902 open. Current main does not implement the PR's central Telegram behavior: `/model` in a Telegram topic/thread is still persisted only on the active session, and new thread sessions only inherit an existing parent `providerOverride`/`modelOverride` if that parent was already set separately. The PR is not obsolete, but it also is not ready to treat as implemented or closeable because its branch contains unrelated memory/gateway/tooling changes and the provided review context flags unresolved Telegram scoping/first-message issues.

## Best Possible Solution

Keep the item open, but ask for a focused Telegram-only PR: add explicit parent-chat future-thread default state, update both `/model` persistence paths and Telegram model-picker/native-command paths if product wants parity, seed only genuinely new Telegram threads from the same chat parent, and cover forum topics plus DM threaded mode including main-scoped keys, first-message metadata precreation, reset-to-default clearing, and cross-chat negative cases. Drop unrelated memory/gateway/recovery tooling from this PR or split it separately.
