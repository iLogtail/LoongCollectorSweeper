## Summary

Keep #25883 open. Current main still implements Gateway-local scheduling, task tracking, and in-process queues rather than Redis/BullMQ-style distributed job orchestration with separate workers, shared queue state, dead-letter handling, and horizontal coordination. The linked ScalyClaw prototype is useful prior art, and the existing maintainer/Codex comment already correctly kept the issue open.

## Best Possible Solution

Keep #25883 open for maintainer design review. The best path is to decide whether OpenClaw needs a core distributed scheduling/job API, then either implement a manifest/plugin-capable queue abstraction with worker, retry, visibility, and cron contracts, or document horizontal orchestration as an external deployment pattern. The ScalyClaw prototype should be treated as useful prior art for that design discussion.
