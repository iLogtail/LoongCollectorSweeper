## Summary

Current main still has the reported product gap: main-session `systemEvent` cron jobs with `wakeMode: "now"` can block on the main heartbeat turn, while timeout policy still assigns all `systemEvent` jobs the generic 10-minute cron timeout. The related duplicate #70325 reinforces the same symptom, and the two referenced fix PRs were closed unmerged, so #50621 remains the canonical open tracker.

## Best Possible Solution

Keep #50621 open and fix the main-session cron contract explicitly. The likely maintainer path is to change `src/cron/service/timeout-policy.ts` and/or `src/cron/service/timer.ts` so `sessionTarget: "main"`, `payload.kind: "systemEvent"`, `wakeMode: "now"` is either genuinely enqueue-and-wake fire-and-forget, or uses a documented/configurable timeout aligned with long agent turns. Add regression coverage beside `src/cron/service/timeout-policy.test.ts` and `src/cron/service.runs-one-shot-main-job-disables-it.test.ts`, then update `docs/automation/cron-jobs.md` if any timeout remains user-visible.

## Risks / Open Questions

- Changing this path from blocking to fire-and-forget affects current `wakeMode: "now"` completion semantics, including one-shot deletion and run-history status timing.
- A narrower fix such as using the 60-minute agent-turn safety timeout would reduce false failures but would not fully satisfy the issue’s requested fire-and-forget semantics.
