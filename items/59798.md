## Summary

Keep open. Current main still contains the ACP `streamError` throw path that this PR targets, and the unrelated OAuth type update is also not on main. The PR is not obsolete, but it needs maintainer/author revision rather than cleanup closure because the existing review comment identifies a plausible failure-bookkeeping regression if the PR moves error handling after fallible cleanup.

## Best Possible Solution

Keep the PR open and request a revision. The ACP manager should avoid the locally caught `streamError` throw while preserving failure bookkeeping before any fallible cleanup can mask the original error, or explicitly contain cleanup failures. Add a focused regression for a stream-error turn with failing reconciliation/cleanup, keep the existing fresh-handle retry behavior covered, and split the `OAuthProviderId` deprecation cleanup into a separate PR unless maintainers accept the broader scope.
