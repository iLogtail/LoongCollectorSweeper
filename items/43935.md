## Summary

Current main still has the coarse channel-reload behavior described in #43935: channel plugin reload prefixes become a channel-wide `restartChannels` entry, and the hot reload handler calls `stopChannel(name)` / `startChannel(name)` without an account id. The lower-level channel manager does support account-scoped start/stop, and the linked PR #43938 remains open rather than shipped, so this is still actionable in the repository.

## Best Possible Solution

Keep #43935 open and land #43938 or an equivalent core fix: add an account-scoped restart representation to `GatewayReloadPlan`, detect strict `channels.<channel>.accounts.<accountId>.*` paths, make full-channel changes win for mixed/global edits, and apply account-scoped reload via `stopChannel(channelId, accountId)` / `startChannel(channelId, accountId)` with regression tests in `src/gateway/config-reload.test.ts` and `src/gateway/server.reload.test.ts`. Treat broader in-flight reply drain/retry behavior as a separate follow-up if it remains observable after account-scoped reload ships.

## Risks / Open Questions

- Current main only adds channel reload deferral around active work; it does not reduce the blast radius for unrelated accounts on the same channel.
- If `gateway.reload.deferralTimeoutMs` is set, active work can still be interrupted once the timeout expires because the eventual action is still a whole-channel restart.
