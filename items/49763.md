## Summary

Keep #49763 open. Current main has a TUI Loader for busy states and a later streaming watchdog, but the reported Ctrl+O/state-mutation path still exits the busy state and stops the loader, and the normal queued-send path does not preserve an active-run spinner across all state transitions.

## Best Possible Solution

Keep the issue open and fix the TUI status model so UI-only mutations such as Ctrl+O do not replace busy run state while activeChatRunId, pendingOptimisticUserMessage, or queued work indicates in-flight processing. Add focused regression coverage for Ctrl+O during waiting/streaming and for sending a normal follow-up while another run is active.

## Evidence
