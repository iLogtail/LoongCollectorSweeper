## Summary

Keep #60275 open. Current main supports one heartbeat configuration per enabled agent, with a single optional session override and transient targeted wake paths, but it does not provide native, persistent, operator-visible per-session heartbeat entries. Cron can persist per-session schedules and wake heartbeat, but docs and code keep that as a separate scheduled-task system rather than first-class heartbeat-entry scheduling.

## Best Possible Solution

Keep the issue open for maintainer product design. A complete solution would extend the core heartbeat contract with persistent session-scoped entries, explicit scheduling/concurrency semantics, restart-safe state, per-entry run status, and CLI/RPC/UI inspection and mutation surfaces, or else explicitly document cron as the supported alternative and close manually with that product decision.

## Risks / Open Questions

- Cron main-session jobs and forced heartbeat wakes cover some operational workarounds, so a future design should explicitly define migration/overlap instead of duplicating scheduler semantics.
- The request is a product/API design feature, not a narrow bug; maintainer judgment is still needed on whether heartbeat entries belong in core or should be reframed around cron plus docs.
