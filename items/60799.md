## Summary

Keep #60799 open. Current main still does not load legacy top-level array cron stores: both async and sync loaders discard arrays by converting them to `{}` and then reading only `parsedRecord.jobs`. The service still persists mutations from that in-memory store, so a first add after an array-shaped `jobs.json` can still rewrite the file without the pre-existing jobs. The related PR #61113 described the right fix but the provided GitHub context says it was closed unmerged, and current main does not contain its proposed normalization helper or regression tests.

## Best Possible Solution

Keep the issue open and implement the focused loader fix: normalize parsed cron stores in one helper that accepts both `{ version: 1, jobs: [...] }` and legacy `[{...job}]` formats, use it from both `loadCronStore()` and `loadCronStoreSync()`, ensure doctor can see and persist the migration, and add regression tests for plain-array load plus load-add-save preserving existing jobs.

## Risks / Open Questions

- Closing now would leave a still-plausible data-loss path untracked for users with legacy top-level array `jobs.json` files.
- The documented `openclaw doctor --fix` guidance is not sufficient for this exact format because doctor uses the same loader and returns early after the array is collapsed to zero jobs.
