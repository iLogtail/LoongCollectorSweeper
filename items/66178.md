## Summary

Keep open. I did not find strong evidence that current `main` fixes the reported LM Studio + MemPalace failure. The current transport still executes tools only from structured OpenAI `tool_calls`, while the UI/history sanitizers only hide leaked XML-style tool text. I found no support for the bracketed `[mempalace_*] ... [END_TOOL_REQUEST]` format reported here, and current docs still warn that local OpenAI-compatible backends can fail on full OpenClaw agent turns even when tiny direct `/v1/chat/completions` tests work.

## Evidence

- **Structured tool calls only:** `src/agents/openai-transport-stream.ts` converts streamed Chat Completions tool use only from `choice.delta.tool_calls`; if `finish_reason` says tool use but no parsed tool calls exist, OpenClaw downgrades the turn back to `stop` instead of recovering a plain-text tool request.
  - file: [src/agents/openai-transport-stream.ts:1308](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/agents/openai-transport-stream.ts#L1308)
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Sanitizer is display-only and XML-specific:** `src/shared/text/assistant-visible-text.ts` explicitly strips XML-style leaked tool payloads such as `<tool_call>`, `<function_call>`, and `<tool_calls>` from visible assistant text. This is a visibility cleanup path, not a tool-execution fallback, and it does not cover the bracketed `END_TOOL_REQUEST` syntax from the issue body.
  - file: [src/shared/text/assistant-visible-text.ts:13](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/shared/text/assistant-visible-text.ts#L13)
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Tests cover Qwen/XML leakage, not bracketed MemPalace blocks:** Current tests assert stripping of `<tool_call>` and Qwen-style XML payloads from visible replies, which confirms the shipped behavior is about hiding leaked XML markup rather than executing raw bracketed tool text.
  - file: [src/shared/text/assistant-visible-text.test.ts:108](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/shared/text/assistant-visible-text.test.ts#L108)
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **No `END_TOOL_REQUEST` support on main:** A repository search for the exact marker reported in the issue returned no matches, so there is no obvious parser/repair path for `[...][END_TOOL_REQUEST]` blocks on current `main`.
  - command: `rg -n -F "END_TOOL_REQUEST" src extensions docs test`
- **Current docs still warn about local backend incompatibility on full agent turns:** The local-model docs now recommend LM Studio with `openai-responses`, and they explicitly warn that some smaller/stricter local OpenAI-compatible backends can work for tiny direct `/v1/chat/completions` tests yet still fail on normal OpenClaw agent turns; suggested mitigations are `localModelLean` or disabling tools, not a claimed fix.
  - file: [docs/gateway/local-models.md:14](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/docs/gateway/local-models.md#L14)
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Prior shipped fix was narrower:** The changelog entry on current `main` says OpenClaw strips Qwen-style XML tool payloads from visible replies. That shipped fix is narrower than this report's bracketed MemPalace block plus hang, and it predates the issue.
  - file: [CHANGELOG.md:1144](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/CHANGELOG.md#L1144)
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
