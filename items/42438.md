## Summary

Keep #42438 open. Current main already has the silent-drop half for WhatsApp `dmPolicy: "allowlist"`, but it does not implement the requested owner notification/webhook path for blocked unauthorized DMs. The blocked DM path returns before normal inbound processing and plugin `message_received` hooks only run after admitted messages, so a ClawHub plugin cannot currently observe these blocked attempts without a new core/channel seam.

## Best Possible Solution

Keep the issue open and implement an explicit, opt-in blocked-DM notification path in the WhatsApp/channel access-control layer. A good design would keep `allowlist` as the silent deny mode, add a separate `notifyOnBlocked`/blocked-attempt event or owner notification setting, and expose a typed plugin/API seam only if plugins are intended to handle blocked attempts without receiving ordinary private WhatsApp traffic by default.

## Risks / Open Questions

- A notification feature needs careful privacy design so blocked sender identifiers/content are not exposed to untrusted plugins or logs by default.
- The implementation should preserve the current no-reply/no-read-receipt behavior for silent allowlist blocks unless the owner explicitly opts into another action.
