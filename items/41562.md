## Summary

Current `main` already contains the routing guards for this report, and the latest shipped release includes them as well. The code now preserves bound thread/conversation delivery targets for announce/session-fallback paths and seeds missing `deliveryContext.to/threadId` on spawned sessions, which matches the failure mode described in #41562.

## Evidence

## Close Comment

Closing as implemented on current OpenClaw. This Codex review found that the current routing code already covers the failure mode from this report: `src/gateway/server-methods/agent.ts:645` now seeds missing session `deliveryContext.to/threadId` from the request so spawned thread sessions do not keep the incomplete channel-only state described here, and the completion path in `src/agents/subagent-announce-delivery.ts:239` plus `src/agents/subagent-announce.ts:526` resolves a bound thread/conversation destination before fallback routing. The shipped regression coverage in `src/agents/subagent-announce.format.e2e.test.ts:852` and `:910` specifically asserts that Discord bound-session completions stay on the thread destination and do not leak back to the main channel when sibling activity is present. This behavior is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), and the changelog also calls out preserved agent-announce/session-fallback thread routing in `CHANGELOG.md:878`.
