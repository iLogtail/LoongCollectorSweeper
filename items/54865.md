## Summary

Keep open. Current `main` does not contain a fix or guard for LiteLLM tier aliases in `tools.media`. The media-understanding path still treats `provider`/`model` as raw strings, performs exact model lookup for image understanding, and lacks the alias-resolution behavior used by normal model-selection paths.

## Evidence

- **Media config still accepts raw provider/model entries:** `MediaUnderstandingModelConfig` is still a freeform `provider` + `model` pair with no LiteLLM-tier alias restriction, warning, or validation hook.
  - file: [src/config/types.tools.ts:53](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/config/types.tools.ts#L53)
  - command: `nl -ba src/config/types.tools.ts | sed -n '45,95p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Docs still present tools.media as direct provider/model config:** Current media-understanding docs still document `tools.media` model entries as direct `provider`/`model` pairs and do not mention LiteLLM routing aliases being rejected or unsafe here.
  - file: [docs/nodes/media-understanding.md:76](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/docs/nodes/media-understanding.md#L76)
  - command: `nl -ba docs/nodes/media-understanding.md | sed -n '40,110p'`

- **Image-model fallback path does not resolve aliases:** `resolveImageModelFromAgentDefaults()` just splits each configured ref at the first slash and forwards the raw provider/model pieces. There is no alias-index lookup or canonicalization step comparable to normal model selection.
  - file: [src/media-understanding/runner.ts:503](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/media-understanding/runner.ts#L503)

  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Media image runtime still requires an exact registered model:** `resolveImageRuntime()` normalizes the provider/model string and then does `modelRegistry.find(...)`; if that exact model is absent it throws `Unknown model`.   - file: [src/media-understanding/image.ts:107](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/media-understanding/image.ts#L107)

  - file: [src/agents/simple-completion-runtime.ts:77](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/agents/simple-completion-runtime.ts#L77)
