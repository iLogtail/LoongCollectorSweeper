## Summary

Current main already implements renewable inbound debounce with same-sender batching and concatenation, and it also has an opt-in BlueBubbles split-send coalescing path that matches the command-plus-payload case described in the issue.

## Risks / Open Questions

- The BlueBubbles split-send path is opt-in via `channels.bluebubbles.coalesceSameSenderDms`; without that flag, only the normal inbound debounce behavior applies.

## Close Comment

Closing from a Codex review: current `main` already implements this behavior. The shared inbound debouncer renews the timer by clearing and rescheduling it when another same-key message arrives (`src/auto-reply/inbound-debounce.ts:149`), and its tests already cover combining multiple items into one flush (`src/auto-reply/inbound.test.ts:286`). The user-facing docs describe `messages.inbound` as batching rapid same-sender text messages into a single agent turn (`docs/gateway/config-agents.md:1241`, `docs/concepts/messages.md:45`). The channels named in this report already concatenate buffered messages during that flush path, including Telegram (`extensions/telegram/src/bot-handlers.runtime.ts:253`), WhatsApp (`extensions/whatsapp/src/inbound/monitor.ts:212`), and Discord (`extensions/discord/src/monitor/message-handler.ts:162`). For the split-send command-plus-payload case, BlueBubbles now has opt-in same-sender DM coalescing with a widened debounce window and merged output, documented in `docs/channels/bluebubbles.md:398` and covered by regression tests in `extensions/bluebubbles/src/monitor.test.ts:1248`.
