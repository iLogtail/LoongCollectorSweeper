## Summary

Current main still has the reported macOS GUI limitation. The macOS composer opens an NSOpenPanel titled for image attachments and restricts selectable files to UTType.image, and the shared Swift chat view model rejects non-image files even if they arrive by drag/drop. This is not implemented, not a duplicate from the provided related context, and it is plausible core app work because the web Control UI and Gateway already support non-video/document attachments.

## Best Possible Solution

Keep this issue open. The likely fix belongs in the shared Swift chat UI used by macOS: change the paperclip affordance from image-only to file attachments, broaden or remove the NSOpenPanel content-type restriction while filtering unsupported video/unsafe types, replace `addImageAttachment`/`loadAttachments` with a generic attachment path that still generates image previews when possible, and add focused Swift tests or smoke coverage for PDF/TXT/DOCX plus image parity. Align the accepted types and size behavior with `ui/src/ui/chat/attachment-support.ts` and the existing Gateway `chat.send` attachment contract.
