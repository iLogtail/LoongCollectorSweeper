## Summary

Current main does not implement the requested Control UI behavior. The New Session button and local `/new`/`/reset` paths still send a chat command on the currently selected session key, while the backend `sessions.create` path can create a separate dashboard session key but is not used by that UI flow. The server reset path also rotates the session id in-place under the same canonical key, which gives a fresh transcript but not the independent session identity requested here.

## Best Possible Solution

Keep the issue open and implement a deliberate Control UI fresh-session path: decide whether New Session, `/new`, and `/reset` should call `sessions.create` with a generated dashboard/webchat key and then switch the UI to that key, or keep reset-in-place semantics but expose a separate clearly named action for independent session creation. Add focused UI/controller tests that prove the selected session key changes and the new key appears in the session selector.
