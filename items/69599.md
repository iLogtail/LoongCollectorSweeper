## Summary

Keep open. Current main still routes the Control UI New Session button plus local /new and /reset handling through the active session reset path, while sessions.create remains the separate backend path that creates a distinct session key/entry. Related #66533 was split/closed for heartbeat selector pollution; it explicitly leaves the true independent fresh-session semantics to #69599.

## Best Possible Solution

Keep #69599 open for maintainer product judgment and implementation. A likely fix is to add a dedicated Control UI fresh-session path that calls sessions.create, switches the UI to the returned key, updates lastActiveSessionKey, clears/reloads chat state, and refreshes the session selector. Maintainers should also decide whether Control UI /new and /reset should adopt that same distinct-session behavior or whether docs/UI should clearly distinguish in-place reset from creating a new selectable chat.
