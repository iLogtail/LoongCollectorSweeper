## Summary

Keep #69599 open. Current main still routes Control UI New Session, /new, and /reset through chat.send on the active session key, while the backend reset path rotates the session id under that same key. The separate fresh dashboard-session behavior exists in sessions.create, but the Control UI fresh-start paths inspected here do not call it or switch to a newly created independent key.

## Best Possible Solution

Keep the issue open for maintainer/product handling. The likely implementation path is to add a Control UI fresh-session action that calls sessions.create, switches state.sessionKey and lastActiveSessionKey to the returned key, refreshes sessions/history, and clarifies the difference between creating a new visible chat session and resetting the current session. If /new is changed in webchat too, add focused UI and gateway tests because current tests explicitly cover reset-in-place semantics.
