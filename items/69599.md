## Summary

Keep open. Current main still wires the Control UI New Session button and local /new//reset handlers through the active-session reset path, while the backend sessions.create API is the separate path that creates a distinct selectable session key. Related #66533 was closed for heartbeat selector pollution and explicitly left this independent fresh-session behavior to #69599; #47561 is adjacent but tracks routing-mode changes after /new, not Control UI direct-session parity.

## Best Possible Solution

Keep #69599 open for maintainer product judgment and implementation. A likely fix is a dedicated Control UI fresh-session path that calls sessions.create, switches the UI to the returned key, updates lastActiveSessionKey, clears/reloads chat state, and refreshes the session selector. Maintainers should decide whether Control UI /new and /reset adopt that same distinct-session behavior or whether docs/UI clearly distinguish in-place reset from creating a new selectable chat.
