## Summary

Keep #34528 open. Current main still creates Feishu synthetic reaction message IDs with the reported `:reaction:<emoji>:<uuid>` suffix, preserves that value as the inbound `ctx.messageId`, can use it as the reply target, and passes message IDs unchanged into Feishu API path parameters. The related fix attempts in the provided context are closed/unmerged except #39558, which remains an open fix candidate rather than shipped behavior.

## Best Possible Solution

Keep #34528 open until a Feishu plugin fix lands on main. The best path is to review, update, or supersede #39558 with a narrow `stripFeishuReactionSuffix` helper applied at Feishu API boundaries, plus regression tests proving synthetic reaction IDs remain internal while clean base `om_...` IDs are sent to Feishu.
