## Summary

Keep #72278 open. Current main has useful low-level SecretRef batching, but it only dedupes exact SecretRef keys and batches refs that already share the same configured source/provider. The public external-vault examples still model 1Password and Vault as one exec provider command per secret with `id: "value"`, which is the configuration shape the issue says prevents cross-secret backend memoization. Related #17311 is adjacent and open, but it does not fully subsume this generic backend-scoped batching/memoization contract.

## Best Possible Solution

Keep the issue open and solve it in the core SecretRef integration model: define a backend-scoped external secret provider pattern, or extend exec/configure/docs so the default external-vault pattern uses one backend/account provider with SecretRef ids as backend paths. Batch all active refs per backend provider during startup/reload, add pass-scoped in-memory memoization for repeated backend entity lookups, and add regression coverage. Coordinate with open #17311 for first-party provider work and #56217 for crash-loop/rate-limit behavior, but do not close #72278 as already implemented or purely plugin-scoped.
