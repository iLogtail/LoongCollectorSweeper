## Summary

Keep open. Current `main` does not implement MCP elicitation handling in the embedded Pi bundle MCP runtime; it still does a direct MCP `callTool()` with no request-handler or approval bridge. The only elicitation bridge I found is in the separate Codex harness path, whose docs explicitly say other MCP elicitation requests still fail closed.

## Evidence

- **Embedded Pi runtime still does direct MCP tool calls:** The embedded Pi bundle MCP runtime creates an MCP `Client` with empty client options and later calls `session.client.callTool(...)` directly. There is no surrounding elicitation/approval handling in this path.
  - file: [src/agents/pi-bundle-mcp-runtime.ts:176](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/agents/pi-bundle-mcp-runtime.ts#L176)

- **Embedded Pi MCP runtime contract has no elicitation hook:** `SessionMcpRuntime` only exposes `getCatalog()`, `markUsed()`, `callTool(serverName, toolName, input)`, and `dispose()`. There is no callback or bridge surface for server-initiated elicitation requests.
