## Summary

Current main still contains the reported behavior: the Overview WebSocket URL input handler updates `gatewayUrl` and clears `settings.token` whenever the typed URL differs from the previous URL. Connect then reads `settings.token`, so the cleared field still affects connection auth. The linked fix PRs remain open/unmerged, so this issue is still the canonical bug report rather than something safely closable.

## Best Possible Solution

Keep the issue open and land one of the linked focused UI fixes, ideally with a regression test that distinguishes manual token entry in Overview from tokenized/deferred gateway URL import flows. The implementation should update only `gatewayUrl` during ordinary field edits, while preserving the existing cross-gateway token safety behavior where it is intentional.
