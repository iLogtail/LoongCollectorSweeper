## Summary

Keep open. Current main still has the reported startup secret-provider behavior: startup activation calls secret resolution once and fails fast on error, with docs still documenting fail-fast startup semantics. The PR is stale and should be rebased/split, but it is not implemented, obsolete, or safely closeable.

## Best Possible Solution

Keep this PR open as useful pending work, but request changes: port the startup-only retry/backoff into createRuntimeSecretsActivator in src/gateway/server-startup-config.ts, add focused regression tests in src/gateway/server-startup-config.secrets.test.ts proving startup retries while reload/restart-check semantics stay unchanged, update docs/gateway/secrets.md if startup no longer strictly fails fast, and split the unrelated WhatsApp append/pairing changes into their own reviewed PR or canonical follow-up.
