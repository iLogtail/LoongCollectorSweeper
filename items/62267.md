## Summary

Keep open. Current main has useful pieces around Ollama context sizing, overflow prechecks, manual prompt-pressure controls, and configurable idle timeouts, but I did not find code, docs, tests, history, or related-item evidence that #62267's specific long-context Ollama prompt-ingestion latency and fallback behavior is fixed or fully tracked by a canonical replacement.

## Best Possible Solution

Keep open for maintainer runtime/performance triage. The best path is to add or document a local-Ollama long-context strategy: clear diagnostics for prompt assembly vs prompt evaluation time, recommended context caps for Qwen 14B-class local models, and fallback behavior that distinguishes slow-but-healthy prompt ingestion from true model failure, while preserving the existing `num_ctx`, overflow precheck, and manual local-model pressure controls.

## Evidence

- **Native Ollama request path still sends assembled context:** The native Ollama stream converts the active context messages plus system prompt into the `/api/chat` request and forwards tools when present. It sets `options.num_ctx` from the model context window, which helps sizing, but it is not a latency-aware prompt-ingestion cap or fallback policy.
  - file: [extensions/ollama/src/stream.ts:620](https://github.com/openclaw/openclaw/blob/8741a86f9303e2e5875093c5a49dfc7a6e41115f/extensions/ollama/src/stream.ts#L620)

That does not address a model that is technically within context but spends 100s in prompt eval.
