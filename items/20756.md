## Summary

Keep #20756 open. Current main still does not auto-select the sole enabled account for `message` sends after channel resolution; it only uses an explicit `accountId`, the tool/default account id, or the first agent binding. WhatsApp's omitted-account path still resolves to the channel default helper, which prefers `default` when present without considering `enabled: false`. The related fix PR #20801 was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep #20756 open. Implement shared outbound account resolution in `runMessageAction` after channel selection: preserve explicit params, tool defaults, and agent bindings; use a configured default only when enabled/configured; auto-select exactly one enabled account when no usable default exists; and return clear zero-or-multiple-enabled-account errors. Add focused regression tests for the shared message action path plus the WhatsApp disabled-`default`/single-enabled-account case, then update the multi-account/channel-routing docs.
