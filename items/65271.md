## Summary

Keep open. Current main still hard-codes Windows exec shell selection to PowerShell, and the docs describe PowerShell-only Windows behavior plus `OPENCLAW_SHELL` as a spawned-process marker rather than a user shell override. The PR is a focused, still-useful attempt to add Windows custom shell support, but it needs maintainer/product review around the env-var contract and the remaining validation concerns before it can land.

## Best Possible Solution

Keep the PR open for maintainer review or revision. A mergeable solution should define the supported shell-selection contract, reconcile or replace `OPENCLAW_SHELL` as a user override, validate Windows configured shell paths and bare names against PATH/PATHEXT, keep child and PTY exec behavior aligned, and update focused tests plus user-facing exec/environment docs.

## Risks / Open Questions

- The proposed `OPENCLAW_SHELL` override conflicts with current docs and call sites that use `OPENCLAW_SHELL` as a spawned-process context marker, so the final contract may need a dedicated config/env name or explicit maintainer approval.
- Windows shell selection changes affect command parsing, quoting, approvals, PTY behavior, and safe fallback behavior; the remaining review comments about validating configured paths and bare shell names should be resolved before landing.
- Related custom-shell issues and PRs remain open, but they are broader config-surface work and do not by themselves supersede this narrower Windows `SHELL` support PR.
