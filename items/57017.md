## Summary

Keep #57017 open. Current main still full-loads cron JSONL logs in the read path before parsing, filtering, sorting, and slicing. The write path prunes after append, but that does not add a hard read cap for a log that grew outside the expected write/prune path or after failed pruning. The related PR #57088 remains open and #57913 was closed unmerged, so the issue is still actionable rather than implemented or superseded.

## Best Possible Solution

Keep the issue open until main ships a bounded read strategy. The conservative implementation is a non-mutating size-capped tail read for `readCronRunLogEntriesPage` and equivalent protection for `readCronRunLogEntriesPageAll`, preserving newest-entry pagination and adding regression tests for oversized logs, query/status filtering, all-scope reads, and concurrent append behavior. If maintainers choose pruning instead, queue that prune through `writesByPath` so it cannot overwrite fresh appends.

## Risks / Open Questions

- A large or externally grown cron run log can still be read fully into memory by the per-job and all-jobs run-history paths.
- A read-time prune fix needs serialization through `writesByPath` or a non-mutating tail-read design to avoid the race already called out in the issue discussion.
