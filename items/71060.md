## Summary

Keep open. Current `main` still gives `stopLaunchAgent` only a 1 second confirmation window before falling back to `launchctl bootout`, and that fallback is intentionally documented/tested as leaving the LaunchAgent unloaded. The report’s core claim still matches current code. `main` does contain a separate mitigation for restart requests issued from inside the managed LaunchAgent process tree, but that only addresses the self-restart transport-loss part of the report, not the unload-on-slow-stop behavior.

## Risks / Open Questions

- Closing this would incorrectly discard a still-present macOS lifecycle bug in `stopLaunchAgent`/`restartLaunchAgent` when shutdown confirmation exceeds the hard-coded 1 second probe window.
- The issue bundles two concerns: self-restart transport loss is partly mitigated on `main`, but the LaunchAgent unload fallback remains; treating the partial mitigation as a full fix would be misleading.
