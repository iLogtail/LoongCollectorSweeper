## Summary

Keep #71060 open. Current main has hardened the macOS restart path, but the stop path still matches the reported failure mode: it waits only 10 × 100ms before treating the LaunchAgent as still running, then falls back to `launchctl bootout` with warnings that explicitly say the service is left unloaded. Existing tests also assert this degraded bootout behavior.

## Best Possible Solution

Keep the issue open and narrow the maintainer work to macOS LaunchAgent stop semantics. The likely fix belongs in `src/daemon/launchd.ts`: replace the hard-coded 1s stop confirmation window with a drain-aware timeout, avoid booting out a loaded LaunchAgent merely because graceful shutdown exceeds that window, and if any bootout fallback remains necessary, re-bootstrap or clearly return a recoverable state. Add regression tests around slow-but-eventually-stopped services and any re-bootstrap fallback, then update the Gateway lifecycle docs if stop semantics change.
