## Summary

Keep #10519 open. Current main still renders Telegram fenced code blocks as static `<pre><code>` HTML, the shared Markdown IR drops the fence language, and the existing Telegram test still expects a `js` fence to render without `class="language-js"`. The related WebChat copy-button work is a separate UI renderer path and does not fix Telegram Bot API HTML output.

## Best Possible Solution

Keep the issue open. Implement fenced-code language preservation for Telegram by carrying optional code-block language metadata through the shared Markdown IR/render path, or by adding a narrowly scoped Telegram fence-aware path if maintainers prefer not to widen the IR yet. Update Telegram format, wrapping/chunking, and outbound tests to expect `class="language-js"` while preserving current escaping and chunk-limit behavior.
