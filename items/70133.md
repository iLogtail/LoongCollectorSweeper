## Summary

Keep #70133 open. Current main still has the reported Feishu same-chat blocking shape: ordinary P2P messages are keyed to one per-chat sequential lane, that lane is released only when the full Feishu message task settles, and the task awaits the agent reply dispatch plus dispatcher idle. I did not find a Feishu queue timeout/release guard, a shipped changelog entry for this bug, or a canonical superseding issue that already tracks the same remaining timeout problem.

## Best Possible Solution

Keep the issue open and fix it in the Feishu plugin, likely by adding a bounded per-chat task timeout or abort-aware release strategy around the Feishu sequential task, with regression coverage proving a stuck same-chat dispatch cannot indefinitely block later P2P messages. Coordinate with the existing #54409/#43512 queue work so command-lane behavior, collect batching, and stale-task recovery do not conflict.
