## Summary

Keep #70133 open. Current main still has a Feishu per-chat promise chain with no per-task timeout or independent release guard; normal P2P messages share the same chat lane and the lane is held until the full agent reply dispatch settles. The related #58953 is a different post-restart replies=0 report, while the related #43512/#54409 reviews reinforce that Feishu queue behavior remains a live area rather than a shipped fix for this timeout issue.

## Best Possible Solution

Keep the issue open and fix it in the Feishu plugin path. The likely direction is a bounded per-chat dispatch watchdog around `extensions/feishu/src/sequential-queue.ts` or `extensions/feishu/src/monitor.message-handler.ts`, with clear behavior for timeout, cancellation/late completion, dedupe release, and user-visible error logging. Add focused Feishu tests proving that a stuck first P2P dispatch does not keep a later same-chat P2P message queued indefinitely, while preserving `/stop` and `/btw` lane behavior.

## Risks / Open Questions

- A naive timeout fix could orphan an in-flight agent run or allow late Feishu replies to interleave with later messages, so the implementation should define cancellation, late-reply, and logging semantics carefully.
- Any fix should account for the existing `/stop` control lane, `/btw` lane, inbound debounce, and core session queue modes so it does not regress the related queue behavior tracked in #43512 and #54409.
