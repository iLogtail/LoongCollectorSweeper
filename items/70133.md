## Summary

Keep #70133 open. Current main still serializes Feishu inbound p2p messages through a per-key promise chain and awaits the full `handleFeishuMessage`/agent dispatch before the next same-chat message can start. I found no timeout or release guard on that queue, and the existing Feishu tests assert strict same-key serialization rather than starvation recovery. The linked #54409 is related but explicitly tracks a different `messages.queue.mode = "collect"` batching failure, so it is not a clean duplicate for this timeout/release-guard bug.

## Best Possible Solution

Keep this issue open and fix it in the Feishu plugin path. The best implementation direction is a bounded per-chat dispatch strategy with explicit timeout/release behavior, plus focused tests under `extensions/feishu/src/` that prove a stuck first p2p dispatch cannot indefinitely block later same-chat messages. The design should account for #54409, but #70133 should retain its own acceptance criteria around starvation recovery and stuck-task handling.

## Evidence

- **Current queue implementation has no timeout/release guard:** `createSequentialQueue` stores `previous.then(task, task)` as the next promise for a key and deletes the key only when that same promise settles. There is no timeout, abort signal, Promise.race, or watchdog that would release later same-key tasks if the active task hangs.
