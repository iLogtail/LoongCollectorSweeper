## Summary

Keep open. Current main still derives the managed Docker sandbox workspace from the gateway process state directory and passes that same path directly into Docker `-v` bind sources. In the documented Docker gateway setup, the process runs with `HOME=/home/node` and `/home/node/.openclaw` is only a container mount target, while the sandbox setup script enables sandbox mode without setting a host-visible `sandbox.workspaceRoot` or adding identical host-path parity for the sandbox root. The docs now warn about DooD host-path constraints, but the reported default Docker Desktop/macOS failure path remains actionable in setup/runtime code.

## Best Possible Solution

Keep the issue open and fix the Docker gateway sandbox path contract. The conservative product path is to make `scripts/docker/setup.sh` configure a host-visible `agents.defaults.sandbox.workspaceRoot` such as `${OPENCLAW_CONFIG_DIR}/sandboxes` and ensure the gateway container has identical host/container path parity for that root when sandboxing is enabled, or add a runtime mapping layer for managed sandbox binds. Add focused tests around Docker setup/config and `docker create` bind sources, then update the Docker/sandbox docs with the supported macOS Docker Desktop shape.
