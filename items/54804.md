## Summary

Keep open. Current `main` at `a6aa151653ab478e9aa23e95c6446c41adc4fce1` still enforces `hooks.token` only in gateway startup/runtime resolution, not during `config.patch`, and `config.schema.lookup` still does not expose dependency metadata for `hooks.enabled`.

## Evidence

- **Runtime-only hooks validation remains:** `resolveHooksConfig()` still throws `"hooks.enabled requires hooks.token"` only after checking `cfg.hooks?.enabled === true`, so the dependency is still enforced in gateway startup/runtime code.
  - file: [src/gateway/hooks.ts:49](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/gateway/hooks.ts#L49)
  - command: `nl -ba src/gateway/hooks.ts | sed -n '1,220p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **config.patch still relies on generic config validation only:** `config.patch` merges the patch, restores redactions, then calls `validateConfigObjectWithPlugins(...)`. There is no follow-up call to `resolveHooksConfig()` or any hooks-specific preflight before writing/restart scheduling.
  - file: [src/gateway/server-methods/config.ts:516](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/gateway/server-methods/config.ts#L516)
