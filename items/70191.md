## Summary

Keep #70191 open. Current main does not implement a native `/health/deep` endpoint, and the existing HTTP/WS health surfaces do not expose the requested model contract fields (`modelId`, `provider`, `fallbacks`, recent errors, or last routing decision). A plugin may be a plausible direction because current plugin APIs support HTTP routes and model-call telemetry hooks, but the issue is a recent design proposal asking for maintainer choices on a core health/readiness surface and auth semantics, so it should not be auto-closed as implemented or ClawHub-only.

## Best Possible Solution

Keep the issue open for maintainer product/API direction. If core should own it, implement a narrow auth-gated model-contract probe in the gateway HTTP layer or as an extension of readiness details, backed by shared model-resolution helpers, explicit tests in the gateway probe suite, and docs at the public health page. If maintainers prefer ClawHub, respond with that direction and identify any missing plugin SDK helper needed to match the desired auth/readiness semantics.

## Risks / Open Questions

- A native endpoint would expose model IDs/provider routing state, so auth, localhost-bypass behavior, redaction, and timing behavior need explicit security review.
- If maintainers steer this to a plugin, the exact `/ready`-style localhost-or-bearer auth behavior may need either a documented different policy or a small SDK/core helper rather than an ad hoc plugin reimplementation.
- The proposal also asks for last routing decision and rolling error counts; those should be wired through existing model-call telemetry hooks or a deliberately scoped core observability helper to avoid duplicating dispatch state.
