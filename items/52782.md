## Summary

Keep open. Current `main` still has the transcript-path mismatch in the reset helper, and the PR as proposed is not a high-confidence complete fix after the refactor: reset cleanup still omits the topic/thread suffix fallback, the reset entrypoint now drops numeric `MessageThreadId` values, and existing session/routing code explicitly supports recovering thread ids from the session key when `MessageThreadId` is absent.

## Evidence

- **Current main still omits topicId during reset cleanup:** `resetReplyRunSession` creates the next transcript path with `resolveSessionTranscriptPath(nextSessionId, agentId, params.messageThreadId)` but, during cleanup, still adds a fallback candidate with `resolveSessionTranscriptPath(prevSessionId, agentId)` and no topic/thread id. That preserves the core mismatch reported in the PR whenever the persisted `sessionFile` is missing or stale.

That means numeric topic ids can still lose their suffix during reset-session transcript handling on current `main`.
