## Summary

Keep open. Current main does not implement PR #70374’s central behavior: `message.send`/`message.poll` still enter `executeSendAction`/`executePollAction`, and `tryHandleWithPluginAction` locally dispatches plugin actions without consulting `resolveExecutionMode` or calling gateway `message.action`. Telegram also does not opt `send` into gateway execution on main. The PR is a focused, still-relevant change, but the posted review/security comments mean it needs maintainer review and probably revisions rather than automated closure.

## Best Possible Solution

Keep PR #70374 open. The best path is to implement plugin-owned `send` gateway routing on the shared outbound send path and have Telegram opt `send` into it, but revise the PR so gateway URL handling, idempotency, and fallback semantics match the hardened existing core gateway send/poll behavior. If fallback remains, limit it to explicit gateway-unavailable transport failures and make non-transient gateway errors visible and non-bypassable.

## Risks / Open Questions

- The PR’s described broad gateway-error fallback can duplicate sends or bypass gateway-side validation if the gateway processed the request but the client timed out, so fallback should be narrowed or made fail-closed.
- Any new `message.action` gateway helper should reuse the existing gateway URL hardening pattern from `src/infra/outbound/message.ts:179` so backend callers cannot forward user-controlled gateway URLs.
- Telegram `message.send` is a user-visible channel path; landing should include focused outbound-send-service and Telegram adapter tests plus maintainer review of the security-bot findings.
