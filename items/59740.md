## Summary

Keep open. Current main has a CI boundary runner that groups existing guard commands, but it has not consolidated the boundary checks into a unified rule framework. The package scripts still invoke many individual `check-*` files, no `scripts/architecture-checks` config/runner exists, and the runner tests cover command aggregation rather than migrated architecture rules.

## Best Possible Solution

Keep the issue open as a low-priority tooling refactor. The best path is to preserve the current CI boundary shard as a transition runner, introduce a real shared architecture-check framework or ESLint-plugin-style rule package, migrate a small tranche of existing guards with parity tests, document how to add new rules, then deprecate individual scripts only after behavior is proven equivalent.

## Risks / Open Questions

- A generic checker DSL could lose specialized AST/type-aware behavior in existing guards unless migrated incrementally with old/new checks running in parallel.
- Current main already reduces some CI overhead by grouping boundary guards, so the remaining work should be scoped to rule implementation reuse, documentation, and deprecation rather than CI grouping alone.
