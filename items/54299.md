## Summary

Keep open. Current main still has the reported Gemini CLI OAuth behavior: token exchange omits `client_secret` when no secret is resolved, and the OAuth redirect URI/listener are fixed to `http://localhost:8085/oauth2callback`. There is a partial local fallback to manual entry after callback-server errors, but no configurable or dynamic redirect port, and the provided related items track different OAuth problems.

## Best Possible Solution

Keep #54299 open. The best path is to update the Google/Gemini CLI OAuth flow, and consider the similar Google Meet OAuth surface, so confidential-client setups fail or warn clearly when a secret is expected while intentional public-client setups are documented. Add a configurable redirect URI/port or another provider-compatible callback strategy, with focused tests for missing-secret handling and occupied-port behavior.

## Risks / Open Questions

- Google OAuth public-client semantics may make a missing `client_secret` intentional for some client types; the fix should distinguish intentional public-client mode from accidental confidential-client misconfiguration instead of blindly requiring a secret everywhere.
- A dynamic redirect port only works if the generated redirect URI is also authorized in Google Cloud, so a configurable redirect URI/port may be safer than `listen(0)` for providers that require exact callback registration.
