## Summary

Issue #54676 asks to add an existing third-party MAX messenger integration as a built-in OpenClaw channel. Current main explicitly routes optional channel/plugin additions to the community plugin path (ClawHub/npm) rather than bundling them in core, so this belongs in ClawHub scope instead of openclaw/openclaw core.

## Close Comment

Closing as `clawhub` based on a Codex review of current `main`. OpenClaw's current scope is to keep optional capabilities out of core when they can ship as plugins: `VISION.md:54` says optional capability should usually ship as plugins, authors should host and maintain them in their own repo, and the bar for adding optional plugins to core is intentionally high. The plugin docs make this explicit for this exact case: `docs/plugins/building-plugins.md:16` says new plugins do not need to be added to this repository and should be published to ClawHub or npm, and `docs/plugins/community.md:9` treats third-party channel integrations as community plugins with ClawHub as the canonical discovery path. There is also no bundled `extensions/max` on current `main` at `a6aa151653ab478e9aa23e95c6446c41adc4fce1`. The MAX adapter sounds useful, but it fits the ClawHub/community-plugin route rather than OpenClaw core. If it is still being maintained, the right path is to publish/list it for `openclaw plugins install ...` discovery rather than track it here as a core-bundling request.
