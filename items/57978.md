## Summary

Keep open. Current `main` does not contain this Feishu WebSocket supervisor/backoff/stall-detection work, and the present tests only cover abort/cleanup paths rather than retry-exhaustion recovery.

## Evidence

- **Current Feishu monitor is still fire-and-forget:** `monitorWebSocket()` creates one WS client, stores it, starts it with `void wsClient.start({ eventDispatcher })`, and then only waits for abort/cleanup. There is no supervisor loop, reconnect backoff, stall polling, or `lastConnectTime` handling in current `main`.
