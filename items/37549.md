## Summary

Keep open. Current `main` still caches subsystem file loggers indefinitely, and the proposed generation-based refresh in this PR does not appear sufficient to handle the midnight rollover path reliably.

## Evidence

- **Current main still holds a stale child file logger forever:** `createSubsystemLogger()` caches `fileLogger` and reuses it for file writes with no root-logger invalidation path in current `main`. That matches the original bug report rather than an already-shipped fix.
  - file: [src/logging/subsystem.ts:325](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/logging/subsystem.ts#L325)
  - command: `nl -ba src/logging/subsystem.ts | sed -n '320,445p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Midnight rollover depends on `getLogger()` rebuilding from a newly resolved dated filename:** The root logger only rebuilds inside `getLogger()` when `resolveSettings()` produces a different dated file path. `isFileLogLevelEnabled()` does not force that rebuild when cached settings already exist.
  - file: [src/logging/logger.ts:354](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/logging/logger.ts#L354)

- **Why this PR is still not obviously merge-ready:** The review discussion already identified that a generation check tied to `getLogger()` is not enough if long-lived subsystem loggers keep writing without first forcing a root refresh after midnight.
