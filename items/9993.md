## Summary

Keep #9993 open. Current main has stronger built-in config validation, SecretRef preflight, base-hash clobber protection, last-known-good/rejected-payload safeguards, and an agent-facing gateway config mutation guard, but it still does not expose the requested synchronous config-write hook/plugin API that receives current and pending config and can veto direct config.set/config.patch/config.apply writes before persistence.

## Best Possible Solution

Keep #9993 open as a plugin API/product-direction issue. The best path is a documented synchronous config-write interception point invoked by config.set, config.patch, and config.apply after the pending config is computed and built-in validation/preflight has passed but before writeConfigFile. It should provide current config, pending config, action, raw patch or full payload, baseHash, actor/client metadata, changed paths, and a typed reject result, with focused tests proving a plugin can reject direct RPC writes without changing the active config.

## Risks / Open Questions

- Built-in validation and the agent-facing mutation guard reduce much of the original crash-loop class, so maintainers may want to narrow the issue title/body to the remaining enforced config-write plugin API.
- A synchronous config-write veto point would touch core persistence, restart scheduling, auth invalidation, SecretRef preflight, and plugin hook failure-policy semantics, so the API shape needs explicit maintainer product/security review.
