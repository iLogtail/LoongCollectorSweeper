## Summary

Keep open. Current main has stronger built-in config validation, base-hash checks, SecretRef preflight, and an agent-facing gateway tool guard, but #9993 asks for a synchronous plugin/API-level veto before config writes. That surface is still missing: direct config.set, config.patch, and config.apply RPC handlers write after built-in validation without invoking a config pre-apply hook, and the public plugin hook catalog has no equivalent hook that receives current and pending config or covers direct RPC writers.

## Best Possible Solution

Keep #9993 open as a plugin API/product-direction issue. The best implementation is a documented synchronous config-write interception point invoked by config.set, config.patch, and config.apply after pending config is computed and built-in validation/preflight succeeds, but before writeConfigFile. It should expose current config, pending config, action, raw patch/full payload, baseHash, actor/client metadata, changed paths, and a typed reject result, with tests proving a plugin can reject direct RPC writes without changing disk or active runtime config.

## Risks / Open Questions

- A synchronous config-write veto point would need explicit maintainer/security design because it touches persistence ordering, SecretRef preflight ordering, runtime reload/restart behavior, shared-auth invalidation, hook failure policy, and recursion/deadlock handling for plugins that also write config.
- The existing agent gateway-tool guard reduces risk for agent-originated config edits, but direct RPC clients, Control UI saves, and plugin/runtime writes still lack the requested plugin-level interception surface.
