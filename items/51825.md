## Summary

Current main still does not show a TUI activity indicator at the start of a system-injected run. TUI-dispatched messages set local status immediately, and external chat deltas are adopted once assistant text starts streaming, but a server lifecycle start for an unknown run is ignored before any chat delta has established `activeChatRunId`. The gateway already has session lifecycle signals (`sessions.changed` / lifecycle `agent` events), but the TUI does not subscribe to or consume the session-scoped start signal for this purpose.

## Best Possible Solution

Keep the issue open. The likely fix belongs in the TUI event path: adopt current-session lifecycle starts from either `agent` events carrying `sessionKey` or the gateway `sessions.changed` start event, set `activeChatRunId` and `running` immediately, then clear on matching lifecycle end/error or chat final/error. Add focused TUI handler tests for an injected current-session lifecycle start with no prior local send or chat delta, plus non-current-session and concurrent-run guards.
