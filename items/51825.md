## Summary

Keep open. Current main still does not show a TUI activity indicator at the start of a system-injected run. The gateway emits lifecycle-start information, but the TUI only accepts lifecycle events for a run it already knows about, and it learns unknown run IDs from chat events/deltas rather than from lifecycle start. That means an externally injected bridge/webhook/cron run can still look idle until assistant chat output or finalization arrives.

## Evidence

- **Issue context:** #51825 reports that bridge/webhook/cron-injected work has no spinner/running/elapsed indicator until a response appears; the only discussion comment identifies the likely gap as TUI listening to user/send-derived state but not treating server-side run-start events as activity.
- **TUI ignores lifecycle events for unknown runs:** `handleAgentEvent` computes `isKnownRun` from the active run, `sessionRuns`, or `finalizedRuns`, and returns before lifecycle handling when the run is unknown. A system-injected lifecycle start for the current session therefore cannot set `running` unless another chat event already registered the run.
  - file: [src/tui/tui-event-handlers.ts:409](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/tui/tui-event-handlers.ts#L409)
  - command: `nl -ba src/tui/tui-event-handlers.ts | sed -n '390,475p'`
  - sha: [8e12c24d1721](https://github.com/openclaw/openclaw/commit/8e12c24d17218dc50023aaf5543df1b7c36520aa)
- **TUI learns new run IDs from chat events, not lifecycle start:** `handleChatEvent` records `noteSessionRun(evt.runId)` and sets `activeChatRunId` when a matching chat event arrives; it sets activity to `streaming` only on `delta`. This covers already-streaming chat output but not the pre-output lifecycle start described in the issue.
  - file: [src/tui/tui-event-handlers.ts:305](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/tui/tui-event-handlers.ts#L305)

- **Tests encode the current limitation:** The TUI tests assert lifecycle events set `running` only when `activeChatRunId` already matches, then separately assert unknown run IDs are captured from chat events. There is no regression test for accepting a lifecycle-start event for the current session before a chat delta.
  - file: [src/tui/tui-event-handlers.test.ts:188](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/tui/tui-event-handlers.test.ts#L188)

- **Gateway emits lifecycle start before chat output path:** The gateway builds an `agentPayload` with `sessionKey`, broadcasts non-tool/non-item agent events, and persists/broadcasts `sessions.changed` for lifecycle `start`; it does not emit a `chat` start state that would let the current TUI register the run before assistant deltas.
