## Summary

Keep open. Current `main` still forks thread/topic sessions from the parent transcript when `ParentSessionKey` is present, and both tests and docs still describe that inherited-parent behavior. The proposed PR is not already implemented on `main` or the latest shipped code in this checkout.

## Evidence

- **Current runtime still branches from parent transcript:** `forkSessionFromParentRuntime()` opens the parent session, gets a leaf id, and calls `createBranchedSession(leafId)`, returning the branched transcript file instead of always creating a clean session.
  - file: [src/auto-reply/reply/session-fork.runtime.ts:62](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/auto-reply/reply/session-fork.runtime.ts#L62)
  - command: `nl -ba src/auto-reply/reply/session-fork.runtime.ts | sed -n '1,220p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Thread/session initialization still performs parent forking:** `initSessionState()` checks `ParentSessionKey` and, unless blocked by the token guard, logs `forking from parent session` and calls `forkSessionFromParent()`. That is the behavior this PR is trying to change.
  - file: [src/auto-reply/reply/session.ts:678](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/auto-reply/reply/session.ts#L678)

  - file: [src/auto-reply/reply/session.test.ts:298](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/auto-reply/reply/session.test.ts#L298)

## Risks / Open Questions

- Slack and Discord expose `thread.inheritParent` with default `false`, so the impact is channel/config dependent; this review only establishes that the parent-forking behavior still exists on `main` where those paths are enabled or where thread/topic sessions set `ParentSessionKey`.
