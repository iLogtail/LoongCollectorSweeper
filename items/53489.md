## Summary

Keep open. Current main no longer defaults Slack/Discord thread sessions to parent inheritance, but the PR is not cleanly obsolete: the core fork path still calls `createBranchedSession()` whenever a channel supplies `ParentSessionKey`, while docs still advertise opt-in parent transcript inheritance. The PR may need revision rather than closure because it changes that documented opt-in behavior and lacks a regression test for the real runtime path.

## Best Possible Solution

Keep the PR open for maintainer review, but request revision before merge: add a focused regression test for `forkSessionFromParentRuntime()` proving the chosen thread/topic behavior, reconcile the documented `thread.inheritParent` semantics, and include affected channel coverage or notes for Slack, Discord, Telegram, and Feishu. If maintainers want to preserve opt-in parent transcript inheritance, replace this with a narrower fix for complete/symmetric inheritance rather than silently making the option no-op.

## Risks / Open Questions

- The PR patch would make the documented `thread.inheritParent` transcript-copy behavior effectively clean-slate unless docs/config semantics are updated with it.
- The PR has no regression test for `forkSessionFromParentRuntime()` itself; current high-level tests mock the fork helper and would not catch dependency-level branch-copy behavior.
- There is a product decision still open: opt-in thread inheritance should either be removed/renamed to clean lineage-only sessions, or fixed to provide complete and symmetric inherited context.
