## Summary

Keep open. Current main does not implement this PR's `fallbackToRestart` contract: external config reload in `hot` mode still logs and returns without queuing a restart for restart-required changes. Existing `hybrid` mode and direct `config.set`/`config.patch` handling cover adjacent cases, but that is not the same as the PR's requested hot-mode fallback behavior.

## Best Possible Solution

Keep the PR open for maintainer judgment. If the intended product contract is still `use hybrid when restart fallback is desired`, close it manually with that rationale. If the new opt-out flag is desired, update the PR to fully implement `gateway.reload.fallbackToRestart` across runtime settings, strict schema, generated config metadata/help/docs, and regression tests for external file edits plus direct config writes.

## Risks / Open Questions

- Changing `hot` to fall back by default would alter semantics that current docs present as intentional: `hot` means apply only hot-safe changes, while `hybrid` handles restart fallback.
- If maintainers want this new flag, the PR needs full public config follow-through beyond the current main surface: schema metadata/help/labels/generated config reference, docs, and tests for both external watcher reloads and control-plane config writes.
- Because current main already documents `hybrid` as the default fallback mode, the remaining question is product semantics, not a high-confidence auto-close case.
