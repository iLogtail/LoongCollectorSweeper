## Summary

Keep open. Current main has substantial Telegram media/proxy/fake-IP fixes, and related reports were correctly closed as implemented, but this issue specifically reports `getaddrinfo ENOTFOUND api.telegram.org` during Telegram media fetches. The current media fallback classifier does not include `ENOTFOUND`/`EAI_AGAIN`, so I cannot say current main fully implements the reported DNS-failure case.

## Best Possible Solution

Keep this open for a narrow maintainer decision: either add a Telegram media regression test for `MediaFetchError` caused by `ENOTFOUND`/`EAI_AGAIN` on `https://api.telegram.org/file/bot...`, then wire those DNS codes into the media transport fallback if that is the desired behavior; or explicitly document/decide that broken host DNS/VPN resolver configuration is external support and close manually with that product-boundary explanation.
