## Summary

Keep open. Current main covers part of #54224 with Slack channel/thread reads that prefer the configured user token, but the requested Slack message search action is still absent, the CLI/docs keep `message search` Discord-only, and the related Slack search PR #58143 was closed unmerged rather than becoming a shipped canonical fix.

## Best Possible Solution

Keep #54224 open and implement the remaining Slack read-only retrieval surface in the bundled Slack plugin: add a `search`/`search-messages` action backed by Slack `search.messages`, advertise it only when a user token/search-read path is available, make the CLI search command channel-agnostic where needed, and add Slack tests/docs. Preserve the existing channel/thread `read` behavior and `userTokenReadOnly` write protection; consider splitting Slack message action gates if maintainers want read and write actions independently disableable.

## Risks / Open Questions

- The issue should probably be narrowed before implementation: history and thread reads already exist, while Slack search and possibly user-token-only action discovery remain open work.
- `listSlackMessageActions` currently filters discoverable Slack accounts by `botTokenSource !== "none"`, so a purely user-token read-only configuration may still fail to advertise read/search actions even though runtime reads can use a user token.
