## Summary

Current main still has a plausible provider-default overwrite path. xAI onboarding uses `applyXaiConfig`, which still applies `applyAgentDefaultModelPrimary` through the default-model preset helper. The newer configure preservation flag exists, but provider API-key auth first merges the provider's `applyConfig(ctx.config)` output as a config patch, so providers whose `applyConfig` already rewrites `agents.defaults.model.primary` can overwrite the original primary before the preservation logic sees it. Classic onboard also calls `applyAuthChoice` with `setDefaultModel: true` and no `preserveExistingDefaultModel` flag. The issue should stay open for a real fix and regression coverage.

## Best Possible Solution

Keep the issue open. The best fix is to separate provider setup patches from default-model mutation: provider auth methods should apply provider/catalog/auth-profile config without embedding a primary-model change, then the caller should set, preserve, or prompt for the default model using one consistent policy. Add regression coverage with a bundled-style provider whose `applyConfig` currently writes `agents.defaults.model.primary`, plus Bedrock-or-other existing-primary input for both `configure` and `onboard`.
