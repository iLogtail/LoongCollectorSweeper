## Summary

Keep #72111 open. Current main still does not support or clearly reject the reported mixed `command` + `url` + `transport: "streamable-http"` MCP configuration; the resolver selects stdio as soon as `command` is present, so the reported spawned-process timeout remains actionable rather than implemented, duplicate, or stale.

## Best Possible Solution

Keep #72111 open. Maintainers should either implement an explicit managed `streamable-http` child-process mode with readiness probing, HTTP client connection, lifecycle cleanup, and regression coverage, or reject mixed `command` + `url`/HTTP transport configs early with a clear diagnostic and docs/schema/test alignment. The current silent stdio precedence should not remain the user-visible timeout path.
