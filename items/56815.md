## Summary

Keep open. Current main still does not implement the central #56815 request: `fetch failed` transport errors are not classified as retryable memory embedding failures, async memory sync logs still flatten errors with `String(err)`, and atomic full reindex still deletes the temp database on any failure. The related PR #56879 describes the right transport-retry/logging direction but is closed unmerged in the provided context, and #65782/#65783 track a different surrogate-pair indexing failure rather than this large remote reindex/progress-loss issue.

## Best Possible Solution

Keep #56815 open and fix it in the memory host/core path: add bounded retry classification for transient fetch/undici transport failures by inspecting nested `cause` and errno/code fields, wrap remote embedding transport failures with provider/error-prefix context while preserving cause, switch async memory sync warning paths to a cause/stack-aware formatter, and add regression tests for `TypeError("fetch failed")` with `ECONNRESET`/timeout/DNS causes. Separately decide a progress-preservation design for safe reindex, such as durable embedding-cache commits outside the temp DB or resumable/checkpointed reindex, so one late batch failure does not waste an hour of successful embeddings.
