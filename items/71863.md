## Summary

Keep PR #71863 open. Current main still has the Signal daemon shutdown race the PR is trying to address: the daemon handle stop API is synchronous, only sends SIGTERM, and the Signal monitor does not await daemon process exit during abort/finally cleanup. Gateway reload/stop paths await the channel task before restarting channels, so this PR’s async daemon stop and regression tests remain directly relevant to the linked open bug #22676 rather than obsolete cleanup.

## Best Possible Solution

Keep PR #71863 open for normal maintainer review/CI. The best path is to land a Signal-owned fix that changes `SignalDaemonHandle.stop` to an idempotent async shutdown, waits for child process exit, escalates to SIGKILL after a bounded timeout, awaits cleanup from `monitorSignalProvider`, and keeps the new daemon/monitor regression coverage. No core/plugin API change appears necessary for this specific bug.

## Evidence

- **Current daemon stop is still fire-and-forget:** `SignalDaemonHandle.stop` returns `void`; `spawnSignalDaemon(...).stop()` only calls `child.kill("SIGTERM")` when the child is not killed/exited and then returns. There is no wait for the child `exit`/`close` promise and no SIGKILL fallback on current main.
