## Summary

Current `main` still shows a plausible surface-specific gap between the documented/configured `tools.fs.workspaceOnly` behavior and how direct-session `read` is assembled and guarded. I did not find strong evidence that issue #70795 is already fixed on `main`, and the existing test coverage does not fully close the reporter’s direct-session/tool-host mismatch claim.

## Evidence

- **Config/docs still define `tools.fs.workspaceOnly` as the switch that restricts FS tools to the workspace:** Schema help says `tools.fs.workspaceOnly` "Restrict filesystem tools (read/write/edit/apply_patch) to the workspace directory (default: false)." The gateway security docs repeat that `tools.fs.workspaceOnly: true` is the optional hardening knob that restricts `read`/`write`/`edit`/`apply_patch` to the workspace.
