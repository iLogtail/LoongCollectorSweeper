## Summary

Keep open. Current main still has no unified read-path policy for approved non-workspace roots: filesystem policy is only a `workspaceOnly` boolean, direct-session read is assembled from a single workspace or sandbox root, and the existing regression test covers only the unwrapped `workspaceOnly=false` helper path rather than the reported direct-session outer tool-host mismatch. The provided related items cover other read-path symptoms and do not supersede this report.

## Best Possible Solution

Keep this issue open. The best maintainer path is to trace the direct-session `read` execution layer, define one effective filesystem read policy that can represent workspace-only plus approved readonly roots, thread that policy through core read/sandbox/tool-host wrappers, and add a regression test that exercises a direct-session `read` against an approved non-workspace root while preserving write/edit/apply_patch hardening.

## Evidence

- **Issue discussion has concrete repro and maintainer-action request:** The issue body reports OpenClaw 2026.4.22 on macOS rejecting `read` for `/opt/homebrew/.../SKILL.md` and `~/.openclaw/openclaw.json` with `Path escapes sandbox root (~/.openclaw/workspace)`, while host `exec` can read the same files. The only comment says current source still looks workspace-root-centric and asks maintainers to trace direct-session `fsPolicy` / tool assembly.
