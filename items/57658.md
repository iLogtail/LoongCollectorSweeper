## Summary

Current main partially addresses the motivation for this issue by preserving down-scoped operator device tokens and surfacing reconnect scope/role upgrades clearly, but it still does not implement custom gateway roles or config-defined role/scope bundles. The gateway role model remains fixed to `operator` and `node`, and there is no `gateway.roles` config or `openclaw roles create` surface on main.

## Evidence

- **Gateway role set is still fixed:** `src/gateway/role-policy.ts` defines `GATEWAY_ROLES = ["operator", "node"]` and `parseGatewayRole()` only accepts those two values, so custom runtime roles are not supported on current main.
  - file: [src/gateway/role-policy.ts:3](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/gateway/role-policy.ts#L3)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Connect helpers still type roles as operator/node only:** Gateway auth helpers still model device connect roles as `"operator" | "node"`, reinforcing that current connection/auth flows are built around the fixed built-in role set rather than config-defined custom roles.
  - file: [src/gateway/server.auth.shared.ts:167](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/gateway/server.auth.shared.ts#L167)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **No config or CLI surface for custom roles found:** Repo search found no `gateway.roles`, `custom roles`, or `roles create` implementation in `docs`, `src/config`, `src/gateway`, or `src/commands`, so the requested config/CLI feature does not exist on main.
  - command: `rg -n "gateway\.roles|roles create|custom roles|configurable scopes" docs src/config src/gateway src/commands`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Scoped operator access is already partially supported:** Gateway method-scope tests show `operator.read`, `operator.write`, `operator.pairing`, and `operator.approvals` are distinct capabilities, so a narrower operator scope set already exists even without custom named roles.
  - file: [src/gateway/method-scopes.test.ts:88](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/gateway/method-scopes.test.ts#L88)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Reconnect overwrite concern was addressed separately:** `docs/cli/devices.md` documents that rotated device tokens reuse cached approved scopes on later reconnects, and `src/infra/device-pairing.test.ts` verifies down-scoping from admin to `operator.read` persists across later rotations/reconnect-style reuse.
  - file: [docs/cli/devices.md:89](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/docs/cli/devices.md#L89)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Current tests enforce persistent down-scoped device tokens:** The device-pairing tests assert that after rotating an operator token down to `operator.read`, subsequent reuse keeps `tokens.operator.scopes` at `operator.read` while preserving the approved baseline, which directly contradicts the specific 'gets overwritten on reconnection' complaint but does not add custom roles.
  - file: [src/infra/device-pairing.test.ts:567](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/infra/device-pairing.test.ts#L567)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)

## Risks / Open Questions

- The issue body combines two separate asks: persistent limited-scope reconnects are now covered, but custom named roles/config-defined role bundles are still missing.
- Closing this as implemented would discard the remaining feature request around true custom roles, pairing assignment for those roles, and a corresponding config/CLI surface.
