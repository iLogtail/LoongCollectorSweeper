## Summary

Keep open. The linked discussion correctly points at cron delivery routing, but current main still defaults Control UI isolated cron jobs to announce delivery, backend-created isolated agent-turn cron jobs also default to announce, and unresolved announce targets still produce delivery-target errors when no channel can be resolved. The related PRs in the provided context were closed/unmerged, and I did not find a main-branch implementation that makes no-channel local cron runs succeed by default while allowing the prompt to save files.

## Best Possible Solution

Keep the issue open and fix the cron product path directly: for local/webchat-only isolated cron jobs that are intended to save artifacts or run internally, either default the Web UI to delivery.mode="none" when no deliverable channel exists, or make requested announce delivery downgrade non-fatally in no-channel setups without masking real explicit delivery failures. Add regression coverage for a Control UI-created isolated cron job in a no-channel config and document the behavior in the public cron docs.
