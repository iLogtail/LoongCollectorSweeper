## Summary

Current main already covers the reported Windows node-exec behavior: remote node exec no longer inherits the gateway cwd when no workdir was requested, `approvals get --node` is explicitly modeled as the merge of node-host approvals plus gateway `tools.exec` policy, wildcard-agent approvals are merged in code, and Windows `cmd.exe /c` wrapper runs are intentionally approval-gated in allowlist mode with explicit guidance.

## Close Comment

Closing from a Codex review because the 2026.3.7 report no longer matches current `main`.

Concrete evidence in the current tree:
- `docs/cli/approvals.md:68` now explains that `openclaw approvals get --node` shows the effective policy as the combination of the node-host approvals file and gateway `tools.exec` policy.
- `src/infra/exec-approvals.ts:750` merges wildcard `agents["*"]` allowlist entries with agent-specific entries on current main.
- `src/node-host/exec-policy.ts:31` and `docs/nodes/index.md:344` make the Windows rule explicit: `cmd.exe /c` shell-wrapper runs are still approval-gated in allowlist mode, so allowlisting the wrapper binary alone is not treated as a no-prompt bypass.
- `CHANGELOG.md:1595` records the remote node exec fix that stopped forwarding the gateway cwd to node hosts when no workdir was requested, specifically to avoid cross-platform `SYSTEM_RUN_DENIED` failures.

Given that current main both implements the relevant node-exec fix path and documents the remaining Windows wrapper behavior as intentional, this is best treated as implemented on main. If this still reproduces on a current release, please open a fresh issue with the exact `openclaw approvals get --node ...` output, `openclaw nodes describe ...`, and the node-host logs from that current build.
