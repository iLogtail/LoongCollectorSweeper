## Summary

Keep #65497 open. The PR is a focused fix attempt for the still-open #62051 CPU/plugin reload regression, and current main does not provide enough evidence to conservatively close it as implemented. Main now routes `resolvePluginProviders()` through `resolveRuntimePluginRegistry()`, and the loader has a compatible-active-registry seam, but that compatibility check still treats a different `workspaceDir` as incompatible. Since #62051’s reported root cause is per-worker/session workspace directories causing cache misses, the current code and tests do not prove the observable regression is fixed.

## Best Possible Solution

Keep the PR open for maintainer/author follow-up. The best path is to add or require a targeted regression test for #62051 that sets an active gateway-bindable registry and then resolves provider hooks from a worker/session workspace with a different `workspaceDir`, proving whether plugin registration is avoided. If current main’s broader loader changes already fixed the real CPU regression by another path, close or supersede this PR manually from #62051 with that evidence; otherwise rework the PR against the current loader compatibility model instead of adding a redundant fast path.
