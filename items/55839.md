## Summary

Keep open. Current `main` still drops `HookContext` on the built-in tool path, so the PR's requested behavior is not implemented on `main`. The PR also has a real follow-up risk: adding `sessionKey` to built-in `before_tool_call` dispatch would start loop-history recording for every built-in tool call, because `runBeforeToolCallHook()` records calls whenever `ctx.sessionKey` is present and `recordToolCall()` still writes history with default config even when loop detection is otherwise unset/disabled.

## Evidence

- **Built-in tool adapter still calls before-hook without context:** `toToolDefinitions()` invokes `runBeforeToolCallHook({ toolName, params, toolCallId })` with no `ctx`, so built-in tools cannot pass `agentId`/`sessionKey`/`runId` into `before_tool_call`.

## Risks / Open Questions

- Merging as-is would change built-in tool behavior beyond `agentId` propagation by starting `sessionKey`-based loop-history tracking on those calls.
- There is no evidence on current `main` of a test that covers the built-in `before_tool_call` path receiving `agentId`, so the intended regression fix remains unproven in-tree.
