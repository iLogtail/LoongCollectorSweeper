## Summary

Keep open. Current main still routes normal `openclaw status` through a CLI-local memory status path that loads the active memory runtime, calls the plugin manager with `purpose: "status"`, probes vector availability, and renders that CLI-local manager status. The related PR #62643 is useful context, but it is open and docs-only per the provided PR body, so it does not remove the reported side effect or make status reflect the live gateway runtime.

## Best Possible Solution

Keep the issue open and fix the status memory contract in core. The likely product path is to make `openclaw status` read-only for memory plugins: prefer live gateway memory status when a gateway is reachable, or add a passive plugin status/snapshot capability that does not call plugin startup/probe paths; when neither is available, report memory/vector state as unknown rather than probing a fresh CLI-local runtime. Add regression coverage that normal status does not execute third-party memory plugin `register(api)` or vector/embedder startup just to render the Memory row. PR #62643 can remain useful as documentation, but it should not be treated as closing this bug unless paired with behavioral enforcement.
