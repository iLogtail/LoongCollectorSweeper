## Summary

Keep open. Current `main` still preserves raw `"\n"` joins between thinking fragments in the TUI formatter path, so the requested behavior is not implemented on `main`, and the PR discussion also surfaced an unresolved CRLF edge case in the proposed patch.

## Evidence

- **Current main still joins thinking fragments with newlines:** At current `main` SHA `250acdd2a28c190ed7934e3ba99022f24123d3c9`, `extractThinkingFromMessage()` returns `parts.join("\n").trim()`, and `extractTextBlocks()` also passes `thinkingParts.join("\n").trim()` into `composeThinkingAndContent()`. There is no whitespace-normalization helper on `main`.
  - file: [src/tui/tui-formatters.ts:247](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/tui/tui-formatters.ts#L247)
  - command: `git show 250acdd2a28c190ed7934e3ba99022f24123d3c9:src/tui/tui-formatters.ts | nl -ba | sed -n '247,333p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Current tests still lock in the pre-fix behavior:** The existing formatter test on `main` still expects two thinking blocks to render as `"alpha\nbeta"`, which matches the unfixed implementation rather than the PR's proposed `"alpha beta"` behavior.
  - file: [src/tui/tui-formatters.test.ts:252](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/tui/tui-formatters.test.ts#L252)

- **This formatter path feeds the live TUI stream output:** `TuiStreamAssembler.updateRunState()` reads `extractThinkingFromMessage(message)` and then renders via `composeThinkingAndContent(...)`, so the whitespace behavior under review is on the active TUI display path, not dead code.
  - file: [src/tui/tui-stream-assembler.ts:127](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/tui/tui-stream-assembler.ts#L127)

That means the PR may still need follow-up before merge.
