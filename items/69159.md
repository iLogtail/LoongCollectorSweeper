## Summary

Keep open. Current main only partially supersedes the PR: it has a dedicated `SessionWriteLockTimeoutError` and failover tests so lock waits are not treated as model failover, but the embedded attempt path still does not pass the agent `--timeout`/`params.timeoutMs` as the write-lock wait timeout. The PR’s central lock-wait behavior remains missing, and the related open #13744 still tracks this broader session-lock contention problem.

## Best Possible Solution

Keep the PR open but ask for a refresh against current main: pass `timeoutMs: params.timeoutMs` at the embedded attempt `acquireSessionWriteLock` callsite, reuse the existing `SessionWriteLockTimeoutError`/`OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT` contract, add or update a callsite regression test proving `--timeout` controls lock wait, and consider exporting the current error helper from the plugin SDK if SDK consumers are expected to branch on it. Cross-link #13744 as the remaining canonical session-lock contention tracker.
