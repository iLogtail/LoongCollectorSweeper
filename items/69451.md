## Summary

Keep #69451 open. Current main has useful partial mitigations for unbounded session storage, but the central memory/performance complaint is still present: `sessions.json` is still read and parsed as one file, whole-store writes still stringify the full object, cache layers retain full store copies, and full history reads still load an entire transcript before pagination.

## Best Possible Solution

Keep this issue open and treat it as the broader session-storage memory/performance tracker. The best path is to implement a real incremental storage layer: shard or database-backed session metadata, avoid retaining full serialized stores in process for large stores, stream or tail-read transcript history with cursor-aware pagination, and coordinate any `skillsSnapshot`/monolithic-write fixes with #71227.

- A complete fix may need an on-disk format migration or compatibility layer for existing `sessions.json` stores and transcript files.
