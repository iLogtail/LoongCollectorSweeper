## Summary

Keep open. Current main has useful session maintenance mitigations, but the core memory complaint remains: `sessions.json` is still loaded and parsed as one whole JSON object, several gateway paths still hold the full store in memory, and full transcript JSONL reads still occur for chat/history paths before limiting the returned messages.

## Best Possible Solution

Keep this issue open and track a real storage/read-path fix: avoid full `sessions.json` read-modify-write on hot paths, shard or index session metadata, and replace full transcript JSONL reads in `readSessionMessages`/history surfaces with bounded tail, paginated, or streaming readers. Preserve the existing maintenance cleanup as a mitigation, but treat it as insufficient for the memory/RSS behavior reported here.
