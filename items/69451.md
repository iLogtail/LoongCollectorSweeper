## Summary

Keep #69451 open. Current main has session-store retention and capping mitigations, but the reported full-file memory behavior still exists: `sessions.json` is synchronously read and parsed, writes stringify the full store, chat history reads full JSONL transcripts before applying pagination, and usage fallback reads the full transcript. Related #71227 covers the latency/run-budget angle, but it does not supersede this memory/RSS report.

## Best Possible Solution

Keep #69451 open. The best maintainer path is to treat #71227 as a related latency signal, then implement a real session-storage performance change: stream or tail-read transcript history/usage instead of full JSONL reads, make pagination disk-backed, reduce or remove full `sessions.json` read-modify-write on hot paths through sharding/indexing or out-of-line large fields such as `skillsSnapshot`, and keep the existing maintenance/disk-budget safeguards with regression tests for large stores and transcripts.

## Risks / Open Questions

- Closing this now would hide a still-valid memory path: current code mitigates unbounded growth but still materializes large session store and transcript files.
- A streaming/paginated fix must preserve current transcript seq/cursor behavior, compaction divider rendering, sanitization, and SSE update semantics.
- A sharded or out-of-line session metadata fix must preserve existing session fields, ACP metadata, plugin/runtime fields, migrations, and cleanup safety for transcript paths.
