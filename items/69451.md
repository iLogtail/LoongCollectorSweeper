## Summary

Keep #69451 open. Current main has session maintenance that mitigates unbounded growth, but the central request is still unresolved: `sessions.json` is still read/parsed and written as one monolithic file, history pagination still starts from a full transcript read, and memory sync still builds whole-file session entries. The open related #71227 corroborates the same storage hotspot from a latency/run-budget angle, but it does not supersede this memory/RSS issue.

## Best Possible Solution

Keep the issue open and treat it as the memory/RSS and streaming-storage tracker. The best fix is a storage/read redesign: shard or index session metadata instead of a monolithic `sessions.json`, avoid retaining both large parsed and serialized stores where possible, and replace history/memory transcript reads with bounded tail/cursor-aware JSONL readers. Keep #71227 linked for the latency/run-budget angle rather than using it as a close target.
