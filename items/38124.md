## Summary

Current `main` already addresses this report’s same-host loopback node-host path and clarifies that WS node-host pairing is handled through device pairing, not `openclaw nodes pending`. The reported `pairing required` + empty `nodes pending` behavior is obsolete against current `main` for the Oracle/VPS-style `openclaw node run --host 127.0.0.1 --port 18789` flow.

## Close Comment

Codex review: current `main` already covers this case. The pairing logic now treats loopback shared-secret node-host connections as local, so trusted same-host tools no longer fail with `pairing required` (`CHANGELOG.md:365`, current `main` `627c19c5cbbe8856a11effe1871c442ac87d56c9`). The gateway handshake code classifies direct/loopback clients as local and allows silent local pairing on that path (`src/gateway/server/ws-connection/handshake-auth-helpers.ts:75`, `src/gateway/server/ws-connection/handshake-auth-helpers.ts:193`), and the regression test suite now expects a local-direct node connection to succeed (`src/gateway/server.auth.control-ui.suite.ts:1207`). The docs also now clarify that WS node hosts pair through device pairing via `openclaw devices list` / `openclaw devices approve <requestId>`, while `openclaw nodes pending` is a separate store and does not gate the WS `connect` handshake (`docs/nodes/index.md:25`). Closing as implemented on `main`. If you still hit this on a current build, please open a new issue with the current `openclaw version`, auth mode, and the `openclaw devices list` output from the same host.
