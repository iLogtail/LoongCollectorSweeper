## Summary

Keep open. Current main still does not support the issue's `messageId` read behavior: Slack `message(action: "read")` forwards `threadId` but drops `messageId`, and the Slack reader returns either thread replies or channel history without filtering to a specific message timestamp. The related closed #44591 covered `threadId` support only, not fetching a specific thread reply by `messageId`.

## Best Possible Solution

Implement this in the Slack plugin read path: add `messageId` to the Slack `read` action dispatch and `readSlackMessages` options, fetch thread replies when `threadId` is present and filter to the requested Slack timestamp, support a channel-scoped specific-message read when only `messageId` is present, and add regression coverage in `message-action-dispatch`, `action-runtime`, and `actions.read` tests. If Slack permalink parsing is part of the user flow, add a focused parser test for extracting both the message timestamp and `thread_ts`.
