## Summary

Keep open. Current main fixed the core gateway stale-socket policy to require transport liveness, but the WhatsApp plugin still has a quiet-connection watchdog based only on inbound/start time and no presence-ping hook. This PR remains useful, but it needs to be rebased from the old src/web paths to extensions/whatsapp and covered with success/failure liveness tests.

## Best Possible Solution

Keep this PR open and rework it as a narrow WhatsApp plugin patch. Add a provider-proven liveness probe, likely backed by Baileys `sendPresenceUpdate("available")`, to the current listener/controller path under `extensions/whatsapp`; refresh watchdog/status liveness only when the probe succeeds, preserve forced reconnect when it fails, and add regression tests for both successful ping suppression and failed ping timeout behavior.

## Risks / Open Questions

- The PR currently targets old `src/web/auto-reply/monitor.ts` and `src/web/inbound/monitor.ts` paths, while current WhatsApp runtime behavior lives under `extensions/whatsapp`; direct merge would not patch the active code path.
- A timer-only refresh would regress silent-dead-socket detection. The rework needs to refresh liveness only after a provider-backed socket round trip succeeds and keep watchdog reconnect behavior when it fails.
