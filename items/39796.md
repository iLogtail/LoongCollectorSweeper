## Summary

Keep open. Current main no longer uses the old `src/web/...` paths and the gateway health policy now ignores stale app events unless a channel reports explicit transport activity, but WhatsApp still has its own 30-minute watchdog based on `lastInboundAt` or connection start time. That means the PR is stale and needs a rebase, but the underlying quiet-channel false-reconnect behavior is still plausibly present in the WhatsApp extension.

## Best Possible Solution

Keep this PR open as a stale but still relevant bug-fix proposal. The maintainable fix should live in the WhatsApp extension: either add a provider-proven transport liveness signal, such as a Baileys round-trip presence probe, and update the watchdog baseline only on success, or redesign the WhatsApp watchdog so quiet but connected accounts are not force-reconnected merely for lack of inbound messages. Add focused tests for successful ping suppressing the quiet-period restart and failed ping preserving watchdog reconnect behavior.
