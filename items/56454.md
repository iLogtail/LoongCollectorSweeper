## Summary

Keep PR #56454 open. Current main still has the exact shared image-tool timeout bug this PR targets: the agent `image` tool passes a hardcoded 30,000 ms timeout into provider calls, and the media-understanding runtime actively aborts those calls with `AbortController`. There is now a `tools.media.image.timeoutSeconds` config surface on main, but it is wired through the separate media-understanding runner path, not the agent image-tool path fixed by this PR. The PR discussion is also substantive: it links open bug #56419, includes focused test/schema coverage, and the author addressed the only concrete review finding by clamping to Node's timer range.

## Best Possible Solution

Keep the PR open and update it against current main. The best product/code path is to make the shared agent `image` tool honor a configurable timeout, preferably by reusing the existing `tools.media.image.timeoutSeconds` surface if maintainers consider it the canonical image-understanding timeout, or by accepting the PR's `agents.defaults.imageTimeoutSeconds` only if a separate agent-default knob is intentional. The final patch should keep the 30-second default, clamp to the safe timer range, update schema/help/generated config docs, and add focused tests for default behavior, configured propagation, and overflow clamping before closing #56419.
