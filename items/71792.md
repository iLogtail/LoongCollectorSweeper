## Summary

Keep #71792 open. Current main still rolls daily/idle-stale sessions to a new session and emits generic plugin session lifecycle hooks, but it does not emit the internal `command:reset` hook that the bundled `session-memory` handler is registered to receive. The PR addresses a real remaining gap and is not superseded by the closed related PRs/issues in the provided context.

## Best Possible Solution

Keep #71792 open and review it as the focused fix for auto daily/idle rollover memory persistence. The likely product path is to emit the same internal `command:reset` event for non-command stale rollovers, with `commandSource` distinguishing `auto:daily` and `auto:idle`, and keep the bundled `session-memory` behavior on its existing command-event contract unless maintainers choose a broader lifecycle-hook redesign.
