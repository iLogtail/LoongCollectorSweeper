## Summary

Keep open. Current main does not implement a per-binding or per-channel Discord nickname override. The binding schema has no `nickname` field, the normal Discord bot send path does not expose any nickname/username override, and the documented supported pattern for distinct Discord identities is separate Discord accounts per agent. The only mutable sender-name path on main is webhook-based delivery for thread bindings, not top-level channel bindings.

## Evidence

- **Binding schema has no nickname/identity override:** `AgentBindingMatch` and `AgentRouteBinding` only support channel/account/peer/guild/team/roles plus `agentId` and `comment`; there is no binding-level `nickname` field.
  - file: [src/config/types.agents.ts:37](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/config/types.agents.ts#L37)

  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Supported Discord identity split is separate bot accounts:** Multi-agent docs explicitly describe 'Discord bots per agent' by mapping each bot token to its own `accountId`; this is the documented way to get distinct Discord identities per agent.
  - file: [docs/concepts/multi-agent.md:274](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/docs/concepts/multi-agent.md#L274)

  - file: [extensions/discord/src/send.outbound.ts:360](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/extensions/discord/src/send.outbound.ts#L360)

## Risks / Open Questions

- Discord nicknames are guild-scoped rather than channel-scoped, so the exact feature as requested may not be implementable via the normal bot identity path without cross-channel races; a future fix likely needs webhook-based delivery or multiple bot accounts.
- There is partial adjacent functionality via thread-binding webhooks, but it does not satisfy the reported top-level channel binding use case.
