## Summary

Keep #41459 open. Current main does not support a `bindings[].nickname` or equivalent per-channel Discord identity override for ordinary Discord route bindings. Discord does have a webhook persona path, but it is currently scoped to bound thread delivery, not normal channel bindings with one bot token across multiple agents/channels.

## Best Possible Solution

Keep the issue open and implement it in the bundled Discord plugin rather than closing as ClawHub. A conservative path is to add a documented Discord route-binding identity override, or reuse `agents.list[].identity` for route-selected agents, then extend Discord outbound delivery to use per-channel webhooks where available with safe fallback to bot-token sends. Update config types/schema/help, Discord docs, and tests covering normal channel bindings, missing webhook permission, and webhook echo suppression.

## Risks / Open Questions

- A direct Discord bot nickname mutation may be guild-scoped rather than channel-scoped; implementation should validate Discord API behavior and likely prefer webhook-based per-message identity where permissions allow it.
- Webhook persona delivery has edge cases that need explicit handling before shipping for normal channel bindings: missing Manage Webhooks permission, fallback to bot send, media/poll/silent delivery behavior, and echo filtering when allowBots is enabled.
