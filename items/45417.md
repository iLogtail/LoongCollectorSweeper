## Summary

Keep open. Current `main` does not implement this PR’s live-tail behavior in the TUI, and the PR discussion itself shows the patch was knowingly scoped short of the reported external-channel/session-mode gap.

## Evidence

- **TUI only consumes `chat`, `chat.side_result`, and `agent` events:** `client.onEvent` in the TUI routes only those three event names. There is no handler for `session.message` / transcript events, so eager transcript updates on the gateway side will not show up live in `openclaw tui --session ...`.
  - file: [src/tui/tui.ts:1041](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/src/tui/tui.ts#L1041)
  - command: `nl -ba src/tui/tui.ts | sed -n '1038,1052p'`
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **TUI chat event model still excludes a user-turn/live-tail state:** `handleChatEvent` assumes run-scoped chat states and `ChatEvent` still only allows `delta | final | aborted | error`. The `user-message` state proposed by the PR is not on `main`.
  - file: [src/tui/tui-event-handlers.ts:288](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/src/tui/tui-event-handlers.ts#L288)
  - command: `nl -ba src/tui/tui-event-handlers.ts | sed -n '200,320p'; nl -ba src/tui/tui-types.ts | sed -n '21,24p'`
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **Gateway `chat.send` now emits transcript updates, not a live-tailed chat user event:** Current `main` calls `emitUserTranscriptUpdate()` around `chat.send` handling. That is a transcript/session-message path, not the PR’s `chat`/`user-message` broadcast path.
  - file: [src/gateway/server-methods/chat.ts:2494](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/src/gateway/server-methods/chat.ts#L2494)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '2488,2525p'`
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **Transcript updates are a separate subscribed event surface:** Gateway tests prove transcript writes arrive as `session.message` events for subscribed operator clients. That supports the code reading above: the current live update mechanism is on a different event surface than the TUI consumes.
  - file: [src/gateway/session-message-events.test.ts:214](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/src/gateway/session-message-events.test.ts#L214)
  - command: `nl -ba src/gateway/session-message-events.test.ts | sed -n '214,245p'`
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **External channel ingress still bypasses `chat.send`:** Current Slack and Discord inbound paths call `dispatchInboundMessage(...)` directly inside extension monitor code rather than routing through gateway `chat.send`, so a fix scoped only to `chat.send` would still miss real external-channel ingress paths.
  - file: [extensions/slack/src/monitor/message-handler/dispatch.ts:963](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/extensions/slack/src/monitor/message-handler/dispatch.ts#L963)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/dispatch.ts | sed -n '958,972p'; nl -ba extensions/discord/src/monitor/message-handler.process.ts | sed -n '934,948p'`
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **PR discussion already acknowledged the remaining gap:** Review comment 2934589954 flagged that external channel ingress does not pass through `chat.send`; the author replied on 2026-03-14 that the PR was intentionally scoped to `chat.send` and that channel-ingress coverage would require follow-up work. That makes the PR knowingly incomplete for the reported external-channel/session-mode problem.
