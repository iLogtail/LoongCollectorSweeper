## Summary

Keep open. Current main still has a mismatch for sandboxed sessions with `workspaceAccess: "none"`: the tool builder treats `none` as write-capable and can expose `write`, `edit`, and `apply_patch`, while the Docker/remote filesystem bridges and mount model only permit writes when `workspaceAccess === "rw"`. This matches the reported symptom of advertised write tools that fail at runtime rather than a fixed or obsolete issue.

## Best Possible Solution

Keep this issue open and fix the sandbox tool-capability contract. The likely product path is to make write-capable file tools depend on effective writeability (`workspaceAccess === "rw"` or an explicitly writable bridge/mount), suppress `write`/`edit`/`apply_patch` for `workspaceAccess: "none"` and `"ro"` unless a real writable target exists, and add regression coverage for `workspaceAccess: "none"` so tool inventory, FS bridge behavior, shell mount mode, memory flush behavior, docs, and diagnostics stay aligned.
