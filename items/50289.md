## Summary

Keep open. Current main partially mitigates the config-parse part by pinning the runtime config snapshot, but the reported `ownerAllowFrom` hot path is still linear: it normalizes the configured list during command authorization and uses array membership checks against the resolved owner list. The issue is recent, author is not a maintainer, labels are unprotected, and the only comment is unrelated commercial advice rather than a resolution.

## Best Possible Solution

Keep the issue open and fix the remaining authorization hot path: precompute/cache normalized owner allowlists per config snapshot, provider, account, and context override where needed; use `Set` membership for sender checks; add a focused regression/performance test with a large `commands.ownerAllowFrom` list; and document scaling guidance for large owner lists or alternate access-control patterns.
