## Summary

Keep open. Current `main` can route TTS audio into WebChat and exposes slash-command fallback diagnostics, but it does not appear to implement the requested explicit WebChat pipeline telemetry or a user-visible WebChat delivery/playback status surface.

## Evidence

- **WebChat audio routing exists:** `buildWebchatAudioContentBlocksFromReplyPayloads()` converts local TTS/media reply payloads into embedded WebChat audio blocks, so the core routing path exists.
  - file: [src/gateway/server-methods/chat-webchat-media.ts:154](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/gateway/server-methods/chat-webchat-media.ts#L154)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **TTS tool results are promoted for WebChat media extraction:** WebChat delivery promotes media-bearing tool results to `final` specifically so downstream audio extraction can pick them up, which matches the related TTS routing problem space.
  - file: [src/gateway/server-methods/chat.ts:2472](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/gateway/server-methods/chat.ts#L2472)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Current WebChat logs are ad-hoc warnings, not explicit stage telemetry:** The WebChat path logs warnings like `webchat audio embedding denied local path`, `webchat transcript append failed for media reply`, and `webchat dispatch failed`; there is no explicit structured `tts.generate.*`, `tts.route.selected`, or `tts.delivery.*` event emission here.
  - file: [src/gateway/server-methods/chat.ts:2417](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/gateway/server-methods/chat.ts#L2417)

  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Requested event names are absent on current main:** A repository search for `tts.generate.started`, `tts.generate.succeeded`, `tts.route.selected`, `tts.delivery.succeeded`, and `tts.delivery.failed` returned no matches.
  - command: `rg -n "tts\.generate\.(started|succeeded)|tts\.route\.selected|tts\.delivery\.(succeeded|failed)" src ui docs extensions test`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Existing status UI is browser-local read-aloud, not WebChat delivery state:** The WebChat UI `Read aloud` button just calls browser `speechSynthesis` and toggles button state on `onEnd`/`onError`; it does not surface gateway TTS routing or delivery outcomes.
