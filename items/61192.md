## Summary

Keep #61192 open. It is a recent regression report with concrete environment and logs, but the stated root cause is still unverified on current main: Telegram display names do flow into sender metadata and prompt/history text, yet the inspected OpenAI-compatible serializers do not show senderName being emitted as a chat message name field. The related fix PR #62441 is still open draft/unmerged and the discussion explicitly requested the actual failing outbound payload before merging a speculative sanitizer.

## Best Possible Solution

Keep the issue open and ask for the redacted outbound provider payload or a minimal failing test. Once the offending field is known, fix sanitization at that exact serializer boundary, preserve raw senderName for display and toolsBySender policy matching, and add regression coverage for both the provider payload and raw-name policy behavior.
