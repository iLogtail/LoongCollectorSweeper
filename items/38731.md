## Summary

Keep PR #38731 open. Current main still does not implement the requested high-frequency cron warning for CLI add/edit or gateway cron.add/cron.update. The linked issue #38730 remains open, and the PR discussion identifies still-relevant rework: cover update paths and use the gateway logger message-first contract. Existing cron session retention mitigates storage growth but does not provide the creation/update-time guardrail requested here.

## Best Possible Solution

Keep the PR open or replace it with a revised implementation that adds one shared high-frequency cron advisory for both create and update paths. CLI cron add/edit should warn non-fatally for --every intervals under 30 minutes, gateway cron.add/cron.update should log the same condition with warn(message, meta), and focused regression tests should cover both add and update without blocking valid schedules.

## Risks / Open Questions

- Merging PR #38731 as-is would leave cron.update uncovered, so users could still silently convert an existing job to a sub-30m schedule.
- Merging the PR as-is would need the gateway warning call corrected to logGateway.warn(message, meta) to match the current SubsystemLogger contract.
- The current tests cover cron add/update validation paths with 60s schedules but do not assert the requested advisory behavior.
