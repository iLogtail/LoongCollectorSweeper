## Summary

Keep open. Current main still implements `doctor.memory.status` as a live embedding-readiness RPC: the gateway handler awaits `manager.probeEmbeddingAvailability()` before responding, and the built-in memory manager still probes by embedding `["ping"]` through the long batch timeout path. I found no cache, cached-result metadata, or strict server-side budget that resolves #71568’s core performance ask.

## Best Possible Solution

Keep #71568 open and fix the core status path. The best maintainer direction is a per-agent/provider TTL cache for embedding probe results plus a short default server-side budget for `doctor.memory.status`, with `lastProbeAt` and cached/stale metadata in the payload. A deep/manual path can still run the full live probe for `openclaw memory status --deep` or an explicit refresh. Add gateway handler tests proving repeated status calls reuse cached results within the TTL, stalled probes return within budget, and docs/protocol text distinguishes cached status from deep probing.
