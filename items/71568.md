## Summary

Keep open. Current main still implements `doctor.memory.status` as a live gateway handler that awaits `manager.probeEmbeddingAvailability()` before responding, and the bundled memory manager still implements that probe by embedding `["ping"]` through the normal batch timeout path. I did not find a shipped TTL cache, strict short-budget default, or separate lightweight memory-health RPC that would resolve this issue; related memory status reports are adjacent but not a narrower canonical duplicate for this latency/cache ask.

## Best Possible Solution

Keep the issue open and fix the memory status control-plane path. A good maintainer path is to cache embedding-readiness per agent/provider/backend with a short TTL and `lastProbeAt`/`lastProbeOk`, or make default `doctor.memory.status` return last-known readiness while moving the long live embedding probe behind an explicit deep path. Add focused regression tests proving repeated status calls do not invoke `probeEmbeddingAvailability()` inside the cache window and that doctor/status do not wait on the full embedding batch timeout by default.
