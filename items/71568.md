## Summary

Keep open. Current main still has `doctor.memory.status` synchronously awaiting a live embedding probe, and the built-in memory manager still implements that probe by calling `embedBatchWithRetry(["ping"])` under the normal batch timeout policy: 120s remote and 600s local. Existing tests assert the live probe behavior, and related ClawSweeper reports #44426 and #47235 were also kept open for the same broader status/probe contract area, but this issue's latency/strict-budget ask is narrower and still actionable rather than a duplicate close.

## Best Possible Solution

Keep #71568 open and fix the control-plane contract. The best maintainer path is to make `doctor.memory.status` return a fast per-agent/provider readiness snapshot by default, including `lastProbeAt`, `lastProbeOk`, and the last error when available; run the expensive `probeEmbeddingAvailability()` only behind an explicit deep path or a short bounded refresh; and add regression tests proving repeated status calls do not call the embedding provider within the cache window and that timeout/failure is reported as stale/inconclusive rather than blocking the gateway response lane.
