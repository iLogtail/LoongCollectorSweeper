## Summary

Keep open. Current `main` still makes `doctor.memory.status` do a synchronous live embedding probe with the normal embedding-batch timeout, and the memory search layer explicitly does not cache status-only managers. I did not find a main-branch fix, a cache/budget split, or a docs/history signal that this report is obsolete.

## Evidence

- **Gateway doctor handler still blocks on a live embedding probe:** `doctor.memory.status` obtains a status-purpose memory manager, then unconditionally awaits `manager.probeEmbeddingAvailability()` before responding. There is no cache lookup, timeout override, or `deep`/`lite` branch in this handler.
  - file: [src/gateway/server-methods/doctor.ts:785](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/gateway/server-methods/doctor.ts#L785)

  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **Embedding probe is a real provider call, not cached state:** `MemoryIndexManager.probeEmbeddingAvailability()` initializes the provider and then calls `embedBatchWithRetry(["ping"])`. The embedding timeout policy still resolves batch probes to `2 * 60_000` ms for remote providers and `10 * 60_000` ms for local providers.
