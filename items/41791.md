## Summary

Current main already supports write-capable isolated filesystem access for isolated cron runs through sandbox configuration: `workspaceAccess: "rw"` for the agent workspace and `sandbox.docker.binds` for additional specific directories, with bind validation and symlink-escape hardening.

## Risks / Open Questions

- The implementation is coarser than the exact proposed schema: it provides `none`/`ro`/`rw` workspace access plus bind mount modes, not per-path CRUD flags, extension allowlists, or dedicated filesystem audit-log settings.

## Close Comment

Closing as implemented on current main after a Codex review. Isolated cron runs already execute under agent-scoped non-main session keys (`src/cron/isolated-agent/run.ts:459`, `src/agents/sandbox/runtime-status.ts:16`), and the sandbox already supports opt-in write access to the agent workspace via `agents.defaults.sandbox.workspaceAccess: "rw"` plus additional specific directory mounts via `agents.defaults.sandbox.docker.binds` (`docs/gateway/sandboxing.md:274`, `src/config/types.sandbox.ts:45`, `src/agents/sandbox/workspace-mounts.ts:23`). The bind path handling is also hardened against blocked roots and symlink-parent escapes (`src/agents/sandbox/validate-sandbox-security.ts:303`). The shipped `v2026.4.22` tree contains the same support, so there is not a remaining core gap to track in this issue. If you still need finer-grained per-path ACLs or audit-specific controls beyond `workspaceAccess` plus bind modes, please open a narrower follow-up with the exact missing behavior.
