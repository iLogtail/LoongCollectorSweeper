## Summary

Keep #71848 open. The report is fresh, concrete, and not covered by a protected maintainer-authored auto-close rule. Current main has substantial macOS launchd hardening, but the specific SIGABRT/background-start incident remains actionable: the current service environment still freezes `TMPDIR` into the managed LaunchAgent, and tests intentionally assert that behavior, which matches one of the reporter's suspected root causes. Existing GUI-session bootstrap errors and macOS permissions docs do not prove a fix for a post-bootstrap launchd SIGABRT, memory pressure, or port-not-bound failure.

## Best Possible Solution

Keep the issue open and investigate on a current macOS launchd environment. The likely product path is to stop persisting session-scoped `TMPDIR` into LaunchAgent plists or add doctor/service-audit repair for stale/nonexistent temp dirs, improve launchd diagnostics for exit 134 and port-not-bound startup, document the exact TCC/Full Disk Access/Aqua-session requirement for the gateway process context, and decide whether memory-cap guidance or heap diagnostics belong in gateway service management.
