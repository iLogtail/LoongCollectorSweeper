## Summary

Keep open. Current main still has a configurable rate-limit profile-rotation cap, defaulting to one same-provider retry before model fallback, and tests explicitly assert fallback after only two primary-provider attempts with three profiles available. That leaves #62022's requested default behavior, trying remaining Codex auth profiles before escalating to fallback models, unresolved. The reporter's workaround using `auth.cooldowns.rateLimitedProfileRotations` is real and is now documented, but it is not the same as fixing the default rotation policy.

## Best Possible Solution

Keep #62022 open as a real product behavior issue. The likely fix belongs in the embedded PI runner failover policy: rate-limit cooldown should advance through eligible `openai-codex` auth profiles for the current model, and model fallback should happen only after the eligible profile set is exhausted or independently cooling down. Any change should update the current cap tests and the model-failover/configuration docs so the default behavior and `auth.cooldowns.rateLimitedProfileRotations` contract stay aligned.
