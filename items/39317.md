## Summary

Keep #39317 open. Current main still treats exit-0 Codex CLI JSONL as a successful CLI run: the runner only throws on non-zero/abnormal process exits, the JSONL parser only returns text/session/usage and has no sandbox-denied semantic failure signal, and the returned output is recorded as `result: "success"`. The prior fix PR #39460 directly addressed this but is closed unmerged; related PR #66081 is scoped to final-message artifacts and explicitly does not change failure classification.

## Best Possible Solution

Keep the issue open. The best fix is to add structured Codex JSONL sandbox-denial detection in the CLI output/runner path, throw a non-success `FailoverError` before `buildCliRunResult`, add a typed failover reason/status policy if maintainers want one, avoid broad assistant-message keyword matching, keep local sandbox denial out of shared auth-profile health, and add parser plus runner regression tests for both detection and false-positive cases.
