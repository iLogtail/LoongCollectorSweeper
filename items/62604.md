## Summary

Keep open. Current main has the general Control UI image attachment path for pasted `image/*` clipboard file items, drag/drop, send serialization, and send-failure draft/attachment restoration. It does not appear to harden the specific reported failure mode where the clipboard provides a `data:image/...;base64,...` payload as text; the paste handler returns without `preventDefault()` when no `image/*` item exists, and the textarea input path will then accept the resulting value as draft text.

## Best Possible Solution

Keep the issue open and harden the Control UI composer paste path. The likely fix belongs in `ui/src/ui/views/chat.ts`: detect clipboard `text/plain`/`text/html` values that are `data:image/...;base64,...`, prevent default insertion, then either convert supported safe image data URLs into `ChatAttachment`s or reject them with a clear UI error. Add focused Control UI tests that prove a pasted data-image string does not become draft text, while preserving the existing image-file paste and send/draft restoration behavior in `ui/src/ui/app-chat.ts`.

## Evidence

A `text/plain` or `text/html` clipboard item containing a `data:image/...;base64,...` string would therefore still be allowed through the browser's normal textarea paste path.
