## Summary

Keep open. Current main has a conservative short-term promotion path and an adjacent memory-wiki belief layer, but it still lacks the requested active memory write resolver: classification into memory types, near-duplicate/conflict search before durable write, create/merge/update/conflict outcomes, source/provenance retention across merges, and revision history. The related #49108 smart-memory/ClawHub report is useful adjacent evidence for classifier/entity extraction problems, but it does not show this core memory-core pipeline is implemented.

## Best Possible Solution

Keep #42648 open and implement it in the active memory/memory-core write path, likely alongside the still-open canonical record, provenance, and review/edit/conflict issues. The best product direction is a resolver-backed durable candidate model that classifies memory type, requires source refs, normalizes entities, searches current durable memory for near duplicates and contradictions, routes outcomes to create/merge/update/conflict-for-review, and records revision/provenance history. memory-wiki’s claim/evidence/conflict concepts can inform the design, but the active promotion/flush path still needs its own enforceable write contract.
