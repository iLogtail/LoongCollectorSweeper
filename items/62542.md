## Summary

Keep PR #62542 open. Current main still lacks `models.providers.*.retry`, `retryOnStatus`, configurable retry backoff, and provider-fetch wiring for status-code-driven model retries. The newer SDK `Retry-After` cap is adjacent but does not implement this PR’s per-provider config surface.

## Best Possible Solution

Keep the PR open for maintainer design review. A good final path would add a validated `models.providers.*.retry` contract, wire it through model resolution into the guarded provider fetch boundary, preserve response-body cleanup on retried responses, avoid leaking model-only fields into channel retry config, and document the interaction with SDK Retry-After/failover behavior.
