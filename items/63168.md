## Summary

Keep open. Current `main` at `f896c3935ef8d706fe4193443c5c5eede5e64178` still lacks the hook-registry pin/refresh/release behavior this PR proposes, so the gateway hook-runner preservation bug does not appear implemented on main and the PR remains materially relevant.

## Evidence

- **Active loads still rebuild the global hook runner from the newly activated registry:** `activatePluginRegistry()` still does `setActivePluginRegistry(...)` followed immediately by `initializeGlobalHookRunner(registry)`. There is no hook-specific indirection here, so a later activating load can still replace the process-global hook runner.
  - file: [src/plugins/loader.ts:1831](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/plugins/loader.ts#L1831)
  - command: `nl -ba src/plugins/loader.ts | sed -n '1815,1855p'`
  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **Main only tracks active, HTTP-route, and channel registry surfaces:** `RegistryState` contains `activeRegistry`, `httpRoute`, and `channel`, but no hook surface. `setActivePluginRegistry()` syncs only the HTTP-route and channel surfaces, so there is no current-main mechanism to preserve a gateway hook registry independently of later default loads.
  - file: [src/plugins/runtime.ts:13](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/plugins/runtime.ts#L13)

- **Gateway bootstrap still pins only the channel registry:** `loadGatewayStartupPlugins()` and `reloadDeferredGatewayPlugins()` both pass `pinActivePluginChannelRegistry` as `beforePrimeRegistry`. Current main does not pin a hook registry or refresh the global hook runner during deferred repins.
  - file: [src/gateway/server-plugin-bootstrap.ts:98](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/gateway/server-plugin-bootstrap.ts#L98)

That matches one of the PR review concerns the contributor addressed in follow-up commits.
