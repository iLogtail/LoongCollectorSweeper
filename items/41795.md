## Summary

Keep open. Current main already supports standard HTTP(S) proxy env vars for Telegram through the current transport layer, but it does not implement or document the requested Telegram-specific `OPENCLAW_TELEGRAM_PROXY` env var. The PR branch is also stale against the current `extensions/telegram` implementation and would need rework rather than direct merge.

## Best Possible Solution

Keep this open for maintainer judgment, but rework it against the current `extensions/telegram` transport if the Telegram-specific env var is desired. The conservative implementation would resolve `channels.telegram.proxy` first, then `OPENCLAW_TELEGRAM_PROXY`, then the existing standard env proxy path, while preserving the current `resolveTelegramTransport` lifecycle, NO_PROXY behavior, TLS/IPv4 fallback handling, and media-download dispatcher attempts. Add tests for priority, blank env values, malformed explicit proxy behavior, and representative send/probe/monitor/audit paths; update Telegram docs/config reference if the env var becomes public.
