## Summary

Keep #42663 open. Current main recognizes Discord webhook IDs but still treats webhook-authored Discord messages as bot messages; with `allowBots` off they are dropped, and with `allowBots="mentions"` they are only mention-gated, not allowlisted by webhook ID. The Discord config types/schema/docs do not expose `trustedWebhooks` or equivalent `mapToUser` behavior. Related #10448 provides direct `/hooks/agent` HTTP ingress as an alternate automation path, but it does not preserve the requested Discord-channel webhook workflow or sender mapping.

## Best Possible Solution

Implement this in the bundled Discord plugin rather than closing it. Add typed/schema-backed Discord config for a scoped `trustedWebhooks` allowlist, update UI metadata and Discord docs, resolve trusted webhook IDs in preflight before the generic bot drop, keep `allowBots` defaulting to false, validate any `mapToUser` against existing guild/channel user policy, and add regression tests for trusted, untrusted, mention-gated, and bound-thread webhook echo cases. Treat `/hooks/agent` as a documented workaround, not as the final fix for this issue.

## Risks / Open Questions

- A fix must preserve the current anti-loop behavior for bot messages and bound-thread webhook echo suppression.
- `mapToUser` is security-sensitive and should be validated against existing guild/user access policy rather than bypassing allowlists or permissions.
- The existing `/hooks/agent` automation surface may be a useful workaround, but closing this issue on that basis would lose the requested Discord-native workflow.
