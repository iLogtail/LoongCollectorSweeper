## Summary

Keep open. Current `main` still requires `after_block_id` for `feishu_doc(action: "insert")`, so the reported prepend/index gap remains real. The PR is directionally useful, but it is not strong enough for conservative cleanup because the proposed schema would admit invalid index values and there is no local test coverage for the new path.

## Evidence

- **Current main still makes `after_block_id` mandatory for `insert`:** `FeishuDocSchema` defines the `insert` variant with required `after_block_id` and no optional `index` field, so callers still cannot request insertion at the beginning of the root document without a reference block.
  - file: [extensions/feishu/src/doc-schema.ts:35](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/extensions/feishu/src/doc-schema.ts#L35)
  - command: `nl -ba extensions/feishu/src/doc-schema.ts | sed -n '1,260p'`
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Runtime path still depends on locating an existing sibling block:** `insertDoc` still takes `afterBlockId: string`, fetches that block, paginates siblings to find its position, and inserts at `blockIndex + 1`. There is no branch that inserts by root-child index when no block id is available.

## Risks / Open Questions

- Merging the PR as posted would weaken validation by allowing non-integer or negative `index` values through the tool schema, pushing avoidable API errors to runtime.
- There is no local test coverage for `feishu_doc(action: "insert")`, so behavior around prepend/index insertion and backward compatibility with `after_block_id` is not yet defended.
