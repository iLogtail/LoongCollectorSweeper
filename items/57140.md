## Summary

Keep open. Current main still does not implement the PR's core normalization fix: the OpenAI WebSocket gate compares exact provider/API strings, while the run path only trims the provider. The PR is useful but stale against current main because stream selection has moved to `stream-resolution.ts`, current code now deliberately keeps `openai-codex` off the native public OpenAI WebSocket path, and only the empty-tools warm-up behavior appears already handled elsewhere.

## Best Possible Solution

Keep the PR open, but rework it against current `main`: normalize provider with `normalizeProviderId` and normalize `modelApi` with `trim().toLowerCase()` inside `shouldUseOpenAIWebSocketTransport`, while preserving the existing `modelBaseUrl` endpoint-classifier gate and current `openai-codex` HTTP exclusion. Add a focused regression test in `src/agents/pi-embedded-runner/run/attempt.spawn-workspace.websocket.test.ts`; if a defensive stream guard is still desired, implement it in `src/agents/pi-embedded-runner/stream-resolution.ts` with a non-shadowing boolean name.
