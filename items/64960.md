## Summary

Keep open. Current main has safeguards that can turn a summarization failure into a generic fallback summary, but it does not implement the requested compaction model fallback chain. `agents.defaults.compaction.model` is still a single string override, the compaction runtime resolves exactly one effective provider/model, the schema is strict and has no `compaction.fallbacks`, and runner model fallback intentionally does not handle context-overflow/compaction errors.

## Best Possible Solution

Keep the issue open and implement a compaction-owned fallback candidate chain, likely under `agents.defaults.compaction`, with schema/docs/tests. The runner should try the resolved compaction model and configured compaction fallback models ephemerally for the compaction operation, without mutating session model selection, then fall back to the existing generic summary or an explicit `compact_then_truncate` escape after all candidates fail. Add regression coverage for a 400 Azure content-filter-style summarization failure and verify related timeout/reduction behavior separately from #64962.

## Risks / Open Questions

- Current main's generic fallback summary is useful as a hard escape, but it can discard meaningful history; a model fallback chain should preserve that escape hatch while first trying configured alternate summarizers.
- The fallback model choice must remain per-attempt/runtime-only so it does not reintroduce the persisted model-lock class called out in related #58303.
