## Summary

Keep #60782 open. Current main disproves the reporter's suspected validation cause, because the Web UI sends an allowed minimal `update.run` payload and the gateway schema makes those fields optional. But the observable Windows Web UI failure is not proven fixed: the gateway still runs the global npm update from the running gateway process, reports `global update (omit optional)` when the fallback install step fails, and only schedules restart after update success. The provided related context for open PR #44614 directly tracks the remaining Windows locked-file/EBUSY class for this same global update surface.

## Best Possible Solution

Keep #60782 open until maintainers either land a Windows-safe Web UI/global npm update path, likely by superseding or fixing #44614, or add clearer error reporting that exposes the underlying npm stderr instead of only `global update (omit optional)`. A focused fix should preserve the existing `--omit=optional` retry, avoid replacing locked files from the running gateway on Windows, schedule/reconcile restart safely, and add regression coverage for the Web UI `update.run` gateway path.
