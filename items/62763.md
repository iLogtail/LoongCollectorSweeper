## Summary

Keep #62763 open. Current main still has the same actionable Signal doctor risk: `doctor --fix` runs the generic compatibility migration, the migration still promotes static top-level keys such as `cliPath`, `dmPolicy`, and `groupPolicy` into `channels.<id>.accounts.default` when named accounts exist but no default account exists, and Signal does not declare a setup-promotion override that would filter those shared defaults for named-account configs.

## Best Possible Solution

Keep the issue open. The best fix is to make Signal's doctor/setup-promotion behavior distinguish account-scoped credentials (`account`, `httpUrl`, `httpHost`, `httpPort`, maybe `cliPath` when truly account-owned) from shared channel defaults (`dmPolicy`, `groupPolicy`, shared `cliPath`), then add a regression test using the issue's two-account Signal config to prove `doctor --fix` does not invent a settings-only `channels.signal.accounts.default`.
