## Summary

Keep #70007 open. Current main has mature inbound audio transcription for existing channel/media paths, but it still lacks a documented, supported custom-client voice ingress contract: `/v1/responses` has no `input_audio`, `input_file` excludes audio MIME types, `chat.send` attachment parsing drops non-image payloads, and WebChat upload controls are image-only.

## Best Possible Solution

Keep the issue open for maintainer/product design. A good implementation would add one canonical custom-client audio ingress path, likely either an additive `input_audio` OpenResponses part or a typed `chat.send` audio attachment/upload flow, then route accepted audio into the existing `tools.media.audio` media-understanding pipeline with schema, docs, tests, limits, and security handling aligned with current image/file attachment code.
