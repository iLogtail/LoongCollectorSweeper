## Summary

Keep open. Current main has channel audio transcription and built-in Control UI voice surfaces, but it still does not define a canonical server-side audio ingress contract for custom chat clients. `/v1/responses` has no `input_audio`, `input_file` defaults exclude audio MIME types, `chat.send.attachments` remains untyped, and the current attachment parser/UI path is image-focused and drops non-image payloads.

## Best Possible Solution

Keep #70007 open as a focused Gateway/API design item. Define one additive, documented server-side audio ingress path, likely `input_audio` on `/v1/responses` or a typed `chat.send.attachments`/media-upload reference flow, then route accepted audio through `tools.media.audio` with protocol docs and regression tests.
