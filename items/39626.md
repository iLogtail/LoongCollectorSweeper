## Summary

Keep open. Current main still does not show source, docs, or regression-test evidence that the exact reported bare pseudo tool-call leak, `exec{...}`, is prevented on shell or Telegram delivery paths. Main has related hardening for XML-style tool-call tags, MiniMax XML, model tokens, memory tags, downgraded `[Tool Call: ...]` markers, and a scoped OpenAI-compatible `tool_choice: "auto"` mitigation, but those do not prove the reported bare JSON-ish reply format is handled across the affected surfaces.

## Best Possible Solution

Keep tracking this as a behavior bug. The best fix is to either prevent the provider from producing bare pseudo tool-call text for the affected route or add a conservative assistant-visible sanitizer for bare JSON-ish pseudo calls such as `exec{...}`, with focused regression coverage for final reply payloads and streaming/block delivery used by shell and Telegram. If the final root cause is provider transport configuration, keep the transport fix but still ensure raw pseudo-call text cannot reach user-visible delivery paths.
