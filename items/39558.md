## Summary

Keep #39558 open. Current main still creates suffixed synthetic Feishu reaction message IDs and still passes raw messageId/replyToMessageId values into multiple Feishu API boundaries. The PR addresses a real open bug linked from #34528, but it needs a current-main refresh because its helper is not present, it references the now-absent send-message.ts file, and newer API sinks such as streaming-card and pins are not covered by the submitted patch.

## Best Possible Solution

Refresh #39558 or replace it with an equivalent Feishu plugin patch that centralizes a reaction message-id normalizer and applies it at every Feishu API-facing message-id boundary: reply/send/read/edit/update, reactions, typing indicators, media resource/replies, streaming-card replies, and pins. Add focused helper tests plus at least one API-boundary regression test proving a `:reaction:<emoji>:<uuid>` ID is stripped before the Feishu SDK call while internal reaction identity remains suffixed.
