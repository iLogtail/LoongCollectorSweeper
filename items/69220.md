## Summary

Keep #69220 open. Current main still wires direct Gemini/Google provider reasoning output to tagged mode while the direct Gemini transport also emits native `thinkingConfig.includeThoughts`, matching the reported conflict. Existing tests and docs still assert the tagged Google behavior rather than a shipped fix, and the issue is recent with multiple affected-user confirmations.

## Best Possible Solution

Keep the issue open and patch the Google/Gemini reasoning-output contract. The likely product fix is to make direct `google-generative-ai` use `native` reasoning output when native Gemini `thinkingConfig` is active, update the Google provider hook/shared family or split the family so replay sanitation is not tied to tagged output, and revise the existing provider-utils/Google plugin tests plus docs that currently assert tagged mode. Add a regression around `google/gemini-2.5-pro` or Gemini 3 with thinking enabled and a tool call to ensure the system prompt does not require `<think>/<final>` while native thoughts still flow.

## Risks / Open Questions

- The fix should separate direct Gemini native API behavior from `google-gemini-cli` and replay-sanitization behavior; the current shared `google-gemini` family couples replay policy and tagged reasoning output.
- A good regression should cover a direct Gemini reasoning model with tools: no tagged prompt/final-tag enforcement, native `thinkingConfig.includeThoughts` preserved, and no empty post-tool assistant turn.
