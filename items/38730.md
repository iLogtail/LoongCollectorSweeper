## Summary

Keep open. Current main has meaningful partial mitigation for cron run-session accumulation through `cron.sessionRetention`, an automatic cron run-session reaper, and session token visibility, but the issue's guardrail request is not fully implemented: `openclaw cron add` and gateway `cron.add` still create high-frequency `--every` jobs without a warning, `sessions cleanup` still uses generic session maintenance rather than invoking the cron run-session reaper, and the linked warning PR #38731 is still open in the provided GitHub context.

## Best Possible Solution

Keep #38730 open until maintainers either land PR #38731 or an equivalent conservative warning in both `src/cli/cron-cli/register.cron-add.ts` and `src/gateway/server-methods/cron.ts` for high-frequency `schedule.kind: "every"` jobs, then decide whether `openclaw sessions cleanup` should invoke the cron run-session reaper or clearly document that cron retention is scheduler-owned through `cron.sessionRetention`.
