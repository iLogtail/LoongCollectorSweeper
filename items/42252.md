## Summary

Keep #42252 open. Current main has useful adjacent improvements, including LaunchAgent/service entrypoint drift repair, gateway status JSON/degraded output, and Telegram getUpdates 409 duplicate-poller hints in runtime logs. It still does not implement the central doctor UX request: a consolidated end-of-run gateway verdict that reconciles service loaded state, runtime PID, RPC health, service drift, duplicate poller signals, and ordered next steps.

## Best Possible Solution

Keep the issue open and implement a doctor-owned gateway diagnosis accumulator. It should collect service installed/loaded state, runtime PID/exit detail, RPC probe result, service audit drift, port/listener conflicts, recent channel/runtime diagnostic hints such as Telegram getUpdates 409 conflicts, then print one final healthy/degraded/down verdict with prioritized next steps and an optional compact JSON output. Reuse the existing gateway status/probe schema and Telegram polling diagnostics where possible.
