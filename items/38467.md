## Summary

Keep #38467 open. Current main still does not persist Baileys `contacts.upsert` data, expose saved WhatsApp contacts through the directory, or resolve outbound WhatsApp targets by contact name. The existing WhatsApp directory remains config-backed from `allowFrom`, while core already has a generic name/handle directory resolver that would work if WhatsApp supplied contact-backed entries.

## Best Possible Solution

Keep the issue open and implement it in the bundled WhatsApp plugin: persist account-scoped Baileys contact events locally, expose contacts through the WhatsApp directory with id/name/handle/phone metadata, make outbound resolution use unique contact-name matches through the existing shared resolver, and add tests/docs for ingestion, filtering, ambiguity, privacy, and message-tool behavior.

## Risks / Open Questions

- Persisting WhatsApp contacts is privacy-sensitive; any implementation should be account-scoped, local-only, documented, and gated by explicit config or clear consent.
- Name-based outbound resolution needs ambiguity handling so a partial or duplicate contact name cannot silently message the wrong recipient.
