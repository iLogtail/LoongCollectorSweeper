## Summary

Keep open. Current main mitigates the original default-600s repro by making the agent runtime timeout 48 hours and adding a separate LLM idle timeout, but the specific misclassification path still exists: the embedded runner only exempts compaction timeouts, and assistant failover still treats any non-compaction `timedOut` as a rotation/fallback signal when fallbacks are configured. There is no tool-execution timeout cause or active-tool exemption in the current runner.

## Best Possible Solution

Keep #52147 open. Implement phase-aware timeout handling in the embedded runner: track active tool execution or a general `timeoutCause`, exempt tool-execution timeouts from model/profile fallback, and reserve fallback for real LLM request or idle timeouts. Add a regression test with a configured fallback chain and a long-running/polling tool that times out the run while no LLM request is in flight, asserting no fallback model call occurs and the user gets an agent/tool timeout surface.
