## Summary

Keep #69513 open. Current main still forwards `--account` into cron delivery, trims/shape-validates it as a non-empty string, and gateway cron validation only checks configured announce channels, not configured account ids. The delivery resolver then gives explicit `job.delivery.accountId` highest precedence, so the reported invalid account can still be persisted and used at delivery time. The linked fix PR #69518 was closed unmerged, so it is not shipped behavior on current main.

## Best Possible Solution

Keep the issue open and land a focused gateway-side validation fix: when `delivery.mode` or `delivery.failureDestination.mode` is `announce`, normalize the channel, look up `cfg.channels[normalizedChannel].accounts`, and reject non-empty account ids that are not configured while preserving existing single-account/no-accounts behavior. Add regression tests in `src/gateway/server.cron.test.ts` for reject/accept on `cron.add`, and ideally the same coverage for `cron.update` and failure destinations.
