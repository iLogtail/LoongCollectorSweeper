## Summary

Keep open. Current `main` still does not implement `extraBody` passthrough for OpenAI-compatible TTS request bodies. The report's validation claim looks partly outdated on `main`, but the requested capability itself is still missing.

## Evidence

- **TTS config no longer hard-rejects unknown provider keys:** `TtsProviderConfigSchema` uses `.catchall(...)`, so arbitrary keys under `messages.tts.providers.<id>` are accepted by the core TTS schema rather than rejected as unrecognized.
  - file: [src/config/zod-schema.core.ts:486](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/config/zod-schema.core.ts#L486)
  - command: `nl -ba src/config/zod-schema.core.ts | sed -n '483,525p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **OpenAI TTS provider config has no extraBody field:** The normalized OpenAI TTS provider config includes `apiKey`, `baseUrl`, `model`, `voice`, `speed`, `instructions`, and `responseFormat`. There is no `extraBody`/`extra_body` field parsed or carried forward.
  - file: [extensions/openai/speech-provider.ts:32](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/extensions/openai/speech-provider.ts#L32)
  - command: `nl -ba extensions/openai/speech-provider.ts | sed -n '32,132p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Outbound OpenAI TTS request body is fixed-shape:** `openaiTTS()` serializes a JSON body containing only `model`, `input`, `voice`, `response_format`, optional `speed`, and optional `instructions`. There is no merge step for provider-specific passthrough JSON.
  - file: [extensions/openai/tts.ts:97](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/extensions/openai/tts.ts#L97)
  - command: `nl -ba extensions/openai/tts.ts | sed -n '93,105p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Docs mention custom TTS baseUrl/model/voice, not TTS extraBody:** The TTS docs say non-default `providers.openai.baseUrl` values are treated as OpenAI-compatible endpoints and allow custom model/voice names, but they do not document any TTS request-body passthrough. By contrast, `extra_body`/`extraBody` is documented only for model/completions config.
  - file: [docs/tools/tts.md:364](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/docs/tools/tts.md#L364)

  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Contract tests reflect the current fixed request shape:** The TTS contract suite's OpenAI mock posts only `input`, `model`, and `voice` for normal synthesis, reinforcing that current behavior has no generic body passthrough path.
  - file: [test/helpers/plugins/tts-contract-suites.ts:298](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/test/helpers/plugins/tts-contract-suites.ts#L298)
