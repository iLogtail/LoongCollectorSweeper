## Summary

Keep open. Current `main` still has the pre-fix `allowlistEnabled = groupPolicy === "allowlist" || hasGroups` logic, and the Telegram runtime still converts that state into a `group-chat-not-allowed` / `This group is not allowed.` rejection. The docs are also internally inconsistent about whether `groupPolicy: "open"` should bypass group allowlists, so this needs maintainer judgment rather than conservative cleanup closure.

## Evidence

- **Current main still enables group allowlist whenever any group entries exist:** `resolveChannelGroupPolicy()` on `main` still computes `allowlistEnabled` as `groupPolicy === "allowlist" || hasGroups`, so explicit `groups` entries continue to turn on allowlist behavior even when `groupPolicy` is `"open"`.
  - file: [src/config/group-policy.ts:345](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/config/group-policy.ts#L345)

That means the current `main` code path still rejects an unlisted group if `resolveChannelGroupPolicy()` turned on allowlist mode because `groups` exists.
