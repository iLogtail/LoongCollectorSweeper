## Summary

Keep open. The issue discussion still matches current main: Telegram inbound body construction only runs `expandTextLinks(...)`, and that helper still filters to `text_link` entities, leaving bold/italic/code/pre/strikethrough/underline formatting entities as plain text before the agent sees the prompt.

## Best Possible Solution

Fix this in the Telegram plugin inbound path, not by closing the issue. Add a helper that converts Telegram message entities from the original text into Markdown in one offset-safe pass, covering links plus bold/italic/code/pre/strikethrough/underline/spoiler as appropriate. Add focused tests for the helper and `resolveTelegramInboundBody`, including combined link + formatting entities, captions, and emoji/UTF-16 offsets so entity ranges do not drift after inserted Markdown markers.
