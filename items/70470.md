## Summary

Keep open. Current main has not clearly fixed the reported WhatsApp read-receipts dependency, and the existing triage comment points at the same code path that is still present. The exact suggested `access.shouldMarkRead` condition is probably not sufficient by itself, because unauthorized messages return before read marking, but accepted WhatsApp messages still await `sock.readMessages()` before the inbound message is enriched and enqueued.

## Best Possible Solution

Keep #70470 open. The best maintainer path is to add a targeted WhatsApp monitor regression where an accepted inbound message's `sock.readMessages()` rejects or does not settle and verify the on-message/calendar pipeline still proceeds. If that reproduces the risk, make read marking best-effort off the critical path or bounded by a short timeout, then live-verify with a sender account that has WhatsApp read receipts disabled.
