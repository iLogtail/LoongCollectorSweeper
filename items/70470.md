## Summary

Keep #70470 open. Current main still puts accepted WhatsApp inbound processing behind an awaited read-receipt call, so the reported dependency on read-receipt behavior is plausible and not already fixed. The existing triage comment's `access.shouldMarkRead` explanation is only partly supported because denied messages are filtered before read receipts, but the central bug remains actionable.

## Best Possible Solution

Keep the issue open and fix it in the WhatsApp plugin by decoupling read-receipt delivery from inbound processing: move the read-receipt side effect after enqueueing or make it bounded/fire-and-forget with logging, preserve `sendReadReceipts` and self-chat behavior, optionally use `access.shouldMarkRead` for clarity, and add a regression test showing `sock.readMessages` rejection or delay cannot prevent `onMessage` and downstream calendar/tool work.

## Evidence

- **Accepted messages await read receipts before processing:** `handleMessagesUpsert` normalizes an accepted inbound message, then calls `await maybeMarkInboundAsRead(inbound)` before enrichment and enqueueing. This means a slow or hanging read-receipt side effect can delay the actual agent/calendar work.

## Risks / Open Questions

- The exact WhatsApp/Baileys behavior when a sender disables read receipts was not live-reproduced in this read-only review, so the provider-side failure mode remains unconfirmed.
- A fix that only adds `access.shouldMarkRead` would probably not address the most important remaining risk, because accepted messages still await `sock.readMessages` before processing.
