## Summary

Keep #58702 open. Current main still has the same Control UI/WebChat message-action overlap contract mismatch described in the issue discussion: the CSS reserves right padding only for `.chat-bubble.has-copy`, but the current grouped message renderer never adds `has-copy` when copy/expand actions are rendered.

## Best Possible Solution

Keep the issue open and fix the current grouped message renderer by adding the padding-reserving action state when `hasActions` is true, then add a focused UI unit test for assistant messages with copy/expand actions. Verify the CSS still works on mobile/touch paths where `.chat-bubble-actions` is always visible.

## Evidence

- **Issue discussion identifies the same root cause:** The provided issue body reports copy/expand action icons covering message text after 2026.3.31, and comment 4169559798 specifically calls out that `.chat-bubble.has-copy` padding exists while the renderer never adds the class. The current source inspection confirms that assessment still matches main.
