## Summary

Keep open. Current `main` does not already implement the requested image zero-output diagnostics, and the PR as proposed still has unresolved control-flow gaps that match the review comments.

## Evidence

  - file: [src/media-understanding/runner.ts:875](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/media-understanding/runner.ts#L875)
  - command: `nl -ba src/media-understanding/runner.ts | sed -n '730,940p'`
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Current main only warns on failed decisions, not skipped zero-output cases:** After processing attachments, `logWarn` runs only when `decision.outcome === "failed"`; otherwise the code only emits a verbose log. That means a zero-output path that resolves to `skipped` is still not covered by the proposed behavior on `main`.
  - file: [src/media-understanding/runner.ts:919](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/media-understanding/runner.ts#L919)

- **Skipped zero-output attempts are a real state in current code:** `runAttachmentEntries` records `outcome: "skipped"` for empty output and for `MediaUnderstandingSkipError`, while only hard errors become `outcome: "failed"`. The model-decision type also explicitly includes `success | skipped | failed`, so an all-skipped/no-output scenario is valid and distinct from a failed scenario.
