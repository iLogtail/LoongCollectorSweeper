## Summary

Keep open. Current `main` still prefers `contextTokensOverride`/`agents.defaults.contextTokens` over a known model window, so the reported over-reporting bug is still present in both resolution and persistence paths.

## Evidence

- **Resolver still returns override before model lookup:** `resolveContextTokensForModel()` immediately returns `contextTokensOverride` when it is positive, before checking configured or discovered model limits. That matches the issue's reported failure mode instead of capping by the known model window.
  - file: [src/agents/context.ts:426](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/agents/context.ts#L426)

  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Status/runtime selection path still prefers agent default override:** `resolveContextTokens()` uses `agentCfg?.contextTokens ?? resolveContextTokensForModel(...)`, so `/status`-style reporting paths still let the override replace a smaller known model window rather than acting as a ceiling.
  - file: [src/auto-reply/reply/model-selection.ts:351](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/auto-reply/reply/model-selection.ts#L351)
