## Summary

Do not close #51429. Current main does not contain a hardcoded `wangtao` path, but the likely bug from the discussion remains: onboarding/configure flows still resolve the default workspace to an absolute home path and persist that value into config. The related PR #56810 explicitly targeted this issue but was closed unmerged, so it does not supersede the open issue.

## Best Possible Solution

Keep the issue open and implement a shared workspace serialization helper for onboarding/setup/configure paths: preserve explicit custom paths, but when the selected value is the canonical default for the current profile, persist `~/.openclaw/workspace` or the profile-aware portable equivalent. Add regression tests for interactive onboarding, noninteractive onboarding, configure, and setup, plus a migration/doctor path for already-baked default absolute paths if maintainers want to repair existing configs.

## Risks / Open Questions

- A fix should distinguish canonical default paths from deliberate custom absolute paths so existing explicit workspace choices are not rewritten unexpectedly.
- Existing configs that already contain a foreign absolute default may need a doctor/setup repair or clear migration guidance in addition to changing future serialization.
- Maintainer response should narrow the issue from “hardcoded username” to “default workspace path baked into config” so the useful report is preserved without endorsing the literal hardcode claim.
