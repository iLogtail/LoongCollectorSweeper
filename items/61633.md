## Summary

Current main contains several relevant mitigations for cron/session accumulation, but the issue should stay open because the startup OOM crash-loop path is not fully fixed: the session store is still read and JSON-parsed in full before load-time maintenance can prune or cap it. A preexisting oversized sessions.json can still be a boot blocker, which is part of the central report.

## Best Possible Solution

Keep this open, but narrow the remaining work to the boot-time recovery path: make session-store startup tolerate oversized existing sessions.json files, either through a streaming/pre-parse pruning path, a size-bounded rescue mode, or an early maintenance command that the gateway can run before normal load. The current cron reaper, runningAtMs recovery, and status-agnostic cap should be kept as the forward-growth fix.

## Evidence

## Risks / Open Questions

- Closing now would leave the reported crash-loop recovery gap untracked: existing oversized sessions.json files still have to be fully read and parsed before cleanup can run.
- The implemented mitigations reduce future accumulation, but they do not prove that a 185 MB legacy sessions.json can be recovered automatically during gateway startup.
