## Summary

Keep #51767 open. Current main still has `cron.sessionRetention` wired only to `isCronRunSessionKey()`, and the cron reaper does not import or check `isSubagentSessionKey()` or `isAcpSessionKey()`. There is separate subagent run cleanup/auto-archive logic, but it is not the requested cron-retention sweep for completed subagent/ACP session-store entries, and the related fix PR #51861 was closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep the issue open and implement a core retention path for completed ephemeral subagent and ACP session-store entries, with regression tests for completed versus running sessions and docs clarifying whether this is governed by `cron.sessionRetention`, a new session/subagent retention field, or the existing `session.maintenance` controls.

## Risks / Open Questions

- A fix must distinguish run-mode, keep-mode, and persistent session-mode subagents so retention does not delete actively used thread-bound sessions.
- ACP and native subagent keys share similar session-store lifecycle concerns, but ACP may need its own retention semantics if transcript ownership differs from native subagents.
- Changing `cron.sessionRetention` from cron-only to broader ephemeral-session retention should be documented because current docs define it as cron-run specific.
