## Summary

Current main should keep this issue open. OpenShell backend creation/execution is implemented, but the reported `openclaw sandbox list` symptom is still plausible: the built-in sandbox CLI path skips plugin runtime loading, while the OpenShell sandbox manager is registered only during full OpenShell plugin registration. The reporter's follow-up that the OpenShell sandbox became ready while OpenClaw still showed it as stopped matches that code path.

## Evidence

- **OpenShell manager registration is plugin-runtime-only:** The OpenShell plugin registers the `openshell` sandbox backend factory and manager only from its full `register()` path; non-full registration returns before installing the manager.
  - file: [extensions/openshell/index.ts:14](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/extensions/openshell/index.ts#L14)
  - command: `nl -ba extensions/openshell/index.ts | sed -n '1,80p'`
  - sha: [b64bfc5d9abf](https://github.com/openclaw/openclaw/commit/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab)
- **Sandbox list marks missing backend managers as stopped:** `listSandboxContainers()` reads registry entries, looks up the backend manager, and if no manager is registered pushes the entry with `running: false`. That would render an OpenShell registry entry as stopped even if OpenShell itself has a ready sandbox.
  - file: [src/agents/sandbox/manage.ts:40](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/src/agents/sandbox/manage.ts#L40)
  - command: `nl -ba src/agents/sandbox/manage.ts | sed -n '1,160p'`
  - sha: [b64bfc5d9abf](https://github.com/openclaw/openclaw/commit/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab)
- **Built-in CLI commands skip plugin registration:** `sandbox` is a built-in subcommand, and `shouldSkipPluginCommandRegistration()` returns true when the primary command is already built in. That means `openclaw sandbox list` does not necessarily load plugin runtime registrars before calling `listSandboxContainers()`.
  - file: [src/cli/command-registration-policy.ts:14](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/src/cli/command-registration-policy.ts#L14)
  - command: `nl -ba src/cli/command-registration-policy.ts | sed -n '1,240p' && nl -ba src/cli/program/register.subclis-core.ts | sed -n '90,125p' && nl -ba src/cli/run-main.ts | sed -n '300,340p'`
  - sha: [b64bfc5d9abf](https://github.com/openclaw/openclaw/commit/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab)
- **OpenShell manager would report status via openshell CLI when loaded:** The current OpenShell manager's `describeRuntime()` calls `openshell sandbox get <name>` and reports `running: result.code === 0`, so the remaining concern is the CLI path reaching this manager, not the manager's status logic itself.
  - file: [extensions/openshell/src/backend.ts:64](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/extensions/openshell/src/backend.ts#L64)

- **Docs still promise normal sandbox CLI lifecycle:** The OpenShell docs tell users to verify with `openclaw sandbox list` and describe OpenShell sandboxes as managed through the normal sandbox CLI, so a stopped status for a ready OpenShell sandbox remains a user-visible defect rather than an expected limitation.
  - file: [docs/gateway/openshell.md:56](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/docs/gateway/openshell.md#L56)
