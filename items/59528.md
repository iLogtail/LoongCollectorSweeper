## Summary

Keep open. Current main has a real OpenShell backend, but the reported `openclaw sandbox list` stopped-status path is still plausible: the sandbox CLI has no command-catalog policy to preload plugins, and sandbox listing marks entries as stopped when the backend manager is not registered. OpenShell creation is also lazy until a sandboxed tool operation, so a chat turn can create the OpenClaw registry entry before `openshell sandbox create` has run.

## Best Possible Solution

Keep the issue open and fix the sandbox CLI/backend lifecycle path. The likely code direction is to make `openclaw sandbox list` and `openclaw sandbox recreate` load enabled sandbox-backend plugins before calling `listSandboxContainers`, or add a targeted backend-manager registration path for sandbox backends. Add a regression test with an `openshell` registry entry proving `sandbox list` invokes the OpenShell manager instead of falling through to `running: false`; separately clarify or adjust lazy creation semantics if maintainers expect an OpenShell sandbox to exist immediately after a chat turn without tool execution.
