## Summary

Current `main` still matches the report. Discord channel/component upload paths still drop the media MIME type when building `Blob`/`files`, and shared media loading still optimizes non-GIF images to PNG/JPEG, so WebP uploads can still be converted and animated WebP frames can still be lost.

## Evidence

- **Discord upload Blob still has no MIME type:** `toDiscordFileBlob` rebuilds any `Uint8Array` as `new Blob([arrayBuffer])` with no `type`, so attachment MIME metadata is discarded before upload.
  - file: [extensions/discord/src/send.shared.ts:340](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/extensions/discord/src/send.shared.ts#L340)
  - command: `nl -ba extensions/discord/src/send.shared.ts | sed -n '340,346p'`
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Channel message uploads still omit contentType:** `sendDiscordMedia` passes `{ data, name }` in `files` and never forwards `media.contentType`, unlike other Discord upload paths that do include `contentType`.
  - file: [extensions/discord/src/send.shared.ts:422](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/extensions/discord/src/send.shared.ts#L422)
