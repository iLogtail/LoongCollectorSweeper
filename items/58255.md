## Summary

Keep open. Current `main` does use the native Gemini embedding endpoints and auth flow, so the report does not look like a bad Gemini request-body/path bug. But the memory embedding HTTP path still goes through `fetchWithSsrFGuard` in its default strict mode, and that mode explicitly ignores ambient env proxies unless the caller opts into trusted env-proxy mode. The Gemini memory path does not opt in, so a proxy-only WSL2 setup can still plausibly fail with `fetch failed` even when direct `NODE_USE_ENV_PROXY=1 fetch()` succeeds.

## Evidence

- **Gemini memory embeddings use the native Gemini API shape:** `createGeminiEmbeddingProvider` builds `models/...:embedContent` and `:batchEmbedContents` endpoints and sends Gemini auth headers via `parseGeminiAuth`, so current `main` is not using the generic OpenAI-style `/embeddings` path for Gemini memory indexing.

  - sha: [0783090d](https://github.com/openclaw/openclaw/commit/0783090d)
- **Memory remote HTTP still uses guarded fetch with default mode:** `withRemoteHttpResponse` calls `fetchWithSsrFGuard` without passing `mode`, `proxy`, or any trusted-env-proxy option. That leaves the fetch guard on its default strict path for memory remote requests.
  - file: [src/memory-host-sdk/host/remote-http.ts:14](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/memory-host-sdk/host/remote-http.ts#L14)

- **Strict guarded fetch explicitly ignores env proxy:** `fetchWithSsrFGuard` only uses `EnvHttpProxyAgent` when `mode === TRUSTED_ENV_PROXY`; otherwise it resolves pinned DNS and builds a direct dispatcher. The regression test explicitly states that strict mode 'ignores env proxy by default'.
