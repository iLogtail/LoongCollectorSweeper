## Summary

Keep open. Current main has deterministic skill source precedence, eligibility filters, per-agent allowlists, and alphabetical prompt/command ordering, but it does not implement user-configurable skill priority, priority metadata, or `openclaw skills priority` CLI management. The request is a core skill loader/prompt-selection policy gap, not just a new ClawHub skill/plugin to publish.

## Best Possible Solution

Keep the issue open and design a core skills policy extension. A conservative implementation would add a documented `skills.entries.<skill>.priority` or `skills.priority` list in `openclaw.json`, validate it in the config schema, apply it consistently before prompt truncation and command catalog generation, preserve existing source precedence for same-name overrides, and add focused tests for loader ordering, per-agent filters, docs, and CLI management if a CLI is accepted.

## Risks / Open Questions

- The design needs maintainer judgment on whether priority should affect prompt ordering, prompt truncation, slash-command ordering, same-name replacement, or all of those surfaces.
- OpenClaw currently relies on the model matching skill descriptions unless a skill is invoked explicitly, so a priority feature should be framed as catalog ordering/visibility policy unless deterministic tool dispatch is added.
