## Summary

Keep #64127 open. Current main still does not implement the requested provider quota circuit breaker: there is no `quota_exhausted` failure reason, periodic quota/usage-window errors are still classified and documented as `rate_limit`, HTTP 429 still wins as `rate_limit`, and only `billing`/`auth_permanent` enter the long disabled lane. The PR is not obviously merge-ready because review comments identify behavior gaps and the diff includes unrelated plugin status/cost surfaces, but it is not obsolete or implemented on main.

## Best Possible Solution

Keep the PR open for maintainer review or replacement. The best path is a narrow core implementation: add a distinct quota-exhausted reason where the failover/auth-profile contracts need it, classify quota text ahead of generic 429 handling, route quota exhaustion into an appropriate persistent disabled/backoff lane, prevent frequent single-provider probes during quota windows, add targeted tests for Gemini/ZhipuAI-style payloads and disabled-window behavior, and update model-failover docs. Split unrelated plugin status/cost API work into its own PR or canonical issue.
