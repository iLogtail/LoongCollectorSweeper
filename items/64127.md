## Summary

Keep #64127 open. Current main still classifies periodic/daily/monthly quota exhaustion as `rate_limit` and only routes `billing`/`auth_permanent` into the long disabled lane, so the PR is not obsolete or implemented on main. It is not merge-ready as-is, though: review comments identify core behavior gaps, and the branch also carries unrelated plugin status/cost surface changes.

## Best Possible Solution

Keep the PR open for maintainer review or replacement. The best path is to decide whether OpenClaw core should add a distinct quota-exhausted disabled lane, then implement it narrowly in the failover/auth-profile pipeline with tests for 429 quota payload precedence, failure-reason scoring, disabled-window behavior, and probe throttling. If maintainers do not want this product behavior, reopen or replace #64085 with the explicit decision rather than silently closing this PR as already implemented.
