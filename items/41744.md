## Summary

Keep #41744 open. Current main preserves and forwards tool media once a media path is present, but it still does not implement the reported `read` image fallback when the tool result has image content but no `details.path`; the related open PRs #46268 and #50359 remain relevant.

## Best Possible Solution

Keep the issue open until one canonical fix lands on main. The likely fix belongs in the embedded tool-result media path: pass the recorded `tool_execution_start` args into media extraction, fall back to trusted `read` args (`path`, `file_path`, `filePath`) only when image content exists and no explicit media path was present, preserve the queued media into final/outbound payloads, and add regression coverage for extraction plus `handleToolExecutionEnd` delivery.
