## Summary

Keep open. Current main has partial allow-always improvements, but it still does not solve the reported dynamic-command workflow. For interpreter-style commands such as the reporter's Python example, default config does not persist broad executable/script trust, and Windows-generated allow-always entries can remain argument-bound, so changing `Day_007.md` to `Day_008.md` can still require another approval. The related docs PR #61687 was closed unmerged, and a second user confirmed the same full-command-hash pain on 2026-04-24.

## Best Possible Solution

Keep this as the canonical tracker for scoped durable exec approvals. The best product/code path is to add an explicit allow-always scope model, such as trust exact command, trust resolved executable, trust a bound script path, or trust a user-visible command pattern, with UI/API affordances and regression tests for Windows and POSIX dynamic script invocations. Exact command hashing should remain a safe fallback, not the only durable option for common parameterized workflows.

## Risks / Open Questions

- A broad executable-level fix could over-trust interpreters such as Python, Node, awk, or shell wrappers unless it is paired with file binding, inline-eval handling, and Windows-specific argv tests.
- The persistence sub-report appears less clear on current main because valid files are loaded and normalized before saving, but the central dynamic allow-always problem is still unresolved.
