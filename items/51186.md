## Summary

Keep #51186 open. Current main has strong evidence for the /hooks/agent session-context fix and the plugin webhook route-registry fix, but the BlueBubbles phone-format mismatch remains: configured/routed conversation matching still treats `+1509...` and `1509...` as different handles, so a peer-specific binding can still fall through to a catch-all.

## Best Possible Solution

Keep the issue open but narrow the remaining work: preserve the shipped /hooks/agent session rewrite and plugin HTTP route-registry fix, then implement BlueBubbles phone-handle canonicalization for configured/runtime conversation matching so `+1...` and `1...` resolve to the same direct conversation when appropriate. Add regression tests covering peer-specific BlueBubbles bindings plus a catch-all, inbound `+1...` vs `1...`, and chat_guid-derived handles.
