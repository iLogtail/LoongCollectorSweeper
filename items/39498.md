## Summary

Keep #39498 open. Current main still has the ByteString-risk behavior in at least one header-bound secret path, but the PR's proposed shared-normalizer change is still too broad because it would globally mutate valid non-header secrets and secret-presence checks. The linked bug #39426 was closed stale, not fixed by current main.

## Best Possible Solution

Keep the PR open or replace it with a narrower fix: add a header-specific secret/header value normalizer or validation path at HTTP-header construction boundaries, cover the Comfy `X-API-Key` path with a regression test, and preserve `normalizeSecretInputString` semantics for non-header secrets and presence checks.
