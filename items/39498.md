## Summary

Keep #39498 open. Current main still has a plausible ByteString crash path for pasted non-Latin1 header-bound secrets, especially Comfy Cloud `X-API-Key`, but the PR's global `normalizeSecretInputString` change is too broad because it would mutate valid Unicode secrets used outside HTTP headers, such as Matrix password login bodies. The existing review comments correctly identify that product boundary, and no narrower canonical replacement is visible in the provided context.

## Best Possible Solution

Keep the PR open, but request a narrower patch. Preserve trim-only semantics for `normalizeSecretInputString`, `hasConfiguredSecretInput`, and non-header credentials. Apply `normalizeSecretInput` or `normalizeOptionalSecretInput` only where values become HTTP headers, starting with Comfy Cloud `X-API-Key` and any shared provider request auth/header sanitizer. Add regression coverage for the Comfy pasted-U+2502 path and a guard proving Matrix-style password/body credentials preserve non-Latin1 characters.
