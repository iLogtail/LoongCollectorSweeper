## Summary

Keep #39498 open. Current main still leaves the shared SecretInput boundary as trim-only and at least one header-bound plugin path can still pass a pasted non-Latin1 API key into `Headers`. The PR also should not be merged as written because globally switching `normalizeSecretInputString` to the Latin1-stripping helper would mutate valid non-header secrets such as Matrix passwords.

## Best Possible Solution

Keep the PR open but request a scoped fix instead of the global normalizer replacement. The best path is to preserve trim-only semantics for generic `SecretInput`, `hasConfiguredSecretInput`, and non-header passwords, then apply `normalizeSecretInput` or a configurable resolver normalizer only at header-bound secret callsites such as Comfy `X-API-Key`, with regression coverage for the affected plugin/provider header paths.
