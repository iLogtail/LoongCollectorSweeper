## Summary

Keep open. Current main has strong session reset and Anthropic replay cleanup, but it does not implement the reported recovery guarantee: `/new` intentionally preserves the selected model and thinking/reasoning settings, and Telegram `/new` is still not a control-lane interrupt for long model inference. The issue is also recent and has a second user confirming a thinking-loop variant with Kimi/Feishu, so this is not a conservative stale close.

## Best Possible Solution

Keep the issue open as a focused recovery-semantics bug. The best path is to reproduce Opus/Kimi thinking-loop behavior, then decide whether `/new` should gain a hard-reset mode that cancels active inference and clears provider-owned transient thinking state, or whether OpenClaw should expose a separate recovery command/workflow that can preserve normal `/new` model-setting semantics while disabling thinking or swapping models when a reasoning loop is detected.

## Evidence

- **Issue discussion has a second affected provider/channel:** The original report says Opus with extended thinking keeps looping after `/new` until switching to Sonnet. A later commenter reports the same class of issue with Kimi 2.5+ on Feishu and says disabling thinking is the only usable workaround.
- **Session reset rotates/aborts but preserves thinking state:** `performGatewaySessionReset` aborts the active embedded run and creates a new session, but the new entry carries over `thinkingLevel` and `reasoningLevel` from the prior entry. That means `/new` does not currently clear the thinking mode implicated by the report.

- The remaining behavior is model/provider-dependent and likely needs a live or synthetic reproduction to avoid overfitting a recovery policy.
- Changing `/new` to clear model or thinking overrides would conflict with current tested persistence semantics, so the product decision needs care.
