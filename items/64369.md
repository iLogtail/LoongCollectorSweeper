## Summary

Keep open. Current main still rejects Claude-style flat read/write/edit aliases, and the PR is a focused compatibility proposal with useful discussion. It is not implemented, not duplicated by the related tool-name-alias PR, and not a safe ClawHub close because the current patch and existing plugin seams need maintainer design work around pre-validation argument normalization.

## Best Possible Solution

Keep the PR open for maintainer design. A maintained fix should either add a supported pre-validation tool-argument normalization seam that survives toToolDefinitions/customTools, or implement a narrowly scoped core compatibility layer with tests for read/write/edit alias normalization, wrapper ordering, canonical-key precedence, and rejection of unsafe/broad aliases. If maintainers prefer plugin ownership, first ensure the plugin API can normalize these arguments before schema validation across arbitrary provider ids without enumerating every provider.
