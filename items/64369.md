## Summary

Keep open. Current main still rejects the Claude Code-style flat argument aliases, and this PR addresses a real compatibility gap, but the submitted implementation is not enough for the production embedded Pi path because OpenClaw converts tools through `toToolDefinitions()` and currently drops `prepareArguments`. This is also not a clean ClawHub close: the documented plugin hook can rewrite params only around execution, while the upstream Pi loop validates arguments before execution; provider schema normalization is provider-scoped and does not provide a provider-agnostic pre-validation argument hook.

## Best Possible Solution

Keep the PR open, but ask for a maintained fix that normalizes Claude Code-style arguments before Pi validation in the actual embedded path. The smallest credible direction is to preserve `prepareArguments` in `toToolDefinitions()` and add regression tests proving read/write/edit aliases work through that adapter, while keeping canonical calls unchanged. The PR should also remove fork-patch comments, add unit/adapter coverage, and decide whether the generic `file` alias is safe. Coordinate with #64150 only for tool-name alias work; it does not supersede this parameter-normalization issue.

## Risks / Open Questions

- As submitted, the PR likely would not fix real embedded agent runs because the alias normalizer is lost when tools are converted to Pi `ToolDefinition`s.
- The broad `file` -> `path` alias could collide with future tool parameters unless it is narrowed or explicitly scoped and tested.
- Unconditional alias normalization changes core read/write/edit validation semantics and needs focused regression coverage through the adapter/pre-validation path.
