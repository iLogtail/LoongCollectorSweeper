## Summary

Keep #43447 open. Current main has partial fixes for related lockout paths, including local slash-command routing and clearing auto-created model overrides on reset, but the central Moonshot/Kimi problem is still plausible: billing-looking messages carried on HTTP 429 still enter the rate-limit lane before billing classification can win. The TeamoRouter comment is a useful external workaround for multi-provider failover, but it does not replace a core/provider classification fix.

## Best Possible Solution

Keep the issue open and fix it in the provider/failover classification path: allow explicit billing indicators such as insufficient balance/credits/quota to override HTTP 429 for Moonshot/Kimi, or move provider-specific failover classification ahead of generic rate-limit matching. Add regression coverage for a Moonshot-style 429 billing payload, user-facing billing copy, and auth-profile/fallback state so billing disables are not treated as model-scoped transient rate limits. Keep the existing reset and slash-command bypass behavior as partial mitigation.
