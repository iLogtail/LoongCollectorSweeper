## Summary

Keep #69319 open. Current main still does not register or implement `skills.uninstall`, so the linked feature request remains real. The PR is a focused attempt to add the missing gateway RPC, but the provided review comments identify fixable safety/semantics issues that should be addressed before merge rather than closing the PR as obsolete.

## Best Possible Solution

Keep the PR open or supersede it with an equivalent focused patch that adds `skills.uninstall` across the TypeBox protocol schema, generated protocol exports/models, gateway method list, admin scope classification, handler, docs, and backend ClawHub uninstall helper. Before merge, require tests for installed removal, idempotent missing removal, malformed slug classification, stale lockfile cleanup, legacy tracked slug removal, and refusal to remove non-ClawHub/manual skill folders.

## Risks / Open Questions

- The PR as described needs provenance gating so `skills.uninstall` only removes ClawHub-tracked skill folders and does not delete manually managed or gateway-installer skill directories with the same slug.
- Invalid slug handling should be split from backend/server failures so clients receive `INVALID_REQUEST` instead of retryable-looking `UNAVAILABLE` responses.
- Lockfile-only cleanup should expose clear result semantics, because dropping a stale lock entry while returning `removed: false` can confuse operator UIs.
