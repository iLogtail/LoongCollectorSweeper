## Summary

Keep open. Current main has adjacent outbound durability work, including a shared write-ahead delivery queue and startup recovery, but the Telegram interactive/final reply path described in #53895 still bypasses that shared queue and sends through `deliverReplies`/direct grammY API calls. A transient `sendMessage` failure in that path is rethrown after emitting failure hooks, and tests still expect generic grammY `sendMessage` network envelopes not to be retried for non-idempotent sends.

## Best Possible Solution

Keep #53895 open. The best fix is to make Telegram final/block reply delivery durable by routing interactive `deliverReplies` sends through the shared outbound delivery queue, or by adding equivalent write-ahead queueing around that direct bot API path, then draining eligible Telegram entries on poll/reconnect using `drainPendingDeliveries` with account/thread matching and backoff. Add regression tests for `Network request for 'sendMessage' failed!` during final reply delivery, queued persistence, and successful later drain without duplicate sends.

## Risks / Open Questions

- Naively retrying ambiguous Telegram send failures can duplicate visible messages; the existing retry policy deliberately avoids that for non-idempotent sends.
- The shared queue already covers some outbound paths, so the eventual fix should avoid double-enqueueing and preserve the current queue claim/ack/fail semantics.
