## Summary

Keep #62307 open. Current main still does not handle WhatsApp `messages.update` events, message edits, or revocations in the bundled WhatsApp inbound monitor. The PR targets a real missing bundled-plugin behavior and is not superseded by the related reports, but the current PR head still needs changes for the in-flight edit race noted in review plus focused tests before it should merge.

## Best Possible Solution

Keep the PR open and request changes. The right path is to land this in the bundled WhatsApp plugin, but split state so queued-buffered messages can be mutated while in-flight messages are no longer considered mutable; late in-flight edits should either produce the same post-processing notice semantics or another deliberate behavior. Add focused tests in the WhatsApp inbox harness for `messages.update` listener lifecycle, buffered edit replacement before debounce flush, revocation ignore, processed-message late-edit notification, unauthorized/unknown edit suppression, self-chat `fromMe` edits, and the in-flight `onMessage` race.
