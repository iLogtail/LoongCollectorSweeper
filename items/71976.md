## Summary

Keep open. Current main still matches the reported memory-core behavior: light dreaming ingests daily/session signals immediately before selecting candidates, then sorts candidates by `lastRecalledAt` before `recallCount`; deep promotion gates on a combined signal count that includes `dailyCount` and `groundedCount`; and rehydration still uses the narrow `preferredSpan + 3` / `8` search span. This is a concrete, same-day bug report with code-level reproduction details, not stale or superseded cleanup.

## Best Possible Solution

Keep the issue open and fix it in `extensions/memory-core`: add regression coverage for light dreaming candidate ordering with fresh daily-only entries versus older recalled entries; decide whether `minRecallCount` should use true `recallCount` or a renamed/explicit total-signal gate; adjust scoring so real multi-query recalls can pass the documented thresholds; widen or otherwise make rehydration tolerant of shifted broad snippets; and update docs/status wording if the gate remains signal-based rather than recall-based.
