## Summary

Current main still matches the core report. The light phase still sorts by `lastRecalledAt` before `recallCount`, daily/session ingestion still records non-recall signals that refresh `lastRecalledAt`, deep promotion still gates on combined `recallCount + dailyCount + groundedCount`, and the relocation search span remains `preferredSpan + 3` / `8`. This is a recent, concrete memory-core bug report with actionable reproduction data, so it should stay open.

## Best Possible Solution

Keep this issue open and fix it in `extensions/memory-core`: make light staging prioritize real recall strength before applying the limit, separate actual recall gates from daily/grounded support signals or rename/recalibrate the gate, make score aggregation match the chosen signal semantics, widen or otherwise harden rehydration for shifted broad snippets, and add regression tests for older high-recall entries outranking fresh daily-only entries plus shifted multi-line candidates.
