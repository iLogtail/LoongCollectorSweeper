## Summary

Keep #71592 open. Current main still advertises shared `/status` and `/compact` in TUI autocomplete/help, but `handleCommand()` has no local `status` or `compact` cases and its default branch sends the raw slash text through `sendMessage()`. In `openclaw tui --local`, that send path uses `EmbeddedTuiBackend`, whose `sendChat()` starts a local embedded run via `agentCommandFromIngress()`, so the reported fallthrough remains a real bug rather than a duplicate or already-shipped fix.

## Best Possible Solution

Keep the issue open and fix the TUI local embedded command boundary. Recognized shared commands such as `/status` and `/compact` should either execute through a shared local command adapter or return an explicit local unsupported/system response in `opts.local` mode, but they should not fall through to `sendMessage(raw)`. Add focused regression tests for local `/status` and `/compact` and keep gateway-connected forwarding behavior covered separately.

## Risks / Open Questions

- A fix that blocks all unknown slash commands in the TUI could regress gateway-connected forwarding for shared commands; the behavior should distinguish local embedded mode from gateway mode.
- Regression coverage should assert local embedded `/status` and `/compact` do not reach `sendChat()` or `agentCommandFromIngress()`, while preserving intentional gateway command forwarding.
