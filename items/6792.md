## Summary

Current main still does not implement manifest-level install-time `configPatch`. The manifest contract and loader omit the field, install result/persistence paths do not carry or apply one, docs still require plugin config to be set manually, and the existing `configPatch` machinery remains provider-auth scoped. The related PR #43421 was closed unmerged, so it does not supersede this issue.

## Best Possible Solution

Keep #6792 open for a core plugin install/API decision. A complete fix should add a typed manifest-level `configPatch` contract, parse and sanitize it from plugin metadata, propagate it through all install result paths, apply a validated merge during install persistence with explicit array semantics and visible diff/skip output, and update tests plus public docs such as `https://docs.openclaw.ai/plugins/manifest` and `https://docs.openclaw.ai/tools/plugin`.

## Risks / Open Questions

- Manifest-declared config mutation is trust-sensitive and needs sanitization, prototype-pollution protection, validation before write, conflict handling, and clear diff/skip UX.
- Array semantics remain unresolved: the issue proposes replacement, while the `tools.alsoAllow` follow-up likely needs additive union behavior to avoid clobbering another plugin's entry.
- A complete implementation needs coverage across linked path, copied path/archive, npm, marketplace, ClawHub, bundled fallback, invalid patch, config include/write-through behavior, and uninstall messaging.
