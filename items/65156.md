## Summary

Keep open. The current main tree does improve degraded search by storing embeddings in `chunks` and falling back to in-process cosine search when sqlite-vec is unavailable, and docs now describe that fallback. But the specific report is about sqlite-vec loading successfully while registering no SQL functions; current loader still treats `sqliteVec.load()`/`loadExtension()` success as sufficient, does not probe `vec_version()`/`vec0`, and the locked dependency remains `sqlite-vec@0.1.9`. The issue also has two fresh independent confirmations on Linux versions after the original report, so this is not a stale low-signal report.

## Best Possible Solution

Keep the issue open and fix the sqlite-vec load contract in `src/memory-host-sdk/host/sqlite-vec.ts`/package mirror: after loading, run a minimal supported probe such as `SELECT vec_version()`/`vec_length(...)` or create/drop a temporary `vec0` table; if the probe fails, return `{ ok: false, error }` so indexing and search enter the existing fallback path. Add a regression test for a load call that appears to succeed while vec0/functions are unavailable, and separately decide whether to bump/rebuild sqlite-vec for Linux if the ABI mismatch is confirmed.
