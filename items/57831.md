## Summary

Keep open: current `main` still persists `sessionTarget="current"` into direct-message session keys and reuses any persisted `session:<id>` at runtime, so the reported isolation bug is not implemented on `main`. The PR is directionally useful, but current review context also shows unresolved edge cases in the proposed guard, so this is not conservative-close material.

## Evidence

- **Create normalization still persists current session keys without a DM guard:** `normalizeCronJobInput` resolves `sessionTarget === "current"` straight to `session:${assertSafeCronSessionTargetId(sessionKey)}` whenever session context exists, with no check for direct-message sessions. That matches the PR's reported bug rather than a fix.
