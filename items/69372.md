## Summary

Keep open. Current `main` still allows raw protected canvas URLs through when `canvasHostUrl` is absent, so the reported behavior is not implemented on `main`. The submitted PR also appears unsafe as written: a timer-driven websocket reconnect would force a disconnect that rejects pending RPCs and clears in-flight chat run state in the current Control UI reconnect path.

## Evidence

- **Current main still renders protected canvas paths without a scoped host:** `resolveCanvasIframeUrl(...)` returns the same-origin protected path unchanged when `canvasHostUrl` is missing, and the unit test explicitly expects `/__openclaw__/canvas/...` to be allowed on current `main`. That means the fail-closed behavior described by the PR is not already shipped.
  - file: [ui/src/ui/canvas-url.ts:52](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/ui/src/ui/canvas-url.ts#L52)
  - command: `nl -ba ui/src/ui/canvas-url.ts | sed -n '1,140p'; nl -ba ui/src/ui/canvas-url.test.ts | sed -n '1,120p'`
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Canvas auth is still time-bounded on the server:** The gateway hello path mints a scoped canvas capability with a 10-minute TTL, and HTTP canvas auth extends that TTL only when the canvas is actively used. This supports the underlying expiration problem still being real on current `main`, rather than already fixed elsewhere.
