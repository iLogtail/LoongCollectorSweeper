## Summary

Keep #41966 open. Current main still intentionally skips `MEDIA:` extraction inside fenced code blocks, the in-tree regression test asserts that fenced `MEDIA:` lines are ignored, and the related PR #41980 is closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep the issue open and implement a deliberate parser/docs fix. The likely path is to update `splitMediaFromOutput()` so a valid standalone `MEDIA:` directive inside a fenced block is either extracted and stripped from visible text or produces a clear warning, then update `src/media/parse.test.ts` and user-facing docs/prompt guidance to cover fenced backtick and tilde blocks without weakening existing unsafe-path rejection.

## Evidence

- **Parser still skips fenced media lines:** `splitMediaFromOutput()` parses markdown fence spans, checks each line offset, and immediately keeps/pushes any line inside a fence before continuing, so a `MEDIA:` line inside the fence is not extracted as media.
  - file: [src/media/parse.ts:501](https://github.com/openclaw/openclaw/blob/6cd047e7c27021d8955e09fa3598029eede5a1a7/src/media/parse.ts#L501)

  - sha: [6cd047e7c270](https://github.com/openclaw/openclaw/commit/6cd047e7c27021d8955e09fa3598029eede5a1a7)
- **Test coverage codifies the current ignored behavior:** The current test is named `returns ordered text and media segments while ignoring fenced MEDIA lines` and expects `MEDIA:https://example.com/ignored.png` inside a fenced block to remain in a text segment, not in `mediaUrls`.
  - file: [src/media/parse.test.ts:117](https://github.com/openclaw/openclaw/blob/6cd047e7c27021d8955e09fa3598029eede5a1a7/src/media/parse.test.ts#L117)

## Risks / Open Questions

- Changing this behavior touches shared media parsing used by reply directives, streaming directive handling, Pi embedded tool media, and UI rendering, so the fix should preserve existing path/URL safety checks and prose false-positive protections.
