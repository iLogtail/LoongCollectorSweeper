## Summary

Keep #61612 open. Current main has some overlapping lazy command and plugin CLI machinery, but it has not shipped this PR's requested subcommand help fast path for the named hotspots. In particular, `pairing` and `plugins` registration still intentionally load plugin CLI commands around their own registration, and plugin-primary help still goes through the full plugin CLI command registry path rather than a help-only metadata path. The PR also has substantive unresolved review comments about async memory registration, legacy plugin CLI help fallback, and pairing help losing channel metadata, so it needs author/maintainer follow-up rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer/author follow-up. A viable version should fit the current lazy registration architecture, avoid async plugin `register(api)`, preserve full-loader fallback for legacy third-party plugin CLI help, and keep pairing help's plugin-backed channel metadata intact. Narrow fast paths could still be added for commands whose help output is proven independent of plugin/runtime registration.
