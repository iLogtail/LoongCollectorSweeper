## Summary

Keep open. Current main has bounded in-run retry handling, provider/auth-profile cooldowns, and a few targeted auto-reset recovery paths, but I did not find the requested gateway/session-level circuit breaker that persists consecutive session failure state, marks a session unhealthy, stops or rotates future retries after a threshold, and exposes operator/UI state. The linked #62614 retry-storm symptom is closed as implemented for narrower retry caps; related #59161 remains open and is narrower than this broader session-failure breaker request.

## Best Possible Solution

Keep this open as a core reliability/product enhancement, or have a maintainer explicitly consolidate it with a canonical session-breaker issue. The likely implementation belongs in the gateway/runtime session layer: persist per-session consecutive failure counters/signatures and breaker cooldown/unhealthy metadata, reset counters on success or explicit reset, trip after configurable thresholds, stop automatic retries or call the existing reset path, emit structured logs/events, and expose clear UI/operator actions such as resume anyway or start fresh session.
