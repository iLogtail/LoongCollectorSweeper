## Summary

Keep open. Current main has related safeguards, including first-search bootstrap for an empty index and manual `memory index --force`, but the central reported behavior is still plausible in source: when an index already has content, `memory_search` does not await dirty/full-reindex sync before querying the existing builtin SQLite index. The related #59137 PR is open and explicitly scoped to rollback/retry/cache preservation rather than query-time behavior, so it does not supersede this issue.

## Best Possible Solution

Keep the issue open and fix it in `extensions/memory-core`: before builtin `memory_search` can return an empty result from an already-populated but dirty or full-reindex-needed store, either await the pending sync/reindex or return an explicit unavailable/reindexing state. Add focused tests for a populated stale index where metadata/config drift requires full reindex and the first search must not silently query the old transitional state.
