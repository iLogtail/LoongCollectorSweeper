## Summary

Keep open. Current `main` still does not render remote Markdown image URLs in WebChat/Control UI. The behavior is enforced in code, covered by tests, and documented as part of the UI CSP/security model, so #55215 is not implemented and remains an actionable product/security tradeoff for maintainers.

## Evidence

- **Markdown renderer drops non-data image URLs:** `md.renderer.rules.image` returns escaped alt text unless the image source matches an inline `data:image/...;base64` URI, so `![alt](https://...)` will not emit an `<img>` tag.
  - file: [ui/src/ui/markdown.ts:409](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/ui/src/ui/markdown.ts#L409)
  - command: `nl -ba ui/src/ui/markdown.ts | sed -n '409,420p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Unit tests lock in current behavior:** Tests explicitly assert that remote Markdown images are flattened to alt text and that only base64 data URI images render as `<img>`.
  - file: [ui/src/ui/markdown.test.ts:268](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/ui/src/ui/markdown.test.ts#L268)
  - command: `nl -ba ui/src/ui/markdown.test.ts | sed -n '268,296p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **WebChat message view uses the sanitized Markdown renderer:** Assistant markdown in chat is rendered through `toSanitizedMarkdownHtml(markdown)`, so the image-flattening rule applies to WebChat output.
  - file: [ui/src/ui/chat/grouped-render.ts:1475](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/ui/src/ui/chat/grouped-render.ts#L1475)
  - command: `nl -ba ui/src/ui/chat/grouped-render.ts | sed -n '1475,1478p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Control UI CSP blocks remote image fetches:** The CSP header sets `img-src 'self' data:` only. Remote `http(s)` image URLs are blocked by the browser, matching the reporter's note that no network request is issued.
  - file: [src/gateway/control-ui-csp.ts:35](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/gateway/control-ui-csp.ts#L35)
  - command: `nl -ba src/gateway/control-ui-csp.ts | sed -n '35,50p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Docs describe this as intentional and non-configurable:** Control UI docs state that only same-origin and `data:` images render, remote `http(s)`/protocol-relative image URLs are rejected without a network fetch, and that this behavior is always on.
  - file: [docs/web/control-ui.md:317](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/docs/web/control-ui.md#L317)
  - command: `nl -ba docs/web/control-ui.md | sed -n '317,327p'`
