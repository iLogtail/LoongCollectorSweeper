## Summary

Keep open. Current main has not implemented the central fix: steer injection is still gated on `isStreaming()` in the embedded queue, reply-run registry, and `runReplyAgent`. The PR addresses a real remaining bug, but it should not be auto-closed as implemented or duplicate; it needs maintainer review/revision because the current PR discussion includes a live P1 concern for handles without `isStopped` and the branch contains broad unrelated changes.

## Best Possible Solution

Keep #65010 open but request a cleaned-up revision or maintainer-authored replacement that fixes the remaining main behavior: use an active/stopped lifecycle predicate instead of `isStreaming` for steer delivery, preserve safe fallback behavior for every backend that lacks `isStopped`, update Codex/app-server or keep the streaming fallback where needed, and add focused regression tests for active-not-streaming, stopped/teardown, compacting, and reply-backend paths. Related PRs #65221 and #65222 should remain separate follow-ups because their bodies describe interrupt scheduling and Telegram delivery bypass work, not a shipped replacement for this core steer gate fix.
