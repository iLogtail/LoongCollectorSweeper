## Summary

Keep open. Current main still runs `memoryFlush` only in the pre-turn runner path, while in-turn SDK/runtime compaction paths can complete and update compaction counts without invoking the flush helper or documenting a clear exclusion. The issue is recent, concrete, unprotected, and not covered by the related ClawSweeper reports, which concern different compaction or channel behaviors.

## Best Possible Solution

Keep the issue open and fix the runtime contract. The best path is to make actual successful auto-compaction share a memory-flush guard, either by running a best-effort flush before timeout/overflow/context-engine compaction and SDK auto-compaction, or by adding a post-compaction audit that detects `compactionCount > memoryFlushCompactionCount` and schedules/records a safe flush. Add regression coverage for in-turn compaction and update the memory/compaction docs to describe the exact guarantee.
