## Summary

Keep #61895 open. Current main has adjacent ACP repair improvements and pins acpx@0.6.1, but the observable process-leak problem is not fixed: the core repair path can re-ensure a dead persistent runtime handle without an explicit old-handle/process-tree cleanup, and the pinned acpx runtime still terminates only the directly spawned adapter child before detaching/unrefing if it does not exit. The related #68916 is open and adjacent for oneshot/parent-reset leaks, but it is not a complete canonical replacement for this persistent ensureSession repair leak.

## Best Possible Solution

Keep the issue open and fix the lifecycle at the ACPX runtime boundary, preferably in openclaw/acpx and then consume the release here: spawn adapter subprocesses in an owned process group or otherwise track the full process tree, close/kill the previous tree before fresh persistent ensureSession replacement, apply the same cleanup on close/reset/gateway shutdown, and add regression tests covering persistent repair plus oneshot cleanup. #68916 can remain as an adjacent child-session/reset case, but #61895 should track the persistent repair leak unless maintainers create a broader canonical ACP process-lifecycle issue.
