## Summary

Keep open. Current main still has the loose gateway-facing CLI timeout behavior this PR targets: the relevant gateway, gateway-rpc, devices, and channels status paths still accept string timeout options and coerce them with Number(...) before passing values to gateway calls. The PR is still plausibly useful, though it needs rebase/adaptation to the current gateway-rpc.runtime.ts split.

## Best Possible Solution

Keep the PR open and ask for a rebase onto current main. The best implementation is to add a shared strict millisecond timeout parser/Commander option parser, wire it into the current gateway-facing CLI surfaces including src/cli/gateway-cli/call.ts, src/cli/gateway-rpc.runtime.ts, src/cli/devices-cli.ts, src/cli/channels-cli.ts, and src/commands/channels/status.ts, preserve existing defaults including the gateway-rpc 30000 ms default, and keep the focused regression tests for invalid input and valid passthrough.
