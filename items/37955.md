## Summary

Current `main` at `2fb9c7e3e586044467f360edf8d8f5170f1c60f5` still does not emit internal `command:new`/`command:reset` hooks for idle or daily session rollovers. The rollover path now emits plugin `session_end`/`session_start` lifecycle hooks, but the bundled `session-memory` hook still subscribes only to command hooks, so the reported gap remains.

## Evidence

- **Command hooks are only emitted by explicit reset helper:** `emitResetCommandHooks()` creates and triggers internal `command` hook events. This is the explicit command path; it is not part of the idle/daily rollover logic.
  - file: [src/auto-reply/reply/commands-reset-hooks.ts:95](https://github.com/openclaw/openclaw/blob/2fb9c7e3e586044467f360edf8d8f5170f1c60f5/src/auto-reply/reply/commands-reset-hooks.ts#L95)
  - command: `nl -ba src/auto-reply/reply/commands-reset-hooks.ts | sed -n '95,145p'`
  - sha: [2fb9c7e3e586](https://github.com/openclaw/openclaw/commit/2fb9c7e3e586044467f360edf8d8f5170f1c60f5)
- **Idle/daily rollover path computes stale reasons but only fires plugin session lifecycle hooks:** `initSessionState()` derives `previousSessionEndReason` from stale idle/daily freshness, then later runs `session_end`/`session_start` plugin hooks when a new session is created. There is no call here to `emitResetCommandHooks()` or `triggerInternalHook()` for stale rollovers.
  - file: [src/auto-reply/reply/session.ts:483](https://github.com/openclaw/openclaw/blob/2fb9c7e3e586044467f360edf8d8f5170f1c60f5/src/auto-reply/reply/session.ts#L483)

- **Bundled session-memory hook still listens only to command events:** The bundled hook metadata declares only `command:new` and `command:reset`, and the handler immediately returns unless `event.type === "command"` with action `new` or `reset`. It has not been migrated to `session_end`/`session_start`.
