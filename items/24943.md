## Summary

Keep open. Current main still does not strip general Markdown before legacy iMessage sends. The active send path converts Markdown tables and strips OpenClaw directive tags, while reply delivery strips assistant-internal scaffolding, but neither applies the existing plain-text `stripMarkdown()` fallback used by comparable channels. The related PRs in the provided GitHub context were closed unmerged, and the latest maintainer-side review also kept this issue open.

## Best Possible Solution

Implement the fallback in the legacy iMessage plugin, most likely in `extensions/imessage/src/send.ts` after table conversion and before directive stripping/RPC send, using the shared `stripMarkdown()` helper or the newer Markdown IR plain-text renderer. Add focused iMessage tests for bold, italic, headers, blockquotes, inline code, syntax-only edge cases, and the reply-delivery path so this does not regress. Keep upstream imsg rich-rendering work separate from OpenClaw's plain-text fallback.
