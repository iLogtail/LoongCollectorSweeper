## Summary

Keep #70643 open. Current main still does not serialize the full read-modify-write path for `openclaw.json`: `writeConfigFile` reads/prepares a snapshot, writes a temp file, and renames it without an advisory lock, while several gateway/CLI paths still build configs from `loadConfig()` or a pre-read snapshot before writing. Existing recovery code can restore invalid configs from last-known-good, but that was already part of the report and does not prevent the concurrent write window. The linked fix PR #70655 proposed the right shape, but it was closed unmerged.

## Best Possible Solution

Keep the issue open and fix it in core by making config writes a real serialized transaction: acquire an advisory lock on the canonical config path before reading/preparing/writing, carry `baseSnapshot` through the exported writer or retry on hash mismatch, migrate stale `loadConfig()` + `writeConfigFile()` call sites to a locked mutation helper, and add a regression test that runs two config-touching writers concurrently and asserts valid JSON plus merged intended changes.
