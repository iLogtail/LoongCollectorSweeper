## Summary

Keep open. Current main has safer config writes than the report describes: OpenClaw-owned writes validate the payload, reject destructive clobbers, and commit through a temp file plus rename. But the requested interprocess serialization/linearized read-modify-write is not implemented: the config write path and mutation helper do not take the existing advisory file lock, and the related fix PR #70655 was closed unmerged. That leaves enough of the reported concurrent-writer class unresolved to avoid conservative auto-close.

## Best Possible Solution

Keep #70643 open and fix it in the config layer, not in each caller. The best path is to serialize the full read/prepare/validate/write transaction for `openclaw.json` with the existing advisory file-lock primitive, or add equivalent conflict-detect-and-retry behavior keyed by the config hash. Add a regression test that forces two disjoint config writers, such as doctor/service repair and an agent/model default update, to overlap and verifies the final file stays valid and preserves both changes without relying on auto-restore.
