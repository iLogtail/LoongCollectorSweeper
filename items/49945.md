## Summary

Keep open. Current `main` still shows the reported TUI `/new` regression: the handler only switches to a fresh `tui-*` session and does not route through `sessions.reset`, and the colocated test still asserts that behavior. This PR is also not a safe auto-close candidate from the other direction, because the review-thread concerns remain credible against current hook consumers: bundled hooks record/archive by `event.sessionKey`, while the PR’s `hookSourceKey` approach would still emit the new `tui-*` key and would also force `sessions.reset` to persist a brand-new empty session entry/file.

## Evidence

- **Current `/new` path skips gateway reset:** On `main`, the TUI `/new` command generates a unique `tui-<uuid>` key, calls `setSession(uniqueKey)`, and never calls `client.resetSession(...)`. That means it does not traverse the hook-visible `sessions.reset` path at all.
