## Summary

Keep open. Current main still matches the reported failure path: interrupt mode aborts an active run, but the aborted run can still build payloads from the last assistant message in the full session snapshot and the reply runner does not suppress aborted-run payload delivery. The linked #50147 PR is directly relevant but remains open/unmerged, so the issue should stay as the canonical bug tracker until a fix lands.

## Best Possible Solution

Keep #50145 open, use #50147 as the most relevant implementation candidate, and land a scoped fix that suppresses aborted-run output before payload delivery. Add regression coverage for an interrupted run with empty `assistantTexts` and a previous assistant in history, plus a reply-runner test that `meta.aborted === true` does not flush or deliver stale payloads.
