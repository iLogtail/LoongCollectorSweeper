## Summary

Keep PR #56391 open. Current main has related tool-name recovery and unknown-tool loop safeguards, but it still preserves unrecoverable blank/whitespace-only live tool-call names rather than replacing them with a safe invalid dispatch value. The linked bug #34129 remains plausibly real, while the PR also has unresolved implementation risks around `_blank` being a dispatchable real tool name and around streamed tool calls whose IDs arrive after an initially blank name.

## Best Possible Solution

Keep #56391 open for maintainer review or replacement. The best fix is a safer blank-tool-name failure path that fails closed without using a dispatchable real tool name, preserves ID-based recovery across streamed deltas, keeps replay sanitization behavior intact, and adds regression tests for blank/no-id calls, whitespace-only calls, `_blank` registered as an allowed tool, delayed ID arrival, and ambiguous ID inference.

## Risks / Open Questions

- Current main may still allow the #34129 blank/no-id live dispatch loop class because blanks are treated as incomplete rather than as a counted unknown tool.
- The PR's literal `_blank` sentinel can collide with a real allowlisted tool name unless the implementation reserves or otherwise makes the sentinel non-dispatchable.
- Mutating streamed blank names to `_blank` too early can make later ID-based recovery impossible when the provider sends the tool-call ID in a later delta.
