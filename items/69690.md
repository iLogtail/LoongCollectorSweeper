## Summary

Keep PR #69690 open. Current main has related gateway-bindable registry reuse, but it does not implement this PR’s specific fix: retaining the active gateway registry’s core gateway method names so helper/runtime loads that omit `coreGatewayHandlers` can still prove cache compatibility. The PR discussion also includes a concrete repro, root-cause explanation, tests, and bot review feedback that was followed by additional commits, so the work is still potentially useful rather than low-signal cleanup.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase it or apply an equivalent current-main patch that persists the active gateway method-name set, uses it only for explicit gateway-bindable compatibility checks, and adds regression coverage for omitted handlers, mismatched method sets, and default-mode non-inheritance.

## Evidence

- **Current main cache key depends on core gateway method names:** `resolvePluginLoadCacheContext` derives sorted `coreGatewayMethodNames` from `options.coreGatewayHandlers` and includes them in the loader cache key. A helper load that omits `coreGatewayHandlers` therefore computes a different key from a gateway startup load that supplied handlers.
