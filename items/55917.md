## Summary

Keep open. Current `main` still degrades some Telegram inbound documents to placeholder-only delivery instead of a saved attachment, so there is no strong basis to close this as fixed or obsolete. The most likely trigger for the reporter's 22.9 MB example is Telegram Bot API's own 20 MB `getFile` download cap, but OpenClaw currently handles that by skipping the attachment and leaving placeholder/body fallback behavior in place rather than providing a real file path. 

## Evidence

- **Oversize Telegram downloads are intentionally skipped:** `resolveTelegramFileWithRetry` treats Telegram `file is too big` as a permanent error and returns `null`, with a log message saying the attachment is skipped.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:127](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/telegram/src/bot/delivery.resolve-media.ts#L127)

  - file: [extensions/telegram/src/bot/delivery.resolve-media-retry.test.ts:243](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/telegram/src/bot/delivery.resolve-media-retry.test.ts#L243)

  - file: [extensions/telegram/src/bot-message-context.body.ts:160](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/telegram/src/bot-message-context.body.ts#L160)
