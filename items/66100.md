## Summary

Keep open. Current `main` has adjacent Android/mobile pairing fixes, but the specific plain-LAN `ws://` failure mode is not clearly resolved: Android still reloads persisted bootstrap auth for reconnects, and bootstrap handoff tokens are only persisted on loopback/TLS, so a successful cleartext LAN bootstrap can still lose durable auth after restart and fall back to the spent bootstrap token.

## Evidence

- **Android reconnect auth still reloads persisted bootstrap token:** `resolveGatewayConnectAuth()` pulls `prefs.loadGatewayBootstrapToken()` back into the reconnect auth bundle. I did not find an Android success-path equivalent to the iOS bootstrap-token clearing logic, so the spent bootstrap token can remain the reconnect fallback.
