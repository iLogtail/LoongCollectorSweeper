## Summary

Keep open. Current main has relay-backed APNs push, LAN/tailnet/manual Gateway connection paths, and plugin infrastructure, but it does not implement the requested cloud account login, gateway registration/heartbeat, or transparent WebSocket relay for mobile Gateway access. This also should not be auto-closed as ClawHub-only: the remaining request changes mobile connectivity, Gateway auth/trust, and relay transport semantics, so it needs maintainer product/API judgment or a defined plugin transport seam first.

## Best Possible Solution

Keep the issue open for maintainer product/API triage. The best path is to first decide the trust model and ownership boundary: either implement a core cloud relay transport with OAuth, heartbeat, mobile discovery, and Gateway protocol bridging, or define a narrow plugin API that lets a relay plugin register/login, maintain heartbeat, and attach relayed mobile WebSocket sessions without weakening existing Gateway auth and device approval.

## Risks / Open Questions

- A plugin-only implementation could accidentally bypass or duplicate Gateway WebSocket handshake, device approval, TLS/origin, and operator-scope semantics unless core exposes an explicit relay transport contract.
- The existing APNs relay may make the word “relay” look implemented, but it solves push delivery after pairing, not remote Gateway connectivity or cloud discovery.
