## Summary

Keep #67884 open. Current main does not implement the PR's `normalizeMessagingConversationId` change, and the PR is a focused bug-fix attempt rather than cleanup noise. However, the current diff has a real unresolved blocker: it strips `user:`/`channel:`/`chat:` from message hook contexts while inbound-claim resolver paths and Discord's production conversation identity intentionally preserve typed prefixes for binding/correlation semantics.

## Best Possible Solution

Keep the PR open for maintainer/contributor rework. The safest direction is a shared conversation-id contract that preserves semantic typed ids (`user:`, `channel:`) for resolver-backed channels, normalizes only provider/channel wrapper prefixes where the channel contract says they are aliases, and adds regression tests covering Discord DM/channel bindings plus the Telegram/Slack-style cases the PR is trying to fix.

## Risks / Open Questions

- Merging the PR as written could make `message_received`/`message_sent` hook `ctx.conversationId` differ from inbound-claim and binding keys for channels that use typed conversation ids, especially Discord DMs versus channels.
- The underlying mismatch the PR is trying to address may still exist for generic/no-resolver channel paths, so closing the PR as cleanup would lose a plausible focused fix thread.
