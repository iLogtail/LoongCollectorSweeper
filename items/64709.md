## Summary

Keep open. Current main still hardcodes UTF-8 decoding in the affected child-process paths, the user-facing FAQ still treats garbled Windows Chinese exec output as an active workaround issue, and PR #64709 has unresolved review concerns including a P1 regression risk. Related PR #50885 overlaps, but it is narrower and uses a different approach, so it is not a strong canonical superseder for conservative cleanup.

## Best Possible Solution

Keep PR #64709 open for maintainer review. The best path is to resolve the P1 encoding-regression concern, decide how this should coordinate with #50885 and #30640, then land a tested fix that preserves UTF-8-producing command output while fixing Windows console-codepage output across the intended process surfaces.
