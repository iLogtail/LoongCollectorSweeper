## Summary

Keep open. Current main still writes every `.clobbered.*` forensic snapshot attempt without a per-path cap in both config I/O implementations, and there is no current regression test or documentation showing the requested 32-file bound. The PR is a focused core config hardening change with concrete incident context; it is not implemented on main and is not safely superseded by the related in-process dedupe work.

## Best Possible Solution

Keep #65649 open and use it as the focused vehicle for the core config hardening. The best fix is to add a per-config-path cap for `.clobbered.*` artifacts across both async and sync clobber persistence paths, keep audit logging intact, emit only one cap warning per path, and add regression tests that cover both `maybeRecoverSuspiciousConfigRead` and observe/last-known-good paths. Before merge, address the review concern about concurrent async count/write overshoot or explicitly narrow the guarantee.
