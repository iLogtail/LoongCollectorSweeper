## Summary

Keep open. The issue is recent, concrete, and tracks core interruptibility semantics that current main does not yet provide as a unified guarantee. Main still has scattered abort/queue primitives, but no run-generation registry, no run-current stale-output fence in the reply delivery path, and no pre-tool generation gate. The linked PR #70363 is useful prior work, but it is an open draft and explicitly partial rather than a superseding canonical tracker.

## Best Possible Solution

Keep #70319 open as the canonical core interruptibility issue. Review or split #70363 into focused PRs: add the session/run generation primitive, wire stale-output fences across reply delivery/typing/tool-result/progress emission sites, add a pre-tool abort/generation gate in the Pi tool path, and make new user messages invalidate/cancel the active run with targeted tests.
