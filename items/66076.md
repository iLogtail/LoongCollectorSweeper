## Summary

Keep open. Current `main` contains a partial #66019 mitigation that clears `nextRunAtMs` to avoid the immediate refire loop, but it does not implement this PR's core change of promoting silent `undefined` cron next-run results into schedule errors. That means repeated silent failures still do not accumulate toward `MAX_SCHEDULE_ERRORS` or auto-disable.

## Evidence

- **Silent `undefined` still is not treated as a schedule error on current `main`:** `computeNextRunAtMs` can still return `undefined` when Croner yields no next run, but `applyJobResult` only calls `recordScheduleComputeError(...)` in the `catch` path. In the silent-`undefined` path it routes through `resolveCronNextRunWithLowerBound(...)`, which logs and returns `undefined` without incrementing `scheduleErrorCount`.
  - file: [src/cron/schedule.ts:105](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/cron/schedule.ts#L105)
