## Summary

Current main still hardcodes every Gateway-backed TUI connection's display name to `openclaw-tui`. The `openclaw tui` CLI, `TuiOptions`, and TUI docs expose no `--client-name`/`--username` equivalent. Gateway already supports an optional client display name, so this remains a focused TUI CLI plumbing request rather than implemented, duplicate, or ClawHub-scoped work.

## Best Possible Solution

Add a focused TUI option such as `openclaw tui --client-name <name>` or `--client-display-name <name>`, carry it through `TuiOptions` and the Gateway-backed TUI connection path, pass a trimmed non-empty value as `GatewayClientOptions.clientDisplayName`, and update TUI docs plus unit coverage around the constructed Gateway client options. Keep `GATEWAY_CLIENT_NAMES.TUI` as the protocol client id.
