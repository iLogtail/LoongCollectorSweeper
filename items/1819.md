## Summary

本 PR 修复了 RFC3164 syslog 解析器中 TAG 字段未限制长度导致的潜在数组越界问题。作者在生产环境中实际触发过该越界异常，修改内容符合 RFC3164 Section 4.1.3 规范（TAG 不得超过 32 字符）。当前 main 分支未见相同修复，且该 PR 提供了明确的边界保护逻辑，属于有价值的真实缺陷修复，应保持开启并进入审查流程。

## Best Possible Solution

建议维护者进行代码审查。需重点确认两点：1. external/github.com/jeromer/syslogparser 目录的依赖管理策略，若为项目内拷贝可直接合并，若已使用 Go Modules 则应引导作者将修复提交至上游仓库或通过 replace 机制引入；2. 解析逻辑中对空格、括号、冒号的处理顺序有所调整，合并前需补充单元测试覆盖 TAG 长度恰好为 32、超过 32、缺失或含特殊字符等边界场景，确保不破坏现有 syslog 采集兼容性。

## Evidence

- **PR Diff:** 在 parseTag() 中引入 math.Min 限制循环上限为 cursor+32，防止越界读取 buff，并调整了终止符判断逻辑。
  - file: [external/github.com/jeromer/syslogparser/rfc3164/rfc3164.go:214](https://github.com/alibaba/loongcollector/blob/1afae850cdcc1ea5882525a0f229022d23565104/external/github.com/jeromer/syslogparser/rfc3164/rfc3164.go#L214)
  - command: `git diff main...bug/tag_litmit`
  - sha: [1afae850cdcc](https://github.com/alibaba/loongcollector/commit/1afae850cdcc1ea5882525a0f229022d23565104)
- **RFC 3164 Specification:** RFC 3164 Section 4.1.3 明确规定：The TAG is a string of ABNF alphanumeric characters that MUST NOT exceed 32 characters.
- **Author Production Report:** 作者评论指出在生产过程中发现解析 tag 时会发生预期之外的数组越界，本 PR 为针对性修复。

## Risks / Open Questions

- 修改位于 external/ 目录，需确认该第三方库是项目内静态拷贝还是应通过 Go Modules 同步。若直接合并，后续上游更新或依赖重构可能覆盖此修复。
- parseTag 的终止条件与游标推进逻辑发生变更（如对空格直接 break，对括号/冒号使用 enough 标志继续），可能影响非标准 syslog 格式的解析兼容性，需测试验证。
