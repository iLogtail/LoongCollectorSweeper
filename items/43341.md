## Summary

Keep open. Current main still does not wire the bundled TUI to the gateway session transcript subscription path needed for real-time messages from Telegram or other channels. The related #43918 fix addressed the opposite-direction routing problem for TUI/UI sends with deliver enabled; it did not implement TUI live inbound session.message handling. The commenter’s standalone gateway-WS TUI is relevant prior art, but it does not make the bundled OpenClaw TUI issue implemented.

## Evidence

- **Bundled TUI ignores session.message events:** The TUI gateway event handler only dispatches chat, chat.side_result, and agent events. There is no branch for session.message or sessions.changed, so subscribed transcript updates would not currently update the TUI chat log.
  - file: [src/tui/tui.ts:1054](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/src/tui/tui.ts#L1054)

  - sha: [e640c0a95fcb](https://github.com/openclaw/openclaw/commit/e640c0a95fcbd06a1311f18fc635518d011d45da)
- **No TUI subscription call exists:** A focused search found no use of sessions.subscribe, sessions.messages.subscribe, or session.message under src/tui, which means the bundled TUI is not registering for the gateway transcript events described by the current protocol.
