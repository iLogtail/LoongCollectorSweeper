## Summary

Keep this issue open. Current main has per-agent isolation in the default memory-core SQLite/QMD paths, but the explicitly selectable memory-lancedb plugin still uses one configured LanceDB database/table without a partition field, per-agent dbPath expansion, or filtered searches/stores.

## Best Possible Solution

Keep the issue open and scope the fix to extensions/memory-lancedb. The best implementation is to make memory-lancedb partition-aware when it owns the memory slot: either derive/cache MemoryDB instances per agent/session partition using plugin tool/hook context, or add a partition column and enforce filtered store/search/delete paths. Add regression coverage for two agents or partitions proving memory_store/auto-capture results from one partition are not returned to another, and update plugin docs/config guidance. In the meantime, docs can recommend memory-core/QMD for multi-agent deployments because those current backends already use per-agent storage.

## Risks / Open Questions

- Closing as implemented would conflate the default memory-core backend with the explicitly selectable memory-lancedb backend that the report targets.
- Because the report describes cross-user memory leakage, the safest maintainer path is to either patch memory-lancedb isolation or clearly deprecate/disable it for multi-user deployments before closing.
