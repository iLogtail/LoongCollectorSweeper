## Summary

Keep open. Current `main` still hardcodes the non-audio fallback body text to `<media:image>` in Telegram inbound-body handling, so the reported document-without-caption bug is still present and the PR's core fix is not implemented on `main`.

## Evidence

- **Hardcoded image fallback still on main:** `resolveTelegramInboundBody` still emits `<media:image>` whenever there is inbound media, no body text, and the media is not audio. That is the exact mislabeling described in the PR for document attachments without captions.

  - command: `nl -ba extensions/telegram/src/bot-message-context.body.ts | sed -n '150,245p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Document placeholder exists separately:** Telegram media classification already knows documents should be labeled `<media:document>`, which makes the fallback mismatch concrete rather than speculative.
  - file: [extensions/telegram/src/bot/body-helpers.ts:56](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/telegram/src/bot/body-helpers.ts#L56)
