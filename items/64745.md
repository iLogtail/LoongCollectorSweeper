## Summary

Keep open. Current main has useful duplicate-instance and gateway de-duplication guards, but they do not prove the reported macOS launch/reboot loop is fixed. A plausible launch-at-login status-sync loop still exists in current code, and there is no regression test or release/history evidence tying this report to a shipped fix.

## Best Possible Solution

Keep the issue open and fix the macOS launch-at-login lifecycle defensively: hydrate launchAtLogin status without invoking the setter side effect, remove the duplicate onboarding onChange write, avoid bootout/kickstart when merely confirming existing LaunchAgent state, and add Swift tests around startup status hydration plus LaunchAgent write calls. Ask the reporter for crash reports or launchd logs only after the obvious write-on-read path is addressed or ruled out on macOS.
