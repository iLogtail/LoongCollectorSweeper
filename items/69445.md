## Summary

Current main does not implement per-agent status-card field suppression. `/status` and `session_status` still resolve provider usage and render `📊 Usage:` whenever provider/auth conditions allow, and the per-agent config type/schema has no `status`, `hideUsage`, `hideSections`, or equivalent presentation policy. The related #10005 context overlaps, but it tracks broader status-card hiding; this narrower per-agent usage/billing privacy request remains concrete and distinct enough to keep open.

## Best Possible Solution

Keep the issue open and implement this as a core status presentation policy, likely under `agents.list[].status` with optional `agents.defaults.status`. Update config types, strict zod schema, generated config help/docs, `buildStatusText`, `buildStatusMessage`, chat `/status`, and `session_status` tests. The implementation should preserve current output by default and avoid fetching provider usage when usage is hidden for the resolved agent.

## Risks / Open Questions

- A display-only implementation would still fetch provider usage; the best fix should skip usage resolution when the resolved agent suppresses that section.
- The product shape needs maintainer judgment: narrow `hideUsage` versus a future-proof `hideSections` or `visible` allowlist that can cover usage, cache, context, and cost consistently.
