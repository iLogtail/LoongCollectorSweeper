## Summary

Keep open. Current main still has the blocking order that this PR is trying to change: both prompt-side and assistant-side rate-limit/profile failover await auth-profile failure marking before attempting `advanceAuthProfile()`, and that marking path still takes the auth profile store file lock. The PR is stale against the current refactor, but the underlying fix remains relevant rather than obsolete or implemented.

## Best Possible Solution

Update the PR for current main by moving cooldown marking after profile rotation in both `src/agents/pi-embedded-runner/run.ts` and `src/agents/pi-embedded-runner/run/assistant-failover.ts`, make deferred marking handle rejections explicitly, preserve timeout no-marking semantics and the `failedProfileId` log guard, then add focused tests proving rate-limit rotation calls `advanceAuthProfile()` before a slow marker can block it on both prompt-side and assistant-side paths. After that validation, the PR can close #57281.
