## Summary

Keep open. Current `main` has generic opt-in compatibility hooks for string-only Chat Completions payloads and `parallel_tool_calls`, but the bundled NVIDIA provider still does not wire any Kimi-specific compat/defaults, so the reported built-in `nvidia/moonshotai/kimi-k2.5` regression is not clearly fixed on `main`.

## Evidence

- **Bundled NVIDIA catalog still has no Kimi-specific compat/defaults:** `buildNvidiaProvider()` only sets `baseUrl`, `api`, and plain model entries for `moonshotai/kimi-k2.5`; there is no `compat.requiresStringContent`, no tool-schema override, and no model params/defaults.
  - file: [extensions/nvidia/provider-catalog.ts:13](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/nvidia/provider-catalog.ts#L13)
- **NVIDIA docs still claim out-of-box OpenAI-compatible behavior:** The provider docs still say NVIDIA should work "out of the box" with the standard `/v1` completions endpoint, which conflicts with treating this as already fixed only via manual opt-in config.
  - file: [docs/providers/nvidia.md:80](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/docs/providers/nvidia.md#L80)
- **String-content flattening is generic and opt-in:** `buildOpenAICompletionsParams()` only flattens `messages` when `compat.requiresStringContent` is true; there is no automatic NVIDIA/Kimi detection in this path.
  - file: [src/agents/openai-transport-stream.ts:1719](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/agents/openai-transport-stream.ts#L1719)
- **Parallel tool call patch is generic and opt-in:** `applyExtraParamsToAgent()` only injects `parallel_tool_calls` when that param is present in effective extra params; it is not automatically enabled/disabled for NVIDIA Kimi models.
  - file: [src/agents/pi-embedded-runner/extra-params.ts:458](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/agents/pi-embedded-runner/extra-params.ts#L458)
- **Current tests cover configured workarounds, not built-in NVIDIA auto-fix:** The NVIDIA-shaped test case asserts `parallel_tool_calls` only when configured under `agents.defaults.models["nvidia-nim/moonshotai/kimi-k2.5"].params`, and the string-content test uses an unrelated model with explicit `compat.requiresStringContent: true`.
  - file: [src/agents/pi-embedded-runner-extraparams.test.ts:728](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/agents/pi-embedded-runner-extraparams.test.ts#L728)
- **Repo search found no NVIDIA plugin hook implementing the reported Kimi fix:** Focused search only found the generic hooks/tests in agent runtime; it did not find any NVIDIA plugin `normalizeToolSchemas`, `contributeResolvedModelCompat`, or similar Kimi-specific wiring.
  - command: `rg -n "clean-for-kimi|normalizeToolSchemas|inspectToolSchemas|contributeResolvedModelCompat|requiresStringContent|parallel_tool_calls" extensions/nvidia src/agents src/plugins -g '!node_modules'`

## Risks / Open Questions

- A manual per-model workaround may now exist via `compat.requiresStringContent` and `parallel_tool_calls`, but that is not the same as the built-in NVIDIA provider being fixed.
- No live NVIDIA NIM repro was possible in this checkout-only review, so the exact remaining failure mode on current `main` is inferred from code paths rather than network execution.
