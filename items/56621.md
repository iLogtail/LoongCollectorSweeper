## Summary

Keep #56621 open. Current main still exposes `openclaw pairing list` for pending requests and `openclaw pairing approve`, but there is no `pairing list-approved` or `pairing revoke` CLI/doc surface. The underlying pairing store can read and remove pairing allowFrom entries, so this is feasible product work rather than obsolete or out-of-repo. Related #56619 is about approval notification and appears separately addressed by `--notify`; it does not supersede approved-sender management.

## Best Possible Solution

Keep the issue open and implement the missing DM pairing management surface in core CLI/docs: add `openclaw pairing list-approved <channel> [--account] [--json]` that merges config allowFrom with account-scoped pairing-store approvals and shows source, and add `openclaw pairing revoke <channel> <sender> [--account]` using `removeChannelAllowFromStoreEntry` for pairing-store entries only. Then update `channels status --probe` or the gateway status snapshot if maintainers want `allow:` to reflect the merged config plus pairing-store view.

## Risks / Open Questions

- Revoke semantics need to be explicit: remove only pairing-store approvals, not configured `channels.<channel>.allowFrom` entries.
- Listing should handle account-scoped default-vs-legacy allowFrom files consistently and label config-derived entries as non-revocable.
- The pairing allowFrom store currently stores sender ids only, so an `Approved` timestamp column may require a store format extension or omission/backfill behavior.
