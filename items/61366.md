## Summary

Keep #61366 open for maintainer judgment rather than auto-closing it. Current main already implements the safer pinned-channel-registry part of the PR, but it does not implement the PR's subagent-specific validation bypass: `agent` still rejects any `channel`/`replyChannel` not known to `isGatewayMessageChannel()` before considering `deliver: false` or `lane: "subagent"`. The PR also is not mergeable as submitted because its head diff duplicated an import, so the useful remaining question is whether OpenClaw should accept metadata-only subagent spawn channels that are not registered through the channel registry, or require plugin authors to use the documented `registerChannel`/channel-entry path.

## Best Possible Solution

Keep the PR open for explicit maintainer/product judgment. The safe path is either to update the PR by removing the duplicate import and adding a regression test for the subagent-spawn channel-metadata case, or to close it manually after confirming that third-party channel plugins must always use the documented `defineChannelPluginEntry`/`api.registerChannel` surface and that unregistered transport-only channels are unsupported.
