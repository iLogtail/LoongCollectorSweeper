## Summary

Keep #71058 open. Current main still models Microsoft Teams as one configured bot identity per gateway: a single `channels.msteams` object, one credential resolution path, one SDK adapter/JWT validator, and one gateway monitor. The request is concrete, recent, unprotected by maintainer labels, and not covered by the provided related items or discussion.

## Best Possible Solution

Keep the issue open as a concrete Teams plugin enhancement. The likely implementation should add a backward-compatible multi-account Teams config shape, preferably aligned with the existing `accounts`/`defaultAccount` channel pattern unless maintainers choose the proposed array form, then make runtime, inbound routing, outbound sends/actions, setup, docs, schema/types, and tests account-aware.

## Risks / Open Questions

- A Teams multi-bot implementation must account for per-bot webhook routing, Bot Framework token audience/JWT validation, outbound account selection, setup UX, and session/account IDs.
- The proposed `agentBinding` field may need product design to align with existing `accounts`/`defaultAccount` and agent account-context patterns used by other channels.
