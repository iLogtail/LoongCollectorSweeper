## Summary

#71058 should remain open. Current main still exposes Microsoft Teams as a single top-level `channels.msteams` bot configuration, and the Teams plugin resolves only one default account. The request is concrete, recent, and not duplicated by the provided related items. This is also not a good ClawHub close: Teams already ships as a bundled OpenClaw plugin, and core already has generic multi-account channel machinery that the Teams plugin can adopt.

## Best Possible Solution

Keep the issue open and implement this in the bundled `msteams` plugin using the existing channel multi-account conventions: add backward-compatible `channels.msteams.accounts.<accountId>` and `defaultAccount`, resolve per-account credentials/policies, start or route each Teams bot account safely, wire account-scoped status/setup/secrets/docs, and use existing agent channel-account bindings for the per-agent identity mapping.

## Risks / Open Questions

- Teams multi-bot support needs careful webhook and JWT-audience routing. Multiple accounts on the same port will need distinct paths or a shared dispatcher that validates each bot identity correctly.
- The issue proposes an array shape, but existing OpenClaw channel-account tooling uses `channels.<channel>.accounts.<accountId>` plus `defaultAccount`; implementing the literal array would likely bypass current account/status/setup conventions.
