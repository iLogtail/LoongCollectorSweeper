## Summary

Keep open. Current `main` does not implement the requested Mattermost WebSocket ping/pong keepalive. The only added health check is a periodic REST poll of the bot account `update_at`, which detects account disable/enable changes but does not detect a silently dropped WebSocket when the server remains otherwise reachable.

## Evidence

- **WebSocket interface has no ping/pong surface:** `MattermostWebSocketLike` only exposes `open`, `message`, `close`, `error`, `send`, `close`, and `terminate`. There is no `ping`, `pong`, or `once("pong")` handling in the current monitor implementation.
  - file: [extensions/mattermost/src/mattermost/monitor-websocket.ts:33](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/extensions/mattermost/src/mattermost/monitor-websocket.ts#L33)
  - command: `nl -ba extensions/mattermost/src/mattermost/monitor-websocket.ts | sed -n '25,230p'`
  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **Current health check is update_at polling, not transport keepalive:** `createMattermostConnectOnce` schedules `runHealthCheck()` around `getBotUpdateAt()` and only calls `ws.terminate()` when the bot account `update_at` value changes. Errors are logged and polling continues. This does not detect a silent WebSocket drop where the account metadata stays unchanged.

## Risks / Open Questions

- A silently dropped WebSocket can still leave OpenClaw believing the Mattermost monitor is connected if `close`/`error` never fires and `fetchMattermostMe()` continues to succeed.
- The existing health check appears targeted at a different failure mode: bot account disable/enable cycles, not NAT/proxy idle expiry or half-open TCP/WebSocket sessions.
