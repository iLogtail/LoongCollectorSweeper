## Summary

Keep open. Current `main` still hardcodes the WhatsApp watchdog timeout/check interval, still forces a 499 reconnect when that timer expires, and still does not expose `messageTimeoutMs` or `watchdogCheckMs` through the WhatsApp config schema/types/docs or the gateway startup wiring.

## Evidence

- **Watchdog values are still hardcoded in current main:** `monitorWebChannel()` still sets `messageTimeoutMs` to `30 * 60 * 1000` and `watchdogCheckMs` to `60 * 1000` from internal tuning defaults, not config.
  - file: [extensions/whatsapp/src/auto-reply/monitor.ts:183](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/whatsapp/src/auto-reply/monitor.ts#L183)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/monitor.ts | sed -n '175,205p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Timeout still forces the reported reconnect behavior:** The connection controller watchdog closes the connection with status `499` once `staleForMs` exceeds `messageTimeoutMs`, matching the issue's restart loop description.
  - file: [extensions/whatsapp/src/connection-controller.ts:563](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/whatsapp/src/connection-controller.ts#L563)

- **Gateway startup does not wire config into those tuning fields:** WhatsApp account startup passes only `statusSink` and `accountId` into `monitorWebChannel()`. It does not pass `messageTimeoutMs` or `watchdogCheckMs` from config.
  - file: [extensions/whatsapp/src/channel.ts:296](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/whatsapp/src/channel.ts#L296)
