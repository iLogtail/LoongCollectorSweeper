## Summary

Keep #52521 open. Current main still uses delivery `storeEntry.channel` as the `sessions.usage` row and `aggregates.byChannel` key before falling back to `origin.provider`, so the linked #52436 behavior is not implemented on main. The PR is a plausible focused bug fix, but the review discussion also identifies a real design risk: `origin` is a mutable session-level snapshot while usage totals are whole-session/date-range totals, so the final fix likely needs a refined attribution path rather than conservative cleanup closure.

## Best Possible Solution

Keep the PR open, but request a maintainer-reviewed revision: either compute an explicit aggregate attribution key while preserving delivery metadata on session rows, or add a more accurate per-turn/per-log origin attribution model for mixed-origin sessions. The final patch should include a regression test for #52436 and a test or documented decision for mixed-origin sessions.

## Risks / Open Questions

- Closing this PR would remove the only visible implementation attempt for #52436 while current main still has the reported attribution behavior.
- Merging the PR as-is may misattribute mixed-origin sessions because it uses a mutable session-level origin snapshot for whole-session totals.
- Changing `sessions[].channel`/`chatType` also changes usage filtering, details badges, and CSV export behavior, so aggregate-only and row-field semantics need explicit maintainer agreement.
