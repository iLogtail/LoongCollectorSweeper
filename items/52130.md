## Summary

Keep open. Current main has partial hardening for the surrounding restart/diagnostic story, but the primary reported crash-loop trigger is still present: `channels.telegram.retry.jitter` is still schema-validated as a number-only field, with no boolean coercion or regression coverage on main. The related PR #52137 proposed exactly that coercion but is closed unmerged in the provided GitHub context, so this issue is not implemented or superseded.

## Best Possible Solution

Keep #52130 open, or split it into focused follow-ups. The highest-value fix is to add schema-level boolean coercion for shared `RetryConfigSchema.jitter` (`true` to the default jitter factor, `false` to `0`) plus regression coverage for `channels.telegram.retry.jitter: true/false` and any shared consumers. Then separately verify whether doctor SecretRef status and CDP restart isolation need narrower issues after the existing service/browser hardening.
