## Summary

Keep open. Current main still implements long-context handling as in-place compaction, memory flush, and manual/daily/idle session reset. I found no automatic context-threshold session rollover, no `session.autoSplit`/`triggerPercent` configuration, and no current plugin/context-engine API that rotates a session binding to a new `sessionId` while injecting continuity context.

## Best Possible Solution

Keep the issue open for maintainer product/design work. The likely path is a core session lifecycle feature, or first an explicit plugin/context-engine API, that waits for active tool calls to finish, performs memory flush/compaction summary, rotates the session key to a new `sessionId`, injects continuity/startup context, routes subsequent messages to the new session, and documents/configures the threshold behavior.
