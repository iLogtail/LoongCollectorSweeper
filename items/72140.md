## Summary

Keep open. Current main still has a credible stale post-compaction status path: compaction can persist a fresh small `tokensAfter`, but `/status` then backfills from transcript usage whenever the transcript prompt candidate is larger than the session total. Since transcript usage scans the full untruncated transcript by default, an older pre-compaction usage snapshot can still win and drive the shared Context line.

## Best Possible Solution

Keep the issue open and fix the shared status accounting path. The narrow fix should make fresh post-compaction `SessionEntry.totalTokens` authoritative over older transcript fallback, or teach transcript usage fallback to respect the latest compaction boundary/checkpoint. Add a regression test where `totalTokensFresh: true` and a small `tokensAfter` value are present while the transcript still contains a larger pre-compaction usage snapshot; `/status` should show the small post-compaction Context value and keep the pre-compaction peak only as compaction metadata if exposed.
