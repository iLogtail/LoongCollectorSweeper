## Summary

Current main has partial protections, but the specific stale post-compaction `/status` path is still plausible. Manual compaction writes `tokensAfter` into the session entry, yet the shared status formatter can replace that smaller fresh value with a larger transcript-derived prompt snapshot, which is exactly the reported pre-compaction-size symptom. Related ClawSweeper reports #44184/#48206 covered sparse or zero status usage fallback, not this post-compaction regression.

## Best Possible Solution

Keep this issue open and fix the shared status usage selection. The likely product/code path is to make `/status` prefer a fresh post-compaction `SessionEntry.totalTokens` over larger transcript fallback usage after compaction, while preserving the existing sparse/zero transcript fallback fixes. A focused regression test should cover `compactionCount > 0`, `totalTokensFresh: true`, small `totalTokens`, and an older larger transcript usage snapshot, plus a formatter update that moves the pre/post compaction delta to the Compactions line when that metadata is available.

## Evidence

After compaction, a smaller fresh `tokensAfter` can therefore be overwritten by larger pre-compaction transcript usage.
