## Summary

Keep open. Current main has meaningful QA harness hardening, including process-tree stop/cleanup helpers, gateway and qa-channel readiness polling, and tests for those helpers. But the core orphan-leak acceptance is not fully covered: QA gateway children are spawned detached, and cleanup depends on normal `runQaSuite`/`gateway.stop()` finally paths. I did not find a suite/gateway-child SIGINT/SIGTERM/exit owner that would reliably reap the detached child if the QA CLI process itself is interrupted or detached. The linked PR #65511 claimed to close this issue, but the provided GitHub context shows it was closed unmerged, so it is not a canonical shipped resolution by itself.

## Best Possible Solution

Keep the issue open and implement a suite-owned lifecycle guard for QA gateway children: register scoped SIGINT/SIGTERM/exit cleanup while a child is running, make cleanup idempotent, add a regression test that simulates interruption and verifies the detached process tree/temp root is reaped, then tighten qa-channel config staging so the suite only writes or enables it after the QA bus/transport readiness contract is satisfied.
