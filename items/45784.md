## Summary

Keep open. Current `main` still treats `OPENCLAW_HOME` as the home root and keeps default config/state under `.openclaw`; this PR changes that contract for every explicit-home user, and the review discussion correctly flags that regression. The nested `~/.openclaw/.openclaw` report may still describe a real edge case, so conservative cleanup should not close this PR.

## Evidence

- **Current main still appends `.openclaw` in `resolveConfigDir`:** On current `main`, `resolveConfigDir` returns `path.join(resolveRequiredHomeDir(env, homedir), ".openclaw")` whenever neither `OPENCLAW_STATE_DIR` nor `OPENCLAW_CONFIG_PATH` is set. That means `OPENCLAW_HOME` changes the base home, not the config dir root itself.
  - file: [src/utils.ts:136](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/utils.ts#L136)
  - command: `sed -n '130,190p' src/utils.ts`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Current path contract explicitly expects `OPENCLAW_HOME/.openclaw`:** `resolveStateDir` builds the default state dir as `newStateDir(effectiveHomedir)`, and tests assert both state and config defaults remain under `OPENCLAW_HOME/.openclaw` and `OPENCLAW_HOME/.openclaw/openclaw.json`. The PR would break those expectations for any non-`.openclaw` `OPENCLAW_HOME` value.
  - file: [src/config/paths.ts:60](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/config/paths.ts#L60)

That is direct evidence the PR's behavior change is broader than the reported bug.

They do not describe `OPENCLAW_HOME` as the config dir itself.
