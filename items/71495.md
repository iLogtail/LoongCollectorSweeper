## Summary

Keep open. The issue is recent, reporter follow-up says it still reproduces on v2026.4.24, and current main has no relevant change after that release beyond a subagent-list sorting tweak. The current slash-command path still builds `/subagents list` entirely from `listControlledSubagentRuns(requesterKey)`, so a registry write/read split or requester/controller key mismatch remains a plausible active product bug rather than an implemented or duplicate cleanup candidate.

## Best Possible Solution

Keep the issue open. The best maintainer path is to add targeted diagnostics/regression coverage around the exact write/read boundary: log or assert `registerSubagentRun` controller/session metadata, `getSubagentRunsSnapshotForRead` size/source, `resolveRequesterSessionKey` output for the affected channel, and parent completion lookup. Then fix the concrete cause, likely in requester/controller session resolution, registry module identity, or persisted snapshot visibility, and add an end-to-end QA scenario for `sessions_spawn` followed by `/subagents list` plus completion delivery.
