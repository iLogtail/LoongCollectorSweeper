## Summary

Current main still appears to route subagent announce follow-up agent calls through the requester session without applying `channels.modelByChannel`. The announce path supplies channel/delivery fields to the gateway, but the agent command model resolver starts from `resolveDefaultModelForAgent()` and only applies stored session overrides or explicit authorized run overrides. The existing status/docs path does know about `resolveChannelModelOverride()`, so the reported status/runtime mismatch is plausible and not covered by the related closed announce-routing reports.

## Best Possible Solution

Keep the issue open and fix the agent ingress model-selection path. The likely implementation belongs in `src/agents/agent-command.ts`: when there is no stored session model override and no explicit authorized run override, resolve `channels.modelByChannel` from the requester session/channel metadata (including thread/group context and parent session key), normalize aliases consistently with existing model selection, honor agent model allowlists, then fall through to stored and explicit priority as documented. Add focused regression coverage for subagent announce delivery plus priority cases: explicit override wins, stored session override wins, channel pin beats agent default, and no channel context preserves default behavior.
