## Summary

Keep open. Current main still only validates cron announce channel IDs on `cron.add`/`cron.update`; it accepts, normalizes, stores, and later gives precedence to explicit `delivery.accountId` and `delivery.failureDestination.accountId` without a configured-account membership check. The PR is a concrete focused bug fix, and the provided review thread shows the author added follow-up commits for the main correctness concerns, so it should get maintainer review/rebase rather than be auto-closed.

## Best Possible Solution

Keep #69562 open and either rebase/review it or reimplement the same narrow gateway fix on main: validate explicit cron announce account IDs for both `delivery.accountId` and `delivery.failureDestination.accountId` against the resolved channel plugin's configured/default account contract, normalize IDs consistently, and skip checks for dynamic/inferred `last` routing where the runtime channel is not statically known. Add focused gateway tests for add/update and failure-destination cases.
