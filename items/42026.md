## Summary

Keep #42026 open. Current main has multi-agent config/routing, low-level runtime selection, hot reload, and some runtime-plan extraction, but it still documents and implements a single Gateway process without a separate `openclaw-runtime`, runtime registration protocol, or per-agent start/stop/restart lifecycle surface. Related #12429 is narrower K8s worker pairing/deployment work, not a canonical replacement for this control-plane/runtime split RFC.

## Best Possible Solution

Keep the RFC open, but narrow it into maintainer-reviewed milestones: Phase 0 per-agent lifecycle commands/RPC and live routing without gateway restart; Phase 1 runtime registration, health, and message protocol; Phase 2 process/container supervisor and secret/workspace isolation. Track #12429 separately for K8s worker pairing because it is a deployment-auth subproblem, not the whole control-plane/runtime split.

## Risks / Open Questions

- The RFC is broad and overlaps existing hot reload, ACP, sandbox, and K8s work; maintainers should probably split implementation into narrower tracked milestones.
- Some immediate pain may be mitigated by current config hot reload and channel reload behavior, so the next step should validate the exact Telegram/drop-session workflow before choosing the Phase 0 scope.
- A true distributed runtime boundary would affect security, secrets, session ownership, tool execution, MCP lifecycle, observability, and backward compatibility, so this needs explicit architecture review rather than automated cleanup.
