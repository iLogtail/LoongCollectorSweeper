## Summary

Keep open. Current main has implemented several major hardening pieces from #53969, including atomic JSON writes, global per-profile OAuth refresh coordination, refresh_token_reused recovery, and safer Codex CLI bootstrap behavior. But the issue’s remaining request is not fully implemented: OAuth auth-profile creation/update paths still use the unlocked upsert helper, and the refresh serialization key is provider+profileId rather than a logical enrollment/account key, so duplicate copies of the same OpenAI Codex enrollment under different profile IDs remain a plausible gap.

## Best Possible Solution

Keep #53969 open but narrow it to the remaining hardening gaps: route OAuth credential creation/update paths through updateAuthProfileStoreWithLock or upsertAuthProfileWithLock, and add duplicate-enrollment detection/repair or a documented single-writer policy for OpenAI Codex OAuth credentials that share the same account/enrollment under different profile IDs or stores. Preserve the already-shipped atomic writes, Codex CLI bootstrap-only behavior, and per-profile refresh locking.
