## Summary

Keep open. Current main has several relevant guardrails, including an outer retry cap, opt-in tool-loop detection, message/idempotency dedupe, model fallback, and compaction failure summaries, but it does not implement the proposal’s central closed-loop self-healing execution design by default. The remaining request is core agent/runtime behavior, not a ClawHub-only plugin idea.

## Best Possible Solution

Keep the issue open as a core runtime design proposal. The best path is a narrow product/design follow-up that builds on existing pieces: make deterministic validation/provider/gateway failures classifyable, add per-turn failed-tool/args/message state outside compacted transcript history, define bounded repair actions and verification, and decide which parts should be default-on versus configurable loopDetection policy.

## Risks / Open Questions

- Closing as implemented would overstate the current guardrails: the shipped loop detector is opt-in and threshold-based, while the proposal asks for default finite self-healing with deterministic-error classification and verification.
- Closing as ClawHub would misclassify this as optional plugin work even though the request targets OpenClaw’s core execution loop, retry policy, gateway/session behavior, and compaction safety.
