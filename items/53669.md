## Summary

Keep #53669 open. Current main has strong scheduler support for cron expressions evaluated in the supplied IANA timezone, including Asia/Shanghai regression coverage, but I did not find a current fix for the reported CronCreate/model-side mismatch where the model converts the hour to UTC while still sending `tz: "Asia/Shanghai"`. The cron tool schema/description still only says `expr` is a cron expression and `tz` is an optional timezone; it does not explicitly tell models to keep the cron expression in the wall-clock time of `tz`, and there is no test covering the natural-language case from the issue.

## Best Possible Solution

Keep the issue open and fix it at the cron tool contract layer: explicitly document in `src/agents/tools/cron-tool.ts` that cron `expr` is interpreted in `schedule.tz` local wall-clock time, add regression coverage for the tool description/schema, and consider a higher-level live or harness test for prompts such as "Remind me every day at 6pm Shanghai time" producing `expr: "0 18 * * *", tz: "Asia/Shanghai"`.
