## Summary

Keep #39428 open. Current main still routes Discord inbound traffic through the shared resolver, which only evaluates explicit bindings and then falls back to the configured default agent. I did not find an accountId-to-agentId name-match fallback on main, and the linked fix PR #39485 is still open/unmerged in the provided GitHub context.

## Best Possible Solution

Keep the issue open and use it as the tracker for landing or revising PR #39485. The likely fix belongs in the shared routing path or a documented Discord-specific route fallback with regression tests for non-default Discord accountId matching, default-account guard behavior, and explicit-binding precedence.
