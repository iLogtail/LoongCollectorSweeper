## Summary

Current main does not implement the requested first-class pre-container Docker bind resolver. Docker binds are still resolved from static global/per-agent config, validated as static entries, and appended during container creation; existing plugin hooks can affect model/prompt/tool/message behavior but cannot mutate sandbox Docker mounts before `resolveSandboxContext()` creates the backend. The related issues in the provided context are closed stale/not-planned or cover different sandbox/config-injection problems, so they do not supersede this request.

## Best Possible Solution

Keep this open for maintainer design. The best path is either a core Docker pre-create bind resolver with strict validation, timeout/nonzero-exit handling, clear merge/replace semantics, and container hash/recreate tests, or an explicit documented plugin sandbox-backend/pre-create API that receives enough session/channel/user context to implement least-privilege mounts outside core.

## Risks / Open Questions

- Dynamic host-side bind resolution is a security-sensitive sandbox API; maintainers need to define validation, timeout/error handling, config-hash/container reuse behavior, auditability, and what channel/user metadata is safe to expose.
- A custom sandbox backend plugin could cover some deployments, but current documented plugin capabilities do not present this as the recommended path and the backend factory does not receive the full session context requested by the issue.
