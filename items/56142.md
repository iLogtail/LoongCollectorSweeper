## Summary

Keep open. Current `main` still routes Slack `block_actions` through queued `Slack interaction: ...` system events rather than the normal inbound message handler, and I did not find any Slack-side wake request paired with that enqueue path.

## Evidence

- **Message events and interaction events still use different dispatch paths:** `registerSlackMonitorEvents()` passes `handleSlackMessage` only to `registerSlackMessageEvents()`. `registerSlackInteractionEvents()` still receives only `ctx` and `trackEvent`.
  - file: [extensions/slack/src/monitor/events.ts:17](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/slack/src/monitor/events.ts#L17)
  - command: `sed -n '1,220p' extensions/slack/src/monitor/events.ts`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Block actions still enqueue a system event and stop there:** `handleSlackBlockAction()` falls through to `enqueueSlackBlockActionEvent()`, which resolves a session key and calls `enqueueSystemEvent(...)`. There is no `handleSlackMessage(...)` call and no `requestHeartbeatNow(...)` in this path.
  - file: [extensions/slack/src/monitor/events/interactions.block-actions.ts:630](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/slack/src/monitor/events/interactions.block-actions.ts#L630)
  - command: `nl -ba extensions/slack/src/monitor/events/interactions.block-actions.ts | sed -n '620,860p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Queued system events are defined as next-prompt context, not immediate inbound delivery:** `src/infra/system-events.ts` describes system events as an in-memory queue that is prefixed to the next prompt, and `enqueueSystemEvent()` only appends to that queue.
  - file: [src/infra/system-events.ts:1](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/infra/system-events.ts#L1)
  - command: `nl -ba src/infra/system-events.ts | sed -n '1,120p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Main-branch docs and tests still codify the system-event behavior:** Slack docs say interactive replies route clicks back as Slack interaction system events, and the interaction tests assert `enqueueSystemEvent` is called for a block action.
  - file: [docs/channels/slack.md:708](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/docs/channels/slack.md#L708)
  - command: `nl -ba docs/channels/slack.md | sed -n '700,790p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Other wake-sensitive paths explicitly request a heartbeat after enqueue:** For comparison, `server-node-events.ts` calls `requestHeartbeatNow(...)` immediately after `enqueueSystemEvent(...)` when an immediate wake is required. I found no analogous wake in the Slack interaction code.
  - file: [src/gateway/server-node-events.ts:687](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/gateway/server-node-events.ts#L687)
  - command: `nl -ba src/gateway/server-node-events.ts | sed -n '680,705p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)

## Risks / Open Questions

- This was a read-only code review; I did not run a live Slack reproduction against credentials or a configured workspace.
- A separate fast heartbeat interval could eventually drain the queued event, but the current Slack interaction path still lacks explicit immediate inbound dispatch or wake behavior.
