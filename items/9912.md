## Summary

Keep #9912 open. Current main still does not provide a global or per-agent hard budget for agent model-tool turns, total tool calls, or a timeout-after-first-tool-call that forces a final response from the last useful tool result. Existing `tools.loopDetection`, general agent timeouts, and the accepted-but-ignored OpenResponses `max_tool_calls` field are related mitigations, but none implement the requested config contract.

## Best Possible Solution

Keep #9912 open as the canonical global/per-agent hard agent iteration budget request. A complete implementation should add typed config/schema/help/docs for max model-tool turns and/or total tool calls, define timeout-after-first-tool-call semantics, enforce the limits in the embedded/tool-call runtime, cover behavior with focused tests, and keep #13271 separate for per-channel budgets and auto-delegation.

## Risks / Open Questions

- Existing `tools.loopDetection` can mitigate some runaway loops, so the eventual fix should clearly document how hard budgets interact with pattern detection and defaults.
- The accepted-but-ignored OpenResponses `max_tool_calls` field may be a useful additional enforcement surface, but it is currently not wired into agent runtime behavior.
- Maintainers still need to choose product semantics for limit exhaustion: block the next tool call, inject a warning, force a final model response, return the last tool result directly, or combine those behaviors.
