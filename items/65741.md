## Summary

Keep open. The PR is a focused Telegram reliability fix and current main does not contain the requested changes: `normalizeChatType()` still does not map `supergroup` to `group`, Telegram group allowlist rejection still happens before message processing without checking configured bindings, and channel hot reload still restarts Telegram immediately after stop. The PR discussion also shows the one bot-review concern was addressed in the PR by scoping the delay to Telegram.

## Best Possible Solution

Keep the PR open, but require a rebase/adaptation to current main. If maintainers accept binding-based group authorization, implement it in the current Telegram group access path with tests for supergroup binding routing, allowlisted sender/group behavior, and a Telegram-only hot-reload restart delay. If maintainers reject the auth semantic change, split out the safer supergroup normalization and Telegram restart-delay fixes or close with that product rationale.
