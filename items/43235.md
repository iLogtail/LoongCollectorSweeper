## Summary

Keep #43235 open. Current main still queues inbound agent runs through a per-session lane and then the shared global `main` lane, with only `agents.defaults.maxConcurrent` controlling that process-wide lane. Manual RPC custom-lane support exists, but there is no configured per-agent or per-agent-group lane policy for Feishu/channel ingress. Related #42686 was already closed in favor of this issue, while #43673 and #49183 are broader open multi-agent/RBAC discussions rather than a shipped or canonical replacement.

## Best Possible Solution

Keep #43235 open as the canonical focused tracker. The best implementation path is a core queue/config change: add typed and documented per-agent or per-agent-group lane policy, validate it in config schema, derive the inbound global lane from the resolved agent for channel dispatch, apply lane concurrency on gateway startup/reload, preserve session-lane ordering, and add generic queue plus Feishu broadcast/@all regression coverage. Manual Gateway RPC `lane` support should remain as a lower-level escape hatch, not the solution for normal channel ingress.
