## Summary

Keep PR #19328 open. Current main at 9a529ca78bfa0e257f662f0e70ab8c681caa6086 still has the stale cached-entry writeback pattern the PR targets: the agent handler builds a session patch from loadSessionEntry() data, including modelOverride/providerOverride and other cache-derived fields, then merges that patch over a fresh updateSessionStore() read where patch fields win. The related open #70575 tracks narrower delivery/channel leftovers, not a replacement for the modelOverride/providerOverride fix.

## Best Possible Solution

Keep PR #19328 open or land an equivalent focused fix. The best implementation is to stop putting cache-derived session fields into nextEntryPatch before merging over the fresh store, preserve fresh store-only modelOverride/providerOverride and related session fields, and add regression tests that simulate stale loadSessionEntry data against a fresh updateSessionStore store. Track the narrower delivery/channel residuals in #70575 separately.
