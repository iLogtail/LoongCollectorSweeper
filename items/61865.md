## Summary

Current main still has the reported `/status` stale-total failure mode. The channel status path defaults to reading transcript usage, the status renderer initializes context usage from `sessionEntry.totalTokens` or `inputTokens + outputTokens` without checking `totalTokensFresh`, and it can replace that value with a larger transcript-derived prompt total. Compaction updates only mark token totals fresh when `tokensAfter` is present, while existing tests explicitly preserve the old total when compaction lacks `tokensAfter`. Related #45268 is still open and adjacent, but #61865 captures a narrower post-compaction status-display case that is not fully superseded.

## Best Possible Solution

Keep #61865 open. The best fix is to make the `/status` renderer use the same freshness contract as other session consumers: only display a current context total from a fresh post-compaction snapshot, avoid `input+output` and whole-transcript fallback as context truth when `totalTokensFresh === false` or when a compacted session lacks a trustworthy post-compaction snapshot, and show unknown instead. Add a focused regression test in `src/auto-reply/status.test.ts` or `src/auto-reply/reply/commands-status.test.ts` for a compacted session with stale `totalTokens` plus a larger transcript estimate.
