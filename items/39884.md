## Summary

Current main already ships first-class browser automation as the bundled `browser` plugin, enabled by default, with Playwright-backed runtime support. The issue's stated current-state assumptions are obsolete; the remaining manual Docker Chromium install step is environment-specific setup, not absence of a default browser capability.

## Close Comment

Closing from a Codex review: current `main` already implements this as shipped product behavior. OpenClaw now includes a bundled `browser` plugin (`extensions/browser/index.ts:9`) with Playwright runtime dependency (`extensions/browser/package.json:7`), and the docs describe it as the default browser tool/plugin surface, enabled by default (`docs/tools/browser.md:46`, `docs/tools/plugin.md:119`). The managed `openclaw` browser profile, CLI, gateway method, and agent tool are already documented as the standard path, and this is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The only remaining manual step is Docker-specific Chromium browser-binary installation (`docs/install/docker.md:279`), which is environment setup rather than lack of first-class browser integration. If there is still a missing browser workflow beyond the shipped default browser plugin, please open a new issue with the exact gap and reproduction path.
