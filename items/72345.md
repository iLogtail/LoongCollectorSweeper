## Summary

Keep #72345 open. Current main still wires `openclaw voicecall call` and the related voicecall CLI commands to a process-local voice-call runtime, and runtime creation still starts a webhook server on the configured bind/port. The existing `voicecall.*` gateway RPC methods are present, but the CLI does not delegate to them or expose a gateway/remote option, so the reported gateway-vs-CLI `EADDRINUSE` path remains actionable.

## Best Possible Solution

Keep #72345 open as the active tracker. Implement a gateway-aware voicecall CLI path that routes `call`/`start`/`continue`/`speak`/`dtmf`/`end`/`status` through the existing `voicecall.*` gateway RPCs when a running gateway is selected or explicitly requested, while preserving standalone local-runtime behavior. Add regression coverage proving gateway-delegated CLI commands do not call `createVoiceCallRuntime()` or start a webhook listener; at minimum, catch webhook `EADDRINUSE` and explain that the gateway voice-call runtime already owns the configured port.
