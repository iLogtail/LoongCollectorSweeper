## Summary

Keep open. Current main still wires `openclaw voicecall call` and related voicecall CLI commands to a process-local voice-call runtime, and that runtime still starts its own webhook listener on the configured bind/port. The existing `voicecall.*` gateway RPC methods are available, but the CLI does not delegate to them or expose a gateway/remote option, so the reported EADDRINUSE path remains actionable on current main.

## Best Possible Solution

Keep #72345 open as the active tracker and implement a gateway-aware voicecall CLI path. Route `call`/`start`/`continue`/`speak`/`dtmf`/`end`/`status` through the existing `voicecall.*` gateway RPCs when a running gateway is selected or explicitly requested, preserve standalone local-runtime behavior where appropriate, and add a regression test proving gateway-delegated CLI commands do not call `createVoiceCallRuntime()` or start a webhook listener. At minimum, catch webhook `EADDRINUSE` and surface a targeted message that the gateway voice-call runtime already owns the configured port.
