## Summary

Current main still appears to drop `event.messages` from `message:received` internal hooks. The docs still say `messages` can be pushed to send to the user, and command reset hooks already consume `hookEvent.messages`, but `dispatchReplyFromConfig` fires the inbound message hook asynchronously without retaining the event or forwarding any pushed messages. The linked PR #61657 was a plausible attempted fix, but it was closed unmerged, and the current tests only assert that the hook is emitted, not that pushed messages are delivered.

## Best Possible Solution

Keep #61432 open. The best path is a narrow core fix in `src/auto-reply/reply/dispatch-from-config.ts`: retain the `message:received` hook event, run it in a controlled async path, then forward non-empty `event.messages` through the same additive notice/routing policy used for other hook notices while honoring send suppression. Add focused regression coverage that proves a `message:received` hook pushing text reaches the dispatcher or route-reply path, and either rebase PR #61657 or use it as the implementation reference.
