## Summary

Keep #72364 open. Current main has compaction lifecycle UI and a manual `/compact` command, but it does not implement the requested lifecycle change: auto-compaction still runs through the direct compaction/runtime paths, and optional `truncateAfterCompaction` still rewrites the JSONL after the post-compaction transcript update and after compaction hooks. The issue is current, has concrete 2026.4.24 operator reports in the discussion, and no canonical shipped fix or duplicate was found in the provided related context or current source.

## Best Possible Solution

Keep the issue open and fix it in core compaction/session lifecycle code. The narrowest maintainer path is to make `truncateAfterCompaction` part of the same visible/atomic compaction envelope: either move the truncation before the post-compaction transcript update and final busy/after hooks, or add an explicit session-history changed/reset signal that WebChat consumes atomically. A broader alternative is the reporter’s proposal to synthesize a system-authorized `/compact`, but the implementation should avoid duplicating command-handler policy and should add regression coverage for event ordering around compaction end, transcript update, and JSONL truncation.

This ordering leaves the truncation outside the earlier post-compaction transcript update path described by the issue.
