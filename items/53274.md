## Summary

Current main still has the core problem reported in #53274: the web Control UI only creates device identity when `crypto.subtle` is available, and the fingerprint path still calls `crypto.subtle.digest`. The secondary scope-bypass pieces described in the issue appear fixed, but the title issue for HTTP LAN/reverse-proxy browser contexts remains intentionally documented and covered by tests.

## Best Possible Solution

Keep the issue open unless a maintainer intentionally consolidates it with the older live tracker #32473. The concrete product path is to decide whether OpenClaw should support browser device identity on non-secure HTTP by replacing the WebCrypto hash dependency with an existing pure-JS hash and removing the `crypto.subtle` gate, then update the current docs/tests/security messaging to match that decision. Preserve the already-shipped `dangerouslyDisableDeviceAuth` scope-bypass and operator scope normalization fixes.
