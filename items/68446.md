## Summary

Keep open. Current main has not implemented this PR: WhatsApp inbound policy still promotes configured `allowFrom` into `groupAllowFrom`, then passes it through the shared fallback resolver without `groupAllowFromFallbackToAllowFrom: false`. The PR also remains security/product-sensitive because the current command authorization path still calls the shared command gate with `policy.dmAllowFrom` and no fallback override, matching the unresolved Aisle review concern in the PR discussion. Existing docs/tests intentionally describe or preserve `allowFrom` fallback, so this needs maintainer judgment and an amended fix rather than conservative auto-close.

## Best Possible Solution

Keep #68446 open for maintainer review or an amended PR. The best fix is to define the WhatsApp contract clearly, then update `extensions/whatsapp/src/inbound-policy.ts` so DM `allowFrom` cannot make `hasGroupAllowFrom` true for group admission and cannot authorize group commands through `resolveDmGroupAccessWithCommandGate`; add regression tests for both normal group message gating and control-command authorization with `groupPolicy: "allowlist"`, configured `allowFrom`, no `groups`, and no explicit `groupAllowFrom`; update WhatsApp/group docs if fallback semantics change.
