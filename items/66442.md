## Summary

Keep #66442 open. Current main still has the reported skills-check mismatch: the bundled Discord skill only gates on `channels.discord.token`, while Discord runtime/docs support account-scoped tokens under `channels.discord.accounts.<id>.token`. The PR is useful but should be revised before merge because its proposed parent-object `channels.discord.accounts` check can false-positive without a concrete token.

## Best Possible Solution

Keep the PR open and ask for a tightened fix: support an OR requirement that accepts `channels.discord.token` or at least one concrete `channels.discord.accounts.<id>.token`, wire that through frontmatter/status/format/runtime eligibility as needed, make the requirement type contract consistent, and add regression tests for both a valid multi-account token and an empty/tokenless accounts object.
