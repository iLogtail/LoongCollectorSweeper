## Summary

Keep open. Current `main` still does not wire Claude CLI into the external-CLI auth sync path, so `anthropic:claude-cli` can remain stale relative to `~/.claude/.credentials.json`, and status/auth health surfaces still derive from the stored profile rather than the live Claude CLI credential.

## Evidence

- **External CLI sync excludes Claude CLI:** `src/agents/auth-profiles/external-cli-sync.ts` defines `EXTERNAL_CLI_SYNC_PROVIDERS` with only Codex and MiniMax entries. There is no `CLAUDE_CLI_PROFILE_ID` / `readClaudeCliCredentialsCached` registration, so the automatic external-CLI overlay/sync path described in the issue is not present on current `main`.
  - file: [src/agents/auth-profiles/external-cli-sync.ts:87](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/src/agents/auth-profiles/external-cli-sync.ts#L87)

  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **Claude CLI credentials are readable, but not used by that sync path:** `src/agents/cli-credentials.ts` still exposes `readClaudeCliCredentials` / `readClaudeCliCredentialsCached` for `~/.claude/.credentials.json` or macOS Keychain, which means the repo has the reader needed for this issue, but current sync registration does not consume it.
  - file: [src/agents/cli-credentials.ts:347](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/src/agents/cli-credentials.ts#L347)
