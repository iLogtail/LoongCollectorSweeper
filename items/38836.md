## Summary

Keep open. Current main has better managed-restart behavior and restart-continuation plumbing, but it still does not implement the issue's requested Feishu-visible notification/fan-out for in-flight sessions interrupted by a direct supervisor/systemctl restart. Plain SIGTERM is still treated as stop unless an OpenClaw restart-intent file exists, main-session restart recovery resumes with `deliver: false`, and restart sentinel handling is scoped to one sentinel payload/session rather than scanning all affected Feishu sessions after boot.

## Best Possible Solution

Keep the issue open until OpenClaw either implements a native cross-channel restart-interruption path or explicitly documents that only OpenClaw-managed restarts are supported. A robust fix should persist the affected external delivery contexts before shutdown or during abort marking, then send a Feishu-visible interruption notice or continuation after startup; it should cover direct supervisor/systemctl restarts or make that unsupported boundary explicit, with regression coverage for Feishu or a generic channel-delivery contract.
