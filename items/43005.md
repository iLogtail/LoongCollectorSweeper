## Summary

Keep #43005 open. Current main still uses a fixed cross-origin redirect safe-header list, explicitly strips custom headers such as `x-api-key`, and exposes no `GuardedFetchOptions` or config hook for opt-in additional safe headers. This is a core/security-sensitive fetch guard API question, not a ClawHub-only extension cleanup; the related #57588 is adjacent security hardening but does not supersede this request.

## Best Possible Solution

Keep the issue open for maintainer/security design. The best implementation would preserve the strict default, add a narrow opt-in API on `GuardedFetchOptions` and the shared redirect-header helper, validate/normalize header names, add regression tests proving default stripping and opt-in preservation, and document the security model. Prefer a per-call or host-scoped API over a broad global `security.fetchGuard` switch unless maintainers explicitly want global config.

## Risks / Open Questions

- Allowing custom redirect headers can leak credentials to attacker-controlled redirect targets unless the design is scoped by caller intent and destination origin/host.
- The shared helper is used by guarded fetch and media downloads, and the guarded fetch function is exported through the plugin SDK, so the blast radius is larger than one integration.
- Related #57588 keeps token-forwarding defaults under active security scrutiny; any opt-in header preservation should be designed with that fail-closed work in mind.
