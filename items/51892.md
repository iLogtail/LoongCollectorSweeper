## Summary

Keep #51892 open. Current main fixes the older inline-tag leak by stripping `[[reply_to:*]]` and sending a separate `reply_to` RPC field, but it still does not pass the requested `reply_to_guid` field and the related PR #51966 was closed unmerged. The current upstream `steipete/imsg` public source/docs also did not confirm a send-side `reply_to_guid`/`reply_to` contract, so this needs maintainer/dependency confirmation rather than auto-close.

## Best Possible Solution

Keep the issue open until maintainers confirm the actual `imsg` send RPC contract. If native threaded replies are supported, wire the sanitized `replyToId` to the exact supported field (`reply_to_guid`/`replyToGUID` or documented successor) in `extensions/imessage/src/send.ts`, keep stripping inline directive tags, and add a focused RPC-param regression test. If upstream `imsg` does not support native reply sends, document that limitation and track the needed upstream dependency work instead of closing this as implemented.
