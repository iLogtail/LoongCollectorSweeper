## Summary

Keep open. Current main still implements thread-bound ACP follow-ups as direct bound-session routing/prompt delivery, and `sessions_send` forwards the caller-supplied message to `chat.send` without a core rewrite/orchestration step. The shipped docs also describe normal bound follow-ups as prompt text sent directly to the ACP session. The related context is ACP-adjacent but does not provide a canonical duplicate for this parent-orchestrated follow-up policy request.

## Best Possible Solution

Keep this issue open for maintainer product judgment. The likely fix belongs in ACP/session orchestration: define whether thread-bound ACP `mode:"session"` should keep direct bound-conversation routing or introduce an explicit parent-orchestrated follow-up mode, then update the accepted note/system prompt/tool docs and add regression coverage proving shorthand follow-ups are rewritten unless the user requests verbatim relay.
