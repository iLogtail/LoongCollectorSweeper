## Summary

Keep #59718 open. Current main still blocks sandbox file-tool writes unless `workspaceAccess` is `"rw"`, even when the resolved target is a custom `docker.binds` mount marked writable. That leaves the requested least-privilege `workspaceAccess: "none"` plus selected `:rw` binds behavior unimplemented.

## Evidence

- **Write gate still requires rw workspace access:** `ensureWriteAccess` rejects when `allowsWrites(this.sandbox.workspaceAccess)` is false, and `allowsWrites` returns true only for `access === "rw"`. This blocks writes under `workspaceAccess: "none"` before bind writability can help.
  - file: [src/agents/sandbox/fs-bridge.ts:274](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/src/agents/sandbox/fs-bridge.ts#L274)
  - command: `nl -ba src/agents/sandbox/fs-bridge.ts | sed -n '1,380p'`
  - sha: [b64bfc5d9abf](https://github.com/openclaw/openclaw/commit/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab)
- **Custom bind writability is parsed but not sufficient:** `buildSandboxFsMounts` adds `sandbox.docker.binds` as mount entries with `writable: parsed.writable`, so bind mode is known, but the fs bridge still applies the global `workspaceAccess === "rw"` write gate.
  - file: [src/agents/sandbox/fs-paths.ts:83](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/src/agents/sandbox/fs-paths.ts#L83)
