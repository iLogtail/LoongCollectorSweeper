## Summary

Keep #50887 open. The current `main` checkout still contains the UTF-8 boundary-unsafe decoding patterns called out by the issue-linked fix attempts, and I did not find a landed regression test or changelog/release entry proving the reported Chinese streaming corruption is fixed. The open #51303 PR remains the most direct active fix attempt for one implicated path, while #50909 was closed unmerged.

## Best Possible Solution

Keep #50887 open until OpenClaw lands a focused UTF-8 boundary-safe streaming fix. The likely implementation is to use `StringDecoder` or streaming `TextDecoder` at byte-stream boundaries such as `src/infra/jsonl-socket.ts` and `src/process/supervisor/adapters/child.ts`, make truncation helpers avoid cutting inside UTF-8/code-point boundaries, and add regression tests that split Chinese/CJK bytes across chunks and assert no U+FFFD reaches the assembled message/transcript. #51303 can be used as the active starting point, but the maintainer should verify the exact WebChat/provider path before closing the bug.

## Evidence

- **Current JSONL socket still decodes each TCP data chunk independently:** `requestJsonlSocket` appends `data.toString("utf8")` inside the socket `data` handler. If a multibyte UTF-8 character is split across TCP chunks, this pattern can emit U+FFFD before the next chunk arrives. The existing tests cover malformed lines, half-close, timeout/error behavior, but not a multibyte split regression.
