## Summary

Keep open. Current main still passes explicit `x-openclaw-session-key` values from the OpenAI-compatible and OpenResponses HTTP endpoints into agent ingress without rejecting internal `subagent:`, `acp:`, or `cron:` namespaces. The related #55928 report is a separate HTTP history endpoint guard, not a replacement for this PR.

## Best Possible Solution

Keep the PR open or recreate its core change on main: add a reserved-session-prefix guard in the shared Gateway HTTP request-context/session-key path, translate only that dedicated error to a 400 `invalid_request_error` in OpenAI-compatible and OpenResponses handlers, and keep tests for normal custom session keys plus `subagent:`, `acp:`, and `cron:` variants. Treat #55928 as a separate HTTP history endpoint hardening item.
