## Summary

Keep open. Current main has partial improvements around OAuth refresh coordination and stale transcript-lock cleanup, but the issue’s end-to-end multi-agent orchestration report is not implemented or superseded: the concurrent `agents.add` lost-update path still appears plausible on current main, session/transcript and session-store locks still have timeout/failover behavior relevant to the report, and linked follow-ups for config serialization and subagent completion loss remain open in the provided GitHub context.

## Best Possible Solution

Keep #43367 open as the umbrella until the remaining core reliability gaps are either fixed or split into canonical tracked issues. The best concrete path is to land or replace #44997 with a locked/mutating config write for `agents.add/update/delete`, add regression coverage for concurrent agent creation and isolated parallel runs, keep #44925/#45200-style subagent completion loss work moving, and make local session-lock failures visible/cancellable without cascading into unrelated provider failover noise.

## Evidence

- **Issue discussion shows active, concrete production impact:** The provided issue body includes a concrete reproduction for concurrent `agents add`, parallel agent runs, lock timeouts, OAuth noise, and detached background work. Later comments add production evidence for subagent completion loss and a 17-agent deployment hitting an idle-race/session-death-spiral mode. Related open items in the provided context include #44997, #45200, #44925, and #42160, so this is not an obsolete single-symptom report.
- **`agents.add` still writes a precomputed config directly:** The current gateway agents create path builds `nextConfig` from the loaded config and calls `writeConfigFile(nextConfig)` directly. This does not use the newer `mutateConfigFile` read-modify-write helper, and it matches the race shape described by #44997.
