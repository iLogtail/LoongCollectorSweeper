## Summary

Keep PR #38290 open. Current main still normalizes browser Origin values through URL.origin, so exact chrome-extension://<id> entries in gateway.controlUi.allowedOrigins still collapse to "null" and cannot match. The PR remains a narrow, relevant fix with regression coverage; the broader wildcard request is separate in #39739.

## Best Possible Solution

Keep PR #38290 open and land it after rebase, or land an equivalent patch in src/gateway/origin-check.ts that normalizes opaque-origin schemes symmetrically for incoming Origin values and configured allowedOrigins. Preserve literal Origin: null rejection, exact matching, global "*" behavior, host-header fallback semantics, and gateway WebSocket regression coverage. Keep scheme-level wildcard support such as chrome-extension://* tracked separately under #39739.
