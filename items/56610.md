## Summary

Keep open. Current main has a shared outbound write-ahead delivery queue and startup recovery, and Discord sends use the shared outbound path, but I do not see the requested Discord-specific drain/retry when the Discord gateway WebSocket reconnects. WhatsApp has that reconnect drain; Discord currently updates gateway status/READY handling without draining pending Discord delivery entries on reconnect.

## Best Possible Solution

Keep the issue open. The best maintainer path is to add Discord-owned reconnect recovery: when a Discord account reaches READY after startup or runtime reconnect, drain pending delivery-queue entries where channel is "discord" and accountId matches, with a narrowly defined bypass for gateway/no-listener reconnect failures. Add regression coverage analogous to the WhatsApp reconnect drain and verify whether Discord REST delivery can actually fail during the WebSocket gap; if not, improve diagnostics and document the corrected failure boundary.
