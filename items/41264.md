## Summary

Keep open. Current `main` still has the narrowed Slack allowlist mismatch described in the issue thread: channel authorization for inbound Slack room messages does not match `channels.slack.channels` keys written as `channel:<CHANNEL_ID>`, so a valid public channel can still fall through to `channel-not-allowed` before persona routing runs.

## Evidence

- **Inbound room auth still relies on raw channel-id matching:** `resolveSlackChannelConfig` builds candidates from the raw channel ID, lowercase form, uppercase form, and optional name forms, but it never adds a `channel:<id>` candidate. A config entry keyed as `channel:C123` therefore does not match here.
  - file: [extensions/slack/src/monitor/channel-config.ts:114](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/extensions/slack/src/monitor/channel-config.ts#L114)
  - command: `nl -ba extensions/slack/src/monitor/channel-config.ts | sed -n '88,155p'`
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **That mismatch still blocks Slack channel ingress on main:** Slack room authorization calls `resolveSlackChannelConfig`; when no match is found under `groupPolicy: "allowlist"`, the code logs and rejects the channel, which matches the report's `channel-not-allowed` symptom before any persona session is created.
  - file: [extensions/slack/src/monitor/context.ts:333](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/extensions/slack/src/monitor/context.ts#L333)
  - command: `nl -ba extensions/slack/src/monitor/context.ts | sed -n '332,364p'`
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Current Slack code is inconsistent about prefixed channel keys:** `resolveSlackChannelType` does recognize configured keys like `channel:<id>` when inferring Slack channel type, which shows the compatibility exists in one path but not in the inbound allowlist matcher.
  - file: [extensions/slack/src/channel-type.ts:40](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/extensions/slack/src/channel-type.ts#L40)
  - command: `nl -ba extensions/slack/src/channel-type.ts | sed -n '40,48p'`
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Docs do not prove a shipped fix:** The Slack docs still document `channels.slack.channels.<id>` as stable channel IDs and do not describe any supported `channel:<id>` compatibility path for inbound allowlist matching, so there is no strong documentation evidence that this report is already resolved on current main.
  - file: [docs/channels/slack.md:477](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/docs/channels/slack.md#L477)
