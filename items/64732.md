## Summary

Keep open. Current main still has the same missing-command guidance path: an unknown primary command under a restrictive `plugins.allow` can still be turned into the misleading bundled-plugin allowlist error. The directly related PR #64859 proposed the right shape of fix, but it was closed unmerged in the provided GitHub context, and the current tests do not cover the reported `help` or `tool` cases.

## Best Possible Solution

Keep the issue open and implement the #64859 approach on current main: only emit `plugins.allow`/`plugins.entries.*.enabled` guidance for actual plugin ids or manifest-backed command aliases, make `openclaw help` route to help normally under restrictive allowlists, let non-plugin roots like `tool` fall through to Commander unknown-command behavior unless the product wants to restore/document a real `tool` CLI, and add focused tests for `help`, `tool`, real plugin ids, and alias-owned commands.
