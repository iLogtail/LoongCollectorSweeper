## Summary

Keep #52826 open. Current main has useful structured agent events (`lifecycle`, `tool`, `item`, `plan`) with per-run sequencing, and plugins can observe them, but it does not expose the requested durable session-level `process.*` / `step.*` event contract with stable step identity and replay/reload semantics. Task Flow is related durable orchestration work, but it is a background-task flow subsystem, not the requested Gateway session event stream for clients such as ClawWork.

## Best Possible Solution

Add a first-class Gateway/session process event contract: define TypeBox/schema docs for `process.started`, `process.replanned`, `step.created`, `step.started`, `step.updated`, `step.blocked`, `step.completed`, `step.failed`, `process.completed`, and `process.failed`; persist/replay a per-session process log so reconnect/reload works; project existing agent lifecycle/item/plan/tool events into that contract where possible; and add gateway protocol plus UI/client tests for ordering, stable step IDs, related `toolCallId`/`messageId`/`artifactId`, and session scoping.

## Risks / Open Questions

- A future implementation should avoid duplicating existing agent `item`/`plan`/`tool` streams without a clear projection layer or migration path.
- Task Flow and agent events are adjacent surfaces; maintainers should decide whether session process events are a new Gateway event family, a durable projection of agent events, or a Task Flow-backed view.
