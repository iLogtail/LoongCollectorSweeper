## Summary

Keep open. Current main still lacks the requested send-level Telegram `NO_REPLY` guard in `sendMessageTelegram`; Slack has the parity guard and tests, while Telegram only has adjacent delivery/streaming handling. The provided related PR #68304 remains the active open fix path rather than shipped behavior.

## Best Possible Solution

Keep #38603 open and review or replace PR #68304 with a focused change in `extensions/telegram/src/send.ts`: use the SDK silent-reply helper before Telegram API work, return a synthetic suppressed result when `text.trim()` is exactly a silent reply and no media or inline keyboard is present, and add regression coverage in `extensions/telegram/src/send.test.ts`. Track broader pre-sentinel streaming buffering separately if maintainers want to solve the visible preview flash beyond the defensive send guard.
