## Summary

#30067 should remain open. Current main still does not show a proactive initial-setup heads-up before the first chat/TUI/Web handoff explaining that setup-related background work may temporarily cause provider API rate-limit errors.

## Best Possible Solution

Keep #30067 open. The likely fix belongs in core onboarding: add a concise proactive notice in the classic finalization path before the hatch prompt/TUI/Web launch, and mirror equivalent guidance in the modern Crestodian first-run or setup handoff. Add focused tests that assert the warning copy appears in the relevant first-run path; optionally align the public first-run/onboarding docs.
