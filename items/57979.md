## Summary

Keep open. Current `main` still logs config warnings unconditionally in both `createConfigIO` warning paths, with only invalid-config errors deduplicated. Recent runtime-config snapshot caching may reduce how often the spam reproduces, but it does not implement the PR’s warning dedupe, and there is still at least one direct `createConfigIO(...).loadConfig()` caller outside the pinned `loadConfig()` wrapper.

## Risks / Open Questions

- Warning spam is still possible anywhere `createConfigIO().loadConfig()` or `writeConfigFile()` is reached repeatedly with the same warnings, because both callsites remain unconditional on `main`.
- Recent runtime snapshot caching makes the original reproduction path less certain, so the exact severity may have changed, but there is not enough evidence to close the PR as already implemented or obsolete.
