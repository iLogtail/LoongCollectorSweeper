## Summary

Keep open. Current `main` does not implement orphan reconciliation, and the PR as proposed appears unsafe because its ownership check would misclassify valid `sessionKey`-owned exec runs as orphans.

## Evidence

- **Current main still has a no-op orphan reconciler:** `createProcessSupervisor()` still returns a deliberate no-op `reconcileOrphans`, so the behavior described by the PR is not implemented on `main`.
  - file: [src/process/supervisor/supervisor.ts:285](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/process/supervisor/supervisor.ts#L285)
  - command: `nl -ba src/process/supervisor/supervisor.ts | sed -n '275,295p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Supervisor contract on main has no reconciliation inputs/results:** The exported `ProcessSupervisor` interface still defines `reconcileOrphans(): Promise<void>`, matching the no-op implementation rather than the PR's proposed reconciliation contract.
  - file: [src/process/supervisor/types.ts:104](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/process/supervisor/types.ts#L104)
  - command: `nl -ba src/process/supervisor/types.ts | sed -n '95,110p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Managed runs are owned by sessionKey when present:** Exec runs are registered with supervisor ownership `sessionId: opts.sessionKey?.trim() || sessionId`, so a non-empty `sessionKey` becomes the supervisor-side owner id.
  - file: [src/agents/bash-tools.exec-runtime.ts:709](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/agents/bash-tools.exec-runtime.ts#L709)
  - command: `nl -ba src/agents/bash-tools.exec-runtime.ts | sed -n '700,760p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Bash process registry is keyed by session slug id, not sessionKey:** The running-session registry stores entries under `session.id` and `getSession(id)` reads that same map. On current main there is no parallel lookup by `sessionKey`.
  - file: [src/agents/bash-process-registry.ts:90](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/agents/bash-process-registry.ts#L90)

- **Existing PR discussion already identified the ownership mismatch:** The provided PR review context includes a Codex P1 on head `03218cc6982c6dd603647f8bbbe8b082f3f25cae`: using `hasSession` for reconciliation would wrongly cancel legitimate runs started with a non-empty `sessionKey`, because supervisor ownership and bash-session registry keys differ.
  - sha: [03218cc6982c](https://github.com/openclaw/openclaw/commit/03218cc6982c6dd603647f8bbbe8b082f3f25cae)

## Risks / Open Questions

- Merging the PR as written could terminate valid background exec runs whenever they are owned by `sessionKey` instead of the session slug id.
- Closing this PR would leave the underlying orphaned-run cleanup gap unresolved on current `main`, because orphan reconciliation is still a no-op.
