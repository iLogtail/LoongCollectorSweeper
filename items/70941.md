## Summary

Keep #70941 open. The issue is recent, author is not protected, and there are no protected labels, but current main still routes `openclaw models set` through the generic config writer without a command-specific exception for the size-drop guard. The writer still treats a >50% shrink as a blocking `CONFIG_WRITE_REJECTED` condition, and the existing `models set` tests mock the config update layer rather than exercising the real write path. The related ClawSweeper items cover config-write races or invalid-config recovery, not this CLI size-drop false-positive path.

## Best Possible Solution

Keep the issue open and fix it in the config write layer or the models command path with a regression test. The likely maintainer path is to reproduce `openclaw models set` against a valid large/comment-heavy or otherwise shrink-prone config, then make the size-drop guard distinguish a narrow semantic model-primary edit from a destructive clobber while preserving rejection for real whole-file data loss.
