## Summary

Keep open. Current `main` still renders TUI assistant and tool output as text/markdown only, with image results reduced to placeholders. I found no terminal-capability detection, inline-image config, or `pi-tui` `Image` integration in the OpenClaw TUI path, so this request is not implemented and is still actionable core TUI work.

## Evidence

- **Assistant chat path is text/markdown-only:** `AssistantMessageComponent` builds the assistant body with `HyperlinkMarkdown` only, and `ChatLog` stores/finalizes assistant entries as plain `text: string` values.
  - file: [src/tui/components/assistant-message.ts:10](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/tui/components/assistant-message.ts#L10)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Tool image outputs are explicitly stringified as placeholders:** `ToolExecutionComponent.extractText()` turns `entry.type === "image"` into placeholder text like `[image/png 123kb]` instead of rendering media.
  - file: [src/tui/components/tool-execution.ts:37](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/tui/components/tool-execution.ts#L37)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **TUI formatter drops non-text content from display text:** `extractTextFromMessage()` and `extractTextBlocks()` only collect `text` and optional `thinking` blocks for rendering; there is no branch that maps assistant image blocks into a renderable TUI component.
  - file: [src/tui/tui-formatters.ts:307](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/tui/tui-formatters.ts#L307)

  - file: [docs/web/tui.md:52](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/docs/web/tui.md#L52)
