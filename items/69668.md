## Summary

Keep issue #69668 open. Current main supports multiple isolated agents and the chat session selector groups session options by agent, but it does not implement a team/project parent concept, configurable team metadata, or a hierarchical team view in the Control UI session list. The related #45086 context explains the already-shipped agent/session switcher work, which is adjacent but not a canonical duplicate for this requested team/project grouping.

## Best Possible Solution

Keep this open for maintainer product/design judgment. The narrow path would be an additive per-agent grouping field such as `agents.list[].group` or `project`, surfaced through `agents.list`/session list contracts and rendered as collapsible groups in the Control UI; if maintainers want real team isolation, that should be designed explicitly as a core config/session boundary rather than inferred from agent-id prefixes.

## Risks / Open Questions

- A correct implementation would likely need coordinated config schema, gateway protocol, session/agent list contracts, Control UI rendering, and docs updates rather than a UI-only grouping heuristic.
- The product direction should distinguish lightweight display grouping from an actual team/project isolation model so it does not imply security or session boundaries that do not exist.
