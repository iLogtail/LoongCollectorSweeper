## Summary

Keep open. Current main has a useful partial mitigation: normal chat/send prompt additions can be separated from the visible transcript via `transcriptPrompt` plus hidden runtime-context delivery. But the `chat.history` sanitizer itself still does not recognize the specific memory/heartbeat prompt blocks reported in #64613, and heartbeat runs still build an agent prompt from `Read HEARTBEAT.md...` style text that can persist as a normal transcript user turn. Harimay23's comment is consistent with the remaining product-safe boundary work: the fix should be at `chat.history` or, better, through structural hidden/internal transcript metadata rather than WebChat-only presentation code.

## Best Possible Solution

Keep the issue open. The best maintainer path is to add a focused customer-visible history boundary fix: either persist memory/heartbeat/system-injection prompts as hidden/internal custom transcript entries, or extend `chat.history`/history-SSE sanitization to omit known internal prompt-only user rows with regression tests for string content, text content blocks, and HTTP/SSE history. The fix should include heartbeat prompt examples beginning with `Read HEARTBEAT.md` and memory-search action blocks, while avoiding broad removal of legitimate user text.

## Risks / Open Questions

- A naive text-only filter for `Read HEARTBEAT.md` could hide a legitimate user-authored message; a structural internal/provenance marker is safer where feasible.
- Any fix must preserve append-only transcript/replay behavior while hiding internal orchestration from customer-visible `chat.history` and SSE history surfaces.
- External memory plugins may inject similar agent-only blocks through current hook surfaces, so the fix should cover known patterns and provide a clear structural API path for plugins.
