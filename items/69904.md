## Summary

Keep PR #69904 open. Current main still lacks the stale transient system-event policy this PR proposes, and the PR is a focused attempt to fix a real late async-notice behavior. It should not be auto-closed, but the current PR head still needs maintainer/author follow-up because review context identifies a remaining regression risk for non-completion exec events such as `Exec denied` / `Exec started`.

## Best Possible Solution

Keep #69904 open. The best path is to revise the PR so stale transient completion/success notices are filtered by explicit event policy and session activity cutoff, while durable blocked/error/timeout notices, cron reminders, and non-completion exec notices (`Exec started`, `Exec denied`) are preserved. Add or keep focused regression tests around stale filtering, mixed cron queues, and non-completion exec-event wakes before maintainer review/merge.
