## Summary

Keep #61986 open. Current main has mitigation around Android notification forwarding: forwarding is disabled by default, Android-side allow/block/quiet-hour controls exist, and there is a burst limiter plus consecutive duplicate suppression on the gateway. But the central bug remains: each accepted Android `notifications.changed` event is still enqueued as a system event and immediately wakes heartbeat through `requestHeartbeatNow`, with only the generic 250 ms wake coalescing window rather than waiting for the configured heartbeat interval. The linked fork PR `https://github.com/KChow-ctrl/openclaw/pull/2` is useful related context, but it is not a merged OpenClaw fix or canonical in-repo superseding item.

## Best Possible Solution

Keep the issue open and fix the OpenClaw gateway/Android notification event path so accepted Android notification changes are accumulated as heartbeat context and surfaced on the next scheduled heartbeat, or behind an explicit bounded wake policy. The likely implementation should change `src/gateway/server-node-events.ts` and associated tests so `notifications.changed` no longer unconditionally calls `requestHeartbeatNow`, add regression coverage for multiple notification events across a 30-minute heartbeat interval, and align Android docs with the final batching/rate-limit semantics.

## Risks / Open Questions

- The current Android-side limiter can reduce volume, but accepted noisy notification streams can still trigger frequent immediate heartbeat runs instead of one run per configured heartbeat interval.
- The docs describe notification forwarding controls and rate limiting, but they do not document a next-heartbeat batching mode for forwarded notification events.
