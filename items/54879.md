## Summary

Keep open. Current `main` still does not expose or implement a dedicated CDP/IME text-insertion browser action for contenteditable editors; the browser tool contract only offers `type`/`fill`-style input and both runtime paths route through those existing mechanisms.

## Evidence

- **Act kinds do not include an IME/CDP insert action:** `ACT_KINDS` on current `main` only lists `batch`, `click`, `close`, `drag`, `evaluate`, `fill`, `hover`, `scrollIntoView`, `press`, `resize`, `select`, `type`, and `wait`.
  - file: [extensions/browser/src/browser/routes/agent.act.shared.ts:1](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/browser/src/browser/routes/agent.act.shared.ts#L1)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Public browser tool schema exposes no insert-text variant:** `BROWSER_ACT_KINDS` and `BrowserActSchema` only allow the existing actions and fields; there is no `insertTextIme`/`cdpInsertText`-style kind or parameters in the agent-facing schema.
  - file: [extensions/browser/src/browser-tool.schema.ts:4](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/browser/src/browser-tool.schema.ts#L4)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Request normalization only accepts `type` text input:** `normalizeActRequest()` has a `case "type"` that validates `ref|selector` plus `text`; there is no separate normalization branch for a CDP-backed insert action.
  - file: [extensions/browser/src/browser/routes/agent.act.normalize.ts:117](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/browser/src/browser/routes/agent.act.normalize.ts#L117)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Playwright path still uses fill/type, not CDP text insertion:** `typeViaPlaywright()` clicks and then uses `locator.type(...)` or `locator.fill(...)`. A repo-wide search of `extensions/browser/src` found no `Input.insertText` implementation in the browser tool runtime.
  - file: [extensions/browser/src/browser/pw-tools-core.interactions.ts:700](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/browser/src/browser/pw-tools-core.interactions.ts#L700)

  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Existing-session path also routes `type` through fill:** For Chrome MCP profiles, `/act` handles `case "type"` by calling `fillChromeMcpElement(...)`, and that helper invokes the Chrome MCP `fill` tool. That is still not a dedicated IME/native insert path.
  - file: [extensions/browser/src/browser/routes/agent.act.ts:428](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/browser/src/browser/routes/agent.act.ts#L428)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
