## Summary

Keep #69453 open. Current main has improved OpenAI Codex GPT-5.4 metadata and runtime resolution, but the exact reported consistency problem is not fully closed: read-only status/session-default paths still use synchronous lookup/fallback behavior, stale persisted context values still override resolver output, and the product currently treats native `contextWindow` and effective `contextTokens` as separate values rather than making GPT-5.4 a 1.05M runtime budget everywhere.

## Best Possible Solution

Keep the issue open and narrow the next fix to context-resolution consistency. Maintainers should decide the intended effective cap for `openai-codex/gpt-5.4`, then make fresh session defaults, status/session listing, persisted session repair, and model catalog display derive from the same provider-aware effective metadata. Add regression coverage for cold/fresh sessions, stale persisted 200k entries, `openclaw status`, model list/catalog, and the native `contextWindow` versus runtime `contextTokens` display contract.

## Risks / Open Questions

- Changing GPT-5.4 to use 1.05M everywhere would conflict with the current documented split between native `contextWindow` and effective runtime `contextTokens`, so maintainers should decide whether the desired fix is consistency at 272k or raising the default cap.
- A fix should preserve read-only behavior for status/model-list paths and avoid reintroducing background discovery or `models.json` writes in commands that are intentionally read-only.
