## Summary

Keep open. Current main has relevant mitigations for MiniMax Anthropic-compatible payloads and WSL2/Undici transport behavior, but the evidence still does not prove the reported MiniMax CN 10+ second per-message delay is fixed. The issue is recent, model/provider-specific, and there is no direct current-main non-repro or issue-linked latency regression test.

## Best Possible Solution

Keep the issue open until a maintainer can reproduce or disprove it against current main with MiniMax CN API/OAuth on WSL2. The best path is to capture timings across `chat.send`, context assembly, `lifecycle:start`, model-call start, first streamed chunk, and first visible assistant text; if the delay persists, fix the implicated MiniMax stream/payload, provider discovery, or transport path and add a focused regression test or documented diagnostic guidance.
