## Summary

Keep PR #39653 open. Current main still contains the hostname fallback this PR removes, current tests still assert hostname `ws://` acceptance when `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` is set, and the docs describe the break-glass flag as private-network IP access rather than arbitrary hostname plaintext transport. The linked issue #37177 is closed stale and prior PR #37883 was closed unmerged, so neither supersedes this PR.

## Best Possible Solution

Keep PR #39653 open for maintainer security review. The best path is to merge or adapt the narrow validator change so `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` accepts loopback/private IP literals but rejects unresolved hostnames, then update the current gateway net, client, connection-details, and onboarding tests to match. If maintainers want private DNS hostname support, track that separately as an explicit allowlist or async DNS-resolution design rather than auto-closing this fix.
