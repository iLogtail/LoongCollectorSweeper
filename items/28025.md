## Summary

Keep #28025 open. Current main still lacks `channels.slack.thread.replyOnToolUse` or an equivalent tool-use-gated Slack reply-routing policy; Slack reply placement is still determined by `replyToMode`, chat type, and existing thread state before agent/tool execution.

## Best Possible Solution

Keep the issue open for explicit maintainer product judgment. If accepted, implement it in the bundled Slack plugin/config surface with a default-false typed `channels.slack.thread.replyOnToolUse` option, carry a reliable per-turn tool-use signal into Slack reply planning before streaming or final delivery starts, preserve existing explicit thread replies and `thread.inheritParent`, and add focused Slack tests for DMs, channels, existing threads, `replyToMode` interactions, streaming, and tool/progress callbacks.
