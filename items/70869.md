## Summary

Keep open. Current main mitigates part of the provider-replay risk by filtering `delivery-mirror` assistant rows before prompt submission, but it still writes outbound delivery mirrors as persisted `role: assistant`, `model: delivery-mirror` transcript entries with no config toggle and no guard against appending after a different assistant turn. That leaves the central requested core behavior unresolved; adjacent reports #63779 and #72141 remain open and #70870 covers the separate LCM/plugin repair scope rather than superseding this mirror-write issue.

## Best Possible Solution

Keep the issue open and add a focused core regression around the message-tool/outbound-route path where a parent session already ends in an assistant turn. The fix should either add an explicit delivery-mirror policy toggle or skip/convert mirror writes when they would append an unpaired assistant transcript-only row, with coverage for direct delivery, plugin-handled send, delivery queue replay, isolated cron/subagent contexts, and Anthropic replay input so the existing provider-side filtering is not the only protection.
