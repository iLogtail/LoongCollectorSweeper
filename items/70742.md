## Summary

Keep PR #70742 open. Current main does not contain the PR’s proposed timeout semantics: `flushPendingToolResultsAfterIdle` still flushes after an idle wait timeout unless `clearPendingOnTimeout` is true and `clearPendingToolResults` exists. The current test suite also still asserts the opposite behavior for the no-cleanup timeout path, so this is not implemented or superseded on main.

## Best Possible Solution

Keep the PR open for maintainer review or rework. The concrete path is to decide the intended helper contract: if any idle wait timeout should suppress synthetic flushing, land a focused version of #70742 with the structured timeout state and updated tests; if only cleanup paths should clear without flushing, preserve current behavior and consider a narrower test/clarification around `clearPendingOnTimeout`.

That differs from PR #70742's proposed unconditional return on timeout.
