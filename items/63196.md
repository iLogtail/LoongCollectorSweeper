## Summary

Keep open. Current `main` still models cron task reconciliation purely from active-job ownership, not cron run history, so a stale terminal `lost` row can continue to audit as an error even if `cron runs` later shows a finished `ok` entry. The related work referenced in the issue timeline (#60299) is present, but it only codifies the active-job ownership rule and does not address this reported mismatch.

## Evidence

- **Cron backing check still ignores run history:** `hasBackingSession()` treats cron tasks as backed only when `isCronJobActive(jobId)` is true; it does not inspect cron run-log state or finished `ok` entries.
  - file: [src/tasks/task-registry.maintenance.ts:124](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/tasks/task-registry.maintenance.ts#L124)
  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **Cron activeness is only an in-memory active-job set:** `isCronJobActive()` reads a process-global `Set<string>` of active job ids. That matches the issue's symptom: once the job is no longer active, maintenance has no alternate source of truth from persisted cron run history.
  - file: [src/cron/active-jobs.ts:3](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/cron/active-jobs.ts#L3)
  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **Audit still reports any terminal `lost` row as an error:** `listTaskAuditFindings()` unconditionally emits a `lost` error for tasks whose persisted status is `lost`, with the stored error text such as `backing session missing`.
  - file: [src/tasks/task-registry.audit.ts:127](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/tasks/task-registry.audit.ts#L127)
  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **Maintenance will not reconcile an already-lost row away:** `runTaskRegistryMaintenance()` only reconciles tasks when `shouldMarkLost(current, now)` is true, and `shouldMarkLost()` returns false for non-active tasks. Terminal rows are only stamped for cleanup or pruned after retention.
  - file: [src/tasks/task-registry.maintenance.ts:163](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/tasks/task-registry.maintenance.ts#L163)
  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **Docs still describe cron reconciliation as active-job ownership plus 7-day retention:** The operator docs say cron tasks use active-job ownership for reconciliation and terminal task records are kept for 7 days before pruning, which is consistent with the reported stale-`lost` behavior rather than a fix.
  - file: [docs/automation/tasks.md:284](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/docs/automation/tasks.md#L284)
  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **Related #60299 coverage does not cover this mismatch:** The regression test added for issue #60299 asserts that stale cron tasks are marked `lost` once the runtime no longer tracks the job as active, reinforcing the same ownership rule but not reconciling against cron run history.
  - file: [src/tasks/task-registry.maintenance.issue-60299.test.ts:133](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/tasks/task-registry.maintenance.issue-60299.test.ts#L133)
