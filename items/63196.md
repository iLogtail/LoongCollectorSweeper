## Summary

Keep open. Current main has fixed the related active cron-task false-lost path from #60299/#71963, but this issue is narrower: a cron task row that is already terminal `lost` while matching cron history says `finished` + `ok`. The current maintenance/audit paths still do not reconcile an already-lost cron row back to `succeeded` or clear it based on cron run history.

## Best Possible Solution

Keep this issue open and implement a focused cron/task-ledger repair path for terminal `lost` cron rows. The safest path is to reconcile only when the row can be matched conclusively to a durable cron run-log or job-state entry for the same execution with `finished`/`ok`, then mark the task `succeeded` or prune/clear the stale audit finding. Add regression coverage for a `runtime: "cron"`, `status: "lost"`, `error: "backing session missing"` row with matching successful cron history so `openclaw tasks maintenance --apply` clears the audit error without hiding genuine failed or unknown runs.
