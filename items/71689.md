## Summary

Keep #71689 open. Current main still opens the task registry directly from `$OPENCLAW_STATE_DIR/tasks/runs.sqlite`, restores by calling `loadSnapshot()`, and on restore failure only logs `Failed to restore task registry`; I found no corruption detection, quarantine/snapshot, `.recover` rebuild path, reopen flow, operator event, or task-registry audit finding for malformed SQLite images. The provided related reports are adjacent storage/registry issues, not a canonical duplicate for corrupted `tasks/runs.sqlite` recovery.

## Best Possible Solution

Keep #71689 open as core storage resilience work. Add a narrow task-registry SQLite corruption recovery path around open/restore: classify malformed-image `ERR_SQLITE_ERROR` failures, close handles, preserve/quarantine the original DB plus WAL/SHM sidecars, attempt a bounded recovery or clean rebuild, reopen and restore what can be recovered, surface an explicit operator/audit finding or event, and add regression coverage using a deliberately invalid/corrupt `runs.sqlite` fixture or mocked `DatabaseSync` failure. Coordinate with adjacent task-flow registry restore/audit behavior so both durable registries expose comparable diagnostics, but do not close this as duplicate of #66747 because that report is about chmod/packaging `EPERM`, not SQLite image corruption.
