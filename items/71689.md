## Summary

Keep open. Current main still treats task registry restore failure as a warning-only path and does not snapshot, recover, rebuild, reopen, or surface a durable operator/audit finding for malformed `tasks/runs.sqlite`. The related items in the provided context are title-overlap only and do not supersede this specific SQLite corruption report.

## Best Possible Solution

Keep the issue open and implement a task-registry-owned corruption recovery path. Detect SQLite corruption on restore/open, close the bad handle, preserve a timestamped snapshot of the DB and sidecars, attempt a safe rebuild/recover or fall back to an explicit empty registry only with an operator-facing event/audit finding, then add regression tests for malformed DB restore and docs/doctor guidance if manual recovery is needed.

## Risks / Open Questions

- Automatic recovery must avoid silent data loss; snapshot `runs.sqlite` plus WAL/SHM sidecars before rebuilding or replacing anything.
- Malformed SQLite can fail at open, PRAGMA/schema migration, or SELECT, so the fix should distinguish corruption from permissions, missing directory, and transient lock/open errors.
