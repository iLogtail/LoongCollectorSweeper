## Summary

PR #41254 should remain open. Current `main` at `e94c0bf51542fe518e49ad4e2ee29882891b5603` still uses the unsafe `loadSessionStore` -> mutate -> `saveSessionStore` sequence for heartbeat dedupe, so the race described in the PR is not implemented on main and remains plausible.

- **Current heartbeat dedupe write is still raw load/save:** `runHeartbeatOnce` still loads the session store outside the write lock, mutates `store[sessionKey]`, and then calls `saveSessionStore`, matching the race described in the PR rather than the proposed `updateSessionStore` fix.
