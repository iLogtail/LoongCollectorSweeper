## Summary

Keep open. Current `main` does not implement the proposed Discord gateway WebSocket handshake timeout, and the newer lifecycle watchdogs still only cover sockets that emit `open` and then fail to reach READY, not sockets that stay stuck in CONNECTING after a reconnect.

## Evidence

- **Current Discord gateway socket creation still has no handshake timeout:** `SafeGatewayPlugin.createWebSocket()` on current `main` throws on empty URL, picks `ws.default`, and constructs the socket with either `{ agent }` or `undefined`. There is no `handshakeTimeout` option in the current implementation.
  - file: [extensions/discord/src/monitor/gateway-plugin.ts:360](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/extensions/discord/src/monitor/gateway-plugin.ts#L360)
  - command: `nl -ba extensions/discord/src/monitor/gateway-plugin.ts | sed -n '340,378p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Current tests still encode the no-timeout behavior:** `provider.proxy.test.ts` asserts the no-proxy path calls the WebSocket constructor with `undefined` options, which would fail if a handshake-timeout option had already been added on `main`.
  - file: [extensions/discord/src/monitor/provider.proxy.test.ts:286](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/extensions/discord/src/monitor/provider.proxy.test.ts#L286)
  - command: `nl -ba extensions/discord/src/monitor/provider.proxy.test.ts | sed -n '286,326p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Lifecycle watchdog only handles opened-but-not-ready reconnects:** The runtime lifecycle observer starts its READY timeout only after a debug message containing `Gateway websocket opened`. On `Gateway reconnect scheduled in ...`, it only pushes disconnected status and clears the ready watch; it does not force-stop or restart a socket that never emits `open`/`close`/`error`.
  - file: [extensions/discord/src/monitor/provider.lifecycle.ts:241](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/extensions/discord/src/monitor/provider.lifecycle.ts#L241)
  - command: `nl -ba extensions/discord/src/monitor/provider.lifecycle.ts | sed -n '235,285p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Upstream Carbon 0.16.0 still depends on close/error to advance reconnects:** OpenClaw currently pins `@buape/carbon` 0.16.0. In that version, `GatewayPlugin.createWebSocket()` creates `new nodeWebSocket(url)` with no options, and reconnect handling advances from socket `close`/`error` paths. That matches the PR discussion's root cause and does not show an upstream fix already present in the pinned dependency.

  - sha: [0.16.0](https://github.com/openclaw/openclaw/commit/0.16.0)
- **The `ws` dependency supports the requested timeout behavior:** `ws` 8.20.0 maps `options.handshakeTimeout` onto the request timeout and aborts the opening handshake on timeout, which is the exact mechanism the PR relies on.
