## Summary

Keep open. Current main still matches the reported behavior for non-backend shared-auth API clients: operator shared auth can allow a device-less connection, but token/password self-declared scopes are cleared unless the client is the narrow direct-local `gateway-client`/`backend` exception.

## Best Possible Solution

Keep the issue open for maintainer product/security judgment. If OpenClaw should support external headless operator WebSocket API clients without device identity, implement an explicit documented trust path in the gateway handshake and protocol docs, with tests for `chat.history`/`chat.send` preserving `operator.read`/`operator.write` after shared-token/password auth. If the current boundary is intentional, maintainers should close it manually as not planned and point external clients to a supported device-identity or trusted-backend path.
