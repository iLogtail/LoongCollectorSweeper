## Summary

Current main still has the reported behavior: `parseConfigValue` only recognizes plain integers/decimals, so exponent notation falls through as a string. The function is used by `/config set`, `/mcp set`, and the CLI `mcp set` path, and main does not have the PR's new focused regression test. The earlier bot review comments were about an accidental heartbeat test that the PR context shows was reverted from the branch, so they do not justify closing this PR.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to land a scoped fix in `parseConfigValue` that treats finite scientific notation as a number, with the added colocated unit test covering exponent forms and existing booleans/null/plain numbers/JSON/quoted strings. Maintainers may want one quick validation pass over `/config set`, `/mcp set`, and CLI `mcp set` behavior before merging.
