## Summary

Keep open. Current main still only retries executeWithApiKeyRotation across configured API keys, using the default rate-limit/quota predicate, and the media transcription path does not provide a transient 5xx/timeout/network retry predicate or same-key retry budget. The related #60421/#60518 discussion covers visibility/registry failures, but it does not supersede this remaining retry-policy gap.

## Best Possible Solution

Keep the issue open and implement a bounded transient retry policy for provider media requests. The likely path is to extend or wrap executeWithApiKeyRotation so rate-limit errors still rotate keys, while retryable 5xx, AbortError, and network/undici timeout errors can retry the same key with exponential backoff. Add focused tests for single-key 503/AbortError retry, multi-key 429 rotation, and media audio call-site behavior.
