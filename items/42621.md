## Summary

Keep #42621 open. Current main still routes `/hooks/agent` and mapped `action: "agent"` webhooks through an isolated cron agent turn, exposes `sessionKey` but not `sessionTarget`/`targetExistingSession`, and documents webhook agent turns as isolated. The related wake-hook PR #70268 is explicitly a workaround and does not solve the broader agent-hook request.

## Best Possible Solution

Keep the issue open and implement a backward-compatible core hook option, likely `sessionTarget` or `targetExistingSession`, for `/hooks/agent`, hook mappings, and transform results. Default to the current isolated behavior, validate request-controlled non-isolated targets with the existing hook session policy, thread the target into `CronJob.sessionTarget`, and add focused gateway/config/docs regression coverage for direct and mapped agent hooks targeting an agent's existing main session.
