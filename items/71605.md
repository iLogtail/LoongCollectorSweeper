## Summary

Keep #71605 open. Current main still has the reported shape: `openclaw agent` waits for a final Gateway `agent` response using a timeout derived from `--timeout`, falls back to embedded execution on any Gateway error, and the embedded run still acquires the same per-session transcript lock with a 10s default acquisition timeout. The docs describe fallback and session write locks but do not document a daemon-vs-embedded same-state limitation or refusal path.

## Best Possible Solution

Keep this open as a real Gateway/CLI integration bug. A maintainer fix should make the Gateway path return/track `runId` reliably and wait via `agent.wait` or equivalent without timing out healthy accepted runs, and should stop falling back to same-state embedded mode when a running daemon owns the session state. Add a clear daemon/state-dir guard or separate state-dir support, update `docs/cli/agent.md`, and add regression coverage for both the Gateway timeout and embedded session-lock fallback behavior.

## Risks / Open Questions

- The exact 60s timeout trigger is intermittent and macOS-daemon-specific, so a fix still needs either a focused live reproduction or a deterministic Gateway/CLI regression harness.
- Changing `expectFinal` or fallback behavior could affect scripted users that currently rely on `openclaw agent` returning the final reply, so the fix should preserve a supported final-wait path such as `agent.wait` while avoiding same-state embedded contention.
