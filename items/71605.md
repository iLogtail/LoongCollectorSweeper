## Summary

Keep open. The issue is recent, concrete, and not protected, and current main still matches the reported Gateway/embedded fallback shape: the CLI waits for a final `agent` Gateway response with a timeout derived from `--timeout`, falls back to embedded execution on Gateway failure, and the embedded runner still takes a session write lock with the 10s default acquisition timeout. The existing ClawSweeper comment already reached the same keep-open conclusion; related Discord issues #38596 and #71546 are adjacent motivation, not canonical duplicates of this Gateway/CLI/session-lock report.

## Best Possible Solution

Keep this open as a real Gateway/CLI integration bug. The maintainer fix should make `openclaw agent` avoid a long final-response Gateway timeout for healthy accepted runs, likely by treating the fast `runId` ack plus `agent.wait` as the control path, and should prevent same-state embedded fallback from contending with a running daemon by refusing clearly, supporting a separate state dir, or adding cooperative lock handling. Update `docs/cli/agent.md` and add regression coverage for the Gateway timeout and embedded session-lock fallback behavior.
