## Summary

Keep #71605 open. The issue is recent, concrete, and the current main checkout still has the reported control-flow shape: `openclaw agent` waits for a final Gateway `agent` response with a timeout derived from `--timeout`, falls back to same-state embedded execution on any Gateway failure, and embedded PI still takes a process/file session write lock with the 10s default acquisition timeout. Current docs explain Gateway fallback and the cross-process session lock, but they do not document a daemon-vs-embedded CLI limitation or provide a guard/refusal path.

## Best Possible Solution

Keep this open as a real Gateway/CLI integration bug. The maintainer path should either make `openclaw agent` use the immediate `agent` ack plus `agent.wait`/events without turning a healthy daemon run into a 60s request timeout, or stop falling back to same-state embedded execution while a daemon owns that state. Add an explicit daemon/state-dir guard or separate state-dir support for embedded fallback, clearer error text, docs under `docs/cli/agent.md`, and regression coverage for the Gateway final-wait/fallback and session-lock behavior.
