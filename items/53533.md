## Summary

Keep open. Current `main` does not implement the Feishu connection-health surfacing from this PR, and the PR discussion still shows an unresolved gateway wiring gap, so there is no conservative basis to close it as implemented, obsolete, or incoherent.

## Evidence

- **Gateway startup still does not pass a health/status sink on main:** `startAccount` only seeds `{ accountId, port }` via `ctx.setStatus(...)` and then calls `monitorFeishuProvider({ config, runtime, abortSignal, accountId })`. There is no callback that would propagate websocket/webhook lifecycle patches into channel runtime status.
  - file: [extensions/feishu/src/channel.ts:1236](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/feishu/src/channel.ts#L1236)

  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Feishu monitor entrypoint on main has no status-sink API:** `MonitorFeishuOpts` only defines `config`, `runtime`, `abortSignal`, and `accountId`; both `monitorSingleAccount(...)` call sites omit any health patch sink, so the monitor cannot surface connection-state updates upstream.
  - file: [extensions/feishu/src/monitor.ts:11](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/feishu/src/monitor.ts#L11)
