## Summary

Keep open. Current main still unconditionally copies ConfigMap-backed `openclaw.json` and `AGENTS.md` into the PVC on every init-container run, so the PR's central data-loss concern is not implemented on main. The PR also has a real unresolved product tradeoff: the existing Kubernetes docs and deploy script describe ConfigMap edits plus redeploy as the update path, while the proposed guard makes those files one-time bootstrap state.

## Best Possible Solution

Keep the item open for maintainer judgment and a fuller Kubernetes ownership decision. The likely fix should preserve user-edited PVC state without silently breaking declared ConfigMap updates, for example by documenting one-time bootstrap semantics plus an explicit reset/force-sync path, or by implementing a safer checksum/merge/sync mechanism across `scripts/k8s/manifests/deployment.yaml`, `scripts/k8s/deploy.sh`, and `docs/install/kubernetes.md`.
