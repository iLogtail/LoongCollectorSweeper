## Summary

Keep open. Current `main` still drops structured connect-handshake errors when the WebSocket closes, so the PR's fix is not implemented on `main` and the report remains valid.

## Evidence

- **Current close handler still flushes pending requests with a generic error:** `GatewayBrowserClient` reads `pendingConnectError` but immediately rejects all pending requests with `new Error("gateway closed (...)")` instead of converting the stored connect error into `GatewayRequestError`. That is the exact bug described in the PR.
  - file: [ui/src/ui/gateway.ts:334](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/ui/src/ui/gateway.ts#L334)
  - command: `nl -ba ui/src/ui/gateway.ts | sed -n '334,341p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Structured connect errors are still recorded before close:** `handleConnectFailure` persists `gatewayCode` and `details` into `pendingConnectError` and then closes the socket, so current `main` still has the stale handoff where the later close path throws away the structured error.
  - file: [ui/src/ui/gateway.ts:492](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/ui/src/ui/gateway.ts#L492)
  - command: `nl -ba ui/src/ui/gateway.ts | sed -n '489,510p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Current tests cover reconnect behavior, not propagation to pending requests:** The nearby auth-failure test asserts reconnect suppression after `AUTH_TOKEN_MISSING`, but there is no corresponding unit test on `main` that verifies a pending RPC rejects with `GatewayRequestError` carrying `details.code` after connect failure.
  - file: [ui/src/ui/gateway.node.test.ts:440](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/ui/src/ui/gateway.node.test.ts#L440)
