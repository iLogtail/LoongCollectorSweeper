## Summary

Keep #65317 open. Current main has a maintained Dockerfile that installs `python3`, but the runtime fallback for the default sandbox image still creates `openclaw-sandbox:bookworm-slim` by pulling and tagging bare `debian:bookworm-slim`. That preserves the reported split: ordinary shell writes can work while the pinned write/edit helper exits with the exact missing-Python error.

## Best Possible Solution

Keep the issue open and fix the default sandbox image contract. The clean path is to stop auto-tagging bare `debian:bookworm-slim` as `openclaw-sandbox:bookworm-slim`; instead build/pull the maintained sandbox image that includes `python3`, or fail with an actionable setup/rebuild message. Add coverage for the missing-default-image path and for the helper dependency, and consider a recreate/doctor check so stale Python-less default containers are detected.

## Risks / Open Questions

- Closing now would leave the automatic default-image fallback able to create a sandbox where `exec` shell writes succeed but `write`/`edit` fail with the reported helper error.
- Existing local images or containers tagged as `openclaw-sandbox:bookworm-slim` from bare Debian can continue to look config-current because the image tag matches, so a fix may need image validation, recreation guidance, or a config/image epoch bump.
