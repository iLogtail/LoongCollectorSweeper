## Summary

Keep open. Current main still does not implement or clearly validate the reported `env:DISCORD_BACKUPS_CHANNEL` cron delivery target. The cron path preserves `delivery.to` literally, resolves it as an outbound target, and Discord normalization only handles bare numeric IDs or passes other strings through unchanged. That leaves the reported false error/unknown delivery state actionable rather than safely closeable.

## Best Possible Solution

Keep the issue open. Maintainers should either implement documented env-ref support for cron `delivery.to` before target resolution, or reject unsupported `env:` targets at cron create/edit time with a clear error. A focused regression test should cover Discord announce delivery with an env-backed target and verify the run status, delivery status, and `delivered` flag.
