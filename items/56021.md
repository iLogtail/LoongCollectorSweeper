## Summary

Keep open. Current `main` contains partial replay/tool-result hardening, especially for Anthropic-compatible retry flows, but the issue's broader claims are not fully implemented away: `github-copilot` still rides the shared OpenAI Responses transport while remaining outside the pairing-preservation allowlist, and the shared tool-call ID sanitizer still synthesizes fallback IDs for orphan tool results.

## Evidence

- **GitHub Copilot still excluded from Responses pairing-preservation set:** `github-copilot` OpenAI-family models are still routed through the shared OpenAI Responses transport, but the allowlist passed to `convertResponsesMessages()` only includes `openai`, `openai-codex`, `opencode`, and `azure-openai-responses`, not `github-copilot`. That is the exact omission called out in the issue.

  - command: `nl -ba src/agents/openai-transport-stream.ts | sed -n '831,846p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Shared sanitizer still invents orphan tool-result IDs:** When no pending assistant tool-call match exists, `resolveToolResultId()` still allocates a replacement ID seeded from the orphan tool-result ID via `${id}:tool_result:${occurrence}` instead of dropping it at that layer. That matches the reported malformed-ID family.
