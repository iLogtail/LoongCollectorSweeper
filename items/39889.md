## Summary

Current main does not provide enough evidence to conservatively close #39889. The Control UI still uses the same served-shell shape described in the report: `ui/index.html` predeclares `<openclaw-app>`, `ui/src/main.ts` imports the app module for side-effect registration, and `OpenClawApp` is still registered through the Lit `@customElement("openclaw-app")` decorator. I found no explicit guarded root `customElements.define("openclaw-app", ...)` fallback and no browser test that loads the real served `ui/index.html` or built bundle path and asserts root registration in Chromium/Vivaldi. A related browser-bundle crash fix exists in the changelog (#48930), but it is not tied to this issue closely enough to prove the reported Vivaldi/Chromium hang is fixed.

## Best Possible Solution

Keep #39889 open. The best maintainer path is to add a served-Control-UI Chromium regression check that loads the real `ui/index.html` or built Control UI bundle, captures page/module/CSP/resource/service-worker errors, asserts `customElements.get('openclaw-app')`, and then hardens the bootstrap path based on the reproduced failure. If #48930 is believed to be the fix, verify it against the original Vivaldi/Chromium scenario or add a regression that proves that root cause cannot reappear.
