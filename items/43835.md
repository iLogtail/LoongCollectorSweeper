## Summary

Keep open. Current main still prunes sandbox runtimes and registry entries without removing workspace directories, so the underlying #43797 leak is not implemented or obsolete. This PR is also not safely superseded by a merged fix: its latest patch still has unresolved correctness concerns around stored scope keys, failed runtime removal, same-scope active entries, and workspaceRoot provenance.

## Best Possible Solution

Keep the bug tracked in #43797 and rework one focused sandbox PR. The best fix belongs in `src/agents/sandbox/prune.ts` plus the backend/registry contract if needed: delete only the correct non-shared main sandbox workspace after confirmed runtime removal, use stored registry scope keys directly, guard against any remaining active same-scope entries, account for per-agent `workspaceRoot`, and add targeted tests for the prune/workspace lifecycle.

## Evidence

- **Current main prune path has no workspace cleanup:** `pruneSandboxRegistryEntries` removes the runtime, then removes the registry entry and optional browser bridge state. It does not import `fs`, resolve a workspace path, or call `fs.rm`, so the reported workspace directory leak remains on main.
  - file: [src/agents/sandbox/prune.ts:38](https://github.com/openclaw/openclaw/blob/ee2ab9a6449219e1c425308a4832ddbb7eeb0ca2/src/agents/sandbox/prune.ts#L38)

  - sha: [ee2ab9a64492](https://github.com/openclaw/openclaw/commit/ee2ab9a6449219e1c425308a4832ddbb7eeb0ca2)
- **Workspace directories are scope-derived:** Sandbox context resolves non-shared workspace directories from `cfg.workspaceRoot` plus a computed `scopeKey`, while shared scope uses `workspaceRoot` directly. Registry entries persist `sessionKey: scopeKey`, so prune cleanup must use the stored scope key carefully rather than re-normalizing it.
  - file: [src/agents/sandbox/context.ts:41](https://github.com/openclaw/openclaw/blob/ee2ab9a6449219e1c425308a4832ddbb7eeb0ca2/src/agents/sandbox/context.ts#L41)

  - sha: [ee2ab9a64492](https://github.com/openclaw/openclaw/commit/ee2ab9a6449219e1c425308a4832ddbb7eeb0ca2)
- **PR head still double-normalizes stored scope keys:** The latest #43835 patch computes `const scopeKey = resolveSandboxScopeKey(params.cfg.scope, entry.sessionKey)` before deleting the workspace. Review comments on the PR correctly point out that registry `entry.sessionKey` is already the normalized scope key, so agent-scoped entries such as `agent:ops` can target the wrong workspace.
  - file: [src/agents/sandbox/prune.ts:66](https://github.com/openclaw/openclaw/blob/c99a72abe3461e6202e80c57d3c69c8e786b9507/src/agents/sandbox/prune.ts#L66)
