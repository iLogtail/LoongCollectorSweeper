## Summary

Keep open. The report is concrete, recent, and security-sensitive. Current main still treats `skills.entries.<skill>.enabled: false` as skill eligibility/prompt/status configuration, not as a filesystem or execution sandbox boundary; the read and exec tool paths are built around workspace/sandbox/tool policy, with no disabled-skill root denylist. Related sandbox-skill items are adjacent but do not supersede this specific disabled-skill enforcement gap.

## Best Possible Solution

Keep #57119 open for a core security/design fix. The best path is to either explicitly document `enabled: false` as visibility-only, or implement real enforcement by deriving disabled skill realpaths and denying read/write/edit/apply_patch plus sandbox/exec access to those paths. For strong security, prefer sandbox/mount or low-privilege filesystem isolation so disabled skill directories are not present or executable in the agent runtime, with tests covering host and sandbox runs, workspace/personal/managed/plugin skills, skillKey aliases, and symlink/realpath escapes.

## Risks / Open Questions

- A command-parser-only exec denylist would be incomplete for shell execution; robust enforcement likely needs filesystem isolation, mount exclusion, or OS/container permissions in addition to tool-level checks.
- Changing `enabled: false` from visibility control to hard access control could be compatibility-sensitive for existing workflows, so docs and UI wording should be updated with the final security contract.
