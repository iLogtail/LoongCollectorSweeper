## Summary

Keep #72394 open. Current main still creates an isolated `:heartbeat` session and only stores the session entry there; pending system-event entries remain keyed to the base session unless the heartbeat preflight reads and consumes them from that base key. Existing tests cover the current workaround behavior for cron prompts and explicitly assert that base-session hook/wake exec events are not classified when isolated sessions are enabled, so the requested generic transfer/lookup fix is not implemented on main.

## Best Possible Solution

Keep the issue open and implement a generic core fix in the heartbeat/system-event boundary: either mirror base-session pending event entries onto the isolated `:heartbeat` key before the model run, or centralize active-heartbeat event lookup so consumers can intentionally read the base queue for pre-isolation events. Add regression coverage for base-queued exec, cron, delivery-context, and untrusted events under `heartbeat.isolatedSession: true`, including no replay on the next heartbeat.
