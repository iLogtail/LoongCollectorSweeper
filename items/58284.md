## Summary

Keep open. Current `main` still implements session pruning as TTL-gated `cache-ttl` behavior only; there is no turn-based/rolling prune option in the config schema, and active eligible attempts keep refreshing the cache-touch timestamp, so the feature gap described in #58284 remains plausible on current code.

## Evidence

- **Pruning is explicitly blocked until TTL expires:** The context-pruning extension returns early when `Date.now() - lastTouch < ttlMs`, so pruning does not run again inside the TTL window.
  - file: [src/agents/pi-hooks/context-pruning/extension.ts:18](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/agents/pi-hooks/context-pruning/extension.ts#L18)

- **Docs still describe TTL-first behavior:** The user-facing docs say session pruning waits for cache TTL expiry before pruning, matching the issue report rather than a rolling-per-turn mode.
  - file: [docs/concepts/session-pruning.md:30](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/docs/concepts/session-pruning.md#L30)
