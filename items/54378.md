## Summary

Keep open. Current main still uses the reported Control UI add-attachment path: FileReader reads images as data URLs, stores them in ChatAttachment.dataUrl, and assigns the resulting array into chatAttachments state. The related #46123/#52361 evidence covers the later chat.send/gateway stack-overflow path, not the immediate browser error when adding an image before it attaches. I did not find a UI-specific source change, object-URL/raw storage replacement, or regression test proving #54378 is fixed.

## Best Possible Solution

Keep #54378 open and treat it as a Control UI attachment-add bug, separate from the already-addressed chat.send/gateway reports. The maintainer path should be to add a focused browser/UI regression around selecting or dropping a large image, then either confirm current main no longer reproduces with evidence or move image payloads out of chatAttachments reactive state, for example by using object URLs or a raw attachment/blob store keyed by id and serializing to base64 only at chat.send time.
