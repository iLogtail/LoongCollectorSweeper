## Summary

Current main still replays all pending outbound delivery queue entries on gateway startup. I found retry/backoff, max retry, and wall-clock recovery budget handling, but no age-based expiry or max replay count that would prevent a 15-hour-old retryCount=0 entry from being resent.

## Best Possible Solution

Keep the issue open and implement an outbound delivery recovery freshness policy in `src/infra/outbound/delivery-queue-recovery.ts`: check `enqueuedAt` before replay, move/drop expired entries with clear logging, optionally cap per-startup replay volume/config if maintainers want that surface, and add regression tests for old retryCount=0 pending entries not being delivered after gateway restart.
