## Summary

Keep #39138 open. Current main still replays pending outbound delivery queue entries on gateway startup without a stale-age cutoff, max replay count, or user-facing delivery recovery policy. The prior ClawSweeper comment reached the same conclusion, and the current checkout still has the same product gap despite later unrelated main changes.

## Best Possible Solution

Implement a conservative outbound delivery recovery policy in src/infra/outbound that expires or moves stale queued deliveries to failed before replay, with focused tests for old retryCount=0 entries, maxRecoveryMs interaction, and restart recovery. If maintainers want operator control, add documented config/schema for max age and possibly max replay count while preserving recovery for recent crash-window sends.
