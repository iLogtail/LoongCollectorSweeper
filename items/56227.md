## Summary

Keep open. Current main has added useful restart guards, but it still does not implement the requested atomic swap/rollback behavior: the existing server is closed before a replacement is started, and a failed post-restart startup can leave the process alive but with no Gateway listener or channel transport.

## Best Possible Solution

Keep the issue open and design a conservative safe-restart path: validate restart-required config before stopping, start a candidate Gateway or equivalent activation check, verify health/readiness, then only switch traffic and stop the old runtime. If true atomic swap is not feasible for the single-port service, add a narrower rollback/rescue design and regression tests around failed restart startup, plugin activation failure, and port/listener conflicts.

## Risks / Open Questions

- Atomic replacement may be non-trivial because the Gateway uses a single configured port; a real fix likely needs a candidate instance on an alternate port, supervisor coordination, or a narrower last-known-good restart design.
- Existing preflight/reload safeguards could make some invalid-config reproductions obsolete, so the remaining work should focus on startup/plugin/port/health failures that occur after the old listener would currently be closed.
