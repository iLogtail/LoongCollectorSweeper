## Summary

Keep #72080 open. The report is recent, specific, and not maintainer/protected-label guarded, and current main still treats ACP runtime turn completion as the task-success signal that wakes the parent. I found no current guard that distinguishes the reported synthetic zero-usage OpenClaw ACP transcript record from a real downstream harness-completion signal before `Background task done` delivery.

## Best Possible Solution

Keep #72080 open and fix the ACP runtime/task lifecycle boundary. ACP child task success should depend on an authoritative harness/ACP terminal condition, not the synthetic `acp-runtime` transcript record or an early zero-usage `stop` placeholder. Add a regression test that simulates initial ACP text/placeholder completion while backing harness work remains active, and assert `sessions_yield`/task delivery does not wake the parent until the true completion signal arrives.
