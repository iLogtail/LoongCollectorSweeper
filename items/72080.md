## Summary

Keep #72080 open. Current main still treats an ACP child run as successful when the OpenClaw/acpx `runTurn` iterator returns and the lifecycle end event is emitted; there is no current guard that distinguishes an `acp-runtime` zero-usage transcript placeholder from authoritative harness completion. The report is fresh, specific, and not superseded by the related ACP reliability items in the provided context.

## Best Possible Solution

Keep the issue open and fix the ACP/acpx completion boundary. Add a regression where an ACP runtime emits preview text and then returns or times out while the harness is still active; do not mark the child task/lifecycle as succeeded until an authoritative ACP completion signal, `agent.wait`/status result, or equivalent harness-settled proof arrives. If the runtime boundary times out or returns ambiguously, surface `failed`/`timed_out` instead of delivering `Background task done`, and do not treat zero-usage `acp-runtime` transcript entries as terminal proof.
