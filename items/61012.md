## Summary

Keep open. Current main still has the exact Telegram message-action/defaultAccount gap described in #61012: the CLI `openclaw message send` path can reach the Telegram action runtime with no `accountId`, and that runtime resolves tokens via `resolveTelegramToken()` rather than the default-account-aware `resolveTelegramAccount()` path. The linked replacement PR #61144 was closed unmerged in the provided GitHub context, and I did not find its fix on current main.

## Best Possible Solution

Keep #61012 open and land the narrow code-path fix from the linked PR direction: make Telegram action-runtime token resolution use the same `resolveTelegramAccount()` default-account fallback as `sendMessageTelegram()`, then add regression coverage in `extensions/telegram/src/action-runtime.test.ts` for omitted `accountId` with `channels.telegram.defaultAccount` across token-gated actions. After that, decide separately whether duplicate-token validation or extra startup/docs warnings are still needed.
