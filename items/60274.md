## Summary

Keep #60274 open. Current main still lacks tapback filtering in the imsg RPC iMessage path: inbound processing dispatches non-empty messages after the existing sender/access/echo/reflection/mention gates, but there is no tapback/reaction branch. The payload type and parser also do not model the imsg reaction fields proposed in the report, and the RPC subscription only passes the attachments option. The related PR #60677 is the focused fix path, but it is open/unmerged in the provided GitHub context, so the bug remains actionable on main.

## Best Possible Solution

Keep the issue open until the imsg plugin implements a minimal tapback filter or reaction-event path. The narrow product/code path is to add typed optional imsg reaction fields to extensions/imessage/src/monitor/types.ts, validate/preserve them in parse-notification.ts, filter tapbacks in resolveIMessageInboundDecision before dispatch, and add regression tests for explicit tapback flags, associated_message_type ranges, quoted text patterns, removal patterns, and false positives. Broader reaction forwarding should remain with the canonical feature discussion in #39031.
