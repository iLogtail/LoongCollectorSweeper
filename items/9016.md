## Summary

Keep #9016 open. Current main has local cost estimates, transcript/session aggregation, `/usage cost`, and status cost display, but it still does not capture OpenRouter provider-reported `usage.cost`, `usage.cost_details.upstream_inference_cost`, or `prompt_tokens_details.cache_write_tokens` from the final streaming usage chunk and expose that exact per-message cost through agent runtime metadata. The ClawRouter comment is a useful external workaround, and #7006/#9696 are related, but neither supersedes this narrower provider-reported cost request.

## Best Possible Solution

Keep #9016 open. The best implementation is to capture OpenRouter `usage.cost`, `usage.cost_details`, and `prompt_tokens_details.cache_write_tokens` at the OpenAI-compatible streaming transport boundary, store provider-reported cost separately from local pricing estimates in transcript/session usage, and expose it through structured agent runtime metadata such as `session_status` details or a dedicated usage/cost tool. Telegram and other reply footers can then render exact provider cost when available and fall back to existing local estimates otherwise.
