## Summary

Current main still has the reported core failure mode: `models auth login` updates `openclaw.json` through the normal config writer, and that writer still blocks writes when the new two-space JSON payload is less than half the raw previous file size. Main now logs/throws `CONFIG_WRITE_REJECTED` and saves `.rejected.*`, so the exact silent-success wording may be partly obsolete, but the auth login/config update path remains blockable by a semantically equivalent verbose/BOM-formatted config.

## Best Possible Solution

Keep this open. The fix should stay in the generic config-write/auth-login path: compare normalized parsed config semantics, or otherwise exempt formatting/BOM-only shrinkage from the destructive size-drop guard, while still blocking real clobbers. Add a regression test with a UTF-8 BOM and exaggerated PowerShell-style indentation that runs the `models auth login` persistence path and verifies both `auth-profiles.json` and `openclaw.json` metadata/default-model changes persist without requiring `allowDestructiveWrite`.
