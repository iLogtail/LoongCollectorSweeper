## Summary

Keep open. Current main still uses a raw byte-size drop as a blocking config-write signal, and `models auth login` persists provider auth through that same guarded writer with no PowerShell/BOM or semantic-equivalence exemption. Main does now throw `CONFIG_WRITE_REJECTED` and save a rejected payload rather than silently replacing nothing, but the reported central problem remains: a valid config whose authored formatting is much larger can still block OAuth auth config persistence until the user reformats it.

## Best Possible Solution

Keep the issue open and fix the config writer rather than special-casing only OpenAI Codex login. The safest product direction is to make the destructive-write guard semantic-aware for valid configs: compare the parsed previous config and proposed config, allow large byte drops caused only by JSON formatting/BOM normalization plus the intended auth/config mutation, and keep the existing hard rejection for true section loss such as `gateway.mode` removal or broad top-level clobbers. Add a regression test that seeds a valid BOM/verbose-formatted `openclaw.json`, performs an auth-profile config update through the normal `replaceConfigFile`/`writeConfigFile` path, and proves the update persists without weakening destructive-clobber coverage.
