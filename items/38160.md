## Summary

Keep open. Current main partially covers hosted-to-local fallback through the standard `agents.defaults.model.fallbacks` chain, but it does not implement this PR's dedicated `localModelFallback` config or response-level semantic cache. The PR also still has unresolved correctness review comments, so there is no conservative auto-close reason.

## Best Possible Solution

Keep the PR open for maintainer architecture review, but split it before merge. Use or document existing `agents.defaults.model.fallbacks` plus cooldown controls for hosted-to-local fallback unless a specific missing API remains. Redesign the semantic response cache against current memory/plugin architecture or add a deliberate core hook, with fail-open cache errors, correct vector persistence, telemetry preservation, docs, and focused tests. Move the onboarding finalize change to its own PR.

## Risks / Open Questions

- The PR bundles two separate agent features plus an unrelated onboarding change, which makes review and rollback harder.
- A response-level semantic cache can return stale or wrong answers if product rules, cache scope, and invalidation are not designed carefully.
- The existing standard model fallback path may already be enough for the local-fallback portion, so merging a second fallback subsystem could duplicate behavior unless a concrete missing capability is identified.
