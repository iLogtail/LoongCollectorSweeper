## Summary

Keep open. Current main has the narrower `context.open_chat_id` fallback, but it still does not implement this PR's nested Schema 2.0 operator identity support (`operator.user_id.open_id/user_id/union_id`). The PR also has positive maintainer signal as the better base compared with #71692.

## Best Possible Solution

Keep the PR open and adapt it on top of current main: add nested Feishu Schema 2.0 identity extraction for `operator.user_id.{open_id,user_id,union_id}` while preserving the existing `context.open_chat_id` fix and malformed-payload guards, keep the lifecycle regression for the nested identity shape, and update the changelog once without duplicating the existing #71670 entry.

It does not dereference nested `operator.user_id.open_id`, `operator.user_id.user_id`, or `operator.user_id.union_id`, so the Schema 2.0 shape described by the PR remains unsupported on current main.
