## Summary

Keep #71787 open. Current main contains the narrower Feishu Schema 2.0 `context.open_chat_id` fallback, but it still does not normalize nested Schema 2.0 operator identity from `operator.user_id.{open_id,user_id,union_id}`. The PR remains useful, and the discussion includes maintainer signal that this PR is the better base for #71670 than the closed narrower PR #71692.

## Best Possible Solution

Keep #71787 open and land the nested Feishu Schema 2.0 identity normalization on top of current main: extract `operator.user_id.{open_id,user_id,union_id}` and equivalent event-level nested identity fields, preserve existing `context.open_chat_id` handling and malformed-payload guards, keep the lifecycle regression test for the nested callback shape, and update the changelog attribution without duplicating the already-shipped narrow #71670 wording.
