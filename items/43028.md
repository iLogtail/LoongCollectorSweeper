## Summary

Current main has several cron hardening pieces, but the requested circuit-breaker behavior is still not implemented: recurring jobs record consecutive errors, can send failure alerts, and apply retry backoff, but they are not automatically disabled/paused after repeated timeouts. Cron session cleanup and run history also exist only as retention/run-log mechanisms, not as the requested `cron rm --cleanup` or `.learnings/changelog.md` behavior.

## Best Possible Solution

Keep this open or split it into narrower maintainer-scoped work: add an explicit recurring-job circuit breaker policy such as `cron.retry.pauseAfterFailures` or `cron.failurePolicy.autoDisableAfter`, persist/announce the disabled state, add tests for repeated timeout failures, and separately decide whether `cron rm --cleanup` should archive/delete cron base and run sessions immediately or whether documented retention plus `openclaw sessions cleanup` is the intended cleanup model. The changelog/audit part should likely map to the existing JSONL-backed `openclaw cron runs` history unless maintainers explicitly want `.learnings/changelog.md` integration.
