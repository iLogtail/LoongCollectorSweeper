## Summary

Keep #14207 open. Current main still hard-codes inline/non-batch memory indexing concurrency at 4, while the only exposed concurrency setting is `remote.batch.concurrency` for provider Batch API jobs. The strict memory config schema, generated help labels, tests, and public docs still lack an inline indexing worker concurrency knob. Related #68812/#68822 are adjacent open retry/concurrency work, but they are not shipped and the provided discussion does not establish them as a canonical replacement for this specific performance request.

## Best Possible Solution

Keep #14207 open until OpenClaw adds an agent-scoped inline/non-batch memory indexing concurrency setting, wires it through `MemorySearchConfig`, `ResolvedMemorySearchConfig`, strict Zod schema, generated schema help/labels, docs, and focused tests, then uses it in `getIndexConcurrency()` without changing the existing provider Batch API meaning of `remote.batch.concurrency`. Coordinate with #68812/#68822 so retry/backoff and concurrency config land coherently.
