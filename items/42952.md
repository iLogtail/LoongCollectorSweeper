## Summary

Keep #42952 open. Current main still has no Feishu `ignoreFileContent` or equivalent inbound-file-skip setting, while the Feishu inbound path still downloads media/file resources and attaches them to the agent context. The related #42947 context-token issue being closed does not supersede this narrower Feishu file-content control request.

## Evidence

- **Feishu config schema lacks ignoreFileContent:** The shared Feishu config shape includes `mediaMaxMb` and other options, then both account and top-level schemas are `.strict()`. `ignoreFileContent` is not a recognized key, so the reported account-level config is not implemented on current main.
  - file: [extensions/feishu/src/config-schema.ts:156](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/extensions/feishu/src/config-schema.ts#L156)

  - sha: [e640c0a95fcb](https://github.com/openclaw/openclaw/commit/e640c0a95fcbd06a1311f18fc635518d011d45da)
- **Inbound Feishu media is still downloaded:** `resolveFeishuMediaList` treats `file`, `audio`, `video`, `media`, `sticker`, image, and post media as downloadable types. For normal file-like messages it calls `downloadMessageResourceFeishu`, saves the buffer, and returns a media placeholder without checking any ignore flag.
  - file: [extensions/feishu/src/bot-content.ts:336](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/extensions/feishu/src/bot-content.ts#L336)

- **Downloaded media is attached to agent context:** The Feishu message handler always calls `resolveFeishuMediaList` with the configured `mediaMaxMb`, builds `mediaPayload`, and spreads that payload into `finalizeInboundContext`, which is the path that sends inbound context to the agent/model.
