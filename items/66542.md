## Summary

Keep PR #66542 open. Current main still lacks the requested bundle MCP reconnect/retry/dead-server/parallel-startup behavior: startup is sequential and `callTool()` performs one direct SDK call without reconnect handling. The PR targets the right runtime surface and is not obsolete, but its current head has unresolved correctness and security concerns, so it should stay open for revision rather than be auto-closed or treated as ready to land.

## Best Possible Solution

Keep the PR open for maintainer review and request a revised, tested implementation. The best path is to reconnect only on verified transport/session-lost failures, let no-session callers join in-flight reconnects or attempt resurrection after the dead window, bound startup concurrency if needed, avoid leaking launch details or sensitive errors into catalog/tool metadata, and coordinate any narrow streamable-http recovery with #71545 instead of treating it as a full replacement.
