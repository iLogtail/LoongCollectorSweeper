## Summary

Keep open. Current main does not implement this PR’s requested behavior: cache-TTL eligibility still falls through only to provider plugin overrides, Anthropic-family/Kilo Anthropic refs, and Google prompt-cache eligibility. The runtime pruning factory and cache-touch marker both depend on that eligibility function, so generic `openai-completions` providers remain excluded unless a provider plugin opts in. The related ClawSweeper reports cover other cache-TTL or plugin API surfaces, not this exact broad OpenAI-compatible eligibility change.

## Best Possible Solution

Keep PR #69245 open for maintainer review. The likely path is either merge a polished version of the generic `openai-completions` eligibility change with focused tests and docs, or narrow it to provider/plugin-owned opt-ins if maintainers do not want all OpenAI-compatible endpoints to enable cache-TTL pruning by API type alone.
