## Summary

Keep PR #70391 open. Current main still reloads chat history after same-session final chat events because `shouldReloadHistoryForFinalEvent` returns true for every final payload, and the existing test suite still asserts that behavior even when the final event carries an assistant message. The PR addresses a real unresolved Control UI reload amplifier, but the submitted patch needs maintainer review because the discussion flags a deferred `session.message` replay edge case that should be preserved rather than dropped.

## Best Possible Solution

Keep the PR open for maintainer review or fold it into the broader #66875 Control UI reload-race work. The best implementation should skip `chat.history` after a final event only when the final payload is already a displayable assistant message, while still replaying any pending `session.message` reload that may represent unrelated subscribed-session transcript updates. Add regression coverage in `ui/src/ui/chat-event-reload.test.ts` and `ui/src/ui/app-gateway.node.test.ts` for final-with-assistant skip, final-without-message reload, non-assistant reload, and pending `session.message` preservation.

## Risks / Open Questions

- The submitted PR patch appears to suppress deferred `session.message` replay after a final assistant payload; that could hide unrelated transcript updates until a later manual refresh or reconnect.
- The branch was based on older gateway code, so any rework should preserve current guards such as different-active-run handling and the shipped active-run deferred reload behavior.
- Current optimistic-tail preservation may reduce visible flicker severity, but it does not remove the redundant final-event reload trigger this PR targets.
