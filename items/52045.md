## Summary

Keep open. The issue has a concrete reporter follow-up: browser hosting works after install, then `openclaw browser --browser-profile openclaw status` reports the gateway closed after Feishu pairing. Current main still routes browser status through the same `browser.request` gateway method and browser status endpoint, and I did not find a landed fix or regression test proving the Feishu-pairing mismatch is resolved. The directly linked fix PR #52057 was closed unmerged, so it is not a shipped/canonical resolution.

## Best Possible Solution

Keep the issue open until maintainers land a focused fix or designate a canonical tracker. The best path is to reproduce the Feishu pairing flow, identify whether pairing rewrites browser/plugin/gateway auth config or leaves browser control stale, then add a targeted regression test around `openclaw browser status`/`browser.request` after channel setup plus a clear user-facing error when browser control is disabled.
