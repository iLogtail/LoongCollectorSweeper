## Summary

Keep #43605 open. Current main still does not implement native cross-gateway `sessions_send`: the tool schema has no per-call gateway target fields, the execution path resolves and sends through the current gateway/session store, and config has `gateway.remote` but no `gateway.peers` or remote-agent registry. The linked PR #43656 is relevant but remains open/unmerged; the xihe-jianmu IPC MCP hub mentioned in related issues is a useful external workaround, not a replacement for the requested native `sessions_send` behavior and session-history integration.

## Best Possible Solution

Keep #43605 open as the canonical native cross-gateway session-messaging tracker. The best maintainer path is to review or replace #43656 with a core design that adds an explicit `gateway.peers` or equivalent config contract, routes `sessions_send` through remote gateway calls with clear auth and visibility semantics, preserves session-history behavior, documents the multi-gateway topology, and adds focused tests for local fallback, peer resolution, auth failure, and remote send success. Keep `xihe-jianmu-ipc` as a useful optional ClawHub/MCP workaround, not as the core fix.
