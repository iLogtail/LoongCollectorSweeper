## Summary

Keep open. Current `main` still resolves `heartbeat.target` by calling `normalizeDeliverableOutboundChannel(rawTarget)` without any plugin-bootstrap path, so plugin-only channel ids can still collapse to `none`. The PR's proposed approach is not present on `main`, and the provided review comments correctly flag the earlier version as ineffective because it depended on the same normalization gate that already failed.

## Evidence

- **Heartbeat target resolution on main still has no bootstrap retry:** `resolveHeartbeatDeliveryTarget` only accepts explicit string targets when `normalizeDeliverableOutboundChannel(rawTarget)` succeeds. There is no fallback that loads plugin channel metadata before deciding `target = "none"`.
  - file: [src/infra/outbound/targets.ts:97](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/infra/outbound/targets.ts#L97)

- **Normalization still depends on already-registered deliverable channels:** `normalizeDeliverableOutboundChannel` returns `undefined` unless `isDeliverableMessageChannel(normalized)` already knows the channel id. It does not accept config and does not bootstrap plugin registration first.
  - file: [src/infra/outbound/channel-resolution.ts:19](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/infra/outbound/channel-resolution.ts#L19)
