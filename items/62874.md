## Summary

Keep open. Current `main` does not show a shipped fix for vLLM/Qwen tool-call auto mode: the OpenAI-compatible completions transport still omits `tool_choice` unless something explicitly sets it, and downstream assistant-tool-call extraction still depends on structured tool-call blocks rather than recovering plain-text/XML tool calls from assistant content. The docs also still frame this class of failure as backend incompatibility or a reason to disable tools, not as a resolved OpenClaw behavior.

## Evidence

  - file: [src/agents/openai-transport-stream.ts:1740](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/agents/openai-transport-stream.ts#L1740)

- **Current test suite locks in the vLLM no-tool_choice default:** The transport test for a `vllm` OpenAI-completions model asserts that when tools are present and no explicit toolChoice option is set, the request must not contain `tool_choice`; a separate test shows `tool_choice: "required"` only when explicitly configured.
  - file: [src/agents/openai-transport-stream.test.ts:2081](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/agents/openai-transport-stream.test.ts#L2081)
