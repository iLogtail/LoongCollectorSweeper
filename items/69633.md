## Summary

Keep PR #69633 open. It is a recent draft adding a standalone SPA bridge module for a proposed WebView2 host contract; current main does not already implement WebView2 native bridge support, and the related host-side work is in external openclaw-windows-node#192 rather than this repository. This is also not a safe ClawHub close because current plugin surfaces document gateway/plugin runtime routes and capabilities, not a documented way for an external plugin to inject or own Control UI SPA bootstrap behavior.

## Best Possible Solution

Keep the PR open for maintainer/product judgment. If OpenClaw wants first-class Windows WebView2 hosting, the best path is a tested core Control UI bridge contract with lifecycle hookup, message validation, cleanup, and docs aligned with the Windows companion host. If maintainers instead want this outside core, first define or document the supported external/plugin injection boundary, then close or redirect this PR accordingly.
