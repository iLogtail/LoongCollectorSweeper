## Summary

Keep open. Current `main` does not implement per-channel+recipient outbound serialization, so there is not strong evidence to close this PR as already fixed or obsolete. The live outbound path only persists a queue entry and claims that entry id; it does not lock same-target sends together, and the recovery drain ordering only applies to replayed queued entries, not concurrent live deliveries.

## Evidence

- **Live outbound path has no same-target serializer:** `deliverOutboundPayloads` enqueues the message, then claims only the resulting `queueId`, and immediately calls `deliverOutboundPayloadsCore`. There is no per-target mutex/queue in this path.
  - file: [src/infra/outbound/deliver.ts:644](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/src/infra/outbound/deliver.ts#L644)
  - sha: [4e4aeacae4a8](https://github.com/openclaw/openclaw/commit/4e4aeacae4a87f2fe09125d462d88cd71a18cb66)
- **Current ordering logic is recovery-only:** `drainPendingDeliveries` sorts pending entries by `enqueuedAt` before replay, but that ordering is scoped to recovery/drain flows and does not serialize concurrent live sends to the same recipient.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:251](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/src/infra/outbound/delivery-queue-recovery.ts#L251)
  - sha: [4e4aeacae4a8](https://github.com/openclaw/openclaw/commit/4e4aeacae4a87f2fe09125d462d88cd71a18cb66)
- **No serializer module exists on main:** The outbound directory on current `main` contains delivery queue files, but no `delivery-serializer.ts` or equivalent per-recipient outbound serialization helper.
  - command: `find src/infra/outbound -maxdepth 1 -type f | sort`
  - sha: [4e4aeacae4a8](https://github.com/openclaw/openclaw/commit/4e4aeacae4a87f2fe09125d462d88cd71a18cb66)
- **Existing ordering fix is for a different path:** Current `main` explicitly serializes tool-result delivery in `agent-runner-execution.ts`, which reinforces that outbound ordering is treated as a separate concern and is not covered by `deliverOutboundPayloads` today.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:1244](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/src/auto-reply/reply/agent-runner-execution.ts#L1244)
  - sha: [4e4aeacae4a8](https://github.com/openclaw/openclaw/commit/4e4aeacae4a87f2fe09125d462d88cd71a18cb66)
- **Open review concerns mean this is not a clear stale close:** The PR discussion still contains substantive review concerns about where the lock is acquired and how the key is derived (queue-before-lock ordering, account/recipient normalization, thread identity). That argues for maintainer review or follow-up work, not conservative closure.

## Risks / Open Questions

- The reported race remains plausible on current `main` because concurrent sessions can still reach `deliverOutboundPayloads` independently for the same target.
- If this PR is revived, it likely needs follow-up on serializer key canonicalization and on whether ordering should begin before `enqueueDelivery(...)` to match send initiation order.
