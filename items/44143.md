## Summary

Current main still does not serialize live outbound deliveries per channel/account/recipient. The PR targets a real remaining gap in the shared outbound delivery path, and the discussion includes useful reproduction evidence. It should not be cleanup-closed, though it needs maintainer review/rework before merge because review comments identify ordering/keying/abort/timeout concerns and the branch includes unrelated Mattermost dependency changes.

## Best Possible Solution

Keep the PR open and ask for a current-main rework: implement a canonical per-destination outbound delivery lane/serializer around the live send path with normalized channel/account/recipient/thread identity, acquire ordering early enough to preserve initiation order, honor aborts while queued, protect against hung adapter calls where feasible, add focused ordering/concurrency/error/cleanup tests, and split unrelated Mattermost dependency changes into their own PR.

## Risks / Open Questions

- The PR as submitted is not obviously merge-ready: locking after enqueue can still invert true initiation order, raw channel/account/recipient strings can produce key misses or collisions, and thread identity is not part of the proposed key.
- Adding a per-recipient mutex can create head-of-line blocking if a channel adapter send hangs; Discord webhook delivery currently uses fetch without an explicit timeout in extensions/discord/src/send.outbound.ts.
- The branch includes unrelated Mattermost typebox dependency/lockfile changes, so maintainers should ask for a narrow rebase or split before landing.
