## Summary

Keep #56850 open. Current main still models cron schedules as static `at`, fixed `every`, or cron-expression schedules, with next-run computation based on those schedule shapes plus error backoff/staggering. There is no declarative `conditional_interval`/deadline-aware scheduler path in the core cron type, normalizer, CLI/docs, or timer flow. The linked zeroclaw-labs PR in the issue timeline is useful related context, but it is not shipped in OpenClaw main.

## Best Possible Solution

Keep the issue open as a core cron scheduling/API design request. A strong implementation would add a backwards-compatible typed schedule or policy for conditional intervals, define a safe condition/evaluation contract, update normalization/protocol/CLI/UI/docs, and add scheduler tests proving `nextRunAtMs` switches between the base and conditional intervals without skipping due runs. If maintainers prefer a plugin-owned approach, first define the missing plugin hook/API for next-run policy rather than closing this as already supported.
