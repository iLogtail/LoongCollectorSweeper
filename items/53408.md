## Summary

Keep open. Current main has important mitigations for malformed tool-call arguments, including boundary-aware transports and a targeted repair wrapper, and the linked Kimi reports (#53591/#53735) appear addressed for that provider family. I did not find enough evidence to close this specific copilot-api/custom Anthropic-compatible long-session report: the repair gate is still provider/API-specific, and there is no regression coverage proving long streamed Anthropic tool arguments with large CJK write content survive end to end for arbitrary `anthropic-messages` proxies.

## Best Possible Solution

Keep the issue open and narrow the next maintainer step to a reproducible transport fixture: capture or synthesize an Anthropic Messages stream for `write`/`exec` with long CJK content across many `input_json_delta` chunks, then assert current main either preserves the complete arguments or fails explicitly with a malformed/incomplete tool-call diagnostic. If the fixture shows the Kimi/openai-completions repair is the right mechanism, make the activation depend on a documented provider capability or API-format compatibility flag rather than a hard-coded provider id.

## Risks / Open Questions

- The remaining root cause could be in the upstream proxy/provider rather than OpenClaw; a raw SSE transcript or recorded fixture is needed to separate provider truncation from OpenClaw parsing/repair behavior.
- Blindly enabling the current repair wrapper for every `anthropic-messages` provider could regress legitimate large write payloads because that repair path has its own bounded buffer and was designed for malformed wrapper text, not general long-content transport correctness.
