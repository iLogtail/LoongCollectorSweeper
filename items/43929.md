## Summary

Keep #43929 open. Current main still exposes `chat.history` as current-session-only: the protocol has no `scope`/logical mode, the handler reads only the current session entry's current transcript, and the transcript resolver does not stitch `.jsonl.reset.*` archives. The related #45003 discussion is useful broader archive-recovery context, but it does not supersede this specific Gateway logical-history API request.

## Best Possible Solution

Keep the issue open and implement an additive Gateway history contract. Preserve current `chat.history` behavior by default, add a logical scope or equivalent option in the protocol schema, resolve ordered reset/daily/idle/manual transcript segments for the canonical logical session key, return lightweight segment metadata (`sessionId`, start/end timestamps, reset reason), and cover it with focused Gateway tests plus Control UI/WebChat wiring. Align `sessions_history` with the same core reader so agent-facing history and UI history do not diverge.
