## Summary

Current `main` does not implement logical chat history across rolled sessions. `chat.history` still accepts only `sessionKey`, `limit`, and `maxChars`, resolves the freshest current session entry, and reads messages from that single current `sessionId`. Reset flow rotates to a new `sessionId` and archives the old transcript on disk, but I found no RPC/HTTP surface that stitches archived segments back together or returns segment metadata for one logical session.

## Evidence

- **`chat.history` API has no logical-history selector:** The protocol schema for `chat.history` only defines `sessionKey`, `limit`, and `maxChars`; there is no `scope`, logical-history flag, or segment-metadata request field.

  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **`chat.history` reads only one current session transcript:** The handler calls `loadSessionEntry(sessionKey)`, takes `entry?.sessionId`, then reads messages only from that single session via `readSessionMessages(sessionId, storePath, entry?.sessionFile)` and returns that `sessionId` plus `messages`.
  - file: [src/gateway/server-methods/chat.ts:1874](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/gateway/server-methods/chat.ts#L1874)

That means the active mapping points at the newest rolled session, not an aggregate across prior segments.

Archived transcripts are renamed on disk with a reset suffix, but I found no history endpoint that follows that chain for clients.
