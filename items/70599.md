## Summary

Keep open. Current main still does not implement the specific cron/embedded path where the latest successful tool result is exact `NO_REPLY` and the final assistant message is empty, but the PR head also has an acknowledged provider-error regression that should be fixed before merge.

## Best Possible Solution

Keep the PR open, but require a revision before merge: inspect only current-attempt successful tool results, treat exact `NO_REPLY` as silent only for clean terminal states, preserve existing provider-error/tool-error/abort/timeout handling, and add regression coverage for both the quiet-success path and the provider-error case. If another PR lands the same corrected behavior first, close this one as superseded by that canonical item.
