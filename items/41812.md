## Summary

Keep open. Current main still lets `openclaw models set` persist any syntactically valid `provider/model` string and create an empty allowlist entry, while `/model` session switching uses catalog/allowlist resolution. PR #41812 is not merged or superseded, but it also needs maintainer review because the latest PR revision changed the proposed behavior from rejection to warning and did not add tests.

## Best Possible Solution

Keep the PR open for maintainer/product review. The best path is a focused update to `src/commands/models/set.ts` plus regression tests that deliberately define the intended semantics: either warn on catalog misses while preserving custom allowlist entries, or add an explicit strict validation mode that rejects only when catalog evidence is authoritative. The empty `{}` entry/routing concern from #24243 also needs to be resolved or explicitly documented as intentional allowlist behavior.
