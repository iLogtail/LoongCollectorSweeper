## Summary

Current main already avoids the reported manual isolated cron self-deadlock by remapping nested cron-triggered agent work off the `cron` lane, so this PR’s separate `cron-dispatch` lane is obsolete.

## Evidence

## Close Comment

Closing as implemented on `main` based on this Codex review. The current manual `cron.run` path still queues on `CommandLane.Cron` (`src/cron/service/ops.ts:697`), but isolated cron execution no longer re-enters that same lane: gateway passes `lane: "cron"` into the isolated runner (`src/gateway/server-cron.ts:333`), and the current nested-agent lane resolver remaps `cron` to `nested` specifically to avoid cron self-deadlocks (`src/agents/lanes.ts:8`, used by `src/cron/isolated-agent/run-executor.ts:153`). The shipped changelog for `v2026.3.13` also records this behavior change at `CHANGELOG.md:2555` (“route nested cron-triggered embedded runner work onto the nested lane so isolated cron jobs no longer deadlock…”). Given that current `main` already avoids the reported deadlock, this PR’s separate `cron-dispatch` lane is obsolete.
