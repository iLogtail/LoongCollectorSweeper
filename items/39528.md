## Summary

Keep open. Current main has meaningful hardening for the owner-only gateway tool, protected config paths, dangerous flags, and prompt guidance, but it still does not fully address the reported behavior. The model-facing gateway restart action can still schedule a restart when restart commands are enabled, and config writes remain allowed for a narrow set of prompt/model/mention-gating paths without a hard explicit-consent token. The context-loss portion also lacks enough reproduction or source-backed fix evidence to close as implemented or cannot-reproduce.

## Best Possible Solution

Keep tracking this as a core behavior/security hardening bug. The best fix is an enforceable consent boundary for model-facing gateway restart/config/update actions, not only prompt guidance: either require a current-turn explicit confirmation token for restart and config writes, or default-deny those gateway actions from autonomous model use unless a maintainer-approved policy enables them. Add regression tests for restart consent, config.patch/config.apply consent, allowed-path behavior, and context retention across compaction/long conversations; link narrower follow-ups such as the config response context-bloat issue if they become the canonical implementation path.
