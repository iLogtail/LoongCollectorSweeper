## Summary

Keep open. Current main still serializes ordinary same-session Telegram messages and holds the embedded run's per-session lane until the full run promise resolves, not until the final reply is delivered. The related failover work has improved provider escalation/profile rotation, but I did not find source or test evidence that #56352's requested lane-lifecycle decoupling is implemented.

## Best Possible Solution

Keep #56352 open. The best maintainer path is to add a focused regression where a fallback attempt delivers a final Telegram/block reply and then stalls in cleanup/error propagation, while a same-session inbound message arrives. Decide whether the product should truly release the session lane at final delivery; if yes, introduce an explicit post-final-delivery boundary that preserves transcript/session write locking and moves remaining cleanup out of the user-facing lane. If maintainers instead want strict one-run-per-session until all cleanup completes, document that as intentional and consider linking this to the busy-ack/session-queue UX work rather than closing it as implemented.
