## Summary

Keep open. Current main has `session.sendPolicy`, which is a useful partial workaround for suppressing normal session replies while still processing inbound messages, but it is not the requested per-account channel read-only mode and it does not provide a hard shared outbound block across all channel delivery surfaces. WhatsApp still has direct pre-dispatch outbound pairing replies, and message/send tool paths still go straight to outbound delivery without `resolveSendPolicy`.

## Best Possible Solution

Keep the issue open and implement a first-class channel/account receive-only policy, either as `channels.<channel>.accounts.<id>.readOnly` or an equivalent account-aware send policy. The enforcement should live at a shared outbound gate and be honored by normal auto-replies, gateway `send`, the message tool, polls/reactions/message actions, recovery/cron delivery where applicable, and channel-specific pre-dispatch sends such as WhatsApp pairing notices. Add config schema/docs and regression tests for at least one direct reply, one message-tool send, and one channel-specific pre-dispatch send.
