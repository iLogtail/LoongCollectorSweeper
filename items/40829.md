## Summary

Keep open: current `main` does not implement this PR's two requested behaviors, and the review thread also raises an unresolved validation tradeoff that still needs maintainer judgment rather than stale cleanup.

## Evidence

- **Current main still converts non-alpha PNGs through JPEG:** `optimizeImageWithFallback` identifies PNGs, checks alpha, and only preserves PNG when alpha exists and the PNG optimization fits under cap. Otherwise it falls through to `optimizeImageToJpeg`, so there is still no fast path that preserves an already-under-cap non-alpha PNG.
  - file: [src/media/web-media.ts:335](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/media/web-media.ts#L335)
  - command: `nl -ba src/media/web-media.ts | sed -n '329,351p'`
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Current main only renames JPEG outputs for HEIC sources:** `loadWebMediaInternal` still rewrites the filename to `.jpg` only when `optimized.format === "jpeg"` and `isHeicSource(meta)` is true. A PNG that gets JPEG-encoded would still keep its original filename on `main`.
  - file: [src/media/web-media.ts:401](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/media/web-media.ts#L401)
  - command: `nl -ba src/media/web-media.ts | sed -n '388,406p'`
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Main test file has no matching regression coverage:** The current `src/media/web-media.test.ts` covers local-path/media-store cases, but there are no assertions for preserving under-cap PNGs or renaming JPEG-fallback outputs from `.png` to `.jpg`; the relevant search terms do not appear in the file.
  - command: `rg -n "under-cap|contentType\)\.toBe\(\"image/png\"\)|toBe\(path.basename\(|tinyPngWrongExtFile|fallbackPngFile|jpg\"\)" src/media/web-media.test.ts`
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)

The contributor replied that they considered that an acceptable pre-existing risk, so the tradeoff is still open for maintainer review rather than cleanup closure.
