## Summary

Keep #50081 open. Current main still has a plausible current-run Control UI/webchat HEIC path where inline attachments are parsed as `image/heic`, passed through `images`/`imageOrder`, returned unchanged when the prompt has no detected image refs, and then submitted to `activeSession.prompt(..., { images })`. The HEIC-to-JPEG normalization that exists today is on the OpenResponses/input-file helper path, not this chat prompt handoff. The linked PR #50320 is directly relevant but is open/unmerged in the provided GitHub context, so the issue remains the right tracker.

## Best Possible Solution

Keep the issue open until the chat/current-run prompt-image path is fixed. The best fix is to share or extract the existing HEIC/HEIF-to-JPEG normalization and apply it before provider submission for all prompt images, including inline `existingImages` when the prompt has no detected refs, then add focused regression coverage for Control UI/chat attachments and `detectAndLoadPromptImages` so an inline `image/heic` attachment becomes `image/jpeg` before `activeSession.prompt`. PR #50320 is a relevant implementation candidate if maintainers want to revive or supersede it.
