## Summary

Keep open. Current main still matches the reported bug: when an SSH tunnel starts, the probe adds the tunnel target and only filters base targets by exact URL, so a direct `gateway.remote.url` for the same gateway remains a second target. The warning builder then emits `multiple_gateways` for any `reachable.length > 1` and does not compare the `self` identity already extracted from presence. The issue is recent, concrete, has no protected maintainer signal, and the provided related items cover different gateway-probe false-negative/timeout paths rather than superseding this false-positive warning.

## Best Possible Solution

Keep the issue open and implement an identity-aware warning path in the gateway status/probe output layer. Collapse reachable targets that share a reliable gateway self identity before emitting `multiple_gateways`, or downgrade same-identity duplicates to clearer “multiple transports to one gateway” copy. Add focused tests for `sshTunnel` plus `configRemote` with the same `self`, and update the gateway probe docs warning semantics after the behavior changes.
