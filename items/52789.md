## Summary

Keep #52789 open. Current main still does not implement `delivery.mode: "session"` for cron jobs: cron delivery is still typed, parsed, documented, and exposed through the CLI as `announce`, `webhook`, or `none`. The existing `sessions_send` / `sessions.send` path exists, but it remains a model/tool or gateway RPC path rather than a cron-runner-owned delivery mode after job completion. The related ClawSweeper context does not provide a canonical duplicate or shipped fix for this specific feature.

## Best Possible Solution

Keep the issue open for maintainer/product design. The most direct implementation would be a first-class core cron delivery mode, e.g. `CronDeliveryMode` adds `"session"`, normalization preserves it, validation defines target-session semantics for `delivery.to`, finalization calls the same trusted gateway/session-send machinery used by `sessions_send`/`sessions.send` with idempotency and provenance, and docs/CLI/tests cover session delivery. If maintainers prefer extension-first, the prerequisite would be an explicit cron completion delivery extension point; current plugin hooks do not appear to provide a stable way to add a new cron delivery mode without relying on model-owned tool calls.
