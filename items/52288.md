## Summary

Keep open. Current main still matches the reported bug: the embedded runner stores only one client tool call, downstream wraps that single value into a one-element pendingToolCalls array, and the /v1/responses HTTP layer emits only pendingToolCalls[0] in both non-streaming and streaming paths. The linked fix PR #52287 is closed and unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep #52288 open and land an updated version of #52287 or equivalent: accumulate client tool calls in attempt.ts, expose an array in the attempt/result types, map every pending tool call into OpenResponses function_call output items for stream and non-stream responses, and add regression tests covering three client tools in one turn plus follow-up function_call_output handling.
