## Summary

Keep open. Current main still lacks the reported dedup boundary: exec approval requests are keyed by approval ID, the exec tool generates a fresh UUID for each approval request, Telegram/native delivery suppresses only duplicate events with the same ID or duplicate delivery targets, and `enqueueSystemEvent` only suppresses consecutive identical text rather than coalescing by run/context intent. The related fix PRs in the provided context were closed unmerged, and the issue is recent with concrete reproduction data.

## Best Possible Solution

Keep the issue open and fix it with an approval-intent dedup/cancellation contract: derive a stable pending-approval fingerprint from host/node, agent/session, command or systemRunPlan, cwd/env binding, and turn/source context; return or reuse the existing pending approval instead of registering a new ID for the same unresolved intent, or cancel prior pending approvals when run failover retries the same intent. Add regression tests for duplicate approval requests under fresh IDs and for Telegram/native delivery emitting only one prompt for that pending intent.
