## Summary

Current main does not implement the requested deduplication. `enqueueSystemEvent` still suppresses only consecutive identical text, exec approval request contexts still get a fresh random approval ID/context key, and the channel approval runtime deduplicates only by that request ID. The related fix PRs in the provided context (#69731 and #69893) were closed unmerged, while #69482 is a distinct approvals schema bug, so there is no shipped or canonical replacement that would justify closing #69478.

## Best Possible Solution

Keep #69478 open and fix it in core. The likely path is to introduce stable idempotency for exec approval prompts based on the run intent, agent/session, command, and context key, or to cancel pending exec approval events when a run fails over before the replacement can enqueue another prompt. Add focused regression coverage for non-consecutive queued duplicate system events, new-approval-ID retries for the same exec command, and the failover path that originally cascaded Telegram prompts.
