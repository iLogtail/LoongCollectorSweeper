## Summary

Keep open. Current main does not implement the PR's central behavior: channel message bodies still flow into auto-reply prompt assembly as `BodyStripped ?? Body` after lightweight inbound sanitization, while `wrapExternalContent()` is only present for existing external-content surfaces such as hook/web content and channel metadata. The PR discussion also shows useful maintainer-reviewable work, not an obsolete or ClawHub-scoped contribution; the remaining concern is to revise the implementation so provider trust is derived from ingress metadata and internal scheduler providers are not misclassified.

## Best Possible Solution

Keep the PR open and ask for a focused revision: implement channel message and thread-context external-content wrapping in core prompt assembly, derive untrusted status from ingress provider rather than reply-routing metadata, explicitly preserve trusted internal providers (`node`, `cli`, `exec-event`, `heartbeat`, `cron-event` as applicable), keep queue summaries based on the raw user body, and add focused tests around those cases.
