## Summary

Keep open. Current `main` does not implement this PR's fail-closed Mattermost routing change: unresolved channel types still collapse to `"channel"`, the `ChatType` union still cannot represent `"unknown"`, and the current tests still assert the old behavior.

## Evidence

- **Mapper still defaults unresolved channel type to channel:** `mapMattermostChannelTypeToChatType` returns `"channel"` when `channelType` is falsy, so a missing/null/undefined Mattermost channel type is still treated as a public channel on current `main`.
  - file: [extensions/mattermost/src/mattermost/monitor-gating.ts:4](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/mattermost/src/mattermost/monitor-gating.ts#L4)
  - command: `nl -ba extensions/mattermost/src/mattermost/monitor-gating.ts | sed -n '1,80p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Current tests lock in the old behavior:** Both Mattermost tests still expect `mapMattermostChannelTypeToChatType(undefined)` to be `"channel"`, which is the exact behavior this PR is trying to change.
  - file: [extensions/mattermost/src/mattermost/monitor.channel-kind.test.ts:15](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/mattermost/src/mattermost/monitor.channel-kind.test.ts#L15)

- **Routing path still consumes the collapsed channel kind:** `resolveSessionKey` and `dispatchButtonClick` both call the mapper and then immediately build routing state from that result. With the current mapper fallback, an unresolvable channel type still flows through as `"channel"` instead of failing closed.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:586](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/mattermost/src/mattermost/monitor.ts#L586)

Current `main` therefore has no typed `unknown` state for this failure mode.
