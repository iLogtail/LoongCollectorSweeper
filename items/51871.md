## Summary

Keep #51871 open. The provided discussion narrowed the bug to malformed persisted cron jobs where `schedule` or `payload.kind` is missing. Current `main` still has Control UI render/filter/edit paths that dereference `job.schedule.kind` and `job.payload.kind` directly, and the cron store load path still casts loaded `jobs.json` entries to `CronJob[]` without rejecting or fully canonicalizing invalid jobs before the UI can receive them.

## Best Possible Solution

Keep the issue open and fix the product boundary rather than relying on manual JSON repair. The best path is to harden the cron list contract at the gateway/store layer so malformed persisted jobs are normalized when possible or returned as explicit invalid rows, and to make the Control UI render valid jobs even if one row has missing `schedule` or `payload`. Add regression coverage with a `jobs.json` entry missing `schedule`/`payload.kind` and verify the dashboard/list path does not crash.

## Evidence
