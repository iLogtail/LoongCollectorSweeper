## Summary

PR #70900 should stay open. Current main still has the exact surface_error throw shape the PR targets, and the proposed regression test is not present. The related #70848/#70124 work shipped the provider-error surfacing path, but it did not add the `failoverFailure` gate needed to avoid throwing on a classified stale errorMessage from an otherwise successful assistant turn.

## Best Possible Solution

Keep PR #70900 open and review it as a focused bug fix. The likely best path is to land the `params.failoverFailure` gate in `src/agents/pi-embedded-runner/run/assistant-failover.ts` with the proposed regression test in `assistant-failover.test.ts`, while preserving #70124/#70848 behavior for genuine `stopReason === "error"` provider failures and the existing external-abort/timeout fallthroughs.
