## Summary

Current main no longer matches the reported stale-socket restart path for iMessage. The iMessage gateway runtime only seeds `cliPath`/`dbPath` status, and the health monitor can only emit `stale-socket` when a channel reports both `connected === true` and a non-null `lastTransportActivityAt`. Repo tests explicitly assert that channels without transport tracking are treated as healthy.

## Evidence

- **iMessage runtime status update is minimal:** `startIMessageGatewayAccount()` only calls `ctx.setStatus({ accountId, cliPath, dbPath })` before starting the monitor; it does not publish `connected`, `lastEventAt`, or `lastTransportActivityAt`.
  - file: [extensions/imessage/src/channel.runtime.ts:70](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/extensions/imessage/src/channel.runtime.ts#L70)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Stale-socket requires transport-liveness fields:** `evaluateChannelHealth()` only enters stale-socket logic when `snapshot.connected === true` and `lastTransportActivityAt != null`. Without those fields, the health monitor will not classify the channel as stale-socket.
  - file: [src/gateway/channel-health-policy.ts:117](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/gateway/channel-health-policy.ts#L117)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Tests lock in non-tracking channels as healthy:** The health-policy tests explicitly verify that a connected channel with `lastTransportActivityAt: null` is healthy rather than stale.
  - file: [src/gateway/channel-health-policy.test.ts:214](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/gateway/channel-health-policy.test.ts#L214)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Health-monitor tests also skip restart without transport liveness:** The health-monitor test suite expects no restart for a connected channel that does not report transport activity timestamps.
  - file: [src/gateway/channel-health-monitor.test.ts:543](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/gateway/channel-health-monitor.test.ts#L543)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **No current iMessage code publishes transport-liveness fields:** Repository search found no non-test iMessage code writing `connected`, `lastEventAt`, or `lastTransportActivityAt`; only the startup `ctx.setStatus(...)` call exists.

## Close Comment

Closing after Codex review because the exact bug report does not match current `main`. On current `main` (`bfa6708c`), iMessage startup only sets `cliPath`/`dbPath` in `extensions/imessage/src/channel.runtime.ts:70`, and the gateway stale-socket path only triggers when a channel reports both `connected === true` and a non-null transport timestamp in `src/gateway/channel-health-policy.ts:117`. The test suite also explicitly treats channels without transport tracking as healthy in `src/gateway/channel-health-policy.test.ts:214` and `src/gateway/channel-health-monitor.test.ts:543`. Based on that, I could not substantiate the reported `health-monitor: restarting (reason: stale-socket)` loop against current main. If this is still happening on a current build, please open a new issue with fresh `/tmp/openclaw/openclaw-*.log` lines, `openclaw health --json`, and the relevant `channels.imessage` health-monitor config so maintainers can investigate the current code path.
