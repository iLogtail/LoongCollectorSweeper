## Summary

Keep #70330 open. Current main still routes WebChat New Session and typed `/new`/`/reset` through the generic `chat.send` path, the gateway still performs reset rotation from slash text inside the agent handler, and there is no explicit `allowSessionControl` parameter or persisted user-visible reset audit marker. Related #59839, #23755, and #70472 cover adjacent symptoms, but none cleanly supersedes this restart/reconnect-triggered silent rotation report.

## Best Possible Solution

Keep #70330 open and treat it as a real WebChat/session-continuity bug. The best fix is to move WebChat New Session/reset onto an explicit session-control RPC or explicit `chat.send` session-control flag, record a visible/auditable reset event with old/new session ids and trigger source, clear stale queued reset/draft/idempotency state on reconnect, and make reset archives/checkpoints discoverable or restorable from the Control UI. Cross-link #59839 and #70472 as related symptoms, but do not close this report as superseded until one canonical issue or PR covers the restart/reconnect failure mode.
