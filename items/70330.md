## Summary

Keep #70330 open. Current main has partial reset lifecycle improvements, but it still leaves the reported WebChat safety issue unresolved: WebChat New Session still goes through `/new` over the generic chat path, gateway `agent` still treats `/new`/`/reset` as session-control commands and can strip a post-reset visible message, resets still rotate the active session id and archive the prior transcript, and the UI surfaces compaction checkpoints rather than reset archives/history. The related open issues cover adjacent behavior, not a clear canonical supersession for this restart/reconnect-triggered report.

## Best Possible Solution

Keep the issue open and fix it in the WebChat/gateway session-control boundary: move the New Session UI off generic `chat.send("/new")` or add an explicit confirmed `allowSessionControl`/trigger field, record a reset audit event with old/new session ids and source, clear stale reset/queue state across ordinary reconnects, and add regression coverage for gateway restart/reconnect followed by a normal visible message preserving the active `agent:main:main` session unless a confirmed reset occurs. Separately, expose reset archives or a restore path in the session UI so archived transcripts/checkpoints are discoverable without manual `sessions.json` repair.
