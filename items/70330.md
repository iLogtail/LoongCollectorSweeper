## Summary

Keep #70330 open. Current main still has the core ingredients reported here: WebChat can initiate reset via the generic chat send path, the gateway still rotates the session id and archives the old transcript, the reset RPC/protocol has no explicit UI trigger/audit fields, and session listing is still based on active store rows rather than reset archive inventory. Related #59839 and #70472 overlap but do not supersede this restart/reconnect failure mode.

## Best Possible Solution

Keep this in OpenClaw core. Add an explicit WebChat session create/reset RPC or an `allowSessionControl` flag for confirmed UI actions, persist a reset/rotation audit record with old/new session ids and trigger, clear stale queued reset state on reconnect, surface reset archives/checkpoints or a restore path in the Sessions/WebChat UI, and add regression coverage for gateway restart/reconnect followed by a normal visible send.
