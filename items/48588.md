## Summary

Keep open. Current main still has the Feishu HTTP cleanup shape this PR targets: both the explicit stop state helper and webhook abort cleanup call `server.close()` and immediately remove shared state. The PR remains the focused open fix for #48183, though it still needs maintainer review around awaitable public shutdown and regression coverage before landing.

## Best Possible Solution

Keep #48588 open as the focused fix for #48183. Refresh it against current main, keep a shared bounded HTTP-server close helper for both `stopFeishuMonitorState()` and `monitorWebhook()` abort cleanup, decide the public `stopFeishuMonitor()` awaitable contract, and add a regression test that proves Feishu webhook server cleanup/restart ordering.

## Risks / Open Questions

- The PR should add targeted Feishu webhook cleanup/restart coverage before landing; current tests do not exercise HTTP server shutdown ordering.
- The EADDRINUSE symptom is not yet strongly reproduced on the current supported Node runtime, so the landing rationale should be cleanup ordering/resource lifecycle rather than an unproven port-bind failure alone.
- Maintainers should decide whether `stopFeishuMonitor()` must return a `Promise<void>`; without that, direct stop/restart callers still cannot await the async cleanup path.
