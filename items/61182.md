## Summary

Keep open. The report is recent, specific, and still plausible for the QMD-backed `openclaw memory search` CLI path. Current main has useful partial protections: QMD child calls are timeout-bounded, the CLI closes managers, and CLI/agent search share the memory manager. But I did not find a source, test, release note, or canonical related issue proving the macOS/QMD CLI hang and wrapper overhead reported here are fixed or superseded.

## Best Possible Solution

Keep #61182 open and investigate the CLI-only QMD search lifecycle. A good maintainer path is to add timing/debug instrumentation around `runMemorySearch`, `getMemorySearchManager`, `manager.search`, result rendering, manager close, and best-effort `recordShortTermRecalls`; then add a focused regression test or QA scenario that exercises a QMD-backed memory CLI search with session collections and verifies the command returns or fails within `memory.qmd.limits.timeoutMs` plus a small wrapper budget. If the wrapper overhead from the comment is distinct, split it into a narrower performance issue linked back to #61182.
