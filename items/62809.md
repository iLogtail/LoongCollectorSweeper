## Summary

Keep open. Current main still derives OpenAI `prompt_cache_key` from the per-run `sessionId`, while isolated cron intentionally forces a fresh session id on each run. I found no current cron cache-affinity config or separate stable prompt-cache key path that would satisfy the requested cross-run first-turn cache reuse.

## Best Possible Solution

Keep the issue open and implement a separate OpenAI prompt-cache affinity key for recurring isolated cron runs. The likely fix is to keep the fresh isolated session id for transcript/runtime isolation, but pass a stable cache-affinity identity, derived from the cron job identity plus provider/model and optionally a stable prefix fingerprint, into OpenAI/OpenAI-compatible transport options. Add regression tests proving two isolated runs for the same job get different session ids but the same prompt-cache key, and that different jobs/models do not collide.

## Risks / Open Questions

- A fix should not simply reuse the full session id, because isolated cron relies on fresh transcript/session ids for context isolation and cleanup.
- The cache-affinity identity should be scoped by at least agent/job/provider/model, and possibly a stable prompt-prefix fingerprint, to avoid accidental cache sharing across unrelated jobs or tenants.
- Live OpenAI cache-hit rates are probabilistic, so the final implementation should include code-level tests for key stability plus a live trace check when credentials are available.
