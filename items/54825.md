## Summary

Keep open. Current `main` still turns provider failures into outbound user-visible reply payloads, Telegram has partial suppression controls, but WhatsApp has no comparable channel error policy and there is no built-in custom/owner-only error message path matching the issue request.

## Evidence

- **Provider errors are still emitted as reply payloads:** `buildEmbeddedRunPayloads` computes `errorText` for assistant failures and unconditionally appends it as a reply item with `isError: true`, so the core path still produces channel-bound error replies rather than suppressing them.
  - file: [src/agents/pi-embedded-runner/run/payloads.ts:163](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/agents/pi-embedded-runner/run/payloads.ts#L163)
  - command: `nl -ba src/agents/pi-embedded-runner/run/payloads.ts | sed -n '130,210p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Billing copy is still a user-facing channel message:** The sanitizer still formats billing failures into a friendly but explicit billing message (`API provider returned a billing error ... top up or switch to a different API key`). This is not raw JSON anymore, but it is still a technical/provider-facing error message sent to end users.
