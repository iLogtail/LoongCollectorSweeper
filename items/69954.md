## Summary

Keep PR #69954 open. Current main still prunes an entry when the resolved stored `sessionFile` path is missing, and `resolveSessionFilePath` only falls back to the canonical `<sessionId>.jsonl` path when metadata is empty or structurally invalid. The PR addresses a real remaining cleanup-safety bug from #50248, but the review comment about repair-only runs not setting `wouldMutate` should be fixed before landing.

## Best Possible Solution

Keep #69954 open. Update the PR so the cleanup helper reports both pruned and repaired entries, dry-run/apply summaries mark repair-only runs as mutations, and tests cover stale-path repair in both modes. Then run the changed command/session cleanup gate before merge. Consider a separate follow-up for doctor warning behavior if maintainers want the same canonical fallback there.
