## Summary

Keep PR #69954 open. Current main still prunes `sessions cleanup --fix-missing` entries after checking only the resolved persisted `sessionFile` path, so a stale-but-valid-looking stored path can still cause deletion even when the canonical `<sessionId>.jsonl` transcript exists. The PR addresses a real remaining bug linked to #50248, though it should fix the review-noted repair-only `wouldMutate` accounting before merge.

## Best Possible Solution

Keep the PR open and revise it into a complete cleanup fix: when a persisted `sessionFile` is missing, check the sessionId-derived canonical transcript before pruning, repair stored metadata when that canonical file exists, report repair-only mutations accurately in dry-run and applied JSON, and keep focused regression tests for stale-path repair, true missing transcripts, and valid stored-path behavior.
