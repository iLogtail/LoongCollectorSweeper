## Summary

Keep PR #69954 open. Current main still trusts a structurally valid persisted `sessionFile` path during `sessions cleanup --fix-missing` and deletes the store entry when that path is missing, without checking the canonical `<sessionId>.jsonl` transcript. The linked issue #50248 has a concrete repro for exactly this stale-metadata case, and the PR is a focused bug fix with relevant tests, though it still needs the review-commented JSON/dry-run mutation accounting gap addressed before landing.

## Best Possible Solution

Keep #69954 open. Ask for a small update that returns or separately tracks repaired transcript paths, sets `wouldMutate` accurately for repair-only preview/apply paths, and keeps the stale-`sessionFile` regression tests. After that, run the changed command/test lanes and use the PR to close #50248.
