## Summary

Keep open. Current `main` does not already implement this fix: the Feishu send/thread-reply path still only builds cards from `presentation`, and the edit path still accepts any object `card` payload and forwards `{}` through to Feishu instead of rejecting it locally.

## Evidence

- **Send/thread-reply path on main has no explicit-card validation branch:** On `main` (`a6aa151653ab478e9aa23e95c6446c41adc4fce1`), Feishu `send`/`thread-reply` builds `card` only from `normalizeMessagePresentation(ctx.params.presentation)` and never reads `ctx.params.card` in that branch, so the PR behavior is not already present there.
  - file: [extensions/feishu/src/channel.ts:695](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/feishu/src/channel.ts#L695)

  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Edit action still accepts empty object cards:** The Feishu `edit` action still treats any object as a card via `ctx.params.card && typeof ctx.params.card === "object"` and passes it into `editMessageFeishu` without checking for content, so `{}` still reaches the runtime layer.
  - file: [extensions/feishu/src/channel.ts:784](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/feishu/src/channel.ts#L784)
