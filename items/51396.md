## Summary

Keep #51396 open. Current main still clears self-declared operator scopes for ordinary device-less shared-token WebSocket connects, including the reported non-local token-auth shape. The current tests explicitly preserve that behavior except for a narrow direct-local backend gateway-client exception. The later issue comments correctly identify related work, but the HTTP scope fix and the direct-local backend exception do not resolve the reported remote WebSocket token-auth regression on current main.

## Best Possible Solution

Keep the issue open and resolve it through a focused gateway WebSocket auth decision. If valid remote shared-token backend/operator clients are intended to retain write/admin authority, update `handleMissingDeviceIdentity()` / `shouldClearUnboundScopesForMissingDeviceIdentity()` and the compat/default-token tests to cover non-local token-auth `chat.send`. If the current clearing is intentional, document the remote WebSocket device-token requirement and close or retarget the open duplicate PRs (#51413, #57889) with that rationale.
