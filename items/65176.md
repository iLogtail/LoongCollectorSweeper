## Summary

Keep open. Current main still has the original shared resolved-IP SSRF message with no remediation hint, so the PR's stated wording change is not implemented. The PR is coherent, but it should not land as-is because the proposed `channels.<plugin>.network.dangerouslyAllowPrivateNetwork` hint is attached to shared SSRF code used by provider, media, web-tool, browser/plugin, and other paths, and the branch also picked up an unrelated cron-model change flagged in review.

## Best Possible Solution

Keep the PR open but ask for a narrower patch: drop the unrelated cron commit, keep any remediation text caller-scoped, and either add a Telegram/media-specific error hint or a generic shared SSRF hint that names the correct surface-specific config paths without pointing every caller at `channels.<plugin>.network.dangerouslyAllowPrivateNetwork`. Add a focused regression test for the chosen wording path.
