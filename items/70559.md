## Summary

Keep #70559 open. Current main still contains the reported unsafe-reindex/sqlite-vec failure path: `runUnsafeReindex()` calls `resetIndex()`, `resetIndex()` drops `chunks_vec` while leaving `this.vector.available` intact, and the subsequent sync prepares a `DELETE FROM chunks_vec ...` statement whenever sqlite-vec is enabled and marked available. The related open PR #71611 addresses the safe-path Windows rename/EBUSY problem, not this unsafe-path missing table crash, and it is not merged into current main.

## Best Possible Solution

Keep the issue open and fix `memory-core` so unsafe full reindex with sqlite-vec either preserves/purges the existing `chunks_vec` virtual table or resets vector availability before sync and recreates the table before any prepared vector DELETE statements. Add a regression test that exercises vector-enabled unsafe reindex after `chunks_vec` exists; handle the safe-path Windows EBUSY work separately under #71611 or its canonical issue.
