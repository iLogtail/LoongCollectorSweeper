## Summary

Keep open. Current main still has the reported unsafe reindex failure shape: `resetIndex()` drops `chunks_vec` without clearing `vector.available`, then `runUnsafeReindex()` calls sync code that prepares `DELETE FROM chunks_vec ...` whenever `vector.available` is still true. The related open PR #71611 addresses the Windows safe-path rename/EBUSY fallback, not the `runUnsafeReindex`/`chunks_vec` crash itself.

## Best Possible Solution

Fix the unsafe reindex path in `extensions/memory-core/src/memory/manager-sync-ops.ts` so `resetIndex()` does not leave `vector.available === true` while `chunks_vec` is absent, either by preserving/purging the vector table or by resetting/revalidating vector state before stale-row delete statements are prepared. Add a focused regression test for unsafe reindex with vector enabled and an existing/dropped `chunks_vec`. Track the Windows safe-path rename fallback separately through #71611 unless maintainers decide that fully replaces the unsafe production workaround.
