## Summary

Current main still matches the reported failure path. `runUnsafeReindex()` calls `resetIndex()`, `resetIndex()` drops `chunks_vec`, and the next sync prepares a `DELETE FROM chunks_vec` statement whenever sqlite-vec is enabled and marked available. The related #71611 is still open and targets Windows safe-reindex rename/EBUSY behavior, not this unsafe sqlite-vec table lifecycle bug on main.

## Best Possible Solution

Keep #70559 open. The targeted fix belongs in `extensions/memory-core/src/memory/manager-sync-ops.ts`: make unsafe reindex preserve or safely purge `chunks_vec`, or make the delete prepare path robust when the vector virtual table was dropped, then add a regression test covering `OPENCLAW_TEST_MEMORY_UNSAFE_REINDEX=1` with sqlite-vec available and vector indexing enabled. Treat #71611 as related safe-path Windows EBUSY work, not as the canonical fix for this issue yet.
