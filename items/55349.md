## Summary

Keep open. Current `main` has render-only support for existing reply metadata in chat history, but the Control UI still does not expose a user reply action or send reply context for `chat.send`, so the requested threaded/quote-reply UX is not implemented.

## Evidence

- **`chat.send` schema has no reply field:** `ChatSendParamsSchema` allows `sessionKey`, `message`, `thinking`, `deliver`, origin fields, `attachments`, timeout/provenance, and `idempotencyKey`, but no `replyToId`, quoted-message payload, or thread-reply field for the Control UI chat path.
  - file: [src/gateway/protocol/schema/logs-chat.ts:35](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/gateway/protocol/schema/logs-chat.ts#L35)
  - command: `nl -ba src/gateway/protocol/schema/logs-chat.ts | sed -n '1,90p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Control UI sender does not send reply context:** `requestChatSend()` calls `chat.send` with only `sessionKey`, `message`, `deliver: false`, `idempotencyKey`, and optional `attachments`. There is no selected-message state or reply target in the outbound request.
  - file: [ui/src/ui/controllers/chat.ts:232](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/ui/src/ui/controllers/chat.ts#L232)
