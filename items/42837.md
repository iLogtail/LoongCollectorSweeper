## Summary

Keep #42837 open. The linked PR #42886 correctly identified the likely root cause for named Feishu accounts, but that PR is closed/unmerged and current main still creates dynamic Feishu DM bindings without `match.accountId`. Since the router intentionally treats account-less bindings as default-account-only, the named-account regression can still route back to `agent:main`.

## Best Possible Solution

Keep the issue open and land a focused Feishu plugin fix: pass the resolved Feishu `account.accountId` into `maybeCreateDynamicAgent`, persist it as `bindings[].match.accountId` for generated DM bindings, treat existing account-less dynamic bindings as default-account-only, and add regression coverage for non-default Feishu accounts plus the default-account compatibility path.

## Evidence

- **Dynamic binding creation is not account-scoped:** `maybeCreateDynamicAgent` has no `accountId` input and writes Feishu DM bindings with `match.channel` plus `match.peer`, but no `match.accountId`; the same shape is used when adding a missing binding for an existing dynamic agent and when creating a new dynamic agent.
  - file: [extensions/feishu/src/dynamic-agent.ts:67](https://github.com/openclaw/openclaw/blob/6cd047e7c27021d8955e09fa3598029eede5a1a7/extensions/feishu/src/dynamic-agent.ts#L67)
  - command: `nl -ba extensions/feishu/src/dynamic-agent.ts | sed -n '1,280p'`
  - sha: [6cd047e7c270](https://github.com/openclaw/openclaw/commit/6cd047e7c27021d8955e09fa3598029eede5a1a7)
- **Feishu DM routing is account-aware before and after dynamic creation:** Inbound Feishu DM handling resolves routes with `accountId: account.accountId`, calls `maybeCreateDynamicAgent` without passing that account id, then re-resolves with the same named account id. An account-less generated binding therefore cannot repair a named-account route miss.
  - file: [extensions/feishu/src/bot.ts:621](https://github.com/openclaw/openclaw/blob/6cd047e7c27021d8955e09fa3598029eede5a1a7/extensions/feishu/src/bot.ts#L621)
