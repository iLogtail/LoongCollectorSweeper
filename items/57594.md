## Summary

Keep open. Current `main` still hardcodes Zalo proactive target detection to numeric-looking IDs, so a hex-like Zalo `chat_id` will miss the id-like fast path and fall through to directory lookup before returning `Unknown target`. The actual Zalo send/runtime code already accepts arbitrary string `chat_id` values and inbound sessions are routed by `chatId`, which matches the report that in-session replies work while proactive CLI sends fail.

## Evidence

- **Zalo plugin still uses numeric-only target detection:** The bundled Zalo plugin sets `messaging.targetResolver.looksLikeId` to `isNumericTargetId` with hint `<chatId>`, so proactive CLI target handling is gated by a numeric-only helper.
  - file: [extensions/zalo/src/channel.ts:201](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/zalo/src/channel.ts#L201)
  - command: `nl -ba extensions/zalo/src/channel.ts | sed -n '198,204p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Numeric helper rejects non-digits:** `isNumericTargetId` returns true only for `/^\d{3,}$/`, so hex-like chat ids such as `3becaa50ae12474c1e03` are not treated as valid explicit ids.
  - file: [src/plugin-sdk/reply-payload.ts:327](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/plugin-sdk/reply-payload.ts#L327)

- **Outbound resolution falls through to unknown-target errors:** `resolveMessagingTarget` only accepts normalized targets directly when `looksLikeTargetId(...)` passes; otherwise it searches the directory and finally returns `unknownTargetError(...)`.
