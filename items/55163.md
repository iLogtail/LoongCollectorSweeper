## Summary

Keep open. Current main does not implement the issue's requested skip for already-allowlisted non-channel plugins. In `src/config/plugin-auto-enable.shared.ts`, `plugins.allow` membership only affects whether the allowlist is amended; for non-channel plugin candidates, `alreadyEnabled` still depends on `plugins.entries.<id>.enabled`, and the code still calls `registerPluginEntry()` afterward. That means an allowlisted local plugin can still be materialized into `plugins.entries.<id>`, which is the behavior the report says causes the reload loop. The latest shipped allowlist note in the changelog is an install-time fix, not this doctor/auto-enable case.

## Evidence

- **Auto-enable still creates plugin entries for non-channel plugins:** Current main computes `allowMissing`, then for non-channel plugins treats `alreadyEnabled` as `next.plugins?.entries?.[entry.pluginId]?.enabled === true`. If that is false, it proceeds to `registerPluginEntry(next, entry.pluginId)` and `ensurePluginAllowlisted(next, entry.pluginId)`. There is no guard that skips auto-enable when the plugin is already present in `plugins.allow`.
  - file: [src/config/plugin-auto-enable.shared.ts:759](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/config/plugin-auto-enable.shared.ts#L759)
  - command: `nl -ba src/config/plugin-auto-enable.shared.ts | sed -n '759,771p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Allowlist is documented as trusting plugin ids:** The plugin architecture docs say `plugins.allow` trusts plugin ids, not source provenance, which matches the issue's expectation that an allowlisted local plugin should already be loadable without doctor forcing a new `plugins.entries.<id>` record.
