## Summary

Current `main` does not implement the requested guardrail. Browser navigation still accepts URLs with embedded credentials, forwards them to the browser runtime, and the browser-specific path is not using the repo's URL-userinfo redaction helper.

## Evidence

- **Navigation validation does not reject URL credentials:** `assertBrowserNavigationAllowed` parses the URL and checks protocol/proxy/SSRF constraints, but there is no branch rejecting `parsed.username` or `parsed.password`. A URL like `https://user:pass@example.com/path` would pass this validator if otherwise allowed.
  - file: [extensions/browser/src/browser/navigation-guard.ts:82](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/browser/src/browser/navigation-guard.ts#L82)
  - command: `nl -ba extensions/browser/src/browser/navigation-guard.ts | sed -n '82,150p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Navigate route forwards the raw URL into Playwright:** The `/navigate` route takes `body.url` and passes it straight into `pw.navigateViaPlaywright`. `navigateViaPlaywright` then forwards that same `url` into `gotoPageWithNavigationGuard`, which calls `page.goto(opts.url, ...)`. There is no redaction or early credential-specific rejection on this path.
  - file: [extensions/browser/src/browser/routes/agent.snapshot.ts:181](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/browser/src/browser/routes/agent.snapshot.ts#L181)
  - command: `nl -ba extensions/browser/src/browser/routes/agent.snapshot.ts | sed -n '181,223p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Browser runtime still uses the raw navigation URL:** `navigateViaPlaywright` validates the URL only with the existing navigation guard, then passes the original string through to `gotoPageWithNavigationGuard` for `page.goto`. This matches the issue report's failure mode rather than an implemented early-safe rejection.
  - file: [extensions/browser/src/browser/pw-tools-core.snapshot.ts:197](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/browser/src/browser/pw-tools-core.snapshot.ts#L197)
  - command: `nl -ba extensions/browser/src/browser/pw-tools-core.snapshot.ts | sed -n '197,245p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Generic text redaction does not cover `user:pass@host` URLs, though a dedicated helper exists:** The default log-redaction patterns cover tokens, passwords, headers, and PEM blocks, but not URL userinfo. Separately, `redactSensitiveUrl` explicitly redacts `parsed.username` and `parsed.password`. That helper exists in the repo, but the browser navigation path above is not using it.
  - file: [src/logging/redact.ts:16](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/logging/redact.ts#L16)
  - command: `nl -ba src/logging/redact.ts | sed -n '1,80p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Supported alternative auth already exists:** OpenClaw already exposes HTTP auth through `openclaw browser set credentials` and the `/set/credentials` route, so the issue's suggested alternative path is consistent with current product capabilities. The missing piece is the early rejection/redacted guidance on navigation URLs.

## Risks / Open Questions

- Credential-bearing navigation URLs can still reach Playwright/Chromium on current `main`, so failures may still include the original URL in browser/runtime error text.
- I could not do a live end-to-end browser reproduction in this checkout because `pnpm` is unavailable in the environment; this review is based on direct source inspection of the current `main` paths.
