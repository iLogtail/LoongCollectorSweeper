## Summary

Keep issue #55365 open. Current main still forwards browser navigation URLs with URL userinfo through the browser tool and `/navigate` route; the shared browser navigation guard validates protocol and SSRF policy but does not reject or redact `username`/`password` before Playwright/Chrome MCP receives the URL. The adjacent cron-webhook credential PR #51822 is related hardening work, but it is a separate surface and does not supersede this browser-tool issue.

## Best Possible Solution

Add a browser navigation/open URL guard, ideally in `assertBrowserNavigationAllowed` or a small helper it calls, that rejects http/https page URLs with `username` or `password`, surfaces a redacted error using the existing sensitive-URL helpers, and points users to `openclaw browser set credentials`, an authenticated browser profile, headers, or cookies. Cover direct tool, `/navigate`, `/tabs/open`, Playwright, Chrome MCP, and node-proxy paths with focused regression tests.
