## Summary

Current main already fixes the Discord 1005/1006 stale-resume loop through the pinned @buape/carbon 0.16.0 gateway implementation. Carbon now tracks failed RESUME attempts, resets stale session state after the threshold, falls back to fresh IDENTIFY, and only resets reconnect counters after READY/RESUMED; OpenClaw pins that dependency, surfaces the fresh-IDENTIFY debug event, and has regression coverage around reconnect exhaustion/status handling. The PR discussion identified a real failure mode, but the branch is now obsolete against the current Discord plugin layout and shipped dependency behavior.

## Best Possible Solution

Close this PR as implemented on main and keep the current Carbon-backed Discord gateway behavior. If a 1005/1006 resume loop still reproduces on @buape/carbon 0.16.0 or later, track it as a fresh issue with logs showing the Carbon 'Gateway reconnect scheduled...' and 'Gateway forcing fresh IDENTIFY...' sequence against the current extensions/discord layout.
