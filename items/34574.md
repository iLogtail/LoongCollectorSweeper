## Summary

Keep open. Current main still does not solve the reported repeated exec loop: exec outcomes are hashed through the generic `{ details, text }` path, while exec details include volatile fields such as `durationMs`, `pid`, and `startedAt`. That prevents the no-progress/global breaker from seeing identical exec results as stable repeats, and the same-args `genericRepeat` path remains warning-only rather than blocking. The issue discussion also points to open PR #41502 as the unresolved fix path; the earlier #34687 fix was closed unmerged in the provided context.

## Best Possible Solution

Keep #34574 open until main lands an exec-specific stable outcome hash, with regression tests for repeated successful exec results that vary only by volatile details. After that ships, either close this as implemented and keep any broader result-similarity work in a narrower follow-up, or explicitly link the canonical issue/PR for the remaining broader detector work.
