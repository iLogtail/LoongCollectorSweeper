## Summary

Keep #72121 open. Current `main` still rejects `sk-ant-api03-...` through `openclaw models auth paste-token --provider anthropic`: the paste-token caller routes Anthropic input through a validator that only accepts `sk-ant-oat01-`. The existing Anthropic API-key login method is real, but the reported paste-token behavior remains unresolved and the linked fix PR #72124 is still open in the provided context.

## Best Possible Solution

Keep the issue open until the linked fix or an equivalent maintainer-approved change lands. The narrow implementation path is to widen the Anthropic paste-token validator and add regression tests for both `sk-ant-oat01-` and `sk-ant-api03-`; the alternative is an explicit setup-token-only error/help update that points API-key users to the existing Anthropic `api-key` login method.
