## Summary

Keep #72121 open. Current main still rejects `sk-ant-api03-...` on the reported `openclaw models auth paste-token --provider anthropic` path because the shared Anthropic token validator only accepts `sk-ant-oat01-...`. A separate Anthropic API-key auth method exists, but it does not fix the reported paste-token behavior. The related PR #72124 is the likely fix path in the provided context, but its changes are not present in the inspected `main` checkout.

## Best Possible Solution

Keep the issue open until #72124 or an equivalent patch lands. The best fix is to widen the Anthropic paste-token validator to accept both `sk-ant-oat01-...` setup tokens and `sk-ant-api03-...` API keys, add focused validator and CLI regression tests for both prefixes and error messages, and clarify CLI/docs wording if `paste-token` is intentionally meant to remain setup-token-only.
