## Summary

Keep PR #52762 open. Current main still has the eager DedupeCache prune path the PR targets, and Discord’s inbound replay guard is backed by that cache with a 5,000-entry memory cap. However, the PR should not be treated as ready/obsolete cleanup: current main already has listener-level non-blocking dispatch for the original slow-listener symptom, while this PR’s proposed lazy prune has unresolved review concerns around saturated-cache behavior and NaN limit semantics for SDK guard callers.

## Best Possible Solution

Keep the PR open for maintainer review, but require a revised implementation that removes the per-message full-cache scan even after saturation, preserves or deliberately documents invalid-limit behavior, and adds targeted DedupeCache/map-size tests plus a small benchmark or regression test covering the Discord replay-guard workload. The shipped listener non-blocking behavior can remain the fix for the older #4453 listener-delay symptom; this PR should stand or fall as an infra cache-performance fix.

## Risks / Open Questions

- Merging the PR as-is may not fix the busiest Discord servers because the lazy-prune condition still triggers a full TTL scan for every distinct message after the cache is saturated.
- The `pruneMapToMaxSize` rewrite may alter malformed numeric config behavior for plugin SDK webhook guards unless NaN handling is specified and tested.
- No live Discord benchmark was run in this read-only review; a focused synthetic/perf regression test would be the right evidence before landing a cache hot-path change.
