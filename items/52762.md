## Summary

Keep open. Current `main` still contains the hot-path `DedupeCache.check()` prune behavior that this PR targets, so the underlying Discord replay-cache cost is not already implemented away. The PR also has unresolved review concerns: it still falls back to a full prune once the 5,000-entry cache saturates, and its `map-size.ts` rewrite would change shared invalid-limit behavior for webhook guard callers.

## Risks / Open Questions

- Merging this PR as-is would not fully resolve the busiest-server case, because once the Discord replay cache stays above 5,000 entries the code still pays the full prune cost on new keys.
- The `map-size.ts` change risks a regression in shared webhook limiter code by turning malformed numeric limits into full cache eviction instead of the current effective no-op.
