## Summary

Keep #37696 open. Current main at b67d9bf still has the reported strict-mode parser behavior: an unclosed <think> tag leaves the reasoning parser inside a thinking block, so the trailing assistant text is not appended. The checked-in tests still assert that strict-mode/unclosed responses collapse to empty or to only the prefix, while shared delivery and channel preview paths still use strict reasoning stripping. The issue discussion’s linked prior fix attempts (#37703, #37861, #46205) are closed unmerged, and #51915 remains an open fix candidate rather than shipped main behavior.

## Best Possible Solution

Keep #37696 open until a focused fix lands. The likely path is to update stripReasoningTagsFromText so malformed/unclosed reasoning tags cannot erase all deliverable assistant text, adjust strict-mode and extractAssistantText tests, and review or supersede open PR #51915 with that behavior. If maintainers want a different safety tradeoff, capture it explicitly in the sanitizer contract and channel tests.
