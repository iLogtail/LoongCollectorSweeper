## Summary

Current main still does not provide a configurable lane wait diagnostic threshold. The queue warning threshold remains only a per-call `warnAfterMs` with a hardcoded 2,000 ms default, the cron enqueue path still hardcodes 5,000 ms, and the diagnostics config/schema/docs do not expose `laneWaitWarnMs` or an equivalent setting. The related implementation PR #16366 is closed unmerged and its head commit is not present in this checkout; a maintainer Codex review already noted the same remaining gap on April 26, 2026.

## Best Possible Solution

Keep #14747 open until a focused maintainer-approved implementation adds a configurable lane wait diagnostic threshold. The likely path is to add a documented config field such as `diagnostics.laneWaitWarnMs` to the config types, zod schema, generated metadata/help/docs, and then wire `src/process/command-queue.ts` so explicit per-call `warnAfterMs` still wins while the default comes from config. The cron manual-run path should either inherit that configured default or expose a documented cron-specific threshold, with regression coverage for the configured default and the long-running cron queue case.
