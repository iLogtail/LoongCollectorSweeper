## Summary

Keep open. Current `main` does not contain the stale-first-attempt recovery guard from PR #54900, and the PR as described still has an unresolved correctness hole around budget-deferred entries. This is not safely closable as already implemented or obsolete.

## Evidence

- **Current main has no stale-entry recovery option or summary field:** `RecoverySummary` only tracks `recovered`, `failed`, `skippedMaxRetries`, and `deferredBackoff`, and `recoverPendingDeliveries()` only accepts `maxRecoveryMs`; there is no `skippedStale`, `maxEntryAgeMs`, or equivalent stale-first-attempt handling on `main`. A focused `rg` for `maxEntryAgeMs|skippedStale|DEFAULT_MAX_ENTRY_AGE_MS` in the recovery code and tests returned no matches.

- **Main still explicitly allows first-attempt crash replay:** The policy test on `main` asserts that an entry with `retryCount=0` and no `lastAttemptAt` is immediately eligible for recovery replay, which is the behavior this PR is trying to change.
  - file: [src/infra/outbound/delivery-queue.policy.test.ts:52](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/infra/outbound/delivery-queue.policy.test.ts#L52)

  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Budget deferral mutates untouched entries into retries:** `deferRemainingEntriesForBudget()` calls `failDelivery(...)`, and `failDelivery()` increments `retryCount` and stamps `lastAttemptAt`. The recovery test on `main` also asserts that budget-deferred entries come back with `retryCount === 1`. That means the PR's proposed stale filter for only `retryCount === 0` / missing `lastAttemptAt` entries would miss old never-sent entries once they roll over through the recovery budget path.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:144](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/infra/outbound/delivery-queue-recovery.ts#L144)
