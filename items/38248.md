## Summary

Keep #38248 open. Current main still has cost estimation, usage summaries, provider spend-limit error classification, cooldown probes, and external LiteLLM spend-limit guidance, but it does not implement a native request-level rolling spend ceiling such as `agents.defaults.costGuard.maxSpendPerHour` or `maxSpendPerDay`. The issue is concrete, unprotected, and the prior discussion’s `npx vibe-billing setup` and Kalibr routing workarounds are useful mitigations but do not supersede the requested OpenClaw failover safety guard.

## Best Possible Solution

Keep the issue open until OpenClaw either adds a native cost guard in the model request/failover path or maintainers intentionally define and document a plugin/API seam that can enforce rolling spend ceilings before provider calls. The likely product path is a strict config surface under agent/model defaults, shared cost accounting over rolling windows, pre-attempt estimated-cost checks, post-response reconciliation when providers return usage/cost, and clear pause/reject plus owner-notification behavior.

## Risks / Open Questions

- Closing as implemented would hide a still-missing financial safety gap: expensive fallback calls can still succeed repeatedly without a native rolling spend ceiling.
- A ClawHub-only close is not high-confidence because cross-provider enforcement before model calls likely needs core request-path participation or an explicit blocking plugin/API hook.
- The prior external mitigations are useful workarounds, but they are outside OpenClaw’s native failover path and do not cover the proposed config/notification behavior.
