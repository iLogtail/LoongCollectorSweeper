## Summary

Keep #38671 open. Current main still has no first-class direct shell command cron payload: the protocol, CLI, UI, service validation, and detached runtime remain built around `systemEvent` and `agentTurn`. The linked PR #62789 is useful related work, but it is open and backend-only in the provided context, so it does not supersede or ship the requested end-to-end command and command-to-agent modes.

## Best Possible Solution

Keep #38671 open until OpenClaw either ships direct command cron support end to end or maintainers explicitly reject that product direction. The likely path is to rework or land the backend/protocol pieces from #62789, add CLI/UI/docs support for pure command and command+message hybrid jobs, and align the executor with existing exec security, timeout, output truncation, run-log, and delivery contracts.

## Risks / Open Questions

- Direct shell cron needs maintainer product/security decisions for argv parsing vs shell wrapping, approval policy, timeout and child termination, output-size truncation, failure handling, run-log shape, delivery semantics, and command-output injection into optional agent analysis.
- A plugin can run commands through runtime helpers, but current core cron lacks a payload extension point; closing this as ClawHub would leave the requested built-in scheduler/protocol/CLI/UI behavior unresolved.
