## Summary

Keep open. Current `main` still preserves the reported failure mode: if `openclaw-sandbox:bookworm-slim` is missing, `ensureDockerImage()` aliases plain `debian:bookworm-slim` to that tag, while sandbox mutation helpers still require Python. The new `Dockerfile.sandbox` and `scripts/sandbox-setup.sh` on `main` help build a compatible image, but they do not replace the incompatible fallback path that triggered this issue.

## Evidence

- **Current fallback still tags plain Debian as the default sandbox image:** `ensureDockerImage()` checks for the image, then for `DEFAULT_SANDBOX_IMAGE` runs `docker pull debian:bookworm-slim` and `docker tag debian:bookworm-slim openclaw-sandbox:bookworm-slim` instead of failing or building the repo Dockerfile. That matches the issue's reported incompatible-image path.

## Risks / Open Questions

- Closing as implemented would be incorrect because the current default-image fallback can still recreate the incompatible sandbox image described by the reporter.
- The presence of `Dockerfile.sandbox` and docs could create a false positive review if the runtime fallback path is not checked.
- Docs are not fully aligned yet: `docs/gateway/sandboxing.md` still describes `python3` as part of the more functional common image, which reinforces that sandbox-image behavior around Python is still in flux.
