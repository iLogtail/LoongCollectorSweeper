## Summary

Current `main` already implements the reported guard: isolated announce/completion delivery is routed through the requester session, and inter-session/system-event turns no longer overwrite the established external route used for later user replies. The relevant code, tests, and docs are present on `main`; I could not tie the fix to a released tag, so this appears implemented on `main` but not yet proven shipped in a release.

## Close Comment

Closing from a Codex review because current `main` already has the guard this report asks for. The announce/completion path now sends back through the requester session, not the isolated child session (`src/agents/subagent-announce-delivery.ts:523`), and session init explicitly preserves the established external route for inter-session/system-event turns instead of letting those turns rewrite `lastChannel` / `lastTo` (`src/auto-reply/reply/session.ts:557`, `src/auto-reply/reply/session-delivery.ts:113`). There is regression coverage for preserving Telegram/external routes (`src/auto-reply/reply/session-delivery.test.ts:4`) and shared-session route preservation during automated turns (`src/auto-reply/reply/session.test.ts:3121`). The docs also now state that isolated cron delivery is shared and that isolated heartbeat delivery still uses the main session context (`docs/cli/cron.md:26`, `docs/gateway/heartbeat.md:227`). I could confirm this on `main` at commit `d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2`; I could not tie it to a released tag yet, so `fixedRelease` is left null. If you still hit this on a fresh build from current `main`, please open a new issue with an updated minimal repro and the exact channel/plugin path involved.
