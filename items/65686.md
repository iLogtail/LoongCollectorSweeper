## Summary

Keep open. Current main only partially mitigates the report: `delivery.mode="none"` can disable runner fallback delivery, and Telegram/Discord can opt into final-assistant text collapse, but there is still no generic `cron.deliveryPayloads` or per-job `full|last|none` payload-selection policy. Existing tests still assert multi-payload announce delivery for Slack, WhatsApp, and iMessage, so the core user request remains actionable.

## Best Possible Solution

Keep the issue open and implement an explicit cron outbound payload policy, likely `full|last|none`, either globally or per job. Thread it through config/schema, CLI/UI job editing if exposed there, `resolveCronPayloadOutcome()`, interim run-executor logic, and `dispatchCronDelivery()` so `none` suppresses synthetic fallback sends while run records/UI output remain populated. Add regression coverage for full, last-only, and none across at least one non-opt-in channel.
