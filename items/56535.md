## Summary

Keep open. Current main has adjacent mitigations, but it does not fully implement the reported Surge/Fake-IP provider endpoint case: strict SSRF still blocks DNS answers in 198.18.0.0/15 unless a caller supplies an explicit policy, and provider HTTP only skips target DNS when an actual HTTP(S)_PROXY env path is selected. The reported DNS-only macOS Surge Enhanced Mode path can still hit the same blocked resolved-IP failure for public Google/OpenAI provider endpoints.

## Best Possible Solution

Keep the issue open and implement a provider-safe Fake-IP path: either add a narrow RFC2544 benchmark-range policy for known public provider base URLs, expose a documented per-provider `allowRfc2544BenchmarkRange`/hostname-scoped policy, or make provider HTTP use trusted proxy mode only when the proxy path is actually responsible for DNS resolution. Add regression tests for Google/OpenAI image generation where public provider hostnames resolve to 198.18.x.x under DNS-only Fake-IP and for the security boundary around private/custom provider base URLs.
