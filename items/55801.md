## Summary

Keep open. Current main still filters tool availability through boolean policy helpers and logs only tool start/end or generic execution diagnostics; it does not preserve or emit the matched policy layer/rule requested in #55801. Related PR #55826 is still open and PR #63547 was closed unmerged, so the issue is not implemented or safely superseded.

## Best Possible Solution

Implement a source-backed policy explanation path: have the matcher/pipeline preserve decision, matchedBy config path, and optional matched rule; thread that through embedded tool execution and gateway /tools/invoke allow/deny paths; emit sanitized DEBUG/INFO audit lines and add regression tests for explicit allow, explicit deny, allowlist miss, gateway default deny, and channel/group policy cases. Existing PRs #55826 and #63547 are useful prior art but should be rebased/reviewed rather than treated as shipped.
