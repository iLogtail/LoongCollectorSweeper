## Summary

Keep open. Current main has partial outbound hook coverage, but it still does not provide the requested unbypassable pre-send boundary across all outbound paths. `deliverOutboundPayloads`, Telegram reply delivery, Discord reply delivery, and the newer generic dispatch helper cover some routes, while Slack, Signal, iMessage, Google Chat, Line, and Microsoft Teams reply paths still send through channel-specific delivery functions without `message_sending` enforcement. The recent issue discussion and related closed/unmerged PRs remain relevant rather than obsolete.

## Best Possible Solution

Keep the issue open and treat it as the canonical architectural invariant until every outbound user-visible delivery route is forced through one pre-send policy boundary. The likely implementation path is to make the core reply dispatcher or `deliverOutboundPayloads` the required delivery seam for agent replies, message-tool sends, auto-replies, and channel adapters; wire remaining channel reply paths such as Slack, Signal, iMessage, Google Chat, Line, and Microsoft Teams through that seam; and add tests or an architecture audit that fails on direct channel sends from reply paths without an approved pre-send hook wrapper.
