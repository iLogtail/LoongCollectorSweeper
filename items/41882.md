## Summary

Do not auto-close this PR. Current main still serializes Discord inbound jobs through an outer per-session `KeyedAsyncQueue` before `processDiscordMessage` reaches the shared reply queue logic, so the reported steer-mode failure remains plausible. The PR itself is stale against the current plugin layout and should be rebased/adapted, especially around `steer-backlog` and per-session `/queue` overrides, but it is still useful work rather than obsolete cleanup.

## Best Possible Solution

Keep the PR open and ask for a rebase/adaptation against the current Discord plugin architecture. The likely fix belongs in `extensions/discord/src/monitor/inbound-worker.ts` or the job-building path: bypass the outer session queue only when the effective Discord queue mode is `steer` or `steer-backlog`, preserve serialized behavior for non-steer modes, honor per-session queue overrides, and add regression tests covering `steer`, `steer-backlog`, and a non-steer session override.
