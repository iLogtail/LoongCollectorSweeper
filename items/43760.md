## Summary

Keep #43760 open. Current main still models `memorySearch.fallback` as a string provider id with a single shared `memorySearch.remote`; both primary provider creation and fallback activation pass the same resolved remote config. The exact Gemini-primary plus OpenAI-compatible SiliconFlow fallback shape requested here is not implemented, and related ClawSweeper context treats #43760 as the canonical narrow tracker after closing PR #44878 and broader issue #55590.

## Best Possible Solution

Keep #43760 open as the canonical narrow tracker. A good implementation should add a backward-compatible fallback object shape such as `{ provider, model, remote }` while preserving the current string fallback form, thread fallback-specific model/remote/secret handling through config types, zod schema, resolved memory config, `createEmbeddingProvider`, and fallback activation, and update docs/help/secret surfaces plus focused regression tests for Gemini primary with an OpenAI-compatible fallback endpoint.
