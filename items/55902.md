## Summary

Issue #55902 proposes a useful external-tool bridge, but current main scope puts this in ClawHub/plugin/MCP territory rather than OpenClaw core. The repo vision explicitly says optional capability should usually ship as plugins, new skills should go to ClawHub first, and first-class MCP runtime should not be added to core when the existing MCP bridge path already exists.

## Risks / Open Questions

- If maintainers later want an official reference implementation or curated docs for third-party IPC bridges, that would be a docs/community-plugin decision, not evidence that this belongs in core.
- The issue body mixes a scope argument with a concrete third-party implementation; closing as `clawhub` assumes the implementation quality/security should be evaluated in its own repository or ClawHub listing, not merged into OpenClaw core.

## Close Comment

Closing as a Codex review: this is useful, but it fits the ClawHub/plugin path rather than OpenClaw core. Current main is explicit that optional capability should usually ship as plugins, new skills should go to ClawHub first, and first-class MCP runtime should not be added to core when the existing MCP bridge path already exists (`VISION.md:54`, `VISION.md:68`, `VISION.md:101`). The docs also already point users to the existing interop surfaces: `openclaw mcp serve` for Codex/Claude Code/other MCP clients talking to OpenClaw conversations (`docs/cli/mcp.md:29`) and ACP/ACPX for OpenClaw running or exposing sessions to external harnesses (`docs/cli/acp.md:18`, `docs/tools/acp-agents.md:13`). Since your proposal is already packaged as an external MCP server / skill and OpenClaw’s documented direction is to distribute this kind of optional integration via ClawHub/community plugins, this should live there rather than in `openclaw/openclaw` core.
