## Summary

Keep #49250 open. Current main has pieces that reduce the race impact, including heartbeat skipping when the target session lane is already busy and Control UI optimistic user messages, but it still does not implement the requested explicit prompt/heartbeat conflict handling. A prompt sent while a heartbeat/non-chat run is active is not represented in the Control UI queue, and the current queue UI only covers locally detected active chat runs.

## Best Possible Solution

Keep the issue open and implement a heartbeat/session-lane-aware queue state for Control UI chat sends. The likely path is to surface an explicit queued/pending event or response when `chat.send` is deferred behind an active heartbeat, render it in `chatQueue` with cancel/steer behavior, and verify that heartbeat final/delta events remain visible while the pending prompt is not mistaken for a lost or completed turn.

## Evidence

- **Issue discussion defines remaining UX contract:** #49250 asks for immediate race detection, visible heartbeat, a visibly queued prompt, and user control to cancel/inject after the current tool boundary. The only comment frames this as a queue-with-explicit-states problem and links adjacent state-drift issues #49247, #49210, and #49236, but none is a canonical tracker for this exact Control UI prompt/heartbeat race.
