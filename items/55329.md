## Summary

Keep open. Current `main` already supports per-agent workspaces and per-agent skill visibility, but it does not implement the issue's requested primitives: sandbox path expansion still resolves `~` to the host home directory, and the SKILL.md/OpenClaw skill metadata surface has no `memoryPath` field or equivalent runtime hook.

## Evidence

- **Sandbox `~` still maps to host home:** `expandPath()` in the sandbox path resolver returns `os.homedir()` for `~` and `~/...`, so sandboxed path handling does not reinterpret `~` as the sandbox workspace root.
  - file: [src/agents/sandbox-paths.ts:27](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/agents/sandbox-paths.ts#L27)
  - command: `nl -ba src/agents/sandbox-paths.ts | sed -n '1,80p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Per-agent workspaces exist today:** Current docs and tests show distinct workspaces per agent are supported (`agents.list[].workspace`, `workspace-home`, `workspace-work`, and per-agent fallback resolution), so the repository already has agent-level workspace isolation but not the requested skill-memory-path bridge.
  - file: [docs/gateway/config-agents.md:892](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/docs/gateway/config-agents.md#L892)

## Risks / Open Questions

- The exact `~/self-improving/memory.md` collision is reported against external skills that are not vendored in this checkout, so I could not verify that specific hardcoded path directly from repository sources.
- There may be a skill-side workaround using workspace-local skill copies or `skills.entries.<skill>.config`, but current `main` does not provide the requested core behavior (`~` remap in sandbox path resolution or SKILL.md `memoryPath` support).
