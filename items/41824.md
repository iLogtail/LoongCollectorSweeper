## Summary

Keep #41824 open. Current main has useful mitigations around Telegram/ACP delivery ordering and progress visibility, but I did not find source, tests, docs, or history proving the reported Claude ACP-on-Telegram behavior is fixed. The central issue is not just early Telegram streaming; it is an external ACP harness claiming side-effectful work without verifiable tool execution, and current main still treats ACP harness filesystem/native-tool behavior as owned by the external harness rather than enforcing tool-first side effects in OpenClaw core.

## Best Possible Solution

Keep the issue open and narrow it into an ACP-on-chat-surface reliability investigation. Best next step is a deterministic repro or fixture around a bound Telegram ACP session where the harness emits final success text without corresponding ACP tool_call/tool_call_update evidence, then decide whether OpenClaw needs an ACP strict-agentic mode, an ACP before-finalize verification hook, clearer docs/config for final-only versus live ACP delivery, or a Telegram-visible warning when no side-effect evidence exists.
