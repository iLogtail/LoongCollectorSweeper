## Summary

Keep open. Current main still has a plausible unresolved Docker sandbox bug for the reported setup: the Docker setup path enables sandboxing with `workspaceAccess: "none"` but does not configure a host-visible `sandbox.workspaceRoot`, while the Docker backend mounts the resolved sandbox workspace at `/workspace` read-only when access is not `rw`. The current docs now describe the DooD host-path parity requirement, and related #59718 tracks selective writable custom binds, but neither fully supersedes this default Docker sandbox read/write failure.

## Best Possible Solution

Keep #59613 open. The likely maintainer path is to make Docker sandbox bootstrap configure a host-visible `agents.defaults.sandbox.workspaceRoot` and matching compose volume parity for DooD, or otherwise fail with an actionable diagnostic before creating unusable `/workspace` mounts. Separately, align `workspaceAccess: "none"` tool exposure with the intended write semantics, and leave #59718 to track optional least-privilege writable custom binds.

## Evidence

- **Issue reproduction and related discussion:** #59613 reports the sandbox container bind source as `/home/node/.openclaw/sandboxes/...:/workspace:ro,z`, with `/workspace` empty and file writes failing as `Sandbox path is read-only`; a later commenter confirmed similar breakage after upgrading to 2026.4.2. The timeline links #59718, but that issue is a broader least-privilege custom-bind feature request, not a complete replacement for this Docker setup failure.
- **Docker setup enables sandbox but does not set sandbox workspaceRoot:** The setup script sets `agents.defaults.sandbox.mode`, `scope`, and `workspaceAccess` to `none`, then restarts the gateway, but there is no matching `config set agents.defaults.sandbox.workspaceRoot` to a host-visible path for Docker-out-of-Docker.
