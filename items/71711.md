## Summary

Keep #71711 open. Current main does not implement a recurring cron expiration field: cron job types, gateway protocol validation, CLI add options, scheduler due/run logic, list/show output, and docs only model `at`/`every`/`cron`, `enabled`, and `deleteAfterRun`. The related #71712 RFC is open and depends on this kind of expiration for an agent-facing scheduling API, but it is broader security/API work rather than a canonical duplicate of the operator-facing recurring-job lifecycle request.

## Best Possible Solution

Keep the issue open and treat it as a core cron feature request. The likely implementation should add a backwards-compatible optional expiration field to the cron job contract and gateway protocol, expose CLI add/edit and list/show support, teach the scheduler to skip expired recurring jobs before execution and either remove or disable them, write an audit/run-log signal, and cover `every`/`cron` plus rejected `at` cases in focused cron service, protocol, CLI, and docs tests. Coordinate naming and agent-facing requirements with the related open #71712 RFC, but do not close this narrower request as a duplicate.
