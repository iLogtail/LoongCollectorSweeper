## Summary

Keep #71711 open. Current main does not implement per-recurring-job expiration: the cron job type, gateway protocol schema, CLI add/edit/list surfaces, and scheduler result path only model schedule, enabled, and deleteAfterRun. The existing auto-cleanup behavior remains limited to successful one-shot `at` jobs, while recurring `every`/`cron` jobs compute another next run. The cross-referenced #71712 is a broader agent-facing scheduling RFC that depends on this idea; it does not supersede the smaller operator-facing cron expiration request.

## Best Possible Solution

Keep the issue open and implement it in core cron: add an optional recurring-only expiration field plus on-expire policy to the cron job type, gateway protocol schema, normalizer/store compatibility path, CLI add/edit/list/show surfaces, scheduler pre-run expiry handling, docs, and focused tests. Coordinate with #71712 because the agent-facing scheduling RFC can reuse the same expiration primitive, but do not close #71711 as a duplicate.
