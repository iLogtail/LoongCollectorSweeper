## Summary

Keep open. Current `main` still lacks any 429-specific backoff/retry path in `syncTelegramMenuCommands`, so the PR addresses behavior that is not yet implemented; the PR discussion also shows an unresolved design tradeoff around what to cache after repeated 429 failures.

## Evidence

- **Current main still rethrows non-BOT_COMMANDS_TOO_MUCH sync failures:** `syncTelegramMenuCommands` only suppresses/logs `BOT_COMMANDS_TOO_MUCH`. Any other `setMyCommands` error falls into `throw err`, and the outer catch only logs `Telegram command sync failed`, so there is no 429 retry/backoff in current `main`.

## Risks / Open Questions

- Current `main` can still hit repeated `setMyCommands` failures on rapid restarts because the command-sync path has no 429 backoff handling.
- The PR's proposed exhaustion behavior appears unresolved: caching the sync hash after repeated 429s could suppress future retries even if `deleteMyCommands` already cleared the menu, so closing it would discard an open design discussion rather than confirm a settled fix.
