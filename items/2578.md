## Summary

本 Issue 详细报告了 flusher_otlp 无法向 OTLP gRPC 后端转发 trace 的问题，提供了完整的交叉验证链路与配置细节。经核查，用户配置中未显式声明 gRPC 明文传输（Insecure/TLS）参数，且 Endpoint 携带 http:// 前缀可能导致 gRPC 客户端初始化或协议协商失败。该问题属于有效的配置咨询与潜在缺陷报告，需维护者介入确认插件默认行为、补充配置指引或修复连接失败时的日志静默问题，不符合自动关闭条件。

## Best Possible Solution

建议维护者或贡献者核查 flusher_otlp 插件的 gRPC 客户端初始化逻辑：1. 确认是否需显式配置 Insecure: true 以支持明文传输，并在文档中补充 OTLP/gRPC 明文与 TLS 的标准配置示例；2. 检查 Endpoint 解析逻辑是否兼容 http:// 前缀，或明确提示用户去除 scheme；3. 若确认为连接失败时日志静默，请增加 gRPC dial 或 send 阶段的 error 日志输出。待配置方案明确或用户验证通过后关闭。

## Evidence

- **config_analysis:** 用户提供的 flusher_otlp 配置中仅包含 Endpoint、Timeout 和 WaitForReady，未配置 TLS 或 Insecure 字段。gRPC 客户端默认启用 TLS，若下游为明文端点且未显式配置 Insecure: true，将导致握手失败且可能无明确错误日志。
