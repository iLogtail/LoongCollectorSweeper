## Summary

本 Issue 详细报告了 flusher_otlp 向 OTLP gRPC 后端转发 trace 失败的问题，提供了完整的交叉验证链路与配置。经核查，该问题涉及 gRPC 客户端 Endpoint 解析与明文传输配置，属于有效的技术咨询与潜在缺陷排查。当前 main 分支未见明确修复或文档说明，且创建不足 60 天，信息充分，故保持开启。

## Best Possible Solution

建议维护者或贡献者核查 flusher_otlp 插件的 gRPC 客户端初始化逻辑：1. 确认 Endpoint 字段是否兼容 http:// 前缀，或需明确使用 host:port 格式；2. 检查是否需显式配置 Insecure: true 以支持 gRPC 明文传输，并在文档中补充标准配置示例；3. 若连接失败时日志静默，请增加 gRPC dial/send 阶段的 error 日志输出。可邀请用户补充 flusher 内部指标或尝试调整 Endpoint 格式与 Insecure 配置进行验证。待配置方案明确或代码/文档更新后关闭。

## Evidence

- **issue_reproduction:** 用户提供了完整的 Docker Compose 环境与 telemetrygen 验证链路，确认 service_otlp 接收正常但 flusher_otlp 转发 gRPC 失败，下游无数据且无错误日志。
- **config_analysis:** 用户配置中 Endpoint 使用了 http:// 前缀指向 4317 gRPC 端口，且未显式配置 Insecure 明文传输选项。flusher_otlp 插件可能因 scheme 解析或默认 TLS 握手失败导致静默丢弃，需维护者确认插件行为与文档示例。
