## Summary

用户反馈 flusher_otlp 无法将接收到的 trace 转发至 OTLP/gRPC 后端，已交叉验证输入端（service_otlp）与下游后端（OTel Collector/Phoenix）均工作正常。问题聚焦于 flusher_otlp 的 Endpoint 配置格式、协议自动识别逻辑或 gRPC 客户端初始化行为。当前为配置咨询与潜在缺陷排查阶段，缺乏足够运行日志或代码证据表明属于已知缺陷或已修复，需维护者或社区介入确认插件配置规范与默认协议回退策略。

## Best Possible Solution

建议维护者核查 flusher_otlp 源码中 Endpoint 解析与 Protocol 默认逻辑，确认是否需显式配置 Protocol: grpc 或调整 Endpoint 格式（如去除 http:// 前缀）。请在评论中提供针对 gRPC 后端的正确配置示例，并建议用户开启 flusher_otlp 详细调试日志或提供容器内网络连通性测试（如 grpcurl）结果。待配置澄清或确认为代码缺陷后，再决定关闭或转为 Bug 跟踪。

## Evidence

- **issue_config_analysis:** 用户配置中 Traces.Endpoint 尝试了 http://otel-collector:4317 与裸主机名格式，但未显式声明传输协议。OTLP 输出插件通常需根据 Protocol 字段或 URL Scheme 切换 HTTP/gRPC 客户端，若默认回退为 HTTP 则向 4317(gRPC) 端口发送请求会静默失败或协议不匹配。
