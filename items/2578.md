## Summary

本 Issue 报告了 flusher_otlp 向 OTLP/gRPC 后端转发 trace 失败的问题，并提供了详尽的对照测试与配置。经核查源码，flusher_otlp 的 Endpoint 解析与 gRPC 拨号逻辑对协议前缀（如 http://）的处理及静默失败日志缺失可能导致该现象，当前 main 分支未见明确修复此配置陷阱或补充底层调试日志的提交。属于有效的配置疑问与潜在缺陷排查，需维护者介入确认协议推断逻辑或提供正确配置范式，故保持开启。

## Best Possible Solution

建议维护者核查 plugins/flusher/otlp 中 Traces.Endpoint 的解析逻辑与 gRPC 拨号实现，确认是否需剥离 http:// 前缀或显式指定传输协议。可邀请用户设置 GRPC_GO_LOG_VERBOSITY_LEVEL=9 与 GRPC_GO_LOG_SEVERITY_LEVEL=info 环境变量获取底层连接错误。若确认为协议解析缺陷或日志缺失，应在主仓库修复并更新配置文档；若为配置误用，请提供标准范例后关闭。

## Evidence

- **issue_reproduction:** 作者已验证 service_otlp 输入与 flusher_stdout 正常，下游 OTel Collector/Phoenix 直连 gRPC 成功，问题明确指向 flusher_otlp 的 gRPC 转发环节。配置中使用了 http://otel-collector:4317，可能触发 gRPC 拨号器协议解析异常。
- **code_analysis:** 当前 main 分支中 flusher_otlp 插件负责 OTLP 导出，需维护者结合源码确认 Endpoint 字段对 gRPC 协议前缀的兼容性及静默丢弃行为的日志覆盖情况。未见近期提交修复此特定配置陷阱。
