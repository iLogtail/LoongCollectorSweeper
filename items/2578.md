## Summary

本 Issue 详细报告了 flusher_otlp 向 OTLP/gRPC 后端转发 trace 失败的现象，并提供了完整的对照测试链路与配置。问题已明确隔离至 flusher_otlp 的 gRPC 客户端初始化或 Endpoint 协议解析环节。当前缺乏底层连接调试日志，且 Endpoint 配置中 http:// 前缀对 gRPC 拨号器的影响需维护者结合源码确认。属于信息充分的有效配置疑问与潜在缺陷排查，证据不足以关闭。

## Best Possible Solution

建议维护者核查 plugins/flusher/otlp 源码中 Traces.Endpoint 的解析与拨号逻辑，确认 gRPC 模式下的正确配置范式（如是否需移除 http:// 前缀或显式指定 Protocol 字段）。可邀请用户开启 gRPC 底层调试环境变量（如 GRPC_GO_LOG_VERBOSITY_LEVEL=99）以获取静默失败原因。若确认为协议推断缺陷或错误日志缺失，应在主仓库修复并补充文档；若为配置误用，请提供正确范例并更新配置指南后关闭。

## Evidence

- **issue_reproduction:** 作者已验证 service_otlp 输入正常且 flusher_stdout 可打印 span，下游 OTel Collector 与 Phoenix 直连 gRPC 均成功，问题明确指向 flusher_otlp 的 gRPC 转发环节。
- **configuration_analysis:** 用户配置 Traces.Endpoint 使用了 http://otel-collector:4317，gRPC 拨号器可能因 http:// 前缀解析失败或协议不匹配导致静默丢弃，当前上下文未见针对该配置错误的明确拦截日志，需维护者确认正确配置范式或修复协议推断逻辑。
