## Summary

Keep open. Current main has minimal single-pod Kubernetes deployment support, but it does not implement the requested secure Kubernetes ServiceAccount Trust pairing path for autoscaled node-host workers.

## Best Possible Solution

Keep this issue open until OpenClaw implements or explicitly supersedes an opt-in Kubernetes ServiceAccount Trust path across gateway config/schema, WebSocket connect protocol, client and node-host credential handling, TokenReview validation with namespace/service-account/audience/TLS policy, tests, docs, and deployment assets. If the distributed runtime roadmap is the intended home, designate a narrower canonical issue that explicitly covers K8s worker authentication and pairing semantics.

## Risks / Open Questions

- This is security-sensitive gateway authentication and pairing work; any implementation should fail closed and get maintainer/security review before bypassing manual device pairing.
- The current plugin surface does not appear sufficient to safely replace connect-time worker authentication outside the core gateway protocol, so treating this as ClawHub-only would likely strand the requested behavior.
- The deployment-manifest part overlaps broader distributed-runtime work; maintainers should decide whether this issue remains the narrow K8s auth tracker or is explicitly linked to a more specific canonical design item.
