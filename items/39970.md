## Summary

Current main only partially covers this PR. Shared outbound delivery now carries `audioAsVoice`, and legacy iMessage text chunking is configured, but the actual `extensions/imessage` outbound path still has no `sendPayload` handler and drops `audioAsVoice` before `sendMessageIMessage`. The PR is stale against moved files, but the underlying iMessage bug remains actionable.

## Best Possible Solution

Keep the PR open and ask for a narrowed rebase/rewrite against `extensions/imessage`. The best fix is an iMessage-owned `sendPayload` path on the current plugin adapter that forwards `audioAsVoice` through `channel.runtime.ts` into `send.ts`, suppresses the generic `<media:audio>` placeholder for voice-intent audio, sends visible caption text separately when needed, preserves configured text chunking for text-only payloads, and adds focused iMessage regression tests.
