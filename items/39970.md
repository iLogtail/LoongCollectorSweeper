## Summary

Keep this PR open. Current main has the shared outbound `audioAsVoice` plumbing and iMessage text chunking, but the bundled iMessage plugin still drops `audioAsVoice` before the low-level sender and still maps empty audio media to the generic `<media:audio>` placeholder. The PR is stale against the current `extensions/imessage` layout, but the central bug-fix remains useful.

## Best Possible Solution

Keep the PR open and narrow/rebase it onto `extensions/imessage`. The best fix is an iMessage-owned `sendPayload` path or equivalent adapter/runtime propagation that carries `audioAsVoice` through `extensions/imessage/src/channel.ts`, `extensions/imessage/src/channel.runtime.ts`, and `extensions/imessage/src/send.ts`, suppresses the generic `<media:audio>` placeholder for voice-intent audio, sends visible caption text separately when needed, preserves text chunking and reply-to semantics, and adds iMessage-specific regression tests.
