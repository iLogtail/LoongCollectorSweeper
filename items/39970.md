## Summary

Keep open. Current main has restored iMessage text chunking and generic audioAsVoice propagation, but it still does not implement this PR's iMessage-specific audioAsVoice delivery semantics. The iMessage plugin only exposes sendText/sendMedia, drops audioAsVoice before the runtime send call, and the low-level iMessage sender still converts audio-only media into generic placeholder text.

## Best Possible Solution

Keep the PR open and rework it against the current extensions/imessage implementation. The best path is to add an iMessage-owned sendPayload or equivalent delivery branch that preserves audioAsVoice into sendIMessageOutbound/sendMessageIMessage, sends the voice audio without placeholder/caption text, sends caption text separately, keeps existing text chunking for long messages, and adds regression tests under extensions/imessage plus the shared outbound delivery layer as needed.
