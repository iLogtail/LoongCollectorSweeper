## Summary

Current main does not implement the RFC's durable next-turn injection API or typed agent-turn preparation hook. The existing SDK has adjacent prompt hooks, plugin-accessible in-memory system events, and an internal runtime-context next-turn custom message path, but those do not provide the requested public host-owned durable queue, exactly-once dequeue, expiry, dedupe, disabled-plugin suppression, or stable preparation boundary. The related #71731 context is an open docs/RFC artifact, not a shipped implementation or superseding implementation tracker.

## Best Possible Solution

Keep the issue open as a core/plugin-SDK RFC. The best path is a focused implementation that adds a generic durable next-turn injection queue and typed `agent_turn_prepare` hook to the public SDK, wires consumption into agent turn preparation before retry/fallback/provider transforms, defines expiry/dedupe/disablement diagnostics, and lands fixture tests proving exactly-once next-turn delivery. #71731 can remain the design packet, but it is not a substitute for the implementation work.
