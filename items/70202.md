## Summary

Keep #70202 open. Current main has the underlying model surfaces: default primary model config, fallback chains, allowlisted `/model` picker entries, CLI `models fallbacks`, Control UI/session model switching, and the later `openclaw configure` model allowlist/fallback multi-select. However the actual `openclaw onboard` flow still prompts for one default model via `promptDefaultModel` and applies only `applyPrimaryModel`; it does not run the multi-select allowlist/fallback step that `openclaw configure` uses. The requested onboarding-time multi-model setup is therefore only partially satisfied by follow-up configuration paths, not implemented in onboarding itself.

## Best Possible Solution

Keep the issue open and implement a focused onboarding enhancement: after the default model prompt in `src/wizard/setup.ts`, reuse `promptModelAllowlist`, `applyModelFallbacksFromSelection`, and `applyModelAllowlist` from the configure flow so users can select multiple available models while keeping one primary default. Consider matching non-interactive flags for primary, allowlist, and fallback models so automated onboarding can express the same policy. The existing Control UI session picker and agents fallback editor should remain as follow-up configuration surfaces, not substitutes for onboarding.
