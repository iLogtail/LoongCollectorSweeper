## Summary

Keep open. Current main at 4a3030df9efa72c44bab567a4390ac0c4876a73f still does not implement PR #39921's central behavior: health snapshots have no runtime snapshot input, default probing is still implicit, gateway health still serves through the cached refresh path, and Discord still only publishes the startup connected patch when the Carbon gateway already reports connected.

## Best Possible Solution

Keep this PR open or consolidate it into a maintainer-chosen canonical health-runtime fix. The product path should thread runtime channel/account state into gateway health snapshots, make probe intent explicit, update cache/broadcast behavior without overwriting runtime-backed fields, remove the Discord startup `gateway.isConnected` gate for the successful-login connected patch, and keep focused regression coverage for both health snapshots and the Discord startup race.
