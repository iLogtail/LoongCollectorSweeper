## Summary

Keep this issue open. Current main still has the reported static 100000-token parent fork guard and the over-limit branch still only logs a WARN while starting the child session fresh. Related compaction infrastructure from #56224 and checkpoint/operator recovery work from #62146 are present, but they are not wired into the parent-fork overflow path.

## Best Possible Solution

Keep open and implement a targeted parent-fork overflow path in `src/auto-reply/reply/session.ts` and `src/auto-reply/reply/session-fork.ts`: either compact/checkpoint the parent before a guarded fork skip, or surface a one-time visible/operator warning when starting a fresh child, and consider making `resolveParentForkMaxTokens()` model-aware. Add focused coverage in `src/auto-reply/reply/session.test.ts` and update the session/config docs to describe the new behavior.

## Risks / Open Questions

- Auto-compacting during session initialization would need bounded timeout/cancellation behavior, especially given the related open compaction-timeout report #43661.
- A channel-visible warning should be one-time or rate-limited per parent/child context so busy Discord or Slack threads do not get noisy repeated notices.
- A model-aware default needs a clear dependency contract for resolving the active model context window at session-init time without loading provider runtimes too broadly.
