## Summary

Keep open. Current main still lacks the requested OpenShell generated-command preflight: OpenShell forwards the model command into the shared SSH sandbox helper, which wraps it as `/bin/sh -c` without rejecting literal placeholders, unterminated quotes, trailing escapes, or backticks first. The loop detector has some critical no-progress breakers, but generic repeated same-tool/same-args calls are still explicitly warning-level until the separate global no-progress breaker, and tests preserve that behavior. This is a core runtime safety request, not a ClawHub/plugin boundary item or a duplicate of the unrelated title-search matches.

## Best Possible Solution

Keep the issue open. The best maintainer path is a focused runtime hardening change: add a shared exec/OpenShell preflight before sandbox command construction that rejects literal generated placeholders in runnable positions and malformed shell structure, then update loop detection so repeated identical failed generic tool calls can become a clear critical block at a configured threshold. Add regression tests for OpenShell/SSH exec command construction and loop detection, and separately lint or rewrite bundled skill examples that look copy-executable but contain placeholders.

## Risks / Open Questions

- A command validator can create false positives for legitimate shell syntax if it is too broad; the fix should be narrow, tested, and scoped to generated-command hazards.
- Changing loop detection from warning to blocking can abort valid repeated workflows; the implementation should distinguish failed/no-progress repeats from intentional polling or idempotent reads.
