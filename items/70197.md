## Summary

Keep open. Current main still ships the default-enabled Bonjour gateway discovery plugin on @homebridge/ciao@1.3.6, has no local ciao patch or dependency upgrade, and the upstream 1.3.6 NetworkManager Windows path still runs the reported `arp -a | findstr` shell command every 15 seconds without `windowsHide`. Current docs/tests prove users can disable Bonjour as a workaround, but they do not fix the default Windows console flash reported in #70197. The provided related context also makes #70197 the fresher/canonical report after the stale #25856 and closed cross-reference #71188.

## Best Possible Solution

Keep #70197 open and fix the bundled Bonjour path rather than closing as stale or workaround-only. The best maintainer path is to land or consume an upstream @homebridge/ciao fix that passes `{ windowsHide: true }` on Windows exec/spawn paths; if upstream is not available quickly, add a narrow pnpm patch for `@homebridge/ciao@1.3.6` and include a regression check that the patched dependency no longer calls the Windows ARP shell pipeline without `windowsHide`. Preserve the existing `OPENCLAW_DISABLE_BONJOUR=1`, `openclaw plugins disable bonjour`, and `discovery.mdns.mode="off"` workarounds as operational mitigations, then verify on Windows gateway startup if feasible.
