## Summary

Keep PR #71612 open. Current main still records loop-detection history against the stable cron agent session key and `recordToolCall` only appends/trims by history size, so identical tool calls can accumulate across isolated cron runs. The PR is a focused, recent bug fix with a concrete reproduction and targeted regression tests; the related session-reset items do not supersede this tool-loop false-positive.

## Best Possible Solution

Keep the PR open for maintainer review. The best product fix is to scope tool-loop detection history to an actual run boundary for isolated cron jobs sharing `agent:<id>:cron:<jobId>`, with regression coverage in `src/agents/tool-loop-detection.test.ts`; the implementation can be this PR's idle-gap reset or a more explicit run-scoped diagnostic reset/key path.
