## Summary

Current main still records and detects tool-loop history against the stable cron agent session key, and `recordToolCall` only appends/trims history with no idle-gap or run-boundary reset. PR #71612 is a recent focused bug fix with a concrete repro and targeted regression tests, and the provided related items do not supersede this cron loop-detection false positive.

## Best Possible Solution

Keep PR #71612 open for maintainer review. The best product fix is to scope tool-loop history to a real run boundary for isolated cron jobs sharing `agent:<id>:cron:<jobId>`, either by passing the per-run key/run boundary into before-tool-call diagnostics or by accepting the PR's idle-gap reset with the added regression coverage.
