## Summary

Keep PR #71612 open. Current main still records and detects tool-loop history on the stable cron agent session key used during tool execution, and `recordToolCall` has no idle-gap or run-boundary reset. The PR is a focused, recent bug fix with a concrete repro and regression tests; the related items in the provided context cover session reset/rollover concerns but do not supersede this cron tool-loop false-positive.

## Best Possible Solution

Keep the PR open for maintainer review. The best product fix is to scope loop-detection history to a real run boundary for isolated cron jobs sharing `agent:<id>:cron:<jobId>`, with regression coverage in `src/agents/tool-loop-detection.test.ts`; the exact implementation can be the PR's idle-gap reset or a more explicit per-run diagnostic key/reset hook.
