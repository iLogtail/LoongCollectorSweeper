## Summary

Current main already implements heartbeat delivery to Telegram; the reported `skipped` / `target-none` result is the expected outcome when heartbeat delivery target is left unset, because heartbeat `target` defaults to `none` rather than auto-reusing Telegram.

## Risks / Open Questions

- The reporter may still want a product change where heartbeat auto-defaults to the last Telegram route, but current OpenClaw behavior is intentionally explicit and internal-only when `target` is unset.

## Close Comment

Closing from a Codex review because current main already implements Telegram heartbeat delivery, and the reported `skipped` / `target-none` state matches the documented default when no heartbeat delivery target is configured. `docs/gateway/heartbeat.md:25` and `docs/gateway/heartbeat.md:232` document that heartbeat `target` defaults to `none`, `src/infra/outbound/targets.ts:92` returns `reason: "target-none"` for that case, and `src/infra/heartbeat-runner.returns-default-unset.test.ts:358` plus `src/infra/heartbeat-runner.returns-default-unset.test.ts:433` cover both the unset-skip path and explicit Telegram routing. The latest shipped release `v2026.4.22` also contains this behavior. This looks like a configuration/support case rather than a missing Telegram transport fix.
