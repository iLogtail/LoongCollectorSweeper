## Summary

Keep open. The issue is recent, concrete, unprotected, and not fixed on current main. Current main still has the Kimi tagged-tool wrapper behavior under discussion, PR #71274 was closed unmerged as the wrong fix, and I did not find a shipped Kimi-specific regression test or implementation that prevents the reported repeated read/exec loop.

## Best Possible Solution

Keep #71273 open and ask for or add a focused reproduction: a captured Kimi anthropic-messages transcript showing the assistant/tool-result sequence that repeats read/exec with identical arguments. The eventual fix should target the verified failure path in `extensions/kimi-coding` or the provider replay/session loop boundary, and add a regression test proving final post-tool-result answers end with `stop` while valid tagged tool-call turns still become `toolUse`.
