## Summary

Keep open. Current main still has the Kimi Coding stream wrapper and tests in the area reported, but I did not find a shipped fix or regression test for this issue; the cross-referenced PR #71274 was closed unmerged. One nuance: the issue's stated root cause is broader than the current code, because `stopReason` is only rewritten after tagged Kimi tool-call text is parsed, not for every text-only final answer. That makes this a real investigation/fix item rather than a safe close.

## Best Possible Solution

Keep the issue open and reproduce against current main with a captured Kimi Coding Anthropic Messages transcript. If Kimi is emitting repeated tagged tool-call sections after tool results, fix `extensions/kimi-coding/stream.ts` in the provider-owned wrapper and add a focused regression test in `extensions/kimi-coding/stream.test.ts` for the final-answer-after-tool-result path. If the model emits text-only final answers with `stopReason: "toolUse"` or the runner replays previous tool calls, the fix likely belongs in the PI runner's stop-reason/tool-call normalization instead of the Kimi tagged-tool parser.
