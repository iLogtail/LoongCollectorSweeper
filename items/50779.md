## Summary

Keep open. Current Control UI handles several adjacent image cases, including assistant/generated managed images, user MediaPath/MediaPaths, assistant MEDIA: attachments, and data-URI/input_image blocks, but the specific read-tool result shape from #50779 is still not rendered as an inline preview. The read tool can return `{ type: "image", data, mimeType }` plus text like `Read image file [image/png]`; the live tool stream reduces tool results to text-only, chat.history strips raw image data from content blocks, and the grouped renderer does not recognize direct image blocks with `data`/`mimeType`. Related #52025 remains the focused implementation attempt, while #55671 and #67533 cover different generated/uploaded image paths.

## Best Possible Solution

Keep #50779 open and use it as the canonical user-visible bug unless maintainers prefer #52025 as the implementation tracker. A conservative fix should preserve or convert trusted read-tool image results into a renderable, bounded Control UI media reference, then add UI tests for direct `{ type: "image", data, mimeType }` tool-result blocks in both live tool cards and chat.history reloads. It should not rely on remote image fetching or expose arbitrary local paths beyond the existing assistant-media/managed-media safety model.
