## Summary

Keep PR #61335 open. Current main still contains the exact Usage-page hour slicing and selected-hour filtering paths that walk to local `:59:59.999` with `setToHourEnd`; a read-only runtime check confirms that this can move backward during the repeated local hour at DST fallback. The PR is a focused fix for a real current bug, not implemented on main and not superseded by the provided related items.

## Best Possible Solution

Keep the PR open and have it land a DST-safe, monotonic hour-boundary helper shared by Usage aggregation and selected-hour filtering, plus focused regression coverage for fallback and spring-forward behavior. Before merge, fix the test environment cleanup issue noted in review.
