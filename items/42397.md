## Summary

Keep #42397 open. Current main supports a single per-run `heartbeat.model` override and `HEARTBEAT.md` task intervals, but it does not support per-task model hints in `HEARTBEAT.md` or an escalation keyword. The related closed reports in the provided context address the global heartbeat model override bug, not task-level model selection.

## Best Possible Solution

Keep the issue open for maintainer product/API design. A complete fix would add a documented optional model/escalation field to `HEARTBEAT.md` task parsing, decide whether mixed-model due tasks are split into multiple model calls or resolved to one model for the tick, update heartbeat docs/config help, and add regression coverage for mixed cheap/primary task runs, skipped tasks, timestamp updates, and delivery behavior.

## Risks / Open Questions

- A per-task heartbeat model design needs a clear execution contract for mixed due tasks: one selected model for the whole tick, grouped model calls, or explicit re-execution/escalation semantics.
- Task timestamp updates and delivery/ack handling would need to remain correct if only some due tasks run under a different model or a second pass fails.
