## Summary

Keep #20950 open. Current main still only supports a single cron definitions store file (`jobs.json`) plus a derived `jobs-state.json` runtime sidecar. It does not expose `cron.storeDir`, `jobsDir`, `jobs.d`, directory scanning, or one-file-per-job load/save behavior. The related discussion points to #41206 as an open implementation candidate, while #70531 was closed into #20950 and #53581 handled only the complementary runtime-state split.

## Best Possible Solution

Keep #20950 open as the canonical product issue until maintainers land #41206 or a replacement. The eventual fix should add an additive, documented directory-backed cron definitions store with deterministic per-file loading, compatibility with existing `cron.store`/`jobs.json`, preservation of the current `jobs-state.json` runtime sidecar behavior, config schema/help/docs/doctor updates, and focused load/save/gateway tests.
