## Summary

Keep open. Current main implements the related cron definitions/runtime-state split, but it still uses one `cron.store` file plus a derived `jobs-state.json` sidecar. I found no `jobs.d` or `storeDir` config contract, and no loader path that enumerates one JSON file per job. Related #53581 is useful context, but it addressed state churn and explicitly described jobs.d as complementary rather than a replacement.
