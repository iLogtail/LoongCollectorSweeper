## Summary

Keep open. Current main still implements cron storage as one definitions file selected by `cron.store`, plus a derived `jobs-state.json` sidecar; there is no accepted `cron.storeDir`/`jobs.d` config, no directory enumeration of per-job JSON files, and no docs/tests for that store shape. Related PR #41206 appears to be the unmerged implementation track, while closed #53581 shipped only the complementary definitions/runtime-state split.

## Best Possible Solution

Keep #20950 open until directory-backed cron definitions are implemented or a maintainer makes a different product decision. The likely path is an additive core cron storage/config change, such as `cron.storeDir` or a store-backend contract, with deterministic per-file loading, compatibility with existing `cron.store`, state sidecar behavior, doctor/config schema/help/docs updates, and focused tests. PR #41206 should be evaluated or updated against that contract rather than treating the issue as cleanup-ready.
