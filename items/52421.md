## Summary

Keep #52421 open. Current main has strong tool_use/tool_result pairing repair in the transcript, compaction, truncation, and sanitizer paths, so the original compaction root cause is partly addressed. But the issue also reports raw Anthropic validation text being posted to Discord with no recovery, and current main still has a generic invalid_request_error path that can surface `LLM request rejected: ...` verbatim without a specific `unexpected tool_use_id` classifier, retry, or friendly session-recovery message.

## Best Possible Solution

Keep #52421 open but narrow it. The best current fix is to add a specific `unexpected tool_use_id` invalid_request_error classifier in the embedded runner/error-formatting path, mark the replay/session state as invalid, attempt one repair/sanitize-and-retry when feasible, and otherwise send a friendly message such as `Session context error - starting fresh from this point.` Add regression coverage proving the raw Anthropic validation string is not emitted to channel payloads.
