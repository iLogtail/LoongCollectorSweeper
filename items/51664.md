## Summary

Keep #51664 open. Current main still renders WebChat code-block Copy buttons, but the click handler still depends solely on `navigator.clipboard.writeText(code)` and swallows failures without an `execCommand('copy')` or textarea-selection fallback. The two related PRs explicitly aimed to add that fallback, but both are closed and unmerged, and current main/release code still shows the same direct Clipboard API path.

## Best Possible Solution

Keep the issue open and fix WebChat code-block copying by adding a shared clipboard helper for code-block Copy that first tries `navigator.clipboard.writeText`, then falls back to a temporary textarea plus `document.execCommand('copy')` or another supported fallback. Add a focused UI test for rejected `writeText` so the Edge/insecure-context path does not regress.
