## Summary

Keep #50563 open. Current main still has the WhatsApp inbound loss window described by the issue: the channel sends a read receipt before durable storage or completed processing, then hands the message to an in-process debouncer/handler path with memory-only dedupe. The KeepMyClaw comment is useful complementary recovery context, but it does not replace an OpenClaw inbound WAL or persist-before-ack implementation.

## Best Possible Solution

Keep the issue open and implement a WhatsApp inbound durability path: persist the normalized inbound message and routing metadata before sending read receipts or otherwise acknowledging user-visible receipt, replay unprocessed records on gateway startup/reconnect, commit only after handler/queue handoff reaches a durable state, and add regression tests for crash points before enrichment, before debouncer flush, and during agent processing. The external KeepMyClaw-style state snapshot idea can remain complementary, but OpenClaw still needs its own inbound message WAL or equivalent persist-before-ack contract.
