## Summary

Current main does not support closing #53858. The reported Nostr restart-loop root cause remains actionable: the Nostr bus still passes an array of filters into `SimplePool.subscribeMany`, while the locked `nostr-tools@2.23.3` API/source treats that parameter as a single `Filter` and pushes it into the relay filter list, producing the same double-wrapped REQ shape the reporter logged. The stop path also only closes the subscription, not the underlying SimplePool relay connections, so the orphaned-connection concern is not clearly fixed either.

## Best Possible Solution

Keep the issue open and fix it in the Nostr plugin. The likely maintainer path is to pass a plain `Filter` object to `SimplePool.subscribeMany` for inbound DMs and profile import, remove the `unknown as Parameters<...>` casts that hide the API mismatch, add a regression test that captures the actual filter argument or serialized REQ shape, and update `NostrBusHandle.close()` to close/destroy the underlying SimplePool relay connections so restart/stop cycles do not leave relay sockets behind.
