## Summary

Keep #10944 open. Current main still hardcodes Telegram `parse_mode: "HTML"` across normal sends, captions, edits, draft previews, and lower-level delivery helpers; the strict Telegram config schema/types do not expose `parseMode`; docs still describe fixed HTML behavior with plain-text fallback only. The latest issue discussion also contains a same-day maintainer/Codex review explicitly keeping this item open.

## Best Possible Solution

Keep #10944 open and implement the feature in the bundled Telegram plugin/config surface. A conservative maintainer path is a typed `channels.telegram.parseMode` setting, with per-account override only if maintainers want that granularity, wired through text sends, captions, edits, delivery helpers, and streaming previews. Update the strict schema, exported Telegram config types, generated config metadata/help, public Telegram docs, and focused tests for HTML, no-parse/plain text, and any supported MarkdownV2/Markdown mode.
