## Summary

Keep open. Current `main` still builds non-direct session keys without `accountId`, and Feishu group routing still feeds group/topic peer IDs into that shared key shape. That means multiple Feishu bot accounts in the same group can still collapse onto one session and reuse the same persisted `lastAccountId`/delivery context.

## Evidence

- **Group session keys still omit accountId:** `buildAgentPeerSessionKey()` includes `accountId` only for direct-message scope `per-account-channel-peer`. For non-direct peers it still returns `agent:<agentId>:<channel>:<peerKind>:<peerId>` with no account component, so group/channel sessions on the same peer still collide across accounts.
  - file: [src/routing/session-key.ts:130](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/routing/session-key.ts#L130)
  - command: `nl -ba src/routing/session-key.ts | sed -n '130,190p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Feishu group routing still uses shared group/topic peer IDs:** Feishu inbound handling passes `account.accountId` to routing, but the routed peer for groups is still just `groupSession.peerId ?? ctx.chatId`. With the current routing/session-key code, `main` and `work` accounts in the same Feishu group still resolve to the same base session key when the peer ID matches.
  - file: [extensions/feishu/src/bot.ts:576](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/extensions/feishu/src/bot.ts#L576)
  - command: `nl -ba extensions/feishu/src/bot.ts | sed -n '570,640p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Shared session entries still drive outbound account identity:** Session init persists `ctx.AccountId` into `deliveryContext.accountId` / `lastAccountId`, and outbound target resolution later falls back to the session entry's stored account. If two Feishu accounts share one session key, the most recent inbound turn can overwrite reply identity for later sends.
  - file: [src/auto-reply/reply/session.ts:581](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/auto-reply/reply/session.ts#L581)
  - command: `nl -ba src/auto-reply/reply/session.ts | sed -n '575,625p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Repo already has a multi-account group-session fix pattern, but only for WhatsApp:** WhatsApp adds an account-specific thread suffix for non-default group sessions via `resolveWhatsAppGroupSessionRoute()`. No analogous Feishu shim exists in `main`, which is further evidence this Feishu report is not already implemented.
