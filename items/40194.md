## Summary

Keep open. Current `main` still routes Feishu group traffic through session keys that omit `accountId`, so multiple Feishu bot accounts in the same group can still collide on the same session. I did not find evidence on `main` that this issue was fixed.

## Evidence

- **Feishu inbound routing passes accountId into route resolution:** The Feishu bot path resolves each inbound message with `channel: "feishu"`, `accountId: account.accountId`, and a group `peer` based on the Feishu conversation/group ID.
  - file: [extensions/feishu/src/bot.ts:590](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/extensions/feishu/src/bot.ts#L590)
  - command: `nl -ba extensions/feishu/src/bot.ts | sed -n '580,610p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Group session keys still ignore accountId:** `buildAgentPeerSessionKey` only includes `accountId` in the `direct` + `dmScope === "per-account-channel-peer"` branch. The group/channel path returns `agent:${normalizeAgentId(params.agentId)}:${channel}:${peerKind}:${peerId}` with no account segment, so Feishu group sessions from different accounts still collapse when agentId/channel/peerId match.
  - file: [src/routing/session-key.ts:130](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/routing/session-key.ts#L130)
  - command: `nl -ba src/routing/session-key.ts | sed -n '126,188p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Feishu conversation bindings are already account-scoped:** The Feishu thread-binding manager keys bindings as `${accountId}:${conversationId}` and looks them up by that composite key, which matches the reporter's root-cause analysis that account separation is expected at the Feishu layer but not preserved by the session-key layer.
  - file: [extensions/feishu/src/thread-bindings.ts:71](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/extensions/feishu/src/thread-bindings.ts#L71)
  - command: `nl -ba extensions/feishu/src/thread-bindings.ts | sed -n '68,76p;140,152p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Existing routing coverage only proves per-account isolation for DMs:** Current routing tests explicitly cover `dmScope=per-account-channel-peer` for direct messages, but I found no corresponding group-session account-isolation coverage. That aligns with the current implementation, which special-cases only direct chats.
  - file: [src/routing/resolve-route.test.ts:545](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/routing/resolve-route.test.ts#L545)

## Risks / Open Questions

- Multiple Feishu bot accounts in the same group can still share one `agent:<agentId>:feishu:group:<peerId>` session, which can mix identity, history, and reply context across apps.
- There is no strong evidence on current `main` of a shipped fix or regression test for per-account Feishu group-session isolation.
