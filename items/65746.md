## Summary

Keep open. Current main still routes the PDF tool through a registry-only resolver and does not include the PR's config fallback or regression test. The PR targets a real remaining gap, though the proposed implementation should address the review comment about normalized provider aliases before merge.

## Best Possible Solution

Keep the PR open or have a maintainer take it over: add a PDF/media model resolver that preserves the current normalized registry lookup, then falls back to `cfg.models.providers` using normalized provider/model matching like `findInlineModelMatch`; add regression tests for a registry miss and a provider-alias custom provider; then use it to close #65742.
