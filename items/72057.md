## Summary

Keep #72057 open. Current main still has a credible Google LLM proxy gap: the direct Gemini stream path uses `buildGuardedModelFetch`, but that helper calls `fetchWithSsrFGuard` without selecting `TRUSTED_ENV_PROXY`, while guarded fetch defaults to strict mode and ignores env proxy unless trusted mode is explicit. The reporter's contrast with Google TTS is supported by current code: the TTS path uses `postJsonRequest`, whose shared provider HTTP helper does auto-upgrade to trusted env-proxy mode when `HTTP_PROXY`/`HTTPS_PROXY` applies.

## Best Possible Solution

Keep the issue open and fix the model transport path, likely in `src/agents/provider-transport-fetch.ts`, by mirroring the conservative provider HTTP helper behavior: select `GUARDED_FETCH_MODE.TRUSTED_ENV_PROXY` for trusted operator-controlled model requests only when `shouldUseEnvHttpProxyForUrl(url)` applies, while preserving strict DNS pinning for `NO_PROXY`, `ALL_PROXY`-only, explicit proxy/TLS dispatcher policies, and private-network safety. Add targeted tests for Google LLM or `buildGuardedModelFetch` proving env proxy is used for `generativelanguage.googleapis.com` without regressing explicit proxy/TLS handling.
