## Summary

Keep #27574 open. Current main still has browser-control primitives and an admin `browser.request` gateway method, but no Control UI/Dashboard browser preview route, tab, or live panel. The existing #63926 is overlapping but narrower around login handoff/browser management, and the April 26 maintainer-side review comment already kept this issue open for the same core gap.

## Best Possible Solution

Continue tracking #27574. The best path is either a core Control UI browser preview backed by the existing `browser.request` and browser-control primitives, or a new supported Control UI panel contribution API followed by a browser-preview plugin. Coordinate with #63926 for login handoff/browser management, but do not treat it as a complete replacement for this broader live remote-debugging preview request.

## Risks / Open Questions

- A live browser preview/input panel would expose browser pixels, URL/title, console/page errors, and possibly manual controls, so it needs operator-admin auth, origin protections, rate limits, and clear remote-control semantics.
- If implemented as a plugin rather than core UI, OpenClaw likely needs a supported Control UI tab/panel contribution API first.
