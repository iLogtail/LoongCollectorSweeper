## Summary

Keep PR #70884 open. Current main still has the trusted-proxy WebSocket scope-clearing behavior the PR documents, and the current docs do not yet include the PR’s HTTP-only `x-openclaw-scopes` clarification, missing-device-identity scope warning, or troubleshooting note. The provided PR discussion is positive evidence rather than a close reason: Greptile found the docs accurate, and later PR commits addressed the review feedback about `dangerouslyDisableDeviceAuth` wording and the `allowInsecureAuth` exception.

## Best Possible Solution

Keep the PR open for normal docs review and merge or adapt it. The best docs outcome is to update `docs/gateway/trusted-proxy-auth.md` and `docs/gateway/protocol.md` so operators understand that `x-openclaw-scopes` is for trusted HTTP API requests, while trusted-proxy Control UI WebSocket sessions without device identity can connect but lose self-declared scopes unless using the documented break-glass or narrow local exception paths.
