## Summary

Keep PR #70884 open. Current main still has the documentation gap the PR targets: the trusted-proxy docs describe `x-openclaw-scopes` for trusted-proxy auth but do not explicitly limit it to HTTP or explain that WebSocket scopes without device identity are cleared. The PR’s central behavior claim is backed by current gateway code and tests, and the provided PR context shows review feedback was addressed in later commits.

## Best Possible Solution

Keep PR #70884 open for maintainer review and likely merge after the usual docs checks. The best product path is to document the actual Gateway boundary in `docs/gateway/trusted-proxy-auth.md` and `docs/gateway/protocol.md`: `x-openclaw-scopes` is an HTTP request header, while WebSocket scopes come from the Gateway connect handshake and are cleared when unbound to device identity except for the existing explicit preservation paths.
