## Summary

Keep open. Current main still has the gateway RPC CLI timeout bug this PR targets: shared gateway-rpc commands advertise a 30000 ms default, but the runtime still coerces `opts.timeout` with `Number(...)` and falls back through `callGateway` to 10000 ms for non-finite values. The PR discussion also shows useful follow-up fixes for partial numeric strings, empty values, and cron timeout side effects, so the work is still plausibly useful rather than obsolete.

## Best Possible Solution

Keep the PR open or supersede it with an adapted patch for current main: add strict positive-integer validation for gateway-rpc `--timeout`, preserve the missing-option default at 30000 ms, reject explicit empty/whitespace and partial numeric strings, add focused gateway-rpc timeout tests, and either make shared parser changes with audited consumers or keep a gateway-specific strict resolver.
