## Summary

Keep open. Current main still does not implement per-spawn tool restrictions for `sessions_spawn`. The issue remains a concrete core API/security gap: the tool schema has no `tools`/allow/deny/none parameter, the native spawn call does not forward a tool policy, child-session metadata has no policy carrier, and the subagent policy resolver only reads global/config-based subagent tool policy. Prior PR #23506 was closed unmerged, and related #23906 is a narrower open follow-up rather than a shipped replacement.

## Best Possible Solution

Keep #15032 open as the core API gap. The best implementation is an additive `sessions_spawn` policy parameter such as `tools: { allow?: string[], deny?: string[] }`, with optional `tools: "none"` if maintainers want to cover the local-model use case in #23906. Thread it through the tool schema, `SpawnSubagentParams`, child-session persistence, and `resolveSubagentToolPolicyForSession`; enforce deny-wins and narrow-only semantics so per-spawn policy cannot re-grant tools removed by profile, provider, sandbox, group, or hard subagent policy. Add focused unit/e2e coverage and update the public Sub-agents docs.
