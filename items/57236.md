## Summary

Keep PR #57236 open. Current main still forces a rebuild when `hasDirtySourceTree()` returns true, before reaching the mtime fallback that this PR is trying to use, and there is a current test asserting that dirty watched trees rebuild. The linked issue #57204 has a concrete reproduction, and the PR is a narrow fix with matching tests in the provided diff, though it needs rebasing/adaptation to the newer structured build-reason behavior on main.

## Best Possible Solution

Keep the PR open and ask for an updated branch that adapts the same core fix to current main: remove the dirty-tree rebuild short-circuit, let dirty watched trees fall through to source mtime comparison, update the explicit dirty reason tests, and keep regression coverage for dirty-but-not-newer skip and dirty-and-newer rebuild.
