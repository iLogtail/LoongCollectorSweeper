## Summary

Keep #72037 open. Current main has moved the bundled ACPX dependency to `acpx@0.6.1`, but the observable Codex ACP startup problem is still present: OpenClaw’s ACPX plugin unconditionally wraps the `codex` agent with `codex-acp-wrapper.mjs`, and that generated wrapper defaults to `node npm-cli.js exec --package @zed-industries/codex-acp@^0.11.1 -- codex-acp` or `npx --package @zed-industries/codex-acp@^0.11.1 -- codex-acp`. The package/lock dependency surface does not include `@zed-industries/codex-acp`, and the docs still describe target harness adapters being fetched on demand with `npx`. The related #60693 comment is directionally consistent but broader/less focused, and #71549 was closed unmerged and would not by itself prove the Codex adapter is bundled.

## Best Possible Solution

Keep this issue open as the focused Codex ACP packaging bug. The likely fix belongs in the bundled ACPX plugin/OpenClaw-acpx integration: make the default Codex ACP command resolve a staged local `@zed-industries/codex-acp` binary while preserving the isolated `CODEX_HOME` wrapper and explicit command override behavior, then add package/lock/runtime-dependency tests proving the adapter is bundled and the default wrapper no longer invokes npm/npx.
