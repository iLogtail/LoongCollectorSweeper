## Summary

Current main still drops the sender envelope on bare `/new` and `/reset` reply runs. The reset branch builds `inboundUserContext` but excludes it from `baseBodyForPrompt`, while the non-reset branch prepends it. Telegram does populate sender identity before this point, and `buildInboundUserContextPrefix` can render the `Sender (untrusted metadata)` block, so the reported product bug remains actionable in OpenClaw core.

## Best Possible Solution

Keep this issue open. The likely fix belongs in `src/auto-reply/reply/get-reply-run.ts`: include `inboundUserContext` in the bare reset prompt composition alongside `startupContextPrelude`, `baseBodyFinal`, and `softResetTail`, then add a regression test for bare `/new` or `/reset` with `SenderName`/`SenderId` proving the agent prompt contains `Sender (untrusted metadata)`.
