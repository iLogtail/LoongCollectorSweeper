## Summary

Keep open. Current main has improved Discord gateway readiness/reconnect supervision, but I do not see an implementation of the requested inbound catch-up/queue for messages sent before the Discord WebSocket is READY or before the OpenClaw message listener is registered. The existing queue only handles events after Carbon has delivered MESSAGE_CREATE into OpenClaw.

## Evidence

- **Discord client starts before message listener registration:** monitorDiscordProvider creates the Carbon Discord client and gateway first, then later builds createDiscordMessageHandler and registers DiscordMessageListener after command deployment and bot identity fetch. That leaves startup work between gateway/client creation and inbound listener registration.
  - file: [extensions/discord/src/monitor/provider.ts:937](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/extensions/discord/src/monitor/provider.ts#L937)
  - sha: [8e12c24d1721](https://github.com/openclaw/openclaw/commit/8e12c24d17218dc50023aaf5543df1b7c36520aa)
- **Initial Carbon client listeners do not include message listener:** createDiscordMonitorClient constructs the Carbon Client with listeners: [readyListener]; the message/reaction/thread listeners are added later by registerDiscordMonitorListeners.
  - file: [extensions/discord/src/monitor/provider.startup.ts:162](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/extensions/discord/src/monitor/provider.startup.ts#L162)
  - sha: [8e12c24d1721](https://github.com/openclaw/openclaw/commit/8e12c24d17218dc50023aaf5543df1b7c36520aa)
- **Inbound queue only queues delivered events:** createDiscordMessageHandler receives a DiscordMessageEvent from Carbon, claims a replay key, and enqueues that already-delivered event into the debouncer/worker. This does not queue messages that Discord never delivered while the gateway was not connected.

## Risks / Open Questions

- A future fix will likely need a Discord-specific READY/reconnect backfill using recent channel history plus the existing inbound replay guard, so replayed messages do not duplicate events Carbon already delivered.
- The cron-side symptom may be partly mitigated by newer gateway startup sequencing, but the issue's central inbound Discord message-loss request remains unimplemented.
