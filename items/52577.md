## Summary

Keep open. Current main has substantial Discord gateway supervision and in-process inbound worker queuing, but that queue only handles events after Carbon delivers a `MESSAGE_CREATE` to the registered listener. I did not find a startup/reconnect catch-up path that fetches recent Discord channel history or otherwise captures user messages sent during a fresh-boot WebSocket gap before READY/listener delivery.

## Best Possible Solution

Keep the issue open and implement this in the Discord plugin surface: either delay gateway connection until inbound listeners are registered, and for fresh boot/reconnect gaps persist last-seen Discord message IDs/timestamps per account/channel and fetch a bounded recent history window after READY. Replayed messages should pass through the existing preflight, authorization, debouncer, and inbound replay guard, with regression tests for missed startup-window messages and duplicate suppression.
