## Summary

Keep #55358 open. Current main still builds inbound Slack DM agent text from the Slack event `text` field plus attachments/files; it does not preserve or parse inbound Slack `blocks`/rich-text blocks for human DM messages. The only current 160-character slice is a system-event preview, but if Slack's DM event `text` is itself the 160-character preview described in the report, the full agent `RawBody`/`BodyForAgent` would still be truncated because it is derived from that same field.

## Best Possible Solution

Keep the issue open and implement a Slack inbound regression around a DM payload whose `text` is a 160-character preview while rich-text/blocks contain the full message. The likely fix belongs in the Slack plugin inbound contract: extend the event typing to carry inbound `blocks`, add a parser for supported Slack rich-text/plain text sections, and make `resolveSlackMessageContent` prefer the fuller reconstructed body when it is longer and source-equivalent. Also cover normal channel messages, DM messages, and Assistant `message_changed` wrappers so #55445 behavior does not regress.
