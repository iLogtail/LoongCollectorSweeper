## Summary

Keep open. Current `main` does not show a fix for inbound Slack DM truncation: the inbound body is still built from `message.text` plus attachment text/fallbacks, with no runtime use of Slack message blocks to recover longer content. The only exact-160 truncation in current code is a system-event preview string, not the stored inbound envelope, so the issue is not safely closable as implemented or non-reproducible.

## Evidence

- **Inbound Slack event model lacks block-based recovery:** `SlackMessageEvent` still exposes `text`, `files`, and `attachments`; there is no top-level inbound `blocks` field, and `message_blocks` exists only on `SlackAttachment`.

## Risks / Open Questions

- This review is code-inspection only; I did not run a live Slack reproduction because the task requires a read-only checkout and no environment-mutating setup.
