## Summary

Keep open. Current main supports video understanding through the separate media-understanding and `openclaw infer video describe` surfaces, but the requested direct video upload path through chat/message attachments remains image-only in the Control UI and Gateway chat pipeline. The existing issue comment already identified this distinction, and current main still matches that assessment.

## Best Possible Solution

Keep #27482 open until OpenClaw has first-class video chat attachments: widen the Control UI accept/type guards and previews, add a Gateway attachment/content model for video instead of `ChatImageContent` only, route accepted videos through provider-native video input or the existing media-understanding video path with capability and size checks, and add focused UI/Gateway/docs coverage for MP4, WebM, and MOV.
