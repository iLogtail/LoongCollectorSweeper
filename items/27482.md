## Summary

Keep #27482 open. Current main still supports video through separate CLI/media-understanding and generated-media surfaces, but the requested direct Control UI chat/message attachment path remains image-only and does not pass MP4/WebM/MOV payloads to provider-native video input.

## Best Possible Solution

Keep #27482 open until OpenClaw has first-class video chat attachments: widen the Control UI accept/type guards and previews, add a Gateway attachment/content model for video rather than `ChatImageContent` only, route accepted videos through provider-native video input or the existing media-understanding video path with model capability and size checks, and add focused UI/Gateway/provider/docs coverage for MP4, WebM, and MOV.
