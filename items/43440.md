## Summary

Keep #43440 open. Current main still runs legacy iMessage by having the Node/OpenClaw process spawn `imsg rpc`, and the user-facing docs still say Full Disk Access is required for the process context running OpenClaw/`imsg`. The requested independent imsg-owned FDA context is not implemented, the issue is recent, and the provided discussion has no duplicate, maintainer disposition, or linked ClawHub workaround.

## Best Possible Solution

Keep this open for maintainer product/security handling. The likely fix belongs in the iMessage plugin/runtime boundary: either run the legacy imsg database access through a stable OpenClaw-owned or imsg-owned macOS process context, document and support a launchd/daemon or signed-app bridge that survives Node upgrades, or explicitly retire this legacy path in favor of BlueBubbles with a maintainer decision. The fix should update `extensions/imessage/src/client.ts`, monitor/probe behavior, setup/doctor guidance, and public iMessage docs, with macOS TCC reproduction evidence.
