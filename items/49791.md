## Summary

Keep #49791 open. The original reporter found a configuration workaround for their immediate Feishu group `/think` failure, but the remaining reported UX bug is still present on current main: an unauthorized text/directive command in a Feishu group can dispatch and complete with no user-visible reply. A later 2026.4.15 report also says Feishu group mentions still produce no response with `groupPolicy: allowall`, so this should not be conservatively closed as implemented or stale.

## Best Possible Solution

Keep the issue open and fix the Feishu group text-command UX. The likely product path is to return a concise group-visible authorization notice for whole-message system/directive commands like `/think medium` when the group is allowed but the sender is not authorized for commands, then add Feishu/group regression tests around `CommandAuthorized=false`, directive-only `/think`, `groupPolicy: open`/legacy `allowall`, and the documented `allowFrom`/group sender configuration. Docs should also explain that group access and command sender authorization are separate controls.
