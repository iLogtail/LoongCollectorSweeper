## Summary

Keep open. Current main partially addresses the issue by prioritizing an explicit/preferred TTS provider first, but it still falls through to other providers when the selected provider does not handle a token, does not add selectedProvider to the parse context, and does not warn for unhandled selected-provider tokens. That leaves a meaningful part of #60131 unresolved.

## Evidence

- **Parser still walks fallback providers:** After parsing provider= into declaredProviderId, parseTtsDirectives builds an ordered list with the declared/preferred provider first, then still loops over every provider in that list and continues on unhandled tokens. Warnings are only copied from a provider that handled the token; there is no selected-provider-only path or unhandled-token warning.
