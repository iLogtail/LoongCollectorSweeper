## Summary

Keep open. Current main still exposes cron jobs through a global store and global Gateway/service CRUD/run paths without a trusted caller ownership scope. The issue also has concrete production reports and a maintainer-backed keep-open comment from April 26, 2026.

## Best Possible Solution

Keep tracking #26370 until core cron ownership is enforced. The likely fix is to pass a trusted caller scope from createCronTool into Gateway cron RPCs, enforce agentId/sessionKey ownership for list/update/remove/run/runs in Gateway/service code, preserve an explicit admin/all-jobs operator path, safely migrate or backfill existing global jobs, and add regression tests plus docs for multi-agent and multi-user cron isolation. Per-agent jobs.json is a viable storage implementation, but hard ownership checks are still needed if storage remains global.
