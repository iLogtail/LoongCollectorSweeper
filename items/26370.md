## Summary

Keep open. Current main at bc24b547 still does not enforce default per-agent or per-session cron ownership for list, update, remove, run, or runs, and cron storage/docs still describe a single Gateway-level jobs.json. The issue also has concrete production reports plus a latest maintainer comment explicitly keeping this open; recent cron changes only addressed delivery-context preservation, not ownership isolation.

## Best Possible Solution

Keep tracking #26370 until core cron ownership is enforced. The likely product/code path is to pass a trusted caller scope from createCronTool into Gateway cron RPCs, enforce agentId/sessionKey ownership for list/update/remove/run/runs in the Gateway/service layer, preserve an explicit admin/all-scope path for CLI/operator use, migrate or backfill existing global jobs safely, and add docs plus regression tests for multi-agent and multi-user cron isolation. Per-agent jobs.json is one viable implementation, but equivalent hard ownership filtering is still required if storage remains global.
