## Summary

Keep #71398 open. Current main still does not resolve the reported `secretref:/...` string form for QQ Bot `clientSecret`; it treats non-empty SecretInput strings as plaintext before SecretRef coercion, while the supported/docs-backed contract is structured `{ source, provider, id }` SecretRef objects. The existing maintainer-thread Codex comment also explicitly kept this open, and I found no current-main code, test, changelog, or release evidence that supersedes that conclusion.

## Best Possible Solution

Keep the issue open. Maintainers should decide whether `secretref:/...` was a supported legacy file-provider shorthand that needs shared migration/runtime compatibility, or invalid syntax that should fail validation/doctor with clear remediation. The likely fix belongs in shared SecretInput validation/migration plus QQ Bot regression tests for both `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`, with docs updated to show the supported structured file SecretRef form or the migration path.
