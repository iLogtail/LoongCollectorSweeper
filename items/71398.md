## Summary

Keep #71398 open. Current main still does not resolve the reported `secretref:/...` string form for QQ Bot `clientSecret`; the string is accepted as a plaintext SecretInput and would remain a literal secret value before QQ token fetch. I found no current-main implementation, test, docs update, release note, or canonical duplicate that closes the reported upgrade regression.

## Best Possible Solution

Keep the issue open. Maintainers should decide the compatibility contract, then either migrate/reject `secretref:/...` with a clear `openclaw doctor --fix` or validation diagnostic, or implement backward-compatible resolution in the shared SecretInput path without reinterpreting legitimate plaintext secrets. The likely code path is shared SecretInput migration/validation plus QQ Bot regression tests for both `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`, with docs and the SecretRef credential target registry aligned if QQ Bot SecretRef support is intended to be first-class.

The only provided issue comment is a maintainer-authored Codex review from 2026-04-26 that also kept the issue open for this exact reason.
- **QQ Bot uses the shared SecretInput resolver for clientSecret:** `resolveQQBotAccount()` resolves `accountConfig.clientSecret` through `adapter.resolveSecretInputString()` at the top-level and account-level paths, with no QQ Bot-specific parser for `secretref:/...`.

## Risks / Open Questions

- The main product decision is whether `secretref:/...` was a supported legacy file-provider shorthand that needs compatibility/migration, or invalid syntax that should be rejected with doctor guidance. Current behavior does neither for QQ Bot.
- QQ Bot `clientSecret` docs and schema advertise SecretRef-shaped input, but the canonical SecretRef credential surface and bundled channel secret-contract registry do not currently expose QQ Bot clientSecret targets, so a fix may need to align runtime collection, docs, tests, and target registry coverage.
