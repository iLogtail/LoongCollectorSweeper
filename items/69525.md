## Summary

Keep #69525 open. Current main still inherits `input.defaultAccountId` for message sends without checking whether the resolved target channel differs from the source provider, then threads that account id into plugin/core outbound delivery. The linked PR #69529 is relevant but remains open/unmerged in the provided context, so the issue is not implemented on main.

## Best Possible Solution

Keep the issue open and land a targeted core fix in `src/infra/outbound/message-action-runner.ts`: distinguish explicit `accountId` from `input.defaultAccountId`, detect allowed cross-provider sends when `toolContext.currentChannelProvider` differs from the resolved target channel, and avoid applying the source default account id in that case. Add a focused regression test proving the target plugin/direct outbound send receives `undefined`/no `params.accountId` for cross-provider sends unless the user explicitly supplied `accountId`.
