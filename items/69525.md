## Summary

Current main still appears to contain the reported bug. `runMessageAction` resolves `accountId` from an explicit param or `input.defaultAccountId`, writes it back into params, and passes it through the send path without checking whether the target channel differs from `toolContext.currentChannelProvider`. The latest release tag has the same code, and the related fix PR #69529 is open/unmerged in the provided context, so this should stay open.

## Best Possible Solution

Keep #69525 open and land a scoped core fix in `src/infra/outbound/message-action-runner.ts`: separate explicit `accountId` from `input.defaultAccountId`, detect `allowAcrossProviders` plus `toolContext.currentChannelProvider !== channel`, and avoid applying the source default accountId in that case. Add focused regression coverage under the outbound message-action runner tests for cross-provider sends with `allowAcrossProviders: true`, while preserving explicit target accountId and same-provider default behavior.

## Risks / Open Questions

- The fix must preserve explicit `accountId` behavior: if the tool call provides a target-channel accountId, core should still pass it through.
- The fix must preserve same-provider/session default behavior and agent binding fallback while only suppressing the source session default for true cross-provider sends.
- A regression test should cover both plugin-handled `send` and the core outbound delivery path, because both currently receive the resolved accountId.
