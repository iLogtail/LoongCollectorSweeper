## Summary

Keep #69525 open. Current main still resolves the target channel and then falls back to the source session `defaultAccountId`, with no cross-provider guard, and that account id is forwarded into plugin/core outbound send paths. The linked PR #69529 is an open, unmerged fix proposal rather than shipped behavior on main.

## Best Possible Solution

Keep the issue open and land a narrow core fix in `src/infra/outbound/message-action-runner.ts`: distinguish explicit `accountId` from `input.defaultAccountId`, suppress the source default on allowed cross-provider sends, then let target-channel agent binding or plugin default resolution handle the account. Add a regression test in the message-action runner send/plugin dispatch coverage proving cross-provider sends do not pass the source account id while same-provider sends and explicit account ids still work. The linked PR #69529 can be used as a starting point, but its unrelated bundled fixes should be split or reviewed separately.
