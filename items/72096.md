## Summary

Keep open. The reporter provided concrete DREAMS.md samples, and current main still has the reported shape: light dreaming selects capped recent short-term entries from the lookback window and appends a narrative whenever any staged entry exists. There is session-ingestion checkpointing and cron/dreaming-transcript filtering, but there is no per-phase last-emitted cutoff or same-body diary dedupe that would prevent repeated DREAMS.md entries across hourly cycles. The related #68882 filter is a different promotion-candidacy layer, and #70737 decoupled cron from heartbeat without adding diary emission dedupe.

## Best Possible Solution

Keep the issue open for a memory-core fix. The likely path is to add a per-phase diary emission checkpoint keyed by candidate keys/snippet hashes and/or last successful emit time, then skip light/REM narrative append when the phase input is unchanged. Add regression coverage for repeated light sweeps with unchanged input and a commit-summary-like snippet, plus a deliberate policy decision on whether the existing manual diary dedupe should ignore timestamps for same-body entries.
