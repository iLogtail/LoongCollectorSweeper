## Close Comment

Thanks for the suggestion. I did a Codex review against current `main`, and this looks better scoped for ClawHub than additional OpenClaw core surface.

Current main already covers the generic core backup cases with `openclaw backup create` / `openclaw backup verify`, including `--only-config` and `--no-include-workspace` (`src/cli/program/register.backup.ts:20`, `docs/cli/backup.md:11`). The docs also already recommend a separate workspace-focused backup path for the user-authored memory files you called out: keep the workspace in a private git repo (`docs/concepts/agent-workspace.md:139`, `docs/help/faq.md:610`).

The remaining request here is a more opinionated, selective “essential markdown / allowlist export” workflow. The project vision explicitly says core should stay lean, optional capability should usually ship as plugins, and new skills should go to ClawHub first (`VISION.md:54`, `VISION.md:69`; `docs/tools/clawhub.md:9`). On that basis, I’m closing this as ClawHub-scope rather than core-scope. If someone wants to build the selective backup flow, ClawHub is the intended path for it.
