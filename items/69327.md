## Summary

Keep #69327 open. Current main does pass a child agent's own `sandbox.docker.env` into Docker create and sandbox exec, but the specific subagent inheritance gap described in the issue comment is still present: spawned context carries workspace/group metadata only, `sandbox="inherit"` is treated as a sandboxing requirement rather than parent container/config inheritance, and the child sandbox is resolved fresh from the target agent config. The recreate/setupCommand part is also not conclusively obsolete because the recreate command removes registry-listed runtimes and there is no regression coverage proving stale subagent runtime/env/setup refresh after recreate.

## Best Possible Solution

Keep the issue open and narrow it into targeted regression coverage plus a fix for native `sessions_spawn` in sandboxed sessions. Maintainers should decide whether `sandbox="inherit"` should thread selected parent `SandboxContext.docker.env`/runtime config into same-agent child spawns, or whether the current target-agent-only semantics should be documented and exposed under a clearer contract. A good fix would add coverage proving a sandboxed parent with Docker env/setup changes spawns a child whose sandbox exec sees the env, and that `openclaw sandbox recreate --agent <id>` invalidates the `agent:<id>:subagent:*` runtime used afterward.
