## Summary

Keep #69327 open. Current main has the generic Docker sandbox env/setupCommand path and agent-scoped recreate filtering, but it still does not thread the requester’s resolved sandbox/Docker context into `sessions_spawn(sandbox="inherit")`. The issue remains actionable because the reported subagent-specific propagation/recreate behavior is not conclusively fixed by current source or tests.

## Best Possible Solution

Keep the issue open and narrow it into a reproducible regression test around `sessions_spawn(runtime="subagent", sandbox="inherit")` from a sandboxed parent with configured `sandbox.docker.env` and changed `setupCommand`. The likely code path is to explicitly thread the requester’s resolved sandbox Docker defaults into subagent spawn, or document and enforce that child runs use only the target agent’s sandbox config, then add tests for same-agent, cross-agent, shared-scope, and recreate behavior.

## Risks / Open Questions

- The issue combines multiple symptoms; same-agent target config env injection may already work, while parent-context inheritance and stale runtime/bootstrap behavior remain unproven.
- A fix needs to define product semantics for `sandbox="inherit"`: inherit only sandboxed/unsandboxed status, inherit the target agent config, or inherit the requester’s resolved container/config defaults.
