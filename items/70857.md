## Summary

Keep #70857 open. The report is recent, concrete, and still plausible on current main: the session-store lock path still derives a 15s max hold for the default 10s store lock timeout, emits the exact warning shape from the issue, and can hold the lock across async session-store work. Related items are useful context, but the open PR #71903 targets a different orphan/self-PID lock cleanup case and does not fully cover the Windows startup plus sessions.json.lock latency report.

## Best Possible Solution

Keep the issue open and treat it as an active Windows/session-store lock latency bug. The likely maintainer path is to collect `OPENCLAW_GATEWAY_STARTUP_TRACE=1` output and session-lock diagnostics from Windows, then narrow or split session-store critical sections so expensive async work and model/catalog/channel startup do not hold `sessions.json.lock`; separately evaluate #71903 for the orphan/self-PID cleanup case without using it as the canonical close for this report.
