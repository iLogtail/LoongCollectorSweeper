## Summary

Keep open. Current `main` still holds session-file locks across the full embedded agent attempt and also runs queued async session-store mutations while the file lock is held, so the reported long-held `sessions.json.lock` and latency remain plausible. I did not find a current-main fix specifically addressing Windows startup/reply latency or issue #70857.

## Evidence

- **Embedded run takes the session write lock before later setup work:** `runEmbeddedPiAgent` acquires `sessionLock` before context/prompt/tool/bootstrap setup continues, so expensive pre-stream work happens while the lock is already held.

- **Lock is released only during final cleanup at attempt end:** The same `sessionLock` is passed into `cleanupEmbeddedAttemptResources(...)` in the final cleanup block, which means the lock lifetime spans the full attempt rather than a narrow transcript-write section.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:3125](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/agents/pi-embedded-runner/run/attempt.ts#L3125)
