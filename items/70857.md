## Summary

Keep #70857 open. Current main still contains the exact session-store lock budget that matches the reported `max=15000ms` warning, and the related open PR is not merged and does not clearly cover the Windows startup/session-store lock symptom. This remains a concrete, recent Windows regression report with useful timing and lock evidence.

## Best Possible Solution

Keep the issue open and investigate the Windows startup path on current main. The likely maintainer path is to instrument session-store lock queue/owner timing around channel startup, verify whether any store mutator can hold `sessions.json.lock` across slow channel/provider work, and then either narrow that lock scope or adjust the session-store lock watchdog semantics. Treat #71903 as related evidence, not a canonical fix for this Windows report unless it lands with direct coverage for the `sessions.json.lock` startup symptom.
