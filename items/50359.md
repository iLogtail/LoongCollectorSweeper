## Summary

Keep open. Current `main` still drops embedded image tool results when the result has image content but no `details.path`; the PR’s start-args fallback is not present on `main`, and the current tests still encode the broken behavior.

## Evidence

- **Current extractor still stops at `details.path`:** `extractToolResultMediaArtifact()` only falls back to `details.path` when image content exists and otherwise returns `undefined`; there is no fallback to `tool_execution_start` args such as `path`, `file_path`, or `filePath`. That means the reported embedded `read` case is still unhandled on `main`.
