## Summary

Keep open. Current `main` still evicts any non-active session key during save-time maintenance and disk-budget cleanup, with no protection for `:topic:` or `:thread:` sessions. Code inspection also found a load-time oversized-store pruning path with no preserve logic, so the report is still actionable and may be understated.

## Evidence

- **Save-time maintenance preserves only the active session key:** `saveSessionStoreUnlocked()` builds `preserveSessionKeys` from `opts?.activeSessionKey` only, then passes that set into both `pruneStaleEntries()` and `capEntryCount()`. There is no topic/thread-specific exemption.
  - file: [src/config/sessions/store.ts:283](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/config/sessions/store.ts#L283)
  - command: `nl -ba src/config/sessions/store.ts | sed -n '240,320p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Prune and cap delete any non-preserved entries:** `pruneStaleEntries()` deletes any entry older than the cutoff unless its key is in `preserveKeys`, and `capEntryCount()` sorts all non-preserved keys by `updatedAt` then deletes the overflow tail. Neither function checks session type, key shape, `chatType`, `:topic:`, or `:thread:`.
  - file: [src/config/sessions/store-maintenance.ts:156](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/config/sessions/store-maintenance.ts#L156)
  - command: `nl -ba src/config/sessions/store-maintenance.ts | sed -n '150,320p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Disk-budget cleanup skips only the active session:** `enforceSessionDiskBudget()` sorts store keys oldest-first and deletes entries until the high-water target is reached. The only exemption in that loop is `activeSessionKey`; topic/thread keys are not treated specially.
  - file: [src/config/sessions/disk-budget.ts:285](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/config/sessions/disk-budget.ts#L285)
  - command: `nl -ba src/config/sessions/disk-budget.ts | sed -n '192,340p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Missing session-store entry causes a new session to be created:** `initSessionState()` reads `sessionStore[sessionKey]`; if there is no reusable entry, it generates a new `sessionId` with `crypto.randomUUID()` and sets `isNewSession = true`. That matches the reported context loss when a topic/thread pointer is evicted.
  - file: [src/auto-reply/reply/session.ts:404](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/auto-reply/reply/session.ts#L404)
  - command: `nl -ba src/auto-reply/reply/session.ts | sed -n '380,560p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **There is also a load-time oversized-store pruning path with no preserve logic:** `loadSessionStore()` applies `pruneStaleEntries()` and `capEntryCount()` when an enforced store exceeds `maxEntries`, and it does so without any `preserveKeys` at all. That broadens the risk beyond the save-time paths called out in the report.
  - file: [src/config/sessions/store-load.ts:131](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/config/sessions/store-load.ts#L131)
  - command: `nl -ba src/config/sessions/store-load.ts | sed -n '123,149p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)

## Risks / Open Questions

- A future fix likely needs to cover both save-time maintenance and load-time oversized-store maintenance; fixing only `saveSessionStoreUnlocked()` would leave another eviction path in place.
- The current default maintenance mode is still `enforce`, so affected users do not need custom config to hit these code paths.
