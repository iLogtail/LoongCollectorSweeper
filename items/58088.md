## Summary

Current `main` still allows session-store maintenance to evict non-active Telegram topic/thread sessions. The code only preserves `activeSessionKey` during save-time maintenance, load-time maintenance preserves nothing, and the disk-budget path still deletes oldest entries without any topic/thread or `chatType` exemption.

## Evidence

- **Topic sessions are intentionally separate long-lived session keys:** Docs state Telegram forum topics add `:topic:<threadId>` so each topic has its own session, and the Telegram parser canonicalizes those ids as `<chatId>:topic:<topicId>`. That matches the issue's premise that topic sessions are persisted as normal store entries.
  - file: [docs/channels/groups.md:73](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/docs/channels/groups.md#L73)
  - command: `nl -ba docs/channels/groups.md | sed -n '68,78p'`
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Save-time prune/cap only preserve the active session key:** `saveSessionStore` builds `preserveSessionKeys` from `opts.activeSessionKey` only, then passes that set into both `pruneStaleEntries` and `capEntryCount`. There is no additional preservation for `:topic:`, `:thread:`, or group/thread chat metadata.
  - file: [src/config/sessions/store.ts:284](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/config/sessions/store.ts#L284)
  - command: `nl -ba src/config/sessions/store.ts | sed -n '250,380p'`
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Maintenance helpers do not special-case topic/thread/chatType entries:** `pruneStaleEntries` and `capEntryCount` only honor an optional `preserveKeys` set. They do not inspect key shape or `SessionEntry.chatType`, even though `SessionEntry` does carry `chatType`.
  - file: [src/config/sessions/store-maintenance.ts:156](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/config/sessions/store-maintenance.ts#L156)
  - command: `nl -ba src/config/sessions/store-maintenance.ts | sed -n '140,330p'`
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Load-time maintenance preserves nothing:** When loading an oversized store, `loadSessionStore` directly calls `pruneStaleEntries` and `capEntryCount` with `{ log: false }` only. No `preserveKeys` are supplied there, so idle topic/thread sessions are fully eligible for eviction on load.
  - file: [src/config/sessions/store-load.ts:132](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/config/sessions/store-load.ts#L132)

Integration tests still assert oldest-first session eviction for both `maxEntries` and `maxDiskBytes`, which is consistent with the reported failure mode rather than a fix.
