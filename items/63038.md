## Summary

Keep open. Current `main` still closes the WhatsApp socket without draining debounced inbound work, and the PR as discussed is not yet strong enough to auto-close because the proposed `flushAll()` approach can miss late or already-running keyed flushes during shutdown.

## Evidence

- **Current main still closes immediately:** `monitorWebInbox().close()` detaches listeners and immediately calls `closeInboundMonitorSocket(sock)`; there is no debounce drain step on current `main`. That means buffered inbound batches can still outlive the socket teardown.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:755](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/extensions/whatsapp/src/inbound/monitor.ts#L755)
  - command: `nl -ba extensions/whatsapp/src/inbound/monitor.ts | sed -n '754,760p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Debouncer on main has no flush-all primitive:** `createInboundDebouncer()` currently returns only `{ enqueue, flushKey }`. The proposed shutdown behavior is not already implemented on `main`.
  - file: [src/auto-reply/inbound-debounce.ts:235](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/auto-reply/inbound-debounce.ts#L235)
  - command: `nl -ba src/auto-reply/inbound-debounce.ts | sed -n '141,147p;235,236p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **In-flight upsert handlers can enqueue after detach:** `handleMessagesUpsert()` is async and does multiple awaited steps before `debouncer.enqueue(...)` (`normalizeInboundMessage`, `maybeMarkInboundAsRead`, `enrichInboundMessage`, dedupe). Detaching the event listener does not cancel a handler already running, so a one-shot `flushAll()` snapshot can miss late enqueues from an in-flight callback.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:638](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/extensions/whatsapp/src/inbound/monitor.ts#L638)
  - command: `nl -ba extensions/whatsapp/src/inbound/monitor.ts | sed -n '638,677p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Active keyed work can exist after buffer removal:** The debouncer stores active per-key work in `keyChains`, and `flushBuffer()` deletes the buffer entry before awaiting `buffer.task`. That matches the review concern that `flushAll()` over only `buffers.keys()` can return while keyed flush work is still running.
  - file: [src/auto-reply/inbound-debounce.ts:84](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/auto-reply/inbound-debounce.ts#L84)

Code inspection on current `main` supports those concerns, so the PR should stay open for maintainer judgment rather than be conservatively closed.
