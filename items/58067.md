## Summary

Keep open. Current `main` still treats `sessions_yield` as an intentional end-of-turn handoff, `agent.wait` still only returns a terminal snapshot, the chat UI clears its active-run state as soon as the parent run ends, and native subagent `streamTo:"parent"` is still rejected. A same-day `main` change only updated agent guidance to prefer `sessions_yield`; it did not add parent-surface progress events, a waiting indicator, or subagent stream forwarding.

## Evidence

- **`sessions_yield` is documented as ending the parent turn:** The public session-tools doc still says `sessions_yield` intentionally ends the current turn so the next message can be the follow-up event, which matches the issue's dormant-parent behavior rather than a streamed/waiting parent session.
  - file: [docs/concepts/session-tool.md:87](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/docs/concepts/session-tool.md#L87)
  - command: `nl -ba docs/concepts/session-tool.md | sed -n '82,92p'`
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **`agent.wait` still has terminal-only semantics:** The agent-loop doc says `agent.wait` waits for lifecycle end/error and returns only the terminal snapshot. I did not find any current protocol path here for intermediate progress or forwarded subagent streaming during the wait.
  - file: [docs/concepts/agent-loop.md:40](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/docs/concepts/agent-loop.md#L40)

- **Control UI busy state depends on `chatRunId`:** The chat host considers the UI busy only while `chatSending` or `chatRunId` is set. Once the parent run ends, there is no separate yielded-wait state to keep a spinner/progress indicator alive.
  - file: [ui/src/ui/app-chat.ts:63](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/ui/src/ui/app-chat.ts#L63)

It can append a later `final` from another run (for example a subagent announce), but there is no persistent waiting state between the parent ending and that later completion message arriving.
