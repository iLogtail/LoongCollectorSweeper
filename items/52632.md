## Summary

Keep issue #52632 open. Current main still has the reported lifecycle bug: the `openclaw configure channels` removal path deletes the raw `channels.<id>` config block while leaving WhatsApp credentials on disk, and gateway startup still treats persisted WhatsApp auth as a configured channel and can write `channels.whatsapp.enabled = true` back to config. The related PR #52672 was closed unmerged, and its behavior is not present on current main.

## Best Possible Solution

Keep the issue open and fix the removal lifecycle in core/commands. The conservative implementation should make `openclaw configure channels` removal go through the plugin-aware account removal contract, and it should also prevent leftover persisted auth from silently re-adding the channel on gateway restart, either by purging/logging out the removed WhatsApp auth state after explicit confirmation or by writing an explicit disabled/removal sentinel that auto-enable respects. Add a regression test for: remove `channels.whatsapp` through the configure wizard, leave `credentials/whatsapp/<accountId>/creds.json` on disk, run the startup auto-enable path, and assert `channels.whatsapp` is not recreated unless the user explicitly re-enables/re-adds it.
