## Summary

Keep open. Current main has Gmail Pub/Sub setup and a Gateway-side watcher around `gog`, plus a bundled `gog` skill, but it still assumes `gog` OAuth is already healthy and does not provide the deterministic revoked-token recovery path, stale auth-state UX, or Gmail read/send + Drive validation requested in #61649.

## Best Possible Solution

Keep the issue open as an operability bug for the existing Gmail/gog surface. The best path is to add an OpenClaw-owned recovery runbook and/or CLI status/doctor command that detects `gog` invalid_grant/stale auth state, guides a single headless-safe reauth path, and optionally validates Gmail read, Gmail send, and Drive before declaring recovery; if the fix belongs in gog itself, OpenClaw docs should still point to the exact upstream recovery command and surface the failure clearly from `webhooks gmail setup/run` and the Gateway watcher.
