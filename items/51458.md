## Summary

Keep #51458 open. Current main still rejects the exact Slack message-action shape reported: `send` with `media` plus generated interactive/presentation blocks. The separate outbound `ReplyPayload` path already has a two-pass media-then-blocks implementation, but the Slack message-action adapter and runtime do not use that path. The related open PR #62792 is scoped to inbound Slack file/image ingestion and explicitly does not claim outbound Slack behavior changes, so it does not supersede this issue. The Ryce comment is useful context because it correctly flags that a fix should preserve the compound send intent for retry/transcript coherence rather than just hide the error with unrelated sends.

## Best Possible Solution

Keep #51458 open. The best path is to update the Slack message-action `send` implementation to treat `media` plus rendered `interactive`/presentation/blocks as one logical action, likely reusing the existing outbound-payload media-sequence/finalize pattern or an equivalent Slack-specific helper, then add regression coverage for `send` with `media` plus `interactive` and for thread/result behavior. Do not close against #62792 because that PR is inbound file hydration, not this outbound message-action failure.
