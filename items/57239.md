## Summary

Keep open. Current `main` does not implement this fast-lane override, and the PR still appears to contain useful work toward a real active-session latency problem. The discussion also surfaced non-trivial queueing/session-isolation risks, so conservative cleanup should not auto-close it.

## Evidence

- **Current docs still describe serialized per-session runs with no fast-lane path:** The shipped queue docs say inbound auto-reply runs are serialized by session key and list only `steer`, `followup`, `collect`, `steer-backlog`, and `interrupt` queue modes. There is no documented immediate fast-lane override for short messages during an active run.
