## Summary

Keep #70578 open. Current main has mitigations for the reported data-loss path, including preserving source-only keys during write projection, rejecting destructive-looking config writes, and making plugin install fail closed on invalid config. But the issue is not fully obsolete: the root OpenClaw config schema is still strict, tests still expect unknown config fields to fail validation, and bundled plugin CLI command surfaces are still gated by plugins.allow.

## Best Possible Solution

Keep the issue open and narrow it into actionable follow-ups. Decide the root config contract explicitly: either preserve inert unknown top-level fields with regression coverage for plugin-install/config-write round-trips, or document the strict fail-closed policy and ensure every writer refuses to clobber invalid/unknown data. Track the bundled plugins.allow command-surface behavior separately as either an intentional documented policy or a targeted exemption for OpenClaw-owned bundled commands.
