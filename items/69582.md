## Summary

Keep #69582 open. Current main has related guardrails, but they do not fully address the reported GLM/local parameter contamination: default loop detection is off, process action params are not runtime schema-corrected before dispatch, edit params only require strings, and the malformed tool-call argument repair path does not repair a trailing `{}` suffix or strip `{}` inside string values.

## Best Possible Solution

Keep the issue open and fix it in the agent tool-call boundary. The best path is to add targeted runtime validation/repair before dispatch for malformed tool arguments from OpenAI-compatible/local models, with tests for `process({ action: {} })`, raw streamed arguments ending in `{}`, and edit params whose string fields end in `{}`. If maintainers prefer not to silently mutate model output, reject these shapes with a clear non-retryable tool error and make loop protection or provider-specific guidance explicit for affected local GLM setups.

## Evidence

- **Issue context is specific enough to remain actionable:** The report names OpenClaw 2026.3.24 on Windows 11 with local/glm-5, gives concrete corrupted parameter shapes (`edit.path`/`edit.newText` ending in `{}` and `process(action={})`), and the reporter followed up that multiple March/April versions showed the same behavior.
- **Process tool does not runtime-correct the reported `action={}` shape:** `processSchema` advertises `action` as a string, but `createProcessTool` casts the incoming args and branches on `params.action`. With `action` as an object and no `sessionId`, the current code returns `sessionId is required for this action`; with a session id it falls through to `Unknown action ${params.action as string}`. That is not the proposed schema validation/correction behavior.

## Risks / Open Questions

- Some infinite-loop symptoms may be mitigated when users explicitly enable `tools.loopDetection`, but that is opt-in and does not sanitize or reject the corrupted arguments themselves.
- The root cause may be model/provider output rather than OpenClaw transport parsing, but the current core behavior still lacks a clear validation, repair, or terminal-error path for the concrete bad parameter shapes in the issue.
