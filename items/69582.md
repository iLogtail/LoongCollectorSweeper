## Summary

Keep #69582 open. Current main has related malformed tool-call argument repair for self-hosted OpenAI-compatible streams and optional tool-loop detection, but it does not clearly solve the reported GLM/local parameter contamination: string values ending in `{}` remain valid strings, `process.action` is still only schema-described as a string and then type-cast at runtime, and the loop-detection safety net is documented as disabled by default.

## Best Possible Solution

Keep the issue open and fix it in the agent tool-call boundary: add schema-aware validation for core tool parameters, make `process.action` a closed/validated action set with a clear wrong-type error, and add provider/model regression coverage for GLM/OpenAI-compatible malformed argument streams. If a sanitizer is added, scope it narrowly to provably malformed tool-call argument framing rather than silently trimming arbitrary user strings.

## Risks / Open Questions

- A broad sanitizer that strips trailing `{}` from every string could corrupt legitimate user text or file names; any fix should be schema-aware and preferably scoped to known malformed tool-call argument patterns/providers.
- Relying only on optional loop detection would leave default installs exposed to repeated malformed tool calls.
