## Summary

Current main does not show a source-backed fix for the reported GLM/local OpenAI-compatible tool-argument contamination. GLM/Z.AI is a current supported path, but malformed argument repair preserves valid JSON values like "value{}", the process tool still treats action={} as a non-list action and returns the same sessionId-required failure, and general loop detection remains opt-in rather than a default protection for this case.

## Best Possible Solution

Keep the issue open and add a focused maintainer fix path: reproduce or mock a GLM/Z.AI/OpenAI-compatible stream that emits the reported argument shapes, add regression tests for process(action={}) and string values ending in {}, make process validate action with the shared typed parameter helpers before sessionId checks, and consider a schema-aware/provider-specific repair only where live GLM proves {} is transport contamination rather than legitimate user content. A hard no-progress guard for repeated tool input errors could also be made independent of opt-in loopDetection.
