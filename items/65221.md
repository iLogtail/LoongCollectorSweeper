## Summary

Keep PR #65221 open. Current main still has the pre-PR interrupt scheduling behavior: after abort+wait it returns the "Previous run is still shutting down" reply when the ReplyOperation registry remains active, and it lacks the PR's force-detach, waitInterrupted, identity-gated force retry, and reply-registry hardening. The PR is also not safely closeable as implemented or superseded because its latest supplied review discussion still contains unresolved P1 feedback about force retry identity in the interrupt burst path.

## Best Possible Solution

Keep #65221 open. The best path is for the contributor or maintainer to narrow and finish the core interrupt scheduling fix on top of current main: add a two-phase embedded/reply-operation wait, introduce a conservative force-detach or equivalent recovery path only for actual interrupt timeouts, and gate any forced ReplyOperation supersede to the exact stale operation identity so rapid interrupt bursts cannot run older input after newer input. Keep #65010 and #65222 as adjacent stack items rather than using them to supersede this core PR.
