## Summary

Keep open. Current main has useful Linux/systemd mitigation for the old `~/.config/openclaw/gateway.env` split: generated units can use `EnvironmentFile=` and dotenv-backed values are moved out of inline `Environment=` lines into a 0600 state-dir file. But #53926 specifically asks for `~/.openclaw/.env` to be the service source of truth, or a symlink to it. Current main instead writes a copied `gateway.systemd.env`; because runtime dotenv loading does not override already-set environment variables, a later edit to `~/.openclaw/.env` can still be shadowed by the stale systemd env file. The related #66248/#66295 discussion is relevant systemd env lifecycle work, but it does not provide a merged canonical fix for this exact install-time source-of-truth mismatch.

## Best Possible Solution

Keep the issue open and fix the Linux systemd install contract so the daemon and CLI share one durable source of truth. The narrowest product fix is to render `EnvironmentFile=-%h/.openclaw/.env` (or the resolved `$OPENCLAW_STATE_DIR/.env`) for the managed state-dir env, avoid generating a stale token copy, and add regression coverage for changing `~/.openclaw/.env` after install followed by `gateway restart`. If the generated `gateway.systemd.env` copy is intentionally retained for permission or filtering reasons, add explicit sync/repair semantics and drift detection for file-backed `OPENCLAW_GATEWAY_TOKEN` before closing #53926.
