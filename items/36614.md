## Summary

Keep #36614 open. Current main still has the reported write-side route contamination path: isolated inbound DM sessions can be recorded with the per-channel session key, while the same inbound handler still writes deliveryContext/lastChannel/lastTo to the main session. Prior discussion correctly separated this from the read-side WebChat stale-route mitigation in #38418; the write-side fix is still incomplete across several plugins.

## Best Possible Solution

Keep #36614 open and finish the write-side fix. The likely maintainer path is to make all inbound direct-message last-route updates derive their target from resolveInboundLastRouteSessionKey, or add an equivalent central guard that skips route-field writes when the update target differs from the recorded inbound session under isolated DM scopes. Add regression coverage for iMessage plus representative recordInboundSession and direct updateLastRoute callers such as Slack/LINE/Signal/Matrix/Mattermost while preserving dmScope=main behavior.

- Some plugins already use safer patterns, so the cleanup should be targeted rather than a blind removal of all main-session route updates.
