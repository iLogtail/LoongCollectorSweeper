## Summary

Keep #62625 open. Current main still has the reported gap for partial/stale session indexes: session syncing depends on process-local dirty state or explicit force/full-reindex/targeted session files, and there is no startup or status-path reconciliation that compares on-disk session transcripts against the SQLite `sessions` rows. The related merged PR #39732 is present, but it only preserves sessions during full reindex; it does not cover clean restarts or missed transcript events. The 2026-04-21 comment’s broader missed-event case also remains plausible under the current event-bus design.

## Best Possible Solution

Keep the issue open and implement durable session reconciliation in memory-core. The likely fix belongs in `MemoryIndexManager`: enumerate `listSessionFilesForAgent`, compare `sessionPathForFile` entries against stored `sessions` source state/hashes, mark missing or changed transcripts dirty or run a targeted session sync, and ensure non-force `openclaw memory index` can perform that catch-up. Status should also compare on-disk totals with the same session filename filter the indexer uses, so `Dirty: no` and `X/Y files` cannot falsely imply freshness.
