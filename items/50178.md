## Summary

Keep #50178 open. Current main has the same class of `reasoning_content` replay handling for DeepSeek/Venice and generic stream capture, but I could not find an equivalent Qianfan or `baiduqianfancodingplan/qianfan-code-latest` fix. The Qianfan plugin is still plain OpenAI-compatible catalog/auth registration with no provider replay policy or stream payload wrapper, and config compat does not expose the upstream `requiresReasoningContentOnAssistantMessages` knob for custom Qianfan-like models.

## Best Possible Solution

Keep the issue open and either add a Qianfan-specific replay/stream wrapper matching the DeepSeek/Venice placeholder behavior, or expose a documented OpenAI-compatible compat option that lets Qianfan-like custom models require `reasoning_content` on replayed assistant tool-call messages. Add focused unit coverage for Qianfan/custom OpenAI-compatible tool-call replay and update the Qianfan docs if the old `baiduqianfancodingplan/qianfan-code-latest` route is retired or renamed.
