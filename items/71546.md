## Summary

Keep #71546 open. The report is recent, detailed, non-maintainer-authored, and has no protected labels. Current main still has Discord reconnect/readiness hardening, but I did not find a shipped catch-up/replay backstop or a fix aimed at the reported 100-400s ingest lag during Carbon RESUME/reconnect windows.

## Best Possible Solution

Keep the issue open for Discord plugin maintainers. The likely path is to reproduce the idle close-1000/RESUME timing on current main, confirm whether Carbon replays MESSAGE_CREATE events to the existing listener across internal reconnects, and then either fix the listener/reconnect ownership path or add an opt-in Discord catch-up cursor that fetches recent channel/DM messages after successful RESUME/reconnect without duplicating already-processed messages.
