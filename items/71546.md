## Summary

Keep #71546 open. Current main includes adjacent Discord gateway hardening for #68159 and #38596, but I did not find a source-backed fix for the specific reported problem: 100-400s delay between Discord MESSAGE_CREATE delivery and OpenClaw inbound-worker enqueue during Carbon reconnect/RESUME windows. The issue is fresh, detailed, and still actionable for the Discord plugin owner.

## Best Possible Solution

Keep #71546 open for Discord plugin owner investigation. The best path is to instrument Carbon close/RESUME/READY and MESSAGE_CREATE delivery, add tests that prove listeners survive internal reconnects and replay reaches the inbound worker promptly, and, if replay is not reliable, add a bounded deduped catch-up read using the existing `readMessagesDiscord(... after ...)` surface after successful reconnect/RESUME. Maintainers should also answer whether the idle close-1000 cadence is expected baseline behavior.
