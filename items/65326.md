## Summary

Keep open. Current main partially addresses the report by accepting Google Workspace Add-on message payloads with `authorizationEventObject.systemIdToken`, binding add-on tokens to `appPrincipal`/JWT `sub`, and logging actionable auth failures. But the public Google Chat setup docs still do not document `appPrincipal` or how to obtain the numeric `sub`, and the add-on wrapper parser is currently message-payload specific rather than the broader format translation described in the issue/workaround.

## Best Possible Solution

Keep this open until the Google Chat docs explicitly tell users configured through the GCP Chat API page to use `audienceType: "app-url"`, configure `appPrincipal` to the numeric JWT `sub`/uniqueId, and provide a safe way to discover that value. Maintainers should also either expand `parseGoogleChatInboundPayload` to normalize the broader Add-ons wrapper variants from the posted workaround or document that OpenClaw only consumes Add-ons `messagePayload` events. If #58541 becomes the canonical docs-only tracker, this issue could later be closed as superseded after confirming the parser work is fully covered or intentionally out of scope.
