## Summary

Keep PR #70877 open. Current main still leaves generic tool-loop detection disabled by default: the runtime default is `enabled: false`, the tool hook receives an undefined/optional loop-detection config unless users configure it, tests still assert the no-config path is disabled, and docs/schema text still advertise default false. The related shipped work for #67401 only enables the unknown-tool guard by default, which is a narrower safety net and does not cover repeated valid tool calls such as repeated `web_fetch` DNS failures. The PR remains directionally relevant, but its latest provided review context shows it still needs fixes around the source default, schema/docs alignment, and inheritance semantics before it is mergeable.

## Best Possible Solution

Keep this PR open or consolidate it with a canonical loop-detection-default PR. The best fix is to move the default-enable behavior into the actual runtime config resolution path used by `detectToolCallLoop`, preserve explicit global `enabled: false` when agent overrides omit `enabled`, update schema/help/generated baselines and public docs from source, and add regression coverage for both no-config default-enabled behavior and explicit opt-out behavior.

## Risks / Open Questions

- The PR's intended behavior change is real, but merging it as-is could preserve a mismatch between direct `detectToolCallLoop` defaults, config UI/schema defaults, and global-to-agent inheritance semantics.
- Default-enabling loop detection is a product behavior change with possible false positives for legitimate repeated tool use, so the final implementation should keep explicit `enabled: false` opt-out and document the default clearly.
