## Summary

Keep #64831 open. The report is recent and bundles several cron, Telegram delivery, approval, exec, and OOM symptoms without logs or a concrete repro. Current main has meaningful cron catch-up, retry, failure-alert, direct-delivery retry, and approval replay protections, but that is only partial evidence; it does not prove the reporter's Telegram/local workflow is fixed end to end. The direct exec/script-heavy cron subset also remains covered by open related trackers such as #18160 and #29907 rather than implemented in core cron.

## Best Possible Solution

Keep #64831 open for now, but ask maintainers or the reporter to split it into reproducible items. The practical path is: collect sanitized cron job JSON, `openclaw cron runs --id` output, Telegram delivery config/target, approval IDs and timing, version >= 2026.4.24 behavior, logs around the failed run, and the script/tool chain that OOMs. Route the deterministic direct-exec cron request to #18160/#29907, and handle any remaining confirmed Telegram delivery or approval lifecycle bug in the owning cron, Telegram plugin, or exec-approval code path with focused regression tests.
