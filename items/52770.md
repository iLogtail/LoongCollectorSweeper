## Summary

Keep open. Current `main` does not implement the PR's core probe-intent preservation, and the provided review discussion shows remaining questions that still need explicit maintainer judgment rather than cleanup auto-close.

## Evidence

- **Current main still drops later probe intent during an in-flight refresh:** `refreshGatewayHealthSnapshot()` keeps a single module-level `healthRefresh` promise and, when one exists, returns it without considering newer caller options. There is no queued follow-up or `pendingProbe` handling on current `main`. That means a later `probe:true` caller can still receive the earlier in-flight result.
  - file: [src/gateway/server/health-state.ts:72](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/server/health-state.ts#L72)
  - command: `nl -ba src/gateway/server/health-state.ts | sed -n '1,140p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **A concrete conflicting caller exists on main:** The health RPC serves a fresh cached response, then kicks off `void refreshHealthSnapshot({ probe: false })` in the background. If a verbose/probing request arrives while that background refresh is in flight, current `main` has no mechanism to preserve the newer `probe:true` intent.
  - file: [src/gateway/server-methods/health.ts:16](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/server-methods/health.ts#L16)
