## Summary

Keep this issue open. Current main has several Discord gateway hardening fixes, but it still does not implement the requested certificate-expiry hard recovery path, and the healthy READY/logged-in line is still not reliably emitted after the gateway becomes ready. The issue discussion also includes a second concrete Raspberry Pi/Debian data point, so this is not stale or incoherent.

## Best Possible Solution

Keep the issue open and fix it in the Discord plugin/gateway lifecycle. The likely path is to add a typed TLS-certificate failure classifier for Discord gateway/REST errors, escalate repeated or terminal `certificate has expired`/CERT_HAS_EXPIRED failures beyond in-process reconnects to an operator-visible hard-restart path or documented supervisor respawn, and add tests proving the lifecycle does not end in a silent disposed state. Separately, emit a `[discord] logged in...` or equivalent READY log when the lifecycle actually observes `gateway.isConnected`, with tests and docs aligned to that operator contract.
