## Summary

Current `main` still appears to miss `session_end` for active sessions during gateway shutdown/restart. The code emits `session_end` on session replacement, reset, and delete paths, but the SIGTERM/SIGINT/restart shutdown path only runs `gateway_stop` plus general teardown and does not iterate active sessions.

## Evidence

- **Hook contract distinguishes session lifecycle from gateway lifecycle:** Plugin docs say `session_start` / `session_end` track session lifecycle boundaries, while `gateway_start` / `gateway_stop` are for plugin-owned Gateway services. That means `gateway_stop` is not a documented substitute for per-session `session_end`.
  - file: [docs/plugins/hooks.md:92](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/docs/plugins/hooks.md#L92)
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Auto-reply session code only fires session_end when a new session replaces an old one:** `session_end` is emitted inside the `isNewSession` branch and only when `previousSessionEntry.sessionId !== effectiveSessionId`, matching the report's claim that normal replacement is handled but process shutdown is separate.
  - file: [src/auto-reply/reply/session.ts:855](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/auto-reply/reply/session.ts#L855)
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Gateway reset/delete paths explicitly emit session_end:** Current Gateway code has dedicated `emitGatewaySessionEndPluginHook(...)` calls for reset and delete flows, which shows the project already treats explicit session teardown as a place to fire `session_end`.
  - file: [src/gateway/session-reset-service.ts:650](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/gateway/session-reset-service.ts#L650)
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Gateway shutdown path does not sweep active sessions:** `server.close()` first runs `gateway_stop`, then `runClosePrelude()`, then the close handler. The close handler stops channels, plugin services, cron, sockets, and broadcasts `shutdown`, but there is no active-session iteration or `session_end` emission in that shutdown path.
  - file: [src/gateway/server.impl.ts:916](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/gateway/server.impl.ts#L916)
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Only current Gateway session teardown call sites are reset/delete helpers:** Repository search found `emitGatewaySessionEndPluginHook(` call sites in `src/gateway/session-reset-service.ts` and `src/gateway/server-methods/sessions.ts`, with no shutdown-path call site under `src/cli/gateway-cli/run-loop.ts`, `src/gateway/server.impl.ts`, or `src/gateway/server-close.ts`.
  - command: `rg -n 'emitGatewaySessionEndPluginHook\(' src/gateway -g '!**/*.test.ts'`
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)

## Risks / Open Questions

- Plugins or downstream systems that rely on `session_end` can keep orphaned sessions after gateway restart/shutdown.
- There does not appear to be shutdown coverage asserting `session_end` for active sessions, so this gap is easy to regress or miss.
