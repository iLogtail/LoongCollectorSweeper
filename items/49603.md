## Summary

Keep #49603 open. Current main has partial session-lock recovery, but it does not implement the reported boot-time/current-PID cleanup when a modern lock file has a valid `starttime` and `HELD_LOCKS` has been lost. Gateway startup only calls `cleanStaleLockFiles`, which removes locks already considered stale; the acquire path only marks same-PID self-locks orphaned when `starttime` is missing. The regression test covers legacy missing-starttime locks, not the issue's stronger same-process/current-PID state-loss case.

## Best Possible Solution

Keep the issue open and land a focused session-lock fix in `src/agents/session-write-lock.ts`: during startup cleanup and/or contended acquisition, treat a lock owned by `process.pid` as orphaned when the normalized session file is not present in `HELD_LOCKS`, regardless of valid `starttime`, while preserving actively held in-memory locks. Add regression coverage for a current-PID lock with matching valid `starttime` but no `HELD_LOCKS` entry, plus a startup cleanup/audit case if boot-time visibility is part of the intended behavior. PR #49660 is the related implementation track, but it should be checked against this valid-starttime scenario before merge.

## Risks / Open Questions

- A naive boot sweep could delete an actively held same-process lock; the fix should preserve locks present in `HELD_LOCKS` and only reclaim current-PID disk locks with no in-memory owner.
- Existing code already handles recycled PIDs and legacy missing-starttime same-PID locks, so the remaining patch should be narrowly targeted to the valid-starttime/current-PID ownership-mismatch case.
