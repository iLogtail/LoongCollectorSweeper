## Summary

Keep open. Current `main` still sends WhatsApp outbound voice notes with only `audio`/`ptt`/`mimetype`; there is no WhatsApp path for `seconds` or `waveform`, and the existing voice-metadata implementation/changelog entry is Discord-only.

## Evidence

- **WhatsApp web send payload lacks duration/waveform:** `createWebSendApi()` still builds WhatsApp audio payloads as `{ audio, ptt: true, mimetype }` with no `seconds` or `waveform` fields before calling Baileys.
  - file: [extensions/whatsapp/src/inbound/send-api.ts:57](https://github.com/openclaw/openclaw/blob/a2221d6b47094cd34c12025ce809aba02c5e1c1c/extensions/whatsapp/src/inbound/send-api.ts#L57)
  - command: `nl -ba extensions/whatsapp/src/inbound/send-api.ts | sed -n '1,220p'`
  - sha: [a2221d6b4709](https://github.com/openclaw/openclaw/commit/a2221d6b47094cd34c12025ce809aba02c5e1c1c)
- **WhatsApp send options have no metadata fields to thread through:** `ActiveWebSendOptions` only exposes `quotedMessageKey`, `gifPlayback`, `accountId`, and `fileName`, so current main has no typed route to carry voice-note duration or waveform metadata into the send API.
  - file: [extensions/whatsapp/src/inbound/types.ts:12](https://github.com/openclaw/openclaw/blob/a2221d6b47094cd34c12025ce809aba02c5e1c1c/extensions/whatsapp/src/inbound/types.ts#L12)

- **Outbound WhatsApp adapter forwards no voice-note metadata:** The bundled WhatsApp outbound adapter passes only `mediaUrl`, local media access, `accountId`, and `gifPlayback` into `sendMessageWhatsApp()`. No voice metadata is computed or forwarded on the outbound path used by message sends/tools.
  - file: [extensions/whatsapp/src/outbound-adapter.ts:74](https://github.com/openclaw/openclaw/blob/a2221d6b47094cd34c12025ce809aba02c5e1c1c/extensions/whatsapp/src/outbound-adapter.ts#L74)
