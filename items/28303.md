## Summary

Keep open. Current main still uses `zh-CN` and `zh-TW` as the Control UI runtime locale identifiers across the public type, lazy registry/imports, browser detection, persisted setting, selector labels, generator/workflow metadata, docs, and tests. The docs publish pipeline has partial `zh-Hans` metadata, but it still maps generated Simplified Chinese docs to `zh-CN` paths and does not implement the requested runtime rename or saved-setting migration. Related PR #42966 was closed unmerged and docs-scoped; PR #63621 is an open zh-TW glossary improvement, not a canonical implementation of this issue.

## Best Possible Solution

Keep the issue open for a focused Control UI i18n change: rename the runtime locale contract to `zh-Hans`/`zh-Hant`, update lazy imports and generated-locale metadata, add a saved-setting migration from `zh-CN`/`zh-TW`, adjust navigator fallback and selector language keys, update docs/workflow matrices, and revise tests that currently assert the old IDs. If maintainers choose to retain region-based IDs for compatibility, document that decision on the issue instead of closing it as implemented.
