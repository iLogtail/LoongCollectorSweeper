## Summary

Keep open. Current `main` at `a6aa151653ab478e9aa23e95c6446c41adc4fce1` still reads `channels.feishu.webhookPath` but does not compare it against `req.url` before parsing the body, validating the signature, and dispatching the event, so the reported bug remains valid by source inspection.

## Evidence

- **Feishu webhook handler still does not enforce the configured path:** `monitorWebhook()` reads `account.config.webhookPath` into `path` and uses it for logging/status/rate-limit keys, but the request listener proceeds straight into generic guards, body read, signature validation, and `eventDispatcher.invoke(...)` without any `req.url` or pathname comparison.
  - file: [extensions/feishu/src/monitor.transport.ts:158](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/feishu/src/monitor.transport.ts#L158)
  - command: `nl -ba extensions/feishu/src/monitor.transport.ts | sed -n '150,240p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Shared ingress guard used here does not check request paths:** `applyBasicWebhookRequestGuards()` only enforces allowed methods, rate limiting, and JSON content type. There is no path validation in the guard Feishu calls.
  - file: [src/plugin-sdk/webhook-request-guards.ts:153](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/plugin-sdk/webhook-request-guards.ts#L153)
