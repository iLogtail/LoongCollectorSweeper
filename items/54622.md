## Summary

Current main still does not surface a session-lane busy/wait-exceeded error to `openclaw agent`. The queue records `waitedMs`/`queueAhead` only through diagnostics, the embedded agent path enqueues on the session lane without a wait deadline or `onWait` reporting, and the Gateway/CLI path waits for a final response or generic gateway timeout rather than returning a typed busy response.

## Best Possible Solution

Keep the issue open. The best fix is to add a typed session-lane wait/busy result through the queue and Gateway agent RPC, then teach the CLI to print an actionable error with lane, queueAhead, waitedMs, and guidance. If maintainers want the current queueing behavior preserved, add an explicit CLI option such as `--force-queue` or a wait policy instead of silently waiting until the generic Gateway timeout.
