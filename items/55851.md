## Summary

Keep PR #55851 open. Current main still surfaces generic user-facing overload, rate-limit, and transport messages without provider/model/profile/trigger/session context, so the PR is not obsolete. The PR does need work before merge because the review discussion identifies a real failover-classification regression risk and missing regression coverage for the new context fields.

## Best Possible Solution

Keep the PR open and ask for an update on top of current main: preserve current transient `api_error` timeout classification, thread sanitized provider/model/profileId/trigger/session/request-id context through the current formatter and payload paths, add focused regression tests for the enriched user-facing messages, and keep strict diagnostic log assertions where they already protect model/provider/rawError output. If the PR is abandoned, the remaining product work can be tracked by the still-open provider-name/error-context request #51336 rather than closing this as implemented.
