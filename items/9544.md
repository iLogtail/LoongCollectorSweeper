## Summary

Keep open. Current main still creates the WhatsApp socket with Baileys `useMultiFileAuthState`; the newer code only hardens `creds.json` writes and auth-state reads, not credentials plus signal-key storage in a SQL/consolidated production auth store. Related PR #66863 is useful prior art, but it was closed unmerged after review feedback and current main does not include its `useAtomicAuthState`/`auth-state.json` path.

## Best Possible Solution

Keep #9544 open until the WhatsApp plugin replaces Baileys `useMultiFileAuthState` for both credentials and signal keys with a production auth store, migrates the existing multi-file layout, preserves or deliberately supersedes the current atomic `creds.json` protection, updates auth presence/logout/self-id helpers, and adds focused fresh-install plus migration tests.
