## Summary

Current `main` implements an opt-in fix for string-only `openai-completions` backends via `compat.requiresStringContent`, with transport wiring, config schema support, regression coverage, and troubleshooting docs for the exact content-array failure mode.

## Close Comment

Closing from a Codex review. Current `main` now implements this through the `models.providers.<provider>.models[].compat.requiresStringContent` compatibility flag. The `openai-completions` transport applies `flattenCompletionMessagesToStringContent(...)` in `src/agents/openai-transport-stream.ts:1688`, and the helper in `src/agents/openai-completions-string-content.ts:20` flattens pure text `content` arrays on any message object before request serialization, which covers assistant history entries as well as user turns. The config surface is wired in `src/config/types.models.ts:65` and `src/config/zod-schema.core.ts:196`, there is regression coverage in `src/agents/openai-transport-stream.test.ts:1502`, and the troubleshooting/config docs now point to this exact fix in `docs/gateway/troubleshooting.md:88` and `docs/gateway/config-tools.md:420`. I did not find a newer tagged release that definitely contains this yet, so I’m closing this as implemented on `main` rather than attributing it to a shipped release.
