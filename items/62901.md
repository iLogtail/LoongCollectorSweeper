## Summary

Current `main` already contains the timeout fix this report was asking for: the old `v2026.4.5` code used a hardcoded 60s LLM idle watchdog that ignored `agents.defaults.timeoutSeconds`, while current `main` and shipped release `v2026.4.23` resolve idle timeout from `agents.defaults.llm.idleTimeoutSeconds`, then `agents.defaults.timeoutSeconds`, and disable the default idle watchdog for cron runs with no explicit timeout.

## Evidence

- **v2026.4.5 had a hardcoded 60s idle timeout:** The `v2026.4.5` version of `resolveLlmIdleTimeoutMs` returned a fixed `60_000` ms default and only read `agents.defaults.llm.idleTimeoutSeconds`; it did not fall back to `agents.defaults.timeoutSeconds`, which matches the reporter's observation that increasing `agents.defaults.timeoutSeconds` had no effect.
  - file: [src/agents/pi-embedded-runner/run/llm-idle-timeout.ts:11](https://github.com/openclaw/openclaw/blob/3e72c0352dde84a0bcb3aabafa99c2d4b12d1c46/src/agents/pi-embedded-runner/run/llm-idle-timeout.ts#L11)
  - command: `curl -fsSL https://raw.githubusercontent.com/openclaw/openclaw/v2026.4.5/src/agents/pi-embedded-runner/run/llm-idle-timeout.ts | nl -ba | sed -n '1,60p'`
  - sha: [3e72c0352dde](https://github.com/openclaw/openclaw/commit/3e72c0352dde84a0bcb3aabafa99c2d4b12d1c46)
