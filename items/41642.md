## Summary

Keep open. Current `main` still runs model fallback candidates sequentially without any per-candidate timeout budget, the public/config surface does not expose the proposed attempt-timeout knob, and the embedded runner's idle-timeout path still derives from the overall run timeout/session timeout rather than reserving budget for later fallback candidates.

## Evidence

- **Sequential fallback candidate execution has no per-attempt timeout:** `runFallbackCandidate` simply awaits `params.run(...)` and catches errors; there is no `Promise.race`, timeout wrapper, or attempt-budget parameter before returning to the fallback loop.
  - file: [src/agents/model-fallback.ts:166](https://github.com/openclaw/openclaw/blob/bae7b54a85b17d180e4bfc28973a7e4fd1646a93/src/agents/model-fallback.ts#L166)
  - command: `nl -ba src/agents/model-fallback.ts | sed -n '160,240p'`
  - sha: [bae7b54a85b1](https://github.com/openclaw/openclaw/commit/bae7b54a85b17d180e4bfc28973a7e4fd1646a93)
- **Fallback API still has no attempt-timeout input:** `runWithModelFallback` accepts `cfg`, `provider`, `model`, `runId`, `agentDir`, `fallbacksOverride`, `run`, and `onError`, then iterates candidates in order. There is no `attemptTimeoutMs` or similar parameter on current `main`. A repo-wide search for `fallbackAttemptTimeoutSeconds|attemptTimeoutMs` returned no matches.
  - file: [src/agents/model-fallback.ts:652](https://github.com/openclaw/openclaw/blob/bae7b54a85b17d180e4bfc28973a7e4fd1646a93/src/agents/model-fallback.ts#L652)
  - command: `nl -ba src/agents/model-fallback.ts | sed -n '652,770p'; rg -n "fallbackAttemptTimeoutSeconds|attemptTimeoutMs" src docs CHANGELOG.md`
  - sha: [bae7b54a85b1](https://github.com/openclaw/openclaw/commit/bae7b54a85b17d180e4bfc28973a7e4fd1646a93)
- **Current callers do not pass any per-candidate budget:** The auto-reply and cron paths call `runWithModelFallback(...)` with the existing options only; neither path supplies a dedicated fallback-attempt timeout or remaining-budget split.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:859](https://github.com/openclaw/openclaw/blob/bae7b54a85b17d180e4bfc28973a7e4fd1646a93/src/auto-reply/reply/agent-runner-execution.ts#L859)
  - command: `nl -ba src/auto-reply/reply/agent-runner-execution.ts | sed -n '859,869p'; nl -ba src/cron/isolated-agent/run-executor.ts | sed -n '109,117p'`

- **Idle timeout still tracks overall run/session timeout rather than reserving fallback budget:** `resolveLlmIdleTimeoutMs` falls back to `agents.defaults.timeoutSeconds` when no explicit LLM idle timeout is set, and the attempt wrapper feeds it the run timeout only when that differs from the configured agent timeout. The docs also state the idle timeout uses `agents.defaults.timeoutSeconds` when configured. That means a hung primary can still consume the same overall timeout budget that later fallback candidates need.
