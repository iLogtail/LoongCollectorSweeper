## Summary

Current main has fixes for several adjacent duplicate-delivery paths, but the specific Telegram duplicate after provider timeout plus assistant retry is not clearly fixed. The timeout failover path can still retry after a streamed/visible Telegram answer, and Telegram dispatch normally rotates the streamed answer lane into a sent message when the retried assistant message starts.

## Best Possible Solution

Keep this open and add a regression around Telegram partial streaming followed by assistant provider timeout/profile rotation where the retry produces the same answer. The fix likely belongs at the Telegram delivery or embedded-runner retry boundary: carry delivered preview/content state across assistant retry attempts and suppress, finalize, or reuse the already-visible Telegram message instead of starting a second outbound message.
