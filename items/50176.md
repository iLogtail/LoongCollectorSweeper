## Summary

Keep open. Current `main` does not implement the historical-replay guard from PR #50176, and the PR discussion already identified a likely correctness gap in the proposed object-identity approach.

## Evidence

- **Current main has no preexisting-session filter:** `subscribeEmbeddedPiSession()` initializes subscription state without any snapshot of `session.messages`, so there is no mechanism on `main` to recognize replayed historical assistant messages when attaching to an existing session.
  - file: [src/agents/pi-embedded-subscribe.ts:72](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/agents/pi-embedded-subscribe.ts#L72)
  - command: `nl -ba src/agents/pi-embedded-subscribe.ts | sed -n '60,130p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Historical replay would still flow through assistant handlers:** `handleMessageStart`, `handleMessageUpdate`, and `handleMessageEnd` only skip non-assistant messages and transcript-only OpenClaw mirrors. They do not check whether an assistant message already existed in `session.messages`, so replayed historical assistant lifecycle events would still be treated as fresh output.
  - file: [src/agents/pi-embedded-subscribe.handlers.messages.ts:349](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/agents/pi-embedded-subscribe.handlers.messages.ts#L349)

  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Existing duplicate tests cover repeat events, not subscribe-time historical replay:** Current tests only cover repeated `text_end` / `message_end` dedupe within a subscription. There is no test on `main` for subscribing to an existing session whose historical assistant events are replayed.
  - file: [src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.does-not-emit-duplicate-block-replies-text.test.ts:12](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.does-not-emit-duplicate-block-replies-text.test.ts#L12)

- **PR discussion already surfaced a plausible gap in the proposed fix:** The provided PR review comments note that the patch keys `preexistingMessages` by object identity. That would miss replay paths that deserialize or clone historical messages instead of reusing the exact `session.messages` object, so the PR is not strong enough to auto-close as already-solved. Current main already uses stable assistant text-signature IDs elsewhere, which reinforces that identity matching is not obviously the right contract.
