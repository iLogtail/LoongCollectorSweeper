## Summary

Current main still treats `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks. Related PR #61485 may supersede the `llm_input`/`llm_output` part if it lands, but it does not cover #20802's adapter-side `after_tool_call` result mutation, so #20802 should not be auto-closed.

## Best Possible Solution

Keep #20802 open unless a maintainer explicitly rejects the broader product direction or lands a canonical replacement. The best path is a focused current-main implementation for mutable `llm_input`/`llm_output` through #61485 or equivalent, plus a separate maintainer decision on whether `after_tool_call` should become result-mutating or whether `tool_result_persist` remains the intended mutation surface.
