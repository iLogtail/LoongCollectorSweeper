## Summary

Keep open. Current main still treats `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks: their public handler contracts return `void`, the runner uses `runVoidHook`, and the embedded/CLI/harness consumers fire them without consuming results. The related discussion shows real unresolved core plugin API work; the newer #61485 may supersede the `llm_input`/`llm_output` portion if merged, but it is still open and does not cover #20802's adapter-side `after_tool_call` mutation.

## Best Possible Solution

Keep #20802 open unless a maintainer explicitly rejects the product direction or lands a canonical replacement. The best path is a fresh, focused implementation against current main: add typed mutable result contracts for `llm_input` and `llm_output`, consume those results in embedded plus CLI/harness paths before provider send and before final delivery, update plugin docs/tests, and handle `after_tool_call` result mutation as a separate design decision against the current exactly-once dispatch and `tool_result_persist` boundaries.
