## Summary

Keep open. Current main serializes same-file edits through the pinned Pi edit tool and has recovery code for post-write/mismatch cases, but it still does not expose the requested distinct noop result. The current recovery path converts a satisfied post-state into normal success, and already-up-to-date unchanged content can still remain an error path.

## Best Possible Solution

Keep the issue open and implement explicit edit result classification around the existing Pi edit path: `changed` when this call writes, `noop` when readback proves the requested final state is already satisfied, and `error` when the target state is not satisfied. Add focused regression coverage for concurrent duplicate `edits[]` calls, already-up-to-date content, true mismatches, and any flat `oldText`/`newText` alias if OpenClaw chooses to support that input mode.
