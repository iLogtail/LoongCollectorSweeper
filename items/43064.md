## Summary

Current main still appears to have the reported heartbeat prefix bug. The heartbeat runner reads the raw effective `responsePrefix`, calls `getReplyFromConfig` without `onModelSelected`, then passes that raw prefix into heartbeat normalization where it is stripped/prepended literally. The normal reply path has the model-selection and `resolveResponsePrefixTemplate` machinery, but heartbeat is not wired into it. Related PRs in the issue timeline propose the right direction, but the provided context shows the direct fix attempts are either open/unmerged or closed unmerged, and the latest release tag still has the same raw heartbeat code.

## Best Possible Solution

Keep #43064 open. The best fix is to wire heartbeat runs into the existing reply-prefix pipeline: create a prefix context for the heartbeat delivery target, pass `onModelSelected` into `getReplyFromConfig`, resolve the configured prefix after model selection, and use that resolved prefix in `normalizeHeartbeatReply` and delivery. Add focused heartbeat regression tests for `{model}`/`{provider}` interpolation on alert replies and for `HEARTBEAT_OK` suppression with the resolved prefix. The existing related PRs are useful starting points, but current main and the latest release do not yet contain the fix.
