## Summary

Keep PR #39507 open. Current main still makes Feishu inbound `thread_id`/`root_id` force `threadReply` and then `reply_in_thread` even when `replyInThread: "disabled"`; tests also lock in that behavior. The PR body includes useful live Feishu verification, and the earlier Greptile review agreed the focused semantic fix is sound, but the branch likely needs a current-main port from `bot.ts` to `bot-content.ts` plus maintainer confirmation of the intended disabled semantics.

## Best Possible Solution

Keep #39507 open and resolve the Feishu semantics explicitly. If disabled is meant to be a hard off switch, port the PR to `extensions/feishu/src/bot-content.ts`, gate non-topic `threadReply` on configured `replyInThread`, preserve `group_topic` and `group_topic_sender` anchoring, and update the current regression test that expects forced thread replies under disabled config. If maintainers prefer follow-existing-thread behavior, clarify the schema/docs and test names so `"disabled"` no longer promises normal inline replies.
