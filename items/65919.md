## Summary

Keep open. Current `main` has partial cleanup wiring for typing shutdown after run completion, but it does not fully implement the requested lifecycle-owned presence lease. The reply typing controller still has a fixed 2-minute TTL during active runs, and that TTL is only refreshed by additional text/reasoning/tool signals, so long quiet phases such as preflight compaction can still drop presence before final dispatch goes idle.

## Evidence

- **Active-run typing still has a fixed 2 minute TTL:** `createTypingController` defaults `typingTtlMs` to `2 * 60_000`, and its TTL timer calls `cleanup()` with `typing TTL reached ...; stopping typing indicator` even while the keepalive loop is running. That is still wall-clock expiry, not the requested `run complete + dispatch idle` lease.
  - file: [src/auto-reply/reply/typing.ts:29](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/auto-reply/reply/typing.ts#L29)
  - command: `nl -ba src/auto-reply/reply/typing.ts | sed -n '20,210p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **TTL refresh depends on extra activity, not on the keepalive lease itself:** `signalRunStart()` only calls `typing.startTypingLoop()` once for `instant` mode. Later TTL refreshes happen only on text, reasoning, or tool signals. There is no separate active-run lease refresh tied to dispatcher idleness.
  - file: [src/auto-reply/reply/typing-mode.ts:78](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/auto-reply/reply/typing-mode.ts#L78)

- **Preflight compaction starts after a single run-start signal:** `agent-runner` fires `typingSignals.signalRunStart()` and then immediately enters `runPreflightCompactionIfNeeded(...)`. There is no compaction-specific typing refresh in this path, so a long quiet compaction can still age out on the 2 minute TTL.
