## Summary

Current `main` already fixes the cron self-deadlock through a different mainline path: cron-triggered isolated runs are still initiated with `lane: "cron"`, but the embedded runner now remaps cron-lane inner work onto `CommandLane.Nested`, and that behavior has dedicated regression coverage. This makes the PR's manual `enqueueRun()` lane swap redundant on current `main`.

## Evidence

- **Gateway cron still marks isolated runs as cron-lane work:** `runIsolatedAgentJob` calls `runCronIsolatedAgentTurn(..., lane: "cron")`, so the production cron path still starts from the cron lane.
  - file: [src/gateway/server-cron.ts:340](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/gateway/server-cron.ts#L340)
  - command: `nl -ba src/gateway/server-cron.ts | sed -n '333,349p'`
- **Embedded runner moves cron-triggered inner work off the cron lane:** `resolveGlobalLane()` explicitly maps `CommandLane.Cron` to `CommandLane.Nested` with a deadlock comment, so the inner embedded agent run no longer waits behind the cron lane slot held by the outer cron execution.
  - file: [src/agents/pi-embedded-runner/lanes.ts:8](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/agents/pi-embedded-runner/lanes.ts#L8)
