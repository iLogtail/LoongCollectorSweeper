## Summary

Keep #56979 open. Current main has meaningful auth-cooldown and fallback-probe mitigations, but the specific log-spam request is not fully implemented: `model-fallback/decision` still emits a warning for every fallback decision, and I found no coalescing/throttling around repeated identical decision logs.

## Best Possible Solution

Keep the existing auth cooldown and probe behavior, but add bounded coalescing in `src/agents/model-fallback-observation.ts` keyed by decision/requested model/candidate/reason/status/error fingerprint/next candidate. Log the first occurrence, suppress repeats for a short window, and emit a summarized suppressed-count record. Add focused tests that repeated identical `model_fallback_decision` calls do not produce one warn per call, plus an auth-cooldown fallback scenario covering `skip_candidate`.
