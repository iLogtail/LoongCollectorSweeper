## Summary

Keep #52596 open. Current main has the pre-tool-call enforcement primitive and plugin hooks that can block tools, but it does not implement the requested framework-owned session PIN gate: there is no `session.auth.pin` config/schema field, no session-authenticated state on `SessionEntry`, and no trusted TypeScript verification path that compares a user PIN outside model context. The issue comment is accurate that the missing piece is not the hook itself but the core session auth state plus trusted verification flow.

## Best Possible Solution

Keep the issue open for a focused core security design: add a sensitive session auth config/secret source, persist a framework-owned authenticated flag on `SessionEntry`, reset it on `/new` and `/reset`, enforce it in the existing before-tool-call path, and provide a trusted native command/gateway/internal verification path that compares PIN attempts in TypeScript without leaking the PIN into model history or tool output.
