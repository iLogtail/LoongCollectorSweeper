## Summary

Keep #52396 open. Current main still does not retry Discord statusless transient network failures such as `fetch failed`, timeouts, connection resets, or DNS failures. HTTP status-bearing 5xx errors are partially covered by the newer delivery wrapper, but the core Discord request runner remains 429-only and the delivery wrapper also only keys off numeric status. The related fix PR #52480 is still open/unmerged in the provided context, while #52409 and #45161 are closed/unmerged, so there is no shipped implementation to close against.

## Best Possible Solution

Keep the issue open until the Discord retry policy is widened on current main. The likely maintainer path is to land or rework #52480 so both direct Discord request runners and outbound delivery wrappers retry clearly transient statusless network failures, while preserving 429 `retry_after` behavior and adding regression tests for `fetch failed`/`ECONNRESET`/timeout plus non-retryable 4xx errors.
