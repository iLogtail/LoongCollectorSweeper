## Summary

Keep open. Current main has partial Discord delivery retry coverage for status-coded 429/5xx failures on the outbound adapter path, but it still does not implement the issue’s requested transient-error policy for timeout/connect/reset/DNS-style failures, and the older Discord request runner remains 429-only. The linked PR #52480 appears to be the active proposed fix, not a merged superseding implementation.

## Evidence
