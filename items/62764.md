## Summary

Keep open. Current main has a manual `OPENCLAW_SERVICE_REPAIR_POLICY=external` escape hatch, but it still does not automatically detect a same-name system-level systemd gateway unit before deciding the managed user-systemd service is missing, and the deep extra-service scan appears to ignore canonical `openclaw-gateway.service` names even in `/etc/systemd/system`.

## Best Possible Solution

Keep the issue open and implement Linux system-unit ownership detection before doctor installs, starts, restarts, or rewrites the managed user-systemd gateway. A conservative fix would check for an active or installed system-scope `openclaw-gateway[-<profile>].service` via `/etc/systemd/system` and/or `systemctl --system`, skip user-unit regeneration when found, and emit an explicit warning if both user and system units exist. Add regression tests for the exact same-name system unit case and document the relationship with `OPENCLAW_SERVICE_REPAIR_POLICY=external`.

## Risks / Open Questions

- Closing this would hide a real remaining product gap: users with a custom system-systemd gateway can still trigger user-service repair unless they already know to set the external repair policy.
- The current deep scan may give a false sense of safety because it scans system directories but suppresses the canonical same-name system unit that matters here.
