## Summary

Current main does transcribe Telegram voice-note audio and marks already-transcribed audio attachments, but it does not expose the requested original-message voice/type flag in the inbound agent metadata/context contract. Telegram currently collapses `msg.audio` and `msg.voice` into the same `<media:audio>` placeholder and only carries path/MIME/transcribed-index data forward, so agents still cannot reliably distinguish typed text, Telegram voice notes, and regular audio attachments from trusted metadata.

## Best Possible Solution

Keep this open. Add an additive per-turn message-source field to the channel/plugin inbound context contract, for example `MessageType`/`InboundMessageType` with values such as `text`, `voice`, `audio`, `image`, then serialize it into the agent-visible current-message metadata path. Populate Telegram from `msg.voice` versus `msg.audio`, extend equivalent channel mappings where available, and add contract plus Telegram regression tests proving typed text, voice note, and regular audio file behavior.

## Risks / Open Questions

- Treating the existing `[Audio transcript ...]` prompt text as the requested signal would be brittle: it is not a typed trusted metadata field and does not distinguish Telegram voice notes from normal audio attachments.
- Adding the field directly to the cache-stable system metadata would conflict with the current design that keeps volatile per-turn identifiers/flags out of `openclaw.inbound_meta.v2`; the implementation should use a dedicated per-turn context path or explicitly account for prompt-cache churn.
