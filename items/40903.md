## Summary

Keep open. Current `main` does not implement this PR's deferred Telegram media-group watermark handling. The current code persists accepted `update_id`s before handler completion, media-group updates are buffered and flushed later on a timer, and the synthetic media-group context still drops the raw update envelope, so the PR's core concern remains unresolved and needs maintainer judgment rather than conservative closure.

## Evidence

- **Current main persists update offsets before completion:** `persistAcceptedUpdateId()` advances the stored watermark as soon as an update enters middleware, before `await next()` finishes. The middleware calls it immediately for every numeric `update_id`.
  - file: [extensions/telegram/src/bot-core.ts:290](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/extensions/telegram/src/bot-core.ts#L290)
  - command: `nl -ba extensions/telegram/src/bot-core.ts | sed -n '280,380p'`
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Buffered media groups complete after the middleware returns:** Media-group updates are buffered in `mediaGroupBuffer`, scheduled with `setTimeout(...)`, and the handler returns early. The actual `processMediaGroup(...)` work happens later, outside the original middleware completion window.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1077](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/extensions/telegram/src/bot-handlers.runtime.ts#L1077)
