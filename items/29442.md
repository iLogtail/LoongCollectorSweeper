## Summary

Keep #29442 open. Current main has real prompt-injection hardening for external content and cron hook payloads, but it still feeds wrapped untrusted content into the same cron agent turn rather than a read-only worker/manager split, and the plugin hook surface still lacks a post-wrap replacement/blocking sanitizer seam like the closed, unmerged #30046 proposed.

## Best Possible Solution

Keep this open as the canonical security/product design item. The best path is either a first-class cron/external-content reader-worker flow with tool-disabled/read-only execution and structured sanitizer output before the privileged manager turn, or a narrow documented plugin API seam that lets ClawHub plugins such as ClawMoat replace or block wrapped external content before privileged agent turns.

## Risks / Open Questions

- Closing now would conflate existing wrapper, token-sanitization, sandbox, and tool-policy hardening with the requested privilege-separated reader/manager architecture.
- A ClawHub-only close would be premature because current plugin hooks can block/rewrite tool calls and add prompt context, but do not expose a post-external-content sanitizer replacement/block hook before the cron agent sees the payload.
- The best design likely needs maintainer product/security judgment on defaults, latency/cost, fail-open versus fail-closed behavior, and whether the core seam should be a first-class cron flow or a narrow plugin hook.
