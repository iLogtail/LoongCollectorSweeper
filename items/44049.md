## Summary

Keep open. Current main still prepends `acp` to all ACP client `serverArgs`, so the duplicated `openclaw acp acp ...` path described by the PR is not implemented or superseded. The production change is narrow and plausible, but the existing bot review is material: the added test in the PR would exercise `resolveAcpClientSpawnInvocation`, not the `buildServerArgs` path where the bug lives.

## Best Possible Solution

Keep the PR open and ask for a corrected regression test that reaches the actual command-construction path, preferably by testing `createAcpClient` with mocked spawn or by exposing a small internal argument-building helper for focused unit coverage. The core code should normalize a single redundant leading `acp` only when building the OpenClaw ACP server command, preserving other args and `--server-verbose` behavior.
