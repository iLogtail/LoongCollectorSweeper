## Summary

Keep open. Current `main` still has the duplicated-`acp` construction in `buildServerArgs`, so the reported bug is not implemented on `main`, and the PR as filed still lacks a regression test that exercises that code path.

## Evidence

- **Current main still prepends `acp` unconditionally:** `buildServerArgs` still constructs `const args = ["acp", ...toArgs(opts.serverArgs)]`, so `serverArgs=["acp", "--verbose"]` becomes `['acp','acp','--verbose']` before spawn.
  - file: [src/acp/client.ts:53](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/acp/client.ts#L53)
  - command: `nl -ba src/acp/client.ts | sed -n '45,75p'`
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **That argument list is used for the spawned ACP server:** `createAcpClient` calls `buildServerArgs(opts)` and passes the resulting `effectiveArgs` into `resolveAcpClientSpawnInvocation`, so the duplicated subcommand would flow into process spawn on current `main`.
  - file: [src/acp/client.ts:121](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/acp/client.ts#L121)
