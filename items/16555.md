## Summary

Keep #16555 open. Current main still uses the file-based outbound `delivery-queue` with startup recovery and reconnect drains that load pending JSON entries, sort by `enqueuedAt`, then apply retry/backoff/max-retry handling without any age/TTL expiry gate. The requested `messages.delivery.maxAgeMs` / `expireAction` config surface is absent from config schema/types/docs, and related SQLite/outbox TTL PRs were closed unmerged. The recent maintainer/Codex comment keeping this open still matches current main at 9a529ca78bfa0e257f662f0e70ab8c681caa6086.

## Best Possible Solution

Keep the issue open until outbound delivery recovery has a bounded, documented expiry policy. The likely core fix is to add optional `messages.delivery` schema/types/help/docs, apply an `enqueuedAt` age check in both startup recovery and reconnect drains before delivery, use a conservative terminal action such as moving expired entries to `failed/`, and add regression tests for default backward compatibility, old entries, reconnect drains, max-retry interaction, and `.delivered` marker cleanup.
