## Summary

Current main has solid manual abort paths (`/stop`, `chat.abort`, `sessions.abort`, queue `interrupt`), but the requested native message recall/edit behavior is not implemented. I found no core or plugin contract that represents inbound message deletion/edit events as cancellable task events, and no bundled-channel mapping from platform edit/delete/recall events to the existing abort machinery. The closest related work, PR #52993 from the provided context, is Feishu-specific message recall/delete tooling and does not supersede this broader cross-channel cancel-on-recall request.

## Best Possible Solution

Keep this open as a cross-channel channel/plugin API and runtime behavior request. The likely implementation path is to add a typed inbound message mutation event for delete/recall/edit with messageId/sessionKey correlation, wire recall/delete to the existing abort paths, define edit fallback behavior as cancel-and-rerun or context update, then implement and test adapters for bundled channels that expose those platform events.
