## Summary

Keep open. Current `main` has in-process follow-up queuing for inbound messages, durable outbound-delivery retry storage, restart continuation, and explicit `sessions_yield` support for subagent waits, but I did not find a durable per-session pending-deliverables queue or automatic resume path for arbitrary interrupted commitments like the issue describes.

## Evidence

- **Inbound follow-up queue is per-process, not durable:** The command queue docs describe serialization of inbound auto-reply runs and follow-up turns for inbound messages only. The implementation stores follow-up queues in a process-global map (`FOLLOWUP_QUEUES`), which means queued work is in memory rather than a durable session deliverables ledger.
  - file: [docs/concepts/queue.md:10](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/docs/concepts/queue.md#L10)
  - command: `nl -ba docs/concepts/queue.md | sed -n '1,120p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Follow-up queue implementation is in-memory:** `FOLLOWUP_QUEUES` is a global singleton map, and `enqueueFollowupRun()` pushes `FollowupRun` items into that map and restarts drain logic if idle. I found no persistence step here for promised outputs or interrupted commitments.
  - file: [src/auto-reply/reply/queue/state.ts:23](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/auto-reply/reply/queue/state.ts#L23)
  - command: `nl -ba src/auto-reply/reply/queue/state.ts | sed -n '1,220p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Durable queue exists for outbound delivery retries, not unfinished agent work:** `enqueueDelivery()` persists completed reply payloads to `delivery-queue/*.json` before send/retry. That covers transport recovery after a message already exists, not resuming an interrupted agent task that promised future output.
  - file: [src/infra/outbound/delivery-queue-storage.ts:129](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/infra/outbound/delivery-queue-storage.ts#L129)
  - command: `nl -ba src/infra/outbound/delivery-queue-storage.ts | sed -n '1,260p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Restart continuation is restart-specific:** `DEFAULT_RESTART_SUCCESS_CONTINUATION_MESSAGE` explicitly tells the resumed agent turn to continue pending work after a gateway restart. I did not find an equivalent generic mechanism for non-restart task switching/interruption.
  - file: [src/infra/restart-sentinel.ts:65](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/infra/restart-sentinel.ts#L65)
  - command: `nl -ba src/infra/restart-sentinel.ts | sed -n '55,90p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Explicit yield support is narrow and subagent-oriented:** Docs and tool code say `sessions_yield` intentionally ends the current turn so the next message can be a subagent follow-up event. That is an explicit tool path, not a general durable pending-deliverables queue for arbitrary interrupted work.
  - file: [docs/concepts/session-tool.md:87](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/docs/concepts/session-tool.md#L87)

- **Compaction preserves commitments in summaries but does not enforce delivery:** Compaction instructions tell summaries to preserve 'Any commitments or follow-ups promised', which helps memory retention, but I found no paired runtime enforcement that guarantees those commitments are later delivered.
