## Summary

Keep open. Current main has useful mitigations: per-session run serialization, inbound follow-up queueing, restart-specific resume, and durable transport delivery retry. It still does not implement the requested durable pending-deliverables/commitment queue that would resume a promised output after arbitrary task switching without a gateway restart.

## Best Possible Solution

Keep the issue open and treat it as a core session/turn-continuity problem. The likely product path is to add typed per-session durable pending-deliverable state or a task-flow-backed commitment record, write it before preemptive task switches/interrupts, clear it only on delivered/deferred/cancelled outcomes, and add tests that simulate Telegram or generic channel interruption without gateway restart.
