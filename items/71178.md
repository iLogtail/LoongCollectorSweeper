## Summary

Keep open. Current `main` has restart-continuation recovery for restart/config flows, but `update.run` still writes a restart sentinel without a continuation and the agent gateway tool still does not forward `continuationMessage` for updates, so the reported mid-update message-loss path remains unresolved.

## Evidence

- **`update.run` sentinel still has no continuation:** `update.run` builds a `RestartSentinelPayload` with `sessionKey`, `deliveryContext`, `threadId`, and `message`, then writes it. There is no `continuation` field in the payload, so restart recovery cannot re-dispatch an owed post-restart reply for update-triggered restarts.
  - file: [src/gateway/server-methods/update.ts:67](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/src/gateway/server-methods/update.ts#L67)
  - command: `nl -ba src/gateway/server-methods/update.ts | sed -n '1,180p'`
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **Gateway tool documents continuation support but skips it for `update.run`:** The gateway tool schema/description includes `continuationMessage`, and the restart path uses it to build a restart continuation, but the `update.run` branch only forwards `sessionKey`, `note`, `restartDelayMs`, and `timeoutMs`. That leaves agent-initiated updates without the continuation contract used by restart/config flows.
  - file: [src/agents/tools/gateway-tool.ts:323](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/src/agents/tools/gateway-tool.ts#L323)
