## Summary

Keep open. Current main still does not honor `OPENCLAW_WRAPPER` when generating daemon service `ProgramArguments`, and the PR is a focused fix for an open, concrete LaunchAgent reinstall/update problem from #69400. The PR should not be closed as obsolete, but it does need revision before landing because the review discussion correctly points out that wrapper resolution must use the install/service environment rather than only ambient `process.env`.

## Best Possible Solution

Keep #69425 open and ask for a revision rather than closing it. The best path is to add wrapper support at the shared daemon argument resolver/installation-plan boundary: accept the relevant install env, validate a trimmed absolute executable wrapper path, wire both gateway and node service plans consistently for LaunchAgent/systemd where intended, preserve or persist `OPENCLAW_WRAPPER` across reinstall/update, and add focused tests plus docs/operator guidance. If maintainers decide arbitrary service wrappers are not a supported product surface, resolve #69400 with a documented alternative before closing this PR.

## Risks / Open Questions

- The PR as written appears to use ambient `process.env`; it should use the merged install/service environment so reinstall/update preserves an already configured wrapper.
- Wrapper validation should likely require an executable regular file, not just an existing regular file, to avoid writing a service that cannot start.
- #69400 also proposes a CLI flag and doctor signal; maintainers may choose to land env support first, but the remaining operator UX should be explicitly accepted or split into follow-up work.
