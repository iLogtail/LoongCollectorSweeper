## Summary

Keep open. Current `main` has a CLI bootstrap that can pick up already-present shell `HTTP_PROXY`/`HTTPS_PROXY`, but the memory embedding transport still does not plumb a memory-specific proxy override, and `config.env` proxy vars are applied after that bootstrap. The reported `openclaw.json` `env` case therefore still appears unresolved on current `main`.

## Evidence

- **CLI proxy bootstrap happens before config env is loaded:** `runCli()` installs the global Undici env-proxy dispatcher at startup, but it does so before `loadConfig()` applies `config.env` values to `process.env`. That means proxy vars defined only in `openclaw.json` are not available when the dispatcher decision is made.
  - file: [src/cli/run-main.ts:187](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/cli/run-main.ts#L187)
  - command: `nl -ba src/cli/run-main.ts | sed -n '176,190p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Config env vars are applied later during config read:** `resolveConfigForRead()` calls `applyConfigEnvVars(...)` while loading config, after CLI startup. This ordering supports the issue's claim for `openclaw.json` `env` values being missed by early proxy bootstrap.
  - file: [src/config/io.ts:1054](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/config/io.ts#L1054)
  - command: `nl -ba src/config/io.ts | sed -n '1050,1060p'`

  - file: [src/infra/net/undici-global-dispatcher.ts:91](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/infra/net/undici-global-dispatcher.ts#L91)

It does not pass `mode: "trusted_env_proxy"` or any explicit proxy dispatcher policy for the memory transport itself.
