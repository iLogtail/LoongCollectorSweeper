## Summary

Keep #52640 open. Current main has adjacent observability pieces: Discord typing keepalive, optional draft-preview editing with tool-progress lines, status reactions, gateway session lifecycle broadcasts, and a durable task ledger for detached ACP/subagent/cron/CLI work. Those do not add the requested first-class persistent per-run channel status surface for normal long-running Discord/chat turns, especially with a single authoritative editable status message and unknown/interrupted semantics after restart or ownership loss. The related #59719 and #71630 threads cover narrower CLI-task liveness and Web UI status surfaces, not the full channel-turn status transport requested here.

## Best Possible Solution

Keep this open and implement a small channel/run status transport path rather than overloading draft reply streaming or detached task records. The likely path is a core lifecycle source plus a channel-gated, best-effort status adapter with Discord first: delayed create, meaningful milestone updates, explicit terminal states, and fail-closed unknown/interrupted handling on restart or lost ownership. Reuse existing session lifecycle, task reconciliation concepts, and Discord edit/fallback helpers where they fit, while keeping #59719's CLI-task liveness and #71630's Web UI chip as narrower related work.
