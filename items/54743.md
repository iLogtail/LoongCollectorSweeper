## Summary

Keep open. Current `main` still has the same recursive warning/logging shape that matches the report: normal CLI startup patches `console.warn`, the patched console path lazily initializes file logging by reading config again, and `config set` still emits config-validation warnings during write preparation. I did not find a fix on `main` or between `v2026.4.22` and `HEAD` for that re-entrant path.

## Evidence

- **CLI commands still patch console before command execution:** Normal CLI startup calls `enableConsoleCapture()` before building/running commands, so `config set` runs with patched `console.warn`/`console.error`.
  - file: [src/cli/run-main.ts:211](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/cli/run-main.ts#L211)

- **Patched console.warn still initializes the logger lazily:** The console capture wrapper calls `getLoggerLazy()`, and the `warn` branch forwards to `resolvedLogger.warn(formatted)`. That means a config warning emitted through `console.warn` can recurse back into logger setup.
  - file: [src/logging/console.ts:221](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/logging/console.ts#L221)
