## Summary

Current main already fixes #60632. The cron/task paths may still pass a pre-captured startedAt, but task registry creation, update, and restore now normalize lifecycle timestamps before records are persisted or exposed, so audit-visible records no longer produce startedAt < createdAt false positives. Regression tests cover create/update/restore timestamp skew and assert zero inconsistent_timestamps findings. The fix was implemented by PR #71871 / commit de5b173546d4405bdc0971222c2dc85ae9e9c5b0; it is on main, but I could not verify a shipped release tag newer than v2026.4.24 for it.

## Best Possible Solution

Keep the merged #71871 registry-boundary normalization. It solves the user-visible invariant for current main without needing cron-specific timestamp special cases; release tracking can happen through the normal changelog/release process.
