## Summary

Keep PR #71063 open. Current main still writes initial input to `child.stdin` after only checking that the stream exists, and there is no current regression coverage for the non-writable/EPIPE stdin case. The PR is a focused gateway/process bug fix with a plausible affected path through CLI backends such as `google-gemini-cli`, so conservative cleanup should not close it as implemented, obsolete, or unactionable.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to land a scoped fix in `src/process/supervisor/adapters/child.ts` that protects the initial stdin write/end path for failed CLI spawns, with a regression test in `src/process/supervisor/adapters/child.test.ts` that models non-writable or throwing stdin behavior closely enough to catch the gateway crash.

## Risks / Open Questions

- The proposed test overrides `PassThrough.writable` but does not fully emulate a real closed/EPIPE stream, so maintainers may want a slightly stronger regression before merge.
- The one-line guard only skips initial input when `writable` is already false; if a stream reports writable and then errors during write/end, a broader try/catch around the initial stdin write path may be safer.
