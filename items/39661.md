## Summary

Keep #39661 open. Current main still has Discord mention gating as a guild/channel boolean, while Discord `users`/`roles` are sender allowlists. The runtime mention resolver is not sender-aware, and the strict config schema would reject the requested per-user override map.

## Best Possible Solution

Keep the issue open for bundled Discord plugin work. A good fix would add a typed sender-specific mention policy to the Discord guild/channel config contract, validate it in zod and generated config metadata, define precedence between guild defaults, guild sender overrides, channel defaults, and channel sender overrides, pass sender identity into Discord mention resolution, and add docs plus regression tests showing two users in the same channel with different trigger behavior.

## Risks / Open Questions

- Changing this behavior needs explicit precedence rules so sender authorization allowlists are not conflated with reply-trigger mention policy.
- A per-user `requireMention: false` path can widen activation in shared Discord channels if defaults and overrides are ambiguous.
