## Summary

Keep open. Current main has generic embedded-runner HTTP timeout/proxy hardening and context-pressure mitigations, but I did not find source, tests, release history, or a canonical superseding item proving the specific large-context OpenRouter “Network connection lost” failure is fixed. The reporter also reproduced it on the main agent and explicitly ruled out bodyTimeout, so PR #61803’s heartbeat routing fix and timeout plumbing are not enough to close this item.

## Best Possible Solution

Keep #53540 open and investigate the embedded OpenAI-compatible transport path under large accumulated tool-result context. A good maintainer fix would add raw error tracing before network-error classification, reproduce with a synthetic or live large-context OpenRouter-compatible stream, then add a regression test around the failing transport/stream lifecycle. Related open reports such as #43113 and #49919 can remain context, but #53540 is currently the most precise reproduction and should not be auto-closed as duplicate without a canonical tracking issue.
