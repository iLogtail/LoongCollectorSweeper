## Summary

Keep #49507 open. Current main has a partial mitigation for cron delivery account selection by falling back to an agent/channel route binding, but it does not implement the requested accountId inference from agentId and it still preserves a session-derived lastAccountId ahead of bindings. The exact attempted fixes in #49623/#49633 were closed unmerged, and no canonical open follow-up in the provided context fully supersedes this Feishu/Telegram multi-account bug.

## Best Possible Solution

Keep the issue open and implement a deterministic cron outbound account resolver. A conservative fix should add tests for Feishu/Telegram-style multi-account cron jobs with agentId but no delivery.accountId, stale session lastAccountId, explicit --account precedence, and configured route bindings. The resolver should prefer explicit job account, then a preserved live/requester delivery account only when it is intentionally tied to the cron target, then agent/channel bindings or a configured account matching the agentId, and only fall back to channel defaults when no agent-specific account can be resolved.

## Risks / Open Questions

- Closing now would hide a remaining multi-account delivery safety issue: current main can still use session lastAccountId, fall back to channel defaults, or leave accountId undefined instead of deterministically selecting the agent's intended outbound account.
- There is a partial fix path through explicit account-scoped bindings, so the eventual issue resolution should preserve that precedence and avoid blindly mapping every agentId to an account unless that account exists for the target channel.
