## Summary

Keep open. Current main still has the reported session-list hot path: rows are fully hydrated before search/active filters/limit, hydration can synchronously read transcripts for usage and title/preview fields, sessionId resolution still reuses the full session list, and MCP getConversation still searches a limit-500 conversation list. Related #57715 overlaps, but the provided context only proves it covers the sessions.list hot path; #72165 also documents direct sessionId lookup and MCP lookup costs, so closing this as a duplicate would be premature.

## Best Possible Solution

Keep #72165 open until the remaining work is fixed or explicitly consolidated into a canonical issue. The likely implementation belongs in the gateway session surface: persist or cache lightweight summary fields at write/update time, apply search/active/limit before expensive transcript hydration, add direct sessionId/key lookup that preserves visibility/deleted-agent checks, and change MCP getConversation to use a direct gateway lookup instead of listConversations(limit: 500). Add regression coverage around list limit/search avoiding per-row transcript reads and around sessionId/MCP lookup avoiding full-list materialization.
