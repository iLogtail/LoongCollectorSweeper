## Summary

Keep open. Current main still hardcodes MiniMax usage polling to the CN host and does not pass configured MiniMax baseUrl into fetchMinimaxUsage, so the PR's central bug is not fixed on main. The PR is plausible but not ready to close or merge blindly because review identified an auth-source/host mismatch risk and the patch adds no URL-routing regression tests.

## Best Possible Solution

Keep the PR open or consolidate it manually with #65054/#63073, but do not auto-close it. The best fix is a maintainer-reviewed MiniMax usage-host resolver that derives the host from the same credential/source chosen by resolveUsageAuth, preserves the /v1/api/openplatform/coding_plan/remains path, keeps CN fallback behavior, avoids unsafe env-only routing, and adds URL-selection regression tests.

## Risks / Open Questions

- Global MiniMax users on api.minimax.io still appear affected on current main because usage polling remains pinned to api.minimaxi.com.
- Merging this PR as-is can mismatch the selected usage token and host when both minimax and minimax-portal configs exist with different regions.
- The patch needs targeted tests for global, CN, fallback, invalid baseUrl, and mixed provider-config cases.
