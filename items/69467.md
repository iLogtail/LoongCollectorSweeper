## Summary

Keep open. Current main still has the risky composition described in #69467: `sessionTarget: "current"` is persisted as `session:<live session key>`, the detached cron runner uses that live key as the run/session binding without forcing a fresh session unless the target is exactly `isolated`, and `deleteAfterRun` direct-delivery cleanup deletes `params.agentSessionKey`. The related #67807 cleanup work is real, but it targeted leaked isolated sessions and did not add a guard for live `current`/`session:*` DM bindings.

## Best Possible Solution

Keep this issue open and fix the core cron lifecycle. `deleteAfterRun` should only delete cron-owned isolated/ephemeral run sessions, or the runner should force a per-run isolated working session when `deleteAfterRun` is true for a live `current`/`session:*` target. Add a regression covering a non-default live channel session key normalized from `current`, multiple one-shot `at` jobs with `deleteAfterRun: true`, direct and message-tool delivery paths, and an assertion that the live DM session binding is not deleted or rebound.
