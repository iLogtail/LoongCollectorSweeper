## Summary

Keep open. Current main still contains the reported core cron/session failure path: `sessionTarget: "current"` is persisted as `session:<live session key>`, one-shot `at` jobs default to `deleteAfterRun: true`, non-`isolated` session targets reuse the resolved live session key, and direct-delivery cleanup still calls `sessions.delete` on that key. Existing tests assert the risky pieces separately but do not cover the live non-default DM preservation case described in #69467.

## Best Possible Solution

Keep #69467 open and fix it in core cron/session handling. The safest direction is to make `deleteAfterRun` cleanup only delete cron-owned isolated/run-scoped sessions, never a live `current` or `session:<channel-session-key>` binding. For current/session-bound jobs, either skip session deletion after reuse or execute in a separate ephemeral run session while preserving delivery to the live channel. Add a regression covering a non-default agent live DM, `sessionTarget: "current"` or `session:<live key>`, one-shot `at` default `deleteAfterRun`, direct delivery, and message-tool/no-delivery paths, asserting the live session binding and transcript remain intact.
