## Summary

Keep #41355 open. Current main still has the Discord adapter pass `ThreadStarterBody` into the finalized inbound context on every thread message, even after reading `previousTimestamp`. The shared reply path then still has a follow-up-session path that can prepend `[Thread starter - for context]` when `ctx.ThreadStarterBody` is present. The related fix PR #50522 is open and unmerged in the provided GitHub context, so this issue remains the canonical tracker rather than cleanup-closeable.

## Best Possible Solution

Keep #41355 open. Land #50522 or an equivalent Discord adapter change that sets `ThreadStarterBody` only when there is no previous session timestamp, then add or update a Discord regression test proving follow-up thread messages record/finalize `ThreadStarterBody` as undefined while first-turn context is preserved. Treat outbound metadata stripping as separate follow-up hardening.
