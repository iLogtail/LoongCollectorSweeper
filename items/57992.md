## Summary

Keep open. Current `main` has generic subagent error propagation and failed-status surfacing, but I did not find strong evidence that the specific sandbox-container-create failure is fully fixed. The Docker sandbox create/start path still has no dedicated catch/log/retry handling, and I did not find a sandbox-create regression test proving this transient failure now logs and terminates cleanly instead of relying on generic downstream lifecycle handling.

## Evidence

- **Sandbox container creation still has no local catch/log/retry:** `createSandboxContainer()` runs `docker create`, `docker start`, and optional setup command sequentially with plain awaits. There is no surrounding `try/catch`, no explicit gateway log on failure, and no retry path in this function.
  - file: [src/agents/sandbox/docker.ts:470](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/agents/sandbox/docker.ts#L470)

  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Sandbox ensure path still just bubbles create failure:** `ensureSandboxContainer()` calls `createSandboxContainer()` when the runtime does not exist, but does not add any sandbox-specific recovery or state transition around that failure path.
  - file: [src/agents/sandbox/docker.ts:578](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/agents/sandbox/docker.ts#L578)

- **Generic agent failures now emit lifecycle error:** If a run throws before normal completion, `agentCommand` emits a lifecycle `error` event with the error message. That is useful generic plumbing, but it is not sandbox-create-specific logging.
