## Summary

Current `main` has generic async agent-run error propagation, but I did not find strong evidence that the specific transient Docker sandbox-create failure path from #57992 is fixed end-to-end. The spawn path still acknowledges the run before sandbox startup, the Docker create/start path still has no local logging or explicit session-state/error annotation, and the nearby tests only cover a mocked post-bind agent-start failure rather than a real sandbox backend/container creation failure.

## Evidence

- **Spawn is acknowledged before sandbox startup:** The gateway `agent` method sends `{ status: "accepted" }` immediately, before dispatching the actual run. That means `sessions_spawn` can still observe an accepted child run even if sandbox initialization fails later during execution.

There is no local `try/catch` here to log Docker/backend creation failures or attach them to session metadata before the error bubbles upward.

## Risks / Open Questions

- A transient Docker API/container-create failure may still present as an accepted spawned run before the backend actually initializes.
- The code reviewed suggests generic error bubbling, but I did not find direct proof that gateway logs, session metadata, and `subagents list` now all show actionable failure details for this path.
- There is no obvious regression test on `main` for the exact reported failure mode, so closing would be higher-risk than leaving the issue open.
