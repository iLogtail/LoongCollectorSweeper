## Summary

Keep open. Current main has a pre-compaction memory flush, but it is still a single-threshold, once-per-compaction-cycle mechanism. The requested progressive 80/90/95-style stages, stage-specific prompts, and forced summary behavior are not implemented, and the current memory plugin API only exposes a single flush plan. The linked PR #43102 was a docs RFC that closed unmerged, and related token-budget/memory-durability issues do not canonically cover this exact progressive flush request.

## Best Possible Solution

Keep #43006 open for maintainer/product judgment. A real implementation would likely belong in the core/Gateway memory-flush pipeline: add staged threshold configuration or a staged flush plan, persist per-stage completion metadata instead of only `memoryFlushCompactionCount`, support stage-specific prompts and a final forced-summary/compaction handoff, and update config schema, docs, and tests. If maintainers want this plugin-owned, the memory flush API first needs stage/token-pressure inputs because the current resolver cannot implement the requested behavior externally.
