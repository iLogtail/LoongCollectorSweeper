## Summary

Keep #17876 open. Current main still has immediate `models set`, manual fallback management, and OpenRouter-oriented scan helpers, but it does not implement the requested staged model migration workflow: test an arbitrary candidate, promote it while moving the old primary to fallback, and rollback in one command. The latest issue discussion also includes an explicit maintainer-authored Codex review saying to keep it open.

## Best Possible Solution

Keep #17876 open for maintainer/product follow-through. The likely implementation belongs in the core models CLI because it coordinates primary/default config writes, fallback ordering, provider/auth probes, sub-agent/tool compatibility checks, and rollback metadata. Build it on the existing `models status`, auth probe, fallback, and scan primitives: add/test a candidate as fallback, promote only after verification while preserving the old primary as fallback, and store enough prior state for `openclaw models rollback`. Add CLI docs and focused config-transition tests.
