## Summary

Keep open. Current `main` still has the silent-drop control-flow hole this PR targets, so the bug is real and unshipped, but the proposed early-return flush is still incomplete: a buffered answer finalized via preview edit would bypass the Telegram internal `message:sent` hook, matching the outstanding PR review comment.

## Evidence

- **Current main still returns before flushing buffered final answers:** In `extensions/telegram/src/bot-message-dispatch.ts`, answer segments are buffered when `shouldBufferFinalAnswer()` is true, reasoning delivery only flushes when the reasoning segment is not `skipped`, and the `segments.length > 0` early-return exits without any `flushBufferedFinalAnswer()` call. That is the silent-drop path described in #53384/#53762.
  - file: [extensions/telegram/src/bot-message-dispatch.ts:859](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/telegram/src/bot-message-dispatch.ts#L859)
  - command: `nl -ba extensions/telegram/src/bot-message-dispatch.ts | sed -n '655,905p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Buffering state machine requires an explicit flush after a hinted-but-undelivered reasoning segment:** `createTelegramReasoningStepState()` buffers the final answer whenever reasoning is only `hinted`, and it only leaves that state via `noteReasoningDelivered()` or `resetForNextStep()`. A skipped reasoning delivery therefore leaves the answer buffered unless a later explicit flush happens.
  - file: [extensions/telegram/src/reasoning-lane-coordinator.ts:98](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/telegram/src/reasoning-lane-coordinator.ts#L98)

- **PR fix path still misses `message:sent` hook for buffered preview finalization:** `emitPreviewFinalizedHook()` is only called on the immediate `deliverLaneText()` result inside the segment loop. The local `flushBufferedFinalAnswer()` helper calls `deliverLaneText()` for the buffered answer but discards the returned `LaneDeliveryResult`, so a `preview-finalized` buffered answer would not emit the internal hook. That matches the existing Codex review comment on the PR.
