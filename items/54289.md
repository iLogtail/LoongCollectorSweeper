## Summary

Keep open. Current main still contains the Gemini CLI OAuth client discovery behavior reported here: it searches for a local `gemini` install, reads Gemini CLI package files for OAuth client config, caches the result, and falls back to it when explicit env vars are absent. Current code does add a Gemini CLI OAuth warning/confirmation before the login flow, so the issue overstates the startup/no-consent path, but the underlying cross-application credential extraction has not been removed or otherwise resolved.

## Best Possible Solution

Keep the issue open for explicit security maintainer judgment. A concrete product fix should either remove the Gemini CLI filesystem scan and require explicit `OPENCLAW_GEMINI_OAUTH_CLIENT_*` / `GEMINI_CLI_OAUTH_CLIENT_*` configuration, or add an explicit, auditable consent/policy surface that tells users OpenClaw will read the local Gemini CLI OAuth client config. Update the Google plugin tests and public provider docs to match the chosen boundary. The `protect-mcp` comment is useful related context for credential isolation, but it is not a replacement for deciding this bundled Google plugin behavior.
