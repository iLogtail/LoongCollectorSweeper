## Summary

Keep #62261 open. Current main still has a source-level path that matches the report: dev update preflight records failed lint attempts while walking candidate commits, then final git-update status scans all recorded failed steps and can report an earlier `preflight lint (...)` failure even after a later candidate succeeds. The linked #63627 is related memory-pressure discussion for the pre-commit hook, but it does not cover this updater-result bug.

## Best Possible Solution

Add a focused updater regression test where an earlier dev candidate's preflight lint is SIGKILL/nonzero and a later candidate passes, then change the updater result calculation so failed preflight candidate steps are diagnostic history rather than final blocking failures once `selectedSha` is found. If memory pressure is also addressed, keep it scoped to update preflight rather than relying on the pre-commit-hook fix from #63627.
