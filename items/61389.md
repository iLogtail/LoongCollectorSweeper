## Summary

Keep #61389 open. Current main already handles the stale-threshold concern with dynamic validation against the configured/default health-check interval, but `gateway.channelMaxRestartsPerHour: 0` is still rejected and a schema-only `min(0)` change would not implement the requested “unlimited” behavior because runtime cap logic would treat 0 as zero allowed restarts.

## Best Possible Solution

Keep the issue open but narrow the fix. Preserve the current dynamic stale-threshold validation, then decide whether `channelMaxRestartsPerHour: 0` should mean unlimited. If yes, implement that sentinel in runtime, Zod schema, generated schema/help/docs, and tests. If no, document that `gateway.channelHealthCheckMinutes: 0` and `channels.<provider>.healthMonitor.enabled: false` are the supported ways to disable health-monitor restarts.

## Risks / Open Questions

- A mechanical `min(0)` schema-only patch would be misleading unless runtime maps 0 to an explicit no-limit sentinel; today it would effectively mean no restarts.
- The stale-threshold part of the issue should not be implemented as a hard `min(5)` without product review, because current schema allows lower health-check intervals and validates the threshold relative to the effective interval.
