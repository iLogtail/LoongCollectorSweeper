## Summary

Keep open. Current main has one-shot cron retry config and a recurring-job error backoff floor, but it does not implement per-job `--retry-count`, `--retry-delay`, or `--retry-backoff`, and a failed daily cron can still wait until its next natural scheduled run. The linked PR #51001 and related issue #38069 track the same feature area but remain open/unmerged, so this issue is still actionable.

## Best Possible Solution

Keep the issue open until recurring cron retry is implemented or the maintainer chooses a canonical tracker. The likely product path is to review/finish PR #51001 or implement an equivalent core scheduler change: add an explicit per-job recurring retry policy, wire it through CLI/API/schema/docs, track retry attempts separately from normal schedule cadence, retry transient failures before the next natural daily/weekly slot, and add focused regression tests for success, exhaustion, persistence, and cadence preservation.

## Evidence
