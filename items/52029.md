## Summary

Keep #52029 open. Current main has heartbeat cost/scope controls such as model override, lightContext, isolatedSession, timeout, and suppressToolErrorWarnings, but there is still no heartbeat-specific tools=false or toolsAllow policy wired from heartbeat config into the agent run.

## Best Possible Solution

Keep the issue open and implement a heartbeat-scoped tool policy. The likely path is to add a strict heartbeat config field in config types/schema/docs, extend GetReplyOptions with disableTools and/or toolsAllow, pass it from runHeartbeatOnce into getReplyFromConfig, then thread it into runCliAgent/runEmbeddedAgent where disableTools/toolsAllow already exist. Add heartbeat-runner tests proving tools:false reaches the agent run without changing normal user turns.

## Risks / Open Questions

- Global or per-agent tools.allow/deny can restrict a model generally, but it is not heartbeat-specific and cannot keep normal interactive turns fully tooled while making only heartbeat runs text-only.
- Cron has a runtime toolsAllow shape, so maintainers should choose whether heartbeat gets a boolean tools:false, a toolsAllow list, or both for consistency.
