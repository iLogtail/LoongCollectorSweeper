## Summary

Keep #31396 open. Current main has safeguards for explicit `/model`, `session_status(model=...)`, `sessions.patch`, fallback retry loops, and overloaded/live-switch conflicts, but the issue still does not identify which model-switch path froze or whether this is actually provider connection latency. The related provider connect-timeout PR remains open/unshipped in the provided context, so implemented/cannot-reproduce would overstate the evidence.

## Best Possible Solution

Keep the issue open and ask for a concrete reproduction: exact command or prompt, channel/UI, provider/model, current/default model, whether a run was active, OpenClaw version/config, and logs around the freeze. If it reproduces in `/model`, `session_status(model=...)`, or `sessions.patch`, add a targeted live-switch regression. If it is provider connection latency, resolve it through the provider timeout design tracked by PR #41936 or a narrower successor.
