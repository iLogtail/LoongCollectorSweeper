## Summary

Keep open. Current `main` has native-command targeting safeguards via `CommandTargetSessionKey`, but the reported core symptom remains plausible because Telegram native commands still persist inbound session metadata onto the slash-session key while carrying group/topic context. I did not find strong evidence that current `main` prevents a helper-agent `agent:<helper>:telegram:slash:<user>` entry from being recorded as a group/topic-shaped shell, and the issue’s linked follow-up PR #57548 is still only a cross-reference in the timeline, not proof of a shipped fix.

## Evidence

- **Telegram native commands still record metadata under the slash-session key:** `recordInboundSessionMetaSafe` is called with `sessionKey: ctxPayload.SessionKey`, and `ctxPayload.SessionKey` is the slash key built by `resolveNativeCommandSessionTargets`. The same payload still carries group/forum fields such as `ChatType`, `From`, and `MessageThreadId`. That means the slash-key store entry can still be hydrated from a group/topic context.
  - file: [extensions/telegram/src/bot-native-commands.ts:938](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/telegram/src/bot-native-commands.ts#L938)
  - command: `nl -ba extensions/telegram/src/bot-native-commands.ts | sed -n '680,1020p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Session metadata derivation still projects group-shaped context into the stored entry:** `deriveSessionMetaPatch` uses `deriveGroupSessionPatch`, which derives group metadata from `ctx.ChatType` and `ctx.From`. If the slash-session update is invoked with Telegram group/topic context, the stored slash entry can still get `chatType=group` and a derived `groupId`.
  - file: [src/config/sessions/metadata.ts:122](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/config/sessions/metadata.ts#L122)
  - command: `nl -ba src/config/sessions/metadata.ts | sed -n '1,260p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Group resolution is driven by group chat markers in the inbound context:** `resolveGroupSessionKey` treats contexts as group sessions when `ChatType` is `group`/`channel` or `From` contains `:group:`/`:channel:`. Telegram native command payloads for topic commands satisfy that shape.
  - file: [src/config/sessions/group.ts:76](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/config/sessions/group.ts#L76)
  - command: `nl -ba src/config/sessions/group.ts | sed -n '1,260p'`

- **Current main does route native-command execution against the target conversation session:** `initSessionState` explicitly prefers `ctx.CommandTargetSessionKey` over the slash-session key for session mutation and loading. This is evidence of partial mitigation, but it does not prove the slash-key persistence bug itself is gone.
