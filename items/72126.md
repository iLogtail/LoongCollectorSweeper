## Summary

Keep #72126 open. Current main still uses the raw `OPENAI_API_KEY` fallback in both OpenAI Realtime browser-session creation and server-side realtime voice bridge creation, so an env value like `keychain:openclaw:OPENAI_API_KEY` can still be passed as the bearer token. The PR is a focused fix for a real current gap, but should address the existing review concern around retry/caching semantics before landing.

## Best Possible Solution

Keep the PR open and finish it as an OpenAI plugin-owned bug fix: resolve supported keychain SecretRef strings before both `createBrowserSession` and `createBridge` use the API key, avoid permanent caching of transient failures, keep `isConfigured` from doing expensive synchronous work if possible, and add regression tests for env-backed `keychain:<service>:<account>` on both affected Realtime paths.
