## Summary

Keep open. Current main still clears the Overview Gateway Token when the WebSocket URL input changes, so this PR is not implemented or obsolete. However, the PR's one-line preservation behavior conflicts with the current credential-scoping contract: tokens are intentionally not reused across gateway targets, and connectGateway sends settings.token to the configured gateway URL. This needs maintainer judgment or a more nuanced UI fix rather than conservative cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer/product judgment, but do not merge the one-line token-preservation change as-is. The better fix is to separate transient URL editing from committed gateway changes: preserve the token while the user edits a draft value for the same effective endpoint, but clear or require explicit confirmation/new token when the normalized gateway target changes, with regression tests covering both same-endpoint edits and different-endpoint credential isolation.
