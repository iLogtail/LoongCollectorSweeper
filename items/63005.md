## Summary

Keep open. Current `main` still has the split-session failure mode described: inbound auto-reply keeps an explicit non-agent session key as-is, while gateway resolution canonicalizes that same key to `agent:<default-agent>:...` and gateway migration prunes the raw alias. That means a later inbound turn on the original raw key can miss the migrated record and create a fresh session. Recent tests added on `main` cover the older `agent:main:*` alias/orphan case, not arbitrary external raw keys like this report.

## Evidence

- **Inbound write path still preserves explicit raw session keys:** `initSessionState` derives `sessionKey` from `resolveSessionKey(...)` and only applies `canonicalizeMainSessionAlias(...)`. For an explicit external key that is not a main alias, the key remains raw and is looked up directly in `sessionStore[sessionKey]`.

That helper lowercases and lets channel plugins normalize, but otherwise returns the raw explicit key unchanged.
