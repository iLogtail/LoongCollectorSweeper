## Summary

Keep open. Current main still implements BlueBubbles inbound webhook auth with `channels.bluebubbles.password`, and the public docs still tell operators to put that password in the webhook URL. The PR is an active, relevant fix for a still-present channel auth problem, not an obsolete cleanup candidate.

## Best Possible Solution

Keep #70757 open and continue review toward a focused fix: add a distinct BlueBubbles `webhookSecret`, authenticate inbound webhooks only against that secret, validate password/secret separation in core and plugin schemas, expose setup/CLI support, update docs, and keep regression coverage for password-reuse and inherited-account cases. If maintainers prefer a smaller patch, split this PR, but #70755 should remain tracked until the current password-as-webhook-secret behavior is fixed on main.
