## Summary

PR #70757 should stay open. Current main still authenticates BlueBubbles inbound webhooks with the BlueBubbles API password, documents publishing that password in the webhook URL, and lacks a `channels.bluebubbles.webhookSecret` config/setup surface. The provided PR context is a focused security fix for linked issue #70755, and later review comments were addressed in the PR head, so this is not conservative cleanup material.

## Best Possible Solution

Keep PR #70757 open or land an equivalent maintainer-approved fix. The best product direction is to add a dedicated `channels.bluebubbles.webhookSecret` surface, authenticate inbound BlueBubbles webhooks only against that secret, enforce that it differs from the API password across inherited account configs, expose it in setup/CLI metadata, and update docs/tests/secretref metadata consistently.
