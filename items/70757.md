## Summary

Keep #70757 open. Current main still uses `channels.bluebubbles.password` as the inbound BlueBubbles webhook credential and public docs still tell operators to put that password in the webhook URL, so the PR is addressing a real remaining bug rather than obsolete work.

## Best Possible Solution

Keep the PR or a narrowed successor open until main has a dedicated BlueBubbles `webhookSecret` surface wired through plugin/core schemas, setup CLI and wizard, secret registration, runtime webhook auth, docs, and regression tests that reject password-only or password-reused webhook auth.
