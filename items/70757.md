## Summary

Keep open. Current main still authenticates inbound BlueBubbles webhooks against `channels.bluebubbles.password`, the same credential used for the BlueBubbles API, and the docs still tell operators to publish `?password=<password>` in the webhook URL. The PR is a focused, still-relevant fix for a live security boundary rather than obsolete cleanup.

## Best Possible Solution

Keep the PR open and review it as the likely implementation path for #70755: add a dedicated `channels.bluebubbles.webhookSecret` surface, authenticate webhooks only against that secret, validate that it exists and differs from the API password, expose setup/CLI support, update secret metadata/docs, and keep regression coverage for inherited account credentials.
