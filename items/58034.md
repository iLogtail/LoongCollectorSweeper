## Summary

Keep open. Current `main` does not contain the follow-up hardening or the explicit `tools.web.fetch.useTrustedEnvProxy` opt-in described in PR #58034; it still has only the earlier baseline trusted-env-proxy behavior.

## Evidence

- **Fetch guard still uses the broad proxy-env gate:** On `main`, `fetchWithSsrFGuard` still enables `TRUSTED_ENV_PROXY` with `hasProxyEnvConfigured()` and does not check protocol-specific HTTP(S) proxy applicability or `NO_PROXY` before taking that branch.

  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Underlying helper still counts ALL_PROXY as sufficient:** `hasProxyEnvConfigured()` returns true for any non-empty proxy env var, including `ALL_PROXY`/`all_proxy`, which is exactly the overbroad gate the PR is trying to narrow.
  - file: [src/infra/net/proxy-env.ts:10](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/infra/net/proxy-env.ts#L10)
