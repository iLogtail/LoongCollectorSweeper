## Summary

Keep open. Current `main` still does not add a hard timeout around `memory_search`, so the liveness bug described by the PR remains plausible. The PR also appears to contain useful follow-up on timeout-specific messaging from review, even though its original file paths are stale after the memory tool moved into `extensions/memory-core`.

## Evidence

- **Current main still awaits memory manager init and search without a tool-bound timeout:** `createMemorySearchTool` directly awaits `getMemoryManagerContext()` and then `memory.manager.search()`; there is no outer `Promise.race`, abort, or deadline around the full `memory_search` execution path.
  - file: [extensions/memory-core/src/tools.ts:211](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/extensions/memory-core/src/tools.ts#L211)

It does not recognize a wall-clock timeout case or return the timeout-specific warning/action that the PR discussion proposed.
