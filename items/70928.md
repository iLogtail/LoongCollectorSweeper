## Summary

Keep open. Current main still appears to bypass outbound lifecycle hook wiring for the WebChat/Gateway TUI path: TUI sends through `chat.send`, `chat.send` constructs a raw reply dispatcher and calls `dispatchInboundMessage`, while the `message_sending` auto-wiring lives only in the higher-level dispatcher helpers. `message_sent` emission is implemented in `deliverOutboundPayloads`, but the WebChat reply dispatcher does not use that path or emit the hook itself.

## Best Possible Solution

Keep #70928 open and either fix it directly or attach it to the broader #50126 hook-coverage work. The practical fix should move WebChat/TUI reply delivery onto the same hook-aware dispatcher seam, or add a shared reply observer that fires `message_sending` before delivery and `message_sent` after success/failure. Add regression coverage for `chat.send` and Gateway-backed TUI so a registered plugin hook observes assistant replies on these default local surfaces.
