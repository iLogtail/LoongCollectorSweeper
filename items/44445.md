## Summary

Current main still does not implement the PR's central behavior for session-memory files. The session-memory transcript reader extracts user text and writes it directly into the memory summary without calling the inbound metadata sanitizer or message-id hint stripper. Current main does have UI/chat-history sanitization for inbound metadata, but that path is separate from the bundled session-memory hook. The PR remains relevant, though its current diff also includes broader scheduled-reset hook behavior with unresolved review concerns, so it should stay open for maintainer review/refinement rather than be auto-closed.

## Best Possible Solution

Keep this PR open, but narrow/refine it before merge. The best implementation likely belongs in `src/hooks/bundled/session-memory/transcript.ts`: sanitize user-role text before writing memory summaries using the existing inbound metadata cleanup plus the shared message-id hint stripping rules, add focused regression tests for Feishu metadata, timestamp-prefixed metadata, message-id envelope handling, and slash-prefixed real content, and handle scheduled/daily reset flushing as a separate lifecycle decision rather than silently reusing `command:new` for non-command rollovers.

## Risks / Open Questions

- The current PR diff includes broader scheduled/daily reset hook dispatch beyond the title, and review comments flag behavioral risks around firing `command:new` hooks without an actual command.
- A correct fix needs to preserve real user content that resembles `[message_id: ...]` while still stripping Feishu/OpenClaw-injected envelope lines, consistent with the invariant in `src/auto-reply/reply/inbound-meta.ts`.
- A fix should decide command filtering order carefully so metadata stripping does not accidentally drop legitimate slash-prefixed user text that was previously retained.
