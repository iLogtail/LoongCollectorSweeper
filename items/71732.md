## Summary

Keep open. Current main still lacks the requested generic plugin session-extension contract. The existing surfaces are adjacent only: `sessions.patch` validates a closed set of core fields, the gateway handler applies core patch logic and emits a post-write internal hook, session rows project hardcoded fields, and `OpenClawPluginApi` exposes gateway RPCs/hooks/runtime store helpers but no typed `registerSessionExtension`-style registration, host-routed patch actions, public projection, or lifecycle cleanup contract.

## Best Possible Solution

Keep this open as a real core/plugin-SDK API gap. The likely implementation belongs in the shared plugin contract: add a typed session-extension registration on `OpenClawPluginApi`, registry storage, gateway protocol support for host-authorized plugin patch actions, schema validation before writes, explicit public projections in session rows/details, reset/delete/compaction/plugin-disable cleanup, stable error codes, and fixture-plugin contract tests. Coordinate the shape with the open #71731 RFC packet and the narrower #71426 sessions.patch hook discussion rather than closing this as already implemented.

## Risks / Open Questions

- The API shape still needs maintainer product judgment, especially `sessions.pluginPatch` versus an additive `sessions.patch.plugin` envelope.
- Plugins can currently write raw session data through runtime helpers or expose separate gateway methods, but that bypasses the host-owned validation, projection, broadcast, reset/delete/compaction, and disabled-plugin semantics requested here.
