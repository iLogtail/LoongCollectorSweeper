## Summary

Keep open. Current main has already shipped substantial web_fetch external-content hardening, including hidden HTML removal and invisible Unicode stripping, plus generic plugin hooks that can rewrite tool-result persistence. But the specific remaining request is not fully implemented: there is no repo-backed content-sanitizer.py/script pipeline, no tools.preprocess config surface, no structured threat_level/blocking result for web_fetch/web_search, and the related broader security issue #54508 remains open rather than superseding this exact API/product decision.

## Best Possible Solution

Keep the issue open but narrow it. Do not integrate the referenced local /home/local Python script directly into core. If maintainers want this capability, define an additive, documented tool-result preprocessing/threat-classification API for web_fetch/web_search, preferably aligned with existing plugin hooks and #54508 taint/exec-guardrail work, with structured threat_level/block semantics and tests for web_search as well as web_fetch.
