## Summary

The request is a useful observability add-on, but current main treats this kind of optional runtime integration as plugin/ClawHub scope rather than OpenClaw core. Main already documents plugin publication via ClawHub, supports hook/service plugins, and exposes a `gateway_start` hook that can load a startup-time integration like burn0 without adding a new core dependency.

## Risks / Open Questions

- If burn0 cannot be initialized reliably from the existing plugin startup hook and would require earlier process bootstrap than plugins get today, this scope call could need revisiting.
- This closes the core request, not the underlying user need; someone still has to build and publish the ClawHub plugin.

## Close Comment

Closing as `clawhub` after Codex review. Current main is explicit that optional capability should usually ship as plugins, not OpenClaw core, and that new skills should go to ClawHub first (`VISION.md:54-70`). The plugin path is already documented and supported here: plugin authors are told to publish to ClawHub/npm instead of adding code to this repo (`docs/plugins/building-plugins.md:16-19`), plugins can register hooks/services (`docs/plugins/building-plugins.md:145-168`), and the public SDK already exposes a `gateway_start` lifecycle seam for startup-time behavior (`docs/plugins/sdk-overview.md:92-100`, `docs/plugins/sdk-overview.md:228-243`, `src/plugins/hook-types.ts:758-765`, `src/plugins/hooks.ts:1038-1045`). OpenClaw also already documents external cost-tracking via LiteLLM rather than a built-in core implementation (`docs/providers/litellm.md:9-18`). So this looks better scoped as a ClawHub plugin that imports burn0 at startup, rather than another optional core dependency and runtime patch in `main`.
