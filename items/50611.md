## Summary

Current main does not yet fix #50611. The memory flush plan still forwards a configured `reserveTokensFloor` directly, and the shared gate returns no runnable threshold when `contextWindowTokens - reserveTokensFloor - softThresholdTokens <= 0`, so `reserveTokensFloor == contextWindowTokens` still prevents the token-based flush path. Existing tests cover normal threshold behavior but not the equality/over-context edge case. The linked PRs #50694 and #51316 are still open in the provided context, so this issue should remain open as the canonical bug until a fix lands.

## Best Possible Solution

Keep #50611 open and land a focused fix that clamps the effective memory-flush reserve/threshold before both preflight compaction and memory flush gating, ideally where the plan is built or where the shared gate normalizes inputs. Add regression tests for `reserveTokensFloor == contextWindowTokens` and `reserveTokensFloor > contextWindowTokens`, and reconcile/close the duplicate implementation PRs after the canonical fix is merged.

With `reserveTokensFloor == contextWindowTokens` and default `softThresholdTokens`, this path cannot trigger the token-based flush.
