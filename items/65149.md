## Summary

Keep open. Current main still does not implement `final_reply_payloads` or an equivalent final-reply hook with resolved provider/model context, and the existing `message_sending` hook still lacks model/provider identity. The PR targets a real core plugin API gap rather than ClawHub-only work, and related issue #69110 is relevant but does not clearly supersede this concrete PR.

## Best Possible Solution

Keep the PR open for maintainer/API review. The best path is a complete core plugin API change that gives plugins a stable, truthful final-delivery model/provider context, with hook type/runner wiring, SDK exports as needed, docs, and regression tests that cover final reply assembly and usage footer ordering. Coordinate the final shape with open issue #69110 so the delivered API addresses the model-identity trust boundary rather than only one call site.
