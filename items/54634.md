## Summary

Keep open. Current main has meaningful partial mitigations for config clobbering and last-known-good recovery, but I could not verify a complete fix for the specific HOME-change path reported in #54634. The update refresh still reruns `gateway install --force` under the current process environment, the daemon install plan writes `HOME` from that invocation environment, and existing service `HOME` / `OPENCLAW_*` values are explicitly not preserved when regenerating the unit. The discussion comment identifying root/HOME switching as the likely cause still matches the current code boundary.

## Best Possible Solution

Keep this issue open and fix the service refresh/install path directly. Before regenerating a loaded service, preserve or validate the existing service's effective config root (`HOME`, `OPENCLAW_HOME`, `OPENCLAW_STATE_DIR`, `OPENCLAW_CONFIG_PATH`), warn or block when it would change unexpectedly, and add a regression test for updating a service installed under `/home/openclaw` while the refresh command runs with `HOME=/root`. If maintainers want migration behavior, it should be explicit and audited rather than silently choosing a fresh root config.
