## Summary

Current `main` does not implement `extraBody` pass-through for the bundled OpenAI TTS provider. The issue body is partially outdated because TTS provider config now accepts unknown provider-specific keys, but the OpenAI TTS request builder still sends a fixed JSON payload with no merge hook for custom body fields. This remains a concrete, scoped feature request for core TTS behavior, not a ClawHub-only item.

## Evidence

- **TTS schema now accepts arbitrary provider-specific keys:** `TtsProviderConfigSchema` uses `.catchall(...)`, so unknown keys under `messages.tts.providers.<id>` are accepted by current config validation rather than rejected outright.
  - file: [src/config/zod-schema.core.ts:486](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/config/zod-schema.core.ts#L486)

  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **OpenAI TTS request body has no extra-body merge:** The bundled OpenAI TTS implementation serializes only `model`, `input`, `voice`, `response_format`, optional `speed`, and optional `instructions` before POSTing to `/audio/speech`. There is no `extraBody`/`extra_body` handling in this path.
  - file: [extensions/openai/tts.ts:97](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/extensions/openai/tts.ts#L97)

The docs do not document any custom request-body passthrough for this TTS surface.
