## Summary

Keep #39900 open. Current main still does not implement `extraBody` / `extra_body` passthrough for the bundled OpenAI-compatible TTS path. The original strict-schema complaint is partly stale because provider TTS config now accepts plugin-owned fields, but those fields are not normalized or merged into the `/audio/speech` request body.

## Best Possible Solution

Implement a narrow provider-owned passthrough in the bundled OpenAI speech provider: accept `extraBody` and `extra_body` as plain JSON objects, preserve them during config normalization, merge them into the `/audio/speech` payload with documented precedence, add focused tests under `extensions/openai`, and document the field on the OpenAI/TTS provider docs. If maintainers want custom OpenAI-compatible TTS transports to be plugin-first instead, this issue should stay open until that explicit product decision or a canonical replacement item exists.

## Risks / Open Questions

- Implementation needs an explicit collision policy for generated OpenAI speech keys such as `model`, `input`, `voice`, `response_format`, `speed`, and `instructions`.
- Tests should cover both official OpenAI TTS behavior and custom `baseUrl` behavior so arbitrary passthrough does not break standard requests.
