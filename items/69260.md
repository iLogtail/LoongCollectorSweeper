## Summary

Keep #69260 open. The report is a concrete, recent hardening request with staged A/B evidence, and current main only strips provider auth env for the OpenClaw ACP client spawning the default OpenClaw bridge. The bundled acpx/Gemini ACP harness path still has no first-class per-agent auth contract, no Gemini-specific ambient env scrub policy, no fail-closed auth preflight hook, and docs still describe vendor auth as host-provided rather than documenting mixed OAuth/API-key drift risk.

## Best Possible Solution

Keep the issue open for maintainer/API design. A good implementation would add a typed ACP auth-contract surface, probably through acp runtime options and the acpx plugin config, with per-agent expected auth mode, fallback allowance, env scrub keys, and a pre-spawn preflight result. Then add targeted Gemini coverage that strips GEMINI_API_KEY, GOOGLE_API_KEY, GOOGLE_GENAI_API_KEY, GOOGLE_GENAI_USE_VERTEXAI, and GOOGLE_GENAI_USE_GCA when the route is pinned to Gemini CLI OAuth, plus docs for mixed OAuth/API-key hosts.

- A narrow Gemini-only command wrapper could reduce the immediate incident class but would not solve the generic ACP auth-contract API requested here.
- Auth env scrubbing needs a compatibility plan because some ACP harnesses intentionally require API-key env in non-interactive runs.
