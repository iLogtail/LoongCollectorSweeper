## Summary

Keep open. Current `main` still appears to have the reported exec loop-detection gap: `hashToolOutcome()` only special-cases `process` poll/log calls and otherwise hashes raw `details` plus `text`, while exec results still include volatile fields and volatile running-status text. I did not find evidence in the checked-out `main` that this PR’s exec-specific normalization landed independently.

## Evidence

- **Current loop hashing still falls back to raw details + text for exec:** `hashToolOutcome()` special-cases `process` poll/log at lines 224-252, then falls back to hashing `{ details, text }` at lines 254-258. There is no exec-specific branch on current `main`. That means exec `resultHash` still changes whenever volatile exec metadata changes.
