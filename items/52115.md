## Summary

Keep #52115 open. Current main still has the reported stale-first-search behavior on the default builtin memory backend: the long-lived manager is cached, dirty file changes trigger search-time sync asynchronously, and the search immediately queries existing SQLite/FTS rows. QMD has a different awaited dirty-search path, but QMD is opt-in; the builtin backend remains the default and fallback path, so the central user-visible problem is not fully implemented on main.

## Best Possible Solution

Keep the issue open and fix the default builtin memory manager path. The likely maintainer path is to make search-time dirty sync deterministic for builtin memory, or add an explicit freshness mode with a bounded await before querying, then add regression tests for a long-lived manager where a new memory file appears on the first post-change `memory_search` and a deleted file no longer appears as a ghost path. The existing QMD awaited-sync behavior should be preserved but should not be treated as closing the builtin default/fallback behavior.

## Risks / Open Questions

- A fix that simply awaits every search-triggered sync could add memory_search latency or surface embedding-provider/indexing failures on tool calls; maintainers should define the intended freshness/latency contract and add bounded regression coverage.
