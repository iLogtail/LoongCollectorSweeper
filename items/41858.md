## Summary

Keep open. Current main partially narrows the skills watcher, but the active memory watcher still recursively watches the workspace memory directory and extraPath globs without a chokidar depth limit or explicit followSymlinks:false, so the FD-exhaustion class reported in #41606 is not clearly fixed on main. The PR is stale against the current memory-core path, but it still contains useful focused work and a follow-up commit that addressed the initial depth:3 regression by raising the memory limit to depth:10.

## Best Possible Solution

Keep the item open for maintainer review or manual consolidation. The best fix is a clean current-main patch against extensions/memory-core/src/memory/manager-sync-ops.ts that bounds memory watcher resource use, handles symlinks and broad extraPaths deliberately, preserves or documents nested memory refresh semantics, and adds watcher-option tests. The skills side should be verified with a chokidar-v5-compatible narrow watch strategy for SKILL.md changes.

## Risks / Open Questions

- The PR no longer applies directly to current main because the memory implementation now lives under extensions/memory-core/src/memory/manager-sync-ops.ts, so it needs a rebase or a clean follow-up PR.
- A naive low depth cap can regress nested memory/extraPaths refresh behavior; the fix should preserve or explicitly document the recursive indexing contract.
- The current skills watcher uses SKILL.md-shaped targets, but chokidar v5 removed broad glob support, so the final skills-side fix should be verified against chokidar v5 rather than relying only on mocked tests.
