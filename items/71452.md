## Summary

Keep #71452 open. Current main still contains the generic 25-row cap in the human-readable message formatter, and the affected `read`, `list-pins`, and `search` CLI text paths still route through that capped formatter. Provider-side pagination/limits are partially implemented in some plugins, including Teams pins/channel-list and Discord/Slack/Matrix reads, but that does not resolve the central user-visible formatter limit or provide a complete pagination contract for all listed actions.

## Best Possible Solution

Keep this issue open and implement a narrow message-format/action-contract follow-up: pass the requested limit and any pagination metadata into `renderMessageList`, remove or make configurable the hardcoded 25-row display cap, pass `limit` through list-pins where providers support it, and add regression tests for more than 25 `read`, `list-pins`, and `search` rows while preserving existing provider pagination such as Teams `@odata.nextLink`.
