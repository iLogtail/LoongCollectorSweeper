## Summary

Keep open. Current main still does not solve the central audit false-positive: bare `$VAR` strings are not recognized as SecretRefs, and auth-profile string values are still reported as `PLAINTEXT_FOUND` even after a ref is detected. The structured SecretRef apply/scrub path is now documented and tested, but that only partially addresses the broader report. The linked #54141 is an open, unmerged PR for part of this issue, not a shipped fix.

## Best Possible Solution

Keep #53998 open, or split it into narrower tracked work: land a tested audit fix for bare `$VAR` and auth-profile inline `${VAR}`/SecretRef strings, make configure/apply handle out-of-scope OAuth profiles with a skip or clear non-fatal warning where appropriate, and preserve the already-shipped structured `keyRef`/`tokenRef` scrub path.

## Evidence

- **Bare env refs are not parsed as SecretRefs:** `ENV_SECRET_TEMPLATE_RE` only matches `${VAR}` form, and `parseEnvTemplateSecretRef()` returns null when that regex does not match. A bare `$OPENAI_API_KEY` / `$OPENCLAW_GATEWAY_TOKEN` value therefore falls through as a normal string on current main.
