## Summary

Keep #39703 open. Current main still has FIFO cron schedule cache behavior: cache hits return without promotion, and capacity eviction deletes the first Map key. The PR remains a focused implementation of the missing LRU promotion and includes the key-level regression coverage requested in review.

## Best Possible Solution

Keep #39703 open as the canonical remaining cron cache LRU fix, or land an equivalent maintainer-owned patch that promotes cache hits with delete/re-set and includes key-level regression coverage proving the promoted cron expression survives eviction while the next-oldest entry is evicted.

## Evidence

- **Current main cache hits do not promote:** resolveCachedCron() builds the timezone/expression key, reads cronEvalCache.get(key), and immediately returns the cached Cron object on hit. There is no delete/re-set step to move the entry to the end of Map iteration order.

  - sha: [4a3030df9efa](https://github.com/openclaw/openclaw/commit/4a3030df9efa72c44bab567a4390ac0c4876a73f)

  - file: [src/cron/schedule.ts:23](https://github.com/openclaw/openclaw/blob/4a3030df9efa72c44bab567a4390ac0c4876a73f/src/cron/schedule.ts#L23)
