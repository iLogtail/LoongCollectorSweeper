## Summary

Keep open. Current main has a partial mitigation for `x-openclaw-sender-is-owner` spoofing, but the broader issue is not fully implemented: loopback MCP still has shared owner/non-owner server tokens, still sends session/account/channel scope as mutable MCP headers, and still resolves those scope fields from request headers. The linked fix PR #67376 remains open, and the issue discussion includes maintainer triage explicitly keeping this open as a real high-severity auth-boundary issue.

## Best Possible Solution

Keep #64993 open until the authoritative loopback MCP scope binding lands. The best path is to finish or replace PR #67376 with a main-branch change that registers per-backend bearer tokens to immutable `{sessionKey, accountId, messageChannel, senderIsOwner}` scope, removes the scope headers/env flow from bundled MCP client config, derives all loopback tool scope from token identity server-side, invalidates scoped tool cache on token revocation, and keeps regression tests for spoofed headers and token isolation.
