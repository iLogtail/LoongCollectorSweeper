## Summary

Keep #67158 open. Current main has newer Codex OAuth defaults/recovery and a narrow stale-model suppression for `gpt-5.3-codex-spark`, but I did not find an implemented fix for the reported ChatGPT/Codex OAuth entitlement rejection of `gpt-5.1`/`gpt-5.2`/`gpt-5.3`. The latest issue discussion also explicitly leaves this open as a separate provider-boundary track.

## Best Possible Solution

Keep this open as an OpenAI Codex provider-boundary investigation. The best path is to collect current-main catalog/status output for an affected ChatGPT/Codex OAuth account, then either suppress/deprioritize no-longer-entitled older Codex rows and fallbacks or classify the upstream unsupported-model response into a clear model-entitlement/failover path. Align docs and live-test target examples if `openai-codex/gpt-5.2`/older Codex ids are no longer valid for subscription OAuth.
