## Summary

Keep #62629 open. Current main still treats `agents.defaults.models` keys as canonical provider/model refs, preserves colon-bearing model ids, looks up params by exact `provider/model` key, and sends the resolved `model.id` to provider transports. Related prefix-normalization work like #59188 does not cover arbitrary variant suffixes, and blanket `:` stripping would conflict with valid model ids such as OpenRouter `:free` and Ollama tags.

## Best Possible Solution

Keep open. Implement an explicit model-variant/profile contract instead of arbitrary `:` stripping: let an `agents.defaults.models` entry carry a canonical target such as `modelId`/`target` while retaining its unique key, alias, and params; update model selection, allowlist/catalog display, `/model`, WebChat, extra-param resolution, and provider transports so the variant key selects params but the canonical target is sent to the provider. Add regression tests for duplicate variants plus colon-bearing real model ids.

## Risks / Open Questions

- A naive delimiter strip would break valid provider model ids that contain colons, including OpenRouter free-model suffixes and local/Ollama-style tags.
- The right fix likely needs a small config/API design decision, not just transport-side string trimming, because params must continue to attach to the selected variant while provider calls use the canonical model id.
