## Summary

Keep #71794 open. Current main fixed the config-schema/CLI-set portion of the report, but it still intentionally blocks `agent_end` and other conversation hooks for non-bundled plugins unless `plugins.entries.<id>.hooks.allowConversationAccess=true` is explicitly set. The remaining upgrade/default/visibility concern is still actionable in OpenClaw core because an existing `plugins.allow` entry alone does not preserve auto-capture for external plugins such as `openclaw-mem0`, and current plugin doctor only surfaces error-level diagnostics, not the warn-level blocked-hook diagnostics.

## Best Possible Solution

Keep the issue open for a narrow core follow-up: preserve the explicit `allowConversationAccess` security gate, but add a user-visible upgrade/doctor/plugins-diagnostics path for enabled non-bundled plugins that register conversation hooks while lacking the opt-in, and document the exact `openclaw config set plugins.entries.<id>.hooks.allowConversationAccess true` remediation. Any default-to-allow behavior for `plugins.allow` should require explicit maintainer security judgment.

## Risks / Open Questions

- Changing the default from deny to allow for non-bundled conversation hooks would weaken the current security gate; a safer maintainer path is likely explicit migration guidance, install/update warnings, or targeted diagnostics rather than silently trusting all allowed plugins.
- Because `openclaw-mem0` is external, core should avoid plugin-specific policy unless there is a generic migration/diagnostic contract for non-bundled conversation hooks.
