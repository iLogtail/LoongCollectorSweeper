## Summary

Keep open. Current `main` still lacks the reported local drag-and-drop workflow: the Control UI only accepts image attachments, converts dropped/selected files into in-memory data URLs, previews them as images, and sends them as `type: "image"` payloads. There is no current path/reference insertion flow for dropped local files, so this is not closable as implemented or non-reproducible.

## Evidence

- **Attachment types are image-only:** `CHAT_ATTACHMENT_ACCEPT` is `"image/*"`, and supported chat attachment MIME types are restricted to strings starting with `image/`.
  - file: [ui/src/ui/chat/attachment-support.ts:1](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/ui/src/ui/chat/attachment-support.ts#L1)

- **Drop/select handlers ignore non-images and build image previews:** Both file selection and drag-drop iterate dropped files, skip anything failing `isSupportedChatAttachmentMimeType(file.type)`, read accepted files with `FileReader`, and render previews with `<img src=...>`.
  - file: [ui/src/ui/views/chat.ts:242](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/ui/src/ui/views/chat.ts#L242)

- **Composer UI advertises image attachments, not path insertion:** The chat surface wires `@drop` to the image-only handler and the hidden file input uses `accept=${CHAT_ATTACHMENT_ACCEPT}`. The queued-message label also renders attachments as `Image (N)`, which matches image-only behavior.
