## Summary

Keep open. Current `main` still treats Windows drive-letter `sessionFile` values as relative on POSIX, so the reported bug is still real, and the submitted PR needs rework because its basename-only rewrite would sidestep the existing cross-agent/session-root resolution behavior and lacks matching test coverage.

## Evidence

- **Current resolver still misses Windows absolute paths on POSIX:** `resolvePathWithinSessionsDir` only normalizes absolute inputs when `path.isAbsolute(trimmed)` is true, then falls through to `path.resolve(realBase, normalized)`. On POSIX, a value like `C:\Users\...\sess-1.jsonl` is not considered absolute, so it is appended under the current sessions directory instead of being recognized as a foreign absolute path.
  - file: [src/config/sessions/paths.ts:181](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/config/sessions/paths.ts#L181)
  - command: `nl -ba src/config/sessions/paths.ts | sed -n '172,233p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Reasonable reproduction still fails on current main:** A direct Node reproduction against the current logic resolves `C:\Users\foo\.openclaw\agents\bot2\sessions\sess-1.jsonl` to `/home/node/.openclaw/agents/bot1/sessions/C:\Users\foo\.openclaw\agents\bot2\sessions\sess-1.jsonl`, matching the PR description's broken concatenation behavior.
  - command: `node -e 'const path=require("node:path"); const sessionsDir="/home/node/.openclaw/agents/bot1/sessions"; const candidate="C:\\Users\\foo\\.openclaw\\agents\\bot2\\sessions\\sess-1.jsonl"; const trimmed=candidate.trim(); const realBase=path.resolve(sessionsDir); const realTrimmed=path.isAbsolute(trimmed)?trimmed:trimmed; const normalized=path.isAbsolute(realTrimmed)?path.relative(realBase, realTrimmed):realTrimmed; console.log(JSON.stringify({isAbsolute:path.isAbsolute(trimmed),normalized,resolved:path.resolve(realBase, normalized)}));'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Existing tests require preserving agent/session-root context:** Current tests assert that cross-agent absolute `sessionFile` paths resolve back to the owning agent transcript, and that unsafe or structurally invalid paths fall back safely. A basename-only rewrite to `<current sessionsDir>/<file>` would not preserve that behavior for Windows-authored paths that encode another agent root.
  - file: [src/config/sessions.test.ts:607](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/config/sessions.test.ts#L607)

- **PR discussion already surfaced unresolved correctness gaps:** The provided review thread on PR #50116 called out missing Windows-path tests and a P1 regression: rewriting `C:\...\agents\bot2\sessions\sess-1.jsonl` to just `sess-1.jsonl` can alias another agent's live transcript into the current agent store. That matches the current test contract and means the proposed patch is not ready to auto-close as complete.
  - file: [src/config/sessions/paths.ts:172](https://github.com/openclaw/openclaw/blob/c6075f5b088732d8a4a0bd3f6536fc08d4293528/src/config/sessions/paths.ts#L172)

## Risks / Open Questions

- Windows-authored `sessions.json` entries mounted on POSIX still resolve to broken transcript paths on current `main`.
- Merging the PR as-is risks collapsing distinct agent transcript paths onto the current agent's basename and creating silent session aliasing.
- There is still no targeted test coverage for `C:\...` or `C:/...` `sessionFile` inputs in `src/config/sessions.test.ts`.
