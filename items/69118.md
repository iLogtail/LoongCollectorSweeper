## Summary

Keep #69118 open. Current main contains a partial fix for volatile inbound metadata, but the specific groupIntro drift reported here is still present: groupIntro is first-turn-only and is still included in the static extra-system-prompt hash used for CLI session reuse, while resolveCliSessionReuse still invalidates on any hash mismatch.

## Best Possible Solution

Keep the issue open and fix the core session identity boundary: continue injecting groupIntro into the actual prompt on first group turns, but exclude first-turn-only groupIntro from the stable CLI session reuse hash/fingerprint, or otherwise split stable session identity from ephemeral prompt guidance. Add a regression that composes a group turn with shouldInjectGroupIntro=true followed by shouldInjectGroupIntro=false and verifies the stored claude-cli binding is reused instead of invalidated with reason=system-prompt.
