## Summary

Keep #69822 open. Current main still lacks the proposed `socket.drain` routing path: session mutation broadcasts only target `sessions.subscribe` event subscribers, `socket.drain` is not registered as a gateway event/scope-guarded broadcast, and the request context does not expose message-subscriber connection IDs to `sessions.delete`/`sessions.reset`. The PR discussion is also substantive: Greptile found no blocking defects and the author addressed the two P2 review comments, while the maintainer comment was about branch workflow rather than rejecting the product change.

## Best Possible Solution

Keep the PR open or replace it with a maintainer-owned focused implementation that registers `socket.drain`, exposes message-subscriber connection IDs to session mutation handlers, routes teardown drains to both event and message subscribers, preserves scope checks, and adds regression tests for `sessions.messages.subscribe` receiving delete/reset drains. If maintainers do not want `socket.drain` as a core gateway contract, that should be an explicit product decision rather than an automated cleanup close.
