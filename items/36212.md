## Summary

Keep open. Current main still has the same queue-continuity gap: Feishu/Lark ingress and queue storage preserve the provider message id, but collect-mode batching synthesizes a followup without carrying `messageId`, the followup runtime does not pass a queued id as `currentMessageId`, and final followup delivery does not provide a current message id to reply-threading. The reporter also reaffirmed production impact on 2026-04-17, and the focused PR #36235 was closed unmerged.

## Best Possible Solution

Keep #36212 open and implement the narrow continuity fix in the queue/followup path: normalize and preserve queued `messageId`, include trusted per-item message_id metadata in collect prompts, carry an appropriate synthetic `FollowupRun.messageId` for summary and collect drains, pass it into `runEmbeddedPiAgent.currentMessageId`, and pass it through followup delivery reply-threading. Add regression coverage for collect prompt metadata, summary/individual drains, followup runner `currentMessageId`, and Feishu/Lark thread-reply behavior where feasible.
