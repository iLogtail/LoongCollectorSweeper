## Summary

Keep open. Current main has native Telegram audio preflight transcription, which may reduce the need for the reporter's local Whisper workaround, but it does not close the reported lock bug. The session write lock remains non-reentrant by default, the embedded attempt path acquires the transcript lock without opting into reentry, and there is no source-level evidence that an external synthetic `bot.handleUpdate()` injection can reuse or wait on the live gateway's held lock.

## Best Possible Solution

Keep this open and fix it in the supported inbound/session boundary rather than closing as stale or implemented. The best path is to add a public, gateway-owned synthetic/user-inbound API or plugin/CLI surface for local transcribers that enqueues by `sessionKey` inside the live gateway process, waits behind any active same-session run, and never deep-imports `dist/extensions/telegram/*`. Add a regression test for an existing Telegram DM session with an active run plus a synthetic/transcribed inbound text message, asserting it queues or steers without `session file locked`. The existing `tools.media.audio` Telegram voice-note preflight path should be documented as the preferred built-in option, but it should not be treated as closing the session-lock bug.

## Evidence
