## Summary

Keep #69790 open. Current main has useful mitigations: Telegram DM voice notes can now be transcribed before dispatch, normal Telegram bot updates are sequentialized, and embedded attempt cleanup releases the session lock. But I did not find source, tests, changelog, or a canonical related item proving that same-session synthetic Telegram inbound now reuses or waits on a live gateway-held session write lock. The current lock contract still contends by default, including same-process acquisitions, unless a caller explicitly opts into `allowReentrant: true`; the embedded run path does not opt in.

## Best Possible Solution

Keep the issue open and treat it as a runtime/API contract question. The likely best fix is a supported Gateway-owned synthetic inbound or session-send API for Telegram that routes through the existing per-session queue and active-run/followup policy, instead of importing built `dist/extensions/telegram/*` and calling `bot.handleUpdate` from a separate process. If maintainers decide external synthetic updates are unsupported, document that boundary and point users at the Telegram audio preflight transcription path or a supported injection API.
