## Summary

Keep open. Current main has adjacent improvements, including Telegram audio preflight transcription and session-lock watchdog cleanup, but I did not find source or tests proving the reported same-session synthetic inbound lock contention is fixed. The embedded runner still acquires the transcript write lock with the default non-reentrant behavior, and the Telegram synthetic-context paths still feed `processMessage` without a lock-reuse or supported inbound-injection API.

## Best Possible Solution

Keep the issue open and narrow it to a maintainer-owned fix or decision: add a supported Telegram inbound injection surface that enters the same session lane as normal gateway inbound, or make the relevant synthetic path explicitly wait/reuse the active session lock with regression coverage for an existing `per-channel-peer` Telegram DM session. If the intended answer is that direct `dist/extensions/telegram/bot-*.js` injection is unsupported, document the supported alternative and close only after the normal-inbound lock claim is disproven or fixed.
