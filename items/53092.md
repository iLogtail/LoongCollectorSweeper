## Summary

Keep open. Current main still does not validate `tools.web.search.provider` against registered web-search provider IDs: the schema accepts any string, and tests explicitly preserve warning plus auto-detect fallback for an unknown configured provider. The manifest contract source exists, so this remains actionable core/plugin-boundary work.

## Best Possible Solution

Keep the issue open. The best path is to add config validation that builds the accepted web-search provider set from manifest `contracts.webSearchProviders` plus bundled providers, rejects or clearly reports unknown active provider IDs, and keeps a deliberate warning-only path for stale missing-plugin configs. Add regression tests covering valid bundled/third-party provider IDs and an invalid typo in `tools.web.search.provider`.

## Risks / Open Questions

- Typos in `tools.web.search.provider` can still validate successfully and change runtime behavior through auto-detect fallback.
- A fix should preserve resilience for stale configs where a previously installed third-party provider/plugin is now missing, likely by warning in that case rather than hard-failing startup.
