## Summary

Keep this PR open. Current main still has the misleading QMD availability path: the shared probe spawns with the configured cwd without validating it first, and doctor still treats every QMD probe failure as a binary-start/install problem. The paired issue was closed as superseded by this open PR, not because the behavior has shipped.

## Best Possible Solution

Keep the PR open and update it against current main: validate the QMD probe cwd before spawning, return a structured QmdBinaryAvailability reason, branch doctor guidance on that reason, and add regression coverage in the current QMD process and doctor test locations. Then review or land the PR as the canonical fix for #63158.
