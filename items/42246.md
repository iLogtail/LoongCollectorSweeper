## Summary

Keep #42246 open. Current main has Telegram retry handling, chunking, and a generic write-ahead delivery queue, but it still delivers Telegram outbound payloads directly and has no configurable digest/aggregation layer. The related open #69165 is complementary transport work for outbound queueing/rate-limit backoff, not a replacement for notification batching.

## Best Possible Solution

Keep the item open and implement it, if accepted, as Telegram-plugin-owned outbound aggregation in `extensions/telegram` or as a narrow generic outbound delivery policy if multiple channels need it. Coordinate with #69165 so digest batching and per-chat rate-limit queueing compose cleanly, and add config schema/docs/tests for opt-in interval, max batch size, shutdown flush, and error bypass behavior.
