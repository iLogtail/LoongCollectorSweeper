## Summary

Keep #49982 open. Current main still rejects the exact reported `chat.send` image attachment path when decoded base64 exceeds 5,000,000 bytes, and the Control UI reads selected/pasted/dropped images as raw data URLs without resizing or recompressing them before sending. Android chat attachments already have a compression helper, but web and shared Apple chat attachments do not solve the requested behavior.

## Best Possible Solution

Keep the issue open. The best path is to add bounded image compression before the 5 MB `chat.send` rejection: client-side for Control UI using a canvas/codec path, shared Apple chat attachment compression via the existing `JPEGTranscoder`, and possibly a gateway fallback that only decodes within a conservative upper bound. Update `chat-attachments.test.ts` to cover oversized-but-compressible images and keep an uncompressible/too-large rejection case.

## Risks / Open Questions

- The current pre-decode gateway size check is a memory-safety guard; any fix should keep bounded decoding and avoid accepting arbitrary large base64 payloads into server memory.
- Android already compresses chat attachments, so the remaining work should avoid regressing that path while adding equivalent behavior for Control UI and Apple shared chat attachments.
