## Summary

Keep open. Current main still does not implement a `tools.exec.shell` override: the typed config, strict runtime schema, config-to-tool plumbing, and exec runtime all lack a shell setting, and Windows gateway exec still resolves to PowerShell through `getShellConfig()`. The related #59998 context tracks the same unimplemented gap, but #49931 is older and has the clearer problem statement plus discussion, so this item should remain open rather than be duplicate-closed in favor of the later issue.

## Best Possible Solution

Keep #49931 open as the canonical core exec feature request. The likely implementation is a typed `tools.exec.shell` contract, possibly with structured executable/args instead of a single string, added to `ExecToolConfig`, zod/generated schema, config docs/help, config-to-tool plumbing, `getShellConfig()`/`runExecProcess()`, and approval/safe-bin tests covering PowerShell, bash-like shells, WSL path/workdir behavior, and validation/failure reporting. Decide explicitly whether node-host exec inherits this setting or gets a separate node-local shell configuration.

## Risks / Open Questions

- Changing the shell affects approval parsing, safeBins trust, PATH handling, command-wrapper detection, and Windows/WSL working-directory semantics, so this needs a core design rather than a plugin-only workaround.
- The scope needs maintainer judgment: gateway exec, node-host exec, sandbox exec, PTY fallback, global config, and per-agent config may not all want the same shell contract.
