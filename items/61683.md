## Summary

Keep open. Current main still treats `plugins.entries.<id>.config` validation failures as invalid global config before channel/plugin startup can continue. The newer last-known-good recovery logic avoids rolling back the whole file for plugin-local invalidity, but it still throws and prevents gateway startup, which is the behavior this issue asks to change.

## Best Possible Solution

Keep this issue open and implement a real plugin-local degradation path: split core config validation from plugin-entry schema validation during gateway startup, keep global config errors fatal, convert isolated `plugins.entries.<id>.config` failures into disabled/degraded plugin records with clear logs/status, and add startup/hot-reload regression tests covering invalid plugin config and the ACPX MCP SecretRef case.
