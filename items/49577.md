## Summary

Current main partially improves channel lifecycle health with transport staleness and channel status timestamps, but it does not solve #49577’s Feishu false-healthy inbound-delivery gap. Feishu’s inbound handler still dispatches events without updating runtime status/activity, channel status only prints inbound/outbound timestamps when some channel has populated them, and readiness explicitly treats stale-socket channels as ready. No provided related item is a canonical tracker for this exact remaining Feishu/gateway observability issue.

## Best Possible Solution

Keep the issue open and narrow the fix around the channel status contract: add or require inbound received and dispatch activity counters/timestamps for channel plugins, wire Feishu to update them on event receipt and before/after dispatch, surface a degraded warning when a running/connected account has no recent inbound/dispatch evidence after a configurable threshold, and add targeted gateway/channel tests for the false-healthy case. Separately verify whether the stale gateway-process-after-restart symptom is already covered by daemon restart-health checks or needs its own narrower tracker.
