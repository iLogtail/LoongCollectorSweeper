## Summary

Keep open. Current `main` has Telegram transport plumbing that knows about inline-query update types, but the Telegram extension does not register or test an `inline_query` handler, does not expose an inline-query config surface, and the docs still describe Telegram as DM/group support rather than inline mode.

## Evidence

- **No inline_query handler on main:** `createTelegramBot` registers `message_reaction`, `callback_query`, `message`, and `channel_post` handlers, but there is no `bot.on("inline_query", ...)` path in the Telegram runtime.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1194](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/telegram/src/bot-handlers.runtime.ts#L1194)
  - command: `rg -n "bot\.on\(|bot\.command\(|bot\.use\(" extensions/telegram/src/bot-handlers.runtime.ts`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Allowed updates include inline_query only at subscription level:** The extension asks Telegram for `inline_query` and `chosen_inline_result` updates, which shows partial transport awareness, but this is not matched by a runtime handler.
  - file: [extensions/telegram/src/allowed-updates.ts:14](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/telegram/src/allowed-updates.ts#L14)
