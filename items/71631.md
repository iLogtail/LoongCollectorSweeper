## Summary

Keep #71631 open. Current main has meaningful pieces of the requested Web UI chat responsiveness, but the issue is broader than those shipped pieces: full click-to-first-visible-response instrumentation, gateway/config waterfall isolation, and more granular config operation responsiveness are still not covered by the inspected source/docs. The related items in the issue body are narrower, not a canonical replacement for this whole request.

## Best Possible Solution

Keep this open as a UX/performance tracker or split it into scoped follow-ups: add client and gateway timing attribution for chat/config RPCs, preserve the existing non-blocking chat flow, make config/status panels load independently where slow calls are known, and cross-link narrower work such as #64004 and #71630 instead of closing this as already implemented.
