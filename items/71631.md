## Summary

Keep open. Current main already has some chat responsiveness work: `chat.send` acks before dispatch, the Web UI adds an optimistic user message and empty stream state immediately, delta events render into the active stream, and session-message reloads are deferred during active runs. The config UI also has coarse loading/saving/applying states. But the requested umbrella is not fully implemented: I found no end-to-end click/RPC/provider/first-visible-token instrumentation, config operations still use broad page/controller busy flags rather than per-control save/test states, and the related open items #64004 and #71630 still cover unresolved Control UI slowness/status work.

## Best Possible Solution

Keep this open as an actionable performance/UX umbrella or split it into focused follow-ups: add end-to-end latency tracing for UI action -> Gateway RPC -> provider/model -> first visible update/final response, extend Control UI config screens with granular pending/test/save states and stale-response guards, and reuse the existing non-blocking/streaming chat path while filling the remaining status and instrumentation gaps.
