## Summary

Keep #51918 open. Current main has adjacent queue controls (`collect`, `steer`, `steer-backlog`, and hard `interrupt`), but it does not implement owner-specific priority, a per-message priority flag, or an `interrupt-soft`/in-turn supplementary-context path. The default `collect` behavior still waits for the current turn to finish, `steer` is a configured mode rather than owner-priority behavior, and current steering is documented as delivery at the next model boundary after current assistant tool calls, not immediate soft preemption at every tool boundary.

## Best Possible Solution

Keep #51918 open as a core agent-loop/queue feature request. The best path is a documented soft-preemption design: add a priority lane or per-message priority metadata tied to trusted owner identity, introduce an explicit mode such as `owner-priority` or `interrupt-soft`, inject at a well-defined safe boundary without discarding pending transcript/tool state, and cover Telegram group owner messages plus ordinary collect/steer/interrupt regressions in tests. Update config types, zod schema/help, queue docs, and channel docs once the maintainer-chosen semantics are settled.

## Risks / Open Questions

- Soft preemption likely needs explicit product semantics for group chats: automatic owner priority can reorder messages from other participants and should be opt-in or carefully scoped.
- Any implementation should preserve existing hard-stop behavior (`/stop`, abort phrases, `interrupt`) while adding a distinct soft path so cancellation and supplementary-context injection do not share ambiguous semantics.
