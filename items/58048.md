## Summary

Current `main` still has the reported Slack duplicate-reaction behavior on the user-facing `react` action, so there is not strong evidence to close this issue.

## Evidence

- **Core Slack reaction helper still calls `reactions.add` without handling duplicate-reaction errors:** `reactSlackMessage` directly awaits `client.reactions.add(...)` and returns no special handling for Slack's `already_reacted` error.
  - file: [extensions/slack/src/actions.ts:95](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/slack/src/actions.ts#L95)
  - command: `nl -ba extensions/slack/src/actions.ts | sed -n '80,135p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **User-facing Slack `react` action still routes straight to that helper:** The action runtime invokes `reactSlackMessage(...)` for `action: "react"` and does not wrap it in an `already_reacted` guard, so a duplicate tool call can still surface the Slack API error.
  - file: [extensions/slack/src/action-runtime.ts:191](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/slack/src/action-runtime.ts#L191)

- **Existing `already_reacted` handling is limited to status-reaction plumbing, not the `react` tool path:** `dispatch.ts` catches `already_reacted` inside the status-reaction adapter, which shows a narrow workaround exists, but that protection does not cover the public Slack `react` action above.
  - file: [extensions/slack/src/monitor/message-handler/dispatch.ts:279](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/slack/src/monitor/message-handler/dispatch.ts#L279)
