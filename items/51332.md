## Summary

Keep open. Current main has useful partial mitigations: Ollama docs now warn that OpenAI-compatible `/v1` mode can emit raw tool JSON as plain text, and assistant-visible text sanitization strips leaked tool-call XML / downgraded tool-call text. But the central request is diagnostics for fake or non-executed tool calls; I found no runtime warning/log/diagnostic that flags assistant text resembling an unexecuted tool invocation, and no docs for the suggested strong-orchestrator/local-text-worker troubleshooting pattern.

## Evidence

- **Issue discussion asks for core diagnostics, not just UI cleanup:** The issue asks OpenClaw to distinguish real tool execution from assistant text that merely resembles a tool call. A later comment specifically says a clear warning in logs would save debugging time. The Keep My Claw comment is an external audit/restore workaround, not an implementation in OpenClaw core.
- **Ollama docs partially address provider reliability:** Current docs warn that remote Ollama users should not use `/v1` because it breaks tool calling and models may output raw tool JSON as plain text.
