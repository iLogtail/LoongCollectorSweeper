## Summary

Keep open. Current main still performs auto-TTS inline before reply payload delivery, with the speech provider call awaited synchronously. The TTS config/schema/docs expose `auto`, `mode`, provider, persona, limits, and timeout settings, but no async/background delivery mode or concurrency control. Related #42539 overlaps Telegram TTS delivery UX, but it is narrower and remains open; it does not supersede this queue/non-blocking request.

## Best Possible Solution

Keep the issue open and implement a core/bundled speech-core delivery mode that sends visible text immediately, schedules TTS audio as background follow-up work outside the active reply run, uses a bounded per-session/provider concurrency queue, and adds config/schema/docs plus regression tests for final replies, streamed block replies, and Telegram voice-note delivery.
