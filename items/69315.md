## Summary

Keep open. Current main still awaits TTS synthesis before reply payload delivery, and Telegram message handling is sequentialized per chat/topic, so slow local TTS remains on the inbound turn's critical path. I found no shipped `messages.tts.delivery` or equivalent async/background delivery mode.

## Best Possible Solution

Keep the issue open and implement this as a core reply-delivery/TTS scheduling feature, not a provider-local patch. Add a documented `messages.tts.delivery` or equivalent generic mode, default to current synchronous behavior, and add a background/concurrency-limited TTS queue that lets visible text delivery and Telegram's inbound sequential lane complete before follow-up voice delivery. Cover it with targeted dispatch and Telegram tests using a slow fake TTS provider.
