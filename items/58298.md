## Summary

Keep open. Current `main` has a shared `presentation` path for Feishu rich sends, but it does not implement agent-sent interactive card buttons/selects or expose a send-time Feishu `card` payload that would let agents use the existing `ocf1` callback routing.

## Evidence

- **Shared message tool send schema has `presentation`, not a Feishu `card` send field:** The send schema adds `presentation` when the channel advertises that capability, but there is no send-time `card` property in the shared message tool schema.
  - file: [src/agents/tools/message-tool.ts:90](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/agents/tools/message-tool.ts#L90)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Docs explicitly steer new message-tool UI away from provider-native `card` fields:** The message-presentation docs say not to add provider-native fields such as Feishu `card` to the shared message tool, and describe button values as routed through the channel interaction path only when clickable controls are actually supported.
  - file: [docs/plugins/message-presentation.md:24](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/docs/plugins/message-presentation.md#L24)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Feishu send path renders presentation as a markdown-only card body:** `buildFeishuPresentationCard()` builds a Feishu card with a header plus a single markdown element derived from fallback text. The send handler uses that helper for `presentation` sends, so button/select payloads are flattened before delivery.
  - file: [extensions/feishu/src/channel.ts:124](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/feishu/src/channel.ts#L124)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Tests confirm button/select content degrades to plain markdown labels:** Feishu channel tests expect button and select presentations to become markdown like `- Run help` and `Pick one:\n- Option A`, not native clickable controls.
  - file: [extensions/feishu/src/channel.test.ts:357](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/feishu/src/channel.test.ts#L357)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **`ocf1` callback routing exists, but current usage is limited to built-in approval/launcher cards:** Feishu card-action handling routes structured `ocf1` envelopes for approval and quick-action flows, and the built-in launcher constructs native Feishu `action` buttons with those envelopes. That plumbing is not wired into the generic message tool send path above.
  - file: [extensions/feishu/src/card-action.ts:325](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/feishu/src/card-action.ts#L325)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)

## Risks / Open Questions

- Docs currently describe Feishu presentation as mapping to interactive cards, but the tested implementation only emits markdown fallback for buttons/selects. Closing this as implemented would likely mis-triage a real gap.
- There is a low-level Feishu `card` object path on `edit`, but not on the shared send schema or the normal agent message-tool send flow, so it does not satisfy the reported use case.
