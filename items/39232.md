## Summary

Keep #39232 open. Current main still lacks a universal observe-only pre-filter raw inbound plugin hook: the typed hook catalog has no `on_raw_inbound`, `onRawInbound`, or `before_channel_filter`; unknown typed hooks are ignored; and WhatsApp still applies access control and group mention gating before the existing opt-in `message_received` hook path. The related PR #39240 attempted a WhatsApp-specific implementation but was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep the issue open for maintainer/API design. A good implementation would add an observe-only, privacy-aware pre-filter inbound contract to the typed plugin/channel SDK, wire participating channel plugins before access control and mention gating only behind explicit opt-in, document the privacy model in public plugin/channel docs, and add contract plus channel-specific ordering tests. If maintainers decide raw ambient observation is unsupported, document that policy and point users to narrower post-gating hooks such as opt-in `message_received`.

## Risks / Open Questions

- A pre-filter raw inbound observer would expose message content, sender identifiers, and group/channel identifiers that current access control and mention gating can suppress, so any implementation needs explicit opt-in, privacy documentation, and security review.
- The product shape remains a maintainer/API decision: a universal typed hook, a channel-scoped observe sink, per-channel config, or a documented policy that ambient raw observation is out of scope.
