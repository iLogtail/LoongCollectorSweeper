## Summary

Keep #39232 open. Current main still does not expose an `on_raw_inbound`, `before_channel_filter`, or equivalent pre-filter passive observation hook, and the existing `message_received`, `inbound_claim`, and `before_dispatch` hooks all sit after channel access/routing or dispatch decisions. The related WhatsApp PR #39240 was a useful concrete attempt, but it closed unmerged and did not land the requested core/plugin API surface.

## Best Possible Solution

Keep #39232 open as the canonical API/design request unless a maintainer chooses a different product direction. The best implementation would be an additive, typed plugin/channel observation seam that fires before allowlist, sender policy, and mention gating, with explicit privacy/security opt-in, channel coverage guidance, docs, and regression tests. PR #39240 can inform the WhatsApp-specific design, but it is closed unmerged and does not supersede this issue.
