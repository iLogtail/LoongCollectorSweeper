## Summary

Keep open. Current main improves observability around model fallback through subsystem logs, runtime lifecycle events, verbose notices, and default-on trajectory sidecars, but I did not find a persisted session-transcript custom entry that records the primary failure before the `model-snapshot` record requested by #64902.

## Best Possible Solution

Keep the issue open and implement a narrow persisted audit entry in the model fallback path, likely a `custom` session entry such as `fallback-trigger` or `model-fallback-decision` written before the fallback model snapshot. Reuse the existing `FallbackAttempt`/`logModelFallbackDecision` fields so the transcript, runtime logs, lifecycle events, and trajectory sidecar agree on provider, model, reason, status/code, sanitized error preview/hash, and selected fallback target. Add focused tests that simulate primary failure plus fallback success and assert the session JSONL ordering: failed primary audit entry, then `model-snapshot`, then successful assistant response.
