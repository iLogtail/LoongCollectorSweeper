## Summary

Keep open. Current `main` still silently skips recoverable Telegram media-group download failures and continues processing the remaining images without any user-visible warning, so the core bug described in #55216 is still present.

## Evidence

- **Media-group handler still skips failed photos silently:** `processMediaGroup` catches recoverable errors from `resolveMedia`, logs `media group: skipping photo that failed to fetch`, and `continue`s. It also only pushes truthy `media` results, so `null` returns are dropped silently before calling `processMessage` with the reduced `allMedia` list.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:389](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/telegram/src/bot-handlers.runtime.ts#L389)
  - command: `nl -ba extensions/telegram/src/bot-handlers.runtime.ts | sed -n '360,455p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Single-message media failures do notify the user:** Outside media groups, the handler catches media download failures and sends `⚠️ Failed to download media. Please try again.` back to the chat. That notification path is not used by `processMediaGroup`, confirming the gap is specific to albums/media groups.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1111](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/telegram/src/bot-handlers.runtime.ts#L1111)

- **`resolveMedia` can return `null` after file resolution fails:** `resolveMedia` returns `null` when `resolveTelegramFileWithRetry` yields no file, and the media-group loop has no user-visible handling for that case.

There is no assertion for a warning about the missing image.
