## Summary

Keep #72114 open. Current main has adjacent bundled runtime-deps hardening, external stage-dir docs, lock cleanup, and update restart deferral for active agent work, but I still do not see a source-backed fix for the specific cold-cache bundled runtime-deps install plus update/service-manager shutdown race. The issue is concrete, recent, external-authored, unprotected, and the related #72058/#72084 context does not clearly supersede the narrower SIGTERM/SIGKILL runtime-deps failure mode.

## Best Possible Solution

Keep #72114 open as a focused runtime-deps/update restart-race report. The likely core fix is to make bundled runtime-deps staging observable as pending restart/shutdown work, avoid service-manager or SIGUSR1 restarts while an npm staging subprocess is active, harden interrupted staging cleanup/retry, and add regression coverage for cold-cache browser runtime-deps staging during an update-driven gateway restart. If maintainers want #72058 to own this, explicitly mark #72114 as a duplicate only after that tracker names the shutdown/SIGKILL failure mode.
