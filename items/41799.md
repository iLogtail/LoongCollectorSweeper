## Summary

Current `main` does not already implement this compatibility fix. The outbound normalization path still only accepts string results from `messaging.normalizeTarget`, so legacy `{ ok, to }` returns are not explicitly tolerated, and current tests on `main` do not cover that case. The PR still appears useful, though it will need rebasing because the old monolithic `message-action-runner.test.ts` file no longer exists on `main`.

## Evidence

- **No legacy-object compat in current normalization:** `normalizeTargetForProvider()` still does a string-only coercion via `normalizeOptionalString(normalizer?.(raw) ?? fallback)`. If a legacy plugin returns an object, there is no runtime branch that extracts `.to`.
  - file: [src/infra/outbound/target-normalization.ts:55](https://github.com/openclaw/openclaw/blob/bae7b54a85b17d180e4bfc28973a7e4fd1646a93/src/infra/outbound/target-normalization.ts#L55)
  - command: `nl -ba src/infra/outbound/target-normalization.ts | sed -n '45,56p'`
  - sha: [bae7b54a85b1](https://github.com/openclaw/openclaw/commit/bae7b54a85b17d180e4bfc28973a7e4fd1646a93)
- **Current resolver falls back to raw input, not canonicalized target:** `resolveMessagingTarget()` uses `resolveNormalizedTargetInput()`, and when normalization does not yield a string it falls back to the raw trimmed input. That means current `main` would not preserve a legacy normalizer's canonical `to` string.
  - file: [src/infra/outbound/target-resolver.ts:355](https://github.com/openclaw/openclaw/blob/bae7b54a85b17d180e4bfc28973a7e4fd1646a93/src/infra/outbound/target-resolver.ts#L355)
