## Summary

Current main has added per-session identity plumbing, but the requested fallback order is not fully implemented: `ui.assistant` still takes priority over `agents.list[].identity` for Control UI name/avatar resolution when the global assistant fields are set.

## Evidence

- **Control UI identity RPC is session-aware:** The browser asks `agent.identity.get` using the current session key and updates `assistantName`, `assistantAvatar`, and related metadata from the response.
  - file: [ui/src/ui/controllers/assistant-identity.ts:31](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/ui/src/ui/controllers/assistant-identity.ts#L31)
  - command: `nl -ba ui/src/ui/controllers/assistant-identity.ts | sed -n '1,120p'`
  - sha: [e640c0a95fcb](https://github.com/openclaw/openclaw/commit/e640c0a95fcbd06a1311f18fc635518d011d45da)
- **Session switch refreshes identity and avatar:** Switching chat sessions resets state, reloads assistant identity, and refreshes the avatar endpoint, so current main does address part of the requested per-session update behavior.
  - file: [ui/src/ui/app-render.helpers.ts:531](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/ui/src/ui/app-render.helpers.ts#L531)

## Risks / Open Questions

- Closing this as implemented would lose the remaining requested behavior for users who already have global `ui.assistant.name` or `ui.assistant.avatar` configured and expect per-agent identity to override it.
