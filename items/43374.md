## Summary

Keep #43374 open. Current main has partial provider-failover and transport hardening, but the specific multi-agent Telegram/LLM concurrency report is not fully resolved: Telegram runner concurrency is still coupled to agent maxConcurrent, LLM/provider requests do not globally apply the requested IPv4-first/DNS policy, lock watchdog defaults remain internal constants, and the existing tests cover fallback behavior but not the reported simultaneous multi-agent timeout scenario.

## Best Possible Solution

Keep this open as a core reliability issue. The best path is to reproduce or simulate the concurrent Telegram-sourced agent workload, then split the fix into scoped changes: independent Telegram runner/polling concurrency, explicit provider-request network policy for LLM calls, and documented/configurable timeout or lock behavior. Add regression coverage for multiple concurrent Telegram-triggered runs with fallbacks and low `timeoutSeconds`.
