## Summary

Keep #8508 open. Current main still supports a single fixed ack reaction value, resolved from account/channel/global/identity fallback, with WhatsApp using a single `ackReaction.emoji` object. There is no `ackReactions` array, contextual selector, or model-driven per-message ack choice, and the existing maintainer/Codex issue comment already reached the same keep-open conclusion.

## Best Possible Solution

Keep #8508 open for a backwards-compatible core/channel behavior design. Preserve existing `ackReaction` string semantics, add a clear configured array or selector contract such as `messages.ackReactions`, decide whether inbound message context is available for selection, and update exported config types, zod schema, generated metadata/help, docs, resolver tests, and focused bundled-channel tests.
