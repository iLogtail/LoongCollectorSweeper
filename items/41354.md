## Summary

Current `main` contains a stale-socket mitigation for quiet channels, but it does not appear to implement the reconnect-grace behavior needed to close this Discord restart-loop report. `src/gateway/channel-health-policy.ts:112` still treats `connected === false` as immediately unhealthy once the startup grace expires, and the only grace window in that policy is tied to `lastStartAt` (`src/gateway/channel-health-policy.ts:106`). I found no reconnect-specific grace keyed off disconnect/reconnect lifecycle state. The shipped v2026.4.22 changelog entry at `CHANGELOG.md:257` addresses false stale-socket restarts from idle channels, which matches part of the follow-up comments, but not the original/disconnected half of the report.

## Evidence

- **Disconnected state still restarts immediately after startup grace:** `evaluateChannelHealth` returns `{ healthy: false, reason: "disconnected" }` whenever `snapshot.connected === false`, with no reconnect-specific grace beyond the earlier `lastStartAt` startup window.
