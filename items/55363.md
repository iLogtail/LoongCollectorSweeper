## Summary

Keep open. Current main still documents `--force-document` as Telegram-only, but the CLI accepts the flag and the WhatsApp core/gateway send path still drops it before the gateway request and WhatsApp adapter payload. The related filename/document issues (#5781/#10862) and HTML-specific issue (#53652) do not cover this broader transport gap.

## Best Possible Solution

Keep the issue open and fix the contract explicitly. The narrowest maintainer-safe path is to reject `--force-document`/`asDocument` when `--channel whatsapp` is used, with CLI/tool docs and tests. If maintainers want WhatsApp support instead, thread a typed `forceDocument`/`asDocument` option through `src/infra/outbound/message.ts`, `src/gateway/protocol/schema/agent.ts`, `src/gateway/server-methods/send.ts`, and the WhatsApp outbound/send API, then force media into a Baileys `document` payload while preserving default audio/image/video behavior.

## Risks / Open Questions

- The remaining product choice is whether to reject `--force-document` for WhatsApp or implement true WhatsApp document-forcing; either path should add a regression test for the dry-run/gateway payload boundary.
- Implementing force-document for WhatsApp would change media routing for audio/image/video files and should preserve existing normal sends, especially voice-note and `gifPlayback` behavior.
