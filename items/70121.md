## Summary

Keep open. #70121 is recent and includes concrete reproduction data for a third-party plugin autoStart subprocess being SIGTERM'd under the Gateway lifecycle. Current main has plugin service and one-shot command helper surfaces, but I found no documented managed long-running subprocess API for plugin services and no Camofox-specific or generic fix that would justify closing this as implemented, duplicate, or ClawHub-only.

## Best Possible Solution

Keep #70121 open as a focused plugin subprocess lifecycle/API investigation. The best path is to add a minimal external-plugin fixture that registers a full-mode service, starts a long-running child server with the reported detached plus piped-stdio shape and an `ignore`-stdio control, then run it under Gateway startup/restart. If core is terminating or orphaning it, add a documented plugin-owned long-running process contract, likely a service-context/process helper with explicit ownership, stop handling, logging, and regression tests. If the real plugin is starting during discovery/top-level evaluation or relying on an unsupported pattern, update plugin docs/diagnostics and route the plugin fix externally without closing the core API question prematurely.

## Evidence

That is enough signal for a current core/plugin-runtime investigation rather than stale or incoherent cleanup.
