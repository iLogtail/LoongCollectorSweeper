## Summary

Keep #56839 open. Current main still appears to permit the reported cross-channel target confusion: cron failure delivery resolves `delivery.to` through the resolved/fallback channel, and the WhatsApp target parser can treat `telegram:<digits>` as a valid WhatsApp E.164 recipient because the shared `normalizeE164` helper strips any provider-like prefix before adding `+`. I found no regression test or source guard that rejects a Telegram-prefixed target before WhatsApp delivery.

## Best Possible Solution

Keep this issue open and add a narrow regression fix. The safest direction is to reject foreign channel-prefixed targets before parsing them with a fallback channel, and add focused tests for cron/failure delivery where `delivery.channel` resolves to WhatsApp but `delivery.to` is `telegram:<numeric-id>`. WhatsApp target normalization should also fail closed for unsupported provider prefixes instead of relying on generic `normalizeE164` prefix stripping.
