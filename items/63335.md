## Summary

Keep open. Current main has useful timeout mitigations, but I did not find a direct implementation or regression proving that the reported Telegram single-turn timeout churn is fixed. The runner still intentionally reuses the same runId across retry/fallback, still rotates auth profiles on assistant timeouts, and still allows bounded same-model/profile/fallback retry paths before a user-visible result appears.

## Best Possible Solution

Keep this open and add a focused embedded-runner/gateway regression that simulates one Telegram direct-chat turn timing out across auth-profile rotation and model fallback with the same runId. The fix should cap repeated same provider/model timeout attempts for one logical turn, surface either a deterministic timeout error or a single fallback result, clear the processing state, and add diagnostics that distinguish same-candidate retry, same-provider different-profile retry, and cross-provider fallback.
