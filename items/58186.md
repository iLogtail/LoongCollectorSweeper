## Summary

Keep open. Current `main` still does not implement hook-level `systemPrompt` plumbing for the Gmail preset or generic hook mappings, so this PR remains materially relevant. The discussion also surfaced a real security constraint: if this lands, `systemPrompt` must stay literal-only rather than payload-templated.

## Evidence

- **Gmail preset on main still only defines `messageTemplate`:** The built-in Gmail hook preset has `sessionKey` and `messageTemplate`, but no `systemPrompt` field, and the resolved hook/action types likewise do not expose one.
  - file: [src/gateway/hooks-mapping.ts:72](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/gateway/hooks-mapping.ts#L72)
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Hook mapping execution path does not build or forward a system prompt:** `buildActionFromMapping` renders `messageTemplate` and returns an agent action without any `systemPrompt`, so mapped hook runs cannot inject extra system instructions on current `main`.
  - file: [src/gateway/hooks-mapping.ts:245](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/gateway/hooks-mapping.ts#L245)
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Hook payload and cron payload contracts lack `systemPrompt`:** `HookAgentPayload` and `CronAgentTurnPayloadFields` only carry `message` plus model/thinking/etc.; there is no hook-to-cron field for a system prompt on current `main`.
  - file: [src/gateway/hooks.ts:213](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/gateway/hooks.ts#L213)
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Async hook dispatch test on main only verifies external-content provenance:** The gateway hook test checks `externalContentSource: "gmail"` survives async dispatch, but there is no assertion or fixture covering `systemPrompt` propagation.
  - file: [src/gateway/server.hooks.test.ts:248](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/gateway/server.hooks.test.ts#L248)
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Config schema/docs on main do not expose `hooks.mappings[].systemPrompt`:** Current schema and docs list mapping fields like `sessionKey`, `messageTemplate`, `deliver`, and `model`, but not `systemPrompt`, so the feature is not shipped or documented on `main`.
  - file: [docs/gateway/configuration-reference.md:549](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/docs/gateway/configuration-reference.md#L549)
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **PR discussion identified and addressed a real security concern:** Review comments on the PR flagged that templating hook `systemPrompt` from webhook payload would let untrusted email content reach the system-role channel; the PR discussion says the later commit changed this to literal-only, which strengthens the proposal rather than obsoleting it.
