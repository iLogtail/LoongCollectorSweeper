## Summary

Keep open. Current `main` still derives the memory filename date and header timestamp from the `/new` or `/reset` event time rather than the session creation time, so the requested behavior is not implemented on `main`. The underlying bug remains plausible, and the PR also appears to need follow-up on the reset-file fallback path noted in review.

## Evidence

- **Current main still stamps memory files from event time:** `saveSessionToMemory` sets `now = new Date(event.timestamp)`, then uses that for `dateStr`, fallback HHMM slug, `timeStr`, and the `# Session:` header. That means a session created on Day 1 but reset on Day 2 will still be written as Day 2 on current `main`.

## Risks / Open Questions

- If this PR lands as originally described without mirroring the `.reset.*` fallback for session creation timestamps, rotated sessions can still be stamped with the `/new` event date instead of the original session date.
- The review context also flagged a minute-boundary race in one new HHMM-slug test, so the proposed test coverage may need adjustment before merge.
