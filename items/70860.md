## Summary

Keep open. Current `main` does not show a landed fix for ACP orphan-process cleanup, and the current docs explicitly acknowledge a related failure mode where ACP work can finish while the harness process remains stuck until killed manually.

## Evidence

- **Current docs acknowledge post-completion ACP stalls:** The ACP troubleshooting table on current `main` says an ACP session can 'stall indefinitely after completing work' because the harness finished but the ACP session did not report completion, and it tells users to inspect processes and kill stale ones manually.
  - file: [docs/tools/acp-agents.md:558](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/docs/tools/acp-agents.md#L558)
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **One-shot ACP completion only calls runtime.close:** After a one-shot ACP run completes, the manager calls `runtime.close({ reason: "oneshot-complete" })` and clears cached runtime state. There is no extra OpenClaw-side PID/process-tree tracking or force-kill fallback in this path.
  - file: [src/acp/control-plane/manager.core.ts:954](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/acp/control-plane/manager.core.ts#L954)
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **ACPX wrapper close is a thin delegate:** The bundled ACPX runtime wrapper's `close()` method only forwards to the underlying delegate and optionally marks session state fresh; it does not add OpenClaw-side subprocess cleanup logic.
  - file: [extensions/acpx/src/runtime.ts:314](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/extensions/acpx/src/runtime.ts#L314)
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Current main still ships the same acpx dependency line:** The bundled ACPX plugin on current `main` still depends on `acpx` version `0.5.3`, matching the reporter's environment, so there is no visible dependency upgrade here that would explain the leak being fixed upstream in OpenClaw's current tree.
  - file: [extensions/acpx/package.json:7](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/extensions/acpx/package.json#L7)
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **No relevant fix landed after the latest release:** Diffing latest release `v2026.4.23` to current `main` for the ACP surfaces only shows changes in docs and ACPX service/plugin metadata; there is no change in `extensions/acpx/src/runtime.ts` or `src/acp/control-plane/manager.core.ts` that would implement orphan-process cleanup.
  - command: `git diff --stat a9797214338ba31c52c796adbb75afb16e0684a9..bc21f500d4c14d8ecbacb992c39c7ff0954b04f2 -- extensions/acpx src/acp docs/tools/acp-agents.md`
- **Checkout remained clean during review:** The repository stayed byte-for-byte clean after inspection.
  - command: `git status --porcelain`
