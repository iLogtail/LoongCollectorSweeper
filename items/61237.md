## Summary

Keep #61237 open. Current main still treats AGENTS.md/SOUL.md/MEMORY.md rules as injected prompt context plus advisory safety guidance, with hard enforcement limited to tool policy, exec approvals, sandboxing, channel allowlists, and plugin/tool-call approval hooks. I did not find a shipped rule-priority or confirmation-gate mechanism that would reliably prevent the reported question-to-action and unauthorized-change patterns for Claude Opus/OpenClaw sessions.

## Best Possible Solution

Keep the issue open for maintainer product/design work. The best path is a core, configurable confirmation/permission policy that can distinguish advisory memory text from enforceable gates: for example, require approval for ambiguous question-to-action transitions, external/mutating tools, outbound messages, commits/pushes/deployments, or user-defined high-risk action classes, while preserving explicit standing-order autonomy. Implementation should include docs and regression coverage around prompt composition, bootstrap-rule loading/truncation, before_tool_call/plugin approvals, exec policy, and at least one harness-level scenario for ambiguous user wording.

## Risks / Open Questions

- A blanket confirmation gate could break documented standing-order, heartbeat, cron, and trusted-operator workflows, so the fix likely needs scoped policy rather than a universal prompt change.
- Intent classification for ambiguous questions is model-dependent; hard enforcement should focus on mutating/external tool surfaces while using prompt changes only as a secondary aid.
- Existing plugin hooks can prototype per-tool approval, but keeping this as core work is appropriate because the report concerns default agent safety and memory-rule priority, not an optional third-party integration.
