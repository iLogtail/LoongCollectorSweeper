## Summary

Keep PR #70707 open. Current main still has the exact device-token revocation/rotation race the PR describes: the relevant handlers respond first, then defer `disconnectClientsForDevice` with `queueMicrotask`, while the per-RPC WebSocket dispatch only revalidates shared gateway auth and has no device-token invalidation gate. The PR is a focused gateway hardening change with proposed regression coverage, and the provided Greptile review also treated the approach as targeted and safe to merge rather than obsolete.

## Best Possible Solution

Keep PR #70707 open and review it as a plausible targeted gateway security hardening fix. The best path is to land an equivalent implementation that synchronously marks affected device-token WebSocket clients invalid before responding to `device.pair.remove`, `device.token.rotate`, and `device.token.revoke`, rejects subsequent RPC frames before method dispatch, and includes ordering-focused regression tests.
