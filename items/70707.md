## Summary

Keep open. Current main still matches the race described by PR #70707: device pair removal and device-token rotate/revoke respond first, then defer disconnect through `queueMicrotask`, while the request context only closes matching sockets and the per-RPC dispatch path only rechecks shared gateway auth generation. The PR’s proposed synchronous per-client invalidation gate is not implemented on main, and the current tests cover deferred disconnects rather than rejection of already-pipelined device-token RPCs.

## Best Possible Solution

Keep PR #70707 open. The best path is to review or reimplement the same focused fix on current main: synchronously mark affected device-token clients invalid before responding in `device.pair.remove`, `device.token.rotate`, and `device.token.revoke`; make `disconnectClientsForDevice` mark clients invalid as defense in depth; reject any later post-handshake RPC from invalidated clients before business dispatch; and add regression tests for the response-flush ordering plus pipelined RPC rejection.
