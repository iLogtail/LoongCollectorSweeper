## Summary

Keep open. Current checked-out main still has the device-token remove/rotate/revoke window described by the PR: successful handlers respond first, then defer client disconnect through queueMicrotask, while the WebSocket per-RPC path only re-checks shared gateway auth generation and has no device-token invalidation gate. The PR discussion and bot review identify the same race and the provided patch is a plausible focused gateway hardening change, so no conservative close reason applies.

## Best Possible Solution

Keep #70707 open for maintainer review. The best product/code path is a focused gateway fix that synchronously marks affected device-token clients invalid before successful device.pair.remove, device.token.rotate, and device.token.revoke responses, adds a per-RPC dispatch guard analogous to the shared-auth generation guard, and preserves tests for ordering plus device/role filtering.
