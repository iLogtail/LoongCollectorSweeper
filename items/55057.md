## Summary

Current `main` already covers the reported failure mode: OpenAI-compatible/custom-provider runs are routed through boundary-aware transport stream wrappers instead of raw `streamSimple`, and the embedded subscriber now preserves final assistant text even when a provider only yields a final message without usable deltas. The same behavior is present in release `v2026.4.22`.

## Close Comment

Closing after a Codex review because current `main` already implements the behavior this report asks for. Embedded runs now route default OpenAI-compatible `streamSimple` paths through boundary-aware transport wrappers (`src/agents/pi-embedded-runner/stream-resolution.ts:44`, `src/agents/pi-embedded-runner/stream-resolution.ts:120`, wired from `src/agents/pi-embedded-runner/run/attempt.ts:1392`), and the OpenAI/OpenAI-compatible transport layer uses `createAssistantMessageEventStream()` instead of handing raw provider events straight to the Gateway path (`src/agents/openai-transport-stream.ts:684`, `src/agents/openai-transport-stream.ts:1116`). On top of that, the embedded subscriber now preserves final assistant text at `message_end` even for non-streaming/no-`text_delta` providers (`src/agents/pi-embedded-subscribe.ts:271`, `src/agents/pi-embedded-subscribe.handlers.messages.ts:713`), with regression coverage in `src/agents/pi-embedded-runner/stream-resolution.test.ts:84` and `src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.does-not-emit-duplicate-block-replies-text.test.ts:87`. This is also reflected in the shipped `v2026.4.22` changelog entry for custom OpenAI-compatible streaming behavior (`CHANGELOG.md:1360`). If you still see this on a newer build, please open a fresh issue with current Gateway logs and the exact provider/model config.
