## Summary

Keep #65168 open. Current main still returns declared commands for an unpaired node, stores them on the live node session, and `node.invoke` authorizes against that live command list. The in-tree test suite still includes a test explicitly expecting allowlisted commands to remain visible before node pairing exists, while docs/changelog say device pairing alone should not expose node commands. The linked PR #65169 is open and unmerged in the provided GitHub context, so it does not make the issue obsolete.

## Best Possible Solution

Keep #65168 open until a fix lands. The best code path is to fail closed for first-time or unapproved node-pair state by publishing an empty live command set until `node.pair.approve` succeeds, add or update regression coverage so `node.invoke` cannot forward pre-approval commands, and align the existing allowlist-update test with the documented 2026.3.31 command-gating contract. PR #65169 is the obvious related implementation to review or supersede, but the issue should remain open until current main actually contains the fix.
