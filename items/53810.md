## Summary

Keep open. Current main has partial adjacent work, but not a complete fix for the reported provider/model-specific failure. The Qwen/legacy ModelStudio provider still exposes the affected Alibaba Coding Plan models through the generic OpenAI-compatible transport, while that transport only turns structured `delta.tool_calls` into `toolCall` blocks and treats `reasoning_content` as thinking. The new Kimi Coding wrapper parses the special `<|tool_call_begin|>` markup only when it appears as a standalone text block on the separate `kimi` provider, not when the affected ModelStudio/Qwen models emit it inside thinking/reasoning content. The MiniMax mitigation also applies to the dedicated `minimax` Anthropic-compatible path, not the Qwen/ModelStudio `qwen/MiniMax-M2.5` route from the issue. The cross-referenced PRs do not supersede this report: #61481 was closed unmerged and concerned phase-aware visible text extraction, and #61801 was merged but scoped to archive delete retry behavior.

## Best Possible Solution

Keep the issue open and address it in provider-owned stream normalization, likely starting with the Qwen/ModelStudio OpenAI-compatible path. A conservative fix would add focused parsing or failure detection for tagged Kimi-style tool-call sections when they appear in reasoning/thinking content, normalize `stopReason` only when valid tool calls are extracted, and add regression coverage for `qwen/kimi-k2.5`, `qwen/MiniMax-M2.5`, and the existing `kimi` wrapper so standalone text parsing and thinking-block behavior are both explicit.
