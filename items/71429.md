## Summary

Keep #71429 open. Current main still has related reload/restart hardening and a durable generic outbound queue, but the normal Telegram same-surface auto-reply path is not visibly journaled before Telegram Bot API delivery and sendChatAction failures are logged through typing callbacks rather than converted into a hard user-visible turn failure. The recent Telegram change on main addresses stale preview finals, not this hot-reload delivery-loss path.

## Best Possible Solution

Keep #71429 open. Add a focused Telegram regression that simulates a live Telegram turn, hot reload/channel restart pressure, sendChatAction failure, and final Bot API delivery failure. Then either route normal Telegram final replies through the existing durable outbound/session-delivery queue before Bot API send, or deliberately fail the turn with clear Telegram/UI status and retry guidance.
