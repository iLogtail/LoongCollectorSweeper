## Summary

Keep #71429 open. Current main has related restart/reload hardening and a durable generic outbound queue, but normal same-surface Telegram auto-replies still go through the in-process reply dispatcher and Telegram deliverReplies path rather than being journaled before Bot API delivery. sendChatAction failures are logged and swallowed by the typing guard, not converted into a hard visible turn failure. The existing maintainer comments also explicitly leave this issue open for the same Telegram-specific gap.

## Best Possible Solution

Keep #71429 open. The next maintainer-quality fix should add a focused Telegram regression for live-turn hot reload with sendChatAction and final delivery failure, then route normal Telegram final replies through a durable queue before Bot API delivery or deliberately fail the turn with visible Telegram/UI retry guidance. The existing generic outbound and restart-continuation queues are likely reusable primitives, but the same-surface Telegram auto-reply path needs explicit wiring and coverage.
