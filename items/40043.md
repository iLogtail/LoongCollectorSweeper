## Summary

Keep #40043 open. Current main still lacks both proposed behaviors: gateway startup does not check selected provider auth, and skill discovery still supports only direct skill roots, immediate child skill folders, plus the narrow `dir/skills/*` unwrap. The PR also remains risky as written because its auth helper would not use the existing runtime auth contract and its recursive scan bypasses the candidate-scan budget called out in review.

## Best Possible Solution

Keep the PR open for maintainer rework or ask the contributor to split it. The recursive skill discovery change should be bounded by a candidate scan budget, preserve root-containment and symlink protections, update watcher behavior or document its limits, and add nested bundle tests. The startup auth warning should be a separate best-effort, read-only diagnostic based on `hasAvailableAuthForProvider()` or an equivalent shared runtime contract so it cannot block gateway startup or warn for valid auth sources.

## Risks / Open Questions

- Merging the PR as written could put write-capable auth-store I/O on the gateway startup path and turn a diagnostic warning into a startup failure.
- Merging the PR as written could emit false missing-auth warnings for valid provider auth sources handled by the existing runtime contract.
- Merging the recursive scanner as written could traverse large non-skill trees without enforcing the existing `maxCandidatesPerRoot` candidate budget.
- Closing this PR as implemented would drop still-missing recursive skill discovery and still-missing startup auth warning behavior.
