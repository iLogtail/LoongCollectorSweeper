## Summary

Keep #40043 open. Current main still lacks arbitrary recursive skill discovery and does not implement the PR's generic startup warning for a selected provider with no usable auth. The exact Anthropic-default symptom from #39903 appears partly addressed on main, but the PR is not fully implemented or fully superseded, and its own review comments identify correctness blockers that require rework before merge.

## Best Possible Solution

Keep #40043 open for maintainer rework or contributor split. The recursive skill discovery portion should either move to the focused #69663 path or be reworked here with bounded traversal, containment and symlink-loop protection, max-candidate budgeting, watcher/docs alignment, and regression tests for grouped managed/workspace/extra skill roots. The startup auth warning should be a separate best-effort diagnostic using the same runtime auth contract as hasAvailableAuthForProvider, through a read-only/non-throwing path, or explicitly dropped if maintainers decide the current default/auth-signal changes sufficiently resolve #39903.

## Risks / Open Questions

- Merging #40043 as written could put write-capable auth-store I/O on the gateway startup path and turn a diagnostic into a startup failure.
- Merging the auth warning as written could produce false warnings for supported auth sources such as custom provider keys, local synthetic auth, Bedrock SDK auth, or broader runtime profile resolution.
- Merging the recursive scanner as written could traverse large non-skill trees without the existing candidate budget.
- Closing #40043 before #69663 or an equivalent recursive-scan fix lands would leave the nested skill-layout request unresolved.
