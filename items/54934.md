## Summary

Keep open. Current main already removes the original Feishu send/thread-reply failure path by rendering cards only from the generic `presentation` contract instead of `params.card`, so the PR is partly obsolete. But the PR still contains useful, unimplemented hardening: Feishu `edit` still accepts `ctx.params.card`, `editMessageFeishu` treats `{}` as a valid card and patches it, and `sendCardFeishu` still stringifies any card without an empty-payload guard. That means the PR should be rebased/narrowed rather than auto-closed.

## Best Possible Solution

Keep the PR open but request a rebase against current main. The best fix is to preserve the shipped `presentation` send path, add a small shared empty-card guard in `extensions/feishu/src/send.ts` for direct card send/edit helpers, make the Feishu edit action reject empty card payloads before network calls, and add focused tests for the edit/action/runtime helper cases without reintroducing native `card` as a generic send parameter.
