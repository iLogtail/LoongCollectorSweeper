## Summary

Keep open. Current `main` has partially moved past the PR's original send-path context, but it does not implement the requested guard end-to-end. `extensions/feishu/src/channel.ts:695-705` now builds send cards only from `params.presentation`, so raw `params.card` is effectively ignored for `send`/`thread-reply`; that makes the PR's send-action hunk stale, not already merged. But the underlying empty-card bug still exists in the runtime helpers and edit path: `extensions/feishu/src/send.ts:495-509` will `JSON.stringify(card)` and send `{}` via `sendCardFeishu`, and `extensions/feishu/src/send.ts:525-528` treats `Boolean(card)` as sufficient for edits, so `card: {}` still passes local validation. `extensions/feishu/src/channel.ts:789-800` still forwards `ctx.params.card` to `editMessageFeishu` unchanged. Current tests only cover non-empty cards (`extensions/feishu/src/send.test.ts:384-395`, `extensions/feishu/src/send.reply-fallback.test.ts:97-106`, `:157-166`, `:258-265`) and do not lock in empty-card rejection. That is not strong evidence for closure; the PR still represents potentially useful bug-fix work, though it needs rebasing to the presentation-based send path.
