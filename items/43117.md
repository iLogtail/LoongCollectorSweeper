## Summary

Keep open. Current main already implements the Gemini Embedding 2 provider path, `outputDimensionality`, and opt-in multimodal indexing for image/audio files under `memorySearch.extraPaths`, matching the helpful newcodebook comment. The issue’s remaining central ask is broader: index conversation-history attachments so text queries can find images from prior chats, plus video/PDF modality coverage. Current session memory is still text-only and multimodal memory is explicitly limited to extra paths and image/audio files.

## Best Possible Solution

Keep the item open but narrow it around the unshipped work: retain the shipped Gemini Embedding 2, `outputDimensionality`, and `extraPaths` image/audio support, then implement or split a focused follow-up for opt-in indexing of eligible inbound/session attachments into memorySearch with tests and docs. If maintainers intentionally want extraPaths-only multimodal memory, document that product boundary and close only after a canonical issue tracks chat-history attachment search separately.

## Risks / Open Questions

- Indexing chat attachments needs explicit opt-in, size limits, retention/privacy handling, and media-store lifecycle review because current inbound media may be represented as opaque `media://` refs or transient offloaded files.
- Adding video/PDF memory modalities would require schema/docs updates and provider capability validation beyond the current image/audio classifier.
