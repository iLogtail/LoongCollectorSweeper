## Summary

Keep open. Current `main` still preserves the 15s Discord voice-ready timeout introduced in 2026.4.5, `/vc join` still waits on that handshake before replying, there is no current config surface for a voice connect-timeout override, and the Discord voice code is unchanged between `v2026.4.23` and the reviewed `main` SHA. The report also has a second reproducer on supported Node 24, so this is not strong close evidence.

## Evidence

- **Join still hard-fails after a 15s voice-ready wait:** `DiscordVoiceManager.join()` still calls `entersState(connection, VoiceConnectionStatus.Ready, 15_000)`, then destroys the connection and returns `Failed to join voice channel: ...` on error. That matches the reported join-then-leave failure mode.

  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Regression timeout is intentionally locked in by test:** Current coverage explicitly asserts that the initial voice connection readiness timeout remains `15_000`, so `main` still expects the shorter post-2026.4.5 timeout rather than a rollback or increase.
  - file: [extensions/discord/src/voice/manager.e2e.test.ts:350](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/extensions/discord/src/voice/manager.e2e.test.ts#L350)
