## Summary

Keep #65668 open. Current main and latest release still preserve the Linux/no-recognized-supervisor detached respawn path that the issue reports, and the doctor/docs wording still frames OPENCLAW_NO_RESPAWN=1 as a startup-overhead optimization rather than a custom-supervisor safety setting. Related #46310 appears to reduce unnecessary config.patch restarts, but restart-required config changes and other SIGUSR1/update paths still reach the same respawn behavior; open #66865 and #68493 are related but do not supersede this Linux custom-supervisor report on main.

## Best Possible Solution

Keep the issue open and fix the product behavior directly. The safest maintainer path is to make unmanaged/custom-supervised Linux restarts use in-process restart by default or require an explicit detached-respawn opt-in, then update doctor/docs to describe OPENCLAW_NO_RESPAWN=1 as a supervisor-safety setting. If maintainers prefer keeping detached respawn, add explicit custom-supervisor detection or a clear config/env contract plus regression tests for Linux custom supervisors.
