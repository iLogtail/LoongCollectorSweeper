## Summary

Keep #72390 open. Current main does not support SecretRef objects in `openclaw.json` `env.vars`: the public config type, Zod schema, generated schema, runtime env merge, and tests are all string-only. Existing SecretRef support covers credential-specific fields like `skills.entries.*.apiKey` and the narrow ACPX MCP env surface, but not the global `env.vars` workflow requested here.

## Best Possible Solution

Keep the issue open and implement `env.vars` SecretRef support end-to-end in core. A conservative path is to widen `env.vars` to `Record<string, SecretInput>` while preserving the existing direct string behavior and dangerous-key filters, add an explicit secrets-runtime collection/materialization step for `env.vars.*`, document the supported surface, update generated schema/help, and add regression tests for env/file/exec refs, unresolved refs, `${VAR}` placeholders, blocked env keys, and non-overwrite precedence. Keep the already-shipped MCP SecretInput support as separate narrow plugin-surface evidence, not a substitute for this global env feature.

## Risks / Open Questions

- Implementing this touches early config/env bootstrapping: `env.vars` currently feeds process/runtime env before many consumers run, while SecretRefs themselves may depend on env/file/exec providers. The design needs explicit ordering and cycle/fail-closed behavior.
- Resolved `env.vars` secrets would enter process or service environment state, so the implementation must avoid persisting plaintext into config, generated service metadata, logs, snapshots, audit output, or broader child process env than intended.
- The change likely needs schema/type/generated docs updates plus focused tests for validation, resolution, blocked-key filtering, service/runtime env collection, unresolved refs, and existing string compatibility.
