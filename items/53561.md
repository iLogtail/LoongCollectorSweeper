## Summary

Keep this PR open for maintainer judgment rather than closing as implemented or superseded. Current `main` has separate pre-auth/connect-challenge timeout hardening, but it still sends the `connect` RPC through the generic request watchdog. The PR's proposed `timeoutMs: null` behavior is not present on `main`, and the existing bot review identified a real concern with disabling the connect response timeout entirely.

## Best Possible Solution

Keep the PR open, but require a maintainer-reviewed revision before merge: preserve a bounded failure path for stalled connect responses, make the connect handshake timeout budget explicit instead of disabling it indefinitely, and add focused tests for both the short generic request-timeout race and the no-response-after-challenge hang case. If maintainers decide the current generic watchdog is the desired contract, close the PR with that rationale rather than as already implemented.
