## Summary

Keep open. Current `main` still has the handshake-timeout behavior this PR is trying to address, but the proposed fix is not safe as written because it removes the only timeout that applies after `connect.challenge` is answered and before `hello-ok` arrives. That leaves startup callers able to hang indefinitely, so this needs maintainer follow-up rather than cleanup closure.

## Evidence

- **Current main still times out the `connect` RPC via the generic request watchdog:** `GatewayClient.sendConnect()` still calls `this.request("connect", params)` with no override, and `request()` falls back to `this.requestTimeoutMs` for non-`expectFinal` requests.
  - file: [src/gateway/client.ts:528](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/client.ts#L528)
  - command: `nl -ba src/gateway/client.ts | sed -n '436,575p' && nl -ba src/gateway/client.ts | sed -n '918,955p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Main has no separate post-challenge handshake timer:** The pre-auth handshake timer only fires while `connectSent` is false. Once the client sends `connect`, `armConnectChallengeTimeout()` no longer protects the handshake.
  - file: [src/gateway/client.ts:801](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/client.ts#L801)

- **Removing the `connect` request timeout can deadlock startup waiters:** `serveAcpGateway()` waits on `gatewayReady` until `onHelloOk`, `onConnectError`, or `onClose` fires. If `connect` is left pending forever after a challenge, this startup path has no bounded failure path.
