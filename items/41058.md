## Summary

Keep #41058 open. Current main still hardcodes the pairing reply text and CLI approval hint, and the channel config/schema surfaces do not expose `channels.*.pairingMessage`. The linked #9897 is the same request but is closed, and #42617 is an open/unmerged partial implementation path, so the remaining user-visible feature is not implemented or fully superseded on main.

## Best Possible Solution

Keep the issue open and use it as the product-level tracker until a maintainer accepts the config shape. The likely implementation belongs in the shared pairing message builder plus exported config types, strict zod schemas, generated channel metadata/docs, and bundled channel callers. #42617 is useful prior work, but a maintainer should decide whether to revive it, supersede it, or broaden it to all pairing-capable bundled channels before closing #41058.
