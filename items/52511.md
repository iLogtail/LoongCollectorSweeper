## Summary

Keep #52511 open. Current main has useful reconnect and token-drift retry behavior, but I do not see a shipped, permission-specific fix for the reported Control UI stale-permission/code-1000 browser-control failure. The related PR #58022 attempted a broad page reload on WebSocket close code 1000, but it was closed unmerged and review feedback called that approach too broad. Current browser control also still requires operator.admin through the plugin gateway method, while the report expects operator.read/operator.write to be sufficient, so this needs a maintainer-scoped product/security fix or a narrower reproduction-backed bug fix rather than cleanup closure.

## Best Possible Solution

Keep the issue open and narrow it into a permission-specific fix. The best path is to reproduce current main with Control UI plus browser.request after gateway restart and token rotation, then either make the reconnect path refresh the correct device-token/scope state on a permission-specific signal, document/repair the required operator.admin approval flow for browser control, or intentionally reduce browser.request scope only if maintainers accept that security change.
