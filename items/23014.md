## Summary

Keep #23014 open. Current main still has no `.m2a` audio extension support in the shared media MIME/extension tables, generic attachment classification, Telegram audio preflight, or adjacent audio allowlists/tests. Some `.m2a` payloads may work if Telegram supplies an `audio/*` MIME or sniffing identifies the container, but the requested extension-only/generic-MIME path is still missing. The related PR #23260 was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep #23014 open until maintainers either implement `.m2a` support or explicitly reject it. The likely fix belongs in the shared media pipeline: add `.m2a` as an audio alias in `src/media/mime.ts`, ensure generic-MIME Telegram/media saves can infer audio from the original filename where appropriate, and add focused tests for `detectMime`, `isAudioFileName`, `resolveAttachmentKind`, and Telegram generic-MIME `.m2a` preflight/context behavior. Maintainers should also decide whether secondary audio allowlists such as media notes, memory multimodal selection, and voice-message compatibility should include `.m2a` for consistency.
