## Summary

Keep open. The PR is trying to fix a real misleading diagnostic from #65083, and current main still has the same product gap: `commands` is a built-in chat slash command, not a registered CLI/plugin command, yet the missing-plugin diagnostic still treats any unknown primary command excluded by `plugins.allow` as a bundled plugin CLI surface. The submitted PR is not ready to merge because the review comments correctly identify a regression in allow-list diagnostics for real plugin commands without explicit `plugins.entries` config.

## Best Possible Solution

Keep the PR open but require changes: preserve the existing allow-list diagnostics for real plugin ids like `browser`, add a test for `openclaw commands`/`resolveMissingPluginCommandMessage("commands", { plugins: { allow: [...] } })`, and fix the user-facing behavior with an explicit command-kind check. The best product outcome is either an accurate diagnostic saying `commands` is a chat slash command (`/commands`) rather than a CLI/plugin command, or a real first-class `openclaw commands` CLI surface if maintainers want that command to work.
