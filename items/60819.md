## Summary

Keep #60819 open. Current main still has the reported diagnostic shape: doctor can still emit the exact “No active memory plugin is registered” message when the command-local active memory backend is null, and status can still render an enabled memory slot as unavailable before it asks a healthy runtime manager for status. The issue's mem0-specific recovery path is not covered by current tests, and the related memory false-positive reports remain unresolved rather than shipped fixes.

## Best Possible Solution

Keep the issue open and fix the diagnostics path directly: status should consult the active memory runtime or gateway-backed memory status before rendering an enabled slot as unavailable, and doctor should distinguish “no runtime loaded in this command path” from “gateway/runtime is healthy but CLI-side inspection is inconclusive.” Add regression coverage with a mem0-like external memory slot whose runtime manager reports healthy status without relying on the built-in default store path.
