## Summary

Keep #41993 open. Current main still has the reported any-blocked DNS behavior in the SSRF pinning path: a public hostname resolving to a valid IPv4 address plus `::`, `::1`, or Teredo-style special-use IPv6 will still throw before `dedupeAndPreferIpv4()` can select the safe public address. The later RFC2544/FakeIP work solves `198.18.0.0/15` opt-in cases, but it does not implement the issue's requested mixed allowed/blocked DNS filtering. The related fix PRs in the provided timeline were closed unmerged, and their proposed behavior is not present in current source.

## Best Possible Solution

Keep the issue open and implement the scoped SSRF DNS-pinning fix in `src/infra/net/ssrf.ts`: replace the void any-blocked assertion with a filter-and-assert helper that returns only allowed `LookupAddress` entries, throws when none remain, feeds the filtered list into `dedupeAndPreferIpv4()` and dispatcher override pinning, and adds regression tests for public IPv4 + `::`, public IPv4 + Teredo, all-blocked answers, duplicate/order preservation, and absence of blocked addresses from the pinned lookup pool.
