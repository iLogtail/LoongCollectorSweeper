## Summary

Keep open. Current main still hardcodes config auto-recovery notices into the next agent prompt and routed outbound delivery, and there is no `gateway.systemEvents.recoveryNotice` config surface. Related #71289 narrowed when whole-file recovery runs, #49579 is an open WhatsApp-specific banner filter, and #39307 asks for broader user-visible reset signaling, so none supersede this focused core gateway knob.

## Best Possible Solution

Keep the issue open and implement this as a core gateway config feature: add a typed `gateway.systemEvents.recoveryNotice` policy, gate prompt-facing and outbound recovery notices in the restart-sentinel/startup/reload recovery paths, leave restart continuations untouched, and add focused tests plus gateway configuration docs. Coordinate wording with #39307 and avoid solving this with another channel-specific hardcoded filter like #49579.

## Risks / Open Questions

- The implementation should preserve the default safety behavior for operator-owned agents while allowing customer-facing deployments to opt down to log-only or off.
- The knob likely needs to cover both restart-sentinel auto-recovery notices and the newer startup/reload `Config recovery warning` path so operators do not get inconsistent behavior.
- Schema/types/help/docs and generated config metadata need to stay aligned if a new public config key is added.
