## Summary

Keep open. Current main still has the reported behavior: config auto-recovery writes a hardcoded restart-sentinel notice, startup processing always enqueues that notice as a system event and sends it through outbound delivery when a route exists, and the strict gateway config schema/types do not expose `gateway.systemEvents.recoveryNotice`. The related items do not supersede this: #71289 narrowed when whole-file recovery happens, #49579 is a WhatsApp-specific banner filter PR, and #39307 asks for broader/more-visible reset signaling rather than this operator-controlled suppression knob.

## Best Possible Solution

Keep the issue open and implement the requested core gateway config surface if maintainers agree: add a typed/schema-validated `gateway.systemEvents.recoveryNotice` policy, gate only the config auto-recovery notice in `server-restart-sentinel.ts` while preserving restart continuations and recovery behavior, add focused tests for `user`/`log`/`off` behavior, and document the operator-facing setting on the public gateway configuration docs.
