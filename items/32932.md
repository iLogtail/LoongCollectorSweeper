## Summary

Keep open. Current main still has no scoped cron webhook private-network opt-in: cron finished webhooks, failure-alert webhooks, and failure-destination webhooks all use the strict SSRF-guarded fetch path without a caller policy, while cron config, Gateway protocol schemas, CLI/UI surfaces, docs, and tests expose only webhook URL/token/delivery fields. This remains an actionable core cron/security design issue, not a ClawHub integration request or a duplicate of the separate inbound GitHub webhook/plugin item.

## Best Possible Solution

Keep the issue open until maintainers either implement a scoped cron webhook private-network opt-in or explicitly decide cron webhook destinations must continue to block intranet targets. A likely implementation belongs in the cron config and/or per-job delivery contract, passes an explicit SSRF policy into postCronWebhook for finished, failure-alert, and failureDestination webhooks, and updates strict schema, generated config help/docs, Control UI/CLI/API surfaces, and regression tests for both default-blocked and opted-in private-network behavior.
