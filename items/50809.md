## Summary

Keep open. Current main has Android SMS send/search support, but it does not implement the requested `sms.read` command, does not register or dispatch `sms.received`, and does not have an SMS channel/plugin/config surface. The linked implementation PR #50808 was closed unmerged, so it does not supersede the issue. This may ultimately become a plugin-backed channel, but the requested Android command/event surface still needs maintainer product and security review.

## Best Possible Solution

Keep #50809 open as the canonical tracker unless maintainers intentionally split it. A good implementation path is to separate the Android node API work (`sms.read` by `_ID`/`THREAD_ID`, command policy, docs, Android tests) from the SMS channel design, and decide whether the channel should be a bundled plugin using `api.runtime.nodes` or needs a new Android event/plugin seam for receiver-driven inbound delivery.

## Risks / Open Questions

- A third-party plugin could approximate SMS ingress by polling `sms.search` through `api.runtime.nodes.invoke`, but that would not satisfy the requested native `sms.read` or receiver-driven `sms.received` flow without additional Android/core work.
- Inbound SMS routing touches Android SMS permissions, node command allowlists, and channel allowlist security, so the product shape should get maintainer review before being bundled.
