## Summary

Keep #15840 open. Current main still has no `messages.suppressMediaPlaceholders` or granular `messages.mediaPlaceholders` config, and the Signal/iMessage outbound paths still synthesize `<media:audio>` for blank audio media sends before dispatch. The issue also has a recent maintainer Codex comment explicitly keeping it open as the canonical product/config request; related implementation PRs #15853 and #15914 were closed unmerged, and narrower Signal issue #48319 does not supersede the shared config request.

## Best Possible Solution

Keep #15840 open as the canonical cross-channel configuration request. A conservative fix should add a documented default-off `messages` config option, validate it in the strict schema and generated metadata, thread it through Signal and iMessage outbound media paths that synthesize placeholders, preserve default accessibility fallback behavior, and add focused tests proving opt-in audio media sends omit visible `<media:audio>` where the transport can support it.
