## Summary

Keep #70451 open. Current main likely fixes the broad #64306-style startup path for `status`/`health` by skipping plugin preload there, but this fresh report specifically includes `openclaw hooks enable/list`. The hooks CLI still builds hook visibility and toggle state through plugin diagnostics, which loads plugin modules, so the Windows hooks timeout is not proven fixed or superseded.

## Best Possible Solution

Keep #70451 open and narrow it to the hooks CLI path. The likely maintainer fix is to make `openclaw hooks list/info/check/enable/disable` use a config/workspace/manifest-only plugin hook snapshot, or otherwise bound and diagnose plugin diagnostics loading on Windows. Add regression coverage proving ordinary hook visibility/toggle operations do not invoke full plugin module loading, and keep the config-only hook fallback documented.
