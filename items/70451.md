## Summary

Keep #70451 open. The related #64306 closure is useful evidence that current main avoids CLI plugin-registry preload for status/health/gateway-status style commands, but #70451 is specifically about `openclaw hooks ...` on Windows. Current `hooks` command execution still builds plugin diagnostics and enters broad plugin module loading, so there is not enough source-backed evidence that the Windows hooks timeout/SIGKILL is fixed.

## Best Possible Solution

Keep #70451 open and narrow the hooks CLI path. The likely maintainer fix is to make `openclaw hooks list/info/check/enable/disable` use cold hook/plugin metadata by default, or load only the selected hook's owning package, instead of calling broad plugin diagnostics during hook management. Add a regression test that `hooks enable <standalone-hook>` and `hooks list --json` do not invoke broad plugin module loading, and document the `hooks.internal.entries.<id>.enabled` config-only fallback for users blocked by the CLI.
