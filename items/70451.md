## Summary

Keep open. Current main has narrowed the generic #64306-style CLI bootstrap preload path for commands such as `status`/`health`, but #70451 specifically reports Windows `openclaw hooks list/enable`. Those hook actions still construct hook state through `buildPluginDiagnosticsReport()`, which loads plugin runtime modules before listing or enabling hooks, so the hooks-specific timeout is not clearly implemented on main.

## Best Possible Solution

Keep #70451 open. The best fix is to make `openclaw hooks list/info/check/enable/disable` avoid full runtime plugin diagnostics for ordinary hook management, using persisted plugin metadata or a lightweight snapshot for plugin-managed hooks plus workspace/bundled hook discovery for local toggles. Add regression coverage proving hooks list/enable do not call `loadOpenClawPlugins()` for config/workspace-only paths, and add a Windows smoke if feasible.
