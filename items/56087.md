## Summary

The reported Telegram multi-mac-profile failure matches a polling `getUpdates` 409 conflict loop that current `main` already handles. OpenClaw now resets the polling webhook latch and rebuilds the Telegram HTTP transport after a 409 conflict so retries use a fresh TCP socket instead of reusing the terminated keep-alive session, and that behavior is covered by targeted tests and documented as shipped in v2026.4.22.

## Close Comment

Closing from a Codex review because this appears implemented on `main` and shipped in `v2026.4.22`. The current Telegram polling code now treats `getUpdates` 409 conflicts as a recoverable restart path, resets the webhook-cleanup latch, and forces a fresh polling transport instead of reusing the terminated keep-alive socket (`extensions/telegram/src/polling-session.ts:366`, `extensions/telegram/src/polling-session.ts:368`, `extensions/telegram/src/polling-session.ts:377`). There is also a targeted regression test for `Conflict: terminated by other getUpdates request` that verifies the webhook cleanup reruns and the transport is rebuilt (`extensions/telegram/src/monitor.test.ts:790`, `extensions/telegram/src/monitor.test.ts:837`, `extensions/telegram/src/monitor.test.ts:841`). The fix is called out in the `2026.4.22` changelog (`CHANGELOG.md:116`, `CHANGELOG.md:222`). If you can still reproduce this on `v2026.4.22` or newer, please open a new issue with current gateway logs around the Telegram polling restart/error lines so maintainers can separate any remaining bug from the already-shipped 409-conflict fix.
