## Summary

Current `main` already routes `update.run`/restart through a deferred SIGUSR1 path and then drains active gateway work before shutdown, which addresses the reported transcript-loss-on-restart behavior.

## Evidence

- **`update.run` uses deferred restart scheduler:** `update.run` no longer restarts directly; on success it calls `scheduleGatewaySigusr1Restart(...)`, so restart goes through the shared idle/deferral path.
  - file: [src/gateway/server-methods/update.ts:107](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/gateway/server-methods/update.ts#L107)
  - command: `nl -ba src/gateway/server-methods/update.ts | sed -n '90,125p'`
- **Scheduler waits for pre-restart deferral check before SIGUSR1:** `setPreRestartDeferralCheck()` registers pending-work accounting, and `scheduleGatewaySigusr1Restart()` consults it via `deferGatewayRestartUntilIdle(...)` before emitting SIGUSR1.
  - file: [src/infra/restart.ts:110](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/infra/restart.ts#L110)
  - command: `nl -ba src/infra/restart.ts | sed -n '107,120p;531,546p'`
- **Gateway defines pending-work count from active queues/replies/runs/tasks:** The gateway's pre-restart deferral check sums queued operations, pending replies, active embedded runs, and active tasks.
  - file: [src/gateway/server.impl.ts:314](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/gateway/server.impl.ts#L314)
  - command: `nl -ba src/gateway/server.impl.ts | sed -n '314,320p'`
- **Restart shutdown drains active work before closing server:** On restart, the run loop marks the gateway draining, waits for active tasks and embedded runs to drain, and only then closes the server; if the timeout hits, it logs and does a best-effort abort.
  - file: [src/cli/gateway-cli/run-loop.ts:165](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/cli/gateway-cli/run-loop.ts#L165)
  - command: `nl -ba src/cli/gateway-cli/run-loop.ts | sed -n '165,207p'`
- **Regression coverage exists for both deferral and restart-drain behavior:** Tests assert that SIGUSR1 is deferred until pending work reaches zero and that restart waits for active tasks/embedded runs before closing.
  - file: [src/infra/infra-runtime.test.ts:318](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/infra/infra-runtime.test.ts#L318)
  - command: `nl -ba src/infra/infra-runtime.test.ts | sed -n '318,360p'; nl -ba src/cli/gateway-cli/run-loop.test.ts | sed -n '289,327p'`

## Close Comment

Closing from a Codex review: current `main` already implements the restart drain needed here. `update.run` now goes through `scheduleGatewaySigusr1Restart()` (`src/gateway/server-methods/update.ts:107`), that scheduler defers SIGUSR1 until the gateway's pending-work check reaches zero (`src/infra/restart.ts:110`, `src/infra/restart.ts:531`), the gateway defines that check from queued operations, pending replies, embedded runs, and active tasks (`src/gateway/server.impl.ts:314`), and the run loop waits for active tasks/runs to drain before closing the server on restart (`src/cli/gateway-cli/run-loop.ts:165`). There is also regression coverage for both the deferral path and the restart-drain path in `src/infra/infra-runtime.test.ts:318` and `src/cli/gateway-cli/run-loop.test.ts:289`. I could not confirm a released tag for the fix from the available local history, so I’m closing this as implemented on `main` rather than tying it to a shipped release.
