## Summary

Keep #14804 open. Current main still has sandbox isolation by mode/scope and per-scope workspaces, but no first-class ephemeral/tmpfs-backed workspace lifecycle for isolated sessions. Workspaces are stable by scope, existing files are preserved, Docker tmpfs only mounts configured container paths such as /tmp, and docs describe manual recreate/prune rather than cleanup-on-completion.

## Best Possible Solution

Keep this open until core sandbox lifecycle/config adds a documented disposable workspace mode. The likely implementation should add explicit AgentSandboxConfig/schema fields, create a fresh per-run workspace where supported, clean it on completion, define Docker/SSH/OpenShell behavior including whether tmpfs is required or best-effort, add lifecycle tests, and update public sandbox/config docs.

## Risks / Open Questions

- A fix needs explicit cross-backend semantics: Docker can use disposable bind roots or tmpfs, while SSH/OpenShell need clear remote workspace creation and cleanup behavior.
- The existing subagent/cron isolated-session paths create fresh transcripts, not fresh filesystem state, so naming could confuse users unless docs distinguish transcript isolation from workspace disposal.
