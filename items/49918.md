## Summary

Keep open. Current `origin/main` still has TUI `/new` allocate `tui-<uuid>` via local `setSession(uniqueKey)` only, so it does not hit the gateway `sessions.reset` path that emits `command:new` hooks. The linked PR #49945 is the right-shaped fix but remains open/unmerged, so the original bug is still actionable.

## Best Possible Solution

Keep #49918 open until the TUI `/new` handler preserves per-client `tui-<uuid>` isolation and also invokes the gateway reset path with reason `"new"` for the newly allocated session. The likely maintainer path is to rebase/review PR #49945 or apply the same narrow change in `src/tui/tui-command-handlers.ts`, update `src/tui/tui-command-handlers.test.ts` to expect `client.resetSession(<tui-uuid>, "new")` for `/new`, and leave `/reset` behavior unchanged.
