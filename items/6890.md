## Summary

Keep open. Current `main` does not implement a per-agent Ralph/self-loop iteration default: the only comparable loop cap in config is the global `session.agentToAgent.maxPingPongTurns`, and the issue body's Kimi-specific file references are obsolete but still salvageable into a valid OpenClaw feature request rather than something already fixed.

## Evidence

- **Per-agent agent config schema has no loop/max-iteration field:** `AgentEntrySchema` exposes per-agent fields such as `model`, `thinkingDefault`, `skills`, `subagents`, `embeddedPi`, `sandbox`, `params`, `tools`, and `runtime`, but no per-agent Ralph or iteration-limit setting.
  - file: [src/config/zod-schema.agent-runtime.ts:812](https://github.com/openclaw/openclaw/blob/a2221d6b47094cd34c12025ce809aba02c5e1c1c/src/config/zod-schema.agent-runtime.ts#L812)
  - sha: [a2221d6b4709](https://github.com/openclaw/openclaw/commit/a2221d6b47094cd34c12025ce809aba02c5e1c1c)
- **Only global session loop cap exists in schema:** The session schema defines `session.agentToAgent.maxPingPongTurns` as an optional global integer in the range `0..5`; there is no agent-scoped variant here.
  - file: [src/config/zod-schema.session.ts:59](https://github.com/openclaw/openclaw/blob/a2221d6b47094cd34c12025ce809aba02c5e1c1c/src/config/zod-schema.session.ts#L59)
  - sha: [a2221d6b4709](https://github.com/openclaw/openclaw/commit/a2221d6b47094cd34c12025ce809aba02c5e1c1c)
- **Runtime reads the global loop cap, not an agent override:** `sessions_send` resolves `maxPingPongTurns` from `resolvePingPongTurns(cfg)` and passes that single config-derived value into the A2A loop flow.
  - file: [src/agents/tools/sessions-send-tool.ts:292](https://github.com/openclaw/openclaw/blob/a2221d6b47094cd34c12025ce809aba02c5e1c1c/src/agents/tools/sessions-send-tool.ts#L292)
  - sha: [a2221d6b4709](https://github.com/openclaw/openclaw/commit/a2221d6b47094cd34c12025ce809aba02c5e1c1c)
- **Docs only describe the global session setting:** The configuration reference documents `session.agentToAgent.maxPingPongTurns` as the maximum reply-back turns between agents and does not document a per-agent loop-iteration default.
  - file: [docs/gateway/configuration-reference.md:2034](https://github.com/openclaw/openclaw/blob/a2221d6b47094cd34c12025ce809aba02c5e1c1c/docs/gateway/configuration-reference.md#L2034)
  - sha: [a2221d6b4709](https://github.com/openclaw/openclaw/commit/a2221d6b47094cd34c12025ce809aba02c5e1c1c)
- **No Kimi/Ralph implementation symbols exist on current main:** A repo search for the issue body's Kimi-era symbols returned no matches, which shows the writeup targets obsolete paths rather than a feature already present on `main`. That makes the issue mismatched, but not implemented.
  - command: `rg -n "Ralph|max_ralph_iterations|kimi_cli|KimiCLI|KimiSoul|FlowRunner" src docs && true`
  - sha: [a2221d6b4709](https://github.com/openclaw/openclaw/commit/a2221d6b47094cd34c12025ce809aba02c5e1c1c)

## Risks / Open Questions

- The issue body is written against obsolete Kimi-era paths, so any implementation would need a maintainer to restate it in current OpenClaw terms (`agents.list[]`, session config, or tool/runtime config).
- Current `main` already has loop-related features (`session.agentToAgent.maxPingPongTurns` and per-agent `tools.loopDetection`), so product intent still needs clarification on whether the ask is self-looping agent runs, agent-to-agent ping-pong limits, or both.
