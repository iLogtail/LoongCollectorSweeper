## Summary

Keep open. Current main still uses regex-based TTS directive parsing and deliberately accepts inline forms; tests and docs also describe inline `[[tts:...]]` / `[[tts:text]]...[[/tts:text]]` behavior. The PR is not obsolete, but it changes a public directive contract and bundles unrelated Slack/config-validation changes, matching the maintainer comment that it should not merge as-is.

## Best Possible Solution

Keep the PR open but require a narrower revision: split Slack/config-validation work into separate PRs if still needed, make the TTS PR only about directive semantics, and have maintainers decide whether inline/bare TTS tags remain supported or are intentionally migrated to standalone-line-only behavior. If standalone semantics are accepted, update `docs/tools/tts.md`, contract tests, and changelog coverage alongside the parser change, using the related inline-literal bug context (#56680) as the behavioral anchor.
