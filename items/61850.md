## Summary

Keep #61850 open. Current main still accepts Slack `file_share` messages and still resolves attached Slack media inside the awaited inbound preparation path, while Socket Mode receiver construction and Slack config do not expose the ping/pong timeout settings discussed in the issue. The related reports/PRs in the provided context remain open and do not prove this specific file-share regression is fixed or safely superseded.

## Best Possible Solution

Keep the issue open for a focused maintainer fix. The best path is to keep the Slack work in the bundled Slack plugin, add a regression test around file-bearing `message`/`file_share` handling, either defer media hydration outside the latency-sensitive Socket Mode path or otherwise prevent ping starvation, and separately consider the `clientPingTimeout`/`serverPingTimeout` config pass-through tracked by #14248/#41473.
