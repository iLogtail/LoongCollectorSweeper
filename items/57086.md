## Summary

Keep PR #57086 open. Current main already fixes the original gateway agent-handler freshness bypass by evaluating reset freshness before choosing the session id, but the PR later expanded to the same stale-sessionId forwarding pattern in `server-node-events`, and current main still forwards stored `sessionId` values for `voice.transcript` and `agent.request` alongside canonical `sessionKey`. That remaining node-event scope is not covered by a narrower canonical item in the provided context.

## Best Possible Solution

Keep the PR open for maintainer review or ask the contributor to rebase and narrow it to the still-missing `server-node-events` behavior: `voice.transcript` and `agent.request` should preserve canonical `sessionKey` while avoiding stale `sessionId` pins so downstream session resolution can roll daily/idle-stale sessions. The already-shipped gateway agent-handler freshness logic on main should be kept rather than replaced wholesale.
