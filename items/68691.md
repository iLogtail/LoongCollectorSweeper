## Summary

Keep open. Current main still creates Docker sandbox containers with `sleep infinity` as PID 1 and no sandbox init/reaper, while sandbox exec runs shell-heavy commands inside that container via `docker exec ... /bin/sh -lc`. The issue discussion has a concrete current observation that PID 1 is `sleep infinity` and that PPID=1 zombies persist; existing zombie-related code only handles PID liveness/stale locks/restart cleanup and does not reap orphaned sandbox children.

## Best Possible Solution

Keep the issue open and fix it in the core Docker sandbox lifecycle: run sandbox containers with a real init/reaper such as Docker `--init` or a bundled tiny init as PID 1, make existing containers recreate when the lifecycle command/options change, and add focused create-args/lifecycle tests plus read-only zombie reporting for diagnostics.

## Risks / Open Questions

- Changing the sandbox PID 1 behavior may require recreating existing containers and should be covered by config-hash/recreate behavior so users do not keep old `sleep infinity` containers silently.
- A health check should remain detection/reporting-only unless the new PID 1 can reap children; killing live orphans under `sleep infinity` can convert them into persistent zombies, as the issue comment notes.
