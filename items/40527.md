## Summary

Keep open. The issue body’s original root-cause claim is partly outdated on current `main` because the macOS runtime now implements `system.which`, but the underlying timeout report is still plausible: `main` still kicks off `refreshRemoteNodeBins()` immediately on every node connect, with a 15s probe timeout and no co-located-node skip or deferred probe path. The April 2 reproduction comment also points to a remaining connect-timing bug for macOS app-backed nodes, and there is a related open follow-up (#53759).
