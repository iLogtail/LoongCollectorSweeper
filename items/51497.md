## Summary

Keep #51497 open. The report is a recent, concrete WebChat transcript durability bug: token usage allegedly proves turns were processed, while the JSONL transcript starts 32-46 minutes after session creation. Current main has related mitigations for optimistic UI history reloads and a header-only transcript SessionManager quirk, but I did not find high-confidence evidence that the reported WebChat token-accounting/JSONL divergence is fixed. The KeepMyClaw comment is a useful external snapshot workaround, and #58088 is related but narrower to topic/thread session pointer eviction, so neither is a safe close reason for this core WebChat report.

## Best Possible Solution

Keep the issue open and narrow it to a current-main durability investigation. Best next path: reproduce with a WebChat-created session, page reload/reconnect, long-running turn, and header-only transcript cases; add a gateway/agent regression test proving accepted WebChat user turns and assistant replies are durable in the SessionManager JSONL before or during dispatch, including failure/abort paths; then either close #51497 with that fix or explicitly link a narrower canonical issue if #58088 or another item becomes the real remaining tracker.
