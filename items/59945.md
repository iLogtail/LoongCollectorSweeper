## Summary

Keep open. The reported runtime-snapshot/redaction failure remains plausible on current main: the active runtime snapshot setter still publishes the supplied config verbatim, while gateway channel startup reads the pinned runtime config and passes it into Slack account/startup resolution. I found no current regression test matching the issue comment’s proposed Slack botToken/appToken sentinel-leak scenario, and the latest release tag has the same `setRuntimeConfigSnapshot` behavior.

## Best Possible Solution

Keep #59945 open and fix it in the shared config/runtime layer rather than adding a Slack-specific workaround. The likely maintainer path is to make runtime snapshot publication restore redacted placeholders against the paired source snapshot, or otherwise reject/avoid publishing redacted snapshots before channels can consume them, then add a focused regression test that simulates redacted Slack `botToken`/`appToken` values in the runtime snapshot and proves `loadConfig()`/channel startup sees the real source/runtime values instead of `__OPENCLAW_REDACTED__`.
