## Summary

Keep open. Current main has adjacent Chrome MCP and node-browser improvements, including node routing for `profile="user"` through the browser tool and better timeout diagnostics, but the exact direct `openclaw nodes invoke ... browser.proxy` existing-session consent failure is not proven fixed. The direct node-host proxy still defaults to a 20s outer request timeout, while the higher-level browser tool now gives existing-session management calls 45s; no regression test covers the direct node proxy consent/attach path reported here.

## Best Possible Solution

Keep the issue open and either add a targeted fix or explicit verification for direct node-host `browser.proxy` existing-session attaches. A strong resolution would align the direct proxy default with the 45s existing-session management budget or document/pass `timeoutMs` for direct `nodes invoke`, then add a regression test around `/tabs?profile=user` through `runBrowserProxyCommand` that proves the node proxy preserves enough attach time and does not poison the Chrome MCP session.

## Evidence

There are no comments or maintainer follow-ups confirming resolution.
