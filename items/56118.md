## Summary

Keep open. Current `main` still intentionally supports `existing-session` / `profile="user"` through a connected browser node, but the Chrome MCP attach path still fails immediately on `client.connect()` / `listTools()` errors and only adds an 8s post-connect tab-readiness poll. That does not show a clear fix for the reported remote-node consent/`Connection closed` failure, and I found no stronger evidence on `main` that this issue was implemented away.

## Evidence

- **Connected browser-node support is still intentional on main:** Docs say existing-session can attach on the selected host or through a connected browser node, and tests assert `profile="user"` routes through `node.invoke` -> `browser.proxy` when a browser node is available.
  - file: [docs/tools/browser.md:478](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/docs/tools/browser.md#L478)

  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Initial Chrome MCP attach still fails fast on connection-close errors:** `createRealSession()` awaits `client.connect(transport)` and `client.listTools()`, and any error immediately becomes `BrowserProfileUnavailableError` with no retry/wait around the initial attach. That matches the issue's `McpError ... Connection closed` symptom more than a fixed flow.
  - file: [extensions/browser/src/browser/chrome-mcp.ts:238](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/browser/src/browser/chrome-mcp.ts#L238)

This wait happens after the MCP session is established, not before/around the initial consent-triggering attach.
