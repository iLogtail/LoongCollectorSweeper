## Summary

Keep open. Current main still lacks the PR's central Feishu `k:"update"` card-action flow: the decoder only accepts button/quick/meta, the card-action handler has no update branch or pending-update registry, and reply dispatch is not wired to patch the original card via `updateCardFeishu`. The PR discussion also contains unresolved latest-review concerns for broadcast dispatch and final card size handling, so this remains potentially useful but not closeable as implemented or superseded.

## Best Possible Solution

Keep the PR open. The best path is for the author or maintainer to rebase on current main, preserve the in-place Feishu card-update design if it is still desired, wire card-update context through both single-agent and broadcast reply dispatch, patch the original message via `updateCardFeishu`, handle long final payloads safely, and keep focused regression tests for decoding, processing-state patching, final patching, duplicate suppression, and broadcast mode.
