## Summary

Current main already contains the gateway-startup websocket-race fix that matches this report, and that fix was shipped in v2026.4.23. The memory CLI still depends on gateway-backed `secrets.resolve`, so the issue’s reported hang is consistent with the already-fixed startup race linked by the reporter.

## Evidence

- **Reporter follow-up linked the same known fix:** The issue’s own follow-up says the behavior looks identical to #43392 and specifically points to the gateway startup websocket race as the cause, with a plan to close once that fix shipped.
- **Gateway now attaches websocket handlers before binding:** Current main creates the `WebSocketServer` and attaches the upgrade handler before HTTP listen starts, explicitly to prevent the startup race where connections arrive before the upgrade handler exists.
  - file: [src/gateway/server-runtime-state.ts:172](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/gateway/server-runtime-state.ts#L172)
  - command: `nl -ba src/gateway/server-runtime-state.ts | sed -n '165,245p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Regression test covers the startup-race symptom:** Current main has a gateway regression test asserting that, during delayed startup, an immediate websocket upgrade probe gets `ECONNREFUSED` instead of hanging on a half-open socket, and then connects successfully once startup completes.
  - file: [src/gateway/server.startup-websocket-race.test.ts:49](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/gateway/server.startup-websocket-race.test.ts#L49)
  - command: `nl -ba src/gateway/server.startup-websocket-race.test.ts | sed -n '45,100p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Memory CLI uses gateway `secrets.resolve` on command startup:** `openclaw memory ...` first loads config via `resolveCommandSecretRefsViaGateway`, and that path calls the gateway `secrets.resolve` RPC with a 30s timeout. That makes the reported memory-command-only hang match the startup websocket race window described in #43392.
  - file: [extensions/memory-core/src/cli.runtime.ts:93](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/memory-core/src/cli.runtime.ts#L93)
  - command: `nl -ba extensions/memory-core/src/cli.runtime.ts | sed -n '70,110p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Gateway RPC details match the issue’s suspected path:** The shared CLI secret-resolution path calls `secrets.resolve` over the gateway RPC transport for command startup, which aligns with the reporter’s zero-output hang theory before the subcommand reaches its own output path.
  - file: [src/cli/command-secret-gateway.ts:688](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/cli/command-secret-gateway.ts#L688)
  - command: `nl -ba src/cli/command-secret-gateway.ts | sed -n '685,745p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Docs and release notes show this was shipped:** The memory CLI docs still note that this command path depends on gateway `secrets.resolve`, and the shipped v2026.4.23 changelog explicitly lists the gateway startup fix: delay HTTP bind until websocket handlers are attached (#43392).
  - file: [CHANGELOG.md:700](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/CHANGELOG.md#L700)
  - command: `nl -ba CHANGELOG.md | sed -n '695,705p'`

## Risks / Open Questions

- I did not run a live Linux-container reproduction in this read-only pass; the close relies on the implemented gateway-startup fix, the shipped release note, and the reporter’s own linkage to #43392.
- If there is a separate post-startup deadlock unrelated to websocket bind timing, it would need a fresh repro on v2026.4.23 or newer.
