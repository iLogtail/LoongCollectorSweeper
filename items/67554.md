## Summary

Keep open. Current main has not implemented or superseded this PR: new workspaces are still seeded from `docs/reference/templates`, those templates still hardcode file-backed memory behavior, and `memory-core`'s `buildPromptSection()` still only emits recall/tool guidance. The active memory plugin prompt seam already exists, so this remains an actionable in-repo template/memory-core boundary cleanup rather than ClawHub-only work.

## Best Possible Solution

Keep the PR open for maintainer review. The best product/code path is to decouple file-backed memory instructions from default workspace templates and move the default file-memory persistence guidance into `memory-core`'s active prompt builder, while keeping platform-wide safety and lesson-learning guidance in the workspace template. A small prompt/template test would make the ownership boundary harder to regress.
