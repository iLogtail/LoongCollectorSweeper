## Summary

Keep open. Current main partially addresses the Discord side by converting Markdown tables to code blocks, but the shared table code-block renderer still pads with UTF-16 string length, so the CJK display-width misalignment described in #55512 remains on the active Telegram and Discord delivery paths.

## Best Possible Solution

Keep #55512 open as the canonical focused bug. Implement a shared display-width helper for code-rendered Markdown tables, use it for width calculation and padding in `src/markdown/ir.ts`, and add regression tests with Chinese/Japanese/Korean table cells that exercise both Telegram HTML/code-block rendering and Discord `convertMarkdownTables` delivery paths.

## Evidence

- **Shared table renderer still uses code-unit length for column width:** `renderTableAsCode` computes each column width with `cell?.text.length` and pads with `widths[i] - (cell?.text.length ?? 0)`, which does not account for East Asian wide characters occupying two display columns.
  - file: [src/markdown/ir.ts:521](https://github.com/openclaw/openclaw/blob/7e376e5aba32eeea8191e70d5e43e293f8b8398a/src/markdown/ir.ts#L521)
  - command: `nl -ba src/markdown/ir.ts | sed -n '400,585p'`
  - sha: [7e376e5aba32](https://github.com/openclaw/openclaw/commit/7e376e5aba32eeea8191e70d5e43e293f8b8398a)
- **Telegram uses the shared table-mode renderer for outbound HTML:** Telegram resolves `markdown.tables` for the account and passes that table mode into `renderTelegramHtmlText`; `markdownToTelegramHtml` then feeds the mode into the shared Markdown IR renderer and maps code blocks to `<pre><code>...`.
  - file: [extensions/telegram/src/send.ts:621](https://github.com/openclaw/openclaw/blob/7e376e5aba32eeea8191e70d5e43e293f8b8398a/extensions/telegram/src/send.ts#L621)
