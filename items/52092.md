## Summary

Keep open. Current main has meaningful cron mitigations for startup catch-up, regular run concurrency, and cron-run session retention, but the central OOM report is not resolved: daemon installs still do not set a heap-size default/config, and isolated cron runs can still persist full session payloads under both base and per-run session entries.

## Best Possible Solution

Keep #52092 open and treat it as the cron-triggered gateway OOM tracker. The best path is to either land a reviewed daemon heap-headroom/config solution such as #42853 or an equivalent, keep the current cron catch-up/concurrency/reaper safeguards, and reduce isolated cron persisted payload size by trimming or deduplicating skillsSnapshot/system-prompt state in per-run session entries. Cross-link #55334 for the broader sessions.json OOM work rather than closing this cron-specific report prematurely.
