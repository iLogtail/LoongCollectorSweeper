## Summary

Keep #70014 open. Current main has multi-workspace dreaming fanout, but the workspace resolver still only enumerates `agents.list` and falls back to the default agent workspace only when the list is empty. That leaves the reported primary workspace gap plausible for configs where AGI/sub-agent workspaces are listed but the primary `agents.defaults.workspace`/main workspace is not represented as an `agents.list` entry. The existing tests cover configured agent fanout and empty-list fallback, but not the reported default-plus-subagents shape.

## Best Possible Solution

Keep the issue open and fix the memory dreaming workspace discovery path. The likely code path is `resolveMemoryDreamingWorkspaces`: include the primary/default workspace in addition to configured agent workspaces, dedupe by normalized path, preserve `agentIds`, and add a regression test for `agents.defaults.workspace` plus AGI/sub-agent entries where `main` is not in `agents.list`. Also verify `dreaming-phases` session ingestion still maps the correct agent IDs after the resolver changes.

## Risks / Open Questions

- A fix needs maintainer judgment on whether `agents.defaults.workspace` should always be included as a memory workspace when `agents.list` is nonempty but lacks `main`, since `resolveDefaultAgentId` currently treats the first listed agent as the default when no entry has `default: true`.
- Changing enumeration could process an additional workspace in existing multi-agent installs, so the regression test should cover deduping when the primary/default workspace is already listed.
