## Summary

#52477 is a valid current-main bug, but the same remaining work is already tracked by open PR #38290, "Gateway: allow extension origins in browser allowlist." Current main still normalizes `chrome-extension://...` through `URL.origin` and compares the resulting `"null"` against `gateway.controlUi.allowedOrigins`; PR #38290 is the active canonical fix path with the same problem statement, exact allowlist behavior, and regression-test scope. Earlier PR #52531 also linked #52477 directly but is closed unmerged, so #38290 is the useful tracker to follow.

## Best Possible Solution

Close #52477 as superseded by open PR #38290 and keep review/implementation focused there. The best fix remains a narrow change in `src/gateway/origin-check.ts` that preserves exact opaque origins for matching while keeping literal `Origin: null` rejected, wildcard semantics unchanged, and regression coverage for allowlisted and non-allowlisted extension origins.

## Evidence

  - file: [src/gateway/origin-check.ts:45](https://github.com/openclaw/openclaw/blob/9a529ca78bfa0e257f662f0e70ab8c681caa6086/src/gateway/origin-check.ts#L45)
