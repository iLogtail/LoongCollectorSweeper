## Summary

Keep open. Current main still has the core behavior reported in #70333: plugin/channel auto-enable materializes `enabled: true` into the config object, gateway startup persists that auto-enabled config back to `openclaw.json`, and doctor repair mode applies the same auto-enable mutation. A later startup hash guard appears to mitigate one restart-loop path, but it does not satisfy this issue's central request to keep auto-enable runtime-only and stop reintroducing `channels.<id>.enabled` / `plugins.entries.<id>.enabled` keys.

## Best Possible Solution

Keep #70333 open and fix auto-enable as a runtime overlay. Gateway startup should activate configured built-in channels/providers from the auto-enabled runtime config without writing those inferred `enabled` flags to disk. Doctor may report auto-enable reasoning, but `doctor --fix` should not persist auto-enable-only changes. Add regression coverage for removing `channels.whatsapp.enabled`, `channels.telegram.enabled`, and `plugins.entries.openai.enabled` from config, then proving startup/doctor do not reintroduce them while runtime activation still works.
