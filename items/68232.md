## Summary

Keep open. Current main does not appear to restart Telegram for unrelated config keys, but it still treats `channels.telegram.*` edits such as `channels.telegram.historyLimit` as a Telegram channel reload, and that reload still recreates the bot/account runtime. The per-bot `groupHistories` buffer is initialized in memory during bot creation, so the reported state-loss problem remains. A recent maintainer comment also explicitly left this open after PR #71465 as a Telegram-specific hot-reload/state-retention issue.

## Best Possible Solution

Keep #68232 open. The best product/code path is to add Telegram-aware reload/state retention: dynamically re-read safe Telegram settings such as group/topic/routing/history policy where possible, preserve or externalize per-account `groupHistories` across required bot recreation, and coordinate with the existing account-scoped reload work (#43935/#43938) so account-local changes only restart the affected account. Add regression coverage showing a `channels.telegram.historyLimit` or equivalent safe edit does not discard active Telegram group history.
