## Summary

Keep #13440 open. Current main still models `session.store` as a filesystem path, resolves it to `sessions.json`, and routes load/save/update/readUpdatedAt through path-based helpers and filesystem locks. Plugins can read/write the current file-backed store through runtime helpers, but there is no session-store backend adapter, driver-shaped config, backend registry, or plugin hook that lets Convex, SQLite, Redis, Postgres, MongoDB, or a custom module own session persistence. The related memory-backend discussions and PRs show demand, but they are memory scoped and do not supersede the session-store persistence request.

## Best Possible Solution

Keep #13440 open until maintainers either add a core `SessionStoreAdapter` contract or expose a stable plugin API that lets external backends own session persistence. The path should keep the current file backend as the default, add driver-shaped `session.store` config/types/help/docs, route load/save/update/readUpdatedAt/session discovery through the abstraction, and add contract tests for the file adapter plus a mocked non-file adapter.
