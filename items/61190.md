## Summary

Keep PR #61190 open. Current main still has the reported resolver behavior: directory matches return the initially inferred/preferred kind instead of the matched entry's `kind`, and main does not include the PR's regression tests. The PR is a focused, plausible fix, though maintainers should resolve the `channel` vs `group` normalization concern before landing.

## Best Possible Solution

Keep the PR open and either land a variant that uses the matched directory entry kind for `user` and selected ambiguous entries, or request a narrow revision that preserves `channel` semantics where downstream routing needs them. The fix belongs in `src/infra/outbound/target-resolver.ts` with focused regression coverage in `src/infra/outbound/target-resolver.test.ts`.

## Evidence

- **Current resolver still preserves guessed kind for single directory match:** On current main, `resolveMessagingTarget` computes `kind` before directory lookup, queries peers/groups from that guess, and returns that same `kind` for a single directory entry match. It does not inspect `entry.kind`, so a `listGroups` result whose entry is `kind: "user"` would still resolve as the guessed group/channel kind.
