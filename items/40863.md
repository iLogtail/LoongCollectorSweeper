## Summary

Keep open. Current `main` does not cleanly resolve the reported `sessions_send` timeout behavior: subagent announce delivery now defaults to `120000` ms, but the A2A wait path still hard-clamps `waitForAgentRun()` to `60000` ms, so the issue's core failure mode remains plausible on current code.

## Evidence

- **Default announce timeout on current main is 120s:** `DEFAULT_SUBAGENT_ANNOUNCE_TIMEOUT_MS` is `120_000`, and `resolveSubagentAnnounceTimeoutMs()` returns that value when no config override is set.
  - file: [src/agents/subagent-announce-delivery.ts:48](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/agents/subagent-announce-delivery.ts#L48)
  - command: `nl -ba src/agents/subagent-announce-delivery.ts | sed -n '40,70p'`
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **`sessions_send` A2A path still caps the initial wait at 60s:** `runSessionsSendA2AFlow()` calls `waitForAgentRun()` with `timeoutMs: Math.min(params.announceTimeoutMs, 60_000)`, so configured/default announce timeouts above 60s are not fully honored in this path.
  - file: [src/agents/tools/sessions-send-tool.a2a.ts:50](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/agents/tools/sessions-send-tool.a2a.ts#L50)
