## Summary

Keep #70853 open. Current main still rotates/rebinds live reply-run sessions to a fresh sessionId without preserving recent user/assistant transcript context, and the directly linked fix PR #70898 is still open/unmerged in the provided GitHub context.

## Best Possible Solution

Keep the issue open and use PR #70898, or an equivalent maintainer-reviewed patch, to preserve a bounded tail of recent user/assistant transcript records across silent in-reply session rotations while leaving explicit /new and /reset behavior fresh. Add regression coverage that proves a Telegram/direct-message stable session key retains recent context after resetReplyRunSession rotates to a new sessionId.

## Evidence

- **Silent reply-run reset still creates a new session without transcript seeding:** resetReplyRunSession copies the previous SessionEntry, assigns a new sessionId, resolves a new transcript path, updates the session store and queued run, then returns. The function does not read the previous transcript or write recent user/assistant turns into the new transcript before continuing.
