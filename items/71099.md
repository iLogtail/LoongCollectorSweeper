## Summary

Keep #71099 open. The issue is recent, has a maintainer comment explicitly saying not to close yet, and current main still lacks a stack-backed root cause or a Telegram regression test for U+25C6/◆. Source inspection shows several Telegram delivery paths that forward rendered text to grammY, including DM draft streaming, but no evidence that main has fixed the reported ByteString failure boundary.

## Best Possible Solution

Keep the issue open and ask for the stack frame around the ByteString throw or a minimal current-main repro that sends a response beginning with ◆ through the Telegram reply path. Once confirmed, add focused regression coverage for the implicated Telegram path, likely normal final delivery and/or DM draft streaming, and fix the exact boundary that is treating user-visible text as a byte-only header/token value. Do not strip Unicode from assistant replies globally.
