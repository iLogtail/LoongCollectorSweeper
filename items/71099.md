## Summary

Keep #71099 open. The issue is recent, has a maintainer comment explicitly saying not to close yet, and current main still lacks a source-level fix, regression test, or stack-frame evidence proving the exact boundary. The Telegram text delivery path still forwards rendered Unicode text through grammY sendMessage; dependency inspection supports the maintainer’s point that this ByteString error is more likely from a header/method/token-like field than normal JSON message body text, so closing as implemented or cannot-reproduce would be premature.

## Best Possible Solution

Keep the issue open and ask for the stack frame around the ByteString throw or a minimal current-main repro that sends a response containing ◆ through the Telegram reply path. Once confirmed, fix the exact boundary that is placing non-Latin1 text into a ByteString-constrained field, and add a Telegram regression test that covers ◆ in normal outbound delivery without stripping valid Unicode.
