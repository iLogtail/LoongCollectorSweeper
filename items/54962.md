## Summary

Keep open. The PR’s one-line fallback rewrite is not safe to land on current `main`: it would break existing Telegram callback buttons that intentionally carry slash-command payloads such as `/models add`, because command routing only recognizes text that still starts with `/`.

## Evidence

- **Current fallback preserves raw callback text unless it is an explicit `tgcmd:` native-command callback:** `registerTelegramHandlers` forwards `nativeCallbackCommand ?? data` into the synthetic message body, and only marks the callback as native when `parseTelegramNativeCommandCallbackData(data)` succeeds.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1762](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/telegram/src/bot-handlers.runtime.ts#L1762)
  - command: `nl -ba extensions/telegram/src/bot-handlers.runtime.ts | sed -n '1762,1770p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Telegram has a dedicated encoded path for callback-based native commands:** `parseTelegramNativeCommandCallbackData` only accepts `tgcmd:` payloads and only returns a command when the decoded text starts with `/`. That is the existing mechanism for command-style callback routing.
  - file: [extensions/telegram/src/bot-native-commands.ts:206](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/telegram/src/bot-native-commands.ts#L206)

- **Existing Telegram UI emits raw slash-command callback payloads:** The models menu builds inline buttons with `callback_data: "/models add"` and `callback_data: "/models add <provider>"`. Those callbacks rely on the fallback path preserving the leading slash.
  - file: [extensions/telegram/src/command-ui.ts:12](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/telegram/src/command-ui.ts#L12)

## Risks / Open Questions

- The underlying user report may still describe a real gap for non-command callback payloads, but this PR does not separate those from existing slash-command callbacks.
- Merging the PR as written would likely regress Telegram inline flows that depend on raw slash-command callback_data, especially the built-in `/models add` buttons.
- The current test coverage does not pin the exact fallback format for non-native callbacks, so either regression direction could slip without a more specific test.
