## Summary

Current main contradicts the issue premise: announce-mode isolated cron jobs still have direct chat-send capability, and announce is only the runner fallback if the agent did not already send. The reported "no Slack API access in announce mode" behavior is obsolete on current main.

## Close Comment

Closing from a Codex review. On current `main` (`7130e56f872aea96577ac9fcbda0c7555a9ecc69`), the issue’s core premise no longer matches OpenClaw: announce-mode isolated cron jobs are documented and implemented as fallback delivery, not as a mode that removes messaging access. `docs/automation/cron-jobs.md:133` and `docs/help/faq.md:265` both state that isolated cron jobs can still use the `message` tool and that announce only covers runner fallback delivery. The runtime also keeps the message tool enabled for `delivery.mode="announce"` in `src/cron/isolated-agent/run.ts:285`, and forwards target/thread routing into the embedded run in `src/cron/isolated-agent/run-executor.ts:155`. Coverage in `src/cron/isolated-agent/run.message-tool-policy.test.ts:408` and `extensions/slack/src/action-runtime.test.ts:451` backs up the direct-send and threaded-send path. Given that current main does provide direct messaging access in announce-mode cron runs, the reported failure mode is not reproducible against today’s code/docs contract.
