## Summary

Keep open. Current `main` has partial protections against Telegram retry storms, but it does not implement the requested per-channel auth-failure circuit breaker/pause behavior across channel polling/runtime. The only explicit circuit breaker I found is Telegram-specific and limited to `sendChatAction`; the broader channel lifecycle still relies on generic restart backoff/attempt caps rather than pausing a broken channel after repeated 401/403 auth failures.

## Evidence

- **Telegram circuit breaker is narrow, not channel-wide:** `extensions/telegram/src/sendchataction-401-backoff.ts` adds exponential backoff and suspension only for `sendChatAction` after consecutive 401s, and `extensions/telegram/src/bot-core.ts` wires that handler into message contexts. That does not cover polling, webhook setup, or general Telegram API calls.
  - file: [extensions/telegram/src/sendchataction-401-backoff.ts:34](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/extensions/telegram/src/sendchataction-401-backoff.ts#L34)
  - command: `nl -ba extensions/telegram/src/sendchataction-401-backoff.ts | sed -n '30,145p'`
  - sha: [4e4aeacae4a8](https://github.com/openclaw/openclaw/commit/4e4aeacae4a87f2fe09125d462d88cd71a18cb66)
- **Gateway has generic restart backoff, not an auth-failure breaker:** `src/gateway/server-channels.ts` backs off crashed channels and gives up after 10 restart attempts, but this is a generic lifecycle restart policy. It is not a per-channel auth-failure counter with pause/re-enable semantics, configurable threshold, or 401/403-specific handling.
  - file: [src/gateway/server-channels.ts:22](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/src/gateway/server-channels.ts#L22)

- **Invalid-token probe result does not stop Telegram startup:** `extensions/telegram/src/channel.ts` probes the bot token before startup, but it only uses a successful probe to decorate the log label. Regardless of `probe.ok`, it still logs `starting provider` and starts the monitor.
  - file: [extensions/telegram/src/channel.ts:883](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/extensions/telegram/src/channel.ts#L883)

- **Polling path still uses long retry windows and only special-cases network/conflict errors:** `extensions/telegram/src/monitor.ts` configures grammY polling with `maxRetryTime: 60 * 60 * 1000`, while `extensions/telegram/src/polling-session.ts` only restarts/backs off for network errors or `getUpdates` conflicts. I did not find a per-channel 401/403 auth breaker in the polling path.
