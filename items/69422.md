## Summary

Keep #69422 open. Current main still has the reported Google Chat thread-routing gap: ingress records the Google Chat thread only as `ReplyToId`, streamed block replies still default their `replyToId` from the inbound message resource name, and queued followups still read `MessageThreadId`, which Google Chat never populates. The newer Google Chat delivery helper correctly sends chunks in-thread only when it is already handed a thread-name `replyToId`, so it does not close the upstream routing bug.

## Best Possible Solution

Keep the issue open and fix the actual routing contract. Google Chat should populate durable thread metadata at ingress, either by setting an appropriate `MessageThreadId` for `message.thread.name` or by adding explicit Google Chat thread fields to the outbound/reply payload contract. That metadata then needs to feed streamed block reply fallback, coalescer identity/conflict checks, `FollowupRun.originatingThreadId`, and the Google Chat delivery adapter. Add regression coverage for long streamed replies, queued followups from a thread, and `messages.queue.byChannel.googlechat` validation if that mitigation is accepted.
