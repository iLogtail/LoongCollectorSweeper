## Summary

Keep open. Current main has better bundled vLLM support and an adjacent fix that classifies bare `404 status code (no body)` errors as `model_not_found`, but I did not find high-confidence evidence that the reported existing `local-vllm` / vllm-mlx GLM 4.7 regression is fixed. The linked fix PRs were closed unmerged, the reporter’s follow-up makes the proposed root cause inconclusive, and current docs still give conflicting API guidance for vLLM/custom local proxies.

## Best Possible Solution

Keep #50719 open until maintainers either reproduce the reporter’s existing `local-vllm`/vllm-mlx GLM 4.7 setup on current main or add a targeted regression test/repair path. The likely cleanup path is to decide the canonical adapter for vLLM/custom `/v1` local proxies, align `docs/providers/vllm.md` and `docs/gateway/local-models.md`, and, if old configs were left in a bad API mode, add doctor/onboarding repair rather than relying only on fallback classification.
