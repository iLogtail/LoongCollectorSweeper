## Summary

Current main already loads local skills from `~/.openclaw/skills` for runtime visibility and local list/info/check flows, but the requested searchable registration/indexing path is still missing. The gateway and CLI `skills.search` surface remains ClawHub-only, with no `register` command or API endpoint for local skill directories.

## Best Possible Solution

Keep this open. The best fix is a focused core CLI/gateway design for local skill discoverability: reuse the existing local skill loader, add tests for `~/.openclaw/skills` results, extend the protocol/result shape if local source/file path metadata is needed, and update the CLI/docs. If maintainers want search to remain ClawHub-only, the issue should be resolved explicitly with documented guidance rather than auto-closed.

## Risks / Open Questions

- The issue says `skill_search` while current code names the gateway method `skills.search`; this appears to be naming drift, not evidence of a separate implemented local-search path.
- A product/API choice is still needed: either merge local skill results into `skills.search` or add a separate local search/register surface so ClawHub search semantics stay clean.
