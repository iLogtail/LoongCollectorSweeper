## Summary

Keep PR #68327 open. Current main overlaps with parts of the PR, but the central auth-store startup-latency issue is not fully implemented on main: the resolver path still reloads through `loadAuthProfileStoreForSecretsRuntime`, which still overlays external auth profiles via provider plugin resolution. The PR also should not be merged as-is because the provided latest PR diff still includes unconditional production profiling logs and bundles several independent changes that need rebase and focused validation.

## Best Possible Solution

Keep the PR open for maintainer judgment, but do not merge it as-is. The best path is to rebase on current main, remove or gate the production profiling logs, split unrelated Windows mDNS/async-lock/UI-spawn changes if needed, and add focused coverage proving the remaining resolver-path SecretRef/auth-store startup flow avoids provider plugin loading without breaking external OAuth recovery.

- The reported startup timing improvement is plausible from source inspection, but this read-only review did not run startup benchmarks or tests.
