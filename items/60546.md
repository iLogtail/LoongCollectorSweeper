## Summary

Current main still leaves the reported built-in `microsoft-foundry` bug open. The provider discovers and persists all succeeded Foundry deployments, but its API type only allows OpenAI-compatible `openai-completions`/`openai-responses`, URL building always returns `/openai/v1`, and connection/runtime auth paths only target `/chat/completions` or `/responses`. Microsoft’s current Claude-on-Foundry docs require the Anthropic base URL and Messages endpoint, matching the reporter’s workaround and making this a real remaining bundled-provider issue rather than a stale or ClawHub-only request.

## Best Possible Solution

Keep #60546 open. The best fix is provider-owned: either add first-class Anthropic/Claude detection in `extensions/microsoft-foundry` that persists `api: "anthropic-messages"`, stores/builds the `/anthropic` base URL, uses `x-api-key` or Entra bearer auth with `anthropic-version`, and tests `/v1/messages`; or add an explicit onboarding/config guardrail that filters or warns on Claude deployments until that support exists. The existing custom-provider workaround should remain documented as a workaround, not a reason to close the bundled provider bug.
