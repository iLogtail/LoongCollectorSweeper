## Summary

Keep open. Current main does force UTF-8 for the transcript headers and some direct transcript file writes, but the actual message append path still delegates to `SessionManager.appendMessage(...)` from the external `@mariozechner/pi-coding-agent` package. That package source is not present in this checkout, there is no local patch for it, and I did not find a Windows/CJK transcript regression test in-tree. The repo also has a separate known Windows GBK/UTF-8 console-output problem, so the report could be related but is not proven duplicate or fixed from the evidence available here.
