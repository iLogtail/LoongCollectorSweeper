## Summary

Keep #56106 open. Current main does not support the issue's suggested GBK transcript-writer root cause: session and memory paths are explicitly UTF-8 or go through the pi-coding-agent SessionManager, whose current package writes JS strings/default UTF-8. But there is still no verified QQBot Windows/CJK transcript roundtrip fix or regression test on main, and QQBot ingress currently decodes gateway buffers as UTF-8 and passes `event.content` through. The related GBK items in the provided context are about Windows process output or web_fetch charset handling, not QQBot transcript persistence.

## Best Possible Solution

Keep the issue open. The best maintainer path is to add a narrow QQBot Windows/CJK transcript reproduction that captures the raw gateway payload, decoded `event.content`, final `agentBody`, and persisted JSONL, then fix the first corrupting boundary. If the payload is already corrupted before OpenClaw receives it, document the provider/upstream constraint; if OpenClaw corrupts it, fix the QQBot ingress decoder or transcript append path and add a regression test.
