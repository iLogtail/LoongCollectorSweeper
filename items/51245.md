## Summary

Keep #51245 open. Current main fixes Telegram native slash metadata persistence for newly handled slash commands, but the reported diagnostic path `openclaw sandbox explain --session telegram:slash:<id>` still remains unresolved: bare colon session keys are returned unchanged, and provider inference only handles agent-scoped keys. The current sandbox explain test file also lacks a regression case for the Telegram slash key documented in the issue.

## Evidence

- **Bare slash key is not normalized:** `normalizeExplainSessionKey` returns any raw session containing `:` unchanged, so `telegram:slash:<id>` is not wrapped into an agent-scoped key before lookup or inference.
