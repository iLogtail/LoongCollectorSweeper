## Summary

Keep open. Current main documents the Tailscale Serve trust boundary and provides `gateway.auth.allowTailscale: false` to force token/password instead of Tailscale header auth, but it still does not implement an optional secondary-auth mode that requires both verified Tailscale/trusted-proxy identity and a shared token/password.

## Best Possible Solution

Keep the issue open as a focused enhancement. The best implementation would be a backwards-compatible Gateway auth policy that explicitly requires verified identity plus shared-secret auth for Tailscale Serve and/or trusted-proxy requests, with schema/types updates, `authorizeGatewayConnectCore` enforcement, tests for success/failure ordering and rate limiting, and docs/security-audit updates explaining the new mode versus `gateway.auth.allowTailscale: false`.

## Risks / Open Questions

- The issue body uses `trusted-proxy` terminology for a Tailscale Serve flow, while current main models Serve header auth through `gateway.auth.allowTailscale`; maintainers should clarify whether the eventual feature is Tailscale-only, trusted-proxy-only, or a generic identity-plus-shared-secret policy.
- Docs and audit output already cover much of the visibility problem, so the remaining feature should be scoped narrowly to avoid weakening the existing simple `allowTailscale: false` hardening path.
