## Summary

Keep #55768 open. Current main has a lightweight `/health`/`/healthz` probe and tests proving it is served before heavier request stages, but the issue's actionable gap remains: public docs do not tell external uptime monitors to use `GET /health` instead of `/v1/chat/completions`, and the OpenAI-compatible endpoint still generates a fresh session key for requests without `x-openclaw-session-key` or `user`. The referenced PR #55785 was closed unmerged, and #62436 is a broader open passthrough-mode request rather than a replacement for this deployment guidance/warning item.

## Best Possible Solution

Keep the issue open and implement the narrow product fix: update `docs/gateway/health.md` and any deployment examples to recommend `GET` or `HEAD /health` or `/healthz` for external uptime checks, explicitly warn that `/v1/chat/completions` is a full agent endpoint that creates sessions and can call an LLM, and consider a small warning/diagnostic in `src/gateway/openai-http.ts` for repeated anonymous session creation. Leave broader true-stateless/passthrough API work to #62436 rather than treating it as the canonical fix for health-monitor misuse.
