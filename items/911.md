## Summary

本 Issue 请求为采集任务（如 File2Kafka）增加运行时暂停与恢复能力，以应对流量高峰期的负载调控。经核查当前主线架构与控制面代码，Pipeline 生命周期仍由配置加载/卸载驱动启停，缺乏独立的 Pause/Resume 状态机、上下文冻结机制及对应的控制面 API。该需求涉及核心调度逻辑改造，无法通过插件系统或外部扩展独立承载，且讨论区与关联记录中未见已合并实现或重复工单。作为明确且有效的核心增强请求，应保持开启以待架构设计与路线规划。

## Best Possible Solution

保持开启。建议维护者或贡献者补充详细设计方案（RFC），明确暂停/恢复的触发入口（如动态配置标记、REST API 或系统信号）、状态机转换边界、Buffer/Checkpoint 保留策略及恢复时的 Offset 衔接逻辑。可评估是否通过扩展现有流控/背压机制间接满足场景，或作为 Pipeline 控制面新特性纳入核心开发路线。经核心维护者评审设计后可分配里程碑跟进。
