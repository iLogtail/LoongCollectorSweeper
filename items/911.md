## Summary

本 Issue 提出了明确的核心功能需求：为采集任务（Pipeline）增加运行时暂停与恢复能力，以应对流量高峰。经核查，当前主线架构仍为配置驱动的启停模型，缺乏独立的 Pause/Resume 状态机与控制面 API。该需求涉及核心调度器与上下文保留策略改造，无法通过插件系统实现，且无重复或已合并的替代方案，属于有效的核心增强请求，应保持开启以待架构设计与路线规划。

## Best Possible Solution

保持开启。建议维护者或贡献者补充详细设计方案（RFC），明确暂停/恢复的触发入口（如 REST API、系统信号或配置标记）、上下文保留策略（Buffer/Checkpoint 冻结机制）及恢复时的 Offset 衔接逻辑。可评估是否通过扩展现有动态配置机制间接满足，或作为 Pipeline 控制面新特性纳入核心开发路线。经核心维护者评审设计后可分配里程碑跟进。

## Evidence

- **core_lifecycle_gap:** 当前 Pipeline 生命周期仅支持 Start/Stop/Reset 等状态转换，任务启停完全由 ConfigManager 动态配置驱动。移除或禁用配置会触发 Pipeline 销毁与重建，缺乏独立的 Pause/Resume 接口与状态机，无法在保留采集上下文（如 Offset、Buffer）的前提下实现任务挂起。
