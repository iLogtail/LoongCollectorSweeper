## Summary

本 Issue 请求为 File2Kafka 采集任务增加运行时暂停与恢复能力，用于流量高峰期的负载调控。经核查当前 main 分支代码与架构设计，Pipeline 生命周期仍为配置驱动的 Start/Stop/Reset 模型，缺乏独立的 Pause/Resume 控制面 API 与状态机。该特性涉及核心调度器改造与上下文保留策略，具有明确业务价值，且无现有等效实现、已合并 PR 或插件替代方案，故保持开启以待核心路线规划或 RFC 设计。

## Best Possible Solution

保持开启。建议维护者或贡献者补充详细设计方案（RFC），明确暂停/恢复的触发入口（如 REST API、系统信号或配置标记）、上下文保留策略（Buffer/Checkpoint 冻结机制）及恢复时的 Offset 衔接逻辑。可评估是否通过扩展现有动态配置机制（如新增 paused 字段触发内部状态机挂起而非销毁）间接满足，或作为 Pipeline 控制面新特性纳入核心开发路线。经核心维护者评审设计后可分配里程碑跟进。

## Evidence

- **pipeline_lifecycle_gap:** 当前 Pipeline 生命周期仅支持 Start/Stop/Reset 等状态转换，任务启停完全由 ConfigManager 动态配置驱动。移除或禁用配置会触发 Pipeline 销毁与重建，缺乏独立的 Pause/Resume 接口与状态机，无法在保留采集上下文（如 Offset、Buffer）的前提下实现任务挂起。
