## Summary

本 Issue 请求为 File2Kafka 采集任务增加运行时暂停与恢复能力，用于流量高峰期的负载调控。需求具有实际运维价值，但当前描述较为简略，未明确触发方式（如 REST API、配置热更新或外部信号）及 Checkpoint/Offset 状态管理策略。当前核心架构主要依赖动态配置下发实现任务的启停，尚无专用的 Pause/Resume 控制面 API。该功能涉及 Pipeline 调度与插件状态机改造，需进一步设计方案或纳入核心路线规划。

## Best Possible Solution

保持开启。建议补充详细设计方案，明确暂停/恢复的触发入口、上下文保留策略及恢复时的 Offset 衔接逻辑。维护者可评估是否通过现有动态配置机制（如修改 enable 字段或移除配置）间接满足，或作为 Pipeline 控制面新特性进行规划。若确认该功能暂不纳入核心路线或已有等效替代方案，可补充说明具体路径后关闭。

## Evidence

- **architecture_gap:** 当前核心 Pipeline 生命周期由配置管理器驱动（Start/Stop/Reset），缺乏独立的 Pause/Resume 状态机与控制面 API。暂停需求目前需通过配置变更间接实现，无法直接满足保留完整运行上下文并快速恢复的场景。
