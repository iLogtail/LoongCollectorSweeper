## Summary

Keep #70895 open. Current main still has the reported behavior: hook agent requests are dispatched as isolated cron-style agent turns with a sessionKey and externalContentSource, but the cron runner still forces a new session for every isolated run, so the lower-level session reuse path is bypassed. The PR is a focused one-commit fix for open issue #70894 and is not implemented or superseded on main.

## Best Possible Solution

Rebase or land a narrow equivalent of #70895: compute forceNew from true scheduled isolated cron semantics rather than all isolated agent turns, allow webhook/Gmail hook-sourced runs to reuse a fresh session entry, and add regression coverage that direct /hooks/agent and mapped hook session keys pass forceNew: false while ordinary isolated cron still passes forceNew: true.
