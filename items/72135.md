## Summary

Keep open. Current main still has the behavior this draft PR is trying to change: `resolveCronPayloadOutcome` treats any error payload as fatal unless a later non-error text payload exists, and there is no `===DONE_OK===` or equivalent sentinel path. That fatal flag directly drives isolated cron run status, persisted job state, and failure notifications, so the PR is not obsolete or already implemented.

## Best Possible Solution

Keep PR #72135 open for maintainer review. The likely implementation belongs in `src/cron/isolated-agent/helpers.ts` with focused tests in `src/cron/isolated-agent.helpers.test.ts`; maintainers should settle the success-override contract before landing.

## Risks / Open Questions

- The remaining product question is the contract shape: maintainers should decide whether a literal `===DONE_OK===` token, a cron-specific token, a config gate, or an explicit outcome signal is the right success override.
- A global sentinel could hide real post-answer tool failures if prompts emit it too eagerly, so regression coverage should include run-level-error precedence and no-sentinel fatal behavior.
