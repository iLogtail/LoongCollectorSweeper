## Summary

Keep open. Current `main` does not implement this PR’s transport-failure retry behavior: memory batch embeddings still retry only rate-limit/HTTP-style errors, and there is no current recursive batch-splitting path or coverage for transient `fetch failed`-style failures.

## Evidence

- **Current batch retry path has no transport-error branch or batch splitting:** `embedBatchWithRetry()` delegates straight into `runMemoryEmbeddingRetryLoop(...)` and returns its result; there is no `fetch failed` matcher and no recursive split/retry path in the function body.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:234](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/extensions/memory-core/src/memory/manager-embedding-ops.ts#L234)
  - command: `nl -ba extensions/memory-core/src/memory/manager-embedding-ops.ts | sed -n '220,330p'`
  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **Retry classifier still excludes transient transport failures:** `isRetryableMemoryEmbeddingError()` only matches `rate limit`, `429`, `5xx`, `cloudflare`, and `tokens per day`. It does not match `fetch failed`, `ECONNRESET`, `socket hang up`, or similar network transport errors from the PR.
  - file: [extensions/memory-core/src/memory/manager-embedding-policy.ts:83](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/extensions/memory-core/src/memory/manager-embedding-policy.ts#L83)
