## Summary

Current main still has the reported hook-to-cron path: hook agent dispatch creates an isolated cron job, cron preparation converts that to forceNew, and CLI execution suppresses stored CLI session reuse whenever the cron session is new. The linked PR #51337 appears to be the focused open fix for #51208, but it is not merged into this checkout, so the issue should remain open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep #51208 open and use #51337, or an equivalent maintainer-reviewed patch, as the implementation path. The fix should add an explicit hook continuation/sessionTarget control through hook config, transforms, dispatch, replay scope, docs, and regression tests proving a second stable hook event invokes `codex exec resume <thread_id>` while true isolated runs still clear stale CLI identity.
