## Summary

Keep open. The report is recent, unprotected only by label/author rules, and current main does not clearly fix the specific failure mode: WhatsApp WebSocket login still passes Baileys an ambient `proxy-agent` instance for the WSS `agent`, which is the exact path the reporter says times out. The newer undici dispatcher path only covers Baileys upload/media `fetchAgent`, not the WebSocket opening handshake.

## Best Possible Solution

Keep the issue open and fix the WhatsApp WSS login proxy path directly. The likely maintainer path is to add a WhatsApp WebSocket proxy resolver that converts `HTTPS_PROXY`/`https_proxy` or fallback `HTTP_PROXY`/`http_proxy` into an explicit `HttpsProxyAgent` for Baileys `agent`, while preserving `NO_PROXY` semantics and existing unsupported-proxy behavior. Add focused tests that distinguish the WebSocket `agent` from upload `fetchAgent` and prove the WSS path no longer uses the ambient `proxy-agent` object that this report identifies as failing.
