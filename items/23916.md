## Summary

Keep #23916 open. Current main still has the same gap called out in the April 26 Codex review comment: pairing metadata is persisted and returned through the raw device pairing RPC, but the human `openclaw devices list` table still does not show client purpose/mode, platform/device family, host/user-agent-style details, editable notes/names, or a `--full` detail view.

## Best Possible Solution

Keep #23916 open. The best path is to add a user-facing device detail workflow: show compact purpose metadata in `openclaw devices list`, add a `--full` or describe-style view for client ID/mode, platform/device family, timestamps, and safe host/IP details, and coordinate editable names/notes with the paired-device naming work instead of treating the storage-layer metadata as sufficient.
