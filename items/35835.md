## Summary

Current main still does not implement audio-file multimodal support in the read tool. The read wrapper and pinned upstream read contract are image-only, while audio support that exists elsewhere is for inbound media understanding or memory indexing, not arbitrary read("*.m4a") tool calls. Related #46601 is adjacent Gemini headless @file work, not a shipped read-tool fix for this issue.

## Best Possible Solution

Keep #35835 open until the read tool can detect supported audio files and return provider-transportable audio content, or until maintainers explicitly choose a different product path. The likely implementation needs either an upstream pi-coding-agent read contract extension or an OpenClaw wrapper that emits audio blocks, provider transport support for those blocks, clear unsupported-model behavior, and focused tests/docs. Coordinate with #46601/#46607 if that work revives, but do not close this issue as implemented or superseded yet.
