## Summary

Keep open. Current main still has the parity gap this PR targets: `openclaw dashboard` builds a token-aware URL for browser/clipboard use, while `openclaw gateway status` still prints only the bare Control UI URL. The PR is a focused attempted fix, but the latest review context also leaves a concrete follow-up concern about avoiding token/SecretRef resolution when Control UI is disabled, so this needs maintainer review or rework rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer review or rework. The best path is a shared dashboard access URL resolver used by both `openclaw dashboard` and `openclaw gateway status`, using the daemon runtime port/TLS/basePath, skipping work when Control UI is disabled, never embedding SecretRef-managed tokens, redacting token fragments from `gateway status --json`, and adding focused status/dashboard parity tests.
