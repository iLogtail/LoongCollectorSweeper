## Summary

Keep open. Current main still has the reported Signal auto-install downloader, now under `extensions/signal`, using raw `node:https` `request()` with manual redirects and no timeout, SSRF guard, proxy-aware guarded fetch, or byte limit. The user-facing wizard still downloads `signal-cli` from GitHub releases, and the existing tests cover asset/archive behavior rather than the download hardening gap.

## Best Possible Solution

Keep the issue open and harden `extensions/signal/src/install-signal-cli.ts`: replace the raw release download path with `fetchWithSsrFGuard` via `openclaw/plugin-sdk/ssrf-runtime`, add an explicit install timeout, validate redirects through the guard, honor trusted env-proxy behavior if appropriate for this operator-controlled GitHub download, enforce a conservative archive byte limit with both content-length and streaming checks, and add focused tests for timeout/redirect SSRF/oversize behavior. The raw release metadata `fetch()` at the same setup boundary should be considered for the same guarded-fetch treatment or a documented exception.

## Risks / Open Questions

- A hung release or redirect target can still stall Signal setup because the direct `https.request()` path has no request timeout.
- A malicious or compromised redirect chain can still point the installer at private-network targets because redirects are followed without the SSRF guard used elsewhere.
- A large or infinite response can still stream to disk without a content-length precheck or incremental byte cap.
- Proxy-configured environments remain inconsistent because this path bypasses OpenClaw's guarded/undici env-proxy fetch handling.
