## Summary

Keep #50205 open. Current main still has no `requestLabels`/`labels` config or Gemini request-body injection path, and the remaining request is a concrete Google-provider enhancement rather than stale, duplicate, or already implemented. The provided related items cover different Gemini/session/retry surfaces and do not supersede this Vertex billing-label ask.

## Best Possible Solution

Keep #50205 open and implement it as a Google-provider-owned enhancement: add a typed strict config slot for Vertex request labels, validate GCP label constraints, inject `labels` only for Vertex AI Gemini `generateContent`/`streamGenerateContent` request bodies, leave the non-Vertex Gemini API path unchanged or explicitly unsupported, and cover schema, transport payload, and docs tests. If maintainers prefer this as an external provider plugin, close only after that product decision or a ClawHub/plugin alternative exists.

## Risks / Open Questions

- Do not inject `labels` into the non-Vertex `generativelanguage.googleapis.com` path unless upstream support is verified; current official docs support labels on Vertex AI, not the Google AI Studio Gemini API path.
- Label key/value validation should follow GCP label constraints so invalid config fails early instead of producing provider-side request errors.
- If the implementation targets `google-vertex`, it may also need a clearer native Vertex Gemini transport/story rather than only the current Google Generative AI transport normalization and OpenAI-compatible Vertex endpoint path.
