## Summary

Current main already solves the Discord 1005/1006 resume-close loop through the pinned @buape/carbon 0.16.0 gateway implementation. Carbon now bounds failed RESUME attempts, clears resume session state, and falls back to fresh IDENTIFY; OpenClaw pins that dependency and promotes the fresh-IDENTIFY debug event in Discord gateway logs. The useful PR discussion and local VPN repro point at a real failure mode, but the PR branch is now obsolete relative to the current Discord plugin layout and shipped dependency behavior.

## Best Possible Solution

Close this PR as implemented on main and keep the current Carbon-backed Discord gateway behavior. If a new 1005/1006 loop still reproduces on @buape/carbon 0.16.0 or later, open a fresh issue with logs showing the Carbon 'Gateway reconnect scheduled...' and 'Gateway forcing fresh IDENTIFY...' sequence so maintainers can debug the remaining case against the current extension layout.
