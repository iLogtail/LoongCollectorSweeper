## Summary

Keep #71389 open. Current main still routes core image operations to the bundled media-understanding image-ops artifact, which imports `sharp` in-process with no AVX/SIGILL CPU preflight, no disabled/none backend option, and no pass-through fallback for `resizeToJpeg`, `convertHeicToJpeg`, or `resizeToPng`. The related #47419 report was a Docker image build regression and does not cover this runtime `sharp` load path.

## Best Possible Solution

Keep the issue open and fix the runtime image backend boundary. Add a Linux/x64 CPU capability gate or safe sharp probe before importing the native module, support an explicit `OPENCLAW_IMAGE_BACKEND=disabled`/`none` pass-through mode, and make `resizeToJpeg`, `convertHeicToJpeg`, and `resizeToPng` degrade predictably where pass-through is valid. Add regression coverage for disabled/probe-failed sharp behavior without requiring non-AVX hardware, and document the operational fallback.
