## Summary

Keep open. Current `main` has partial image support for GitHub Copilot models, but it does not actually register `github-copilot` as a media-understanding image provider, so the specific request in #57259 is not clearly implemented.

## Evidence

- **GitHub Copilot plugin manifest:** The bundled GitHub Copilot plugin declares only `memoryEmbeddingProviders`; it does not declare `mediaUnderstandingProviders`, so Copilot is not registered in the plugin manifest as a media-understanding provider.
  - file: [extensions/github-copilot/openclaw.plugin.json:5](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/github-copilot/openclaw.plugin.json#L5)
  - command: `nl -ba extensions/github-copilot/openclaw.plugin.json | sed -n '1,80p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Media-understanding registry behavior:** `buildMediaUnderstandingRegistry()` only loads declared `mediaUnderstandingProviders`, then adds generic image support only for config providers that already have image-capable models in `models.providers.*.models[]`. There is no GitHub-Copilot-specific registration path here.
  - file: [src/media-understanding/provider-registry.ts:30](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/media-understanding/provider-registry.ts#L30)
  - command: `nl -ba src/media-understanding/provider-registry.ts | sed -n '1,90p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Current docs still exclude Copilot from bundled image fallback:** The media-understanding docs say auto-detect uses the active reply model, `agents.defaults.imageModel`, or configured image-capable providers, and the bundled image fallback order is `OpenAI -> Anthropic -> Google -> MiniMax -> MiniMax Portal -> Z.AI`. GitHub Copilot is not listed there.
  - file: [docs/nodes/media-understanding.md:154](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/docs/nodes/media-understanding.md#L154)
  - command: `nl -ba docs/nodes/media-understanding.md | sed -n '148,190p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Partial workaround exists, but it is not the requested registration:** The image runner falls back to generic `describeImageWithModel` for image requests even when no media-understanding provider is registered, and GitHub Copilot dynamic/default models are marked `input: ["text", "image"]`. That supports explicit/native-vision use, but it is not equivalent to registering Copilot in the media-understanding provider matrix.
  - file: [src/media-understanding/runner.entries.ts:541](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/media-understanding/runner.entries.ts#L541)
  - command: `nl -ba src/media-understanding/runner.entries.ts | sed -n '535,575p' && nl -ba extensions/github-copilot/models.ts | sed -n '58,76p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)

## Risks / Open Questions

- Users may already be able to use `github-copilot/<model>` for explicit image runs or as a native-vision primary model, so the remaining gap is specifically provider registration and auto-detect behavior.
- `pnpm docs:list` could not be run in this environment because `pnpm` was unavailable, so doc discovery was limited to direct file inspection.
