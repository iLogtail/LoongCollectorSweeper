## Summary

Keep #50880 open. Current main still has a plausible Discord steer-mode failure: Discord inbound jobs are queued by session before they reach the shared reply pipeline, while the documented `steer` mode requires messages to reach `runReplyAgent()` during an active streaming run. The related #60604 context is Telegram-specific and closed unmerged, and current main does not show a Discord fix or regression coverage for mid-turn steer injection.

## Best Possible Solution

Keep the issue open. The likely product fix belongs in the Discord inbound worker/job-building path: allow effective `steer` and `steer-backlog` messages to reach the shared reply runner while a session run is active, preserve serialization for non-steer modes, honor inline/session/per-channel/global queue resolution, and add Discord regression tests plus diagnostics for steer-to-followup fallback or failed steer injection.
