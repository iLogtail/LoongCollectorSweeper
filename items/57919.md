## Summary

Keep open. Current `main` still has the early-crash gateway-lock gap this PR targets: `runGatewayLoop` acquires the file lock before signal handlers and before server startup, while `acquireGatewayLock` on `main` only cleans up in async `release()` and does not register any synchronous `process.on("exit")` cleanup. The PR discussion also matters here: the bot review caught the Windows `unlinkSync`-with-open-handle bug, and the follow-up PR commit updates the patch to use `fsSync.closeSync(handle.fd)` before unlinking, so this is still useful work rather than a superseded or obsolete PR.

## Evidence

There is no `process.on("exit")` registration or other synchronous early-exit cleanup in the current implementation.
