## Summary

Keep #71412 open. Current main still contains the reported stop-timeout path: `stopChannel` returns before clearing `store.aborts` and `store.tasks`, leaves `running: true`, and `startChannelInternal` still no-ops when the stale task entry remains. The existing regression test also still asserts the buggy behavior. Related PR #71456 was closed unmerged in the provided context, so it does not supersede or fix this issue on main.

## Best Possible Solution

Keep #71412 open and land PR #71456 or an equivalent maintainer-reviewed patch that clears `store.aborts` and `store.tasks` on stop timeout, marks the runtime stopped instead of running, and updates the regression test to prove a subsequent start registers a fresh task. Treat the `openclaw doctor` liveness/probe improvement as a separate follow-up unless maintainers want it bundled.
