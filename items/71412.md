## Summary

Keep #71412 open. Current main still implements the reported failure mode: the stop-timeout path returns before clearing `store.aborts` and `store.tasks`, marks the runtime `running: true`, and `startChannelInternal` still silently returns when the stale task entry remains. The existing test still asserts that a second account task does not start after a stop timeout, while the related fix PR #71456 is closed and unmerged in the provided context.

## Best Possible Solution

Keep #71412 open and land PR #71456 or an equivalent core gateway fix: on stop timeout, release the stale `store.aborts` and `store.tasks` entries before returning, mark the runtime stopped rather than running, and update the regression test to prove a subsequent start registers a fresh task. Treat the proposed `openclaw doctor` liveness check as a separate follow-up unless maintainers choose to bundle it.
