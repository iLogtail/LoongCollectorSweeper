## Summary

Keep open. Current `main` has explicit ordered `agents.defaults.model.fallbacks`, provider-owned smart-routing refs like `openrouter/auto` and `kilocode/kilo/auto`, and some OpenAI Codex forward-compat for renamed model IDs, but it does not implement a generic provider-family `provider/auto` or wildcard selector such as `openai-codex/auto` that automatically searches a provider-local pool before cross-provider fallback.

## Evidence

- **Fallback chain is explicit, not wildcard-based:** Runtime fallback candidates are built from the requested `provider/model` plus explicit configured fallback strings from `agents.defaults.model.fallbacks`; there is no generic provider-family auto expansion in this path.
  - file: [src/agents/model-fallback.ts:397](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/src/agents/model-fallback.ts#L397)
  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **Model config only exposes primary + fallbacks:** Agent model config supports `primary` and ordered `fallbacks`; there is no config type here for wildcard or provider-local auto-selection semantics.
  - file: [src/config/types.agent-defaults.ts:181](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/src/config/types.agent-defaults.ts#L181)
  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **Only OpenRouter has a built-in compat alias for `:auto`:** Model parsing special-cases `openrouter:auto` and otherwise falls back to normal exact ref parsing. This is provider-specific compatibility logic, not a generic `provider/auto` feature.
  - file: [src/agents/model-selection-shared.ts:156](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/src/agents/model-selection-shared.ts#L156)
  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **Docs describe explicit fallback behavior:** The failover docs say OpenClaw moves to the next model in `agents.defaults.model.fallbacks` and builds the chain from the requested model plus configured fallbacks.
  - file: [docs/concepts/model-failover.md:213](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/docs/concepts/model-failover.md#L213)
  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **`auto` exists today only as provider-owned routes:** Docs show `openrouter/auto` and `kilocode/kilo/auto` as provider-owned smart-routing refs. That is narrower than the issue’s requested generic `openai-codex/auto` / wildcard provider-pool behavior.
  - file: [docs/providers/openrouter.md:25](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/docs/providers/openrouter.md#L25)
  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **OpenAI Codex has partial forward-compat, not provider-pool auto selection:** The OpenAI Codex plugin synthesizes some renamed/new model IDs and canonicalizes `gpt-5.4-codex` to `gpt-5.4`, which mitigates some rename churn for that provider, but it is not a generic `openai-codex/auto` fallback pool.
  - file: [extensions/openai/openai-codex-provider.ts:182](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/extensions/openai/openai-codex-provider.ts#L182)
  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)

## Risks / Open Questions

- OpenAI Codex rename/compat handling may reduce the original pain point for a few specific model IDs, so maintainers could choose to narrow the issue scope to generic provider-family auto selection rather than provider-specific rename repair.
- Provider-owned `auto` routes already exist for some providers, which may overlap with part of the request, but there is no evidence on `main` of a generic core feature that would justify closing this issue as implemented.
