## Summary

Keep #69132 open. Current main removed the exact `/api/experimental/web_search` call, but the reporter's observable failure is not fully fixed: the bundled Ollama web-search provider still defaults to the local Ollama host, makes one request to `/api/web_search`, and throws on a 404 instead of trying the documented cloud fallback/workaround discussed in the issue. The open related PR #69222 still tracks the broader compatibility/auth-scoping fix.

## Best Possible Solution

Keep the issue open and finish a focused fix equivalent to the #69222 direction: preserve local compatibility if desired, handle local 404s with clear guidance or an `OLLAMA_API_KEY`-gated `https://ollama.com/api/web_search` fallback, scope bearer tokens to the correct host, and add regression tests plus docs/changelog updates for the chosen contract.

## Risks / Open Questions

- The current main change may have fixed a narrower endpoint-name bug, but closing #69132 now would lose the remaining local-404/cloud-fallback and auth-scoping work captured in the reopened issue and #69222.
- Maintainers still need to choose the product contract: local-host web search with fallback, cloud-only default, or explicit configuration only. The docs and tests should then be aligned with that choice.
