## Summary

Keep open. Current main has advisory prompt guidance and optional loop detection, but I did not find a targeted fix for the reported MiniMax/Feishu loop where `sessions_list` can return `count: 0` for the active session and the model keeps sending duplicate messages. The default hard loop detector remains disabled, and the session list path still returns only rows present in the gateway/store result rather than synthesizing the requester/current session when absent.

## Evidence

- **Issue report is concrete and recent:** #51133 reports OpenClaw 2026.3.13 with MiniMax-M2.5 on Feishu, where the model repeatedly called `message` and `sessions_list`; `sessions_list` sometimes returned `count:0`, leading to 15+ duplicate messages. A follow-up comment specifically identifies current-session visibility/misinterpretation as the suspected cause.
- **sessions_list does not synthesize current session:** The tool calls gateway method `sessions.list`, converts only the returned `sessions` array into rows, and returns `count: rows.length`; there is no fallback row for the requester/current session if the gateway result is empty.
  - file: [src/agents/tools/sessions-list-tool.ts:114](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/agents/tools/sessions-list-tool.ts#L114)

  - sha: [8e12c24d1721](https://github.com/openclaw/openclaw/commit/8e12c24d17218dc50023aaf5543df1b7c36520aa)
- **Gateway list is store-driven:** `listSessionsFromStore` enumerates `Object.entries(store)`, applies filters, and returns `count: sessions.length`; this supports that an absent current-session store row can still produce an empty list.
  - file: [src/gateway/session-utils.ts:1511](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/gateway/session-utils.ts#L1511)
