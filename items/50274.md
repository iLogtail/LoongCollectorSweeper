## Summary

Keep open. Current main has a real model fallback chain and fallback docs, but the central follow-up is not complete: there is no benchmark/ranking report for the three named OpenRouter fallbacks, no latency/quality/cost ordering policy for those models, and the config schema still only enum-validates model input modalities rather than fail-fast validating whether a valid modality such as audio is supported by the configured model.

## Best Possible Solution

Keep #50274 open. The best path is to implement a focused config validation contract for model input capabilities, add regression coverage for unsupported-but-valid modalities such as audio on a non-audio model, and either add the benchmark-backed fallback ordering policy for the named OpenRouter models or split that benchmark/report work into a narrower canonical follow-up with docs for the chosen latency/quality/cost policy.
