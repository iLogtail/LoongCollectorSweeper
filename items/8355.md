## Summary

Keep #8355 open. Current main still batches the classic voice-call STT/agent/TTS path: it awaits one completed embedded-agent text response, then awaits one completed telephony TTS audio buffer, then chunks that buffer to Twilio. Realtime voice is implemented, but it is a separate full-duplex provider path rather than sentence-level LLM -> TTS -> audio streaming for the classic pipeline. The related streaming TTS PR #42187 is closed and unmerged, while #42159 was closed as the narrower duplicate/superseded report.

## Best Possible Solution

Keep #8355 open as the canonical provider-generic voice-call feature request. The best implementation path is to add a sentence-level pipeline in the voice-call plugin: wire embedded-agent partial replies or sentence chunks into the voice response path, add a streaming telephony TTS capability with buffered fallback, stream chunks through Twilio and Telnyx media handlers where supported, and preserve queue clearing, abort, and barge-in behavior for in-flight synthesis and playback.
