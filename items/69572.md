## Summary

Current main still implements Feishu typing as message/comment reactions, and the docs/tests describe and assert that reaction-based behavior. The related #8722 cleanup confirms the reaction implementation exists, but it does not supersede this newer bug because #69572 specifically reports that reactions are the wrong UX and asks for native Feishu typing instead.

## Best Possible Solution

Keep #69572 open. The best maintainer path is a Feishu plugin fix or feasibility investigation: replace normal chat typing reactions with the native Feishu/Lark typing API if it exists and supports bot replies, keep or explicitly separate comment-reaction behavior if native typing does not cover document comments, update `typingIndicator` docs to say native typing rather than reactions, and add regression tests proving the normal reply path no longer calls `im.messageReaction.create` for typing.

## Risks / Open Questions

- Implementing this should first confirm the exact Feishu/Lark native typing endpoint, scopes, payload, supported chat types, and SDK availability; current repo evidence is enough to keep the issue open but not enough to choose the final API contract.
- Changing away from reactions may affect existing rate-limit/backoff handling and comment-thread cleanup paths, so the fix should include focused Feishu plugin tests for start, refresh, stop, disabled config, stale message suppression, and fallback behavior.
