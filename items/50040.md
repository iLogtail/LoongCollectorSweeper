## Summary

Keep open. Current main has useful Telegram delivery hardening, including strict non-idempotent send retries and preview-finalization ambiguity handling, but it still does not implement the issue's central runtime recovery path for already-computed Telegram reply chunks that fail during polling/network recovery. Standard `sendMessage` failures still bubble out of `deliverReplies`; the dispatcher only tries an immediate fallback reply through the same send path, with no durable pending/ambiguous state or polling-restart-triggered retry worker for the lost final chunks.

## Best Possible Solution

Keep the issue open and implement a Telegram-owned runtime delivery recovery path for assistant replies: persist failed standard reply chunks or route them through a durable queue before send, classify failures as safe retry/ambiguous/permanent, retry only safe or rate-limited sends with backoff, hold ambiguous entries for operator-visible recovery, and trigger a drain after Telegram polling restart/recovery. Add regression tests for a multi-chunk final reply where an already-computed later `sendMessage` fails during a polling recovery window.
