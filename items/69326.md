## Summary

Keep #69326 open. Current main still treats browser WebChat `chat.send` turns as internal WebChat replies: the Control UI sends `deliver: false`, and the Gateway explicitly blocks WebChat clients from inheriting an external `deliveryContext` even on channel-scoped sessions. The WeChat plugin is external, but the reported routing block is in core WebChat/Gateway behavior, so this is not safely closeable as plugin-owned cleanup.

## Best Possible Solution

Keep the issue open for maintainer/product handling. The likely fix is an explicit WebChat/Control UI send-to-channel path for channel-scoped sessions, using stored `deliveryContext` or explicit `originatingChannel`/`originatingTo` fields only when the user intentionally targets that external conversation, with regression tests for a third-party plugin channel id such as `openclaw-weixin` and safeguards preserving the existing route-overwrite protections.

## Risks / Open Questions

- Changing WebChat to inherit external delivery routes by default could regress the existing route-stealing protections referenced by the current tests and comments.
- A fix should remain generic for third-party channel ids such as `openclaw-weixin`; hardcoding WeChat in core would violate the plugin boundary.
