## Summary

Keep #69326 open. Current main still treats browser Control UI/WebChat turns as internal WebChat replies: the Control UI connects in `webchat` mode, sends `chat.send` with `deliver: false`, and the gateway explicitly prevents WebChat clients from inheriting external delivery routes even when a session has `deliveryContext` for a channel. This is a real core/UI routing policy question, not something already implemented or safely closeable as an external WeChat-plugin issue.

## Best Possible Solution

Keep the issue open for maintainer-owned product/design work. A good fix would define an explicit, user-visible cross-surface reply path for Control UI/WebChat when a selected session has a routable external `deliveryContext`, then update `ui/src/ui/controllers/chat.ts` and `src/gateway/server-methods/chat.ts` with tests that cover WeChat-style plugin channels while preserving the #38957 guard against accidental external sends from plain WebChat browsing.

## Risks / Open Questions

- Changing this naively could regress the existing #38957-protected behavior where merely viewing or typing from WebChat should not unexpectedly send messages to an external conversation.
- The desired UX needs a maintainer decision: either keep WebChat strictly internal, add an explicit “reply to external channel” affordance, or reclassify specific Control UI channel-session sends as operator UI sends with clear safeguards.
