## Summary

Keep #71728 open. The RFC is fresh, coherent, and asks for a core trust-signaling contract: per-factual-claim confidence tiers, source attribution, known-vs-inferred distinction, and autonomous completion reporting for unverified claims. Current main has adjacent truthfulness guidance, memory citations, memory-wiki claim metadata, and plugin hooks, but it does not expose claim-level confidence/evidence metadata in reply payloads or task-completion events. This is not a safe ClawHub-only close because the request cuts across core reply, gateway protocol, SDK, and autonomous completion surfaces.

## Best Possible Solution

Keep the RFC open for maintainer product/design judgment. A credible path would define a core or SDK-visible trust signal for agent output: claim-level confidence/source metadata where available, known-vs-inferred guidance, propagation through ReplyPayload/runtime-plan/channel delivery, and task-completion fields for unverified claims. Existing memory citations and memory-wiki claim metadata should be reused as inputs, but they do not solve the whole output trust contract.
