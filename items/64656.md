## Summary

Keep #64656 open. Current main partially covers the auto-failover stickiness through auto-source reset/self-healing, but it does not implement the PR’s full behavior: there is no dedicated `failoverState`, no separate LCM `memoryConversationKey`, and context-engine lifecycle calls still use the runtime `sessionKey`. The overflow path has a bounded retry loop and diagnostics on main, so that part may need rebasing or narrowing rather than automatic closure.

## Best Possible Solution

Keep the PR open but require a rebase and likely split/narrowing: preserve the already-shipped auto-override self-healing, decide whether a dedicated `failoverState` is still worth adding, implement the LCM conversation identity split with consistent context-engine keys and legacy reset semantics, and only add overflow diagnostics/circuit changes that are not already covered by the current three-attempt overflow guard.
