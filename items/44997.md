## Summary

Keep #44997 open. Current main has newer optimistic base-hash protection, but it still does not implement the PR's locked read/transform/write path for concurrent `agents add`; the PR addresses a real remaining race and needs a current-main rebase rather than auto-close.

## Best Possible Solution

Keep the PR open and ask for a current-main rebase that moves the shipped config mutation path to a serialized read-snapshot -> transform -> write critical section, preserving `commitConfigWithPendingPluginInstalls`, base-hash conflict semantics, include handling, runtime snapshot refresh, and hardened config directory permissions. Add a regression test proving parallel non-interactive `agents add` calls do not lose `agents.list` entries.

## Risks / Open Questions

- The PR is likely stale against current main because `agents add` now commits through `commitConfigWithPendingPluginInstalls` and `replaceConfigFile`; merging as-is could bypass newer plugin install, include-write, or base-hash behavior.
- The current base-hash guard can reject some stale writes, but without a lock it does not guarantee that two writers cannot both validate the same snapshot before either rename commits.
- A safe fix should include a focused concurrent `agents add --non-interactive` regression test or an equivalent lower-level config mutation race test.
