## Summary

Current `main` does not implement strict9 tool-call ID sanitization for Mistral models routed through OpenRouter, and this PR is still addressing a real gap even though its original patch target (`src/agents/provider-capabilities.ts`) no longer exists on `main` after replay-policy ownership moved into provider plugins.

## Evidence

- **Replay policy now comes from provider plugins, not legacy capability tables:** `resolveTranscriptPolicy` resolves the provider runtime plugin and returns its `buildReplayPolicy` result when present; only providers without a replay hook use the generic transport fallback.
  - file: [src/agents/transcript-policy.ts:197](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/agents/transcript-policy.ts#L197)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **OpenRouter plugin has no Mistral strict9 replay hook:** The OpenRouter provider registers `PASSTHROUGH_GEMINI_REPLAY_HOOKS`, which do not sanitize tool call IDs or set `toolCallIdMode`; there is no Mistral-family override in this plugin.
  - file: [extensions/openrouter/index.ts:142](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/openrouter/index.ts#L142)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Mistral strict9 handling is only on the native Mistral provider:** The Mistral plugin's replay policy hard-codes `sanitizeToolCallIds: true` with `toolCallIdMode: "strict9"`, but that policy is attached only to provider `mistral`.
  - file: [extensions/mistral/index.ts:11](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/mistral/index.ts#L11)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Current tests still expect OpenRouter to avoid the stricter replay path:** `transcript-policy.test.ts` explicitly asserts that OpenRouter keeps its existing turn-validation path and does not enable the stricter OpenAI-compatible replay behavior.
  - file: [src/agents/transcript-policy.test.ts:495](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/agents/transcript-policy.test.ts#L495)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **The old static capability bag is deprecated and no longer read at runtime:** `ProviderCapabilities` is marked deprecated with a note that replay/runtime ownership now lives on explicit hooks like `buildReplayPolicy`; this makes the PR's original `provider-capabilities.ts` edit stale relative to current architecture.
  - file: [src/plugins/types.ts:670](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/plugins/types.ts#L670)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Cross-provider Mistral detection exists for transport compat, not replay policy:** `extensions/mistral/provider-compat.ts` detects Mistral-family model IDs across providers, but only to contribute transport compat such as `supportsStore`/`maxTokensField`; it does not set replay `toolCallIdMode`.
  - file: [extensions/mistral/provider-compat.ts:26](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/mistral/provider-compat.ts#L26)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
