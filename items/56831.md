## Summary

Keep PR #56831 open. Current main still sends non-empty-looking Discord component objects into `readDiscordComponentSpec`, where `modal: { fields: [] }` throws before the send can fall back to plain text. The PR addresses a real current behavior, but the submitted patch also has review concerns around preserving malformed-field validation and container-only component specs, so it should be revised rather than closed as obsolete.

## Best Possible Solution

Keep the PR open and ask for a narrow revision: treat empty LLM-default component specs as absent, skip empty `blocks: []` and empty/nullish modal defaults, keep validation errors for present non-array `modal.fields`, preserve container-only specs so fallback text still uses the component path, and add regression tests for all of those cases in the Discord plugin tests.

## Risks / Open Questions

- The PR patch as described by review comments may silently swallow non-array `modal.fields` values; the final fix should tolerate empty arrays/nullish empty defaults without hiding malformed field types.
- A broad `return null` for specs with no text/blocks/modal can regress container-only component styling when text is supplied through the top-level message fallback.
