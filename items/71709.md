## Summary

Keep open. Current main still has the reported slug-generator path: `generateSlugViaLLM` calls `runEmbeddedPiAgent` for `temp:slug-generator` without `authProfileFailurePolicy: "local"`, while the embedded runner persists classified non-timeout failures such as `billing` into shared auth-profile state. The current classifier also intentionally maps Anthropic extra-usage messages to `billing`, so the issue’s profile-wide 5-hour disable path remains reachable from the helper lane.

## Best Possible Solution

Keep #71709 open. Fix the session-memory slug generator call site by passing `authProfileFailurePolicy: "local"` or an equivalent helper-local failure scope into `runEmbeddedPiAgent`, then add a regression test proving slug-generation billing/auth failures fall back to timestamp/null without writing shared auth-profile failure state. Preserve existing billing classification for normal user-facing provider runs unless maintainers intentionally change that product behavior.
