## Summary

Keep open. Current main still has the reported slug-generator path: `generateSlugViaLLM` calls `runEmbeddedPiAgent` on `sessionKey: "temp:slug-generator"` without `authProfileFailurePolicy: "local"`, while the embedded runner only suppresses shared auth-profile writes when that local policy is set. Anthropic extra-usage messages are still intentionally classified as billing, and billing failures still create a profile-wide disabled window by default, so a slug-helper billing-classified failure can still poison the shared auth profile.

## Best Possible Solution

Keep the issue open and fix `src/hooks/llm-slug-generator.ts` by passing `authProfileFailurePolicy: "local"` to the slug-generator `runEmbeddedPiAgent` call. Add a regression test in `src/hooks/llm-slug-generator.test.ts` asserting the local policy is passed, and ideally a focused runner/auth-state test proving a billing-classified slug-helper failure falls back to timestamp/null without writing shared auth-profile failure state. Preserve the existing billing classifier for normal user-facing/provider runs unless maintainers choose a broader classifier policy change.
