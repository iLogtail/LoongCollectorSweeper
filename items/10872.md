## Summary

Keep #10872 open. Current main still only includes iMessage reply context when the imsg payload already contains resolved `reply_to_*` fields; it does not model or resolve `thread_originator_guid` / `reply_to_guid` into `ReplyToId`, `ReplyToBody`, and `ReplyToSender`. The issue discussion already identifies this remaining Layer 3 gap, and related PR #42683 is still only proposed work in the provided context, not shipped behavior on main.

## Best Possible Solution

Keep the issue open until the bundled iMessage plugin resolves imsg reply GUID metadata such as `thread_originator_guid` or `reply_to_guid` into the existing inbound reply fields. The likely fix belongs in `extensions/imessage/src/monitor/*`: add a narrow resolver, preferably via a documented imsg RPC capability or a carefully scoped read-only chat.db lookup when `dbPath` is configured, then add parser/monitor regression tests for GUID-only inline replies. PR #42683 can be used as the starting point or superseded by a maintainer-approved replacement.
