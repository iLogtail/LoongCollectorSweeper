## Summary

Keep #10872 open. Current main still handles iMessage reply context only when the legacy imsg payload already includes resolved `reply_to_*` fields; it does not accept or resolve imsg's GUID-only `thread_originator_guid` path. Related PR #42683 is open and proposes the missing chat.db fallback, but it is not merged. ([github.com](https://github.com/openclaw/openclaw/pull/42683))

## Best Possible Solution

Keep the issue open until the iMessage plugin models imsg reply GUID metadata (`thread_originator_guid` and/or `reply_to_guid`) and resolves it into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` via a documented imsg RPC path or a carefully scoped chat.db lookup with local/remote runtime gating. Add parser and inbound-processing regression tests. If maintainers decide legacy imsg should not gain this, make that an explicit product decision in favor of BlueBubbles rather than an automated cleanup close.
