## Summary

Keep open. Current `main` does not implement the PR’s requested DNS domain blocklist layer or its supporting artifacts. The SSRF guard on `main` still only blocks built-in hostname patterns (`localhost`, `.local`, `.internal`) plus private/special-use IPs, and `resolvePinnedHostnameWithPolicy()` performs no domain-blocklist check before DNS resolution.

## Evidence

- **Current SSRF hostname blocking is still limited to fixed local/internal patterns:** `isBlockedHostnameNormalized()` only blocks entries from `BLOCKED_HOSTNAMES` plus `*.localhost`, `*.local`, and `*.internal`; `isBlockedHostnameOrIp()` combines that with private/special-use IP checks. There is no configurable or suffix-matching DNS domain blocklist in current `main`.
  - file: [src/infra/net/ssrf.ts:182](https://github.com/openclaw/openclaw/blob/b98a6a46fb54145a33e753101c02f0d7d7e0bc3b/src/infra/net/ssrf.ts#L182)

  - sha: [b98a6a46fb54](https://github.com/openclaw/openclaw/commit/b98a6a46fb54145a33e753101c02f0d7d7e0bc3b)
- **No pre-DNS domain-blocklist hook in the pinned-resolution path:** `resolvePinnedHostnameWithPolicy()` calls `resolveHostnamePolicyChecks()` and then immediately performs DNS lookup. The pre-DNS checks only enforce hostname allowlist and `assertAllowedHostOrIpOrThrow()`, which throws plain `SsrFBlockedError`; there is no `DnsBlocklistError` or equivalent domain-blocklist check before resolver access.
