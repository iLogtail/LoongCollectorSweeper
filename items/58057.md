## Summary

Current `main` does not implement the requested `dmPolicy/groupPolicy: dynamic` behavior, an external resolver hook before allowlist checks, or dynamic group admission based on existing bindings. OpenClaw does have dynamic DM onboarding via `pairing`, but `allowlist` remains config/store-driven and group authorization is still explicit, so this issue is not closable as already implemented.

## Evidence

- **Config schemas exclude `dynamic`:** Core config types and schemas only allow `pairing|allowlist|open|disabled` for DMs and `open|disabled|allowlist` for groups. There is no `dynamic` policy value on main.
  - file: [src/config/zod-schema.core.ts:438](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/config/zod-schema.core.ts#L438)

- **`allowlist` still blocks unknown DM senders:** `readStoreAllowFromForDmPolicy` skips store reads when `dmPolicy === "allowlist"`, and the shared access resolver only returns the dynamic `pairing` path for unknown senders when `dmPolicy === "pairing"`. Otherwise unknown senders are blocked.
  - file: [src/security/dm-policy-shared.ts:95](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/security/dm-policy-shared.ts#L95)
