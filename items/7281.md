## Summary

Current main still lacks configurable `fallbackChannel` or `fallbackChannels` support for hook mappings. The hook mapping config/schema, request normalization, mapping pipeline, dispatch bridge, cron delivery model, and direct delivery path all carry a single primary `channel`/`to` target. The existing `fallbackChannel` local variable in cron delivery target resolution is only an implicit pre-send channel-selection fallback, not a user-configurable retry chain after primary delivery failure. The same-day #7281 discussion also already contains a maintainer/Codex review keeping the issue open, and the related closed Slack availability report #54029 is distinct rather than a canonical fallback-chain tracker.

## Best Possible Solution

Keep #7281 open for maintainer/product handling. A complete fix should add typed `fallbackChannel` or `fallbackChannels` config and validation, propagate ordered fallback targets through hook mapping resolution and `dispatchAgentHook` into cron/direct delivery, retry alternate targets only after primary delivery failure, keep this distinct from `channel: "last"` and `failureDestination`, document it in the public configuration reference, and add focused regression tests for primary failure with fallback success.
