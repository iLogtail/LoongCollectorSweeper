## Summary

Keep #38714 open. Current main still receives Discord reaction add/remove events and queues them as session-scoped system events, but it does not expose reaction add/remove as internal Hooks events or typed plugin hooks, and it does not bridge Discord reactions into the `message_received` hook payload. Related #11747 remains a separate open auto-response/wake request; closed #52658 and #60805 overlap but do not provide a shipped canonical replacement for this hook/API gap.

## Best Possible Solution

Keep #38714 open until maintainers choose and implement a reaction event contract. The likely path is to add a typed internal/plugin hook such as reaction add/remove, or bridge Discord reactions into a structured `message_received`/`message:received` payload with emoji, action, reactor, target message, route/session, account/guild/channel/thread metadata, and removal state. Add Discord listener tests, plugin/internal hook tests, and docs. Keep #11747 as the separate auto-response/wake behavior discussion.
