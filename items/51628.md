## Summary

Keep #51628 open. Current main partially mitigates adjacent issues: Telegram live sends use a stricter non-idempotent retry predicate, and delivery-mirror transcript appends can dedupe against an explicit idempotency key or the latest equivalent assistant message. But the central reported bug remains: startup delivery recovery still replays eligible queued Telegram entries based only on retry count/backoff, without classifying `sendMessage` network failures as uncertain or quarantining them before replay.

## Best Possible Solution

Keep the issue open and implement a focused recovery-policy fix: classify Telegram `sendMessage`/non-idempotent network failures as uncertain when queue metadata is updated, skip or quarantine those entries during startup/reconnect recovery unless delivery can be confirmed, preserve a stable mirror idempotency key on recovery, and add a regression test with a queued Telegram `Network request for 'sendMessage' failed!` entry asserting no replay and no duplicate transcript mirror append. The KeepMyClaw comments are useful validation/cross-checking context, but they do not replace the core queue recovery fix.
