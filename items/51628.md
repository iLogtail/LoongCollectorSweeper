## Summary

Keep #51628 open. Current main has partial transcript-mirror dedupe, but the central Telegram delivery-queue replay problem remains: recovery still retries pending outbound entries after backoff without classifying uncertain Telegram `sendMessage` failures or quarantining them. The KeepMyClaw comments are useful corroborating detection context, but they explicitly do not replace the root-cause fix in OpenClaw core.

## Evidence

- **Recovery still replays pending entries generically:** Startup recovery skips only max-retry and backoff-ineligible entries, then calls `drainQueuedEntry` for the current queued delivery. There is no check for `channel === "telegram"`, `lastError`, delivery uncertainty, or a quarantine policy before replay.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:386](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/infra/outbound/delivery-queue-recovery.ts#L386)
  - command: `nl -ba src/infra/outbound/delivery-queue-recovery.ts | sed -n '380,620p'`
  - sha: [8e12c24d1721](https://github.com/openclaw/openclaw/commit/8e12c24d17218dc50023aaf5543df1b7c36520aa)
- **Replay path invokes the original delivery with queue disabled:** `drainQueuedEntry` calls the stored delivery through `buildRecoveryDeliverParams`; that builder preserves channel, target, payloads, mirror, session, and sets `skipQueue: true`. This prevents re-enqueueing, but it still performs the send again.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:112](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/infra/outbound/delivery-queue-recovery.ts#L112)

- **Queue storage records lastError but no uncertainty state:** Queued delivery entries have `lastError`, `retryCount`, and `lastAttemptAt`; `failDelivery` records the formatted error string, but there is no `deliveryState`, `uncertain`, or equivalent metadata for recovery to distinguish unsafe Telegram network failures.
