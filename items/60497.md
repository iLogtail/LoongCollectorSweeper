## Summary

Keep open. Current main captures exec failures internally, but it still does not enforce the behavior requested by #60497 for the local CLI path: a non-timeout exec failure such as `python: command not found` can be suppressed in the final payload when verbose mode is off, and assistant success text can still be delivered. The linked NemoClaw/NVIDIA sandbox-side python symlink fix reduces one trigger, and #60507 is complementary command-approval work, but neither is a canonical fix for OpenClaw's final-answer/tool-failure honesty behavior.

## Best Possible Solution

Keep the issue open and fix the local/embedded agent finalization path. The likely product-safe fix belongs around `src/agents/pi-embedded-runner/run/payloads.ts` and tool-failure state handling: failed `exec`/`bash` actions should produce a user-visible error or block/annotate contradictory assistant success text unless a later matching successful retry resolves the same action. Add a regression test for `lastToolError: { toolName: "exec", error: "... command not found" }` plus assistant text like `the script is ready`, and decide whether CLI/default channels should include exact stderr or a redacted summary outside verbose mode.
