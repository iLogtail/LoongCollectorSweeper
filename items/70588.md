## Summary

Keep #70588 open. Current main has internal Discord webhook creation/reuse for thread bindings, but it is not exposed as an agent-callable `message(action="webhook-create")` action, the shared action enum does not include that action, and the Discord docs/skill guidance do not document an equivalent workflow.

## Best Possible Solution

Keep the issue open for maintainer/product handling in the bundled Discord plugin. A complete fix would add a first-class Discord-owned shared message action, likely `webhook-create`, extend `CHANNEL_MESSAGE_ACTION_NAMES` and target-mode handling, add Discord discovery/gating and runtime dispatch, accept channel/name/avatar parameters, return the created webhook URL carefully because it is a credential, and update tests plus Discord/CLI docs. If maintainers prefer this as a ClawHub plugin instead, OpenClaw first needs a generic plugin action/API seam because the current shared message-tool action surface is a closed typed list.
