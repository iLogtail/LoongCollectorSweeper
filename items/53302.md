## Summary

Keep PR #53302 open. Current main does not already provide the requested Discord-only escape hatch for cross-context components v2 containers, and the user problem still exists: Discord cross-context markers are still rendered through the channel presentation path by default. The PR itself is stale against main because the old `buildCrossContextComponents`/`channel-adapters.ts` surface has been replaced by `buildCrossContextPresentation`, but that makes it a rebase/update request rather than a conservative close.

## Best Possible Solution

Keep the PR open but ask for a rebase/update: add a Discord-owned config helper and schema/docs entries for a narrowly named cross-context presentation/container opt-out, thread it through `buildDiscordCrossContextPresentation` or the plugin registration so disabled accounts return no presentation builder and core falls back to the existing text marker, and add focused tests for base config, per-account override, default behavior, and preservation of interactive components/forms.

## Risks / Open Questions

- The PR patch is stale against current main's message-presentation architecture and would need to be reworked around `buildCrossContextPresentation`, not the deleted `channel-adapters.ts`/`buildCrossContextComponents` path.
- The per-account behavior should be implemented in a Discord-owned helper that handles base config, account overrides, and default-account resolution consistently without leaking Discord-specific config resolution into core outbound policy.
- Any implementation should preserve the review-thread fixes already discussed: disabling cross-context containers must not disable Discord buttons, selects, forms, modals, exec approvals, or explicit presentation/components sends.
