## Summary

Keep open. Current `main` still limits `sendChatAction` backoff/circuit-breaking to 401-style errors and still resets the typing failure guard on every `onReplyStart()`, so the reported Telegram network-error keepalive loop is not clearly fixed on `main`.

## Evidence

- **`sendChatAction` handler is still 401-only:** `extensions/telegram/src/sendchataction-401-backoff.ts:32-49,59-67,69-76,83-135` defines `maxConsecutive401`, detects only `is401Error()`, and only increments/suspends inside the 401 branch. There is no network-error backoff or max-retry path there.
  - file: [extensions/telegram/src/sendchataction-401-backoff.ts:32](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/telegram/src/sendchataction-401-backoff.ts#L32)

  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Current tests still codify that non-401 errors do not trip suspension:** `extensions/telegram/src/sendchataction-401-backoff.test.ts:100-114` explicitly asserts `does not count non-401 errors toward suspension`, which is the opposite of the issue's requested network-error protection.
  - file: [extensions/telegram/src/sendchataction-401-backoff.test.ts:100](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/telegram/src/sendchataction-401-backoff.test.ts#L100)
