## Summary

Keep open. Current `main` still persists errored assistant turns, and the Moonshot/OpenAI-completions replay path does not repair or rewrite duplicate assistant tool-call IDs across a failed turn plus retry. The reported session-corruption path remains plausible on `main`, so there is not strong evidence to close this conservatively.

## Evidence

- **Moonshot replay keeps native tool-call IDs:** The Moonshot plugin explicitly opts out of tool-call ID sanitization for OpenAI-compatible replay because Kimi uses native IDs like `functions.<name>:<index>`. That means replay does not rewrite duplicate IDs away for this provider family.
  - file: [extensions/moonshot/index.ts:60](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/extensions/moonshot/index.ts#L60)
  - command: `nl -ba extensions/moonshot/index.ts | sed -n '60,66p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **OpenAI-compatible replay policy lacks pairing repair:** `buildOpenAICompatibleReplayPolicy()` only enables assistant-first ordering and turn validation for `openai-completions`; it does not enable `repairToolUseResultPairing`. So Moonshot completions replay does not get the repair path that drops tool results from errored assistant turns.
  - file: [src/plugins/provider-replay-helpers.ts:27](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/plugins/provider-replay-helpers.ts#L27)
  - command: `nl -ba src/plugins/provider-replay-helpers.ts | sed -n '27,42p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Replay sanitizer only repairs when policy flag is on:** Replay history calls `sanitizeToolUseResultPairing(..., { erroredAssistantResultPolicy: "drop" })` only when `policy.repairToolUseResultPairing` is true. For the Moonshot OpenAI-completions policy above, that branch is skipped and duplicated assistant tool-call IDs are left untouched.
  - file: [src/agents/pi-embedded-runner/replay-history.ts:520](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/agents/pi-embedded-runner/replay-history.ts#L520)
  - command: `nl -ba src/agents/pi-embedded-runner/replay-history.ts | sed -n '520,525p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Consecutive assistant turns are merged without deduping tool-call blocks:** `validateGeminiTurns()` merges consecutive assistant messages by concatenating their content arrays. If a failed assistant turn and its retry are adjacent and both contain the same tool-call ID, this merge preserves both blocks instead of dropping or remapping the duplicate.
  - file: [src/agents/pi-embedded-helpers/turns.ts:320](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/agents/pi-embedded-helpers/turns.ts#L320)
  - command: `nl -ba src/agents/pi-embedded-helpers/turns.ts | sed -n '320,345p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Errored assistant messages are still persisted:** The session persistence guard skips extracting pending tool calls from assistant messages with `stopReason` `error`/`aborted`, but it still appends the assistant message itself to session history. That matches the issue's claim that the failed assistant response remains stored.
