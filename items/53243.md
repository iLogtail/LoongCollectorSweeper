## Summary

Keep open. Current main does not implement the PR's requested behavior: chat.history still truncates by a character cap before byte budgeting, the default is now 8,000 chars rather than the proposed 64,000, and oversized entries can still fall through to the placeholder path. Main added configurable maxChars in #58900, but that is a mitigation/knob, not the byte-aware preservation fix proposed here.

## Best Possible Solution

Keep the PR open for maintainer judgment, but rework it against the current chat-display-projection path. The robust fix should decide the intended default cap, align docs/config help with that default, truncate text by the same JSON-byte budget used by the per-message hard cap, and add regression tests for 50k ASCII text plus UTF-8/escape-heavy assistant text that remains readable instead of becoming the oversized placeholder.
