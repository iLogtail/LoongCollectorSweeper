## Summary

Keep open. Current `main` does not already implement BlueBubbles approval-by-reaction, and the proposed PR still conflicts with the repo’s current approval and SDK patterns. Real BlueBubbles reaction webhooks are routed through `processReaction`, not the message path the PR patches; the current approval runtime still does not bind pending approvals to a delivered channel message GUID; and the latest maintainer review explicitly asked for that hardening before merge.

## Evidence

- **BlueBubbles routes real reaction webhooks to a separate handler:** `handleBlueBubblesWebhookRequest` treats `message-reaction` and `reaction` payloads as reactions and dispatches them to `processReaction(...)`, while only non-reaction payloads go through the normal message/debouncer path. That means approval resolution added only to `processMessage` would not cover the main reaction webhook path on current `main`.

## Risks / Open Questions

- A tapback resolver keyed from replied-to text instead of tracked delivered message ids can resolve the wrong pending approval or accept forged/quoted approval text.
- Adding the feature only on the message-processing path would miss real BlueBubbles `message-reaction`/`reaction` webhooks and still fail in mention-gated group cases.
- Exporting low-level `callGateway` through `plugin-sdk/gateway-runtime` would widen the public plugin SDK surface beyond the narrower approval helper the repo already documents and uses.
