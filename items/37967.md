## Summary

Keep #37967 open. Current main still only creates dynamic Feishu agents for direct-user flows, while Feishu group handling provides sender/topic session routing and thread-bound delivery for already spawned subagents. It does not automatically create per-sender group subAgents with isolated agent config, workspace, skills, memory, or admin hot configuration.

## Best Possible Solution

Keep the issue open for Feishu plugin product/design work. A complete implementation should add an explicit opt-in group-chat mode that maps sender or role identity to isolated agent/subagent profiles, safely provisions workspace/agentDir/skills/model/config/memory, routes replies back to the originating Feishu group/thread, provides bounded admin controls, documents the trust model, and adds regression coverage for automatic provisioning and isolation. If maintainers want this outside the bundled Feishu plugin, first define the needed plugin/API contract clearly rather than closing the request as already satisfied.

## Risks / Open Questions

- The requested feature overlaps security/product semantics: per-user group memory and skills are useful, but current docs warn that session or memory scoping is not a per-user authorization boundary.
- Manual bindings and thread-bound subagent hooks solve adjacent routing problems, so a fix should avoid treating those as sufficient for automatic per-sender provisioning.
