## Summary

Keep #37967 open. Current main has Feishu sender/topic-scoped group sessions, DM-only dynamic Feishu agent creation, and explicit thread-bound subagent hooks, but it still does not automatically create and route each Feishu group sender/role to an isolated `__mc_*` subagent with separate workspace, skills/config, memory, and admin controls.

## Best Possible Solution

Keep the issue open for Feishu plugin product/design work. A good implementation would add an explicit opt-in group mode that maps sender or role identity to bounded isolated agent/subagent profiles, provisions workspace/agentDir/skills/model/config/memory, routes replies back to the originating group/thread, adds admin controls, and covers provisioning/isolation with tests and security docs. If maintainers prefer this outside the bundled Feishu plugin, first define the stable Feishu inbound-routing/plugin API and authorization model needed for an external ClawHub implementation.

## Risks / Open Questions

- The requested feature changes Feishu group trust boundaries: per-user memory, workspace, skill enablement, model/config overrides, and admin controls need explicit opt-in authorization and security documentation.
- A ClawHub/plugin approach may be possible for a plugin that owns the full Feishu channel path, but this issue targets bundled Feishu group behavior and current evidence does not show an already-posted external implementation or a settled maintainer API/product decision.
