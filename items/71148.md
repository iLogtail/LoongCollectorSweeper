## Summary

Keep open. Current main still bundles and loads the affected Lobster runtime path: the OpenClaw Lobster plugin runs the published `@clawdbot/lobster/core` runtime in-process, and both `extensions/lobster/package.json` and `pnpm-lock.yaml` still pin `@clawdbot/lobster@2026.4.6` with `ajv@8.18.0`. The latest release tag also pinned the same Lobster dependency, and the main-branch changes since then adjust loader/package metadata rather than adding an Ajv compile cache or bumping Lobster to a fixed version. The issue also has independent production confirmation, so there is not enough evidence to close as stale, duplicate, or implemented.

## Best Possible Solution

Keep #71148 open and fix it in the bundled Lobster path: either patch `@clawdbot/lobster` to cache/reuse Ajv validators at the reported `sharedAjv` and `llm_invoke` call sites, then bump `extensions/lobster` to the fixed Lobster version, or carry a narrow OpenClaw-side integration workaround if an upstream release is delayed. Add regression coverage that repeatedly validates identical workflow/LLM schemas and proves compile count or retained validator growth is bounded.
