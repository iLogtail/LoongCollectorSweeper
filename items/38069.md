## Summary

Keep #38069 open. At the provided main SHA, OpenClaw still does not implement configurable delayed re-execution for failed recurring cron jobs; current behavior only gives one-shot jobs transient retry semantics and applies an error backoff floor to recurring jobs, which still lets daily/weekly jobs wait for their natural next schedule.

## Best Possible Solution

Keep #38069 open until recurring cron execution retry is designed and shipped. The likely maintainer path is an additive retry policy in the cron job/config contract, CLI/API editing support, explicit retry attempt state separate from normal schedule state, transient-error classification for recurring runs, scheduling retries before the next natural daily/weekly slot while avoiding unresolved-schedule refire loops, and regression tests plus docs that distinguish one-shot retry from recurring execution retry.

## Evidence

For daily or weekly schedules, the natural next run is later than the retry delay, so this does not re-execute the failed run after 15/30/60 minutes as requested.
