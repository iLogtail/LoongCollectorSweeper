## Summary

Keep #50298 open. Current main only solves the simple direct-run mapping: `chat.send` returns the client/idempotency runId and the immediate active run can use it. The collect/followup queue path still creates a fresh execution runId later and the queued item/protocol surfaces do not preserve or expose the original `chat.send` runId set. The related #50296 appears to be an earlier duplicate context, not a shipped fix or canonical superseding item.

## Best Possible Solution

Keep the issue open and implement this as an additive Gateway contract change: store the client `chat.send` runId on `FollowupRun`, preserve all source ids when collect mode batches queued messages, expose them on agent/chat events with a documented optional field such as `sourceRunIds` or `originalRunIds`, and add regression coverage for direct, followup, and collect-mode WebSocket flows.
