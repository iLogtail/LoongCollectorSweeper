## Summary

Keep open. Current `main` still carries `memorySearch.outputDimensionality` into provider selection, but the OpenAI embedding implementation on `main` never forwards that value to the `/embeddings` request body, and the vector table dimension is derived from the returned embedding length. The reported bug still matches current code.

## Evidence

- **Memory search request still includes outputDimensionality:** `resolveMemoryPrimaryProviderRequest` copies `settings.outputDimensionality`, and `MemoryIndexManager.loadProviderResult` passes that request into `createEmbeddingProvider`, so the config value still enters provider creation.
  - file: [extensions/memory-core/src/memory/manager-provider-state.ts:30](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/memory-core/src/memory/manager-provider-state.ts#L30)
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Provider creation uses that request unchanged:** `createEmbeddingProvider` forwards the full options object into the selected adapter's `create(...)` call, so any loss happens inside the provider implementation, not before.
  - file: [extensions/memory-core/src/memory/manager.ts:148](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/memory-core/src/memory/manager.ts#L148)
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **OpenAI provider does not store or forward dimensions:** `OpenAiEmbeddingClient` only has `baseUrl`, `headers`, `ssrfPolicy`, `fetchImpl`, and `model`. `resolveOpenAiEmbeddingClient` returns only those fields, and `createRemoteEmbeddingProvider` posts `body: { model: client.model, input }` with no `dimensions` field.
  - file: [extensions/openai/embedding-provider.ts:10](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/openai/embedding-provider.ts#L10)
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **OpenAI batch embedding path also omits dimensions:** The OpenAI batch adapter builds each request body as `{ model: client.model, input: chunk.text }`, again with no `dimensions`, so the same bug exists for batch sync paths too.
  - file: [extensions/openai/memory-embedding-adapter.ts:44](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/openai/memory-embedding-adapter.ts#L44)
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Vector dimension comes from returned embedding length:** Memory sync uses `sample.length` from the actual returned embeddings to call `ensureVectorReady(sample.length)`, and `ensureVectorTable` creates `FLOAT[${dimensions}]`. If OpenAI returns its default length, the vector table will use that default rather than the configured `outputDimensionality`.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:664](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/memory-core/src/memory/manager-embedding-ops.ts#L664)
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Other providers do wire outputDimensionality correctly:** The Gemini path resolves `options.outputDimensionality` into the client and includes it in embedding requests and batch requests, which matches the reporter's comparison and highlights the OpenAI gap.
  - file: [extensions/google/embedding-provider.ts:343](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/google/embedding-provider.ts#L343)
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
