## Summary

Keep #41111 open. Current main still exposes `openclaw backup create` and `openclaw backup verify` only; there is no first-party `openclaw backup restore <archive>` command, implementation, test coverage, or docs. The linked keepmyclaw ClawHub plugin is a useful workaround, and related PR/issue activity shows demand, but nothing has shipped or cleanly superseded this core CLI request.

## Best Possible Solution

Keep the issue open until OpenClaw either ships a safe first-party `openclaw backup restore <archive>` workflow or maintainers explicitly close it in favor of a canonical tracker. The implementation should extend the existing backup CLI with archive validation, dry-run/force behavior, staging/rollback on publication failure, docs updates, and focused command plus restore-path tests; keepmyclaw can remain a documented community workaround if maintainers want to point users there.

## Risks / Open Questions

- A maintainer may later consolidate this under the broader native backup/restore umbrella (#13616) or an active restore PR, but current main has no shipped restore command to justify cleanup closure now.
- Restore is state-mutating and needs validation, staging, conflict handling, and rollback semantics; the best fix should be reviewed as a safety-sensitive core backup workflow, not treated as a trivial command alias.
