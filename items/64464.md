## Summary

Keep open. Current main has a related Linux no-display fallback that runs local managed browser profiles headless by default and reports a clearer error/warning when headed mode is explicitly requested, but it does not implement the reported WSL2 visible Chrome behavior. The Chrome spawn environment still preserves the parent env and HOME only; it does not synthesize DISPLAY=:0 for headless:false WSL2 launches. The linked fix PRs were closed unmerged, and the maintainer comment correctly noted that the landed fallback does not close this exact request.

## Best Possible Solution

Keep the issue open for a targeted browser plugin fix or maintainer decision. The best implementation would add a small, tested Chrome spawn-env helper that preserves existing DISPLAY/WAYLAND_DISPLAY, applies any WSL-specific DISPLAY fallback only for local managed Linux/WSL visible launches with no display env, and updates browser troubleshooting docs to explain when users should set DISPLAY, use WSLg/Xvfb, or run headless.

## Risks / Open Questions

- A naive DISPLAY=:0 injection could overwrite or misroute users with non-default X/Wayland display setups if it does not preserve existing DISPLAY/WAYLAND_DISPLAY values.
- Visible WSL2 Chrome behavior depends on the user having a working Windows/WSLg/X server; setting DISPLAY alone may not be sufficient on every WSL configuration.
- The existing current-main behavior intentionally treats explicit headed no-display mode as an actionable configuration error, so changing it needs a narrow product decision and regression tests.
