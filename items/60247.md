## Summary

Keep #60247 open. Current main has a safer Control UI WebSocket parser that catches malformed JSON frames, and the gateway broadcasts each event as a single JSON-stringified WebSocket frame, but I did not find a source-backed fix or regression test that proves long CJK Control UI streaming content is preserved across the reported UTF-8 boundary failure. Dropping malformed frames would prevent the visible JSON.parse exception, but it is not the same as buffering/reassembling the reported split payload.

## Best Possible Solution

Keep the issue open and add a focused regression around the Control UI gateway path: send a long CJK chat delta/final event through the gateway/WebSocket client path and assert the UI receives the complete text without JSON.parse errors or truncation. If a malformed partial-frame path is still possible, fix it at the transport boundary with explicit buffering/reconnect diagnostics rather than only swallowing parse errors.
