## Summary

Keep open. Current `main` still does not validate `gateway.nodes.denyCommands` during `openclaw config set`/`unset`; unknown entries remain schema-valid strings and runtime policy treats them as exact-match no-ops. The PR’s core fix is not implemented on `main`, and main still only surfaces this as a post-write security-audit warning.

## Evidence

- **`config set` has no denyCommands validation hook on main:** `runConfigSet` builds `nextConfig`, runs SecretRef policy/schema dry-run handling, then writes via `replaceConfigFile(...)`. There is no `denyCommands`-specific validation step before write.
  - file: [src/cli/config-cli.ts:1195](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/cli/config-cli.ts#L1195)
  - command: `nl -ba src/cli/config-cli.ts | sed -n '1188,1340p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Runtime policy uses exact command-name matching only:** `resolveNodeCommandAllowlist` removes denied commands by exact trimmed string from the allow set. A typo/unknown deny entry is therefore a silent no-op at runtime.
  - file: [src/gateway/node-command-policy.ts:178](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/gateway/node-command-policy.ts#L178)

- **Schema still accepts any string array for denyCommands:** Current schema constrains `gateway.nodes.denyCommands` only to `string[]`; it does not validate against known command IDs.
  - file: [src/config/zod-schema.ts:923](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/config/zod-schema.ts#L923)
