## Summary

Keep PR #39489 open. Current main still lacks Feishu runtime connection/activity reporting into the gateway channel health snapshot, and this PR remains relevant rather than obsolete. It should be revised before merge because the current health policy uses `lastTransportActivityAt` for stale-socket detection, while the PR context only adds `connected`, `mode`, `lastEventAt`, and `lastError` reporting and does not add focused Feishu status tests.

## Best Possible Solution

Keep the PR open and ask for a rebase/revision that uses the existing channel status snapshot fields, threads a Feishu status sink from `startAccount` through `monitorFeishuProvider`, `monitorSingleAccount`, event handlers, and webhook/websocket transports, reports `connected`, `mode`, `lastConnectedAt`, `lastEventAt`, `lastTransportActivityAt`, and `lastError` at validated lifecycle/activity points, preserves the webhook guard-ordering fix from the PR discussion, and adds focused Feishu tests for websocket/webhook status behavior. Maintainers should also pick the canonical branch among the overlapping Feishu health PRs before requesting final changes.
