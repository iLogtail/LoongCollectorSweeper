## Summary

Issue #41760 reads as a one-off workstation/Tailscale outage, not a reproducible OpenClaw defect on current `main`. The report and follow-up both point to the remote host being offline and explicitly say no gateway-side action was needed.

## Evidence

- **Issue describes host outage, not app behavior:** The report says `rusty` was `offline` in `tailscale status`, with 100% ping loss and SSH timeouts. The follow-up comment adds `No action needed on gateway side - awaiting physical access to machine.` That is an external workstation availability incident, not a code-path failure in OpenClaw.
- **Current Tailscale docs assume the remote machine itself must be reachable:** `docs/gateway/tailscale.md:117` says remote browser control works by running a node host on the browser machine and keeping both machines on the same tailnet; if that machine is offline, browser automation is unavailable by design.
