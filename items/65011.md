## Summary

Keep #65011 open. Current main has real updater primitives, auto-update config, an update notification banner, and a generic Config > Updates section, but it still lacks the requested first-class Control UI Update Manager with status, auto-update toggles, dry-run/update progress, changelog link, last failure/success visibility, and post-update health summary. Related updater issues remain open rather than superseding this UI/discoverability tracker.

## Best Possible Solution

Keep #65011 open as the first-class Control UI/onboarding Update Manager tracker. The best path is to reuse the existing updater engine and config model, add the missing gateway/API surfaces for status, dry-run, release notes, persisted outcomes, and health summaries, then build a dedicated Control UI surface that can enable auto-update, explain channels, run updates with visible outcomes, and link narrower follow-ups for safe-update and skipped-result behavior.
