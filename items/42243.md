## Summary

Keep open. Current main has improved raw provider-error formatting in the embedded runner path, but cron announce delivery still treats an all-error cron result as deliverable content and sends it through the outbound adapter without a run-status or fatal-error guard. The linked attempted fix PR #46236 is closed unmerged, so there is no canonical landed fix for the requested "failed runs should not announce" behavior.

## Best Possible Solution

Keep the issue open and fix the cron isolated-agent delivery boundary: when `resolveCronPayloadOutcome` reports `hasFatalErrorPayload` or the final run outcome will be `status: "error"`, skip primary `delivery.mode: "announce"` payload delivery and record the run error/delivery state internally. Preserve explicit failure-notification paths (`failureAlert`/`failureDestination`) as a separate operator opt-in surface, and add a regression test covering a Codex/OpenAI `server_error` payload with `delivery.mode: "announce"` and `bestEffort` unset.

## Risks / Open Questions

- A broad raw-error regex guard could suppress legitimate cron reports that discuss errors; the safer fix is to use structured fatal-run/error-payload state rather than text matching alone.
- Cron failure alerts and `delivery.failureDestination` are now a separate documented feature, so the fix should suppress primary announce fallback for failed runs without breaking explicitly configured failure notifications.
