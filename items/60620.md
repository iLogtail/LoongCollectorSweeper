## Summary

Keep #60620 open. Current main has a related fix for broad runtime ensure calls after a gateway-bindable boot load, but the explicit deferred-channel startup path still reloads the same startup plugin scope and can re-run plugin registration for non-deferred sidecar/memory plugins during the same gateway boot.

## Best Possible Solution

Keep the issue open and fix the deferred-channel startup path so non-deferred startup plugins are registered once per gateway process. A conservative implementation would narrow the post-listen reload to only deferredConfiguredChannelPluginIds or otherwise merge/upgrade deferred channel records without re-running sidecar, memory, context-engine, or provider plugin register() calls. Add a regression test with a non-bundled startup sidecar or memory plugin plus a deferred configured channel and assert the non-deferred plugin register count stays at 1 while the deferred channel reaches full runtime after listen.
