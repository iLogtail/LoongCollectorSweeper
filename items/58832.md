## Summary

Keep open. Current main still does exactly what #58832 reports for Slack top-level DMs with `replyToMode="all"`: non-roomish DM routing derives `canonicalThreadId` from the message's own timestamp, and `resolveThreadSessionKeys` appends `:thread:<ts>` to the base DM session key. The earlier channel fix is present, but the DM case is not fixed; current tests and changelog also show this behavior was deliberately preserved, so this needs maintainer product judgment rather than automated cleanup.

## Best Possible Solution

Keep #58832 open for Slack plugin maintainer judgment. If the report is accepted, change `extensions/slack/src/monitor/message-handler/prepare-routing.ts` so top-level direct messages do not use the message timestamp as a canonical thread session id when the desired DM scope is the base peer session, then update the Slack routing tests, reply-threading tests, and docs/changelog to distinguish DM reply delivery from DM session scoping. If the current auto-threaded DM behavior is intentional, close only with an explicit maintainer rationale and point users to `replyToModeByChatType.direct` or `dm.replyToMode` for flat DM sessions.
