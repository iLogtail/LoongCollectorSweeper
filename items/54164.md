## Summary

Keep open. Current main still pre-resolves and forwards the stored gateway sessionId for keyed agent runs, so downstream session freshness can be bypassed exactly as reported. The related PR #57086 appears to target this bug, but it is open/unmerged and current main does not include that behavior change.

## Best Possible Solution

Keep #54164 open until the gateway agent handler stops pinning the stored sessionId for canonical sessionKey dispatch or explicitly evaluates freshness before dispatch. The likely fix belongs in src/gateway/server-methods/agent.ts with regression coverage in src/gateway/server-methods/agent.test.ts for a stale keyed sessions_send/inter_session run proving the old sessionId is not forwarded and rollover cleanup semantics are preserved.
