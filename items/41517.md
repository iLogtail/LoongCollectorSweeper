## Summary

Keep PR #41517 open. Current main has broader model-ref normalization in some session/config paths, but the embedded resolver still does not safely handle the PR's direct case where `resolveModel`/`resolveModelAsync` receives `provider="google"` and `modelId="google/gemini-2.5-pro"`. The PR is relevant, but the submitted implementation needs adjustment because review comments correctly flag overbroad stripping for valid nested same-provider model IDs such as NVIDIA models.

## Best Possible Solution

Keep the PR open and ask for a rebase/revision that fixes the resolver-level Google same-provider prefix case without stripping valid nested provider-native IDs. A safe direction is to canonicalize only where the provider contract expects bare model IDs, or only when the stripped suffix matches configured/discovered metadata, and add tests for Google same-provider stripping, OpenRouter cross-provider pass-through, and NVIDIA nested ID preservation.

## Risks / Open Questions

- Closing this PR as implemented would leave a resolver-level gap for hook/simple-completion/other call paths that pass a provider-qualified Google model after the provider is already known.
- The current PR patch is overbroad as submitted: it can strip valid nested model IDs whose first path segment matches the provider, so it needs a narrower rule and regression coverage before merge.
