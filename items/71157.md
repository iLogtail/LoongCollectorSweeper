## Summary

Current `main` does not implement a WhatsApp disappearing-message expiration setting or propagate an `ephemeralExpiration`-style option through the bundled WhatsApp outbound send paths. This is still an in-scope enhancement request for the bundled WhatsApp plugin, not something that should be conservatively auto-closed.

## Evidence

- **WhatsApp config types do not expose a disappearing-message expiration field:** The shared WhatsApp config shape includes delivery controls like `textChunkLimit`, `chunkMode`, `mediaMaxMb`, `ackReaction`, and `replyToMode`, but there is no `disappearingMessagesSeconds` or similar outbound expiration setting in the public TypeScript config surface.
  - file: [src/config/types.whatsapp.ts:53](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/config/types.whatsapp.ts#L53)
  - command: `nl -ba src/config/types.whatsapp.ts | sed -n '40,140p'`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **WhatsApp zod schema likewise has no expiration knob:** The runtime schema for WhatsApp accepts fields such as `sendReadReceipts`, `textChunkLimit`, `chunkMode`, `ackReaction`, `replyToMode`, `mediaMaxMb`, and `actions`, but no outbound disappearing-message expiration field at either channel or account scope.
  - file: [src/config/zod-schema.providers-whatsapp.ts:51](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/config/zod-schema.providers-whatsapp.ts#L51)
  - command: `nl -ba src/config/zod-schema.providers-whatsapp.ts | sed -n '30,170p'`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Outbound send options do not carry an expiration value:** `ActiveWebSendOptions` only supports `quotedMessageKey`, `gifPlayback`, `accountId`, and `fileName`. In `createWebSendApi`, outbound text/media sends build a Baileys payload and optionally quoted-message options, then call `sock.sendMessage(...)` without any expiration-related option. Poll sends also call `sock.sendMessage(...)` with only the poll payload.
  - file: [extensions/whatsapp/src/inbound/types.ts:12](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/extensions/whatsapp/src/inbound/types.ts#L12)
  - command: `nl -ba extensions/whatsapp/src/inbound/types.ts | sed -n '1,40p'`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Current docs document other WhatsApp delivery controls, not disappearing-message expiry:** The WhatsApp docs describe outbound media behavior, `mediaMaxMb`, and `replyToMode`, and the high-signal config list calls out `textChunkLimit`, `chunkMode`, `mediaMaxMb`, `sendReadReceipts`, `ackReaction`, and `reactionLevel`. There is no documented disappearing-message expiration setting, which matches the missing code/schema support.
  - file: [docs/channels/whatsapp.md:353](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/docs/channels/whatsapp.md#L353)
