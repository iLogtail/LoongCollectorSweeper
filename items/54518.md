## Summary

Keep open. Current main has an opt-in `tools.fs.workspaceOnly` policy and sandbox configurations, but the default multi-agent workspace is still documented and implemented as a cwd, not a hard filesystem boundary. With `workspaceOnly` unset/false, host read/write/edit paths can still reach outside the agent workspace, and broader isolation still requires sandboxing or denying runtime tools such as `exec`.

## Best Possible Solution

Keep this open for maintainer product judgment on the desired default and UX. A strong fix would provide a clear per-agent confinement profile or wizard option that sets `tools.fs.workspaceOnly: true`, denies or sandboxes runtime tools, and documents how skill setup should use the active agent workspace; if maintainers want a hard default boundary, that should be handled as an explicit behavior change with migration notes.

## Risks / Open Questions

- Changing the default to workspace-only would be a product compatibility decision because current behavior intentionally preserves legacy host-file access when unset.
- `tools.fs.workspaceOnly` does not by itself constrain shell/runtime tools; a complete no-outside-workspace posture also needs sandboxing or tool policy that denies `exec`/`process` and unsafe MCP/plugin tools.
