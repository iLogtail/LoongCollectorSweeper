## Summary

Keep open. Current `main` still treats silent replies as exact single-token matches, so `NO_REPLY\n\nNO_REPLY` is not recognized as silent and can leak through core normalization/delivery paths.

## Evidence

- **Exact-match silent token regex remains on main:** `isSilentReplyText()` still builds `^\\s*${token}\\s*$`, so it only accepts one token with surrounding whitespace.
  - file: [src/auto-reply/tokens.ts:16](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/auto-reply/tokens.ts#L16)
  - command: `nl -ba src/auto-reply/tokens.ts | sed -n '10,42p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Core normalization does not collapse repeated silent tokens:** `normalizeReplyPayload()` first checks exact silence, then only strips one trailing token from mixed text. For `NO_REPLY\n\nNO_REPLY`, the exact check fails and the trailing-strip path leaves a visible `NO_REPLY` behind instead of suppressing the payload.
  - file: [src/auto-reply/reply/normalize-reply.ts:57](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/auto-reply/reply/normalize-reply.ts#L57)
  - command: `nl -ba src/auto-reply/reply/normalize-reply.ts | sed -n '55,79p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Direct reproduction of the current regex behavior:** Replaying the current regexes shows the doubled token case is neither an exact silent match nor a leading-attached silent match, and the trailing-strip logic reduces it to a single visible `NO_REPLY`: `{"exact":false,"leading":false,"stripped":"NO_REPLY"}`.
  - command: `node -e "const exact=/^\\s*NO_REPLY\\s*$/i; const trailing=/(?:^|\\s+|\\*+)NO_REPLY\\s*$/i; const leading=/^\\s*(?:NO_REPLY\\s+)*NO_REPLY(?=[\\p{L}\\p{N}])/iu; const text='NO_REPLY\\n\\nNO_REPLY'; console.log(JSON.stringify({exact:exact.test(text), leading:leading.test(text), stripped:text.replace(trailing,'').trim()}));"`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Outbound directive parsing still relies on the same exact silent-token check:** `parseReplyDirectives()` marks payloads silent via `isSilentReplyPayloadText()`, which delegates to the same exact-match helper. A doubled token payload therefore stays non-silent in outbound planning.
  - file: [src/auto-reply/reply/reply-directives.ts:33](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/auto-reply/reply/reply-directives.ts#L33)
  - command: `nl -ba src/auto-reply/reply/reply-directives.ts | sed -n '33,47p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Public plugin runtime exports the exact-match helper unchanged:** Channel plugins consuming the SDK helper still receive `isSilentReplyText` from core without any repeated-token handling.
  - file: [src/plugin-sdk/reply-runtime.ts:30](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/plugin-sdk/reply-runtime.ts#L30)
  - command: `nl -ba src/plugin-sdk/reply-runtime.ts | sed -n '23,31p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)

## Risks / Open Questions

- Any fix should preserve the existing guard against suppressing substantive replies that merely end with `NO_REPLY` (`src/auto-reply/tokens.ts:39-41`).
- The same repeated-token edge case likely applies to `HEARTBEAT_OK`, because both control-token paths reuse the same exact-match helper.
