## Summary

Keep open. Current main covers adjacent OpenClaw-internal selected-vs-active fallback display, but it does not implement the central request here: capturing the backend model returned by an OpenAI-compatible proxy such as LiteLLM/OpenRouter and exposing it as separate resolved model/provider metadata.

## Best Possible Solution

Keep #51441 open. The best product path is to add a separate backend-resolution contract: capture returned model identity from OpenAI-compatible Chat Completions/Responses where providers expose it, thread it through agent runtime metadata without overwriting the selected/requested model, persist it on session state/GatewaySessionRow, and surface it in `session_status`, `/status`, and runtime context. Coordinate naming and semantics with the still-open OpenRouter actual-model/cost requests so proxy routing, OpenClaw failover, and selected model state remain distinct.
