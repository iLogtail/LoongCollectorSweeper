## Summary

Keep #61445 open. Current main still creates a fresh in-memory iMessage sent-message cache per monitor start, and the self-chat `is_from_me=true` path only drops when that cache has a match. With an empty cache after restart, the current decision logic can still dispatch the same message shape described by the issue. Related PR #62191 targets a separate U+FFFD/control-character echo-key normalization problem, not startup cache warming or a rowid fence.

## Best Possible Solution

Keep the issue open and fix it in the bundled iMessage plugin. The likely path is a targeted startup guard in `extensions/imessage/src/monitor`: record a Messages DB high-water rowid or persist outbound echo identifiers, apply that fence before `resolveIMessageInboundDecision` dispatches self-chat `is_from_me=true` rows, and add a regression test for gateway restart with an empty sent-message cache while preserving the existing real self-chat dispatch behavior.

## Risks / Open Questions

- The fix needs to distinguish replayed outbound self-chat rows from legitimate real user self-chat messages, because current tests intentionally allow the empty-cache self-chat dispatch case.
- A broad startup drop of all `is_from_me=true` self-chat rows could hide valid user messages; a rowid/high-water fence or persisted outbound identity cache is safer.
