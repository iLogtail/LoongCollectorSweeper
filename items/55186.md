## Summary

Keep open. Current `main` still suppresses `threadRootId` when `replyToMode` is `"off"`, and the test suite explicitly locks that behavior in. This PR is not already implemented on `main`, but the proposed one-line change is also not safe evidence for closure because the same helper drives both reply delivery and thread session-key derivation.

## Evidence

- **Current main still drops thread roots when replyToMode is off:** `resolveMattermostEffectiveReplyToId()` returns `threadRootId` only when `replyToMode !== "off"`; otherwise threaded messages fall through and direct messages return `undefined`.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:349](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/mattermost/src/mattermost/monitor.ts#L349)
  - command: `nl -ba extensions/mattermost/src/mattermost/monitor.ts | sed -n '338,392p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Session routing is coupled to that same helper:** `resolveMattermostThreadSessionContext()` calls `resolveMattermostEffectiveReplyToId()` and uses the result as `threadId` when deriving `sessionKey` and `parentSessionKey`, so removing the guard would also fork sessions for `replyToMode: "off"` threads.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:373](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/mattermost/src/mattermost/monitor.ts#L373)
