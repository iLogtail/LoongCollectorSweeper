## Summary

Keep open. Current `main` still sends a user-facing warning for mutating tool validation failures such as `write` with missing `content`: parameter validation throws an internal error, the adapter converts it to a tool error result, the subscription stores it as `lastToolError`, and payload construction deliberately shows mutating tool failures as `⚠️ ✍️ Write failed` in normal chat surfaces while hiding only the raw detail suffix unless verbose is enabled. Related ClawSweeper context for #68070 also says parameter-validation retryability plumbing remains unshipped, so this issue is not obsolete.

## Best Possible Solution

Keep the issue open and implement a narrow validation-error classification path. Parameter validation failures should still be returned to the agent/runtime and logged, but `buildEmbeddedRunPayloads` should suppress user-facing warning payloads for validation-class failures even when the tool is mutating, with regression tests for `write` missing `content`, `exec` missing `command`, and a non-validation mutating failure that must remain visible.
