## Summary

Current `main` does not implement cron job tags/categories. The persisted cron job model, Gateway RPC schema, server-side list API, CLI output, and Control UI filters all lack a `tags` field or tag-based filtering, so this remains valid feature work rather than maintainer cleanup.

## Best Possible Solution

Keep the issue open. The best implementation is an additive core cron contract change: add optional normalized `tags?: string[]` to the cron job model, Gateway add/update/list schemas, persistence/normalization, CLI add/edit/list/show, Control UI create/edit/display/filter controls, and docs/tests. Tag-filtered bulk enable/disable can either be included behind the same list/update contract or tracked as a follow-up if the initial tags field lands first.
