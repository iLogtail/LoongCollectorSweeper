## Summary

Keep open. Current main improves embedded-run timeout handling by making idle timeouts configurable, retrying one silent idle timeout, and returning explicit user-facing timeout payloads, but it still does not emit the plugin-observable `lifecycle:error` agent event with structured `errorType: "timeout"` requested by #52353 when the embedded run timeout handler fires.

## Best Possible Solution

Keep #52353 open. The best fix is to add a single, deduplicated embedded-run timeout lifecycle event at the timeout/abort boundary, with a backwards-compatible event shape such as `stream: "lifecycle"`, `data.phase: "error"`, `data.errorType: "timeout"`, user-readable `error`, duration/timing, provider/model/profile context where available, and regression coverage proving `api.runtime.events.onAgentEvent()` receives it without double-emitting for normal assistant errors, cancellation, compaction grace, or later timeout payload synthesis.
