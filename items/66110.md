## Summary

Keep open. Current `main` does not implement root `--log-file` / `OPENCLAW_LOG_FILE`, and the PR discussion identifies real correctness gaps in the proposed patch: routed commands return before Commander `preAction` runs, and daemon status currently reports the CLI config path rather than the inspected service's effective env/config.

## Evidence

- **Current main still routes before Commander preAction:** `runCli()` returns through `tryRouteCli()` before the Commander program is built. Any env promotion added only in `registerPreActionHooks()` would be skipped for routed commands such as `status` and `config get`.
  - file: [src/cli/run-main.ts:206](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/cli/run-main.ts#L206)
  - command: `nl -ba src/cli/run-main.ts | sed -n '180,215p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Existing preAction hook only promotes --log-level:** The root-option handling in `registerPreActionHooks()` currently reads `logLevel` and sets `OPENCLAW_LOG_LEVEL`; there is no existing routed-command path that would apply a root `--log-file` override before `tryRouteCli()` exits.
  - file: [src/cli/program/preaction.ts:83](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/cli/program/preaction.ts#L83)
  - command: `nl -ba src/cli/program/preaction.ts | sed -n '35,95p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Daemon status currently reports CLI config log path:** `gatherDaemonStatus()` returns `logFile: resolveConfiguredLogFilePath(cliCfg)`, even though it separately computes `mergedDaemonEnv` for the inspected service. That matches the review concern that status can drift from the service's actual environment.
  - file: [src/cli/daemon-cli/status.gather.ts:465](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/cli/daemon-cli/status.gather.ts#L465)
  - command: `nl -ba src/cli/daemon-cli/status.gather.ts | sed -n '450,475p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Current main has no root --log-file or OPENCLAW_LOG_FILE surface:** Root CLI options still only include `--profile`, `--log-level`, and `--container`; root help also only documents `--log-level`. The public docs list `OPENCLAW_LOG_LEVEL` but not `OPENCLAW_LOG_FILE`, and gateway logging docs only mention `logging.file` / `logging.level`.
  - file: [src/infra/cli-root-options.ts:4](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/infra/cli-root-options.ts#L4)

  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **PR discussion already identified unresolved correctness gaps:** The provided review comments call out two material issues: routed commands bypass the new `preAction`-only env promotion, and daemon status should resolve the service's merged environment rather than the caller's process env. Those are substantive review findings, not cleanup nits.

## Risks / Open Questions

- Merging the PR as written would likely leave `openclaw --log-file ... status` and other route-first commands ignoring the override entirely.
- Even if log-file override support is added, status output can still misreport the effective service log path unless the resolution path accepts the inspected service environment explicitly.
