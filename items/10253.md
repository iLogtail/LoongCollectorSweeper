## Summary

Keep open. Current main still does not provide a configurable hook/webhook session lifecycle policy: hook config has no cleanup surface, hook dispatch still runs isolated cron-style sessions without delete-after-run or hook retention settings, and the dedicated cron reaper only targets cron run aliases, not `hook:` session keys. Generic `session.maintenance` can prune old sessions globally, but it does not satisfy the requested hook-specific on-complete/delayed cleanup, transcript-retention, or session-list behavior.

## Best Possible Solution

Keep #10253 open. The best product path is a maintainer-approved hook session lifecycle policy in core, likely under `hooks`, with schema/help/docs updates, dispatch/finalization wiring for `/hooks/agent` and Gmail mappings, explicit archive/delete/transcript-retention semantics, and focused tests covering per-message Gmail keys, delayed cleanup, session-list visibility, and interaction with existing `session.maintenance` and cron retention.
