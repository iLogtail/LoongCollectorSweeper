## Summary

Keep #14376 open. Current main has useful adjacent pieces, including `lastErrorReason`, one-shot retry config, generic recurring backoff, failure alerts, and provider auth-profile cooldowns, but recurring cron still lacks the requested reason-aware guard policy. The issue discussion also includes real retry-storm reports and an Apr 26, 2026 maintainer review explicitly keeping this open; current main has no relevant diff from that reviewed surface.

## Best Possible Solution

Keep #14376 open until recurring cron scheduling has an explicit reason-aware guard policy. The likely implementation should update `src/cron/service/timer.ts`, cron state/config types, gateway protocol/schema, UI/docs, and tests: classify terminal run errors once, persist/display reason and cooldown state, respect Retry-After where available, auto-pause or circuit-break recurring jobs for billing/auth/format with safe configurable defaults and one alert, and keep timeout/network/rate-limit jobs enabled with bounded backoff coordinated with existing auth-profile cooldowns.
