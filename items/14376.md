## Summary

Keep #14376 open. Current main has adjacent partial work: cron records lastErrorReason, one-shot jobs retry transient errors and disable on permanent/exhausted errors, recurring jobs get generic consecutive-error backoff, and failure alerts exist. It still lacks the requested reason-aware recurring cron guard policy: billing/auth/format circuit breaking, Retry-After-aware rate-limit scheduling, job-level cooldown/disabled-until state, and configurable reason thresholds.

## Best Possible Solution

Keep #14376 open until recurring cron scheduling has an explicit reason-aware guard policy. The likely path is src/cron/service/timer.ts, cron state/config types, gateway protocol/schema, UI/docs, and tests: classify terminal run errors once, persist/display the reason, respect Retry-After where provider errors expose it, auto-pause or circuit-break recurring jobs for billing/auth/format with configurable safe defaults and one alert, keep timeout/network/rate-limit jobs enabled with bounded backoff, and coordinate with existing auth-profile cooldowns/provider-wide rate-limit handling.
