## Summary

Keep #14376 open. Current main has partial pieces, including `lastErrorReason`, one-shot transient retries, generic recurring backoff, failure alerts, and auth-profile cooldowns, but recurring cron still lacks the requested reason-aware guard policy: no billing/auth/format circuit breaker, no Retry-After-aware recurring scheduling, and no job-level cooldown/disabled guard state. The supplied discussion also includes concrete retry-storm reports and an Apr 26 maintainer review already keeping the issue open.

## Best Possible Solution

Keep #14376 open until recurring cron scheduling has an explicit reason-aware guard layer. The likely implementation spans `src/cron/service/timer.ts`, cron state/config types, gateway protocol/schema, UI/docs, and regression tests: classify terminal run errors once, persist/display reason and guard state, respect Retry-After where available, auto-pause or circuit-break recurring jobs for billing/auth/format with safe configurable defaults and one alert, and keep timeout/network/rate-limit jobs enabled with bounded backoff coordinated with existing auth-profile cooldowns.
