## Summary

Current main already solves the PR's observable Slack bug through the shipped inbound placeholder path: Slack file references include the native Slack `fileId`, downloaded and fallback file-only messages put those placeholders into `RawBody`/`BodyForAgent`, and the reply prompt uses `BodyForAgent` plus media notes. The Slack docs also now state that `download-file` accepts the file IDs shown in inbound placeholders. The exact proposed `MediaFileIds` field is not present, but the user problem described by the PR is covered on main and in release `v2026.4.24`.

## Best Possible Solution

Close this PR as obsolete on current main. Keep the shipped placeholder-based Slack workflow unless a maintainer wants a separate, tested cross-channel structured media-ID API; that would need downstream prompt/tool-context consumption, not just adding `MediaFileIds` to `MsgContext`.
