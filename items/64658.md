## Summary

Keep open. Current main still does not expose the requested `sessions_spawn` announcement control for top-level channel-bound sessions. The advertised tool schema and docs list no `announce`/`completionDelivery` parameter, and the implementation still registers normal run-mode subagents for completion announcements back to the requester origin. Existing behavior intentionally suppresses external delivery only for nested requester subagent sessions or thread-bound persistent session spawns, which does not solve the reported iMessage/DM top-level orchestration use case.

## Best Possible Solution

Add an explicit, documented `sessions_spawn` delivery policy for native and ACP run-mode spawns, likely a small enum such as `completionDelivery: "channel" | "parent" | "none"` with current `channel` behavior preserved as default. The implementation should thread that value through tool schema, spawn registration, task delivery status, announce dispatch, docs, and regression tests for top-level channel-bound sessions such as iMessage/BlueBubbles or WhatsApp. Reuse the existing nested-subagent internal injection path for the `parent` behavior where possible, rather than adding a separate polling-only mechanism.
