## Summary

Keep #72314 open. Current main still does not dedupe auto-reply inbound retries when `MessageSid` is missing: `buildInboundDedupeKey()` returns `null`, `claimInboundDedupe()` reports `invalid`, and dispatch proceeds instead of treating the retry as duplicate/in-flight. The PR is recent, focused, includes tests for that missing idless path, and the author already addressed the bot review concern about preserving legacy provider-id key suffixes.

## Best Possible Solution

Keep #72314 open for maintainer review. The best path is to land or refine the PR's content/timestamp fallback in `src/auto-reply/reply/inbound-dedupe.ts`, preserving the current raw `MessageSid` key format for provider-id messages and keeping regression tests for idless retries plus same-body/different-timestamp non-collapse.
