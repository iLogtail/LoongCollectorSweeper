## Summary

Keep PR #72314 open. Current main still treats inbound messages without `MessageSid` as invalid for inbound dedupe, so the PR's central idless-retry fallback is not implemented or superseded. The related poison-dedupe work on main covers replay-unsafe failures after a valid dedupe claim, but it does not create a key for idless retries.

## Best Possible Solution

Keep PR #72314 open and continue review toward a narrow implementation in `src/auto-reply/reply/inbound-dedupe.ts` that preserves the existing provider `MessageSid` key format while adding a deterministic content/timestamp fallback for idless retries, with regression tests for duplicate idless retries and distinct same-body messages at different timestamps.
