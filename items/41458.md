## Summary

Current main does not match the reported Feishu TTS failure path. Feishu outbound media is uploaded to Feishu first via `im.file.create`, then sent using the returned `file_key`; Feishu is also explicitly treated as a native voice-note TTS channel, with tests and docs covering Opus/audio delivery. The issue's stated root cause (passing a local file path directly to Feishu without upload) is not true on current main or the latest release snapshot inspected.

## Evidence

- **Feishu upload-before-send path on main:** `sendMediaFeishu` uploads non-image media with `uploadFileFeishu(...)`, which calls `client.im.file.create(...)`, then sends the returned `file_key` via `sendFileFeishu(...)`. This is the opposite of the issue's claimed root cause.
  - file: [extensions/feishu/src/media.ts:367](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/extensions/feishu/src/media.ts#L367)
