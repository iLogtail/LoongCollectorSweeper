## Summary

Keep open. Current main still preserves `--history-limit 0` through the TUI history path and sends it as `chat.history.limit: 0`, while the Gateway protocol validates `chat.history.limit` with `minimum: 1`. The PR is a narrow fix for the implicated TUI path and adds the missing regression coverage; related history-limit reports in the provided context cover group/channel config, not this TUI Gateway request path.

## Best Possible Solution

Keep the PR open and land this narrow TUI-side guard plus regression test, or recreate the same behavior on main. The Gateway protocol should continue rejecting `chat.history.limit: 0`; the TUI should interpret `--history-limit 0` locally as skip history loading and refresh the local session marker without calling `chat.history`.
