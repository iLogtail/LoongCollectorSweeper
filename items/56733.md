## Summary

Keep #56733 open. Current main has targeted mitigations for related surfaces, but I did not find a source-backed implementation of the issue’s central request: detecting or recovering from a gateway-wide event-loop freeze where Feishu and provider HTTP requests all silently time out while the process remains alive. The closed #65517 active-memory/Telegram report is useful related evidence, but it is narrower and does not supersede the Feishu/LLM/systemd zombie-gateway failure described here.

## Best Possible Solution

Keep the issue open as a gateway resilience bug. A conservative maintainer path is to add source-backed diagnostics/recovery for the reported class: gateway event-loop delay monitoring or an external supervisor-compatible health signal, Feishu transport liveness/status publication if Feishu websocket idleness is a trigger, and tests or documented operational guidance showing how a silent hung gateway is detected and recovered. Treat #65517 as related narrower evidence, not as the canonical close target for #56733.

## Evidence

- **Gateway health monitor is snapshot-based, not an event-loop freeze watchdog:** The monitor runs from an in-process setInterval and restarts channels only after evaluating channel runtime snapshots. If the Node event loop is frozen, this timer cannot run; and even when it runs, it only reacts to status fields surfaced by channels.

## Risks / Open Questions

- The original report is environment-specific to WSL2, systemd user services, Feishu, and low-activity hours; a final fix should be validated with fresh diagnostics or a reproducible stall harness.
- Existing Telegram and active-memory mitigations may remove some related triggers, so maintainers should avoid assuming one global root cause without isolating the Feishu/provider HTTP freeze path.
