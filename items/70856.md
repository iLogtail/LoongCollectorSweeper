## Summary

Keep open. Current main includes the related #70678 quiet-session watchdog fix, but #70856 still describes broader WhatsApp listener instability: 408/428 reconnect churn, no-listener outbound gaps, health-monitor overlap, Bad MAC/libsignal errors, and a newer Linux/Docker group-delivery failure with `Failed hydrating participating groups on connect: 1006`. Those remaining symptoms are not fully implemented away on main.

## Best Possible Solution

Keep #70856 open as a real WhatsApp plugin regression. The best maintainer path is to add focused coverage for retryable 408/428 reconnects, ensure the generic health monitor does not restart WhatsApp while its own reconnect loop is healthy and bounded, make direct outbound sends wait or queue through short reconnect gaps, and investigate/recover from `groupFetchAllParticipating` 1006 plus libsignal `Bad MAC` failures so group events and sends do not silently disappear.

## Evidence

- **Issue discussion has active cross-platform reproduction:** The provided comments include the original Windows/npm global report plus a 2026-04-26 Linux/Docker confirmation. The later comment says direct WhatsApp DMs still work while group messages produce zero ingested `@g.us`/group events and repeatedly hit hydrate-1006 -> 30m idle -> 499 reconnect, so this is not just the already-fixed quiet Windows-only case.
- **Quiet-session slice fixed, but only as a related partial:** Current main changelog records the #70678 fix: WhatsApp/Web now bases the watchdog on WhatsApp Web transport activity so quiet healthy linked-device sessions are not restarted solely for lack of inbound messages. That maps to the issue's 30-minute idle/status-499 symptom, but not to 408/428 disconnects, group delivery loss, Bad MAC, or direct outbound no-listener gaps.
