## Summary

Keep open. The issue is recent, has concrete Windows/WhatsApp logs, and current main still contains the same reconnect/no-active-listener and watchdog paths. Related closed reports cover older outbound-listener and delivery-retry fixes, and #49305 covers the narrower 401/logout cleanup path, but this report also includes 408/428/499 reconnect stalls and a Bad MAC/libsignal crash shape that is not clearly fixed or superseded.

## Best Possible Solution

Keep #70856 open as a current WhatsApp reliability bug. Maintainers should investigate the Windows Baileys/libsignal Bad MAC path and the interaction between watchdog reconnects, health-monitor restarts, and direct outbound sends during reconnect gaps. A good fix would add regression coverage for libsignal-only Bad MAC recovery, clarify or change direct-send behavior while reconnect is pending, and coordinate with #49305 only for the separate 401/logged-out credential cleanup path.
