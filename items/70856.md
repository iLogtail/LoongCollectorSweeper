## Summary

Keep open. Current main partially addresses the related quiet-session watchdog issue (#70678), but #70856 still describes broader WhatsApp listener instability: 408/428 reconnect churn, outbound gaps while the listener is inactive, health-monitor overlap during reconnect, libsignal/Bad MAC symptoms, and an active Linux/Docker report where group events disappear after `groupFetchAllParticipating` fails with 1006.

## Best Possible Solution

Keep #70856 open as a real WhatsApp plugin regression. The best maintainer path is to preserve the #70678 quiet-session transport-activity fix, then add focused coverage and fixes for retryable 408/428 reconnect churn, prevent the generic health monitor from restarting WhatsApp while its own reconnect loop is healthy and bounded, make outbound sends wait or queue through short reconnect gaps, and investigate/recover from `groupFetchAllParticipating` 1006 plus libsignal Bad MAC failures so group events and sends do not silently disappear.

## Evidence

- **Active cross-platform reproduction in issue discussion:** The provided issue discussion includes the original Windows/npm global report plus a later Linux/Docker confirmation. The later report says direct WhatsApp DMs still ingest, but group messages produce zero `@g.us`/group events and repeatedly hit hydrate-1006 -> 30m idle -> 499 reconnect, so this is not only the already-fixed quiet-device watchdog case.
- **Quiet-session watchdog was only a partial fix:** Current main changelog records the #70678 fix: WhatsApp/Web now bases the watchdog on WhatsApp Web transport activity while retaining a longer app-silence cap. That maps to the 30-minute idle/status-499 slice, but not to 408/428 disconnects, group delivery loss, Bad MAC/libsignal, or outbound no-listener windows.
