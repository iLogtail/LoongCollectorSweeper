## Summary

Keep open. Current main still resolves `openclaw tui` Gateway targets from config and process env only; it does not apply an active Gateway service/runtime port before building the local loopback URL. The related fix attempts in the provided timeline are not merged into main: #42496 is open/unmerged, while #42666 and #46115 are closed/unmerged.

## Best Possible Solution

Keep the issue open. The best fix is to make the TUI Gateway connection resolver apply the active local Gateway runtime port before calling `buildGatewayConnectionDetails()`, preferably by sharing the existing daemon service-argument/env resolution used by status/lifecycle flows. Add a focused regression in `src/tui/gateway-chat.test.ts` proving a running Gateway on `--port 48789` resolves to `ws://127.0.0.1:48789`, while preserving explicit `--url` and env/config precedence, then update the TUI/gateway docs if the behavior is user-facing.
