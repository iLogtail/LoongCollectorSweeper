## Summary

Keep open. Current `main` still builds reply payloads from collected `assistantTexts`, messaging-tool dedupe only removes textual duplicates, same-target suppression only drops the final auto-reply payloads after that step, and the proposed `replyMode/tool-only` config is not present on `main`.

## Evidence

- **Embedded runner still emits all collected assistant texts:** `buildEmbeddedRunPayloads` filters `assistantTexts` to `nonEmptyAssistantTexts`, then uses those texts as `answerTexts` and pushes each one into `replyItems`. That means non-duplicate narration text remains eligible for outbound delivery.
  - file: [src/agents/pi-embedded-runner/run/payloads.ts:296](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/agents/pi-embedded-runner/run/payloads.ts#L296)
  - command: `nl -ba src/agents/pi-embedded-runner/run/payloads.ts | sed -n '121,380p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Dedupe is text-based, not blanket suppression:** `filterMessagingToolDuplicates` only filters payloads when `isMessagingToolDuplicate(payload.text, sentTexts)` returns true. Non-matching narration text is not removed by this path.
  - file: [src/auto-reply/reply/reply-payloads-dedupe.ts:13](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/auto-reply/reply/reply-payloads-dedupe.ts#L13)

  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)

[] : filteredPayloads`. This suppresses the final auto-reply payloads, but it does not change the earlier `buildEmbeddedRunPayloads` behavior that created assistant-text payload candidates in the first place.
