## Summary

Keep open. Current `main` does not show a conservative fix for repeated Control UI image submits: the client still sends full base64 attachments with a fresh UUID per send, and gateway dedupe only short-circuits identical `idempotencyKey` values. That means the reported duplicate-submit/session-bloat failure mode is still plausible on current code.

## Evidence

- **Control UI send path still posts raw base64 image attachments with a fresh per-send idempotency key:** `sendChatMessage` generates a new UUID for every send and passes `buildApiAttachments(params.attachments)` directly into `chat.send`; attachments are serialized from `data:` URLs into base64 content, with no client-side content dedupe or image shrinking in this path.
  - file: [ui/src/ui/controllers/chat.ts:464](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/ui/src/ui/controllers/chat.ts#L464)

- **UI busy-state logic is not a duplicate-submit fix:** `handleSendChat` only queues when a run is already busy; otherwise it immediately calls `sendChatMessageNow`. A repeated trigger after the prior request clears `chatSending` would create another distinct send with another UUID.
