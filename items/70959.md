## Summary

Keep open. Current main has shipped/landed part of the adjacent fix: duplicate attachment-backed `chat.send` calls with the same idempotency key now return the documented in-flight path. That does not fully cover #70959’s reported failure mode: repeated same-content image sends with fresh run IDs are still accepted, typical ~1.6 MB pasted images can still be sent inline to the provider, and overflow recovery still retries the same image-bearing prompt without an evident image-drop/resize fallback.

## Best Possible Solution

Keep #70959 open and fix the remaining product boundary: add a Control UI in-flight guard keyed by the submitted draft plus attachment identities, add a gateway-side short-window duplicate-content/image digest guard or coalescing path for different idempotency keys, resize or offload inbound webchat images before provider submission, and make overflow recovery drop/downscale already-processed current-turn image payloads rather than retrying the same oversized request body. Treat #70139/8bc4d4bc as already-landed supporting work, not the canonical close for this report.
