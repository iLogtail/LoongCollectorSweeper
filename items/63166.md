## Summary

Current `main` already protects the reported gateway-startup hook-runner path: equivalent later default-mode loads reuse the active `gateway-bindable` registry, gateway tool/runtime entrypoints explicitly request gateway-bindable loading, and tests cover the default-vs-gateway-bindable reuse behavior. The shipped `v2026.4.23` release already contains these guards.

## Evidence

- **Gateway-bindable/default registry compatibility guard:** `getCompatibleActivePluginRegistry()` now treats a later default-mode load as compatible with the active registry when the only difference is `allowGatewaySubagentBinding`, so the loader reuses the startup registry instead of replacing it.

- **Hook runner still follows activation target:** Activation still initializes the global hook runner from whichever registry is activated, which means the effective fix is preventing conflicting later activations on gateway paths rather than adding a separate hook-runner pin.
  - file: [src/plugins/loader.ts:1831](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/plugins/loader.ts#L1831)

## Risks / Open Questions

- This is not a universal hook-runner pin: a future gateway path that performs a truly different activating default-mode load without `allowGatewaySubagentBinding` could regress the same class of bug.
