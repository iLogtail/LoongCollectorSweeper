## Summary

Keep open. Current main still intentionally gates macOS `screen.record`: the macOS default allowlist includes `screen.snapshot` but not `screen.record`, and the regression test asserts `screen.record` remains blocked by default. PR #57192 is directionally useful for linked bug #57169, but its own review identified a real gap: wizard-generated configs seed `denyCommands` from `DEFAULT_DANGEROUS_NODE_COMMANDS`, and `denyCommands` is applied after platform defaults, so the PR as described would not allow `screen.record` end-to-end for new gateway setups. The unrelated OpenAI WebSocket empty-tools change is already present on current main through the shared warm-up payload helper.

## Best Possible Solution

Keep the PR and linked bug open, but revise the patch so macOS `screen.record` is allowed by default end-to-end. The fix should update `src/gateway/node-command-policy.ts`, wizard seeding in `src/wizard/setup.gateway-config.ts`, and the related gateway/wizard/audit tests so both an empty config and a wizard-generated config allow `screen.record` for `macOS 26.4.0` while preserving intentional deny/audit behavior for other high-risk commands and non-macOS platforms.
