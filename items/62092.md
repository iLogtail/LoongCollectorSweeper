## Summary

PR #62092 is still useful and not implemented on current main. The current memory-core auto-selection path still treats an adapter that omits shouldContinueAutoSelection as a hard failure, so a remote third-party adapter auth/config error can abort provider initialization before FTS-only search is reached. The submitted patch needs maintainer review/changes because the existing review correctly notes that transport is the safer predicate than adapter id, but this is a plausible focused bug fix rather than a cleanup-close candidate.

## Best Possible Solution

Keep the PR open and request changes: implement the fallback heuristic in memory-core using adapter.transport !== "local", add focused tests for remote third-party adapters that omit shouldContinueAutoSelection and for local-transport adapters that should not be soft-skipped, and consider whether src/gateway/embeddings-http.ts should receive the same contract behavior for consistency.

## Risks / Open Questions

- The patch as submitted could mask auth-looking errors from a custom local-transport adapter if it keeps the adapter.id !== "local" guard instead of using adapter.transport !== "local".
- The same explicit-only auto-selection helper exists in src/gateway/embeddings-http.ts, so maintainers should decide whether the OpenAI-compatible embeddings endpoint needs parity or whether this PR should stay scoped to memory_search.
- The heuristic should stay narrow and covered by regression tests so it does not turn unrelated provider failures into silent skips.
