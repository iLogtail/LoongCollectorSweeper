## Summary

Current main still has the reported webhook session-continuity bug. `/hooks/agent` dispatches an isolated cron job with the caller-approved `sessionKey`, but `prepareCronRunContext` still passes `forceNew: input.job.sessionTarget === "isolated"` to `resolveCronSession`, and `resolveCronSession` only reuses an existing `sessionId` when `forceNew` is false. The linked PR #70895 directly targets this but is open/unmerged in the provided GitHub context, so the issue should remain open.

## Best Possible Solution

Keep #70894 open and land #70895 or an equivalent maintainer-reviewed patch that passes `forceNew: false` for webhook/hook-sourced isolated agent runs with accepted stable session keys, while preserving forced fresh sessions for ordinary isolated cron jobs. Add a focused regression test proving webhook-sourced isolated jobs can reuse fresh sessions and non-hook isolated cron still forces a new one.
