## Summary

Keep open. Current `main` still routes `/hooks/agent` through an isolated cron job and then forces a fresh session for `sessionTarget: "isolated"`, so the reported loss of webhook conversation continuity is not implemented/fixed on `main`. Docs and tests also describe that isolated behavior as intentional, which means this needs maintainer product judgment rather than conservative auto-close.

## Evidence

- **Hook endpoint always dispatches isolated jobs:** `/hooks/agent` constructs a `CronJob` with `sessionTarget: "isolated"` even when a request/session mapping provides a stable `sessionKey`.
  - file: [src/gateway/server/hooks.ts:64](https://github.com/openclaw/openclaw/blob/734748d4f4fef7436def57f7a7ae3350bba9a091/src/gateway/server/hooks.ts#L64)
  - command: `nl -ba src/gateway/server/hooks.ts | sed -n '43,90p'`
  - sha: [734748d4f4fe](https://github.com/openclaw/openclaw/commit/734748d4f4fef7436def57f7a7ae3350bba9a091)
- **Isolated cron runs force a new session:** The isolated runner passes `forceNew: input.job.sessionTarget === "isolated"` into cron session resolution, so webhook agent jobs skip normal session reuse.
  - file: [src/cron/isolated-agent/run.ts:487](https://github.com/openclaw/openclaw/blob/734748d4f4fef7436def57f7a7ae3350bba9a091/src/cron/isolated-agent/run.ts#L487)
