## Summary

Current main still routes `/hooks/agent` through an isolated cron job and still calls `resolveCronSession` with `forceNew: true` for every isolated job. That bypasses the existing session reuse path even when hooks accept a stable `sessionKey`, so #70894 remains a real bug. The linked PR #70895 is directly relevant but is open and unmerged, so it does not make the issue cleanup-closeable.

## Best Possible Solution

Keep #70894 open and review a focused fix like #70895: preserve forced fresh sessions for ordinary isolated cron jobs, but allow webhook-sourced isolated runs with an allowed stable `sessionKey` or hook external-content source to pass `forceNew: false`. Add a regression test at the `runCronIsolatedAgentTurn` or gateway hook boundary proving repeated `/hooks/agent` calls with the same allowed `sessionKey` reuse the existing session ID, and update hook docs if `sessionKey` should be listed among supported `/hooks/agent` fields.
