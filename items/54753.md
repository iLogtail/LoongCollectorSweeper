## Summary

Keep open. Current `main` still has a routing mismatch for Slack thread interactions: inbound Slack thread replies resolve to `:thread:` session keys, but Slack block-action system events still resolve only from channel/user scope and ignore the interaction `threadTs`. That matches the reported failure mode for `message` tool interactive buttons rendered in a thread.

## Evidence

- **Inbound thread replies are thread-scoped:** Slack inbound routing derives `canonicalThreadId` from `thread_ts` and builds a thread session key from it; the adjacent test asserts `:thread:1770408518.451689` for a thread reply.
  - file: [extensions/slack/src/monitor/message-handler/prepare-routing.ts:65](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/slack/src/monitor/message-handler/prepare-routing.ts#L65)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare-routing.ts | sed -n '55,120p'; nl -ba extensions/slack/src/monitor/message-handler/prepare.thread-session-key.test.ts | sed -n '88,100p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **System-event session resolver has no thread input:** `resolveSlackSystemEventSessionKey` only accepts `channelId`, `channelType`, and `senderId`, then returns the base route session key; it has no way to select the thread session used by inbound thread turns.
  - file: [extensions/slack/src/monitor/context.ts:178](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/slack/src/monitor/context.ts#L178)
  - command: `nl -ba extensions/slack/src/monitor/context.ts | sed -n '170,215p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Block-action handler drops thread scope at enqueue time:** The Slack interaction payload includes `threadTs`, but `enqueueSlackBlockActionEvent` calls `resolveSlackSystemEventSessionKey` without passing any thread identifier before enqueueing the system event.
  - file: [extensions/slack/src/monitor/events/interactions.block-actions.ts:614](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/slack/src/monitor/events/interactions.block-actions.ts#L614)
  - command: `nl -ba extensions/slack/src/monitor/events/interactions.block-actions.ts | sed -n '612,646p'`

- **Existing regression test does not verify thread-session delivery:** The reply-button test uses a threaded Slack action payload (`thread_ts: "100.100"`) but only checks that `enqueueSystemEvent` was called, not that it targeted the thread session.
  - file: [extensions/slack/src/monitor/events/interactions.test.ts:463](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/slack/src/monitor/events/interactions.test.ts#L463)
