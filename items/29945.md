## Summary

Current main still has the reported behavior: DM pairing creates a user-visible branded OpenClaw reply, shared challenge issuance always sends a reply for newly created pairing requests, and there is no `unpairedResponse`/silent/code-only/branded config surface. The directly linked maintainer PR #39065 remains open and unmerged, so this issue should stay open.

## Best Possible Solution

Keep #29945 open until #39065 or an equivalent maintainer-approved change lands. The fix should add an explicit unpaired-DM response policy, wire it through shared pairing helpers and affected channel call sites, update config schema/generated metadata/public docs, and add regression coverage for silent, code-only, and branded behavior.
