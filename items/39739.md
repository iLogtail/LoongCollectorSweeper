## Summary

Keep open. Current main still does not support scheme-level wildcard matching for `gateway.controlUi.allowedOrigins`; the checker only accepts the global `"*"` wildcard or an exact normalized `URL.origin`, and `chrome-extension://...` still normalizes to `origin: "null"`. The related exact-origin PR #38290 is open and narrower, while the direct wildcard PR #39867 is closed unmerged.

## Best Possible Solution

Keep #39739 open for explicit gateway product/security judgment. If accepted, implement a conservative matcher in `src/gateway/origin-check.ts` for `chrome-extension://*` only, preserve literal `Origin: null` rejection and existing HTTP(S)/global-wildcard behavior, add focused unit and WebSocket handshake tests, and update the Control UI docs plus config help. Coordinate with open PR #38290 for exact non-standard origin normalization; closed PR #39867 can inform test cases but does not supersede the issue.
