## Summary

Keep open. Current main still only accepts the global "*" wildcard or exact normalized `URL.origin` values for `gateway.controlUi.allowedOrigins`; `chrome-extension://*` is neither documented nor implemented, and Chrome extension origins still normalize to `"null"` in the current checker. The directly related PR #39867 is closed unmerged, while PR #38290 is open and intentionally narrower because it handles exact extension origins, not scheme-level wildcards.

## Best Possible Solution

Keep #39739 open for explicit gateway product/security judgment. If accepted, implement a narrowly scoped `chrome-extension://*` matcher in `src/gateway/origin-check.ts`, preferably restricted to parsed `chrome-extension:` origins rather than a generic prefix wildcard. Preserve literal `Origin: null` rejection, existing HTTP(S) exact matches, host-header fallback, and loopback behavior. Add focused unit and WebSocket handshake tests for exact extension origins, `chrome-extension://*`, non-extension negatives, malformed origins, and docs/config help updates. Coordinate with PR #38290 for exact non-standard origin normalization, but keep this issue as the wildcard tracking item unless a narrower maintainer-approved follow-up replaces it.
