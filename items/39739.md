## Summary

Keep open. Current main still does not implement scheme-level wildcard matching for `gateway.controlUi.allowedOrigins`: the matcher only accepts global `"*"` or an exact normalized `URL.origin`, and Node normalizes `chrome-extension://<id>` to origin `"null"`. The related open PR #38290 explicitly excludes wildcard/prefix expansion, while PR #39867 targeted this issue but is closed and unmerged.

## Best Possible Solution

Keep the issue open for a focused gateway fix or explicit maintainer product/security decision. A strong implementation would add narrowly scoped support for `chrome-extension://*` or a documented pattern contract in `src/gateway/origin-check.ts`, cover exact and wildcard extension origins plus negative cases in `src/gateway/origin-check.test.ts`, and update the public gateway configuration docs if the behavior is accepted.

## Risks / Open Questions

- Scheme-level origin wildcards are security-sensitive; the implementation should avoid a generic broad prefix matcher unless maintainers explicitly want that policy, and should include negative tests for non-extension origins.
- Exact extension-origin support in PR #38290 would reduce reliance on `"*"`, but it does not solve this issue's ID-churn use case unless wildcard support is also added or the product direction changes.
