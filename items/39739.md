## Summary

Keep open. Current main still does not implement scheme-level wildcard matching for `gateway.controlUi.allowedOrigins`; it only accepts the global `"*"` wildcard or an exact parsed URL origin, and Chrome extension origins still normalize to `"null"` through `URL.origin`. Related PR #38290 is narrower exact-origin work and PR #39867, which targeted `chrome-extension://*`, is closed unmerged.

## Best Possible Solution

Keep the issue open for maintainer product/security judgment. If accepted, implement a conservative matcher in `src/gateway/origin-check.ts` for `chrome-extension://*` only, preserve exact `http(s)` matching and literal `Origin: null` rejection, add focused unit and WebSocket handshake coverage, and update the Control UI/Gateway configuration docs.

## Risks / Open Questions

- A generic prefix-wildcard implementation could broaden browser-origin access more than intended; a scheme-specific matcher for `chrome-extension://*` would be safer if maintainers accept the feature.
- This is part of Gateway browser-origin hardening, so the product/security tradeoff should be intentional and covered by focused tests and docs.
