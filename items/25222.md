## Summary

Keep open. Current main still serializes active sessions by queuing/collecting follow-up messages and offers adjacent manual controls like `/status` and `/stop`, but a normal message sent during an active run still does not receive the requested immediate scripted busy/waiting reply with cancel/new-session options.

## Best Possible Solution

Implement a generic core reply-pipeline path for active-session follow-ups that can return a channel-safe scripted busy/status payload before or alongside enqueueing. It should preserve existing queue modes, keep `/stop` and `/new` as safe next actions, expose a small typed busy reason if maintainers want it, and add regression coverage in `src/auto-reply/reply/agent-runner*` plus at least one channel test such as Telegram/Webchat to prove the immediate payload is delivered.
