## Summary

Keep open. Current main still does not teach `resolveGroupSessionKey()` to extract the real provider from agent-prefixed group/thread keys like `agent:builder:slack:channel:...`; it still falls back to the context provider/surface when the first segment is not a known message surface. Some announce/restart fallback behavior has been improved, but the core Slack A2A session-normalization bug described here remains unimplemented on main.

## Best Possible Solution

Keep the issue open and land a focused fix that adds a shared agent-prefixed session-key parser for `resolveGroupSessionKey()` or reuses the existing session-conversation parser, with regression tests for `agent:<id>:slack:channel:<channelId>:thread:<threadTs>` under webchat/provider-fallback contexts. The existing restart/announce field-level fallback behavior can remain, but the Slack session metadata normalization still needs source and test coverage.
