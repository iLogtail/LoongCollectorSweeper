## Summary

Keep open. Current main still resolves Mattermost inbound attachments only from the WebSocket post's `file_ids`; when that array is empty there is no REST re-fetch path for `GET /api/v4/posts/{post_id}`. The linked PR #59591 proposed that exact workaround but, per the provided GitHub context, was closed unmerged.

## Evidence

- **Inbound handler still uses event file_ids directly:** The Mattermost monitor calls `resolveMattermostMedia(post.file_ids)` directly after mention gating; there is no delay or REST re-fetch before media resolution.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:1451](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/extensions/mattermost/src/mattermost/monitor.ts#L1451)

- **Empty file_ids produces no media:** `resolveMattermostMedia` normalizes the passed IDs and returns `[]` immediately when the list is empty, matching the reported failure mode when Mattermost sends `file_ids: []`.
