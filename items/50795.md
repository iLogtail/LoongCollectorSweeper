## Summary

Keep #50795 open. Current main still has the live compaction handler behavior reported in the issue: `clearStaleAssistantUsageOnSessionMessages()` runs after non-retry compaction and unconditionally replaces every assistant message usage snapshot with zeros. The replay-history sanitizer has the timestamp/index-gated behavior the issue asks for, but that logic has not been applied to the live subscription compaction handler. The related fix PR #50845 and UI PR #51231 were closed unmerged, and the later merged #68711 discussion explicitly treated #50795 as a distinct post-compaction zeroing bug needing a separate timestamp-gated fix.

## Best Possible Solution

Keep the issue open and apply the replay-history timestamp/index-gated stale-usage logic to `clearStaleAssistantUsageOnSessionMessages()`. Add a focused regression test for the live subscription compaction handler that zeroes pre-compaction assistant usage while preserving assistant usage created after the latest compaction summary, plus a status/session-store assertion if the maintainer wants to lock the TUI footer behavior directly.
