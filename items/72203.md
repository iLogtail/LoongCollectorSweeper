## Summary

Keep open. This is a fresh, concrete report against v2026.4.24 QuickStart on Ubuntu, and current main has not shipped a relevant post-release fix for the token-auth setup/dashboard path. Existing code documents and implements the intended generated-token, health-probe, and dashboard-fragment flow, but that does not disprove the reported first-run token mismatch.

## Best Possible Solution

Keep the issue open and reproduce the v2026.4.24/current-main QuickStart install-daemon path on Ubuntu. Compare the generated gateway.auth.token, any OPENCLAW_GATEWAY_TOKEN in the daemon service environment, the token passed to waitForGatewayReachable/healthCommand, and the token that openclaw dashboard places in the URL fragment/sessionStorage. If this is service-env or installer token drift, fix daemon install/repair and add a regression test around QuickStart token auth plus post-install health/dashboard connection.
