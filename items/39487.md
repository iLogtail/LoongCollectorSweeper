## Summary

Keep open. Current main still does not implement Feishu lifecycle/liveness status reporting into the gateway runtime state, so the PR is not obsolete or implemented. It is also not clearly superseded: #41940 is a parallel open attempt, while this PR still represents useful work but needs the existing review comments addressed before merge.

## Best Possible Solution

Keep the PR open or ask the author to update it. The best implementation should wire a status sink from `extensions/feishu/src/channel.ts` through `monitor.ts`, `monitor.account.ts`, and `monitor.transport.ts`; publish `connected` only after WebSocket start or webhook listen succeeds; publish disconnected/error status on abort and failures including early-abort paths; update app-event liveness on inbound Feishu events; report `lastTransportActivityAt` for actual transport liveness; and add focused lifecycle/inbound regression tests. If #41940 becomes the maintainer-approved canonical PR, close this one as a duplicate at that point.

## Risks / Open Questions

- The current PR head should not be merged as-is until it removes premature connected-state reporting and handles early-abort/error disconnect status correctly.

- If maintainers later choose #41940 as the canonical implementation, this PR can then be closed as duplicate, but the current evidence does not establish that yet.
