## Summary

Keep PR #39487 open. Current main still does not implement the Feishu liveness/status plumbing proposed by the PR: the Feishu gateway start path only publishes the webhook port, and no status callback is threaded through monitor startup, inbound event handlers, or transport lifecycle paths. Current main has a generic health-policy mitigation that avoids stale-socket restarts when transport liveness is absent, but that does not make this Feishu-specific PR implemented or clearly superseded. The related PR #41940 is still open context, not a merged or maintainer-selected canonical replacement, and #39487 also has unresolved review comments about premature connected reporting and early-abort status gaps.

## Best Possible Solution

Keep #39487 open for Feishu plugin follow-up. The best path is to thread the gateway setStatus callback through extensions/feishu/src/channel.ts, monitor.ts, monitor.account.ts, and monitor.transport.ts; report connected only after WebSocket start or webhook listen succeeds; report disconnect/error state on abort, startup failure, runtime failure, and early-abort paths; update inbound event liveness for meaningful Feishu events; publish transport activity when Feishu transport activity proves socket liveness; and add focused lifecycle plus inbound regression tests. If #41940 is later merged or explicitly selected by maintainers as canonical, #39487 can then close as duplicate/superseded.
