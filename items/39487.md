## Summary

Keep PR #39487 open. Current main has added conservative generic health-policy behavior that avoids stale-socket restarts when a channel has not reported transport liveness, but Feishu still does not thread `ctx.setStatus` into its monitor stack or publish connection lifecycle, inbound-event, disconnect/error, or transport-activity status. The overlapping PR #41940 remains separate context rather than an explicitly chosen or merged superseder, and PR #39487 itself needs updates before merge because review comments identified incorrect connected-state timing and early-abort gaps.

## Best Possible Solution

Keep #39487 open for maintainer/contributor follow-up rather than closing it as implemented or superseded. The best path is a Feishu plugin update that threads the gateway `setStatus` callback through `channel.ts`, `monitor.ts`, `monitor.account.ts`, and `monitor.transport.ts`; reports `connected` only after WebSocket start or webhook listen succeeds; reports disconnected/error state on abort, startup failure, runtime failure, and early-abort paths; updates inbound event liveness for all meaningful Feishu events; publishes `lastTransportActivityAt` when Feishu transport activity proves socket liveness; and adds focused lifecycle plus inbound regression tests. If #41940 is later merged or explicitly selected as canonical, #39487 can then close as duplicate/superseded.
