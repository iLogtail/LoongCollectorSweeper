## Summary

Current main does not have strong evidence for closure. Feishu still catches `sendMediaFeishu` failures and returns a successful text fallback, which leaves the misleading-success behavior from #51347 plausibly present. The related PR #51374 claimed to fix this but is closed unmerged in the provided context, and the current source does not contain that fail-fast behavior.

## Evidence

- **Feishu media failures are swallowed into fallback text:** `feishuOutbound.sendMedia` catches any `sendMediaFeishu` error, logs `[feishu] sendMediaFeishu failed`, and returns `sendOutboundText({ text: `📎 ${mediaUrl}` })` instead of rethrowing. That preserves a successful delivery result even when the media upload failed.
  - file: [extensions/feishu/src/outbound.ts:267](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/extensions/feishu/src/outbound.ts#L267)
  - command: `nl -ba extensions/feishu/src/outbound.ts | sed -n '230,300p'`
  - sha: [8e12c24d1721](https://github.com/openclaw/openclaw/commit/8e12c24d17218dc50023aaf5543df1b7c36520aa)
- **Shared delivery treats returned fallback as success:** After a channel media send returns a delivery, `deliverOutboundPayloadsCore` pushes the result and emits `message_sent` with `success: true`; it only emits failure if the handler throws. Since Feishu catches the media error, this layer cannot distinguish failed media from a real media delivery.
  - file: [src/infra/outbound/deliver.ts:1169](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/infra/outbound/deliver.ts#L1169)

- **CLI summary is always a success string for returned results:** `formatOutboundDeliverySummary` formats any returned delivery result as `✅ Sent via ...`, so a Feishu fallback result after media failure still produces a success-looking CLI summary.
  - file: [src/infra/outbound/format.ts:45](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/infra/outbound/format.ts#L45)
