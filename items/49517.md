## Summary

Keep #49517 open. Current main still treats `/new` as a reset-style rollover of the same session key, with `/reset` as the documented alias, so the requested additive “spawn a new session while preserving the old one as accessible history” behavior is not implemented or superseded.

## Best Possible Solution

Decide the intended product contract. If `/new` should be additive, implement it as a separate session creation/switch path that preserves the current session entry as listable history, keep `/reset` on the existing reset path, and update docs/tests across `ui/src/ui/app-chat.ts`, `src/auto-reply/reply/session.ts`, gateway session methods, and `docs/tools/slash-commands.md`. If maintainers intend `/reset` to remain an alias, close manually as intended behavior rather than by automated cleanup.
