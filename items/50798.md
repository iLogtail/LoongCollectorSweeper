## Summary

Keep open. Current main partially addresses adjacent ACP/Discord thread behavior: bound-thread inbound messages, including allowed bot relay posts, can route to the bound ACP session, and `sessions_send` has an A2A announce path for visible follow-up delivery. The requested capability is still missing: `sessions_send` has no `visible` option and still injects via internal `deliver:false`; the `message`/gateway outbound send paths still derive and record an outbound Discord session entry before delivery; and Discord outbound routing does not consult bound ACP conversation state or offer a `proxyOnly` skip. That means the reporter's proxy workaround can still create the route-polluting main Discord thread session they described.

## Best Possible Solution

Keep the issue open and implement a core-supported visible coordinator-to-ACP-worker path. The safest direction is a small, explicit contract: either add `sessions_send({ visible: true })` that posts the coordinator instruction to the target session's bound delivery context without changing routing ownership, or add a `proxyOnly`/no-session-recording outbound send mode and an ACP-bound-thread exclusion guard. Add regression coverage proving Discord bound-thread relay posts still reach the ACP session and do not create or mirror into `agent:main:discord:channel:<thread_id>`.
