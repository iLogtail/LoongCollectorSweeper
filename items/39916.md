## Summary

Current `main` already suppresses the reported retry-noise case: mutating tool failures are recorded, then cleared when the same action later succeeds, and the user-facing warning path only runs if that failure state remains. The current release tag `v2026.4.22` already contains the same implementation and regression test coverage for the `edit` mismatch-then-retry scenario.

## Close Comment

Closing this as implemented on `main` after a Codex review. Current OpenClaw records tool failures in `lastToolError`, then clears that state when the same mutating action later succeeds (`src/agents/pi-embedded-subscribe.handlers.tools.ts:808`, `src/agents/tool-mutation.ts:155`, `src/agents/tool-mutation.ts:217`). The final `⚠️ … failed` chat warning is only emitted if that failure state still exists when reply payloads are built (`src/agents/pi-embedded-runner/run/payloads.ts:339`). There is also a regression test for the reported `edit` mismatch-then-retry path, including alias changes like `file_path` -> `file`, and it asserts the failure is cleared before user-facing payload generation (`src/agents/pi-embedded-subscribe.handlers.tools.test.ts:195`). The latest release tag `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) already contains the same implementation, so there is no remaining maintainer action on this issue.
