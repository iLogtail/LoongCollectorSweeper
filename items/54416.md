## Summary

Keep #54416 open. Current main has an exec approval framework and some model-facing guidance for the first-class gateway config tool, but the reported shell path is still possible under the documented/current default host exec policy: gateway exec defaults to full access with ask off unless the operator tightens policy. I did not find a current-main guard that specifically prevents an agent from running example `openclaw config set ...` or `openclaw gateway restart` commands via the shell after merely answering a model-switching question.

## Best Possible Solution

Keep the issue open and treat it as a product/security hardening request. The likely fix belongs in the exec/tool-policy layer or system prompt contract: require explicit user intent/approval before shell-running config-mutating OpenClaw commands such as `openclaw config set`, `openclaw approvals set`, and `openclaw gateway restart`, especially when those commands appear as examples in an answer. Add focused regression coverage for an agent turn that discusses switching models but must not execute the example config commands unless the user explicitly asks to apply them.
