## Summary

Keep open. The linked fix path in #52140 identified the right failure mode, but that PR was closed unmerged, and current main still does not keep the ClawHub download timeout alive while the archive body is consumed. A server that returns headers and then stalls can still leave skill installation waiting in the body read, which matches the reported silent/unresponsive behavior.

## Evidence

- **Current download timeout ends before body read:** `clawhubRequest` creates an AbortController and clears its timeout immediately after `fetch` resolves, returning the `Response` before any archive body is consumed.
  - file: [src/infra/clawhub.ts:371](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/infra/clawhub.ts#L371)
  - command: `nl -ba src/infra/clawhub.ts | sed -n '360,390p'`
  - sha: [8e12c24d1721](https://github.com/openclaw/openclaw/commit/8e12c24d17218dc50023aaf5543df1b7c36520aa)
- **Skill archive body is read after timeout cleanup:** `downloadClawHubSkillArchive` awaits `response.arrayBuffer()` after `clawhubRequest` returns. There is no separate body-read timeout or pipeline abort signal here, so the stalled-body case described in the issue discussion remains plausible on main.
  - file: [src/infra/clawhub.ts:625](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/infra/clawhub.ts#L625)
