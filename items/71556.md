## Summary

Keep open. Current `main` still appears to drop Z.AI thinking/reasoning params: the OpenAI-completions compat layer marks Z.AI as `thinkingFormat: "zai"` while disabling `supportsReasoningEffort`, the transport layer only emits reasoning payloads for OpenRouter or generic `reasoning_effort`, and the bundled Z.AI plugin exposes only the binary UI/profile without a Z.AI-specific request wrapper. The `open.bigmodel.cn` CN/Coding-Plan endpoints are configured by the plugin, but endpoint attribution still hardcodes only `api.z.ai` as `zai-native` and otherwise depends on manifest `providerEndpoints`, which the Z.AI manifest does not declare.

## Evidence

- **ZAI compat defaults disable `reasoning_effort` while marking the provider as `thinkingFormat: "zai"`:** `resolveOpenAICompletionsCompatDefaults()` sets `isZai` from `endpointClass === "zai-native"` or the default `zai` provider route, then forces `supportsReasoningEffort` to `false` for ZAI while setting `thinkingFormat` to `"zai"`. That leaves ZAI outside the normal `reasoning_effort` path.
  - file: [src/agents/openai-completions-compat.ts:72](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/agents/openai-completions-compat.ts#L72)

There is no `thinkingFormat === "zai"` handling.
