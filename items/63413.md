## Summary

Keep this PR open. Current main still accepts `ltm search --limit` as `Number.parseInt(opts.limit, 10)` without checking `NaN`, zero, or negative values before passing the result into the memory-lancedb database search path. The PR remains a focused fix candidate, though it should be reviewed against current main and ideally add a small regression test for invalid CLI limit input.

## Best Possible Solution

Keep this PR open and review it as a still-relevant bug fix. The maintainer path is to adapt the patch onto current main so `extensions/memory-lancedb/index.ts` validates the parsed CLI limit, warns on invalid input, falls back to the default limit, and ideally adds a focused test around `ltm search --limit abc` or the CLI handler path.
