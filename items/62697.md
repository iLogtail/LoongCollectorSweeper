## Summary

Keep open. Current main still lacks WhatsApp Newsletter/Channel inbound handling: the WhatsApp monitor only routes messages as direct or group, non-group JIDs must resolve to a direct E.164/LID identity, and there is no `newsletterPolicy` or newsletter allowlist/config surface. The related #13417 report supports that newsletter support remains incomplete, but it covers outbound/message-tool targeting rather than superseding this inbound receiving request.

## Best Possible Solution

Keep this issue open and implement newsletter support in the bundled WhatsApp plugin: add a newsletter JID recognizer, model newsletter/channel chat type and session routing separately from direct/group, add a `newsletterPolicy` or equivalent access-control surface with allowlist semantics, preserve Baileys-specific newsletter handling, and cover inbound routing/access behavior with focused tests and WhatsApp docs updates. Coordinate with #13417 so inbound and outbound newsletter semantics do not diverge.
