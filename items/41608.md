## Summary

Keep #41608 open. Current main has later group/channel peer-kind compatibility, but the exact Slack binding shape reported in the issue (`peer.id: "channel:C..."`) still does not appear to match inbound Slack room routes, which pass the raw Slack channel id (`C...`) into route resolution. The related PR #42162 was closed unmerged, and its target-style peer-id normalization is not present in the inspected main code path.

## Evidence

- **Slack inbound routing uses raw channel id:** For non-DM Slack messages, `resolveSlackRoutingContext` calls `resolveAgentRoute` with `peer.kind` set to `channel` for rooms and `peer.id` set to `message.channel`, i.e. the raw Slack channel id such as `C0...`, not the target-style `channel:C0...` form from the issue config.
  - file: [extensions/slack/src/monitor/message-handler/prepare-routing.ts:52](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/extensions/slack/src/monitor/message-handler/prepare-routing.ts#L52)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare-routing.ts | sed -n '1,260p'`
  - sha: [e640c0a95fcb](https://github.com/openclaw/openclaw/commit/e640c0a95fcbd06a1311f18fc635518d011d45da)
- **Binding peer id normalization only trims/coerces:** Route binding peer normalization calls `normalizeId(peer.id)` and returns the resulting id as-is for valid peers; `normalizeRouteBindingId` only trims/coerces strings/numbers. A configured `peer.id` of `channel:C0...` therefore remains `channel:C0...` instead of being canonicalized to `C0...`.
  - file: [src/routing/resolve-route.ts:483](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/src/routing/resolve-route.ts#L483)
