## Summary

Keep open. Current main still appears to miss route-resolution support for wrapped peer ids like `channel:C...` on inbound Slack bindings, so the reported fallback to `agent:main` remains plausible rather than clearly fixed or obsolete.

## Evidence

- **Slack inbound routing uses bare channel ids:** Inbound Slack messages call `resolveAgentRoute` with `peer.id = message.channel` and `peer.kind = "channel"`/`"group"`; no alias such as `channel:<id>` is passed into route resolution.
  - file: [extensions/slack/src/monitor/message-handler/prepare-routing.ts:52](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/extensions/slack/src/monitor/message-handler/prepare-routing.ts#L52)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare-routing.ts | sed -n '40,80p'`
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Route binding id normalization is trim-only:** `normalizeRouteBindingId()` only trims strings and stringifies numbers; it does not peel wrappers like `channel:` before matching.
  - file: [src/routing/binding-scope.ts:25](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/routing/binding-scope.ts#L25)
  - command: `nl -ba src/routing/binding-scope.ts | sed -n '1,60p'`
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Route matching still requires exact peer-id equality:** `resolveAgentRoute()` normalizes both sides with the trim-only helper, then `matchesBindingScope()` requires `scope.peer.id === match.peer.id`. A binding peer id of `channel:C0...` will not match an inbound Slack peer id of `C0...`.
  - file: [src/routing/resolve-route.ts:589](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/routing/resolve-route.ts#L589)
  - command: `nl -ba src/routing/resolve-route.ts | sed -n '576,626p'`
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Adjacent bound-account path has alias support that route resolution lacks:** `resolveFirstBoundAccountId()` accepts `exactPeerIdAliases`, and its test covers a binding stored as `peer.id = "channel:conversation-a"` matching a bare inbound `peerId = "conversation-a"`. That compatibility exists for account lookup, but there is no equivalent path in `resolveAgentRoute()`.
  - file: [src/routing/bound-account-read.ts:42](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/routing/bound-account-read.ts#L42)
  - command: `nl -ba src/routing/bound-account-read.ts | sed -n '35,130p'`
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Alias behavior is explicitly tested only for bound-account reads:** The existing test proves wrapper-alias matching for bound-account resolution, reinforcing that current route resolution is the missing piece rather than the issue being clearly obsolete.
  - file: [src/routing/bound-account-read.test.ts:328](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/routing/bound-account-read.test.ts#L328)
