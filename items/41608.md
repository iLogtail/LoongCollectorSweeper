## Summary

Keep #41608 open. Current main still appears to miss the reporter's exact Slack binding shape: Slack inbound routing passes the bare Slack channel/user id into `resolveAgentRoute`, while binding peer ids are only trimmed and then compared exactly. A binding peer id like `channel:C0AJUGWG5L6` therefore would not match an inbound Slack channel id of `C0AJUGWG5L6`, which matches the reported fallback to `agent:main`. The related PR #42162 was explicitly opened to fix this by normalizing Slack route binding peer ids, but the provided GitHub context says it was closed unmerged; current main tests cover raw `C123456` group/channel kind compatibility, not target-prefixed Slack ids.

## Best Possible Solution

Implement the missing normalization on the active routing path, preferably as a provider-aware route binding peer-id alias for Slack target forms (`channel:<id>` and `user:<id>`) without changing stored session key shape. Add regression tests that reproduce #41608 with `match.peer.id: "channel:C0AJUGWG5L6"` and assert the resolved route/session key uses the bound agent, plus a DM `user:U...` case if intended. Update schema/help or channel-routing docs to state whether bindings should use raw provider ids or accepted target aliases.
