## Summary

Keep #41346 open. Current main has mitigations around specific managed jobs, but the general cron safety request is not implemented: cron creation still defaults missing `enabled` to true, the agent-facing cron tool documents that default, isolated cron jobs still fall back to agent/default model selection when no per-job model is set, and cron config/runtime surfaces do not include a usage quota circuit breaker or model-cost guard.

## Best Possible Solution

Keep the issue open for a core cron safety design. The best path is to add a provenance-aware cron creation policy: manual CLI/UI jobs can preserve explicit current behavior, while plugin/Gateway/agent-created recurring agentTurn jobs should require explicit enablement or a reviewed policy; doctor/update should show a diff before managed jobs are enabled; cron should warn or block inherited high-cost defaults for polling jobs unless a per-job model is set; and runtime usage telemetry/provider quota data should feed an auto-pause or circuit-breaker for non-critical polling jobs.

## Risks / Open Questions

- Changing default enablement is a behavior change for existing reminder, CLI, UI, and plugin workflows; a source/provenance-aware migration or explicit compatibility path is likely needed.
- A quota circuit breaker probably needs provider-usage and local telemetry semantics defined carefully, because not all providers expose comparable session quota data.
