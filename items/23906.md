## Summary

Keep open. Current main still does not expose a per-spawn `sessions_spawn` tool profile override, and the issue has an explicit same-day maintainer/Codex comment saying to keep it open. The runner has adjacent `disableTools`/`toolsAllow` support, but the public tool schema, native spawn params, gateway RPC schema, and subagent docs only expose global/config-level policy, not the requested per-call `tools: "none" | string[]` contract.

## Best Possible Solution

Keep #23906 open until maintainers implement or explicitly reject a backward-compatible per-spawn tool selection contract. A likely fix is to add a `sessions_spawn` parameter such as `tools: "none" | string[]` or explicit `disableTools`/`toolsAllow`, validate and document precedence with global subagent policy, carry it through `SpawnSubagentParams` and the gateway `agent` schema/server method, map tool-less spawns to runner `disableTools: true`, map subsets to runner `toolsAllow`, and add tests for core tools, plugin/bundle MCP tool names, sandbox/security interactions, and the existing #15032 overlap.
