## Summary

Keep open. Current main still does not expose a per-spawn `sessions_spawn` tool profile override, and the issue has a same-day maintainer/Codex comment explicitly keeping it open. The existing global subagent tool policy and runner-level `toolsAllow`/`disableTools` machinery are adjacent, but the requested per-call schema, native spawn params, and gateway RPC plumbing are still missing.

## Best Possible Solution

Keep #23906 open until maintainers implement or explicitly reject a backward-compatible per-spawn tool selection contract. A likely implementation path is to add a `sessions_spawn` parameter such as `tools: "none" | string[]` or explicit `disableTools`/`toolsAllow`, validate it in `sessions-spawn-tool`, carry it through `SpawnSubagentParams`, extend the internal gateway `agent` schema/server method to accept the policy for native subagent starts, map tool-less spawns to runner `disableTools: true`, map subsets to `toolsAllow`, and add tests/docs covering core tools, plugin/bundle MCP tool names, global subagent policy precedence, and sandbox/security interactions.
