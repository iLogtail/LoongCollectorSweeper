## Summary

Keep #23451 open. Current main has related exec-specific approvals, ACP prompt approvals, block-reply flushing before tool-start events, and plugin-driven `before_tool_call.requireApproval`, but it still does not provide the requested built-in configurable `tools.confirmation` risk-tier policy or a same-response assistant-text-aware gate before the first tool executes.

## Best Possible Solution

Keep #23451 open for maintainer/product work in the agent tool-execution boundary and config contract. The likely path is to add a documented `tools.confirmation` schema/help/docs surface, classify tools by risk, make same-response assistant text available before the first tool executes, flush/stream that text, wait for explicit approval or timeout, and then deterministically execute or skip the held tool calls. If maintainers only want static per-tool approval, document a ClawHub/plugin recipe using `before_tool_call.requireApproval`, but that would not satisfy the issue's text-aware `auto` mode.

## Risks / Open Questions

- A static always-confirm policy for selected tools can already be built as a plugin with `before_tool_call.requireApproval`; core work should focus on the missing built-in config contract and same-response text-aware ordering, not duplicate the whole hook system.
- The requested `auto` confirmation mode depends on heuristic question/confirmation detection, so maintainers should define opt-in behavior, timeout semantics, transcript effects, and allow/deny defaults explicitly.
