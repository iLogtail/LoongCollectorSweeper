## Summary

Keep open. Current main contains multiple Telegram polling/webhook reliability fixes after v2026.3.2, but this read-only review does not strongly prove the report is obsolete, and the Control UI schema symptom in the issue still appears plausible on current main.

## Evidence

- **Post-2026.3.2 Telegram reliability fixes shipped:** The changelog records later fixes aimed at this exact class of failure: skip stale-socket restarts for Telegram polling health monitoring, validate/confirm persisted polling offsets, and reset webhook cleanup after getUpdates 409 conflicts.
  - file: [CHANGELOG.md:3159](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/CHANGELOG.md#L3159)
  - command: `nl -ba CHANGELOG.md | sed -n '3140,3205p'`
- **Later watchdog fixes shipped:** The changelog also records follow-up polling hardening: recent non-polling API activity now counts as liveness, the default stall threshold was raised from 90s to 120s, and the persisted-offset confirmation probe was bounded by a client timeout.
  - file: [CHANGELOG.md:1462](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/CHANGELOG.md#L1462)
  - command: `nl -ba CHANGELOG.md | sed -n '1438,1463p'; nl -ba CHANGELOG.md | sed -n '360,381p'`
- **Current main implements liveness-aware polling restarts:** The polling session now tracks getUpdates plus other API activity, runs a watchdog, forces restart on detected stalls, and explicitly treats 409 getUpdates conflicts as a restart/recovery path.
  - file: [extensions/telegram/src/polling-session.ts:246](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/extensions/telegram/src/polling-session.ts#L246)
