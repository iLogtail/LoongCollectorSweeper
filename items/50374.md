## Summary

Keep open. Current main still does not strip outgoing OpenAI-compatible chat message objects down to provider-accepted keys for direct `models.providers` entries using `api: "openai-completions"`. The code has adjacent compatibility support for string-only content, tool-call-id sanitation, replay ordering, and provider plugin replay policies, but not the strict message-key sanitizer requested for Infomaniak-style providers. Ryce's KeepMy Claw gateway comment is a useful workaround, not an implementation in OpenClaw core/config behavior.

## Best Possible Solution

Keep the issue open and implement a narrow OpenAI-compatible outgoing message sanitizer, likely as an opt-in `compat` flag or documented provider payload/replay hook that also works for direct `models.providers` entries. Add tests that build an `openai-completions` payload from a session with prior tool calls and assert the strict provider mode removes unsupported message keys without regressing normal OpenAI-compatible tool calling.

## Risks / Open Questions

- A provider plugin or external gateway can work around this today, but direct custom `models.providers` configuration still lacks the requested built-in sanitizer.
- A blanket automatic sanitizer could break providers that legitimately accept OpenAI tool-call keys, so the best fix likely needs an explicit compat flag or scoped provider-owned hook plus regression tests.
