## Summary

Keep open. Current main has refactored this area into the centralized OpenAI Responses payload policy, but it still does not implement the PR's central behavior: `compat.supportsStore: true` does not force `store: true` for proxied `openai-responses` routes. The PR discussion is useful because the author addressed both requested tests and the compaction-coupling review concern, but the work is not on main.

## Best Possible Solution

Keep the PR open or request an updated patch against the current policy layer. The best fix is to make `compat.supportsStore === true` an explicit Responses `store: true` opt-in for proxied OpenAI-family routes, while keeping service tier, reasoning compatibility, prompt-cache stripping defaults, and server compaction safely scoped. Add regression tests for proxy opt-in, proxy opt-out, unknown provider, no compat, and compaction staying off unless explicitly configured.

## Risks / Open Questions

- The current main implementation has moved from `shouldForceResponsesStore()` to `resolveOpenAIResponsesPayloadPolicy()`, so the PR likely needs rebasing or translation into the centralized policy layer.
- A naive `supportsStore: true` change could reintroduce the review bug by enabling server compaction on proxies; the final fix should keep proxy store opt-in separate from default `context_management` injection.
