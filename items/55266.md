## Summary

Keep open. Current `main` does not implement the PR’s requested behavior; the centralized request-capability/payload-policy path still treats proxied OpenAI Responses routes as ineligible for forced `store=true`, and current tests assert that behavior.

## Evidence

- **Central capability gate still blocks proxied store opt-in:** `allowsResponsesStore` is only true when the route is a known native OpenAI endpoint. A proxied `baseUrl` therefore stays false even if `compat.supportsStore` is true.
  - file: [src/agents/provider-attribution.ts:685](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/agents/provider-attribution.ts#L685)
  - command: `nl -ba src/agents/provider-attribution.ts | sed -n '676,692p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Payload policy still derives `store=true` only from `allowsResponsesStore`:** In provider-policy mode, `explicitStore` becomes `true` only when `capabilities.allowsResponsesStore` is true, so proxied OpenAI Responses routes are still not forced to `store=true`.
  - file: [src/agents/openai-responses-payload-policy.ts:113](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/agents/openai-responses-payload-policy.ts#L113)
