## Summary

Keep open. Current `main` still contains the sticky Telegram IPv4 fallback path the issue describes, with no code-level guard for `/etc/hosts` loopback `ECONNREFUSED` or IPv6-preferred `verbatim` cases. The transport still latches to a forced IPv4 dispatcher and the polling monitor intentionally preserves that sticky state across clean/conflict restarts.

## Evidence

- **Fallback classifier still matches timeout/socket envelopes:** `extensions/telegram/src/fetch.ts:96` includes `ETIMEDOUT` and `UND_ERR_SOCKET` in `FALLBACK_RETRY_ERROR_CODES`. `extensions/telegram/src/fetch.ts:436` enables fallback whenever the error message contains `fetch failed` and any collected nested code matches that allowlist. There is no special-case for loopback `ECONNREFUSED`, `/etc/hosts` overrides, or `dnsResultOrder=verbatim`.
