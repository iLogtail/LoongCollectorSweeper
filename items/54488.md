## Summary

Keep open. Current main still has an unbounded followup drain and each followup turn still runs through the per-session lane. Recent changes mitigate heartbeat and some context-maintenance latency, but they do not cap consecutive followup turns, yield to newly arrived user messages, or add inbound priority over queued followups.

## Best Possible Solution

Keep the issue open and implement a bounded/fair followup-drain policy in the auto-reply queue: cap consecutive followup turns per session, reschedule/yield between batches, let newly arrived user messages or interrupts preempt queued system/followup work where safe, and add regression coverage for same-session Discord/WhatsApp-style mixed followups plus cross-channel collect routing.

## Risks / Open Questions

- Some latency contributors have changed since the report, so a fix should include a focused regression test that separates followup-drain starvation from heartbeat and context-maintenance effects.
- Changing priority or preemption in the session lane can affect cross-channel ordering, dedupe, /stop, and collect-mode routing, so the implementation needs narrow queue tests plus one reply-pipeline integration test.
