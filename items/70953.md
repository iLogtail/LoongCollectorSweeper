## Summary

Keep #70953 open. Current main still treats foreground exec approvals as an asynchronous pending state: the exec tool registers the approval, returns `status: "approval-pending"`, and runs the approved command later from a detached follow-up path. The docs and tests also describe/lock in immediate return behavior, so the requested pause-until-approved semantics are not implemented. The related no-prompt and pending-list reports do not supersede this issue; they cover different surfaces.

## Best Possible Solution

Keep the issue open and implement an explicit blocking approval mode for foreground interactive exec calls. The likely fix belongs in the exec tool/agent-runner boundary: when an approval is required for a foreground call, await `exec.approval.waitDecision`, then run or deny the command before returning the tool result; reserve the current detached follow-up path for explicit background, headless, or no-interactive-route cases. Update `docs/tools/exec.md`, `docs/tools/exec-approvals.md`, and the approval-id/gateway/node tests to distinguish blocking foreground approvals from async background approvals.

## Risks / Open Questions

- Changing this behavior could break existing async approval workflows that rely on later `Exec finished` / `Exec denied` system events, so the fix should preserve an explicit background/headless mode or migration path.
- Heartbeat/cron behavior needs separate product handling: current code only resolves some no-route cron cases inline and otherwise still treats routed approvals asynchronously.
