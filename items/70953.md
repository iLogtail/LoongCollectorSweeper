## Summary

Current `main` still implements exec approvals as an immediate `approval-pending` return plus asynchronous post-approval followup/resume, so the reported non-pausing behavior remains present and is not safe to auto-close.

## Evidence

- **Exec tool surface explicitly models a non-blocking pending state:** `ExecToolDetails` includes an `approval-pending` result shape rather than a suspended/awaiting turn state.
  - file: [src/agents/bash-tools.exec-types.ts:45](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/agents/bash-tools.exec-types.ts#L45)
  - command: `nl -ba src/agents/bash-tools.exec-types.ts | sed -n '1,180p'`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Gateway exec path returns pending immediately and continues approval handling in a detached async task:** `processGatewayAllowlist` starts a `void (async () => { ... })()` to wait for approval, run the command, and send followup later, then immediately returns `pendingResult: buildExecApprovalPendingToolResult(...)`. That is the behavior the issue reports.
  - file: [src/agents/bash-tools.exec-host-gateway.ts:291](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/agents/bash-tools.exec-host-gateway.ts#L291)
