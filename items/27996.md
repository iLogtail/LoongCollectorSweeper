## Summary

Keep #27996 open. Current main recomputes `nextRunAtMs` for the normal `openclaw cron edit` / `cron.update` path, but the concrete remaining reproduction from the issue comments still appears valid: editing `jobs.json` directly while `jobs-state.json` retains a still-future `nextRunAtMs` is not drift-detected on load/startup.

## Best Possible Solution

Keep the issue open and implement drift-aware rescheduling for external cron definition edits. A conservative fix would persist a schedule fingerprint derived from schedule kind/expression/timezone/every/at/stagger, or otherwise recompute and compare on load/reload, then update `jobs-state.json` only when the schedule identity changed. Add regression coverage for editing `jobs.json` while `jobs-state.json` contains a still-future stale `nextRunAtMs`, while preserving the existing API edit behavior and restart catch-up protections.
