## Summary

Keep #27996 open. Current main fixes the normal `openclaw cron edit` / `cron.update` path by recomputing `nextRunAtMs` when `patch.schedule` changes, but it still does not handle the concrete remaining reproduction from the comments: direct `jobs.json` schedule/timezone edits with an existing future `jobs-state.json` `nextRunAtMs`. Startup and timer reload paths preserve still-future values and there is no schedule fingerprint or drift comparison in persisted state.

## Best Possible Solution

Keep the issue open and implement drift-aware rescheduling for external cron definition edits. A conservative fix would persist a schedule fingerprint derived from schedule kind/expression/timezone/every/at/stagger, or recompute and compare the stored next run against the current schedule on load/reload, then update `jobs-state.json` when the schedule changed. Add regression coverage for a `jobs.json` schedule edit with a still-future `jobs-state.json` `nextRunAtMs`, while preserving the already-correct `cron.update` schedule recompute and non-schedule edit behavior.
