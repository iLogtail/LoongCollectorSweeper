## Summary

Keep #7679 open. Current main still defaults Telegram DM access to pairing, still documents pairing as the default, and still sends pairing challenges to unknown DM senders. The setup surface can prompt for numeric allowFrom IDs and then write dmPolicy="allowlist", but it does not token-verify and auto-capture the intended owner's Telegram ID during onboarding.

## Best Possible Solution

Keep #7679 open. The maintainer path should add a Telegram onboarding flow that generates a short verification token, asks the intended owner to send that token to the bot, captures that sender's numeric Telegram user ID, writes dmPolicy="allowlist" plus allowFrom for the correct account, and updates schema/runtime/docs/tests with a backcompat story for existing pairing-mode configurations.

## Risks / Open Questions

- A schema-only default change would make allowlist validation fail or lock out users unless onboarding captures and writes a valid allowFrom owner ID.
- Blindly accepting the first inbound Telegram message would preserve the race condition raised in the issue; the onboarding flow needs a displayed verification token or equivalent sender proof.
- Existing pairing-mode installs need an explicit compatibility path so current users are not unexpectedly locked out.
