## Summary

Keep PR #38170 open. Current main still lacks the Telegram `download-file` action and still does not embed parser-safe `file_id` attributes in Telegram media placeholders. The PR is stale against the current `extensions/telegram` plugin layout and needs rework, but it addresses a real remaining capability gap from #8843 and the prior `tg-media-resolve` workaround discussion.

## Best Possible Solution

Keep this PR or a refreshed follow-up open until the Telegram plugin implements the capability on current main: expose `download-file` from `extensions/telegram/src/channel-actions.ts`, handle it in `extensions/telegram/src/action-runtime.ts` through the existing proxy/apiRoot-aware Telegram transport, expose inbound media file identifiers in a parser-safe placeholder/body format, update audio-placeholder consumers, and add Telegram action/body/docs coverage.

## Risks / Open Questions

- The PR patch targets older paths such as `src/telegram/*`, `src/agents/tools/telegram-actions.ts`, and `src/channels/plugins/actions/telegram.ts`; current main routes Telegram through `extensions/telegram/src/*`, so it cannot merge as-is.
- If placeholders gain attributes, exact-match audio/media consumers need coordinated updates so inbound audio behavior does not regress.
- The `download-file` implementation should use the existing Telegram proxy/apiRoot transport and avoid the deterministic filename collision flagged in the PR review.
