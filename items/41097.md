## Summary

Keep open. Current `main` still lacks correct Feishu doctor coverage for `groupPolicy: "allowlist"` with missing `groupAllowFrom`. Runtime still silently drops those group messages, config validation does not catch it, and the new generic empty-allowlist doctor path is not wired with Feishu-specific capabilities, so it can miss or misstate the warning.

## Evidence

- **Runtime still drops Feishu groups when allowlist has no group IDs:** Feishu group handling reads `groupAllowFrom` directly, checks the chat ID against it, and returns early when the group is not allowed. There is no fallback to top-level `allowFrom` here, so `groupPolicy: "allowlist"` plus empty `groupAllowFrom` still blocks all group traffic.
  - file: [extensions/feishu/src/bot.ts:433](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/extensions/feishu/src/bot.ts#L433)
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Docs still describe allowlist as requiring `groupAllowFrom`:** The Feishu docs on `main` say `groupPolicy: "allowlist"` means only groups in `groupAllowFrom` are allowed.
  - file: [docs/channels/feishu.md:58](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/docs/channels/feishu.md#L58)
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Feishu config schema does not validate empty `groupAllowFrom` for group allowlist mode:** The Feishu schema currently adds a doctor/config issue for `dmPolicy: "open"` without `allowFrom`, but there is no analogous validation for `groupPolicy: "allowlist"` with empty `groupAllowFrom`.
  - file: [extensions/feishu/src/config-schema.ts:302](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/extensions/feishu/src/config-schema.ts#L302)
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Generic doctor warning path assumes fallback-to-allowFrom by default:** Shared doctor capabilities default `groupAllowFromFallbackToAllowFrom` to `true`, and the shared empty-allowlist warning suppresses the group warning when `allowFrom` is populated. That does not match Feishu runtime semantics.
  - file: [src/commands/doctor/channel-capabilities.ts:17](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/commands/doctor/channel-capabilities.ts#L17)
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Feishu package metadata does not override doctor capabilities:** `extensions/feishu/package.json` declares the channel metadata but has no `doctorCapabilities` override, so Feishu inherits the incorrect default fallback behavior from the shared doctor path.
  - file: [extensions/feishu/package.json:22](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/extensions/feishu/package.json#L22)
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **No Feishu-specific doctor implementation exists on current main:** A direct grep on `main` for `doctor-channels-feishu`, `noteFeishuChannelDiagnostic`, or `diagnoseFeishuChannel` returned no matches under `src`, `docs`, or `extensions`.
  - command: `git grep -n "doctor-channels-feishu\|noteFeishuChannelDiagnostic\|diagnoseFeishuChannel" e94c0bf51542fe518e49ad4e2ee29882891b5603 -- src docs extensions`
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)

## Risks / Open Questions

- Users can still end up with Feishu group messages silently blocked on current `main` when `channels.feishu.groupPolicy` is `"allowlist"` and `groupAllowFrom` is unset.
- The newer shared doctor warning path is only partial coverage for Feishu and can be wrong when `allowFrom` is populated but `groupAllowFrom` is empty.
- This PR as opened is very noisy (`changedFiles: 1484` in the provided GitHub context), so the feature likely needs a smaller re-cut even though the underlying cleanup is still valid.
