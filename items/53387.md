## Summary

Keep open. The related PRs #53452 and #53462 described the right bug but were closed unmerged in the provided context, and current main still snapshots state-dir `.env` values into the gateway install environment. Source and tests now explicitly treat `~/.openclaw/.env` as a durable service env source, merge it into the install plan, and render that environment into the macOS LaunchAgent plist, so the stale-value restart problem reported in #53387 is still plausible on current main.

## Best Possible Solution

Keep #53387 open and fix the gateway install boundary. The likely code path is `src/config/state-dir-dotenv.ts` and `src/commands/daemon-install-helpers.ts`: stop copying state-dir `.env` values into service metadata, keep only the intended internal service vars and any explicitly supported service-env contract, and add regression coverage proving a key like `TAVILY_API_KEY` from state-dir `.env` is absent from the install plan and LaunchAgent plist while runtime dotenv loading still works after restart. Also consider cleanup for already-installed stale plist entries via `openclaw gateway install --force` or doctor repair.

## Risks / Open Questions

- Changing this now may affect users who currently rely on install-time service metadata to carry durable env values, so the fix should clearly separate internal service env, explicit config/service env, and state-dir `.env` runtime loading.
- Existing installed plists may already contain stale managed keys; a complete fix likely needs reinstall or doctor repair behavior that removes previously snapshotted `.env` entries.
