## Summary

Keep open. The report is clear, not protected/maintainer-authored, and current main still does not implement Mattermost WebSocket ping/pong or any Mattermost transport-activity timestamp that the gateway stale-socket policy can act on. PR #57621 is useful related work, but it is still open and the bug remains observable in main.

## Evidence

- **Mattermost WebSocket interface lacks ping/pong hooks:** The current Mattermost WebSocket wrapper only models open/message/close/error, send, close, and terminate. There is no ping(), pong listener, ping interval, or pong timeout path in the monitored socket lifecycle.
  - file: [extensions/mattermost/src/mattermost/monitor-websocket.ts:33](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/extensions/mattermost/src/mattermost/monitor-websocket.ts#L33)
  - command: `nl -ba extensions/mattermost/src/mattermost/monitor-websocket.ts | sed -n '1,240p'`
  - sha: [e640c0a95fcb](https://github.com/openclaw/openclaw/commit/e640c0a95fcbd06a1311f18fc635518d011d45da)
- **Mattermost health check only detects bot update_at changes:** On open, Mattermost starts a REST-based getBotUpdateAt poll to catch disable/enable cycles and calls ws.terminate() only when update_at changes. That does not prove inbound WebSocket delivery or detect a half-open/deaf socket.
  - file: [extensions/mattermost/src/mattermost/monitor-websocket.ts:267](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/extensions/mattermost/src/mattermost/monitor-websocket.ts#L267)

  - sha: [e640c0a95fcb](https://github.com/openclaw/openclaw/commit/e640c0a95fcbd06a1311f18fc635518d011d45da)
- **Generic stale-socket policy requires transport activity:** The gateway stale-socket evaluator only checks staleness when connected is true and lastTransportActivityAt is non-null; stale app events alone are explicitly ignored.
  - file: [src/gateway/channel-health-policy.ts:115](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/src/gateway/channel-health-policy.ts#L115)

- **Mattermost does not publish transport activity:** Mattermost status updates set connected/lastConnectedAt on open and disconnected/error fields on close/error, but do not set lastTransportActivityAt, so the generic stale-socket policy has no Mattermost transport liveness signal to evaluate.
  - file: [extensions/mattermost/src/mattermost/monitor-websocket.ts:247](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/extensions/mattermost/src/mattermost/monitor-websocket.ts#L247)
