## Summary

Keep open. Current main still does not detect the reported Mattermost deaf WebSocket condition: the Mattermost WebSocket wrapper has no ping/pong surface, the connection path only tracks open/close/error plus the existing bot update_at REST watchdog, and the generic stale-socket monitor only fires for channels that publish lastTransportActivityAt. The related fix PR #57621 is still open/unmerged in the provided context, so this issue remains actionable rather than implemented on main.

## Best Possible Solution

Keep this issue open until Mattermost publishes provider-proven transport liveness or implements WebSocket ping/pong keepalive with a regression test. The likely fix belongs in extensions/mattermost/src/mattermost/monitor-websocket.ts, with status/health integration only if Mattermost can provide a real transport heartbeat signal rather than ordinary app-message activity.
