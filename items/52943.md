## Summary

Keep open. Current main still does not wire the bundled legacy iMessage channel through typing-aware reply dispatch, and there is no iMessage typing helper on main. The PR addresses a real missing capability, but the review discussion identifies concrete unresolved fixes needed before merge, so it should stay open for contributor or maintainer follow-up rather than be closed.

## Best Possible Solution

Keep the PR open. The best path is to update the PR against current main by adding an iMessage-owned typing helper, wiring it through the existing shared typing dispatcher/callback lifecycle, preserving `opts.cliPath` and `opts.dbPath`, serializing start/stop subprocess calls, and adding focused tests for target resolution, graceful missing-CLI behavior, and reply lifecycle cleanup. Tapback/reaction event work should be handled separately because this PR is scoped to typing indicators.
