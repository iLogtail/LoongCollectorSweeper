## Summary

Keep open. Current `main` still does not implement the requested cron-status normalization, and the PR's blanket `delivered === true => ok` approach remains semantically unsafe and only partially propagated through cron events/notifications.

## Evidence

- **Main still uses raw run status everywhere in `applyJobResult`:** At `84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5`, `src/cron/service/timer.ts` still writes `result.status` directly into `lastRunStatus`/`lastStatus`, derives `lastErrorReason` from raw error status, increments `consecutiveErrors` on raw error status, and applies one-shot / recurring retry-backoff logic from raw error status. This means the behavior requested by PR #50189 is not implemented on current `main`.
  - file: [src/cron/service/timer.ts:418](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/cron/service/timer.ts#L418)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '391,585p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Downstream cron events still key off raw status:** `emitJobFinished` in `src/cron/service/timer.ts` and the manual-run path in `src/cron/service/ops.ts` both emit `status: coreResult.status`. `src/gateway/server-cron.ts` then triggers failure-destination handling when `evt.status === "error"`. So even if the PR normalized stored job state, downstream event consumers and failure notifications would still observe the raw error status unless those paths were changed too.
  - file: [src/gateway/server-cron.ts:468](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/gateway/server-cron.ts#L468)

  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Current cron isolated-run semantics treat `delivered` separately from success:** `src/cron/isolated-agent/run.ts` resolves run status from `hasFatalErrorPayload`, not from `delivered`, and still records `delivered` as separate metadata. In `src/cron/isolated-agent/helpers.ts`, `pickDeliverablePayloads()` falls back to the last deliverable payload when there are no successful deliverables, so an error payload can still be delivered. That means `delivered === true` is not a reliable proxy for a successful run, which supports the review concern that the PR could mask genuine failures.
  - file: [src/cron/isolated-agent/run.ts:824](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/cron/isolated-agent/run.ts#L824)

  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **PR discussion already identified real unresolved gaps:** The supplied PR review history is materially useful here: review comments on 2026-03-19 flagged that the original patch missed recurring-job backoff normalization and that event/webhook paths still used raw status; another review comment argued that delivered error payloads are legitimate failures and should not be rewritten to `ok`. One of those code gaps was reportedly patched in the PR branch, but the core semantic concern remains consistent with current `main`'s isolated-run design.
