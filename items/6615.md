## Summary

Keep open. Current `main` still lacks core exec-approval denylist support: the runtime types, docs, CLI validation, and macOS UI all expose only `deny`, `allowlist`, and `full`. Two linked PR attempts (#43222 and #60684) were closed unmerged. A plugin workaround was linked in discussion, but there is no strong maintainer evidence that this core request was intentionally redirected out of scope.

## Evidence

- **Core runtime still defines only three exec security modes:** `ExecSecurity` is currently `"deny" | "allowlist" | "full"`, and `normalizeExecSecurity` accepts only those three values. No denylist mode or denylist field exists in the current exec-approvals file types here.
  - file: [src/infra/exec-approvals.ts:22](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/infra/exec-approvals.ts#L22)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **User-facing docs still document only deny / allowlist / full:** The exec-approvals docs list exactly three security modes and three ask-fallback modes. There is no documented `denylist` security mode, denylist array, `argsMatch`, or `on-match` prompt behavior.
  - file: [docs/tools/exec-approvals.md:206](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/docs/tools/exec-approvals.md#L206)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **macOS approvals UI still exposes only deny / allowlist / always allow:** The macOS companion app `ExecSecurity` enum has only `deny`, `allowlist`, and `full`, and the titles map to `Deny`, `Allowlist`, and `Always Allow`. No denylist editing surface is present in the current UI model.
  - file: [apps/macos/Sources/OpenClaw/ExecApprovals.swift:6](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/apps/macos/Sources/OpenClaw/ExecApprovals.swift#L6)
