## Summary

Keep open. Current main has a bundled browser plugin with managed/headless browser control, tests for headless behavior, proxy/custom launch args, remote CDP providers, and optional Docker/sandbox Chromium paths. That addresses several related pain points, including #41019-style headless support, but #53763 specifically asks for a built-in bundled Chromium with no dependency on the user's browser or third-party services. Current main still uses `playwright-core` plus a discovered/installed Chrome/Brave/Edge/Chromium executable, and the Docker/browser image paths are optional rather than the default OpenClaw browser dependency model.

## Best Possible Solution

Keep #53763 open, but narrow it to the remaining product decision: whether the browser plugin should ship, download, or officially preinstall a pinned Chromium binary by default across supported package/Docker/server installs. Any implementation should build on the existing `extensions/browser` managed profile, headless, CDP, proxy, and sandbox-browser surfaces rather than adding a separate web access stack. If maintainers do not want a default bundled binary, create a narrower canonical issue or docs decision that says the supported paths are installed system Chromium, optional Docker build arg, sandbox browser image, node proxy, or remote CDP.

## Risks / Open Questions

- Bundling Chromium by default would add platform, package size, update cadence, CVE response, sandboxing, and installer/Docker policy obligations that need maintainer product/security judgment.
- A bundled browser would not by itself guarantee access to protected sites; login, CAPTCHA, anti-bot, proxy, and session persistence behavior may still need separate provider/profile/API decisions.
