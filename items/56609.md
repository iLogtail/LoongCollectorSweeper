## Summary

Keep #56609 open. Current main still uses `.jsonl.reset.<timestamp>` and `.jsonl.deleted.<timestamp>` as the active archive convention, and some maintenance paths can hard-remove archived or unreferenced transcript artifacts under retention or disk-budget cleanup. Related reports show `.reset`/`.deleted` indexing and usage were fixed, but they do not implement this issue’s requested naming/no-delete policy.

## Best Possible Solution

Keep this open for maintainer product judgment and a focused session-archive contract change. The likely implementation should choose either a durable `sessions/archive/` layout or new `.archived.*` / `.archived-reset.*` suffixes, preserve compatibility for legacy `.reset`/`.deleted` readers, update memory/usage/session hooks and tests, and revise maintenance/docs so transcript hard-deletion is either removed or made explicitly opt-in with clear backup semantics.

## Risks / Open Questions

- Renaming the archive convention is compatibility-sensitive because current readers, usage aggregation, memory indexing, hooks, and tests recognize `.jsonl.reset.*` and `.jsonl.deleted.*` as meaningful transcript artifacts.
- A strict never-delete policy conflicts with existing retention and disk-budget cleanup behavior, so maintainers need to decide whether cleanup becomes opt-in, moves files to a durable archive/trash location, or preserves legacy cleanup for explicitly configured constrained environments.
