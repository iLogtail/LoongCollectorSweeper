## Summary

Keep #69799 open. Current main has partial macOS app TCC hardening for app-owned actions, but it does not implement the requested dedicated/bundled Node boundary for the gateway/npm path. The current macOS docs explicitly say OpenClaw.app no longer bundles Node/Bun or the gateway runtime, the app installs the gateway LaunchAgent through `openclaw gateway install --runtime node`, and the gateway daemon code still resolves the active/system Node path rather than an OpenClaw-scoped runtime. Legacy iMessage/FDA docs still require Full Disk Access for the process context running OpenClaw/`imsg`, so the core security concern is still actionable.

## Best Possible Solution

Keep the issue open as core macOS security architecture work. The best path is to decide and implement an OpenClaw-scoped runtime boundary for TCC-sensitive macOS workflows, such as a signed app-owned gateway/runtime, a dedicated installed Node binary with stable identity, or a documented supported isolation mode, while coordinating with the open #7227 security thread and the legacy iMessage/FDA path.
