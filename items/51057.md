## Summary

Keep #51057 open. Current main has timeout-aware rotation/fallback paths, but the central request is still not implemented: plain timeout reasons are explicitly excluded from persisted auth-profile failure/cooldown state, and there is no separate timeout failover budget or timeout-specific rotation cap. The related open #71682 is broader timeout/model-selection UX, not a clean canonical duplicate of this auth-profile failover request.

## Best Possible Solution

Keep the issue open and implement a narrow timeout failover policy: either count repeated same-profile/provider timeouts toward a short transient cooldown or add a configurable timeout-specific same-provider retry budget before cross-provider model fallback. The change likely belongs in `src/agents/pi-embedded-runner/run.ts`, `run/failover-policy.ts`, and `run/auth-profile-failure-policy.ts`, with regression tests and docs clarifying that timeout cooldowns are transient and do not imply permanent auth failure.
