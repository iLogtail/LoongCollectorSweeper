## Summary

Keep open. Current `main` still has the bug this PR fixes: Mattermost threaded messages with `replyToMode: "off"` lose thread routing because `effectiveReplyToId` becomes `undefined`, and the code still reuses that value for `MessageThreadId`, typing, and reply delivery. The latest shipped release shows the same logic, the dependency work around `ChatType: "unknown"` is also not on `main`, and current tests still do not cover the proposed `messageThreadId` behavior.

## Evidence

- **Thread context still drops thread identity on `replyToMode: "off"`:** `resolveMattermostEffectiveReplyToId` returns `undefined` when a threaded message arrives with `replyToMode: "off"`, and `resolveMattermostThreadSessionContext` only returns `effectiveReplyToId` plus session keys. There is no separate `messageThreadId` on current `main`, so delivery code has no preserved thread root to use.
