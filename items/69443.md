## Summary

Keep #69443 open. Current `main` has strong support for routing an `agent` RPC to an explicit `sessionKey`, and the built-in subagent announce path uses that field correctly. However, the reported raw RPC shape sets `to` to a canonical WeChat session key, and current `main` still treats `to` as an outbound delivery target, not as the session selector. I did not find source or tests showing that `to=<agent:...>` is rejected, remapped to `sessionKey`, or otherwise fixed. The related #69576 delivery-mirror/accountId issue appears related but does not supersede this raw `agent` RPC routing complaint.

## Best Possible Solution

Keep the issue open and clarify the intended RPC contract. A conservative fix would add a regression test for `agent` calls where `to` looks like a canonical `agent:<id>:<channel>:direct:<peer>` session key, then either route it as `sessionKey` for backwards compatibility or reject it with an actionable error telling callers to use `sessionKey`. The built-in subagent announce path should continue to pass the requester session via `sessionKey` and reserve `to` for channel delivery targets.
