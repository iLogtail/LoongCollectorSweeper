## Summary

Keep #51805 open. Current main still only supports `dmScope` and `identityLinks` for direct-message session keys; group/channel peers remain isolated session keys. The issue discussion adds a real privacy requirement, merging only the linked user's group messages plus bot replies, and that filtering/session-routing behavior is not implemented. Related #19929 and #7524 overlap, but neither is an exact canonical replacement for this group-to-DM per-identity request.

## Best Possible Solution

Keep #51805 open as a distinct core session-routing/product-design gap. A good implementation would add an explicit opt-in config/API for per-identity group/DM session sharing, route it through core session-key resolution and channel adapters, preserve reply delivery to the originating chat, and add privacy filtering so only the linked user's group messages plus bot responses enter that shared session. It should coordinate with broader shared-session trackers such as #19929/#7524 without losing this issue's per-user filtering requirement.
