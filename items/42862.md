## Summary

Keep #42862 open. Current main does not auto-render plain image URLs in chat, and the closest existing behavior intentionally goes the other direction for remote images: explicit markdown image syntax is flattened unless it is a data image, and Control UI CSP blocks browser fetches of remote http(s) images. This needs maintainer product/security judgment rather than conservative cleanup.

## Best Possible Solution

Keep the issue open for maintainer product/security triage. The best implementation, if accepted, should live in the Control UI chat rendering/normalization path and preserve the existing remote-fetch hardening by using a deliberate preview policy rather than direct <img src="https://..."> insertion. It should include tests for bare URLs, markdown images, CSP behavior, and MEDIA/attachment interactions, plus Control UI docs updates.

## Risks / Open Questions

- A naive regex-to-<img> implementation would conflict with the current CSP and previously shipped privacy/security hardening against automatic remote image fetches.
- If maintainers want this UX, it likely needs an explicit policy decision plus a safe preview path, such as opt-in same-origin/proxied previews with SSRF checks, caching, size limits, and updated docs/tests.
