## Summary

Keep open. Current `main` still does not preserve destination file mode in the sandbox fs bridge write path, so this is not implemented on main, and the PR is not obsolete. The PR also mixes that sandbox fix with unrelated Feishu logic and very large generated churn, so there is not a strong maintainer-evidence basis to close it as already handled.

## Evidence

- **Sandbox write path still replaces files with a temp file created as 0600:** `create_temp_file()` opens the temp file with mode `0o600`, and `write_atomic()` writes, fsyncs, and `os.replace()`s it without first reading or restoring the destination mode.
  - file: [src/agents/sandbox/fs-bridge-mutation-helper.ts:70](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/agents/sandbox/fs-bridge-mutation-helper.ts#L70)
  - command: `nl -ba src/agents/sandbox/fs-bridge-mutation-helper.ts | sed -n '60,110p'`
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **OpenClaw already preserves existing mode in the non-sandbox pinned write path:** `resolvePinnedWriteTargetWithinRoot()` reads the existing file mode from `opened.stat.mode & 0o777`, and `runPinnedWriteHelper()` passes that mode into the helper so the replacement temp file is created with the preserved mode. That behavior exists in core fs-safe writes, but not in the sandbox bridge helper above.
  - file: [src/infra/fs-safe.ts:863](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/infra/fs-safe.ts#L863)

- **Pinned write helper uses the supplied mode for atomic replace:** The helper's `create_temp_file(parent_fd, basename, mode)` opens the temp file with that explicit mode before `os.replace()`, showing the repo already has a mode-preserving pattern elsewhere.
  - file: [src/infra/fs-pinned-write-helper.ts:60](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/infra/fs-pinned-write-helper.ts#L60)
