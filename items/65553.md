## Summary

Keep open: current main fixes the stale `memory-alt-*` collection sub-bug and adds QMD timeout/config plumbing, but the standalone CLI can still create full QMD managers whose boot/update/embed work is awaited during `search`/`index` lifecycle, matching the ARM/QMD hang-timeout class reported here.

## Best Possible Solution

Keep this issue open and fix the standalone CLI lifecycle: make `openclaw memory search/status/index` avoid starting unrelated QMD boot/embed maintenance when possible, make any required QMD child work bounded and cancellable on CLI close, and add regression tests that simulate a hanging `qmd embed` so the CLI exits cleanly on ARM/QMD setups. The already-shipped `memory-alt-*` repair should remain, but it should not be treated as closing the full report.

## Evidence

- **Issue evidence shows a remaining CLI lifecycle problem beyond the stale collection bug:** The issue reports `openclaw memory search`, `status --json`, and `index` hanging or timing out while direct QMD and direct memory-manager calls succeed. The follow-up comment adds ARM/Pi 5 evidence that `qmd embed` repeatedly times out around 120s due node-llama-cpp/Vulkan compilation, and asks for configurable embed/interval timeouts.
- **Status path is partly mitigated, but search/index use full managers:** `runMemoryStatus` passes `purpose: "status"`, while `runMemoryIndex` and `runMemorySearch` create managers without status mode. This means search/index still activate the full QMD lifecycle rather than the read-only status path.
