## Summary

Keep open. I did not find a main-branch fix for this report, and current ACP session rehydrate code still trusts persisted `cwd` state instead of reconciling it with the configured agent workspace. The current tests on `main` explicitly preserve persisted `cwd` across restart, while `/new`/reset still works by forcing a fresh ACP session, which matches the reporter’s workaround rather than disproving the bug.

## Evidence

- **Fresh gateway/main runs still derive workspace from agent config:** Normal gateway session runs resolve `runtimeWorkspaceDir` from `resolveAgentWorkspaceDir(...)` unless there is an explicit spawned-workspace override, and the embedded runner then resolves the workspace from that input before starting the run.

- **ACP rehydrate path still reuses persisted cwd verbatim:** When rehydrating an ACP session, `AcpSessionManager` still takes `cwd` from persisted runtime options or persisted metadata (`runtimeOptions.cwd ?? meta.cwd`) and uses that value for subsequent runtime reuse/re-ensure. There is no comparison here against `resolveAgentWorkspaceDir(...)` or the configured workspace.
