## Summary

Keep open. Current `main` still rejects `channels.telegram.accounts.<id>.agentId` in Telegram account configs, while the docs route multi-account ownership through top-level `bindings[]`. The workaround in the issue comments matches current docs, but there is no evidence on `main` of a compatibility or doctor migration for old per-account `agentId` configs, so the reported upgrade regression remains plausible and unresolved.

## Evidence

- **Telegram account schema is strict and has no account-level `agentId`:** `TelegramAccountSchemaBase` enumerates allowed per-account fields and ends with `.strict()`. `agentId` exists for topic configs, but not for account configs; `TelegramConfigSchema` uses that same strict account schema for `accounts`, which matches the reporter's `must NOT have additional properties` error for `channels.telegram.accounts.<id>.agentId`.
  - file: [src/config/zod-schema.providers-core.ts:115](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/config/zod-schema.providers-core.ts#L115)
  - command: `nl -ba src/config/zod-schema.providers-core.ts | sed -n '80,380p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Type definitions also omit account-level `agentId`:** `TelegramConfig` is `TelegramAccountConfig` plus `accounts?: Record<string, TelegramAccountConfig>`, and `TelegramAccountConfig` does not include `agentId`; only `TelegramTopicConfig` does. That aligns with the current schema behavior being intentional, not an already-fixed regression on `main`.
  - file: [src/config/types.telegram.ts:223](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/config/types.telegram.ts#L223)
  - command: `nl -ba src/config/types.telegram.ts | sed -n '200,340p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Current docs route Telegram multi-account ownership via top-level bindings:** The multi-agent docs show Telegram bots per agent using `bindings: [{ agentId, match: { channel: "telegram", accountId } }]` and keep `channels.telegram.accounts.<id>` for account config like `botToken` and policy only. This matches the workaround comment on the issue.
  - file: [docs/concepts/multi-agent.md:325](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/docs/concepts/multi-agent.md#L325)
  - command: `nl -ba docs/concepts/multi-agent.md | sed -n '320,380p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Docs and CLI guidance describe bindings-based routing, not account `agentId` migration:** The generic multi-account channel docs say to use `bindings[].match.accountId` to route each account to a different agent, and the CLI docs say the interactive wizard writes account-scoped routing bindings. The doctor docs list current config migrations, but there is no migration for `channels.telegram.accounts.<id>.agentId` to top-level `bindings[]`.

That discussion supports keeping the issue open as an unresolved compatibility/regression item rather than closing it as incoherent or obsolete.
