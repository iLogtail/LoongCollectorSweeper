## Summary

Keep #62486 open. Current main has strong evidence that the WebChat/Control UI interruption path was rebuilt: chat.send is non-blocking, Stop calls chat.abort, /stop is handled out-of-band, and queued/steered follow-ups are covered by UI tests. Current main also shares direct-message context with the agent main session by default, and WebChat can view that selected agent session. But the issue also asks for QQ-side and Web-side visible chat-history synchronization, specifically mentioning QQ/NapCat/OneBot. Current main routes WebChat replies back to WebChat and does not prove literal QQ client history mirroring or a bundled OneBot/NapCat path, so closing the whole multi-part regression would be too aggressive.

## Best Possible Solution

Keep the issue open but narrow it. Maintainers should ask the reporter to retest current main or the next release for the Stop/Queue/Steer behavior, then split the remaining QQ/Web history request into a precise item: identify official qqbot versus OneBot/NapCat, direct versus group session, configured session.dmScope, and whether the expected behavior is shared agent context in WebChat or actual mirrored messages inside the QQ client. If literal OneBot/NapCat mirroring is desired and no missing core API is found, that narrower work may belong in a plugin/ClawHub discussion rather than this broad regression issue.
