## Summary

Keep PR #39772 open. Current main still does not implement the PR's first-preview-only reply gating in Telegram draft/message streaming: `replyParams` is built once and reused for every new preview send after `forceNewMessage()`. The author and reviewers addressed useful early concerns in the PR, but later review concerns about first-send failure recovery and `replyToMode: "all"` semantics are still relevant rather than superseded by main.

## Best Possible Solution

Keep the PR open for maintainer or author revision. The best fix should suppress the transient Telegram `Deleted message` preview artifact while preserving configured reply-threading behavior: consume reply context only for the appropriate stream mode, restore or defer consumption on retryable first-send failure, preserve `message_thread_id` after rotation, and add regression coverage for `forceNewMessage`, thread params, first-send failure/materialize recovery, and `replyToMode: "all"`.
