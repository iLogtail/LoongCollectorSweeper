## Summary

Keep PR #39772 open. Current main still reuses one Telegram draft-stream reply parameter object for every preview send and materialize send, so the proposed first-preview reply gating is not implemented. The PR is a plausible focused fix for linked issue #39718, but review discussion and current contracts show it still needs maintainer/author work around failed first-send recovery and `replyToMode: "all"` behavior before it should be closed or landed.

## Best Possible Solution

Keep the PR open for maintainer review or author revision. The best fix should suppress the transient Telegram preview "Deleted message" artifact while preserving the configured reply-threading contract: keep `replyToMode: "all"` behavior where intended, consume reply context only after an actually delivered first successful message or otherwise recover it on retry/materialize paths, preserve forum/thread params after rotation, and add regression coverage for forceNewMessage, thread params, retryable first-send failure/materialize recovery, and all-mode streaming behavior.
