## Summary

Keep open. Current main still reuses the same Telegram reply parameters for every message-transport stream preview, so this PR is not implemented or obsolete. The PR also still needs maintainer-quality handling for failed first-send/materialize recovery and `replyToMode="all"` semantics before it is safe to close or land.

## Best Possible Solution

Keep the PR open for maintainer handling. The best path is to update the Telegram draft-stream API or dispatch plumbing so streaming previews can intentionally handle `replyToMode` (`first` vs `all`) while avoiding the transient `Deleted message` artifact, and to consume the reply target only after a successful first delivered message or explicitly recover it for materialize/fallback sends.
