## Summary

Keep open. Current main has the docs correction that QMD follows QMD scanner behavior, but I did not find an OpenClaw-side runtime fix for QMD symlink-loop traversal. The QMD manager still registers configured collection roots/patterns directly with QMD and runs `qmd update`; watcher ignores for `node_modules` only affect dirty-event scheduling, not the update traversal that the issue reports. Maintainer and reporter comments also explicitly say the runtime repro remains active.

## Best Possible Solution

Keep #54463 open. The best next maintainer path is a focused regression test for a workspace-visible temp monorepo symlink loop, then either harden OpenClaw’s QMD collection/update layer with pre-traversal pruning or first-class excludes, or land an upstream QMD traversal fix that prevents symlink cycles/ENAMETOOLONG before OpenClaw calls `qmd update`. Keep the current docs wording, but do not treat the docs correction as resolving the runtime bug.
