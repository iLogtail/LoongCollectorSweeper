## Summary

Keep open. Current `main` partially supersedes the PR by handling MiniMax-style JSON `api_error` payloads with transient-signal matching and regression coverage, but it still does not add explicit HTTP 520 transient/status classification. The PR’s original broad `api_error` change is also stale against current `main`, because `main` now intentionally avoids retrying non-transient `api_error` payloads such as schema/format errors.

## Risks / Open Questions

- The remaining explicit 520 gap means status-driven callers can still miss timeout classification on current `main`.
- The PR branch is stale relative to current `main`; merging it as written would discard newer non-transient `api_error` safeguards unless it is rebased and revised.
