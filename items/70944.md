## Summary

Keep PR #70944 open. Current main still only applies `session.identityLinks` to session routing; it does not resolve a Discord sender to a canonical trusted principal, thread that value through Discord preflight/process/native command/component contexts, or render `sender_principal` in the trusted inbound metadata prompt. The PR discussion also looks live and relevant: reviewers found two concrete issues, and the author responded with follow-up commits, including the PluralKit sender-ID fix at `1eca439126dd3e9d67b2cfa096f9e82a98c650a9`. There is no implemented-on-main, ClawHub, or duplicate/superseding basis to close it.

## Best Possible Solution

Keep the PR open and continue review. The right product path is to land a carefully reviewed implementation in the existing Discord plugin plus shared inbound-meta plumbing, with an intentional SDK helper if plugins need the identity-link lookup. If maintainers do not want the SDK export, keep the helper internal or add a narrower plugin-facing seam rather than closing the user-facing capability.
