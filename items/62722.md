## Summary

Keep open. Current main still has the exact fallback that PR #62722 proposes to fix: `stopChannel` aborts an existing controller before calling `stopAccount`, but when no controller exists it passes a fresh, non-aborted `AbortController` signal instead. The PR is a focused one-line fix for a real gateway lifecycle inconsistency; the best next step is to add a regression test and land the fallback change if CI stays green.

## Best Possible Solution

Keep #62722 open. Add a focused regression test in `src/gateway/server-channels.test.ts` for calling `stopChannel` on a configured account with `stopAccount` but no active task/controller, asserting `ctx.abortSignal.aborted === true`, then land the one-line `AbortSignal.abort()` fallback if the changed gateway lane is green.
