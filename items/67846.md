## Summary

Keep open. Current main still lacks the PR's requested memory-core upfront validation for `dreaming.frequency`/`dreaming.timezone`; the existing cron service can record schedule errors after cron evaluation, but memory-core still reconciles the managed dreaming job from unvalidated config and has no `invalid` result path. The PR discussion is active and focused, with prior review concerns addressed rather than evidence that the work is obsolete.

## Best Possible Solution

Keep #67846 open. The best path is a memory-core-owned validator exported through the existing memory host SDK facade, called before managed dreaming cron add/update/prune work, with tests for parse errors, inline `TZ=`/`CRON_TZ=`, invalid IANA timezone, and preserving last-known-good managed schedules while config is fixed.
