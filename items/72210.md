## Summary

Current main still has the reported run-record trace bug. Bare `delivery.mode="none"` cron runs correctly resolve delivery state as `not-requested`, but the isolated-agent trace path still constructs a non-ok `resolvedDelivery` with `Error("delivery is disabled")` and serializes it as `delivery.resolved.error`. The linked PR #72219 is the narrow fix, but it is still open/unmerged in the provided GitHub context, so #72210 should remain open.

## Best Possible Solution

Keep #72210 open until PR #72219 or an equivalent maintainer-approved change lands. The fix should be narrow in `buildCronDeliveryTrace()`: for explicit bare `delivery.mode="none"`, omit `delivery.resolved` or emit neutral skipped metadata instead of `ok:false/error`, while preserving `deliveryStatus: "not-requested"`, explicit no-deliver message-tool targets, and existing announce/webhook trace behavior. Add a regression test that inspects the emitted `delivery` trace or run-log entry for the no-deliver case.
