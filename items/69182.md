## Summary

Current main has fixed the more severe related #70481 timeout-enforcement failure, but it still intentionally runs Active Memory as a blocking `before_prompt_build` pre-step. A configured `timeoutMs` remains user-visible reply-path latency until the recall sub-agent completes or the hard timeout fires, so #69182's non-blocking/shorter-default/first-visible-output request is still a real product/performance decision rather than something already implemented.

## Best Possible Solution

Keep #69182 open as the active-memory latency/product follow-up. The best fix would be an explicit maintainer decision to either make Active Memory non-blocking relative to first visible model output, add concurrent/late injection semantics, lower or auto-tune default budgets for common modes, or document that bounded blocking latency is accepted behavior and close manually with that product rationale. The existing #70481-style hard-timeout enforcement should be kept as a separate shipped safety improvement.
