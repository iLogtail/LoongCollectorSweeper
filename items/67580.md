## Summary

Keep #67580 open. Current main has partial mitigations, including stripping managed Light/REM blocks during daily ingestion, but it still does not fully solve the reported MEMORY.md promotion leak: promotion appends the selected snippet verbatim, the contamination detector misses inline metadata before `Candidate:`, rehydration does not reject ranges inside dreaming fences, and `minRecallCount` is enforced against aggregate signal count rather than actual recalls.

## Best Possible Solution

Keep the issue open until memory-core lands a tested fix. The best path is to land or rework #68774's detector and dreaming-fence rehydration guard, then explicitly decide the recall-threshold contract: either enforce real `recallCount >= minRecallCount` for durable `MEMORY.md` promotion or rename/document the aggregate signal threshold. Add regression tests for inline metadata before `Candidate:`, rehydrated ranges inside managed dreaming fences, and recall=0 candidates with daily/grounded signal counts.
