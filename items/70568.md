## Summary

Keep PR #70568 open. Current main still appears to allow Telegram-origin exec approvals without a bound accountId to match every eligible Telegram account in a multi-account setup, so the core bug is not implemented on main. However, the PR should not be merged as-is without addressing the existing P1 review concern around `mode: "both"` and likely splitting the unrelated local-check runtime change.

## Best Possible Solution

Keep the PR open and ask for a narrow update: add a regression for Telegram-sourced unbound multi-account approvals, preserve explicit `turnSourceAccountId` routing, apply ambiguity handling only where it does not break `mode: "both"` session delivery, and split out the unrelated local-heavy-check runtime change. If #69916 remains the canonical bug, this PR or a successor should close it after those changes land.
