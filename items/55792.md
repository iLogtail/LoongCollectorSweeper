## Summary

Keep open. Current main has reconnect/readiness, duplicate-delivery guards, Telegram polling offsets, and some restart-continuation delivery recovery, but it does not implement the requested post-startup provider-history catch-up for missed inbound messages. The Discord path named in the report still processes live MessageCreate events only; Slack similarly registers live socket/http events, and the existing queues/history buffers are in-process rather than durable provider replay.

## Best Possible Solution

Keep this issue open and implement a bounded, configurable provider-history catch-up path. The likely design is a channel/plugin SDK contract for providers that can list recent inbound messages, plus per-provider implementations starting with Discord: persist or resolve a safe per-conversation anchor such as last outbound bot message, fetch a small time-bounded window after startup/READY, filter bot/self/duplicates through a durable dedupe key, and enqueue accepted messages through the existing inbound worker. Add focused Discord tests first, then Slack/other provider tests where their APIs permit history reads, and document which channel modes support catch-up.
