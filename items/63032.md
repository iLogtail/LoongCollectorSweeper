## Summary

Close as plugin-scope, not OpenClaw core: current main documents WeChat as an external plugin, current DM behavior is pairing-gated by default, and the reported missing `allowFrom` file after login does not match how core writes pairing state.

## Evidence

- **WeChat runtime lives outside core:** Current docs say the WeChat code does not live in the OpenClaw core repo and that WeChat login, iLink API calls, and monitoring are owned by the external `@tencent-weixin/openclaw-weixin` plugin.
  - file: [docs/channels/wechat.md:27](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/docs/channels/wechat.md#L27)
  - command: `nl -ba docs/channels/wechat.md | sed -n '25,43p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **WeChat DM access is pairing-based:** The WeChat docs explicitly say direct messages use the normal OpenClaw pairing/allowlist model and instruct users to approve senders with `openclaw pairing list openclaw-weixin` and `openclaw pairing approve openclaw-weixin <CODE>`.
  - file: [docs/channels/wechat.md:84](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/docs/channels/wechat.md#L84)
  - command: `nl -ba docs/channels/wechat.md | sed -n '84,96p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Unknown paired DMs are intentionally not processed:** The pairing docs state that when DM policy is `pairing`, unknown senders get a short code and their message is not processed until approval; the same page lists `openclaw-weixin` as a supported pairing channel and documents the `~/.openclaw/credentials/` allowFrom store files.
  - file: [docs/channels/pairing.md:20](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/docs/channels/pairing.md#L20)
  - command: `nl -ba docs/channels/pairing.md | sed -n '18,46p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Core defaults inbound DM access to pairing:** The generic plugin DM access path uses `dmPolicy ?? "pairing"`, reads the pairing allow store only for that policy, and returns a `pairing` decision for unknown senders instead of creating a normal session.
  - file: [src/plugin-sdk/direct-dm-access.ts:44](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/plugin-sdk/direct-dm-access.ts#L44)
  - command: `nl -ba src/plugin-sdk/direct-dm-access.ts | sed -n '31,140p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Pairing-store writes create the credentials directory on demand:** Core pairing writes go through `writeJsonFileAtomically`, and that writer is configured with `ensureDirMode`, so the parent `credentials` directory is created when pairing state is first written. A missing directory immediately after login does not by itself indicate a core bug.
  - file: [src/pairing/pairing-store.ts:98](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/pairing/pairing-store.ts#L98)
  - command: `nl -ba src/pairing/pairing-store.ts | sed -n '98,112p' && nl -ba src/plugin-sdk/json-store.ts | sed -n '33,39p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Project scope keeps optional channel behavior in plugins:** The project vision says core should stay lean, optional capability should usually ship as plugins, and plugins should be hosted and maintained in their own repositories, with plugin promotion living outside core.
  - file: [VISION.md:52](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/VISION.md#L52)
  - command: `nl -ba VISION.md | sed -n '52,73p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)

## Risks / Open Questions

- I did not run a live WeChat reproduction in this review, so this does not rule out a separate bug inside `@tencent-weixin/openclaw-weixin` after pairing is completed.
- If the external plugin is failing to issue or surface the pairing challenge itself, that would still be actionable upstream in the plugin package rather than in `openclaw/openclaw` core.
