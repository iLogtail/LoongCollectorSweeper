## Summary

Keep open. Current main still treats `write`/`edit`/`apply_patch` denial as tool-name filtering, not as a semantic no-write boundary for an allowed `exec` command. Docs now describe the safer read-only pattern as denying `exec` too, but there is still no command-level enforcement for the reported configuration, and one multi-agent doc example still labels `allow: ["read", "exec", "process"]` plus write-tool denial as “no file modifications.”

## Best Possible Solution

Keep this issue open for maintainer/security design. The best path is either to add an explicit exec no-write/readonly mode with regression coverage for `allow: ["read", "exec", "process"]` plus `deny: ["write", "edit", "apply_patch"]`, or to make a deliberate docs/product decision that read-only agents must deny `exec`/`process` and update the misleading “Safe Execution Agent” example accordingly.

## Evidence

## Risks / Open Questions

- Changing `deny: ["write"]` to implicitly deny or constrain `exec` would be a behavior change for configurations that intentionally allow read+exec but deny OpenClaw file tools.
- Shell command content inspection is likely incomplete across shells, interpreters, redirection forms, and Windows PowerShell; a robust fix probably needs an explicit exec mode, approval policy, sandbox/OS-level read-only boundary, or clear documentation that read-only profiles must deny `exec` and `process`.
