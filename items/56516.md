## Summary

Current main still matches the reported behavior. `sessions_spawn(mode:"run")` defaults to `cleanup: "keep"`, keep-mode run records are not scheduled for archive/delete, `sessions.list` still returns completed subagent session rows from the store, and the Control UI Sessions page renders that `sessions.list` result.

## Best Possible Solution

Keep this issue open as the focused tracker for completed subagent sessions remaining visible in the Control UI. The best fix is a maintainer-chosen product contract: either make default one-shot subagents transient after completion, add a gateway/UI filter or cleanup path for completed subagent child sessions, or document and expose the retention behavior clearly. Any fix should cover `sessions_spawn`, the subagent registry cleanup/archive path, `sessions.list`, Control UI Sessions behavior, and focused regression tests.
