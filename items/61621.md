## Summary

Current main does not provide strong evidence that the Feishu metadata-leak report is fixed. The Feishu path still builds an agent-facing body with a leading `[message_id: ...]` line and `senderOpenId`/sender-name prefix, and that body is passed as `BodyForAgent`. Core display-history sanitizers strip message_id hints from stored user messages, but the normal reply sanitizer only strips inbound metadata blocks/internal runtime context from outgoing text; it does not prove Feishu user-visible replies are protected from the reported metadata being echoed.

## Best Possible Solution

Keep the issue open. The best fix is to move Feishu message IDs and sender IDs out of `BodyForAgent` freeform text and into the existing structured inbound metadata/context fields, then add Feishu regression coverage showing quoted/replied messages and outgoing replies do not expose `[message_id: ...]`, `sender_id`, or `ou_xxx` metadata to users while preserving reply/thread/tool targeting.
