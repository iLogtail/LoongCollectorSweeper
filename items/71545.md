## Summary

Keep open. Current `main` still does not implement streamable-http MCP session-loss recovery: `callTool` uses the cached session client directly and has no lost-session detection, reconnect, metadata refresh, or retry path. The PR is a focused, recent fix for a real supported surface, and the maintainer force-pushed head appears to address the bot review concerns with a preserved custom schema validator, a shared reconnect guard, narrower error matching, and regression tests. It is not implemented on main, not a duplicate, and not out of repo scope.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to land this hardened streamable-http recovery behavior, or an equivalent main-branch implementation, with the current safeguards: preserve the custom JSON schema validator on rebuilt clients, share one reconnect across concurrent lost-session calls, refresh SDK tool metadata after reconnect, leave failed reconnect attempts retryable, and keep focused regression tests around the stateful streamable-http probe.
