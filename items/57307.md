## Summary

Keep open. Current `main` has optional recency-based temporal decay for memory search and weighted dreaming promotion, but it does not implement the requested write-time importance scoring, persisted importance metadata, access-based boosts, or importance-weighted retrieval. The issue also has an open cross-referenced PR (#71590), so this is still active rather than obsolete.

## Evidence

- **Hybrid retrieval only combines vector/text scores plus decay/MMR:** `mergeHybridResults()` computes ranking from `vectorWeight * vectorScore + textWeight * textScore`, then applies temporal decay and optional MMR. There is no importance field or importance-weighted rerank in the current search path.
  - file: [extensions/memory-core/src/memory/hybrid.ts:129](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/memory-core/src/memory/hybrid.ts#L129)
  - command: `nl -ba extensions/memory-core/src/memory/hybrid.ts | sed -n '1,220p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Current decay is simple half-life decay, not the requested write-time importance system:** `TemporalDecayConfig` only exposes `enabled` and `halfLifeDays`, and the score adjustment is a simple exponential multiplier based on age. There is no stored 1-10 importance score, no access-recency boost, and no Weibull shape parameter in the shipped implementation.
  - file: [extensions/memory-core/src/memory/temporal-decay.ts:4](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/memory-core/src/memory/temporal-decay.ts#L4)

- **Builtin memory schema has no persisted importance/access columns:** The SQLite schema stores file/chunk path, hash, text, embedding, and timestamps, but no importance, pin, or access-frequency metadata that could support the requested long-term ranking model.
  - file: [src/memory-host-sdk/host/memory-schema.ts:19](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/memory-host-sdk/host/memory-schema.ts#L19)

That is related work, but it is distinct from the issue's request to score memories at write time and use that score directly during retrieval.
