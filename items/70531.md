## Summary

Keep open. Current main has only the related runtime-state split: cron definitions still live in one `jobs.json` array, while runtime state lives beside it in `jobs-state.json`. The requested per-job definition files under `~/.openclaw/cron/jobs/` are not implemented or superseded by the related reports in the supplied context.

## Best Possible Solution

Keep the issue open and treat it as a core cron store-layout feature. The best path is a cron store abstraction that can read/write per-job definition files under `~/.openclaw/cron/jobs/`, migrate or fall back from legacy `jobs.json`, preserve the current runtime-state sidecar semantics, update doctor/docs/config help, and add focused store/service migration tests.

## Risks / Open Questions

- Changing this storage layout would touch migration/fallback behavior, doctor repairs, scheduler reload semantics, CLI/UI assumptions, and docs. The existing `jobs-state.json` sidecar should be preserved or deliberately migrated so runtime state churn does not return to job definition files.
