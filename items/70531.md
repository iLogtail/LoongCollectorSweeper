## Summary

Keep #70531 open. Current main still stores cron job definitions in one `jobs.json` array and does not discover or persist individual per-job definition files under `~/.openclaw/cron/jobs/`. Main has partially addressed runtime churn by splitting execution state into `jobs-state.json` and using per-job run logs, but the requested per-job config-file layout, migration, and CLI/Gateway save behavior remain missing.

## Best Possible Solution

Keep this open as a core cron storage enhancement. If maintainers accept the direction, implement a versioned cron store that loads legacy `jobs.json`, discovers deterministic per-job definition files under `~/.openclaw/cron/jobs/`, saves add/edit/delete operations atomically per job, preserves the existing `jobs-state.json` runtime-state split and per-job run logs, and updates doctor/config docs plus cron store/service/Gateway tests.

## Risks / Open Questions

- The current `jobs-state.json` split partially addresses runtime-state churn and should be preserved or deliberately migrated if per-job definition files are added.
- A storage migration would need deterministic file naming/load order, atomic add/edit/delete behavior, fallback from legacy `jobs.json`, doctor repair coverage, and docs/test updates.
