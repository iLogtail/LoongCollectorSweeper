## Summary

Keep #55640 open. Current main still only commits messaging-tool text/target state after a successful tool result, while reply suppression reads only the committed `messagingToolSent*` arrays. A failed send followed by an in-flight retry can therefore still reach `message_end` before the retry result commits, which matches the reported duplicate-delivery window. Existing tests cover success suppression and single-error non-suppression, but not the failed-then-retry same-turn case.

## Best Possible Solution

Keep the issue open and fix it in the embedded messaging-tool lifecycle. The likely path is to track same-turn pending/retried message sends separately from committed deliveries, then let both block-reply and final-payload suppression consider a pending same-target same-text retry while still falling back to normal replies when no successful or in-flight retry exists. Add a focused regression test for failed first `message` send, retry start, `message_end` before retry result, and retry success.

## Risks / Open Questions

- A naive pending-send suppress set could hide the only user-visible reply when the messaging tool ultimately fails and is not retried successfully.
- The fix needs to preserve cross-target behavior: sends to a different channel should not suppress the current conversation's final reply.
