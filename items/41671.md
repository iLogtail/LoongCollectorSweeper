## Summary

Current main still appears to have the reported one-way runtime fallback. Telegram network config is honored for the primary dispatcher, but direct Telegram fetches still build sticky fallback attempts that force IPv4 (`family: 4`, `autoSelectFamily: false`, `dnsResultOrder: ipv4first`) after broad `fetch failed` network errors. The inbound media path passes those dispatcher attempts into the SSRF-guarded media downloader, and current tests explicitly assert fallback attempts even when network config is supplied. The linked PR #42059 proposes the exact skip-explicit-config fix, but it is still open and not present on main.

## Best Possible Solution

Keep the issue open and fix it in the Telegram plugin transport layer. The narrow product-safe path is to make sticky IPv4 fallback opt out when `channels.telegram.network.autoSelectFamily` or `channels.telegram.network.dnsResultOrder` is explicitly configured, or add an explicit fallback policy switch if maintainers want operator control. The implementation likely belongs near `resolveTelegramTransport` in `extensions/telegram/src/fetch.ts`, with regression coverage proving `autoSelectFamily: true` plus `dnsResultOrder: "verbatim"` does not arm the forced IPv4 dispatcher for Bot API or inbound media fetches. PR #42059 can be mined for the intended behavior, but the fix should be landed as current-main code with focused tests.
