## Summary

Current main still has the Control UI chat final/aborted ordering that #71992 reports: `handleChatEvent` writes committed assistant messages to `chatMessages` while `chatStream` is still non-null, and the renderer still appends a live stream item whenever `props.stream !== null`. The issue is recent, has additional reproduction on v2026.4.24, and the candidate fixing PRs are open rather than shipped on main.

## Best Possible Solution

Keep #71992 open until #72004, #72290, or an equivalent patch lands on main. The narrow fix should snapshot fallback stream text, clear `chatStream`/`chatRunId`/`chatStreamStartedAt` before any `chatMessages` write in both `final` and `aborted`, and add a regression test that observes `chatStream` is already null at the moment `chatMessages` mutates. Track persisted transcript duplicate cases separately under the broader duplicate-transcript work.
