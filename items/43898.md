## Summary

Keep open. Current `main` at `81666e586aa6d4305e6472d7075e62f63f701831` still skips `memory-lancedb` auto-capture whenever a user message contains `<relevant-memories>`, and the current test suite does not cover the mixed injected-context-plus-user-text case described in the PR. The bug remains plausible on `main`, so this is not closable as implemented or obsolete.

## Evidence

- **Current capture guard still rejects any message containing recall context:** `shouldCapture()` still returns `false` for any text containing `<relevant-memories>`, which matches the reported failure mode when recall context is prepended to a real user message.
  - file: [extensions/memory-lancedb/index.ts:247](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/extensions/memory-lancedb/index.ts#L247)
  - command: `nl -ba extensions/memory-lancedb/index.ts | sed -n '220,255p'`
  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **Auto-capture path still uses raw extracted user text:** In the `agent_end` hook, extracted user text is passed straight into `shouldCapture()` with no stripping/sanitization step before filtering, so injected recall context would still suppress capture.
  - file: [extensions/memory-lancedb/index.ts:663](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/extensions/memory-lancedb/index.ts#L663)

- **Main only tests pure markup rejection, not the reported mixed-content regression:** The current test coverage asserts that a standalone `<relevant-memories>...</relevant-memories>` string is rejected, but there is no regression test for a prepended recall block followed by real user text.
  - file: [extensions/memory-lancedb/index.test.ts:1545](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/extensions/memory-lancedb/index.test.ts#L1545)

That argues for follow-up review, not cleanup closure.

## Risks / Open Questions

- Leaving this PR open means the current `main` behavior can still miss auto-capturing genuine user preferences/facts whenever recall context is injected.
- Closing this item would discard a live bug report without a replacement fix on `main`.
- Merging the PR unchanged also carries risk because its global stripping approach could remove user-authored tagged text from stored memories.
