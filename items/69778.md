## Summary

Keep open. The reported behavior still exists on current main: persisted task records are loaded without an age filter, task maintenance only marks active tasks lost when backing state is missing after the 5-minute grace period, and subagent orphan recovery resumes any active run whose session entry has `abortedLastRun: true` without checking task or run age. Existing tests even preserve a 3-hour-old aborted restored subagent run for restart recovery, which is the opposite of the requested stale-resurrection bound.

## Best Possible Solution

Implement an explicit stale recovery policy in core task/subagent recovery: add a documented config surface such as `agents.defaults.taskRecovery.maxAgeMinutes`, apply it before dispatching restart-aborted subagent or CLI recovery prompts, mark stale candidates `lost` with cleanup metadata, log a startup/maintenance summary like `Skipped N stale tasks older than Xm during recovery`, and update the existing tests that currently preserve a 3-hour-old aborted run.
