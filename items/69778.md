## Summary

Keep open. Current main still implements unbounded restart recovery for restart-aborted subagent work: task records are restored without an age filter, maintenance only marks active tasks lost when backing state is gone, and subagent orphan recovery re-sends the synthetic "gateway reload" resume prompt for any active run whose child session has `abortedLastRun`. Existing tests and docs explicitly preserve recovery for stale aborted runs, so #69778 is not implemented or superseded by the closed #66909 docs clarification.

## Best Possible Solution

Keep #69778 open and fix it in core. Add a bounded recovery policy for restart-aborted task/subagent recovery, likely under `agents.defaults.taskRecovery.maxAgeMinutes` or an equivalent clearly documented config surface; apply it before `scheduleSubagentOrphanRecovery()`/synthetic resume dispatch so stale aborted runs are finalized as `lost` instead of auto-executed. Add regression tests for old `abortedLastRun` subagent runs and restored task records, and update the Background tasks/Subagents docs to state the max-age behavior.
