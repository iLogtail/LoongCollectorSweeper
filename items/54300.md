## Summary

Keep open. Current main does not support a high-confidence cleanup close for #54300: the Docker subclaim appears partly obsolete because current source uses a hardcoded inspect template rather than a generic label-parameter helper, but the thread-ownership URL still interpolates raw path components, and the gateway plaintext-WS break-glass remains intentionally broad, documented, and covered by tests. This is a real security/product hardening request rather than stale or external cleanup.

## Best Possible Solution

Keep the issue open or split it into narrower maintainer-owned hardening tasks. A conservative code path would encode or strictly validate `channelId` and `threadTs` before calling the thread-ownership forwarder, confirm/close the obsolete Docker label-helper subclaim against current source, and make an explicit gateway security/product decision on whether `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS` should stay as documented break-glass or move to a narrower allowlist and less revealing error text.
