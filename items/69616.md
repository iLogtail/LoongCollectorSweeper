## Summary

Keep #69616 open. The report is recent, concrete, and environment-specific, with a strace loop on Ubuntu 24.04.4 / kernel 6.8.0-107 while running as root. The only discussion comment is a useful negative repro, but it used OpenClaw 2026.4.21, Ubuntu 24.04.3 / kernel 6.8.0-90, a non-root user, and a tarball swap rather than the reporter's service/root setup. Current main has several startup and plugin-loading improvements after 2026.4.15, but I did not find a source, test, changelog entry, or linked PR that directly addresses the reported capget/getuid busy loop or proves the current release fixes that exact startup path.

## Best Possible Solution

Keep the issue open and ask for a current-version reproduction on v2026.4.24 or newer, ideally on Ubuntu 24.04.4 / kernel 6.8.0-107 with the same install method and service/root mode. Request `OPENCLAW_GATEWAY_STARTUP_TRACE=1` gateway logs, `openclaw diagnostics export` output if available, and a short strace sample with the OpenClaw startup phase visible. If it still reproduces, investigate the pre-bind path first: CLI server import/config load, `prepareGatewayPluginBootstrap`, bundled plugin runtime-dependency staging/Jiti loading, and any root-only filesystem access loops.

## Evidence

- **Issue has concrete recent repro and root-specific syscall evidence:** #69616 reports OpenClaw 2026.4.15 on Ubuntu 24.04.4 / Linux 6.8.0-107, with `openclaw gateway run` and systemd service startup stuck in a tight `getpid`/`capget`/`getuid`/`getgid` loop. The trace returns uid/gid 0, so the observed path is root/service-sensitive.
- **Only discussion datapoint is a negative repro with material environment/version differences:** The lone comment reports no repro on OpenClaw 2026.4.21, Ubuntu 24.04.3 / kernel 6.8.0-90, Node 22.x, non-root, and a fresh tarball swap. That is useful evidence but not enough to close a root/systemd Ubuntu 24.04.4 report.
- **Current main has startup tracing and progress, but this is diagnostic plumbing, not a proven fix:** Gateway CLI startup now has `OPENCLAW_GATEWAY_STARTUP_TRACE` measurement around server import, config loading, and later startup phases, which would help localize a hang, but it does not by itself prove the capget/getuid loop is fixed.
