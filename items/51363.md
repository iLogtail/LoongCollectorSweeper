## Summary

Keep #51363 open. Current main still derives Docker and sandbox-browser container names from sandbox scope only, not from an instance-unique value such as HOME/state/workspace identity. The reported collision string is reproducible from the current slug algorithm for `agent:main`, and the related fix PR #51380 was closed unmerged, so there is no shipped fix to close against.

## Best Possible Solution

Implement the fix in core sandbox naming: include a stable instance discriminator, preferably the resolved OpenClaw state/HOME-derived workspace root or passed `workspaceDir` identity, in Docker and sandbox-browser container slug derivation while preserving deterministic names across restarts. Add regression tests proving two identical `agent:main` scopes with different instance/workspace roots produce different Docker and browser container names, and add a short migration note that old containers may be orphaned and pruned.

## Evidence

- **Docker container name still ignores instance identity:** `ensureSandboxContainer` resolves `scopeKey`, then sets `slug` to `shared` or `slugifySessionKey(scopeKey)` and builds `containerName` from `containerPrefix + slug`. `workspaceDir` is included in the config hash, but not in the container name used for Docker lookup/create, so two co-hosted instances with the same agent scope still target the same name.
