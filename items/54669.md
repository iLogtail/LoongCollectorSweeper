## Summary

Keep #54669 open. Current main has generic remote-CDP reachability checks, but it does not implement the requested Chrome-on-Windows IPv6-only CDP guidance or doctor diagnostics. The WSL2 browser guide still tells users to verify only `127.0.0.1:9222` locally and the WSL-reachable address from WSL2; browser status/doctor probes only the configured `cdpUrl` and surfaces generic CDP HTTP/WebSocket failures. The linked #68666 discussion is useful confirmation of the same v4tov4-to-v4tov6 problem, but #68666 is closed and does not track the remaining docs/diagnostics work.

## Best Possible Solution

Keep #54669 open. The best maintainer path is a narrow docs plus diagnostics update: add a Chrome 136+/147+ Windows section to `docs/tools/browser-wsl2-windows-remote-cdp-troubleshooting.md` that tells users to test both `curl http://127.0.0.1:9222/json/version` and `curl http://[::1]:9222/json/version`, inspect `netstat -ano | findstr :9222`, use `netsh interface portproxy add v4tov6 ... connectaddress=::1` when Chrome listens on IPv6 loopback, and mention the non-default `--user-data-dir` requirement if validated. Then extend browser doctor or remote-CDP failure hints so a WSL2/Windows remote profile that fails `/json/version` points users to the IPv4-vs-IPv6 binding and portproxy checks instead of only reporting generic CDP reachability failure.
