## Summary

Keep open. Current main still lacks the requested Chrome-on-Windows IPv6 CDP guidance and does not diagnose the specific 127.0.0.1 vs [::1] portproxy failure; the browser doctor/status path only probes the configured CDP URL. Related reports confirm the same area remains active, but no canonical linked item is strong enough to close this original field report automatically.

## Best Possible Solution

Keep the issue open until OpenClaw either implements the targeted docs/diagnostic work or maintainers deliberately consolidate it. The concrete path is to update the public WSL2 + Windows remote CDP guide with Chrome 136+/147 IPv6-loopback checks, netstat/curl examples for 127.0.0.1 and [::1], v4tov6 portproxy guidance, and current browser CLI flag order; then extend browser/openclaw doctor to detect or at least clearly hint when a configured IPv4 CDP endpoint fails while the Windows Chrome listener is likely on IPv6 loopback.
