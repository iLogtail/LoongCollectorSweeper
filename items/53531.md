## Summary

Current main has auth-profile rotation for normal embedded model runs, but the compaction path still binds a single auth profile for the compaction attempt and current regression coverage explicitly preserves non-rotation for failover-looking errors that originate from compaction wait/cleanup. The issue’s requested same-provider compaction failover remains real, and the existing discussion’s idempotency-boundary concern is important before implementing it.

## Best Possible Solution

Keep #53531 open. The right maintainer path is to add compaction-specific same-provider auth-profile failover around a stable transcript snapshot: assign a compaction attempt id, resolve eligible same-provider profiles using the existing auth-profile order/cooldown policy, mark failed profiles through the existing failure policy, retry summarization against the same snapshot, and commit at most one successful summary/checkpoint. Tests should cover rate-limit/cooldown failures during compaction without relying on normal prompt replay.

## Risks / Open Questions

- A naive fix could replay an already-completed model/tool turn or apply two compaction summaries; the balthazar-bot comment’s stable attempt/snapshot boundary is a real correctness requirement.
- Compaction can flow through the context-engine delegate and plugin-owned compaction surfaces, so a fix limited only to one direct Pi call path could leave other current compaction paths without failover.
