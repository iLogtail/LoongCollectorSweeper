## Summary

Keep #71190 open. Current main has related queue-path improvements, including collect-mode followups being enqueued while a run is active, but the specific orphan repair helper still merges the full extracted orphan text unchanged. If an aborted run leaves a previously merged queued block as the trailing user leaf, current code can still prepend another queued marker instead of stripping accumulated markers/abort notes. Existing tests cover basic duplicate detection and structured content preservation, but not the reported repeated-abort snowball case.

## Best Possible Solution

Keep the issue open and implement a targeted orphan-repair fix in `src/agents/pi-embedded-runner/run/attempt.prompt-helpers.ts`: preserve orphan delivery, but normalize/strip existing queued markers and abort notes or otherwise cap accumulated orphan text before merging. Add a focused regression test for repeated aborted turns so the merged prompt contains one queued marker and only the latest legitimate orphan content, plus a cross-channel preservation case to avoid the no-op regression.
