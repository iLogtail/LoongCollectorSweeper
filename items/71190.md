## Summary

Keep #71190 open. Current main still merges the full orphan user text back into the next prompt without stripping previously generated queued-message markers or the abort hint, so the reported accumulation path is not clearly fixed. Related reports cover adjacent queue/abort problems but do not supersede this specific collect-mode snowball behavior.

## Best Possible Solution

Keep the issue open and fix it in `src/agents/pi-embedded-runner/run/attempt.prompt-helpers.ts` with focused tests. The best path is to add a regression test for an orphan leaf that already contains OpenClaw's queued marker and abort note, then normalize only those generated prefixes before merging so the latest orphan content is preserved, structured/media orphan handling still works, and the session leaf can still be removed safely.
