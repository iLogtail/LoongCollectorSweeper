## Summary

Keep open. Current `main` still appears vulnerable to the reported abort-driven snowball: orphaned user leaves are still re-merged verbatim into the next prompt, and the only transcript cleanup path runs after a successful prompt submission, not after an abort/error.

## Evidence

- **Merge helper still re-wraps the full orphan text:** `mergeOrphanedTrailingUserPrompt()` still prepends the entire extracted orphan text unless the new prompt already contains that exact queued block or orphan text. There is no logic here to strip a prior queued marker or collapse an already-merged orphan before re-merging it.
  - file: [src/agents/pi-embedded-runner/run/attempt.prompt-helpers.ts:299](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/agents/pi-embedded-runner/run/attempt.prompt-helpers.ts#L299)
  - command: `nl -ba src/agents/pi-embedded-runner/run/attempt.prompt-helpers.ts | sed -n '299,332p'`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Transcript rewrite only happens on successful prompt completion:** `rewriteSubmittedPromptTranscript()` is called only after `await abortable(activeSession.prompt(...))` returns. The surrounding `catch` handles abort/error cases without invoking the rewrite, so an aborted run can leave the submitted merged prompt in the transcript/session state.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:2583](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/agents/pi-embedded-runner/run/attempt.ts#L2583)
  - command: `nl -ba src/agents/pi-embedded-runner/run/attempt.ts | sed -n '2580,2617p'`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Rewrite helper has no abort-path repair of its own:** `rewriteSubmittedPromptTranscript()` is a synchronous post-submit rewrite that only runs when explicitly called and only rewrites a user entry whose text exactly matches `submittedPrompt`. It does not independently repair aborted submissions.
  - file: [src/agents/pi-embedded-runner/run/transcript-prompt-rewrite.ts:56](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/agents/pi-embedded-runner/run/transcript-prompt-rewrite.ts#L56)
  - command: `nl -ba src/agents/pi-embedded-runner/run/transcript-prompt-rewrite.ts | sed -n '56,94p'`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Existing tests do not cover the reported abort regression:** Current tests cover duplicate detection for already-present orphan text and successful transcript rewriting, but I found no test combining orphan-merge logic with an aborted/yielded prompt path.
  - command: `rg -n "mergeOrphanedTrailingUserPrompt|rewriteSubmittedPromptTranscript" src/agents/pi-embedded-runner/run/*.test.ts src/auto-reply/reply/*.test.ts | rg "abort|aborted|yield"`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Queued followups still pass a separate transcript prompt, implying success-only cleanup is relied on:** Collect/followup queue runs still pass both `prompt` and `transcriptPrompt` into the embedded runner. That supports the interpretation that transcript repair depends on the later rewrite step, which is skipped on abort.
