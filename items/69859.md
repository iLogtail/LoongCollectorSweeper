## Summary

Keep open. The PR is stale and currently unsuitable as-is, but the underlying /hooks/agent threadId gap still exists on current main, and there is no open canonical replacement in the provided context. Current main has the lower-level cron delivery.threadId plumbing, so the best path is a narrower rework that parses threadId at the hook endpoint and passes it through the existing delivery contract.

## Best Possible Solution

Keep this PR open for maintainer/author follow-up, but request a focused rework: rebase on current main, remove the backup file and unrelated model-selection change, add threadId parsing to HookAgentPayload/normalizeAgentPayload, pass it through dispatchAgentHook as delivery.threadId, and add focused gateway hook tests plus docs. Current main already has cron delivery.threadId plumbing, so the implementation should reuse that contract rather than expanding CronAgentTurnPayload.
