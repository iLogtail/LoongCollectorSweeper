## Summary

Keep #71066 open. Current main has adjacent Telegram polling hardening, including sticky IPv4 fallback, watchdog restart, dirty transport rebuild, and the #71502 runner-liveness fix, but the issue discussion explicitly distinguishes this report from the generic watchdog bug and asks for current-main diagnostics on the affected India host. There is no source, test, docs, or history evidence that the exact sticky IPv4 fallback loop with reachable Telegram API has been fixed or superseded.

## Best Possible Solution

Keep #71066 open and ask for current-main `OPENCLAW_DIAG=telegram` logs around a full stall/rebuild cycle plus a probe using OpenClaw's `resolveTelegramFetch` on the affected host. If that probe succeeds while gateway polling still wedges, investigate polling-session lifecycle/runner ownership; if it fails, focus on the Telegram fetch/dispatcher fallback path and LaunchAgent environment.
