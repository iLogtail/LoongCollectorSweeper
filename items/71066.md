## Summary

Keep #71066 open. Current main has several Telegram polling/network hardening fixes, including sticky IPv4 fallback, transport rebuilds, liveness diagnostics, and the merged #71502 watchdog fix, but the issue is a recent region/host-specific sticky IPv4 fallback loop and two maintainer comments explicitly say the current-main local transport probe does not invalidate the report. There is not enough source-backed or live affected-host evidence to close it as implemented or non-reproducible.

## Best Possible Solution

Keep the issue open and use #71502 as the retest baseline. Ask for current-main c74fb781 logs with OPENCLAW_DIAG=telegram through one full stall/rebuild cycle, plus an affected-host probe using OpenClaw's resolveTelegramFetch for getMe/getUpdates. If that probe succeeds while gateway polling still wedges, investigate polling-session/monitor lifecycle ownership; if it still loops in fetch fallback, instrument dispatcher construction/reuse and nested undici error causes in extensions/telegram/src/fetch.ts.
