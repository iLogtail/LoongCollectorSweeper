## Summary

Keep open: current `main` still appears to route built-in `openclaw help` through plugin allowlist gating, while config validation still treats `help` as a stale plugin id. The `plugin` part of the report is less clear because the built-in subcommand is `plugins` (plural), but the `help` inconsistency itself is still present in code.

## Evidence

- **Commander help is registered as a built-in help command:** `buildHelpProgram` registers `program.helpCommand("help [command]", "Display help for command")`, so `help` is intended to be a framework/built-in CLI surface rather than a plugin id.
  - file: [src/cli/program/help.ts:72](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/cli/program/help.ts#L72)
  - command: `nl -ba src/cli/program/help.ts | sed -n '60,90p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **CLI gating only recognizes built-ins that appear in `program.commands`:** `runCli` computes `hasBuiltinPrimary` from `program.commands.some(command => command.name() === primary || command.aliases().includes(primary))` and uses that to decide whether to skip plugin command registration. There is no explicit exemption for Commander’s built-in help command here.
  - file: [src/cli/run-main.ts:255](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/cli/run-main.ts#L255)
  - command: `nl -ba src/cli/run-main.ts | sed -n '239,289p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Unknown primary commands still fall through to plugin allowlist errors:** If the primary command is still missing after lazy plugin registration, `runCli` calls `resolveMissingPluginCommandMessage(primary, config)`. That helper emits the `plugins.allow excludes "<id>"` message for names not present in the allowlist, which matches the reported `openclaw help` failure mode.
  - file: [src/cli/run-main.ts:123](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/cli/run-main.ts#L123)
  - command: `nl -ba src/cli/run-main.ts | sed -n '74,140p'; nl -ba src/cli/run-main.ts | sed -n '278,286p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Config validation still warns on non-plugin ids in `plugins.allow`:** Validation only accepts real plugin ids or manifest command aliases. Otherwise it pushes `plugin not found: <id> (stale config entry ignored; remove it from plugins config)`. There is no special-case for built-in framework commands like `help`.
  - file: [src/config/validation.ts:1098](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/config/validation.ts#L1098)
  - command: `nl -ba src/config/validation.ts | sed -n '1098,1117p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **The built-in plugin manager command is `plugins`, not `plugin`:** Current subcli descriptors and user docs expose the management surface as `plugins` plural. That means the issue’s `plugin` wording is partly muddied by command naming, but it does not invalidate the separate `help` bug.
  - file: [src/cli/program/subcli-descriptors.ts:132](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/cli/program/subcli-descriptors.ts#L132)
