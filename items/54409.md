## Summary

Keep open. Current main still matches the reported root cause: Feishu inbound messages are serialized per chat before entering the gateway, and the serialized task awaits the full reply dispatch. The gateway collect path still only batches when a later message reaches it while the session is active, so same-chat Feishu followups can still arrive after the active run has ended and be treated as separate `run-now` turns.

## Best Possible Solution

Keep the issue open and fix the Feishu/gateway boundary. A conservative fix should ensure later same-chat Feishu messages reach `dispatchReplyFromConfig` while the session is still active, either by bypassing/shortening the Feishu per-chat queue for already-active chats or by moving active-session followup enqueueing before channel-local serialization blocks it. Add a focused regression test around `createFeishuMessageReceiveHandler`/`handleFeishuMessage` proving multiple same-chat messages during an active run produce one collect followup turn.
