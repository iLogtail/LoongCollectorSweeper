## Summary

Keep open. Current main still serializes normal Feishu messages through a per-chat promise queue before they reach `handleFeishuMessage`, while the gateway only enters the followup/collect path when it sees an active run. The issue’s reported failure mode therefore still matches current code.

## Best Possible Solution

Keep #54409 open and fix the Feishu inbound dispatch path so followup-eligible same-chat messages can reach the gateway while a run is active, without regressing the intentional ordering/dedup behavior for the first message and the dedicated control/`/btw` lanes. Add a Feishu-focused regression test that simulates an active session with `messages.queue.mode = "collect"` and verifies multiple later same-chat messages are batched into one followup turn; coordinate with #70133 only if a shared queue refactor addresses both failure modes.
