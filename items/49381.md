## Summary

Keep #49381 open. Current main still contains the Feishu transient-error path that #59771 identifies: `onError` closes the active streaming card, while the recovered final path later closes/sends separately. Existing tests cover exact duplicate final-text suppression, but they do not cover partial content -> reply error -> recovered final, and current non-streaming tests intentionally allow distinct final payloads. The linked PR #59771 is an open candidate fix, not merged or shipped evidence.

## Best Possible Solution

Keep the issue open until a Feishu regression fix lands. The likely product/code path is to update `extensions/feishu/src/reply-dispatcher.ts` so transient reply errors do not finalize the active streaming card, add a regression test for partial content -> error -> recovered final, and separately verify whether the streaming=false failover case needs a core single-final coordination guard.
