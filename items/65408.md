## Summary

Keep open. Current main does not satisfy the central request: the Control UI and gateway still intentionally support a bounded cached device-token retry after `AUTH_TOKEN_MISMATCH`, and gateway tests explicitly assert that existing device-token websocket sessions remain connected after shared gateway token rotation. Some adjacent hardening exists, including sessionStorage-only gateway token persistence and device token revoke/rotate surfaces, but there is still no first-class Control UI logout that clears both the tab token and persisted device auth, and the render path still gates on `connected=false`, which can show the login gate during restore/connect.

## Best Possible Solution

Keep the issue open for maintainer/product-security judgment. If gateway token rotation is intended to invalidate Control UI browser sessions, update the gateway and Control UI so shared-token mismatch cannot silently recover with a cached Control UI device token after rotation, add a first-class logout that stops the client and clears the scoped sessionStorage token plus current device-role auth cache, add an explicit restoring/bootstrap render state, and update the protocol/docs/tests that currently document and assert the retry behavior.

## Risks / Open Questions

- Changing this behavior affects the documented paired-device recovery flow; maintainers need to decide whether gateway token rotation should also invalidate Control UI device-token sessions.
- A fix likely touches a public gateway protocol/client contract because retry hints and device-token auth are shared beyond this one UI path.
- Adding logout must be scoped carefully so it clears the current browser/device auth state without accidentally revoking unrelated paired devices unless explicitly requested.
