## Summary

Keep open. Current main has some repair coverage for legacy config and cron job shapes, but it does not fully solve this report: malformed cron stores still throw, the cron doctor path depends on the same throwing loader, and the related plain-array cron migration/data-loss report remains separate and open. The legacy config keys named in this issue are not first-class migrations; generic doctor repair can drop unknown keys, but that does not close the broader upgrade regression.

## Best Possible Solution

Keep this issue open and fix it as an upgrade-recovery bug: add a clear doctor repair/quarantine path for malformed cron stores, decide whether `jobs.json` backup restore should be automated or guided, and add explicit legacy handling or documented doctor repair coverage for `agents.list[*].allowedModels` and `tools.exec.allowlist`. Keep #60799 as the canonical follow-up for the separate valid plain-array cron-store clobber case unless maintainers choose to consolidate both cron migrations under one issue.

## Risks / Open Questions

- Changing startup-time config migration policy could conflict with the repository rule that legacy config repair should generally live in doctor/fix paths rather than core load-time migrations.
- Cron recovery must avoid silently discarding or overwriting operator data; malformed-store handling should prefer quarantine, explicit backup discovery, or operator-guided repair over treating the store as empty.
