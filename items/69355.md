## Summary

Keep #69355 open. Current main does not implement the proposed `openclaw doctor` local-rebuild/pristine-npm diagnostic, and the supplied related items are adjacent doctor/install reports rather than a canonical replacement. The PR is a focused, plausible doctor improvement, but the open review comments identify real merge-follow-up areas around metadata mismatch, stale cache behavior, malformed JSON reporting, and network timeout handling.

## Best Possible Solution

Keep the PR open for maintainer review or author revision. The best path is a warn-only doctor install freshness check wired after `noteSourceInstallIssues`, preferably reusing existing package-root/version and `fetchWithTimeout` utilities, with tests for pristine, rebuild, unreleased, corrupt metadata, inconclusive network failure, cache TTL/offline behavior, package/build-info version mismatch, and docs/changelog coverage.

## Risks / Open Questions

- A doctor-path registry request must stay bounded and silent on inconclusive data so normal `openclaw doctor` does not hang or produce noisy false positives.
- The implementation should treat package.json/build-info version mismatch and malformed build metadata as explicit corrupt metadata, otherwise it can suppress or mislabel the exact divergence signal being added.
- Offline/cache fallback semantics need a clear TTL policy to avoid stale registry data producing incorrect unreleased-version warnings.
