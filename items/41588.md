## Summary

Keep open. Current main has useful mitigations since the report: the default agent runtime timeout is now 48 hours, LLM stream idleness has its own `agents.defaults.llm.idleTimeoutSeconds`, and timeout-only turns now surface an explicit error instead of disappearing silently. But the requested activity-aware agent run timeout is still not implemented: the embedded run still uses an absolute wall-clock abort timer, with only a special compaction grace extension, and `agents.defaults.timeoutSeconds` still rejects 0 in the generated config schema.

## Best Possible Solution

Keep the issue open and implement the commenter’s Option 2 as a deliberate config/API change: add a run-level idle timeout that resets on assistant/tool/lifecycle activity, preserve a separate absolute max runtime cap, update schema/docs/tests, and decide explicitly whether global `agents.defaults.timeoutSeconds: 0` should be allowed or remain limited to per-run overrides.

## Risks / Open Questions

- The old 600-second default symptom is likely mitigated by the current 48-hour default, so maintainers may want to retitle or narrow the issue around configured shorter timeouts and true run-idle semantics.
- Changing `agents.defaults.timeoutSeconds` from an absolute cap into idle behavior would alter an existing safety contract; the best fix should add explicit run idle/max fields rather than silently changing the meaning of the current key.
