## Summary

Keep open. Current main partially covers the no-output/silent-token UX through NO_REPLY filtering, but the main requested capability remains missing: Control UI chat sends still have no structured reply-to/parent metadata field, the browser send path does not send one, and the gateway builds the agent message context from plain parsed text plus origin/thread routing, not a replied-to message id or parent summary.

## Evidence

- **chat.send schema lacks reply metadata:** The current chat.send parameter schema is closed with additionalProperties: false and lists sessionKey, message, thinking, delivery/origin fields, attachments, timeout/provenance, and idempotencyKey. There is no replyTo, replyToId, replyToCurrent, parentMessageId, or equivalent field for Control UI clients to pass a replied-to message.
  - file: [src/gateway/protocol/schema/logs-chat.ts:35](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/src/gateway/protocol/schema/logs-chat.ts#L35)
  - command: `nl -ba src/gateway/protocol/schema/logs-chat.ts | sed -n '35,52p'`
  - sha: [81a41fe5bee3](https://github.com/openclaw/openclaw/commit/81a41fe5bee350c6a035bfcc458eb8be380afe2c)
- **Control UI send path does not send reply metadata:** requestChatSend only accepts message, attachments, and runId, then sends sessionKey, message, deliver:false, idempotencyKey, and attachments to chat.send. No reply target or parent message metadata is collected or forwarded from the UI path.
  - file: [ui/src/ui/controllers/chat.ts:460](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/ui/src/ui/controllers/chat.ts#L460)

There is no replied-to message id or parent summary in the session context for the agent to consume.
