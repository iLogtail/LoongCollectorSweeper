## Summary

Keep open. Current main has not implemented PR #70682's central read-tool change: host reads are still wired through the upstream `createReadTool()` path, with workspace enforcement applied only by an outer guard when `tools.fs.workspaceOnly` is true, while that setting still defaults to false. The linked #70573 privacy/security report is still an open, security-labeled bug, so this is not safe maintainer cleanup to close. The PR likely needs narrowing/rebase/review because it bundles unrelated changes and has unresolved review feedback, but the core bug remains actionable.

## Best Possible Solution

Keep the PR open for maintainer/security review. The ideal path is to split or rework #70682 so the read-tool workspaceOnly enforcement and regression tests land as a focused agents/security change, while unrelated web UI, gateway, config, Telegram, media, and transport fixes move to separate PRs or are dropped from this one. The implementation should explicitly decide whether the default `tools.fs.workspaceOnly=false` privacy model is acceptable for #70573 or whether a safer default/agent isolation policy is needed.
