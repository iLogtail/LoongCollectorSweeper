## Summary

Keep open. Current `main` still has the reported `loadState` bug in both pairing loaders, `readJsonFile` still accepts array-shaped JSON without validation, and current tests do not show the recovery path the PR is trying to add.

## Evidence

- **Device pairing loader still passes through arrays:** `loadState` in `device-pairing.ts` still assigns `pendingById: pending ?? {}` and `pairedByDeviceId: paired ?? {}`. If `pending.json` or `paired.json` parses as `[]`, that array is truthy and survives unchanged.
  - file: [src/infra/device-pairing.ts:143](https://github.com/openclaw/openclaw/blob/7bd74758c5f54c546b52708359a25c08cae74ad7/src/infra/device-pairing.ts#L143)

  - sha: [7bd74758c5f5](https://github.com/openclaw/openclaw/commit/7bd74758c5f54c546b52708359a25c08cae74ad7)
- **Node pairing loader has the same unguarded path:** `node-pairing.ts` still uses the same `pending ?? {}` / `paired ?? {}` pattern, so the same array-typed state-file bug remains there too.
  - file: [src/infra/node-pairing.ts:134](https://github.com/openclaw/openclaw/blob/7bd74758c5f54c546b52708359a25c08cae74ad7/src/infra/node-pairing.ts#L134)
