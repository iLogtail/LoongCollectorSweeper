## Summary

Keep open. Current `main` still does not implement Slack-specific `[[reply_to_current]]` thread mapping in the message-action send path, and this PR is not merge-ready because the proposed patch appears to redeclare an imported helper, adds tests that call undefined helpers, and likely leaves a `replyToMode: "first"` edge case broken for lowercase Slack targets.

## Evidence

- **Current main still lacks explicit `reply_to_current` handling in the send path:** `handleSendAction` strips reply directives and applies `parsed.replyToId`, but there is no branch that handles `parsed.replyToCurrent` before normal reply/thread resolution. The flow goes from `parsed.replyToId` straight into `resolveAndApplyOutboundReplyToId(...)` and `prepareOutboundMirrorRoute(...)`.
  - file: [src/infra/outbound/message-action-runner.ts:532](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/infra/outbound/message-action-runner.ts#L532)
  - command: `nl -ba src/infra/outbound/message-action-runner.ts | sed -n '528,590p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Current Slack auto-threading will not satisfy the PR's intended explicit behavior by itself:** The shared outbound thread resolver only uses an explicit `threadId` or the channel plugin's `resolveAutoThreadId(...)`. Slack's resolver returns `undefined` unless `replyToMode` is `all` or a single-use mode, so current `main` still misses the explicit `[[reply_to_current]]`-in-`replyToMode: off` behavior described in the PR.
  - file: [src/infra/outbound/message-action-threading.ts:16](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/infra/outbound/message-action-threading.ts#L16)

- **PR patch appears to redeclare an imported binding:** On `main`, `message-action-runner.ts` already imports `resolveAndApplyOutboundThreadId` from `./message-action-threading.js`. The PR patch shown in the review context adds a same-scope `function resolveAndApplyOutboundThreadId(...)` near the top of the same file, which would redeclare the imported ESM binding and block the module from loading.
  - file: [src/infra/outbound/message-action-runner.ts:63](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/infra/outbound/message-action-runner.ts#L63)

A repo search under `src/infra/outbound` finds no definitions for those symbols, but the PR patch adds tests that call both names, so the added coverage would fail as written.

## Risks / Open Questions

- Merging the PR as posted would likely introduce a module-load failure from the duplicated `resolveAndApplyOutboundThreadId` binding.
- The added threading tests do not appear runnable in their current form, so the claimed coverage is not reliable.
- Even after fixing the compile/test issues, the Slack `replyToMode: first` accounting still needs a case-insensitive match on the explicit-thread send path.
