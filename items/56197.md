## Summary

Keep open. Current main has a generic embedded-run runtime-auth retry path, but it only runs when a provider has prepared refreshable runtime auth. The bundled Z.AI provider remains API-key/Bearer-auth only and does not register a runtime-auth refresh hook, so the reported Feishu + ZAI/AutoGLM expired-token path is not clearly fixed on main.

## Best Possible Solution

Keep the issue open. The best fix is a narrow embedded-run regression test for Feishu or channel-agnostic embedded dispatch where the first Z.AI/AutoGLM request returns `HTTP 401: Invalid token`, refreshed auth/config is re-resolved, and the same user turn retries successfully without sending the raw HTTP error. If AutoGLM needs expiring JWT exchange, add a provider-owned `prepareRuntimeAuth`/refresh hook or a documented plugin/provider seam rather than relying on delayed config reload after the user-visible failure.
