## Summary

Keep #51264 open. Current `main` still eagerly builds a combined gateway session store by discovering all agent session-store targets and loading every store, and `loadSessionStore` still clones whole stores through the cache/read path. The later session-maintenance work reduces stale/oversized-store pressure, but it does not implement the requested lazy per-agent loading, pagination, or clone reduction. The related PR #51362 claimed to close this issue, but the provided GitHub context says it was closed unmerged and current code does not contain its described `readOnly` option.

## Best Possible Solution

Keep this issue open and fix the distinct eager-loading problem. Add targeted session-store APIs for known agent/session/run/transcript lookups so those paths load only the relevant agent store, keep global `sessions.list` behavior through pagination/indexing or an explicit all-agent scan, and reduce avoidable whole-store `structuredClone` use with a safe read-only/no-clone contract. Add regression coverage with multiple agent stores proving single-agent lookup paths do not read unrelated stores while preserving disk-only agent visibility.
