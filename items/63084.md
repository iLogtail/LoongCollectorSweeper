## Summary

Keep open. Current `main` does not implement `gateway:shutdown` or `gateway:pre-restart` internal hook emission, and the PR discussion leaves a substantive shutdown-blocking risk unresolved: `triggerInternalHook()` awaits handlers sequentially, so the proposed synchronous shutdown dispatch can hang restart/exit if any hook stalls.

## Evidence

- **Current main closes the gateway without emitting shutdown internal hooks:** `createGatewayCloseHandler()` computes `reason` and `restartExpectedMs`, broadcasts the external `shutdown` event, and proceeds with socket/server cleanup, but there is no import or call to `createInternalHookEvent()` or `triggerInternalHook()` in the shutdown path on current `main`. That means the PR's requested behavior is not already implemented.
  - file: [src/gateway/server-close.ts:105](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/gateway/server-close.ts#L105)

  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **User-facing hook docs still expose only gateway startup:** The automation hooks table documents `gateway:startup` but does not list `gateway:shutdown` or `gateway:pre-restart`, which matches the current implementation gap and shows this PR is not obsolete via docs or shipped behavior.
  - file: [docs/automation/hooks.md:36](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/docs/automation/hooks.md#L36)

- **Existing gateway hook dispatch pattern is non-blocking:** Gateway startup hooks are dispatched from `setTimeout(...); void triggerInternalHook(hookEvent);` after startup work completes. Current `main` therefore treats gateway lifecycle hooks as best-effort background work rather than something that can block the lifecycle transition.
  - file: [src/gateway/server-startup-post-attach.ts:268](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/gateway/server-startup-post-attach.ts#L268)

- **Internal hook runtime awaits handlers sequentially:** `triggerInternalHook()` loops handlers and `await`s each one. If the PR's shutdown-path patch awaits `triggerInternalHook(...)` directly, one hung hook can block shutdown indefinitely until additional timeout or fire-and-forget protection is added.
  - file: [src/hooks/internal-hooks.ts:286](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/hooks/internal-hooks.ts#L286)

- **PR discussion contains an unresolved shutdown-timeout concern:** The provided review thread includes a Codex P1 review note that the PR's synchronous shutdown hook dispatch needs a bounded wait or fallback because otherwise restart/exit can hang. The follow-up comments in the thread address timer cleanup, CI, and unrelated webhook regressions, but do not resolve that shutdown-hook timeout concern.
