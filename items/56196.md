## Summary

Keep open. Current main still contains the exact no-model-alias skill-command failure described by #56196/#56188: `resolveReplyDirectives` gates skill discovery on `rawAliases.length > 0`, and `handleInlineActions` treats an empty upstream `skillCommands` array as loaded. The docs also confirm this is a supported surface for `/skill`, user-invocable skills, and direct tool-dispatch skills, matching the later reproduction comment. The PR may need a small revision for the duplicate-scan review comment, but the underlying fix is not obsolete.

## Best Possible Solution

Keep #56196 open, but revise before merging if needed: remove the `rawAliases.length > 0` gate so slash skill discovery runs whenever text commands are allowed and the command text contains a slash; avoid unnecessary duplicate skill scans in `handleInlineActions`; add a no-model-alias regression test for workspace skill invocation and direct tool dispatch; then close the linked #56188 when the fix lands.

## Risks / Open Questions

- The PR's `params.skillCommands.length > 0` fallback can rescan skills when upstream already loaded an empty list; the final fix should distinguish 'not provided' from 'loaded empty' or avoid changing the fallback once the upstream alias gate is removed.
- A stronger regression test should cover the full no-alias path through `resolveReplyDirectives` into `handleInlineActions`, ideally including both `/skill <name>` rewrite and direct tool-dispatch workspace skills.
