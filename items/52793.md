## Summary

Keep open. Current main still models `tools.fs.workspaceOnly` as one boolean guard for the filesystem tool family. The separate `tools.exec.applyPatch.workspaceOnly` setting only covers apply_patch and does not provide the requested per-tool/per-operation policy for read/search/write/edit, so the reported security-vs-usability tradeoff remains.

## Best Possible Solution

Keep the issue open for a core filesystem-policy change. A good implementation would extend `tools.fs` and `agents.list[].tools.fs` with a backwards-compatible per-operation policy, optionally including read/search allowed roots, then thread that policy through config types, strict Zod schema, generated help/docs, `resolveToolFsConfig`, read/write/edit/apply_patch wiring, media/image/PDF consumers where applicable, and regression tests proving external read/search can be allowed while write/edit/apply_patch remain workspace-confined.

## Risks / Open Questions

- The fix is security-sensitive: it should preserve backwards compatibility for existing boolean configs, define precedence with `tools.exec.applyPatch.workspaceOnly`, and use canonical path checks so read exceptions do not weaken write/edit/apply_patch containment.
