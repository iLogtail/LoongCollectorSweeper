## Summary

Keep #70515 open. Current main still has the reported bug in both async and sync suspicious-read recovery: backup copy failures are swallowed, the success warning is emitted unconditionally, and the audit record is written with `valid: true`. The audit schema also has no `restoreErrorCode` or `restoreErrorMessage` fields, and current tests cover successful restore paths but not failing `copyFile`/`copyFileSync`. The PR discussion is materially useful: bot review identified only minor follow-ups, and the author addressed them in `4da2addf170813f934fd0de8208882dc5eb2b223`.

## Best Possible Solution

Keep the PR open and either land it after maintainer validation/rebase, or apply an equivalent fix on main: capture async and sync copy errors, log a distinct restore-failed warning, make audit `valid` reflect actual restore success, persist nullable restore error details, and keep both async and sync failure-injection tests.
