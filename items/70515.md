## Summary

Keep #70515 open. Current main still has the reported config restore false-success bug: both async and sync suspicious-read recovery swallow backup copy failures, log a successful restore, and write observe audit records with `valid: true`. The PR is a focused, still-relevant fix, and the provided review context shows the author addressed the bot follow-ups rather than abandoning the work.

## Best Possible Solution

Keep the PR open and review/land it, or reimplement the same behavior on main: capture async and sync backup copy errors, log a restore failure instead of success, write `valid: false` with nullable restore error details, update non-recovery audit constructors, and keep both async and sync regression tests. Coordinate with #68462 only for rebase/conflict handling; it is not a canonical replacement for this fix.
