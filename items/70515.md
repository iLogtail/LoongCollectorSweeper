## Summary

Keep #70515 open. Current main still has the false-success behavior the PR describes: both suspicious-read recovery paths swallow backup copy failures, then log a successful restore and write `valid: true` audit records. The PR discussion is relevant and positive: bot review agreed the fix was correct, and the author addressed the two review follow-ups with a sync mirror test and warning-format cleanup.

## Best Possible Solution

Keep the PR open and review it as a focused bug fix. The likely best path is to land the behavior the PR proposes: capture copy errors in both async and sync recovery, emit a failure warning, set audit validity from the actual restore result, persist additive restore error details, and keep the async/sync regression tests added after review feedback.
