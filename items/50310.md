## Summary

Keep #50310 open. Current main has partial cleanup for archived Discord threads and a non-agent `sessions.delete` RPC, but it does not implement the requested stale/orphaned Discord session GC for deleted/inaccessible threads, nor does it expose an agent-facing `sessions_delete` or `sessions_gc` tool. The related #49259 ClawSweeper context also found broader dashboard/session pruning still missing, with #50310 remaining the narrower Discord cleanup item.

## Best Possible Solution

Keep the issue open and implement a narrowly scoped Discord session GC. The likely maintainer path is to reuse shared session deletion/archival semantics while adding Discord-aware reachability checks for `discord:channel:*`/thread session keys: handle deleted-thread events if the Carbon/Discord listener surface supports them, add a startup or explicit cleanup sweep that probes `GET /channels/{id}` for Discord thread sessions with the right account token, remove confirmed 404/403 orphan entries from `sessions.json`, and add tests for archived, deleted/inaccessible, permission-error, and multi-agent/multi-account cases. If exposing self-maintenance to agents is desired, add a dedicated permission-gated `sessions_delete` or `sessions_gc` tool rather than widening the generic gateway tool.

## Risks / Open Questions

- A naive Discord GC could delete valid private or permission-restricted sessions on transient 403s; the fix should distinguish confirmed deletion/inaccessibility from temporary auth or outage states.
- Current main's archived-thread path may already prune some stale entries through default session maintenance, so remaining work should be scoped to deleted/inaccessible thread cleanup and/or an explicit agent/admin cleanup surface rather than duplicating the archived event handler.
