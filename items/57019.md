## Summary

Keep #57019 open. Current main still has the reported ordering: `releaseHeldLock` removes the session from `HELD_LOCKS` before awaiting handle close and lock-file removal, while same-PID/no-starttime contention is reclaimed when `HELD_LOCKS` no longer contains the session. Existing tests cover active held locks and watchdog handoff, but not the overlapping async release/reacquire window described in the issue.

## Best Possible Solution

Keep the issue open and fix the lock helper directly: either keep `HELD_LOCKS` populated until close/rm completes, or add an explicit releasing guard checked by `shouldTreatAsOrphanSelfLock`. Add a focused regression test that blocks release cleanup, starts a concurrent same-session acquire with no valid `starttime`, and proves the old release cannot delete the newly acquired lock file.
