## Summary

Keep open. Current main has limited doctor-related plugin surfaces for plugin legacy config contracts and channel-specific doctor adapters, plus plugin-owned CLI commands, but I did not find a public extension point that lets a plugin add arbitrary environment-specific checks and `--fix` repairs into the built-in `openclaw doctor` flow. This issue is recent, author is not a maintainer, and it has no protected labels, so it is eligible for cleanup review but not closeable on the evidence.

## Best Possible Solution

Keep the issue open for maintainer product/API design. The likely path is an additive Plugin SDK doctor contribution surface, for example a versioned `api.registerDoctorCheck` or manifest-backed doctor contribution contract with clear `check` and `fix` modes, non-interactive behavior, ordering, trust gating, docs, and tests. Existing channel doctor adapters and plugin legacy config contracts should remain the narrow owner-specific paths where they already fit.

## Risks / Open Questions

- A generic doctor extension API could duplicate existing channel-doctor and plugin legacy-migration paths unless it is designed as an additive, narrow surface.
- Doctor repairs can mutate config and state, so plugin-provided fixes would need explicit trust, ordering, dry-run, non-interactive, and failure semantics.
