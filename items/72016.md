## Summary

Keep open. Current main has partial doctor extension surfaces for plugin legacy config migrations and channel-scoped doctor adapters, but it does not expose a general installed-plugin/custom-environment doctor contribution API for `openclaw doctor` and `--fix`, which is the central request in #72016. The related #68348 context is a doctor contract loading optimization, not a canonical implementation of custom doctor checks.

## Best Possible Solution

Keep #72016 open and design an additive plugin SDK doctor contribution surface: installed/enabled/trusted plugins should be able to register named doctor checks, warnings, and repair actions that integrate with `openclaw doctor` and `openclaw doctor --fix` using deterministic ordering, preview/repair separation, noninteractive semantics, docs, and contract tests. Reuse the existing channel doctor adapter for channel-specific checks and the doctor contract registry for legacy config migrations.

## Risks / Open Questions

- A general `doctor --fix` plugin API needs explicit trust, ordering, preview-vs-repair, noninteractive behavior, and config/state write contracts before implementation.
- Some channel-owned cases are already covered by `ChannelDoctorAdapter`, so the design should reuse that path rather than creating overlapping channel-specific hooks.
