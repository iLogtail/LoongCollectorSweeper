## Summary

Current `main` at `350d3221800a18283915cd2f3b612ae04ddd490c` still appears to have the reported Signal reply-path bug. The inline reply delivery path never threads agent-scoped `mediaLocalRoots` into `sendMessageSignal`, so local attachments from `workspace-*` agent workspaces can still fall back to default roots and trip the explicit `workspace-*` deny guard.

## Evidence

- **Signal reply delivery still omits mediaLocalRoots:** `deliverReplies` sends media via `sendMessageSignal(...)` with `cfg`, `baseUrl`, `account`, `mediaUrl`, `maxBytes`, and `accountId`, but no `mediaLocalRoots` field.
  - file: [extensions/signal/src/monitor.ts:306](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/extensions/signal/src/monitor.ts#L306)
  - command: `nl -ba extensions/signal/src/monitor.ts | sed -n '300,350p'`
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Signal event-handler contract still has no mediaLocalRoots slot:** The `deliverReplies` callback type only accepts `cfg`, `replies`, `target`, `baseUrl`, `account`, `accountId`, `runtime`, `maxBytes`, and `textLimit`, and the call site passes exactly those fields.
  - file: [extensions/signal/src/monitor/event-handler.types.ts:107](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/extensions/signal/src/monitor/event-handler.types.ts#L107)
  - command: `nl -ba extensions/signal/src/monitor/event-handler.types.ts | sed -n '95,120p'`
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Attachment resolution depends on opts.mediaLocalRoots:** `sendMessageSignal` forwards `opts.mediaLocalRoots` into `resolveOutboundAttachmentFromUrl`. If callers omit it, attachment loading uses default local roots instead of agent-scoped roots.
  - file: [extensions/signal/src/send.ts:155](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/extensions/signal/src/send.ts#L155)
  - command: `nl -ba extensions/signal/src/send.ts | sed -n '145,165p'`
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Default local-root enforcement still rejects workspace-* when roots are implicit:** `assertLocalMediaAllowed` has an explicit branch for `localRoots === undefined` that rejects paths whose first segment under state dir starts with `workspace-`. That matches the reported `workspace-bookkeeper` failure mode.
  - file: [src/media/local-media-access.ts:52](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/media/local-media-access.ts#L52)
  - command: `nl -ba src/media/local-media-access.ts | sed -n '40,70p'`
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Agent-scoped roots exist and other channels already thread them through:** `getAgentScopedMediaLocalRoots` appends the resolved agent workspace, and Discord's reply path computes and passes `mediaLocalRoots`, which is the contrast the report describes.
  - file: [src/media/local-roots.ts:54](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/media/local-roots.ts#L54)
