## Summary

Keep open. Current main has cron scheduling and an agent-facing cron tool, but it does not implement the RFC's central request: a hardened gateway-owned scheduling API with playbook allowlists, non-forgeable agent/session provenance, recurring-job expiration, per-agent quotas, and schedule-creation audit entries. The AS76 comment also supports the multi-agent orchestration use case, so this needs maintainer product/security design rather than conservative cleanup closure.

## Best Possible Solution

Keep the RFC open as core scheduler/security design work. The likely path is a maintainer-reviewed gateway-owned scheduling API, separate from unrestricted cron.add, with operator-registered playbooks and parameter schemas, gateway-injected actor/session provenance, recurring expiration aligned with #71711, per-agent quotas, creation/update audit records, and docs/tests covering the new trust boundary.

## Evidence

- **Issue discussion requests a distinct hardened path:** #71712 asks for a gateway-exposed schedule_task path, not direct jobs.json writes or unrestricted cron creation. It requires playbook allowlisting, gateway-injected createdBy/createdAt, activeUntil for recurring schedules, per-agent quota, and audit ledger entries. The provided AS76 comment confirms the multi-agent orchestration relevance.
- **Cron protocol schema lacks requested fields:** Current schedule variants are only at/every/cron, and CronAddParamsSchema accepts the ordinary cron job fields. There is no playbook, params, activeUntil, createdBy, provenance source, or quota field; additional properties are rejected.
