## Summary

Keep open. Current main has cron scheduling and an agent-facing cron tool, but it does not implement the RFC’s central request: a hardened gateway-owned scheduling API with playbook allowlists, non-forgeable agent/session provenance, recurring-job expiration, per-agent quotas, and schedule-creation audit entries. The AS76 comment also supports the multi-agent orchestration use case, so this should remain available for maintainer product/security design rather than be auto-closed.

## Best Possible Solution

Keep the RFC open as core scheduler/security design work. The best path is a maintainer-reviewed gateway-owned scheduling API, likely separate from unrestricted `cron.add`, with operator-registered playbooks and parameter schemas, gateway-injected actor/session provenance, recurring expiration aligned with #71711, per-agent quotas, creation/update audit records, and docs/tests covering the new trust boundary.

## Evidence

- **Issue discussion requests a distinct hardened path:** The issue asks for `schedule_task` exposed by the gateway, not direct `jobs.json` writes or unrestricted cron creation. It requires playbook allowlisting, gateway-injected `createdBy`/`createdAt`, `activeUntil` for recurring schedules, per-agent quota, and audit ledger entries. AS76 confirmed the multi-agent orchestration relevance in the provided issue comments.
- **Cron schedule schema lacks requested fields:** Current protocol schedule variants are only `at`, `every`, and `cron`, and `CronAddParamsSchema` accepts the ordinary cron job fields. There is no `playbook`, `params`, `activeUntil`, `createdBy`, provenance source, or quota field in the add schema; additional properties are rejected.

## Risks / Open Questions

- Closing as implemented would conflate the existing broad cron tool with the narrower security boundary requested here.
- This proposal spans gateway protocol, cron store compatibility, task/audit semantics, and agent tool policy, so it needs maintainer product/security judgment before implementation.
