## Summary

Keep open. Current `main` still treats HTTP MCP `url` and `headers` as literal strings with no `${ENV_VAR}` expansion step, while separate CLI bundle adapters already contain placeholder-decoding logic. The report is fresh, another user corroborated it, and the issue is already linked to an open fix PR, so conservative cleanup should not auto-close it.

## Evidence

- **HTTP MCP runtime passes headers through unchanged:** `resolveHttpMcpServerLaunchConfig()` validates the URL, then copies `raw.headers` via `toMcpStringRecord()` and returns the resulting strings directly. There is no env-template or SecretRef resolution in this path.
