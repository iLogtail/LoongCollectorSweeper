## Summary

Keep open. Current main still only supports `agents.defaults.userTimezone`; the per-agent config type and strict per-agent schema do not include `agents.list[].userTimezone`, and runtime timezone resolution for system prompt context and heartbeat active-hours fallback still reads the global default. The issue discussion's proposed resolution path remains valid and is not superseded by the related per-agent override items in the provided context.

## Best Possible Solution

Keep the issue open and implement a first-class per-agent timezone override in core config: add `userTimezone?: string` to `AgentConfig` and `AgentEntrySchema`, expose it through `resolveAgentConfig` or a dedicated `resolveAgentTimezone(cfg, agentId)` helper, then update system prompt, heartbeat active-hours, cron-style current-time, startup/session context, and envelope/user-timezone fallback sites to use the per-agent value before `agents.defaults.userTimezone` and host timezone. Add focused schema/resolution and runtime tests, plus docs for `agents.list[].userTimezone`.
