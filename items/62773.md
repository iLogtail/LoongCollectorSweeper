## Summary

Keep PR #62773 open. Current main still does not expose a first-class `list` directory-enumeration tool in OpenClaw’s coding tool set: the fs catalog/docs only cover `read`, `write`, `edit`, and `apply_patch`, and `createOpenClawCodingTools` composes those filesystem tools without adding `list`/`ls`. The PR remains relevant, but it should not be closed or treated as merge-ready because the existing PR review comments flag concrete sandbox-boundary, symlink, and truncation issues in the proposed implementation.

## Best Possible Solution

Keep the PR open and ask for a corrected implementation. The best path is a first-class directory listing tool wired through the same OpenClaw tool catalog, `group:fs` policy, required-param validation, host/sandbox filesystem bridge, workspace guards, docs, and focused tests. The implementation should resolve the existing review comments before maintainer review: deterministic sort-before-truncation, safe symlink metadata/type semantics, and no host filesystem bypass in sandbox sessions. Reusing or adapting upstream `createLsTool` operations may reduce custom code, but the final OpenClaw surface still needs explicit policy/catalog/docs coverage.
