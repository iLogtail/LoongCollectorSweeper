## Summary

Keep open. Current main still has no configurable URL/hostname deny list for `web_fetch`: the config type/schema expose normal fetch limits plus only `ssrfPolicy.allowRfc2544BenchmarkRange`, the tool passes only that policy into the guarded fetch path, and the shared SSRF guard only has hardcoded blocked hostnames plus allowlist-style exceptions. The provided related items do not supersede this request; #68631 is broader tool/path scoping and was itself kept open for remaining gaps.

## Best Possible Solution

Keep the issue open. The likely fix belongs in core `web_fetch` configuration and enforcement: add a documented `tools.web.fetch.denyHosts` field to config types, zod/generated schema/help, and web-fetch docs; normalize host patterns case-insensitively; enforce exact and `*.domain` matches before DNS resolution and on redirects; return a short blocked-host error; and add focused tests for exact, wildcard, case-insensitive, redirect, cache-key, and provider-fallback interactions. Consider whether the matcher should live in the shared SSRF helper so future guarded fetch callers can opt in without duplicating policy logic.
