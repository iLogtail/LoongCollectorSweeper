## Summary

Keep open. Current `main` still has a macOS LaunchAgent scan that can flag unrelated plists as "gateway-like" based on a raw `openclaw` substring anywhere in the plist, and there is no evidence of a fix shipped after the reported version. The false-positive detection part of the report remains plausible on `main`, even though the reported self-unload behavior is not directly explained by the current scan call sites.

## Evidence

- **Darwin scan still uses broad substring marker detection:** `detectMarker` returns `openclaw` or `clawdbot` if that substring appears anywhere in the plist contents, without requiring a gateway command or matching label.
  - file: [src/daemon/inspect.ts:56](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/daemon/inspect.ts#L56)
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Unrelated launchd plists can still be reported as extra gateway services:** `scanLaunchdDir` reports any plist with a detected marker unless it is the current launch agent label or passes the narrow `isOpenClawGatewayLaunchdService` check. For `openclaw` markers, that check only auto-whitelists plists with explicit service markers or labels starting `ai.openclaw.` and containing `gateway`.
