## Summary

Keep open. Current main still treats JSON-stringified chat content-block arrays as ordinary string content, so the vLLM double-serialization fix is not implemented on main or in the latest release. The PR remains useful, especially after the author addressed the mixed-content review in e415c6b, but the current heuristic still needs a narrower chat-block shape check before it is safe to merge.

## Best Possible Solution

Keep this PR open and revise it against current main. The best fix is a conservative shared normalization path that only treats a stringified JSON array as chat content when it has a recognizable chat content-block shape, extracts text consistently with the real-array path, preserves arbitrary JSON strings, routes string callers that currently bypass the helper through that same logic where needed, and adds focused tests in `src/shared/chat-content.test.ts` plus a targeted transport/display regression for the vLLM `openai-completions` response shape.

## Risks / Open Questions

- Current main can still carry raw JSON-stringified vLLM content blocks through extraction, prompt, display, or preview-style paths.
- Merging the PR as-is could regress arbitrary JSON string content whose objects merely contain a `type` key.
- Regression coverage is still needed for vLLM text blocks, mixed text/media blocks, invalid JSON fallback, and non-chat typed JSON arrays.
