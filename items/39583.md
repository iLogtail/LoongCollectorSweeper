## Summary

Keep open. Current main still treats JSON-stringified chat content-block arrays as ordinary strings, so the vLLM double-serialization bug is not implemented on main or clearly superseded. The PR remains a focused, useful fix, but the current PR discussion also identifies a real heuristic risk that should be addressed before merge.

## Best Possible Solution

Keep this PR open and revise it against current main. The best path is a conservative shared normalization helper that only parses stringified JSON arrays with a recognizable chat content-block shape, extracts text consistently with the real-array path, preserves arbitrary JSON strings, routes affected string-body callers through the same behavior where appropriate, and adds focused regression tests in `src/shared/chat-content.test.ts` plus a targeted OpenAI-compatible/vLLM transport or display regression.

## Risks / Open Questions

- Closing would discard a focused PR for a behavior that still exists on current main by source inspection.
- Merging the PR as-is may regress arbitrary JSON string content because the updated guard treats any JSON array of objects with a `type` key as chat content blocks.
- The final fix needs tests for text-only stringified blocks, mixed text/media stringified blocks, invalid JSON fallback, and non-chat typed JSON array preservation.
