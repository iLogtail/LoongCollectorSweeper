## Summary

Current `main` no longer uses the legacy Chrome Browser Relay extension code path that issue #40037 reports against. OpenClaw removed that relay path, migrated legacy `driver: "extension"` configs to Chrome MCP `existing-session`, and now documents the signed-in browser flow through Chrome MCP instead of the old `background.js` extension worker.

## Risks / Open Questions

- Users still running older pre-2026.3.22 builds may still hit the legacy extension bugs there, but that is not a current-main bug anymore.
- If there is a new bug in the Chrome MCP `existing-session` path that looks similar from the user side, it should be tracked as a new issue against the current architecture, not this removed extension worker.

## Close Comment

Closing from a Codex review because current `main` no longer ships the Chrome Browser Relay extension path this report targets. The legacy relay was removed in `CHANGELOG.md:2188` for release `2026.3.22`, `extensions/browser/src/browser/server-lifecycle.ts:13` makes the old relay startup helper a no-op, `src/commands/doctor/shared/legacy-config-core-normalizers.ts:27` migrates legacy `driver: "extension"` configs to Chrome MCP `existing-session`, and `extensions/browser/src/browser/routes/basic.ts:202` now rejects `driver: "extension"` entirely. The browser docs also state that `openclaw doctor` migrates old extension-based browser config to the current Chrome MCP flow (`docs/tools/browser.md:459`). Because the reported `~/.openclaw/browser/chrome-extension/background.js` worker is no longer the current implementation, this issue is already resolved on current OpenClaw. If you can reproduce similar tab-loss or browser-close behavior on the current Chrome MCP `existing-session` path in a current release, please open a fresh issue with the exact version, profile type, and reproduction steps.
