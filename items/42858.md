## Summary

Current main does not contain a `bailian` provider auth env mapping or `BAILIAN_API_KEY` candidate. The current Qwen/Alibaba provider docs and manifests support Alibaba Model Studio through canonical `qwen`/`alibaba` ids with `MODELSTUDIO_API_KEY`, `DASHSCOPE_API_KEY`, and `QWEN_API_KEY`, and current SecretRef support can resolve arbitrary custom env vars, but that is not the same as the reported `bailian` + `BAILIAN_API_KEY` compatibility path. The related fix PRs are unmerged/closed except #42978, which remains open, so the issue should stay open for maintainer judgment or a narrow compatibility fix.

## Best Possible Solution

Keep this issue open until maintainers choose the compatibility boundary. If `bailian` should keep working, add a narrow manifest/core compatibility entry or provider auth alias that maps `bailian` to the current Qwen/Model Studio auth surface, include `BAILIAN_API_KEY` only if that env name is accepted as public compatibility, and add a regression test around `resolveEnvApiKey('bailian')`/custom provider auth. If `bailian` is intentionally obsolete, update the Qwen/Alibaba docs and close this with a maintainer note directing users to `qwen` plus `MODELSTUDIO_API_KEY`/structured SecretRef for custom env names.
