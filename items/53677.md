## Summary

Keep open. The PR's optimization is still relevant on current main: the builtin memory read path still reads the entire Markdown file with `fs.readFile` before slicing, and the current focused tests verify slice behavior but do not enforce a streaming path. The PR is stale because the target file moved, but the product problem remains actionable.

## Best Possible Solution

Keep the PR open or ask the contributor to port it to the current memory host SDK layout. The best fix is to stream only the requested line window in the builtin `readMemoryFile()` path, feed the collected slice into `buildMemoryReadResultFromSlice`, update the mirrored package source or generation path consistently, and add a regression test in `extensions/memory-core/src/memory/manager.read-file.test.ts` that proves windowed reads do not call `fs.readFile` while preserving truncation metadata.
