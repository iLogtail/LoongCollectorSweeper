## Summary

Keep #26772 open. Current main still defaults to the builtin SQLite memory engine and still performs forced/full builtin reindexes by creating a fresh temp DB, indexing all sources into it, writing meta only after indexing completes, and then swapping it into place. I found progress reporting, embedding-cache reuse, QMD as an opt-in alternate backend, and atomic cleanup tests, but no durable checkpoint cursor, resumable temp DB discovery, resume-on-restart config, or regression coverage for interrupted full-reindex resume. The related #70567 is closed as an adjacent transient embedding-failure fix, not a canonical replacement for process-crash checkpoint/resume.

## Best Possible Solution

Keep #26772 open until memory-core has a durable builtin full-reindex checkpoint/resume design. A good fix should persist generation/checkpoint state or safely reuse/discard incomplete temp DBs, resume from committed file/chunk progress after process restart, avoid serving incomplete generations as clean results, expose status/config/docs if new controls are added, and add regression coverage for interrupted forced/full reindex behavior. Keep #70567 separate unless maintainers intentionally fold transient embedding-failure dirty-state reporting and process-crash resume into one broader memory reliability project.
