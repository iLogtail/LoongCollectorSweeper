## Summary

Keep #26772 open. Current main still uses a fresh temp SQLite database for safe full memory reindexing, swaps it only after the full build succeeds, deletes temp files on failures, and has no persisted checkpoint/resume schema, config, or tests for interrupted long-running indexing. The related #70567 is adjacent and open, but it tracks silent partial success after embedding failures, not process-crash resume.

## Best Possible Solution

Keep the issue open until memory-core has a durable full-reindex checkpoint/resume design or a maintainer explicitly chooses a different product direction. A good fix would persist indexing generation/checkpoint state, resume or safely discard incomplete work on restart, prevent search from serving incomplete generations, update memory status/config/docs if new controls are added, and add regression coverage for interrupted full reindex behavior.
