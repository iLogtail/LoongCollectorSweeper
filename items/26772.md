## Summary

Keep #26772 open. Current main still lacks durable checkpoint/resume for long-running memory full reindexing: safe reindex creates a fresh temp SQLite DB, builds the full index there, swaps only after success, and deletes temp files on failure. Progress remains in-memory callback state, and the schema/config/docs/tests do not define resumable generations or checkpoint controls. Related #70567 is adjacent and still open, but it tracks silent partial success after embedding 500s, not process-crash resume.

## Best Possible Solution

Keep #26772 open until memory-core has an explicit durable full-reindex checkpoint/resume design or maintainers choose a different product direction. A good fix should persist generation/checkpoint state, resume or safely discard incomplete work on restart, prevent search from serving incomplete generations, update memory status/config/docs if new controls are added, and add regression coverage for interrupted full reindex behavior. Keep #70567 separate unless a broader reliability fix intentionally covers both process-crash resume and transient embedding failures.
