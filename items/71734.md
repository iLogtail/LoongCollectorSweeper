## Summary

Keep #71734 open. Current main has adjacent `before_tool_call` hook behavior, config-driven tool filtering, and basic plugin tool catalog projection, but it does not expose the requested trusted pre-plugin policy tier, first-class tool metadata/display/safety registry, or an explicit core-tool decoration model. The linked RFC PR #71731 is open and described as docs-only/proposed, so it does not supersede the implementation work.

## Best Possible Solution

Keep the issue open and use #71731 as the design artifact. The best implementation path is one focused host-hook PR that adds an additive trusted pre-plugin tool policy registration contract, a first-class plugin tool metadata projection into catalog/UI/effective inventory, an explicit rule for whether bundled plugins may decorate core-owned tools such as `update_plan`, and fixture tests for ordering, external-plugin rejection, metadata projection, and disablement cleanup.

## Risks / Open Questions

- The RFC is broad and should not land as Plan Mode-specific core code; the host seam should stay generic and fixture-tested.
- The privileged policy stage needs explicit bundled/trusted authorization and negative tests for external plugins.
- Adding display/safety metadata affects gateway protocol, UI lookup, docs, disablement cleanup, and plugin contract tests.
