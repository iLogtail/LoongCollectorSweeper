## Summary

Keep PR #72124 open. Current main still rejects `sk-ant-api03-` through the exact Anthropic `paste-token` path reported in #72121, while the PR is a focused additive fix with relevant tests and supportive review comments. The separate Anthropic API-key login method exists, but it does not make this paste-token bug obsolete.

## Best Possible Solution

Keep #72124 open for maintainer review. The best path is to land a narrow additive change that widens the Anthropic pasted-token validator to accept both `sk-ant-oat01-` and `sk-ant-api03-`, updates the user-facing validation message, and adds regression coverage for both prefixes. If maintainers prefer `paste-token` to remain setup-token-only, then the alternative should be an explicit docs/CLI message that routes standard Anthropic API keys to `openclaw models auth login --provider anthropic --method api-key`; current main does neither for the failing path.
