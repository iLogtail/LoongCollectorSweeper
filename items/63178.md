## Summary

Current main already distinguishes explicit per-session model selections from transient fallback state, and the Telegram/main-session fallback path now has regression coverage showing user-selected overrides are preserved instead of being clobbered. I could not tie the fix to a released tag; latest release v2026.4.23 appears older than the code now on main.

## Evidence

- **Issue discussion provided consistent repro scope:** The issue comments from Adam-Researchh and darconadalabarga consistently narrowed this to persisted session-state clobbering across Telegram main sessions and TUI/webchat direct sessions, not just status/UI drift. That matches the regression-focused code now present on main.
- **Session entries now record whether an override is user-selected or auto-fallback:** `SessionEntry` now carries `modelOverrideSource` and documents that `/model`/`sessions.patch` are user-driven while system fallbacks are not. `liveModelSwitchPending` is likewise documented as user-driven only.

- **Auto fallback overrides self-heal instead of staying sticky:** `createModelSelectionState` explicitly treats `modelOverrideSource === "auto"` as transient, clears that stored override, and resets the in-memory provider/model back to the configured primary.
