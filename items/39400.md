## Summary

Keep #39400 open. Current main still exposes provider quota in `session_status` as an optional human-readable status line, not as stable structured usage fields with explicit availability/source/reset metadata. Some provider-window normalization exists, including Codex Day/Week labeling, but the issue’s monitor-facing contract remains missing and related issues are adjacent rather than canonical duplicates.

## Best Possible Solution

Keep the issue open. If maintainers want cron/monitor use of `session_status` supported, extend the provider usage snapshot/status contract with explicit availability, source, and reset metadata; return those fields in `session_status.details` while preserving the existing human-readable status line. Add focused tests for no-auth, error, empty-window, timeout, and Codex 5h/Day/Week mapping behavior.
