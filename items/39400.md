## Summary

Keep #39400 open. Current main still treats `session_status` provider quota as an optional human-readable status line, not a stable structured usage contract. It can omit the usage window line on errors, empty snapshots, missing auth, and timeouts, and the tool details still do not expose `usage.window5h`, `usage.windowDay`, source, reset reason, or explicit unavailable metadata. Current code has mitigations for some Codex Day-vs-Week labeling cases, but that does not satisfy the core request.

## Best Possible Solution

Keep the issue open until maintainers either add a stable typed provider-quota payload to `session_status` or explicitly define another supported monitor-facing API. A good implementation would extend the provider usage snapshot contract with availability/source/reset metadata, expose normalized 5h/Day windows in `session_status.details.usage` while preserving the existing text card, and add focused tests for success, provider error, empty windows, auth unavailable, timeout, and Codex Day/Week labeling.
