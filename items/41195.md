## Summary

Do not close. Current main already restores Feishu dedupe state from disk on startup, so the issue body's "cache not restored on restart" root-cause claim is outdated. But the actual inbound dedupe path still keys exclusively on Feishu `message_id`, and the only timestamp-based replay handling I found is for suppressing typing indicators, not for blocking inbound dispatch. If Feishu reconnect replay really arrives with fresh message IDs, current main would still process those as new messages, so there is not strong evidence to close this item.
