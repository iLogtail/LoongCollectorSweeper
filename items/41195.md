## Summary

Keep open. Current main has fixed the narrower Feishu same-message-id replay path: persistent per-account dedupe exists, it is warmed on monitor startup, and duplicate message IDs are dropped before dispatch. But #41195 specifically also claims Feishu WebSocket reconnect can replay recent messages with new message IDs and asks for a startup grace period or timestamp-age filtering. I found no current Feishu reconnect replay window, ignore-offline option, or stale-message timestamp filter, so the remaining requested behavior is not implemented or cleanly superseded.

## Best Possible Solution

Keep the issue open as a focused Feishu plugin bug/behavior request. The best path is to add a Feishu-owned reconnect replay guard, likely in the monitor/message-handler layer, that uses connection lifecycle plus message create_time or an explicit ignore-offline/replay-window option, with tests proving same-ID replay remains deduped and new-ID stale reconnect replay does not dispatch unexpectedly.

## Risks / Open Questions

- A grace-period or timestamp-age filter can drop legitimate offline messages, so the product decision should be explicit and likely opt-in or scoped to reconnect replay handling.
- Before implementation, maintainers should confirm whether current Feishu WebSocket replay really emits new message IDs under the reported reconnect conditions, because current code already handles same-ID replay well.
