## Summary

Keep #70364 open. Current main has a partial MCP cleanup for `runAgentStep` follow-up/announce steps, but the first `sessions_send` target run still enters the gateway as a nested lane without `cleanupBundleMcpOnRunEnd`, and the embedded runner only retires bundled MCP runtimes when that flag is true. The open #70465 PR remains the direct fix path for the reported gateway ingress leak.

## Best Possible Solution

Keep #70364 open until the gateway agent handler or the primary `sessions_send` path makes completed nested gateway runs retire their bundled MCP runtime. The likely maintainer path is to rebase/review #70465 or land an equivalent minimal fix that defaults cleanup for nested lanes while preserving explicit opt-out for persistent sessions, plus a gateway/server-method regression test for nested vs non-nested lanes.
