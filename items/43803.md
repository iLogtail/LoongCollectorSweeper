## Summary

Keep #43803 open. Current main fixed the broad unconditional config.patch SIGUSR1 problem, but not the browser-profile case this issue tracks: browser config is still registered as restart-required, the reload planner still treats browser paths as gateway-restart paths, and config.patch still directly schedules SIGUSR1 in hot mode when the planner says restart is required. The attempted PR #45624 was closed unmerged, and #64204 was closed as a duplicate of this issue.

## Best Possible Solution

Keep #43803 as the canonical tracker. Implement granular browser reload planning so `browser.profiles.*` and likely `browser.defaultProfile` do not trigger a gateway restart, while service-affecting browser settings still restart as needed. Add reload-plan and config.patch regression tests for browser profile `cdpUrl` changes, and update docs to match the final browser reload contract.
