## Summary

Keep #7575 open. Current main still does not implement Sysbox/sysbox-runc, socketless Docker-in-Docker gateway isolation, or a supported `sandbox.docker.runtime` selector. Main does now expose a plugin sandbox backend seam, so a Sysbox backend plugin is plausible, but the latest issue discussion explicitly treats this as the canonical Sysbox/runtime-architecture tracker that needs maintainer product judgment rather than conservative cleanup.

## Best Possible Solution

Keep #7575 open as the canonical Sysbox/runtime-architecture tracker. A good resolution would either add a conservative `sandbox.docker.runtime` path with schema, docs, tests, and host setup guidance, or endorse a Sysbox sandbox backend plugin using the existing `registerSandboxBackend` SDK if maintainers decide this belongs outside core.

## Risks / Open Questions

- Closing as implemented would conflate current Docker-socket sandbox support with the requested socketless Sysbox/sysbox-runc architecture.
- Adding a Docker runtime selector in core would be a config/API/security surface change requiring types, strict schema, docs, tests, and host setup guidance.
- A ClawHub/plugin path is plausible now that sandbox backends are pluggable, but maintainer judgment is still needed on whether Sysbox should be endorsed, documented, or left as an external backend plugin.
