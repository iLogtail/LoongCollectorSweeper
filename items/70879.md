## Summary

Keep #70879 open. Current main still lacks bounded retry/backoff for Feishu outbound message sends: text/card/media send paths call the Feishu SDK once and fail on non-zero API responses, while the only Feishu rate-limit/backoff handling found is scoped to typing indicators. This is a bundled Feishu plugin delivery reliability bug, not a stale report or ClawHub-only request.

## Best Possible Solution

Keep the issue open and implement a Feishu-owned bounded retry layer around outbound create/reply/send-media operations in extensions/feishu. Centralize retryable Feishu API classification, include HTTP 429 and confirmed flow-control/rate-limit codes, preserve withdrawn-reply fallback semantics, avoid unsafe retries for ambiguous post-delivery failures, and add focused tests for non-throwing API responses plus thrown SDK/Axios error shapes.

## Risks / Open Questions

- A naive retry can duplicate Feishu messages if the original request succeeded but the SDK response was lost; the fix should retry only well-classified rate-limit/pre-delivery failures with bounded attempts and backoff.
- Feishu error code taxonomy should be verified against upstream docs or SDK behavior before treating broad codes like 2200 as safely retryable.
