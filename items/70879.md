## Summary

Keep #70879 open. Current main still sends Feishu text/card/media messages with single SDK calls and converts non-zero Feishu API responses into thrown errors; there is no bounded retry/backoff path for the requested rate-limit codes. Existing Feishu backoff handling is limited to typing-indicator reactions, not outbound message delivery.

## Best Possible Solution

Keep the issue open and implement this inside the Feishu plugin. Add a Feishu-owned bounded retry runner around outbound message delivery paths in extensions/feishu/src/send.ts and extensions/feishu/src/media.ts, with strict retry predicates for Feishu rate-limit/flow-control codes such as 11232, 2200 where appropriate, HTTP/SDK 429, and Retry-After hints. Add focused tests for non-throwing API responses and thrown SDK/Axios shapes, plus negative tests proving ordinary send failures and withdrawn-message fallback behavior are not retried incorrectly.
