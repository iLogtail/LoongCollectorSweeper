## Summary

Keep open. Current `main` still sends Feishu messages through one-shot `im.message.create` / `im.message.reply` calls with immediate error propagation and no retry wrapper for rate-limit/backoff conditions. The repo already has shared retry infrastructure, and Telegram uses it for outbound sends, so this looks like a real Feishu gap rather than an already-implemented feature. One nuance: the issue body overstates the failure mode slightly, because current Feishu send code throws on failure instead of silently marking the send successful.

## Evidence

- **Feishu send path is one-shot:** `sendFallbackDirect()` and `sendReplyOrFallbackDirect()` call `client.im.message.create()` / `client.im.message.reply()` directly and only special-case withdrawn reply targets; there is no retry runner or backoff loop in this path. `sendMessageFeishu()` and `sendCardFeishu()` both route through that helper.
  - file: [extensions/feishu/src/send.ts:103](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/extensions/feishu/src/send.ts#L103)
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **Feishu send failures throw immediately:** `assertFeishuMessageApiSuccess()` throws on any non-zero Feishu response code. That means current behavior is fail-fast, not retry-on-rate-limit.
  - file: [extensions/feishu/src/send-result.ts:9](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/extensions/feishu/src/send-result.ts#L9)
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **Test suite expects rate-limit-like Feishu send errors to be rethrown:** The reply-fallback test injects SDK error code `99991400` (`rate limited`) and asserts the send rejects and does not fall back to another send path. That confirms current intended behavior is propagation, not retry.
  - file: [extensions/feishu/src/send.reply-fallback.test.ts:169](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/extensions/feishu/src/send.reply-fallback.test.ts#L169)
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **OpenClaw already has shared retry infrastructure:** The plugin SDK exports shared retry helpers (`retryAsync`, `createRateLimitRetryRunner`, `createChannelApiRetryRunner`), so a retry-based fix fits existing repo patterns instead of needing new core scope.
  - file: [src/plugin-sdk/retry-runtime.ts:1](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/src/plugin-sdk/retry-runtime.ts#L1)
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **Telegram already uses the shared retry runner for outbound send requests:** Telegram wraps non-idempotent outbound sends in `createTelegramRetryRunner(...)` with explicit retry predicates for safe send errors and rate limits. Feishu has no analogous wrapper today.
  - file: [extensions/telegram/src/send.ts:492](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/extensions/telegram/src/send.ts#L492)
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **Published Feishu SDK does not add automatic retries:** Inspection of the published `@larksuiteoapi/node-sdk@1.61.1` tarball showed a plain `axios.create()` default HTTP client and a request path that performs a single `httpInstance.request(...)` then rethrows on error; no built-in retry middleware was present in the package entrypoint inspected.
  - command: `curl -fsSL https://registry.npmjs.org/%40larksuiteoapi%2Fnode-sdk/-/node-sdk-1.61.1.tgz | tar -xzOf - package/es/index.js | sed -n '1,150p;83970,84040p'`

## Risks / Open Questions

- The issue body slightly overstates the symptom: current code propagates send failures instead of silently reporting success, so the user-visible impact depends on how higher layers surface that exception.
- This was a read-only code review, not a live Feishu rate-limit reproduction; the decision is based on current source, tests, and the published SDK package behavior.
