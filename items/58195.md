## Summary

Keep open. Current `main` does not show a clear end-to-end fix for Ollama-style reasoning in Control UI chat: the webchat event path still emits text-only assistant chat payloads, the UI only renders reasoning from structured `thinking` blocks, and the OpenAI-compatible stream parser still looks for `reasoning*` fields rather than an Ollama `thinking` field.

## Evidence

- **UI only renders structured thinking blocks:** `extractThinking()` only pulls assistant reasoning from `content[]` items with `type: "thinking"` (or legacy `<think>` tags), and `grouped-render` only shows the reasoning panel when that extracted thinking exists.
  - file: [ui/src/ui/chat/message-extract.ts:44](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/ui/src/ui/chat/message-extract.ts#L44)
  - command: `nl -ba ui/src/ui/chat/message-extract.ts | sed -n '44,73p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Webchat live chat events are text-only:** `emitChatDelta()` and `emitChatFinal()` build Control UI chat payloads with `content: [{ type: "text", text: ... }]` only; they do not forward any `thinking` blocks into the webchat `chat` event stream.
  - file: [src/gateway/server-chat.ts:732](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/gateway/server-chat.ts#L732)
  - command: `nl -ba src/gateway/server-chat.ts | sed -n '683,845p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Control UI chat handler consumes only visible text during streaming:** For `delta` events, the chat controller calls `extractText(payload.message)` and stores plain streamed text; on `final`, it appends the provided assistant message or falls back to a synthetic text-only assistant message.
  - file: [ui/src/ui/controllers/chat.ts:451](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/ui/src/ui/controllers/chat.ts#L451)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '451,469p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **OpenAI-compatible stream parser still ignores an Ollama `thinking` field:** The streaming parser maps `reasoning_details`, `reasoning_content`, `reasoning`, and `reasoning_text` into thinking blocks, but there is no matching handler here for a provider delta field named `thinking`, which is the field the report shows from Ollama.
  - file: [src/agents/openai-transport-stream.ts:1445](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/agents/openai-transport-stream.ts#L1445)
  - command: `nl -ba src/agents/openai-transport-stream.ts | sed -n '1445,1470p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)

## Risks / Open Questions

- I did not run a live Ollama reproduction, so this is based on current code paths rather than runtime proof.
- Control UI does reload `chat.history` after final events; if some provider path already persists `thinking` blocks into transcript history, the final post-run state could differ from the live streamed state.
