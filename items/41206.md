## Summary

Keep #41206 open. Current main still uses a single cron store file and does not implement the requested per-job cron/jobs/ directory store. The PR is a plausible implementation path tied to the still-open GitOps feature request #20950, but it has unresolved review-identified data-integrity bugs around filename keying, so the right maintainer action is revision/review rather than automated closure.

## Best Possible Solution

Keep the PR open for maintainer review or contributor revision. The implementation should key per-job files by immutable job id or otherwise load existing files to match persisted ids before save/remove, add regression tests for id/name divergence and rename/delete behavior, then align docs/config/doctor behavior with the chosen directory-store contract. If this PR is abandoned, keep #20950 as the canonical feature request rather than auto-closing the implementation as obsolete.
