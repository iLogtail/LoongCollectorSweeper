## Summary

Keep #65726 open. Current main still does not implement the requested ACP binding `notifyPolicy` config or a thread-level duplicate-response guard. ACP binding config remains limited to mode/label/cwd/backend (and per-agent ACP runtime defaults omit notifyPolicy), ACP prompt turns create task records without a policy so session-owned ACP tasks still default to `done_only`, and Slack/runtime binding code routes inbound thread replies to the bound ACP session but does not suppress independent sessions that also target the same Slack thread.

## Best Possible Solution

Add a real ACP notify policy contract (`done_only`/`state_changes`/`silent`) to `AgentRuntimeAcpConfig` and `AgentAcpBinding.acp`, update strict schema/help/docs/generated schema, and propagate it through configured/runtime ACP binding resolution into ACP task creation. Separately, design a conservative thread ownership guard around session binding/routing or outbound delivery so a non-owning session targeting an actively bound Slack thread is suppressed, warned, or explicitly requires override. Cover with schema tests, ACP manager/task tests, and Slack bound-thread routing/outbound tests.
