## Summary

Keep open. Current main has a generic `openclaw pairing approve ... --notify` path, but the WhatsApp plugin does not implement an approval notifier, so approving a WhatsApp pairing code still only adds the sender to the allow store. Related #56621 is separate pairing-management work and does not supersede this notification gap.

## Best Possible Solution

Implement this in the WhatsApp plugin using the existing channel pairing notification seam: add a WhatsApp approval notification hook/message, wire `pairing approve --notify` through the selected account, add focused WhatsApp tests for the notifier and CLI account forwarding, and update the WhatsApp/pairing docs. Keep #56621 separate because list-approved/revoke does not cover sender notification.

## Risks / Open Questions

- The current notify path does not pass the CLI `--account` value into `notifyPairingApproved`, so a WhatsApp implementation should handle multi-account delivery before relying on the default account.
- Replaying the original blocked message is a larger behavior change than sending a confirmation and should be treated separately unless maintainers explicitly want it.
