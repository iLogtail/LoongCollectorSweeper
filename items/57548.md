## Summary

Keep open. Current `main` still persists Telegram native-command session metadata to the slash-envelope session key instead of the routed target session key, and the current tests still encode that old behavior. This PR appears to address a real unfixed bug rather than work already implemented on `main`.

## Evidence

- **Current main still writes session metadata to the slash session key input:** `registerTelegramNativeCommands` builds `ctxPayload` with both `SessionKey` and `CommandTargetSessionKey`, but the metadata write still passes `ctxPayload.SessionKey ?? route.sessionKey` into `recordInboundSessionMetaSafe` on current `main`. There is no fallback to `CommandTargetSessionKey` in this callsite.
