## Summary

Keep open. Current main has opt-in provider request proxy plumbing and better provider-error formatting, but it does not prove the reported Gemini + ambient `http_proxy`/`https_proxy` hang is fixed. The Gemini LLM path still goes through `buildGuardedModelFetch`, which calls `fetchWithSsrFGuard` without selecting `TRUSTED_ENV_PROXY` or `TRUSTED_EXPLICIT_PROXY`; the guard defaults to strict DNS pinning, and tests explicitly assert strict mode ignores env proxy settings. That means the reported Fake-IP/private-resolution block can still happen before the proxy gets to resolve the target.

## Best Possible Solution

Keep the issue open and make the product decision explicit. If Gemini model-provider requests should support operator-approved proxy routing, wire `models.providers.google.request.proxy` through `buildGuardedModelFetch` to the appropriate trusted guarded-fetch mode, add regression coverage for Gemini + env/explicit proxy + Fake-IP DNS, and ensure CLI/WebUI show DNS/proxy/SSRF failures. If ambient env vars must stay ignored by default, document the required config and emit a clear diagnostic when proxy env vars are present but strict DNS pinning blocks the provider target.
