## Summary

Keep open. Current main has relevant proxy plumbing, but the Google Gemini LLM path still does not automatically switch model-provider fetches into trusted env-proxy mode when only HTTP_PROXY/HTTPS_PROXY are set. That means strict guarded fetch can still DNS-pin generativelanguage.googleapis.com locally and hit the Fake IP SSRF block before the configured proxy is used. A later open report, #72057, appears to track the same Google LLM proxy mismatch, but without maintainer consolidation this older, more detailed report should remain open.

## Best Possible Solution

Keep the issue open. The likely fix belongs in the model-provider LLM transport path: make buildGuardedModelFetch use the same conservative shouldUseEnvHttpProxyForUrl/TRUSTED_ENV_PROXY behavior already used by provider HTTP helpers when HTTP_PROXY or HTTPS_PROXY applies and NO_PROXY does not, without dropping explicit TLS/proxy overrides. Add focused regression coverage for Google Gemini LLM requests behind an env HTTP proxy with local Fake IP DNS, and verify CLI/WebUI surface a descriptive error when the proxy itself fails.
