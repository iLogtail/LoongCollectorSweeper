## Summary

Keep #65223 open. Current main has partial abort plumbing around the embedded prompt and passes a run AbortSignal into tools, but it still does not implement the requested framework-level tool execution interruption or the key per-tool cancellation paths called out in the issue. `process poll` still ignores the signal during its timeout loop, and gateway-backed `sessions_send` still has only timeout-based cancellation.

## Best Possible Solution

Keep the issue open as the canonical design discussion. The best implementation path is to add an explicit tool-execution interruption contract, probably in the OpenClaw tool adapter or upstream pi-agent-core boundary, then add cooperative signal handling for `process poll` and a signal-aware gateway request path for `sessions_send`. Tests should cover ignored-signal tools, process poll partial/early return behavior, transcript history semantics, and the distinction between interrupt and steer.

## Risks / Open Questions

- A tool-level Promise.race would change transcript semantics: the agent may record an interrupted tool result while the real tool continues in the background.
- Per-tool cooperative cancellation needs careful handling for exec/process so aborting a poll does not kill a background process that should remain resumable.
- Gateway cancellation likely needs an API contract change in `callGateway`/`GatewayClient.request`, not just a sessions_send-local change.
