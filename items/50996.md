## Summary

Keep open. Current main mitigates the exact 300000ms example by clamping `process poll` waits to 120 seconds and documents completion wakes / non-polling guidance, but it still does not implement the requested interruptible or parallel handling. The `process` tool receives an abort signal but ignores it, and steering/inbound queue docs say messages are delivered only after the current tool calls finish. The KeepMyClaw comment is useful operational context, but it is a workaround for session recovery, not a native fix for tool-call interruptibility.

## Best Possible Solution

Keep this issue open. The best core path is to make long-running tool waits observe the existing run abort signal: add an abortable sleep/check to `process(action: "poll")`, ensure `/stop` and queue `interrupt` unwind the pending tool call with a clean tool error/result, and add a regression test that an in-flight poll is cancelled promptly when the run is aborted. Docs should then clarify the max poll timeout, completion-wake guidance, and how queue modes interact with abortable tools.
