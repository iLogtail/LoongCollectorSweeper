## Summary

Keep open. Current main still has no `defaultUser`/`fallbackUser` trusted-proxy config, and the trusted-proxy auth path still fails closed when the configured identity header is absent. The provided discussion also explicitly says #25293 did not fix this and distinguishes #17761 as a separate fallback problem.

## Best Possible Solution

Keep this open for maintainer/security product decision. A complete implementation would add an explicit trusted-proxy fallback/default user option to `GatewayTrustedProxyConfig`, zod and generated schema/help, CLI configuration, docs, security audit wording, and auth tests, then update `authorizeTrustedProxy` so fallback is allowed only after the request source and required headers satisfy the trusted-proxy contract and `allowUsers` semantics remain explicit. If maintainers decide fail-closed identity headers are intentional, the best outcome is to document Cloudflare IP-whitelist/no-identity and loopback cloudflared setups as unsupported rather than auto-close this as already solved.
