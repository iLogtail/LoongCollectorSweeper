## Summary

Keep #23585 open. Current main still has no trusted-proxy default/fallback user support: the auth helper requires the configured identity header, the dispatcher returns `trusted_proxy_user_missing` directly, the typed/zod/generated config surfaces expose only `userHeader`, `requiredHeaders`, and `allowUsers`, and docs/tests still describe and assert fail-closed identity-header behavior. The related discussion also explicitly says #25293 and #17761 do not cover this Cloudflare/WebSocket missing-header case.

## Best Possible Solution

Keep #23585 open for maintainer/security product judgment. If accepted, add an explicit trusted-proxy fallback/default user option across `GatewayTrustedProxyConfig`, zod/generated schema, config help/docs/security audit guidance, and auth tests, then update `authorizeTrustedProxy` so fallback is allowed only after trusted proxy source and required-header checks pass and `allowUsers` semantics remain explicit. If maintainers want identity headers to remain fail-closed, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported rather than closing this as implemented.
