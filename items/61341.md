## Summary

Current main still contains and routes the exact exec-approval follow-up prompt shown in #61341’s screenshots, and the Control UI/history filters do not hide that prompt class. The issue is recent, has concrete screenshots, is not maintainer-authored, has no protected close label, and should remain open for a product/implementation fix rather than conservative cleanup.

## Best Possible Solution

Keep this open. The likely fix belongs in the exec-approval follow-up path: represent completion follow-ups as internal/system provenance or internalEvents, persist a user-facing summary instead of the raw continuation prompt, and update gateway/Control UI history projection tests so the raw `An async command...` instruction block never appears as normal chat content while approved async execs can still resume the session once per approval id.
