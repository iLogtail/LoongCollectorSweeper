## Summary

Keep open. Current main has a partial mitigation for zero-valued/default poll params, and the Discord `components` schema is no longer exposed on the generic message tool in the scoped schema test. But the core issue is not fully fixed: poll creation fields are still present in the shared `message` tool schema, Telegram poll extras can be merged into schemas for other configured/current channels, and the `action="send"` runtime guard still rejects non-empty poll params instead of stripping or action-scoping them. The issue discussion also has multiple Telegram reproductions plus related open PRs attempting the broader strip/schema split.

## Best Possible Solution

Keep #43015 open until maintainers either land a broader fix or explicitly decide the current policy is intended. The likely fix belongs in the shared message tool/action layer: scope poll creation fields to `action="poll"` or strip model-populated poll params before `send`, keep the zero-default mitigation, and add focused regression tests for Telegram/Feishu/Discord simple sends with model-style extra poll/components payloads. Discord components should remain off the generic send schema unless an explicit advanced action/API is chosen, and empty modal skeleton handling should be decided in the Discord plugin runtime.
