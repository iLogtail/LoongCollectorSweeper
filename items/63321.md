## Summary

Keep this PR open. Current main has not implemented the proposed parse hardening, and the PR is a plausible focused bug-fix candidate, but its current approach needs maintainer review because it changes error semantics for Tailscale whois and funnel status parsing. The provided review discussion is relevant: Greptile’s reuse-helper suggestion was addressed by the author, while the later Codex review comments identify concrete behavioral risks that are still unresolved by the PR as described.

## Best Possible Solution

Keep the PR open for review or revision. The best fix is a targeted parse-result path that preserves current error semantics: malformed whois JSON should stay on the short error-TTL path, malformed funnel status should report an unknown/parse failure rather than disabled Funnel, and regression tests should cover malformed Tailscale status, whois, and funnel status output.
