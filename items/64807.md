## Summary

Current main still contains the reported divergence path. The gateway hydrates `updateAvailable` from `update-check.json`, then returns early for any fresh `lastCheckedAt` before resolving the npm channel; a stale/equal `lastAvailableVersion` or missing `lastAvailableVersion` can therefore suppress the Control UI banner while `openclaw update status` uses the live registry path. The issue is recent, concrete, and actionable, so it should remain open.

## Best Possible Solution

Keep the issue open and fix the gateway update-check path. The likely fix belongs in `src/infra/update-startup.ts`: do not let a fresh empty or non-newer persisted cache suppress the banner for the full normal interval when the CLI live registry path would report a newer package; preserve the last known good available version on transient empty registry results, schedule a near-term retry, and add focused tests for stale/equal `lastAvailableVersion`, missing `lastAvailableVersion`, and CLI/gateway availability parity.

## Evidence

- **Gateway trusts persisted cache during throttle window:** `runGatewayUpdateCheck` reads `update-check.json`, derives `updateAvailable` from `lastAvailableVersion`, then returns when `now - lastCheckedAt < checkIntervalMs` before calling `resolveNpmChannelTag`. A fresh cache with no usable newer `lastAvailableVersion` leaves the banner null for the interval.

There is no short retry or invalid-cache marker, so an empty result can make the next gateway pass skip for the normal interval.
