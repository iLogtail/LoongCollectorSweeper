## Summary

Keep #61502 open. Current main still does not reliably route Slack interactive reply clicks to the originating thread session: block actions enqueue a system event with the parsed thread timestamp in the payload, but session-key resolution omits `threadTs`, Slack interactions do not receive the normal inbound message dispatcher, and no Slack interaction path calls the heartbeat wake scheduler. Related cleanup has already treated #61502 as the canonical tracker for this remaining bug, with active fix PRs #67751 and #61374 still open.

## Best Possible Solution

Keep #61502 open as the canonical Slack interactive-reply routing bug. The fix should make Slack block-action clicks create exactly one agent turn for the correct resolved session and thread, preserve outbound delivery context for replies back to the same Slack thread, avoid duplicate/retry storms for busy sessions, and add focused regression coverage in the Slack interaction tests. Active PRs #67751 and #61374 are relevant candidate fixes or design references, but current main has not shipped the behavior yet.
