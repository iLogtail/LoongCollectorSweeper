## Summary

Current `main` already contains the WhatsApp auth-state hardening and bounded reconnect behavior described in #56054, and the same behavior is present in the shipped `v2026.4.22` release.

## Evidence

## Close Comment

Thanks for the report. This was a Codex review against current `main` and the shipped `v2026.4.22` code, and this looks implemented already.

Concrete evidence:
- OpenClaw now writes WhatsApp `creds.json` atomically through a temp file + rename, with cleanup on failure (`extensions/whatsapp/src/creds-persistence.ts:32`).
- It no longer rotates `creds.json.bak` when the current `creds.json` is corrupted, which directly addresses the corruption/backup cycle described here (`extensions/whatsapp/src/session.ts:83`).
- Backup restore is constrained to a parseable regular file and restores via temp file + rename (`extensions/whatsapp/src/auth-store.ts:73`).
- The reconnect monitor now treats unstable auth as a bounded retry path and stops with an explicit error after max attempts instead of looping indefinitely (`extensions/whatsapp/src/auto-reply/monitor.ts:317`).
- The issue timeline also links commit `45e230a59ccd8d3332ebd04a48906f10fb4682f0` / PR #66863 (`fix(whatsapp): atomic auth state + socket keepalive tuning`), and the same behavior is present in `v2026.4.22`.

If you still see a fresh 499 loop on `v2026.4.22` or newer, please open a new issue with current logs, exact OpenClaw version, and the active WhatsApp channel/account config, since that would likely be a new regression rather than the original corruption-cycle bug.
