## Summary

Keep #69408 open. Current main still treats loop-detection findings as warning/critical only, and the cron isolated-run context needed for the requested downgrade is not wired into the before-tool-call hook. The issue is recent, concrete, and describes a plausible core logging/diagnostics behavior gap rather than stale or ClawHub work.

## Best Possible Solution

Keep the issue open and implement a narrow core fix: add explicit loop-detection run context to HookContext, pass sessionTarget/schedule interval/run bounds from isolated cron execution, downgrade only bounded isolated cron warnings to info while preserving critical blocking, and add regression tests around pi-tools before-tool-call plus cron isolated runs. If the interval/lifetime rule is too ambiguous, a documented config option for cron-isolated warning suppression would be the conservative fallback.

## Risks / Open Questions

- A fix needs explicit run context passed through the tool wrapper path; inferring isolated cron from session-key shape alone could misclassify persistent session targets or non-cron runs.
- Changing warning entries to info also requires updating the diagnostic event/logging contract, which currently only models warning and critical loop events.
- The lifetime <= cron interval rule needs a defined behavior for one-shot jobs, manual cron runs, delayed retries, and overlapping/concurrent cron executions.
