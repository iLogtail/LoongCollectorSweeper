## Summary

Keep open. Current main has several related partial session capabilities, but it does not fully satisfy #59109's requested OpenClaw-level session fork/resume/continue surface. Explicit resume by session id is partly available, ACP has `resumeSessionId`, and native subagents can fork the current requester transcript, but there is no general `forkSession(id)`-style public API for arbitrary sessions and no first-class `continue: true` / most-recent-session option.

## Best Possible Solution

Keep the issue open and implement a deliberate session-control surface: a generic `sessions.fork`/branch operation that accepts a resolved key or session id and creates a new transcript/session key from that point, a documented resume-by-id path across CLI/gateway/client APIs, and a `continue latest` selector that deterministically chooses the most recent eligible session for an agent. Add docs and tests covering duplicate ids, agent scoping, stale/missing transcripts, and ACP-vs-OpenClaw id namespaces.

## Risks / Open Questions

- The eventual design should distinguish OpenClaw session keys, OpenClaw session ids, and upstream ACP/Codex session ids so resume and fork do not accidentally target the wrong namespace.
- Existing ACP resume, subagent fork, and checkpoint-branch features are useful building blocks, but closing this issue now would hide the missing first-class user/API workflow.
