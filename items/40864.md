## Summary

Keep open. Current `main` still treats `sessions_send` agent-to-agent delivery as best-effort: timeout/error paths return without durable queue or retry semantics, and announce-send failures are only logged. This is a coherent core gap, not a high-confidence close.

## Evidence

- **`sessions_send` exits on timeout/error without recovery flow:** `createSessionsSendTool()` starts the nested run, then waits once. In the waited path it returns `status: "timeout"` or `status: "error"` immediately, while `startA2AFlow()` is only invoked for fire-and-forget acceptance or after an `ok` wait result. That matches the report's "unknown if delivered after timeout" concern.
  - file: [src/agents/tools/sessions-send-tool.ts:338](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/agents/tools/sessions-send-tool.ts#L338)

  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Current tested contract is only `delivery.status = pending`:** The session-tool tests assert that both fire-and-forget and waited `sessions_send` calls return announce delivery as `pending`, not confirmed/durable delivery.
