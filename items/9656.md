## Summary

Keep #9656 open. Current main still folds `envelope.editMessage?.dataMessage` into the normal Signal inbound path, drops `editMessage.targetSentTimestamp` from the typed envelope, derives `MessageSid` from the new envelope timestamp, and dispatches the resulting message through the regular reply pipeline. The previously linked PR #9701 is closed and unmerged, and the existing maintainer-side review comment also concluded the issue remains open.

## Best Possible Solution

Keep the issue open and implement the behavior in the bundled Signal plugin: type and validate `editMessage.targetSentTimestamp`, preserve edit metadata or an `[edited]` marker in agent/session context, use the target timestamp for dedupe or transcript replacement, suppress edit-triggered replies unless the edit explicitly mentions the agent, and add focused Signal tests for edit envelopes, `MessageSid`/dedupe behavior, and mention-gated group edits.
