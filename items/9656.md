## Summary

Keep #9656 open. Current main still normalizes Signal `editMessage` envelopes as ordinary inbound messages, drops `editMessage.targetSentTimestamp`, assigns `MessageSid` from the new envelope timestamp, and dispatches through the normal reply path. The related PR #9701 is closed and unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep the issue open until the bundled Signal plugin explicitly handles `editMessage`: type and validate `editMessage.targetSentTimestamp`, preserve edit metadata or an `[edited]` marker in agent/session context, use the target timestamp for dedupe or transcript replacement, and suppress edit-triggered replies unless the edit explicitly mentions the agent. Add focused Signal plugin tests for edit envelopes, `MessageSid`/dedupe behavior, and group mention-gated edit handling.
