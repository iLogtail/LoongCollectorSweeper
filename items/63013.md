## Summary

Keep open. Current `main` has a fresh provider-ownership refactor, but it does not clearly implement this PR's hook-alias ownership preservation path. Explicit provider ownership still resolves from manifest activation metadata plus manifest `providers`/`cliBackends`, while unresolved hook lookups still fall back to an unscoped provider load. The alias cases called out in the PR discussion (`google-vertex`, `google-antigravity`, `azure-openai-responses`, `minimax-cn`) are still runtime-only `hookAliases` in bundled plugins rather than manifest-owned provider ids, so there is not strong evidence that the PR is obsolete or already implemented on `main`.

## Evidence

- **Current explicit owner resolution is still manifest-first only:** `resolveExplicitProviderOwnerPluginIds` on current `main` consults `resolveManifestActivationPluginIds(...)` and then falls back only to `resolveOwningPluginIdsForProvider(...)`; there is no active-runtime or setup-registry alias-owner lookup in this path.

It does not inspect runtime `hookAliases`, which is the gap this PR discussion spent most of its revisions addressing.

## Risks / Open Questions

- Current main may have reduced the original unrestricted-load failure mode via explicit empty plugin scopes, but the hook-alias ownership path from this PR is still not clearly covered.
- A future manifest migration could supersede this PR by declaring these aliases in manifest activation/setup metadata, but that migration is not present in the checked-out `main`.
