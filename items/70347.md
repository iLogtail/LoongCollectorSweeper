## Summary

Keep #70347 open. Current main still has the reported cron outer-timeout gap: the timer aborts the isolated run and rejects the race, but there is no cron-side lifecycle event or synchronous session-store finalization path, and timeout result handling can drop the cron run session identity.

## Best Possible Solution

Keep the issue open and implement a cron-owned timeout finalization path. Preserve the active isolated cron session identity before the outer timer can fire, emit or synthesize a terminal lifecycle error with timeout metadata through the existing gateway lifecycle persistence path, and add a regression that proves the matching session row becomes terminal immediately when the outer wall timer wins.

## Risks / Open Questions

- A fix should be idempotent with the embedded runner's later lifecycle end/error event so timeout finalization does not double-finalize or regress normal completion.
- The cron run-specific key and the base agent session key can diverge; the fix needs to preserve the key that owns the row being finalized.
- Regression coverage should include a timeout after session start, not only queue-before-start timeout or cron job-state timeout.
