## Summary

Keep open. Current main still aborts and rejects the cron outer timeout as a plain error, then records cron/job task state, but I found no synchronous lifecycle.error emission or session-store finalization path tied to that timeout. The existing gateway persistence is ready to consume lifecycle events, so the remaining work is the missing cron timeout event/identity bridge described by the issue.

## Best Possible Solution

Keep this issue open and implement a narrow cron-owned terminal lifecycle bridge for outer wall-timeout aborts. The best fix is to preserve or expose the active isolated cron run identity as soon as the session is prepared, then emit a timeout-specific lifecycle error/end event with sessionKey, runId/sessionId, endedAt, errorKind/stopReason before the cron timeout result is returned. Add regression coverage that the matching sessions.json row becomes terminal in-band, without relying on later watchdog repair.

## Risks / Open Questions

- A fix should avoid emitting duplicate terminal lifecycle events if the embedded agent later observes the abort and emits its own lifecycle terminal event.
- The timeout wrapper currently has job identity, while the precise persisted runSessionKey is derived deeper in isolated cron setup; the implementation may need an early session identity handoff instead of reconstructing that key in timer.ts.
