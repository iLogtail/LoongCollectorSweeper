## Summary

Keep open. Current `main` still treats Control UI reconnect as a cleanup boundary for active WebChat runs: reconnect hello clears `chatRunId`/`chatStream`, and the browser WebSocket client rejects pending RPCs on socket close before reconnecting. Server-side `chat.send` tracking and recent changelog entries are useful partial hardening, but they do not implement the issue's requested behavior of preserving or reattaching an in-flight Control UI session across a reconnect.

## Best Possible Solution

Keep tracking this until Control UI and Gateway can preserve or recover an active WebChat run across reconnect. A good fix would avoid clearing `chatRunId`/`chatStream` until terminal state is known, reattach the reconnecting client to active server-side run state or replay a terminal state from transcript/run state, and add regression coverage for reconnect during an active `chat.send` run. If maintainers decide stream resumption is intentionally out of scope, document the expected behavior before closing.
