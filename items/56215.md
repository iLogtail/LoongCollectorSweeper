## Summary

Keep open. Current main has relevant WebSocket lifecycle cleanup, client stop teardown, and shutdown hardening, but I did not find source, tests, release notes, or related-item evidence proving the reported long-running Windows/Node 24 CLOSE_WAIT/FIN_WAIT_2 accumulation and crashes are fixed. The issue is recent, and a follow-up comment says CLOSE_WAIT still appears on a latest build, so conservative cleanup should not close it.

## Best Possible Solution

Keep #56215 open until a targeted Windows/Node 24 investigation either reproduces and fixes the socket accumulation or proves it no longer occurs on current main. A good maintainer path is to add diagnostics around Gateway WebSocket close/error/terminate transitions, reproduce with repeated CLI/UI/channel clients while collecting netstat and gateway ws logs, and add a regression test or documented manual verification for long-running client-initiated close and keepalive-failure cleanup.

This is relevant lifecycle cleanup, but it does not prove Windows CLOSE_WAIT/FIN_WAIT_2 accumulation under steady-state use is gone.
