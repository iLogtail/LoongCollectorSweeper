## Summary

Keep open. The report is fresh and underspecified, but it maps to real core-owned session lifecycle behavior: current main intentionally creates a new session after the default 4:00 AM local daily reset boundary, which matches the reported 11 PM to 5 AM gap. That may be expected behavior, but it is not enough evidence to auto-close as fixed, external, duplicate, stale, or unreproducible.

## Best Possible Solution

Keep the issue open for maintainer triage. Ask the reporter for `session.reset`, `session.resetByChannel`, `session.dmScope`, WeChat plugin version, `/status` before/after the overnight gap, the affected session key, redacted `sessions.json` timestamps (`sessionStartedAt`, `lastInteractionAt`, `updatedAt`), and gateway logs around 4:00-5:00 AM. If the default daily reset explains it, the best fix is likely a clearer first-turn reset notice or docs/config guidance; if not, add a focused regression test around channel direct-session reuse or WeChat session-key stability.
