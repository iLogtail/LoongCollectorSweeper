## Summary

Keep open. The report is fresh and underspecified, but it maps to core-owned session lifecycle behavior: current main defaults to a new session after the 4:00 AM local daily reset boundary, which matches the reported 11 PM to 5 AM WeChat gap. That may be intended behavior, but it is not enough evidence to auto-close as fixed, duplicate, external, stale, or unreproducible.

## Best Possible Solution

Keep the issue open for maintainer triage. Ask the reporter for session.reset, session.resetByChannel, session.dmScope, WeChat plugin version, /status before and after the overnight gap, the affected session key, redacted sessions.json timestamps, and gateway logs around 4:00-5:00 AM. If the default daily reset explains it, the best product path is clearer first-turn reset notice or docs/config guidance; if not, add a focused regression test around WeChat direct-session reuse or session-key stability.
