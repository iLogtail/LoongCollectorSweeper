## Summary

Keep open. Current `main` does not implement the Discord MIME-preservation fix, and the PR’s WebP change still needs maintainer judgment because it modifies the shared `loadWebMedia` path used outside Discord.

## Evidence

- **Discord send path still drops MIME type on main:** `toDiscordFileBlob` still accepts only `Blob | Uint8Array` and returns `new Blob([arrayBuffer])` with no `type`; `sendDiscordMedia` still calls it as `toDiscordFileBlob(media.buffer)`.
  - file: [extensions/discord/src/send.shared.ts:340](https://github.com/openclaw/openclaw/blob/bae7b54a85b17d180e4bfc28973a7e4fd1646a93/extensions/discord/src/send.shared.ts#L340)
  - command: `nl -ba extensions/discord/src/send.shared.ts | sed -n '336,442p'`
  - sha: [bae7b54a85b1](https://github.com/openclaw/openclaw/commit/bae7b54a85b17d180e4bfc28973a7e4fd1646a93)
- **Other Discord upload paths also omit content type:** Component sends still call `toDiscordFileBlob(media.buffer)`, and interaction replies still build `new Blob([arrayBuffer])` from loaded media without threading `loaded.contentType`.
  - file: [extensions/discord/src/send.components.ts:197](https://github.com/openclaw/openclaw/blob/bae7b54a85b17d180e4bfc28973a7e4fd1646a93/extensions/discord/src/send.components.ts#L197)

- **Shared media loader on main does not have the WebP bypass:** `clampAndFinalize` special-cases only GIF (`params.contentType === "image/gif"`); all other images continue through `optimizeAndClampImage`. That means the PR is not already implemented on `main`.
