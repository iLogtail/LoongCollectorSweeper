## Summary

Keep PR #39323 open. Current main still lacks proactive owner notifications for fresh DM pairing requests, lacks a `pairing:request` internal/plugin hook, and has no `pairing.notify` config surface. Existing notification paths cover requester replies, approval confirmations, or device pairing, not owner alerts when an unknown DM sender first creates a pending pairing request.

## Best Possible Solution

Keep PR #39323 open as the active implementation candidate, or replace it with a maintainer-owned narrower design. The best product path is a documented, versioned `pairing:request` event or plugin hook emitted only for freshly created DM pairing requests, followed by owner notification through a plugin or minimal core config with fixed-format sanitized text, PII-safe logs, target normalization, and focused tests.
