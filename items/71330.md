## Summary

Keep #71330 open. Current main still hard-codes short-term/deep promotion output to workspace `MEMORY.md`, exposes no `promotionTarget` or equivalent memory-core config key, and documents/tests `MEMORY.md` as the durable promotion destination. The linked #59095/#64097/#63829 items are adjacent memory lifecycle/vault discussions, but none is a canonical tracker for configurable memory-core promotion output paths.

## Best Possible Solution

Keep the issue open and implement a memory-core-owned configurable promotion target, likely under `plugins.entries["memory-core"].config.dreaming` or its deep-promotion subsection. The implementation should default to `MEMORY.md`, share the same resolver between managed dreaming and manual `openclaw memory promote --apply`, validate/expand safe workspace-relative date templates, create missing target files with an appropriate header/frontmatter, update docs/schema/help, and add tests proving target files remain searchable for both builtin and QMD memory backends.
