## Summary

Current main still does not fully fix #69187. The memory-core dreaming narrative path still uses timestamped `dreaming-narrative-*` session keys, uses the same key as the `idempotencyKey`, and relies on `params.subagent.deleteSession()` for cleanup. The gateway runtime still dispatches that cleanup as plain `sessions.delete`, which remains `operator.admin`-scoped, and the existing scrubber only removes missing-file stale entries or unreferenced orphan transcripts, not valid leaked store entries with transcripts. The linked #70464 PR remains the canonical open work for this exact bounded cleanup/scope issue.

## Best Possible Solution

Keep #69187 open and land or replace #70464 with a tested fix: bounded/stable memory-core dreaming session keys or separate idempotency, a least-privilege lifecycle-independent way for memory-core to delete only its own `dreaming-narrative-*` transient sessions, and regression coverage proving cron/fallback cleanup removes both the referenced session-store entry and transcript without granting broad plugin admin rights.
