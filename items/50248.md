## Summary

Keep open. Current main has partially fixed the stale-writer side for new cron/session rollovers, but `sessions cleanup --fix-missing` still prunes from a single persisted `sessionFile` path and does not fall back to the canonical `<sessionId>.jsonl` transcript that normal readers already prefer for stale generated paths. The linked PR #69954 is still open in the provided context and tracks the missing cleanup repair/fallback, so this issue should remain open until that behavior lands.

## Best Possible Solution

Keep the issue open and land a core cleanup fix, likely via #69954 or an equivalent patch. `sessions cleanup --fix-missing` should use the safe transcript-candidate contract used by readers, keep entries when any current candidate exists, repair stale generated `sessionFile` metadata to the canonical path when appropriate, and only prune when no acceptable transcript exists. Add focused regression coverage for stale generated `sessionFile` plus custom/fork/topic transcript cases, and update the doctor missing-transcript check or track that follow-up explicitly.

## Risks / Open Questions

- Existing stores with stale but structurally safe `sessionFile` values can still be falsely pruned by `openclaw sessions cleanup --enforce --fix-missing` if the stale path is missing and the canonical `<sessionId>.jsonl` exists.
- `openclaw doctor` still uses the same single-path `resolveSessionFilePath` check for recent missing transcript warnings and can continue steering users toward unsafe cleanup until the same candidate-resolution logic is applied there or tracked separately.
