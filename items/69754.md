## Summary

Keep #69754 open. Current main still does not provide Feishu-specific built-in approval fallback clarity: Feishu has same-chat approval authorization and interactive card infrastructure, but built-in exec/plugin approval replies are still rendered through generic text/media delivery rather than Feishu approval buttons or explicit degraded-state copy.

## Best Possible Solution

Keep open. The best fix is in the bundled Feishu channel: either add a Feishu approval render/native adapter that maps shared approval actions to Feishu interactive cards and callbacks, or explicitly mark/render degraded Feishu approval prompts with clear text explaining why buttons are unavailable and whether `/approve` works in-channel versus Web UI/terminal. Add Feishu-focused tests for approval payload rendering/fallback and document the behavior in `docs/channels/feishu.md`; coordinate with #58298 only for reusable card/message-tool infrastructure.

## Risks / Open Questions

- A wording-only fix could still leave `channels.feishu.capabilities = ["inlineButtons"]` misleading unless Feishu either renders shared approval buttons or reports that this approval surface is text-only.
- Feishu already has quick-action/card callback code, so implementation should avoid conflating those metadata action cards with the built-in exec/plugin approval lifecycle.
