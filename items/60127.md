## Summary

Keep open. Current main has partial adjacent capabilities such as multi-agent routing, per-agent auth/session state, per-agent skill/tool restrictions, and coarse gateway operator scopes, but it explicitly does not provide the secure tenant-isolated multi-tenancy requested here. The KinthAI comment is useful supporting context: it describes real external demand and names concrete missing core primitives such as tenant_id propagation, per-tenant quotas, tenant-scoped plugin enablement, and tenant-attributed audit logs.

## Best Possible Solution

Keep this issue open for maintainer product/security triage. A real implementation would need a design first: tenant identity propagated through gateway auth and request context, tenant-scoped state/config/plugins/skills/model auth/quotas/audit logs, authorization tests for cross-tenant denial, and updated public docs clarifying how this differs from today’s multi-agent routing. If maintainers decide this is out of scope, the clean path is an explicit maintainer decision and docs wording that points users to separate gateways or managed external services such as KinthAI rather than treating the current multi-agent system as strict multi-tenancy.
