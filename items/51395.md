## Summary

Keep open. Issue #51395 asks for a generic browser upload fallback for non-standard upload widgets such as ByteUploader. Current main still routes the browser `upload` action through `/hooks/file-chooser`, then either Playwright file chooser/input-file APIs or Chrome MCP `upload_file`. I found no DataTransfer/drop, clipboard/paste, native drag-event, ByteUploader, or ByteDance-specific fallback in the browser upload implementation or docs. Related #38844 is nearby but narrower: it tracks flaky file-chooser/stale-click behavior, not custom uploader fallback support.

## Best Possible Solution

Keep #51395 open. The likely maintainer path is a generic browser-plugin fallback design in `extensions/browser`: preserve current upload-root path validation, add an explicit fallback sequence for non-standard upload targets such as DataTransfer/drop, paste, or CDP native drag where technically viable, add focused route/tool tests, and document when agents should use the fallback versus ordinary `ref`/`inputRef` file-input upload. If maintainers want ByteDance-only behavior, it should be split into a narrower plugin/skill issue only after confirming the core browser upload API exposes enough primitives.
