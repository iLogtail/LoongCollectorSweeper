## Summary

Keep #34252 open. Current main still has only partial Ollama multi-provider support: the Ollama hook can read a secondary provider's baseUrl when invoked directly, but runtime activation, native `api: "ollama"` streaming, synthetic auth, docs, and memory-search routing still center on the canonical `ollama` provider id. The issue remains the canonical tracker for multi-host/multi-instance Ollama support.

## Best Possible Solution

Keep #34252 open as the canonical tracker until OpenClaw implements or explicitly documents supported multi-Ollama instances. The likely fix should add provider-family ownership for configured provider ids with `api: "ollama"` or another explicit activation contract, cover synthetic/local auth and runtime stream hook selection for a secondary id such as `ollama-jetson`, and update docs/tests for model selection plus memory-search or heartbeat routing.
