## Summary

Keep #65494 open. The report is concrete and recent, and the legacy `imsg` iMessage path is still documented as a supported channel even though BlueBubbles is recommended for new setups. Current main has a related iMessage send/config-threading cleanup, but I did not find source, test, changelog, or release evidence proving the reported `ReferenceError: accountInfo is not defined` in final reply delivery has been fixed; the final-reply path still depends on `accountInfo.accountId` inside the dispatcher delivery callback.

## Best Possible Solution

Keep the issue open and add a focused regression around `monitorIMessageProvider` final reply delivery with mocked RPC, mocked dispatch, and mocked send delivery so the callback actually executes after dispatch. If the ReferenceError reproduces in the built bundle, patch the iMessage monitor/delivery boundary and then validate with the legacy `imsg` channel on macOS; BlueBubbles can remain the recommended new setup but should not replace fixing a concrete supported legacy-channel bug.
