## Summary

Keep open. Current main still lacks the HTTP internal-session guard: after auth, the REST/SSE session-history handler resolves the requested key, loads the session store, and returns history with no check for subagent, cron, or ACP namespaces. Existing tests cover ordinary history, SSE, pagination, and scope enforcement, but not rejection of internal namespace keys. The PR discussion also resolved a real WebSocket over-blocking concern by narrowing back to HTTP-only, which matches current internal subagent-control usage of `chat.history`.

## Best Possible Solution

Keep the PR open for maintainer review and either land a narrow HTTP/SSE guard or implement the same behavior directly on main: add a shared internal-session-key predicate covering raw and agent-scoped subagent/cron/ACP keys, reject those keys in `src/gateway/sessions-history-http.ts` after auth/scope checks and before store I/O, and keep WebSocket `chat.history` available to internal runtime flows unless a separate caller-boundary design is added. Include regression tests for canonical and raw internal keys.
