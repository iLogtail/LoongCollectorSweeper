## Summary

Keep open. Current `main` and the latest release still serve HTTP session history without blocking internal subagent/cron/ACP session namespaces, so this PR is not implemented on `main` and should not be conservatively closed.

## Evidence

There is no check rejecting internal session namespaces before the store read/response path.

## Risks / Open Questions

- Closing this PR would leave the current HTTP endpoint able to expose internal subagent/cron/ACP transcript history on `main`.
- The alias canonicalization path means the gap is not limited to fully qualified `agent:*` keys; bare keys like `cron:*` can still resolve to internal sessions.
- Because internal runtime code still uses WebSocket `chat.history` for subagent flows, the remaining useful work here is the HTTP-only restriction, not a blanket close-out.
