## Summary

Keep open. Current main still has the reported product gap: inbound/channel-triggered runs default into the process-wide `main` lane, the gateway only configures fixed `cron`, `main`, and `subagent` lane concurrency, and there is no config/schema/docs surface for a dedicated browser lane or per-channel lane routing. The existing browser tab cleanup work helps the related browser resource leak but does not address scheduler starvation.

## Best Possible Solution

Keep the issue open. The best path is a core scheduling/config design: add a validated custom-lane concurrency surface, apply it on gateway startup and reload, route inbound runs by channel/account/session policy before `runEmbeddedPiAgent` enters the global lane, and document/test the interaction with existing session serialization, `agents.defaults.maxConcurrent`, cron/subagent lanes, and browser tab cleanup. Auto-detecting browser-tool use may be a follow-up, but it cannot replace pre-run lane assignment for preventing main-lane starvation.
