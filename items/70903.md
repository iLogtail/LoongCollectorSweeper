## Summary

Keep open. Current main has useful partial fixes: active cooldown/disabled windows no longer extend on retry, billing-disabled profiles can be probed when there are no fallback candidates, successful runs clear usage state, and `models auth login` clears stale lockouts. But the reported topology includes same-provider fallback candidates sharing one auth profile; current fallback logic treats that as “fallbacks exist” and skips billing-disabled profiles when the cooldown is far from expiry. The added `openclaw configure --section model` re-auth scenario also still appears uncovered because the configure/auth-choice path does not call the lockout-clearing helper used by `models auth login`.

## Best Possible Solution

Keep the issue open and narrow the fix to the remaining gaps: treat same-provider-only fallback chains that share disabled auth profiles like single-provider billing recovery so they get a throttled probe before hours-long expiry, and move the stale-lockout clearing helper into the shared provider auth-choice/configure path so successful re-auth clears the affected provider’s `auth-state.json` usage entry. Add regression coverage for an Anthropic single-profile primary+same-provider fallback billing disable and for a GitHub Copilot/configure re-auth clearing `cooldownReason: "format"`.
