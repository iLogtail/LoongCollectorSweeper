## Summary

Keep open. Current main has partial mitigations: repeated failures no longer extend an active cooldown/disabled window, single-provider billing can probe on a short throttle, and `openclaw models auth login` clears stale lockouts before re-auth. But the central reported cases are not fully solved: billing disables are still persisted with a 5h default/24h cap, same-provider fallback chains still skip billing-disabled candidates until near expiry, and the `openclaw configure --section model` plugin auth path still upserts refreshed credentials without clearing the provider's persisted `auth-state.json` cooldown entry.

## Best Possible Solution

Keep the issue open and fix it in the auth-profile/fallback boundary. The strongest path is to clear provider/profile usage state after successful auth setup through both `models auth login` and configure/onboarding plugin auth choices, plus add a documented user-facing reset or health-probe path for billing-disabled profiles. Add regression coverage for same-provider fallback on `billing`, configure re-auth clearing a `format` cooldown, and the status/UX message that shows the remaining disabled window.
