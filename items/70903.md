## Summary

Keep open. Current `main` partially addresses billing recovery by probing billing-disabled providers in true no-fallback setups, but the reported same-provider fallback chain is still blocked because same-provider model fallbacks count as fallback candidates and suppress the immediate billing probe path.

## Evidence

- **Docs and config still describe persisted long billing disables:** The current docs say billing failures are stored in `auth-state.json` as `disabledUntil`/`disabledReason`, with a default backoff starting at 5 hours and capped at 24 hours.
  - file: [docs/concepts/model-failover.md:178](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/docs/concepts/model-failover.md#L178)

  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Same-provider fallbacks are always included:** `resolveFallbackCandidates()` explicitly keeps the full configured fallback chain when the current model is on the same provider, so model-version fallbacks like `anthropic/opus-*` are still treated as fallback candidates.
  - file: [src/agents/model-fallback.ts:492](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/agents/model-fallback.ts#L492)

## Risks / Open Questions

- The new billing recovery path may give a false sense of closure because it helps only when there are literally no fallback candidates, not when all fallbacks stay on the same provider/auth profile.
- There is still no proof on `main` that a same-provider fallback chain clears a persisted billing disable promptly after credits are restored.
