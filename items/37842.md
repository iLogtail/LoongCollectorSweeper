## Summary

Keep #37842 open. Current main at b67d9bf7f06f4c655b1e0869395056c7d631abc2 still has source-only InputProvenance for sessions_send and the A2A flow remains bounded only as a requester-target ping-pong, not as a graph-aware multi-agent chain. The linked PR #37945 is useful prior work but was closed unmerged, and related #51832 tracks a different Slack gateway error-cascade path rather than this sessions_send provenance-chain guard.

## Best Possible Solution

Keep #37842 open for a core sessions_send fix. The likely product/code path is to add a backwards-compatible chain or trace field to InputProvenance and the gateway schema, propagate it through createSessionsSendTool, nested agent runs, and A2A follow-up steps, block or return a clear error when the target agent/session already appears in the chain, and add regression coverage for A->B->C->A while preserving existing direct maxPingPongTurns behavior. Do not close as duplicate of #51832 because that issue covers gateway-posted Slack error cascades, not agent-initiated sessions_send chains.
