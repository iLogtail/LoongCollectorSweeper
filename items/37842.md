## Summary

Keep #37842 open. Current main still lacks graph-aware sessions_send cycle detection: the proposed PR #37945 was closed unmerged, provenance remains source-only, the gateway schema has no chain field, and the existing A2A limiter still bounds only the direct requester-target ping-pong flow.

## Best Possible Solution

Keep #37842 open for a core sessions_send fix. Add an explicit chain or trace field to InputProvenance and the gateway schema, propagate it through createOpenClawTools/createSessionsSendTool, runAgentStep, and A2A follow-up runs, reject or warn when the target agent/session already appears in the chain, and add regression coverage for A->B->C->A while preserving existing maxPingPongTurns behavior for direct A<->B exchanges.

## Risks / Open Questions

- Multi-agent deployments with tools.agentToAgent enabled can still form triangular or polygonal sessions_send cascades because current main has no visited-agent guard.
- A future implementation needs an additive protocol/schema-backed provenance design; the current gateway schema uses additionalProperties: false for InputProvenance.
- Related #51832 covers a Slack gateway error-message cascade, not active sessions_send graph loops, so it does not supersede this issue.
