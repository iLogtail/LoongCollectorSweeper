## Summary

Keep open. Current main partially addresses the narrowed request by adding per-channel queue debounce via `messages.queue.debounceMsByChannel`, while `messages.queue.byChannel` remains mode-only and queue `cap`/`drop` still resolve only from inline, session, or global config. The user-visible gap from the issue comment, per-channel cap/drop queue options, is still real.

## Best Possible Solution

Keep the issue open and narrow it to the remaining core config/runtime work: add validated per-channel queue cap/drop overrides, decide whether to extend `messages.queue.byChannel` to accept objects or add `capByChannel`/`dropByChannel`, then update queue resolution, schema/help/docs, and focused tests for fallback order and backward compatibility. The already-shipped `debounceMsByChannel` support should be preserved.
