## Summary

Keep #54831 open. Current main still lacks config-time validation for `hooks.enabled=true` without `hooks.token`, and the startup path still throws the old terse error. The PR addresses a real remaining product gap, but it should be revised before merge because its suggested remediation command is not a valid top-level CLI command and the new validation path needs regression tests.

## Best Possible Solution

Keep the PR open and ask for a small revision: add the cross-field check in `validateConfigObjectWithPluginsBase` so merged `config.patch` payloads reject `hooks.enabled=true` without a non-empty `hooks.token`, add focused tests for reject/accept cases, and replace the remediation text with the supported `openclaw gateway call config.patch --params ...` flow or a valid `openclaw config set hooks.token ...` command.
