## Summary

Keep open. Current `main` still lacks config-time enforcement for `hooks.enabled=true` without `hooks.token`, but this PR is not ready to close out conservatively because its remediation text points users to a nonexistent top-level `openclaw config.patch` CLI command and it adds no regression coverage for the new cross-field rule.

## Evidence

- **Current runtime still fails late with bare hooks token error:** `resolveHooksConfig` on `main` still throws only `"hooks.enabled requires hooks.token"` when hooks are enabled without a token, so the startup-time failure described in the PR remains present.
  - file: [src/gateway/hooks.ts:49](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/gateway/hooks.ts#L49)

- **`config.patch` validates the merged config before write:** The gateway `config.patch` handler merge-patches the current snapshot, then runs `validateConfigObjectWithPlugins(restoredMerge.result)`. That means the right fix surface is config validation, not only the startup hook resolver.
  - file: [src/gateway/server-methods/config.ts:516](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/gateway/server-methods/config.ts#L516)

## Risks / Open Questions

- Merging the PR as written would replace one unclear failure with a misleading recovery command: users would be told to run nonexistent `openclaw config.patch` instead of `openclaw gateway call config.patch` or `openclaw config set`.
- Without regression tests for the new cross-field rule, future validation refactors could silently drop the config-time protection and reintroduce the startup-only failure.
