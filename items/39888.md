## Summary

Keep open. Current main still has the observability gap described by PR #39888: the reply-runner path emits `model.usage`, but the CLI agent command path updates session state and persists CLI transcripts without emitting that diagnostic event. The PR is stale in file location and context-token logic, but the underlying requested behavior remains useful and unimplemented.

## Best Possible Solution

Keep the PR open or ask for a refresh. The best implementation is a small update in `src/agents/agent-command.ts` that emits a `model.usage` diagnostic event for CLI agent runs after the session-store update, using the same usage/cost fields as reply-runner and the current resolved context-token value from runtime/session-store logic. Add a focused test that runs the CLI agent command path with diagnostics enabled and verifies a `model.usage` event with `channel: "cli"` or another maintainer-approved CLI channel marker.
