## Summary

Keep open. Current main still only supports `userTimezone` as a global `agents.defaults` setting; it does not accept or resolve `agents.list[].userTimezone`, and the runtime/docs paths that PR #39610 targets remain global-only. The linked feature issue #39579 was the request this PR attempts to implement, not a superseding canonical item, and the PR discussion shows concrete follow-up work rather than a reason for conservative cleanup closure.

## Best Possible Solution

Keep PR #39610 open as the active implementation candidate. The best path is to rebase or recreate the change against current main, add `agents.list[].userTimezone` to the typed and zod config contract with validation, centralize agent-aware timezone resolution, thread agent context through prompt, session status, gateway timestamp, envelope, system-event, heartbeat, cron/current-time, and memory/post-compaction paths, then update docs/generated schema and targeted tests.
