## Summary

Keep #39610 open. Current main at 4a3030df9efa72c44bab567a4390ac0c4876a73f still has only global `agents.defaults.userTimezone`; it does not define, validate, document, or resolve `agents.list[].userTimezone`. The provided PR context shows #39610 is the canonical unmerged implementation for closed issue #39579, not obsolete cleanup.

## Best Possible Solution

Keep #39610 open as the canonical implementation path for per-agent `userTimezone`. The maintainer-ready solution is to rebase it onto current main, add `agents.list[].userTimezone` to config types, strict runtime schema, generated schema/help, and docs; centralize resolution as per-agent override -> `agents.defaults.userTimezone` -> host timezone; thread the active agent id through system prompt params, `session_status`/current-time helpers, heartbeat active-hours and heartbeat prompt injection, envelope/system-event timestamps, gateway/chat timestamp injection, cron, memory flush, post-compaction context, and plugin inbound-envelope surfaces; validate IANA timezone strings consistently; then run the appropriate changed gates.
