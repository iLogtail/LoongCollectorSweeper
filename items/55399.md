## Summary

Current `main` clearly sanitizes the exact `Unexpected event order, got message_start before receiving "message_stop"` error into a generic retry message, and that behavior shipped in `v2026.4.22`, but I did not find strong evidence that the underlying stuck-streaming condition itself was fixed. The linked work appears to address error surfacing rather than the root cause, so this should stay open conservatively.
