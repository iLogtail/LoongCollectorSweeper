## Summary

Keep #70968 open. Current main does not contain the PR’s behavior: the gateway install plan still includes literal managed env values alongside OPENCLAW_SERVICE_MANAGED_ENV_KEYS, and runtime dotenv loading still skips already-present process env keys. The PR remains a focused, relevant fix for #70612, but the existing review concern about preserving service-owned vars like PATH should be resolved before merge.

## Best Possible Solution

Keep the PR open and revise it rather than closing it. The best fix belongs in the gateway install-plan boundary: stop serializing literal values for managed .env/config/auth-profile keys into macOS LaunchAgent environments while preserving service-owned/generated env entries such as PATH, HOME, TMPDIR, OPENCLAW_PORT, and service metadata. Add focused regression coverage for config vars, auth-profile env refs, state-dir .env values, and the PATH/system-var edge case flagged in review.

## Risks / Open Questions

- The current PR implementation described in the review thread deletes any final environment key listed in OPENCLAW_SERVICE_MANAGED_ENV_KEYS. That should be adjusted or explicitly guarded so curated service values such as PATH, HOME, TMPDIR, and generated OPENCLAW_* service metadata are not removed accidentally.
- Current main’s auto-refresh handling for embedded OPENCLAW_GATEWAY_TOKEN is narrower than this PR. It does not solve arbitrary managed .env/auth values such as DISCORD_TOKEN, TAVILY_API_KEY, OPENROUTER_API_KEY, or provider API keys being serialized into the LaunchAgent plist.
