## Summary

Keep open. Current main still defaults Discord streaming/block delivery off unless explicitly configured, and the current non-streaming block reply path intentionally buffers text-only blocks into the final reply while tool/progress messages can be dispatched earlier. That matches the reported Discord ordering problem rather than proving it fixed. The reporter’s workaround is valid, but it is not the current default behavior.

## Best Possible Solution

Keep the issue open for maintainer/product handling. The best path is to add a Discord-specific ordering fix or explicit default policy: either enable Discord block delivery with conservative coalescing, or flush/order pre-tool text segments before tool/progress messages even when block streaming is otherwise disabled. Add a regression that covers `text1 -> tool -> text2 -> tool -> final` delivery on Discord and checks the #57875 fragmentation concern.
