## Summary

Keep PR #70288 open. Current main still has the exact plugin-bound `handled` branch returning `queuedFinal: false`, and current tests still assert that old value. Telegram still treats `queuedFinal: false` plus no delivered reply as the condition for its silent-reply fallback, so the PR’s one-line behavior change remains relevant rather than obsolete or superseded.

## Best Possible Solution

Keep the PR open and land this behavior change, or an equivalent maintainer-authored patch, after normal validation. The best fix is narrow: return `queuedFinal: true` only for the plugin-bound `handled` outcome and update the existing plugin-owned binding tests, while leaving `declined`, `missing_plugin`, `no_handler`, and `error` outcomes as `queuedFinal: false`. A follow-up Telegram regression test would further lock the original #70286 symptom if maintainers want end-to-end coverage.
