## Summary

Keep PR #70288 open. Current main still has the reported `plugin-bound-handled` path returning `queuedFinal: false`, and Telegram still uses `!queuedFinal` to trigger the silent-reply fallback. The PR is relevant, but its current one-line core return may need additional coverage or adjustment because the channel path goes through the buffered dispatch wrapper, whose finalizer can collapse `queuedFinal: true` back to false when no final reply was queued.

## Best Possible Solution

Keep the PR open and ask for maintainer review or an update that proves the full `dispatchReplyWithBufferedBlockDispatcher` -> Telegram path suppresses the silent-reply fallback for plugin-bound `{ handled: true }` with no immediate reply. The likely durable fix is to represent this as a committed async response through the full dispatch result contract, with regression coverage for the linked `openclaw-codex-app-server` style case, then close #70286 when landed.
