## Summary

Keep open. Current main has not implemented this PR: the plugin-bound `handled` branch still returns `queuedFinal: false`, and current regression tests still assert that behavior for plugin-owned bindings. Telegram still uses `!queuedFinal && !deliverySummary.delivered` to send the silent-reply fallback, so the PR addresses a live, plausible code path rather than obsolete work.

## Best Possible Solution

Keep PR #70288 open and merge or adapt it after review: change only the `plugin-bound-handled` dispatch result to `queuedFinal: true`, update the existing plugin-bound regression expectations, and ideally add/confirm Telegram coverage that a plugin-claimed async turn does not trigger silent-reply fallback. The linked issue #70286 remains the canonical bug until that lands.
