## Summary

Keep #71638 open. Current main still does not surface Telegram forward, via-bot, or reply provenance on `MessageReceivedHookEvent.context.metadata`. The hook metadata escape hatch exists, and Telegram already computes forward/reply context for agent prompt/session payloads, but that data is not threaded through the canonical received-hook mapper or Telegram received-hook emit paths. No current source/test/doc evidence shows `via_bot` or `viaBot` support in the hook payload.

## Best Possible Solution

Keep the issue open and implement a Telegram-owned additive fix. Add a typed provider-metadata path through the canonical received-hook context or an equivalent narrow mapper seam, populate `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo` from inbound Telegram `msg` in both the normal dispatch path and mention-skip ingest path, reuse existing `normalizeForwardedContext` / `describeReplyTarget` where appropriate, add explicit via-bot normalization, and cover the mapper plus Telegram emit paths with focused tests.
