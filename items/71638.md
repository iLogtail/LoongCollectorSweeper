## Summary

Keep #71638 open. Current main still has the generic message hook metadata escape hatch, and Telegram already derives forward/reply provenance for agent prompt context, but that provenance is not threaded into `MessageReceivedHookEvent.context.metadata`; `via_bot`/`viaBot` is not implemented anywhere in the searched Telegram hook path. The prior Codex review comment reached the same conclusion, and the current main SHA does not appear to have changed this surface.

## Best Possible Solution

Keep the issue open and implement a Telegram-owned additive fix: derive bounded `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo` from inbound Telegram messages, thread it through the canonical received-hook mapper used by both plugin and internal `message_received` events, reuse existing forward/reply normalization and visibility logic, and add focused tests for the mapper plus Telegram emit paths.

## Risks / Open Questions

- Adding reply/forward metadata must preserve Telegram group supplemental-context visibility rules so hidden or allowlist-only provenance is not leaked through hooks.
- The requested `replyTo.text` should be bounded/truncated before entering hook metadata to avoid oversized event payloads.
- `via_bot` needs fresh normalization because current Telegram code does not appear to read it at all.
