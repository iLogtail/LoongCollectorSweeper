## Summary

Keep #71638 open. Current main still does not expose Telegram forward, via-bot, or reply provenance on `MessageReceivedHookEvent.context.metadata`. The generic metadata escape hatch exists, and Telegram already normalizes reply/forward context for agent prompt construction, but the canonical hook context and received-event mappers only carry generic routing/sender metadata. A targeted search found no `via_bot`/`viaBot` implementation in the Telegram hook path.

## Best Possible Solution

Keep the issue open and implement a Telegram-owned additive fix: derive bounded `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo` from inbound Telegram `msg`, thread those fields through the canonical received-hook mapper path, and add focused tests for both `toInternalMessageReceivedContext` and the Telegram received-hook emit path. Reuse existing reply/forward normalization where possible, and add explicit via-bot normalization because current main has no `via_bot` handling.
