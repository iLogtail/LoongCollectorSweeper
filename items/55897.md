## Summary

Keep open. Current main still intentionally uses oldest-first round-robin auth profile ordering and explicitly does not prioritize the last successful profile, so the MRU-sticky part of #55897 remains unresolved. The 402 classification/failover path appears partially improved on main, but that does not close the whole issue because the primary `/new` key-selection behavior still matches the report.

## Best Possible Solution

Keep the issue open and narrow it to the remaining product decision: add a supported auth-profile selection policy for API-key profiles, such as MRU/lastGood sticky ordering or an explicit config option, in `src/agents/auth-profiles/order.ts`; update the model-failover docs and regression tests to cover `/new` selection, cooldown skipping, and final-only 402 surfacing.
