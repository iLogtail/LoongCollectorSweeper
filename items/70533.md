## Summary

Keep #70533 open. Current main has useful startup scoping and persisted plugin-registry helpers, but it still does not apply the plugin allowlist before manifest materialization: the runtime loader discovers candidates and builds a full manifest registry before `onlyPluginIds` filtering, and the gateway startup planner still rehydrates all indexed manifests in key paths. The issue's `tools.allow` wording appears imprecise; the current plugin-load gate is `plugins.allow`, but the central discovery-time filtering request remains actionable.

## Best Possible Solution

Keep the issue open and implement discovery-time narrowing. The startup/plugin planner should compute required plugin ids from `plugins.allow`, deny rules, explicit bundled-channel config exceptions, memory slots, configured channels, provider/harness/setup triggers, and then pass that scope before manifest parsing/validation. Reuse the persisted installed-plugin index where possible, pass `pluginIds` into installed-index manifest registry calls when a scope is known, and add regression coverage proving skipped plugin manifests are not read, not only that skipped runtime modules are not imported.
