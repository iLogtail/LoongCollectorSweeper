## Summary

Keep #50442 open. Current main has improved atomic archive publication and exception-path cleanup for `openclaw backup create`, and that likely resolves the related stalled/failed tar-write cases such as #41830. It still does not fully address this report's timeout-kill failure mode: the backup temp archive is written as `<output>.<uuid>.tmp`, cleanup only runs from an in-process `finally`, and the backup command/register path has no SIGTERM/SIGINT or stale-temp cleanup path. The issue discussion's cron/workaround remains relevant rather than obsolete.

## Best Possible Solution

Keep the issue open and implement a backup temp-file lifecycle fix in `src/infra/backup-create.ts`/`src/commands/backup.ts`: retain the existing atomic publish behavior, add graceful SIGINT/SIGTERM cleanup for active backup temp archives, and add conservative stale cleanup for OpenClaw-owned backup `.tmp` files from previous interrupted runs. Document that SIGKILL/power loss cannot be caught, and cover tar rejection, SIGTERM-style cancellation, and stale-temp cleanup with focused regression tests.
