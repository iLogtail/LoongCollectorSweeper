## Summary

Keep open. Current main partially addresses parts of the report, especially exec-event prompt content/consumption, scheduler retry/reload behavior, and lightweight bootstrap filtering. But the broader requested heartbeat receipt/visibility boundary is still not implemented: `last-heartbeat` still returns the in-memory heartbeat singleton, and exec-event heartbeat runs still use the same heartbeat event surface. The related open issues #66382 and #68386 cover narrower exec-event paths, but they do not fully supersede the durable receipt / `system heartbeat last` separation requested here.

## Best Possible Solution

Keep #65161 open but narrow it to the remaining product/API work: define a durable heartbeat receipt surface for scheduled/manual heartbeat runs, make `openclaw system heartbeat last` read that surface instead of the generic in-memory event singleton, and explicitly separate exec-event wake handling from canonical scheduled heartbeat receipts. Link #66382 and #68386 as narrower exec-event follow-ups rather than using them to close this broader issue.
