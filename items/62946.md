## Summary

Current `main` already supports multiple Telegram bot accounts routed to different agents via `channels.telegram.accounts.<accountId>` plus `bindings[].match.accountId`. The reported use case is implemented, although the supported shape is multi-account under one `telegram` channel key rather than separate top-level keys like `telegram-2`.

## Evidence

- **Multi-agent docs show one Telegram bot per agent:** The multi-agent docs include a concrete Telegram example with `default` and `alerts` bot accounts, each bound to a different agent through `bindings[].match.accountId`.
  - file: [docs/concepts/multi-agent.md:325](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/docs/concepts/multi-agent.md#L325)
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Telegram docs document per-account config and default selection:** Telegram channel docs document `accounts.*` config, per-account overrides, and multi-account precedence with `channels.telegram.defaultAccount`, including an explicit note that `openclaw doctor` warns when default routing is ambiguous.
  - file: [docs/channels/telegram.md:909](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/docs/channels/telegram.md#L909)
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Telegram config schema exposes per-account bot settings:** `TelegramAccountConfig` includes per-account `botToken`, `dmPolicy`, `allowFrom`, `groupPolicy`, `defaultTo`, and related settings, matching the requested independent bot configuration surface.
  - file: [src/config/types.telegram.ts:94](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/config/types.telegram.ts#L94)
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Telegram account selection is built around multiple account IDs:** `listTelegramAccountIds` combines configured Telegram accounts and bound account IDs, and `resolveDefaultTelegramAccountSelection` chooses the default account while warning only when multi-account routing is ambiguous.
  - file: [extensions/telegram/src/account-selection.ts:114](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/extensions/telegram/src/account-selection.ts#L114)
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Gateway starts Telegram accounts independently:** The Telegram channel gateway starts each resolved account separately via `startAccount`, carrying that `accountId` through probe, status, and monitor startup.
  - file: [extensions/telegram/src/channel.ts:866](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/extensions/telegram/src/channel.ts#L866)
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Implementation is already in the latest release:** The release tag `v2026.4.23` already contains the same Telegram multi-account docs and account-selection code, so this is shipped behavior, not just unreleased `main` state.

## Risks / Open Questions

- The requested syntax (`channels.telegram-2` or `agentId` nested directly under each Telegram config) is not the supported API; OpenClaw solves the same use case with `channels.telegram.accounts.<id>` plus top-level `bindings`.
