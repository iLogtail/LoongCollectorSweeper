## Summary

Keep open. Current main already documents stability as a priority and has beta-first release gates, and the related #13238 quality-gate proposal was closed as implemented, but #67239 asks for broader maintainer policy: a feature freeze, maintenance-only/stabilization releases, longer beta/RC windows, stability metrics, safe-upgrade communication, and an LTS/stable branch model. I did not find those specific policies implemented or superseded on current main.

## Best Possible Solution

Keep this open for explicit maintainer release-policy judgment. If maintainers accept it, the likely path is to update docs/reference/RELEASING.md and docs/install/development-channels.md with any maintenance-only cadence, longer beta/RC window, safe-upgrade checklist, metrics, and LTS/stable-branch policy, plus adjust release workflows/templates only where those policies need enforcement. If maintainers do not want that model, it should be closed manually with a product/release-governance explanation rather than by automated cleanup.
