## Summary

Current `main` still does not implement #30828’s requested generic model throttling surface. OpenClaw has adjacent reactive handling for 429s, `Retry-After`, auth-profile cooldown/rotation, and fallback models, but there is still no documented provider/model configuration or request-path scheduler for manually configured tokens-per-interval and requests-per-interval budgets.

## Best Possible Solution

Keep #30828 open for a core model request-path design: add a documented per-provider/per-model, and possibly per-auth-profile, schema for request and token budgets with interval windows; enforce it before shared model calls; reconcile usage after responses and streams; optionally ingest provider rate-limit headers; and add focused tests/docs covering scheduler, fallback, retry, streaming, and auth-profile interactions.

## Risks / Open Questions

- A provider plugin or external proxy could implement private throttling for a narrow setup, but that would not satisfy the issue’s generic OpenClaw configuration request.
- A complete core design still needs product decisions for prompt/output token estimation, queued waiting behavior, streaming usage reconciliation, fallback/retry interactions, auth-profile rotation, and how provider `x-ratelimit-*` headers interact with manual budgets.
