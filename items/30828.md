## Summary

Keep #30828 open. Current main has reactive rate-limit handling through auth-profile cooldowns, profile rotation caps, Retry-After bypass, model fallback, and usage/status probes, but it still does not provide the requested manual per-model tokens-per-interval or requests-per-interval limiter. The OpenAI x-ratelimit header suggestion in the discussion is only observed/logged in a live header probe and is not used for request scheduling.

## Best Possible Solution

Keep the issue open for core provider/model request-path design. A complete fix should add a documented config schema for per-provider or per-model request and token budgets with interval windows, enforce it before model calls in the shared request/fallback path, update budgets from normalized usage, decide whether provider headers such as OpenAI x-ratelimit remaining/reset auto-adjust those windows, and add scheduler/failover tests plus docs.
