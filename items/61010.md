## Summary

Keep #61010 open. Current main has an opt-in `tools.fs.workspaceOnly` guard and sandboxing support, but the default behavior still allows filesystem tools to read/write/edit outside the workspace when sandboxing/workspace-only mode is not enabled. That leaves the issue’s central request, limiting model/tool file visibility to the workspace by default or with a default-on override, unresolved.

## Best Possible Solution

Keep this open for maintainer product/security judgment. The likely fix path is to make new or all local setups default to workspace-scoped filesystem access, or add a clearly default-on guard with an explicit opt-out, then update config docs, onboarding defaults, and tests that currently codify unrestricted legacy behavior.

## Risks / Open Questions

- Changing the default to workspace-only would be a behavior change for users and plugins that intentionally rely on absolute host paths, so it needs a migration/compatibility plan.
- Docs currently frame sandboxing and workspace-only filesystem policy as opt-in controls, so resolving this may require explicit product/security direction rather than a narrow bug fix.
