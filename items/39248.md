## Summary

Keep #39248 open. Current main has relevant sandbox/subagent plumbing, but I still do not see source, regression-test, release-note, or live evidence strong enough to conclude the reported `agents.defaults.sandbox.mode: "non-main"` child-run stall is fixed. The issue is concrete, recent, and only has a stale label, so it should remain actionable until a maintainer lands or verifies a targeted fix.

## Best Possible Solution

Keep the issue open and add a focused regression around native `sessions_spawn` with `agents.defaults.sandbox.mode: "non-main"`: spawn from the main agent, require the child subagent session to initialize in the sandbox, and assert a transcript/message or `systemSent`/token progress instead of only an accepted run id. If sandbox setup fails after acceptance, surface a clear lifecycle/error event and gateway log so users do not see a silent running-at-zero-tokens state.
