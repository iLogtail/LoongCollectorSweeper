## Summary

Keep #39248 open. The issue is concrete, author/labels are not protected, and current main still lacks code, test, docs, release, or live evidence proving `sessions_spawn` child sessions initialize and produce transcript/message/token state when `agents.defaults.sandbox.mode` is `"non-main"`. The current code recognizes the child as sandboxed and dispatches the run, but the accepted response still precedes actual child execution, matching the reported accepted-then-stalled shape.

## Best Possible Solution

Keep the issue open until maintainers add or identify a targeted regression proof for native `sessions_spawn` under `agents.defaults.sandbox.mode: "non-main"`. The best fix path is a focused integration or high-fidelity mocked test that exercises child session sandbox context creation and verifies the child run starts, writes transcript/message state, updates `systemSent`/token state, and surfaces sandbox initialization failures instead of silently timing out. If the repro still fails, patch the Docker-backed initialization/dispatch path implicated by `resolveSandboxContext`; if it now passes live, close only with that source/test/manual evidence.
