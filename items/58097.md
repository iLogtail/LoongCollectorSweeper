## Summary

Keep open. Current `main` still has the manual early-return and XOR-based Nextcloud Talk signature check that this PR is trying to replace, so the request is not implemented on `main` and is still actionable.

## Evidence

- **Current main still has the vulnerable comparison path:** `verifyNextcloudTalkSignature` returns early on length mismatch and then compares bytes with a manual XOR loop instead of `timingSafeEqual`.
  - file: [extensions/nextcloud-talk/src/signature.ts:28](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/nextcloud-talk/src/signature.ts#L28)
  - command: `nl -ba extensions/nextcloud-talk/src/signature.ts | sed -n '1,220p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Main lacks the PR's regression test:** Current tests cover valid and invalid signatures, but there is no focused test asserting the compare path still runs on a mismatched signature length.
  - file: [extensions/nextcloud-talk/src/core.test.ts:82](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/nextcloud-talk/src/core.test.ts#L82)
  - command: `nl -ba extensions/nextcloud-talk/src/core.test.ts | sed -n '1,160p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Repo already uses the safer pattern elsewhere:** `extensions/line/src/signature.ts` pads buffers and calls `crypto.timingSafeEqual` unconditionally, with a regression test for length mismatch, which matches the direction of this PR and shows the proposal fits existing codebase practice.
  - file: [extensions/line/src/signature.ts:12](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/line/src/signature.ts#L12)

## Risks / Open Questions
