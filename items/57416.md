## Summary

PR #57416 should stay open. Current main still has the reported product gap: workspace `.env` entries for `OPENCLAW_CONFIG_PATH` are blocked, and tests assert that behavior. The PR is focused on the right related bug (#57408), but it needs maintainer/security rework against current main because the blocklist now also has a broad `OPENCLAW_` prefix and CLI bootstrap uses `loadCliDotEnv`.

## Best Possible Solution

Keep the PR or the canonical bug #57408 open and rework the fix against current main. The best path is an explicit, documented workspace-dotenv allowlist exception for `OPENCLAW_CONFIG_PATH`, continued blocking of `OPENCLAW_STATE_DIR`, precomputed trusted global dotenv paths where needed, and regression coverage for `pnpm openclaw config file` / CLI bootstrap behavior plus the redirection-attack cases.

## Risks / Open Questions

- A direct rebase of PR #57416 is likely insufficient on current main because removing the explicit `OPENCLAW_CONFIG_PATH` entry would still leave the broad `OPENCLAW_` prefix blocking the variable.
- The final fix needs tests around both `loadDotEnv` and CLI startup `loadCliDotEnv`, preserving the invariant that workspace `.env` cannot redirect trusted global `.env` loading while allowing only the intended config-path override.
