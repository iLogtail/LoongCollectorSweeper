## Summary

Keep PR #61443 open. Current main still does not expose `models.providers.<id>.retry` in the model-provider type or strict Zod schema, and the OpenAI-compatible transports still call the provider SDK directly without a provider-level retry runner. Existing main only caps long SDK `Retry-After` sleeps and then relies on SDK retry/model failover, which does not satisfy the PR's configurable provider API retry goal. The PR is not obviously obsolete or superseded, but it still needs maintainer review and follow-up on the unresolved review concerns before it would be mergeable.

## Best Possible Solution

Keep this open and ask for a maintainer-quality revision rather than auto-closing it. The best path is a core implementation or reworked PR that adds the `models.providers.<id>.retry` config contract across types, strict schema, labels/help/generated docs, and transport tests; registers and looks up retry runners by canonical provider id; clears stale registry state across all config reload paths; handles `Retry-After` from both plain objects and `Headers`; and explicitly decides whether Anthropic/Google/Bedrock transports are in scope or covered by a follow-up.
