## Summary

Keep #68493 open. The issue is recent, has a detailed Windows config-edit restart failure report, and a maintainer explicitly noted after PR #71465 that the Windows stale-lock/EADDRINUSE mode is different and should remain open. Current main has partial restart mitigations, but it still contains a Windows Scheduled Task full-process restart handoff path and I did not find a direct Windows config-edit stale-lock/EADDRINUSE regression proving this specific report is fixed.

## Best Possible Solution

Keep the issue open for a targeted core gateway fix and regression test. The likely path is to reproduce Windows config-file restart under both unmanaged and Scheduled Task gateway modes, then make the restart handoff release/reacquire the gateway lock and port deterministically or prefer the in-process restart path where safe. Add a Windows-focused unit/integration test around config reload, lock cleanup, and EADDRINUSE recovery before closing.
