## Summary

Keep PR #38162 open. Current main still does not implement `sessionSaveRedirectPath`, and it also lacks the dependent `postHookActions`, `blockSessionSave`, and `sessionSaveContent` foundation. This is concrete core session-memory filesystem policy work with active security-review context, not conservative cleanup closure material.

## Best Possible Solution

Keep #38162 open. Resolve or explicitly supersede #38161 first, then review #38162 as the redirect-specific core follow-up with maintainer-approved workspace-containment semantics, symlink/retraction behavior, late block/unblock behavior, public hook context documentation, and redirect-focused tests. If maintainers decide not to add this bundled-hook policy surface, close only after that product/security decision and point plugin authors to an approved external-hook or plugin API replacement path.
