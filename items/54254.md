## Summary

Keep open. Current main has meaningful partial coverage for this report: Telegram routes true aborts and read-only status commands onto a control lane, uses grammY runner sink concurrency from `agents.defaults.maxConcurrent`, and has abort-fence tests that prevent stale finals after an abort. But the issue and follow-up comment explicitly report `/new` plus all slash commands being blocked during long inference, and current main still classifies `/new` and `/reset` as session commands that fall back to the normal chat/topic sequential lane. That leaves a real part of the requested behavior unresolved rather than safely implemented-on-main.

## Best Possible Solution

Keep the issue open and narrow the fix around Telegram responsiveness during active runs: preserve the existing `/stop` control-lane and abort-fence behavior, then add a safe live-control path or explicit acknowledgement path for session commands such as `/new` and `/reset` while a same-chat turn is active. Add tests that simulate an active Telegram dispatch plus later `/stop`, `/status`, and `/new` updates, including `agents.defaults.maxConcurrent: 1`, so the desired behavior is not only dependent on default runner concurrency.
