## Summary

Keep open. Current main has a related single-action-then-narrative guard, but it only retries when the single non-plan tool is in a small read-only allowlist. The issue's specific dummy `sleep`/bash evasion is still not covered because `exec`/`bash` are classified as mutating/potential side-effect tools, which makes the planning-only retry path return null instead of retrying and eventually blocking.

## Best Possible Solution

Keep #64883 open. The best fix is a focused command/tool-progress classifier that can distinguish harmless dummy shell/no-op calls from real mutating shell work, then thread that into `resolvePlanningOnlyRetryInstruction` without weakening replay safety. Add a regression test where strict-agentic GPT-5 emits one dummy sleep/no-op tool call plus planning prose and the runner retries up to the strict-agentic blocked exit.
