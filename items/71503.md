## Summary

Keep PR #71503 open. Current main still uses first-match LAN IPv4 selection after only exact `en0`/`eth0` preference, and the pairing/setup-code plus bundled `device-pair` plugin paths still have the same first-private-interface behavior. The PR is a focused, recent bug fix with follow-up commits/comments addressing review concerns, and I did not find an equivalent fix already landed on main.

## Best Possible Solution

Keep #71503 open and either land it after maintainer review or adapt its approach onto main. The best code path is to make gateway status/control UI, pairing setup-code generation, and the bundled `device-pair` plugin choose LAN IPv4 candidates consistently: prefer the non-bridge default-route interface when available, then physical/default-looking LAN interfaces, then other non-deprioritized candidates, and use Docker/bridge/CNI interfaces only as fallback. Preserve explicit configuration behavior and keep regression tests for Docker bridge, CNI overlay, fallback-only bridge, and setup-code/plugin URL resolution.
