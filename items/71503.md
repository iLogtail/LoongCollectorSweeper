## Summary

Keep PR #71503 open. Current main at 21c51bc140295e4286f1adc871a96b9b33c6a8b9 still has the older LAN selection behavior: gateway display/status only prefers exact en0/eth0 before taking the first external IPv4, while pairing setup and the bundled device-pair plugin still return the first matching private IPv4. The linked issue #71493 is closed, but the requested fix is not present on main, and the PR discussion shows active follow-ups addressing review feedback around CNI prefixes and default-route preference.

## Best Possible Solution

Keep the PR open for maintainer review or land an equivalent fix. The best implementation should make gateway status/dashboard, pairing setup, and device-pair use consistent LAN address selection that prefers default-route or physical/default-looking LAN interfaces, deprioritizes Docker/bridge/veth/CNI/container interfaces as fallback candidates, preserves explicit configuration overrides, and includes regression tests for Docker bridge, CNI overlay, and default-route cases.
