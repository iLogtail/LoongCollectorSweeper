## Summary

Keep open. Current `main` does not contain the PR's main latency changes in reply dispatch or session-message reads, and the ACPX portion of the PR is partly stale because the relevant runtime path has since moved into the external `acpx/runtime` package.

## Evidence

- **Reply dispatcher still uses old human-delay defaults with no webchat fast path:** `createReplyDispatcher` still uses `DEFAULT_HUMAN_DELAY_MIN_MS = 800` and `DEFAULT_HUMAN_DELAY_MAX_MS = 2500`, and block delivery still calls `getHumanDelay(options.humanDelay)` without any channel/webchat branch.
  - file: [src/auto-reply/reply/reply-dispatcher.ts:31](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/auto-reply/reply/reply-dispatcher.ts#L31)
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Prefix context on main still has no channel field:** `ResponsePrefixContext` only exposes model/provider/thinking/identity fields, and `createReplyPrefixOptions` returns only `responsePrefix`, `responsePrefixContextProvider`, and `onModelSelected`. That means the PR's channel-based webchat delay optimization is not already present on `main`.
  - file: [src/auto-reply/reply/response-prefix-template.ts:10](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/auto-reply/reply/response-prefix-template.ts#L10)
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Gateway/webchat path still passes the reply pipeline without a built-in webchat delay override:** The webchat chat handler spreads `replyPipeline` into `createReplyDispatcher`, but the pipeline comes from `createReplyPrefixOptions`, which does not include a channel field in the current main code.
  - file: [src/gateway/server-methods/chat.ts:2462](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/gateway/server-methods/chat.ts#L2462)
