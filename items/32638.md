## Summary

Keep #32638 open. Current main still has only narrow Groq reasoning-effort handling and does not preserve or map provider/model-native reasoning vocabularies such as LM Studio/Qwen `on`/`off`; the recent issue comments reporting ongoing LM Studio/Qwen warnings remain consistent with current source and tests.

## Best Possible Solution

Keep this open and implement reasoning-effort compatibility at the provider/model contract level. A strong fix would preserve discovered native vocabularies, expose typed compat metadata for provider-specific reasoning maps, broaden Groq handling beyond one hardcoded model id if Groq still requires `none/default`, and add regression tests for Groq plus LM Studio/Qwen-style `on/off` models in the OpenAI-compatible completions path.
