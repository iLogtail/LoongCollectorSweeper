## Summary

Keep open. The report is recent, includes a concrete live Discord/PluralKit repro shape, and the linked fix PR #52007 was closed unmerged. Current main has explicit OpenClaw webhook-echo suppression, but it still does not implement the requested fallback for original/proxy copies with different message IDs or the PluralKit-specific typing suppression described in the issue.

## Evidence

- **Issue/PR context:** The issue discussion acknowledges the source already has explicit bound-thread webhook suppression, but the reporter reaffirmed the lived duplicate-turn/typing behavior. The directly linked PR #52007, "Fix Discord PluralKit duplicate bound-thread turns", is closed and unmerged, so it is not a canonical shipped fix.
- **PluralKit lookup is skipped for webhook-authored messages:** Preflight resolves webhookId first and only calls fetchPluralKitMessageInfo when PluralKit is enabled and no webhookId is present, leaving webhook-authored proxy copies outside the PluralKit identity path.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts:409](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/extensions/discord/src/monitor/message-handler.preflight.ts#L409)
  - sha: [8e12c24d1721](https://github.com/openclaw/openclaw/commit/8e12c24d17218dc50023aaf5543df1b7c36520aa)
- **Webhook suppression is limited to the bound OpenClaw webhook identity:** shouldIgnoreBoundThreadWebhookMessage returns true only when the inbound webhook ID matches the thread binding webhook ID, or the recently-unbound echo cache matches; it does not broadly suppress PluralKit proxy webhooks in bound threads.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts:232](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/extensions/discord/src/monitor/message-handler.preflight.ts#L232)
  - sha: [8e12c24d1721](https://github.com/openclaw/openclaw/commit/8e12c24d17218dc50023aaf5543df1b7c36520aa)
- **Discord replay guard keys by message id:** The Discord inbound replay key is accountId + channelId + messageId, so original/proxied PluralKit copies with different Discord message IDs will not collapse at this layer.
  - file: [extensions/discord/src/monitor/inbound-dedupe.ts:22](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/extensions/discord/src/monitor/inbound-dedupe.ts#L22)
  - sha: [8e12c24d1721](https://github.com/openclaw/openclaw/commit/8e12c24d17218dc50023aaf5543df1b7c36520aa)
- **Generic inbound dedupe also keys by MessageSid:** Generic inbound dedupe builds its key from provider/account/session/peer/thread/messageId, while the Discord process path sets MessageSid to message.id. That matches the concern that different original/proxy IDs evade generic dedupe.
  - file: [src/auto-reply/reply/inbound-dedupe.ts:58](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/auto-reply/reply/inbound-dedupe.ts#L58)
  - sha: [8e12c24d1721](https://github.com/openclaw/openclaw/commit/8e12c24d17218dc50023aaf5543df1b7c36520aa)
- **Discord typing path has no PluralKit/bound-thread suppression:** The Discord reply pipeline always installs a typing start callback for the resolved Discord channel; there is no guard here for sender.isPluralKit or bound-thread PluralKit turns.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:562](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/extensions/discord/src/monitor/message-handler.process.ts#L562)
