## Summary

Keep PR #69932 open. Current main still computes `privateApiEnabled` in the BlueBubbles inbound monitor, but the typing restart, typing start, and final stop-typing paths still call `sendBlueBubblesTyping` without that guard. The shipped #65447-related send/attachment work covers outbound Private API decisions and cache refresh, not these inbound monitor typing callbacks. The PR remains a focused BlueBubbles plugin bug fix and adds the missing disabled-path monitor test; its remaining policy tradeoff is whether unknown Private API status should be treated like disabled for typing indicators.

## Best Possible Solution

Keep #69932 open for maintainer review or merge after choosing the intended unknown-status policy. The likely product path is to land the narrow BlueBubbles monitor-side guard and disabled-path regression test, or adjust the PR to distinguish explicit `false` from `null` if maintainers want to preserve typing attempts while capability is unknown.

Maintainers should choose whether that tradeoff is acceptable or whether a tri-state/refresh approach is preferred.
