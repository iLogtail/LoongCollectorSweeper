## Summary

Keep open. Current `main` still validates the entire config in one `OpenClawSchema.safeParse(...)` pass, so one invalid `models.providers.<id>` entry can still reject the whole config; the PR discussion also includes a recent shipped-version reproduction of that exact symptom.

## Evidence

- **Whole-config validation still short-circuits on any schema error:** `validateConfigObjectRaw` normalizes the raw config and then immediately calls `OpenClawSchema.safeParse(normalizedRaw)`. On failure it returns `ok: false` with schema issues; there is no provider-by-provider sanitization step before that parse.
  - file: [src/config/validation.ts:646](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/config/validation.ts#L646)
  - command: `nl -ba src/config/validation.ts | sed -n '614,650p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Provider entries are strict schema objects:** `ModelProviderSchema` is a strict object schema for each `models.providers` entry. Any unknown field or malformed provider payload fails that provider parse, and because validation is whole-config, that failure bubbles up to the entire config.
  - file: [src/config/zod-schema.core.ts:347](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/config/zod-schema.core.ts#L347)
  - command: `nl -ba src/config/zod-schema.core.ts | sed -n '347,378p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **User-facing docs expect custom provider entries to be preserved:** The CLI docs describe `models.providers` as a protected map and tell users to use `--merge` when adding entries, which matches the PR's concern that user-added provider definitions and overrides should not disappear because one sibling entry is invalid.
  - file: [docs/cli/config.md:110](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/docs/cli/config.md#L110)
  - command: `nl -ba docs/cli/config.md | sed -n '108,123p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Recent discussion reports the bug on a shipped build:** In the PR comments, `jw8957` reported on 2026-04-11 that OpenClaw `2026.4.10` dropped unrelated custom providers from `gateway models.list` because an implicit `codex` provider in `models.json` failed validation, and that manually adding a valid override restored them. That is the same failure mode this PR addresses.
- **Current main has no provider-sanitization implementation or test surface:** A repo search on current `main` found no `sanitizeInvalidProviders`, no `validation.provider-sanitize` test, and no `skipping invalid provider` warning path in `src/config`, which is consistent with the bug still being unresolved rather than already implemented elsewhere.
  - command: `rg -n "provider-level validation sanitization|validation\.provider-sanitize|skipping invalid provider|sanitizeInvalidProviders" src/config`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)

## Risks / Open Questions

- The underlying bug remains user-facing on current `main`: one bad provider entry can still wipe out unrelated valid custom provider config at validation time.
- The posted PR has unresolved review concerns in the discussion, including raw `console.warn` logging of untrusted config-derived text.
- The posted PR also has review feedback about storing raw provider values instead of `result.data` and missing an all-providers-invalid test case, so it should not be auto-closed as settled.
