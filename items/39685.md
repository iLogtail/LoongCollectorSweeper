## Summary

Keep #39685 open. Current main still has lower-level SSRF hostname allowlist machinery and a Codex-native `openaiCodex.allowedDomains` option, but it does not expose or enforce the requested global/per-agent `allowedDomains`/`denyDomains` egress firewall for `web_fetch` and managed `web_search`. The prior Codex review comment in the issue reached the same conclusion, and the current checked-out main SHA `dc05c93c024c652c410fb1fb430da4cb7442151b` still shows the same product gap.

## Best Possible Solution

Keep the issue open and implement a core web tool network policy: add typed/schema/docs support for global and per-agent `allowedDomains`/`denyDomains`, resolve it consistently with other tool policies, map allowlists into `SsrFPolicy.hostnameAllowlist` for `web_fetch`, add deny precedence, and define enforceable behavior for managed/native `web_search` providers with focused config and guard tests. If provider-native search cannot be fully constrained locally, document that limitation and fail closed where feasible.
