## Summary

Keep #69570 open. Current main has partial mitigations for async exec completion spam: heartbeat wake requests coalesce by target, pending exec events can be batched into one prompt, heartbeat skips a busy session lane, and recent runtime-only work keeps synthetic wake prompts out of visible transcripts. Those changes do not fully implement the reported stop/cancel and preemption behavior: non-empty exec completions still enqueue session system events and wake heartbeat, Control UI `/stop` maps to `chat.abort`, and `chat.abort` aborts active runs but does not clear pending system events or pending heartbeat wake state. The related closed #69564 is explicitly just a duplicate of #69570, not a canonical fix; #69386 overlaps stop/queue behavior but tracks a denial retry loop rather than this approved-completion cascade.

## Best Possible Solution

Keep the item open and narrow the implementation to the remaining runtime gap: add a session-scoped cancellation path for pending exec-completion system events and pending heartbeat wakes, wire Control UI `chat.abort`/`/stop` to that cleanup, and add regression tests for multiple approved async exec completions where a new user message or stop command preempts/cancels the remaining completion cascade. Preserve the current batching, wake coalescing, busy-lane skip, and runtime-only transcript fixes.
