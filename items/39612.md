## Summary

Keep open. Current main improves the default path with generic direct-chat copy and silent non-direct chat handling, but external direct chats still intentionally receive sanitized provider/runner failure details when `/verbose` is `on` or `full`. That conflicts with the issue’s central request to never expose LLM/API failure details through WhatsApp, Telegram, or similar messaging channels.

## Best Possible Solution

Keep this open for an explicit maintainer decision on external-channel failure policy. The likely core fix is to keep generic default direct-chat copy and silent group/channel behavior, but stop sending provider/runner exception details to external chat users even under `/verbose`; reserve details for logs, Control UI, or operator-only surfaces. Add regression coverage for WhatsApp/Telegram direct verbose LLM failures. Track multi-agent escalation and deeper session migration/recovery as separate follow-ups if maintainers want those parts in scope.
