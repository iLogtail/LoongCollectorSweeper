## Summary

Keep open. The related #52118 zero-default poll case is addressed, and media workspace-root handling appears partly improved, but current main still matches the central GPT-5.4 failure modes: `pollDurationHours: 24` is treated as poll intent on `send`, and an all-empty Discord `components` object with `modal.fields: []` still routes into component parsing and throws instead of being normalized away.

## Evidence

- **Non-zero poll duration still triggers poll intent:** `hasPollCreationParams` treats any finite non-zero numeric poll parameter as poll creation intent. A GPT-5.4 default like `pollDurationHours: 24` therefore still returns true even when `pollQuestion` is empty/absent.
  - file: [src/poll-params.ts:67](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/poll-params.ts#L67)
  - command: `nl -ba src/poll-params.ts | sed -n '1,140p'`
  - sha: [8e12c24d1721](https://github.com/openclaw/openclaw/commit/8e12c24d17218dc50023aaf5543df1b7c36520aa)
- **Send action still rejects when poll intent is detected:** `runMessageAction` still throws `Poll fields require action "poll"` for `action === "send" && hasPollCreationParams(params)`, so the non-zero default duration path still blocks normal sends.
  - file: [src/infra/outbound/message-action-runner.ts:927](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/infra/outbound/message-action-runner.ts#L927)
