## Summary

Keep #69249 open. Current main has restart deferral for managed restarts, restart-sentinel notices/continuations for tool/config/update restarts, and internal main-session restart recovery. The remaining gap is still the issue's central user-visible problem: an arbitrary gateway restart that aborts an in-flight Discord turn can leave no Discord-channel recovery message, because main-session recovery resumes with `deliver: false` and sentinel outbound delivery only runs when a restart sentinel plus routable context exists.

## Best Possible Solution

Keep the issue open and implement a narrow core/channel-generic recovery path for interrupted routed main sessions: persist enough in-flight delivery context and run state before shutdown, detect aborted turns on startup, then either resume with delivery back to the original Discord route or post a deterministic recovery notice when safe resume is not possible. Add focused tests for `agent:main:discord:*` sessions, restart-sentinel dedupe, and the existing internal `deliver: false` recovery path.
