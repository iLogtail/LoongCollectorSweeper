## Summary

Current main does not yet solve the reported upgrade break. The memory-core manifest still validates `dreaming.storage` as an object only, config loading validates plugin config before the CLI can run normally, and the existing legacy/doctor migration paths do not include a memory-core `dreaming.storage` string-to-object migration. The related default-storage issue #68468 addresses the default mode change, not legacy string config compatibility.

## Best Possible Solution

Keep the issue open and add a narrow compatibility repair: before plugin schema validation, or through a memory-core doctor compatibility contract, rewrite `plugins.entries.memory-core.config.dreaming.storage: "inline" | "separate" | "both"` to `{ "mode": <same value>, "separateReports": false }`. Add regression coverage for config loading/doctor repair with the legacy string shape and an upgrade-facing hint so affected users are not left with an opaque plugin-schema error.

## Risks / Open Questions

- A schema-only fix would accept legacy input but still risk silently changing explicit `"both"` to the default unless the resolver or migration preserves the selected mode.
- Because current docs now call dreaming storage behavior internal, maintainers should decide whether to keep this as a compatibility-only migration rather than re-documenting storage as a supported user setting.
