## Summary

Keep open. Current main still rejects the reported legacy `dreaming.storage: "both"` shape during memory-core plugin schema validation, and I did not find a raw migration, doctor compatibility rule, or plugin doctor contract that rewrites the old string form before validation. The related ClawSweeper item #68468 only covered the default storage mode change, not this upgrade-breaking schema migration.

## Best Possible Solution

Add a narrow compatibility migration before plugin schema validation, and wire it into `openclaw doctor --fix`, that rewrites legacy `plugins.entries.memory-core.config.dreaming.storage` string values `inline`, `separate`, or `both` to `{ mode: <value>, separateReports: false }`. Add a regression test using a v2026.4.15-style config to prove `openclaw status`/config load and doctor repair no longer fail. Keep the public schema object-shaped going forward; accept the string only at the migration boundary.
