## Summary

Keep #68596 open. Current main still has the TUI streaming watchdog defaulting to 30 seconds, with only an internal `streamingWatchdogMs` context override used by tests. The production TUI path does not pass a user/config value, the TUI CLI/docs do not expose a watchdog threshold, and the ACP/server-side timeout settings cited in the issue are separate controls. The issue is recent, coherent, and corroborated by several users; related #69978 is overlapping UX work, not a clear superseding canonical tracker for this older central configurability request.

## Best Possible Solution

Keep #68596 open as the main configurable TUI streaming-watchdog threshold request. A good fix would add a user-facing setting, wire it through `TuiOptions`/`runTui` into `createEventHandlers`, validate and document the config/CLI or `/watchdog` surface, and add regression tests proving long delta-silent reasoning can raise or disable the client watchdog without changing unrelated server-side LLM idle timeout behavior. Coordinate with #69978 for duplicate-warning suppression and runtime command UX, and with #69081 for tool/reconnect desync fixes.
