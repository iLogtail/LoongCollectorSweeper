## Summary

Keep open. Current main has partial improvements around missing auth detection and fallback logging, but the issue's requested behavior is not fully implemented: remote custom providers without an apiKey still participate in model fallback, auth errors are still fallback-eligible, override/per-agent failures can still land on the configured default primary, and there is no gateway-level credential-sharing surface for multiple AI Gateway-backed providers.

## Evidence

- **Remote missing apiKey is detected before transport:** Current tests assert that a remote custom provider with baseUrl/api/models but no apiKey rejects with "No API key found". That prevents a truly unauthenticated request in this path, but it does not by itself prevent the model fallback path from trying another model.
