## Summary

Keep #43567 open. Current main has a narrow compaction divider for synthetic compaction checkpoint entries, but the reported pre-compaction memory-flush prompt is still submitted through the normal embedded prompt path and has no display provenance/role that would let Control UI or TUI render it as a distinct system/runtime block. The related #7532 cleanup is about suppressing noisy exec lifecycle events, not this visual/provenance distinction.

## Best Possible Solution

Keep the issue open and implement a real display-provenance path for runtime-originated prompts such as memory flushes and queued system events. The likely fix is to tag these transcript/history entries with stable OpenClaw metadata or a dedicated display role, preserve that through `chat.history`/SSE, and render them as a muted System/Runtime block in Control UI and TUI, with regression tests covering the pre-compaction memory flush example.
