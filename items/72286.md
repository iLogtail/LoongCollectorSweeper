## Summary

Keep #72286 open. Current main does not implement Google Live browser Talk sessions: the public realtime browser session contract and gateway response only expose the OpenAI-style client-secret fields, the Control UI Talk client hard-requires WebRTC and posts SDP to OpenAI, and the Google realtime provider only implements backend bridge sessions. The PR is a draft, but it targets a real missing core/UI/provider contract rather than something already shipped or cleanly closeable as ClawHub-only work.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is a narrowed, provider-agnostic browser realtime transport design that extends `RealtimeVoiceBrowserSession`, the Gateway schema/handler, and Control UI transport handling, then adds Google `createBrowserSession` support with focused provider/UI tests and updated public docs.
