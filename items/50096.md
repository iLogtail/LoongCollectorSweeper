## Summary

Keep open. Current main now covers much of the long-term memory surface through memory-core search, opt-in session transcript indexing, scheduled dreaming promotion, memory-wiki, and broader bootstrap injection controls. However, #50096 also asks about core reliability/config behavior that is still not fully satisfied: memoryFlush is documented and implemented as a pre-compaction/turn-time path, not a graceful gateway-shutdown flush, and current injection controls are broad workspace-bootstrap controls rather than a MEMORY.md-specific full/recall-only mode.

## Best Possible Solution

Keep the issue open or replace it with explicit canonical follow-ups: one for graceful shutdown/session-stop memory persistence semantics, one for MEMORY.md-specific injection mode or documented recall-only configuration, and one docs/product note that points users to the shipped QMD + dreaming + memory-wiki path while keeping UAML/Engram-style external memory engines in the plugin/ClawHub lane.
