## Summary

Keep open. Current main still runs the `runtime-assets` `pnpm prune --prod` step without the shared pnpm store cache mount, while the build-stage `pnpm install` step uses `id=openclaw-pnpm-store` and the repo is configured with `node-linker=hoisted`. The PR remains a coherent Docker build-cache fix, though it now needs rebasing onto the newer multi-command `runtime-assets` RUN block.

## Best Possible Solution

Keep the PR open, rebase it onto the current `runtime-assets` Dockerfile block, add the shared pnpm store cache mount to the prune RUN, and consider adding a small Dockerfile regression test that the `pnpm prune --prod` RUN itself includes `id=openclaw-pnpm-store`. Merge only after current Docker build/plain-log and runtime smoke evidence.
