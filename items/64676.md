## Summary

Keep open. The issue comment's config-cascade concern still matches current main for the reported manual `models.json` model-list edit: gateway/PI startup paths still call `ensureOpenClawModelsJson`, merge mode only preserves selected existing provider fields such as `apiKey` and `baseUrl`, and current docs say other provider fields are refreshed from config/catalog data. Related cleanup items cover cron persistence, invalid config recovery, restart loops, provider secrets, and Tailscale config, but none is a canonical fix for this model-list persistence report.

## Best Possible Solution

Keep the issue open for maintainer product judgment and a focused fix or clarification. The likely durable path is to decide whether manual per-agent `models.json` model-list removals are supported. If supported, update `mergeWithExistingProviderSecrets`/planning tests so matching provider model arrays can be preserved or removed intentionally across gateway restarts. If unsupported, document `models.json` as generated state more explicitly and point users to `agents.defaults.models` for allowlisting plus `models.providers.<id>.models` edits through `openclaw config set --merge/--replace`, with a test covering that restart behavior.
