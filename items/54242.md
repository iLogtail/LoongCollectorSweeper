## Summary

Current main does not appear to fix #54242. Backup creation still remaps archive entry paths only, while verification validates entry paths, manifest coverage, and duplicate paths but does not inspect or reject hardlink link targets. Related #40786 is broader exclude support, and #57515 concerned symlink duplicate entries, not the reported hardlink target with `..`.

## Best Possible Solution

Keep the issue open and fix it in the core backup CLI. The narrow path is to make backup archives restore-safe for hardlinks by dereferencing or rewriting hardlink targets into normalized in-archive paths, and to extend `openclaw backup verify` with link-target validation that catches `..` targets. Add a regression fixture for a hardlink target containing `..`; keep #40786 as separate exclude-pattern follow-up rather than treating it as the fix for this bug.

## Evidence
