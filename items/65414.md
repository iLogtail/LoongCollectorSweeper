## Summary

Keep open. Current main still parses ACTION_ASSIST as an assist launch with no prompt, but MainViewModel unconditionally routes every assistant launch to HomeDestination.Chat. VoiceTabScreen only enables the mic from manual Voice-tab controls or Talk Mode permission flow; there is no one-shot assistant-launch voice auto-activate state to consume. The latest release tag shows the same routing, and the provided related ClawSweeper items are unrelated TTS/Discord routing reports rather than a canonical fix or superseding Android issue.

## Best Possible Solution

Implement the Android app change in the existing assistant-launch path: route source == "assist" with prompt == null to HomeDestination.Voice, add a one-shot pending voice auto-activate state on MainViewModel, consume it from VoiceTabScreen with the existing RECORD_AUDIO permission flow, preserve ASK_OPENCLAW/text-prompt launches to Chat, and add focused unit/UI coverage plus a docs note for Android assistant entrypoints.
