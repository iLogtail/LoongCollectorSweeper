## Summary

Keep open. The PR targets real `prompter.text()` null-safety gaps that are still present on `main`, but the proposed `src/channels/plugins/setup-wizard.ts` change has a reviewed regression: it would treat an intentional empty string as `undefined` and preserve the old optional value instead of clearing it.

## Evidence

- **PR change would regress optional-field clearing:** Current `setup-wizard` logic trims `rawValue` and then explicitly clears optional fields when the trimmed value is empty. The reviewed PR replacement would route through `normalizeOptionalString`, which collapses `""` to `undefined`; that would fall back to `initialValue` and skip the clear path.
  - file: [src/channels/plugins/setup-wizard.ts:458](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/channels/plugins/setup-wizard.ts#L458)
  - command: `nl -ba src/channels/plugins/setup-wizard.ts | sed -n '445,470p' ; nl -ba src/shared/string-coerce.ts | sed -n '13,15p'`
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **String normalization proves the regression:** `normalizeOptionalString` is defined as `normalizeNullableString(value) ?? undefined`, and `normalizeNullableString` returns null for trimmed-empty strings. That means `normalizeOptionalString("")` becomes `undefined`, exactly the behavior called out in review.
  - file: [src/shared/string-coerce.ts:13](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/shared/string-coerce.ts#L13)
