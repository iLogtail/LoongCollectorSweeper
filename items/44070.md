## Summary

Keep open. Current `main` has partial WSL/systemd hardening for probe-time bus errors like `Failed to connect to bus` and `No medium found`, but it still hard-fails `systemctl --user daemon-reload` during gateway install, and the reported `Transport endpoint is not connected` / `Reload daemon failed` path is not explicitly handled anywhere I could find.

## Evidence

- **Systemd-unavailable classifier does not cover the reported daemon-reload error:** The Linux systemd unavailable matcher only recognizes bus/session cases such as `failed to connect to bus`, `failed to connect to user scope bus`, `DBUS_SESSION_BUS_ADDRESS`, `XDG_RUNTIME_DIR`, and `no medium found`. It does not match `Transport endpoint is not connected` or `Reload daemon failed`.
  - file: [src/daemon/systemd-unavailable.ts:23](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/daemon/systemd-unavailable.ts#L23)
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Unknown nonzero `systemctl --user status` results can still pass availability gating:** `assertSystemdAvailable()` throws only for known unavailable classifications. If `systemctl --user status` fails with some other detail, the function returns instead of blocking install, so install can proceed into later `daemon-reload` failure paths.
  - file: [src/daemon/systemd.ts:414](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/daemon/systemd.ts#L414)
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Gateway install still hard-fails on `daemon-reload`:** `activateSystemdService()` immediately throws `systemctl daemon-reload failed: ...` on any nonzero reload result, with no fallback or special handling for the reported transport-endpoint error.
  - file: [src/daemon/systemd.ts:541](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/daemon/systemd.ts#L541)
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Current install hardening only covers pre-install probe failures:** `runDaemonInstall()` treats only `service.isLoaded()` probe failures as non-fatal via `isNonFatalSystemdInstallProbeError()`. That does not cover failures raised later by `service.install()` / `installSystemdService()`.
  - file: [src/cli/daemon-cli/install.ts:92](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/cli/daemon-cli/install.ts#L92)
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Tests added on main cover `No medium found`, not daemon-reload transport-endpoint failures:** Regression coverage exists for continuing when the service-status probe reports `systemctl --user unavailable: Failed to connect to bus: No medium found`, which is narrower than the issue’s reported `systemctl daemon-reload failed: Reload daemon failed: Transport endpoint is not connected`.
  - file: [src/commands/configure.daemon.test.ts:157](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/commands/configure.daemon.test.ts#L157)
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **No code/docs/changelog match for the reported error string:** A repository search for `Transport endpoint is not connected` and `Reload daemon failed` returned no matches, which supports that this exact failure mode has not been explicitly addressed on current `main`.
  - command: `rg -n "Transport endpoint is not connected|Reload daemon failed" src docs CHANGELOG.md`
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
