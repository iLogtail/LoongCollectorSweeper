## Summary

Keep #39554 open. Current main still sends any truthy Google Chat thread value directly as `body.thread.name` and still has outbound paths that forward raw `threadId`/`replyToId` values. The exact validation PR #46519 was closed unmerged, and the open related PR #67055 only addresses `replyToMode: "off"` propagation, not provider-local validation of invalid Google Chat thread resource names.

## Best Possible Solution

Keep this issue open until Google Chat outbound code validates or normalizes thread resource names at the plugin boundary. The likely fix belongs in `extensions/googlechat/src/api.ts` or a shared Google Chat threading helper used by API sends, monitor reply delivery, attached-results sends, and message actions: preserve valid `spaces/{space}/threads/{thread}` values, drop invalid raw message/session ids for flat sends, and add focused regression tests for text, media, actions, and reply delivery.
