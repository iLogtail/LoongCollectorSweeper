## Summary

Keep #39554 open. Current main still forwards unvalidated Google Chat thread values into `thread.name`, and several outbound paths can still pass raw `threadId`, `replyTo`, or `replyToId` values instead of only valid Google Chat thread resources.

## Best Possible Solution

Keep #39554 open until the Google Chat plugin preserves valid `spaces/{space}/threads/{thread}` resources, omits or converts invalid raw message/session/action ids before API calls, respects `replyToMode: "off"` across typing/text/media paths, and adds regression tests for API sends, message actions, attached results, and buffered streamed/block replies.

## Evidence

- **Google Chat thread contract:** Google's REST docs distinguish message resource names (`spaces/{space}/messages/{message}`) from thread resource names and describe `Thread.name` with example `spaces/{space}/threads/{thread}`; `spaces.messages.create` threads messages via `thread.name` or `threadKey`. 
  - command: `web open official Google Chat REST docs`
- **API boundary still trusts raw thread input:** `sendGoogleChatMessage` accepts `thread?: string`, writes it directly to `body.thread = { name: thread }`, and sets `messageReplyOption` whenever the value is truthy; there is no resource-name validation, normalization, or invalid-value omission at the provider API boundary.
