## Summary

Keep open. Current main still forwards non-empty Google Chat thread strings directly into `thread.name` without validating the required `spaces/{space}/threads/{thread}` resource shape, and current tests still assert raw `threadId` forwarding. The exact validation PR #46519 is closed unmerged; related PR #67055 is open and only partially overlaps with `replyToMode: "off"` behavior.

## Best Possible Solution

Keep the issue open until Google Chat outbound sends validate or normalize thread resource names before calling `spaces.messages.create`, suppress `thread` when `replyToMode` is `off` across typing/text/media paths, and add regression tests for raw message IDs, session-derived IDs, valid `spaces/.../threads/...` values, and flat DM replies.
