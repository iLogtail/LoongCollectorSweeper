## Summary

Keep #39554 open. Current main still serializes any truthy Google Chat `thread` string as `body.thread.name` and several outbound paths can pass raw `threadId`, `replyTo`, or `replyToId` values through without validating the required `spaces/{space}/threads/{thread}` resource format. The exact validation PR #46519 was closed unmerged; #67055 and #69422 are related but do not fully supersede this provider-boundary validation bug.

## Best Possible Solution

Keep #39554 open until the Google Chat plugin validates or normalizes thread identifiers at the provider boundary. The best fix is a shared Google Chat thread helper used by every `sendGoogleChatMessage` caller: preserve valid `spaces/{space}/threads/{thread}` resources, omit invalid raw message/session IDs for flat sends, respect `replyToMode: "off"`, and add regression tests for API sends, message actions, attached-results text/media, monitor reply delivery, and streamed/block reply paths.
