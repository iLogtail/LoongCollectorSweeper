## Summary

Keep #41676 open. Current main only caps the structured `logging.file` sink with `logging.maxFileBytes`; the reported macOS service stderr file `gateway.err.log` is still a LaunchAgent `StandardErrorPath` target with no repo-level size cap, rotation, truncation, or duplicate heartbeat-line suppression visible in source/docs/tests.

## Best Possible Solution

Keep the issue open and fix the daemon-facing surface: add bounded rotation/truncation for `gateway.log` and `gateway.err.log` or route service stdout/stderr through a bounded OpenClaw-managed writer, then add focused launchd/service-log tests. Separately, trace the WebSocket heartbeat timeout emitter and rate-limit or aggregate repeated timeout lines so one noisy socket state cannot consume disk or bury useful diagnostics.
