## Summary

Keep open. Current `main` does not contain the PR's fix: `evaluateChannelHealth()` still only marks a channel disconnected after startup grace when `connected === false`, and it does not inspect `lastConnectedAt` at all. Because channel runtime updates are patch-merged on start, stale `connected`/`lastConnectedAt` state from a prior lifecycle can still survive into a new lifecycle on `main` and evade this health check.

## Evidence

- **Post-grace disconnect check still only handles explicit false:** After startup grace, `evaluateChannelHealth()` returns `disconnected` only for `snapshot.connected === false`; any tracked-but-not-true state otherwise falls through to later checks and can end as `healthy`.
