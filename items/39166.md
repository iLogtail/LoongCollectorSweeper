## Summary

Current main only contains the explicit ACP runTimeoutSeconds plumbing. It does not contain the PR's central config-level acp.defaultRunTimeoutSeconds fallback, and the linked issue #38419 was closed stale rather than replaced by a canonical implementation issue. The PR remains actionable, though it needs rebasing because part of its diff has already landed elsewhere.

## Best Possible Solution

Keep this PR open and ask for a narrow rebase against current main. The best implementation is to preserve the already-landed explicit ACP timeout behavior, add acp.defaultRunTimeoutSeconds to the config contract and user-facing docs, resolve ACP timeout as explicit per-call value -> config default -> 0, apply the resolved value to ACP dispatch/runtime paths without breaking explicit 0, and add regression tests for config default, explicit override, and explicit 0 over a non-zero default.
