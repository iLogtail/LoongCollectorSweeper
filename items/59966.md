## Summary

Current main still contains the exact hardcoded `Agent response timeout (120s)` race in the Synology Chat channel webhook path. The generic agent runtime timeout is configurable and is wired into reply execution, but Synology Chat wraps that delivery in an independent 120s `Promise.race`, so a configured `agents.defaults.timeoutSeconds: 600` can still be cut off by this channel handler. Related timeout cleanup reports do not supersede this narrower remaining Synology/channel-layer bug.

## Best Possible Solution

Keep #59966 open. The likely fix belongs in the Synology Chat plugin/channel handler: remove the independent 120s `Promise.race`, or derive any channel-side wait budget from the same resolved agent timeout used by reply dispatch, then add a focused Synology webhook test proving `agents.defaults.timeoutSeconds` above 120 seconds is not capped by the channel layer.
