## Summary

Keep open. Current main has a file-backed write-ahead outbound delivery queue and startup crash-recovery, but the specific health-monitor channel-restart path in this issue is not clearly covered for Telegram/iMessage. The monitor still performs in-process stop/start restarts, and the only reconnect-drain wiring I found is WhatsApp-specific, so there is not strong evidence that messages queued during Telegram/iMessage channel restarts are retried promptly on current main.

## Evidence

- **Health monitor still does in-process channel stop/start restarts:** The stale-socket health monitor calls `stopChannel(...)` and then `startChannel(...)` for the affected account. This is the restart path described in the issue, and it is not a full gateway process restart.
  - file: [src/gateway/channel-health-monitor.ts:160](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/gateway/channel-health-monitor.ts#L160)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Outbound sends are persisted before send and recovered on gateway startup:** `deliverOutboundPayloads()` now enqueues a write-ahead delivery entry before sending, and `activateGatewayScheduledServices()` runs `recoverPendingDeliveries(...)` on startup. That addresses crash/gateway-restart loss, but only on startup recovery.
  - file: [src/infra/outbound/deliver.ts:649](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/infra/outbound/deliver.ts#L649)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Startup recovery is wired after sidecars are ready:** Gateway scheduled-service activation imports `recoverPendingDeliveries` and runs it once after startup.
  - file: [src/gateway/server-runtime-services.ts:55](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/gateway/server-runtime-services.ts#L55)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Reconnect drain exists as an SDK helper but I found only WhatsApp using it:** The shared SDK exposes `drainPendingDeliveries(...)`, and WhatsApp calls it on reconnect to flush queued entries for that account.
  - file: [extensions/whatsapp/src/auto-reply/monitor.ts:393](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/extensions/whatsapp/src/auto-reply/monitor.ts#L393)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **No reconnect-drain wiring found for Telegram or iMessage:** A repo search for `drainPendingDeliveries(` under `extensions/telegram` and `extensions/imessage` returned no matches, so I did not find equivalent reconnect recovery for the channels named in the issue.
  - command: `rg -n "drainPendingDeliveries\(" extensions/telegram extensions/imessage`
- **Changelog shows earlier queue work, but the report post-dates it:** The changelog already records a write-ahead delivery queue in `2026.2.13`, yet this issue was filed later against OpenClaw 2026.3.2, which weakens any claim that the existing queue work fully resolved this specific channel-restart case.
  - file: [CHANGELOG.md:5019](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/CHANGELOG.md#L5019)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)

## Risks / Open Questions

- There could be an indirect Telegram/iMessage recovery path outside `drainPendingDeliveries(...)` that was not obvious from the code search, but I did not find one.
- Current main likely improves crash/gateway-restart recovery, so the remaining bug surface may be narrower than the issue body suggests; still, that is not strong enough to close conservatively.
