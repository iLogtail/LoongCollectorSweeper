## Summary

Keep open. Current main still contains the exact VoiceWakeOverlay render-loop shape described in the issue: attributed text changes synchronously call window frame recomputation, and frame measurement synchronously mutates observable overlay state. I found no current async/debounced guard, no merged superseding fix, and no regression test covering the SwiftUI update-loop warning or pinwheel symptom.

## Best Possible Solution

Keep #43480 open and fix the macOS VoiceWakeOverlay path directly. The best product/code direction is to make layout measurement pure, update `isOverflowing` only when the value changes and outside the active SwiftUI render/update cycle, and defer or consolidate attributed-text window-frame updates. Add a focused regression test or manual verification note that the overlay no longer emits SwiftUI 'modifying state during view update' / repeated onChange warnings under rapid transcript updates.
