## Summary

Keep open. Current `main` still intentionally treats an ended subagent run with pending descendants as active, and that same active-count path is used to enforce the `sessions_spawn` max-children quota. The reported behavior is therefore still present on `main`, even though related recovery/archive mitigations now exist.

## Evidence

- **Active-list predicate still marks ended parents as active when descendants are pending:** `isActiveSubagentRun` returns true when a run has no `endedAt` or when its `childSessionKey` still has pending descendants. `buildSubagentList` uses that predicate to populate the `active` list.
  - file: [src/agents/subagent-list.ts:132](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/agents/subagent-list.ts#L132)
  - command: `nl -ba src/agents/subagent-list.ts | sed -n '120,290p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Quota gate still uses active-run count:** `sessions_spawn` computes `activeChildren = countActiveRunsForSession(requesterInternalKey)` and blocks new spawns with the exact `max active children` error once the configured limit is reached.
  - file: [src/agents/subagent-spawn.ts:629](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/agents/subagent-spawn.ts#L629)

## Risks / Open Questions

- The current behavior is partly intentional for descendant-cleanup and announce bookkeeping, so any fix likely needs to separate user-facing 'active' status/quota semantics from cleanup-pending lifecycle semantics.
- The report's restart angle may be partially mitigated by newer orphan-recovery paths, but those paths do not change the core active/quota logic that this issue reports.
