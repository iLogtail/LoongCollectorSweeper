## Summary

Keep open. Current `main` still prefers the active session model in the safeguard compaction hook, so `agents.defaults.compaction.model` is not reliably honored for safeguard/auto-compaction. The latest shipped release also still contains the same precedence.

## Evidence

- **User-facing docs promise a compaction model override:** The docs say `agents.defaults.compaction.model` can select a different model for compaction summarization, and that the primary model is only used when the override is unset.
  - file: [docs/concepts/compaction.md:45](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/docs/concepts/compaction.md#L45)
  - command: `nl -ba docs/concepts/compaction.md | sed -n '40,76p'`
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Current safeguard hook still prefers `ctx.model` over runtime override:** `session_before_compact` resolves `const model = ctx.model ?? runtime?.model;`. In an active session, that makes the session model win whenever both are present, which matches the reported bug.
  - file: [src/agents/pi-hooks/compaction-safeguard.ts:912](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/agents/pi-hooks/compaction-safeguard.ts#L912)

That path does not resolve `agents.defaults.compaction.model` before wiring the safeguard runtime.

## Risks / Open Questions

- Auto-closing would hide a live cost/config bug: safeguard auto-compaction can still use the expensive session model despite a configured compaction override.
- A complete fix likely needs more than swapping precedence in the hook; the normal run/attempt path also appears to seed safeguard runtime with the active model instead of a resolved compaction target.
