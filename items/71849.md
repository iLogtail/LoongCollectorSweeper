## Summary

Keep #71849 open. Current main still routes realtime voice context questions through the shared full `openclaw_agent_consult` embedded-agent path, with memory tools allowed only inside that slower consult. I found no `realtime.fastContext` config, bounded direct memory/session lookup, partial-transcript accumulation for context triggers, or explicit fast miss/timeout path in the voice-call realtime handler. The existing maintainer comment also explicitly reviewed the same surface and concluded this is a real remaining gap, not a duplicate of the merged realtime voice work.

## Best Possible Solution

Keep the issue open and implement a default-off `realtime.fastContext` path in the Voice Call plugin: add config schema/manifest/docs, accumulate and normalize partial user transcript chunks, detect context-seeking turns, call the active memory/session search runtime under a short hard deadline, inject either top snippets or an explicit miss/timeout result into the realtime session, and preserve `openclaw_agent_consult` as fallback for deeper work. Add focused tests for partial chunks, multiple context questions in one call, timeout/miss behavior, and fallback-to-consult behavior.
