## Summary

Keep #50213 open. Current main still does not implement a Gemini web_search baseUrl override: the bundled Google plugin schema only allows `apiKey` and `model`, the Gemini web-search config type/runtime ignore any endpoint field, and the runtime always builds requests from the fixed Google Generative Language base URL. This is a concrete missing provider-owned config enhancement, not a stale or duplicate report.

## Best Possible Solution

Add an optional `plugins.entries.google.config.webSearch.baseUrl` field to the bundled Google plugin schema/UI hints/docs, parse it in the Gemini web-search config, pass it into the runtime endpoint builder, include it in cache-key behavior if endpoint-specific results can differ, and add config/runtime tests proving both the default Google endpoint and a custom proxy endpoint work. Prefer the current plugin-owned config path, not the legacy `tools.web.search.gemini` shape from the issue body.

## Risks / Open Questions

- Adding arbitrary Gemini search base URLs needs an explicit trust/SSRF policy and tests, because this path sends API keys to the configured endpoint.
- Third-party Gemini-compatible providers may not all support Google Search grounding exactly like the official Generative Language API, so the implementation should keep the default endpoint unchanged and document compatibility expectations.
