## Summary

Keep #71555 open. Current main still pins `@mariozechner/pi-coding-agent` 0.70.2 and still opens embedded-run transcripts through `SessionManager.open()` before handing the manager to `createAgentSession()`. The pinned dependency still performs whole-file `readFileSync(...).split("\n")` parsing in `loadEntriesFromFile()`, calls that from `SessionManager.open()`, and then constructs a manager whose constructor calls `setSessionFile()` and loads the same file again. Existing stale-lock cleanup explains the restart log, but it is recovery behavior, not a fix for the large-transcript crash path.

## Best Possible Solution

Keep the issue open and fix it in the embedded Pi/session boundary. The best path is to eliminate the second whole-file load in the pinned Pi `SessionManager.open()` path, either by upgrading/patching `pi-coding-agent` to header-read once and pass parsed entries into the manager, or by adding a narrow OpenClaw adapter/seam that avoids duplicate parsing for existing session files. Add focused regression/perf coverage with a large JSONL or long tool-result transcript that proves embedded context rebuild/truncation returns an error or compacts cleanly without gateway exit. Keep stale-lock cleanup as recovery, not the primary fix.
