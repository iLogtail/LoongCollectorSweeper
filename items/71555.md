## Summary

Keep #71555 open. Current main still pins `@mariozechner/pi-coding-agent@0.70.2`, and that upstream `SessionManager.open()` still reads/parses the whole JSONL once to discover the header/cwd and then constructs a `SessionManager` whose `setSessionFile()` reads/parses the same file again. OpenClaw’s embedded attempt path still calls `prewarmSessionFile()` and then `SessionManager.open()` before creating the embedded agent session; the prewarm helper only reads 4096 bytes for OS page-cache warmup and does not reuse parsed entries or a manager instance. The related stale-lock cleanup item #52929 explains the post-crash dead-pid cleanup symptom, but it does not supersede the large-transcript crash/memory-pressure path reported here.

## Best Possible Solution

Keep #71555 open. The best fix is to remove the double full-file parse before embedded context rebuild, preferably by fixing/upgrading `pi-coding-agent` so `SessionManager.open()` does header-only cwd discovery or passes the first parsed entries into the constructed manager. If upstream cannot move quickly, add an OpenClaw-side adapter for large transcripts plus a focused regression that proves embedded startup/tool-result truncation does not read and parse a large JSONL twice or crash before returning a controlled error.
