## Summary

Keep open. Current main has a real partial mitigation for the shared-token/password backend path, but it still does not implement the requested same-process `sessions_spawn` shortcut and still leaves no-secret `gateway.auth.mode="none"`/missing-shared-auth backend calls on the pairing/device path. The issue is recent, concrete, and not fully superseded by the related embedded-tool in-process dispatch PR.

## Best Possible Solution

Keep this open and narrow the maintainer decision: either implement a true same-process `sessions_spawn` dispatch path into the existing agent ingress/queue lane, or explicitly define the supported boundary as direct-loopback backend shared token/password auth only. If the latter is intended, update the issue/docs with that decision and add regression coverage showing auth-none/no-shared-auth behavior is intentionally rejected rather than accidentally hitting the pairing gate.

## Risks / Open Questions

- Current main may already be good enough for normal token/password server installs because startup auth can generate a token and the backend shared-auth path now avoids stale paired-device scope baselines.
- Maintainers may decide `gateway.auth.mode="none"` should remain outside backend subagent RPC support for security/product reasons, but that decision should be explicit rather than auto-closing this concrete report.
