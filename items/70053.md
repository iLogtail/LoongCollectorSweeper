## Summary

Keep open. Current main has a partial shared-token/password backend-loopback mitigation, but native `sessions_spawn` still enters the Gateway through the WebSocket client path and no true same-process subagent dispatch shortcut exists. The issue's no-shared-auth / `auth.mode="none"` concern is not clearly implemented or intentionally rejected, and related PR #42497 covers embedded gateway tool calls rather than this native subagent path.

## Best Possible Solution

Keep the issue open and make the product boundary explicit: either implement a true same-process `sessions_spawn` dispatch path into the existing agent ingress/`subagent` queue lane, or document and test that only direct-loopback backend shared token/password auth is supported and that auth-none/no-shared-auth deployments must seed credentials or pair a device.

## Risks / Open Questions

- The shared-token/password backend loopback path may already be good enough for many server installs, so the remaining issue should be narrowed to true in-process dispatch and no-shared-auth/auth-none behavior.
- Maintainers may intentionally want `auth.mode="none"` backend subagent RPC to require device/pairing approval; if so, the best outcome is an explicit docs/test decision rather than auto-closing this report.
