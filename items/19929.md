## Summary

Keep #19929 open. Current main still isolates non-direct group/channel peers by channel and peer id, only exposes DM-oriented session sharing controls, and has no strict config/API surface for selective shared sessions across chosen channels or threads. The existing maintainer/Codex comment on the issue also explicitly kept it open, noting PR #20078 closed unmerged and #7524 is narrower group-to-main work rather than a replacement.

## Best Possible Solution

Keep #19929 open as the concrete tracker for opt-in shared sessions across selected group/channel/thread chats. A good implementation belongs in core routing/config: add a backward-compatible session grouping config, thread it through buildAgentSessionKey/resolveAgentRoute and thread/outbound paths, update strict schemas/types/help/docs, and add regression tests for grouped routing, ungrouped compatibility, reply delivery, queue serialization, and thread inheritance. Reuse any useful design from closed PR #20078, but re-review the trust-boundary and sandbox semantics before merging.
