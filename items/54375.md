## Summary

Keep open. Current main does not implement this PR's broad behavior change: `isSystemdServiceEnabled` still throws when `systemctl --user is-enabled` ultimately reports a DBus/user-bus transport failure. Main has scoped handling that treats known bus-unavailable `is-enabled` probe errors as non-fatal during install/onboarding, but globally returning `false` from `isLoaded` would change other service lifecycle callers such as uninstall/reset.

## Best Possible Solution

Do not merge the one-line global false-return change as-is. Keep `isSystemdServiceEnabled` fail-loud for unresolved transport errors, continue handling known bus-unavailable probes only in install/onboarding paths, or introduce a richer service-state result that distinguishes disabled, unknown/unreachable, and installed states. Add regression coverage for the intended install behavior and for uninstall/reset not skipping cleanup when the systemd user bus is unreachable.
