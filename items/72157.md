## Summary

Keep #72157 open. The report is new, concrete, and targets a real WebChat keydown surface on current `main`: the Enter handler only gates composition and connection state, while the send button separately disables during `sending`. I found app-layer behavior that may already prevent the exact duplicate by clearing `chatMessage` and setting `chatRunId`/`chatSending` before the network await, but there is no focused rapid-Enter regression test proving the UI path, and adding the reporter's proposed `stream` guard would conflict with existing intentional queued-follow-up behavior.

## Best Possible Solution

Add a focused Control UI/WebChat regression test that dispatches two rapid plain-Enter keydowns before a rerender and asserts only one `chat.send` for the same draft. If it fails, fix the composer path with a narrow duplicate-submission guard or live draft/state check that preserves deliberate queued follow-up behavior for distinct messages during active runs.
