## Summary

Keep open. Current main still builds browser snapshots from the page/accessibility tree rather than the current scrolled viewport: the managed-browser path routes default snapshots to `snapshotAiViaPlaywright`, which calls `page.ariaSnapshot({ mode: "ai" })` and then only applies URL appending/ref storage/character truncation. The ARIA fallback similarly calls CDP `Accessibility.getFullAXTree`. The docs point users to `--labels`/screenshots for viewport evidence, but there is no implemented viewport-position-aware snapshot mode that would satisfy the reported behavior after `window.scrollTo(...)`.

## Best Possible Solution

Keep the issue open and implement a browser-owned viewport-aware snapshot path or explicit snapshot option. The likely change belongs in `extensions/browser/src/browser/routes/agent.snapshot.ts` and `extensions/browser/src/browser/pw-tools-core.snapshot.ts`: capture current scroll/viewport bounds, filter or prioritize visible nodes after scroll, and add a regression test that scrolls a long page before snapshotting. Docs should then clarify the difference between full tree snapshots, viewport snapshots, and labeled screenshots.
