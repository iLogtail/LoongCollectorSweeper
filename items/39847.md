## Summary

Keep open. Current `main` still has the reported gap: outbound reply normalization and channel delivery never call `stripInboundMetadata()`, while the inbound prompt builder still emits the exact metadata sentinels the report described.

## Evidence

- **Inbound metadata is still generated with the reported sentinels:** `buildInboundUserContextPrefix()` still prepends `Conversation info (untrusted metadata):`, `Sender (untrusted metadata):`, and related blocks into message text.
  - file: [src/auto-reply/reply/inbound-meta.ts:236](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/auto-reply/reply/inbound-meta.ts#L236)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Sanitizer exists and strips those sentinels plus timestamps:** `stripInboundMetadata()` is implemented to remove the inbound metadata sentinel blocks and leading timestamp prefix, matching the issue description.
  - file: [src/auto-reply/reply/strip-inbound-meta.ts:183](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/auto-reply/reply/strip-inbound-meta.ts#L183)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Outbound normalization still does not invoke the sanitizer:** `createOutboundPayloadPlanEntry()` parses `payload.text` with `parseReplyDirectives(payload.text ?? "")` and never calls `stripInboundMetadata()`. The file imports `parseReplyDirectives` but not `stripInboundMetadata`.
  - file: [src/infra/outbound/payloads.ts:2](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/infra/outbound/payloads.ts#L2)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Delivery entrypoint still routes through the unsanitized plan:** `normalizeReplyPayloadsForDelivery()` is only `projectOutboundPayloadPlanForDelivery(createOutboundPayloadPlan(payloads))`, so the missing strip step remains on the main delivery path.
  - file: [src/infra/outbound/payloads.ts:347](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/infra/outbound/payloads.ts#L347)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Final channel delivery only applies handler text sanitization:** `normalizePayloadsForChannelDelivery()` only runs `handler.sanitizeText()` / `handler.normalizePayload()` and does not apply metadata stripping as a safety net before send.
  - file: [src/infra/outbound/deliver.ts:382](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/infra/outbound/deliver.ts#L382)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **No regression coverage for echoed inbound metadata on outbound normalization:** Current `normalizeReplyPayloadsForDelivery` tests cover directives, silent replies, and relay placeholders, but no case asserts stripping `Conversation info` / `Sender` metadata from outbound text.
  - file: [src/infra/outbound/payloads.test.ts:32](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/infra/outbound/payloads.test.ts#L32)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
