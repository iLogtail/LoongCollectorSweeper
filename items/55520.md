## Summary

Keep open. Current main has partial mitigations, especially Z.AI default thinking-off payloads, but it does not implement the issue's central generic fix: OpenAI-compatible streaming still classifies non-empty `reasoning_content`/`reasoning`/`reasoning_text` as `thinking`, and normal reply extraction still ignores thinking-only assistant content for visible answers. The provided related items address narrower delivery or provider-control cases, not a canonical fix for this multi-provider reasoning/text misclassification report.

## Best Possible Solution

Keep the issue open and implement a conservative OpenAI-compatible reasoning-only recovery path. The likely fix belongs around `src/agents/openai-transport-stream.ts` plus final run/payload handling: keep provider reasoning fields classified as thinking, add regression coverage for `delta.content === ""` with non-empty `reasoning_content`, and decide per provider whether to disable provider thinking, retry for a visible answer, or surface a clear error rather than silently sending an empty reply or leaking chain-of-thought as text.
