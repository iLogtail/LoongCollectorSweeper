## Summary

Keep open. The issue is fresh, detailed, and reports a real remaining entry/bootstrap boundary: current main still has a main-only `src/entry.ts` prelude before delegating to `runCli()`, while provider code/tests/docs show Codex OAuth image generation itself is supported. I found no targeted fix, duplicate, or live/regression proof that the Linux entry-path fetch failure is resolved on current main.

## Best Possible Solution

Keep the issue open and investigate the entry bootstrap/runtime boundary directly. The best fix is to make `dist/entry.js` invocation and direct `runCli(argv)` share the same effective network and process state for `openclaw infer image generate`, then add a regression or smoke test that exercises the entry main path, not only the provider and direct `runCli()` path. Focus on entry-only side effects: respawn flags, compile cache, warning filter, profile/container argv normalization, env/proxy setup, and global Undici/fetch behavior on Linux dual-stack hosts with unreachable IPv6 and working IPv4.

## Evidence

- **issue_differential_diagnosis:** The supplied issue body reports that raw fetch, guarded `postJsonRequest`, direct OpenAI image provider `generateImage`, and direct `runCli(argv)` all succeed on the same Ubuntu host, while `openclaw`, `openclaw.mjs`, and direct `dist/entry.js` main execution fail with `TypeError: fetch failed`. That is concrete, current reproduction data pointing at entry/bootstrap equivalence rather than generic Codex OAuth, provider, or backend failure.
- **entry_main_path_still_has_distinct_bootstrap:** Current main still performs main-module-only setup before `runCli`: it sets process title, installs the OpenClaw exec marker and warning filter, normalizes env, enables compile cache, may respawn, normalizes argv/profile/container state, mutates `process.argv`, and only then calls `runMainOrRootHelp(process.argv)`. This is not yet a no-op wrapper around direct `runCli(argv)`.
