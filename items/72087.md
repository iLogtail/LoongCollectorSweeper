## Summary

Keep #72087 open. The report is new, detailed, and maps to a real distinction in current main between the CLI entry bootstrap path and direct runCli/provider execution. Current main has nearby changes since v2026.4.24, including awaiting runMainOrRootHelp and moving Codex image Responses traffic to gpt-5.5, but I did not find source, tests, docs, or history that specifically proves the reported Linux entry-path fetch failure is fixed or obsolete.

## Best Possible Solution

Keep the issue open and reproduce against a built current-main package via the real openclaw.mjs/dist entry on a Linux host with Codex OAuth and the same dual-stack/broken-IPv6 conditions. If it reproduces, fix the entry bootstrap or process-global network/dispatcher interaction and add regression coverage that compares packaged entry invocation with direct runCli/provider execution for the OpenAI Codex image route. If current main no longer reproduces, close later with explicit packaged-entry live proof and the fixing commit.
