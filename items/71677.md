## Summary

Keep #71677 open. Current main still handles Feishu `card.action.trigger` through a `Promise<void>` path that awaits the synthetic command dispatch and returns no immediate card callback payload, while tests and docs cover duplicate-token mitigation/streaming but not instant disabled-button card updates.

## Best Possible Solution

Keep the issue open and implement this in the bundled Feishu plugin: extend the `ocf1` interaction contract with an opt-in disable-state field or equivalent typed metadata, return an immediate Feishu card callback response for `card.action.trigger` while dispatching the synthetic command asynchronously, preserve duplicate-token protection, and add focused tests for the returned WebSocket/webhook callback payload plus no-regression coverage for existing quick-action and approval cards.
