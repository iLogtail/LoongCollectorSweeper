## Summary

Keep #9637 open. Current main still has no accepted TUI plain-text/accessibility config, and the affected TUI/tool-display paths still emit emoji and Unicode glyphs without a config-controlled ASCII branch. The linked PR #9685 is useful prior art, but it is closed and unmerged; another user confirmed the screen-reader problem, and the latest maintainer comment already kept the issue open.

## Best Possible Solution

Keep the issue open and implement this in core TUI/config. Add a real ui.accessibility.plainText or equivalent config contract, regenerate config schema/help surfaces, document the user-facing option, load it during TUI startup, thread it through ChatLog, ToolExecutionComponent, selector/theme code, and tool summary formatting, and add focused tests for config validation plus ASCII/plain-text rendering. Use closed PR #9685 as prior-art context, but refresh the approach against current main.
