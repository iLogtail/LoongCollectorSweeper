## Summary

Keep #69916 open. Current main still has the native Telegram approval path described by the maintainer comment: explicit target forwarding preserves `targets[].accountId`, but per-account native approval handlers still decide handling through `shouldHandleTelegramExecApprovalRequest`, whose account binding only considers turn-source/session data and does not consult `approvals.exec.targets[].accountId`.

## Best Possible Solution

Keep #69916 open until a fix like #69936 or #70568 lands. The best implementation is to add account-target filtering before Telegram native approval delivery, so an explicit `approvals.exec.targets[].accountId` for Telegram makes only that account's native runtime handle the request, with a multi-account regression covering the exact fan-out scenario. Reconcile the open PR approaches rather than closing the issue prematurely.
