## Summary

Current main still leaves the reported Telegram native exec approval fan-out path open. The explicit approval forwarder preserves `approvals.exec.targets[].accountId`, but Telegram native approval runtimes are registered per bot account and still decide handling through `shouldHandleTelegramExecApprovalRequest(...)`. For Telegram/internal approval requests with no resolved turn-source or persisted session account, the matcher returns true for each eligible account and does not consult the explicit `approvals.exec.targets[].accountId` target. The provided context also has a maintainer comment confirming this is still open and two related fix PRs that are open, not merged.

## Best Possible Solution

Keep #69916 open until one maintainer-approved fix lands on main. The fix should scope Telegram native exec approval delivery by explicit `approvals.exec.targets[].accountId` when that target is unambiguous, or otherwise reject ambiguous multi-account native handling, with a regression test where multiple Telegram accounts allowlist the same approver but only the configured target account sends the approval prompt.
