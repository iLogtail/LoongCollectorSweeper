## Summary

Current main still appears affected. The Control UI starts with `chatRunId` null after a page load, clears run state again on gateway hello/reconnect, and renders Stop/Queue only when `chatRunId` is truthy. Incoming delta events update the stream text but do not re-associate `payload.runId` into `chatRunId`, so the reported refresh-state gap remains actionable.

## Best Possible Solution

Keep the issue open and fix the Control UI bootstrap/reconnect path. A good fix would derive an abortable active-run state from a current gateway/session source, or re-associate `chatRunId` from stream events, then render Stop/Queue after refresh and call `chat.abort` by session key when no exact run id is available. Add a regression test that simulates page-load state with an active/running session and asserts Stop + Queue are shown while New session + Send are not.
