## Summary

Current main still has the reported Volcengine/BytePlus catalog problem. At the target main SHA, Kimi K2.5 remains `reasoning: false` with `maxTokens: 4096`, every shared Volc coding model still uses `maxTokens: 4096`, and the Volcengine/BytePlus providers consume that shared catalog. Together still defines Kimi K2.5 as `reasoning: true` with `maxTokens: 32768`, so the inconsistency described in #54149 remains. The provided related item #54157 is a distinct cache-pricing issue, not a canonical duplicate or superseding fix.

## Best Possible Solution

Keep this open. The likely fix belongs in `src/plugin-sdk/volc-model-catalog-shared.ts`, with Volcengine/BytePlus catalog tests that lock the verified `reasoning`, `contextWindow`, and `maxTokens` values. Scope the patch to vendor-verified models if only Kimi K2.5/Kimi thinking can be confirmed, and update provider docs if user-visible catalog tables need output-limit or reasoning columns.
