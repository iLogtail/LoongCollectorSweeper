## Summary

Keep #62790 open. Current main still declares the WebChat draft as Lit reactive state and normal textarea input still assigns every draft change back through the host render path, so the requested keystroke render fix is not implemented. The linked #61660 is a closed predecessor rebased into this PR, not a canonical replacement. Greptile's supplied review is useful positive signal for the PR direction, but this remains active work rather than cleanup-closeable.

## Best Possible Solution

Keep #62790 open for maintainer review or rework. The best fix is to decouple ordinary WebChat textarea typing from Lit host reactivity and any draft-only host `requestUpdate()` calls, while preserving programmatic draft sync for send failure restore, abort clear, input history, slash commands, STT, attachments, token count, and run controls. Add a focused regression test that non-slash draft typing does not rebuild the chat thread/message groups, then land #62790 or an equivalent narrow fix.
