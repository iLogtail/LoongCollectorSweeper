## Summary

Keep open. Current `main` still appears to have the underlying `chat.abort` cleanup gap, but this PR's fix is not safe as written: it would apply session-scoped queue / embedded-run cleanup in paths that are intentionally run-scoped or owner-filtered, so one abort could stop other active runs or another device's run on the same session.

## Evidence

- **chat_abort_on_main_still_only_aborts_controller_runs:** On current `main`, the `chat.abort` handler aborts authorized controller runs and persists partials, but neither the sessionKey path nor the runId path calls `abortEmbeddedPiRun`, `clearSessionQueues`, or `stopSubagentsForRequester` before responding.
  - file: [src/gateway/server-methods/chat.ts:1976](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/gateway/server-methods/chat.ts#L1976)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '1750,2045p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **session_abort_authorization_is_intentionally_owner_scoped:** The current contract explicitly leaves another device's run alive on the same session: `chat.abort` by `sessionKey` aborts `run-mine` but keeps `run-other` when ownership differs. Any unconditional session-wide cleanup after a partial abort would break that isolation.
  - file: [src/gateway/server-methods/chat.abort-authorization.test.ts:84](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/gateway/server-methods/chat.abort-authorization.test.ts#L84)

  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **multiple_runs_can_share_one_session:** Tests on current `main` create two active runs (`run-a`, `run-b`) for the same `sessionKey` and same `sessionId`, and session-scoped abort expects both run IDs. That means a targeted `runId` abort can coexist with other active work on the same session.
  - file: [src/gateway/server-methods/chat.abort-persistence.test.ts:151](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/gateway/server-methods/chat.abort-persistence.test.ts#L151)

- **cleanup_helpers_are_session_scoped_not_run_scoped:** `clearSessionQueues` clears follow-up queues and command lanes by session keys/session IDs, and `abortEmbeddedPiRun` aborts by `sessionId`. Those helpers do not accept a `runId`, so adding them to `chat.abort(runId)` would widen a targeted abort into session-wide cleanup.

## Risks / Open Questions

- Current `main` still appears to leave embedded Pi runs, sub-agents, and queued follow-ups alive when webchat uses `chat.abort`, so the original bug is likely still real.
- Merging the PR as written could let a single `runId` abort or a partially authorized session abort cancel unrelated work sharing the same session lane.
