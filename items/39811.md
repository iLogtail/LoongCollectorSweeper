## Summary

Current main only partially addresses #39811. Fallback visibility is better now through lifecycle events, persisted fallback state, `/status` output, and verbose-mode channel notices, but `openclaw config validate` still does not validate configured model refs against provider catalogs or upstream availability. The schema and type surface continue to accept arbitrary model strings, so the core misconfiguration class remains open.

## Best Possible Solution

Keep #39811 open. Preserve the shipped fallback lifecycle/status improvements, but continue tracking the missing config-time validation path: `openclaw config validate` should either validate configured primary/fallback refs against provider/plugin catalogs when feasible, or the product/docs should explicitly define why upstream model existence checks belong in `models list`/startup warnings instead. If maintainers accept the current validation boundary, split the remaining user-facing fallback notice behavior into a narrower follow-up.
