## Summary

Keep open. Current `main` still accepts arbitrary manually declared provider model IDs during config validation, and fallback on `model_not_found` only emits a logger warning rather than a clear user-facing notification.

## Evidence

- **Schema only checks model IDs are non-empty strings:** `ModelDefinitionSchema` accepts `id` and `name` as `z.string().min(1)` with no provider-catalog or upstream existence check.
  - file: [src/config/zod-schema.core.ts:307](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/config/zod-schema.core.ts#L307)
  - command: `nl -ba src/config/zod-schema.core.ts | sed -n '307,345p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Config validation is local/schema validation only:** `validateConfigObjectRaw` runs legacy checks, `OpenClawSchema.safeParse`, duplicate-agent-dir checks, avatar checks, and tailscale bind checks; it does not query provider model catalogs or verify configured model IDs upstream.
  - file: [src/config/validation.ts:573](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/config/validation.ts#L573)

## Risks / Open Questions

- The report bundles two behaviors: config-time upstream validation and runtime/user-surface fallback notification. Current main only partially addresses the second via logs, so closing would hide a still-open product gap.
- Some providers intentionally support manual model declaration, so any eventual fix likely needs provider-specific validation semantics rather than a blanket reject of all non-catalog IDs.
