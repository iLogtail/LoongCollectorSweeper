## Summary

Keep open. The current multi-account path appears better than the 2026.4.10 report, but the issue is recent, the reporter's linked PR #64712 is still open, and current main still has code paths that can synthesize or report a `default` account when Feishu account discovery falls back to the default id. That is not strong enough evidence for conservative auto-close.

## Best Possible Solution

Keep the issue open and let a maintainer decide whether to land or revise #64712. A robust fix should add a targeted Feishu regression around gateway restart/status with multiple configured accounts and no explicit `default`, then either suppress implicit default status rows where they are only read-only fallbacks or document and validate when `channels.feishu.defaultAccount` may legally point at an unlisted account.
