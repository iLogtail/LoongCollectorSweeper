## Summary

Keep open. Current main still has Feishu full registration without an idempotency/dedup boundary, and the legacy single-account promotion path still treats top-level Feishu groupPolicy as a movable account key when named accounts exist. The PR also has a substantive unresolved review concern: its module-level guard would be unsafe across activating reloads, so the work is not ready to close as implemented or superseded.

## Best Possible Solution

Keep the PR or a maintainer follow-up open, but revise the fix before merge: preserve Feishu top-level policy by declaring/auth-scoping Feishu named-account promotion keys and covering the early legacy migration path, and solve duplicate Feishu registration with registry/load-scoped idempotency or loader-level cache behavior rather than a process-lifetime module flag. Keep focused regression tests for repeated full registration and named-account Feishu configs with top-level groupPolicy.
