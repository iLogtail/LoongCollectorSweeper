## Summary

Current `main` does not contain this startup-hardening change. Relative `OPENCLAW_STATE_DIR` is still resolved from the current process cwd on each call, and neither gateway startup entrypoint pins explicit overrides first, so this PR still represents unshipped work and should stay open.

## Evidence

- **State-dir resolution is still cwd-sensitive on main:** `resolveStateDir()` still reads `OPENCLAW_STATE_DIR` at call time and sends it through `resolveUserPath()`. That helper ends in `path.resolve(trimmed)`, so a relative override is reinterpreted against whatever the process cwd is at the moment of each call.
  - file: [src/config/paths.ts:60](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/config/paths.ts#L60)

  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **CLI gateway startup does not normalize overrides up front:** `runGatewayCommand()` begins with profile/dev handling and then continues into gateway startup. There is no `normalizeStateDirEnv(process.env)` call or equivalent env pinning at function entry.
  - file: [src/cli/gateway-cli/run.ts:358](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/cli/gateway-cli/run.ts#L358)
