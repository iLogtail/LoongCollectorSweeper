## Summary

Current main still runs BOOT.md from the boot-md gateway startup hook whenever the hook receives gateway:startup, and there is no sentinel/idempotency guard in the boot-md path. The docs currently describe BOOT.md as a startup checklist and distinguish BOOTSTRAP.md as the one-time first-run ritual, so this is not implemented-on-main and does not fit an auto-close reason. Related ClawSweeper reports cover adjacent boot-md dispatch/delivery issues, but not this one-shot/idempotency request.

## Best Possible Solution

Keep this open for maintainer/product triage. The likely paths are either implement an explicit once-per-workspace/per-agent mode with a documented reset mechanism and regression tests, or clarify/close manually as by-design by pointing users to BOOTSTRAP.md for first-run rituals and BOOT.md for recurring startup checks.
