## Summary

Keep #54413 open. Current main has important adjacent hardening for Discord outbound delivery, including shared outbound queueing, delivery diagnostics, and transient Discord API retry. However, I did not find source/test/history evidence that the exact reported silent drop after a terminated run plus successful retry is fixed. The Discord final delivery callback still has an abort path that returns before calling `deliverDiscordReply`, which would not produce the delivery error logging/retry behavior the issue asks for.

## Best Possible Solution

Add a targeted Discord regression for a terminated or aborted first attempt followed by a successful retry where the final assistant payload exists. The fix should ensure the final reply either reaches `deliverDiscordReply`/shared outbound delivery or records a durable delivery failure and queue entry; avoid silent returns after a final reply has been produced.

## Evidence
