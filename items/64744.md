## Summary

Keep open. Current main has adjacent hardening, but the reported default Codex harness initialize hang is not proven fixed: `initialize()` still has no per-request timeout, the default app-server client factory still calls the shared client without `timeoutMs`, and the run-attempt startup guard uses the full agent timeout rather than the Codex app-server `requestTimeoutMs`. The open #66251 tracker references #64744 as a child item, not as a superseding canonical fix.

## Best Possible Solution

Keep #64744 open and fix the Codex harness startup path directly: forward `appServer.requestTimeoutMs` into the default client factory/shared initialize call, apply bounded timeouts to `thread/start` and `thread/resume`, close/reap the app-server child on startup timeout, and add a regression test for the normal `codex/gpt-5.4` harness path with default config so it fails clearly within the configured app-server timeout instead of waiting for the full agent timeout.
