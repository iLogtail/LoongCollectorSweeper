## Summary

Keep open. Current main has adjacent compaction timeout and liveness safeguards, but the issue discussion includes a later Feishu reproduction where successful timeout compaction retried the prompt and re-executed already-completed message-tool sends. Current source still retries after successful timeout compaction without checking prior side effects, and the message send paths do not provide a persisted per-turn idempotency guard for that replay case.

## Best Possible Solution

Keep #43661 open and narrow the fix around retry safety for compaction recovery. The runner should not blindly retry after timeout compaction when the timed-out attempt already recorded user-visible side effects, or the message tool/outbound layer should carry a stable per-turn idempotency key/cache across compaction retries for text and media sends. Add focused coverage in `src/agents/pi-embedded-runner/run.timeout-triggered-compaction.test.ts` and message-tool/outbound tests for a timed-out attempt that already sent text/media, then compacts successfully.
