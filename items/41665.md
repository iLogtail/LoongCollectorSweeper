## Summary

Current `main` does not already implement this PR’s fix, and the report still matches the shipped metadata path. The core filter exists, but representative bundled skills on `main` still have unscoped Homebrew install entries, so Windows onboarding can still surface brew-only installers. The PR also carries a useful `xurl` metadata correction that is still absent on `main`.

## Evidence

- **Core OS-filter logic exists but depends on per-skill metadata:** `normalizeInstallOptions` filters install specs by `spec.os`, so the requested behavior only happens when each skill declares OS metadata.
  - file: [src/agents/skills-status.ts:106](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/src/agents/skills-status.ts#L106)
  - sha: [a53fea39050b](https://github.com/openclaw/openclaw/commit/a53fea39050beba21ac5468d876c37323cfdd2e6)
- **Parser still rejects `npm`/`apt` install kinds on main:** `parseInstallSpec` only accepts `brew`, `node`, `go`, `uv`, and `download`, so shipped `kind: "npm"` and `kind: "apt"` entries are ignored today.
  - file: [src/agents/skills/frontmatter.ts:112](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/src/agents/skills/frontmatter.ts#L112)
  - sha: [a53fea39050b](https://github.com/openclaw/openclaw/commit/a53fea39050beba21ac5468d876c37323cfdd2e6)
- **Representative brew-only skill on main is still unscoped:** `skills/1password/SKILL.md` still ships a Homebrew install entry with no `os` filter.
  - file: [skills/1password/SKILL.md:11](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/skills/1password/SKILL.md#L11)
  - sha: [a53fea39050b](https://github.com/openclaw/openclaw/commit/a53fea39050beba21ac5468d876c37323cfdd2e6)

  - file: [skills/obsidian/SKILL.md:11](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/skills/obsidian/SKILL.md#L11)
