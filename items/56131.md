## Summary

Keep open. Current `main` still does not make reset archives discoverable through `sessions_history` or `sessions_list`. The only reset-archive fallback I found is in the bundled session-memory hook, not in the session retrieval surfaces named in the issue.

## Evidence

- **`sessions_history` has no archive-following path:** The tool schema only accepts `sessionKey`, `limit`, and `includeTools`, and the execute path forwards directly to `chat.history` with `{ sessionKey, limit }`. There is no `includeArchived`, `followResets`, or archive metadata path here.
  - file: [src/agents/tools/sessions-history-tool.ts:26](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/agents/tools/sessions-history-tool.ts#L26)
  - command: `nl -ba src/agents/tools/sessions-history-tool.ts | sed -n '20,280p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Gateway history reads the active transcript only:** `chat.history` loads the current session entry and reads messages via `readSessionMessages(sessionId, storePath, entry?.sessionFile)`. I found no chaining to prior reset fragments in this handler.
  - file: [src/gateway/server-methods/chat.ts:1891](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/gateway/server-methods/chat.ts#L1891)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '1874,1945p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Transcript candidate resolution excludes `.jsonl.reset.*` archives:** `readSessionMessages` picks the first existing path from `resolveSessionTranscriptCandidates(...)`. That candidate builder enumerates current/stale transcript paths and legacy locations, but not reset-archive filenames, so archived fragments are not discoverable through the normal history read path.
  - file: [src/gateway/session-transcript-files.fs.ts:70](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/gateway/session-transcript-files.fs.ts#L70)
  - command: `nl -ba src/gateway/session-transcript-files.fs.ts | sed -n '60,115p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Reset fallback exists only in a separate hook-specific path:** The bundled session-memory hook has a dedicated helper that falls back to the latest `.jsonl.reset.*` file when the active transcript is empty, and there is a focused test for that behavior. That shows archive recovery was added in one subsystem, not in `sessions_history`/`sessions_list`.
  - file: [src/hooks/bundled/session-memory/transcript.ts:64](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/hooks/bundled/session-memory/transcript.ts#L64)
  - command: `nl -ba src/hooks/bundled/session-memory/transcript.ts | sed -n '50,95p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Docs still describe `sessions_history` as a bounded current-session recall view:** The docs describe truncation/redaction behavior and explicitly say to inspect the transcript file on disk for exact transcript contents. They do not mention reset-archive discovery, previous fragments, or archive metadata on the session tools.
