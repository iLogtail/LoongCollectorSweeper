## Summary

Keep #56131 open. Current main still archives reset transcripts as .reset files, but sessions_history ultimately reads only the current session entry through chat.history/readSessionMessages and does not expose reset archives, previous fragments, or archive metadata. Related PR #60409 is still open and is described as a partial symptom fix, not shipped main behavior.

## Best Possible Solution

Keep the issue open until a core gateway/session-tool change lands that makes reset archives discoverable from the logical session. The best fix should add archive-chain recovery or explicit includeArchived/followResets behavior to chat.history/sessions_history, return metadata when archived fragments exist, and add regression tests for reset followed by history/continue recovery and exec-completion provenance across the reset boundary.
