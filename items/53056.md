## Summary

Keep #53056 open. Current main has not implemented the linked ampersand-preservation fixes: Slack outbound formatting still converts `&` to `&amp;`, and tests still assert that behavior. The related PRs #53091 and #53144 were closed unmerged, and the issue discussion itself says the proposed removal of ampersand escaping needed reconsideration because Slack documents `&`, `<`, and `>` as control characters that should be entity-escaped exactly once. The remaining work is to reproduce the live/display pipeline and identify where `&#38;` is introduced, not to auto-close the report.

## Best Possible Solution

Keep the issue open and ask for or perform a live Slack send/readback reproduction that compares raw `R&D`, pre-escaped `R&amp;D`, and already-escaped entity inputs through the actual OpenClaw Slack send and preview/edit paths. The eventual fix should preserve Slack’s documented exactly-once escaping contract, normalize only the path that is double-encoding, and add regression coverage in `extensions/slack/src/format.test.ts`, `extensions/slack/src/send.blocks.test.ts` or the relevant dispatch/preview test for the confirmed path.
