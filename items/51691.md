## Summary

Keep #51691 open. Current main has useful building blocks: shared gateway-backed sessions, session.message WebSocket events, and session routing/identity linking. But the requested real-time Telegram <-> TUI sync is not fully implemented: the TUI does not subscribe to or handle session.message/sessions.changed updates, docs still say history is not fully synced back to every client, and main-session UI sends intentionally do not inherit external delivery routes. This is a core TUI/gateway behavior gap, not a ClawHub-only item.

## Best Possible Solution

Keep the issue open and implement this as a core TUI/gateway feature: have the TUI subscribe to session index/message updates, update the active transcript and session/inbox UI when Telegram or another channel writes to a shared session, and define explicit opt-in reply routing for channel-scoped or main-session handoff. Reuse the existing sessions.subscribe/session.message primitives rather than inventing a new transport, and keep delivery routing explicit enough to avoid cross-channel leaks.

## Risks / Open Questions

- Implementing this by simply allowing UI clients to inherit main-session external routes could leak replies from shared main sessions into the wrong channel or person. The fix should preserve the existing explicit delivery safeguards.
- The existing gateway primitives make the implementation feasible, but the product behavior still needs a clear TUI UX for active-session updates, channel badges/inbox state, and explicit reply routing.
