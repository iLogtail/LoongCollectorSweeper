## Summary

Keep open. Current `main` still sends model-only `thinkingDefault` values in gateway session rows/defaults, while the runtime chat path resolves `agents.defaults.thinkingDefault`. The Control UI dropdown reads those gateway values first, so the reported `Default (off)` label mismatch remains a real current-code inconsistency. The timeline also fits that reading: the UI-side PR was closed, but the gateway-side PR referenced from the issue is still the missing piece.

## Evidence

- **Control UI uses session/default payload thinkingDefault for the dropdown label:** `resolveChatThinkingSelectState()` builds `Default (<level>)` from `activeRow?.thinkingDefault` first, then `state.sessionsResult?.defaults?.thinkingDefault`, and only falls back to model-derived detection after that.
  - file: [ui/src/ui/chat/session-controls.ts:208](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/ui/src/ui/chat/session-controls.ts#L208)

- **Gateway defaults still ignore config-level thinkingDefault:** `getSessionDefaults()` populates `thinkingDefault` with `resolveThinkingDefaultForModel(...)`, which does not read `cfg.agents?.defaults?.thinkingDefault`. That means the UI default payload is still model-only on current `main`.
  - file: [src/gateway/session-utils.ts:1041](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/gateway/session-utils.ts#L1041)
