## Summary

Keep #51832 open. Current main has useful Slack self-event filtering, but it only drops events from the same Slack bot account. The reported cascade uses two separate Slack bot accounts in one shared channel with `allowBots: true`, so sibling bot error messages can still enter the normal Slack inbound path. Current error handling also still emits visible OAuth/model-login failure replies for known auth failures, and there is no Slack-channel error dedupe or auth-failure cooldown covering the reported mechanism.

## Best Possible Solution

Keep the issue open and implement a bounded Slack/core guard. The best fix is to mark or classify OpenClaw-generated failure replies and suppress them as triggers across configured sibling Slack bot accounts, while preserving intentional inter-agent bot traffic. Add regression coverage for two Slack accounts in one channel with `allowBots: true` and a visible auth/OAuth failure, plus either per-channel identical-error dedupe or per-agent auth-failure cooldown as defense in depth.
