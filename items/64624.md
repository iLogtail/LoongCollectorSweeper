## Summary

Current main still leaves the requested feature unimplemented. WhatsApp reconnect handling unconditionally queues both connected and disconnected system events into the resolved agent route session, the shared queue only suppresses consecutive duplicate text, and the formatter turns queued entries into `System:` prompt lines. Config/docs expose health-monitor restart controls, but not a notification threshold, verbosity setting, or surface-aware filter for transient channel status events.

## Best Possible Solution

Keep the issue open. The likely fix belongs in the WhatsApp channel/runtime system-event path, with a documented config contract such as per-channel `statusEvents`/verbosity or `notifyAfterMs`, plus tests covering short reconnect suppression and routing so unrelated sessions do not receive transient channel lifecycle noise.
