## Summary

Keep open. Current `main` does not implement this noise suppression, and the PR's blanket `WhatsApp gateway disconnected...` filter would hide the only queued prompt-visible channel-health signal for logged-out/conflict/max-attempt-stop cases. The prior PR review comments correctly identified a real regression risk, so this is not conservative-close material.

## Evidence

- **Current prompt compaction on main does not suppress WhatsApp gateway banners:** `drainFormattedSystemEvents` currently filters periodic/heartbeat noise and trims `Node:` lines, but has no WhatsApp-specific suppression path.
  - file: [src/auto-reply/reply/session-system-events.ts:22](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/auto-reply/reply/session-system-events.ts#L22)
  - command: `nl -ba src/auto-reply/reply/session-system-events.ts | sed -n '1,80p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **WhatsApp monitor enqueues disconnect banner before emitting stop guidance separately:** The monitor enqueues `WhatsApp gateway disconnected (status ...)` as a system event, then emits logged-out/conflict/max-attempt guidance via `runtime.error`. A blanket disconnect filter would remove the queued prompt-visible event while the actionable text is not enqueued into system events.
