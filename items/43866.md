## Summary

Keep #43866 open. Current main still drops `accountId` on the `/hooks/agent` path before constructing the isolated cron job. The cron delivery layer now supports `delivery.accountId`, and the message-tool policy has changed since the report, but the hook bridge still does not preserve the explicit account requested by the caller. The related open PR #43951 may be the right vehicle, but the bug is not implemented on main.

## Best Possible Solution

Keep the issue open and land a focused hook bridge fix: add a normalized `accountId` field to direct `/hooks/agent` payloads, hook mappings if intended, idempotency replay scope, `HookAgentDispatchPayload`, and `dispatchAgentHook()` so it writes `delivery.accountId`. Add regression tests proving `/hooks/agent` with `accountId` creates a cron delivery carrying that account, and update the public hooks docs.
