## Summary

Keep open. Current main has related startup-performance work and a plugin-level deferred full-load hook, but it does not implement the requested `channels.lazyConnect` behavior: configured channel accounts are still started during Gateway sidecar startup, readiness remains gated on sidecars by default, and there is no consumed `channels.lazyConnect` config or registered-vs-connected lifecycle state.

## Best Possible Solution

Keep the issue open and scope an explicit lazy channel lifecycle design. The likely path is an additive config/manifest contract that lets channel plugins declare whether `startAccount` can be deferred, makes `channels.lazyConnect` or a successor setting control startup behavior, starts/connects on first outbound or inbound route where safe, reports a clear registered/not-connected state in `channels.status` and readiness, and adds regression tests proving configured channels are registered but not `startAccount`-started at boot when lazy mode is enabled.

## Risks / Open Questions

- The current plugin-level deferred full-load hook could be mistaken for this feature, but it does not provide the requested global default, first-message connect semantics, or status-state split.
- Implementing this naively can break webhook channels, native slash/HTTP callbacks, outbound delivery, readiness semantics, and channel health monitoring unless each channel declares whether lazy connection is safe.
