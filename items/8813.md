## Summary

Keep #8813 open. Current main still emits read-based Skills guidance unconditionally, even though registered plugin tool names are passed into system-prompt construction and normalized into `availableTools`. Tests and public docs still lock in the read-SKILL.md behavior, so the requested Skill-tool preference remains actionable in core prompt composition.

## Best Possible Solution

Keep #8813 open. The likely fix belongs in `src/agents/system-prompt.ts`: pass the normalized available-tool set and casing resolver into `buildSkillsSection`, prefer invoking a registered Skill tool when present, preserve the existing read-based fallback when absent, and update system-prompt tests plus public Skills/system-prompt docs for both behaviors.
