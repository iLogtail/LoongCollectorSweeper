## Summary

Keep open. This is a fresh, concrete report against the bundled QQ Bot channel, and current main still has an observable triage gap around QQ access-token endpoint timeouts. The code fetches the token from `https://bots.qq.com/app/getAppAccessToken` during connection startup, so a network timeout there plausibly leaves the channel disconnected. No close reason is strong enough yet: it is not stale, not a duplicate from the supplied context, not ClawHub-only, and not clearly outside the repo because OpenClaw can still improve diagnostics/docs or token-fetch timeout behavior.

## Best Possible Solution

Keep this open for QQ Bot maintainer triage. The best path is to review the reporter's curl/route/channel-status artifacts, confirm whether direct `bots.qq.com:443` reachability is the blocker, document the current Gateway `HTTP_PROXY`/`HTTPS_PROXY` workaround if applicable, and consider adding bounded token-fetch diagnostics/probe output around `extensions/qqbot/src/engine/api/token.ts` plus QQ troubleshooting docs for token endpoint timeouts.
