## Summary

Keep open. The report is recent, specific, and maps to the bundled QQ Bot token-fetch path. Current main still fetches the same Tencent token endpoint, and the token fetch path lacks the bounded timeout/diagnostic handling that the QQ REST client already has. The related ClawSweeper timeout items are different channel/runtime failures and do not supersede this QQ token-stage report.

## Best Possible Solution

Keep the issue open and handle it in the bundled QQ Bot plugin: add a bounded timeout to `TokenManager.doFetchToken()` consistent with `ApiClient`, preserve a clearer lastError/status for token-stage failures, and document or expose diagnostics for `bots.qq.com:443` reachability, IPv4/IPv6, proxy/env, DNS, and egress-policy cases. If maintainers decide the root cause is external service policy only, close later with a support-oriented docs/workaround answer.
