## Summary

Keep #71583 open. The reporter provided concrete QQ token-endpoint timeout evidence and current main still has a QQ Bot plugin reliability/docs gap: token acquisition is a raw hardcoded fetch with no explicit timeout, proxy-aware fetch path, endpoint override, or QQ-specific egress troubleshooting. The independent curl timeout strongly suggests QQ/network reachability is the root cause, but OpenClaw can still improve bounded failure behavior and diagnostics for this plugin-owned path.

## Best Possible Solution

Keep the issue open as a QQ Bot plugin reliability/docs item. The best path is plugin-owned: add an explicit bounded timeout and clearer error metadata around `TokenManager.doFetchToken()`, consider routing token fetch through the existing plugin/runtime proxy-aware fetch surface or documenting supported proxy/egress configuration, and update QQ Bot troubleshooting with `bots.qq.com` diagnostics and expected failure signatures. Core should stay generic unless the plugin needs a missing SDK fetch/diagnostic seam.
