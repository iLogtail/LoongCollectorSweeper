## Summary

Keep open. The report is fresh and concrete, and current `main` still depends on the same hard-coded QQ token fetch path before the QQ gateway can connect. I found no post-`v2026.4.23` fix, config override, or documented workaround that would make this safe to close as implemented, unreproducible, or stale.

## Evidence

- **Fresh issue with concrete repro and follow-up:** The issue was opened on 2026-04-25 with explicit repro commands, expected/actual behavior, and A/B uplink evidence; the follow-up comment offers more diagnostics immediately. That is enough to avoid `stale_insufficient_info`, and there is no contradictory discussion resolving it.
- **Current main still hard-codes the QQ token endpoint:** `TokenManager` still fetches the QQ app token from `https://bots.qq.com/app/getAppAccessToken` and surfaces fetch failures as network errors. This is the exact surface the report says is timing out.
