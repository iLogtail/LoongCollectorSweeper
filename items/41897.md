## Summary

PR #41897 should stay open. Current main still applies the 4,000-character error-detail cap uniformly to all non-OK web_fetch responses, including 401/403/407/451, and the current tests do not cover the requested access-denied truncation behavior. The PR is a focused two-file bug fix with targeted tests and prior review feedback already addressed in its latest head commit.

## Best Possible Solution

Keep PR #41897 open. The best path is for maintainers to rebase/review and merge the narrow status-aware cap, or independently implement the same behavior in src/agents/tools/web-fetch.ts with regression tests covering short caps for access-denied statuses and preserving the full cap for diagnostic statuses such as 500.
