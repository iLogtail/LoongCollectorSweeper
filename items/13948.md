## Summary

Keep #13948 open. Current main still models tool policy as whole-tool allow/deny, while `message` and `gateway` are multiplexed tools whose `action` is read only at execution time. The supplied issue discussion also includes a recent maintainer-thread Codex review keeping this open, and related gateway action-level requests (#54260, #46684) were closed back toward this canonical tracker rather than shipped as fixes.

## Best Possible Solution

Keep #13948 open as the canonical tracker. The likely product/code path is to extend config schema/types/docs and the shared tool-policy pipeline with action-aware matching for multiplexed tools such as `message` and `gateway`, define deny/allow precedence and `/tools/invoke`/Lobster semantics, and add regression coverage proving `message:send` or `gateway:config.patch` can be denied while read-only actions remain available.
