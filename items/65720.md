## Summary

Keep #65720 open. Current main serves the Control UI from the Gateway HTTP process and has service-manager restart support, but it does not implement an opt-in static fallback/maintenance page that remains reachable on the Gateway port after the Gateway process crashes. This is a core/supervisor resilience request, not a ClawHub/plugin item, because an in-process plugin would die with the Gateway.

## Best Possible Solution

Keep the issue open for maintainer product/design judgment. The strongest implementation path would be a supervisor-owned or sidecar-owned fallback mode for installed services, with a minimal static maintenance page, clear health/readiness semantics, and tests around crash/restart/port handoff. Document reverse-proxy or service-manager alternatives if the project decides not to own this in core.

## Risks / Open Questions

- A crash fallback cannot be implemented purely inside the Gateway process; the durable part must live in a service supervisor, sidecar, wrapper, or deployment layer that outlives the crashed process.
- Binding the same host/port for a fallback page can interact with Gateway auth, TLS, allowed origins, health probes, and restart port-race behavior, so it needs explicit product/security design rather than a small config toggle.
