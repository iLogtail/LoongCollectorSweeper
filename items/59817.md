## Summary

Keep open. Current main still classifies inbound Discord DMs solely from fetched channel info; when that lookup fails and `guild_id` is absent, the code falls through to channel routing and persists channel-shaped context, which matches the reported failure mode. The existing session-key normalization only helps when the context is already marked `ChatType: direct`, so it does not cover this channel-info failure path. The provided commenter also treated this as a Discord provider/session bug with manual cleanup workaround, not as evidence that it is fixed.

## Evidence

- **DM classification still depends on fetched channel type:** `preflightDiscordMessage` sets `isDirectMessage` only from `channelInfo?.type === ChannelType.DM` after `resolveDiscordChannelInfo`; if channel info is null after a fetch/network failure, a non-guild DM is not inferred as direct from the missing `guild_id` or sender context.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts:442](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/extensions/discord/src/monitor/message-handler.preflight.ts#L442)

  - sha: [b64bfc5d9abf](https://github.com/openclaw/openclaw/commit/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab)
- **Channel lookup failures become null:** `resolveDiscordChannelInfo` catches fetch errors, logs them, negative-caches `null`, and returns `null`; this is exactly the degraded channel-info/network condition described by the issue.
  - file: [extensions/discord/src/monitor/message-utils.ts:164](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/extensions/discord/src/monitor/message-utils.ts#L164)
