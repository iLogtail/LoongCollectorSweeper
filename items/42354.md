## Summary

Keep open. Current main does not implement the issue's suggested Feishu WebSocket constructor change, and the later issue comment points to a Lark/Feishu domain and transport-mode problem that current main still does not clearly handle for `domain: "lark"` plus default WebSocket mode. The open PR #45674 tracks one proposed fix, but current pinned Lark SDK types do not expose a `wsConfig` constructor parameter, so this should not be closed as implemented or simply delegated to that PR without maintainer/root-cause review.

## Best Possible Solution

Keep #42354 open and narrow it to the actual Feishu/Lark WebSocket failure modes. The best maintainer path is to reproduce with current main against Feishu CN and Lark international, add focused tests around `createFeishuWSClient()` domain selection and Lark/WebSocket transport handling, and implement either an actionable early error or webhook fallback for `domain: "lark"` if long connection is unsupported. Do not land a `wsConfig` constructor patch unless the pinned Lark SDK contract or a version bump proves that option is valid.
