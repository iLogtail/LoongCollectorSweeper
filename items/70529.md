## Summary

Keep #70529 open. It is recent, has no protected label but has an explicit maintainer comment saying to keep it open as the Desktop-specific variant, and current main still routes existing-session Chrome MCP through an ambient `npx chrome-devtools-mcp@latest` default unless users manually configure `mcpCommand`. The linked #62288 and #63926 overlap, but they do not supersede the packaged Desktop `spawn npx ENOENT` failure called out here.

## Best Possible Solution

Keep this issue open and fix the Desktop/browser runtime path so existing-session attach does not depend on an ambient `npx` in packaged Desktop environments. A good fix would bundle or explicitly resolve the Chrome MCP driver, or at least detect `ENOENT` and return a structured Desktop-specific diagnostic that points users to the managed `openclaw` profile and the login handoff path while #62288 and #63926 continue tracking the broader follow-ups.
