## Summary

Keep open. Current main still has the reported Mattermost split: direct messages are documented and resolved as non-threaded, but the outbound root resolver is kind-agnostic and can still forward a payload reply id into Mattermost `root_id`. The later comments and #65729 broaden this into configurable DM threading, but that related issue is open and does not represent a shipped fix for #53186.

## Best Possible Solution

Keep the issue open and fix it in the Mattermost plugin. The conservative path is to make the outbound root resolver or its call sites chat-kind aware so direct messages cannot turn a fallback `payload.replyToId` into `root_id` under the current documented contract, with a regression test for DM delivery. If maintainers prefer the later requested behavior, add an explicit Mattermost DM threading config surface, schema/docs, session/history handling, and tests instead of relying on the current accidental fallback.

## Risks / Open Questions

- The issue discussion now contains two product directions: preserve the documented flat-DM contract, or add explicit configurable DM threading. Closing this before that decision would lose a real Mattermost behavior gap.
- A narrow fix that strips all DM `payload.replyToId` values could also affect explicit reply directives in DMs, so the patch should include tests for implicit replies, explicit `[[reply_to_*]]` tags, preview/block streaming, and the chosen DM config semantics.
