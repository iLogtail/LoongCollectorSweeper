## Summary

Current main already prevents this failure mode: bootstrap/subagent message runs require explicit targets, Telegram target resolution rejects path-like/non-chat targets before any API call, and REACTION_INVALID is handled as a soft non-fatal result.

## Close Comment

Closing from a Codex review: current `main` already covers this well enough. Bootstrap/subagent runs now require explicit message targets (`src/agents/pi-embedded-runner/run/attempt.ts:558`, `extensions/codex/src/app-server/run-attempt.ts:620`), and the message tool enforces that guard for send/upload-style actions (`src/agents/tools/message-tool.ts:722`). On the Telegram side, outbound target resolution now rejects non-numeric and non-resolvable targets locally before any API call (`extensions/telegram/src/send.ts:301`, used by `extensions/telegram/src/send.ts:617`), with coverage in `extensions/telegram/src/send.test.ts:692` and `extensions/telegram/src/send.test.ts:734`. `REACTION_INVALID` is also downgraded to a soft warning/non-fatal result instead of surfacing as a hard tool failure (`extensions/telegram/src/send.ts:1081`, `extensions/telegram/src/action-runtime.ts:285`). The current main proof point is commit `60d892d700440a2e9df65a80cbe856f727437c1a`; this appears to be after `v2026.4.22`, so I am not attaching a shipped release tag.
