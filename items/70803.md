## Summary

Keep PR #70803 open. Current main still has the exact WhatsApp normalizer gap the PR fixes: runtime JID conversion accepts `@hosted.lid`, but `normalizeWhatsAppTarget` and `isWhatsAppUserTarget` still only recognize bare `@lid`. The related ClawSweeper items are adjacent WhatsApp/LID topics, not a canonical duplicate for this narrow hosted-LID target-normalization fix.

## Best Possible Solution

Keep PR #70803 open and proceed with maintainer review/CI for the narrow WhatsApp plugin fix. The likely correct implementation is the PR's scoped regex/test update in `extensions/whatsapp/src/normalize-target.ts` and `extensions/whatsapp/src/resolve-target.test.ts`, unless maintainers want a broader follow-up to reconcile LID-to-phone mapping semantics across target normalization and runtime identity resolution.
