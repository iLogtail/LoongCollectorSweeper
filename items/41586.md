## Summary

Current main still keeps raw `cron` owner-only for subagents, but the underlying workflow gap from #41586 is now covered by the subagent completion-delivery pipeline: completion notices go direct first, active requester sessions are steered immediately, and queued fallback now wakes heartbeat right away instead of waiting for the next natural tick.

## Close Comment

Closing from a Codex review: current `main` now covers the workflow this request was trying to unlock. Subagent completions are push-driven: completion delivery goes direct first (`src/agents/subagent-announce-dispatch.ts:88`), active requester sessions are steered immediately via `queueEmbeddedPiMessage(...)` (`src/agents/subagent-announce-delivery.ts:491`), dormant requesters get an immediate `agent` call with delivery routing (`src/agents/subagent-announce-delivery.ts:517`), and the docs now state that task completion triggers an immediate heartbeat wake instead of waiting for the next scheduled tick (`docs/automation/tasks.md:127`). `cron` itself is still owner-only (`src/agents/tools/owner-only-tools.ts:1`), so this landed as runtime completion-wake behavior rather than exposing raw `cron wake` to subagents. Best proof on `main` is commit `a903438707f11de28f65c33511ac1da9ee0641cb`; I could not verify a released tag containing it from this checkout.
