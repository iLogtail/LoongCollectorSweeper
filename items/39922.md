## Summary

Keep open. Current `main` has restart mitigations for explicit restart flows and subagent recovery, but it still does not implement durable automatic recovery for interrupted main-agent runs. Startup recovery only checks for a persisted restart sentinel and then runs subagent orphan recovery; I found no generic persisted pending-main-run record or startup scan that would resume an in-flight main run after crash/reboot/watchdog restart. The restart loop still drains active work, proceeds on timeout, and then resets lane state for the next lifecycle rather than restoring the interrupted main run.

## Evidence

- **Startup recovery is limited to restart sentinel + subagent recovery:** Gateway startup wires exactly two restart-related sidecars here: restart-sentinel handling and `scheduleSubagentOrphanRecovery()`. There is no corresponding startup pass for interrupted main-agent runs.
  - file: [src/gateway/server-startup-post-attach.ts:333](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/gateway/server-startup-post-attach.ts#L333)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Restart sentinel only runs when a sentinel file already exists:** `loadRestartSentinelStartupTask()` begins by consuming a restart sentinel file and returns `null` when none exists, so there is no generic recovery path for arbitrary restarts/crashes unless some earlier code explicitly wrote a sentinel.
  - file: [src/gateway/server-restart-sentinel.ts:307](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/gateway/server-restart-sentinel.ts#L307)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **The only automatic post-restart resume code is subagent-specific:** `subagent-orphan-recovery.ts` is explicitly scoped to 'subagent sessions' and resumes them by sending a synthetic `lane: "subagent"` message. This does not cover interrupted main-session runs.
  - file: [src/agents/subagent-orphan-recovery.ts:1](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/agents/subagent-orphan-recovery.ts#L1)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Restart path still aborts/proceeds instead of persisting and restoring the main run:** During restart the run loop drains active tasks, but if drain time expires it logs 'proceeding with restart'; on the next in-process iteration it calls `resetAllLanes()` to clear old queue state. That is recovery of bookkeeping, not recovery of the interrupted main run.
  - file: [src/cli/gateway-cli/run-loop.ts:166](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/cli/gateway-cli/run-loop.ts#L166)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Sentinel creation is limited to explicit restart/config/update flows:** A repo-wide search for `writeRestartSentinel(` shows call sites in chat `/restart`, gateway tool restart, config apply/patch, update flow, and config auto-recovery paths; there is no generic 'accepted main run' persistence hook.
  - command: `rg -n "writeRestartSentinel\(" src -g '!**/node_modules/**'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Current restart UX still assumes explicit continuation instructions, not automatic generic main-run replay:** The gateway tool description tells agents to pass a one-shot `continuationMessage` when restarting during a user task. That is a scoped restart-sentinel mechanism, not durable automatic recovery of any in-progress run after arbitrary restart causes.
  - file: [src/agents/tools/gateway-tool.ts:356](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/agents/tools/gateway-tool.ts#L356)
