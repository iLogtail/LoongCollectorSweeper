## Summary

Keep open. Current main has adjacent runtime-deps hardening and DeepSeek V4 replay handling, but the reported production symptom is gateway startup CPU/memory growth with DeepSeek V4 on 2026.4.24. The issue discussion and related PR both say the DeepSeek-specific root cause is not proven and should not be closed without logs/profile or a latest-main/next-release reproduction result.

## Best Possible Solution

Keep the issue open. Ask for a retry on current main or the next release with `openclaw --version`, the last gateway logs, whether logs repeat `[plugins] ... staging bundled runtime deps` or model-pricing refresh lines, and a CPU profile or stack sample from the pegged gateway. If logs show the runtime-deps staging loop, close/link it as the fixed #71818 class; otherwise investigate the DeepSeek V4 provider/wrapper startup path directly and add a targeted regression.
