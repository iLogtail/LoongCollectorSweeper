## Summary

Keep open. Current main does not implement `tools.exec.denyPatterns` or an equivalent command-regex deny guardrail. Existing controls can deny the whole `exec` tool or require host approval/allowlist policy, but they do not provide the PR’s proposed per-command regex block for otherwise allowed exec. The PR may need rebase and further review, but it still tracks useful unmerged core exec/config work.

## Best Possible Solution

Keep the PR open for maintainer/product review. If accepted, rebase it onto current main, add `denyPatterns` to the current config type/schema/generated docs flow, enforce it early in `createExecTool`, wire all exec call sites, document it as a guardrail rather than a sandbox, and add regression tests for global plus per-agent merging, options-path behavior, invalid regex handling, chat-bash wiring, and shell-segment or explicitly documented bypass semantics.
