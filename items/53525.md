## Summary

Keep open. Current main has generic safeguards for silent empty `stopReason="error"` turns, but I did not find a Qianfan/Baidu error payload normalization fix or regression coverage proving `error_msg`/`error_code` is surfaced to the user. The issue is recent and has a concrete transcript, so it is still actionable.

## Best Possible Solution

Keep the issue open for a targeted provider/error-boundary fix. Add Qianfan-compatible error extraction for `error_msg`, `msg`, `message`, nested `error.message`, and `error_code` on the OpenAI-compatible transport path or in a Qianfan-owned wrapper, then add focused regression coverage proving a Baidu error response becomes a visible assistant/user error instead of `error: null` plus `content: []`.
