## Summary

Keep open. Current `main` still contains the close/response race path this PR targets, and the PR discussion shows the follow-up commit addressed the one concrete review concern about preserving close diagnostics on real disconnects.

## Evidence

- **Current wrapper still lets `onClose` win before the awaited request continuation:** In `executeGatewayRequestWithScopes`, `ignoreClose` is only set after `await client.request(...)` resolves. If a normal close arrives in the same turn as the response, `onClose` still immediately calls `stop(new Error(formatGatewayCloseError(...)))`. There is no `requestInFlight`/deferred-close logic on `main`.
  - file: [src/gateway/call.ts:522](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/gateway/call.ts#L522)
  - command: `nl -ba src/gateway/call.ts | sed -n '500,570p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **Gateway client rejects pending requests before invoking `onClose`:** The WebSocket close handler flushes pending request errors via `flushPendingErrors(...)` and only then calls `this.opts.onClose?.(...)`. That ordering matches the PR review comment: a real mid-request disconnect should still surface request failure details unless the close is the benign post-response race.
  - file: [src/gateway/client.ts:303](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/gateway/client.ts#L303)

  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **Response handling can resolve before the async caller resumes:** Successful response frames synchronously call `pending.resolve(parsed.payload)` inside `handleMessage`. That means a server can send the response and then close immediately, leaving the `await client.request(...)` continuation in `call.ts` to run in a later microtask while `onClose` runs first.
