## Summary

Keep #72081 open. Current main does not print a tokenized dashboard URL; it logs only the clean URL and sends the tokenized URL only to clipboard/browser when allowed. In headless or clipboard-failure paths, the user is told to use the clean URL above, which matches the reported WSL-style failure. There is also a docs/UI mismatch: user-facing text still tells users to get or copy a tokenized URL from `openclaw dashboard --no-open`.

## Best Possible Solution

Keep the issue open and fix the dashboard access UX without reintroducing token leakage into default logs. The likely path is to update `src/commands/dashboard.ts` so headless/`--no-open`/clipboard-failure cases provide an actionable safe path, such as explicit opt-in token reveal or clear `openclaw config get gateway.auth.token` guidance, then align the Control UI strings and dashboard/remote-access docs with that policy.
