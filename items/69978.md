## Summary

Keep #69978 open. Current main still has the TUI streaming watchdog hardcoded to a local 30s default unless tests inject `streamingWatchdogMs`, no `/watchdog` command is registered or handled, and the event path can re-adopt the same run after a watchdog timeout and re-arm the warning. #68596 overlaps on configurability and #69081 overlaps on streaming-status desync, but neither fully covers the per-run duplicate-warning suppression plus TUI runtime command requested here.

## Best Possible Solution

Keep this open or explicitly fold it into a canonical watchdog UX issue. The concrete implementation likely belongs in the TUI surface: add a persisted local watchdog preference, register and handle `/watchdog [off|0|<duration>]`, plumb the setting into `createEventHandlers` dynamically or recreate handlers safely, and add a warned-run set cleared on final/abort/error/session changes with regression tests for same-run duplicate suppression.
