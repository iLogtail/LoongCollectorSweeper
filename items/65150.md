## Summary

Close as ClawHub/plugin-surface work. The useful goal, intercepting and observing tool calls, is already covered by the public plugin hooks (`before_tool_call` / `after_tool_call`), while adding new internal `tool:*` hooks would duplicate the supported extension path. The PR review also found the proposed blocking path was non-functional because internal hooks return void.

## Best Possible Solution

Close this PR and keep tool-call policy/observation on the supported plugin hook surface. A tool preflight/audit plugin can use `before_tool_call` and `after_tool_call` today and be published via ClawHub; any remaining gap should be filed as a focused plugin API design issue rather than adding parallel internal hooks.
