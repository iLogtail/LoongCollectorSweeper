## Summary

Current `main` does not implement account-scoped channel config form resolution, so the underlying bug is still live. This PR also has unresolved review concerns around schema fallback and account-path alignment, so it is not strong enough for conservative cleanup closure.

## Evidence

- **Current UI still resolves channel config only at the channel root:** `resolveChannelConfigValue` only accepts `(configForm, channelId)` and reads `configForm.channels[channelId]` or the legacy top-level fallback. There is no `accountId` parameter and no lookup under `channels.<provider>.accounts.<accountId>` on current `main`.
  - file: [ui/src/ui/views/channel-config-extras.ts:3](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/ui/src/ui/views/channel-config-extras.ts#L3)
  - command: `nl -ba ui/src/ui/views/channel-config-extras.ts | sed -n '1,220p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **Form schema, value, and patch paths are hard-coded to `channels.<id>`:** `renderChannelConfigForm` resolves the schema at `["channels", props.channelId]`, resolves values with `resolveChannelConfigValue(config, channelId)`, and patches back to `["channels", props.channelId]`. `ChannelConfigFormProps` also has no `accountId`, so current `main` cannot render/edit account-scoped channel config forms.
  - file: [ui/src/ui/views/channels.config.ts:8](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/ui/src/ui/views/channels.config.ts#L8)
  - command: `nl -ba ui/src/ui/views/channels.config.ts | sed -n '1,240p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **The UI already has default-account information but does not thread it into config forms:** `resolveDefaultChannelAccount` reads `snapshot.channelDefaultAccountId`, but channel cards still call `renderChannelConfigSection({ channelId, props })` with no account context. That means multi-account channels still render the root form even when the default account is known.
  - file: [ui/src/ui/views/channels.shared.ts:28](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/ui/src/ui/views/channels.shared.ts#L28)
  - command: `nl -ba ui/src/ui/views/channels.shared.ts | sed -n '1,220p'; nl -ba ui/src/ui/views/channels.discord.ts | sed -n '1,120p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **Review comments identified a real regression risk in the proposed patch:** The supplied PR review flagged that the patch scopes schema lookup to `channels.<id>.accounts.<accountId>` without falling back to the root schema. That would break channels whose runtime exposes a default account but whose schema remains top-level-only.
  - file: [ui/src/ui/views/channels.config.ts:101](https://github.com/openclaw/openclaw/blob/12544ee51092418398efdde41e3539a2dc2b02a3/ui/src/ui/views/channels.config.ts#L101)
  - sha: [12544ee51092](https://github.com/openclaw/openclaw/commit/12544ee51092418398efdde41e3539a2dc2b02a3)

That makes the schema-fallback review comment concrete, not theoretical.
  - file: [extensions/msteams/src/channel.ts:103](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/extensions/msteams/src/channel.ts#L103)

- **The status panel helper would also need account-aware plumbing:** `resolveChannelExtras` still reads extras via `resolveChannelConfigValue(params.configForm, params.channelId)`, and the agent status panel calls it without any account id. That aligns with the PR review comment that the patch is incomplete even if the form pathing is fixed.
