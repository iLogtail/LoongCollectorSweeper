## Summary

Keep open. Current main still lacks the requested ACP session-lane timeout behavior, so this is not implemented or obsolete, but the PR is not conservative cleanup: its timeout wrapper appears to let the per-session queue advance before the timed-out operation actually stops, and the branch also bundles unrelated public plugin-SDK and Slack changes.

## Evidence

- **Current main has no session-lane task timeout:** `SessionActorQueue.run()` on `main` just delegates to `KeyedAsyncQueue.enqueue()` with enqueue/settle hooks; there is no timeout path in the queue itself.
  - file: [src/acp/control-plane/session-actor-queue.ts:23](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/acp/control-plane/session-actor-queue.ts#L23)
  - command: `nl -ba src/acp/control-plane/session-actor-queue.ts | sed -n '1,220p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **A hung ACP actor task would still block the lane on main:** `initializeSession()` and other ACP operations go through `withSessionActor()`, which enqueues work on the per-session actor queue. A never-settling `ensureSession()` or similar op therefore still holds that lane indefinitely on current `main`.
  - file: [src/acp/control-plane/manager.core.ts:311](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/acp/control-plane/manager.core.ts#L311)
  - command: `nl -ba src/acp/control-plane/manager.core.ts | sed -n '230,380p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Serialization depends on the returned task promise:** `KeyedAsyncQueue` stores the per-key tail from the promise returned by `task` and advances cleanup from that tail. If a wrapper promise settles before the underlying operation finishes, same-key work can start early.
  - file: [src/plugin-sdk/keyed-async-queue.ts:14](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/plugin-sdk/keyed-async-queue.ts#L14)
  - command: `nl -ba src/plugin-sdk/keyed-async-queue.ts | sed -n '1,220p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **The PR timeout wrapper appears to detach the underlying operation:** In the PR patch, `runWithTaskTimeout()` rejects with `SessionActorQueueTimeoutError` on timer expiry but starts `params.op()` via `void params.op().then(...)`. That means the queue-facing promise can reject before the underlying op stops, which conflicts with the documented per-session lane guarantee and the queue contract above.
  - file: [src/acp/control-plane/session-actor-queue.ts:15](https://github.com/openclaw/openclaw/blob/7108b894ff1cf0d134d69aa47c4ae76e60f77457/src/acp/control-plane/session-actor-queue.ts#L15)
  - sha: [7108b894ff1c](https://github.com/openclaw/openclaw/commit/7108b894ff1cf0d134d69aa47c4ae76e60f77457)
- **The PR scope is broader than the ACP bug:** Besides the ACP timeout work, the PR adds new public plugin-SDK subpaths/exports in `package.json` and `scripts/lib/plugin-sdk-entrypoints.json`, plus a new writer export in `src/plugin-sdk/channel-pairing.ts` and Slack runtime rewiring across many files. That is materially broader than a conservative stuck-lane fix and should stay open for maintainer review rather than auto-close.

- Merging or auto-resolving this PR as-is risks breaking per-session serialization and expanding public plugin-SDK surface area without owner review.
