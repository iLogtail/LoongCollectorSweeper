## Summary

Current `main` has shipped first-party Windows gateway service management and docs that steer users to `openclaw gateway install` with Scheduled Tasks or a Startup-folder fallback, but the specific PM2 behavior in #55298 is not clearly documented as fixed or obsolete. PM2 is still explicitly recognized in runtime code, and there is no current doc note saying PM2 is unsupported or that `pm2 reload --update-env` requires a delete/start/save cycle, so this is not strong enough for a conservative close.

## Evidence

- **Windows docs now point users to first-party managed startup:** Native Windows guidance says managed startup should use `openclaw gateway install` / `openclaw gateway status --json`, with Scheduled Tasks first and a Startup-folder fallback.
  - file: [docs/platforms/windows.md:40](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/docs/platforms/windows.md#L40)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Gateway runbook documents Windows service flow, not PM2:** The Windows tab documents `openclaw gateway install/status/restart/stop` and names the managed supervisor as the `OpenClaw Gateway` Scheduled Task with Startup-folder fallback.
  - file: [docs/gateway/index.md:255](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/docs/gateway/index.md#L255)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **CLI now exposes built-in service lifecycle commands:** `addGatewayServiceCommands` registers `status`, `install`, `uninstall`, `start`, `stop`, and `restart` for launchd/systemd/schtasks, showing a first-party replacement for external process-manager workflows.
  - file: [src/cli/daemon-cli/register-service-commands.ts:49](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/cli/daemon-cli/register-service-commands.ts#L49)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Runtime still has explicit PM2 handling:** `isMainModule` still contains PM2-specific logic and comments for `pm_exec_path`, so PM2 is not clearly removed or declared unsupported.
  - file: [src/infra/is-main.ts:42](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/infra/is-main.ts#L42)
- **No current PM2 guidance found in docs/runbooks:** A repo search for `pm2`, `ecosystem.config`, `reload --update-env`, and `pm2-ecosystem` found only PM2 test references, not user-facing docs or recovery instructions.
  - command: `rg -n "pm2|ecosystem\.config|reload --update-env|pm2-ecosystem" docs src scripts README.md .github --glob '!**/dist/**' --glob '!extensions/**'`

## Risks / Open Questions

- Closing now would require inferring that the new built-in daemon flow supersedes the requested PM2 note, but the repo does not explicitly say that.
- Users who still manage `openclaw-gateway` with PM2 on Windows could still hit the reported behavior because there is no explicit workaround note on current `main`.
