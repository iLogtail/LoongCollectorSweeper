## Summary

Keep open. Current main still has the reported Synology Chat webhook behavior: `HEAD` falls into the non-POST 405 path, and successful POST handling still sends a 204 empty ACK. Existing tests also assert the old 204/405 behavior. Related PRs #53441 and #53522 propose fixes for this issue, but the provided GitHub context says both are open/unmerged, and current `origin/main` remains separate from their PR heads.

## Best Possible Solution

Keep #53439 open until a focused fix lands on main: handle Synology Chat `HEAD` probes with 200 OK, return `200 {"success":true}` for successful POST ACKs, and update `extensions/synology-chat/src/webhook-handler.test.ts` plus the integration assertions away from 204. PR #53441 is the narrowest linked implementation; PR #53522 also contains the same Synology fix but bundles unrelated changes.
