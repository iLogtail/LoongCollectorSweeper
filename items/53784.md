## Summary

Keep PR #53784 open. Current main still does not implement the PR title's behavior for ordinary MP3/WAV/M4A/FLAC/AAC outbound media: Feishu only routes native Opus/Ogg as `msg_type: "audio"`, while a current test explicitly expects MP3 with `audio/mpeg` to remain a generic file. The PR is also not directly merge-ready against current main because main now has a safer `audioAsVoice` transcode path, so the best outcome is maintainer-guided rework rather than cleanup closure.

## Best Possible Solution

Keep the PR open and rework it against current main. The maintainer path should decide the product boundary first: if ordinary audio attachments should become voice bubbles, update `resolveFeishuOutboundMediaKind`, `detectFileType`, and the existing MP3 fallback test accordingly; if only voice/TTS should be native audio, adapt the PR to the existing `audioAsVoice` transcode path instead. Duration handling should be coordinated with the narrower Feishu duration work so audio/video uploads get one focused, tested implementation.
