## Summary

Keep #38907 open. Current main still has a credible unresolved ACP bridge failure path matching the report: `openclaw acp client` defaults to a bare `acp:<uuid>` Gateway session key without persisting ACP runtime metadata, `chat.send` canonicalizes that key to an agent-scoped ACP key, and the ACP reply-dispatch path treats ACP-shaped sessions without `entry.acp` as terminal `ACP_SESSION_INIT_FAILED` errors instead of falling back to normal assistant generation.

## Best Possible Solution

Keep the issue open until the ACP bridge/runtime boundary is fixed and covered. The likely fix is either to stop default Gateway-backed bridge sessions from using ACPX-shaped `acp:<uuid>` keys that the ACP runtime claims, or to persist valid `SessionAcpMeta` for those sessions before first prompt. Add a focused regression test for default `openclaw acp client` / `AcpGatewayAgent.newSession` first prompt producing assistant chunks instead of `ACP_SESSION_INIT_FAILED`.
