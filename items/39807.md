## Summary

Current main still does not attach a durable cooldown key to inline `models.providers.<id>.apiKey` auth. Inline-key providers resolve without `profileId`, auth failure recording is skipped when `profileId` is missing, and model fallback only skips cooldowned providers when auth profile ids exist. The related PR #39836 proposed the expected synthetic inline-profile fix, but it was closed unmerged and that behavior is absent from main.

## Best Possible Solution

Keep the issue open and implement the missing durable cooldown identity for inline `models.providers.<id>.apiKey` credentials. A conservative fix would mirror PR #39836's direction: return a synthetic profile id such as `inline:<provider>` for inline-key auth, ensure `markAuthProfileFailure()` can persist a minimal matching profile/usage entry, and add regression coverage proving a 402 billing failure disables the inline provider and lets configured fallbacks run.
