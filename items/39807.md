## Summary

Keep open. Current `main` still does not put inline `models.providers.*.apiKey` credentials onto any cooldown-tracked profile path, so a 402/billing failure can still miss `markAuthProfileFailure` and repeat on later runs.

## Evidence

- **Inline provider auth still returns no profile id:** `resolveApiKeyForProvider` returns the configured provider API key with only `{ apiKey, source, mode }` and no synthetic `profileId` for `models.providers.*.apiKey` credentials.
  - file: [src/agents/model-auth.ts:552](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/agents/model-auth.ts#L552)
  - command: `nl -ba src/agents/model-auth.ts | sed -n '540,566p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Runner records no profile for that auth path:** `applyApiKeyInfo` writes `lastProfileId` from `apiKeyInfo.profileId`; for inline configured API keys that stays `undefined`, so the run has no cooldown-tracked profile identity.
  - file: [src/agents/pi-embedded-runner/run/auth-controller.ts:447](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/agents/pi-embedded-runner/run/auth-controller.ts#L447)
  - command: `nl -ba src/agents/pi-embedded-runner/run/auth-controller.ts | sed -n '360,448p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Failure marking is skipped without a profile id:** `maybeMarkAuthProfileFailure` returns immediately when `profileId` is missing, so billing/auth failures from the inline-key path never enter `markAuthProfileFailure`.
  - file: [src/agents/pi-embedded-runner/run.ts:561](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/agents/pi-embedded-runner/run.ts#L561)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '561,580p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Explicit config API-key path still bypasses auth-profile candidates:** When `shouldPreferExplicitConfigApiKeyAuth(...)` is true, the runner skips `resolveAuthProfileOrder(...)` and falls back to `[undefined]` candidates instead of a real/synthetic profile id.
  - file: [src/agents/pi-embedded-runner/run.ts:397](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/agents/pi-embedded-runner/run.ts#L397)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '397,409p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Linked proposed fix is not present on main:** The linked PR patch for #39836 adds `INLINE_PROFILE_PREFIX`, synthetic inline profiles in `markAuthProfileFailure`, and a synthetic `profileId` in `model-auth.ts`; current `main` has none of those identifiers (`rg` found no matches).
  - command: `curl -L --silent https://patch-diff.githubusercontent.com/raw/openclaw/openclaw/pull/39836.patch | sed -n '1,260p'; rg -n "INLINE_PROFILE_PREFIX|inline:" src/agents/auth-profiles.ts src/agents/auth-profiles/usage.ts src/agents/model-auth.ts src/agents/auth-profiles.markauthprofilefailure.test.ts`

## Risks / Open Questions

- Inline API-key providers can still avoid billing/auth cooldown tracking, so repeated 402 failures may continue to hit the same primary provider on subsequent runs.
- Because the linked fix PR is closed and its synthetic-profile changes are absent from `main`, closing this issue as implemented would be incorrect.
