## Summary

Keep #39807 open. Current main still resolves inline `models.providers.<id>.apiKey` credentials without a durable auth profile id, while billing cooldown recording and provider skipping are still keyed to auth profiles. General 402 fallback behavior exists, but it does not solve the reported persistent no-backoff/circuit-breaker gap for inline-apiKey providers.

## Best Possible Solution

Keep the issue open and implement a maintained cooldown key for inline provider credentials. The most direct path is to have inline `models.providers.<id>.apiKey` resolution produce a stable synthetic profile id or provider-level cooldown id, ensure billing/auth failure marking persists state for that id, and add regression coverage proving later turns skip the disabled inline provider and go directly to configured fallbacks.
