## Summary

Keep open. Current `main` has partial layered diagnostics and fresh troubleshooting docs, but the in-product UX requested by #41553 is not fully landed: raw browser transport signals exist, Control UI auth/origin errors are differentiated, yet the human CLI surfaces still do not present the layered remote-CDP health breakdown, and the existing browser-doctor path is limited to Chrome MCP readiness rather than WSL2/remote-CDP diagnosis.

## Evidence

- **Browser status API already computes layered transport state:** The browser status route returns separate `cdpHttp` and `cdpReady` booleans, so the raw signal for HTTP reachability vs websocket readiness exists on `main`.
  - file: [extensions/browser/src/browser/routes/basic.ts:82](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/extensions/browser/src/browser/routes/basic.ts#L82)
- **Remote CDP failures are already split by layer:** Remote profiles surface different errors for HTTP-unreachable CDP endpoints vs websocket-unreachable endpoints.
  - file: [extensions/browser/src/browser/server-context.availability.ts:241](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/extensions/browser/src/browser/server-context.availability.ts#L241)
- **Control UI origin and secure-context failures are explicitly differentiated:** Gateway handshake errors distinguish `origin not allowed` from `control ui requires device identity`, and the Control UI maps those detail codes into user-facing messages.
  - file: [src/gateway/server/ws-connection/message-handler.ts:483](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/src/gateway/server/ws-connection/message-handler.ts#L483)
- **Control UI formatter has user-facing auth/origin hints:** The Control UI formatter maps detail codes to specific messages like `device identity required` and `origin not allowed`, which is good partial UX coverage for the auth/origin side.
  - file: [ui/src/ui/connect-error.ts:65](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/ui/src/ui/connect-error.ts#L65)
- **Human `openclaw browser status` output still hides the layered browser health fields:** The CLI prints `running`, transport, `cdpPort`, and `cdpUrl`, but it does not print the `cdpHttp` or `cdpReady` values that the status API returns.
  - file: [extensions/browser/src/cli/browser-cli-manage.ts:150](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/extensions/browser/src/cli/browser-cli-manage.ts#L150)
- **`openclaw browser profiles` still shows only a shallow summary:** The profiles command reports running/tab count/basic connection info, but not a layer-by-layer health view for remote CDP profiles.
  - file: [extensions/browser/src/cli/browser-cli-manage.ts:443](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/extensions/browser/src/cli/browser-cli-manage.ts#L443)
- **Browser doctor is not the requested remote-CDP diagnostics surface:** The current browser doctor helper only checks Chrome MCP existing-session readiness and local Chrome version/remote-debugging setup; it does not diagnose WSL2/Windows remote CDP reachability or Control UI/CDP layer splits.
  - file: [extensions/browser/src/doctor-browser.ts:54](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/extensions/browser/src/doctor-browser.ts#L54)
- **Docs now cover the layered troubleshooting model:** A new WSL2+Windows troubleshooting guide explicitly documents overlapping layers and a step-by-step separation of Control UI/auth issues from CDP reachability issues. That mitigates the problem, but it is documentation rather than the product-side UX requested in the issue.
  - file: [docs/tools/browser-wsl2-windows-remote-cdp-troubleshooting.md:56](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/docs/tools/browser-wsl2-windows-remote-cdp-troubleshooting.md#L56)
