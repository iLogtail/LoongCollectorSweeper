## Summary

Keep #43903 open. Current main still has a single shared gateway token/password model, while trusted-proxy and Tailscale are the only identity-bearing gateway auth modes. The requested per-token identity/scopes mapping and Control UI session attribution are not implemented, and the +1 comment's team-use workaround remains a valid product gap.

## Best Possible Solution

Keep the issue open for maintainer design. The likely core path is an additive `gateway.auth.tokens` model with per-token labels/scopes, token-to-principal resolution in gateway auth, propagation into WS/HTTP request context, Control UI `chat.send` sender metadata and session routing, and updated docs/tests while preserving the existing single `gateway.auth.token` contract.

## Risks / Open Questions

- Adding this naively could imply least-privilege semantics that shared-secret HTTP surfaces do not currently enforce; the scope model would need to be propagated through WS connect, HTTP request auth, method authorization, and plugin routes.
- The feature touches public config/schema/docs and security-sensitive gateway auth behavior, so it needs a backward-compatible design that preserves `gateway.auth.token` and current device-pairing behavior.
- Browser-local Control UI identity is not a substitute for server-resolved token identity because it is user-controlled and not an authorization boundary.
