## Summary

Keep open. Current `main` still creates per-lane `LaneState` entries on demand and does not prune idle dynamic lanes after task completion, `clearCommandLane()`, or `resetAllLanes()`, so the memory-growth concern described in PR #58244 remains plausible and unfixed on `main` at `90877e0d4274c53fd91bf5787265c4d6889dc9a7`.

## Evidence

- **Lane states are created and retained in a global Map:** `getLaneState()` creates a new `LaneState` and stores it in `queueState.lanes` whenever a lane name is first seen. There is no corresponding production deletion path in this function.
  - file: [src/process/command-queue.ts:97](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/process/command-queue.ts#L97)
  - command: `nl -ba src/process/command-queue.ts | sed -n '97,113p;155,224p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Task completion does not prune idle lanes:** `drainLane()`/`pump()` removes active task IDs and recurses, but the current main code never deletes the lane after `state.queue.length === 0` and `state.activeTaskIds.size === 0`. The PR's proposed prune-on-idle logic is absent.
  - file: [src/process/command-queue.ts:167](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/process/command-queue.ts#L167)
