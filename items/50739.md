## Summary

Keep #50739 open. Current main still routes `openclaw system event --mode now` through the existing wake/heartbeat path: the CLI only accepts `--text`, `--mode`, and `--json`, the wake handler enqueues a system event and calls `requestHeartbeatNow`, and the heartbeat runner explicitly skips when the target session lane is busy. The KeepMyClaw comments are useful recovery/snapshot context, but they are complementary; they also state the system-event path still needs a higher-priority injection mechanism.

## Best Possible Solution

Keep the issue open and implement a core-owned urgent system-event path if maintainers accept the product tradeoff: add an explicit CLI/RPC flag such as `--priority high` or `--bypass-queue`, define how it appears in the TUI/session timeline during active turns, preserve trust metadata, and add regression tests for a congested session lane. Treat KeepMyClaw-style snapshots as complementary recovery tooling, not a replacement for reliable in-session alerts.

## Risks / Open Questions

- A bypass path needs careful transcript/session-state design so urgent alerts do not corrupt an active streaming turn or create inconsistent session history.
- Trust handling should stay explicit for untrusted hook/node-originated system events so a priority path does not reopen spoofing or alert-injection issues.
- TUI behavior, gateway RPC schema, docs, and tests should be updated together if a new priority or bypass mode is added.
