## Summary

Keep open. Current main does not implement the requested `old_text`/`new_text` aliases. The edit tool now validates canonical `path` plus `edits[]`, parses only `oldText`/`newText` inside each edit entry, and existing regression coverage explicitly rejects legacy/top-level alias payloads including `old_text` and `newString`. The related alias PRs were closed unmerged, and the issue discussion remains relevant because the follow-up comment reports similar failures from Claude with `oldString`/`newString`.

## Best Possible Solution

Keep the issue open for maintainer/product decision. If compatibility is desired, implement a narrow normalization layer before `REQUIRED_PARAM_GROUPS.edit` validation that maps supported legacy/top-level aliases into canonical `{ path, edits: [{ oldText, newText }] }`, update the provider-facing schema/prompt surface accordingly, and replace the current alias-rejection tests with explicit compatibility coverage. If OpenClaw intentionally wants canonical-only tool calls, close this manually with that product decision and update any remaining docs or diagnostics that imply legacy aliases are supported.
