## Summary

Keep open. Current `main` still treats `notifyOnExitEmptySuccess` as explicit opt-in only, so chat-backed background execs that exit 0 with empty output still go silent unless config sets the flag. The PR remains a focused, still-relevant change, and the provided review context shows the author addressed the two semantic concerns by respecting explicit `false` and limiting the new default to real reply-capable channels.

## Best Possible Solution

Keep tracking PR #39726 until the chat-context default is either merged or explicitly rejected by a maintainer. The best implementation path is the PR's refined shape: default `notifyOnExitEmptySuccess` to true only when the exec tool has a real reply-capable channel from `resolveGatewayMessageChannel`, preserve explicit `false`, keep generic provider labels like `openai` silent, and land regression coverage for all of those cases plus a changelog entry with proper attribution.
