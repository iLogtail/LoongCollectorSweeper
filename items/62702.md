## Summary

Current main has typed plugin hooks and the context-engine slot, but it does not implement the requested non-blocking hook policy. Pre-prompt/context hooks are still awaited before prompt submission, modifying hooks run sequentially, and the public hook registration API exposes priority only, with no timeout/fallback/critical option or registerContextProvider surface. This is a valid core plugin SDK/runtime feature request, not a ClawHub-only plugin item.

## Best Possible Solution

Keep the issue open and design an additive core plugin SDK/runtime feature: extend typed hook registration and `plugins.entries.<id>.hooks` config with per-hook timeout/fallback/critical semantics, preserve fail-closed defaults for safety-sensitive hooks, add a `plugin:hook:timeout` diagnostic/log event, and cover `before_prompt_build`/legacy `before_agent_start` plus other decision hooks with focused tests and docs. If async memory recall is the primary use case, consider a dedicated non-exclusive context-provider API rather than relying only on the exclusive context-engine slot.

## Risks / Open Questions

- Adding this casually could weaken safety hooks; security-sensitive hooks need explicit fail-closed/critical behavior and tests.
- Timeouts need cancellation/diagnostic semantics, not just Promise.race, or timed-out work may keep running and mutate plugin state later.
- Context-engine assembly is also awaited today, so maintainers should decide whether the feature covers only hooks or a broader context-provider/deadline abstraction.
