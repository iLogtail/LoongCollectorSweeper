## Summary

Keep #69910 open. Current main has partial Ollama argument normalization for replay/OpenAI-compatible request paths, but the exact native inbound `buildAssistantMessage` path targeted by this PR still passes `tool_calls[].function.arguments` through unchanged and lacks the regression coverage added by the PR.

## Best Possible Solution

Keep the PR open and land or recreate its narrow fix: normalize `response.message.tool_calls[].function.arguments` inside `buildAssistantMessage` using the existing unsafe-integer-preserving helper with `{}` fallback for malformed/non-object strings, broaden the local response type enough to reflect runtime inputs, and add regression tests for valid string args, object passthrough, unsafe integers, and malformed strings.
