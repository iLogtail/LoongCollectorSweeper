## Summary

Keep #69910 open. Current main and v2026.4.24 still pass native Ollama inbound `tool_calls[].function.arguments` through unchanged in `buildAssistantMessage`; only the replay/outbound Ollama paths normalize string arguments today. The PR remains a plausible fix, especially after its follow-up commit addressed the unsafe-integer and malformed-JSON review comments.

## Best Possible Solution

Keep the PR open for maintainer review or land an equivalent rebased fix: allow native Ollama response tool-call arguments to be `unknown` or `Record<string, unknown> | string`, normalize them with the existing `ensureArgsObject` in `buildAssistantMessage`, and add focused regression tests for valid JSON strings, unsafe integer preservation, and malformed/non-object fallback.
