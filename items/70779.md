## Summary

Keep #70779 open. Current main still sends `stream_options: { include_usage: true }` unconditionally on the OpenAI-compatible Chat Completions streaming path, even when compatibility detection/model normalization says generic custom OpenAI-compatible providers should not opt into streaming usage. That directly overlaps the reporter’s suspected regression and there is no source-backed fix on main or in v2026.4.24.

## Best Possible Solution

Keep the issue open and fix the OpenAI-compatible Chat Completions payload builder to honor `compat.supportsUsageInStreaming`: omit `stream_options.include_usage` for generic/custom providers unless the model/provider explicitly opts in or detection identifies a compatible native/local endpoint. Add regression tests for generic custom providers with unset/false compat omitting `stream_options`, explicit true preserving it, and the reported Gemini-style custom backend path if a fixture can model it.
