## Summary

Keep #61238 open. Current main still defaults sessions to a daily 4:00 AM rollover and rotates to a new sessionId on the next user message after the boundary. The code/docs do not provide a supported `mode: "off"`, a pre-reset warning/cancel/postpone flow, or a general restore/rebind command for reset archives. Related #29536 was closed stale rather than shipped, and #63732 tracks the opposite reset-not-firing regression, so neither supersedes this user-facing warning/control request.

## Best Possible Solution

Keep the issue open for maintainer/product handling. The strongest path is to add a supported reset opt-out or manual-only mode in the config schema and docs, surface the active reset policy during setup/status, add a user-visible pre-rollover notice or explicit first-message-after-boundary confirmation where channels can support it, and decide whether reset archives should be restorable/rebindable through a CLI or Control UI workflow.

## Risks / Open Questions

- Session rollover is also used to bound context and cost, so a fix should preserve a safe default or make the tradeoff explicit when users disable automatic resets.
- Reset archives are currently retained and cleaned by maintenance policy; a restore/rebind feature should define archive retention and UI expectations before promising recovery.
