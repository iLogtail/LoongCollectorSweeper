## Summary

Keep #61430 open. Current main still does not expose an optional per-call exec `description`/`intent` field in the model-facing tool schema, execution params, approval request schema/payload, or public docs. Plugin hooks can require approvals with their own generated descriptions, but hooks currently receive only `toolName` and raw `params`, so they cannot consume a model-supplied exec intent unless core adds the field. The related #61562 is a newer duplicate-style report of the same remaining work, not a better canonical supersession of this older issue.

## Best Possible Solution

Keep the issue open. The best implementation is an additive optional `description` or `intent` field on the exec tool schema, threaded through exec param typing and `before_tool_call` params, and, if approvals/audit logs should use it, through the exec approval request schema, stored request payload, view model/UI, docs, and regression tests. The field should be clearly documented as informational and never as a substitute for command-based safety checks.
