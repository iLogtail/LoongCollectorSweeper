## Summary

Keep #71784 open. Current main still has a single live-query embedding path for builtin memory search: `manager.search()` calls `embedQueryWithTimeout()`, which calls the selected provider's `embedQuery()` once under a timeout but without retrying transient fetch/socket failures. The OpenAI-compatible remote embedding helper and Gemini embedding provider both route through `withRemoteHttpResponse()` / `fetchWithSsrFGuard()` without a query-level retry. Related #44166/#44167 cover batch/reindex transient embedding failures; this issue is specifically the per-query live search path and even states bulk reindex is not affected, so it is not superseded.

## Best Possible Solution

Keep this issue open and fix the live memory-search query path in current `extensions/memory-core` / `src/memory-host-sdk` code. A conservative implementation would add a narrow transient transport-error classifier for embedding HTTP calls, retry `embedQueryWithTimeout()` or the shared single-embed remote HTTP helper with bounded attempts/backoff, preserve existing auth/quota error behavior, and add regression tests for `fetch failed`, `other side closed`, and TLS socket disconnects on `openclaw memory search`/`memory_search`. Dispatcher keep-alive tuning can be added if live validation confirms it is the root cause, but a scoped retry is the direct user-visible reliability fix.

## Risks / Open Questions

- A live reproduction against OpenAI/Gemini is still needed before choosing between a query-level retry, dispatcher keep-alive tuning, or both.
- The fix should distinguish retryable transport resets from permanent auth/configuration failures so bad credentials do not become slow repeated failures.
