## Summary

Keep #60858 open. Current main contains fixes for adjacent problems (#71325 precheck/windowing and #71130 empty provider submission), but the reported compaction state disagreement is not proven fixed: the same no-real-conversation guard paths still no-op, tests intentionally preserve the empty-preparation boundary behavior, and the latest issue comment reports repeated empty-message compaction attempts on v2026.4.24 after the related safeguard fix.

## Best Possible Solution

Keep #60858 open and narrow the maintainer work to the remaining upstream state mismatch: add diagnostics around persisted transcript messages, prepared compaction messages, and trigger inputs; then add a regression test where a populated transcript would previously reduce to an empty compaction input or repeatedly schedule boundary-only compactions. The existing boundary/no-op safeguard should stay as damage control, but it should not be treated as the complete fix.
