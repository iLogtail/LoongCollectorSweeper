## Summary

Keep open. Current main still matches the reported root cause: queued followups store the inbound message id, and the normal immediate reply path passes a currentMessageId into applyReplyThreading, but the followup delivery path neither accepts nor forwards currentMessageId, so replyToMode="all" cannot synthesize replyToId for drained followup replies. The linked #63637 appears to target the right fix, but it is still open/unmerged and the fix is not present on current main.

## Best Possible Solution

Keep #54456 open and land the small queued-followup fix, likely via #63637 or an equivalent maintainer patch: add currentMessageId to resolveFollowupDeliveryPayloads, pass queued.messageId from createFollowupRunner, and add regression coverage that a drained followup with replyToMode="all" produces payload.replyToId for Matrix/channel delivery.
