## Summary

Keep #9986 open. Current main still does not route context-overflow failures to a larger configured model fallback: the outer fallback loop explicitly rethrows likely context-overflow errors, the embedded Pi runner handles overflow with compaction/tool-result truncation on the active provider/model, and exhausted recovery returns a blocked `context_overflow` result rather than selecting `agents.defaults.model.fallbacks`. The same-day issue discussion also already includes a Codex review comment keeping this open, and the related closed items cover different fallback or graceful-overflow cases rather than this larger-context fallback behavior.

## Best Possible Solution

Keep #9986 open until maintainers either explicitly choose the documented compaction-only policy or implement a guarded context-overflow fallback path. The likely implementation belongs after overflow compaction and oversized-tool-result truncation are exhausted: if replay is safe and a configured fallback has known larger context headroom, switch to that candidate, log the event, and add regression coverage plus docs for smaller-primary to larger-fallback behavior.

## Risks / Open Questions

- Naively replaying a context-overflow turn on another model can duplicate tool side effects or outbound messages unless it is gated by replay-safety metadata.
- Selecting a fallback by context headroom requires reliable context-window metadata for both primary and fallback candidates; current code deliberately avoids trying an arbitrary fallback that may have a smaller window.
