## Summary

Keep #9986 open. Current main still deliberately excludes context-overflow errors from model fallback, handles them through compaction/tool-result truncation on the active model, and returns a terminal context_overflow/compaction_failure payload when recovery is exhausted. The related closed items cover overloaded fallback, output-token length, or graceful overflow messaging, not automatic selection of a larger configured fallback model after context overflow.

## Best Possible Solution

Keep #9986 open until maintainers either explicitly affirm the current documented policy or implement a guarded context-overflow fallback path. The likely fix belongs between embedded-runner overflow exhaustion and model-fallback policy: after compaction/truncation are exhausted, replay only when safe, choose a configured fallback with sufficient context headroom, log the transition, and add docs/tests for smaller-primary to larger-fallback behavior.

## Risks / Open Questions

- A naive fallback-on-overflow implementation could replay side-effecting turns or completed tool work; any fix needs replay-safety and side-effect gating.
- Selecting a larger fallback requires trustworthy context-window metadata for the primary and fallback candidates, including behavior for equal or smaller fallback windows.
