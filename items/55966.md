## Summary

Keep open. Current `main` has a downstream mitigation for zero-byte/tiny audio files, but the Telegram inbound media path still saves directly to the final inbound file path instead of writing to a temp file and renaming on success. That means the reported ENOSPC orphan-file behavior is not clearly fixed on `main` even though the later transcription crash is partially mitigated.

## Evidence

- **Telegram inbound media still saves straight through `saveMediaBuffer`:** `downloadAndSaveTelegramFile()` fetches the Telegram file and then calls `saveMediaBuffer(..., "inbound", ...)` for inbound attachments; there is no temp-file staging in this path.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:188](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/telegram/src/bot/delivery.resolve-media.ts#L188)
  - command: `nl -ba extensions/telegram/src/bot/delivery.resolve-media.ts | sed -n '170,250p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **`saveMediaBuffer` still writes the final destination directly:** `writeSavedMediaBuffer()` writes the final inbound path with `fs.writeFile(dest, ...)`. By contrast, `saveMediaSource()` uses a `.tmp` path and `fs.rename(...)` for URL sources. The direct-write buffer path is the remaining gap for ENOSPC leaving a created/partial file behind.
  - file: [src/media/store.ts:301](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/media/store.ts#L301)
  - command: `nl -ba src/media/store.ts | sed -n '300,410p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Current main does contain a transcription-side guard for empty/tiny audio:** `assertMinAudioSize()` rejects audio smaller than `MIN_AUDIO_FILE_BYTES`, and both provider and CLI audio paths use it before transcription. There is also a test that explicitly covers a 0-byte file being skipped. This addresses the later EOF-style transcription crash but not the orphaned inbound file.
  - file: [src/media-understanding/runner.entries.ts:494](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/media-understanding/runner.entries.ts#L494)

- **Zero-byte skip is covered by test:** `runner.skip-tiny-audio.test.ts` includes `it("skips audio transcription for empty (0-byte) files", ...)`, confirming the downstream mitigation exists on current `main`.
