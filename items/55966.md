## Summary

Keep open. Current `main` partially mitigates the downstream EOF symptom by skipping empty/tiny audio before transcription, and Telegram media fetch failures are caught before normal inbound dispatch. However, the media store still writes inbound buffers directly to the final `media/inbound` path with `fs.writeFile`; there is no temp-file plus atomic rename path or ENOSPC cleanup for `saveMediaBuffer`, so the core zero-byte/partial-artifact report remains actionable.

## Best Possible Solution

Keep the issue open and fix it in core media storage plus Telegram-facing diagnostics: make `saveMediaBuffer` stage buffer writes to a same-directory temp file, fsync/close if the project has an existing helper for that, rename only after a complete write, and unlink the temp/final candidate on ENOSPC or other write failure. Add regression coverage that simulates a buffer write failure and verifies no final inbound file is left, then keep the existing tiny-audio transcription skip as the downstream guard. A small Telegram log/message improvement for ENOSPC would complete the operator-facing part.
