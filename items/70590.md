## Summary

Keep open. Current main already has a Control UI i18n framework, locale picker, generated non-English locale bundles, tests, and docs, but the specific user problem is still real: production web UI surfaces still contain many user-visible English strings outside the translation map. The provided related ClawSweeper matches are title-search false positives and do not supersede this request.

## Best Possible Solution

Keep the issue open and treat it as a Control UI i18n completion task: move remaining production user-visible strings into ui/src/i18n/locales/en.ts, update call sites to use t(), add or extend a guard/test that catches raw English in high-traffic Lit UI surfaces, then run the established ui:i18n:sync pipeline to regenerate locale bundles and metadata. Good first targets are chat, Quick Settings, update banners, and the Dreaming/Memory Wiki views because current main has concrete untranslated strings there.
