## Summary

Keep #41657 open. Current main no longer simply strips Slack files: it attempts media download, adds filename/fileId placeholders, and exposes downloaded media through MediaPath/MediaPaths/MediaTypes. The requested attachment metadata contract is still missing: agents do not receive a structured per-file block with Slack file id, filename, MIME type, and size, and the related open PR #62792 is narrower file-access hardening rather than this metadata feature.

## Best Possible Solution

Keep the issue open and implement an additive inbound attachment metadata contract. Preserve the existing BodyForAgent placeholders, MediaPath/MediaPaths behavior, and download-file action for compatibility, but add a structured field such as InboundAttachments with provider, providerFileId, fileName, mimeType, sizeBytes, downloaded/localPath, and placeholder. Slack should populate it from message.files, while follow-up work can map Discord and Telegram into the same shape.

## Risks / Open Questions

- A Slack-only metadata shape could create channel drift; the cleaner fix is probably a shared inbound attachment model across Slack, Discord, Telegram, and other media-capable channels.
- Do not treat #62792 as the canonical closure for this issue unless it is expanded to add the structured metadata contract requested here.
