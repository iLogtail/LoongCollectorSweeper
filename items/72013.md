## Summary

Keep open. Current main has a partial fix that reconciles identity before closing newly completed one-shot ACP runs, but startup reconciliation still scans persisted ACP records based only on pending identity plus a stable session id. There is no current-main skip for `acp.mode === "oneshot"` or historical terminal/closed metadata, so stale one-shot records like the issue describes can still reach the warn-producing startup path. The linked PR #72023 is open and targets this exact remaining gap.

## Best Possible Solution

Keep #72013 open until the focused startup behavior is fixed. The best implementation path is to land a validated version of #72023: skip terminal one-shot and historical closed ACP records inside `AcpSessionManager.reconcilePendingSessionIdentities` before incrementing `checked` or touching the runtime, and add regression tests proving those records do not call `ensureSession`/`getStatus` while persistent pending sessions still reconcile.
