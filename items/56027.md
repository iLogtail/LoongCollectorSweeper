## Summary

Current `main` already covers both reported failure modes: plugin trust/allowlist decisions now read from the raw source config snapshot during runtime-snapshot reloads, and the gateway pins the startup channel registry so later plugin-registry churn does not evict configured channels like `whatsapp` from tool send resolution.

## Evidence

- **Allowlist reload now uses source snapshot:** `resolvePluginActivationSourceConfig(...)` prefers the raw activation-source config, and when only the runtime snapshot is available it falls back to `getRuntimeConfigSourceSnapshot()`. That matches the report’s false `plugins.allow is empty` symptom during reloads.
  - file: [src/plugins/activation-source-config.ts:7](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/plugins/activation-source-config.ts#L7)
  - command: `nl -ba src/plugins/activation-source-config.ts | sed -n '1,40p'`
- **Shipped fix note for false empty allowlist warnings:** The changelog entry for `2026.4.22` says plugin trust decisions now come from the source config snapshot so `plugins.allow` remains enforced and gateway startup/doctor no longer warn that the allowlist is empty when it is configured.
  - file: [CHANGELOG.md:250](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/CHANGELOG.md#L250)
  - command: `nl -ba CHANGELOG.md | sed -n '240,255p'`
- **Channel registry pinning exists in runtime state:** `pinActivePluginChannelRegistry(...)` and `getActivePluginChannelRegistry()` explicitly preserve the startup channel snapshot across later `setActivePluginRegistry(...)` calls, which addresses the reported post-reload loss of `whatsapp` as a valid send target.
  - file: [src/plugins/runtime.ts:151](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/plugins/runtime.ts#L151)
  - command: `nl -ba src/plugins/runtime.ts | sed -n '145,180p'`
- **Gateway uses channel pinning for startup and deferred reloads:** Both `loadGatewayStartupPlugins(...)` and `reloadDeferredGatewayPlugins(...)` route through `beforePrimeRegistry: pinActivePluginChannelRegistry`, so the gateway keeps the pinned channel registry across later plugin reload activity.
  - file: [src/gateway/server-plugin-bootstrap.ts:98](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/gateway/server-plugin-bootstrap.ts#L98)
  - command: `nl -ba src/gateway/server-plugin-bootstrap.ts | sed -n '60,118p'`
- **Regression tests cover both behaviors:** Tests assert that a pinned startup channel registry survives later active-registry replacement (`src/plugins/runtime.channel-pin.test.ts:99`) and that loader trust checks use the source snapshot without emitting `plugins.allow is empty` warnings (`src/plugins/loader.test.ts:6277`). Gateway tests also verify the pinned startup registry survives later churn (`src/gateway/server-plugins.test.ts:365`).
  - file: [src/plugins/runtime.channel-pin.test.ts:99](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/plugins/runtime.channel-pin.test.ts#L99)

## Risks / Open Questions

- I did not run a live gateway reproduction in this read-only review, so the decision is based on current code, shipped changelog entries, and targeted regression tests.
- Issue #70524 suggests there is still a separate hot-reload limitation around newly adding channels/allowlist entries without restart; that is different from this report’s regression where an already-configured channel disappeared after reload.

## Close Comment

Closing from a Codex review because current `main` already implements the needed fix. The reported false `plugins.allow is empty` behavior is covered by the source-snapshot trust change in `src/plugins/activation-source-config.ts:7` and the shipped `2026.4.22` changelog entry at `CHANGELOG.md:250`. The reported channel-loss side is also covered: the gateway now pins the startup channel registry (`src/plugins/runtime.ts:151`, wired from `src/gateway/server-plugin-bootstrap.ts:98` and `:107`) so later plugin-registry churn does not evict configured channels from message-send resolution. There are matching regression tests in `src/plugins/runtime.channel-pin.test.ts:99`, `src/plugins/loader.test.ts:6277`, and `src/gateway/server-plugins.test.ts:365`. If you still hit this on `v2026.4.22` or newer, please open a fresh issue with updated logs and exact reload steps.
