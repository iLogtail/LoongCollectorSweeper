## Summary

Keep #54183 open. Current main has the broader write-ahead outbound delivery queue and WhatsApp reconnect drain, so the persistent-loss reports are largely addressed, but the PR’s distinct configurable WhatsApp send retry surface is not implemented on main. The latest maintainer comment also explicitly treats configurable send retry as adjacent rather than identical to the shipped queue work.

## Best Possible Solution

Keep the PR open for maintainer review against current main. A current-main implementation should either deliberately add the WhatsApp retry config/schema/docs and wrap the actual resolved text/media/poll send path with a conservative pre-connect retry policy, or the maintainer should explicitly decide that the shipped delivery queue supersedes configurable immediate send retry and close it manually.
