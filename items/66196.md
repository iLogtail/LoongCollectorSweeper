## Summary

Keep open. Current `main` still feeds raw WhatsApp inbound content into the agent envelope, so the reported hardening gap is not already implemented, and the PR should not be auto-closed. But the proposed patch as submitted is not a safe `implemented_on_main`/`cannot_reproduce` close either: it would make WhatsApp combined-body echo deduplication non-deterministic and it wraps trusted prefix context together with untrusted user text.

## Evidence

- **Current main still passes raw inbound WhatsApp text into the envelope:** `buildInboundLine` constructs `baseLine` from `messagePrefix`, `msg.body`, and reply context, then passes that raw string directly as `body` to `formatInboundEnvelope(...)`. There is no `wrapExternalContent(...)` call on current `main` in this path.
  - file: [extensions/whatsapp/src/auto-reply/monitor/message-line.ts:35](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/extensions/whatsapp/src/auto-reply/monitor/message-line.ts#L35)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/monitor/message-line.ts | sed -n '1,220p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **WhatsApp echo suppression keys on exact combined-body bytes:** `processMessage` builds `combinedEchoKey` from `combinedBody`, and the echo tracker stores/checks keys as `combined:${sessionKey}:${combinedBody}`. Any nondeterministic change to the rendered body changes the dedupe key and can permit duplicate replies on redelivery.
  - file: [extensions/whatsapp/src/auto-reply/monitor/process-message.ts:307](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/extensions/whatsapp/src/auto-reply/monitor/process-message.ts#L307)

  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **`wrapExternalContent` output is intentionally nondeterministic:** The wrapper generates a fresh marker id with `randomBytes(8)` and embeds that id in the start/end markers for every call, so wrapping the same text twice does not produce stable output.
