## Summary

Keep #71142 open. Current main has Control UI chat uploads, but the requested configurable upload limit is not implemented: `chat.send` still passes a literal 5,000,000-byte attachment cap to the shared parser, and `gateway.controlUi` config has no max upload/media limit field.

## Best Possible Solution

Keep the issue open and implement a real upload policy. A maintainable fix would add a documented config key, likely under `gateway.controlUi` for the requested UI scope or under a generic `gateway.chat.attachments` surface if maintainers want the shared `chat.send` RPC covered too. Wire that value through config types, zod schema, help/generated docs, the Control UI client display/validation path, and the `chat.send` attachment parser, with tests for default 5 MB behavior, raised limits, and oversized rejection. If limits above the current WebSocket payload budget are supported, add or adjust the transport/media upload path accordingly.

## Risks / Open Questions

- Raising raw upload limits over the existing base64-over-WebSocket path may run into the current 25 MiB WebSocket payload policy, especially for 20 MB images after base64 expansion.
- The implementation should not reuse `tools.media.*.maxBytes` directly without a clear contract, because those limits govern media understanding after ingestion, not operator upload transport policy.
