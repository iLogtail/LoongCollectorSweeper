## Summary

Keep open. Current main still has a hard 5,000,000-byte Gateway attachment cap on Control UI chat.send, no gateway.controlUi upload/media limit config in the runtime schema, and docs still show higher media-understanding/WhatsApp limits than the Control UI send path. Related prior cleanup only confirms that image upload exists; it does not make the upload size configurable.

## Best Possible Solution

Keep the issue open and implement a bounded Gateway-owned upload-size policy, likely under gateway.controlUi or a shared chat attachment limit. Wire it through config types/schema/help/docs, pass it into all chat attachment parse call sites used by Control UI, expose the effective value to the browser for client-side validation/error copy, and add regression tests for a configured limit above 5 MB plus rejection above the configured cap.

## Risks / Open Questions

- Raising this limit affects Gateway memory, WebSocket payload, and media-store pressure; any implementation should bound values against MAX_PAYLOAD_BYTES and account for base64 overhead.
- The current Control UI upload surface is image-only, so the issue's audio/video/document examples may need a separate product decision if the requested mediaLimits shape is adopted broadly.
