## Summary

Keep open. Current main still enforces a hardcoded 5,000,000-byte chat attachment limit on the Gateway chat path used by Control UI uploads, and the Control UI/gateway config contract has no configurable upload-size field. The request is recent, author is not a maintainer, and there are no protected labels, so it remains an actionable product/code enhancement rather than a stale cleanup close.

## Best Possible Solution

Keep the issue open. A good implementation would add a documented Gateway config field for Control UI chat upload limits, validate it in the config schema and generated docs, expose the effective value through the Control UI bootstrap/config path for client-side validation messaging, apply it to the server-side chat attachment parser, add focused tests for default and configured limits, and separately decide whether Control UI should remain image-only or grow typed audio/video/document attachment support.

## Risks / Open Questions

- Raising the Control UI limit must stay below the effective WebSocket payload budget after base64 expansion, or separately redesign upload transport away from inline chat.send payloads.
- The media-understanding maxBytes settings and OpenResponses HTTP image/file limits are separate config surfaces; reusing them directly could unintentionally change non-Control-UI behavior.
- Audio, video, and document upload support is not currently part of the Control UI attachment picker, so per-media-type limits need explicit product scope.
