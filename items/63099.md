## Summary

Current `main` already handles `/reset` as a full session rollover in the shared gateway session layer: it generates a new `sessionId`, archives the prior transcript, retires the prior session runtime, and has regression coverage proving the transcript path rotates with the new session id. That is sufficient to address the reported post-`/reset` stale-session loop, and the implementation is already present in release `v2026.4.23`.

## Evidence
