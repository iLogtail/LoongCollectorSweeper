## Summary

Keep open. Current main still has the cron delivery path that can treat a numeric Telegram chat ID in `delivery.channel` as the channel itself, and the PR’s central runtime fix is not present. The PR is still useful, but its target-confusion/security review concern needs maintainer resolution before merge.

## Best Possible Solution

Keep the PR open and have maintainers review a constrained cron delivery fix: preserve support for explicit numeric Telegram `delivery.to`, add regression coverage for the misplaced numeric `delivery.channel` case if OpenClaw wants to recover old/bad jobs, and resolve the Aisle target-confusion concern before merge.
