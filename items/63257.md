## Summary

Keep this issue open. Current main does not prove the Windows Feishu startup timeout is fixed: the Feishu startup path still awaits a bot-identity probe before starting each account, the default startup probe timeout is still 30 seconds, and the timeout path explicitly logs that startup continues after the timeout. The newer HTTP timeout/config work may make the behavior bounded or tunable, but it does not satisfy the reported expectation that gateway startup connect to Feishu quickly by default. The provided related title matches are generic Windows gateway/WebSocket handshake reports, not the Feishu tenant-token REST timeout path described here.

## Best Possible Solution

Keep this open and handle it in the Feishu plugin startup path. The best fix is to make gateway startup avoid blocking on the Feishu bot-info or tenant-token probe when that REST call hangs on Windows: start the transport with unknown bot identity when safe, keep cancellable/background identity recovery, add a regression test proving startup does not wait the full 30s on a hanging token/probe request, and document any `httpTimeoutMs` or startup-probe timeout knobs in the public Feishu channel docs and plugin metadata.

## Risks / Open Questions

- A fix should preserve the existing protections around Feishu startup: sequential bot-info probing avoids API bursts, and strict unknown-bot behavior protects mention-gated groups.
- Lowering the timeout alone may mask the symptom while leaving the Windows/Node/Lark-SDK REST path slow in the background.
