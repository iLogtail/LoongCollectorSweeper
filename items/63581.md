## Summary

Keep this PR open. Current main still normalizes iMessage echo-cache text with newline replacement plus trim only, and JavaScript trim does not remove leading NUL bytes. The PR is a narrow implementation candidate for the closed linked iMessage NUL-prefix bug and is not obsolete on main.

## Best Possible Solution

Review and land this narrow iMessage echo-cache normalization fix, or land an equivalent focused patch with a regression test for single and repeated leading NUL bytes. Broader control-character/U+FFFD normalization can remain a separate maintainer decision, but current main does not yet solve this PR's NUL-prefix behavior.
