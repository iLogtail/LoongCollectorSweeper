## Summary

Keep open. Current `main` still appears to miss this fix: isolated cron direct delivery sends payloads without resolving or prepending `messages.responsePrefix`, while the normal reply path still applies that prefix.

## Evidence

- **Cron delivery file has no prefix-resolution import:** `src/cron/isolated-agent/delivery-dispatch.ts` imports outbound/session helpers and silent-token utilities, but not `resolveEffectiveMessagesConfig`, which the proposed fix depends on.
  - file: [src/cron/isolated-agent/delivery-dispatch.ts:1](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/cron/isolated-agent/delivery-dispatch.ts#L1)
  - command: `nl -ba src/cron/isolated-agent/delivery-dispatch.ts | sed -n '1,24p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Cron direct delivery only strips silent tokens before send:** `deliverViaDirect()` builds `rawPayloads`, maps them through `normalizeSilentReplyText()`, filters with `hasReplyPayloadContent()`, and never applies `responsePrefix` before delivery.
  - file: [src/cron/isolated-agent/delivery-dispatch.ts:519](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/cron/isolated-agent/delivery-dispatch.ts#L519)
  - command: `nl -ba src/cron/isolated-agent/delivery-dispatch.ts | sed -n '502,536p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Cron path passes payloads straight to outbound delivery:** The direct cron send call forwards `payloadsForDelivery` into `deliverOutboundPayloads()` with no intermediate prefix normalization step.
  - file: [src/cron/isolated-agent/delivery-dispatch.ts:601](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/cron/isolated-agent/delivery-dispatch.ts#L601)
  - command: `nl -ba src/cron/isolated-agent/delivery-dispatch.ts | sed -n '596,620p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Normal reply path still applies responsePrefix:** Interactive reply routing resolves `responsePrefix` via `resolveEffectiveMessagesConfig()`, then `normalizeReplyPayload()` prepends it when the text is not already prefixed. That inconsistency matches the PR’s bug description.
  - file: [src/auto-reply/reply/route-reply.ts:112](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/auto-reply/reply/route-reply.ts#L112)
  - command: `nl -ba src/auto-reply/reply/route-reply.ts | sed -n '100,145p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Prefix application lives in normalizeReplyPayload, not outbound delivery:** `normalizeReplyPayload()` prepends `effectivePrefix`, but `deliverOutboundPayloads()` itself is a transport layer and does not add any `responsePrefix`, so cron direct delivery does not get it automatically.
  - file: [src/auto-reply/reply/normalize-reply.ts:112](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/auto-reply/reply/normalize-reply.ts#L112)
