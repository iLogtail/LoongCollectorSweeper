## Summary

Keep open. Current `main` does not already filter heartbeat user prompts out of `chat.history`, so there is no strong basis to close this PR as implemented or obsolete. At the same time, the proposed fix is not strong enough for conservative cleanup: it relies on text-prefix matching for user messages, and current main already documents that this kind of coupling is spoofable by real user input.

## Evidence

- **`chat.history` still reads raw transcript messages on current `main`:** The handler pulls `localMessages` from `readSessionMessages(...)`, then runs `stripEnvelopeFromMessages(...)` and `sanitizeChatHistoryMessages(...)`. There is no user-side heartbeat filter in this path on current `main`.

  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Existing history sanitization only drops assistant-side noise:** `shouldDropAssistantHistoryMessage(...)` exits immediately unless `role === "assistant"`, so the current history sanitizer does not remove heartbeat prompts stored as user messages.
  - file: [src/gateway/server-methods/chat.ts:1213](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/gateway/server-methods/chat.ts#L1213)

That is the same class of predicate this PR proposes to use for hiding user messages from history, which is unsafe for a conservative close-out.
  - file: [src/auto-reply/heartbeat-filter.ts:36](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/auto-reply/heartbeat-filter.ts#L36)

That is direct evidence against treating user-prefix matches as a safe filter.

## Risks / Open Questions

- Closing this PR would drop a still-relevant bug report: current `main` still leaves heartbeat user prompts in the `chat.history` path.
- Treating the PR as effectively done would miss that its prefix-match approach can hide legitimate user messages that begin with the heartbeat prompt text, creating a chat-history integrity risk.
