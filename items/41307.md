## Summary

Current `main` already supports `peer.id: "*"` wildcard routing by peer kind, with the expected priority over channel/account fallbacks and tests covering the behavior. The implementation uses the name `binding.peer.wildcard` instead of `binding.peer.kind`, but the requested feature is present.

## Evidence

- **Wildcard peer binding is implemented in route normalization:** `src/routing/resolve-route.ts` normalizes `peer.id === "*"` into a dedicated wildcard-kind state, so bindings can target any peer of a given kind without enumerating IDs.
  - file: [src/routing/resolve-route.ts:483](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/routing/resolve-route.ts#L483)

- **Wildcard tier already exists in route priority order:** `resolveAgentRoute` evaluates exact peer, then parent peer, then wildcard peer bindings before guild/role, guild, team, account, and channel fallbacks. That matches the substance of the PR's requested routing tier.
  - file: [src/routing/resolve-route.ts:726](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/routing/resolve-route.ts#L726)
