## Summary

Keep PR #41866 open. Current main still lacks an equivalent defense for leading `to=<tool> ... {json}` text leaks: the shared user-facing sanitizer handles final tags, internal metadata, error payloads, leading blank lines, and duplicate blocks, but does not strip this Codex-style prefix. The PR needs a rebase and a tighter matcher, but the underlying sanitizer gap is not implemented on main.

## Best Possible Solution

Keep the PR open, rebase it onto `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts`, implement a narrower balanced-prefix parser with regression tests in `src/agents/pi-embedded-helpers.sanitizeuserfacingtext.test.ts`, and separately verify or track the root Discord cron execution failure if it remains reproducible.
