## Summary

Keep #72120 open. Current main still uses the raw `process.env.OPENAI_API_KEY` fallback in both reported OpenAI Realtime paths, so an unresolved `keychain:...` placeholder can still be sent as the bearer token. PR #72126 tracks the exact fix but is open and unmerged, so closing the issue now would be premature.

## Best Possible Solution

Keep the issue open and review/land #72126 or an equivalent focused fix in the OpenAI plugin. The fix should resolve or locally reject known OpenClaw keychain/SecretRef env placeholders before both `createBrowserSession` and `createBridge`, cache successful keychain resolutions for the gateway process, and add regression coverage for the env fallback plus config-backed Realtime API-key paths. If voice-call realtime provider config is intended to support SecretRefs, align the SecretRef surface docs/registry as part of the fix.
