## Summary

Keep #20934 open. Current main still does not expose REST session-management endpoints for `/v1/chat/completions` integrations. Session reset, patch/model, compaction, list/status, and usage data exist through Gateway RPC/control-plane surfaces, and there are narrow HTTP routes for session history/kill plus `/tools/invoke`, but not the requested `/v1/sessions/reset`, `/v1/sessions/status`, `/v1/sessions/compact`, `/v1/sessions/model`, or `/v1/sessions/reasoning` API. The related discussion supports keeping this as the canonical tracker: #23941 attempted the REST endpoints but was closed unmerged, #29985 was described as plugin-SDK reset only with REST still a follow-up, and the latest issue comment kept this item open.

## Best Possible Solution

Keep #20934 open as the canonical REST session-management tracker. A strong implementation would add authenticated Gateway HTTP endpoints for reset, status/list or usage, compact, and likely model/reasoning updates by reusing existing `sessions.reset`, `sessions.list`/`sessions.usage`, `sessions.compact`, and `sessions.patch` semantics, with route/auth/scope tests and docs under the OpenAI-compatible HTTP API. If maintainers prefer plugin ownership, first add a documented public plugin/runtime session-control API for reset, compact, and patch operations; route registration alone is not enough.
