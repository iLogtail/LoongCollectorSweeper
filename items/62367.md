## Summary

Keep #62367 open. Current main distinguishes direct OpenAI `openai/gpt-5.4-nano` from Codex OAuth `openai-codex/*`; direct OpenAI nano has forward-compat support, but the OpenAI Codex provider only has explicit GPT-5.4, GPT-5.4-pro, and GPT-5.4-mini handling. I did not find source, tests, docs, changelog, or history evidence that `openai-codex/gpt-5.4-nano` is implemented or intentionally rejected with a precise unsupported-model error.

## Best Possible Solution

Keep the issue open. Maintainers should decide whether `openai-codex/gpt-5.4-nano` is meant to be first-class; if yes, add Codex-provider nano constants, dynamic resolver metadata, catalog augmentation, modern/reasoning capability coverage, and regression tests. If not, add a precise unsupported-model error and ensure session/model-fallback paths do not silently route an explicit Codex nano request to an unrelated provider.
