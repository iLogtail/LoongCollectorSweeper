## Summary

Keep #53920 open. Current main still performs the exact in-place mutation the PR targets: `setup-auth-system.sh` runs `sed -i` against the tracked `scripts/systemd/openclaw-auth-monitor.service` template before copying it into the user systemd directory. The tracked template also still hardcodes `/home/admin/openclaw/scripts/auth-monitor.sh`, so the generated unit can inherit the wrong checkout path. The PR is a focused, plausible fix, but review discussion also raised an unresolved idempotency concern for already-mutated templates, so it should be revised/reviewed rather than auto-closed.

## Best Possible Solution

Keep the PR open and ask for a small revision: preserve the copy-then-patch behavior, render the current checkout's auth monitor path on the copied unit only, and make the notification placeholder replacement idempotent for both pristine and previously-mutated templates. After that, validate with shell syntax plus a read-only/fake-home install check that proves the repository template remains unchanged and the generated user unit has the expected ExecStart and environment lines.
