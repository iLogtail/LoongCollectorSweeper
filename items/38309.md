## Summary

Keep open. Current `main` does not implement SecretRef support for `sandbox.docker.env`: the config type and Zod schema still require plain strings, the secrets runtime only has sandbox SSH collectors, and the documented SecretRef credential surface/target registry do not include `agents.defaults.sandbox.docker.env` or `agents.list[].sandbox.docker.env`.

## Evidence

- **Sandbox docker env type is still string-only:** `SandboxDockerSettings.env` is declared as `Record<string, string>`, not `SecretInput`.
  - file: [src/config/types.sandbox.ts:21](https://github.com/openclaw/openclaw/blob/627c19c5cbbe8856a11effe1871c442ac87d56c9/src/config/types.sandbox.ts#L21)
  - sha: [627c19c5cbbe](https://github.com/openclaw/openclaw/commit/627c19c5cbbe8856a11effe1871c442ac87d56c9)
- **Schema still rejects SecretRef objects in sandbox.docker.env:** `SandboxDockerSchema` validates `env` as `z.record(z.string(), z.string())`, so object-valued SecretRefs are not accepted.
  - file: [src/config/zod-schema.agent-runtime.ts:112](https://github.com/openclaw/openclaw/blob/627c19c5cbbe8856a11effe1871c442ac87d56c9/src/config/zod-schema.agent-runtime.ts#L112)
  - sha: [627c19c5cbbe](https://github.com/openclaw/openclaw/commit/627c19c5cbbe8856a11effe1871c442ac87d56c9)
- **Secrets runtime only handles sandbox SSH SecretRefs:** Core secret collectors include `collectSandboxSshAssignments` for `identityData`/`certificateData`/`knownHostsData`; there is no collector for `sandbox.docker.env`. Existing runtime tests cover SSH SecretRefs only.
  - file: [src/secrets/runtime-config-collectors-core.ts:530](https://github.com/openclaw/openclaw/blob/627c19c5cbbe8856a11effe1871c442ac87d56c9/src/secrets/runtime-config-collectors-core.ts#L530)
  - sha: [627c19c5cbbe](https://github.com/openclaw/openclaw/commit/627c19c5cbbe8856a11effe1871c442ac87d56c9)
- **Secret target registry omits sandbox.docker.env:** The core registry enumerates supported config targets such as memory search and gateway credentials, but no `agents.defaults.sandbox.docker.env` / `agents.list[].sandbox.docker.env` entries are present.
  - file: [src/secrets/target-registry-data.ts:91](https://github.com/openclaw/openclaw/blob/627c19c5cbbe8856a11effe1871c442ac87d56c9/src/secrets/target-registry-data.ts#L91)
  - sha: [627c19c5cbbe](https://github.com/openclaw/openclaw/commit/627c19c5cbbe8856a11effe1871c442ac87d56c9)
- **Docs still treat sandbox.docker.env as the workaround, not a SecretRef surface:** `docs/tools/skills-config.md` tells users to use `agents.defaults.sandbox.docker.env` for sandboxed skills, while the canonical supported SecretRef surface doc does not list sandbox docker env.
  - file: [docs/tools/skills-config.md:128](https://github.com/openclaw/openclaw/blob/627c19c5cbbe8856a11effe1871c442ac87d56c9/docs/tools/skills-config.md#L128)
  - sha: [627c19c5cbbe](https://github.com/openclaw/openclaw/commit/627c19c5cbbe8856a11effe1871c442ac87d56c9)
- **Linked implementation PR is not merged into current main:** The cross-referenced PR still exists as a pull ref on origin, which is consistent with the feature not being present on `main`.
