## Summary

Keep open. Current main still has the native Discord voice response-contract shape described in this issue: the Discord VC path sends transcribed speech into the agent as Discord ingress, then requires a returned text payload before it performs its own TTS/playback. The shared tool policy only hides the core `tts` tool for `messageProvider: "voice"`, while this path resolves as `discord`, so a tool-only `tts` plus `NO_REPLY` turn can still leave Discord VC with no `replyText` to play. The supplied discussion also includes a same-symptom user report and a maintainer deep-review comment saying this remains valid on current main; related #60236 is about the earlier receive/capture failure, not this post-STT response/playback failure.

## Best Possible Solution

Implement a native Discord VC spoken-output contract instead of closing this issue. The likely fix belongs in the Discord voice path and/or generic agent ingress context: preserve Discord routing metadata, but run VC turns under a voice-specific tool/output policy so the core `tts` tool is hidden and the agent is prompted to return spoken text for DiscordVoiceManager to synthesize and play. Add focused regression coverage for a VC segment ensuring the agent context/tool policy is voice-safe and that tool-only `tts` plus `NO_REPLY` cannot silently satisfy the turn.
