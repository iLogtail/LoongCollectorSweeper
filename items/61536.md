## Summary

Keep open. Current main still matches the reported Discord VC output-contract gap: native Discord voice transcribes audio, invokes the agent as normal Discord ingress, can expose the `tts` tool, and only synthesizes playback after receiving non-empty reply text. A maintainer comment on 2026-04-25 also explicitly says this exact issue remains valid on current main.

## Best Possible Solution

Keep the issue open and fix native Discord VC as a first-class voice-output path: preserve Discord routing metadata separately, but run the agent under a voice-specific output/tool policy that hides `tts`, add a Discord VC spoken-output prompt contract equivalent to Voice Call, and add regression coverage around `/vc` ingress ensuring a tool-only `tts` plus `NO_REPLY` turn cannot leave playback with no reply text.
