## Summary

Keep #13364 open. Current main still exposes `before_tool_call` and `after_tool_call` only through the plugin hook runner, while managed/internal `HOOK.md` hooks remain void-returning event handlers for command/session/agent/gateway/message events. The issue also has explicit recent maintainer signal to keep it open as the canonical tracker, with related #17008 closed as a duplicate and #29590 closed unmerged as only a partial after-tool-call attempt.

## Best Possible Solution

Keep #13364 open for maintainer product/API design. A complete implementation should add typed managed/internal before/after tool-call events, define blocking and parameter-rewrite return semantics, validate unsupported `HOOK.md` event names or document the boundary clearly, bridge both relevant tool execution paths without double-firing plugin hooks, and add focused internal-hook tests plus docs. If maintainers decide this should remain plugin-only, close later with an explicit product decision and point users to plugin hooks and ClawHub packaging.
