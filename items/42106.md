## Summary

Keep #42106 open. Current main still matches the reported framework-level behavior: `EmbeddedBlockChunker` emits text before paragraph breaks and advances past the consumed newlines, while the coalescer can only restore `\n\n` when chunks remain in the same buffered flush. Once a flush has happened, the block reply pipeline delivers plain text payloads without boundary metadata, so a progressive client concatenating deliveries can still lose paragraph structure. The provided related items cover other block-streaming problems and do not supersede this issue.

## Best Possible Solution

Keep the issue open and fix the streaming boundary contract in core. The likely path is to make `EmbeddedBlockChunker` preserve consumed separators in emitted chunks or carry explicit boundary metadata through `BlockReplyPipeline`/`BlockReplyCoalescer`, then add regression coverage for a paragraph-separated Markdown reply split across separate block-streamed deliveries and reconstructed by concatenating visible updates.
