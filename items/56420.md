## Summary

Keep PR #56420 open. Current main still stores pending Claude permission requests by short request_id only and accepts any matching yes/no reply from any session, so the spoofing issue described by the PR remains present. The later leak issue/PR #71646/#71648 is orthogonal TTL/cleanup work and does not supersede session binding.

## Best Possible Solution

Keep the PR open and finish it as a session-binding security fix: require or otherwise reliably propagate a nonempty normalized session_key with Claude permission_request notifications, store it alongside each pending permission, compare it against the inbound message sessionKey before accepting yes/no replies, add attacker-session regression coverage, and update the MCP e2e/docs surfaces that currently show permission_request without session_key. Treat #71648 as parallel cleanup work, not the canonical replacement.
