## Summary

Keep PR #71671 open. Current main still has the stale-history behavior the PR targets: `getNoProgressStreak` scans backward past non-matching or missing-result records with `continue`, so older matching no-progress tool results can still be counted after an intervening tool call. Existing main tests cover contiguous no-progress blocking and progressing output, but I found no current-main regression test for another tool breaking the no-progress tail. The PR is a focused bug fix with matching regression coverage and no protected author/label reason to auto-close it.

## Best Possible Solution

Keep the PR open for maintainer review and land a scoped fix that makes no-progress streak counting use contiguous current-tail semantics, with the PR's regression tests or equivalent coverage for known polling and the global no-progress circuit breaker.
