## Summary

Keep #14317 open. Current main at 9a529ca78bfa0e257f662f0e70ab8c681caa6086 still does not support Slack groupPolicy "private": the shared type/schema only accept "open", "disabled", and "allowlist"; Slack message and slash routing do not pass private-channel metadata into policy evaluation; and Slack docs/tests still document and cover only the existing three modes. The prior Codex review comment on the issue remains consistent with this newer checkout.

## Best Possible Solution

Implement the feature in the bundled Slack plugin and aligned config/security contract: add a supported Slack groupPolicy "private" mode, carry or fetch Slack is_private metadata for message and slash routing, allow invited private channels while preserving explicit public-channel allowlist entries and explicit denies, update security audit guidance so private is not treated like open, and add schema, runtime, docs, and tests for private channels, public fallback, slash commands, and Slack Connect caveats.

## Risks / Open Questions

- Adding "private" to the shared GroupPolicy union would expose the mode to channels that may not have a room privacy concept; implementation should either define generic semantics or scope the schema/runtime carefully to Slack.
- Slack Connect/shared private channels can include external participants, so the feature needs an explicit caveat and tests/docs should not imply private means internal-only.
