## Summary

Keep #60576 open. Current main still lacks the proposed undefined/null guard in the process-level unhandled-rejection handler, so an unhandled rejection with no reason falls through to the generic fatal path. The PR is a focused core infra bug fix, not a ClawHub/plugin candidate, because the crash path is installed by core startup code and affects the process before Slack's existing reconnect loop can continue reliably.

## Best Possible Solution

Keep the PR open for maintainer review or rebase it onto current main. The best implementation is the narrow core guard proposed here: after custom `registerUnhandledRejectionHandler` hooks run, treat `undefined` and `null` rejection reasons as non-fatal with a warning, and add the two regression tests in `src/infra/unhandled-rejections.fatal-detection.test.ts`. This should remain in core infra rather than ClawHub because the failing behavior is the shared process-level unhandled-rejection policy.
