## Summary

Keep #60832 open. Current main still labels the composer plus action as “New session”, wires it to `/new`, and tests explicitly expect that tooltip. The implementation/docs also show `/new` and `/reset` both reset the current session key to a new sessionId rather than creating a visibly separate Sessions-tab row, so the reported UX ambiguity remains actionable.

## Best Possible Solution

Keep the issue open. The narrow fix is to rename the composer plus tooltip/aria-label and related test expectations to something like “Reset context” or “Start fresh context” so users understand it resets the current session key. If maintainers instead want the button to create a separate Sessions-tab conversation, wire the UI to `sessions.create`, switch `state.sessionKey` to the returned key, and update docs/tests to distinguish create versus reset.
