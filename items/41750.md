## Summary

Keep #41750 open. Current main has better CDP readiness probing, singleton-lock cleanup, and owned-process restart paths, but it still does not automatically identify and kill an unresponsive, non-owned local Chrome process that keeps the managed CDP port bound after a gateway restart. The issue comments provide useful manual/external recovery workarounds (`lsof`/kill and ClawAid), but the core/browser plugin behavior requested by the reporter remains missing.

## Best Possible Solution

Keep the issue open and implement a narrowly-scoped browser plugin fix: for local managed profiles only, when `/json/version`/CDP readiness fails but the managed CDP port is still occupied after gateway restart, inspect the listener, verify it is an OpenClaw-managed Chromium command/profile/port, terminate it with a bounded graceful-then-force path, wait for the port to become bindable, and then relaunch. Add tests for the unresponsive non-owned port holder path and ensure attachOnly/remote CDP profiles remain non-destructive.
