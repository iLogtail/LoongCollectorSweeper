## Summary

Keep #41750 open. Current main has stronger CDP health diagnostics, but it still does not implement the requested recovery for a loopback managed Chrome process that is alive, holds the CDP port, and does not answer /json/version. The existing comments provide useful workarounds, including ClawAid and manual lsof/kill recovery, but they do not make the core bug obsolete.

## Evidence

- **Startup still fails on occupied port before stale owner cleanup:** launchOpenClawChrome calls ensurePortAvailable(profile.cdpPort) before any launch or stale-process recovery. ensurePortAvailable only probes bind/listen availability and throws PortInUseError on EADDRINUSE, so a dead Chrome still holding 18800 blocks startup rather than being killed and replaced.

If no runtime state is recorded, it does a short retry and then calls launchOpenClawChrome; there is no code in this branch to identify or terminate the process holding the CDP port.
