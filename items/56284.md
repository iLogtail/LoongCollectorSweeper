## Summary

Keep open. Current main has useful restart-intent and drain infrastructure, but the normal Windows Scheduled Task restart path still does not prove the issue is fixed: the Scheduled Task runtime usually does not expose a PID for a running task, so the generic service restart intent is not written for the common Windows path, and `restartScheduledTask()` still uses `schtasks /End` plus listener termination before `/Run` rather than routing through the in-gateway idle drain/session-recovery restart path.

## Best Possible Solution

Keep the issue open and implement a Windows-specific graceful restart that reliably reaches the gateway-owned drain path before task termination. Concretely, either make the Scheduled Task runtime resolve the active gateway PID before `runServiceRestart()` writes restart intent, or use an RPC/SIGUSR1-equivalent restart request that runs inside the gateway; then wait for active tasks/embedded runs to drain, persist any required restart continuation/session recovery state, perform the supervisor handoff, wait for health, and add a Windows Scheduled Task regression test for a normally running task.
