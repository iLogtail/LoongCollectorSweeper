## Summary

Keep open. Current `main` still has the misleading QMD availability path in the doctor/deep-status probe: `checkQmdBinaryAvailability` spawns with the requested `cwd` but does not preflight whether that workspace exists, and `noteMemorySearchHealth` still reports any `ENOENT` as a qmd-binary startup failure with reinstall guidance. `extensions/memory-core/src/memory/search-manager.ts` does guard one runtime path by creating the workspace first, so the report is partially mitigated there, but the underlying shared probe behavior and user-facing warning remain unresolved. The issue timeline’s linked PR `#63167` also suggests this is recognized as real work rather than obsolete noise.

## Evidence

- **Shared QMD probe still does not validate `cwd` before spawn:** `checkQmdBinaryAvailability` resolves the spawn invocation and then immediately calls `spawn(..., { cwd: params.cwd ?? process.cwd() })`; there is no existence check or targeted workspace-missing error path before the probe runs.
