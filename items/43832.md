## Summary

Current `main` does not contain this PR’s graceful cron shutdown path or an equivalent fix. Gateway hot reload still stops cron synchronously and immediately replaces the service, while cron timer ticks and manual runs still do work outside the lock and persist state afterward, so the reported overwrite race remains plausible on `main`.

## Evidence

- **Hot reload still uses synchronous cron stop:** Gateway hot reload still calls `state.cronState.cron.stop()` and immediately builds/starts a replacement cron service. There is no awaited graceful drain before replacement.
