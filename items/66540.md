## Summary

Not safe to close. Current `main` still contains the reported `replyToCurrent === false` veto, the directive parser still defaults missing reply tags to `false`, and the queued followup delivery surface has no regression test proving this path is fixed.

## Evidence

- **Inline directive parsing still defaults missing reply tags to `replyToCurrent: false`:** `parseInlineDirectives` returns `replyToCurrent: false` both for empty text and for text without any `[[...]]` tags, so a missing tag is still encoded as a false value rather than as an explicit opt-out.
  - file: [src/utils/directive-tags.ts:170](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/utils/directive-tags.ts#L170)
  - command: `nl -ba src/utils/directive-tags.ts | sed -n '165,186p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **Reply threading helper still treats `replyToCurrent === false` as a hard stop for implicit `replyToId`:** `resolveReplyThreadingForPayload` still short-circuits when `params.payload.replyToCurrent === false`, matching the issue's reported condition exactly.
  - file: [src/auto-reply/reply/reply-payloads-base.ts:39](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/auto-reply/reply/reply-payloads-base.ts#L39)
