## Summary

Keep #22021 open. Current main still lacks `X-Actual-Model` or an equivalent HTTP transport response contract for the resolved runtime provider/model. The OpenAI-compatible Gateway handlers continue to echo the request-side agent target model in response bodies and SSE events, while the actual selected-model callback exists deeper in the reply pipeline and is not wired into these HTTP surfaces. The existing steipete/Codex issue comment remains accurate on current main.

## Best Possible Solution

Keep the issue open. Thread the resolved provider/model from `GetReplyOptions.onModelSelected` or a new run-result metadata seam into `src/gateway/openai-http.ts` and `src/gateway/openresponses-http.ts`; expose it consistently for non-streaming and streaming responses with a documented contract; add focused coverage in `src/gateway/openai-http.test.ts` and `src/gateway/openresponses-http.test.ts`; then update the public Gateway docs at https://docs.openclaw.ai/gateway/openai-http-api and https://docs.openclaw.ai/gateway/openresponses-http-api.

## Risks / Open Questions

- Streaming semantics need maintainer design: current SSE headers flush before runtime fallback selection is known, so the solution may need delayed header flush, trailers, or an SSE metadata event instead of only a header.
- The exact value contract should be documented before implementation, especially whether it is `provider/model`, a bare provider-reported model id, or includes fallback metadata.
