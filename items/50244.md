## Summary

Keep open. Current `main` still lacks the Feishu outbound success logging this PR targets, and the proposed PR behavior is not safe to auto-close as-is because reply sends in Feishu can transparently fall back to top-level direct sends. The dispatcher only knows whether a reply target was requested, not whether the send actually landed as a reply, so the PR's `mode: reply|send` logging can misreport successful fallback deliveries.

## Evidence

- **Current main has no outbound-success delivery log in the Feishu reply dispatcher:** `createFeishuReplyDispatcher` sends text, card, and media replies, but on current `main` it only logs streaming-session messages and failure paths. There is no success-side `runtime.log` emitted after `sendMessageFeishu`, `sendStructuredCardFeishu`, or `sendMediaFeishu` complete.
  - file: [extensions/feishu/src/reply-dispatcher.ts:405](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/extensions/feishu/src/reply-dispatcher.ts#L405)
  - command: `nl -ba extensions/feishu/src/reply-dispatcher.ts | sed -n '405,560p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Feishu text/card reply sends can silently degrade to direct sends:** `sendReplyOrFallbackDirect` calls `im.message.reply`, but for withdrawn/not-found targets it falls back to `im.message.create` whenever `replyInThread` is false. That means a successful result does not guarantee the message was delivered as a reply.
  - file: [extensions/feishu/src/send.ts:126](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/extensions/feishu/src/send.ts#L126)

- **Returned send results do not expose actual delivery mode:** `FeishuSendResult` only contains `messageId` and `chatId`. There is no field telling the dispatcher whether the send ultimately used reply-vs-create, so logging `mode` from `replyToMessageId` alone is not source-of-truth.
  - file: [extensions/feishu/src/types.ts:58](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/extensions/feishu/src/types.ts#L58)

That matches the misclassification concern already raised in the PR discussion: a requested reply is not always an actual reply delivery.
