## Summary

Keep open. Current main still reuses the in-memory cron store on normal operations without checking the backing file mtime, so the external-edit reload behavior proposed by this PR is not implemented. The later jobs-state split changes the exact patch shape, but it does not make the request obsolete because list/read/manual-run paths still call non-forced loading and can keep stale in-memory data.

## Best Possible Solution

Keep the PR open or ask for a rebase. The best implementation is a small cron-store invalidation fix in `src/cron/service/store.ts` that checks the relevant persisted snapshots before non-forced cache reuse, covers both `jobs.json` and the derived `jobs-state.json`, avoids the stat-before-read stale-cache issue noted in review, and keeps focused regression coverage for external deletion and manual runtime-state recovery.
