## Summary

Keep #54607 open. Current main appears to have the runtime side guarded: explicit channel disablement and explicit plugin disablement both suppress plugin auto-enable. The requested documentation is still not implemented in the BlueBubbles docs or examples: the BlueBubbles page only documents the channel `enabled` key, plugin docs cover generic enablement rules elsewhere, and the safe loopback/token examples do not show BlueBubbles disabled at both the channel and plugin layers or explain startup mutation/auto-enable semantics in one place.

## Best Possible Solution

Keep this open as a documentation task. Add a short section to `docs/channels/bluebubbles.md`, with links from the configuration examples or gateway security page if useful, that explains `channels.bluebubbles.enabled=false` versus `plugins.entries.bluebubbles.enabled=false`, includes the requested loopback/token JSON5 example with BlueBubbles disabled at both layers, and explicitly documents current startup behavior around auto-enable, config repair/metadata writes, and whether extra channel keys count as configured when `enabled=false`. The code path already appears to suppress auto-enable correctly, so this should likely be docs-only unless further verification finds a runtime mismatch.
