## Summary

Keep #71216 open. Current main has adjacent controls, but not the requested config schema/runtime surface: sandboxing is under `agents.defaults.sandbox` / per-agent config rather than root `sandbox`; top-level `routing` is legacy and current routing is agent/binding-based, not tag-based provider selection; multi-gateway operation still uses separate profiles/config/state/ports rather than an `instances` map; and `gateway.nodes` supports command allow/deny only, with no `denyPaths`.

## Best Possible Solution

Keep the issue open and split the path deliberately: document or alias root sandbox only if maintainers want a root policy surface over `agents.defaults.sandbox`; design tag-based provider/model routing in the agent model-selection/runtime layer with clear metadata propagation; decide whether multi-instance belongs in runtime config or gateway/profile management; and, if `gateway.nodes.denyPaths` is accepted, add a generic gateway-enforced path guard that covers node/tool filesystem access consistently, with zod/types/generated schema, docs, doctor/security audit, and focused regression tests updated together.
