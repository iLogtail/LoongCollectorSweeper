## Summary

Current main already ships the supported shared rich-message path through `ReplyPayload.presentation` / `MessagePresentation`, with native channel renderers for Slack, Discord, Telegram, and Teams. That supersedes this PR’s older `<!--adaptive-card-->` marker-parsing approach.

## Risks / Open Questions

- Current main does not preserve this PR’s exact `<!--adaptive-card-->` marker transport format; external integrations using that hidden format would need to emit the supported `presentation` contract instead.
- Teams uses the shared presentation path through its message action/card builder rather than the exact outbound-adapter wiring proposed in this PR.

## Close Comment

Closing after Codex review. Current main already ships the supported shared rich-message path through `ReplyPayload.presentation` / `MessagePresentation`, and the docs now explicitly direct new shared UI work to that contract instead of provider-native card/block/component fields (`docs/plugins/message-presentation.md:11`, `docs/plugins/architecture-internals.md:615`). The released tree in `v2026.4.22` already includes native renderers for Slack (`extensions/slack/src/outbound-adapter.ts:133`), Discord (`extensions/discord/src/outbound-adapter.ts:157` and `extensions/discord/src/shared-interactive.ts:87`), Telegram (`extensions/telegram/src/outbound-adapter.ts:120`), and Teams Adaptive Cards (`extensions/msteams/src/presentation.ts:4`, `extensions/msteams/src/actions.ts:290`). Because that shared implementation is already on main and shipped, this older comment-marker-based PR is redundant, so I’m closing it as implemented on main.
