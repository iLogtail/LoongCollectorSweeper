## Summary

Keep open. Current main has partial failover observability, but it does not satisfy the issue's central request: the terminal wrapper failure still lacks structured session_id/sessionKey/lane metadata, and the failover error types/logging API do not carry session attribution. Related ClawSweeper context such as #42469 covers provider error/fallback behavior, not this structured logging attribution gap.

## Best Possible Solution

Add session attribution to the model-fallback surface: pass sessionId/sessionKey or lane into runWithModelFallback from agent-command/auto-reply, include it in FallbackSummaryError or the final logging context, and extend model_fallback_decision / exhausted-wrapper logging tests to assert attempted provider chain plus normalized session_id/sessionKey fields. Keep the existing per-attempt provider/model logging as the base rather than replacing it.
