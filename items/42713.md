## Summary

Current main has per-candidate fallback metadata, but the reported wrapper/terminal failure path still does not carry the requested session_id metadata, and the final user-facing error logging path remains string-only. The issue should stay open.

## Evidence

- **FailoverError metadata lacks session id:** FailoverError currently carries reason, provider, model, profileId, status, code, and rawError, but no sessionId/session_id field, so propagating the originating session through this wrapper is still not implemented there.

- **Fallback summary attempts carry provider/model but not session id:** FallbackAttempt records provider, model, error, reason, status, and code. FallbackSummaryError stores attempts and soonestCooldownExpiry, but there is no sessionId/session_id field on the summary or attempt shape.
