## Summary

Keep open. Current `main` does not implement this PR’s heartbeat timer-overflow fix: heartbeat intervals are still returned without a 32-bit timer clamp, the scheduler still passes the raw delay to `setTimeout`, and config validation still does not reject oversized `heartbeat.every` values.

## Evidence

- **Interval resolution still returns oversized delays unchanged:** `resolveHeartbeatIntervalMs` parses the configured duration and returns it as long as it is positive; there is no max-timer clamp or warning path. A `30d` interval is 2,592,000,000 ms, which exceeds Node’s signed 32-bit timer ceiling of 2,147,483,647 ms.
  - file: [src/infra/heartbeat-summary.ts:44](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/infra/heartbeat-summary.ts#L44)
  - command: `nl -ba src/infra/heartbeat-summary.ts | sed -n '40,72p'; node -p '({requestedMs: 30*24*60*60*1000, maxTimerMs: 2**31-1, exceeds: 30*24*60*60*1000 > 2**31-1})'`
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Scheduler still feeds the raw delay into setTimeout:** `scheduleNext` computes `const delay = Math.max(0, nextDue - now)` and passes that directly to `setTimeout(...)`; there is no defensive clamp before scheduling.
  - file: [src/infra/heartbeat-runner.ts:1368](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/infra/heartbeat-runner.ts#L1368)
