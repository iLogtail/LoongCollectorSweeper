## Summary

Current `main` does not implement `memorySearch.remote.nonBatchConcurrency`, and the PR still appears incomplete because the runtime config schema would reject the new key. This is still actionable work, not a conservative close candidate.

## Evidence

- **Non-batch indexing still uses the old fixed fallback:** `getIndexConcurrency()` on current `main` still returns `this.batch.concurrency` when batch mode is enabled, otherwise the hardcoded `EMBEDDING_INDEX_CONCURRENCY` default. There is no provider-specific or configurable non-batch concurrency path.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:466](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/memory-core/src/memory/manager-embedding-ops.ts#L466)

- **Resolved memory config has no `nonBatchConcurrency` field:** Current `ResolvedMemorySearchConfig.remote` only contains `baseUrl`, `apiKey`, `headers`, and `batch`, and the merge logic only propagates those fields. The proposed setting is not present on `main`.
  - file: [src/agents/memory-search.ts:21](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/agents/memory-search.ts#L21)
