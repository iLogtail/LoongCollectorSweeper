## Summary

Keep #53319 open. Current main has moved far beyond the reported OpenClaw 2026.3.22 / acpx 0.1.16 stack and includes ACPX first-turn/dead-handle recovery work, but I did not find a direct implementation, regression test, or release note proving the specific concurrent `sessions_spawn(runtime:"acp")` first-child launch race is fixed. The current ACP manager still allows different ACP sessions to run in parallel, and the tests explicitly lock that in, so a backend concurrent-init race would still be actionable if it reproduces on current acpx.

## Best Possible Solution

Keep the issue open until a maintainer or contributor verifies current main with the bundled acpx 0.6.1+ and two rapid `sessions_spawn(runtime:"acp", agentId:"claude")` calls. If it still reproduces, add a focused regression around concurrent ACP spawn initialization/run dispatch for distinct session keys and fix the ACPX runtime/control-plane launch path; if current acpx no longer reproduces, close later with that live proof and the exact fixing SHA/release.
