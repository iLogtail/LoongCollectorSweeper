## Summary

Keep #50590 open. Current main does include a sandbox skill-copy mechanism, and the related #46257 was closed as implemented, but this issue’s narrower failure can still occur in the current source path: `agent-command` builds and persists a skill snapshot from the original agent workspace before the embedded run resolves the sandbox workspace, then the run reuses that snapshot prompt even after sandbox skill sync. Ryce’s KeepMyClaw comment is useful as external validation/workaround context, but it does not remove the remaining OpenClaw runtime contract gap.

## Best Possible Solution

Keep the issue open and fix the remaining snapshot/prompt ordering gap. The safest product path is to resolve the sandbox workspace before building the skill snapshot for sandboxed non-`rw` sessions, or to rebuild/rewrite `skillsSnapshot.prompt` and `resolvedSkills` from the sandbox-local `skills/` copy after `syncSkillsToWorkspace`. Add a regression that creates a managed/global skill, starts a non-main sandboxed session with a prebuilt snapshot, and asserts `<available_skills><location>` points to the sandbox-local `skills/<name>/SKILL.md` path that the read tool can access.
