## Summary

Current main and the v2026.4.24 release already solve the PR's central MCP context-budget/tool-policy problem through the shipped final effective tool-policy pass: bundled MCP/LSP tools are filtered by the normal allow/deny pipeline before they are exposed to the model. The exact nested `mcp.servers.<server>.tools` discovery-time config shape from this PR is not present, but users can filter individual MCP tools via their exposed `serverName__toolName` IDs or the `bundle-mcp` plugin key, and the linked #65612 bypass is now fixed.

## Best Possible Solution

Close this PR as implemented/superseded by the shipped final tool-policy implementation. Keep the documented general `tools.allow` / `tools.deny` approach for MCP tools; if maintainers later want a nested per-server MCP config alias or discovery-time startup optimization, that should be a fresh product/API discussion with docs and schema changes.
