## Summary

Keep open. Current main has named browser profiles, an optional sandboxed browser path, Chrome MCP session caching by profile, and per-session tab cleanup, but the ordinary host browser tool still defaults all agents to the same configured browser profile/Chrome instance. `agentSessionKey` is used for tab tracking and lifecycle cleanup, not for selecting or creating an isolated per-agent Chrome process/profile.

## Best Possible Solution

Keep #61831 open for an explicit browser isolation decision. The likely maintainer path is either to implement first-class per-agent browser profile resolution for the host browser tool, including deterministic CDP port/user-data-dir allocation and lifecycle/MCP cache separation, or to document sandboxed browser `scope: "agent"`/`"session"` as the supported isolation model and make the host shared-profile boundary clear in browser docs and tool guidance.

## Risks / Open Questions

- Do not close this as implemented based only on per-session tab cleanup; tabs can be cleaned up while cookies, localStorage, selected page state, and the underlying Chrome/MCP process remain shared for callers using the same profile.
- Optional sandboxed browser containers may be an acceptable documented workaround for some deployments, but maintainers still need a product decision for unsandboxed host browser behavior and Chrome MCP existing-session concurrency.
