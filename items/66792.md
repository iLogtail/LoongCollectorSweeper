## Summary

Keep open. Current main improves Active Memory fail-open behavior and default memory-slot startup, but it does not implement or prove the requested lightweight `runEmbeddedPiAgent` bootstrap improvement. The issue is recent and has fresh v2026.4.24 evidence that the Active Memory embedded run still can spend its budget before any useful model call.

## Best Possible Solution

Keep this open and narrow it to embedded Pi runner cold-path profiling for blocking plugins. Preserve the shipped Active Memory hard timeout and default memory-core startup load, then add stage timings around lane wait, runtime plugin load, auth/profile resolution, context-engine/bootstrap resolution, session lock/session manager creation, tool materialization, resource loader reload, and prompt start. After measuring, cache or bypass safe per-run invariants for `bootstrapContextMode: "lightweight"`, and update Active Memory docs with measured latency guidance.
