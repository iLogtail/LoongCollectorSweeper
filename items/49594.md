## Summary

Keep open. Current `main` still checks `node` before any nvm bootstrap, so the `curl | bash` path described in #49556 remains plausible, and the PR itself has unresolved review concerns that make it unsafe to treat as already handled.

## Evidence

- **Current main still runs Node detection before any nvm setup:** `main()` goes from Homebrew setup straight to `check_node()` and only afterwards calls `ensure_default_node_active_shell()`. There is no `NVM_DIR` or `nvm.sh` bootstrap before detection, so a non-interactive `curl | bash` shell still depends on its inherited `PATH`.
  - file: [scripts/install.sh:2323](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/scripts/install.sh#L2323)
  - command: `nl -ba scripts/install.sh | sed -n '2310,2345p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Node detection still uses the active PATH only:** `check_node()` gates on `command -v node` and `node -v`. In the reported scenario, that means the installer will inspect whatever `node` the unsourced shell can already see, not an nvm-managed runtime hidden behind shell init.
  - file: [scripts/install.sh:1376](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/scripts/install.sh#L1376)
  - command: `nl -ba scripts/install.sh | sed -n '1230,1415p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **The later nvm hint is conditional on nvm already being visible:** `ensure_default_node_active_shell()` only recognizes nvm when `NVM_DIR` is set, the active `node` path already contains `/.nvm/`, or `command -v nvm` succeeds. In a `curl | bash` shell where rc files were not sourced, all three checks can miss nvm and fall back to generic PATH guidance.
  - file: [scripts/install.sh:1353](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/scripts/install.sh#L1353)
  - command: `nl -ba scripts/install.sh | sed -n '1230,1415p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Docs still describe the same edge case rather than a shipped fix:** The Node install docs warn that version managers must be initialized from `~/.zshrc` or `~/.bashrc`, while the installer docs still present `curl -fsSL https://openclaw.ai/install.sh | bash` as the standard flow. That aligns with the report; it does not show the problem was already solved on `main`.
  - file: [docs/install/node.md:68](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/docs/install/node.md#L68)
  - command: `nl -ba docs/install/node.md | sed -n '1,180p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **PR discussion identifies unresolved correctness problems in the proposed patch:** The provided review comments flagged that sourcing `nvm.sh` with `2>/dev/null` can still leak stdout into installer output, and sourcing plain `nvm.sh` can switch the active Node version unless loaded in `--no-use` mode. That is evidence to keep the PR open for revision, not to close it as obsolete.
  - file: [scripts/install.sh:2350](https://github.com/openclaw/openclaw/blob/631caec633a6ba486301d8a93fd6ee3aa47062df/scripts/install.sh#L2350)
