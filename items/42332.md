## Summary

Current main does not implement the requested sessionKey-based 409 concurrency guard for core webhook deliveries. The core /hooks handler resolves sessionKey and has idempotency replay dedupe, but it dispatches a new async isolated-agent run and returns 200; repeated non-idempotent deliveries can still start another run. Plugin SDK in-flight helpers exist for plugin-owned webhook ingress, which is useful related work, but they are not wired into the core /hooks sessionKey dispatch path and default to 429, not the requested active-session 409. Related PR #27193 was closed unmerged and addressed persistent session reuse rather than this concurrency guard, so it does not supersede the issue.

## Risks / Open Questions

- A deeper runtime path may avoid some duplicate work by returning an in-band warning when the same session is already active, but that still happens after HTTP 200 acceptance and does not satisfy the webhook caller contract requested in #42332.
- The plugin SDK in-flight limiter is useful related infrastructure, so a future fix might reuse it or add a small core-specific sessionKey limiter rather than inventing a separate mechanism.
