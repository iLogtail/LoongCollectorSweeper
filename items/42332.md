## Summary

Keep #42332 open. Current main supports hook sessionKey resolution, prefix policy, and idempotency-key replay dedupe, but it does not implement a sessionKey active-run guard or return 409 for overlapping webhook deliveries. Hook agent dispatch still starts an isolated cron-style turn asynchronously and returns 200 with a runId. The related #27193 PR was closed unmerged and addressed persistent session reuse, not the requested active-conflict semantics; #54325 is cron-specific and does not supersede this webhook contract request.

## Best Possible Solution

Keep the issue open and implement a webhook-specific active-run guard around the normalized dispatch sessionKey in the hook agent path, covering both /hooks/agent and mapped agent hooks. The likely fix belongs near createHooksRequestHandler/createHooksRequestHandler dispatch: register an active run before runCronIsolatedAgentTurn starts, return HTTP 409 for a second active delivery with the same normalized sessionKey, clear the guard in finally, and add gateway hook tests for direct hooks, mapped hooks, idempotency interaction, and agentId-scoped sessionKey normalization.

## Risks / Open Questions

- The implementation needs a clear HTTP contract for how this interacts with idempotency-key replay: same idempotency key may still reasonably return cached 200, while same sessionKey with a different/no idempotency key should return 409 if the guard is enabled.
- The guard must release reliably on success, error, timeout, and process-local failure paths, and should account for normalized agent-scoped session keys so multi-agent webhook deliveries do not block each other incorrectly.
