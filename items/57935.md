## Summary

Keep open. I did not find strong evidence that current `main` fixes this report. The Telegram polling path on `main` still reuses a persisted update offset on restart, skips any update at or below that watermark, and only does a non-fatal `getUpdates(offset+1)` probe before starting the runner. If the stored offset is stale-but-valid, the exact symptom in the issue remains plausible: outgoing sends still work while inbound `getUpdates` stay empty after a gateway restart.

## Evidence
