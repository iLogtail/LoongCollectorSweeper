## Summary

Do not close PR #65606. Current main still treats QQBot close code 4009 as a session-invalidating re-identify path, while the QQ Bot WebSocket docs say 4009 should reconnect with resume. The PR is a plausible focused bug fix, but it needs to be ported to the current refactored gateway files rather than the old `extensions/qqbot/src/gateway.ts` path.

## Best Possible Solution

Keep the PR open and request a port to current main: update `extensions/qqbot/src/engine/gateway/reconnect.ts` so `GatewayCloseCode.SESSION_TIMEOUT` preserves session state and describes a resume reconnect, update the `SESSION_TIMEOUT` comment in `constants.ts`, add a focused `ReconnectState.handleClose` test for 4006, 4007, 4009, and unrelated codes, then run the QQBot extension test lane and changed checks.
