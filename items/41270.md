## Summary

Keep open. Current `main` still preserves different semantics: `config.set` writes validated config without scheduling restart, while `config.patch`/`config.apply` are the restart-capable control-plane writes. The report is still actionable as a docs/UX consistency issue, even though one detail is outdated: `config.patch` no longer always restarts and now hot-reloads when possible.

## Risks / Open Questions

- Part of the original report is stale: current `main` no longer unconditionally restarts on `config.patch`; it hot-reloads when possible and restarts only when required.
- Because the current behavior is explicitly tested, closing as `cannot_reproduce` or `implemented_on_main` would be incorrect; the behavioral difference still exists.
- Docs partially explain the split for Control UI, but CLI-facing docs still do not clearly frame `config.set` as a write-only change that requires a manual restart.
