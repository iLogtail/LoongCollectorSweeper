## Summary

Keep #71487 open. Current main still has backend/CLI TTS state and a per-message browser-native Read aloud button, but it does not provide the requested first-class Control UI Settings TTS toggle, default provider selector, default voice picker, or sample playback surface. The existing maintainer-thread comment also explicitly kept this issue open, and the related items in the provided context do not supersede this request.

## Best Possible Solution

Keep the issue open. Implement a first-class Text-to-Speech section in Control UI Settings that calls the existing tts.status, tts.providers, tts.enable, tts.disable, tts.setProvider, and tts.convert RPCs; clearly shows configured versus unavailable providers; filters voices by provider; and adds a small persistence/API design for default voice or uses documented TTS persona/provider config deliberately. Cover the UI state and RPC behavior with focused tests.

## Risks / Open Questions

- The generic advanced config editor may expose some messages.tts provider fields, so the eventual UI should avoid creating conflicting sources of truth between openclaw.json provider config and local ~/.openclaw/settings/tts.json preferences.
- Persistent default voice selection likely needs either a narrow new TTS prefs/RPC surface or a deliberate provider/persona config-write path; current RPCs support one-shot voiceId on tts.convert but not a saved default voice preference.
