## Summary

Keep #71487 open. Current main has backend TTS state/RPCs, CLI commands, documentation, provider voice-listing support, generic advanced config labels, and a per-message browser-native “Read aloud” button in webchat. It still does not provide the requested first-class Control UI Settings surface that reflects and persists the TTS enabled state, default provider, or default voice, and the latest issue discussion already recorded the same keep-open conclusion.

## Best Possible Solution

Keep the issue open and implement a first-class Text-to-Speech panel in Control UI Settings. It should use the existing tts.status/providers/enable/disable/setProvider/convert RPCs where possible, clearly show configured versus unavailable providers, filter voice choices by selected provider using provider voice metadata or listVoices support, add or choose a safe persistence path for default voice, and cover the UI/RPC behavior with focused tests.
