## Summary

Keep open. Current main has a manifest-backed command-alias mechanism that fixes this class only when a plugin advertises the alias owner, but the currently published lossless-claw package still does not declare `lcm` or `lossless`/`/lossless` in `commandAliases`. For this specific report, the CLI can still fall through to the generic `plugins.allow excludes "<command>"` message when the owner is unknown, while validation still treats command tokens as non-plugin ids.

## Best Possible Solution

Keep this open until the contradiction is removed for the lossless case. The best path is to stop emitting `Add "<command>" to plugins.allow` unless the token is a real plugin id or a manifest-backed alias with a known parent, add regression coverage with a lossless-like external plugin fixture, and coordinate with lossless-claw to publish `commandAliases` for `lcm`/`lossless` as runtime slash/native command aliases if OpenClaw should give plugin-aware diagnostics for those names.
