## Summary

Keep #50642 open. Current main and latest release v2026.4.24 still set `allowTOFU` for a fresh macOS node `wss://` gateway and accept/store the first certificate before `SecTrustEvaluateWithError`. The linked #50643 is a relevant open fix attempt, but it is unmerged. Ryce's comment also frames snapshot restore as resilience only; first-connection TLS hardening remains the actual fix.

## Best Possible Solution

Keep the issue open and route it through security/product review. The best code path is to land a macOS/shared TLS fix that either performs normal `SecTrustEvaluateWithError` before first-use pinning or requires an explicit configured/confirmed fingerprint, wire `gateway.remote.tlsFingerprint` into the macOS direct-node path, and add focused macOS/shared TLS regression coverage. PR #50643 is a useful starting point but should be reviewed against the intended self-signed/reverse-proxy workflow before merge.
