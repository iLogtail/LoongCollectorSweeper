## Summary

Keep #71170 open. Current main does not implement the PR’s `agents.defaults.reminderGuard` or per-agent `reminderGuard` opt-out, and the existing auto-reply reminder guard still only suppresses the disclaimer for core `cron.add` success or an existing enabled session cron job. The linked #70666 report describes a real plugin-tool false positive, and the PR’s proposed config-based workaround is still relevant even though a future plugin SDK signal may be the cleaner long-term contract.

## Best Possible Solution

Keep #71170 open and either merge or adapt its config-based `reminderGuard` opt-out after maintainer review. Longer term, a plugin SDK signal for scheduler success may be worth a separate CODEOWNER-reviewed proposal, but the current core auto-reply guard still needs a shipped mitigation for external schedulers.
