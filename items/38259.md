## Summary

Keep open. Current `main` still loads `.env` values literally and does not expand `${VAR}` or `$VAR` references before copying them into `process.env`, so the reported behavior remains plausible and is not implemented on `main`.

## Evidence

- **Dotenv loader copies parsed values verbatim:** `readDotEnvFile` uses `dotenv.parse(content)` and stores the returned `value` directly in `entries`; the loaders then assign that value straight into `process.env` with no interpolation pass.
  - file: [src/infra/dotenv.ts:148](https://github.com/openclaw/openclaw/blob/53aac30f5104052db56da2b26bcee0f4e5140325/src/infra/dotenv.ts#L148)
  - command: `nl -ba src/infra/dotenv.ts | sed -n '146,230p'`
  - sha: [53aac30f5104](https://github.com/openclaw/openclaw/commit/53aac30f5104052db56da2b26bcee0f4e5140325)
- **Both workspace and runtime dotenv paths are raw assignments:** `loadRuntimeDotEnvFile`, `loadWorkspaceDotEnvFile`, and `loadParsedDotEnvFiles` all set `process.env[key] = value` from parsed dotenv entries; there is no expansion step anywhere in the dotenv loading path.
  - file: [src/infra/dotenv.ts:166](https://github.com/openclaw/openclaw/blob/53aac30f5104052db56da2b26bcee0f4e5140325/src/infra/dotenv.ts#L166)
  - command: `nl -ba src/infra/dotenv.ts | sed -n '166,230p'`
  - sha: [53aac30f5104](https://github.com/openclaw/openclaw/commit/53aac30f5104052db56da2b26bcee0f4e5140325)
- **Env substitution is documented and implemented for config strings, not `.env` files:** Docs say `${VAR_NAME}` substitution applies to config string values, and `src/config/env-substitution.ts` implements that config-time substitution separately. That is a different path from dotenv loading.
  - file: [docs/help/environment.md:77](https://github.com/openclaw/openclaw/blob/53aac30f5104052db56da2b26bcee0f4e5140325/docs/help/environment.md#L77)
  - command: `nl -ba docs/help/environment.md | sed -n '77,95p'`
  - sha: [53aac30f5104](https://github.com/openclaw/openclaw/commit/53aac30f5104052db56da2b26bcee0f4e5140325)
- **Tests cover config substitution after dotenv load, not dotenv self-expansion:** `src/config/config.env-vars.test.ts` verifies a plain `.env` value like `BRAVE_API_KEY=from-dotenv` can later satisfy a config string `${BRAVE_API_KEY}` via `resolveConfigEnvVars`, which confirms substitution happens in config resolution rather than while parsing `.env` files.
  - file: [src/config/config.env-vars.test.ts:104](https://github.com/openclaw/openclaw/blob/53aac30f5104052db56da2b26bcee0f4e5140325/src/config/config.env-vars.test.ts#L104)
  - command: `nl -ba src/config/config.env-vars.test.ts | sed -n '104,132p'`
  - sha: [53aac30f5104](https://github.com/openclaw/openclaw/commit/53aac30f5104052db56da2b26bcee0f4e5140325)
- **Runtime consumers still read process env directly:** The OpenAI realtime transcription provider checks `process.env.OPENAI_API_KEY` directly when configuring a session, so a literal placeholder loaded from `.env` would propagate into runtime auth rather than being expanded first.
  - file: [extensions/openai/realtime-transcription-provider.ts:156](https://github.com/openclaw/openclaw/blob/53aac30f5104052db56da2b26bcee0f4e5140325/extensions/openai/realtime-transcription-provider.ts#L156)
