## Summary

Keep open. Current main still loads dotenv values literally: `src/infra/dotenv.ts` parses `.env` content with `dotenv.parse(...)` and assigns the parsed string directly into `process.env` for workspace, runtime, and global fallback loaders. Config-level `${VAR}` substitution exists, but it only applies to parsed config values and does not expand aliases inside loaded `.env` entries. The related fix PR #38254 was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Implement dotenv-level expansion in `src/infra/dotenv.ts` before assignment to `process.env`, preserving shell/process precedence and CWD-over-global fallback behavior. Add focused regression tests in `src/infra/dotenv.test.ts` for shell-backed aliases, same-file aliases, chained references, missing references, and both workspace and global `.env` paths, reusing the intent from closed PR #38254 while re-checking workspace `.env` security constraints.
