## Summary

Keep open. Current main still contains the reported empty-catch pattern in several of the named areas, and broader source search shows the repo has not adopted the requested logging/comment convention or a lint guard for empty catch bodies. The predecessor #63359 was closed as superseded by this issue, not by a merged fix.

## Best Possible Solution

Keep this issue open and do a scoped cleanup rather than a blanket mechanical change: add comments for intentionally ignored fallback probes, log unexpected failures where a logger already exists or can be injected cleanly, and consider an oxlint/no-empty policy only after auditing intentional catch bodies. Pairing allowFrom, config IO recovery, and other developer-facing failure paths should get targeted regression coverage where the swallowed error affects diagnostics.
