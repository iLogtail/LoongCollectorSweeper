## Summary

Keep #19362 open. Current main still does not register or implement a Discord channel-delete gateway listener, while Discord guild channels are documented and persisted as isolated channel session keys. The PR remains relevant, but it needs to be ported to the current `extensions/discord` layout and should address the transcript archival/cleanup issues already raised in review.

## Best Possible Solution

Keep the PR open and refresh it against current main. The best path is to implement deleted-channel cleanup in `extensions/discord/src/monitor`: add/register a Carbon channel-delete listener, validate the deleted channel ID, remove or reset matching `agent:<agentId>:discord:channel:<channelId>` entries through lock-aware session-store helpers, archive and clean up transcripts with the same semantics as current session deletion, and add focused tests for listener registration, multi-agent/channel-key cleanup, and transcript retention.

## Risks / Open Questions

- Merging the current PR as-is would be risky because it targets the old `src/discord` layout in its body/diff, while current main has the Discord plugin under `extensions/discord`.
- Transcript handling needs care: the implementation should collect deleted entries inside the `updateSessionStore` lock and call the current archive cleanup helper so deleted-channel cleanup does not lose or leak transcript files.
- The Discord plugin must stay within the plugin/SDK boundary; if a refreshed implementation needs an agent-listing or transcript-archive seam not currently exported to plugins, that seam should be added deliberately rather than deep-importing core internals.
