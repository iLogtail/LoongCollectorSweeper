## Summary

Keep open. Current main still rejects any node_modules symlink whose real target escapes the plugin install root, and the install flow scans the package source before the host openclaw peer link is created. That leaves the PR's reported workspace/package-manager peer-link case unsolved on main. The related symlink-discovery PRs are about plugin discovery/profile install behavior, not this install security-scan exception.

## Best Possible Solution

Keep the PR open and review it as the current candidate fix for package-manager-created openclaw peer symlinks. The best product/code direction is a narrow scanner exception for exactly node_modules/openclaw and node_modules/.bin/openclaw when their real targets resolve under the trusted host OpenClaw package root, with regression tests proving spoofed, nested, non-exact, and arbitrary escaping node_modules symlinks remain blocked.
