## Summary

Keep open. Current `main` still has the `assistantTexts` vs. post-hook `lastAssistant` mismatch this PR is trying to address, and the PR itself has an unresolved regression in the blocked-turn fallback noted in review.

## Evidence

- **Current main still builds final payloads from raw `assistantTexts`:** `run.ts` on `main` passes `attempt.assistantTexts` and `attempt.lastAssistant` straight into `buildEmbeddedRunPayloads()` with no `before_message_write` hook reconciliation.
  - file: [src/agents/pi-embedded-runner/run.ts:1748](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/agents/pi-embedded-runner/run.ts#L1748)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '1728,1768p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Payload construction prefers `assistantTexts` over `lastAssistant` text:** When `assistantTexts` is non-empty, `buildEmbeddedRunPayloads()` uses it before the fallback text extracted from `lastAssistant`, so stale pre-hook text would win over the post-hook snapshot.
  - file: [src/agents/pi-embedded-runner/run/payloads.ts:304](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/agents/pi-embedded-runner/run/payloads.ts#L304)
