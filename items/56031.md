## Summary

Keep open. Current `main` still routes embedded-run final payload construction through the raw collected `attempt.assistantTexts` plus `attempt.lastAssistant`, while `before_message_write` can mutate or block the persisted session message before it appears in `messagesSnapshot`. The PR is a plausible focused fix for a real core runner mismatch, but the existing review comment about falling back to a previous assistant when the latest turn is fully blocked means the PR should be revised rather than auto-closed.

## Best Possible Solution

Keep the PR open and ask for a corrected version of the post-hook payload selection. The fix should make `--expect-final` use the post-hook current-turn assistant from the persisted snapshot when `before_message_write` hooks are present, avoid raw `assistantTexts` in that path, and add regression coverage for replacement, assistant block, and the reviewed edge case where both the current user and assistant are blocked so no previous-turn assistant is emitted.
