## Summary

Keep open. Current `origin/main` still matches the reported behavior: Discord guild/channel `users` allowlists are enforced as the primary guild sender gate, the preflight path still drops disallowed senders before bot mention handling, and the recognized Discord allowlist prefixes still exclude `bot:`. The docs on `main` mention `allowBots` but do not document either of those gotchas yet. The issue is also actively tracked in open PR #59904 per the issue discussion.

## Evidence

- **Current Discord docs still omit the reported allowlist caveats:** `docs/gateway/config-channels.md` shows `allowFrom` and `guilds.<id>.users` in the example config and documents `allowBots`, but it does not say that non-empty `guilds.<id>.users`/channel `users` is the primary guild sender gate or that `bot:` is unsupported in `allowFrom`.

## Risks / Open Questions

- Bot-to-bot Discord setups on current `main` can still fail with silent guild sender drops if operators follow the docs but omit the bot ID from `guilds.<id>.users`.
- Operators can still reasonably try `bot:<id>` in `allowFrom`; current runtime will not treat that as a recognized ID prefix unless `dangerouslyAllowNameMatching` behavior happens to mask it.
- Closing now would discard an issue that still matches current `main` and is already linked to an active docs PR.
