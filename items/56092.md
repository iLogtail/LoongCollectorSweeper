## Summary

Keep #56092 open. Current main still has the core routing bug the PR targets: `runHeartbeatOnce` passes wake `sessionKey` as `forcedSessionKey`, and `resolveHeartbeatSession` resolves that forced key before consulting configured `heartbeat.session`. The public config/types docs still describe `heartbeat.session` as the session key for heartbeat runs, so the configured-session contract remains meaningful. Current tests cover configured and forced overrides separately, but not their conflict, and there is no current-main migration path for pending system events when a wake-origin session differs from the configured heartbeat session. The PR is still relevant, though it should be rebased and tightened for current main, especially around the latest unresolved review concern about canonicalizing the source session before queue migration.

## Best Possible Solution

Keep the PR or a replacement fix open, but rebase and narrow it against current main: resolve configured `heartbeat.session` before ordinary forced wake sessions, keep the subagent/global guards, migrate only validated/canonicalized origin system events into the actual heartbeat run queue, preserve `trusted` and delivery context, bypass dedup only for deliberate migration, and add focused regression tests for configured-session-vs-forced-session conflict plus migration/skip-restore edge cases.
