## Summary

Keep PR #71820 open. Current main still lacks an inbound BlueBubbles reply-context HTTP API fallback: it extracts reply metadata from the webhook, checks the process-local reply cache, then proceeds without fetching the original message on cache miss. The related #43764/#65447 work shipped outbound Private API status refresh for reply threading/effects, but not inbound reply-context recovery. The draft PR is still useful, though its SSRF and config-inheritance review comments need fixes before merge.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to revise the fallback so the original-message lookup reuses BlueBubblesClient or an equivalently guarded request path for GET /api/v1/message/{guid}, preserve channel-level defaults when account configs omit the new flag, keep the behavior opt-in unless maintainers choose otherwise, and retain focused tests for cache misses, failures, dedupe, config inheritance, and SSRF policy.
