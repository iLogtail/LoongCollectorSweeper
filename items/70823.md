## Summary

Keep #70823 open. Current main still models plugin slots as bare string IDs, normalizes and compares those strings directly, and writes selected slot IDs back as strings. The RFC asks for a public config/plugin-contract change with read-side object normalization, provenance fields, and overwrite invariants; that is not implemented or superseded. This is also not a ClawHub close because the missing piece is a core config/schema/plugin API capability that external plugins cannot provide while the core schema rejects object slot values.

## Best Possible Solution

Keep the RFC open for maintainer design review. The likely implementation path is a backwards-compatible core/plugin-config contract change: add a typed slot ownership object union, normalize bare strings to `{ owner }` on read, update install/update/uninstall/selection paths to write canonical ownership metadata, add an explicit overwrite policy with force/allow-override semantics, and update config docs plus focused tests for legacy strings, object slots, default slots, `"none"`, and conflict handling.
