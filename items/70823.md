## Summary

Keep open. Current main still treats `plugins.slots.*` as bare plugin-id strings across the public config type, Zod/runtime schema, normalization, activation checks, install slot selection, and docs. The issue is a coherent core config/plugin-contract RFC with a concrete `@plur-ai/claw` repair use case and a plur-ai/plur#51 cross-reference, not a ClawHub-only plugin request or an already-implemented behavior.

## Best Possible Solution

Keep the RFC open for maintainer design review. A conservative implementation path would introduce a shared slot-claim type, accept both string and object forms on read, expose helpers for normalized owner/provenance access, update install and slot mutation paths deliberately, document the migration, and only then let ecosystem plugins adopt `plugins.slots.<slot>.owner`.
