## Summary

Keep #70823 open. Current main still models `plugins.slots.memory` and `plugins.slots.contextEngine` as bare plugin-id strings, rejects the proposed object slot form, and lacks provenance fields or an overwrite invariant. This is a core config/plugin-contract RFC, not ClawHub-only work, because external plugins cannot add the proposed `plugins.slots.<slot>.owner` schema or make core install/activation paths honor it.

## Best Possible Solution

Keep the RFC open for maintainer API/product judgment. A good implementation path would add a typed slot-owner union or normalized internal slot-owner shape, accept legacy strings on read, update Zod/schema/help/docs, teach validators/resolvers/install-selection/uninstall paths to use `owner`, and add regression tests before deciding whether provenance fields and overwrite enforcement ship together or as follow-ups.

## Risks / Open Questions

- The proposal changes a public config contract, so any implementation needs backward-compatible string acceptance plus generated schema/help/docs and focused migration tests.
- The overwrite invariant needs maintainer design for inferred default slots, manual config edits, plugin install/update/uninstall, and any explicit force or allow_override semantics.
