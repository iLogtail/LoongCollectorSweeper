## Summary

Issue #70678 should remain open. Current main still has the WhatsApp extension-local no-message watchdog that treats inbound silence as stale activity and force-closes the connection with status 499 after the default 30-minute timeout. The discussion correctly identifies this as separate from the gateway-level stale socket policy; the active fix path appears to be open PR #72145, not a shipped main-branch fix.

## Best Possible Solution

Keep #70678 open as the canonical parent bug until the WhatsApp plugin no longer treats inbound-message silence as a socket-liveness failure. The likely fix belongs in the WhatsApp plugin: record transport/Baileys socket activity separately from handled inbound messages and use that for liveness, or disable/configure the inbound-message watchdog for quiet linked-device sessions. Review or land open PR #72145 if it implements that behavior with regression coverage.
