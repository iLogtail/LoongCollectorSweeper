## Summary

Keep #71770 open. The request is a concrete new model-routing feature: proactive token/quota/cost-threshold failover with ordered fallbacks, optional return to primary, and notifications. Current main has reactive model fallback on failures/cooldowns and usage/quota reporting surfaces, but I did not find a supported config/runtime path that switches providers before a call based on real-time token consumption thresholds. The related rate-limit rotation work covers sustained 429/error handling, not this proactive resource-management policy.

## Best Possible Solution

Keep the issue open for product/design work in the model request and fallback path. A solid implementation would add a documented config surface for provider/model usage or cost thresholds, evaluate provider usage snapshots and local transcript cost before selecting each candidate, integrate with `runWithModelFallback` without weakening existing cooldown/error fallback, emit lifecycle/user-visible notifications when a threshold switch happens, and define recovery/reverse-failover semantics when quota windows reset. Adjacent open budget/rate-limit items can inform the design, but they do not fully supersede this request.

## Risks / Open Questions

- Provider usage data is uneven across providers, so any implementation needs explicit semantics for providers with no quota API, subscription-style quotas, local providers, or only post-response token usage.
- Automatic switching can surprise users or increase costs unless thresholds, fallback order, reverse-failover behavior, and notifications are explicit and auditable.
- This overlaps adjacent budget/rate-limit work, so maintainers should coordinate with open token/request budget and spend-ceiling discussions rather than adding a one-off failover path.
