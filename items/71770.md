## Summary

Keep #71770 open. Current main supports ordered model fallbacks after auth/rate-limit/timeout/billing failures and exposes usage/cost reporting surfaces, but I did not find a supported config or runtime path that proactively switches models before a call based on real-time token, quota, or cost thresholds.

## Best Possible Solution

Keep the issue open for product/design work in the model request and fallback path. A good implementation would add a documented threshold policy surface, evaluate provider usage/cost snapshots before candidate selection, integrate with runWithModelFallback without weakening existing cooldown/error fallback, emit user-visible failover notifications, and define reset/reverse-failover behavior for quota windows.

## Risks / Open Questions

- Provider usage/quota data is uneven across providers, so any implementation needs explicit semantics for providers without quota APIs or with only post-response token usage.
- Automatic provider switching can surprise users or increase costs unless thresholds, fallback order, reverse-failover behavior, and notifications are explicit and auditable.
- This overlaps existing usage reporting and rate-limit fallback work, so maintainers should coordinate the design rather than adding a one-off routing path.
