## Summary

Keep open. Current main still does not implement the requested protection for all active/running sessions: enforce-mode save maintenance preserves only one caller-supplied activeSessionKey, pruning/capping remain driven by updatedAt and entry count, load-time maintenance has no active-session preservation, and disk-budget cleanup skips only a single active key. The related PR #59932 tracks a focused fix but is still open and was previously reviewed as not implemented on main.

## Best Possible Solution

Keep the issue open and land a focused core session-store fix that preserves all currently active/running session entries across stale pruning, maxEntries capping, load-time maintenance, and disk-budget eviction. The implementation should define one shared protected-key source, likely combining the caller activeSessionKey with persisted/in-memory active subagent or cron run state, and add regression tests for multiple running sessions under age, count, and disk-budget pressure. PR #59932 is the closest related work but still needs maintainer review or a corrected follow-up before this issue can close.

## Risks / Open Questions

- Concurrent running subagent or cron-created sessions can still be evicted when another session save triggers enforce-mode maintenance and only that caller's activeSessionKey is preserved.
- Oversized-store load/restart can still prune or cap running sessions because load-time maintenance has no active-session preservation path.
- There is still no targeted regression test proving multiple status:"running" session entries survive pruneAfter, maxEntries, and maxDiskBytes enforcement.
