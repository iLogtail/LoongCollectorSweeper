## Summary

Keep open. Current `main` does not implement this PR’s alias-normalization, inferred-model-id, or error-hint behavior, so there is no strong basis to close it as already handled or obsolete.

## Evidence

- **Alias lookup still uses exact lowercase keys only:** `buildModelAliasIndex` stores aliases with `normalizeLowercaseStringOrEmpty(alias)`, and `resolveModelRefFromString` looks them up with the same exact-lowercase key. There is no space/underscore-to-hyphen normalization, so inputs like `gpt 5.4` or `gpt_5.4` will not hit an alias `gpt-5.4`.
  - file: [src/agents/model-selection-shared.ts:255](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/agents/model-selection-shared.ts#L255)

  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Provider inference still compares only exact/lowercased model ids:** `inferUniqueProviderFromConfiguredModels` only checks `parsed.model === model` or lowercase equality. It does not normalize separator variants, so bare-model inference still misses allowlist entries like `openai-codex/gpt-5.4` when the user enters `gpt 5.4` or `gpt_5.4`.
