## Summary

Keep open. Current main still lacks a WhatsApp channel/account readOnly config and still has multiple direct WhatsApp outbound paths outside generic session.sendPolicy. The PR is active, tied to #71166, and contains useful transport-layer work, but the provided review discussion also identifies unresolved follow-ups that should be addressed before merge.

## Best Possible Solution

Keep #71382 open. The best path is to finish the transport-level WhatsApp readOnly implementation on this PR or an equivalent maintainer branch: add channel/account config schema and docs, enforce a hard gate before all WhatsApp outbound transport calls, preserve pairing request creation while suppressing WhatsApp replies, and land focused tests for messages/media, polls, reactions, composing/presence, read receipts, pairing replies, and account inheritance. Generic session.sendPolicy can remain documented as a reply-delivery control, but it does not currently replace a WhatsApp transport-level no-outbound guarantee.
