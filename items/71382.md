## Summary

Keep PR #71382 open. Current main does not implement WhatsApp `readOnly` at the config, account-resolution, outbound send, inbound monitor, read-receipt, or pairing-reply layers. The related issue #71166 being closed does not make this PR obsolete because the PR is the active implementation attempt, and the PR discussion still has concrete actionable review follow-ups.

## Best Possible Solution

Keep the PR open and ask the branch owner or a maintainer with push access to apply the already-triaged follow-ups: add the WhatsApp `readOnly` config contract and resolver state, enforce it at the WhatsApp transport/outbound API boundary, gate presence/read receipts/polls/reactions/typing/reply/media/pairing replies, preserve pairing request creation in read-only mode, and add focused regression tests for the empty-message, composing, and pairing-request cases. `sendPolicy` can remain complementary session delivery policy, but it is not sufficient for the transport-level guarantee requested here.
