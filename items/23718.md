## Summary

Keep open. Current main still does not accept `{ primary, fallbacks }` for `heartbeat.model`: validation and public config types remain string-only, heartbeat execution forwards only a string `heartbeatModelOverride`, and fallback execution still derives `fallbacksOverride` from the agent/default model config rather than a heartbeat-specific fallback chain. The related maintainer/Codex comment on #23718 correctly describes a real remaining gap on current main.

## Best Possible Solution

Implement heartbeat-specific fallback-chain support in core: change the heartbeat config schema and exported types to accept the shared `AgentModelConfig` shape while preserving string compatibility, resolve heartbeat primary/fallbacks with existing model-input helpers, thread a heartbeat fallback override through reply/run options into `runWithModelFallback`, update pricing-reference collection and docs, and add focused heartbeat override/config-honor tests.
