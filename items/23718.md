## Summary

Current main still does not implement fallback-chain syntax for `heartbeat.model`. The schema and public types keep heartbeat model as a string, the heartbeat runner/reply options only pass a string `heartbeatModelOverride`, and docs still describe a single `provider/model` override. Related fallback work covers agent model fallbacks used by heartbeat runs, but not a heartbeat-specific `{ primary, fallbacks }` config object.

## Best Possible Solution

Implement heartbeat-specific fallback-chain support in core: change the heartbeat config contract to accept the shared `AgentModelConfig` shape while preserving string compatibility, resolve the heartbeat primary and fallback list with the same helpers used for agent model config, forward heartbeat fallbacks through explicit reply/run options into `runWithModelFallback` as `fallbacksOverride`, and update schema/types, pricing reference collection, heartbeat docs, reload/config-honor coverage, and focused heartbeat override tests.
