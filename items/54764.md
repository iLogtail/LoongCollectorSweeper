## Summary

Keep PR #54764 open. Current main still has the same core cleanup shape the PR is trying to fix: kill termination deduplicates cleanup by child session key, while delete-mode cleanup deletes only the single passed run id. Orphan-prune cleanup also remains a separate helper path that does not trigger deferred announce retries. The PR is stale against refactored files, but the underlying work still looks useful rather than obsolete.

## Best Possible Solution

Keep the PR open, but ask for a rebase onto the current split registry modules. The best implementation is to make final delete/keep cleanup bookkeeping run per terminated run id in `src/agents/subagent-registry-run-manager.ts`/`src/agents/subagent-registry-lifecycle.ts`, route orphan pruning through equivalent final bookkeeping or explicitly retry deferred completed announces, and add regression coverage for multiple delete-mode runs sharing a childSessionKey plus the orphan-prune deferred-retry path.
