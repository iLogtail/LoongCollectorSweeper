## Summary

Keep open. Current main does not implement a loopback-specific pre-auth WebSocket handshake timeout, and the PR's raw-socket loopback check conflicts with current gateway locality/proxy handling that intentionally treats forwarded-header evidence as overriding loopback. This is still a live code review item, not a cleanup close.

## Evidence

- **Current main still has one shared pre-auth handshake timeout:** `getPreauthHandshakeTimeoutMsFromEnv` only returns the global env override or the shared 10s default; there is no loopback-specific branch or `isLoopback` option on current `main`.
  - file: [src/gateway/handshake-timeouts.ts:36](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/gateway/handshake-timeouts.ts#L36)

- **WebSocket pre-auth handshake still uses that shared timeout directly:** The gateway WS connection handler still calls `getPreauthHandshakeTimeoutMsFromEnv()` before the client is authenticated, so the behavior requested by PR #52142 is not already present on `main`.
  - file: [src/gateway/server/ws-connection.ts:366](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/gateway/server/ws-connection.ts#L366)

That aligns with the review concern that `isLoopbackAddress(remoteAddr)` alone is not enough.
