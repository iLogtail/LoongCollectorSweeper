## Summary

Keep #60448 open. Current main still sends sender metadata into the model-facing user prompt: gateway `chat.send` copies connected client identity into `SenderId`/`SenderName`/`SenderUsername`, TUI identifies itself as `openclaw-tui`, Telegram fills real sender name/id/username, `buildInboundUserContextPrefix` renders those fields as `Sender (untrusted metadata)` JSON, and `get-reply-run` prepends that prefix to the prompt body. Existing sanitizers strip the blocks from UI/history display boundaries, not from the AI context. The related #42859 remains a plausible fix path, but that work is not implemented on current main.

## Best Possible Solution

Keep the issue open and implement a scoped core fix: preserve internal routing/requester identity for authorization and delivery, but stop exposing internal UI client identity and sensitive channel sender identifiers in the model-facing prompt unless a channel explicitly needs a privacy-minimized label. Update `chat.send`, `buildInboundUserContextPrefix`, Telegram/TUI coverage, and display/history sanitizer tests so model-context and UI-context behavior are verified separately.

## Risks / Open Questions

- A fix should not simply delete sender identity from `MsgContext`; those fields also support routing, command authorization, approvals, and lifecycle behavior. The safer direction is to separate trusted internal requester identity from prompt-visible metadata.
