## Summary

Keep open. `main` still does not start typing for inline `dispatch.kind === "tool"` skills, so the user-visible gap is real, but PR #52434 is not safe to auto-close or merge-as-is: the proposed `typing.startTypingLoop()` call would sit inside the existing tool `try/catch`, which would make typing-start failures abort the tool execution, and it would bypass the existing `suppressTyping`/`typingMode: "never"` contract for contexts that must stay silent.

## Evidence

- **Current main still lacks inline tool typing start:** `handleInlineActions` finds the inline tool and immediately calls `tool.execute(...)` inside the tool `try/catch`; there is still no typing start before execution on `main`, so this PR is not already implemented.
  - file: [src/auto-reply/reply/get-reply-inline-actions.ts:292](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/auto-reply/reply/get-reply-inline-actions.ts#L292)
  - command: `nl -ba src/auto-reply/reply/get-reply-inline-actions.ts | sed -n '286,307p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **Typing start errors are currently fatal:** `createTypingController` builds its start guard with `rethrowOnError: true`, and `startTypingLoop()` awaits that guarded start path. If `onReplyStart` throws, `startTypingLoop()` throws. Placing that call inside the existing tool `try/catch` would convert transient typing-indicator failures into tool-execution failures.
  - file: [src/auto-reply/reply/typing.ts:116](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/auto-reply/reply/typing.ts#L116)
  - command: `nl -ba src/auto-reply/reply/typing.ts | sed -n '116,178p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **Typing suppression is an existing contract:** `resolveTypingMode` forces `never` when `suppressTyping` is true or the turn is heartbeat/system-event/internal-webchat. An unconditional `typing.startTypingLoop()` in the inline tool-dispatch path would bypass that policy.
  - file: [src/auto-reply/reply/typing-mode.ts:26](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/auto-reply/reply/typing-mode.ts#L26)
  - command: `nl -ba src/auto-reply/reply/typing-mode.ts | sed -n '18,117p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **The codebase already has a safer pattern for tool-start typing:** The main agent-runner tool path emits typing through `params.typingSignals.signalToolStart()`, which routes tool-start typing through the resolved typing mode instead of calling the controller directly. That is the behavior-aligned pattern to mirror here.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:1176](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/auto-reply/reply/agent-runner-execution.ts#L1176)

Current `main` code confirms those concerns are valid.
