## Summary

Keep #69505 open. Current main has outbound `messages.responsePrefix`, but it is post-generation reply formatting, not a per-cron or agent-default provider-native assistant prefill. The requested `assistantPrefix` surface is absent from the cron payload contract, agent defaults schema/types, agent command options, CLI cron/agent flags, and cron isolated execution path.

## Best Possible Solution

Keep the issue open and implement it as an additive core/protocol feature: add a nullable/string assistant-prefix field to cron payloads, agent defaults, agent command options, CLI/UI/docs, and the strict schemas; thread it through cron and ad-hoc agent execution into provider/runtime request construction; initially support Anthropic plus one OpenAI-compatible path with tests proving delivered cron content starts with the configured prefix and unsupported providers fail open.

## Risks / Open Questions

- Existing `messages.responsePrefix` may look similar, but closing on that would leave the reported pre-generation narration leak unsolved because it runs after model output is already available.
- Provider-native continuation semantics differ across Anthropic, OpenAI-compatible, Ollama, CLI, and plugin runtimes, so the eventual fix should be additive, no-op where unsupported, and covered by provider-specific contract tests.
