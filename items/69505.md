## Summary

Keep open. Current main has an outbound `messages.responsePrefix`, but it is post-generation delivery formatting. The requested per-cron/agent-default `assistantPrefix` that is passed into provider-native assistant prefill is not present in cron payload types, protocol validators, agent defaults config, agent run options, or the cron isolated execution path.

## Best Possible Solution

Keep #69505 open for maintainer/product judgment. The best implementation path is an additive core change: extend cron payload schema/types and agent defaults, thread the value through `AgentCommandOpts` and cron isolated execution, add provider/runtime-specific support where the dependency contract supports continuation from a prefix, no-op or clearly degrade where unsupported, and document how this differs from `messages.responsePrefix`.

## Risks / Open Questions

- Provider-native assistant-prefill semantics vary across Anthropic, OpenAI-compatible, Ollama, and plugin runtimes, so implementation needs provider-contract checks instead of assuming one message shape works everywhere.
- The existing `messages.responsePrefix` and block-streaming settings are adjacent mitigations but do not satisfy the requested protocol-level guarantee; docs should make that distinction explicit if this is implemented.
