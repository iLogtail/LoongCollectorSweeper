## Summary

Keep #11326 open. Current main still decodes built-in web_fetch response bytes through the default UTF-8 TextDecoder/res.text() path, exposes no tool/config charset or encoding parameter, and passes the already-decoded string into Readability/basic HTML extraction. The supplied prior Codex review remains accurate on current main; the related GBK search hits are about Windows process output or unrelated features, not this web_fetch charset gap.

## Best Possible Solution

Keep #11326 open until core web_fetch decodes response bytes with charset awareness before extraction. The likely fix belongs in src/agents/tools/web-shared.ts or a narrow helper used by web_fetch: preserve SSRF and maxResponseBytes behavior, detect charset from Content-Type and early HTML meta bytes, support labels such as gbk/gb2312 when the runtime can decode them, consider a manual encoding parameter/config only if maintainers want that public surface, and add focused web_fetch tests plus docs/tools/web-fetch.md updates.
