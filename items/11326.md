## Summary

Keep #11326 open. Current main still decodes web_fetch response bytes through default UTF-8 paths before HTML extraction, exposes no tool/config charset parameter, and has no GBK/GB2312 web_fetch regression coverage. The existing maintainer comment remains accurate on current main. The related GBK work in #56538 is about Windows node-host/process output decoding, not web_fetch response decoding.

## Best Possible Solution

Keep the issue open until core web_fetch performs charset-aware byte decoding before extraction. A narrow fix should live in readResponseText or a helper it calls: preserve SSRF and maxResponseBytes behavior, detect charset from Content-Type and early HTML meta bytes, support labels such as gbk/gb2312 where the runtime can decode them, decide whether a manual encoding parameter is worth adding, and add focused web_fetch tests plus docs updates.
