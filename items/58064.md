## Summary

Keep open. Current `main` contains a targeted fix for explicit macOS LaunchAgent restarts, but the evidence does not show that the reported self-`launchctl bootout/bootstrap` path is handled end-to-end. Separately, the status table still marks a channel `ok` whenever it is enabled/configured even if its runtime summary says `running: false`, which matches the reporter's misleading `ON / OK` symptom.

## Evidence

- **Explicit LaunchAgent restart path was hardened earlier:** `CHANGELOG.md` records a 2026.3.11 fix for explicit macOS launchd restarts (`Fixes #43311, #43406, #43035, and #43049`), but this predates issue #58064, which was reported on 2026-03-31 against 2026.3.28.
  - file: [CHANGELOG.md:2735](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/CHANGELOG.md#L2735)
  - command: `nl -ba CHANGELOG.md | sed -n '2730,2742p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Current code protects only the managed restart entrypoint:** `restartLaunchAgent()` now detects when it is running inside the current LaunchAgent and hands the restart off to a detached helper instead of directly `kickstart`ing itself. The helper script then does `launchctl enable` plus `kickstart`/`bootstrap` recovery. This is strong evidence for the `openclaw gateway restart` path, but it does not prove recovery from an arbitrary raw `launchctl bootout/bootstrap` initiated inside the managed process.
  - file: [src/daemon/launchd.ts:635](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/daemon/launchd.ts#L635)
  - command: `nl -ba src/daemon/launchd.ts | sed -n '625,685p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Detached helper implementation is specific to restart modes:** The handoff helper only exposes `kickstart` and `start-after-exit` modes and is wired through `scheduleDetachedLaunchdRestartHandoff()`. I did not find a corresponding guard that intercepts or forbids raw `launchctl bootout/bootstrap` shell commands from the agent/runtime side.
  - file: [src/daemon/launchd-restart-handoff.ts:78](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/daemon/launchd-restart-handoff.ts#L78)
  - command: `nl -ba src/daemon/launchd-restart-handoff.ts | sed -n '63,179p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Status rendering still ignores `running:false` when deciding overall state:** `buildChannelsTable()` returns `ok` once a channel is enabled/configured unless there are explicit issues, unavailable secrets, or link warnings. The state machine never checks the channel summary's `running` field, even though the shared status summary carries `running`, `lastStartAt`, `lastStopAt`, and `lastError`. That means stopped-but-configured channels can still look healthy in aggregated status output.
  - file: [src/commands/status-all/channels.ts:263](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/commands/status-all/channels.ts#L263)
  - command: `nl -ba src/commands/status-all/channels.ts | sed -n '248,315p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Agent prompt steers toward `openclaw gateway restart`, not raw launchctl:** The system prompt quick reference tells agents to manage the daemon with `openclaw gateway start/stop/restart`. That reduces exposure to the self-decapitation path, but it is guidance rather than a hard runtime safeguard, so it is not enough to close this bug as implemented.
  - file: [src/agents/system-prompt.ts:751](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/agents/system-prompt.ts#L751)
  - command: `nl -ba src/agents/system-prompt.ts | sed -n '748,770p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)

## Risks / Open Questions

- I could not perform a macOS LaunchAgent reproduction in this checkout, so the lifecycle failure itself remains unverified here.
- The raw self-`launchctl bootout/bootstrap` scenario described by the reporter still appears distinct from the protected `openclaw gateway restart` path.
- The misleading `openclaw status` symptom appears still plausible on current `main` because aggregate channel state does not consider `running:false`.
