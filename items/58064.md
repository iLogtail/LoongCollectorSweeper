## Summary

Current `main` has a targeted LaunchAgent restart handoff for the managed `openclaw gateway restart` path, but I did not find proof that it prevents or recovers from the reporter's narrower root cause: an agent-issued `launchctl bootout/bootstrap` against its own LaunchAgent followed by channels staying uninitialized. Given that gap, this is not a high-confidence close.

## Evidence

- **Reporter narrowed root cause to self-bootout/bootstrap:** The provided issue follow-up says the breakage started after the agent edited its LaunchAgent plist and ran `launchctl bootout/bootstrap` from within its own process, and explicitly links that to the self-decapitation pattern from #43311/#40905.
  - command: `Provided GitHub issue comment context for #58064`
- **Managed LaunchAgent restarts now use detached handoff:** `restartLaunchAgent()` detects when it is running inside the current LaunchAgent and schedules a detached helper instead of calling `launchctl kickstart -k` in-process.
  - file: [src/daemon/launchd.ts:669](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/daemon/launchd.ts#L669)
  - command: `nl -ba src/daemon/launchd.ts | sed -n '660,710p'`
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Detached helper waits for caller PID before relaunch:** The helper script waits until the caller PID exits, then runs `launchctl enable` plus `kickstart`/`bootstrap` and logs the result. This addresses self-restart via the managed service path.
  - file: [src/daemon/launchd-restart-handoff.ts:85](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/daemon/launchd-restart-handoff.ts#L85)

- **Tests cover managed restart handoff, not arbitrary shell-issued bootout/bootstrap:** Unit tests verify that `restartLaunchAgent()` hands work off when invoked from the current LaunchAgent and that no direct `launchctl` calls happen in that path.
