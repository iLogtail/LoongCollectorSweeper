## Summary

Keep open. Current main has substantial session transcript indexing for both builtin memory and QMD, so the old “pipeline does not execute” part is likely improved. But the reported user problem is broader: a WhatsApp DM should recall work from an unrelated same-agent gateway-dispatched session. Current `memory_search` still filters session hits through session-tool visibility, whose default is `tree`; that excludes same-agent sessions that are neither the requester nor spawned children. QMD also still requires separate `memory.qmd.sessions.enabled` configuration before session transcripts become a QMD collection, so the issue’s shown `agents.defaults.memorySearch` config alone is not enough for QMD session indexing.

## Best Possible Solution

Keep the issue open for maintainer/product handling. The best path is to define the intended contract for session-memory visibility: either make same-agent gateway transcript recall work from DM when `experimental.sessionMemory` and `sources:["sessions"]` are enabled, or document the required extra configuration (`memory.qmd.sessions.enabled` for QMD and `tools.sessions.visibility=agent` or `all` where appropriate). Add a regression covering a gateway-style same-agent session key searched from a DM session through `memory_search`, including QMD and builtin fallback behavior.
