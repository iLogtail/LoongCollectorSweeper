## Summary

Keep #43794 open. Current main has useful SecretRef mitigation and `openclaw secrets` audit/configure/apply tooling, but it has not implemented config encryption at rest with a system keychain or master password. Plaintext credential strings remain valid, several flows still default to plaintext storage, and the current threat model still lists token theft from config storage as high residual risk with the recommendation to add token encryption at rest. This is a real core/security hardening request, not a ClawHub/plugin cleanup candidate.

## Best Possible Solution

Keep the issue open for maintainer security/product design. The best path is a core secrets-at-rest design that layers OS keychain and/or master-password encryption over supported config/auth-profile/credential stores, preserves SecretRef and plaintext migration compatibility, updates the threat model once shipped, and adds tests for encrypted persistence, fallback/migration, audit behavior, and rotation guidance.
