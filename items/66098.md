## Summary

Keep open. Current `main` still does not expose or consume effective chat defaults the way this PR proposes, so the underlying bug remains live. The PR is directionally useful, but the review discussion raises stale-cache regressions around model/session updates that still need maintainer judgment before merging.

## Evidence

- **`chat.history` still omits effective reasoning state on `main`:** Current `chat.history` only computes and returns `thinkingLevel`, `fastMode`, and `verboseLevel`. There is no `reasoningLevel` in the response, so the UI cannot learn the effective reasoning default for a fresh session from history.
  - file: [src/gateway/server-methods/chat.ts:1936](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/gateway/server-methods/chat.ts#L1936)
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Web UI history loader still stores only `thinkingLevel`:** `loadChatHistory()` requests `{ messages?, thinkingLevel? }` and writes only `state.chatThinkingLevel`; there is no `chatReasoningLevel` state on current `main`. That means the active chat view still cannot reflect history-resolved reasoning defaults.
  - file: [ui/src/ui/controllers/chat.ts:149](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/ui/src/ui/controllers/chat.ts#L149)
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Reasoning visibility still comes only from the session row:** `renderChat()` derives `reasoningLevel` from `activeSession?.reasoningLevel ?? "off"`. A fresh session with only effective agent defaults still renders from persisted row data, not the resolved active-chat state described in the PR.
  - file: [ui/src/ui/views/chat.ts:774](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/ui/src/ui/views/chat.ts#L774)
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Thinking default label still comes from row/default-model fallback, not active effective state:** The chat thinking select computes `defaultLabel` from `activeRow?.thinkingDefault`, session defaults, or `resolveThinkingDefaultForModel(...)`. It does not use the active chat history state, so the header can still disagree with runtime-resolved defaults for a new session.
  - file: [ui/src/ui/chat/session-controls.ts:195](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/ui/src/ui/chat/session-controls.ts#L195)
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Docs and config types define per-agent effective defaults that should apply without session overrides:** The public config contract says `thinkingDefault` overrides `agents.defaults.thinkingDefault` for the agent, and `reasoningDefault` applies when no session override is set. That makes the current UI/gateway gap a real contract mismatch rather than an obsolete report.
  - file: [docs/gateway/config-agents.md:948](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/docs/gateway/config-agents.md#L948)
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **PR review discussion identifies unresolved stale-cache regressions:** The PR already has review comments flagging that a history-cached thinking/reasoning value can go stale after `switchChatModel()` or later session-row updates. That concern matches current `switchChatModel()` behavior, which patches the session and refreshes session options but does not reload chat history.
  - file: [ui/src/ui/chat/session-controls.ts:260](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/ui/src/ui/chat/session-controls.ts#L260)
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)

## Risks / Open Questions

- Merging the PR as-is may leave the header label stale after model changes, because `switchChatModel()` refreshes session rows without reloading `chat.history`.
- Prioritizing history-cached reasoning over live session-row updates could desync reasoning visibility after `sessions.patch` or `sessions.changed` events unless the cache invalidation story is tightened.
