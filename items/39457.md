## Summary

Keep #39457 open. Current main still refreshes an already-loaded gateway service with `gateway install --force`, which rewrites the LaunchAgent plist, then immediately proceeds to the restart handoff. I found post-core plugin update work and plugin lifecycle hooks, but no user post-update hook or equivalent post-refresh/pre-restart customization slot for the reported LaunchAgent wrapper use case. The existing ClawSweeper comment reached the same conclusion, and the related plugin-update hook reports do not cover this updater-owned service refresh path.

## Best Possible Solution

Keep the issue open. The best maintainer path is to add a core updater-owned post-update hook, or an equivalent durable service customization mechanism, after `refreshGatewayServiceEnv` succeeds and before `runRestartScript`/`runDaemonRestart` in `src/cli/update-cli/update-command.ts`, with focused tests for the loaded macOS LaunchAgent update path and user-facing docs for the supported contract.

## Risks / Open Questions

- A shell-style lifecycle hook would need explicit executable, timeout, failure, logging, environment, and security semantics so it cannot silently hang or obscure updater failures.
- Current env preservation helps some service-environment customizations, but it does not solve the wrapper `ProgramArguments` use case described in the issue.
