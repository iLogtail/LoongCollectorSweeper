## Summary

Keep open. Current main already has a partial provider-scoped configured-model match that covers the central bare-request to provider-prefixed-configured-id case, but this PR is not cleanly obsolete: the PR discussion contains an unresolved exact-priority concern, and the current main implementation has the same order-sensitivity for bare requests when both bare and provider-prefixed configured rows exist. The PR should be revised rather than auto-closed or merged as-is.

## Best Possible Solution

Keep the PR open and ask for a rebase/update that replaces both the PR helper and current main's single-pass matching with a true priority search: exact id across the full configured list first, provider-prefixed configured id for bare runtime requests second, and any symmetric fallback only after those fail. Add mixed-list regression tests for both request shapes, including the bare-request case where an exact bare entry appears after a prefixed entry, and adapt the docs note to the current docs structure.
