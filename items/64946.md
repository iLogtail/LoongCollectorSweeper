## Summary

Keep open. Current main does not provide enough close-level evidence that the reported false-positive plugin config spam is fixed or fully superseded. The loader still validates `entry?.config` after coercing missing config to `{}`, and it logs `[plugins] <id> invalid config` from both normal validation and CLI metadata paths. The repo docs and config schema agree that plugin schemas validate `plugins.entries.<id>.config`, so the reported top-level workaround is intentionally rejected, but the noisy validation path still needs a targeted fix or canonical merge.

## Best Possible Solution

Keep #64946 open until maintainers either merge a loader fix that defers validation when config is unavailable in non-activating paths, or add a targeted regression test showing a strict-schema plugin configured under `plugins.entries.<id>.config` does not emit repeated invalid-config logs while loading successfully. If #70811 is expanded or merged to cover this exact agentmail/wrapped-config case, then close #64946 as duplicate/superseded with that PR as the canonical item.
