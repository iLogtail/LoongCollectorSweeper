## Summary

Keep open. Current main still lacks OAuth support for remote MCP servers: config/types/docs describe only URL, transport, headers, and timeout; the HTTP resolver drops any auth mode; transport construction does not pass an MCP SDK authProvider; and the session runtime treats UnauthorizedError like any other connection failure instead of running an OAuth callback/finishAuth/retry flow. The PR is not implemented on main or superseded by a merged change, and it addresses a real core MCP runtime gap rather than a ClawHub-only plugin idea. It still needs maintainer/product/security review because the PR discussion contains unresolved P1/P2 concerns around callback ordering, SSE reconnect/header behavior, OAuth state validation, token storage, and state namespacing.

## Best Possible Solution

Keep #62134 open or move the same work into a maintainer-reviewed replacement PR. The best implementation path is core MCP runtime support for an opt-in remote-server OAuth mode: add the config/schema/docs surface, preserve the auth mode through MCP config resolution, pass an SDK OAuthClientProvider into both SSE and streamable HTTP transports, implement a hardened localhost callback/token store with state validation and restrictive permissions, handle UnauthorizedError with finishAuth and reconnect semantics, and cover both transports plus stale-token and static-header cases in tests.
