## Summary

Keep open. Current main still allows the session-memory hook to write conversation-derived Markdown into `memory/` while `memorySearch.experimental.sessionMemory` plus `sources: ["memory", "sessions"]` indexes raw session transcripts separately. The user-facing docs describe each feature, but they do not warn that enabling both creates overlapping recall entries or recommend using only one path.

## Best Possible Solution

Keep the issue open and fix it in core docs or behavior. The conservative path is to add an explicit warning in `docs/reference/memory-config.md` and `docs/automation/hooks.md` saying the `session-memory` hook and `memorySearch.sources: ["sessions"]` overlap, with recommended configurations for each mode. A stronger product fix would add provenance-aware query/index deduplication for session-memory-generated files, but that needs a deliberate design because current chunk ids and source paths are independent.
