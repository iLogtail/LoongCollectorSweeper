## Summary

Keep open. Current `main` still implements cron delivery as a single `delivery` object with one channel/target, and there is no fanout or multi-delivery contract in the type definitions, Gateway schema, CLI surface, docs, or delivery-planning runtime.

## Evidence

- **Cron job model is singular:** `CronDelivery` is a single object with scalar `channel`, `to`, `threadId`, and `accountId` fields; `CronJob` carries one `delivery` value, not an array or repeated targets.
  - file: [src/cron/types.ts:24](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/cron/types.ts#L24)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Gateway protocol only accepts one delivery object:** `CronDeliverySchema` is a union of three single-target object shapes (`none`/`announce`/`webhook`), and `CronDeliveryPatchSchema` is likewise a single object, not a list.
  - file: [src/gateway/protocol/schema/cron.ts:180](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/gateway/protocol/schema/cron.ts#L180)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **CLI add command exposes one channel/to pair:** `openclaw cron add` only offers `--channel`, `--to`, and `--account`, then builds exactly one `delivery` object in the RPC params.
  - file: [src/cli/cron-cli/register.cron-add.ts:100](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/cli/cron-cli/register.cron-add.ts#L100)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **CLI edit command patches one delivery object:** `openclaw cron edit` only patches a single `delivery` object from one set of `--channel`/`--to`/`--account` flags; there is no `--also-channel`, `extraDelivery`, or similar fanout path.
  - file: [src/cli/cron-cli/register.cron-edit.ts:233](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/cli/cron-cli/register.cron-edit.ts#L233)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Runtime delivery planning resolves one target:** `resolveCronDeliveryPlan()` reads one `job.delivery` object and returns one `CronDeliveryPlan` with a single `channel`/`to` target.
  - file: [src/cron/delivery-plan.ts:10](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/cron/delivery-plan.ts#L10)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Docs still describe single-target delivery:** The cron docs describe delivery in singular terms and show one `--channel ... --to ...` route per job, with no multi-target syntax documented.
  - file: [docs/automation/cron-jobs.md:129](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/docs/automation/cron-jobs.md#L129)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **No fanout implementation surfaced in code search:** Repository search found no cron-specific `extraDelivery`, `additionalDelivery`, `--also-deliver`, `--also-channel`, or `delivery: [...]` support; matches were unrelated fanout usages elsewhere.
  - command: `rg -n "extraDelivery|additionalDelivery|also-deliver|also-channel|delivery\s*:\s*\[|delivery\[|multi-channel|fanout" src docs ui`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)

## Risks / Open Questions

- Closing as implemented would be incorrect; the current contract still requires duplicating jobs to deliver the same cron output to multiple channels.
- This is a real core feature gap, not a ClawHub-only request, because it requires changes to cron job schema, CLI, Gateway protocol, and runtime delivery semantics.
