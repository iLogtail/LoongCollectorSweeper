## Summary

Keep #53242 open. Current main does not implement the requested behavior: `chat.history` still truncates assistant text before the byte-budget pass, and the default cap is now 8,000 chars, lower than the reported 12,000-char cap. A later change made the limit configurable, but Control UI/WebChat still uses the default unless config or an RPC override is supplied, so the default 50k-character reproduction would still truncate. The companion PR #53243 remains open and unmerged, so the bug is not implemented on main.

## Best Possible Solution

Keep the issue open until maintainers either merge or rework the companion fix (#53243), or make an explicit product decision that the supported solution is `gateway.webchat.chatHistoryMaxChars`/RPC `maxChars`. A robust fix should align the default or UI-requested cap with the 128 KiB byte budget, add a 50k-character regression test for assistant history, handle UTF-8-heavy text without replacing the whole message with the oversized placeholder, and update config docs/generated help if the default changes.
