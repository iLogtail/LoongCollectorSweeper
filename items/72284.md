## Summary

Keep #72284 open. Current main does not provide enough evidence to close this as implemented or duplicate: REM report creation is independent from `DREAMS.md`, the Dream Diary write is conditional on the best-effort narrative subagent path, and the Control UI reads only root `DREAMS.md`/`dreams.md` rather than falling back to `memory/dreaming/rem/YYYY-MM-DD.md`. The related #70523 discussion is useful context but targets raw candidate pollution and explicitly leaves separate diary writes unchanged, so it does not supersede this bug.

## Best Possible Solution

Keep the issue open and fix the memory-core/Control UI contract. The best path is to make a successful REM run either reliably create/update `DREAMS.md` with a human-readable diary entry or return an explicit no-diary status that the UI can surface, then add a regression test for a managed cron REM run where `memory/dreaming/rem/YYYY-MM-DD.md` exists and `doctor.memory.dreamDiary`/the Dreams tab show the expected content. Also consider logging or surfacing narrative-subagent failures instead of silently leaving the diary absent.
