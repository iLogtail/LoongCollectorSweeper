## Summary

Keep #71948 open. Current main still emits the generic `Discord inbound worker timed out.` reply and logs only channel/message IDs on the timeout path, while the issue asks for actionable diagnostics such as session key, run id, queue depth, and blocked subsystem state. Related closed reports cover narrower listener-blocking, attachment, and websocket-lifecycle fixes, but they do not supersede this fresh, concrete diagnostics/timeout-observability report.

## Best Possible Solution

Keep this as an actionable Discord plugin/core-channel observability issue. The likely fix is to extend the Discord inbound worker/process observer contract so timeout handling can capture and log/surface account id, resolved session key, agent/run id once the agent starts, queue key/depth or blocked predecessor state, timeout source/default-vs-override, and last known phase such as preflight, dispatch, model/tool, qmd, session lock, gateway connect, or context-overflow recovery. Tests should be updated from the current generic fallback expectations to cover the richer timeout reply/log while preserving safe redaction.
