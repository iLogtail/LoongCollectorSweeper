## Summary

Keep open. Current `main` has partial recovery primitives, but not the requested native self-healing workflow. OpenClaw still relies on supervisor restart semantics plus manual `doctor`/config repair, and the config surface does not expose a `gateway.selfHeal` feature or equivalent restart-attempt/recovery-brief controls.

## Evidence

- **No `gateway.selfHeal` config surface on main:** Gateway config currently exposes `gateway.reload` with `mode`, `debounceMs`, and `deferralTimeoutMs`; there is no `selfHeal`, `configValidateOnRestart`, `recoveryBrief`, `maxRestartAttempts`, or `restartIntervalSeconds` field.
  - file: [src/config/types.gateway.ts:201](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/config/types.gateway.ts#L201)
  - command: `nl -ba src/config/types.gateway.ts | sed -n '190,235p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Invalid startup config still points users to manual doctor flow:** Docs say invalid config blocks gateway boot, leaves only diagnostic commands available, and instructs users to run `openclaw doctor` / `openclaw doctor --fix`. This is not an automatic validate-on-restart workflow.
  - file: [docs/gateway/configuration.md:80](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/docs/gateway/configuration.md#L80)

That is narrower than the requested configurable restart-time validation and auto-fix pipeline.

That matches the issue's description of current behavior rather than implementing native crash health monitoring.

## Risks / Open Questions

- There is overlapping recovery work on `main` (last-known-good restore, restart sentinel, diagnostics), so the eventual solution may partially reuse existing pieces, but the requested feature set is not shipped today.
- Closing as implemented would be incorrect because operators still do not have a documented/configurable native self-healing workflow for gateway crash recovery.
