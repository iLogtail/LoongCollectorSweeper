## Summary

Keep #55347 open. Current main has useful adjacent pieces: channel-level health restarts, restart sentinels for intentional restarts, stability bundles, and last-known-good config recovery. It still does not implement the requested native gateway process self-healing workflow: no `gateway.selfHeal` config surface, no gateway liveness probe/restart-attempt loop, no pre-restart doctor/config workflow tied to crash recovery, and no configurable recovery brief target. Related PR #64684 is an open narrow diagnostic slice and explicitly does not supersede the broader request.

## Best Possible Solution

Keep the issue open and scope it as a design/product item. The best path is to build on existing config recovery, stability bundles, restart sentinels, and channel health monitoring: add an explicit `gateway.selfHeal` policy only if maintainers want core-owned process recovery, implement supervisor-aware process liveness/retry behavior separately from channel restarts, run doctor/config diagnostics in failure paths without unsafe broad auto-repair, and route any recovery brief through the existing generic delivery/session mechanisms. Treat #64684 as a related diagnostic slice, not the canonical closure.

## Risks / Open Questions

- The issue overlaps several existing partial systems, so implementation should be split into narrow milestones to avoid duplicating channel-health, restart-sentinel, and config-recovery behavior.
- Crash recovery that auto-fixes config or restarts processes can create loops or hide plugin schema/API incompatibilities unless the policy is conservative and observable.
