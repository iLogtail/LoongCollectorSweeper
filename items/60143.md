## Summary

Keep open. Current main still treats plugin manifest config-schema failures as fatal config validation issues, and gateway startup still rejects invalid plugin-local snapshots instead of disabling only the affected plugin for the session. Main does have graceful handling for plugin register/load failures and a runtime migration for the old Brave web-search config path, but those do not implement the requested non-fatal plugin config validation path.

## Best Possible Solution

Keep the issue open and implement a config-validation/startup policy that classifies plugin-local manifest configSchema errors separately from core config errors. For invalid `plugins.entries.<id>.config`, record a warning, mark that plugin disabled for the session/runtime snapshot, continue startup for other plugins, and add regression coverage for cold startup and reload. Preserve fatal handling for root/core schema failures, missing critical subsystem config, and ambiguous `plugins.entries` shape failures.

## Risks / Open Questions

- A fix should avoid silently ignoring security- or auth-sensitive plugin misconfiguration; the disabled-for-session state needs clear logs/status and should not rewrite the user's config without an explicit repair flow.
- Reload and startup need consistent semantics: hot reload can keep the previous active runtime, while cold startup needs a defined runtime config that omits only the invalid plugin.
