## Summary

Current `main` still warns and skips symlinked skill paths that resolve outside the configured root, including bundled-skill symlink escapes, so the reported false-positive logging issue is not implemented away on `main` and should remain open.

## Evidence

- **Containment check still warns on realpath escape:** `resolveContainedSkillPath` calls `fs.realpathSync`, accepts only paths inside `rootRealPath`, and otherwise calls `warnEscapedSkillPath` before returning `null`. See `src/agents/skills/workspace.ts:268` and `src/agents/skills/workspace.ts:281`.
  - file: [src/agents/skills/workspace.ts:268](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/agents/skills/workspace.ts#L268)
  - command: `sed -n '240,560p' src/agents/skills/workspace.ts`
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Bundled symlink escapes are still treated as warnings:** `buildEscapedSkillPathReason` explicitly classifies bundled symlink cases as `bundled-symlink-escape` with the hint `likely-stray-local-symlink-or-checkout-mutation`, which matches the current warning behavior rather than suppressing it. See `src/agents/skills/workspace.ts:203-231`.
  - file: [src/agents/skills/workspace.ts:203](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/agents/skills/workspace.ts#L203)

- The issue body focuses on `~/.agents/skills/` symlinks to bundled skills; current tests directly cover workspace and bundled roots, but the same containment-and-warn logic applies to all loaded roots, so personal-skill symlink noise remains plausible rather than disproven.
- `gh` inspection was unavailable in this environment because `GH_TOKEN` was unset, so this review relies on current `main` code, tests, docs, and local git history rather than live PR metadata.
