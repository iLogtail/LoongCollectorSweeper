## Summary

Keep #49380 open. Current main still has the reported visibility gap: plugin context engines can run proactive/background compaction from `afterTurn()`, but the contract has no notification, return value, or stats method for reporting those compactions into `SessionEntry.compactionCount`, and `/status` / `session_status` still render that session-store value.

## Best Possible Solution

Keep the issue open and add a backward-compatible compaction accounting seam for context engines. The best path is an optional `notifyCompaction(count?: number)` runtime callback or equivalent optional stats/reporting method, persisting the result into `SessionEntry.compactionCount`, then wiring `/status` and `session_status` to that persisted value with focused tests for `ownsCompaction` engines that compact in `afterTurn()` or maintenance.

## Risks / Open Questions

- A fix needs to avoid double-counting when OpenClaw-triggered `compact()` calls and plugin-owned background compactions both occur in one turn.
- If status queries live plugin stats directly instead of persisting counts, it must handle inactive plugins, cross-process Gateway state, and slow plugin stores without making `/status` flaky.
