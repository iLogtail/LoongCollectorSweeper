## Summary

Keep PR #71651 open. Current main still lacks the reload-time schedule/enabled-change invalidation, while the timer path force-reloads with skipRecompute and the recompute helpers deliberately preserve existing future nextRunAtMs values. The linked issue #71607 has a concrete cron drift report, and the PR discussion shows maintainer fixups addressing the review gaps, so this is active useful work rather than cleanup-close material.

## Best Possible Solution

Keep #71651 open for maintainer review/merge. The right product/code path is to land the PR's semantic schedule comparator and reload invalidation after jobs-state hydration, preserving unchanged/key-order-only pending slots while clearing nextRunAtMs when schedule or enabled actually changes, with the regression tests and cron docs/changelog updates from the PR branch.
