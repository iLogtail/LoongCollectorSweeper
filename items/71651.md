## Summary

Keep open. Current `main` still reloads cron jobs with `forceReload: true, skipRecompute: true` and never invalidates a persisted future `state.nextRunAtMs` when the job's `schedule` or `enabled` state changes on disk, so the reported stale-`nextRun` bug remains plausible. The PR adds a targeted store-layer fix plus regression coverage; the existing review comments are minor follow-up points, not reasons to close.
