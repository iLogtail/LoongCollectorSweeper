## Summary

Keep PR #71651 open. Current main still lacks the proposed cron reload invalidation: timer ticks force-reload the split cron store with recompute skipped, `loadCronStore` merges `jobs-state.json` runtime state into externally edited `jobs.json`, and due-job collection still trusts an existing `state.nextRunAtMs`. The PR is also not obsolete: the maintainer discussion added semantic schedule comparison, coverage for key-order-only rewrites and all schedule kinds, docs, changelog, and local validation on the PR head.

## Best Possible Solution

Keep #71651 open and integrate the PR's maintainer-fixup shape: add semantic schedule/enable-change detection in `src/cron/service/store.ts` after `loadCronStore` merges runtime state but before due-job collection, preserve unchanged/key-order-only pending slots, and keep the regression tests plus cron docs/changelog entry. If maintainers prefer a different patch shape, this PR should remain the canonical vehicle or be explicitly superseded by a merged follow-up.
