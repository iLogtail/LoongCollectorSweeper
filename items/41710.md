## Summary

Current main already strips `[[reply_to...]]` directives before outbound delivery in the shared reply pipeline, so the iMessage bug this PR targets is already handled. The PR's separate LanceDB rejected-promise retry change is also already present on main in the newer runtime loader.

## Close Comment

Closing from a Codex review because current `main` already handles this. Reply tags are stripped earlier in the shared outbound reply pipeline: `src/auto-reply/reply/agent-runner-payloads.ts:146` always normalizes final payloads, `src/auto-reply/reply/reply-directives.ts:23` parses reply directives with `stripReplyTags: true`, and streamed/block replies go through the same normalization in `src/auto-reply/reply/reply-delivery.ts:119`. This branch is also stale on the LanceDB side: the rejected-promise retry reset already exists on main in `extensions/memory-lancedb/lancedb-runtime.ts:320`. With current main, there is nothing left here that still needs to land.
