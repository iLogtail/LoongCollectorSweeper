## Summary

Keep PR #70056 open. Current main still has the reported gateway stop-timeout behavior: the timeout branch marks the channel as running and returns before deleting the task/abort entries, and the existing regression test still expects a timed-out stop to block a second start. The PR targets a real bug linked to #70024, but the review discussion identifies a credible race in the proposed cleanup: deleting the task entry before the old task settles can allow a replacement start while the old task's unconditional finally still writes running=false. This needs revision rather than cleanup closure.

## Best Possible Solution

Keep #70056 open and ask for a small revision: preserve the intended timeout runtime fix for #70024, but add an identity/generation guard around old task cleanup/status writes or otherwise prevent the timed-out task from clobbering a replacement start. Add a regression test where the old task settles after a timeout and after a successful restart, proving the new channel remains running and no duplicate restart chain fires.
