## Summary

Keep PR #70056 open. Current main still has the reported timeout behavior, so the PR is not obsolete; however, the submitted patch needs revision because the review comments identify a real race with the old task's unconditional `finally` handler after the PR deletes `store.tasks` on timeout.

## Best Possible Solution

Keep the PR open and revise it in `src/gateway/server-channels.ts`: make the timeout snapshot honest and allow recovery, but guard stale task completion so an old `trackedPromise` cannot mutate runtime or trigger restart behavior after a replacement task has started. Add a regression test where the timed-out task eventually settles after a successful restart, not only a never-settling task.

## Evidence

- **Current main still reports timed-out stops as running:** On current main, `stopChannel` waits up to `CHANNEL_STOP_ABORT_TIMEOUT_MS`; when the task does not settle, the timeout branch sets `running: true`, records the timeout error, and returns before deleting `store.aborts` or `store.tasks`. That is the bug PR #70056 is trying to fix, so the PR should not be closed as already implemented.
