## Summary

Keep PR #70056 open. Current `main` still has the stop-timeout bug the PR targets, so the work is not obsolete or implemented elsewhere. The submitted patch is also not clearly merge-ready because the existing task finalizer can still clobber runtime state if the PR deletes the task entry and a replacement channel starts before the old task settles.

## Best Possible Solution

Revise PR #70056 rather than close it: fix the timeout path so runtime state no longer lies and future recovery is possible, but preserve task identity or guard every old-task finalizer/restart path so a timed-out task cannot clobber a replacement. Add a regression test where the timed-out task eventually settles after a restart attempt.
