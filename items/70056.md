## Summary

Keep open. Current main still has the reported stop-timeout behavior, and PR #70056 targets the right gateway lifecycle path, but the submitted patch has unresolved review feedback about a zombie task clobbering a replacement run after timeout.

## Best Possible Solution

Keep the PR open and ask for a revision that fixes the timeout path without allowing a timed-out original task to mutate the replacement lifecycle. A good patch should set the timeout snapshot to stopped, make restart possible, guard stale `trackedPromise` status writes with task identity or a lifecycle token, and add a regression test where the timed-out task eventually settles after a restart.
