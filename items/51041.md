## Summary

Keep #51041 open. Current main still does not expose Discord slash interaction response controls to `api.registerCommand` handlers: plugin command handlers receive the generic `PluginCommandContext` and return one `ReplyPayload`; the Discord native-command path defers the interaction, executes the plugin command with generic context fields, then sends the returned payload using the configured slash-command ephemeral default. There is a Discord `respond` helper for component interactive handlers, but it is not wired into native plugin command handlers. The reporter's webserver workaround is useful context, but it does not mean the requested core/plugin API exists.

## Best Possible Solution

Keep the issue open and design a small typed responder seam for Discord native plugin command handlers, preferably modeled after the existing Discord interactive `respond` helper but scoped to `registerCommand`. Preserve current authorization and routing before handler execution, avoid exposing raw Carbon interactions, allow defer/reply/followUp/edit semantics with explicit ephemeral flags, and add regression tests in the Discord native command plugin-dispatch suite for mixed ephemeral initial response plus non-ephemeral follow-up in guild channels and group DMs. Update the plugin SDK types and public docs once the API shape is settled.

## Risks / Open Questions

- Exposing raw Discord interaction objects would couple third-party plugins to Carbon/Discord internals and could bypass OpenClaw routing, auth, and delivery accounting if not designed as a narrow responder abstraction.
- A Discord-only command context field would expand the public plugin SDK; it needs backwards-compatible typing, docs, and tests for guild channels, group DMs, deferred interactions, and expired interaction handling.
