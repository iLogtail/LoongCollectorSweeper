## Summary

Keep #55450 open. Current main still returns the OpenAI-compatible request `model` value, which is the documented OpenClaw agent target (`openclaw`, `openclaw/default`, or `openclaw/<agentId>`), not the resolved backend provider/model. The related #53150 work implemented request-side dynamic model override via `x-openclaw-model`, but it does not satisfy this response-side attribution request.

## Best Possible Solution

Keep this open and implement response-side model attribution for `POST /v1/chat/completions`. The conservative path is to preserve the documented agent-target `model` field unless maintainers choose an intentional API contract change, and add an explicit attribution surface such as `X-OpenClaw-Model: <provider>/<model>` and matching SSE/non-stream test coverage using `result.meta.agentMeta.provider/model`, including fallback behavior when those fields are unavailable.

## Risks / Open Questions

- Changing the existing `model` field from the agent target to a raw provider/model could conflict with the documented agent-first `/v1/models` contract and clients that select `openclaw/default`. A response header or additive field may be the safer compatibility path.
