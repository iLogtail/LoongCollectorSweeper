## Summary

Keep #71696 open. Current main still lacks Telegram chat-type-based reply mode and related reply-threading plumbing: Telegram config accepts only global `replyToMode`, Telegram message/native-command delivery carries one resolved mode, outbound payload planning does not receive `currentMessageId`, and plugin command context still lacks provider message IDs. The PR is a focused continuation of the closed unmerged #70579 and addresses a real gap rather than obsolete work.

## Best Possible Solution

Keep #71696 open for maintainer review. The best path is to rebase it onto current main, keep runtime `replyToMode` precedence while adding Telegram `replyToModeByChatType` to schema/types/generated metadata/docs, resolve reply mode per direct/group in message and native-command delivery, pass `currentMessageId` through outbound planning/delivery, expose Telegram message IDs to plugin command handlers, and run the changed-lane checks for Telegram, config, plugin commands, and outbound payloads.
