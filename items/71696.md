## Summary

Keep #71696 open. Current main still does not implement Telegram chat-type-based reply mode: Telegram schema/types/docs/generated metadata expose only the global `replyToMode`, message and native-command delivery still use one captured mode, outbound Telegram delivery still lacks `currentMessageId` plumbing for `reply_to_current`, and plugin command context still lacks provider message ids. The provided PR context shows #71696 supersedes the earlier closed #70579 and includes targeted commits/tests for the missing behavior, so it is not obsolete cleanup.

## Best Possible Solution

Keep #71696 open for maintainer review. The best path is to rebase it onto current main, preserve runtime `replyToMode` fallback behavior, add Telegram `replyToModeByChatType` to schema/types/generated metadata/docs, resolve reply mode per direct/group in message and native-command delivery, pass `currentMessageId` through outbound Telegram planning/delivery, expose Telegram message ids to plugin command handlers, and run the changed Telegram/config/plugin/outbound checks before merge.
