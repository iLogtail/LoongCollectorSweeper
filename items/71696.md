## Summary

Keep #71696 open. Current main at 7e376e5aba32eeea8191e70d5e43e293f8b8398a still lacks Telegram chat-type-based reply mode and still has the reply-threading/plugin-command context gaps this PR is trying to address. The PR is recent, focused, and supersedes the earlier closed #70579 rather than being superseded by main.

## Best Possible Solution

Keep #71696 open for maintainer review. The best path is to rebase it onto current main, preserve the runtime replyToMode fallback while adding Telegram replyToModeByChatType to schema/types/generated metadata/docs, resolve Telegram reply mode per direct/group for message and native-command delivery, pass currentMessageId through outbound planning/delivery, expose Telegram messageId/replyToMessageId to plugin command handlers, and run the changed Telegram/config/plugin/outbound checks.
