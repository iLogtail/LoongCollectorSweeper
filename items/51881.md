## Summary

Keep #51881 open. Current main still lets a sniffed `image/apng` win over the provided MIME in gateway attachment parsing, and there is no APNG-to-PNG normalization or regression test on the implicated path. The related PRs/comments correctly identify the fix direction, but the fix is not present on current main.

## Evidence

- **Current attachment parser still prefers sniffed MIME:** `parseMessageWithAttachments` normalizes the provided and sniffed MIME values, warns on mismatch, then sets `finalMime` to `sniffedMime ?? providedMime ...`; because `isImageMime` accepts any `image/*`, `image/apng` is not rejected or normalized before being used.
  - file: [src/gateway/chat-attachments.ts:344](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/gateway/chat-attachments.ts#L344)
  - command: `nl -ba src/gateway/chat-attachments.ts | sed -n '335,370p'`
  - sha: [8e12c24d1721](https://github.com/openclaw/openclaw/commit/8e12c24d17218dc50023aaf5543df1b7c36520aa)
- **No APNG normalization in local MIME normalization:** `normalizeMime` only strips parameters and lowercases/trims through `normalizeOptionalLowercaseString`; it does not map `image/apng` to `image/png`. A repo search found no `image/apng` handling in `chat-attachments.ts` or its tests.
  - file: [src/gateway/chat-attachments.ts:133](https://github.com/openclaw/openclaw/blob/8e12c24d17218dc50023aaf5543df1b7c36520aa/src/gateway/chat-attachments.ts#L133)
