## Summary

Keep open. Current main still appears to have the reported `llm-task` bug: the plugin selects `provider` and `model` independently, treats `params.model` as the raw model id, and passes both directly to `runEmbeddedPiAgent` without using the same alias/model-ref resolver that sessions and cron use. The related issue comment is directionally right: #26018/#26175 were about a stale Haiku alias, and #18556/#48576 cover other resolver surfaces, but this `llm-task` path remains separate.

## Best Possible Solution

Fix `extensions/llm-task` to resolve the effective model through the same model-ref/alias path used by sessions and cron before dispatch. Preserve explicit `provider` when supplied, otherwise parse `model` as either an alias or `provider/model`, enforce `allowedModels` against the resolved key, pass the resolved provider plus bare model id to `runEmbeddedPiAgent`, and add regression tests for `gemini-flash`, `google/gemini-3-flash-preview`, and a configured `haiku` alias with an Anthropic default provider.
