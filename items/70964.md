## Summary

Keep open. Current `main` still contains the reported Gemini CLI OAuth deadlock path: the Google provider only prints the auth URL when `ctx.openUrl(...)` throws, while the browser-open helper reports failure by returning `false`, and the models-auth wrapper discards that return value. The linked PR #71469 in the issue timeline is consistent with the gap not being fixed on `main` yet.

## Evidence

- **Google OAuth flow still logs the URL only in `catch`:** `loginGeminiCliOAuth` updates progress, calls `await ctx.openUrl(authUrl)`, and only prints `Open this URL in your browser` inside the `catch` block. There is no unconditional URL print before browser launch.
  - file: [extensions/google/oauth.ts:43](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/extensions/google/oauth.ts#L43)
  - command: `nl -ba extensions/google/oauth.ts | sed -n '1,120p'`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Browser-open helper signals failure with `false`, not an exception:** `openUrl(url)` returns `false` for invalid URLs, unsupported environments, and any `runCommandWithTimeout(...)` failure; it swallows errors in its own `catch` block. That means callers cannot rely on an exception to trigger fallback UX.
  - file: [src/infra/browser-open.ts:98](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/infra/browser-open.ts#L98)

- **Models-auth wrapper discards the browser-open result:** The provider auth command passes `openUrl: async (url) => { const { openUrl } = await import("../onboard-helpers.js"); await openUrl(url); }`. Since the underlying helper returns `boolean`, this wrapper converts both success and silent failure into a resolved `Promise<void>` unless the import itself throws.
  - file: [src/commands/models/auth.ts:305](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/commands/models/auth.ts#L305)
