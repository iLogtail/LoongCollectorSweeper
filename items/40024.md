## Summary

Keep #40024 open. Current main still does not satisfy the reporter's baseUrl-only local OpenAI-compatible proxy case: configured fallback models without an explicit provider API still resolve as `openai-responses`, the embedded runner only uses `/v1/chat/completions` for `openai-completions`, and loopback/private endpoints still require explicit `request.allowPrivateNetwork: true`. The earlier ClawSweeper keep-open comment remains consistent with current main at 4a3030df9efa72c44bab567a4390ac0c4876a73f.

## Best Possible Solution

Keep the issue open and make the provider contract explicit. Either infer/select `openai-completions` for custom local OpenAI-compatible `/v1` providers that do not support Responses, or fail early with an actionable configuration error requiring `models.providers.<id>.api: "openai-completions"` and `models.providers.<id>.request.allowPrivateNetwork: true` for loopback/private endpoints. Add a regression test for a `local-agent-proxy` style provider at `http://127.0.0.1:3000/v1` covering the selected transport path and error behavior.
