## Summary

Keep open. Current main still has a plausible unresolved embedded-agent routing gap for this report: when a configured provider has a local OpenAI-compatible `baseUrl` but no explicit `api`, the fallback/configured resolution path still defaults to `openai-responses`, and the embedded stream transport sends Responses API requests rather than `/v1/chat/completions`. The issue is not old enough for stale-insufficient-info, has no protected maintainer signal, and no canonical duplicate in the provided related context.

## Best Possible Solution

Keep the issue open until maintainers choose and implement the intended contract for local OpenAI-compatible providers. A good fix would either infer or validate `openai-completions` for explicit local `/v1` custom providers, or add a clear doctor/config error and docs path requiring `models.providers.<provider>.api = "openai-completions"`; either path should include an embedded-runner resolver/transport regression test for `local-agent-proxy` with `baseUrl: http://127.0.0.1:3000/v1`.
