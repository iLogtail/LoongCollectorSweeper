## Summary

Keep open. Current main still defaults an embedded-runner provider entry with no explicit `api` to `openai-responses`, so a proxy that only expects `/v1/chat/completions` would still not receive the reported POST. I did not find strong evidence on `main` that this specific routing/config problem was fixed; the only clearly relevant recent fix addresses a 60s timeout path, not OpenAI-compatible endpoint selection.

## Evidence

- **Embedded runner still defaults missing provider API to Responses:** `resolveConfiguredFallbackModel()` builds the embedded-runner model with `providerConfig?.api ?? "openai-responses"`, then carries `requestConfig.api ?? "openai-responses"` into the resolved model. If a custom provider entry omits `api`, current main still routes it through the Responses transport, not Chat Completions.
  - file: [src/agents/pi-embedded-runner/model.ts:560](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/agents/pi-embedded-runner/model.ts#L560)
  - command: `nl -ba src/agents/pi-embedded-runner/model.ts | sed -n '520,610p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Responses vs Chat Completions are separate transport paths:** The Responses transport calls `client.responses.create(...)`, while the Chat Completions transport calls `client.chat.completions.create(...)`. That means the exact HTTP path depends on the resolved `api` mode, not just `baseUrl`.
  - file: [src/agents/openai-transport-stream.ts:703](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/agents/openai-transport-stream.ts#L703)
  - command: `nl -ba src/agents/openai-transport-stream.ts | sed -n '660,745p' && nl -ba src/agents/openai-transport-stream.ts | sed -n '1090,1145p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Docs are internally inconsistent for generic local OpenAI-compatible proxies:** One doc recommends `api: "openai-completions"` for local proxies, while another says generic local proxies should use `api: "openai-responses"`. That inconsistency makes it unsafe to close this as mere user misconfiguration.
  - file: [docs/concepts/model-providers.md:581](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/docs/concepts/model-providers.md#L581)

Its commit message and touched paths describe a 60s timeout fix, not a change to OpenAI-compatible proxy endpoint selection or `api` defaulting.
