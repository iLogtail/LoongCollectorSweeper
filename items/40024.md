## Summary

Keep open. Current `main` does not show a clear fix for this embedded-agent proxy report: the embedded runner still defaults custom configured providers to `openai-responses` when no explicit `api` is set, and the transport still chooses `/v1/responses` vs `/v1/chat/completions` strictly from that resolved API. The docs also still describe local OpenAI-compatible backends as a live troubleshooting area rather than a closed regression.

## Evidence

- **Embedded fallback transport still defaults custom providers to Responses API:** `resolveConfiguredFallbackModel()` still resolves `providerConfig?.api ?? "openai-responses"`, and the attached fallback model also keeps `requestConfig.api ?? "openai-responses"`. That means a custom provider without an explicit `api` does not automatically route to Chat Completions on current `main`.
