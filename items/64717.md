## Summary

Keep #64717 open. Current main still hardcodes 4000 inside Telegram send-time HTML/plain-text chunk planning, while the PR targets exactly that remaining gap by using the configured Telegram text chunk limit with the Telegram 4096 hard cap. The provided Greptile review is consistent with the local code read: this is a narrow, plausible bug fix, not obsolete or already implemented.

## Best Possible Solution

Keep the PR open and land a targeted version of it: compute the Telegram send-time chunk limit with `resolveTextChunkLimit(cfg, "telegram", account.accountId, { fallbackLimit: 4000 })`, clamp it to 4096, use that value for HTML chunks, plain fallback chunks, and fallback alignment, and add regression coverage for both 4096 and a lower configured limit before running the Telegram extension/changed gates.
