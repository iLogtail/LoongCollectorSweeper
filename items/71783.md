## Summary

Keep open. Current main has partial QMD conflict rebinding, but it only reconciles a different-name collection when `qmd collection list` exposes path+pattern metadata. The current tests explicitly preserve the failure path where conflict metadata is unavailable and OpenClaw only logs `qmd collection add skipped`, which matches the reported boot failure shape for `life` vs `life-main`.

## Best Possible Solution

Keep the issue open and add a narrow memory-core fix: parse the conflicting collection name from QMD add errors like `A collection already exists for this path and pattern` / `Name: life-main (qmd://life-main/)`, then safely rebind that stale entry to the managed name when the add command path+pattern are the current managed collection. Add regression coverage for the exact `life` vs `life-main` shape and verify the missing-collection search retry succeeds after repair. If maintainers decide not to remove/rename external QMD entries automatically, surface a targeted diagnostic and `openclaw doctor` repair instead.

## Evidence

- **Issue reports a concrete unresolved failure shape:** The issue log says OpenClaw tries to add `/Users/clark/life` as `life`, QMD rejects it because the same path/pattern is already registered as `life-main`, and later searches fail with `Collection not found: life`. That is specific enough to evaluate against current QMD manager behavior.
- **External explicit QMD path names are intentionally unscoped:** For paths outside the workspace, current config resolution preserves an explicit collection name instead of appending the agent id.
