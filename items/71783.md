## Summary

Keep #71783 open. Current main can rebind QMD path+pattern conflicts only when `qmd collection list` exposes usable path and pattern metadata; it still does not parse the conflicting collection name from add errors like `Name: life-main (qmd://life-main/)`. Searches still target the configured managed name, so a stale `life-main`-only state can still fail with `Collection not found: life`.

## Best Possible Solution

Keep the issue open and add a narrow memory-core repair in `extensions/memory-core/src/memory/qmd-manager.ts`: when `qmd collection add` reports `A collection already exists for this path and pattern` and includes a conflicting `Name: <old> (qmd://<old>/)`, parse that old name, verify the failed add command was for the configured managed path/pattern, then safely remove/rebind the stale collection to the managed name. Add focused coverage in `extensions/memory-core/src/memory/qmd-manager.test.ts` for `life` versus `life-main`, including the missing-collection search retry. If automatic rebinding is considered too destructive, add a targeted diagnostic and `openclaw doctor` repair path instead.
