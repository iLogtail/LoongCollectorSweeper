## Summary

Keep open. Current main still passes compaction reserveTokens into Pi summarization without clamping to the selected model output limit, while the pinned pi-coding-agent still computes summary maxTokens as floor(0.8 * reserveTokens). The PR is not obsolete: the provided discussion shows the initial missing-test feedback was addressed with a dedicated reserve-token clamping test commit, and related issue #61852 was closed as a duplicate specifically to keep #54383/#54392 as the canonical remaining work.

## Best Possible Solution

Keep #54392 open or land an equivalent refreshed fix. The best path is to clamp the effective reserveTokens before any Pi generateSummary call so floor(0.8 * effectiveReserveTokens) cannot exceed model.maxTokens, preserve a safe fallback for models without maxTokens, and include the PR's focused regression coverage for 300000 reserveTokens, 128000 maxTokens, pass-through, fallback, and staged summarization behavior.
