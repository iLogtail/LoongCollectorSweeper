## Summary

Keep open. Current main still fails startup SecretRef activation immediately when required secrets are unavailable, and the 1Password exec-provider path has no startup retry/backoff, circuit breaker, or rate-limit-aware pause. The related PR #56499 directly targets this issue but is still open/unmerged in the provided GitHub context, so the bug is not implemented on main.

## Best Possible Solution

Keep #56217 open until main has a targeted startup SecretRef resilience fix. The likely path is to land or rewrite PR #56499 with tests that prove bounded startup retry/backoff around prepareSecretsRuntimeSnapshot, clear logging for repeated failures/rate limits, and no regression to reload last-known-good behavior. A stronger follow-up would also document safer 1Password patterns, such as a single protocol-aware exec resolver or `op run` wrapper, but the core bug is the fatal unbounded supervisor restart loop on startup failure.
