## Summary

Keep #11665 open. Current main still sends `/hooks/agent` and mapped hook agent actions through an isolated cron-agent turn, and that path still forces a new session id even when the caller supplies a stable `sessionKey`. The request remains concrete, core-owned, and not superseded by a shipped fix.

## Best Possible Solution

Keep the issue open and implement a backward-compatible persistent hook-session option in core. The likely path is an explicit `sessionMode: "isolated" | "persistent"` or equivalent for `/hooks/agent`, hook mappings, and transform results, defaulting to isolated. Thread that through hook normalization, schema/types/help, mapping merge, dispatch, docs, and tests, and disable `forceNew` only for the persistent hook path while preserving isolated cron behavior.
