## Summary

Keep #11665 open. Current main still routes /hooks/agent and mapped hook agent actions through an isolated CronJob, and that isolated path passes forceNew=true into resolveCronSession, so a stable hook sessionKey cannot reuse the existing sessionId. The code has generic reuse support when forceNew is false, but no hook sessionMode/reuseSession/sessionTarget surface exists to select it.

## Best Possible Solution

Keep the issue open and implement a backward-compatible persistent hook-session option in core. The likely path is sessionMode: "isolated" | "persistent" or equivalent on /hooks/agent payloads, hook mappings, and transform results, defaulting to isolated. Thread it through hook normalization, config types/schema/help, mapping merge, dispatch, docs, and regression tests, then pass forceNew=false only for the persistent hook path.
