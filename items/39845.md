## Summary

The requested behavior is already implemented via queue mode `steer` (and `steer-backlog`): active streaming runs are steered immediately, while non-streaming/idle cases fall back to followup or run normally. This is documented and already shipped in release `v2026.4.22`.

## Close Comment

Closing from a Codex review because current OpenClaw already implements this behavior through `messages.queue.mode: "steer"` (or per-channel via `messages.queue.byChannel`). The runtime marks `steer`/`steer-backlog` for active runs in `src/auto-reply/reply/get-reply-run.ts:633`, injects the message into an active streaming run with `queueEmbeddedPiMessage(...)` in `src/auto-reply/reply/agent-runner.ts:969`, and still runs normally when the session is idle (`src/auto-reply/reply/queue-policy.ts:11`). The docs already describe the same behavior in `docs/concepts/queue.md:29`. This is also present in the shipped release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
