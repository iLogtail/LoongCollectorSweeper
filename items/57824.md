## Summary

Keep open. Current `main` still skips URL-only ACP image attachments and still locks Synology invalid-token guesses one request late; the PR also includes a necessary zero-budget clamp raised in review, and none of that is present on `main` or in the latest release tag inspected.

## Evidence

- **URL-only ACP images are still skipped on main:** `normalizeAttachments()` produces entries from `ctx.MediaUrl` with `path: undefined`, but `resolveAcpAttachments()` still hard-gates on `attachment.path`, so URL-only image attachments never reach ACP turn forwarding.
  - file: [src/auto-reply/reply/dispatch-acp-attachments.ts:52](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/auto-reply/reply/dispatch-acp-attachments.ts#L52)
  - command: `nl -ba src/auto-reply/reply/dispatch-acp-attachments.ts | sed -n '1,220p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **The underlying media cache already supports remote URLs:** `MediaAttachmentCache.getBuffer()` reads `entry.attachment.url` and, when there is no usable local path, falls through to remote fetch handling. That means the ACP skip is a caller-side bug, not a missing runtime capability.
  - file: [src/media-understanding/attachments.cache.ts:93](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/media-understanding/attachments.cache.ts#L93)
  - command: `nl -ba src/media-understanding/attachments.cache.ts | sed -n '80,165p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Synology invalid-token throttling is still off by one on main:** `InvalidTokenRateLimiter.isLocked()` and `recordFailure()` still use strict `>` comparisons. In `authorizeSynologyWebhook()`, that means the request that exactly exhausts the configured budget still returns 401, and 429 starts only on the next invalid guess.
  - file: [extensions/synology-chat/src/webhook-handler.ts:75](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/synology-chat/src/webhook-handler.ts#L75)
  - command: `nl -ba extensions/synology-chat/src/webhook-handler.ts | sed -n '1,220p'; nl -ba extensions/synology-chat/src/webhook-handler.ts | sed -n '404,448p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **The zero-budget edge case from review is real in current tests/config:** Current Synology tests assert `SYNOLOGY_RATE_LIMIT=0` resolves to `rateLimitPerMinute = 0`, matching the PR review warning that a threshold-based fix must clamp zero to avoid pre-auth lockout of valid requests. The PR discussion says the branch added that clamp, but current main does not have it.
