## Summary

Keep #62276 open. Current main has better EADDRINUSE diagnostics, stop hints, generated systemd restart limits, and diagnostic deep service discovery, but the Linux follow-up describes an install-time conflict between `openclaw onboard --install-daemon` user-systemd service and an existing system-level gateway service. Current `installSystemdService` still writes/enables/restarts the user unit without first scanning or warning about system-level gateway units/listeners, and related #28191/#70664 do not fully supersede this Windows/Linux report.

## Best Possible Solution

Keep the issue open and implement a focused install-time guard for Linux daemon onboarding/gateway install: reuse or extend the existing deep service/listener discovery to detect system-level OpenClaw gateway units or verified listeners on the target port, then skip with a clear warning or require an explicit override. Add targeted systemd install tests and update the gateway/onboarding docs to explain user-service versus system-service ownership.
