## Summary

Keep open. Current main does not implement the requested `MEDIA:/path#doc` override, and the existing force-document support is Telegram-only rather than solving the WhatsApp/Baileys compression case reported in #41965.

## Evidence

- **Media loader has no #doc handling:** `loadWebMediaInternal()` strips a leading `MEDIA:` prefix, resolves URLs/paths, detects MIME, derives `kind` from MIME, and finalizes that kind. There is no fragment parsing or `#doc` override in this path.
  - file: [src/media/web-media.ts:354](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/src/media/web-media.ts#L354)
  - command: `nl -ba src/media/web-media.ts | sed -n '354,590p'`
  - sha: [e640c0a95fcb](https://github.com/openclaw/openclaw/commit/e640c0a95fcbd06a1311f18fc635518d011d45da)
- **WhatsApp sends images through the image path:** WhatsApp outbound delivery loads media, normalizes it, then sets document filename only when `media.kind === "document"`; image media remains image media. The send options do not include a force-document override.
  - file: [extensions/whatsapp/src/send.ts:119](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/extensions/whatsapp/src/send.ts#L119)
  - command: `nl -ba extensions/whatsapp/src/send.ts | sed -n '1,210p'`
  - sha: [e640c0a95fcb](https://github.com/openclaw/openclaw/commit/e640c0a95fcbd06a1311f18fc635518d011d45da)
- **WhatsApp adapter does not propagate forceDocument:** Core defines and passes `forceDocument`, but the WhatsApp outbound adapter destructures and forwards media options without `forceDocument`; only Telegram consumes that field.
  - file: [extensions/whatsapp/src/outbound-base.ts:175](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/extensions/whatsapp/src/outbound-base.ts#L175)
