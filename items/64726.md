## Summary

Keep #64726 open. Current main already has exact-hash embedding caching, inline batching, opt-in provider batch jobs, session-start/search sync warming, and max-entry cache pruning, but the requested smart cache work remains materially incomplete: fuzzy reuse for minor edits, hit/miss/latency metrics, gradual model-switch migration or dual-write, vector compression, and benchmarked acceptance criteria are not implemented. The related ClawSweeper items in the provided context are narrower batch timeout/retry issues and do not supersede this broader cache-invalidation/observability request.

## Best Possible Solution

Keep this open as a memory-core enhancement, preferably split into narrower tracked work: cache telemetry in status/CLI, an explicitly configured fuzzy-reuse strategy with safety tests, a model/provider migration design, and storage pruning/compression behavior. Update docs/reference/memory-config.md and memory-core cache/indexing tests when implemented.

## Risks / Open Questions

- Fuzzy embedding reuse can make recall stale or semantically wrong if introduced without an opt-in threshold, content provenance, and regression tests.
- A dual-write or migration window could increase provider cost during reindexing, so it needs explicit configuration and status visibility.
