## Summary

Keep open. Current main still runs the package-manager update, verifies the installed package, and runs post-update doctor, but it does not reinstall the previous global package version when doctor fails. The PR is still addressing a real missing safety behavior, although its patch needs rework for the current refactored update path and the existing review concern about only rolling back after a successful update.

## Best Possible Solution

Keep the PR open and revise it against the current package update helper. The best implementation is to perform rollback only after a successful package-manager update when post-install verification or post-update `openclaw doctor --non-interactive --fix` fails and `beforeVersion` is known, reinstall `${packageName}@${beforeVersion}`, rerun doctor if the rolled-back entrypoint exists, preserve the overall error result, and update tests/docs if maintainers accept automatic rollback as product behavior.

## Risks / Open Questions

- The submitted patch targets an older inline update flow; current main has refactored package update behavior into `src/infra/package-update-steps.ts`, so the PR likely needs a rebase and design adjustment.
- Rollback should be gated to successful package-manager updates so failed network/registry/package-manager attempts do not trigger redundant reinstalls of the already-installed previous version.
- A complete fix should add focused tests for successful update plus doctor failure, verification failure after install, rollback doctor behavior, and failed install with no rollback.
