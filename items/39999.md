## Summary

Keep #39999 open. Current main still does not roll back a global package install to the previous version when post-update verification or `openclaw doctor --non-interactive --fix` fails. The PR remains a plausible core update-safety fix, but it needs rework for the current refactored package update helper and should incorporate the existing review concern that rollback should only run after a successful package-manager update.

## Best Possible Solution

Keep the PR open and revise it against the current package update helper. The best implementation is to roll back only after a successful package-manager update when post-install verification or post-update `openclaw doctor --non-interactive --fix` fails and `beforeVersion` is known; reinstall `${packageName}@${beforeVersion}`, rerun doctor if the restored entrypoint exists, preserve the overall error result, and add focused CLI/helper tests plus docs if maintainers accept automatic rollback as product behavior.

## Risks / Open Questions

- The submitted patch targets an older inline update flow; current main has split shared package update behavior into `src/infra/package-update-steps.ts`, so the PR likely needs a rebase and design adjustment.
- Rollback should be guarded to successful package-manager updates so failed network/registry/package-manager attempts do not trigger redundant reinstalls of the already-installed previous version.
- A complete fix needs focused coverage for successful update plus doctor failure, verification failure after install, rollback doctor behavior, and failed install with no rollback.
