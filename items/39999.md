## Summary

Keep open. Current main still does not implement the PR's automatic rollback-to-previous-version behavior for package/global updates after post-update doctor failure. The current package update path updates, verifies, runs doctor, then reports the first failing step; docs still describe rollback as a manual version pin. The PR remains potentially useful but needs revision because the existing review found its rollback guard can fire even when the update step itself failed.

## Best Possible Solution

Keep this PR or a replacement tracking item open. The best implementation is a narrow package-install rollback path in `src/cli/update-cli/update-command.ts`: after a successful global update, if post-update verification or doctor proves the install unhealthy and `beforeVersion` is known, reinstall `${packageName}@${beforeVersion}` with the same package manager/env, re-resolve the package root/version, optionally rerun doctor on the rolled-back entrypoint, and add focused regression tests for successful rollback, rollback failure, and the guard that prevents rollback when the update step itself failed.

## Risks / Open Questions

- The submitted patch likely needs an additional `updateStep.exitCode === 0` guard so rollback only runs after an update succeeded and then left the install unhealthy.
- Rollback should preserve the original failure in the result while adding clear rollback step evidence, otherwise it may obscure whether the update, verification, doctor, or rollback failed.
