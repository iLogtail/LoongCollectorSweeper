## Summary

Keep open. Current main intentionally rejects the remote plain-HTTP Control UI path and has docs/UI/handshake guidance, but it does not emit the requested server-side startup warning. The closest open PR, #56830, only clarifies protocol/UI copy; #32473 remains the broader policy/support tracker.

## Best Possible Solution

Keep #71669 open and implement a targeted gateway startup warning after config, bind, port, and TLS/runtime proxy facts are known. The likely home is `src/gateway/server-startup-log.ts` or the surrounding startup/runtime-state path, with regression coverage in `src/gateway/server-startup-log.test.ts`. The warning should point users to HTTPS/Tailscale Serve, an SSH tunnel to localhost, or the documented break-glass `gateway.controlUi.dangerouslyDisableDeviceAuth` path, while leaving the broader policy decision on remote plain HTTP tracked by #32473 and copy cleanups coordinated with #56830.

## Risks / Open Questions

- The warning predicate needs careful false-positive handling because TLS may terminate outside the gateway through Tailscale Serve or a reverse proxy while the local gateway still serves HTTP.
- The fix should keep auth policy unchanged unless maintainers explicitly decide to change the remote plain-HTTP security model tracked by #32473.
- #56830 may overlap copy/constants if it lands first, but it does not remove the need for a startup-log implementation.
