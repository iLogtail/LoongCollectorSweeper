## Summary

Keep open. Current `main` still treats `agent.request` attachments as agent input only; outbound delivery sends media only when the agent reply itself contains `mediaUrl`/`mediaUrls`. There is an existing direct `send` RPC for dumb relay use cases, but that is a separate surface and does not implement the requested `agent.request` auto-forward behavior.

## Evidence

- **`agent.request` payload has no auto-forward field:** The node-event parser accepts `message`, `deliver`, `attachments`, routing fields, and timeout, but no `auto_forward_attachments` or equivalent flag.
  - file: [src/gateway/server-node-events.ts:384](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/gateway/server-node-events.ts#L384)
  - command: `nl -ba src/gateway/server-node-events.ts | sed -n '384,401p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Node event forwards attachments only into agent input:** After parsing attachments into `message`, `images`, and `imageOrder`, the gateway calls `agentCommandFromIngress(...)` with those agent-input fields plus delivery route metadata. It does not carry original attachments into outbound delivery state.
  - file: [src/gateway/server-node-events.ts:523](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/gateway/server-node-events.ts#L523)

- **Agent delivery path uses reply payloads only:** `deliverAgentCommandResult` builds `deliveryPayloads` from the agent result and passes those payloads to `deliverOutboundPayloads(...)`. There is no fallback that injects original inbound attachments when the reply is text-only.
  - file: [src/agents/command/delivery.ts:345](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/agents/command/delivery.ts#L345)

  - file: [src/infra/outbound/deliver.ts:951](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/infra/outbound/deliver.ts#L951)
