## Summary

Current main already restores Telegram DM voice-note preflight transcription, includes a targeted regression test for that exact case, and the latest shipped release v2026.4.23 contains the same Telegram/audio implementation.

## Evidence

- **DM voice notes now trigger preflight transcription:** `resolveTelegramInboundBody` sets `needsPreflightTranscription` for audio-only messages when `!isGroup`, then calls `transcribeFirstAudio(...)` and replaces the `<media:audio>` placeholder with the transcript when one is returned.
  - file: [extensions/telegram/src/bot-message-context.body.ts:201](https://github.com/openclaw/openclaw/blob/7bd74758c5f54c546b52708359a25c08cae74ad7/extensions/telegram/src/bot-message-context.body.ts#L201)
  - sha: [7bd74758c5f5](https://github.com/openclaw/openclaw/commit/7bd74758c5f54c546b52708359a25c08cae74ad7)
- **Regression test covers the reported path:** A dedicated test named `transcribes DM voice notes via preflight (not only groups)` asserts that a Telegram DM voice note is transcribed once and that the resulting body text is the transcript rather than `<media:audio>`.
  - file: [extensions/telegram/src/bot-message-context.body.test.ts:149](https://github.com/openclaw/openclaw/blob/7bd74758c5f54c546b52708359a25c08cae74ad7/extensions/telegram/src/bot-message-context.body.test.ts#L149)
  - sha: [7bd74758c5f5](https://github.com/openclaw/openclaw/commit/7bd74758c5f54c546b52708359a25c08cae74ad7)
- **Preflight transcription is enabled by default unless explicitly disabled:** `transcribeFirstAudio` returns early only when `tools.media.audio.enabled === false`; otherwise it normalizes attachments, finds the first audio attachment, and runs the audio transcription pipeline.
  - file: [src/media-understanding/audio-preflight.ts:24](https://github.com/openclaw/openclaw/blob/7bd74758c5f54c546b52708359a25c08cae74ad7/src/media-understanding/audio-preflight.ts#L24)
  - sha: [7bd74758c5f5](https://github.com/openclaw/openclaw/commit/7bd74758c5f54c546b52708359a25c08cae74ad7)
- **Docs match the current default behavior:** The audio docs state that if no explicit audio models are configured and `tools.media.audio.enabled` is not set to `false`, OpenClaw auto-detects a working transcription path.
  - file: [docs/nodes/audio.md:23](https://github.com/openclaw/openclaw/blob/7bd74758c5f54c546b52708359a25c08cae74ad7/docs/nodes/audio.md#L23)
  - sha: [7bd74758c5f5](https://github.com/openclaw/openclaw/commit/7bd74758c5f54c546b52708359a25c08cae74ad7)
