## Summary

Issue #56094 proposes a higher-level multi-agent helpdesk orchestration layer. Current main keeps that kind of optional orchestration out of core and provides plugin/runtime seams to build it externally, so this is better classified as ClawHub/plugin scope than an OpenClaw core issue.

## Evidence

- **Vision scope excludes heavy orchestration in core:** The project vision says optional capability should usually ship as plugins, new core skills should go to ClawHub first, and 'heavy orchestration layers that duplicate existing agent and tool infrastructure' are out of scope for core by default.
  - file: [VISION.md:55](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/VISION.md#L55)

- **Plugin/runtime seam already supports silent background agent work:** The documented plugin runtime exposes session-store helpers plus api.runtime.subagent for background runs, including `deliver: false`, waiting for completion, and reading session messages. That is the right extension point for a silent helpdesk workflow.

## Close Comment

Closing after Codex review. This looks like a useful idea, but on current `main` it fits ClawHub/plugin scope rather than OpenClaw core. The project vision explicitly keeps optional capability in plugins, says new core skills should go to ClawHub first, and calls out heavy orchestration layers that duplicate existing agent/tool infrastructure as out of scope for core (`VISION.md:55-70`, `VISION.md:99-107`). The repo already exposes the seams needed to build this externally: plugin runtime session helpers plus background subagent runs with `deliver: false` and session message access (`docs/plugins/sdk-runtime.md:69-75`, `docs/plugins/sdk-runtime.md:87-113`), along with allowlisted and bounded inter-agent controls (`src/config/schema.help.ts:337-342`, `src/config/schema.help.ts:1346-1349`). Because of that, this should be pursued as a ClawHub skill/plugin instead of a new built-in runtime.
