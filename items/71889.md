## Summary

Keep #71889 open. Current main still matches the reported bug: the interactive `agents add` wizard seeds the workspace prompt from `resolveAgentWorkspaceDir(cfg, agentId)`, and that resolver appends non-default agent ids under any `agents.defaults.workspace`. Since `setup` writes the normal main workspace into `agents.defaults.workspace`, accepting the wizard default still produces `~/.openclaw/workspace/<id>` while the docs continue to document peer-level `~/.openclaw/workspace-<agentId>`. The related PRs #71899 and #71941 are open, not merged, so this issue remains the right canonical tracker.

## Best Possible Solution

Keep #71889 open and land a narrow wizard/default-helper fix, likely along the lines of #71941: for brand-new agents after normal setup, prefill the documented peer-level `workspace-<agentId>` path, while preserving existing-agent updates, explicit `agents.list[].workspace`, and custom shared `agents.defaults.workspace` bases. Add regression coverage in `agents.add.test.ts` plus helper/resolver coverage for the documented default and custom-base cases.
