## Summary

Keep #71889 open. Current main at 21c51bc140295e4286f1adc871a96b9b33c6a8b9 still seeds the interactive `agents add` workspace prompt from `resolveAgentWorkspaceDir(cfg, agentId)`, and that resolver still appends non-default agent IDs under `agents.defaults.workspace`. Setup/onboarding writes the main workspace into `agents.defaults.workspace`, while current docs say added agent workspaces should use peer-level `workspace-<agentId>` paths. The issue is also actively tracked by open fixing PRs #71899 and #71941, so it should stay open until a fix lands.

## Best Possible Solution

Keep the issue open and land a narrow regression-tested fix in the agents owner surface: add a helper for the `agents add` wizard that suggests peer-level `~/.openclaw/workspace-<agentId>` when `agents.defaults.workspace` equals the normal main workspace written by setup, while preserving explicit per-agent workspaces and deliberately configured shared workspace bases. PR #71941 appears to be the most targeted current implementation path because it changes the wizard prefill without changing runtime resolution contracts.
