## Summary

Current `main` already routes gateway restarts through authorized `SIGUSR1` paths, coalesces duplicate restart requests, skips no-op config writes, and carries a macOS launchd safeguard specifically documenting and avoiding the `kickstart -k`/`SIGTERM` pattern that made the gateway look crash-looped.

## Evidence

- **Gateway loop: `SIGTERM` stops, `SIGUSR1` restarts:** The gateway run loop logs `signal SIGTERM received` and handles it as `request("stop", "SIGTERM")`, while authorized restart requests flow through `SIGUSR1` and `request("restart", "SIGUSR1")`. That means the current restart path is no longer modeled as repeated `SIGTERM` self-restarts.
  - file: [src/cli/gateway-cli/run-loop.ts:222](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/cli/gateway-cli/run-loop.ts#L222)
  - command: `nl -ba src/cli/gateway-cli/run-loop.ts | sed -n '1,360p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **CLI lifecycle: unmanaged restart uses `SIGUSR1`, not `SIGTERM`:** For unmanaged gateways, `runDaemonStop` sends `SIGTERM`, but `runDaemonRestart` sends `SIGUSR1` to the verified gateway PID. This cleanly separates stop vs restart semantics in the current code.
  - file: [src/cli/daemon-cli/lifecycle.ts:110](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/cli/daemon-cli/lifecycle.ts#L110)

- **Config/UI writes now avoid unnecessary restart churn:** `config.patch` short-circuits identical writes (`changedPaths.length === 0`) and skips the file write and restart entirely; when a restart is needed it schedules `scheduleGatewaySigusr1Restart(...)` instead of issuing a stop signal.

  - command: `nl -ba src/gateway/server-methods/config.ts | sed -n '520,710p'`

- **Restart scheduler coalesces duplicate restart requests:** The shared restart runtime guards against duplicate emitted restarts, and tests assert that back-to-back scheduled restarts coalesce into a single pending `SIGUSR1` emission.
  - file: [src/infra/restart.ts:115](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/infra/restart.ts#L115)

- **macOS launchd path contains the exact safeguard for the reported symptom:** The LaunchAgent install path explicitly avoids `launchctl kickstart -k` because, per the in-code comment, it can `SIGTERM` the freshly booted gateway. Restarting from inside the LaunchAgent is handed off to a detached helper instead of killing the current process inline. The same behavior is present in the shipped `v2026.4.23` release commit.
  - file: [src/daemon/launchd.ts:590](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/daemon/launchd.ts#L590)

- **User-facing docs match the new stop-vs-restart contract:** The Gateway CLI docs state that `SIGUSR1` triggers an in-process restart, while `SIGINT`/`SIGTERM` stop the gateway. Plugin docs also explain that config writes typically restart automatically under the watch path, aligning with the audited behavior.
