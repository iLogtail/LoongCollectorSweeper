## Summary

Current main does not fully fix #60712. Telegram inbound routing still enters the generic `resolveAgentRoute` path, and a matched `bindings[]` rule whose `agentId` is absent from `agents.list` is normalized through `pickFirstExistingAgentId`, which falls back to the default agent. Current code has a narrow Telegram topic-agent override that preserves unknown topic `agentId`s, but that does not cover the reported configured-binding case.

## Best Possible Solution

Keep the issue open. The best fix is to make matched route bindings fail closed or validate `bindings[].agentId` against `agents.list` before runtime, add a Telegram regression test for a binding to a missing agent, and document any intentional catch-all behavior if maintainers choose that path.

## Evidence

## Risks / Open Questions

- Changing this globally may break setups that accidentally rely on stale binding IDs falling back to the default agent, so the fix should be explicit and covered by migration/doctor guidance if needed.
- A product decision is still needed between hard fail, config validation error, or a documented configurable catch-all agent; the current code does not implement any of those for route bindings.
