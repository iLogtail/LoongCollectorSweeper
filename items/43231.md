## Summary

Keep open. Current main has the #65973 partial fix: it can attach a known Telegram forum topic name to the current topic context, prompt metadata, and plugin hook metadata. It still does not provide the requested first-class topic-name -> topic_id resolution, topic directory, list action, or metadata endpoint for routing to another topic by human-readable name.

## Evidence

- **Issue discussion says the linked PR is partial:** The provided issue comment says PR #65973 maps chatId:threadId to a human-readable name and exposes topic names alongside IDs, but explicitly says it does not solve name->ID resolution directly. That matches current main behavior rather than superseding the request.
- **Current implementation stores id-to-name cache only:** The Telegram topic-name cache keys entries as chatId:threadId and exposes update/get APIs keyed by chatId and threadId. There is no exported list or reverse lookup API from name to thread ID in this module.
