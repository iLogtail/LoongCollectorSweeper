## Summary

Keep open. Current `main` has newer WhatsApp `fromMe` handling, but it does not implement the requested human-takeover behavior of pausing/suppressing bot replies for a conversation after a manual outgoing WhatsApp message.

## Evidence

- **No human-takeover config surface on current main:** `WhatsAppConfig` exposes `dmPolicy`, `selfChatMode`, allowlists, debounce, reply threading, etc., but there is no `humanTakeover`, `pauseOnOutgoingMessage`, or pause-duration setting in the WhatsApp config type.
  - file: [src/config/types.whatsapp.ts:53](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/src/config/types.whatsapp.ts#L53)
  - command: `nl -ba src/config/types.whatsapp.ts | sed -n '1,220p'`
  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **Outbound echo filter only covers gateway-sent messages:** The inbox monitor records outbound message IDs only from `sendTrackedMessage(...)`, then skips `fromMe` inbound echoes only when `isRecentOutboundMessage(...)` matches that tracked ID. That covers bot-sent echoes, not arbitrary manual sends from WhatsApp Web.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:245](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/extensions/whatsapp/src/inbound/monitor.ts#L245)
  - command: `nl -ba extensions/whatsapp/src/inbound/monitor.ts | sed -n '240,340p'; nl -ba extensions/whatsapp/src/inbound/monitor.ts | sed -n '360,382p'; nl -ba extensions/whatsapp/src/inbound/dedupe.ts | sed -n '110,132p'`
  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **Manual outbound DMs are dropped rather than used to pause the bot:** For direct chats, `checkInboundAccessControl` explicitly rejects `fromMe` messages that are not same-phone self-chat: `if (params.isFromMe && !policy.isSamePhone(params.from)) { ... allowed: false }`. That prevents pairing side effects, but it also means there is no pause/handoff action on a human operator's outgoing DM.
  - file: [extensions/whatsapp/src/inbound/access-control.ts:112](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/extensions/whatsapp/src/inbound/access-control.ts#L112)

  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **No cancellation or pause path for already-queued replies:** Inbound handling still routes messages through the shared debounce queue; the debouncer can batch and flush inbound work, but there is no path here for a manual outbound WhatsApp message to cancel a pending flush or suspend future auto-replies for that conversation.
  - file: [src/auto-reply/inbound-debounce.ts:166](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/src/auto-reply/inbound-debounce.ts#L166)

- **Docs describe self-chat safeguards, not operator takeover:** Current WhatsApp docs mention self-chat protections such as skipping read receipts and handling `fromMe` DMs, but they do not document any conversation pause or human takeover feature triggered by manual outgoing replies.
