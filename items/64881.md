## Summary

Keep open. Current main still derives `fullAccessAvailable` from config/allowlist state rather than a real provider/runtime capability signal, and tests still allow the ambiguous enabled+allowed case to resolve as available. The prompt layer correctly honors an explicit `false`, but the remaining issue is that current callers do not source that value from provider/runtime probes.

## Best Possible Solution

Keep the issue open and implement a real runtime-backed full-access capability signal. Thread that signal into `resolveElevatedPermissions` or the prepared-run path before `bashElevated` is built, make undefined `fullAccessAvailable` fail closed in `resolveEmbeddedFullAccessState`, and add regression coverage for ambiguous input plus provider/runtime blocked states and prompt integration.
