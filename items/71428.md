## Summary

Keep #71428 open. Current main implements and documents the config-level `plugins.entries.<id>.hooks.allowConversationAccess` path, but it still does not implement the requested manifest-owned `permissions.conversationAccess` contract. The loader passes only config entry hooks into hook enforcement, the manifest type/normalizer does not carry a `permissions` field, and tests/docs cover only the existing host-config opt-in. The prior Codex review comment in the issue is consistent with current source at 21c51bc140295e4286f1adc871a96b9b33c6a8b9.

## Best Possible Solution

Add an explicit, additive manifest permission contract, likely `permissions.conversationAccess`, in `src/plugins/manifest.ts` and manifest docs; normalize it through manifest loading/registry records; merge it with the existing `plugins.entries.<id>.hooks.allowConversationAccess` policy in the loader/registry path with documented config override precedence; and add tests for manifest-only allow, config-only backward compatibility, both-present precedence, and the rejected default case.

## Risks / Open Questions

- A manifest-owned conversation-access permission changes the trust model from operator config toward plugin-supplied metadata, so implementation should include explicit precedence, docs, and security review.
- Leaving only the config-level opt-in means third-party plugins like Smarter-Claw still need installation/config patching or operator setup instead of a self-contained manifest declaration.
