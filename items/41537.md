## Summary

Keep #41537 open. Current main has doctor-side repair coverage that preserves Telegram `dmPolicy: "allowlist"`, but the interactive configure path still offers the DM policy prompt with `pairing` as the default option instead of defaulting to the current policy. That leaves a meaningful remaining path where `openclaw configure` can revert an explicitly configured allowlist if the user accepts the prompt default.

## Evidence

- **Configure prompt still defaults to pairing:** `maybeConfigureDmPolicies` asks to configure DM access with message text saying default is pairing, then renders the policy select with `pairing` as the first option and no `initialValue` derived from `policy.getCurrent(...)`. The current policy is read only after the selection, so accepting the prompt default on an existing Telegram allowlist can choose `pairing`.
  - file: [src/flows/channel-setup.prompts.ts:106](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/src/flows/channel-setup.prompts.ts#L106)

  - sha: [e640c0a95fcb](https://github.com/openclaw/openclaw/commit/e640c0a95fcbd06a1311f18fc635518d011d45da)
- **Selected Telegram policy is persisted:** Telegram setup exposes `getCurrent`, but `setPolicy` writes the selected `dmPolicy` into the channel/account patch. If the configure prompt returns `pairing`, this setter is the write path that can replace an existing allowlist policy.
