## Summary

Keep open. The reported 2026.4.22 invalid generated thinking bundle appears superseded by the reporter’s own 2026.4.24 syntax check and current main has moved on, but the remaining Telegram webhook startup outage still matches current main. `startTelegramWebhook` binds the local HTTP listener, then calls Telegram `setWebhook`; if that call throws, current code closes the local server, stops the bot, and rethrows. The channel manager then logs `channel exited` and waits for restart backoff, leaving the reverse proxy with no local webhook listener during the backoff window.

## Best Possible Solution

Keep this issue open and fix the Telegram webhook startup path. The best product/code direction is to decouple the local listener lifetime from Telegram webhook advertisement: once the local listener successfully binds, keep it serving health/webhook routes while `setWebhook` retries recoverable network failures with bounded backoff, and only tear it down for permanent configuration/auth errors or explicit shutdown. Add a regression test that forces `setWebhook` to fail transiently and verifies the local listener remains available or is restored before channel restart backoff. Release/package syntax validation for generated JS can remain a separate publish-time guard because the issue’s syntax-error half is already reported as fixed in 2026.4.24.
