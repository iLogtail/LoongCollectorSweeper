## Summary

Current `main` already supports per-group and per-direct WhatsApp prompt injection via `channels.whatsapp.groups.*.systemPrompt` and `channels.whatsapp.direct.*.systemPrompt`, and wires that value into the runtime system prompt for the session. This addresses the reported cold-start context gap, even though the shipped field name is `systemPrompt` rather than `bootMd`.

## Close Comment

Closing from a Codex review: current `main` already implements the underlying request with WhatsApp per-conversation system prompts. The config schema now accepts `channels.whatsapp.groups.<jid>.systemPrompt` in `src/config/types.whatsapp.ts:25` and `src/config/zod-schema.providers-whatsapp.ts:21`, the WhatsApp inbound path resolves that prompt in `extensions/whatsapp/src/auto-reply/monitor/process-message.ts:239`, passes it through `GroupSystemPrompt` in `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts:142`, and the runner prepends it into the actual system prompt in `src/auto-reply/reply/get-reply-run.ts:297`. The docs also cover the feature in `docs/channels/whatsapp.md:495`. This is not the exact `bootMd` field proposed in the issue, but the reported cold-start group-context gap is implemented on `main` via `systemPrompt`, so this item is no longer actionable as an open feature request.
