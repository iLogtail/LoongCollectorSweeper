## Summary

Current main already implements Feishu bot-identity recovery for the reported `open_id = unknown` startup failure, and that implementation was shipped in `v2026.4.22`.

## Evidence

## Close Comment

Closing from a Codex review because current `main` already implements the reported Feishu recovery path. In `extensions/feishu/src/monitor.account.ts:421`, startup now detects an unresolved bot identity and calls background recovery instead of leaving `open_id` stuck as `unknown`, and `extensions/feishu/src/monitor.bot-identity.ts:29` retries the bot-info probe with backoff until it recovers, logging `bot open_id recovered via background retry` on success. This code is already present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

One note: the LaunchAgent part of the report was not fixed by raising `ThrottleInterval`; current main still keeps `ThrottleInterval=1` in `src/daemon/launchd-plist.ts:3`. So if you still see a distinct launchd restart-loop problem on `v2026.4.22` or later, please open a new issue with fresh logs from the current version. But the Feishu `open_id resolves as unknown and never recovers` regression described here is already implemented on main.
