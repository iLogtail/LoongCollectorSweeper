## Summary

Keep #60509 open. Current main still does not expose the incoming user message, prompt, or query string on the `agent:bootstrap` internal hook context, and the existing prompt-aware alternatives are separate surfaces rather than a drop-in way for `agent:bootstrap` hooks to query-condition and replace `bootstrapFiles`. The related ClawSweeper context also points in the same direction: similar missing hook/message metadata requests such as #52664 and #50482 were kept open when the exact event payload was still absent.

## Best Possible Solution

Keep open for a narrow API decision. The most direct implementation is an additive optional `userMessage` or `prompt` field on `AgentBootstrapHookContext`, threaded from CLI and embedded run call sites where the current prompt already exists, with docs and tests for `agent:bootstrap`. If maintainers prefer the plugin route, document SMELT-style compression as a context-engine plugin pattern using prompt-aware `assemble()` and `agents.defaults.contextInjection: "never"`.

## Risks / Open Questions

- Maintainers may prefer to redirect SMELT-style query-conditioned compression to a context-engine plugin plus `contextInjection: "never"`, but that is a product/API decision rather than evidence that the requested `agent:bootstrap` field is already present.
- An implementation needs to define which text is exposed (`prompt`, `BodyForAgent`, raw command body, or a normalized query) and keep it undefined for non-message runs such as heartbeat/startup paths.
