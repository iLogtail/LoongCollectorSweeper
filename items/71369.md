## Summary

Keep open. The report is recent, non-maintainer-authored, has no protected labels, and describes a plausible default-runtime failure mode: repeated no-output tool calls causing token waste on a Feishu/Qwen setup. Current main has opt-in loop detection and no-output poll backoff, but the core repetitive tool-call guard remains disabled by default, so current code does not prove the reported default behavior is fixed.

## Best Possible Solution

Keep the issue open and ask for a redacted trajectory/log bundle or exact tool sequence from the Feishu/Qwen run. Maintainers should then either reproduce on current main/latest release and add a regression around the implicated tool path, or decide whether to enable a narrow default no-progress guard for repeated no-output tool calls while preserving the existing opt-in loopDetection controls for broader cases.
