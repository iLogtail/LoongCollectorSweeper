## Summary

Keep #50773 open. The linked PR #50774 attempted the exact cron/lanes busy heartbeat behavior but was closed unmerged, and current main still only skips heartbeats for the main command lane and the resolved target session lane. The requested always-on cron-lane skip and opt-in `heartbeat.skipWhenBusy` config are not implemented in source, schema, docs, or tests.

## Best Possible Solution

Implement this in core heartbeat scheduling: always skip heartbeat model calls while cron work is active, using both cron active-job state and cron command-lane depth as needed; add an opt-in `agents.defaults.heartbeat.skipWhenBusy` / per-agent equivalent that checks all relevant lanes; update strict config schema, generated/help docs, and focused tests for cron active, cron lane queued, subagent/nested busy, wake retry, and scheduler no-advance behavior.

## Risks / Open Questions

- Leaving this open is appropriate because local-LLM resource contention can still happen when a cron job is active but the main/session lanes are idle.
- A correct fix must cover both scheduled cron active-job tracking and manual cron command-lane depth; checking only one would leave part of the report reproducible.
- If new skip reasons are introduced, the wake retry and scheduler no-advance logic must be extended consistently, or the heartbeat schedule may advance instead of retrying promptly.
