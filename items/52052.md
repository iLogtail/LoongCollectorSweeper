## Summary

Keep open. Current `main` still lacks the requested inbound `sticker_file_id` projection, and the PR as described is not yet a complete fix because Telegram sticker sending still accepts only `fileId`/`stickerId`, so echoing the newly surfaced `sticker_file_id` key would still fail.

## Evidence

- **Current main still omits sticker id from inbound conversation metadata:** `buildInboundUserContextPrefix()` builds `conversationInfo` with `chat_id`, `message_id`, `reply_to_id`, and `sender_id`, but there is no `sticker_file_id` field on current `main`. That means the behavior requested by the PR is not already implemented.
  - file: [src/auto-reply/reply/inbound-meta.ts:200](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/auto-reply/reply/inbound-meta.ts#L200)
  - command: `nl -ba src/auto-reply/reply/inbound-meta.ts | sed -n '170,260p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **Telegram send path still rejects the surfaced key:** `handleTelegramAction()` resolves sticker sends from `fileId` or `stickerId` only, then throws `fileId is required.` if neither is present. A model that copies `sticker_file_id` verbatim from prompt context would still miss this read path.
  - file: [extensions/telegram/src/action-runtime.ts:545](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/extensions/telegram/src/action-runtime.ts#L545)
  - command: `nl -ba extensions/telegram/src/action-runtime.ts | sed -n '537,549p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **Shared message-tool schema still advertises only stickerId:** The tool schema for sticker actions exposes `stickerId` but not `fileId` or `sticker_file_id`, so the proposed prompt key is not aligned with the documented/typed tool surface on current `main`.
  - file: [src/agents/tools/message-tool.ts:282](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/agents/tools/message-tool.ts#L282)

  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)

Reading `fileId` can match `file_id`, but it will not match `sticker_file_id`, so there is no hidden alias that makes the PR's surfaced key work today.
