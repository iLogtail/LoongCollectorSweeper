## Summary

Keep PR #68677 open. Current main at a35d259719cc7aecd1854fdc830ad6390b15f02a still lacks the resize-result memoization proposed by the PR: session history replay still routes historical image blocks through `sanitizeSessionMessagesImages`, `sanitizeContentBlocksImages` still calls `resizeImageBase64IfNeeded` directly for each valid image block, and that helper still decodes metadata with `getImageMetadata` before any early return. Main has an older-image pruning path, but it is not the same fix: it only targets completed turns older than the recent-turn window and is invoked after the replay sanitization path has already run in the attempt flow. The supplied PR discussion also supports keeping this open: #64514 was closed unmerged, #68677 is the active follow-up, and the Greptile parseInt nit was addressed in the PR head rather than shipped on main.

## Best Possible Solution

Keep PR #68677 open for maintainer review. The best path is to review or adapt its byte-bounded, full-payload-keyed resize cache in `src/agents/tool-images.ts`, keep the repeated-call/cache-bound regression tests, and fix the changelog attribution before merge.
