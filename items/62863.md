## Summary

Keep open: current `main` adds edit-tool recovery and better mismatch diagnostics, but it still preserves exact-match `oldText` semantics and still uses `read` paging via `offset`/`limit` rather than the requested line/keyword locators.

## Evidence

- **Edit recovery is explicitly non-semantic:** `wrapEditToolWithRecovery` says it recovers failure modes "without changing edit semantics" and only adds mismatch hints plus post-write success recovery.
  - file: [src/agents/pi-tools.host-edit.ts:145](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/agents/pi-tools.host-edit.ts#L145)

  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Exact-match mismatch is still the expected edit behavior:** Current tests still expect the edit error text `Could not find the exact text ... old text must match exactly including all whitespace and newlines`, which matches the issue's reported pain point rather than a fixed behavior.
  - file: [src/agents/pi-tools.read.host-edit-recovery.test.ts:89](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/agents/pi-tools.read.host-edit-recovery.test.ts#L89)
