## Summary

Current main still matches the reported failure mode. The shared promotion helper still moves all common keys when named accounts exist and no channel-specific `namedAccountPromotionKeys` is present; doctor then deletes those keys from the channel root and writes them to `accounts.default`. Slack is a concrete affected channel: it has no named-account promotion filter, and its tests assert named accounts inherit top-level `allowFrom` but do not inherit `accounts.default.allowFrom`. Related merged work fixed Telegram and Matrix only, not the broader fallback or affected channels.

## Best Possible Solution

Keep this issue open and fix the shared promotion contract. The most conservative path is to make named-account promotion default to credentials-only when a channel lacks `namedAccountPromotionKeys`, or add explicit `namedAccountPromotionKeys` for every affected channel, then add doctor/setup regression tests proving shared policy/delivery defaults stay at the channel root while credential/auth keys move into the promoted account.
