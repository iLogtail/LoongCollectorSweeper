## Summary

Keep open. Current `main` still filters `ollama-local` as a non-secret marker and only applies Ollama synthetic auth to the canonical `ollama` provider id, so the reported custom-named Ollama/subagent auth gap is not clearly fixed on `main`.

## Evidence

  - file: [src/agents/model-auth-markers.ts:7](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/src/agents/model-auth-markers.ts#L7)

  - file: [src/agents/model-auth.ts:114](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/src/agents/model-auth.ts#L114)

## Risks / Open Questions

- Subagents using custom-named Ollama providers can still miss auth and fall through to other providers.
- Because the fallback path is success-preserving, privacy/cost regressions can remain invisible to users until logs or billing are inspected.
- There is still no clear regression test on `main` for custom-named `api: "ollama"` providers with `apiKey: "ollama-local"`.
