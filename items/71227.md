## Summary

Keep open. Current `main` still uses a single monolithic `sessions.json` store, parses it synchronously, rewrites it wholesale, and persists `skillsSnapshot` inline. Recent maintenance controls reduce some growth cases, but they do not implement sharding, snapshot dedupe, or off-critical-path writes, so the reported timeout path is still plausible on current code.

## Evidence

- **Whole-store synchronous read/parse is still the load path:** `loadSessionStore()` still does `fs.readFileSync(storePath, "utf-8")` followed by `JSON.parse(raw)` for the entire store before access. There is no per-session shard or incremental read path here.
  - file: [src/config/sessions/store-load.ts:102](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/config/sessions/store-load.ts#L102)
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Whole-store stringify/write is still the save path:** `saveSessionStoreUnlocked()` still serializes the full in-memory store with `JSON.stringify(store, null, 2)` and writes it atomically. That means every update pays whole-file serialization cost.
  - file: [src/config/sessions/store.ts:358](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/config/sessions/store.ts#L358)
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **`skillsSnapshot` is still stored inline in each session entry:** `SessionEntry` still includes `skillsSnapshot`, and session update flow still writes that snapshot back into the persisted session entry on first-turn and refresh paths.
  - file: [src/config/sessions/types.ts:261](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/config/sessions/types.ts#L261)
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Session update flow still persists refreshed snapshots into the store:** `maybeSendAgentSystemIntro()` rebuilds or reuses the snapshot, assigns it to `nextEntry.skillsSnapshot`, and persists the entry. This matches the issue’s claim that snapshot payloads accumulate inside `sessions.json`.
  - file: [src/auto-reply/reply/session-updates.ts:167](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/auto-reply/reply/session-updates.ts#L167)
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Current safeguards are retention/capping only, not the requested architectural fix:** The built-in maintenance defaults are `pruneAfter=30d`, `maxEntries=500`, and `rotateBytes=10MB`. That can help stale or very large stores, but it does not dedupe inline snapshots, shard the store, or move writes off the request path.
  - file: [src/config/sessions/store-maintenance.ts:12](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/config/sessions/store-maintenance.ts#L12)
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **No post-release fix on current main for this architecture:** Diffing `v2026.4.23` against current `main` in the relevant session-store files shows only minor maintenance changes (for example resolving config at load time and copying backups instead of renaming). The monolithic parse/stringify architecture is unchanged.
