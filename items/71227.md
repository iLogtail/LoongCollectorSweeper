## Summary

Keep #71227 open. Current main has session maintenance, caching, and atomic writes, but the hot path still re-reads/parses and re-stringifies the monolithic sessions.json during updates. skillsSnapshot is still stored inline on each SessionEntry and written into the session row, so the reported latency/run-budget failure for active stores with large repeated snapshots is not solved. #69451 overlaps on the monolithic-store problem, but it tracks the memory/RSS angle and does not supersede this latency-specific repro.

## Best Possible Solution

Keep this open and fix it in core session storage. The likely maintainer path is to dedupe or content-address skillsSnapshot first, then consider per-session sharding/SQLite or off-critical-path writes for the broader monolithic-store issue. Add regression coverage for a large sessions.json below maxEntries with repeated skillsSnapshot data, verifying updates avoid whole-store latency growth or no longer consume the agent wait/run budget.
