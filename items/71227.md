## Summary

Keep #71227 open. Current main has session maintenance and caching mitigations, but the reported latency path still exists: session updates still synchronously parse the full `sessions.json` inside the write lock and serialize/write the whole store, while `skillsSnapshot` and `systemPromptReport` remain inline per session entry. The existing pruning/capping/rotation work can reduce some oversized stores, especially stale or >500-entry stores, but it does not solve the 286-entry/18.5 MB or 98-entry/4.7 MB inline-snapshot case described in the issue.

## Best Possible Solution

Keep the issue open and fix the core session-store hot path. The best product/code direction is to move repeated `skillsSnapshot` data out of line with content-addressed references or shard/session-index metadata per session, and ideally avoid full-store parse/stringify/write on ordinary session updates. Keep existing age/count/disk-budget maintenance as a fallback cleanup layer, but do not treat it as sufficient for the reported Slack timeout failure.
