## Summary

Keep open. Current main still uses a monolithic `sessions.json` read-modify-write path for session updates, still stores `skillsSnapshot` inline on session entries, and the existing maintenance controls only prune/cap/delete entries rather than solving the reported parse/write latency for large but still-retained stores. The related #69451 remains relevant but is broader memory/storage work and does not cover the targeted `skillsSnapshot` duplication and 60s run-budget failure well enough to supersede this issue.

## Best Possible Solution

Keep the issue open as a focused core storage/performance bug. The best fix is in `src/config/sessions/*` and the agent snapshot persistence path: either move `skillsSnapshot` to a content-addressed out-of-line store with migration/backfill tests, or shard session metadata per session/SQLite so active updates do not parse and rewrite unrelated entries. Add regression coverage for large stores with repeated snapshots, preserve current maintenance behavior as a fallback, and cross-link #69451 as the broader memory/storage tracking issue rather than closing this item.
