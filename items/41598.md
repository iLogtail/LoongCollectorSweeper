## Summary

Keep open. Current `main` still does not pass `tools.exec.host`, `tools.exec.security`, or `tools.exec.ask` into chat-bash exec creation, so the PR's main bugfix is not implemented on `main`. The extra bang-alias change in the PR also appears to narrow `!` routing relative to current arbitrary-command support, so there is not strong evidence to close this PR as obsolete or already fixed.

## Evidence

- **Chat-bash still omits host/security/ask on main:** `handleBashChatCommand` forwards `timeoutSec`, `notifyOnExit`, and `notifyOnExitEmptySuccess` into `createExecTool`, but it does not forward `tools.exec.host`, `tools.exec.security`, or `tools.exec.ask` on current `main`.
  - file: [src/auto-reply/reply/bash-command.ts:350](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/auto-reply/reply/bash-command.ts#L350)

  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Those omitted defaults materially affect exec policy:** `createExecTool` resolves host from `defaults?.host`, and resolves exec approval behavior from `defaults?.security` and `defaults?.ask` when present. If chat-bash does not pass those defaults, it cannot inherit the user's `tools.exec.*` policy settings.

- **Current main still supports arbitrary !-prefixed bash commands:** `handleBashCommand` currently treats any normalized command beginning with `!` as bash alias input, and `parseBashRequest` accepts any remaining text as a run command unless it matches `poll|stop|help`. That means the PR's narrower bang-matching change is not already reflected on `main` and would need careful follow-up because punctuation-prefixed commands are currently supported.
