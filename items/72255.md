## Summary

Keep open. Current main still allows Pi/model auth resolution to use stored auth-profiles entries when no declared auth.profiles entry or explicit auth.order constrains that provider, and the runtime path sets the selected credential without an orphan-profile warning. Related #40363 and #43945 cover adjacent stale/implicit fallback symptoms, but neither supersedes this Pi orphan-credential billing case.

## Best Possible Solution

Keep the issue open and fix the owner path in auth resolution/Pi runtime: make declared auth.profiles or explicit auth.order act as an allowlist for runtime profile selection, or emit a high-visibility warning and doctor/secrets-audit finding for stored profiles whose ids are no longer declared. Add a regression test that stages an undeclared anthropic setup-token in auth-profiles.json and proves Pi does not silently select it after the declaration is removed.
