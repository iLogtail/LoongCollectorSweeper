## Summary

Issue #55920 asks for a Neo4j long-term memory integration that the reporter already built as a standalone plugin. Current main explicitly treats memory as a plugin slot and directs optional integrations/community extensions to third-party plugins/ClawHub rather than OpenClaw core, so this belongs outside the core issue tracker.

## Close Comment

Closing this as a Codex maintainer review because it fits OpenClaw's third-party plugin/ClawHub scope rather than core backlog. Current main already treats memory as a plugin slot and exposes dedicated memory-plugin APIs (`src/plugins/types.ts:2130`, `src/plugins/registry.ts:1469`), while the project vision explicitly says optional capability should usually ship as plugins maintained in their own repo (`VISION.md:54`) and plugin docs say you do not need to add your plugin to the OpenClaw repository (`docs/plugins/building-plugins.md:16`). Since this Neo4j memory integration is already described here as a standalone plugin in its own repository, the right path is to keep publishing/maintaining it externally and, if useful, list it via the community plugin flow (`docs/plugins/community.md:9`).
