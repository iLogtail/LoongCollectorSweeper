## Summary

Keep open. Current main still consumes Discord interactive component entries by the clicked component ID, not by message/panel, so a `reusable:false` panel with multiple buttons can still deliver a second callback through a sibling button. The related #54233 report overlaps, but this issue is the cleaner focused tracker for the same still-real bug rather than a safe duplicate to close.

## Best Possible Solution

Keep this issue open as the focused tracker. The likely fix belongs in the Discord component registry/interaction path: assign a panel/message consumption group when registering component entries, and when any `reusable:false` entry is consumed, invalidate all sibling entries for that same component message or panel before dispatching the callback. Add a regression test with two buttons in one non-reusable component message that proves the second sibling click returns the expired response and does not dispatch to the agent.
