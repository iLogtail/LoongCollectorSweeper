## Summary

Keep #41685 open. Current main still lets Gateway/Control UI session listing aggregate disk-only agent stores, and the Control UI Sessions tab renders the `sessions.list` result without filtering to configured agents. Agent-delete/session-targeting guards now mitigate part of the deleted-agent path, but the requested registered-agent-only Control UI behavior is not implemented and the closed PR #41687 was not merged.

## Best Possible Solution

Keep the issue open and implement a scoped registered-agent boundary for Control UI: either add a `sessions.list` option such as `configuredAgentsOnly` and have the UI use it, or filter UI rows against `agents.list`/session defaults while preserving broader Gateway/ACP disk-only discovery for callers that need it. Also normalize or reject persisted/URL `agent:<id>:...` session keys when `<id>` is no longer configured, falling back to the default main session.

## Risks / Open Questions

- A registered-only backend change could regress the documented ACP/disk-only session visibility from #32831, so the fix likely needs a scoped Control UI behavior or an explicit `sessions.list` filter rather than removing discovery globally.
- The issue now includes two surfaces: Sessions tab visibility and chat restoration of explicit unknown-agent keys. A complete fix should cover both or split the restoration behavior into a narrower follow-up.
