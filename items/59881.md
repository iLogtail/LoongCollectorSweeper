## Summary

Keep open. Current main still builds agent hook context with `ctx.channelId` from the provider/channel name path (`params.messageChannel ?? params.messageProvider`) rather than the concrete conversation/channel target, while the actual target is carried separately as `currentChannelId` for tool context and is not exposed on `PluginHookAgentContext`. This matches the reported Discord failure mode closely enough that the item should not be auto-closed.

## Best Possible Solution

Keep the issue open and fix the hook contract in core: define provider versus conversation fields explicitly, populate `ctx.messageProvider` with the provider/plugin channel name, and expose the concrete channel/conversation id to agent hooks, preferably using `currentChannelId`/`OriginatingTo` and session-key parsing as fallbacks. Add regression coverage for Discord-style `agent:main:discord:channel:<id>` on `before_prompt_build` and `agent_end`, then update plugin hook docs/types so third-party plugins can rely on the contract.

## Risks / Open Questions

- Changing `ctx.channelId` semantics may affect plugins that have worked around the current provider-name behavior, so the fix should be documented and covered by compatibility-focused tests.
- The same hook context shape is built in embedded and CLI paths; a partial fix in only one runner could leave `before_prompt_build`, `agent_end`, or `before_agent_reply` inconsistent across runtimes.
