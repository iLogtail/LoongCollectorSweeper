## Summary

Keep open. Current `main` still does not implement the PR’s core security hardening: non-voice, non-node message providers such as Telegram/Discord/Slack pass through the message-provider filter unchanged, so the PR is not obsolete. The PR also is not a conservative close candidate because the supplied review discussion identifies unresolved correctness issues in the proposed patch around internal providers, memory flush, and ineffective `exec` owner-only enforcement.

## Best Possible Solution

Keep the PR open and ask for a narrower revision: scope the default dangerous-tool deny list to untrusted external channel auto-reply providers, explicitly preserve internal providers such as `webchat`, `heartbeat`, `cron-event`, and `exec-event`, preserve memory-flush `read`/`write` behavior or move that flow outside the untrusted tool surface, and wire any `exec` owner-only change through `src/agents/tool-policy.ts` or the exec tool definition. Add focused tests for external providers, internal providers, memory flush, and runtime owner-only enforcement.

## Risks / Open Questions

- Closing this PR would discard an unresolved security-hardening path that current `main` still lacks.
- Merging the PR as described without revision could regress trusted/internal runs or memory flush behavior because the default deny is applied by `messageProvider` before later policy stages.
- The defense-in-depth `exec` owner-only change needs to be wired through the runtime owner-only policy, not only the exported owner-only name constant.
