## Summary

Keep open. Current main still does not implement the issue's requested behavior: Signal group access is evaluated against the sender allowlist, not the Signal group ID, and the shared requireMention resolver still falls back to true when no explicit boolean is configured. The current docs also define groupAllowFrom as a group sender allowlist, so this needs maintainer/product resolution rather than automatic cleanup.

## Best Possible Solution

Keep the issue open. The best fix is for the Signal owner to clarify the product contract, then make Signal group access use a distinct group-id allowlist from channels.signal.groups or another documented route allowlist while preserving groupAllowFrom as sender authorization, and add regression tests for group ID allowlisting plus the chosen requireMention default for explicitly configured Signal groups.
