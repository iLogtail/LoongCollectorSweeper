## Summary

Keep open. Current main detects 429/rate-limit conditions and routes them through SDK short-retry, auth-profile cooldown/rotation, and configured model fallback, but it does not implement the issue’s requested paused-rate-limited run state, persisted retry_at metadata, or automatic same-step resume after a provider retry hint. The related closed reports cover fallback behavior, not this requested pause-and-resume workflow.

## Best Possible Solution

Keep #62540 open and implement it in the core model-run orchestration path: preserve structured retry-after/retry_at metadata from provider 429s, add a rate-limit paused/retrying run state visible in session/status surfaces, schedule a bounded jittered retry of the pending model call, and integrate it explicitly with existing auth-profile rotation/model fallback so retries do not duplicate final replies.
