## Summary

Keep PR #62740 open. Current main still calls `response.json()` in the Zalo API client before checking `response.ok`, so the reported non-2xx/non-JSON failure is not fixed on main. The PR targets the right plugin code path, but it still needs maintainer review and regression coverage, especially around bounded/sanitized error-body handling and HTTP 408 versus app-level polling timeouts.

## Best Possible Solution

Keep the PR open and ask for a final patch that adds the `response.ok` guard in `extensions/zalo/src/api.ts`, reads/surfaces HTTP error bodies with bounded and sanitized handling, preserves structured Zalo JSON errors, keeps HTTP status separate from app-level `error_code`, and adds focused tests in `extensions/zalo/src/api.test.ts` for those cases.
