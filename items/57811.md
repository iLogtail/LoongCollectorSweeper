## Summary

Keep open. Current `main` appears to have fixed the issue's secondary cron `payload.model` precedence problem, but the main Ollama failure is still plausible on `main`: provider runtime lookup only matches a plugin by provider id/aliases/hookAliases, and the bundled Ollama plugin still registers only canonical `ollama` with no aliases for custom `models.providers.<id>` entries using `api: "ollama"`. That means custom ids like `ollama-mini` can still miss `createStreamFn` resolution and never register the `ollama` API transport before pi-ai streams.

## Evidence

- **Embedded runner now does attempt lazy registration:** `runEmbeddedPiAgent` still calls `registerProviderStreamForModel(...)` before building the active stream function, so the original report's 'cron never reaches registration' explanation is no longer the full story on current `main`. The remaining question is whether plugin lookup finds the Ollama provider for custom ids.

- **Registration depends on provider-runtime plugin resolution:** `registerProviderStreamForModel` resolves a provider-owned stream via `resolveProviderStreamFn({ provider: params.model.provider, ... })` and only calls `ensureCustomApiRegistered` if that lookup returns a stream function. If provider lookup misses, no `ollama` API transport is registered.

- **Provider-runtime matching is id/alias based only:** `matchesProviderId` accepts only the plugin's canonical `id`, `aliases`, or `hookAliases`. `resolveProviderRuntimePlugin` uses that matcher, so a custom provider id only works if the plugin advertises that id as an alias/hook alias. There is no fallback match by configured `models.providers.<id>.api`.
