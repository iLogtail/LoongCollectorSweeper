## Summary

Keep #71195 open. Current main has the shared backend/browser Realtime Talk path and Voice Call realtime bridge, but the native macOS Talk client still uses the classic SFSpeechRecognizer -> chat.send -> chat.history -> TTS/talk.speak flow. The maintainer discussion also explicitly narrowed this issue to native macOS integration rather than closing it.

## Best Possible Solution

Keep the issue open and scope the implementation to native macOS Talk consuming the existing `talk.realtime.session` path: add a `GatewayConnection` wrapper, a native WebRTC/AVAudioEngine realtime session, overlay phase/cancellation wiring, and keep the current STT -> chat -> TTS path as fallback. Avoid adding parallel gateway PCM streaming unless maintainers explicitly choose a backend-mediated provider-agnostic design.
