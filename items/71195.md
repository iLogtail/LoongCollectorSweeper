## Summary

Keep #71195 open. Current main already has the backend/browser Realtime Talk path and OpenAI realtime provider support, but native macOS Talk still runs the older SFSpeechRecognizer -> chat.send -> history polling -> TTS/talk.speak flow. Maintainer discussion explicitly narrowed the issue to native macOS consuming the existing realtime Talk session flow, not a new gateway realtime stack.

## Best Possible Solution

Keep the issue open as the native macOS Realtime Talk tracking item. The best implementation path is to add a GatewayConnection wrapper for talk.realtime.session, add a native macOS realtime audio/WebRTC session that uses AVAudioEngine capture and streaming playback, wire TalkModeRuntime/overlay phase/cancellation into that session, and keep the current SFSpeechRecognizer -> chat.send -> talk.speak/TTS path as fallback. Split the ElevenLabs repeated PCM 403 behavior into a smaller Mac Talk TTS issue or PR.

## Risks / Open Questions

- Closing this now would discard an active maintainer-narrowed macOS app enhancement; the backend/browser pieces are not enough to satisfy the native macOS user problem.
- The ElevenLabs PCM 403 subthread is a separable Talk TTS fallback/config issue and should not be used to close or block the native realtime Talk request.
- A future implementation should avoid adding a parallel gateway PCM protocol unless maintainers decide macOS must use a provider-agnostic backend bridge instead of the existing browser-style realtime session path.
