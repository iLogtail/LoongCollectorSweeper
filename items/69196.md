## Summary

Keep #69196 open. Current main still implements the Control UI microphone dictation button through browser `SpeechRecognition`/`webkitSpeechRecognition`, and the UI still silently clears recording state on STT errors. The shipped same-origin `Permissions-Policy` header helps ordinary microphone access, but the reporter specifically narrowed this bug to Chrome Web Speech returning `not-allowed` after microphone permission was granted and after trying a local microphone-header change.

## Best Possible Solution

Keep the issue open and fix the Control UI dictation path directly. A good resolution would either add a robust fallback from browser audio capture to a gateway/provider STT path, or at minimum detect and surface `not-allowed` with actionable UI/docs so users are not left with a silent stop. The existing same-origin `Permissions-Policy` header should remain, but it is not sufficient to close this Web Speech failure report.
