## Summary

Keep #70567 open. Current main has retry and atomic full-reindex safeguards, but `memory status` still reports `Dirty` from in-memory flags plus metadata existence rather than from a source-file/index coverage audit. A separate status process can therefore still print `Dirty: no` when the SQLite index is partial, even though the CLI scans enough source files to show an indexed/total mismatch. The related #26772 is adjacent checkpoint/resume work, not a canonical fix for this status correctness bug.

## Best Possible Solution

Keep the issue open and fix the builtin memory status/index contract: persist failed or incomplete indexing state, or teach status/deep status to compare configured source files against `files`/`chunks` coverage and surface missing paths as dirty. Add a regression test that simulates a transient embedding 500 during incremental indexing, then creates a fresh status/index manager and verifies missing files are reported or reprocessed without requiring `--force`.

## Risks / Open Questions

- A fix should avoid making plain `memory status` unexpectedly expensive on large `extraPaths`; a `--deep` audit or cached completeness metadata may be safer.
- The builtin and QMD memory backends have different status semantics, so the fix should be scoped to the builtin SQLite index path unless a shared status contract is added.
