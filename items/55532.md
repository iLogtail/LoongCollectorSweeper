## Summary

Keep open. Current main still delegates Feishu WebSocket reconnect behavior to the pinned Lark SDK without an OpenClaw-side exponential backoff, PingInterval guard, closed-socket race fix, or token-cache poisoning mitigation. The issue overlaps with #55531 and open PR #55619, but #55619 is not merged and the issue comment adds a separate SDK token-cache failure mode that should not be discarded automatically.

## Best Possible Solution

Keep the issue open until a maintainer either lands a tested Feishu WebSocket fix or deliberately consolidates it into a canonical Feishu reconnect issue/PR. The best code path is a narrowly tested Feishu-owned mitigation around the Lark SDK: bounded exponential backoff with jitter/reset semantics, a narrow PingInterval guard, closed-client race handling, and a decision on whether OpenClaw should patch or upstream the token-cache poisoning behavior in `@larksuiteoapi/node-sdk`.
