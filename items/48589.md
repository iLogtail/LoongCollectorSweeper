## Summary

Keep #48589 open. Current main still has the pre-PR TUI behavior: the TUI gateway client uses a fresh random instance ID per process, default/main TUI sessions resolve to the ordinary agent main key, gateway chat routing does not synthesize a TUI affinity origin, and the affinity/instance modules from the PR are absent. The provided PR discussion is useful, but the PR needs revision for the P1 orphan short-reply reattachment concern rather than closure.

## Best Possible Solution

Keep the PR open, or replace it with a revised implementation that preserves TUI main-session continuity through a stable per-terminal TUI instance ID, a documented affinity session-key/origin contract, and regression tests. The revision should address the P1 review by gating orphan short-reply reattachment to default TUI main-session aliases only and should align current TUI client naming/mode behavior with gateway routing tests.
