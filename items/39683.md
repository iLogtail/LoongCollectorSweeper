## Summary

Keep open. Current main still does not implement the PR’s requested create/update-time rejection for Feishu/Lark announce jobs without a resolvable delivery target, and the linked canonical bug #39599 remains open. The PR is still a plausible focused bug fix, but its blanket `delivery.to` requirement is not the right current-main implementation because target resolution can come from session history or plugin-owned/default resolution, and the PR discussion already identified a legacy update/disable regression.

## Best Possible Solution

Keep this PR open, but do not merge the current blanket `delivery.to` check as-is. The best path is a narrower current-main implementation, likely in the gateway/cron create and target-changing update validation path, that dry-runs the same delivery target resolution used by execution/list previews and rejects Feishu/Lark announce jobs only when neither explicit `delivery.to` nor deterministic context/plugin resolution can provide a valid target. It should include regression coverage for create failure, explicit target success, session-derived target success, and disabling legacy missing-target jobs. Keep #39599 as the canonical bug until that implementation lands.
