## Summary

Keep open. Current main has provider-scoped proxy routing via `models.providers.*.request.proxy` and related global/env proxy work from #67228, but it does not implement a per-auth-profile `proxyUrl`. Auth profiles only carry credential/provider metadata, profile selection returns profile IDs/credentials, and transport proxy dispatchers are attached from static model-provider config rather than the active auth profile.

## Best Possible Solution

Keep the issue open and implement this as a core auth-profile transport feature if maintainers want it: add a typed per-profile proxy field with secret/redaction handling, validate it in auth-profile config/store paths, and thread the active profile's proxy into provider request dispatcher construction for supported LLM transports. Existing `models.providers.*.request.proxy` should remain the provider-level override for static provider aliases.

## Risks / Open Questions

- A provider-alias workaround with separate `models.providers.<alias>.request.proxy` entries may cover some single-account-per-alias deployments, but it does not preserve the requested auth-profile rotation semantics for one provider.
- A complete implementation would cross auth-profile persistence/schema, SecretRef/redaction policy for proxy credentials, provider auth selection, stream/WebSocket transports, and provider plugin contracts.
