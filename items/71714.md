## Summary

Keep open. Current main clearly handles ordinary unshifted Enter in the Control UI textarea, but the report is recent, concrete, Windows Chrome/Edge-specific, and plausibly tied to browser/IME key events that the current tests do not cover. Source inspection alone is not strong enough to close this as implemented or not reproducible.

## Best Possible Solution

Add a focused Control UI chat composer regression around the textarea keydown behavior: ordinary Enter prevents default and sends, Shift+Enter remains newline, and composition-confirmation Enter does not send. Then verify the reported Windows Chrome/Edge path and adjust the IME/composition guard or add a configurable send shortcut if that environment still cannot send from the keyboard.
