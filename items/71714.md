## Summary

Keep #71714 open. Current main handles ordinary non-Shift Enter in the Control UI textarea, but that same handler is already present in the latest release tree, so its presence does not prove the reported v2026.4.22+ Windows Chrome/Edge regression is fixed. The issue is recent, concrete, and may involve a browser/IME path because the handler returns before preventDefault when `isComposing` or `keyCode === 229`, which can still allow the textarea newline behavior the reporter describes.

## Best Possible Solution

Keep the issue open and reproduce against current main on Windows Chrome/Edge, including Chinese/Japanese IME composition and plain ASCII input. If it reproduces, fix `ui/src/ui/views/chat.ts` so completed composition Enter does not fall through to textarea newline while preserving Shift+Enter and active-composition candidate confirmation, then add a jsdom/browser regression test that dispatches Enter on `.agent-chat__input textarea` and covers the composition guard.
