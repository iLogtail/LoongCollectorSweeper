## Summary

Keep PR #71237 open. Current main has related orphan handling in doctor and disk-budget cleanup, but it does not include the proposed `pruneOrphanedTranscripts` utility or an equivalent general sessions-cleanup path that scans a sessions directory and prunes already-unindexed `.jsonl` transcripts on demand. The PR remains a plausible focused follow-up rather than something clearly implemented, duplicated, or superseded.

## Best Possible Solution

Keep the PR open for maintainer review. The best product path is to turn the primitive into an explicit `openclaw sessions cleanup` orphan-transcript sweep, collecting preserved paths from all relevant stores plus compaction checkpoints before deleting anything. If maintainers prefer the already-shipped doctor/disk-budget approach, close this only after creating or pointing to a canonical follow-up that explains why direct cleanup-command pruning is not wanted.
