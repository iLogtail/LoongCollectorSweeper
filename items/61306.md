## Summary

Keep open. Current main does not implement the Claw mission-control backbone, and the PR is not cleanly superseded by an existing core implementation or canonical related item. It also is not a high-confidence ClawHub close: the mission engine, gateway RPC/events, durable state, and Control UI surface are product/core-shaped, while current plugin surfaces do not expose a Control UI tab/panel extension point. The PR is very broad and has serious review concerns, so the right maintainer path is explicit product review and decomposition, not automated closure.

## Best Possible Solution

Keep the PR open for maintainer judgment, but do not treat it as ready. The best path is to decide whether Claw mission control belongs in core; if yes, split it into focused PRs such as specs/docs, browser hardening, mission state/service, gateway RPC/events/scopes, Control UI, and CI/tests, with all P1/P2 review findings addressed. If maintainers want this outside core, first define the missing plugin/UI extension seam and then move the mission runner to a ClawHub/plugin package using existing Task Flow and plugin runtime surfaces where possible.
