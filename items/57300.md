## Summary

Keep PR #57300 open. Current `main` has not implemented the proposed direct-delivery bypass: `agentCommandInternal` still runs the model attempt before calling delivery for `opts.deliver === true`. The PR is a focused attempt to fix a real linked bug (#57284), but the final PR discussion also identifies a current regression risk: other `deliver: true` call sites without `internalEvents`, especially exec approval followups, intentionally rely on the LLM turn to rewrite instruction-heavy prompts into user-safe messages.

## Best Possible Solution

Keep the PR open, but require a narrower implementation before merge: preserve the existing LLM-backed `deliver: true` contract for internal followups and automation prompts, add an explicit direct-delivery path for the CLI notification use case (or a distinct option/ingress flag), keep gateway-first routing, and add regression tests for direct CLI delivery, subagent announcements with `internalEvents`, and exec approval followups without `internalEvents`.

## Risks / Open Questions

- Closing this PR would drop a plausible fix path for open bug #57284 while current main still routes `agent --deliver` through the LLM turn.
- Merging the PR as-is is also risky because a broad `deliver: true && no internalEvents` bypass conflicts with existing exec approval followup workflows that expect model synthesis before external delivery.
- The current public CLI wording says `--deliver` sends the agent's reply, so any semantic change to direct message delivery needs a narrowly named mode or coordinated docs/help updates.
