## Summary

Keep #57326 open. Current main has fixed the primary/direct agent execution pattern for CLI runtime aliases, but the original helper surfaces named in the issue still contain direct embedded-runner calls, and the newer cron comment identifies a real remaining mismatch: cron checks `isCliProvider()` on the canonical provider before resolving the configured CLI runtime.

## Best Possible Solution

Keep #57326 open. The best fix is to apply the established `resolveCliRuntimeExecutionProvider(...) ?? provider` plus `isCliProvider(...)` dispatch pattern to the remaining helper paths, route CLI-backed helper runs through `runCliAgent()`, preserve auth/session/runtime context, and add focused regression coverage for queued followups, `models status --probe`, LLM slug generation, and cron canonical-provider runtime selection. Prewarm should also be reviewed so configured CLI runtimes do not produce startup harness warnings or failures.
