## Summary

Keep #57326 open. Current main still contains the direct embedded-runner paths named in the issue for queued followups, `models status --probe`, and LLM slug generation. The newer cron report also matches current code: cron checks `isCliProvider()` on the fallback provider directly, while the established main-runner and command paths first resolve the configured CLI runtime provider.

## Best Possible Solution

Keep #57326 open. The best fix is to apply the established `resolveCliRuntimeExecutionProvider(...) ?? provider` plus `isCliProvider(...)` dispatch pattern to queued followups, auth probes, slug generation, and cron execution; route CLI-backed attempts through `runCliAgent()` while preserving auth profile, runtime config, session binding, and fallback behavior; then add focused regression tests for each helper path and the canonical-provider cron case.
