## Summary

Keep open. Current main still has no configurable umask/file-mode surface for agent-created user-facing files, and the relevant write paths either rely on process defaults or hardcode owner-only modes. The related permission fixes are for media/temp/sensitive state, not general agent output from write/edit/apply_patch/exec.

## Best Possible Solution

Keep the issue open and design a core-owned permissions policy for agent-created user output. The likely path is a documented tools.fs or tools.outputPermissions setting with validated octal modes/umask, threaded through write/edit/apply_patch, sandbox fs bridge, and exec sandbox/gateway launch environments, while explicitly excluding sensitive internal state and documenting Docker UID/GID limits.

## Risks / Open Questions

- A naive global process.umask change could weaken sensitive OpenClaw config, credential, transcript, and state-file protections.
- File mode alone will not solve UID/GID ownership mismatches in Docker/NAS deployments; implementation should document when docker.user, bind-mount ownership, or ACLs are still required.
- Exec-created files are harder than file-tool-created files because arbitrary child processes create files directly; support likely needs scoped shell/container umask handling rather than only wrapping OpenClaw fs helpers.
