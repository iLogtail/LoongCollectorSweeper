## Summary

Keep PR #69363 open. Current main still has the parser behavior the PR is trying to fix: `parseCliJsonl` captures the first `session_id`, while `parseCliJson` and the streaming parser still accept later `session_id`s without distinguishing transient Claude hook records. The existing #61390-era handling catches `No conversation found` as a stale-session error, but it does not prevent persisting the wrong hook id and losing Claude CLI conversation continuity.

## Best Possible Solution

Keep the PR open and land this behavior after maintainer review, either by merging/rebasing #69363 or by applying an equivalent helper in `src/agents/cli-output.ts` that prefers non-hook Claude records across `parseCliJson`, `parseCliJsonl`, and `createCliJsonlStreamingParser`, with the PR's focused regression tests. Before merge, run the changed-agent lanes and verify the live Claude CLI resume path if available.
