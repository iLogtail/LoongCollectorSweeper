## Summary

Keep open. Current `main` and the latest release still use session-id selection logic that can capture transient Claude hook `session_id` values, and current tests do not cover the reported `SessionStart:resume` ordering. I did not find an equivalent fix landed on `main`.

## Evidence

- **Current parser still accepts any `session_id` without filtering hook records:** `pickCliSessionId()` returns the first matching string field from the parsed record and has no special case for Claude `hook_started`/`hook_response` records, so transient hook ids are eligible everywhere this helper is used.

- **`parseCliJsonl()` still has the reported first-wins bug on main:** `parseCliJsonl()` only sets `sessionId` when it is unset. With the PR's reported ordering (`hook_started`/`hook_response` before `init`), that logic preserves the transient hook id instead of the resumable conversation id.
