## Summary

Keep #71581 open. Current main still processes forced exec-event and cron-event heartbeat wakes in the literal same-agent session key after only main-alias canonicalization; it does not consult `session.dmScope` or the single-owner allowlist helper used by inbound DM routing. The provided related items cover adjacent routing surfaces but do not supersede this specific internal heartbeat wake behavior.

## Best Possible Solution

Keep the issue open and implement this in core heartbeat/cron/exec session resolution, not as a plugin. A good fix would add a shared helper that, when `session.dmScope` is `main` and the same single-owner allowlist precondition holds, maps same-agent direct forced wake session keys to `resolveAgentMainSessionKey`; it should also decide whether system events are re-keyed at enqueue time or migrated during preflight. Add focused regression coverage for background exec completion and main-session cron wakes showing the model context `SessionKey` and queued event history land in `agent:<id>:main` while multi-owner or non-main `dmScope` deployments remain unchanged.
