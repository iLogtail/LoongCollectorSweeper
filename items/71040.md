## Summary

Keep PR #71040 open. Current main still has the specific producer-side cron active-job gap described by the PR: normal due jobs and executeJob mark the job active, but startup catch-up and prepared manual runs do not. The newer durable cron recovery path helps after terminal run history exists, but it does not provide a live in-flight signal during a long startup catch-up or manual run.

## Best Possible Solution

Keep the PR open. The best path is to merge or apply the same scoped fix on main: mark startup catch-up and prepared manual runs active immediately before job execution, clear them in finally, and keep regression coverage that observes isCronJobActive mid-flight without stubbing task-registry maintenance.
