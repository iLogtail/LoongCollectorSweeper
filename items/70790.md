## Summary

Keep PR #70790 open. Current main has shipped related fixes for toolsAllow forwarding and the BTW tool-less path, but it has not implemented this PR's broader embedded-runner payload guard. The remaining idea is still plausible because embedded stream resolution still returns unwrapped stream functions, while OpenAI-compatible transport code can still build payloads with empty `tools` in tool-less or replay contexts.

## Best Possible Solution

Keep the PR open, but ask for a rebased, focused payload-boundary fix. The best implementation would add a shared embedded/provider payload sanitizer that omits `tools` when empty and removes `tool_choice`/related tool-only fields when no non-empty tools remain, with tests for boundary-aware, custom/session fallback, explicit tool-less, and tool-history replay paths. It should also reconcile the overlap with #47947 and preserve existing `onPayload` composition semantics.
