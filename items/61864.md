## Summary

Current main still does not guarantee the invariant requested by #61864. Compaction accounting only refreshes token state when `tokensAfter` is available; otherwise it preserves the existing `totalTokens` and freshness state. The memory-flush and auto-compaction paths can hit that no-`tokensAfter` branch, and the next memory-flush gate can continue preferring a larger persisted “fresh” total over a smaller transcript-derived post-compaction estimate.

## Best Possible Solution

Keep #61864 open and fix the compaction accounting invariant directly: whenever compaction completes, persist a trustworthy `tokensAfter` if available; otherwise clear or mark `totalTokensFresh=false` so subsequent memory-flush/preflight gates cannot reuse a pre-compaction total as fresh. Add tests for memory-flush and normal auto-compaction paths where compaction increments without `tokensAfter`, including the case where transcript-derived post-compaction tokens are lower than the previous cached total.
