## Summary

Keep #69519 open. Current main still renders the Sessions Limit control as a plain string-backed input and stores the filter as string state, while the gateway protocol expects a positive integer limit. The PR targets a real Control UI validation gap, but the provided review discussion shows the current PR needs cleanup around state typing and blank/invalid input handling before merge.

## Best Possible Solution

Keep the PR open and request a coherent Sessions limit implementation: either keep string UI state and parse only at the controller boundary, or migrate the whole Control UI Sessions limit state to `number | undefined`; in either case, use a numeric input with `min=1`/`step=1`, preserve a clear blank/invalid state, and add focused view/controller tests for the input and resulting `sessions.list` params.
