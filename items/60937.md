## Summary

Keep PR #60937 open. Current main still does not add the requested baseline headers to the standalone `startCanvasHost` HTTP server or cover that behavior in canvas-host tests. The shipped gateway path already applies default security headers before routing to canvas, but that is a different path from the standalone server this PR modifies. The PR should get maintainer review rather than be auto-closed, especially because `X-Frame-Options: DENY` may need reconciliation with the gateway comment that canvas/A2UI can be framed.

## Best Possible Solution

Keep the PR open and have a maintainer review the standalone server hardening. The likely code direction is to add at least `X-Content-Type-Options: nosniff` and `Referrer-Policy: no-referrer` at the top of `startCanvasHost` before routing, add regression coverage for 200/404/500-style paths, and explicitly decide whether standalone canvas responses should also send `X-Frame-Options: DENY` despite the gateway's existing frame-compatibility policy.
