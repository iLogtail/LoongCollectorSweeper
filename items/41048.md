## Summary

Keep #41048 open. The underlying Telegram null-media path is still present on current main: getFile failures can return null, the inbound handler only warns on thrown media errors, and body construction still emits a media placeholder even when no media was resolved. The PR is not merge-ready as-is, but it still targets a real, current bug tracked by #40991.

## Best Possible Solution

Keep the PR or its linked issue active, then port the fix to the current extensions/telegram implementation. The best product fix is to surface a user-visible Telegram media download failure when resolveMedia() returns null for inbound media, avoid bare placeholder-only agent bodies, preserve captions, add regression coverage for single-message and media-group null-return cases, and decide whether the placeholder should become explicit metadata such as attachment-unavailable rather than implying an attached file.
