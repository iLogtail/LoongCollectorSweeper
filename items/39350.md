## Summary

Keep PR #39350 open. Current main still has the gateway final-message gap for a lifecycle ending after only a partial `NO_REPLY` fragment: delta/flush paths suppress lead fragments, but `emitChatFinal` resolves the buffer with lead-fragment suppression disabled and would emit buffered `NO` as the final assistant message. The PR identifies a real bug and adds a useful partial-fragment regression test, but its current one-line `isSilentReplyPrefixText` guard would also suppress a legitimate uppercase `NO` final reply.

## Best Possible Solution

Keep the PR open, but require a revised gateway final-path fix. The best direction is to distinguish buffered control-token provenance from a deliberate final `NO` answer, then add regression coverage for lifecycle end after only `NO`/`NO_` is buffered and coverage preserving an intentional uppercase `NO` final response.
