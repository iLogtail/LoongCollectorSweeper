## Summary

Keep PR #39350 open. Current main still appears to have the narrow final-path gap: a buffered partial `NO_REPLY` fragment such as `NO` is recognized as a suppressed lead fragment on delta/flush paths, but `emitChatFinal` only suppresses exact control-token text before constructing the final chat payload. The PR is also not clearly mergeable as-is because its proposed `isSilentReplyPrefixText` guard would drop deliberate uppercase `NO` final replies, matching the existing review-comment concern.

## Best Possible Solution

Keep the PR open, but do not merge its current one-line guard unchanged. The best path is a gateway final-path fix with a regression test for lifecycle end after only `NO` is buffered, plus explicit handling or maintainer-approved policy for deliberate uppercase `NO` replies so the final-message fix does not trade one visible bug for another.
