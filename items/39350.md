## Summary

Keep PR #39350 open. Current main still allows a lifecycle final event to emit a buffered partial NO_REPLY lead fragment such as "NO", but this PR should not be closed as obsolete or merged unchanged because its proposed prefix guard would also suppress a legitimate final assistant reply of uppercase "NO".

## Best Possible Solution

Keep the PR open and ask for a revised gateway fix. The best implementation is to carry provenance that a buffered value came from a suppressed control-token lead fragment during streaming, then suppress only that final payload while preserving intentional uppercase `NO` replies. Add focused regression coverage for lifecycle end after only `NO` and `NO_`, plus a case proving a deliberate final uppercase `NO` remains visible.
