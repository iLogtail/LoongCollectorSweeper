## Summary

Keep open. Current main still resolves auto-reply exec overrides without sender ownership, the call site does not pass `command.senderIsOwner`, and exec remains a normal tool rather than an owner-only tool. The PR is still a useful security fix path, but its latest review context also identifies an unresolved edge case around partial `agentExecDefaults`, so this needs maintainer review/changes rather than cleanup closure.

## Best Possible Solution

Keep the PR open and request a focused revision: thread `command.senderIsOwner` into reply exec override resolution, make non-owner senders ignore user-injectable inline/session exec settings, and only let explicit operator-configured policy loosen the deny default when `security`/`ask` are safe and complete. Add regression tests for non-owner no-overrides, inline injection, persisted session overrides, and partial agent `tools.exec` defaults. Coordinate with the broader channel tool restriction work in #64719, but keep this PR as the narrow exec-default hardening path unless a canonical replacement lands first.
