## Summary

Keep open. Current main has SecretRef-based secret management, plaintext auditing, one-way scrubbing, file permission hardening, and redaction, but it does not implement the requested transparent encrypted `.env`/config storage with `ENC:` values, master key/keyring handling, decryption on read, or key rotation. The repo’s own threat model still lists token encryption at rest as a remaining recommendation, so this is a real core/security feature request rather than ClawHub cleanup or an implemented item.

## Best Possible Solution

Keep this open as a core security/design issue. The best path is to build on the existing SecretRef/secrets runtime: either add a first-class encrypted local secret provider or a carefully scoped encrypted-at-rest store with documented keyring/master-key behavior, migration/audit support, and tests for `.env`, `openclaw.json`, `auth-profiles.json`, startup/reload, rotation, and failure recovery. Do not close it as ClawHub work unless maintainers explicitly decide encrypted local storage should be implemented as an external provider with sufficient core API support.

## Risks / Open Questions

- The existing SecretRef system partially addresses the user problem by moving many secrets out of plaintext config, so any implementation should avoid duplicating or weakening that contract.
- Transparent `ENC:` support would need explicit product/security design for key derivation, OS keychain behavior, headless/server installs, backup/restore, rotation, failure modes, and migration from existing SecretRefs/plaintext values.
