## Summary

Keep open. Current `main` appears to cover the DM takeover part through current-conversation binding, but it does not clearly implement the issue's requested parent-handoff lifecycle, especially reverting back to the parent on `/new` or preserving/restoring a previous binding.

## Evidence

- **Telegram supports current-conversation binding:** The Telegram channel plugin advertises `supportsCurrentConversationBinding: true` and `defaultTopLevelPlacement: "current"`, which means Telegram DMs/topics can bind directly to the current conversation instead of requiring a child thread.
  - file: [extensions/telegram/src/channel.ts:620](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/telegram/src/channel.ts#L620)
  - command: `nl -ba extensions/telegram/src/channel.ts | sed -n '618,636p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **ACP bind path is generic, not topic-only:** `bindSpawnedAcpSessionToCurrentConversation()` accepts any active conversation with current-placement support and only checks for a resolved `conversationId`; there is no Telegram-DM-specific block in the current code path.
  - file: [src/auto-reply/reply/commands-acp/lifecycle.ts:185](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/auto-reply/reply/commands-acp/lifecycle.ts#L185)

  - file: [src/auto-reply/reply/commands-acp.test.ts:1268](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/auto-reply/reply/commands-acp.test.ts#L1268)
