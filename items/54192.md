## Summary

Keep open. Current main still resubmits the effective prompt to an existing CLI backend session on model fallback retries; the orphaned trailing-user repair exists for the embedded runner, but there is no equivalent guard in the CLI runner path.

## Best Possible Solution

Keep the issue open and implement a CLI-specific resumed-session retry guard. The likely fix belongs around `src/agents/command/attempt-execution.ts` or `src/agents/cli-runner.ts`: when `isFallbackRetry` and a reusable `cliSessionId` are present, either port the embedded `mergeOrphanedTrailingUserPrompt` pattern against the relevant transcript/session state or skip re-submission when the same user prompt is already the last queued turn. Add focused regression coverage for CLI fallback retries with an existing session id, plus a negative test that normal first attempts and fresh-session retries still submit the prompt.
