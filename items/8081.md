## Summary

Issue #8081 should stay open. Current main still documents and implements OpenClaw as a one-trusted-operator gateway, with partial controls for shared inboxes, operator scopes, trusted proxies, config redaction, and sender/tool policy, but not the requested shared-gateway admin/user/guest RBAC, per-user credential visibility/isolation, or audit model.

## Best Possible Solution

Keep #8081 open as the broad product/security tracker. A complete direction should define gateway/channel caller identity, role- or scope-based authorization for sensitive config and credential operations, per-user or per-tenant secret isolation, audit logging, channel/group identity mapping, documentation of the trust boundary, and regression tests across gateway auth, config read/write paths, HTTP APIs, and channel command surfaces. If maintainers choose an external-wrapper or ClawHub-first direction, document that product decision and identify any missing plugin/API surfaces needed to support it outside core.

## Risks / Open Questions

- Partial controls such as trusted-proxy allowUsers, operator scopes, config redaction, per-agent auth stores, sandboxing, and toolsBySender can be mistaken for RBAC, but current docs and code do not define them as a per-user tenant or secret-visibility boundary.
- External wrapper projects mentioned in the issue may be good deployment paths, but closing #8081 would hide the unresolved product/security decision about whether OpenClaw core should define shared-gateway identity, authorization, secrets isolation, and audit behavior.
