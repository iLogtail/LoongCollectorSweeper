## Summary

Keep #39669 open. Current main still lacks a generic user-visible queued-message feedback path for busy-session followups: the busy `enqueue-followup` branch enqueues, touches the session, cleans up typing, and returns without any queued callback, receipt, or reaction. Feishu/Lark still only gets typing feedback from reply-start callbacks, so the reported channel remains uncovered. The related PR #39676 is closed unmerged, and #39117 is still open/unmerged in the provided context, so there is no shipped fix to close against.

## Best Possible Solution

Keep the issue open. The best fix is a generic, non-blocking queued-followup feedback seam on the active-run `enqueue-followup` path, likely an `onQueued` or equivalent typed callback in `GetReplyOptions` and the shared reply dispatcher pipeline. It should fire before typing cleanup, swallow/log callback failures so enqueue reliability is preserved, wire Feishu/Lark to an immediate reaction or receipt, avoid double-signaling in Telegram/Slack/Discord paths that already set initial status reactions, and add focused tests proving the callback fires only for busy queued followups.
