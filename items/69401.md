## Summary

Keep #69401 open. Current main has useful partial defenses: channel metadata/history are marked as untrusted, `wrapExternalContent` has nonce-style external-content boundaries, and Discord has a plugin-local untrusted-context copy of guild message bodies. But the shared inbound channel path still sends the current message body as ordinary user prompt text, and normal channel transcripts/dreaming export still treat external channel user turns as plain `User:` content rather than carrying channel-trust provenance. The issue comment also adds relevant design concerns around tag-boundary breakouts and memory persistence, so this is not conservative cleanup material.

## Best Possible Solution

Implement this as a core/shared inbound safety feature, not as a cleanup close. The likely path is to wrap the current inbound body after command/directive parsing but before model prompt assembly, preserve clean `RawBody`/`CommandBody` for command handling, use the existing nonce-based `wrapExternalContent` pattern or a channel-message-specific equivalent, avoid duplicating raw and wrapped body content, and persist `InputProvenance` such as `kind: "external_user"` plus `sourceChannel` for normal channel turns so memory/dreaming can filter or downgrade untrusted channel-derived material. Add regression tests for at least Slack/Telegram-style inbound bodies, tag-breakout text, transcript replay, and session-memory export.
