## Summary

Keep #69401 open. Current main has useful adjacent protections: structured untrusted metadata blocks, a nonce-based external-content wrapper, special-token sanitization, and a partial Discord guild-context wrapper. The central request is still not implemented: ordinary inbound channel message bodies for Slack, Telegram, WhatsApp, Google Chat, and other channels still flow into `BodyForAgent` as plain text, and the memory/session ingestion path strips channel metadata before retaining plain user text without ordinary channel provenance.

## Best Possible Solution

Keep the issue open and implement a shared inbound-channel body trust boundary in the core reply pipeline or finalized channel context contract. Reuse `wrapExternalContent` or an equivalent nonce-safe wrapper for the agent-facing message body, keep raw/command fields stable for command detection and display, and carry structured channel/sender/timestamp provenance into memory/dreaming retrieval so untrusted inbound content can be filtered or down-weighted later. Add cross-channel regression tests for Slack, Telegram, WhatsApp, Discord, and at least one memory/session ingestion path.

## Risks / Open Questions

- A fix is security-sensitive and should preserve command parsing, transcript/UI rendering, and channel-specific reply behavior while changing only the agent-facing body boundary.
