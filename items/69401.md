## Summary

Keep #69401 open. Current main has a reusable nonced external-content wrapper and some untrusted metadata handling, plus a Discord-only guild message body wrapper, but the shared inbound reply path still appends ordinary channel message bodies as raw user text. Slack and Telegram are clear examples: they pass raw inbound text into `BodyForAgent`, while only channel metadata goes through `UntrustedContext`. The memory ingestion path also strips inbound metadata before normalization, so channel/source attribution is not preserved through dreaming-style session ingestion.

## Best Possible Solution

Keep the issue open and implement a core/shared inbound-content boundary in the reply prompt assembly or `finalizeInboundContext` contract: wrap model-facing external channel bodies with `wrapExternalContent` or an equivalent nonced marker format, escape/sanitize marker spoofing, keep clean command bodies for command detection, and carry `external_user`/channel/source provenance into transcript and memory ingestion so Dreaming/retrieval can distinguish untrusted channel-originated content. Add regression tests for at least Slack and Telegram plus a shared contract test so plugins do not each have to invent wrappers.

## Risks / Open Questions

- The fix should preserve command parsing and routing behavior: `RawBody`/`CommandBody` likely still need clean text, while the model-facing `BodyForAgent`/prompt body should get structural boundaries.
- There is already partial Discord behavior, so a generic fix should avoid double-wrapping Discord guild message bodies or should retire the Discord-specific wrapper cleanly.
- Docs currently say external-content wrapping applies to inbound channel content, but the source inspection shows several major channel paths still pass raw body text; implementation should align code and docs.
