## Summary

Keep open. Current main still has a plausible path for the exact #56818 failure: the Feishu card-action parser prefers the top-level `open_message_id` over `context.open_message_id`, so a callback payload containing top-level `card-action-c-*` plus a real context message id can still route streaming replies through Feishu `message.reply` with the temporary callback id.

## Best Possible Solution

Keep #56818 open and fix it in the Feishu plugin: preserve/prefer the real `context.open_message_id` over the temporary top-level card-action callback id, fall back to plain send when only a callback id is available, and add a lifecycle regression test where top-level `open_message_id` is `card-action-c-*` and `context.open_message_id` is the original `om_*` target.

## Evidence

- **Issue-reported payload/error shape:** The issue body and first comment report that Feishu card action callbacks can carry a temporary top-level `open_message_id` like `card-action-c-xxx`, while the original message id is available in `context.open_message_id`; Feishu rejects the temporary id when used for streaming replies.
  - command: `provided GitHub context for #56818`

The returned event then writes that selected value back into both `open_message_id` and `context.open_message_id`, losing the original context id.
