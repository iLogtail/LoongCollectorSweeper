## Summary

Keep open. Current main has improved Twitch setup/status/docs, but the reported runtime symptom still appears present: Twitch can log a connection, return from `startAccount`, get marked stopped by the channel manager, and enter the auto-restart loop.

## Evidence

- **Twitch startup awaits a short-lived monitor call:** `startAccount` sets the account running and then awaits `monitorTwitchProvider`; there is no long-lived await or blocking lifecycle after monitor setup.
  - file: [extensions/twitch/src/plugin.ts:169](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/extensions/twitch/src/plugin.ts#L169)
  - command: `nl -ba extensions/twitch/src/plugin.ts | sed -n '169,190p'`
  - sha: [b64bfc5d9abf](https://github.com/openclaw/openclaw/commit/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab)
- **Monitor returns immediately after connecting and registering handlers:** `monitorTwitchProvider` obtains the client, registers the message handler and abort listener, then returns `{ stop }` immediately. That makes the gateway `startAccount` promise settle even though the Twitch client is meant to stay alive.
  - file: [extensions/twitch/src/monitor.ts:195](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/extensions/twitch/src/monitor.ts#L195)

- **Channel manager treats a settled start task as channel exit and restarts:** When the `startAccount` task settles, the manager marks `running: false`, increments restart attempts, and logs `auto-restart attempt ...`, matching the issue's observed post-connect restart loop.
