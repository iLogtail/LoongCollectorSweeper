## Summary

Keep open. Current main has fixed the Telegram reasoning/commentary lane issues, but the later repro in this issue is a different path: a successful same-target message-tool send, including file/media sends, can still suppress the following final reply even when the text is distinct. Maintainer discussion explicitly called this out as still valid, and current code/tests still encode that behavior.

## Best Possible Solution

Narrow same-target message-tool suppression in buildReplyPayloads/reply-payloads-dedupe so media/file sends only suppress actual duplicate media or matching captions/text. Preserve a distinct final text payload after a successful same-target filePath/media send, and add regression tests for Telegram same-target file/media send followed by different [thinking,text] final output.
