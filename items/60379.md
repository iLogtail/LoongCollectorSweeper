## Summary

Keep #60379 open. Current main has an internal per-run provider/model override path, but the requested user-facing per-message chat/slash-command override is not implemented: `/model` still persists the session model, Control UI model pickers persist through `sessions.patch`, and `chat.send` has no provider/model fields.

## Best Possible Solution

Keep the issue open and implement a user-facing one-turn model override that reuses the existing per-run `provider`/`model` plumbing without writing `providerOverride`/`modelOverride` to the session entry. A conservative path is to add a parsed `/model <provider/model> --once <message>` or `/ask <provider/model> <message>` directive for authorized senders, extend the appropriate chat/slash-command contract and UI/TUI/channel docs, and add regression tests proving the selected model is used for that turn while the session default remains unchanged.

## Risks / Open Questions

- Adding this feature needs careful authorization and allowlist handling because the internal per-run override path is gated by `allowModelOverride`.
- The UX should define whether the one-shot model applies to followups queued during an active run, `/btw`, attachments/vision support checks, and delivery from channel surfaces such as Telegram.
