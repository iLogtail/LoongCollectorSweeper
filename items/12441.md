## Summary

Keep #12441 open. Current main only supports Control UI shared-secret auth from UI-provided state (`#token=`, settings/sessionStorage, or password) and from `connect.params.auth` on the WebSocket handshake. Bearer auth is accepted for HTTP Control UI read endpoints after the UI already has a credential, and Tailscale/trusted-proxy identity modes are separate paths, but a reverse proxy injecting `Authorization: Bearer ...` or `X-OpenClaw-Token` is still not mapped into shared-secret Control UI WebSocket auth or a short-lived browser session/bootstrap credential.

## Best Possible Solution

Keep #12441 open as a core Gateway/Control UI auth design issue. The best fix is to validate `Authorization: Bearer ...` and/or `X-OpenClaw-Token` on the Control UI WebSocket upgrade and feed it into the existing shared-secret auth path, or validate the header on a same-origin HTTP bootstrap request and issue a short-lived UI session/bootstrap credential for the browser WebSocket. Add focused WS/server and UI bootstrap tests, then update the public Control UI/dashboard docs.

## Risks / Open Questions

- Closing this would leave shared-secret reverse-proxy deployments without the requested header-to-WebSocket auth path; users still need URL fragment/sessionStorage/manual auth or a separate Tailscale/trusted-proxy identity mode.
- A fix needs careful origin, proxy-trust, rate-limit, and credential-leakage design if it issues a browser-readable bootstrap token or accepts proxy-injected headers on WebSocket upgrade.
