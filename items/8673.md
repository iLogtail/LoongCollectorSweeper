## Summary

Keep #8673 open. Current main still performs the built-in OAuth refresh as a single `getOAuthApiKey(...)` call without bounded transient retry; the manager adds locking/timeout and a `refresh_token_reused` recovery path, but not retries for network, timeout, 429, or 5xx refresh failures. The earlier fix PRs were closed unmerged, and the latest issue discussion already has maintainer-facing review evidence that this remains relevant.

## Best Possible Solution

Keep #8673 open and implement a bounded, tested retry policy around the actual OAuth refresh operation on the current auth path. Prefer reusing `src/infra/retry.ts`, retry only classified transient refresh failures, preserve immediate handling for permanent auth errors, keep the `refresh_token_reused` adoption path intact, and add focused tests under `src/agents/auth-profiles/` for success after a transient failure, retry exhaustion preserving the final error, and non-retryable OAuth failures.

## Risks / Open Questions

- A future fix should classify retryable transient network, timeout, rate-limit, and 5xx-style failures explicitly; retrying permanent OAuth failures such as `invalid_grant` would add latency without recovery.
- The existing `refresh_token_reused` adoption path and main-agent credential fallback solve a different token-rotation race and should be preserved when adding transient retry.
