## Summary

Keep open. Current main still enforces a finite CLI no-output watchdog in both the normal CLI runner and the Claude live-session path, with timers reset only by CLI stdout/stderr. I found no pending-Monitor/tool-wait exemption or heartbeat path, and the related #40982/#41022 cap work is broader/partial rather than a complete fix for this Monitor-specific expected-idle case.

## Best Possible Solution

Keep the issue open and fix the CLI backend watchdog behavior directly. The maintainer path is to make the normal CLI runner and Claude live-session watchdog aware of expected idle while a long-running Monitor/tool wait is pending, or emit a narrowly scoped heartbeat from that state, while preserving timeout protection for real stuck/interactive CLI stalls. Add focused regression coverage that simulates a Claude CLI turn entering a pending Monitor/tool wait with no stdout for longer than the watchdog and verifies the session is not killed, plus a separate test that a truly silent stalled CLI still times out. Coordinate with #40982/#41022 for default cap tuning, but do not treat those as a complete replacement for the Monitor-aware fix.
