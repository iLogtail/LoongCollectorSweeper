## Summary

Current main already implements the timestamp-ordering fix requested by PR #70293. The shipped direction is broader than this PR's narrow `createdAt = min(now, startedAt)` clamp: task records are normalized at create, update, and restore time, and regression tests assert that an earlier pre-captured `startedAt` no longer produces `inconsistent_timestamps` audit findings.

## Best Possible Solution

Close #70293 as implemented on current main. Keep the broader registry-level timestamp normalization from #71871, which covers create, update, and restore paths; any remaining task-audit timestamp issue should be filed as a fresh regression against current main with current audit output and task records.
