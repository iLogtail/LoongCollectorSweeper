## Summary

Keep open. Current main has partial cleanup paths, but the core reported risk still exists: session lock stale detection still defaults to 30 minutes, live-PID locks are only stale by age, and embedded runs can set watchdog maxHoldMs to the 2,147,000,000 ms ceiling for effectively no-timeout runs. The embedded attempt path now releases through cleanup finally blocks, but that does not close the broader process-alive leaked-lock case described in the issue discussion.

## Best Possible Solution

Keep #49157 open as a focused hardening bug. The likely maintainer path is to cap embedded session-lock maxHoldMs to a short bounded window, add regression coverage for a process-alive abandoned lock context, and then consider heartbeat/lease semantics or explicit nonfatal unhandled-rejection lock cleanup if the bounded watchdog is not sufficient. Coordinate with #13744 for configurability and lock-scope work, but do not close this as a duplicate unless a canonical issue explicitly covers the leaked live-PID lock behavior.
