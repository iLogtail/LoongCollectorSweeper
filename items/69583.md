## Summary

Keep open. Current main still runs multi-collection QMD search serially, so PR #69583 addresses a real unshipped performance gap. The PR is recent and focused, but it should get maintainer review/rework rather than auto-close because the proposed Promise.all fan-out introduces resource/backpressure questions raised in review.

## Best Possible Solution

Keep PR #69583 open and review it as the active implementation path. The best landing shape is a bounded parallel multi-collection QMD search helper that preserves current merge/dedup semantics, aborts sibling subprocesses on first failure before fallback, adds robust runCliCommand abort cleanup, and keeps regression coverage for parallel start spread, fallback call sets, and sibling abort behavior.

## Evidence

- **Current main still serializes per-collection qmd calls:** runQueryAcrossCollections iterates collectionNames with a for-of loop and awaits each runQmd call before starting the next collection, then merges/dedupes results afterward. This matches the PR's reported bottleneck and is not implemented on main.

## Risks / Open Questions

- The proposed implementation fans out one qmd subprocess per collection; maintainers should decide whether to add a per-request or global concurrency cap before landing.
- The abort support shown in the PR discussion should be reviewed for single-settle cleanup of timers/listeners and for safe fallback behavior across platforms.
