## Summary

Keep #56880 open. Current main still enforces one active reply run per session key, queues normal follow-ups while a run is active, and waits for an active run to end before starting another run-now path. The related PR #57239 directly targets this request, but it is open/unmerged in the provided GitHub context, so the behavior is not implemented on main.

## Best Possible Solution

Keep the issue open and evaluate PR #57239 or a successor as the implementation path. The best fix should add a deliberate fast-lane or concurrent-turn policy with tests covering active-session contention, queued follow-up draining, interrupt behavior, streaming, transcript coherence, and Discord/channel routing.
