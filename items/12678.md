## Summary

Keep #12678 open. Current main has useful narrower controls, including tool-name allow/deny profiles, per-agent/provider/sender policies, owner-only gateway filtering, HTTP operator scopes, plugin before_tool_call hooks, and skill/config gating. It still does not implement the requested manifest-declared capability permission model with default-deny grants/revocations before risky tools are exposed plus durable high-risk invocation audit semantics. The linked @vainplex/openclaw-governance plugin is useful call-time governance prior art, but the issue discussion itself notes it is not a replacement for the missing API-layer capability authorization model.

## Best Possible Solution

Keep #12678 open as a core/plugin-API security hardening item. The best path is to add a manifest-declared capability schema, enforce default-deny authorization before risky tools are exposed or invoked, provide clear grant/revoke UI and CLI flows, and record high-risk capability invocations in a durable audit log. Treat @vainplex/openclaw-governance as useful call-time governance prior art, not as a substitute for API-layer capability authorization.
