## Summary

Keep open. Current main now has useful adjacent Web Push infrastructure from merged PR #44590, including subscription management and a `push.web.test` RPC that broadcasts to registered browser push subscriptions. That does not fully implement the requested product surface: there is still no top-level `openclaw notify <message>` command, the existing web-push RPC is named and documented as a test path, it is write-scoped rather than admin-scoped, and the existing chat injection path remains session-targeted/transcript-based rather than a transient notification bubble to connected UI sessions.

## Best Possible Solution

Keep the issue open for maintainer product/security judgment. The likely implementation path is a first-class `openclaw notify <message>` CLI backed by an explicit admin-scoped Gateway RPC or event that shows a transient notification/toast across eligible UI clients without writing chat transcripts. If maintainers prefer Web Push as the supported transport, rename or wrap `push.web.test` as a non-test notify API, align its scope, document the shell-script workflow, and add protocol/CLI/UI tests.
