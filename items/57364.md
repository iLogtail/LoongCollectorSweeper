## Summary

Keep PR #57364 open. Current main already replaced the accepted-upload success path with an in-place FileInfoCard update, which overlaps the related #47268 cleanup, but it still does not clean up stale FileConsentCards on decline, expired/not-found upload, conversation mismatch, or upload failure. The PR’s idea remains useful, though it needs to be rebased/adapted from the old inline `monitor-handler.ts` flow to current `file-consent-invoke.ts`.

## Best Possible Solution

Keep the PR open, but ask for a rebase onto current main. The best path is to adapt the cleanup into `extensions/msteams/src/file-consent-invoke.ts`, preserve the existing success-path `updateActivity` FileInfoCard replacement unless product wants deletion, and add best-effort cleanup for decline, expired/not-found, conversation mismatch, and upload-failure paths using the stored `consentCardActivityId` with a safe fallback to `activity.replyToId` where appropriate.
