## Summary

Keep #51049 open. Current main has adjacent WhatsApp reliability work, including recent `append` upsert handling, reconnect watchdogs, runtime status, and ambient proxy support, but I did not find a merged source or docs change that resolves the specific Docker -> k3s/OpenShell nested-container inbound failure. The related OpenShell deployment PR #54959 is still open in the supplied context, and the other WhatsApp inbound reports are not exact canonical superseders for this k3s networking symptom.

## Best Possible Solution

Keep the issue open as a WhatsApp plugin plus Kubernetes/OpenShell deployment investigation. The maintainer path should be to reproduce or instrument the nested Docker -> k3s case, capture Baileys `connection.update` and `messages.upsert` behavior inside the pod, test the comment-suggested `hostNetwork` and plain Docker `--network host` isolation steps, and then either fix the WhatsApp transport/runtime diagnostics or land a focused deployment guide if the root cause is environment networking.
