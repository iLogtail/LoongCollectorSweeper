## Summary

Keep #72202 open. Current main appears intended to route no-Codex-profile OpenAI image requests through the Images API when an OpenAI API key resolves, but the report is recent, Windows-specific, and points at a possible gateway restart/config-env ordering problem that is not covered by the current tests or release evidence. Related #70703 added Codex OAuth image support and routing hardening, but it does not prove this no-OAuth Windows/config-env regression is fixed.

## Best Possible Solution

Keep the issue open and add a targeted regression around image_generate with agents.defaults.imageGenerationModel.primary = openai/gpt-image-2, OPENAI_API_KEY supplied through openclaw.json env/env.vars, and an auth store with no openai-codex profile, asserting the provider calls /v1/images/generations and never /responses. If that passes locally, investigate the Windows gateway restart/config-write path that may apply env too late or prune explicit models.providers.openai config; also clarify in docs/doctor that the supported direct-route override is models.providers.openai.apiKey, not apiKeyEnvVar.

## Evidence
