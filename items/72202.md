## Summary

Keep #72202 open. It is recent, concrete, and not superseded by #70703. Current main has the intended OpenAI image routing and config-env plumbing, but I did not find source/test/history evidence that the exact reported Windows case is already fixed: `OPENAI_API_KEY` supplied only via `openclaw.json` env, no `openai-codex` profile, and `openai/gpt-image-2` still taking the Codex Responses path.

## Best Possible Solution

Keep the issue open. Add a targeted regression around `image_generate` with `agents.defaults.imageGenerationModel.primary = "openai/gpt-image-2"`, `env.OPENAI_API_KEY` or `env.vars.OPENAI_API_KEY`, no `openai-codex` profile, and a Windows-compatible gateway/tool path. Assert direct OpenAI Images API routing and no Codex Responses call. If that passes, ask for sanitized config/auth-store shape and gateway logs; track provider-config persistence separately under the provider-config overwrite issue.
