## Summary

Keep open. Current main has useful partial behavior: non-blocking `chat.send`, `/stop`/`chat.abort`, `sessions.steer`, queue modes, `/btw` side questions, and non-blocking subagent spawns. But same-session normal agent turns are still serialized, and current docs explicitly say steering waits until the active assistant turn finishes its tool calls. That does not satisfy concurrent user turns during long-running tool execution.

## Best Possible Solution

Keep this open as a core runtime/gateway feature request. The likely product path is to preserve the existing queue, steer, `/stop`, `/btw`, and subagent behavior while adding a true same-session preemption path for long-running tool execution: accept the new user turn out-of-band, propagate cancellation or suspension into active tool/backend handles, make tool result delivery safe after cancellation, and add focused tests around blocking exec/tool calls, `interrupt`, `steer`, and followup ordering.
