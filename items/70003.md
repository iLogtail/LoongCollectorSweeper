## Summary

Keep open. Current main has useful adjacent support for `/stop`, `chat.abort`, queue modes, `sessions.steer`, and non-blocking `sessions_spawn`, but it still does not implement the core request: immediate concurrent same-session user turns while a long-running active turn/tool call is still executing. Same-session PI runs are serialized through a per-session lane, default inbound queue behavior is `collect`, and interrupt paths abort/wait rather than allowing a new turn to run concurrently.

## Best Possible Solution

Keep the issue open for explicit maintainer product/design work. The best path is to define a same-session preemption/concurrency contract: decide whether new user turns abort/suspend or run alongside active turns, make transcript ordering and tool-result ownership explicit, preserve current `/stop`, `sessions.steer`, and queue-mode behavior, and add targeted tests for foreground `exec`, non-streaming tool waits, queued follow-ups, and UI/API steering. If the desired product answer is interrupt-only rather than true concurrent turns, document that clearly and track any UI/default-queue changes separately.

## Evidence

- **Issue discussion asks for more than existing stop/steer:** #70003 asks for subsequent user messages to start immediately during long-running tool execution, with examples like correcting constraints or cancelling a misdirected multi-minute run. The existing related reports (#42993, #57208, #60635) cover Telegram/control-lane stops, queue steering, and cooperative aborts, but the new issue and follow-up comment explicitly ask for general interleaved turns while the assistant is busy.
- **Same-session agent runs are serialized:** `runEmbeddedPiAgent` resolves a per-session lane from the session key/session id and wraps each run in `enqueueSession(...)` before entering the global lane. That means a second run for the same session waits behind the active one rather than running concurrently.

## Risks / Open Questions

- Implementing true same-session concurrency without a transcript/turn transaction model could interleave tool results, user messages, abort metadata, and final replies incorrectly.
- The existing stop/steer/interrupt affordances are useful but can be mistaken for the broader requested capability; closing this would likely hide a real remaining product gap.
