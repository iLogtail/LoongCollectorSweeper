## Summary

Keep #71136 open. Current main partially implements a conservative context cap for `agents.defaults.contextTokens`, but several requested surfaces remain unresolved: some live paths still let an agent/default override win before model metadata, `agents.list[].contextTokens` is defined but not propagated through resolved agent config, and `maxTokens` clamping is inconsistent across providers. The provided related items are title-search overlaps and do not supersede this issue.

## Best Possible Solution

Keep the issue open and implement a shared effective token-limit resolver that computes the minimum of available agent default/per-agent config, provider model config (`contextTokens`/`contextWindow` and `maxTokens`), discovered runtime model metadata, and defaults. Wire it through run setup, auto-reply, status/session persistence, and transport stream params, and add focused tests for a high `agents.defaults.contextTokens`, a high `agents.list[].contextTokens`, and an explicit `maxTokens` above the selected model limit.

## Risks / Open Questions

- There is already a partial context-token implementation; a fix should preserve the deliberate provider-qualified discovery/cache behavior while applying the minimum rule uniformly.
- `maxTokens` semantics differ by provider, so the shared clamp should be introduced at the narrow transport or stream-parameter boundary with provider-specific regression tests.
