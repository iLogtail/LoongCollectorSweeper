## Summary

Keep #61572 open. Current main still has the behavior this PR is trying to harden: message-action normalization accepts legacy `to`/`channelId`, can infer the current target for target-taking actions, channel selection can fall back to the tool-context provider when `channel` is actually an id-like value, and Discord reads return the raw normalized payload without the proposed compact/default-limit/detail handling. The PR is not obsolete, but it should not be auto-closed or landed as-is because the current PR head has a boundary review concern around importing core `src/logger.js` from Discord extension production code.

## Best Possible Solution

Keep the PR open and ask for a focused revision: preserve the Discord read/message-targeting hardening, route logging through a plugin-local or public SDK/runtime seam instead of importing core `src/**`, verify or fix the `detail` read adapter and internal edit caller concerns, then update the normalization/channel-selection/Discord read tests and run the changed OpenClaw gates.
