## Summary

Keep open. Current main has adjacent string-content compatibility work, but it does not implement this PR's specific normalization of assistant tool-call messages from `content: null` to `content: ""` for OpenAI-compatible Chat Completions providers. The linked bug context remains plausible for custom/string-strict providers even though newer DeepSeek V4 replay handling covers a separate `reasoning_content` requirement.

## Best Possible Solution

Keep the PR open or land an equivalent narrow fix on main: normalize OpenAI-compatible Chat Completions assistant messages that have `tool_calls` and `content == null` to `content: ""`, then add targeted tests in the transport/wrapper path and ensure it does not affect Responses API payloads or assistant messages without tool calls.
