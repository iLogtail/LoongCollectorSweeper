## Summary

Keep #52201 open. Current main still does not support a `workspace` field on route bindings or an equivalent binding-level workspace override. The existing product boundary is still agent-scoped workspace plus binding-to-agent routing; the current workaround remains separate agents/workspaces bound to conversations, which is the config sprawl the issue is explicitly trying to avoid. The related PR attempts in the supplied context are closed/unmerged, and #61460 is also closed/unmerged and describes a different outbound-firewall feature, so there is no canonical shipped fix to close against.

## Best Possible Solution

Keep the issue open for a core config/routing implementation: add a validated binding-level workspace override (or an equivalent explicit per-binding context scope), thread it through route resolution and reply runtime, ensure outbound interaction persistence and rebind clearing do not retain stale overrides, and cover precedence plus multi-channel behavior with targeted tests. The existing multi-agent workaround should remain documented, but it does not solve the same-agent/per-conversation isolation request.
