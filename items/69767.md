## Summary

Keep open. Current main has partial hardening for direct fallback delivery, but the reported early-completion/missing-payload race is still plausible and not fully fixed: subagent completion can still proceed with `(no output)` after retry, and tests explicitly assert that a completion announce is delivered in that state. Related PR #68464 is open/unmerged and appears relevant, but it is not shipped behavior on main.

## Best Possible Solution

Keep #69767 open and link it to PR #68464 or its successor. The best code direction is to make completion-message cleanup fail closed when there is no usable child result: persist/read a user-safe completion payload, return `false` from `runSubagentAnnounceFlow` when the payload is still missing, retry through the existing lifecycle backoff, and only mark `completionAnnouncedAt` after a real payload, structured delivery payload, or intentional error fallback has been delivered.

## Risks / Open Questions

- Current main includes a useful direct fallback when the announce-agent returns no visible payload but `internalEvents` already contain real child output; the eventual fix should preserve that behavior while deferring genuinely missing child-output cases.
- A good fix needs targeted regression coverage for `expectsCompletionMessage` with no child reply, no descendant findings, and no non-silent frozen/fallback result, asserting the cleanup remains retryable instead of marking delivery complete.
