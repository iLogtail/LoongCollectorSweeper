## Summary

#54199 should stay open. Current main still exposes the Control UI chat attachment picker as image-only, ignores non-image selected/dropped files, serializes UI attachments as image attachments, and the Gateway attachment parser/test path still drops non-image payloads. The related fix PR #54248 was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep #54199 open and implement a real generic Control UI upload path if that is the intended product behavior: expand or remove the file picker accept filter, preserve filename/type/content for non-images, render non-image attachment chips instead of image thumbnails, add Gateway/workspace upload handling with size/MIME/security validation, cover `.txt`, `.md`, and `.pdf` in UI and Gateway tests, and document the supported file types and storage location. If maintainers decide chat attachments should remain image-only, then close only after updating docs/UI copy to make that product boundary explicit.
