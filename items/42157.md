## Summary

Current main still acknowledges Telegram webhook requests before durable persistence or processing completion, and no webhook replay queue exists on the Telegram path. The linked fix attempts were closed unmerged, so the reported restart-loss window remains actionable.

## Best Possible Solution

Keep #42157 open and implement the durable Telegram webhook queue in the bundled Telegram plugin: after secret/body validation, persist the raw update to a per-account durable queue before returning 200, dispatch through the normal bot middleware, dequeue only after all deferred work completes, and replay pending entries during webhook startup before accepting new requests. Add focused tests for enqueue-before-ACK, dequeue-after-completion/failure behavior, restart replay ordering, malformed/stale file cleanup, and duplicate/idempotency handling.

- Webhook mode still has an at-most-once window: a gateway crash or restart after HTTP 200 but before bot.handleUpdate and deferred work complete can drop the update permanently.
- The existing Telegram update-offset persistence is on the polling monitor path and does not replay webhook payloads after restart.
