## Summary

Keep #71299 open. Current main has Telegram API failure/success logging once `sendMessage` is actually invoked, and v2026.4.24 shipped the related polling offset/restart fix, but the issue was explicitly left open because the reported symptom may be a broader outbound-dispatch/no-attempt path. The related PR #71377 was closed unmerged, and I did not find a shipped change that conclusively closes that remaining gap.

## Best Possible Solution

Keep the issue open and ask for a retest on v2026.4.24 or current main. If a missing reply still has no `telegram sendMessage ok` or `telegram sendMessage failed` log, add targeted instrumentation around the dispatcher/no-attempt paths and a regression test that proves assistant replies reaching Web UI either call Telegram delivery or emit an operator-visible skip/failure reason.
