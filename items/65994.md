## Summary

Keep open. Current `main` still routes async exec completion events into the heartbeat path for webchat/control-ui, and that path is still configured to relay alert content to the user-facing chat surface by default. The shipped webchat history fix in `CHANGELOG.md` only covers the narrower synthetic transcript-repair leak from #65247, not this exec-event leak.

## Evidence

- **Exec completions still enqueue untrusted system events and wake `exec-event` heartbeats:** `exec.finished`/`exec.denied` still build `Exec ...` text, enqueue it as an untrusted system event, and immediately call `requestHeartbeatNow(... reason: "exec-event" ...)`. That is the same control path described in the issue.
