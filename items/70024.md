## Summary

Keep open. Current main still contains the exact timeout behavior reported in #70024: `stopChannel` records `running: true` on stop timeout and returns before deleting `store.tasks`/`store.aborts`, while later `startChannel` calls silently return when `store.tasks.has(id)`. The linked PR #70056 is useful and directly relevant, but it is still open/unmerged in the provided GitHub context, so the issue remains the canonical tracker for a real gateway lifecycle behavior that has not landed on main.

## Best Possible Solution

Keep #70024 open until PR #70056 or an equivalent gateway lifecycle fix lands. The best fix should explicitly define what a stop timeout means for channel ownership, avoid permanent stale `store.tasks` entries, preserve protection against duplicate live provider tasks, and update `server-channels.test.ts` plus health-monitor coverage to match the chosen contract.
