## Summary

Keep open. Current `main` still does not implement a Telegram inbound-to-Control-UI hook, but this PR no longer matches the current architecture: today’s live UI refresh path is transcript-driven `session.message`, while the PR discussion/patch shape relies on a custom chat event path that current protocol/UI contracts do not accept.

## Evidence

- **Current main has no Telegram inbound callback seam:** `MonitorTelegramOpts` and `TelegramBotOptions` have no `onInboundMessage`-style option, and Telegram gateway startup passes only the existing runtime/webhook/status fields. That means the PR’s proposed callback path is not present on `main`.
