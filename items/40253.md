## Summary

Current main already solves the PR's central problem: inbound Telegram user turns are persisted as normal session transcript messages and surfaced to WebChat/Control UI through the session transcript event stream, rather than through the PR's proposed new `chat` event state. The implementation is present in current main and in the v2026.4.24 release tag, and the open PR's own review discussion identified contract/UI risks in its `state: "inbound"` approach.

## Best Possible Solution

Close PR #40253 as implemented on main and keep the shipped session transcript event path as the canonical solution. Do not merge the PR's `onInboundMessage`/`chat state: "inbound"` approach; any future cross-channel live-display work should build on the documented `session.message`/`sessions.changed` session stream and keep protocol/UI handling aligned.
