## Summary

Keep #51572 open. Current main still does not fire the bundled `session-memory` hook on automatic idle/daily reset or before `session.maintenance.pruneAfter` maintenance removal. The reset half overlaps open trackers/PRs (#50891, #11960, #61675, #71792), but the maintenance-prune half is not shipped and is not fully covered by those reset-only items. The issue discussion also usefully points to KeepMyClaw as an external snapshot workaround, confirming the underlying state-loss concern rather than making the core request obsolete.

## Best Possible Solution

Keep #51572 open for now. Land one of the focused reset fixes for idle/daily rollover, then either narrow this issue to the remaining maintenance-prune memory boundary or close it only after a canonical prune-specific tracker exists. The best implementation would emit a documented lifecycle/internal hook or memory extraction path before maintenance removes/archives stale sessions, with regression tests for `pruneAfter` and `maxEntries` maintenance plus docs for the chosen hook/event semantics.

## Risks / Open Questions
