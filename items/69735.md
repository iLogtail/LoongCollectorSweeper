## Summary

Keep open. Current main still forwards native Ollama response `tool_calls[].function.arguments` unchanged in `buildAssistantMessage`, so the reported inbound string-argument mismatch remains unresolved. Existing fixes and tests normalize replay/outbound arguments, but they do not cover fresh provider-returned string arguments; the related #69751 PR is closed unmerged and #69910 appears to be the still-useful focused fix path.

## Best Possible Solution

Keep #69735 open until a minimal fix lands in `extensions/ollama/src/stream.ts` that applies the existing `ensureArgsObject()`/unsafe-integer-preserving parser to `buildAssistantMessage` tool-call arguments. Add regression coverage for provider-returned string JSON arguments, malformed string fallback, and existing object passthrough; #69910 is the closest useful implementation path, likely with a small rebase onto current main.
