## Summary

Keep #69735 open. Current main still has the reported native Ollama inbound bug: `buildAssistantMessage` copies `response.message.tool_calls[].function.arguments` directly into the assistant `toolCall`, so JSON-string arguments from Ollama can still reach tool execution unparsed. Existing normalization only covers replay/outbound paths, and the exact fix remains in open PR #69910 rather than shipped on main.

## Best Possible Solution

Keep #69735 open as the bug tracker. Land or rework PR #69910, or an equivalent narrow fix, by widening the native Ollama response tool-call argument type to accept string/unknown input, normalizing `toolCall.function.arguments` inside `buildAssistantMessage` with the existing unsafe-integer-preserving helper, and keeping regression tests for JSON-string args, object pass-through, malformed strings, unsafe integers, and streamed final-message delivery.
