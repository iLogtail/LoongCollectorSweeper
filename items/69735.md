## Summary

Keep #69735 open. Current main still forwards native Ollama response `tool_calls[].function.arguments` unchanged in `buildAssistantMessage`, so the reported inbound string-argument path remains unresolved. Existing normalization and tests cover replay/outbound conversion, not fresh provider-returned string arguments. The linked #69751 fix was closed unmerged, and #72102/#69910 are still the relevant implementation paths rather than proof that this issue is fixed.

## Best Possible Solution

Keep #69735 open until a minimal Ollama fix lands in `extensions/ollama/src/stream.ts` that applies `ensureArgsObject()` to `buildAssistantMessage` tool-call arguments. Add focused regression coverage for inbound provider-returned JSON-string arguments, object passthrough, malformed string fallback, and the streaming accumulation path; then close #69735 and whichever duplicate PR is not used.
