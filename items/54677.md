## Summary

Keep open. Current `main` still has the underlying docs gap this PR addresses: `docs/install/hetzner.md` shows a loopback-only `ports:` example but does not warn readers that adapting the repo-root default `docker-compose.yml` can preserve the base public bindings, while the default `docker-compose.yml` still publishes both gateway and bridge ports publicly.

## Evidence

- **Hetzner guide lacks the warning on main:** The Hetzner install guide shows a loopback-only mapping and then immediately continues to the `--allow-unconfigured` note. There is no warning about replacing, rather than merging, the base `ports:` block when starting from the repo default compose file.
  - file: [docs/install/hetzner.md:189](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/docs/install/hetzner.md#L189)
  - command: `nl -ba docs/install/hetzner.md | sed -n '150,210p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Repo default compose still exposes ports publicly:** The repo-root default compose file still publishes `${OPENCLAW_GATEWAY_PORT:-18789}:18789` and `${OPENCLAW_BRIDGE_PORT:-18790}:18790` with no loopback host prefix, so naively layering an override remains a plausible footgun.
  - file: [docker-compose.yml:24](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/docker-compose.yml#L24)
