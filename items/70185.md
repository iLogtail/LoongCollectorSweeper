## Summary

Keep PR #70185 open. Current main still does not register memory-core public artifacts from the lightweight CLI metadata entry, and the CLI-metadata loader still only wires `registerCli`, so the PR's core missing capability is not implemented on main. The related destructive-prune behavior appears mitigated separately by skipping bridge pruning when no memory capability is loaded, but that does not solve the reported 0-artifacts import path.

## Best Possible Solution

Keep the PR open, but require an updated validation against current main: either ensure the CLI wiki bridge import loads/registers the active memory plugin capability before syncing, or adapt the CLI metadata/public-artifacts registration path so `memory-core` artifacts are visible during `openclaw wiki bridge import`. Add a focused regression test for CLI bridge import seeing memory-core artifacts, while preserving the existing no-prune safety behavior when no memory capability is available.
