## Summary

Keep open. Current main still does not register memory-core publicArtifacts for CLI/wiki bridge flows: the lightweight memory-core CLI metadata only registers the `memory` command, the CLI-metadata loader only wires `registerCli`, and `registerMemoryCapability` defaults to a no-op without an explicit handler. Main has a separate no-prune guard that mitigates the destructive-removal part, but the 0-artifact import problem remains plausible and the PR is still useful, though it should be validated against the actual `wiki` CLI execution path.

## Best Possible Solution

Keep #70185 open and adapt or merge the narrow fix only after validating the real CLI bridge path. The best implementation should register the selected memory plugin's public artifact provider in CLI-safe loading with the same memory-slot gating and rollback semantics as full registration, and add focused coverage for `wiki bridge import` returning memory-core artifacts rather than 0 artifacts.
