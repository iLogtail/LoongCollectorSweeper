## Summary

Keep open. I did not find strong evidence that current `main` fixes or obsoletes this report. Telegram startup on `main` still performs the same `deleteWebhook` cleanup and command-menu API calls, and non-network failures still bubble out. There is also still a plausible token/account-selection path where OpenClaw can use a different implicit default token than the one a reporter manually tested, which matches the report’s symptom pattern.

## Evidence

- **Polling startup still fails on non-network `deleteWebhook` errors:** `TelegramPollingSession` still calls `bot.api.deleteWebhook({ drop_pending_updates: false })` during startup, and only retries errors classified as recoverable network errors. A Telegram 404 is not handled here as a special case or downgraded.
