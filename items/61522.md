## Summary

Keep PR #61522 open. Current main still has the config IO observability gap this PR tries to address: health-state reads/writes, audit-log appends, observe-recovery health persistence, and backup maintenance still swallow IO failures without logging. The PR is not merge-ready as-is because the provided review comments identify real implementation risks, but those are reasons to request changes, not to close the item as implemented, superseded, stale, or out of scope.

## Best Possible Solution

Keep the PR open for maintainer/contributor follow-up. The best product direction is a narrow, bootstrap-safe config IO observability fix: add low-noise logging for serious config health, audit, and backup IO failures without changing best-effort semantics; avoid subsystem logging paths that depend on config-driven logger initialization; classify disk-full only from `ENOSPC`/clear no-space messages; and add targeted tests for classification, log routing, and no recursive logging. If a newer PR such as #62810 becomes the maintained canonical implementation, close duplicates then with that explicit link.
