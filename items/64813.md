## Summary

Keep open. Current main does not implement the PR's direct `params.user` passthrough, and the unmerged PR is still a plausible starting point, but it needs changes before merge: the proposed wrapper injects `user` into every provider payload without the API-family guard used by nearby payload patches, and it has no regression tests. Main now has a documented `extra_body`/`extraBody` path for OpenAI-compatible Completions payloads, but that is only a partial workaround and does not fully close the PR's requested `user` behavior.

## Best Possible Solution

Keep the PR open or request changes. The best path is a guarded implementation that only injects `user` for provider/API families that support it, reuses a shared API-family helper like the `parallel_tool_calls` path, normalizes the configured value, and adds tests proving injection for supported OpenAI/OpenRouter-style payloads and no mutation for unsupported providers. If maintainers prefer the existing `extraBody` escape hatch instead, close this PR only after explicitly resolving or superseding #64812 with that product decision.
