## Summary

Current `main` has a targeted mitigation for the clobbered-config symptom: config reads now detect an `update.channel`-only root and can restore the last-known-good backup, with a test that matches this report's shape. But the macOS companion still defaults Sparkle auto-checks/downloads to `true`, so the auto-update part of the report is not clearly resolved, and I could not do a runtime macOS reproduction in this read-only environment. That is not strong enough evidence to conservatively close.

## Evidence

- **macOS companion still defaults auto-update on:** `makeUpdaterController()` reads `autoUpdateEnabled` and falls back to `true`, then wires both `automaticallyChecksForUpdates` and `automaticallyDownloadsUpdates` from that value.
  - file: [apps/macos/Sources/OpenClaw/MenuBar.swift:369](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/apps/macos/Sources/OpenClaw/MenuBar.swift#L369)
- **UI toggle also defaults to true:** The About screen stores `autoUpdateEnabled` with `@AppStorage(...)=true` and syncs both Sparkle auto-check and auto-download from that toggle on appear/change.
  - file: [apps/macos/Sources/OpenClaw/AboutSettings.swift:6](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/apps/macos/Sources/OpenClaw/AboutSettings.swift#L6)
- **Core now treats update-only config roots as recoverable clobbers:** Config observe recovery marks `update-channel-only-root` as suspicious/recoverable and restores the last-known-good backup by copying it back over the live config path.
  - file: [src/config/io.observe-recovery.ts:431](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/config/io.observe-recovery.ts#L431)
- **Exact regression shape is covered by a test:** There is a test named `auto-restores suspicious update-channel-only roots from backup` that seeds a full config, replaces it with `{ update: { channel: "beta" } }`, and expects recovery from backup.
  - file: [src/config/io.observe-recovery.test.ts:162](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/config/io.observe-recovery.test.ts#L162)
- **Recovery is on the normal config read path:** Both sync and async config reads call `maybeRecoverSuspiciousConfigRead*` before resolving/validating the config snapshot.
  - file: [src/config/io.ts:1209](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/config/io.ts#L1209)
- **Latest shipped macOS app feed notes a related fix:** `appcast.xml` for `2026.4.22` includes: `Config/gateway: restore last-known-good config on critical clobber signatures...` which is adjacent to this symptom, but it does not prove the auto-update behavior is fixed.
  - file: [appcast.xml:95](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/appcast.xml#L95)

## Risks / Open Questions

- No macOS/Sparkle runtime reproduction was possible here without writing artifacts or running the app, so the review is based on current code and release metadata only.
- The checkout is shallow and `gh` was unavailable without `GH_TOKEN`, so I could not trace the exact fixing commit beyond current tree evidence and the shipped `appcast.xml` note.
- The report combines two behaviors: config loss and automatic beta updating. Current `main` clearly mitigates the former, but the latter is not clearly implemented away because the companion still defaults auto-update on.
