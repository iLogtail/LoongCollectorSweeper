## Summary

Keep open. Current main still matches the reported Feishu silent-failure path: Feishu inbound media uses a 30 MB default cap, oversized buffers throw in the shared media store, `resolveFeishuMediaList` catches those failures and only logs them, and `handleFeishuMessage` continues to normal agent dispatch with an empty media payload and no Feishu user-facing warning. The provided related items are adjacent media bugs, not a canonical replacement for this oversized-inbound-media notification/blocking issue.

## Best Possible Solution

Keep the issue open and fix it in the bundled Feishu plugin. The best path is to make Feishu media resolution return structured success/failure state or throw a typed recoverable media error, send a chat-visible Feishu warning that includes the configured `mediaMaxMb` limit, and avoid normal agent dispatch when the triggering input was dropped media. Add focused regression tests for oversized single media and embedded post media failures asserting both user notification and no unqualified agent dispatch.
