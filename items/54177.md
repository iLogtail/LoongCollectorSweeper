## Summary

Keep #54177 open. Current main does not provide a request-header pass-through from the OpenAI-compatible HTTP endpoint into agent/skill execution. The endpoint accepts a few OpenClaw control headers for model, agent, session, and message-channel routing, and skills can receive statically configured env/apiKey values, but there is no request-scoped `custom-token`/arbitrary header context for a Python skill script to read. Related #16387 is adjacent but tracks outbound provider header templating, not this ingress-to-skill header propagation request.

## Best Possible Solution

Keep the issue open for maintainer product/design judgment. A robust fix would define an explicit, security-reviewed request-scoped metadata path: configure which inbound headers may be captured, redact/log them as sensitive, carry them through the gateway request context into the agent/tool/skill runtime, and expose them to skills through a documented env or context API. For static auth needs, docs can point users to `skills.entries.<skill>.env` or `skills.entries.<skill>.apiKey`, but that does not replace dynamic per-request header propagation.
