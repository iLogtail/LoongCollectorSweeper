## Summary

Keep open. Current main partially mitigates the original report by tagging session-corpus lines with an agent id and using an isolated cron target, but it still intentionally groups agents by shared workspace and ingests every agent attached to that workspace into the same dreaming corpus. That matches the maintainer comment narrowing #65374 to a real shared-workspace isolation bug rather than an obsolete report.

## Best Possible Solution

Keep #65374 open as a narrowed core memory-core bug. Maintainers should decide whether dreaming is supposed to be agent-isolated even when agents share a workspace; if yes, change the dreaming workspace/session ingestion model so each dreaming run filters to the current agent or stores per-agent corpora, and add a regression test with two active agents sharing one workspace proving one agent's transcript does not appear in the other's session corpus, light/REM candidates, or dream diary narrative. If workspace-level pooling is intended, document that explicitly and make the UI/output provenance obvious enough that first-person dream prose cannot silently impersonate another agent's experience.
