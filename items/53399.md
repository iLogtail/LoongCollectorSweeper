## Summary

Keep open. Current main has partial Chrome MCP mitigations, including a 30s attach handshake guard, tool-call timeout support, and docs for overriding the default `npx` launch. It does not fully close #53399: the default existing-session path still spawns `npx -y chrome-devtools-mcp@latest`, and the `/tabs` reachability path still ignores its 300ms timeout for Chrome MCP by calling `listChromeMcpTabs` without timeout or abort options. There is also no regression test proving a hung `client.connect`/`npx` spawn is bounded and cleaned up without orphaned subprocesses.

## Best Possible Solution

Keep the issue open and make the Chrome MCP attach/list path explicitly bounded: propagate route/client AbortSignal and timeoutMs into `listChromeMcpTabs`/`callTool`, make failed pending sessions close the spawned transport reliably, add a regression for hung `client.connect` or hung `list_pages` that verifies no request exceeds the browser client timeout and no subprocess is left behind, and document Windows Defender-specific guidance separately if confirmed for Windows users.
