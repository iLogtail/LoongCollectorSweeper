## Summary

Keep #71988 open. Current main still lacks the fallback `inbound_claim` broadcast this PR is trying to add, and the linked #48434/#71644 context makes this the active focused implementation path rather than obsolete work. The PR is not ready as-is because the proposed terminal broadcast path needs synthetic-reply handling and inbound-dedupe cleanup coverage before merge.

## Best Possible Solution

Keep #71988 open and request a narrow follow-up patch: add the fallback broadcast in `src/auto-reply/reply/dispatch-from-config.ts`, mirror the targeted plugin-bound handled path by delivering `broadcastResult.reply`, explicitly clean up inbound dedupe before any new terminal return, and add regression tests for missing-plugin/no-handler broadcast, handled-with-reply, and dedupe behavior. This belongs in core dispatch because external plugins already have the hook surface but cannot make core call the broadcast runner on fallback.
