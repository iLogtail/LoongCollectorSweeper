## Summary

Keep #35119 open. Current main already handles the original structured HTTP 500 / JSON `server_error` path, but the later April 20 reproduction is a different raw OpenAI Codex message (`The server had an error processing your request...`). That phrase is still absent from the classifier, and assistant-stage fallback still needs a failover signal, classified reason, or timeout before it throws into the model fallback chain. The generic unrecognized-error fallback path only protects thrown errors with remaining candidates, not assistant error payloads returned as a run result.

## Best Possible Solution

Keep the issue open. Add a focused regression for an OpenAI Codex assistant turn with `stopReason: "error"`, the raw text `The server had an error processing your request...`, and configured model fallbacks. Then either classify that wording as a transient server failure or adjust the assistant-result fallback path so unclassified provider error stops with remaining candidates reach model fallback without broadly treating unrelated generic text as retryable.
