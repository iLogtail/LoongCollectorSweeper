## Summary

Keep #35119 open. Current main fixes the original direct HTTP 500/status path and the JSON `server_error` Codex path, but the latest concrete reproduction in the issue is a raw OpenAI Codex message variant (`The server had an error processing your request...`) that still is not classified as failover-worthy. Assistant-side fallback still depends on classification or timeout, so this remaining path can still miss the model fallback chain.

## Best Possible Solution

Add a focused regression for the April 20 raw OpenAI Codex message (`The server had an error processing your request...`) with `stopReason: "error"` and configured model fallbacks, then either add that wording to the server-error classifier or normalize assistant-side provider server failures into the existing failover path without replaying side-effectful turns.
