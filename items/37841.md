## Summary

Current main still appears to have the reported same-target media-only suppression bug. The production reply path passes message-tool sent text/media/target metadata into `buildReplyPayloads`; that function still computes same-target suppression from target metadata and drops all final payloads when the target matches, without requiring that the message tool actually sent text. Media-only sends can populate target/media state while leaving `messagingToolSentTexts` empty, and the proposed fix PR #37900 was closed unmerged.

## Best Possible Solution

Keep #37841 open and implement the narrow reply-payload fix: same-target reply suppression should only drop final/follow-up session text when the message tool actually sent text to that same origin target. Media URL dedupe should continue stripping duplicate media. Add focused coverage for same-target media-only sends preserving final session text, and check the follow-up delivery path because it uses the same target-based suppression helper.
