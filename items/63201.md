## Summary

Current `main` already routes dispatcher-backed multipart requests through a runtime FormData normalization layer, which fixes the reported undici/global FormData mismatch for audio transcription. The fix is present in release `v2026.4.23` and is covered by a targeted regression test.

## Evidence

- **audio transcription still uses FormData through the guarded request path:** `transcribeOpenAiCompatibleAudio` builds a `FormData` body and sends it via `postTranscriptionRequest`, so the reported failure mode would only persist if the guarded fetch path still mishandled `FormData`.
  - file: [src/media-understanding/openai-compatible-audio.ts:42](https://github.com/openclaw/openclaw/blob/7bd74758c5f54c546b52708359a25c08cae74ad7/src/media-understanding/openai-compatible-audio.ts#L42)

  - sha: [7bd74758c5f5](https://github.com/openclaw/openclaw/commit/7bd74758c5f54c546b52708359a25c08cae74ad7)
- **guarded fetch now normalizes global FormData into undici runtime FormData:** `fetchWithRuntimeDispatcher` detects FormData-like bodies, rebuilds them with the runtime `undici` FormData constructor, and clears `content-length`/`content-type` so the runtime fetch can generate the correct multipart boundary.
  - file: [src/infra/net/runtime-fetch.ts:21](https://github.com/openclaw/openclaw/blob/7bd74758c5f54c546b52708359a25c08cae74ad7/src/infra/net/runtime-fetch.ts#L21)

- **dispatcher-backed requests use the normalized runtime fetch path:** When SSRF/DNS pinning attaches a dispatcher, `fetchWithSsrFGuard` routes the request through `fetchWithRuntimeDispatcher`, so audio transcription requests now benefit from that normalization automatically.
