## Summary

Keep open. Current main has useful partial protections: sub-agent child runs use a gateway agent idempotency key, gateway agent calls dedupe by idempotencyKey, and the embedded runner avoids some automatic retries after recorded mutating side effects. But I did not find a stable idempotency/recovery identity for the `sessions_spawn` tool call itself or a session-scoped dedupe/last-N guard that would prevent a parent timeout retry from spawning a second sub-agent or repeating an ambiguous external POST. The linked ClawHub skill is a useful workaround, but the remaining gap is in core sub-agent recovery semantics.

## Evidence

- **sessions_spawn creates fresh child identity per call:** Native sub-agent spawn generates a new random `childIdem` and passes it as the gateway `agent` idempotency key for that child run. This protects the child agent start call, but it is not a caller-supplied/reused recovery key for repeated `sessions_spawn` attempts with the same task.
