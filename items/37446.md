## Summary

Current main still has the core `sessions_spawn` idempotent recovery gap. Gateway `agent` calls and plugin runtime subagent calls can dedupe when the same idempotency key is reused, but the public `sessions_spawn` path has no caller-supplied or deterministic recovery key and native/ACP spawns still generate fresh child session and gateway idempotency identities per call. The linked ClawHub multi-agent deployment skill is useful operational guidance, but it does not replace the missing core recovery contract.

## Best Possible Solution

Keep this issue open until core sub-agent orchestration has an idempotent recovery contract. The likely path is to add a stable caller-supplied or deterministic `sessions_spawn` recovery/idempotency key, persist the key-to-child-run mapping, return or reuse the existing child session/run/task on replay, and add regression coverage for lost-ack or parent-timeout retries. The ClawHub skill can remain recommended operational guidance, but it should not replace the missing core `sessions_spawn` recovery API.
