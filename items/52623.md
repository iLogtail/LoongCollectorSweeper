## Summary

Keep open. Current `main` still uses the old Feishu WebSocket monitor that starts the SDK client and then waits only for abort, so the requested supervisor/recovery behavior is not implemented on `main`. This PR also still has credible unresolved review findings around reconnect backoff and reconnect-cycle state cleanup, so it is useful but not safe to auto-close.

## Evidence

- **Current main still uses park-on-abort WebSocket monitor:** `monitorWebSocket()` creates one `WSClient`, calls `void wsClient.start({ eventDispatcher })`, logs startup, and then only resolves on abort. There is no supervisor loop, health check, or client recreation path in current `main`. That means the PR's requested behavior is not already implemented.
