## Summary

Keep open. Current main has useful restart-drain mitigations, including SIGUSR1 restart deferral and service-manager restart intent handling, but it does not implement the durable SQLite inbound message store/requeue path requested here. Slack inbound events still flow through in-memory prepare/dispatch, and a plain SIGTERM is still treated as a stop unless a restart intent is present, so the broader SIGTERM-resilience request remains real.

## Best Possible Solution

Keep this issue open and implement a durable inbound-message persistence/replay path for Slack or a generic channel-inbound queue. The fix should persist the accepted inbound event before agent dispatch, mark completion only after terminal delivery/failure, requeue orphaned processing rows at startup, and add regression coverage for Slack Socket Mode event accepted followed by restart/SIGTERM during a tool-running turn. It should reuse the existing restart deferral where possible rather than replacing it.
