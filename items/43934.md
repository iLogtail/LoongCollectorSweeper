## Summary

Current `main` already implements this behavior in the MS Teams handler: it builds the stored conversation reference up front and persists it inside the `access.decision === "pairing"` DM path before returning. Current tests also assert that the pairing path both creates the pairing request and stores the conversation reference, which is the mechanism `user:`/notify-style proactive sends later read.

## Evidence

- **Pairing branch now persists the conversation reference:** `createMSTeamsMessageHandler` builds `conversationRef` once, then in the DM `pairing` branch calls `conversationStore.upsert(conversationId, conversationRef)` before continuing to the drop/return path. This directly covers the PR’s claimed gap.
