## Summary

Keep #70296 open. Current main still has actionable evidence for at least two of the three reported problems: restart-sentinel notices are still routed to channel delivery with no suppress config surface, and BOOT.md runs still append a hard-coded `NO_REPLY` footer after the workspace BOOT.md content. The Control UI model readback path appears improved on main, but the compound issue should not be closed while the sentinel and BOOT behaviors remain unresolved.

## Best Possible Solution

Keep the issue open or split it into focused follow-ups. The strongest product/code path is to add a documented restart-sentinel notification policy or suppress flag wired through config schema, formatter/delivery logic, and tests; then resolve the BOOT.md footer conflict by either documenting/rejecting noncanonical tokens clearly or changing the boot prompt so workspace BOOT.md does not fight the injected silent-reply instruction. Add a focused Control UI regression test if maintainers want to close the model-display portion separately.
