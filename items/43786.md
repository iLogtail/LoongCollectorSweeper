## Summary

Keep #43786 open. Current main recognizes `gateway.auth.mode="none"` at the auth layer, but the requested trusted internal network/Kubernetes deployment remains blocked by both runtime config validation and CLI startup preflight for non-loopback binds. The related #43820 attempted this exact change but is closed/unmerged in the provided GitHub context, and the later #69066 RFC is broader auth/service-identity design work rather than a shipped replacement for this feature request.

## Best Possible Solution

Keep #43786 open for maintainer product/security judgment. If accepted, implement a deliberate private-ingress no-auth path by updating both `src/gateway/server-runtime-config.ts` and `src/cli/gateway-cli/run.ts` so explicitly configured `gateway.auth.mode="none"` can start on intended non-loopback binds, add focused LAN/container/custom-bind tests, preserve loud startup warnings and security-audit coverage, and reconcile the docs that currently restrict none auth to loopback with the API docs that mention private-ingress none. If maintainers reject the trust model, close manually with that rationale rather than treating it as already implemented.
