## Summary

Keep open. Current main still has the broad whole-file heuristic described by the report: `potential-exfiltration` fires when any `readFile*` text appears in a file that also contains `fetch`/`post`/`http.request`, and the audit still reports it as generic `plugins.code_safety` without source/sink confidence or suppression guidance.

## Best Possible Solution

Refine `src/security/skill-scanner.ts` so `potential-exfiltration` requires stronger source/sink evidence than same-file regex co-occurrence, avoids matching helper names like `readFileSegment` as file-read APIs, and surfaces source, sink, confidence, and suppression guidance in audit details/docs. Add a regression test based on local read-only parsing plus unrelated network use, while keeping a positive read-and-send exfiltration test.

## Evidence

- **Current heuristic is whole-file, not source-to-sink:** `potential-exfiltration` is defined as `/readFileSync|readFile/` plus a separate whole-source network-send context regex, so a helper name like `readFileSegment` can satisfy the file-read side when the same file has any matching network call elsewhere.
  - file: [src/security/skill-scanner.ts:176](https://github.com/openclaw/openclaw/blob/9a529ca78bfa0e257f662f0e70ab8c681caa6086/src/security/skill-scanner.ts#L176)
  - sha: [9a529ca78bfa](https://github.com/openclaw/openclaw/commit/9a529ca78bfa0e257f662f0e70ab8c681caa6086)
- **Finding points at first readFile-pattern line:** For source rules, `scanSource` reports the first line matching the read pattern as evidence; it does not report a paired sink line, confidence level, or dataflow relationship.
  - file: [src/security/skill-scanner.ts:280](https://github.com/openclaw/openclaw/blob/9a529ca78bfa0e257f662f0e70ab8c681caa6086/src/security/skill-scanner.ts#L280)
  - sha: [9a529ca78bfa](https://github.com/openclaw/openclaw/commit/9a529ca78bfa0e257f662f0e70ab8c681caa6086)
- **Deep audit still applies this scanner to installed plugins:** `collectDeepCodeSafetyFindings` runs plugin code-safety collection only for deep audits, and plugin warnings are emitted as `plugins.code_safety` when scanner warnings exist.
  - file: [src/security/audit-deep-code-safety.ts:11](https://github.com/openclaw/openclaw/blob/9a529ca78bfa0e257f662f0e70ab8c681caa6086/src/security/audit-deep-code-safety.ts#L11)
  - sha: [9a529ca78bfa](https://github.com/openclaw/openclaw/commit/9a529ca78bfa0e257f662f0e70ab8c681caa6086)
- **Plugin audit output remains generic:** The plugin collector formats warning findings as suspicious code patterns with remediation to review the flagged code; it does not distinguish file-read source, network sink, confidence, or suppressibility metadata requested by the issue.
  - file: [src/security/audit-extra.async.ts:808](https://github.com/openclaw/openclaw/blob/9a529ca78bfa0e257f662f0e70ab8c681caa6086/src/security/audit-extra.async.ts#L808)
