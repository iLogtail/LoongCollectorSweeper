## Summary

Keep open. Current main partially supports remote Ollama discovery when an explicit provider uses `models: []`, but it still does not implement the issue's documented/expected config shape: `models.providers.ollama` with `baseUrl` and no `models` array is still rejected by the core schema, and the Ollama docs still show a custom-base-url provider entry without `models` while also saying explicit config requires manual models. The related PRs #71515 and #71636 attempted the direct schema fix but were closed unmerged, so there is no canonical landed fix to close against.

## Best Possible Solution

Keep the issue open and narrow the fix to aligning schema, types, docs, and tests. The most direct product path is to make `models.providers.*.models` optional where callers can safely handle `undefined`, add a config-schema regression for `models.providers.ollama` with `baseUrl` and no `models`, and update the Ollama docs to clearly state whether remote autodiscovery is enabled by omitting `models` or by using `models: []`. If maintainers prefer `models: []` as the contract, the docs and schema/help should explicitly require it and the issue can close only after that behavior is documented.
