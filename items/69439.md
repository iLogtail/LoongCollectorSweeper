## Summary

Keep #69439 open. Current main now has meaningful partial coverage for the report: per-agent tool policy, per-agent exec defaults, and per-agent sandbox/workspace isolation are present and documented. However the requested per-agent command-level `tools.exec` deny policy and variant/embedded command blocking are still not implemented; `tools.exec` only accepts host/security/ask/node/safe-bin/style defaults, and the docs explicitly say there is no heuristic command-obfuscation prefilter in broad host-exec mode. The related ClawSweeper context also shows narrower per-agent/isolation gaps remain open rather than a canonical shipped replacement.

## Best Possible Solution

Keep the issue open, but narrow it for maintainer/product judgment: document the existing workaround using `agents.list[].tools.deny`, `agents.list[].tools.exec.security`, `tools.exec.strictInlineEval`, `tools.fs.workspaceOnly`, and `agents.list[].sandbox`; then decide whether OpenClaw should add a real per-agent exec command policy API or explicitly reject shell-text deny filters in favor of sandboxed allowlist/approval flows.

## Risks / Open Questions

- A command substring denylist for shell text is easy to bypass; the remaining design should prefer allowlist/approval/sandbox semantics over promising strong security from string matching.
- Current main already solves part of the request with different config keys, so any implementation should avoid adding a parallel policy surface unless it clearly closes the per-agent command-level gap.
