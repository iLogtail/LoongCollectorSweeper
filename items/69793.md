## Summary

Keep open. Current main still has the reported Telegram inbound mismatch: Telegram `photo` messages seed `bodyText` from an unconditional `<media:image>` placeholder, and the fallback for non-audio `allMedia` still emits `<media:image>` without inspecting `contentType`. The related PR #69893 is reported in the provided context as closed and unmerged, so it does not supersede the open issue.

## Best Possible Solution

Keep #69793 open and land a focused fix in the Telegram bundled plugin, ideally by adding a small content-type-aware media placeholder helper used by `resolveTelegramInboundBody` after media resolution. Add regression tests for media-only Telegram photos whose resolved `allMedia` is non-image or unknown, multiple real images, and mixed media. If maintainers want the broader cleanup noted in the discussion, extract the helper for other channel plugins in the same or a follow-up PR.
