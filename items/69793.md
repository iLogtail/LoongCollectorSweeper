## Summary

Keep open. Current main still derives Telegram inbound body placeholders from Telegram message shape rather than the resolved media contentType, so a photo-shaped message with a non-image or unknown saved payload can still produce BodyForAgent/bodyText as <media:image>. The related PR #69893 was closed unmerged in the provided context, and local history/search did not show a landed #69793 fix.

## Best Possible Solution

Keep #69793 open and implement the Telegram fix in the extension: add a small media-placeholder helper that derives image/video/audio/document/sticker wording from resolved allMedia contentType/kind, use it for raw body and empty-body fallback paths, avoid hardcoded image/jpeg for msg.photo, and add targeted resolveTelegramInboundBody regression tests for application/octet-stream/document, mixed media, and multiple-image cases. A later shared helper across other channels would be useful if maintainers want broader parity.
