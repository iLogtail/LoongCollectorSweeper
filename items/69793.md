## Summary

Keep open. Current main still builds Telegram inbound placeholders from Telegram message shape rather than the resolved media content type, so a photo-shaped message can still produce `<media:image>` even when the saved payload contentType is non-image or unknown. The related PR #69893 is closed/unmerged in the provided context, and current source/tests do not show a landed replacement fix.

## Best Possible Solution

Implement the Telegram inbound placeholder from resolved media metadata, preferably through a small shared helper that classifies `allMedia` by saved contentType and handles mixed media intentionally. Add focused tests for photo-shaped messages whose saved contentType is `application/octet-stream` or document-like, mixed media, and multiple images. Do not add the hardcoded `image/jpeg` photo MIME fallback that the issue discussion later retracted.
