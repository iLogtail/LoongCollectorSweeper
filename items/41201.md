## Summary

Keep open. Current main fixes much of the local workspace-avatar path, but it still intentionally does not render remote http(s) agent avatars in the Control UI. That conflicts with #41201’s central expectation and with docs that still describe `set-identity --avatar` as accepting http(s) URLs.

## Best Possible Solution

Keep #41201 open until maintainers decide the intended Control UI contract. The best path is either to restore remote avatar display through a safe same-origin proxy/cache with SSRF, MIME, size, auth, and CSP controls, or to document and validate that Control UI only renders workspace-local and `data:image/...` avatars while http(s) avatar values are accepted only for non-Control-UI surfaces. Local workspace avatar handling appears substantially implemented on current main, so the remaining actionable gap is remote-avatar policy and docs/UI consistency.
