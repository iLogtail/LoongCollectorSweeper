## Summary

Keep open. Current `main` still does not render remote `http(s)` agent avatars in Control UI, despite config/docs allowing them, and oversized local avatar files still resolve to no avatar without a user-facing signal. The referenced fix PR #41239 was closed unmerged for PR-queue policy, not because the behavior landed.

## Evidence

- **Config/docs still promise remote avatars:** Agent config docs explicitly allow `identity.avatar` to be a workspace-relative path, `http(s)` URL, or `data:` URI.
  - file: [docs/gateway/config-agents.md:951](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/docs/gateway/config-agents.md#L951)
  - command: `nl -ba docs/gateway/config-agents.md | sed -n '944,956p'`
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Gateway serves remote/data only in meta response, not the actual avatar route:** `/avatar/:agent?meta=1` returns the remote/data URL, but the real `/avatar/:agent` handler immediately 404s anything that is not `kind === "local"`. That means remote avatars still are not actually served from the Control UI avatar endpoint.
  - file: [src/gateway/control-ui.ts:564](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/gateway/control-ui.ts#L564)
