## Summary

Keep #55694 open. Current main has partial/optional mitigations, but the reported default behavior is not fully fixed: generic tool-loop detection is disabled unless configured, deterministic exec validation failures are still just ordinary tool errors, and the default thresholds would allow many repeats before a hard stop even when enabled. The issue discussion also reports cross-channel/cross-model impact, including Feishu/Kimi, Telegram/Qwen, and rate-limit loops, so this is not safely reducible to one plugin or provider.

## Best Possible Solution

Keep the issue open and fix it in the core agent/tool execution path: classify deterministic schema/parameter validation failures as non-retryable by default, add a low per-turn identical failure breaker for same tool + same params + same error, add a total tool-failure cap for a turn, and ensure channel streaming/status delivery coalesces or suppresses repeated pre-retry messages while still surfacing one clear final failure to the user.
