## Summary

Current `main` still preserves `HEARTBEAT_OK` when it appears mid-string. The stripping helper only removes leading/trailing tokens, the unit test explicitly locks that behavior in, and reply-normalization paths still rely on that helper in `message` mode. The same logic is present in the latest release `v2026.4.23`, so there is no strong basis to close this as already fixed or obsolete.

## Evidence

- **Current helper only strips edges:** `stripTokenAtEdges` checks `startsWith(token)` and an end-of-string regex, then returns `didStrip: false` when neither matches. A middle `HEARTBEAT_OK` therefore survives unchanged.
  - file: [src/auto-reply/heartbeat.ts:76](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/auto-reply/heartbeat.ts#L76)

  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Test suite encodes the reported bug:** `heartbeat.test.ts` contains `it("does not touch token in the middle", ...)` and expects `hello HEARTBEAT_OK there` to come back unchanged with `didStrip: false`. That matches the issue report, not a fix.
  - file: [src/auto-reply/heartbeat.test.ts:72](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/auto-reply/heartbeat.test.ts#L72)
