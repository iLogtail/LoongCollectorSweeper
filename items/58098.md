## Summary

Keep open. Current `main` still only strips `HEARTBEAT_OK` at the start or end of a reply; mid-string tokens are intentionally left untouched in code, tests, and docs, so the reported behavior is still present.

## Evidence

- **Edge-only stripping in current implementation:** `stripTokenAtEdges` only checks `startsWith(token)` and a token-at-end regex inside its loop. If the token appears only in the middle, `didStrip` stays false and `stripHeartbeatToken` returns the original text unchanged.
  - file: [src/auto-reply/heartbeat.ts:76](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/auto-reply/heartbeat.ts#L76)
  - command: `nl -ba src/auto-reply/heartbeat.ts | sed -n '70,190p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Current tests codify the reported behavior:** The heartbeat unit test explicitly expects mid-string `HEARTBEAT_OK` to remain untouched, returning `didStrip: false` and the original text.
  - file: [src/auto-reply/heartbeat.test.ts:72](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/auto-reply/heartbeat.test.ts#L72)
  - command: `nl -ba src/auto-reply/heartbeat.test.ts | sed -n '50,90p'`

- **Current docs describe middle tokens as unsupported:** The heartbeat docs state that `HEARTBEAT_OK` is treated specially only at the start or end of a reply, and that a middle occurrence is not treated specially.
  - file: [docs/gateway/heartbeat.md:82](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/docs/gateway/heartbeat.md#L82)
