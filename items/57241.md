## Summary

Keep open. Current `main` still appears to have both reported behaviors: gateway startup re-applies `tailscale serve` when `gateway.tailscale.mode = "serve"`, and `gateway.tailscale.mode = "funnel"` still hard-fails unless gateway auth mode is `password`. I did not find a fix or a regression test that preserves an existing manual Funnel config across gateway restarts.

## Evidence

- **Gateway startup still re-runs Tailscale exposure:** Post-attach startup always calls `startGatewayTailscaleExposure(...)` whenever Tailscale mode is not `off` (or cleanup is requested), so the Gateway reconfigures Tailscale on each start/restart.
  - file: [src/gateway/server-startup-post-attach.ts:471](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/gateway/server-startup-post-attach.ts#L471)
  - command: `nl -ba src/gateway/server-startup-post-attach.ts | sed -n '460,490p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Serve mode still issues a fresh `tailscale serve` command:** `startGatewayTailscaleExposure` dispatches `enableTailscaleServe(port)` for `serve` mode, and `enableTailscaleServe` runs `tailscale serve --bg --yes <port>`; the unit test asserts that exact command.
  - file: [src/infra/tailscale.ts:409](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/infra/tailscale.ts#L409)

  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Funnel mode still rejects token auth:** `resolveGatewayRuntimeConfig` throws `tailscale funnel requires gateway auth mode=password...` when `tailscaleMode === "funnel"` and the resolved auth mode is not `password`.
  - file: [src/gateway/server-runtime-config.ts:133](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/gateway/server-runtime-config.ts#L133)
  - command: `nl -ba src/gateway/server-runtime-config.ts | sed -n '128,140p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Current OpenClaw docs still describe Funnel as password-only:** The user-facing Tailscale docs on `main` still say `funnel` is public HTTPS and that OpenClaw requires a shared password, while `serve` is the tailnet-only mode.
  - file: [docs/gateway/tailscale.md:15](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/docs/gateway/tailscale.md#L15)
  - command: `nl -ba docs/gateway/tailscale.md | sed -n '9,18p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Upstream Tailscale semantics make the overwrite plausible:** Official Tailscale docs say the same port cannot be Serve and Funnel at the same time, and whichever command ran most recently determines whether that port is private or public; they also document `-bg` persistence across restarts. That matches the risk in OpenClaw re-running `tailscale serve` on gateway startup. 
  - command: `web search/open: Tailscale Serve/Funnel docs`
