## Summary

Keep #50220 open. Current main still sends an explicit `host=node` workdir into `system.run.prepare`, and the node-host approval planner still fails closed with the exact reported `approval requires an existing canonical cwd` message when its parent process cannot complete `lstat`/`stat`/`realpath` for the requested cwd. I found no current code, tests, docs, changelog entry, or related-item context that proves the macOS `/Users` launchd/FDA edge case is fixed.

## Best Possible Solution

Keep the issue open and fix it in the node-host approval planner. A good maintainer path is to reproduce under a macOS launchd node host, then make `resolveCanonicalApprovalCwdSync` Darwin-aware without weakening the symlink/TOCTOU checks: handle APFS `/Users` versus `/System/Volumes/Data/Users` canonicalization or move/duplicate cwd identity resolution into the execution context that can actually see the path. Add focused tests around the resolver, plus a macOS/manual regression note for `exec host=node workdir=/Users/<user>`.
