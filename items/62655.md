## Summary

Current main still appears to contain the reported OpenRouter model-id mismatch. The bundled OpenRouter provider registers the auto model as bare `auto`, while onboarding/docs and the pinned pi-ai catalog use `openrouter/auto`; the picker and Telegram callback path still carry raw catalog ids through to persisted session overrides. Related PR #62745 addressed this directly but was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep the issue open and land a scoped OpenRouter fix: change the bundled provider catalog auto id to `openrouter/auto`, add an OpenRouter provider-catalog regression test that prevents bare ids for bundled OpenRouter text models, and consider a defensive repair for existing persisted `openrouter`/`auto` session overrides plus a fallback-loop guard if the runtime loop is still reproducible.
