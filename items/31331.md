## Summary

Keep #31331 open. Current main still creates Docker sandbox and sandbox-browser bind mounts from gateway-visible workspace paths without remapping those paths to host Docker paths for Docker-outside-of-Docker. The docs now acknowledge a host-path parity constraint, but the Docker install/setup flow still mounts host storage at `/home/node/.openclaw/...` and does not automate that parity. The issue is also canonical (`dedupe:parent`) with recent same-problem reports, while the related fix PRs #31355 and #31457 were closed unmerged in the provided context.

## Best Possible Solution

Keep #31331 open as the canonical Docker-install/DooD sandbox workspace bind issue. Best fix is to make Docker sandbox and sandbox-browser creation host-path aware by remapping managed workspace, sandbox workspace, and agent workspace bind sources through the gateway container mount table before validation, hashing, and `docker create`; add regression coverage for `workspaceAccess` `rw`, `ro`, and `none`. If maintainers prefer host-path parity as the contract, update Docker setup/onboarding to set host-canonical workspace paths and identical volume maps by default, with docs/tests proving the Docker install sandbox path works.
