## Summary

Keep #32473 open. Current main still intentionally rejects non-localhost plain-HTTP Control UI sessions without browser device identity, including the common VPS/NAS/Docker path discussed in the issue. The thread documents working HTTPS, localhost/SSH tunnel, Tailscale/Traefik, trusted-proxy, and break-glass workarounds, but the broader product/security decision for homelab remote HTTP ergonomics remains unresolved; #56830 and #71669 are narrower copy/warning follow-ups, not full replacements.

## Best Possible Solution

Keep #32473 open for maintainer product/security judgment. The best path is either to design and document a safe homelab-friendly connection path that preserves device identity where browsers allow it, or to make an explicit policy decision that remote plain HTTP must remain rejected and then land the narrower operator-facing warning/copy improvements in #71669/#56830 before closing this support-heavy issue.

## Risks / Open Questions

- Closing as implemented because break-glass `dangerouslyDisableDeviceAuth` works would ignore commenters who specifically want device authentication preserved on non-127.0.0.1 HTTP access.
