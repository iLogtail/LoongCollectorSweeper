## Summary

Current main still intentionally rejects Control UI sessions that lack browser device identity from non-localhost insecure HTTP contexts. The issue discussion documents valid HTTPS, localhost/SSH tunnel, trusted-proxy, and break-glass workarounds, but the requested homelab/VPS non-127.0.0.1 HTTP device-auth path remains unresolved and needs maintainer product/security judgment rather than conservative cleanup closure.

## Best Possible Solution

Keep #32473 open for maintainer product/security judgment. The likely resolution is either a deliberately designed safe homelab-friendly path for non-localhost Control UI access, or an explicit decision to keep rejecting remote insecure HTTP while landing clearer server/UI guidance through narrower follow-ups such as #71669 and any copy-only improvements from #56830.
