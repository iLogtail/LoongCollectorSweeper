## Summary

Keep open. Current main partially avoids one older inflation path by ignoring cumulative-only Codex token usage unless a `last/current` breakdown is present, but the core reported bug remains plausible and unproven-fixed: the Codex app-server projector still stores `inputTokens` directly as `usage.input` while separately storing `cachedInputTokens` as `usage.cacheRead`, and shared context accounting still treats `input + cacheRead + cacheWrite` as the prompt/context total. The regression test currently locks in that additive shape rather than the reporter's observed `inputTokens`-already-includes-cache case. The maintainer comment also explicitly says the related #70749 checkpoint-usage PR is not the Codex cached-input fix and reports fresh 2026.4.24-era user signal.

## Best Possible Solution

Keep this issue open. Maintainers should verify the Codex app-server `TokenUsageBreakdown` contract for `last.inputTokens`; if it includes cached input, normalize in `extensions/codex/src/app-server/event-projector.ts` before storing usage, add a regression test for the reporter's arithmetic (`inputTokens == totalTokens - outputTokens` with nonzero `cachedInputTokens`), and preserve the shared `derivePromptTokens` additive contract for providers whose `input` is already uncached.
