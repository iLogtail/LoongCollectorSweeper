## Summary

Keep #69298 open. Current main still maps Codex app-server `inputTokens` directly to OpenClaw `input` and `cachedInputTokens` to `cacheRead`; shared session/context accounting still treats `input + cacheRead + cacheWrite` as prompt/context tokens. The current tests cover per-turn Codex token projection but do not cover the observed inclusive-input shape from the issue, and the maintainer comment explicitly notes fresh 2026.4.24-era signal plus that related PR #70749 is a different checkpoint-usage bug.

## Best Possible Solution

Keep the issue open and fix it in the Codex harness usage adapter, preferably by normalizing Codex `inputTokens` to uncached input when `cachedInputTokens` is present or by deriving prompt context from trusted `totalTokens - outputTokens` for this harness. Add a regression test using the observed shape (`inputTokens` equals prompt-side total while `cachedInputTokens` is a subset) and verify `/status` plus persisted `SessionEntry.totalTokens` no longer double-count cached input.
