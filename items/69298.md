## Summary

Keep open. Current main has improved the Codex app-server projector by preferring current-turn `last` usage over cumulative `total` usage, but I do not see source, tests, or docs proving the reported cached-input double-counting is fixed. The Codex adapter still maps `inputTokens` directly to normalized `input` and `cachedInputTokens` directly to `cacheRead`, while shared session/context/status math still treats `input + cacheRead + cacheWrite` as the prompt/context total. The provided maintainer comment also explicitly says the related #70749 work is a different checkpoint-usage inflation fix and does not prove this issue fixed.

## Best Possible Solution

Keep #69298 open and fix it in the Codex app-server usage adapter or with an explicitly documented app-server contract. The best code path is to add a regression fixture matching the reported shape (`inputTokens == totalTokens - outputTokens`, with nonzero `cachedInputTokens`) and normalize Codex `inputTokens` to cache-exclusive `input` when the app-server field is prompt-total-inclusive. If upstream `last.inputTokens` is already cache-exclusive, document that in generated protocol comments/tests and add a guard proving `deriveSessionTotalTokens` and status do not inflate Codex context snapshots.
