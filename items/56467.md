## Summary

Current main does not implement two-strike or configurable N-strike enforcement for agents or subagents. Subagent runs persist terminal outcome and emit `subagent_ended`, and agent runs emit `agent_end`, but there is no consecutive failure counter, no `agents.defaults.failureThreshold` / `agents.defaults.onFailure` config, and no core disable/escalate/notify action triggered by repeated failures.

## Best Possible Solution

Keep this issue open. A core implementation should define failure semantics, persist/reset consecutive failure counts for agent and subagent runs, add typed config/schema/docs for threshold and action policy, expose sufficient hook context, and add focused lifecycle tests for error, timeout, killed, and success-reset behavior.

## Risks / Open Questions

- The eventual design needs maintainer judgment on what counts as failure: error, timeout, killed/manual stop, startup failure, and provider retry exhaustion do not all have the same semantics.
- The best persistence key is a product/API decision: per configured agent id, per session key, per subagent child session, or per requester-child relationship each has different operator impact.
