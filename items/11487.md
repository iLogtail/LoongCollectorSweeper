## Summary

Keep #11487 open. Current main still does not provide a canonical optional human-readable `name`, `label`, or `description` field for group config entries. Representative core and bundled plugin group schemas remain strict behavior-only objects, generated config metadata still omits the requested field, and public docs list only behavior fields for group entries.

## Best Possible Solution

Keep the issue open. The best implementation is to choose one canonical optional string metadata field, add it to core and bundled channel group-entry types/Zod schemas, preserve it while ignoring it functionally at runtime, regenerate config metadata, update public channel/group docs, and add focused validation tests proving the field is accepted without affecting group behavior.
