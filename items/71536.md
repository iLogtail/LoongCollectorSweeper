## Summary

Keep open. Current main has the expected Telegram native command authorization path, but #71536 is a recent regression report and the available evidence does not prove it is fixed, duplicated, stale, or purely misconfiguration. `/restart` is a native built-in command and is owner-gated after normal sender authorization, and both gates can surface the same native reply: "You are not authorized to use this command." The report lacks sanitized config/logs, but it specifically claims both Telegram allowlists and owner-related config were updated, so the best maintainer action is a targeted Telegram `/restart` reproduction/regression test rather than auto-closing.

## Best Possible Solution

Keep #71536 open and add a focused regression path for Telegram `/restart`: cover DM and, if relevant, group/topic cases where the sender is in `channels.telegram.allowFrom` or `commands.allowFrom.telegram` and also in `commands.ownerAllowFrom`, with `commands.restart` enabled. If the test fails, fix the Telegram native command path or owner normalization/reload path; if it passes, ask the reporter for sanitized `commands.*` and `channels.telegram.*` config, chat type, numeric Telegram `from.id`, account id, and gateway logs around the rejected command.
