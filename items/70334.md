## Summary

Keep #70334 open. The issue is recent, concrete, corroborated by another user, and still matches current main: overflow recovery can compact and retry, but there is no direct source/test evidence that this exact post-compaction retry path always clears the active embedded run and diagnostic state back to idle when the retry hangs or loses terminalization. Related #70591 was closed as a duplicate of this tracker; #71455 is a narrower provider-specific DeepSeek replay problem and does not supersede this general session-lock report.

## Best Possible Solution

Keep #70334 open as the canonical tracker for the compaction retry/session-lock freeze. Add a focused regression that exercises explicit overflow compaction followed by retry success, retry error, and a lost/hanging retry terminalization path; assert the active embedded run clears, diagnostics return to idle or a bounded abort state, `queueDepth` drains, and follow-up messages can proceed. If the root cause is stale active-run state rather than compaction wait resolution, recovery should use the existing abort/cleanup primitives instead of leaving stuck-session diagnostics warning-only.
