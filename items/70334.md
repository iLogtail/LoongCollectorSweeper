## Summary

Keep #70334 open. The report is recent, concrete, and corroborated, and #70591 has already been closed as the ClawX/WebChat duplicate of this same compaction/session-freeze failure. Current main has relevant mitigations and generic active-run cleanup, but I did not find direct source, test, changelog, or history evidence proving the exact successful overflow-compaction retry path can no longer leave diagnostics in `processing` with `queueDepth=0`.

## Best Possible Solution

Keep #70334 open as the canonical tracker. Add focused regression coverage for successful explicit overflow compaction followed by retry success, retry error, and lost retry/lifecycle resolution; assert the diagnostic session state returns to idle or the run is boundedly aborted, the active run registry clears, and queued follow-up messages can drain. If the root cause is stale active-run state rather than compaction wait resolution, add a conservative recovery path tied to existing abort/cleanup primitives instead of leaving `session.stuck` warn-only.
