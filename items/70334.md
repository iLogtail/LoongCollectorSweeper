## Summary

Keep #70334 open. The issue is recent, concrete, corroborated by a second reporter, and is now the canonical tracker for the related #70591 ClawX/webchat freeze. Current main has useful partial safeguards around compaction waits and generic timeouts, but I did not find source, test, history, or release evidence proving that the specific `processing` + `queueDepth=0` lock after successful overflow compaction and prompt retry is fixed.

## Best Possible Solution

Keep #70334 open as the canonical tracker. Add a focused regression around the embedded overflow-compaction retry lifecycle: simulate successful explicit overflow compaction followed by retry success, retry error, and lost retry/lifecycle resolution, then assert the diagnostic session state returns to idle or the run is boundedly aborted, the active run registry clears, and queued followups can drain. If the root cause is stale active-run state rather than compaction wait resolution, add a conservative watchdog/recovery path tied to existing abort primitives instead of leaving `session.stuck` warn-only.
