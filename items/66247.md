## Summary

Keep #66247 open. Current main still wires Telegram through the plain per-chat/topic sequential key, so ordinary same-topic follow-up messages remain blocked before core can apply immediate `steer`, `steer-backlog`, or `interrupt` queue policy. The PR head contains relevant unmerged work for that bypass and is not obsolete or superseded, though the remaining embedded-run and per-instance config review comments still need maintainer/author follow-up before merge.

## Best Possible Solution

Keep the PR open and finish the Telegram-specific busy-session bypass on top of the current PR shape: preserve existing control lanes, match core queue-mode precedence and aliases, cover text and media-only turns, and resolve the embedded-run and instance-config review comments with focused regression tests before merge.
