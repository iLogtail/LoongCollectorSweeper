## Summary

Keep open. This is a fresh, unprotected issue with a concrete failure mode against v2026.4.24, and current main only has partial in-process mitigations. Telegram polling can restart on getUpdates stalls, and the Gateway can restart stale channel runtimes, but neither is a process-level/event-loop watchdog for a live Gateway that is spinning CPU so hard that WebSocket status probes and the reply pipeline stall. The stability bundle path also still records only safe error metadata, not the useful rejection stack/context requested here.

## Best Possible Solution

Keep #72338 open. The maintainer path should collect a diagnostics export plus CPU/profile evidence from a current-main or latest-release Gateway, then fix the narrowed cause. Likely work areas are a process-level event-loop/CPU stall watchdog or supervisor health signal, isolation/cleanup for interactive channel login subprocesses launched from agent sessions, richer safe stability metadata for unhandled rejections, and regression coverage proving Telegram polling, reply dispatch, and `channels status --probe` stay responsive or recover cleanly.
