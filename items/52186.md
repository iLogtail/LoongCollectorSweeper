## Summary

Keep #52186 open. The useful observability fixes from #57953/#57954 are present on current main, and ElevenLabs now has voice-note-aware synthesis output, but the remaining behavior called out by the maintainer comment is still not implemented: normal TTS paths still build a provider fallback chain and can play a fallback provider such as OpenAI when the selected ElevenLabs attempt fails or is skipped.

## Best Possible Solution

Keep the issue open and implement a deliberate TTS fallback policy. A good fix would either make configured-primary provider playback sticky by default or add a documented `messages.tts` fallback policy that `/tts audio`, auto-TTS, and channel delivery all honor. Preserve the structured diagnostics from #57953/#57954 and add regression coverage where `messages.tts.provider: "elevenlabs"` plus configured OpenAI does not silently play OpenAI unless fallback is explicitly allowed.
