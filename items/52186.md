## Summary

Keep #52186 open. Current main has the diagnostics and partial fallback observability from #57953/#57954, but the remaining behavior called out by the maintainer comment is still not implemented for configured auto TTS: `messages.tts.provider` remains a primary provider with automatic fallback, so ElevenLabs can still fall through to OpenAI instead of failing closed or enforcing provider stickiness.

## Evidence

- **Maintainer discussion leaves remaining scope open:** Issue comment 4159720380 says #57953 and #57954 partially addressed diagnostics, but explicitly leaves the issue open for enforcing provider/playback stickiness or equivalent provider-selection policy changes.
- **Configured provider still has automatic fallbacks:** `resolveTtsRequestSetup` builds the provider list as `[provider]` only when `disableFallback` is true; otherwise it calls `resolveTtsProviderOrder(provider, cfg)`, so a configured ElevenLabs primary can still fall through to OpenAI or another registered provider.
