## Summary

Keep #56708 open. Current main still treats Slack reactions as queued system events rather than turn triggers: the Slack reaction handler authorizes/routes the reaction and calls `enqueueSystemEvent`, while normal Slack messages go through `handleSlackMessage` and dispatch the message pipeline. The Slack config schema and docs expose `reactionNotifications`/`reactionAllowlist`, but no `reactionTriggersTurn`, `reactionWakeMode`, or equivalent wake-mode setting. Related reaction reports in the provided context, especially #64752 for Telegram, also remain open because the same class of inbound reaction wake behavior is not implemented.

## Best Possible Solution

Keep the issue open and implement a backward-compatible reaction wake mode, likely on the bundled channel reaction-notification path. A good fix would add a documented config such as `channels.<channel>.reactionWakeMode: "next-turn" | "now" | "off"` with `next-turn` as the default, then wire Slack reaction events to request a session-scoped heartbeat/agent turn when set to `now`. The same contract should be considered for Telegram and other channel reaction handlers, with tests proving both default queue-only behavior and immediate wake behavior.
