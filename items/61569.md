## Summary

Keep #61569 open. Current main still carries `--node-id` only as `GatewayClient.instanceId`, while the gateway node registration and node-pairing reconciliation still derive the effective node ID from the paired device ID or client ID. That means a custom CLI node ID is still not the key used by `node.list`, `node.describe`, or `tools.exec.node`. The two linked fix PRs (#61587 and #61719) were closed unmerged, and the current checkout does not contain their proposed `connect.client.instanceId` priority change.

## Best Possible Solution

Implement the identity fix in core gateway/node handling: normalize a non-empty `connect.client.instanceId` from `openclaw node run --node-id`, use it as the effective `NodeSession.nodeId` and node-pairing ID where appropriate, preserve current device/client ID fallback behavior, and add focused tests for registration, pairing reconciliation, `node.list`/`node.describe`, and `tools.exec.node` selection.
