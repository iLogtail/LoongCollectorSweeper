## Summary

Keep open. Current main has ACP thread binding, output streaming, and explicit `/acp steer`, but the issue asks for ordinary Discord/thread messages to reach a running ACP or sub-agent session while it is still working. Current ACP dispatch sends bound messages through `AcpSessionManager.runTurn(mode: "prompt")`, and the manager serializes all turns for a session through the per-session actor queue, so a second message waits behind the active turn. The ACP runtime contract also lacks a separate live-input method. Native embedded session steering is only available through configured `steer`/`steer-backlog` queue modes, while the default queue mode is `collect`; subagent tool steer aborts/restarts the child run rather than delivering normal thread input into the active turn.

## Best Possible Solution

Keep the issue open and implement a focused live-input path for bound ACP/sub-agent conversations. For ACP, add a capability-aware runtime/control-plane API for mid-turn input or concurrent steer, wire bound thread messages to it when a session is running, and preserve serialized `runTurn` for normal idle turns. For native sub-agent sessions, decide whether bound thread messages should default to embedded-run queue steering or require an explicit queue mode, then document the behavior and add Discord/thread-bound tests covering an active turn receiving input before completion.

## Risks / Open Questions

- Adding this naively by reusing `runTurn` will keep the current serialized behavior and still miss true mid-turn input.
- Changing default queue behavior for all chat sessions could regress existing collect/follow-up semantics; this likely needs scoped behavior for bound ACP/sub-agent threads.
- ACP backends may differ in whether they support live input while a turn is active, so the runtime contract needs capability-aware fallback behavior.
