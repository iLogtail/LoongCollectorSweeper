## Summary

Keep open. Current `main` still treats Nextcloud Talk rich-object payloads as plain text and never threads attachment URLs through the inbound context, so the underlying bug this PR targets is still present. The PR discussion also surfaced follow-up correctness concerns around rich-object fallback behavior and MIME/type propagation for extensionless share links, so this is not a conservative auto-close candidate.

## Evidence

- **Current main still passes raw webhook content through as text:** `payloadToInboundMessage()` assigns `text` directly from `payload.object.content` and only carries the webhook `mediaType`; there is no rich-object JSON parsing or attachment extraction on current `main`.

## Risks / Open Questions

- If this PR is revived, it likely needs `MediaTypes` propagation in addition to `MediaUrls` so image/audio/video handling can classify extensionless Nextcloud share links reliably.
- If rich-object parsing is kept, the parser should return the human-readable `message` for successfully parsed rich objects even when no `{placeholder}` tokens match.
