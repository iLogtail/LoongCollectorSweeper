## Summary

Keep #60084 open. Current main still hard-wires the builtin memory engine and task registries to file-backed `node:sqlite` `DatabaseSync`, only accepts the `sqlite` memory store driver, and has no `better-sqlite3`, env-var driver, or lock-probe fallback. The related items in the provided context cover missing FTS5 or a file-scan memory fallback, not this broader ARM64/container file-lock failure that also affects task registries.

## Best Possible Solution

Keep the issue open and treat it as core storage resilience work, not ClawHub/plugin cleanup. The best path is a shared SQLite storage seam or lock probe used by the builtin memory backend and task registries, with either automatic fallback or an explicit supported driver setting, plus regression coverage for `DatabaseSync` construction/first-statement `SQLITE_BUSY` failures and aligned config/docs.
