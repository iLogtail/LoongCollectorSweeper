## Summary

Keep open. Current main still has the unsupported-platform gateway service throw, and the main status paths still resolve the gateway service before fallback status handling can apply. The linked fix PR #25690 was closed unmerged, and the latest issue discussion already contains a maintainer Codex review keeping this open for the same reason.

## Best Possible Solution

Keep #25621 open until OpenClaw either adds a graceful unsupported-platform GatewayService adapter/fallback, guards read-only status and gateway status resolver calls so diagnostics work on OpenBSD, or makes an explicit documented product decision that unsupported service-manager platforms must use manual foreground gateway startup outside managed service flows. The likely implementation belongs in `src/daemon/service.ts` plus regression coverage for `openclaw status` and `openclaw gateway status` on an unsupported `process.platform`; PR #25690 remains useful prior art but needs revival against current main.
