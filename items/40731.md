## Summary

Keep open. Current `main` still builds the Telegram `/think` arg menu from the default provider/model context instead of the active session override, so the reported bug remains plausible, but this PR now needs rework because current thinking semantics and session-store access patterns have changed since the patch was proposed.

## Evidence

- **Telegram native command path still uses default arg-menu context:** The Telegram handler calls `resolveCommandArgMenu({ command, args, cfg: runtimeCfg })` with no active-session `provider` or `model`, so `/think` menu choices are still derived from config defaults rather than the routed session.
  - file: [extensions/telegram/src/bot-native-commands.ts:793](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/extensions/telegram/src/bot-native-commands.ts#L793)
  - command: `nl -ba extensions/telegram/src/bot-native-commands.ts | sed -n '789,804p'`
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Arg-menu resolution falls back to default provider/model:** `resolveCommandArgMenu` picks the choice-bearing arg via `resolveCommandArgChoices`, and `resolveCommandArgChoices` uses `resolveDefaultCommandContext(cfg)` unless the caller explicitly provides `provider` and `model`. That means function-based choice lists do not see session overrides today.
  - file: [src/auto-reply/commands-registry.ts:297](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/auto-reply/commands-registry.ts#L297)

- **The `/think` command menu depends on provider/model-specific choices:** The built-in `think` command still defines its choices as `({ provider, model }) => listThinkingLevels(provider, model)`, so passing the wrong context directly affects which levels appear in Telegram menus.
  - file: [src/auto-reply/commands-registry.shared.ts:697](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/auto-reply/commands-registry.shared.ts#L697)

The test suite also covers provider profiles exposing `max` directly. That makes the PR's `max -> xhigh` rewrite stale against current main semantics.
