## Summary

Keep #56502 open. Current main still does not implement SecretRef support for the requested phone-number/PII channel identifiers: WhatsApp `allowFrom`, `defaultTo`, and `groupAllowFrom` remain plain string fields, voice-call `fromNumber`, `toNumber`, and `allowFrom` remain E.164 string fields, and the SecretRef credential surface does not register these paths. The maintainer comment saying the issue is still valid matches the current code.

## Best Possible Solution

Keep the issue open and implement this as channel/plugin-owned SecretRef support, not a startup migration. Widen the relevant WhatsApp and voice-call config schemas/types/manifests to accept plain strings plus SecretRef-backed values, register explicit secret target entries and runtime collectors, resolve before access-control/outbound routing reads the fields, keep plain strings backward-compatible, redact/report status safely, update generated config metadata/docs, and optionally add a doctor rewrite that helps users move existing plaintext phone numbers into SecretRefs without silently rewriting config at load time.
