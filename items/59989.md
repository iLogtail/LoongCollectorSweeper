## Summary

Keep open. Current main still removes the Pi session listener immediately after `abortCompaction()`, so the PR is not implemented or superseded on main. The branch appears directionally useful, but the provided review discussion identifies blocking correctness issues in the PR itself, so this should be reworked rather than auto-closed.

## Best Possible Solution

Keep the PR open and ask for a targeted rework: preserve the existing synchronous unsubscribe contract or explicitly await a shared cleanup promise in all teardown callers; register the cleanup waiter without the check/register race; resolve only after the final non-retry compaction end; clear timeout handles; and add focused tests for post-abort compaction-end delivery, `willRetry=true` followed by the final end event, and timeout behavior with fake timers.

## Risks / Open Questions

- The PR as written changes `unsubscribe()` completion semantics; current production cleanup still calls it synchronously, so a merge without caller/API rework could make teardown less deterministic.
- The PR review's `willRetry=true` concern is substantive because current main treats retry compaction as a multi-event lifecycle and only final non-retry completion should release a cleanup wait.
- The branch likely needs a rebase to current main event naming and handler structure before it can be evaluated as mergeable.
