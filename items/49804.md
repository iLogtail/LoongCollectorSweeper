## Summary

Keep #49804 open. Current main has several targeted lifecycle and cleanup fixes, but it does not implement the requested shared ResourceManager-style shutdown protocol across retry, cleanup, listener disposal, and plugin/channel resources. Related narrow work remains open, and no single canonical item fully supersedes this broader architecture discussion.

## Best Possible Solution

Keep #49804 open as the broader maintainer design tracker. The best path is a narrow, backward-compatible lifecycle protocol that builds on existing channel `startAccount`/`stopAccount`, plugin SDK lifecycle helpers, chat abort tracking, and pi-runner cleanup hooks: register async work with ownership, stop accepting new work during shutdown, propagate cancellation where appropriate, drain pending operations with bounded timeouts, and migrate the known Feishu, retry/tool, outbound replay, and listener-disposal paths with regression tests. Coordinate with #48183, #48512, #48793, and the stale-replay canonical tracker rather than closing this umbrella now.
