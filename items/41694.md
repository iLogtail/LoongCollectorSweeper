## Summary

Keep open. Current `main` still lacks the requested `targetResolver.defaultTargetKind` API and the related explicit/provider-prefix handling in outbound target resolution, so this PR is not already implemented on `main` and is not obsolete.

## Evidence

- **Public channel-plugin contract still has no `defaultTargetKind`:** `ChannelMessagingAdapter.targetResolver` only exposes `looksLikeId`, `hint`, and `resolveTarget`; there is no `defaultTargetKind` field on current `main`. That means the PR's requested contract change is not present.
  - file: [src/channels/plugins/types.core.ts:540](https://github.com/openclaw/openclaw/blob/bae7b54a85b17d180e4bfc28973a7e4fd1646a93/src/channels/plugins/types.core.ts#L540)
  - command: `nl -ba src/channels/plugins/types.core.ts | sed -n '520,555p'`
  - sha: [bae7b54a85b1](https://github.com/openclaw/openclaw/commit/bae7b54a85b17d180e4bfc28973a7e4fd1646a93)
- **Kind detection still defaults through `inferTargetChatType` before explicit-prefix handling:** `detectTargetKind` calls plugin `inferTargetChatType` first, then only recognizes `@`/`user:` as user and `#`/`channel:` as group before falling back to `group`. It does not implement the PR's explicit `group/chat/room/conversation/dm` preservation or provider-prefix stripping.
