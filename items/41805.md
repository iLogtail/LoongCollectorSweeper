## Summary

Keep #41805 open. Current main partially improves the surrounding area: Discord component entries can store callbackData/messageId, plugin interactive handlers receive sender/message metadata, and normal inbound prompts include sender_id. The ordinary agent fallback still turns a button click into plain BodyForAgent/RawBody text and sets MessageSid to the Discord interaction id, without surfacing a structured interaction payload containing original message id, component/custom id, button label, and clicker metadata. The related closed items cover sender id and routing, not this remaining structured callback metadata request.

## Best Possible Solution

Keep the issue open and implement an additive Discord component interaction metadata path. A conservative fix would preserve the current fallback text/command behavior while adding structured metadata, likely via UntrustedStructuredContext and/or explicit ctx fields, containing schema, channel, interaction kind, original messageId, interactionId, component/custom id or callbackData, button/select label, selected values, and sender id/username. Update Discord component tests and docs to lock the contract.

## Risks / Open Questions

- Changing BodyForAgent or CommandBody semantics could break existing callbackData-as-command workflows; additive metadata should preserve the current plain-text fallback.
- The implementation needs to distinguish Discord interaction id from the original component message id and any reply_to id.
- Interaction metadata should be exposed as untrusted user-context metadata, not as trusted system instructions.
