## Summary

Keep #49524 open. Current main still fails open for thrown memory_search errors, but it does not add the requested hard timeout around manager initialization or the full manager.search call, so a never-resolving memory path can still hold the tool result hostage. The related #49794 PR appears to be the active fix attempt, but it is still open and not merged into the checked-out main.

## Best Possible Solution

Keep the issue open. Land a focused fix that bounds the whole memory_search tool path, including getMemoryManagerContext/getMemorySearchManager and manager.search, and returns the existing disabled/unavailable payload on timeout. Add regression coverage with fake timers or controlled deferred promises for stalled manager initialization and stalled search, then close #49524 via the merged PR.
