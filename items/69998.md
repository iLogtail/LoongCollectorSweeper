## Summary

Keep #69998 open. Current main still returns the updated `skills.update` entry as `config: current`, so the plaintext `apiKey` and secret-looking `env` values reported in #66769 are not fixed on main. The existing redaction utility and tests already cover `skills.entries.*.env.*`, and the closed duplicate PRs #66928/#66986 do not supersede this active implementation PR.

## Best Possible Solution

Keep the PR open and finish it as the focused gateway fix: return a redacted `skills.update` response while persisting full values, add regression coverage for `apiKey` and secret-looking `env` response redaction, and either restore/keep existing values when `REDACTED_SENTINEL` is submitted to `skills.update` or stop returning a config object so the sentinel cannot be round-tripped into storage.

## Risks / Open Questions

- The PR should address the sentinel round-trip review before merge: returning `__OPENCLAW_REDACTED__` from `skills.update` without restore/keep-existing handling can overwrite a valid saved secret if a client later resubmits the response config.
- The current Control UI ignores the `skills.update` response result, but `skills.update` is still a gateway RPC surface used by clients and wrappers, so relying only on current UI behavior is not a sufficient product boundary.
