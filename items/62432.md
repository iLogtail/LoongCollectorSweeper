## Summary

Keep open. Current main still does not provide the requested runtime guardrail for repeated `exec` relaunches after a `Command still running` result. The exec result includes volatile `sessionId`, `pid`, and `startedAt` details, while `hashToolOutcome()` still falls through to hashing the whole details object, so no-progress streak detection can miss this exact pattern. Generic repeated `exec` detection is warning-only, and loop detection is disabled by default unless configured. Related PR #41502 directly proposes the missing exec-result canonicalization but remains open/stale, so this issue should not be closed as implemented.

## Best Possible Solution

Keep #62432 open as the product-level bug until main has a shipped guardrail. The likely fix should canonicalize `exec` running/completed outcomes so volatile process metadata does not defeat no-progress detection, make repeated `exec` relaunches after a running result block or coalesce rather than just warn, and add regression coverage for `Command still running` results with changing `sessionId`/`pid`. PR #41502 is a relevant candidate implementation but does not yet close the issue because it is unmerged and stale.
