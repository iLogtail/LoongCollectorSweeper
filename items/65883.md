## Summary

Keep open: current `main` still has a plausible overwrite path for this report. A dequeued queued message is removed from the queue, shown optimistically as a user turn, and then the client reloads `chat.history` on every final event and replaces the in-memory transcript wholesale. There is no targeted regression test proving queued user turns survive that reload.

## Evidence

- **Dequeued queue items are removed before resend:** `flushChatQueue()` removes the queued entry from `chatQueue` before sending it, so the queued bubble disappears from queue state immediately and must survive via transcript state instead.

  - file: [ui/src/ui/controllers/chat.ts:294](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/ui/src/ui/controllers/chat.ts#L294)

## Risks / Open Questions

- No live webchat reproduction was run in this read-only review, so this decision is based on current code and tests rather than an end-to-end gateway session.
- The backend persistence timing behind `chat.history` was not inspected here; if the server now guarantees the dequeued user turn is present before final, the UI overwrite path would be masked even though the client still lacks a protective merge/guard.
