## Summary

Keep open. Current `main` has gateway-side work to package TTS replies as WebChat `content` blocks with `type: "audio"`, but the Control UI/WebChat rendering path still does not normalize or render those audio blocks, so there is not high-confidence evidence that the reported no-playback/no-download WebChat bug is fixed.

## Evidence

- **TTS success still depends on downstream media delivery:** On success, the core TTS reply path returns `mediaUrl: result.audioPath` and `audioAsVoice`, so WebChat still has to surface the generated audio attachment correctly.
  - file: [extensions/speech-core/src/tts.ts:1188](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/speech-core/src/tts.ts#L1188)

  - file: [src/gateway/server-methods/chat-webchat-media.ts:158](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/gateway/server-methods/chat-webchat-media.ts#L158)
