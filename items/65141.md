## Summary

Keep #65141 open. Current main still wires iMessage startup through the generic per-account list and does not contain the proposed default-account dedupe. With a config containing both `accounts.default` and a named iMessage account, the plugin can still list both account IDs and the gateway can start one provider task per listed ID. The linked fix PR #65194 is open and unmerged, so the bug should remain tracked until that or an equivalent patch lands.

## Best Possible Solution

Keep the issue open and land PR #65194 or an equivalent iMessage-owned fix. The fix should make `default` act as fallback-only when a named iMessage account exists and `default` has no distinct runtime/backend fields, preserve `default` when it defines a distinct `cliPath`/`dbPath`/runtime identity, and add regression coverage in `extensions/imessage/src/accounts.test.ts` for both cases.
