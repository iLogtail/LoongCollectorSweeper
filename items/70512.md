## Summary

Keep #70512 open. Current main still routes Feishu DMs by sender Open ID only; `root_id`/`thread_id` are parsed into context but not used to build a direct-message session peer or conversation id. Existing Feishu topic-session support is implemented and tested for groups/topic groups, not per-topic DM isolation, and the related #10242 cleanup covered Feishu DM thread creation/reply sending rather than inbound DM topic session keys.

## Best Possible Solution

Keep the issue open as a concrete Feishu bundled-plugin feature request. A good fix would live in `extensions/feishu`: add a Feishu DM session-scope config such as `per-user`/`per-topic`, derive p2p/private peer and conversation ids from `thread_id` or `root_id` when enabled, preserve correct DM delivery targets, and add docs plus tests for two topics from the same `open_id` producing isolated sessions while the default remains sender-scoped.

## Risks / Open Questions

- Before implementation, confirm the Feishu/Lark DM topic event contract: whether p2p/private topic messages consistently include `thread_id`, `root_id`, or both, and which id is valid for reply delivery.
- The current Feishu delivery-target parser treats `:topic:` conversation ids as chat/topic targets, so a direct-topic conversation grammar would need an explicit direct-topic form rather than reusing group semantics blindly.
