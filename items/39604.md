## Summary

Keep #39604 open. Current main still lacks the requested general `web_fetch` private-network opt-in: schema, generated config metadata, runtime wiring, tests, and docs only expose `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange`, which is a narrow fake-IP proxy exception and does not allow localhost, LAN, or private/internal service fetches. The related implementation PR #39630 is still open/unmerged in the provided context, and #43700 was closed unmerged, so the requested behavior is not shipped or superseded.

## Best Possible Solution

Keep #39604 open until maintainers either land an explicit default-off `web_fetch` private-network opt-in or make a product decision not to support it. The likely path is to update or replace PR #39630 against current main, preferably under `tools.web.fetch.ssrfPolicy` with a deliberately named option such as `dangerouslyAllowPrivateNetwork`, then wire schema/types/generated help, runtime guarded-fetch policy, cache-key isolation, docs, and SSRF tests while preserving the existing RFC2544-only option.

## Risks / Open Questions

- A broad private-network opt-in expands SSRF reach when enabled, so any eventual implementation should use explicit dangerous naming or a similarly clear default-off contract, include cache-key isolation, docs, generated config metadata, and regression tests.
- The shipped RFC2544 setting only addresses `198.18.0.0/15` fake-IP proxy compatibility; treating it as a full fix would leave the original localhost/LAN/internal-service workflow unresolved.
