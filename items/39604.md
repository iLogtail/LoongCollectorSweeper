## Summary

Current main still does not expose a web_fetch opt-in for localhost/LAN/private-network targets. The schema and public type only accept `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange`, runtime only forwards that RFC2544 flag, tests still assert localhost/private IPs are blocked, and docs still describe private/internal hostnames as blocked. The lower-level SSRF guard supports `allowPrivateNetwork`, so the missing work is config/runtime/docs/test wiring for web_fetch. The related implementation PR #39630 remains the active path to evaluate or supersede, not evidence that the issue is fixed on main.

## Best Possible Solution

Keep #39604 open until maintainers either land or explicitly reject a web_fetch private-network opt-in. The likely path is to evaluate or supersede PR #39630 with a current-main implementation that updates `src/config/zod-schema.agent-runtime.ts`, `src/config/types.tools.ts`, config labels/help/generated schema, `src/agents/tools/web-fetch.ts`, `docs/tools/web-fetch.md`, and SSRF tests for default blocking plus explicit localhost/private-DNS opt-in. Given current naming, maintainers may prefer a scoped `tools.web.fetch.ssrfPolicy.dangerouslyAllowPrivateNetwork` shape over the originally proposed flat key, but the user problem remains the same.
