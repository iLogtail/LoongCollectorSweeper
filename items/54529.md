## Summary

Keep open. Current main does support `channels.telegram.apiRoot` passthrough, but the specific local Bot API migration failure is not fully implemented: persisted Telegram update offsets are still scoped by account/bot ID rather than `apiRoot`, and polling still treats non-network `deleteWebhook` failures as fatal before polling starts.

## Best Possible Solution

Keep #54529 open and fix it in the Telegram plugin: include normalized `apiRoot` in the update-offset state/key or reset offsets when it changes, handle `deleteWebhook` `400: Logged out` as a transient/nonfatal migration state, add focused regression tests for both paths, and document the local Bot API migration steps.
