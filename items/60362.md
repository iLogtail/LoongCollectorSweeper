## Summary

Current main appears to fix the recovery chunking path: startup recovery replays through `deliverOutboundPayloads`, the shared outbound core applies channel chunkers before `sendText`, and Telegram advertises a markdown chunker with a 4000/4096 effective limit plus a send-layer 4000-char safety split. However, the exact fallback requested in this issue is still missing: `PERMANENT_ERROR_PATTERNS` does not include Telegram's `message is too long` error, and the policy test has no coverage for that string. Because the title and expected behavior explicitly ask for both recovery chunking and immediate permanent classification, this should stay open for a small targeted fix or a maintainer decision to intentionally rely only on chunking/bounded retries.

## Best Possible Solution

Keep the issue open for a narrow maintainer fix: add an explicit Telegram oversized-message permanent-error classifier, preferably `/message is too long/i`, plus policy and recovery tests proving the entry moves to `failed/` immediately. Keep the already-shipped recovery chunking path and Telegram send-layer chunking as the primary prevention mechanism.
