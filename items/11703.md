## Summary

Keep #11703 open. Current main still preserves WhatsApp `messageTimestamp` for access/context, but inbound batching remains a receive-time debounce keyed by account/conversation/sender. There is no timestamp bucket, coalescing window, config option, or regression test for late Baileys deliveries sharing the same WhatsApp server timestamp. The related #19716 context remains unrelated because that PR was closed unmerged and concerned exposing timestamps to agent context, not timestamp-based batching.

## Best Possible Solution

Keep #11703 open. Implement a WhatsApp-owned timestamp-aware coalescer, or an additive shared inbound debouncer capability if maintainers want it reusable, that groups same account/chat/sender messages with the same or near `messageTimestamp` even when Baileys delivers them after the normal receive-time debounce window. Add fake-timer regression coverage for delayed same-timestamp deliveries and update config/docs if a new option is exposed.
