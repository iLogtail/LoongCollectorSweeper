## Summary

Keep open. Current main already distinguishes Gateway clients at connect time, including browser Control UI and native macOS, but that metadata still does not reach the agent as trusted runtime info. `chat.send` only maps client identity into sender/untrusted message context and scopes; the runtime prompt type/line still expose channel and capabilities without a client type.

## Best Possible Solution

Keep #39597 open and implement a narrow trusted Gateway client-context path: define sanitized client metadata, populate it from `client.connect.client` in `chat.send`, propagate it through the auto-reply/run params into `RuntimeInfoInput`, render fields such as `client=openclaw-macos`, `client_mode=ui`, and `client_platform=macos` in `buildRuntimeLine`, then add prompt and Gateway chat tests plus WebChat/Control UI docs explaining trusted/privacy-safe fields.

## Risks / Open Questions

- Raw userAgent, modelIdentifier, or instance identifiers could be privacy-sensitive if exposed directly; the fix should use sanitized fields such as client id, mode, platform, and device family.
- Existing prompts/tests may already observe client ids through SenderId/SenderName in some paths, so preserve compatibility while adding a separate trusted runtime client context.
