## Summary

Keep open. Current main has Gateway client metadata that distinguishes browser Control UI and native macOS clients at connect time, but the agent-facing runtime/prompt context still does not expose a trusted client type such as `client=macapp`; WebChat turns still surface only `channel=webchat` plus capabilities, and client identity is only mapped into sender metadata/untrusted context.

## Best Possible Solution

Implement a narrow trusted Gateway client-context path in core: define sanitized client metadata, propagate it from connect.client through chat.send and agent run setup into RuntimeInfoInput or trusted inbound metadata, update buildRuntimeLine/prompt tests, and document which WebChat/native client fields are trusted and privacy-safe.

## Risks / Open Questions

- Agents can sometimes infer openclaw-control-ui from untrusted sender metadata today, but that does not satisfy the requested trusted runtime/client context.
- A future implementation should avoid exposing raw userAgent to agents without an explicit privacy/product decision; sanitized id/mode/platform fields are safer first fields.
