## Summary

Keep open. Current `main` still has the underscore-only copy-sensitive guard in `src/tui/tui-formatters.ts`, so long hyphen-separated tokens are still split in the TUI formatter path this PR targets. The existing tests cover paths, URLs, underscore tokens, and mixed alnum credentials, but not hyphenated file-like/package-name tokens.

## Evidence

- **Current main still excludes hyphenated file-like tokens:** `isCopySensitiveToken` preserves paths, URLs, slashes, and underscore-containing file-like tokens, but the guard is still `token.includes("_") && FILE_LIKE_RE.test(token)`. `FILE_LIKE_RE` itself allows `-`, so hyphen-only package names are not treated as copy-sensitive on `main`.
  - file: [src/tui/tui-formatters.ts:82](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/tui/tui-formatters.ts#L82)

  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Reasonable local reproduction still shows the bad split:** Replaying the current `isCopySensitiveToken`/`normalizeLongTokenForDisplay` logic with `some-long-package-name-with-many-segments` produces `some-long-package-name-with-many -segments`, which matches the PR's reported failure mode.
