## Summary

Keep open. Current `main` still does not fail fast on missing default-model auth, but this PR's approach would also introduce false startup failures because it gates on a narrower auth heuristic than the runtime actually supports. The review comments also identified an unresolved misleading error-message example.

## Evidence

- **Current main still swallows startup-model failures and keeps going:** `startGatewaySidecars` computes `skipChannels`, calls `prewarmConfiguredPrimaryModel()`, then always proceeds to `startChannels()` inside a `try/catch` that only logs `channel startup failed` instead of aborting startup. That does not implement the PR goal of failing fast before channel startup.
  - file: [src/gateway/server-startup-post-attach.ts:255](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/gateway/server-startup-post-attach.ts#L255)
  - command: `nl -ba src/gateway/server-startup-post-attach.ts | sed -n '220,340p'; nl -ba src/gateway/server-startup-post-attach.ts | sed -n '468,520p'`
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Warmup path is model-resolution only, not auth-resolution:** `prewarmConfiguredPrimaryModel()` resolves the configured model via `resolveModel()` / `resolveModelAsync()` and only warns on failure. The model resolver path builds/queries model registries; it does not call `resolveApiKeyForProvider()`, so current main still lacks a true startup auth guard for the default model.

Current runtime auth resolution also accepts provider config overrides/custom API keys, AWS SDK auth for Bedrock, and synthetic local auth. Merging the PR as written would reject valid working configurations at startup.
