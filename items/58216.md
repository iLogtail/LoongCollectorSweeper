## Summary

Keep open. Current `main` does not implement the requested Discord shutdown suppression, and the existing tests do not cover the reported stale-socket abort race.

## Evidence

- **Current main still treats reconnect exhaustion as fatal:** `extensions/discord/src/monitor/provider.lifecycle.ts:388-409` has no abort-aware suppression branch. `handleGatewayEvent` logs gateway errors and returns `"stop"` for `event.shouldStopLifecycle`, which includes `reconnect-exhausted`.
  - file: [extensions/discord/src/monitor/provider.lifecycle.ts:388](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/discord/src/monitor/provider.lifecycle.ts#L388)
  - command: `nl -ba extensions/discord/src/monitor/provider.lifecycle.ts | sed -n '352,469p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Live lifecycle stop events still reject the wait path:** `extensions/discord/src/monitor.gateway.ts:57-60` converts any lifecycle event whose handler returns `"stop"` into `finishReject(new DiscordGatewayLifecycleError(event))`. There is no special-case here for abort-driven `reconnect-exhausted` events arriving before the abort wait path settles.
  - file: [extensions/discord/src/monitor.gateway.ts:57](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/discord/src/monitor.gateway.ts#L57)
  - command: `nl -ba extensions/discord/src/monitor.gateway.ts | sed -n '18,75p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Current tests do not cover the reported abort race in provider lifecycle:** `extensions/discord/src/monitor/provider.lifecycle.test.ts:374-392` still asserts queued `reconnect-exhausted` errors throw, and current `main` has no test for suppressing `reconnect-exhausted` during stale-socket shutdown. `extensions/discord/src/monitor.gateway.test.ts:143-161` only covers the later ordering where disconnect emits after that helper's own abort handler has already started resolving.
  - file: [extensions/discord/src/monitor/provider.lifecycle.test.ts:374](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/discord/src/monitor/provider.lifecycle.test.ts#L374)

## Risks / Open Questions

- The reported crash remains plausible on current `main` because the lifecycle handler still treats `reconnect-exhausted` as a fatal stop event during shutdown.
- If this PR is reconsidered, its regression coverage still needs careful review so the live-event abort ordering is actually exercised.
