## Summary

Keep open. Current `main` still has the live-switch restart gap described here: `canRestartForLiveSwitch` only checks per-attempt side-effect buffers, while reply payload construction can still emit a user-visible assistant answer from `lastAssistant` when `assistantTexts` is empty. The new `LiveSessionModelSwitchError` fallback handling on `main` is narrower and only covers candidate-failure paths, not this successful-fallback reply path.

## Evidence

- **Live-switch restart gate still ignores successful `lastAssistant` output:** `canRestartForLiveSwitch` is still computed from `didSendViaMessagingTool`, approval/tool state, and `assistantTexts.length === 0`; it does not check whether the attempt already has a visible assistant answer in `lastAssistant`.
  - file: [src/agents/pi-embedded-runner/run.ts:961](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/agents/pi-embedded-runner/run.ts#L961)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '930,1035p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Payload assembly still emits assistant replies when `assistantTexts` is empty:** `buildEmbeddedRunPayloads(...)` falls back to `extractAssistantVisibleText(lastAssistant)` when `assistantTexts` is empty, so a successful reply can exist even when the live-switch guard still considers the attempt restartable.
