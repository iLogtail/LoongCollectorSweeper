## Summary

Keep open. Current main still lacks WhatsApp per-group `allowFrom`: the WhatsApp group config type, strict Zod schema, generated channel metadata, runtime inbound policy, and tests/docs all still model sender authorization only through channel/account-level `groupAllowFrom` with fallback to `allowFrom`. Related #63302 is a global denylist PR and does not satisfy the requested per-group allowlist. The issue also intersects the open maintainer-labeled #69874 multi-account `groups` inheritance decision, so this needs a real WhatsApp plugin change and maintainer/product review rather than cleanup closure.

## Best Possible Solution

Keep #69926 open. The best path is a focused WhatsApp plugin change: add optional `allowFrom` to root and account-scoped `groups.<jid>` config schema/types/generated metadata, resolve the effective group entry for each inbound group conversation, make group-level `allowFrom` override channel/account `groupAllowFrom` when defined, keep fallback behavior when undefined, update command authorization consistently, add regression tests for root/account scopes plus wildcard and fallback semantics, and document the behavior alongside the #69874 multi-account inheritance decision.
