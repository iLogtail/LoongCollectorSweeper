## Summary

Keep open. Current `main` does not implement per-field length limits for hook agent metadata, and the PR as described would only validate direct `/hooks/agent` bodies. Mapped hook endpoints are a shipped surface and still synthesize and dispatch `name`, `sessionKey`, `to`, `model`, and `thinking` without going through `normalizeAgentPayload()`, which matches the substantive review concern already raised on the PR.

## Evidence

- **Direct hook normalization still has no per-field length checks on main:** `normalizeAgentPayload()` trims and returns `name`, `agentId`, `sessionKey`, `to`, `model`, and `thinking`, but current `main` has no max-length validation for those fields.

## Risks / Open Questions

- Closing this PR would leave the security hardening request unresolved on `main` while the mapped-hook bypass remains in shipped code.
- If this PR is revived as-is, it should likely factor the length validation into a shared path that covers both direct `/hooks/agent` requests and mapped hook dispatches; the existing review also noted missing exact-boundary coverage for `agentId` and `sessionKey`.
