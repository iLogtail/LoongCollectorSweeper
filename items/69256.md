## Summary

Keep open. Current main still deletes `deleteAfterRun` direct cron sessions without checking active descendants, and `sessions.delete` cleanup cascades to stop subagents for that requester. The PR is not safely obsolete, but the current patch also has unresolved review feedback: it skips premature deletion without adding a later cleanup retry or regression coverage for the active-descendant direct-delivery case.

## Best Possible Solution

Keep the PR open but require a fuller fix: guard direct `deleteAfterRun` cleanup against active descendants, add an active-descendant regression test for structured/threaded direct delivery, and add or prove a second-stage cleanup path that deletes the isolated cron session after descendants drain. If maintainers decide the existing text-path behavior is the intended retention model, that should be stated explicitly in review before merging.
