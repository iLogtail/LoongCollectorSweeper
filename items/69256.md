## Summary

Keep open. Current main still deletes `deleteAfterRun` direct cron sessions without checking active descendants, while structured/threaded deliveries are routed through that immediate cleanup path. The PR is not obsolete, but the current patch also has unresolved correctness feedback: it needs regression coverage and either a deferred cleanup retry or an explicit maintainer decision that skipped cleanup is acceptable while descendants are active.

## Best Possible Solution

Keep the PR open, but require a fuller fix before merge: guard direct `deleteAfterRun` cleanup against active descendants, add an active-descendant regression test in `src/cron/isolated-agent/delivery-dispatch.double-announce.test.ts`, and either implement or prove a second-stage cleanup path that deletes the isolated cron session after descendants drain. If maintainers want the session retained in this case, state that explicitly in review and adjust tests around the intended retention model.
