## Summary

Keep #71260 open. Current main still has only the existing shallow `mergeSessionEntry` behavior and the two existing merge policies, and there is no typed `pluginMetadata` field, `merge-plugin-metadata` policy, or gateway/session-store adoption. This is not a ClawHub-only request because the requested fix is a core session merge/API seam needed for third-party plugin coexistence, and the linked Smarter-Claw/OpenClaw RFCs are related but do not supersede this exact merge-policy problem.

## Best Possible Solution

Keep the issue open as a focused plugin/session-store API RFC. The likely product direction is an additive, documented session metadata merge contract, with tests for namespace preservation, overwrite, and deletion semantics, then adoption at the agent-run and gateway mutation paths that can carry plugin-owned session state.

## Risks / Open Questions

- The right API shape needs maintainer design because merge semantics, null/deletion behavior, and whether `pluginMetadata` should become a public `SessionEntry` field affect all session-store writers.
- A narrow implementation only at one call site would not solve the reported coexistence problem if other `mergeSessionEntry` paths can still overwrite plugin-owned metadata.
