## Summary

Keep open. Current main still implements the strip-and-deliver behavior that this PR is trying to change: a payload like `File's there. Not urgent.\n\nNO_REPLY` is normalized to visible text rather than suppressed. The PR is not obviously superseded by current main or by the narrower open #66755 work, but the submitted patch needs revision because review correctly points out a false-positive risk in the `stripped.length < text.length` heuristic.

## Best Possible Solution

Keep the PR open for maintainer/author follow-up. The best implementation is a precise shared trailing-silent-token detector for `normalizeReplyPayload()` that suppresses the full text-only turn only when a trailing token was actually present, preserves media/channelData behavior intentionally, keeps non-trailing mentions like `Use NO_REPLY when...\n` deliverable, and adds regression tests for raw reasoning preambles, false positives, mixed case, and media payloads. After maintainers choose the policy, consolidate or close duplicates such as #66755/#42873 against the canonical fix.

## Risks / Open Questions

- The PR patch as submitted is not merge-ready: `stripped.length < text.length` can become true from whitespace trimming even when `NO_REPLY` was only a non-trailing mention, which would drop legitimate content.
- The inline `(#TBD)` code comment noted by review should be removed or replaced before merge.
- The PR explicitly does not address streaming block delivery that may already have sent intermediate content before a final trailing `NO_REPLY` arrives.
