## Summary

Close as implemented on main: the reported WhatsApp false-success/caption-only media-send bug matches the shipped gateway `--media` routing fix in `v2026.4.23`, and current `main` still contains that implementation and its regression test.

## Evidence

- **Release note matches the exact symptom:** `v2026.4.23` changelog explicitly says gateway-mode WhatsApp `--media` sends were fixed because they could previously drop the attachment while still delivering the caption, which is the same behavior reported here.
  - file: [CHANGELOG.md:942](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/CHANGELOG.md#L942)
  - command: `nl -ba CHANGELOG.md | sed -n '936,948p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Current main routes media sends through the media path:** Current `main` sends any outbound message with `mediaUrl` through `outbound.sendMedia(...)` instead of the text-only path, preserving the attachment handling path that the changelog calls out.
  - file: [src/cli/send-runtime/channel-outbound-send.ts:60](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/cli/send-runtime/channel-outbound-send.ts#L60)
  - command: `nl -ba src/cli/send-runtime/channel-outbound-send.ts | sed -n '40,68p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Regression test covers preserved media context:** The test asserts that a WhatsApp gateway send with `mediaUrl` invokes `sendMedia` and preserves `mediaAccess`, `mediaLocalRoots`, and `mediaReadFile`, covering the caption-without-attachment regression shape.
  - file: [src/cli/send-runtime/channel-outbound-send.test.ts:16](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/src/cli/send-runtime/channel-outbound-send.test.ts#L16)
  - command: `nl -ba src/cli/send-runtime/channel-outbound-send.test.ts | sed -n '12,60p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **WhatsApp sender still builds real attachment payloads once routed correctly:** The WhatsApp send API on current `main` converts routed media into actual `image`, `video`, or `document` payloads rather than text-only sends, so the fixed CLI/runtime path reaches the attachment transport.
  - file: [extensions/whatsapp/src/inbound/send-api.ts:49](https://github.com/openclaw/openclaw/blob/cf858258c71701ef62dddf88aa777873a4962046/extensions/whatsapp/src/inbound/send-api.ts#L49)
  - command: `nl -ba extensions/whatsapp/src/inbound/send-api.ts | sed -n '45,74p'`
  - sha: [cf858258c717](https://github.com/openclaw/openclaw/commit/cf858258c71701ef62dddf88aa777873a4962046)
- **Issue discussion matched the fixed gateway-media path:** The reporter and follow-up commenters narrowed this to `message send --channel whatsapp --media ...` / gateway-style media sends that returned success while only the caption arrived, which aligns directly with the `v2026.4.23` changelog entry for the shipped fix.

## Risks / Open Questions

- If someone can still reproduce caption-only WhatsApp media sends on `v2026.4.23` or newer, that would indicate a separate regression or a different non-CLI send path and should be reported with a fresh reproduction on a current build.
