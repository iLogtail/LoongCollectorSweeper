## Summary

Keep open. Current main has ACP `sessions_spawn` support, parent relay via `streamTo: "parent"`, and session/thread binding infrastructure, but it still does not expose the requested explicit `delivery` object for routing an ACP spawn directly to an arbitrary channel/topic such as Zulip/Discord. The related Zulip plugin reference in the issue is useful because the existing binding adapter surface can likely participate, but the missing piece is still a core `sessions_spawn` API/runtime delivery path, not something already implemented on main.

## Best Possible Solution

Keep this open for maintainer/product design. A good implementation would add a typed `delivery` object to `sessions_spawn` for `runtime: "acp"`, reuse or align with cron delivery parsing where appropriate, validate channel/account/target permissions, route through existing SessionBindingAdapter/outbound delivery seams, and add tests/docs for Discord/Zulip-style topic delivery plus negative cases for unsupported channels and unsafe cross-channel targets.

## Risks / Open Questions

- Adding this should be treated as a core runtime/API change because explicit channel targets affect authorization, cross-channel misdelivery protection, and plugin/channel capability checks.
- The design should avoid silently accepting ignored `delivery` input; current tool parsing would otherwise ignore unknown `delivery` fields unless schema/runtime validation is tightened.
