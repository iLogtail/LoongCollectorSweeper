## Summary

Keep #70325 open. Current main still applies the 10 minute default cron execution timeout to main-session `systemEvent` jobs, and `timeoutSeconds` is only modeled/exposed for `agentTurn` jobs. The memory-core managed dreaming path has been moved to isolated `agentTurn` jobs, which helps that bundled plugin path, but it does not solve the reported general main-session/systemEvent behavior or document/configure the hard limit.

## Best Possible Solution

Keep the issue open and address the core cron contract directly: either add a documented timeout policy for main-session `systemEvent` jobs, support a per-job timeout for that payload/session type, or change recurring main-session cron accounting so successful long heartbeat turns are not recorded as timeout errors. Any implementation should include a regression test for a main-session systemEvent run whose heartbeat exceeds the current 10 minute ceiling.

## Risks / Open Questions

- A fix should preserve the cron lane safety net; simply removing the timeout for all main-session jobs could let stuck heartbeat work hold cron execution indefinitely.
- Memory-core managed dreaming already moved to isolated agentTurn, so maintainers should avoid conflating that plugin-specific migration with the still-missing generic main-session/systemEvent behavior.
