## Summary

Keep open. Current main fixes the built-in memory-core managed dreaming cron by moving it from a main-session system event to an isolated lightweight agent turn, and v2026.4.24 documents that shipped fix. However, the issue also reports the broader main-session `systemEvent` timeout behavior, and current main still gives non-agent cron jobs the 10-minute default with no `timeoutSeconds` support on `systemEvent` payloads. The remaining main-session behavior is still actionable unless maintainers explicitly narrow this issue to managed memory-core dreaming only.

## Best Possible Solution

Keep this open but narrow it. The best fix is either to add a documented timeout policy for main-session `systemEvent` cron jobs, or explicitly document that long-running work should use isolated/current/custom `agentTurn` cron jobs with `timeoutSeconds` instead. The memory-core managed dreaming case should be treated as already fixed by the v2026.4.24 isolated-agent migration and doctor repair path.
