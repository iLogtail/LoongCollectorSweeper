## Summary

Keep open. Current main no longer shows the old literal Feishu-specific tool-call log path, but the requested behavior is still not implemented: embedded tool lifecycle logs are emitted through a module-level `agent/embedded` logger, not through the originating channel subsystem such as `gateway/channels/telegram`.

## Best Possible Solution

Keep the issue open and implement a narrow logging fix in the embedded tool lifecycle path: derive the origin channel from `messageChannel` or the parsed `sessionKey` when present, include `sessionKey`/`runId` metadata, use `gateway/channels/<channel>` only for real channel-originated runs, fall back to `agent/embedded` otherwise, and add regression coverage for at least Telegram plus a non-channel/local run.

## Risks / Open Questions

- A broad fix that blindly uses a channel logger for all embedded runs could misattribute CLI, local, subagent, compaction, or channel-less runs; fallback behavior should remain explicit.
- Tool lifecycle logs can expose arguments/results depending on log level, so any added channel metadata should preserve existing redaction and verbosity boundaries.
