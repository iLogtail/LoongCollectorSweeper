## Summary

Keep open. Current `main` does not include the HarmonyOS app or gateway client-id support, and the PR head still has unresolved functional regressions in auth/setup-code handling and connection state.

## Evidence

- **Not implemented on main:** `main` has no `apps/HarmonyOS` subtree; the only app directories are `apps/android`, `apps/ios`, `apps/macos`, `apps/macos-mlx-tts`, and `apps/shared`. This is not already shipped on current `main`.
  - command: `find apps -maxdepth 1 -mindepth 1 -type d | sort | nl -ba`
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Gateway does not accept a HarmonyOS client id on main:** `src/gateway/protocol/client-info.ts` defines iOS and Android app ids but no HarmonyOS id, and `src/gateway/server.ios-client-id.test.ts` only accepts iOS/Android. The PR’s server-side support is not present on `main`.
  - file: [src/gateway/protocol/client-info.ts:3](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/gateway/protocol/client-info.ts#L3)
  - command: `nl -ba src/gateway/protocol/client-info.ts | sed -n '1,40p'; nl -ba src/gateway/server.ios-client-id.test.ts | sed -n '1,40p'`
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **PR head exposes a pre-auth connected state and short mobile timeout:** In PR head `apps/HarmonyOS/entry/src/main/ets/api/GatewaySession.ets`, `awaitConnectNonce` defaults to 2000 ms, and `connect()` sets `this.connected = true` before `performAuthentication()` finishes. ## Risks / Open Questions

- Setup-code onboarding looks incompatible with current bootstrap-token pairing, which is a user-visible first-run failure.
- Talk mode gating would still exclude a HarmonyOS-only client unless core mobile-node detection changes alongside the app.
