## Summary

Keep open. Current main does not implement the requested two-phase `bootout -> wait old pid/port gone -> bootstrap/kickstart` ordering. `restartLaunchAgent` still uses `launchctl kickstart -k` as the primary restart operation, and the unit tests explicitly assert that the normal restart path performs no `bootout` or `bootstrap`.

## Best Possible Solution

Keep this issue open for maintainer judgment. The best implementation path is a focused macOS launchd lifecycle change in `src/daemon/launchd.ts` that either adds the requested two-phase stop/wait/start sequence with explicit ordering tests and detached-handoff coverage, or documents why the current kickstart-plus-stale-port-cleanup approach is the intentional product behavior and closes the issue manually.

## Evidence

- **Current restart path:** `restartLaunchAgent` resolves the service, optionally runs stale-port cleanup, enables the service, then calls `launchctl kickstart -k`; it only bootstraps on a not-loaded fallback and does not do the requested bootout/wait/bootstrap sequence.

## Risks / Open Questions

- A two-phase launchd refactor could regress the current installed-but-unloaded recovery and detached handoff paths unless those paths get targeted tests.
- The existing stale-port cleanup may already reduce the practical EADDRINUSE race, so maintainers should decide whether the stronger launchd service-state sequencing is worth the macOS `bootout`/`bootstrap` risk described in the issue.
