## Summary

Keep open. Current main still does not implement explicit web-search provider fallback configuration: configured providers are treated as explicit selections and fail fast, while fallback retry is only allowed for non-explicit auto-detect paths. The PR is a plausible core/runtime config change rather than obsolete ClawHub work.

## Best Possible Solution

Keep this PR open as the current implementation candidate. The best path is to review or rebase it onto current main, keep the fallback list as an explicit `tools.web.search.fallbacks` config/schema/help addition, and land it only after focused runtime tests cover configured fallback order, unknown fallback ids, duplicate/case-normalized ids, per-call provider override fail-fast, unavailable providers, thrown errors, and structured error payloads.
