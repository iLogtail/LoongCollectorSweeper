## Summary

Keep open. Current main has implemented the per-agent TTS half via `agents.list[].tts`, but the Android and iOS Talk/Voice paths still route through the resolved main session rather than exposing a Talk/Voice agent/session picker. The maintainer comment on this issue matches the current code state and explicitly says the Talk/Voice UI session-switching request remains open.

## Best Possible Solution

Keep this issue open for the remaining product work: add a Talk/Voice session or agent selector, wire the selected key into Android and iOS Talk mode routing, decide whether macOS needs an explicit Talk selector beyond its current active WebChat fallback, and add focused UI/routing tests. Treat `agents.list[].tts` as the canonical already-shipped per-agent voice configuration path rather than adding the proposed `talk.voiceId` shape.
