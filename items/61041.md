## Summary

Keep open. Current main still rejects inbound chat image attachments above the 5,000,000-byte gateway limit before any decode/resize path, and existing resize/sanitization helpers cover adjacent provider/client paths rather than this requested gateway behavior.

## Best Possible Solution

Keep the issue open and implement the behavior in the gateway attachment pipeline: validate MIME/base64 safely, use the existing image-ops resize/recompress helpers for image payloads over the configured budget, preserve offload semantics for large accepted images, and update chat/agent/node-event tests to assert resize success and failure only when the image cannot be reduced safely. Coordinate with #67031 for configurability and #68524 for iOS composer-side resizing rather than closing this as covered by either one.
