## Summary

Current main still exposes Feishu plugin tools and their schemas to embedded runs, and the strongest follow-up reproduction in the issue resolved as Feishu app permission/publish misconfiguration rather than an OpenClaw tool-injection bug.

## Close Comment

Closing from a Codex review because I can’t substantiate an OpenClaw tool-exposure bug on current `main`. Feishu still registers `feishu_doc`/`feishu_wiki` with concrete schemas in `extensions/feishu/src/docx.ts:1429` and `extensions/feishu/src/wiki.ts:178`, plugin tools are still appended into the agent tool set in `src/agents/openclaw-tools.ts:325`, and embedded runs still pass the effective tool set through tool-definition assembly in `src/agents/pi-embedded-runner/run/attempt.ts:510` and `src/agents/pi-tool-definition-adapter.ts:216`.

The strongest reproduction follow-up on this issue was also resolved without an OpenClaw code change: after correcting Feishu scopes and republishing the Feishu app, `feishu_doc` and `feishu_wiki` worked normally. That points to Feishu-side configuration rather than a persistent embedded-agent exposure defect. Current `main` also has a separate hardening change in `86fa8eeb68c64eb0ae25e1eb8c6196061862a010` so streamed JSON tool args no longer collapse to `{}` before dispatch.

If someone can still reproduce this on a current build, please open a fresh issue with the exact OpenClaw version, model/provider, Feishu app scopes, whether the app was republished after scope changes, and one failing session log showing the emitted tool definitions and the raw tool-call payload.
