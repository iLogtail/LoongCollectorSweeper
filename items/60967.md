## Summary

Keep #60967 open. Current main still does not expose or pass a per-tool MCP request timeout for bundled `mcp.servers`: startup uses `connectionTimeoutMs`, but `callTool` still invokes the MCP SDK without request options. Related #57969 closed a remote-transport/startup gap, and open PR #62249 targets this timeout area, but it is unmerged and only proposes a fixed 5-minute default rather than the configurable override requested here.

## Best Possible Solution

Keep the issue open and fix the bundle MCP runtime by adding a documented, bounded per-server tool/request timeout field, resolving it alongside transport config, and passing it as MCP SDK `RequestOptions` to `session.client.callTool(...)`. PR #62249 could be revised into that configurable fix with focused regression coverage for `mcp.servers.<name>` long-running tool calls.
