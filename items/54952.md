## Close Comment

Closing from a Codex review because current `main` already has an operator-facing way to confirm the routing decision for this case. The router exposes `matchedBy` outcomes in `src/routing/resolve-route.ts:46`-`66`, logs every candidate binding plus the winning `matchedBy`/`agentId` in `src/routing/resolve-route.ts:711`-`809`, and those debug lines are available through gateway file logs via `OPENCLAW_LOG_LEVEL=debug` or `openclaw --log-level debug ...` (`docs/logging.md:149`-`152`, `docs/cli/logs.md:9`-`23`, `src/logger.ts:87`-`90`). On the Feishu side, `extensions/feishu/src/bot.ts:651`-`666` logs bound-conversation reroutes and `extensions/feishu/src/bot.ts:1214` logs the final session dispatched. That means an operator can already tell whether a Feishu group chat hit an explicit binding/bound conversation path or fell through to the default route on current `main`. If you want a richer UX beyond logs, like binding IDs or a doctor-style explainer, that would need a separate enhancement request.
