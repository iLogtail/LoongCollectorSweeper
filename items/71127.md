## Summary

Keep #71127 open. Current main still implements stuck-session diagnostics as warning/telemetry only: the heartbeat emits `session.stuck`, but there is no `stuckSessionAbortMs` config, no `session.aborted` event, and no diagnostic path wired to the existing run-abort/queue-clear mechanisms. The related #71455/DeepSeek replay work appears to address one provider-specific freeze source, not the core self-recovery behavior requested here.

## Best Possible Solution

Keep the issue open and implement a conservative core recovery path: add a documented optional abort threshold, emit a distinct diagnostic event/log when recovery fires, wire it to existing reply-run abort and queue cleanup primitives, and cover the heartbeat-to-abort behavior with focused diagnostics/runtime tests. Treat provider-specific fixes like the DeepSeek V4 replay work as useful related mitigation, not as a substitute for the requested generic session self-healing.
