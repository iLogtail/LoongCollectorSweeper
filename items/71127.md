## Summary

Keep #71127 open. Current main still treats stuck-session detection as diagnostics-only: the heartbeat emits `session.stuck` warnings but does not invoke the existing active-run abort machinery, and the public diagnostics config still only exposes a warning threshold. The related #71455 DeepSeek issue appears to cover one provider-specific poisoning cause, not the broader core recovery request here.

## Best Possible Solution

Keep the issue open as a core runtime recovery bug. The likely path is to add a conservative, opt-in or well-bounded stuck-session recovery threshold that maps diagnostic session state back to the active reply/embedded run, aborts via the existing run registry/embedded-run abort path, releases queue state, emits a structured `session.aborted` diagnostic event, and adds focused tests proving that stale processing sessions recover while normal long-running turns are not aborted by the warning-only threshold.
