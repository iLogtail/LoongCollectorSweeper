## Summary

Keep #41495 open. Current main has a structured Telegram inline-button path through the shared `presentation` contract, but it does not address the reported Gemini failure mode where button JSON is emitted in final message text. The current system prompt still advertises an unsupported `buttons=[[...]]` shape, while the message tool schema exposes `presentation` and intentionally omits provider-native `buttons`.

## Best Possible Solution

Keep the issue open and fix the product path: align `src/agents/system-prompt.ts` with the shared `presentation` schema, add tests proving Telegram `presentation` blocks render as inline buttons, and add either a narrowly scoped legacy parser or explicit non-support decision for raw `[[...callback_data...]]` final text.

## Risks / Open Questions

- A prompt-only change may reduce but not eliminate Gemini final-text fallbacks; this needs a regression test around Gemini-like text output or tool-use compliance.
- If maintainers choose to parse raw button JSON from assistant text, the parser should be tightly scoped so arbitrary JSON in chat is not converted into callback actions.
