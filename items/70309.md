## Summary

Keep open. Current main has adjacent pieces: an Infrastructure > MCP config section, a quick-settings MCP server count, and an Agents tools panel that lists session-effective tools. It does not provide the requested per-MCP-server connectivity panel, status/error details, or one-click reconnect/refresh. Failed MCP server starts are currently logged and omitted from the successful catalog, which leaves the reported debugging gap intact.

## Best Possible Solution

Keep the issue open. The likely maintainer path is to add a redacted gateway MCP status/probe surface for configured servers, including per-server state, tool names/count, last probe/error, and an explicit refresh/reconnect action, then render it in Control UI under Infrastructure > MCP with focused protocol, runtime, and UI tests. The existing `tools.effective` panel can be reused as supporting context but does not satisfy the feature by itself.

## Risks / Open Questions

- Implementing this needs a clear runtime contract because probing MCP servers may spawn stdio processes or open remote connections outside an active agent turn.
- Error details must be redacted before reaching Control UI because MCP URLs, headers, and env vars can contain tokens.
- A refresh/reconnect button should define whether it invalidates only cached session MCP runtimes or also probes saved global `mcp.servers` entries independent of sessions.
