## Summary

Current `main` still appears to have the reported gap. The PI discovery bridge only converts inline `key`/`token` fields, while auth-profile persistence strips those plaintext fields when `keyRef`/`tokenRef` are present. The runtime resolver in `src/agents/auth-profiles/oauth.ts` already handles secret refs, but `resolvePiCredentialMapFromStore()` does not reuse it. `/models` still loads its catalog through `discoverAuthStorage()`, so the picker path remains exposed to this mismatch.

## Evidence

- **PI discovery bridge still ignores secret refs:** `convertAuthProfileCredentialToPi()` only reads inline `cred.key`, `cred.token`, and inline OAuth fields. `resolvePiCredentialMapFromStore()` just loops over store profiles and uses that converter, so `keyRef`-only or `tokenRef`-only profiles still resolve to `null`.

  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)

That means a normal secret-backed stored profile can arrive at PI discovery without the inline fields its converter requires.
