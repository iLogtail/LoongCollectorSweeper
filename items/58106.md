## Summary

Keep open. Current `main` at `90877e0d4274c53fd91bf5787265c4d6889dc9a7` still drops secret-ref-backed `api_key`/`token` auth profiles during PI discovery, so the reported OpenRouter model-picker/auth-gating failure is not implemented or disproven.

## Evidence

- **PI discovery converter still ignores `keyRef`/`tokenRef`:** `convertAuthProfileCredentialToPi()` only reads inline `cred.key`, `cred.token`, and OAuth plaintext fields; empty inline values return `null`. There is no `keyRef`/`tokenRef` resolution in this path.
  - file: [src/agents/pi-auth-credentials.ts:16](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/agents/pi-auth-credentials.ts#L16)
  - command: `nl -ba src/agents/pi-auth-credentials.ts | sed -n '1,120p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **PI auth discovery still uses the sync converter directly:** `resolvePiCredentialsForDiscovery()` loads the auth profile store, then immediately calls `resolvePiCredentialMapFromStore(store)` before adding env/synthetic credentials. That means persisted ref-backed profiles are filtered by the converter above.
  - file: [src/agents/pi-auth-discovery.ts:14](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/agents/pi-auth-discovery.ts#L14)
