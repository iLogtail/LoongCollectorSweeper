## Summary

Keep open. Current `main` mitigates invalid-config outages by restoring a last-known-good snapshot during startup/reload, but unknown keys are still rejected by the normal config read path and startup still hard-fails when no recovery snapshot is available. The requested strict-write/tolerant-read behavior is not implemented on `main`.

## Evidence

- **Raw config validation is still strict:** `validateConfigObjectRaw` still calls `OpenClawSchema.safeParse(normalizedRaw)` and returns schema issues directly on failure; there is no alternate tolerant read-mode that strips unknown keys before validation.
  - file: [src/config/validation.ts:646](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/config/validation.ts#L646)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Gateway config load still fails closed on invalid schema:** `loadConfig()` validates the read config with `validateConfigObjectWithPlugins(...)` and throws on invalid config. The catch block explicitly says invalid configs must fail closed rather than falling back permissively.
  - file: [src/config/io.ts:1265](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/config/io.ts#L1265)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Current mitigation is last-known-good recovery, not tolerant parsing:** Gateway startup now attempts `recoverConfigFromLastKnownGood(...)` before asserting validity, but the recovery test also proves startup still throws when that recovery is unavailable.
  - file: [src/gateway/server-startup-config.recovery.test.ts:105](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/gateway/server-startup-config.recovery.test.ts#L105)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Docs still describe unknown keys as validation failures removed by doctor:** The public config docs say removed `bridge.*` keys are no longer in schema and that validation fails until they are removed; `openclaw doctor --fix` can strip unknown keys. That matches a workaround, not a tolerant read path.
  - file: [docs/gateway/configuration-reference.md:978](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/docs/gateway/configuration-reference.md#L978)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Doctor support exists specifically to strip unknown keys after the fact:** The doctor helper intentionally strips unknown keys from a cloned config, and the CLI docs say `openclaw doctor --fix` drops unknown keys and lists each removal.
  - file: [src/commands/doctor-config-analysis.ts:62](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/commands/doctor-config-analysis.ts#L62)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)

## Risks / Open Questions

- Existing installs are better protected now because startup/reload can restore a last-known-good config, so the original outage severity is reduced for users who already have a promoted good snapshot.
- The environment did not have `pnpm` or authenticated `gh`, so this review relied on current source, tests, docs, and local history rather than executing a live reproduction or hydrating related GitHub threads beyond the provided context.
