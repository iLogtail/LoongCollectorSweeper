## Summary

Keep #41783 open. Current main mitigates the cron self-deadlock path by mapping cron-owned embedded work onto the nested lane, but it has not implemented the requested timeout semantics: the cron execution timeout is still armed before `executeJobCore(...)` runs, and the isolated-agent call chain has no execution-start/lane-acquired callback to defer that timer.

## Evidence

- **Timeout still starts before cron job core execution:** `executeJobCoreWithTimeout()` resolves the job timeout, creates an AbortController, and immediately races `executeJobCore(state, job, signal)` against a `setTimeout(...)`. This is the same structural behavior the issue identifies: timeout begins before nested isolated-agent work has acquired its downstream execution lanes.

  - sha: [e640c0a95fcb](https://github.com/openclaw/openclaw/commit/e640c0a95fcbd06a1311f18fc635518d011d45da)
- **No execution-start callback in cron core signature:** `executeJobCore(...)` currently accepts only `state`, `job`, and an optional `AbortSignal`; there is no `onExecutionStart` or `onLaneAcquired` parameter that would arm the timeout after actual isolated-agent execution begins.
  - file: [src/cron/service/timer.ts:1133](https://github.com/openclaw/openclaw/blob/e640c0a95fcbd06a1311f18fc635518d011d45da/src/cron/service/timer.ts#L1133)
