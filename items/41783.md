## Summary

Keep #41783 open. Current main still starts the cron job timeout before isolated-agent execution reaches the queued embedded runner path, so cron/session/global lane wait can still consume the configured execution timeout. The related fix PRs are unmerged, and this issue remains the clearest canonical bug report for the remaining work.

## Best Possible Solution

Keep #41783 open and land a focused cron timeout fix: defer the execution timeout until isolated `agentTurn` work actually acquires the embedded runner session/global lanes, or split queue-wait and execution budgets with distinct errors. The likely code path is `src/cron/service/timer.ts` through `runIsolatedAgentJob`, `runCronIsolatedAgentTurn`, and `runEmbeddedPiAgent`; add a regression test that simulates queue wait longer than `timeoutSeconds` before execution begins.
