## Summary

Keep open. Current `main` still starts the cron timeout before isolated job execution actually begins, so queued time can still consume the timeout budget. Manual runs are queued on the shared `cron` lane, and isolated agent work is then moved onto the shared `nested` lane; the timeout race still wraps that whole path rather than starting after lane acquisition.

## Evidence

- **Timeout still starts before isolated execution:** `executeJobCoreWithTimeout()` immediately races `executeJobCore(...)` against `setTimeout(...)`, so the timeout clock starts before any isolated-run lane acquisition or execution work completes.
