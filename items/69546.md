## Summary

Keep #69546 open. Current main still starts QQBot with a captured config snapshot and evaluates inbound routes against that snapshot, while `bindings` changes are classified as no-restart dynamic config changes. The reporter's observed split between updated CLI bindings and stale QQBot runtime routing is still plausible on main.

## Best Possible Solution

Keep the issue open and fix QQBot routing freshness. The most direct product/code direction is to have QQBot resolve the active runtime config snapshot for each inbound route decision, or otherwise make `bindings` changes restart the QQBot channel if true dynamic reads are not feasible. Add a regression test that starts QQBot with an account binding, updates top-level `bindings` with a peer-specific C2C binding, and verifies the next inbound C2C route uses `matchedBy: "binding.peer"` without a gateway restart.
