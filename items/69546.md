## Summary

Keep #69546 open. Current main still has the reported QQBot runtime/config split: CLI binding writes update top-level `bindings`, but gateway hot reload treats `bindings` as a no-op and QQBot inbound routing resolves against the config snapshot passed at account startup. Other channels already use a fresh config specifically for bindings lookup, so the likely fix is to apply that pattern to QQBot and cover it with a regression test.

## Best Possible Solution

Keep the issue open and fix QQBot inbound routing to resolve route bindings from fresh runtime config per message, mirroring Telegram/WhatsApp, while preserving payload-derived account and peer fields. Add a focused QQBot regression test that starts with an account-level binding, updates top-level `bindings` with a C2C `peer.kind: "direct"` / `author.user_openid` binding, and verifies the next inbound route resolves `matchedBy: "binding.peer"` without gateway restart.
