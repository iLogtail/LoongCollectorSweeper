## Summary

Keep #69546 open. The current main code still matches the reported failure mode: root `bindings` changes are classified as dynamic/no-restart, but QQBot inbound routing uses the config object captured when the QQBot account gateway started. Core routing would prefer a peer binding over an account binding if it received the updated config, so the remaining bug is the QQBot runtime config freshness path rather than the route precedence logic. The provided related ACP/plugin binding PRs do not supersede this issue because this report is about ordinary `openclaw agents bind` routing for the bundled QQBot channel.

## Best Possible Solution

Keep the issue open and fix QQBot's inbound routing freshness. The preferred path is to thread a current config reader or runtime config accessor into QQBot `handleMessage`/`buildInboundContext` for route resolution and reply dispatch, then add a focused regression test that starts with an account binding, updates root `bindings` with a peer binding, and verifies a C2C inbound message resolves `matchedBy: "binding.peer"` without restarting the QQBot gateway. The core resolver precedence should remain unchanged.
