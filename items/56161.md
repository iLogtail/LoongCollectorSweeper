## Summary

Keep open. Current `main` still resolves `openclaw skills install` and `openclaw skills update` against the default agent workspace rather than inferring the agent from the current working directory, so the reported multi-agent workspace bug is still present.

## Evidence

- **CLI resolver still hard-codes default agent workspace:** `resolveActiveWorkspaceDir()` loads config and returns `resolveAgentWorkspaceDir(config, resolveDefaultAgentId(config))`; it never consults `process.cwd()` or a workspace-path lookup.
  - file: [src/cli/skills-cli.ts:44](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/cli/skills-cli.ts#L44)
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Install and update both use the default-agent resolver:** `skills install` uses `const workspaceDir = resolveActiveWorkspaceDir()` before calling `installSkillFromClawHub(...)`, and `skills update` does the same before reading tracked skills / updating them.
  - file: [src/cli/skills-cli.ts:101](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/cli/skills-cli.ts#L101)
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Default-agent selection ignores cwd:** `resolveDefaultAgentId()` chooses the first `default: true` agent or the first listed agent; it has no workspace-path inference logic.
  - file: [src/agents/agent-scope-config.ts:78](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/agents/agent-scope-config.ts#L78)
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Workspace-path inference helper exists elsewhere:** `resolveAgentIdByWorkspacePath(cfg, workspacePath)` is implemented and returns the most specific agent whose workspace contains the given path, matching the reporter's suggested direction for a fix.
  - file: [src/agents/agent-scope.ts:228](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/agents/agent-scope.ts#L228)
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Docs still promise active-workspace behavior:** The CLI docs say `search`/`install`/`update` install into the "active workspace `skills/` directory," which does not match the current resolver behavior above.
  - file: [docs/cli/skills.md:40](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/docs/cli/skills.md#L40)
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Tests only cover the default workspace path:** The command test asserts install/update use `/tmp/workspace` from the mocked default resolver, but there is no coverage for cwd-based agent inference or multi-agent workspaces.
  - file: [src/cli/skills-cli.commands.test.ts:189](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/cli/skills-cli.commands.test.ts#L189)
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)

## Risks / Open Questions

- I did not run a live CLI reproduction because this checkout does not have `pnpm` available, so the conclusion is from direct inspection of current code, docs, and tests.
- The same resolver is also used by `skills update`, and `loadSkillsStatusReport()` likewise defaults to `resolveDefaultAgentId(config)`, so the user-visible inconsistency may extend beyond `install`.
