## Summary

Keep open. Current `main` still has the reported display-name bug path: `buildGatewaySessionRow()` uses any truthy `entry.channel` to call `buildGroupDisplayName()` before falling back to `origin.label`, with no `chatType` guard. For a direct main session whose stored `channel` is `telegram`, that still yields a group-style label instead of `openclaw-tui`, and the TUI footer renders that `displayName` directly.

## Evidence

- **Unconditional group display-name fallback remains on main:** `buildGatewaySessionRow()` still computes `channel = entry?.channel ?? parsed?.channel` and then uses `channel ? buildGroupDisplayName(...) : undefined` before `entry?.label` and `origin?.label`. There is no `chatType === group|channel` guard in this path.
  - file: [src/gateway/session-utils.ts:1249](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/gateway/session-utils.ts#L1249)

  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Group-name builder still turns fallback keys into `g-*` labels:** `buildGroupDisplayName()` still falls back to `id ?? key` and prefixes non-`#`/`@` tokens with `g-`. For a direct main-session key with provider `telegram`, that logic still produces a group-style label such as `telegram:g-agent-main-main`.
  - file: [src/config/sessions/group.ts:61](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/config/sessions/group.ts#L61)
