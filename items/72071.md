## Summary

Keep #72071 open. The report is recent, concrete, and not maintainer/protected, but current main does not provide a strong close reason for the Kimi K2.6 Telegram leak. Main has generic guards that keep structured `toolCall` and `thinking` blocks out of normal extracted assistant text, but it still treats Ollama `message.content` text as user-visible. If Kimi places reasoning/preface text in `content`, the Telegram reply path can still deliver it, and there is no current regression test for `ollama/kimi-k2.6:cloud` producing mixed visible text plus tool calls.

## Best Possible Solution

Keep the issue open and add a focused regression around the Ollama Kimi K2.6 path: simulate a stream with `message.content` containing reasoning-like preface text plus `tool_calls`, verify the tool calls execute as structured calls, and verify Telegram/block reply delivery receives only the final user-facing answer. If Kimi emits reasoning in `content`, fix the provider wrapper or agent extraction boundary to classify/suppress that preface before it reaches `assistantTexts`/`payload.text`.

## Evidence

- **Ollama content mapping:** The Ollama adapter maps `message.thinking`/`message.reasoning` into hidden `thinking` blocks and `message.tool_calls` into structured `toolCall` blocks, but any `message.content` is appended as a visible `text` block. That leaves the reported plain-text reasoning preface actionable if Kimi emits it in `content`.
