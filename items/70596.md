## Summary

Keep open. Current main does not implement builtin explicit-local memory prewarm: post-attach startup only schedules memory startup for QMD, and startGatewayMemoryBackend still skips every non-QMD backend. The PR is recent and plausibly addresses open performance issue #70460, but the provided diff appears incomplete and has an unresolved P1 risk around poisoning the cached builtin manager on failed provider initialization.

## Best Possible Solution

Keep the PR open and request changes: update the post-attach startup gate so builtin explicit-local configs actually schedule memory startup; avoid caching a failed default manager during prewarm, for example by using a noncached/status-style probe, evicting/closing failed default managers, or clearing failed provider init promises; add an integration test through `startGatewayPostAttachRuntime`; and split or justify the unrelated provider-runtime test-support change.
