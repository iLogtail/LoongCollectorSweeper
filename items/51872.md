## Summary

Keep open. Current main supports OpenClaw gateway auth tokens/passwords inside the WebSocket `connect` RPC and supports trusted-proxy identity headers for browser/control UI deployments, but it does not implement the requested transport-level proxy authentication fields in `openclaw qr` or Android/iOS clients before the HTTP WebSocket upgrade.

## Best Possible Solution

Keep this issue open. A good path is a narrow, typed `transportAuth` addition to the pairing setup payload and QR CLI for Basic/Bearer/fixed header/cookie cases, secure storage and redaction in Android/iOS, and HTTP-upgrade header injection in OkHttp and the Swift WebSocket client. Treat OIDC/AppAuth as a follow-up design item unless maintainers want it in the same feature.

## Risks / Open Questions

- Transport proxy credentials in QR/setup payloads need careful secret storage, redaction, and expiry semantics on both mobile platforms.
- OIDC/AppAuth support is materially broader than static header/cookie support and likely needs separate product/security design.
