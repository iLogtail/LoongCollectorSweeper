## Summary

Keep open. Current `main` has a partial mitigation for repeated `LiveSessionModelSwitchError` loops, but `runWithModelFallback()` still treats that error as a normal failed candidate and walks the precomputed fallback list sequentially. The direct redirect/jump behavior described in #57471 is not implemented on `main`.

## Evidence

- **Fallback candidates are precomputed and iterated linearly:** `runWithModelFallback()` builds `candidates` once via `resolveFallbackCandidates(...)` and then uses `for (let i = 0; i < candidates.length; i += 1)`. There is no code in that loop that inserts or jumps directly to the model carried by `LiveSessionModelSwitchError`.
  - file: [src/agents/model-fallback.ts:727](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/agents/model-fallback.ts#L727)
  - command: `nl -ba src/agents/model-fallback.ts | sed -n '715,760p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **LiveSessionModelSwitchError is still recorded as a failed candidate:** On `err instanceof LiveSessionModelSwitchError`, current `main` wraps the switch error in `FailoverError`, stores it as `lastError`, calls `recordFailedCandidateAttempt(...)`, and then `continue`s to the next candidate. That is the behavior the issue reports as wasteful.
  - file: [src/agents/model-fallback.ts:926](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/agents/model-fallback.ts#L926)

- **Observation schema has no redirect decision:** The fallback decision logger only defines `skip_candidate`, `probe_cooldown_candidate`, `candidate_failed`, and `candidate_succeeded`. There is no `live_session_redirect` or equivalent event on `main`.
