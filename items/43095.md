## Summary

Keep #43095 open. Current main still returns `sessions_send` timeout as `{ runId, status: "timeout", error, sessionKey }` after the target agent run has already been accepted, without target-delivery metadata or a distinct delivered/no-reply state. The cross-referenced closed #53527 is the same symptom, but it does not supersede this remaining behavior on current main.

## Best Possible Solution

Keep the issue open and implement an additive clarification in the session-send contract: once the target `agent` RPC has accepted a run, a later reply wait timeout should return explicit target-delivery state, for example `status: "timeout"` plus `deliveryStatus: "accepted" | "delivered"` and `replyStatus: "timeout"`, or a clearer `status: "accepted"/"pending"` with compatibility preserved. Add regression coverage for an accepted `agent` call followed by `agent.wait` timeout, and update the session-tool docs to distinguish send acceptance, delivery failure, and reply wait timeout.

## Evidence

- **Issue discussion confirms the user-facing ambiguity:** The issue and marcus-daemon's comment describe a reproduced path where `sessions_send` returns `status: "timeout"` while the target run remains queued/running and can complete later. This is exactly a reply-wait timeout, not necessarily delivery failure.
- **Current timeout branch still conflates wait timeout with result status:** After `waitForAgentRunAndReadUpdatedAssistantReply(...)`, the `result.status === "timeout"` branch returns only `runId`, `status: "timeout"`, optional `error`, and `sessionKey`; it does not include `deliveryStatus`, target-delivery state, queued/running state, or the existing announce `delivery` metadata.

## Risks / Open Questions

- Changing the top-level `status` values may break existing agents that branch on `timeout`; an additive field such as `deliveryStatus` or `replyStatus` is safer.
- The implementation must not mark delivery as successful when the initial `agent` RPC fails before returning a `runId`; only post-acceptance wait timeouts should be classified as delivered/accepted/pending.
