## Summary

Current main still has the reported ambiguity: `sessions_send` starts/accepts the target agent run, waits via `agent.wait`, and returns plain `status: "timeout"` when the wait expires without adding delivery state or a pending/accepted distinction. The related #20939 fallback/rate-limit issue is secondary and closed; it does not supersede this remaining result-shape problem.

## Evidence

- **Issue discussion confirms mechanism:** The issue body and marcus-daemon comment describe a reproduced case where `sessions_send` timed out while the target run had been accepted/queued and could complete later; the requested fix is to distinguish delivery failure from reply-wait timeout.
