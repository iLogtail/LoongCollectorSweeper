## Summary

Keep #53526 open. Current main still defaults `resolveSessionResetPolicy()` to daily for group/thread sessions unless users explicitly configure reset policy, and the current regression test still asserts the group-with-dm-override case remains daily. The PR targets a real remaining core behavior gap raised by #32109, but its current patch is not merge-ready because review comments found the `hasExplicitReset` guard suppresses the new group/thread default when unrelated `resetByType` entries exist.

## Best Possible Solution

Keep the PR open, but require a corrected patch before merge: implement type-aware implicit defaults in `src/config/sessions/reset-policy.ts` so group/thread sessions use idle/10080 only when no channel override, base `session.reset`, or type-specific reset applies; do not let unrelated `resetByType.direct`/`dm` entries suppress that default. Add exact regression tests for no-config group/thread defaults, unrelated direct/dm overrides, direct-session daily behavior, and explicit group/thread overrides. Update the user-facing session docs and changelog if maintainers accept the behavior change.
