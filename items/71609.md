## Summary

Keep #71609 open. Current main still contains the browser Control UI auth behavior called out by the issue: stale cached device-token cleanup only happens in a narrow fallback case, `AUTH_TOKEN_MISMATCH` can still schedule reconnects when no bounded device-token retry was consumed, checked-in tests assert that reconnect behavior, and the gateway-facing device-token mismatch message still says to rotate/reissue rather than guiding stale browser credential cleanup. The related closed Control UI auth/signature reports cover partial overlap, but they do not resolve this scope-upgrade stale device-token loop/rate-limit path.

## Best Possible Solution

Keep the issue open and fix the Control UI browser auth recovery path. The likely maintainer path is to clear stale cached device auth on `AUTH_DEVICE_TOKEN_MISMATCH` and stale stored-token failures, stop automatic reconnects when no safe bounded retry is possible or after the one-shot retry is consumed, avoid burning the auth rate limiter, keep the app-level `clientInstanceId` stable, and replace the `rotate/reissue` mismatch wording with reconnect/stale identity cleanup guidance. Add focused browser tests for the scope-upgrade stale-token case and the non-looping recovery path.
