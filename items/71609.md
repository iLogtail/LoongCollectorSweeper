## Summary

Keep #71609 open. Current main has partial Control UI/gateway auth hardening, including URL/shared-token precedence and a bounded cached-device-token retry, but the specific stale browser device-token recovery path is not fully fixed: the browser clears cached device auth only in the explicit shared-token fallback case, `AUTH_DEVICE_TOKEN_MISMATCH` is not treated as terminal for reconnect scheduling, a checked-in browser test still preserves reconnecting after an auth mismatch when no retry was attempted, and the server/docs still contain the rotate/reissue guidance the issue calls misleading.

## Best Possible Solution

Keep the issue open and fix the browser Control UI auth recovery path. The likely maintainer path is to clear stale cached browser device auth when the server reports `AUTH_DEVICE_TOKEN_MISMATCH` for the stored-token path, stop automatic reconnect loops when no safe one-shot retry is available or after it fails, avoid burning the gateway auth rate limiter, preserve the stable app-level `clientInstanceId`, and replace `rotate/reissue` guidance with reconnecting/stale-identity cleanup guidance. Add focused browser tests for the scope-upgrade stale-token case and for the non-looping recovery path.
