## Summary

Keep #71609 open. Current `main` contains partial Control UI/gateway auth hardening for related stale-token and token/signature mismatch reports, including URL/shared-token precedence and a bounded cached-device-token retry, but the specific stale cached device-token recovery path can still loop: browser cleanup is narrow, `AUTH_TOKEN_MISMATCH` can still schedule reconnects when no retry was used, tests preserve that reconnect behavior, and the gateway still emits the rotate/reissue wording called out by the issue.

## Best Possible Solution

Keep the issue open and fix the browser Control UI auth recovery path. The likely maintainer path is to align `ui/src/ui/gateway.ts` with the protocol guidance and the stronger core client behavior: clear stale cached device auth when the server reports `AUTH_DEVICE_TOKEN_MISMATCH` or when stored-token auth is known stale, stop automatic reconnect loops after a failed one-shot retry or when no safe retry is possible, avoid burning the auth rate limiter, preserve the stable app-level `clientInstanceId`, and replace the gateway/client-facing `rotate/reissue` mismatch wording with reconnecting or stale identity cleanup guidance. Add focused browser tests for the scope-upgrade stale-token case and for the non-looping recovery path.
