## Summary

Keep #71609 open. Current main has partial Control UI auth hardening, including explicit token precedence and a stable app-level client instance id, but the stale cached device-token path reported here still is not fully handled: primary stored device-token mismatches are not cleared, `AUTH_DEVICE_TOKEN_MISMATCH` is not treated as terminal for browser reconnects, and the gateway-facing message still says to rotate/reissue the token.

## Best Possible Solution

Keep the issue open and fix the browser Control UI auth recovery path. In `ui/src/ui/gateway.ts`, clear stale cached device auth when `AUTH_DEVICE_TOKEN_MISMATCH` occurs for the active stored device-token path, stop or tightly bound automatic reconnects after stale-token auth failures to avoid rate-limit lockout, keep the stable app-level `clientInstanceId`, and replace the rotate/reissue wording with stale identity cleanup or reconnect guidance. Add focused browser tests for stale primary device-token mismatch after scope/device approval and for the non-looping recovery path.
