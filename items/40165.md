## Summary

Keep open. Current `main` has partial NO_REPLY hardening in agent reply normalization, but the gateway/display and outbound-delivery paths still treat mixed prefix/suffix NO_REPLY text as visible content rather than suppressing or stripping it.

## Evidence

- **Gateway control-token suppression is exact-only:** `isSuppressedControlReplyText` delegates to `isSilentReplyText`, which only matches the exact trimmed token. That does not cover `NO_REPLY ...` or `... NO_REPLY`.
  - file: [src/gateway/control-reply-text.ts:32](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/gateway/control-reply-text.ts#L32)
  - command: `nl -ba src/gateway/control-reply-text.ts | sed -n '29,60p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Outbound delivery test locks in visible suffix/prefix leakage:** Current tests explicitly expect `normalizeReplyPayloadsForDelivery` to keep both `NO_REPLY thanks for the update` and `thanks NO_REPLY` visible, while only dropping the exact sentinel.
  - file: [src/infra/outbound/payloads.test.ts:135](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/infra/outbound/payloads.test.ts#L135)
