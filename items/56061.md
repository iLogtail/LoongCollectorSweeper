## Summary

Current `main` already implements the requested Telegram long-poll hard timeout and related recovery hardening. The Telegram client now assigns a 45s hard timeout to `getUpdates`, wraps fetches with an aborting controller, has regression coverage for that abort path, and the changelog records the fix as shipped in `2026.3.22`.

## Close Comment

Closing as implemented on `main` after Codex review. Current OpenClaw already hard-times out stuck Telegram `getUpdates` requests: `extensions/telegram/src/request-timeouts.ts:6` sets `getupdates` to `45_000`, and the Telegram client fetch wrapper aborts method-specific requests so wedged long-poll calls do not wait for the stall watchdog (`extensions/telegram/src/bot-core.ts:181`). Regression coverage also checks that wrapped `getUpdates` fetches abort after 45s (`extensions/telegram/src/bot.fetch-abort.test.ts:72`), and polling recovery now bounds the persisted-offset confirmation probe as well (`extensions/telegram/src/polling-session.test.ts:283`). The fix is already called out in the shipped changelog under `v2026.3.22` (`CHANGELOG.md:2329`).
