## Summary

Keep open. Current main has the relevant plugin hook surfaces, and related prior work implemented `before_tool_call`, `after_tool_call`, and `agent_end`, but the typed hook payloads and runtime emitters still do not expose the requested structured progress metrics (`stepIndex`, `stepTotal`, `percent`, `etaSec`). This is a real plugin API gap, not a ClawHub-only item, because the issue is asking core to expose missing run-progress data to plugins.

## Best Possible Solution

Keep the issue open and design a small, backward-compatible progress contract. A good implementation would add an optional `PluginHookProgressMetrics` type to `src/plugins/hook-types.ts`, thread it through the relevant hook emitters, document it in `docs/plugins/hooks.md`, and add focused tests proving monotonic metrics, success-to-100 only when a step model exists, and no false 100% on failures/cancellations. Maintainers should decide whether the metrics belong directly on `before_tool_call`/`after_tool_call`/`agent_end` as proposed or on a dedicated progress observation hook, but current main does not yet satisfy the request.

## Risks / Open Questions

- Progress totals may be hard to define consistently across Pi, Codex/native harness, CLI, subagents, cancellations, and tool-only turns.
- A naive percent implementation could mislead users if plans change mid-run or if failed/cancelled runs are reported as complete.
