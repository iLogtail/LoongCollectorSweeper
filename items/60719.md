## Summary

Keep #60719 open. Current main has pre-compaction memory flush and opt-in session transcript indexing, but it still does not solve the reported short-session/heartbeat failure mode: default memory flush is threshold-driven, explicitly skips heartbeat runs, and the bundled session-memory hook only runs on /new or /reset rather than on heartbeat/session close or after N sessions/hours.

## Best Possible Solution

Keep the issue open for maintainer product judgment and a scoped implementation. The likely fix should live in the memory-core/Gateway boundary: add a configurable end-of-run or heartbeat-safe memory persistence path, or count/time-based forced flush/compaction, with tests proving isHeartbeat=true short sessions persist durable notes without needing context-pressure compaction. Docs should clarify how heartbeat isolatedSession, cron/task sessions, memoryFlush, and session transcript indexing interact.

## Risks / Open Questions

- A custom ClawHub/plugin implementation may be the preferred product direction, since plugin hooks can observe session lifecycle, but the current bundled/default behavior still leaves the reported heartbeat path unsolved.
- Opt-in session transcript indexing can improve recall for users who enable it, but it is asynchronous, off by default, and not equivalent to a guaranteed heartbeat/session-close memory write.
