## Summary

Keep open. Current main has meaningful mitigations for Active Memory timeouts, but it still runs Active Memory as a foreground blocking before-prompt hook and uses the configured timeout as the user-turn wait budget. The issue asks for direct-chat turns not to block for about 30 seconds; with `timeoutMs: 30000`, current code can still spend that full budget before failing open.

## Best Possible Solution

Keep the issue open until Active Memory either gains a non-blocking/async direct-chat mode, a separate small fail-open foreground budget with background recall, or a documented readiness/prewarm gate that prevents the first direct-chat turn from paying cold memory/indexing cost. If maintainers prefer narrower tracking, explicitly mark #67878 and/or #69892 as canonical only after confirming they cover the remaining direct-chat blocking behavior.
