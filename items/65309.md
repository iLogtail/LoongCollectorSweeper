## Summary

Keep open. Current main has useful mitigations for Active Memory, including toolsAllow forwarding and a hard timeout deadline, but the central reported behavior remains: Active Memory still runs in the foreground before prompt build and can block a direct-chat turn until the configured timeoutMs fires. The issue also has fresh 2026-04-26 confirmation, so the remaining product bug is not conclusively obsolete.

## Best Possible Solution

Keep #65309 open as the canonical Active Memory direct-chat latency/reliability tracker unless a maintainer intentionally splits it. A real fix should make Active Memory safe for primary chat by adding a non-blocking/background recall mode, a much smaller foreground fail-open budget with optional late/background memory update, or an explicit prewarm/readiness gate so cold memory/indexing cost is not paid inside the first user turn.
