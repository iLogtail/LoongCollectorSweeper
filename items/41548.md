## Summary

Keep open. Current main intentionally routes replies back to the originating channel and explicitly documents that session history is not fully synced back to every client, so the requested unified cross-surface inbox is not implemented. The behavior is in OpenClaw core scope, not a ClawHub/plugin-only idea.

## Risks / Open Questions

- The issue body mixes a by-design routing behavior (`TUI -> Telegram` route preservation) with a real product gap (full cross-surface transcript sync). Closing it would discard the unresolved feature request.
- Any future implementation needs explicit privacy and session-boundary rules, because current session settings (`dmScope`, `identityLinks`) already control cross-channel context sharing and can make a unified inbox security-sensitive.
