## Summary

Current main still has the reported behavior for raw Markdown placed under `sources/`: compile reads every non-index `.md` file in `sources/`, `toWikiPageSummary` infers `kind: "source"` from the path while leaving `id`, `pageType`, and `updatedAt` empty without YAML frontmatter, and lint unconditionally reports missing `id`, missing `pageType`, and unknown freshness for every non-report page. The supported ingest/bridge/unsafe-local paths wrap imported content with source frontmatter, so those paths avoid the lint errors, but the docs still describe `sources/` as imported raw material and do not clearly say raw files are invalid. This needs a maintainer product/docs decision rather than cleanup closure.

## Best Possible Solution

Keep the issue open for a focused maintainer decision. The best fix is either to document that `sources/` contains OpenClaw source pages whose raw imported content is wrapped with required provenance frontmatter, or to add an explicit raw-source mode/path convention that lint, compile, search, and docs all understand. If raw mode is chosen, add targeted tests proving raw `sources/*.md` files do not produce missing-frontmatter lint errors while generated bridge/ingest source pages still get provenance validation.

## Risks / Open Questions

- Changing lint to skip all `sources/` frontmatter checks could weaken provenance, duplicate-id, freshness, and bridge-page validation for generated source pages.
- Changing docs only would preserve current behavior, but may disappoint users who interpreted `sources/` as a raw evidence folder rather than an OpenClaw-wrapped source-page folder.
