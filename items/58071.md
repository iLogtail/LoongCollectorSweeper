## Summary

Keep open. Current `main` still hardcodes a 30s timeout for the agent `image` tool path, and the media-understanding runtime still aborts the provider call when that timeout elapses. The shipped `#55072` change was for embedded-runner LLM idle-stream timeouts, not this image-tool/media-understanding path.

## Evidence

- **Image tool still hardcodes 30s per request:** `runImagePrompt(...)` passes `timeoutMs: 30_000` on every image-analysis call path (single image, multi-image batch, and per-image fallback loop).
  - file: [src/agents/tools/image-tool.ts:221](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/agents/tools/image-tool.ts#L221)
  - command: `nl -ba src/agents/tools/image-tool.ts | sed -n '180,290p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Abort is still enforced in media-understanding runtime:** `describeImagesWithModel(...)` creates an `AbortController` and calls `controller.abort()` after `params.timeoutMs`, so a slow vision provider request is still force-aborted when the timeout is hit.
  - file: [src/media-understanding/image.ts:278](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/media-understanding/image.ts#L278)
