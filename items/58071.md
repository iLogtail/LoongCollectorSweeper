## Summary

Current `main` does not appear to implement this fix. The `image` tool still hardcodes a 30s timeout for vision requests and the media-understanding path still aborts the underlying model call when that timer fires. The reporter's Gateway-RPC explanation looks outdated for current `main` because `chat.send`/`expectFinal` requests are no longer auto-timed out by the default 30s client RPC watchdog, but the user-visible bug remains plausible in the image-tool path itself.

## Evidence

- **Image tool still hardcodes 30s vision timeout:** `runImagePrompt()` passes `timeoutMs: 30_000` into both `describeImages()` and `describeImage()` for single-image, multi-image, and per-image fallback paths, so slow vision responses still hit a fixed 30s cap.
