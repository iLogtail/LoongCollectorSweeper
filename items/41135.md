## Summary

Keep open. Current main has useful auth-profile ordering, cooldown, and retry-time rotation, but it does not implement the requested opt-in provider-pool routing layer with configurable strategies, per-profile concurrency/in-flight state, pool inspectability, or batch-worker stickiness semantics. The issue is concrete, recent, author is not protected, and only has the stale label, so it should remain available for maintainer product/API design rather than be auto-closed.

## Best Possible Solution

Keep the issue open and scope it as a maintainer design item: either add an opt-in core/provider-runtime pool routing contract that extends auth.order/cooldown with explicit strategies, task stickiness, in-flight/concurrency accounting, status JSON, and tests, or first add a documented plugin API that can implement those semantics without bypassing core auth safety.
