## Summary

Keep open. Current main still accepts arbitrary Nostr relay strings at the config schema, only does protocol checks in setup surfaces, and forwards configured relays into the Nostr runtime. The PR addresses a real remaining gap, and its latest discussion says the earlier remote ws:// concern was addressed, but it needs refresh against current main and should share one validator across schema, CLI setup, and Control UI setup.

## Best Possible Solution

Keep the PR open and refresh it against current main. The best implementation is one shared Nostr relay URL validator that allows wss:// remote relays and ws:// only for localhost/loopback, then uses it from NostrConfigSchema, CLI setup, Control UI setup, and focused regression tests in the current Nostr test files.
