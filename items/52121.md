## Summary

Keep open. Current `main` does not implement the requested startup conflict detection, and the proposed PR shape still appears incomplete against OpenClaw's current per-agent/runtime config resolution and startup error semantics.

## Evidence

- **No startup conflict detector on current main:** `startGatewayServer()` loads `cfgAtStart` from `prepareGatewayStartupConfig()` and continues startup, but there is no `detectConfigConflicts()`/`logConfigConflicts()` hook or equivalent warning pass at that point.
  - file: [src/gateway/server.impl.ts:289](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/gateway/server.impl.ts#L289)

That means current conflict/exposure guidance is manual/read-only health output, not automatic startup feedback.
  - file: [src/flows/doctor-health-contributions.ts:290](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/flows/doctor-health-contributions.ts#L290)

That is useful evidence that the UX gap is specifically about startup surfacing, not that the whole problem space is absent.

Calling that unconditionally during gateway startup would turn some config states into boot failures rather than non-fatal warnings.
