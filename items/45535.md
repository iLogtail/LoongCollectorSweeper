## Summary

Keep open. Current `main` still has the reported normalization gap for OpenAI-compatible usage, the CLI-side alias support is still missing, and the PR as discussed would need rework against current parser semantics to avoid overwriting previously captured nonzero usage.

## Evidence

- **Current main still prefers `input_tokens` over `prompt_tokens` even when the former is zero:** `normalizeUsage` still resolves `rawInputValue` with a nullish chain: `raw.input ?? raw.inputTokens ?? raw.input_tokens ?? raw.promptTokens ?? raw.prompt_tokens ...`. That means a payload like `{ input_tokens: 0, prompt_tokens: N }` normalizes to `0`, not `N`.
  - file: [src/agents/usage.ts:114](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/agents/usage.ts#L114)
  - command: `nl -ba src/agents/usage.ts | sed -n '100,170p'`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **CLI parser on main still lacks `prompt_tokens` / `completion_tokens` fallback:** The current CLI usage reader only looks at `input_tokens` / `inputTokens` and `output_tokens` / `outputTokens`; it does not read `prompt_tokens`, `promptTokens`, `completion_tokens`, or `completionTokens`, so the CLI half of the report is not implemented on `main`. The equivalent logic now lives in `cli-output.ts`, not the PR's older `cli-runner/helpers.ts` path.
  - file: [src/agents/cli-output.ts:123](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/agents/cli-output.ts#L123)

- **Current parsers retain prior usage only when later reads return `undefined`:** Both the streaming parser and the JSONL parser update usage with `usage = ... ?? usage`. If the helper is changed to return all-zero usage objects instead of `undefined`, later bookkeeping events can replace earlier nonzero totals.

## Risks / Open Questions

- The underlying bug still appears real on current `main` for OpenAI-compatible providers that emit both `input_tokens: 0` and a positive `prompt_tokens` value.
- The CLI path still misses `prompt_tokens` / `completion_tokens` aliases, so usage can remain wrong even if core normalization is fixed elsewhere.
- A straightforward rebase of this PR onto current files would need extra guards so zero-valued terminal events do not erase earlier nonzero usage in JSONL streams.
