## Summary

Current main has related fixes for stale/auto fallback model state, but the specific greeting mismatch is not high-confidence resolved. Bare /new and /reset greetings are still generated from a prompt that tells the model to mention default_model when runtime model differs, while the system prompt separately exposes both the active runtime model and configured default. User-selected overrides are also intentionally preserved across resets, so a runtime/default mismatch can still be a valid state.

## Best Possible Solution

Keep the issue open and make the reset greeting deterministic about model identity: either ensure bare /new and /reset switch the active runtime to the configured default before the greeting, or change the reset prompt/status prelude to say the active runtime model and default model separately. Add a regression test with configured default minimax-portal/MiniMax-M2.5 and active/preserved runtime volcengine/ark-code-latest so the prompt or reply path cannot imply the default is the current runtime model.
