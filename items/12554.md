## Summary

Keep #12554 open. Current main still has useful per-process-session machinery and subagent-run kill/cleanup paths, but it does not expose the requested session-wide Gateway/CLI operator surface to list and terminate all running exec processes for a target OpenClaw session key. The existing `sessions.abort` only aborts active chat work, and background exec sessions intentionally survive tool-call aborts unless killed one process session at a time through the agent `process` tool.

## Best Possible Solution

Keep the issue open and implement an additive operator surface: a Gateway/protocol method such as `sessions.exec-list` plus `sessions.kill --kill-exec`, or an explicit `sessions.abort` option, with CLI wiring and docs. It should resolve the canonical session key, list active process registry/supervisor records scoped to that session, terminate all matching records using supervisor cancellation plus process-tree fallback, enforce operator/requester authorization, and add regression tests for same-session kill, unrelated-session isolation, background exec behavior, and CLI/protocol validation.

## Risks / Open Questions

- A correct implementation needs explicit operator/requester authorization so one session cannot inspect or terminate another session's exec processes.
- The implementation should preserve existing semantics where ordinary foreground chat/tool abort does not silently kill intentionally backgrounded exec sessions unless the caller requests the stronger kill behavior.
- Process termination should use the existing supervisor/process-tree machinery and prove that unrelated sessions with different `scopeKey`/session keys are not affected.
