## Summary

Keep open. The report is recent, externally authored, and the current main code still has the exact product gap: a saved MCP server with a URL but no OpenClaw `transport: "streamable-http"` is resolved as legacy SSE, while the issue's working proof is a POST/HTTP MCP endpoint and Claude Code `type: "http"` path. Current main has no auto-detect/fallback from SSE to streamable HTTP and still waits up to the per-server connection timeout before logging failure.

## Best Possible Solution

Keep this issue open and fix the bundled embedded MCP path. The likely maintainer path is to either canonicalize CLI-native `type: "http"` to OpenClaw `transport: "streamable-http"` for `mcp.servers`, or reject it with a clear diagnostic, and add a regression test using a streamable HTTP MCP server at `/mcp`. A stronger fix would also consider SSE-to-streamable detection/fallback for URL-only hosted MCP endpoints and avoid serial gateway-startup stalls from failed remote MCP handshakes.
