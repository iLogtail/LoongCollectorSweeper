## Summary

Keep open. Current `main` still sends raw telephony text through the voice-call TTS path without parsing `[[tts:...]]` directives or applying per-call overrides, so the requested feature is not implemented on `main`.

## Evidence

- **Twilio telephony playback forwards raw text:** `playTtsViaStream()` accepts `text` and races `ttsProvider.synthesizeForTelephony(text)` directly. There is no `parseTtsDirectives()` call, no cleaned-text step, and no per-call override merge before synthesis.
  - file: [extensions/voice-call/src/providers/twilio.ts:620](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/voice-call/src/providers/twilio.ts#L620)
  - command: `nl -ba extensions/voice-call/src/providers/twilio.ts | sed -n '540,740p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Telephony TTS provider only supports static config:** `TelephonyTtsProvider.synthesizeForTelephony` only accepts `(text: string)`. `createTelephonyTtsProvider()` computes one `mergedConfig` up front, then always calls `runtime.textToSpeechTelephony({ text, cfg: mergedConfig })`, so there is no API for per-call directive overrides.
  - file: [extensions/voice-call/src/telephony-tts.ts:22](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/voice-call/src/telephony-tts.ts#L22)
  - command: `nl -ba extensions/voice-call/src/telephony-tts.ts | sed -n '1,220p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Telephony runtime params have no override channel:** The general TTS request type includes `overrides?: TtsDirectiveOverrides`, but `TtsTelephonyRequestParams` only has `text`, `cfg`, and `prefsPath`. That matches the current voice-call implementation gap.
  - file: [src/plugin-sdk/tts-runtime.types.ts:65](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/plugin-sdk/tts-runtime.types.ts#L65)
  - command: `nl -ba src/plugin-sdk/tts-runtime.types.ts | sed -n '1,150p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Message-channel TTS already parses directives:** `maybeApplyTtsToPayload()` calls `parseTtsDirectives()`, uses `cleanedText`, and passes `directives.overrides` into `textToSpeech()`. This contrast supports the report that the directive infrastructure exists elsewhere but is not wired into telephony.
  - file: [extensions/speech-core/src/tts.ts:1054](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/speech-core/src/tts.ts#L1054)
  - command: `nl -ba extensions/speech-core/src/tts.ts | sed -n '1030,1175p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Voice-call docs describe static deep-merge config, not directive parsing:** The voice-call docs say call TTS uses core `messages.tts` config and plugin `tts` overrides with the same shape via deep-merge. They do not describe model-driven per-reply directive handling on the telephony path.
  - file: [docs/plugins/voice-call.md:413](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/docs/plugins/voice-call.md#L413)
