## Summary

Current main still has the reported Control UI/backend mismatch. The Agents view's Set Default action writes `agents.defaultId`, while the strict config schema only accepts `agents.defaults` and `agents.list`, with the supported default marker living on `agents.list[].default`. The runtime still resolves default agent from that per-agent flag or, if absent, the first `agents.list` entry, matching the reporter's observed fallback. Related reports in the provided context cover model selection/save behavior, but they do not supersede this default-agent persistence bug.

## Best Possible Solution

Keep the issue open. The Control UI should persist Set Default by updating `agents.list`: ensure the target agent entry exists, set that entry to `default: true`, clear `default` from the other entries, then save through the existing config path. Add a focused UI/controller regression test that clicks or invokes Set Default and asserts the saved raw config uses `agents.list[].default` and not `agents.defaultId`; add a schema/runtime assertion if needed to protect the contract.
