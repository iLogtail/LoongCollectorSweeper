## Summary

Keep open. Current main has session maintenance and fresh-cron sanitization mitigations, but the central reported problem is still present: `SessionSkillSnapshot` still contains the rendered `prompt`, normal session setup persists that snapshot, isolated cron persists a cloned run entry under a `:run:` key, and the session store still serializes and caches the full entry payload. There is even a current test asserting the cron run snapshot copy rather than guarding against prompt duplication.

## Best Possible Solution

Keep this issue open and fix the storage contract: stop persisting rendered `skillsSnapshot.prompt` in long-lived session entries and per-run cron rows, persist only lightweight identity/metadata such as version, filter, skill names, and a hash, and rebuild or cache the rendered prompt lazily for runtime use. Add regression coverage that isolated cron base/run entries and reset/follow-up paths do not duplicate large rendered prompt payloads in `sessions.json`. If maintainers want one canonical tracker, consolidate explicitly with the broader open #55334 rather than auto-closing this detailed repro.
