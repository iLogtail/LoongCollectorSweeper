## Summary

Keep #62249 open. Current main still does not override the MCP SDK request timeout on bundled MCP tool execution, so the central bug fixed by the PR remains present. The patch should likely be refreshed with a regression test, and its progress-reset claim should be checked because SDK v1.29.0 only registers a progress token/handler when `onprogress` is provided.

## Best Possible Solution

Keep the PR open or replace it with an equivalent main-branch fix. The likely best implementation is to pass explicit MCP SDK `RequestOptions` from `src/agents/pi-bundle-mcp-runtime.ts` when invoking `client.callTool`, add a focused regression test that asserts the third argument includes the longer timeout, and separately decide whether progress reset needs a no-op `onprogress` handler, a `maxTotalTimeout`, or a documented fixed timeout only.
