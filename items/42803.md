## Summary

Keep open. Current main has a Feishu sequential-key path that lets `/stop` abort requests use a separate `:control` lane, but the reported command set is not fully fixed: `/status` is explicitly tested to stay on the normal chat lane, and `/new` also falls through to the normal lane because the Feishu key resolver only special-cases abort and `/btw` text. That means `/status` and `/new` can still sit behind an active same-chat Feishu run.

## Best Possible Solution

Keep #42803 open until Feishu command-only queue semantics are settled and implemented consistently. A likely fix belongs in `extensions/feishu/src/sequential-key.ts` plus Feishu tests: route safe/read-only status commands such as `/status` to a dedicated control lane like Telegram, decide whether session-reset commands such as `/new` should also bypass or intentionally remain serialized, and add concurrency tests proving the chosen commands execute while a normal same-chat task is blocked.
