## Summary

Keep #9865 open. Current main still lacks first-class provider Batch API routing for background LLM turns in cron, heartbeat, or subagent execution. Existing batch support is scoped to memory embeddings and other non-LLM mechanics, while the provider/plugin contract is still immediate StreamFn-based. The related #49326 is closed and the existing discussion treats #9865 as the canonical tracker, not something superseded.

## Best Possible Solution

Keep #9865 open as the canonical product/API tracker. The likely maintainer path is a provider/runtime batch-turn contract with submit, poll, cancel, timeout, reconcile, and delayed-delivery semantics, then explicit opt-in surfaces for cron jobs, heartbeat runs, sessions_send, and sessions_spawn/subagents, plus docs and regression coverage. Treat #49326 as a closed duplicate/follow-up reference rather than a replacement.
