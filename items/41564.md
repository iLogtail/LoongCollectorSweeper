## Summary

Current `main` already supports re-exposing `cron` to subagent sessions via `tools.subagents.tools.allow` or `tools.subagents.tools.alsoAllow`, and the `cron` tool still includes the `wake` action.

## Evidence

- **Subagent policy now honors explicit allow/alsoAllow entries before default denies:** `resolveSubagentToolPolicy()` and `resolveSubagentToolPolicyForSession()` build `explicitAllow` from both `allow` and `alsoAllow`, then remove matching tools from the subagent base deny list before returning the policy. That means `cron` is no longer unconditionally filtered out when explicitly allowlisted.
  - file: [src/agents/pi-tools.policy.ts:81](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/src/agents/pi-tools.policy.ts#L81)
- **Cron tool still exposes wake:** The built-in `cron` tool description/action list includes `wake` as a supported action, so surfacing `cron` to a subagent also surfaces the requested wake behavior.
  - file: [src/agents/tools/cron-tool.ts:421](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/src/agents/tools/cron-tool.ts#L421)
- **Regression tests cover subagent re-allow behavior:** Tests verify that `tools.subagents.tools.alsoAllow` and `tools.subagents.tools.allow` can re-enable tools from the default subagent deny set, which is the mechanism needed for `cron`.
  - file: [src/agents/pi-tools.policy.test.ts:55](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/src/agents/pi-tools.policy.test.ts#L55)
- **Released build contains the same implementation:** The latest release commit already contains the same `explicitAllow` logic in `src/agents/pi-tools.policy.ts`, so this is not just unreleased `main` behavior.
  - command: `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:src/agents/pi-tools.policy.ts | sed -n '75,110p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Risks / Open Questions

- `cron` remains owner-only; this fix applies to owner-authorized subagent runs, not untrusted/non-owner senders.

## Close Comment

Closing from a Codex review because this request is already implemented on current `main` and in the released `v2026.4.22` build. In `src/agents/pi-tools.policy.ts:81`, subagent tool policy now folds both `tools.subagents.tools.allow` and `tools.subagents.tools.alsoAllow` into `explicitAllow`, then removes explicitly allowed tools from the default subagent deny list before returning the policy. That means `tools.subagents.tools.alsoAllow: ["cron"]` or `allow: ["cron"]` now re-exposes `cron` for owner-authorized subagent sessions. The `cron` tool still includes the `wake` action (`src/agents/tools/cron-tool.ts:421`), and the re-allow behavior is covered in `src/agents/pi-tools.policy.test.ts:55`. If you still hit the old behavior on a current build, please open a fresh issue with the exact config and whether the subagent run is owner-authorized.
