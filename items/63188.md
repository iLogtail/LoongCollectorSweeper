## Summary

Keep open. Current main preserves provider `stopReason: "length"` in parts of the model transport and exposes it in run metadata, but the native subagent completion path still treats a normal lifecycle end as successful and can announce the latest assistant text as "completed successfully" even when that text is partial. There is no `onLengthTruncation` / `maxContinuations` config surface. The related reports in the thread are useful corroboration, but #63210 is the complementary main-session tracker rather than a replacement for this subagent-specific request.

## Best Possible Solution

Keep this issue open and implement it in the core agent/subagent runtime. A conservative first phase should treat `stopReason:"length"` in native `mode:"run"` subagents as a partial terminal failure, propagate that through `agent.wait`, subagent registry outcome, task status, and completion announce text, and expose an opt-in config surface. A later phase can add bounded continuation prompts with `maxContinuations`, gated to avoid replaying unsafe partial tool calls or side effects. Coordinate but do not collapse this into #63210, which tracks the main interactive-session variant.

## Risks / Open Questions

- A future continuation implementation needs a side-effect guard: continuing after partial tool emission or external writes can duplicate actions unless the runtime first distinguishes safe text continuation from unsafe partial tool calls.
