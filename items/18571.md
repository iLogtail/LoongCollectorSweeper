## Summary

Keep open. Current main has a `message_sending` hook that can rewrite or cancel outbound content, but the shared outbound delivery path still does not pass `sessionKey` or `runId` into that hook, and the hook event has no tool-call history/context payload. That leaves the core request for deterministic pre-delivery verification correlation unresolved.

## Evidence

- **Shared outbound hook invocation omits session/run correlation:** `applyMessageSendingHook` builds the `message_sending` event with `to`, `content`, reply/thread fields, and metadata, then passes a context containing only `channelId`, `accountId`, and `conversationId`; no `sessionKey` or `runId` is supplied on this central delivery path.
  - file: [src/infra/outbound/deliver.ts:666](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/infra/outbound/deliver.ts#L666)
  - command: `nl -ba src/infra/outbound/deliver.ts | sed -n '666,745p;1000,1075p'`
  - sha: [bd796d1c85c7](https://github.com/openclaw/openclaw/commit/bd796d1c85c7dec0dcb89ed11502e4271ded1314)
- **Central delivery call site has no session/run arguments for message_sending:** The call to `applyMessageSendingHook` from `deliverOutboundPayloads` passes payload, target, channel, account, reply/thread data only, so even when `params.session` or mirror context exists nearby it is not forwarded to the plugin hook.
  - file: [src/infra/outbound/deliver.ts:1000](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/infra/outbound/deliver.ts#L1000)
