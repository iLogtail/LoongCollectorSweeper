## Summary

Keep #71593 open. Current main has SecretRef support for the built-in env/file/exec providers and manifest-declared plugin config SecretInput paths, but it still does not expose a plugin-defined SecretRef provider contract and does not provide universal SecretInput coverage for core schemas. The related AWS Secrets Manager provider PR #43643 was closed unmerged in the provided GitHub context, and current main does not contain an equivalent provider.

## Best Possible Solution

Keep this open as a core/plugin API design issue. Maintainers should either document env/file/exec, especially exec, as the supported external SecretRef provider model, or add a versioned plugin SecretRef-provider contract. If adding the contract, update core SecretRef types and Zod schemas, plugin SDK/runtime registration, Gateway protocol schemas, resolver dispatch and validation, docs, tests, and remaining requested SecretInput surfaces such as skills.entries.*.env.

## Risks / Open Questions

- Treating this as ClawHub-only would be premature because the requested plugin-defined SecretRef provider requires a missing public core/plugin/Gateway contract, not just an external plugin package.
- Adding the requested capability would touch public config shape, Zod schemas, Gateway protocol schemas, SDK/runtime registration, resolver dispatch, docs, and tests; it needs maintainer design rather than conservative cleanup.
- The existing exec provider may be the intended extensibility model, but that is not equivalent to a plugin-defined provider contract and would need explicit maintainer documentation if chosen as the answer.
