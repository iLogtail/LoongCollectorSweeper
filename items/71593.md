## Summary

Keep #71593 open. Current main supports SecretRef primitives for env/file/exec sources and manifest-declared plugin config SecretInput paths, including external plugins, but it still does not expose a plugin-defined SecretRef provider contract or universal SecretInput coverage for core schemas. The previously linked AWS Secrets Manager PR #43643 was closed unmerged and no equivalent AWS Secrets Manager provider exists on current main.

## Best Possible Solution

Keep this open as a core/plugin API design issue. Maintainers should either document env/file/exec, especially exec, as the supported extensibility model, or add a versioned plugin SecretRef-provider contract. If adding the contract, update core SecretRef types and Zod schemas, plugin SDK/runtime registration, Gateway protocol schemas, resolver dispatch and validation, docs, tests, and remaining requested SecretInput surfaces such as `skills.entries.*.env`.

## Risks / Open Questions

- Adding plugin-defined SecretRef providers would change public config, SDK, and Gateway protocol contracts; it needs explicit maintainer design rather than conservative cleanup.
- The existing exec provider may be the intended extensibility answer, but that would need maintainer confirmation and documentation because it is not the same as a plugin-defined provider contract.
