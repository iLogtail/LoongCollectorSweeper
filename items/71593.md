## Summary

Keep #71593 open. Current main still has a closed SecretRef provider/source model (`env`, `file`, `exec`) and no plugin-defined SecretRef provider contract. Plugin manifest `secretInputs` help plugin config fields, but they only mark paths under `plugins.entries.<id>.config`; they do not let a plugin implement a new provider source and they do not cover the requested `skills.entries.*.env` surface.

## Best Possible Solution

Keep this open as a core/plugin API design issue. Maintainers should decide whether `exec` is the supported universal secret-provider extension point and document that clearly, or add a versioned plugin SecretRef-provider contract. If adding the contract, align core SecretRef types/Zod schemas, plugin SDK/runtime registration, Gateway protocol schemas, resolver dispatch, config validation ordering, docs, tests, and remaining core SecretInput surfaces such as `skills.entries.*.env`.

## Risks / Open Questions

- Adding plugin-defined SecretRef providers would affect public config, plugin SDK, resolver behavior, Gateway protocol schemas, docs, and tests, so this needs maintainer API design rather than conservative cleanup.
- The existing exec provider can cover many external secret managers today, but closing this as implemented would leave the requested plugin-defined provider model and `skills.entries.*.env` coverage unresolved.
