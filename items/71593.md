## Summary

Keep #71593 open. Current main supports built-in SecretRef sources (`env`, `file`, `exec`) and manifest-declared plugin config SecretInput paths, but it still does not provide a plugin-defined SecretRef provider contract or universal core-schema coverage. The linked AWS Secrets Manager PR #43643 was closed unmerged, so it does not supersede this design request.

## Best Possible Solution

Keep this open as a core/plugin API design issue. Maintainers should either document env/file/exec, especially exec, as the supported extensibility model, or add a versioned plugin SecretRef-provider contract. If adding the contract, update SecretRef types and Zod schemas, plugin SDK/runtime registration, resolver dispatch, Gateway protocol schemas, docs, tests, and any intended additional SecretInput surfaces such as `skills.entries.*.env`.

## Risks / Open Questions

- The existing `exec` provider may be the intended extensibility answer, but that is a maintainer product decision and is not the same as a plugin-defined provider contract.
- Adding plugin-defined SecretRef providers would change public config, SDK, resolver, and Gateway protocol contracts, so it needs explicit design rather than conservative cleanup.
