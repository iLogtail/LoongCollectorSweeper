## Summary

Keep #71593 open. Current main has useful SecretRef primitives and plugin-declared SecretInput path support, including external plugin config paths, but it still does not expose a plugin-defined SecretRef provider contract and does not cover all requested core schema surfaces such as `skills.entries.*.env`. The linked AWS Secrets Manager PR #43643 is closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep this open as a core/plugin API design issue. Maintainers should decide whether to document `exec` providers as the supported external secret-provider extension point, or add a versioned plugin SecretRef-provider contract. If adding the contract, update core SecretRef types and Zod schemas, plugin SDK/runtime registration, Gateway protocol schemas, resolver dispatch and validation, docs, tests, and remaining requested SecretInput surfaces such as `skills.entries.*.env`.

## Risks / Open Questions

- Adding plugin-defined SecretRef providers would touch public config, plugin SDK, Gateway protocol schemas, resolver dispatch, docs, and tests, so it needs explicit maintainer design rather than conservative cleanup.
- The existing exec provider may be the intended extensibility answer, but that is a product/docs decision and is not the same as a plugin-defined provider contract.
