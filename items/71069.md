## Summary

Current main has Gemma 4 support for the canonical Google model id and has documented paths for local/self-hosted models, but it does not clearly resolve the reporter's exact case. A stored/session override outside the agent allowlist still triggers the reported generic reset message, and the issue asks for either accepting this specific variant or explaining why it is blocked. That remaining UX/config ambiguity is actionable enough to keep open.

## Best Possible Solution

Leave open and ask for the reporter's model config/session override source and intended backend. If the intended model is Google Gemma 4, guide or normalize toward the canonical `google/gemma-4-26b-a4b-it` form. If it is a local GGUF model, confirm it is configured under a provider and allowlisted. Independently, improve the reset message to include the rejected provider/model ref plus a `/models` or local-model docs hint.
