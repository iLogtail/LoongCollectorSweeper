## Summary

Keep open. Current main gates cross-agent session messaging behind explicit config and records sender provenance metadata, but it does not implement HMAC/token-based mutual authentication or any cryptographic verification for `sessions_send` provenance. The request is still unimplemented and belongs in core session/gateway security, not ClawHub.

## Evidence

- **`sessions_send` does not accept any auth/signature material:** The tool schema exposes only `sessionKey`, `label`, `agentId`, `message`, and `timeoutSeconds`; there is no token, signature, or HMAC field in the public tool contract.
  - file: [src/agents/tools/sessions-send-tool.ts:37](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/agents/tools/sessions-send-tool.ts#L37)
  - command: `nl -ba src/agents/tools/sessions-send-tool.ts | sed -n '37,48p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Built-in sender identity is runtime-bound metadata, not cryptographic auth:** `createOpenClawTools` passes the current `agentSessionKey` into `createSessionsSendTool`, and `sessions_send` injects that value into `extraSystemPrompt` and `inputProvenance.sourceSessionKey`. That means the built-in tool path already binds provenance to the current session context, but only as metadata.
  - file: [src/agents/tools/sessions-send-tool.ts:270](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/agents/tools/sessions-send-tool.ts#L270)

- **Gateway protocol provenance fields are unsigned metadata only:** `InputProvenanceSchema` contains only `kind`, `originSessionId`, `sourceSessionKey`, `sourceChannel`, and `sourceTool`. There is no signature, MAC, nonce, or verification field in the wire schema.
  - file: [src/gateway/protocol/schema/primitives.ts:22](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/gateway/protocol/schema/primitives.ts#L22)

## Risks / Open Questions

- Closing would overstate current guarantees: main has provenance tagging and access gating, but no cryptographic sender verification for inter-agent session messages.
- This is a recent security hardening request (opened 2026-03-30), so `stale_insufficient_info` does not apply.
- The request targets core session/gateway behavior; ClawHub/plugin closure would be a scope mismatch because the missing primitive is protocol/authentication, not optional capability packaging.
