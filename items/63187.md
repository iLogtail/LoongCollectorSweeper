## Summary

Current `main` exposes token and aggregate latency telemetry, but not a documented TTFT metric or a Pi subscribe callback payload that includes first-chunk timing. The request remains plausible and not clearly implemented end-to-end on `main`.

## Evidence

- **Usage/docs surface lists token usage but no TTFT field:** User-facing usage docs describe `/status`, `/usage`, and transcript-backed token/cost recovery for input/output/cache counters, with no TTFT or first-token metric documented.
  - file: [docs/reference/token-use.md:62](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/docs/reference/token-use.md#L62)
  - command: `nl -ba docs/reference/token-use.md | sed -n '50,110p'`
  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **Current latency summary is full-turn latency stats only:** `SessionLatencyStats` contains only aggregate latency fields (`count`, `avgMs`, `p95Ms`, `minMs`, `maxMs`), with no TTFT/first-chunk field.
  - file: [src/infra/session-cost-usage.types.ts:81](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/infra/session-cost-usage.types.ts#L81)

  - file: [src/infra/session-cost-usage.ts:626](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/infra/session-cost-usage.ts#L626)

## Risks / Open Questions

- There is partial underlying timing data on the internal agent-event bus, so a maintainer could reasonably decide this is close to implemented for some consumers, but the requested Pi streaming callback/API surface is still incomplete.
- I did not find a public doc or test asserting TTFT availability, so closing as implemented would be weak.
