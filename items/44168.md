## Summary

Keep open. Current main does not implement the PR's OpenAI WebSocket first-response watchdog: the stream options only carry manager options and an abort signal, the WS run waits on abort/close/message events after send without an early first-server-event timer, and existing fallback tests cover connect/error/send failures rather than a sent request that receives no initial server event. The PR discussion is a positive signal: reviewers treated the fix as scoped and useful, with earlier bot review concerns addressed in the PR branch.

## Best Possible Solution

Keep the PR open and rebase the same product fix onto current main: add a first-server-event watchdog inside createOpenAIWebSocketStreamFn, make auto transport reset/degrade the WS session and fall back to HTTP without duplicate start events, keep forced websocket mode surfacing a timeout, and wire a derived timeout budget through the current stream-resolution path with regression tests for no initial event, disabled/edge timeout values, forced websocket errors, late WS completions, and signal forwarding.
