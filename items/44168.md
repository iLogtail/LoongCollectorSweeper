## Summary

Current `main` does not contain the first-response WebSocket watchdog or the earlier auto-fallback behavior described in PR #44168. The OpenAI WS path still waits for normal WS events or the outer run/idle timeout, so this is neither implemented-on-main nor obsolete.

## Evidence

- **No first-response guard in WS stream options:** `OpenAIWebSocketStreamOptions` on `main` only defines `managerOptions` and `signal`; there is no `firstResponseTimeoutMs` or similar guard input in the current WS stream factory.
  - file: [src/agents/openai-ws-stream.ts:239](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/src/agents/openai-ws-stream.ts#L239)
  - command: `nl -ba src/agents/openai-ws-stream.ts | sed -n '230,330p'`
  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **WS request lifecycle has no timer for 'sent but no first server event':** After `session.manager.send(...)`, the code emits `start` and then awaits a Promise that only settles on abort, socket close, `response.completed`, `response.failed`, or `error`. There is no timer guarding the case where the socket stays open but produces no initial server event.
  - file: [src/agents/openai-ws-stream.ts:910](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/src/agents/openai-ws-stream.ts#L910)

  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **Embedded runner still wires only signal + manager options into OpenAI WS:** Current stream resolution creates the OpenAI WS stream with `signal` and `managerOptions.request` only. The PR's proposed first-response timeout is not being passed from the runner on `main`.
  - file: [src/agents/pi-embedded-runner/stream-resolution.ts:105](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/src/agents/pi-embedded-runner/stream-resolution.ts#L105)
