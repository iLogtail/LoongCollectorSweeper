## Summary

Keep open. Current main has partial cron delivery hardening for silent tokens, stale interim acknowledgements, descendant/subagent follow-up, and explicit error payloads, but it does not implement the requested fail-closed cron/session delivery policy, hallucination-marker suppression, or ledger-backed verification of unsupported success claims.

## Best Possible Solution

Keep this issue open and implement a core delivery/session-boundary policy for unattended cron runs, likely in `src/cron/isolated-agent/helpers.ts`, `run.ts`, and `delivery-dispatch.ts`: track tool/result failures and required-resource preconditions, suppress or fail output when critical calls fail or completion is unverifiable, and coordinate with #45049/#68051-style ledger-backed claim verification rather than relying only on text markers.

## Risks / Open Questions

- A regex-only marker filter would miss clean unsupported success claims and may over-suppress legitimate summaries.
- A cron-only fix may leave the related interactive-session success-claim failure mode from the comments unresolved.
- Strict fail-closed behavior needs careful operator feedback via run logs or failureDestination so silent suppression is diagnosable.
