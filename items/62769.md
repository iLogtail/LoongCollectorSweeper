## Summary

Keep open. Current main still scopes Telegram configured ACP bindings to topic-qualified group/forum conversations and rejects positive Telegram DM chat IDs, so the requested direct-message configured binding behavior is not implemented. The closed related #70177 is adjacent DM session-continuity work, not a canonical replacement for this configured ACP binding gap.

## Best Possible Solution

Keep the issue open and implement Telegram configured ACP direct-message support in the Telegram plugin adapter: accept `match.peer.kind: "direct"` with a positive numeric peer id as a valid configured-binding target, match inbound non-group Telegram conversations by that direct peer id, and ensure the resulting ACP session key stays distinct from topic-qualified group/DM-topic bindings. Add regression coverage for regular DM messages and native commands through a configured `bindings[].type="acp"` DM binding, then update the Telegram and ACP docs to document the supported DM shape.

## Risks / Open Questions

- Adding DM support changes Telegram ACP configured-binding identity semantics; the fix should preserve existing forum-topic binding IDs and current `/acp spawn --thread here` runtime DM bindings.
- Configured bindings should distinguish plain DMs from DM topics and group topics so positive chat IDs do not accidentally inherit topic-only parentConversationId behavior.
