## Summary

Keep #63074 open. Current main still classifies any bind source that does not start with `/` as `non_absolute` before normalizing or checking blocked credential paths, so a Windows home credential bind is not promoted into the critical blocked-path lane. The PR discussion describes a real classification gap, though the current PR should probably be narrowed because it includes unrelated changes outside the sandbox security surface.

## Best Possible Solution

Keep the PR open, but ask for a focused sandbox-security patch: move the blocked-path check before the non-absolute fallback, include Windows home roots such as `USERPROFILE`, add targeted regression coverage for Windows credential binds and generic Windows non-absolute behavior, and split unrelated Mattermost/postpublish/snapshot changes into separate PRs or drop them from this one. Treat broader Windows bind-source support as a separate follow-up, likely related to the existing Windows bind PR #42174.

## Risks / Open Questions

- The PR currently changes Mattermost policy exports/tests, npm postpublish verification, and unrelated snapshot/update tests in addition to the sandbox security classifier; that increases review risk and should be split unless each change is necessary for the Windows CI proof.
- Because `src/config/zod-schema.agent-runtime.ts` still rejects drive-letter bind sources at config validation, #63074 should be reviewed as a classification/audit hardening patch, not as the canonical fix for full Windows Docker bind support.
- This is security-sensitive path classification; the focused patch should land with a platform-aware regression test that proves dangerous Windows home credential binds classify as `targets` while generic Windows paths retain the intended non-absolute behavior.
