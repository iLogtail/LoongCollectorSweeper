## Summary

Keep open. Current main gives plausible configuration explanations, but not strong close evidence: dashboard chat replies still pass through the auto-TTS pipeline, compaction itself appears to mutate only compaction/token metadata rather than `ttsAuto`, and the reporter supplied concrete steps plus a same-day follow-up saying the issue repeated after a full reset. That is not enough to close as cannot-reproduce or implemented without a focused regression test, logs, or effective TTS state from the affected session.

## Best Possible Solution

Keep #69777 open until maintainers can capture the effective TTS state around compaction. The best next step is a focused dashboard/webchat regression test: long or compacted session, no `messages.tts.auto`, no local prefs/session `ttsAuto`, no TTS directives/tool call, and assert the final chat payload has no `mediaUrl`. If the behavior is configuration-driven, surface the effective TTS mode in the dashboard and document `/tts status` plus `/tts chat off/default` as the recovery path.

## Risks / Open Questions

- The issue may ultimately be a user preference/session override (`/tts on` or `/tts chat on`) rather than a code bug, but the current issue lacks `/tts status`, redacted prefs, session entry, and verbose TTS logs to establish that.
- Because block streaming can intentionally send a TTS-only final media payload when auto-TTS is enabled, a regression test should cover the dashboard compaction path with auto-TTS off and no TTS directives.
