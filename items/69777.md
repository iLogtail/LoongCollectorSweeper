## Summary

Keep open. Current main shows TTS defaults to off and manual compaction store updates do not directly set `ttsAuto`, but that is not enough to close this recent, concrete report: the reporter confirmed recurrence, and current ACP/dashboard delivery has an intentional audio-only final TTS path whenever effective TTS auto is enabled. The remaining question is which effective TTS source, local prefs, session override, config, or model/directive path, flips on after long-session compaction in the reporter's setup.

## Best Possible Solution

Keep #69777 open and ask for a targeted repro bundle: `/tts status` before and after compaction, the redacted session entry fields `ttsAuto`, `lastTtsLatestHash`, `compactionCount`, effective `messages.tts`/agent/channel/account TTS config, and whether ACP/block streaming was active for the dashboard session. If confirmed, the likely fix belongs in the dashboard/ACP reply path: ensure text remains the primary visible final reply unless the user explicitly enabled auto-TTS, and consider a clearer explicit-only default or status warning when `ttsAuto`/local prefs make audio-only final delivery possible after block streaming.
