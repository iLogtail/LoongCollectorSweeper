## Summary

Keep open. Current main does not contain a clear fix or a strong enough cannot-reproduce result for this recent dashboard/TTS-after-compaction report. The code does show that compaction count updates preserve existing session fields rather than directly setting `ttsAuto`, which supports the prior bot triage comment, but current main still has multiple legitimate paths for automatic audio replies when session prefs/config enable TTS. The reporter also followed up that the issue happened twice, so conservative cleanup should not close this without a current-main reproduction or targeted fix.

## Best Possible Solution

Keep this issue open and ask for a focused reproduction bundle: redacted `messages.tts` config, `~/.openclaw/settings/tts.json`, the affected session entry fields including `ttsAuto` and `compactionCount`, `/tts status` output before and after compaction, and a short transcript around the first audio-only reply. If reproduced, add a targeted regression around the dashboard `chat.send`/compaction path and decide whether internal webchat should suppress automatic TTS unless explicitly requested, or whether `buildTtsSystemPromptHint` should more strongly tell the model not to call/use TTS unless the user asks.
