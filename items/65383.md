## Summary

Keep #65383 open. The underlying iMessage duplicate default/named watcher problem is still present on current main, but this PR should not be closed as implemented or superseded because no equivalent fix has landed. The PR also should not be landed unchanged: the provided review discussion identifies startup-vs-global enabled-state concerns and an unrelated outbound-target change with regression risk.

## Best Possible Solution

Keep the PR open but request a narrower patch: implement iMessage duplicate-backend suppression at the gateway watcher startup boundary, not as the plugin-wide enabled predicate; only shadow the default when the competing named account is actually startup-eligible; add focused regression tests for default plus named same-backend and placeholder-account cases; and drop the unrelated outbound target resolver changes from this PR.
