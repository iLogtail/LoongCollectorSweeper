## Summary

Current main still matches the reported problem. `pickSummaryFromOutput` hard-codes a 2000-character cap, cron run-log config/schema/docs expose only `maxBytes` and `keepLines`, and the run-log writer persists only the already-truncated `summary` field for `cron runs`. No duplicate or shipped fix was found in local code/history evidence.

## Best Possible Solution

Keep the issue open. A targeted fix should add a documented `cron.runLog.maxSummaryChars` contract, validate it in the config schema/help/generated docs, plumb it into the isolated cron summary selection or run-log append path, and add regression coverage proving long cron outputs can be retained up to the configured cap while existing `maxBytes`/`keepLines` pruning still bounds disk usage.
