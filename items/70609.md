## Summary

Keep open. Current main still lacks the requested outbound metadata route: gateway send rejects unknown metadata, sendMessage has no Hermes or opaque metadata parameter, direct delivery does not expose caller metadata in ChannelOutboundContext, and delivery queue storage/recovery does not persist or replay such metadata. The PR is a relevant core contract proposal, but the branch should not merge as-is because the provided review context identifies a central forwarding gap and the PR mixes Hermes-specific hardcoded values plus an unrelated auth fix into a generic outbound/API change.

## Best Possible Solution

Keep the PR open, but ask for a narrow rework. The landable core part should add a generic opaque outbound metadata contract through gateway send, sendMessage, DeliverOutboundPayloadsParams, queue storage/recovery, and ChannelOutboundContext, with tests proving gateway, direct delivery, and recovery replay expose the same metadata to channel plugins. Hermes-specific builders, hardcoded topic/bot defaults, task/media/approval injection, and the unrelated auth change should be moved behind configuration or a plugin/ClawHub implementation and split from the core API change.
