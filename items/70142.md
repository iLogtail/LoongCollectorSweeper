## Summary

Keep open. Current main still has two awaited attachment-processing steps between the initial `chatAbortControllers` in-flight check and the later registration point, so same-`idempotencyKey` attachment retries can still miss the documented `{ status: "in_flight" }` response path. Current main does mitigate the exact map-overwrite/orphan-controller claim because `registerChatAbortController` refuses to overwrite an existing run id, but that does not make this PR obsolete: the PR still targets a real contract gap, duplicate started acks, duplicate attachment parsing/persistence work, and missing regression coverage. The PR discussion is concrete and useful, including linked issue #70139, a focused regression plan, Greptile's positive review, and a follow-up fix for offloaded media cleanup after Codex review feedback.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to land or adapt the post-parse `in_flight` re-check with offloaded media cleanup and regression coverage, while reframing the rationale around the current-main behavior: preserve the documented idempotency response and avoid duplicate attachment work, not primarily controller-map overwrite. If broader duplicate transcript effects remain, track those under the existing #69208 umbrella rather than using it to close this focused PR.
