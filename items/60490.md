## Summary

Keep open. Current main has partial Windows gateway service hardening, but it does not conclusively solve the reported native Windows auto-start-after-reboot failure. The latest issue comment also reports a closely related current Windows state with task registered/ready but no listener, plus an external workaround reference, so this still needs maintainer/Windows verification rather than cleanup closure.

## Evidence

- **Recent issue discussion reports related current behavior:** The Apr 25 comment reports native Windows cases where `openclaw gateway status` can show the task registered/ready while nothing listens on `127.0.0.1:18789`, and links an external helper/workaround at https://github.com/sherlock-huang/openclaw-windows-silence-run plus related writeups. That is aligned with the issue's remaining auto-start/listener symptom rather than obsolete noise.
- **Scheduled Task remains the primary persistent startup path:** Current main creates a Windows Scheduled Task with `/SC ONLOGON` and only writes a persistent Startup-folder login item when `/Create` fails with access denied/timeout/no-output. If the task can be created but does not actually run after login, this path does not install a persistent fallback for future Windows restarts.
  - file: [src/daemon/schtasks.ts:827](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/src/daemon/schtasks.ts#L827)
  - command: `nl -ba src/daemon/schtasks.ts | sed -n '827,883p'`
  - sha: [b64bfc5d9abf](https://github.com/openclaw/openclaw/commit/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab)
- **Run-no-op fallback is immediate, not persistent:** When `schtasks /Run` succeeds but the task remains in the not-yet-run state without launch evidence, current main calls `launchFallbackTaskScript(...)`. That can recover the current install/restart attempt, but it does not write the Startup-folder fallback used for future logins.
