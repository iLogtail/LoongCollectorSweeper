## Summary

Keep PR #55225 open. Current main still lacks Feishu recalled-message handling: no recalled event is registered, no reply-to-bot-message mapping exists, and the current send path can fall back to a direct send when a reply target is withdrawn. The PR is focused on the existing bundled Feishu channel and addresses a real remaining behavior gap, though it still needs maintainer review and focused tests before merge.

## Best Possible Solution

Keep this PR open for Feishu plugin owner or maintainer review. The right implementation belongs in extensions/feishu: register the correct recalled-message event, record successful bot replies across relevant reply paths, avoid direct fallback when that would answer an already-recalled user message, delete all tracked bot replies idempotently, retain retryable failures, and add tests plus Feishu event-subscription docs. If maintainers decide recalled-message cleanup should not be bundled, that should be an explicit product decision rather than automated cleanup closure.
