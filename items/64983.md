## Summary

Keep open. Current main partially mitigates the Telegram context-overflow no-reply path by surfacing explicit overflow fallback text, but the broader report is not fully implemented: isolated cron still relies on model-produced final output before NO_REPLY suppression, and gateway PID/port mismatch is still handled as diagnosis/restart/fail-fast rather than a clear self-healing single-owner state.

## Best Possible Solution

Keep the issue open or split it into focused follow-ups. The best product path is to retain the shipped Telegram context-overflow fallback, add a cron no-op/silent-outcome path that can complete when dispatch logic determines no send is needed without requiring a final NO_REPLY model generation, and decide whether gateway PID/port mismatch should self-heal automatically or remain documented restart/doctor remediation.
