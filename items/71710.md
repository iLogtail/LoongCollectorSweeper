## Summary

Keep open. Current main does not clearly implement either requested mitigation: `openclaw agent` still waits on the gateway with no CLI-level SIGTERM cancellation path, falls back to embedded execution on gateway failure, and the agent runtime still has process-level SIGTERM cleanup handlers that can avoid re-raising when more than one handler is registered. The CLI docs describe `--timeout` but do not warn cron users to use `timeout -k` or an equivalent supervisor kill escalation. Related #66399 is adjacent supervisor work, not a complete duplicate of this CLI entry-point leak.

## Best Possible Solution

Keep #71710 open and fix it in the CLI/agent runtime. The likely path is to centralize SIGTERM/SIGINT handling for `openclaw agent`, cancel in-flight gateway and embedded agent work with an AbortSignal, run bounded cleanup, and always exit or re-raise after a short grace window even when multiple cleanup listeners exist. Until that lands, add CLI/cron documentation that recommends `timeout -k <delay> <duration>` or an equivalent supervisor escalation for scripted one-shot agent runs.
