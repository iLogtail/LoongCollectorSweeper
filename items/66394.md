## Summary

Keep open. Current main fixed the most serious size-cap behavior by rotating file logs instead of suppressing writes, but the issue and related PR discussion ask for production-grade, configurable rotation. Current main still exposes only `logging.maxFileBytes` and fixed archive behavior, with no configurable interval, compression, or max retained file count.

## Best Possible Solution

Keep the issue open as a focused logging product/design follow-up. The shipped size rotation should remain, but maintainers should decide whether to add documented config for rotation interval, archive count/retention, compression, and possibly human-friendly size strings, with matching schema, docs, and logging tests.
