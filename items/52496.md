## Summary

Keep #52496 open. Current main still resolves a visible `sessions_send` target and starts an internal agent run with the resolved session key, with no refusal or warning for Slack/Discord-style `:thread:` session keys. The related items in the provided context cover preserving thread delivery, timeouts, or reply mirroring; none supersedes this requested guardrail.

## Best Possible Solution

Implement a narrow guard in `sessions_send` after target resolution and before `startAgentRun`, so both direct `sessionKey` and label-resolved targets are covered. Return a clear error for user-facing thread-suffixed session targets, add regression tests in the session tool tests for direct and label-resolved thread keys, and update the session-tool docs/tool description if the policy becomes user-visible.

## Risks / Open Questions

- A naive substring ban on `:thread:` could block custom/non-channel session keys unintentionally; the implementation should use existing session/thread parsing where possible.
- Maintainers should explicitly decide whether the same policy should apply only to Slack/Discord-style `:thread:` keys or also to other user-facing topic/thread forms such as Telegram `:topic:`.
