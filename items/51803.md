## Summary

Current `main` still does not persist or replay followup queues across gateway restart, and the PR discussion identifies unresolved behavioral gaps in the proposed system-event replay approach. This is useful unfinished work, not a safe auto-close.

## Evidence

- **Shutdown still clears run state without persisting followups:** On current `main`, gateway shutdown proceeds to `params.chatRunState.clear()` and websocket teardown without any followup-queue persistence step. The core behavior this PR proposes is not implemented on `main`.

  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **Startup has no persisted-followup replay hook:** Current startup schedules restart-sentinel wake and aborted-main-session recovery, then returns. There is no `replayPersistedFollowups()` call or equivalent recovery path on `main`.
  - file: [src/gateway/server-startup-post-attach.ts:341](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/gateway/server-startup-post-attach.ts#L341)

## Risks / Open Questions

- Queued followups and drain-window inbound messages can still be lost on restart on current `main` because recovery persistence is not implemented there.
- If this PR is revived, the system-event replay approach still needs explicit maintainer judgment on routing fidelity, session triggering, repeated-restart durability, overflow behavior, and removal of the committed log artifact.
