## Summary

Keep open. Current main still omits the `file` alias from the shared tool-display path resolver and from the embedded read-start path check, so the display/logging part of PR #55121 is not implemented. The PR should be updated against current main because runtime validation now enforces canonical `path` and rejects legacy aliases, which changes the PR body's execution-contract assumptions.

## Best Possible Solution

Keep the PR open but ask for a rebase/update against current main. If maintainers want display-only alias compatibility, add `record.file` to `resolvePathArg()`, align the embedded read-start warning behavior intentionally, and add the proposed regression test. If maintainers instead want strict canonical `path` only everywhere, close the linked bug after explicitly deciding that stale alias display is out of scope and adjust existing `file_path` display/warning compatibility for consistency.
