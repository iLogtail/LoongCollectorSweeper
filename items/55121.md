## Summary

Keep open. Current `main` at `34896839ba22f3a71a3d5a8fb4ca5f4c6989a534` still does not recognize the `args.file` alias in the tool-display path resolver or the embedded `read` warning path check, so this PR appears to address a real unmerged bug rather than something already fixed on `main`.

## Evidence

- **Tool-display path resolver still omits `file`:** `resolvePathArg()` only checks `record.path`, `record.file_path`, and `record.filePath`; it does not check `record.file`. That means `read`/`write`/`edit` summaries can still drop the path when models send `{ file: ... }`.
  - file: [src/agents/tool-display-common.ts:179](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/agents/tool-display-common.ts#L179)

- **Read/write detail generation depends on that resolver:** `resolveReadDetail()` and `resolveWriteDetail()` both call `resolvePathArg()`, so the missing alias directly affects displayed tool detail on current `main`.
  - file: [src/agents/tool-display-common.ts:197](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/agents/tool-display-common.ts#L197)
