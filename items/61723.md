## Summary

Keep #61723 open. Current main has a related, merged Gateway/TUI startup retry for transient `chat.history unavailable during gateway startup`, but that does not prove the reported no-render hang after gateway restart or the alternate `kill -9` terminal-state path is fixed. The issue is recent, has a second Ubuntu confirmation, and I did not find a canonical duplicate or source-backed full fix for the no-render/plugin-log hang.

## Best Possible Solution

Keep #61723 open and narrow the next maintainer step to a reproducible TUI lifecycle regression: exercise `openclaw tui` across gateway restart and forced prior-TUI termination, ideally with a pseudo-terminal harness or WSL2/Linux manual proof. If the hang reproduces, fix the TUI startup path to render or fail visibly even after plugin log output and add terminal-state recovery before first render; if it no longer reproduces on current main, close later with explicit live evidence and the exact fixing commit/release.
