## Summary

Keep open. Current main still does not expose `--session-key` on `openclaw agent`, even though the resolver, embedded command options, Gateway protocol, Gateway handler, and Gateway tests already support explicit session keys. The PR addresses a real core CLI parity gap with concrete downstream demand in the discussion, so no conservative close reason applies.

## Best Possible Solution

Keep the PR open. The best path is a scoped core CLI change that registers `--session-key` on `openclaw agent`, adds it to `AgentCliOpts`, includes it in the selector guard and resolver call, preserves embedded fallback behavior, aligns validation with the Gateway handler, and updates CLI docs/help plus focused tests. If maintainers do not want raw session keys exposed, that should be an explicit product decision rather than an automated stale close.
