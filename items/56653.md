## Summary

Keep #56653 open. Current main has Slack reaction handlers, and #34045 fixed session-key routing after a reaction event reaches OpenClaw, but this issue reports a newer post-#34045 failure where Slack Socket Mode delivers zero reaction events/logs. I did not find current source, tests, docs, or changelog evidence proving that Socket Mode ingress for multi-account reaction events has since been fixed or that the report is obsolete.

## Best Possible Solution

Keep the issue open and investigate the Slack Socket Mode ingress path for `reaction_added`/`reaction_removed` in multi-account setups. The best maintainer path is to add temporary/raw envelope diagnostics around Socket Mode `events_api` processing, verify per-account `api_app_id`/`team_id` filtering against real reaction payloads, add a regression or live-smoke that drives a reaction event through the receiver into `registerSlackReactionEvents`, and then update Slack troubleshooting docs if the root cause is Slack-side subscription/app installation state rather than OpenClaw code.
