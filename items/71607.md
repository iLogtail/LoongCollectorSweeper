## Summary

Current main still appears to have the reload drift reported in #71607. The timer force-reloads jobs with `skipRecompute: true`, `ensureLoaded` does not compare the previous in-memory schedule/enabled inputs or clear stale `state.nextRunAtMs`, and the runnable-job path can still treat a persisted stale timestamp as due. The linked PR #71651 is an active unmerged fix attempt, so the issue should remain open until that or an equivalent fix lands.

## Best Possible Solution

Keep #71607 open. The best fix is to land #71651 or an equivalent patch in `src/cron/service/store.ts` that compares reloaded jobs against the previous in-memory job by id, invalidates `state.nextRunAtMs` when scheduling inputs such as `schedule` or `enabled` change, and adds a regression test for `forceReload` plus `skipRecompute` with a stale persisted future timestamp.

## Evidence

- **Reload path lacks invalidation:** `ensureLoaded` rehydrates jobs and replaces `state.store`, then only calls `recomputeNextRuns` when `skipRecompute` is false. There is no previous-job comparison and no clearing of `hydrated.state.nextRunAtMs` when `schedule` or `enabled` changed on disk.
