## Summary

Current main already implements the task timestamp ordering fix through a broader normalization path: task records are normalized at create, update, and restore time, with regression coverage for earlier startedAt values and zero inconsistent_timestamps audit findings. PR #69356 is now obsolete even though its root-cause analysis and test feedback were useful.

## Best Possible Solution

Close #69356 as implemented on main and keep the broader #71871 normalization path. It solves the same user-visible audit noise while also covering update and restore paths; any separate Task Flow timestamp issue should stay in a narrower follow-up rather than merging this now-obsolete PR.
