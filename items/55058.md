## Summary

Current `main` already adds a configurable `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange` opt-in that keeps `web_fetch` strict by default but allows the FakeIP `198.18.0.0/15` range when explicitly enabled, which resolves the concrete problem described in #55058.

## Close Comment

Closing from a Codex review because current `main` already implements the reported FakeIP SSRF-policy escape hatch for `web_fetch`. There is now a `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange` setting in `src/config/types.tools.ts:561`, documented in generated config help at `src/config/schema.help.ts:757`, and the runtime threads that flag into the guarded fetch path in `src/agents/tools/web-fetch.ts:390`. The regression case in this issue is also covered directly by `src/agents/tools/web-fetch.ssrf.test.ts:159`, which verifies that `198.18.0.153` stays blocked by default and is allowed only when the opt-in is set. This Codex review was against current `main` at `34896839ba22f3a71a3d5a8fb4ca5f4c6989a534`. The implementation is narrower than full browser-schema parity, but it does address the concrete FakeIP `web_fetch` gap described here.
