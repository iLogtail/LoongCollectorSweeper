## Summary

Keep open. Current main still preserves the reported failure class: packaged gateway install paths do not provide a launchd WorkingDirectory, the plist only emits that key when one is supplied, and explicit relative agentDir values still resolve through process cwd. The provided triage comments match the current source. Related #58061 is broader launchd-default work, not a shipped canonical fix for this ENOENT path.

## Best Possible Solution

Implement the bug fix in core: set a deterministic WorkingDirectory for packaged gateway LaunchAgent installs, preferably via buildGatewayInstallPlan or resolveGatewayProgramArguments, and add regression coverage that a normal packaged macOS install emits the plist key. For defense in depth, also anchor explicit relative agents.list[].agentDir values to the loaded config/state directory or migrate/doctor-repair legacy relative agentDir configs, then document the behavior if the public config contract changes.
