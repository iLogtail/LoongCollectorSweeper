## Summary

Keep #70223 open. Current main still does not set a `WorkingDirectory` for the packaged gateway install path, and explicit relative `agents.list[].agentDir` values are still resolved through `path.resolve()` against the process cwd. The current default agent directory is now derived from the state dir, which helps fresh configs, but it does not fix existing relative `agentDir` configs or the reported launchd cwd=/ failure mode.

## Best Possible Solution

Keep the issue open and implement both layers: set a deterministic `workingDirectory` for packaged gateway service installs before rendering launchd/systemd/schtasks metadata, and normalize or repair explicit relative `agents.list[].agentDir` values so they are anchored to the config/state directory rather than `process.cwd()`. Add regression coverage for packaged gateway install plist output and a relative-agentDir/cwd=/ resolution case.

## Risks / Open Questions

- Closing this now would leave existing users with explicit relative `agents.list[].agentDir` vulnerable under launchd cwd=/ even if fresh configs no longer write that relative default.
- A WorkingDirectory-only fix would address macOS launchd, but the more durable fix should also anchor or repair relative agentDir values so the same class of failure does not recur under other supervisors.
