## Summary

Keep #53008 open. Current main still dispatches pre-compaction memory flush as an awaited embedded run without an explicit non-main lane, so `runEmbeddedPiAgent` falls back to the default global `main` lane. The related PR #53154 proposed a dedicated `memory-flush` lane, but it was closed unmerged as stale and its lane/coverage changes are not present on main.

## Best Possible Solution

Keep the issue open and implement a focused core fix: route pre-compaction memory flush/compaction work through a dedicated, bounded background lane or equivalent scheduler path, add gateway lane concurrency setup, and add regression coverage proving memory flush does not use the default main lane. The closed PR #53154 is useful prior art, especially its later `memory-flush` lane direction, but it should be reworked cleanly without unrelated test churn and with attention to the active-run/session-ordering behavior noted in review comments.
