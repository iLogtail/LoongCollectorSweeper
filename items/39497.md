## Summary

Keep #39497 open. Current main still has the reported `/agent` read-guard gap in the sandboxed workspace-only read path: the guard maps only `containerWorkdir`, the sandboxed read-tool wiring passes only that option, and existing regression coverage still expects `/agent/secret.txt` to fail with `Path escapes sandbox root` when the workspace-root guard is enabled. The related fix PRs in the provided GitHub context were closed unmerged, and merged #44307 explicitly did not address `/agent` read-guard remapping.

## Best Possible Solution

Keep the issue open. The likely fix belongs in the sandboxed file-tool guard/wiring: make `wrapToolWorkspaceRootGuardWithOptions` validate against the sandbox mount table, or pass an explicit `/agent` mapping only for the `workspaceAccess: "ro"` agent workspace mount. Add regression coverage proving `/agent/...` reads resolve in the intended read-only path while write/edit/apply_patch remain blocked. Treat `/tmp` reads as a separate policy decision.
