## Summary

Keep open. Current main still does not implement the requested `/agent` read-tool workspace-root guard behavior. The ordinary sandbox bridge can resolve `/agent` mounts, but the `tools.fs.workspaceOnly` guard still maps only `containerWorkdir` and current tests explicitly reject `/agent/...` under that guard. Related fix PRs were closed unmerged, and merged PR #44307 explicitly scoped out this issue.

## Best Possible Solution

Keep tracking this for maintainer product/security judgment. If `/agent` should be readable in `workspaceAccess: "ro"` sessions even when `tools.fs.workspaceOnly` is true, implement a mount-aware guard that validates the real resolved sandbox mount target rather than blindly remapping `/agent` to the sandbox workspace root, and add regression coverage for both allowed read-only `/agent` reads and rejected writes/unsafe mounts. If maintainers want the current boundary, close only after documenting that `/agent` is intentionally outside `tools.fs.workspaceOnly`.
