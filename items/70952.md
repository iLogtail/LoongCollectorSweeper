## Summary

Keep #70952 open. Current main has useful SecretRef batching and some pass-scoped cache plumbing, but it does not implement the requested provider-level memoization for repeated vault/container lookups, item/name lookups, fetched item payloads, or a startup-wide diagnostic. The reported 1Password Connect quota/rate-limit issue remains a plausible core/runtime performance gap rather than an already-shipped fix or duplicate.

## Best Possible Solution

Keep the issue open and implement a focused SecretRef performance fix in core if maintainers want OpenClaw to own this behavior: extend the startup-resolution cache so resolveSecretRefValues consults and populates resolvedByRefKey for batched calls, avoid duplicate preflight/final activation work where safe, add tests proving repeated refs/shared provider paths do not re-invoke the resolver unnecessarily, and add non-secret diagnostics for total refs/provider calls/cache hits. If the 1Password vault/item decomposition is owned by an external exec resolver, clarify that contract in docs and provide a sample resolver-side memoization pattern.
