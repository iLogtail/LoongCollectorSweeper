## Summary

Keep open. Current `main` already batches identical SecretRef resolutions per provider during runtime snapshot prep, but it does not implement the issue’s requested provider-scoped memoization for shared backend lookups (for example vault-name or item-name sublookups inside an exec-backed resolver). With no live 1Password Connect reproduction in this review environment, there is not enough evidence to close as fixed or non-reproducible.

## Evidence

- **Gateway startup resolves all collected SecretRefs in one runtime-snapshot pass:** `prepareSecretsRuntimeSnapshot()` collects config/auth-store assignments, then resolves them through a single `resolveSecretRefValues(refs, { cache: context.cache })` call during startup/reload preparation.
  - file: [src/secrets/runtime.ts:337](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/secrets/runtime.ts#L337)
  - command: `nl -ba src/secrets/runtime.ts | sed -n '296,347p'`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Resolver de-duplicates exact SecretRefs and groups by provider:** `resolveSecretRefValues()` first keys refs by `secretRefKey(ref)` and then groups them by `source:provider` before dispatching provider work. That proves exact-ref dedupe and provider batching exist already on `main`.
  - file: [src/secrets/resolve.ts:838](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/secrets/resolve.ts#L838)

- **Exec providers already receive batched `ids` requests:** `resolveExecRefs()` builds one stdin payload per provider with `protocolVersion`, `provider`, and an `ids` array, so OpenClaw already avoids one process spawn per ref when refs share an exec provider.
  - file: [src/secrets/resolve.ts:648](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/secrets/resolve.ts#L648)

- **Built-in cache surface is narrower than the requested memoization:** `SecretRefResolveCache` currently exposes only `resolvedByRefKey` and `filePayloadByProvider`. There is no core cache for provider-specific intermediate lookups such as 'vault name -> id' or 'item name -> id'.
  - file: [src/secrets/resolve-types.ts:1](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/secrets/resolve-types.ts#L1)

## Risks / Open Questions

- No live 1Password Connect or custom exec-provider reproduction was possible in this read-only environment, so I could not directly validate the reported duplicate backend-path counts against current `main`.
- The reported duplication may still be real for distinct SecretRef ids that share provider-specific path components; current core batching does not obviously eliminate that class of duplicate backend lookup.
- `gh` inspection beyond the provided issue context was blocked because `GH_TOKEN` is not configured in this environment.
