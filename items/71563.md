## Summary

Keep #71563 open. Current main still routes Telegram updates through `getTelegramSequentialKey`, but that resolver only puts abort/status/btw/approval cases on special lanes; registered `/new` and `/reset` session commands fall through to the normal chat/topic lane. The PR is a focused, still-relevant Telegram fix and is not implemented or superseded on main.

## Best Possible Solution

Keep the PR open for maintainer review. The likely best path is a scoped Telegram change in `extensions/telegram/src/sequential-key.ts` that routes only registered `/new` and `/reset` session commands to the control lane, with regression coverage in `extensions/telegram/src/sequential-key.test.ts` and, if feasible, an active-run Telegram middleware test proving the command bypasses the busy lane without reviving the prior non-abort command/reply race.
