## Summary

Keep PR #71563 open. Current main does not route Telegram `/new` or `/reset` onto the control lane: `getTelegramSequentialKey` only special-cases abort, read-only status commands, BTW, and approval callbacks before falling through to the normal chat/topic key. The PR is a focused Telegram bug fix with relevant tests; the provided related items cover different `/new`/`/reset` symptoms and do not supersede this queue-bypass work.

## Best Possible Solution

Keep the PR open and review it as the likely correct fix: add a narrow session-control predicate for registered lifecycle commands (`/new` and `/reset`) in `extensions/telegram/src/sequential-key.ts`, share command-resolution logic with the status bypass, and land regression coverage proving those commands use `telegram:<chat>:control` while non-control session commands remain serialized.
