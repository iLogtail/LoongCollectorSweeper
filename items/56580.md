## Summary

Keep open. Current main still models native slash commands with a single English/default `description`, builds Discord Carbon command objects without `descriptionLocalizations`, and redeploys those descriptors through the Carbon reconcile path. The reporter's requested `description_localizations` support remains a real bundled Discord plugin/SDK gap, and the related ClawSweeper context covers other Discord slash-command routing/defer/visibility items rather than this localization request.

## Best Possible Solution

Keep #56580 open. The best implementation is an additive native-command localization contract: extend `ChatCommandDefinition`/`NativeCommandSpec` and, if plugin-owned commands should participate, `OpenClawPluginCommandDefinition`; preserve the fields through `listNativeCommandSpecsForConfig` and Discord `appendPluginCommandSpecs`; set Carbon `descriptionLocalizations`/`nameLocalizations` and any supported option localization fields in `createDiscordNativeCommand`; add regression tests proving the serialized Carbon command contains `description_localizations` and that startup reconcile treats localized payloads as desired state; update the Discord/slash-command docs once the config/API shape is settled.
