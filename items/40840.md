## Summary

Current `main` still has a plausible delete-then-reregister race: `sessions.delete` waits only for the run to stop being active, while normal agent teardown later calls `updateSessionStoreAfterAgentRun(...)`, which recreates `store[sessionKey]` if the row was deleted. I did not find a tombstone/kill-boundary guard or a regression test covering that post-run persistence window, so this should stay open.

## Evidence
