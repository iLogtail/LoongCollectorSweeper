## Summary

Keep #61278 open. Current main fixed the misleading model-pricing part by running pricing refresh asynchronously, and the issue discussion positively narrowed the real bottleneck to Gmail watcher startup. But the requested non-blocking Gmail hook startup is not implemented on the default gateway path: sidecar startup still awaits Gmail watcher setup, and the default CLI/server path still waits for sidecars unless an explicit internal defer option is set.

## Best Possible Solution

Keep the issue open and make the Gmail watcher bootstrap non-blocking on the default startup path. A narrow fix would fire-and-forget or shortly bound startGmailWatcherWithLogs inside sidecar startup, preserve existing watcher logs/errors, decide whether /readyz should wait for Gmail watcher or only channels/plugin services, and add a regression test with a slow injected Gmail watcher proving gateway startup does not wait on the 120s gog watch start path.
