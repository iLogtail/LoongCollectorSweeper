## Summary

Keep #59808 open. Current main has not implemented the reported mcporter stdout/noisy-first-call hardening: the mcporter search path still parses raw `result.stdout` with `JSON.parse`, while tests cover only clean mcporter JSON and clean daemon-start stdout. The related ClawSweeper reports are different QMD fallback/stale-collection cases, not a canonical duplicate for mcporter stdout contamination after gateway restart.

## Best Possible Solution

Keep the issue open and fix it in the memory-core QMD mcporter bridge. Add defensive handling for mcporter stdout noise: discard daemon-start stdout where possible, parse mcporter call output by extracting the structured JSON object before `JSON.parse`, and add a regression test for `startDaemon=true` first search where mcporter emits `[mcporter]` log text before valid JSON. The test should assert QMD results are used and `FallbackMemoryManager` does not switch to builtin/FTS for this startup-noise case.
