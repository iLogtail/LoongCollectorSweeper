## Summary

Keep open. Current main does not contain this PR's requested behavior: `listSessionsFromStore` still applies `activeMinutes` solely by `updatedAt`, so a stale `agent:<id>:<mainKey>` row is filtered out like any other stale session. The web chat path still asks for the compact active list after completed chat runs, so the underlying bug remains relevant. The PR discussion also shows the custom `mainKey` review concern was acknowledged and addressed on the PR branch, which makes this useful work rather than obsolete cleanup.

## Best Possible Solution

Keep the PR open and review or land a focused gateway fix that exempts configured top-level agent main sessions from `activeMinutes` filtering while preserving compact filtering for non-main stale sessions. The final patch should include the default-main regression and the custom-`mainKey` regression described in the PR discussion.
