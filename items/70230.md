## Summary

Keep open. Current main still exposes `health` as a cached service-level gateway snapshot with channels, agents, sessions, heartbeat seconds, and duration, but not the requested per-connection `connected`, `rttMs`, or `lastHeartbeatAt` fields. There is transport tick/watch logic in the Node gateway client, and the browser client has a local `connected` getter, but those are not surfaced through `openclaw gateway health` or a `gateway.ping` RPC. The provided related reports are channel-health-specific and do not supersede this connection-level gateway UI request.

## Best Possible Solution

Keep #70230 open. The best implementation path is an additive connection-health surface: either a lightweight `gateway.ping` RPC that returns server time and lets clients compute rolling RTT/last heartbeat, or a per-request `connection` object added to `health` after the shared health cache is read. Update `HealthSummary`/protocol docs, gateway CLI JSON tests, and UI client code so polling UIs can consume `connected`, `rttMs`, and `lastHeartbeatAt` without confusing them with channel health or the existing agent `last-heartbeat` method.
