## Summary

Keep open. Current main does not implement an explicit one-shot import/clone flow from an existing-session browser profile into the managed openclaw profile. The browser CLI, control routes, client API, and profile service cover list/create/delete/reset/start/stop and existing-session attach, but there is no import command, route, or API that copies source browser state into the managed user-data dir. The related reports in the provided context concern existing-session attach reliability or limits, not this feature request.

## Best Possible Solution

Keep the issue open and implement it in the bundled browser plugin's profile-management surface: add an explicit CLI command and control route such as `openclaw browser profiles import` / `POST /profiles/import`, stop/reset the destination managed profile, copy from a resolved same-user local source, strip Chrome Singleton/runtime artifacts and other unsafe transient files, leave the source untouched, relaunch only when requested, and document the security model. If maintainers prefer an external ClawHub workflow, first expose a safe browser-profile maintenance API; current public surfaces are not enough for a robust plugin-only implementation.

## Risks / Open Questions

- Importing browser profile state is security-sensitive because cookies, sessions, and password-manager state may be copied into an agent-controlled profile.
- A safe implementation needs platform-aware source resolution, destination lifecycle control, and a conservative exclusion list for lock/runtime/cache files.
- This does not currently fit a clean ClawHub close because the public browser profile SDK facade is resolve-only, while the requested behavior needs browser-owned profile mutation and lifecycle handling.
