## Summary

Keep #72273 open. Current main still hardcodes the Discord gateway startup READY timeout at 15s and runtime reconnect READY timeout at 30s, and the Discord config schema exposes other Discord timeout knobs but not these two. The reportâ€™s multi-account stagger context also matches current code: Discord accounts are staggered by 10s per startup index, so large account sets can reasonably exceed the fixed READY window unless the lifecycle timeout becomes configurable or derived from the stagger.

## Best Possible Solution

Implement this in the Discord plugin boundary, not generic core: add typed account-aware config such as `channels.discord.gateway.readyTimeoutMs` and `channels.discord.gateway.runtimeReadyTimeoutMs` or a similarly named Discord-owned section, validate/clamp it in the Discord config schema and metadata, thread it through `monitorDiscordProvider` into `runDiscordGatewayLifecycle`, and add tests covering configured startup/runtime READY windows plus the default behavior. A derived default that accounts for the configured multi-account startup stagger would also address the central failure mode.
