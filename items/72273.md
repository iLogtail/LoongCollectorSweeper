## Summary

Keep #72273 open. Current main still hardcodes Discord gateway startup and runtime READY deadlines at 15s and 30s, while multi-account Discord startup intentionally staggers later accounts by 10s per account before `monitorDiscordProvider` starts. The Discord config schema/types expose other per-account runtime knobs such as `inboundWorker` and `eventQueue`, but no ready-timeout config or env override exists on this surface.

## Best Possible Solution

Keep the issue open and implement this inside the bundled Discord plugin, likely as strict per-account/channel config fields under `channels.discord` plus docs/schema/type/metadata/test coverage. The implementation should pass resolved timeout values into `runDiscordGatewayLifecycle`, preserve current defaults for existing users, and cover multi-account stagger cases where the startup READY deadline must exceed the largest account startup delay.
