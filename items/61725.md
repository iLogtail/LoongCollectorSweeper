## Summary

Current main does not contain PR #61725's nullish fallback or its direct legacy-schema regression tests. The linked bug report may be closed, but I found no source, test, or history evidence that the underlying write path has been fixed or superseded on main, so this PR should not be auto-closed.

## Best Possible Solution

Keep PR #61725 open for maintainer review. The likely product-safe path is a targeted SQLite binder hardening that preserves empty-string requester keys while coalescing only nullish values for legacy `requester_session_key TEXT NOT NULL` databases, plus regression coverage for both direct upsert and the `markTaskLostById` legacy-schema path. If review finds the nullish value originates earlier, fix the restore/normalization boundary instead of closing this as obsolete.

## Risks / Open Questions

- The PR's stated root cause may be imprecise for Node v24.14.1: `undefined` is rejected by `node:sqlite`, while `null` triggers the observed NOT NULL constraint failure. That argues for maintainer review and a precise regression test, not automatic closure.
- The normal TypeScript creation path requires and normalizes `requesterSessionKey`, so the real failure likely depends on legacy, restored, or malformed task rows. The regression should keep using a real legacy schema and the actual persistence path.
