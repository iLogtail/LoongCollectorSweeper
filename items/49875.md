## Summary

Keep open. Current `main` still lacks the general `runInboundClaim()` dispatch that this PR proposes, so the underlying gap is not already implemented, but the PR is not a safe cleanup close either: current tests still encode the old no-broadcast behavior, the plugin-fallback path promises an OpenClaw fallback before the proposed insertion point, and fast-abort handling still depends on reaching the post-`markProcessing()` branch.

## Evidence

- **Current main still has no general inbound-claim broadcast:** After plugin-bound handling and the fire-and-forget `message_received` hooks, `dispatchReplyFromConfig` goes straight to `markProcessing()` and then `tryFastAbortFromMessage()`. There is no main-path `runInboundClaim()` call on `main`.

## Risks / Open Questions

- Closing this PR would discard a still-plausible core behavior gap that is not implemented on `main` today.
- Merging the PR as-is would conflict with current tested behavior and could redirect plugin-fallback turns away from the promised OpenClaw fallback path.
- Merging the PR as-is could let broad `inbound_claim` handlers intercept `/stop`-style fast-abort messages before the current abort path runs.
