## Summary

Keep open. Current `main` still exposes `write` as an overwrite-only file primitive, and the only append-only protection on `main` is a narrow memory-flush wrapper. That does not address the issue’s reported isolated-cron/shared-file data-loss path.

## Evidence

- **`write` is still overwrite-only on main:** `src/agents/tool-catalog.ts:62-64` and `src/agents/system-prompt.ts:478-480` both describe `write` as "Create or overwrite files". In implementation, `src/agents/pi-tools.read.ts:751-787` writes with `fs.writeFile(...)` or `writeFileWithinRoot(...)`; there is no general append mode in the exposed write path.
  - file: [src/agents/pi-tools.read.ts:751](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/agents/pi-tools.read.ts#L751)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Append-only logic is scoped only to memory flush:** `src/agents/pi-tools.ts:637-655` only wraps `write` when `isMemoryFlushRun && memoryFlushWritePath`. The wrapper in `src/agents/pi-tools.read.ts:535-579` hard-restricts writes to one memory-flush path and appends there only. `src/agents/pi-tools.workspace-only-false.test.ts:205-240` tests that narrow memory-triggered case, not general cron/session writes.
  - file: [src/agents/pi-tools.ts:637](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/agents/pi-tools.ts#L637)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Issue discussion still matches the current gap:** The provided issue comments describe real production data loss from isolated cron sessions writing shared `memory/YYYY-MM-DD.md` files. Current `main` has no general `append` tool and no cron-specific shared-file guard. Ryce’s `keepmyclaw` ClawHub backup skill is a mitigation/workaround mentioned in discussion, not an OpenClaw core fix.
- **No local history evidence of a landed fix for this issue:** A local history scan with `git log --oneline --decorate --grep='40574|40001|append mode|append-only write|memory flush' --all -n 40` returned no matching fix commit in this checkout.
  - command: `git log --oneline --decorate --grep='40574|40001|append mode|append-only write|memory flush' --all -n 40`
