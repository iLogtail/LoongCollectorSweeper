## Summary

Keep open. Current `main` added an append-only guard for memory-flush runs, but it did not implement a general append-capable file tool or append mode for normal/isolated cron sessions. The reported failure mode remains plausible on current `main` because cron runs still use the normal overwrite-oriented `write` tool.

## Evidence

- **Write tool contract still has no append/mode parameter:** `REQUIRED_PARAM_GROUPS.write` still requires only `path` and `content`; there is no `mode`, `append`, or separate append-tool parameter surface in the OpenClaw wrapper.
  - file: [src/agents/pi-tools.params.ts:75](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/agents/pi-tools.params.ts#L75)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Normal write tool construction is still overwrite-oriented:** Both sandboxed and host workspace write tools are still built from upstream `createWriteTool(...)`, and their write operations call plain `writeFile` paths rather than an append primitive.
  - file: [src/agents/pi-tools.read.ts:637](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/agents/pi-tools.read.ts#L637)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Underlying host/sandbox write operations still overwrite:** Sandbox writes call `bridge.writeFile(...)`; host non-workspace writes call `fs.writeFile(...)`; host workspace writes call `writeFileWithinRoot(...)`. The dedicated append primitive exists separately as `appendFileWithinRoot(...)`, but it is not what the general `write` tool uses.
  - file: [src/agents/pi-tools.read.ts:720](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/agents/pi-tools.read.ts#L720)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Append-only protection is memory-triggered only:** The append-only wrapper is enabled only when `options.trigger === "memory"` and `memoryFlushWritePath` is set; otherwise tools are left unchanged.
  - file: [src/agents/pi-tools.ts:345](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/agents/pi-tools.ts#L345)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Isolated cron runs do not use the memory-only guard:** Isolated cron execution calls `runEmbeddedPiAgent(...)` with `trigger: "cron"`, so it does not qualify for the memory-flush append-only wrapper in `src/agents/pi-tools.ts`. That leaves cron sessions on the normal `write` behavior described above.
  - file: [src/cron/isolated-agent/run-executor.ts:160](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/cron/isolated-agent/run-executor.ts#L160)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Shipped changelog entry is scoped to memory flush, not a general append tool:** The changelog documents append-only protection for memory-flush runs and embedded attempts, which is a partial mitigation, but it does not claim a general append tool/mode for shared workspace files or isolated cron sessions.
  - file: [CHANGELOG.md:1445](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/CHANGELOG.md#L1445)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
