## Summary

Keep #72318 open. The older related ClawSweeper reports (#62374/#62564) show a generic Windows ESM loader fix, but this new report is against v2026.4.24 and specifically fails while the Telegram channel task is running. Current main still has channel-specific plugin/runtime loading paths that are separate from the generic `toSafeImportPath` fix, so there is not enough evidence to close this as already implemented.

## Best Possible Solution

Keep the issue open and investigate the Windows packaged Telegram channel start path. The best fix, if reproduced, is to share the Windows-safe import specifier normalization across channel/plugin runtime boundaries or add it to `src/channels/plugins/module-loader.ts` and related plugin-boundary loaders, then add a regression test that exercises a Windows `C:\...` bundled Telegram/channel runtime load rather than only unit-testing the generic helper.
