## Summary

Keep open. Current main still has the same keystroke-sensitive WebChat draft path: `chatMessage` is Lit `@state()`, the chat view receives it as `draft`, and each textarea input writes the new value back through `onDraftChange`, which updates host state. `renderChat` also still rebuilds chat items and the thread template during render; I did not find the guarded/thread-cache fix described in earlier PR attempts. The linked #54905 and #54997 attempts were closed unmerged, and the related #62790 remains open work rather than shipped behavior.

## Best Possible Solution

Keep #54874 open as the canonical bug until the Control UI draft path is fixed. The likely product/code path is to decouple normal textarea typing from Lit host `@state()` updates, or otherwise isolate/cache the chat thread so draft-only changes do not rebuild message groups, while preserving send/clear/programmatic draft sync, slash commands, token count, input history, STT, attachments, and keyboard behavior. Add a focused regression test that draft-only input does not re-render the chat thread/message groups, then review and land #62790 or an equivalent narrow fix.
