## Summary

Keep open. Current main has an internal `argPattern` field and Windows-specific matching coverage, but manual exec allowlist matching remains effectively binary/path-only for non-Windows hosts. The reported use case is a Homebrew/macOS Node invocation, and current docs still describe allowlist argument scope as path-only.

## Best Possible Solution

Keep the issue open and implement a deliberate cross-platform argument-level exec allowlist contract. The likely path is to formalize a public field (`argPattern` or an `argsPattern` alias), apply it in `matchAllowlist()` for POSIX and Windows without widening legacy path-only entries, update CLI/Control UI/docs, and add regression tests proving `/opt/homebrew/bin/node` only matches the intended `mute_user.mjs <chat_id> <user_id>` argv shape while other Node scripts, inline eval, and extra arguments miss.
