## Summary

Keep open. Current main has renewable same-sender inbound debounce/concatenation, which covers related #55835, but it still uses fixed debounce windows. There is no burst-length-based increment, max cap, or config/schema/docs support for the adaptive timing requested in #57123.

## Best Possible Solution

Leave #57123 open for a maintainer/product decision. A complete fix likely belongs in the shared inbound debouncer/config surface: add count-aware timing options such as an initial debounce, increment, and max cap; validate them in zod/types/help; compute the window from the current per-key buffer length; preserve existing fixed-window behavior by default; and add core debouncer tests plus docs for single-message and rapid-burst timing.
