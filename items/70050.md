## Summary

Keep #70050 open. Current main still performs default load-time session maintenance synchronously inside `loadSessionStore`, and `chat.history` reaches that path through `loadSessionEntry`. The provided issue comments correctly point at #69404’s default-enforce/load-time prune behavior; current source and tests still preserve that behavior rather than deferring, chunking, or moving it off the startup/request path.

## Best Possible Solution

Keep the issue open and fix the startup/request-path behavior in core: make load-time maintenance non-blocking or explicitly deferred/chunked, keep a manual/debug opt-out via `session.maintenance.mode: "warn"`, add a regression/performance-oriented test around oversized stores, and update the session-maintenance docs/contracts so the default behavior is consistent.
