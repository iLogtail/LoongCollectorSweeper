## Summary

Keep #70050 open. Current main still has the blocking load-time session maintenance path that the issue and triage comments identify: `chat.history` synchronously calls `loadSessionEntry`, which synchronously loads the whole session store and, by default, prunes and caps oversized stores before returning. A later change appears to add a config/workaround path by honoring `session.maintenance`, but it does not solve the default startup responsiveness problem.

## Best Possible Solution

Keep #70050 open and fix the session-store startup path directly. The best product/code direction is to preserve bounded session maintenance while making load-time cleanup nonblocking: defer it until the gateway is responsive, chunk/yield during prune and cap, or replace the full sort with a bounded selection algorithm. Add regression coverage around oversized stores showing startup or `chat.history` is not blocked for the full maintenance pass, keep the explicit warn-mode escape hatch, and update session maintenance docs/schema text to match the final runtime default.
