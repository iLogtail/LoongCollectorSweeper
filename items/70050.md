## Summary

Keep #70050 open. Current main still performs default enforced session maintenance synchronously inside `loadSessionStore`, including a full store scan and full sort before returning. `chat.history` and several startup paths still call that loader synchronously, so the reported startup/history blocking failure mode remains plausible and is not superseded by the later #71356 config-honor fix.

## Best Possible Solution

Keep the issue open and fix the session-store load path: avoid synchronous O(n)/O(n log n) maintenance during gateway startup and `chat.history` reads, either by deferring enforcement to a post-ready background task, chunking/yielding large maintenance work, or replacing full sorting with bounded selection. Preserve the explicit `session.maintenance.mode: "warn"` opt-out, add a regression/performance test for oversized stores, and separately instrument the stdin-unblock symptom if it persists after the load-time maintenance change.
