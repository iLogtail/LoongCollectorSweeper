## Summary

Keep #56752 open. Current main still has a singleton macOS WebChat window/panel manager: opening chat for another session replaces the existing window controller, and the shared SwiftUI chat view switches sessions inside one view model rather than supporting independent parallel windows. The gateway protocol is already session-keyed, so the missing piece is the desktop multi-window/session UX rather than a closeable core cleanup item.

## Best Possible Solution

Keep the issue open. The likely fix belongs in the macOS/WebChat layer: replace the singleton WebChatManager window slot with a per-session or per-window controller collection, preserve independent OpenClawChatViewModel instances per window, and add tests proving that opening a second session window does not close or clear the first pending run and that session-keyed events route to the correct window.
