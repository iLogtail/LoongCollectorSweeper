## Summary

Keep open. Current `main` still implements the reported behavior: session-targeted Gateway RPCs and subscriptions are keyed by `sessionKey` plus operator method scope, without caller device/connection ownership checks. The repo docs also say this is not treated as a security vulnerability by design, so this looks more like an authorization-model/product decision than a stale or already-fixed item, and none of the permitted conservative auto-close reasons fit.

## Evidence

- **Session lookup is keyed only by sessionKey:** `loadSessionEntry(sessionKey)` resolves the target store and freshest session entry from the provided key alone; it does not take caller, device, or connection identity.
  - file: [src/gateway/session-utils.ts:475](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/gateway/session-utils.ts#L475)
  - command: `nl -ba src/gateway/session-utils.ts | sed -n '475,493p'`
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Session read/subscribe handlers have no ownership gate:** `sessions.list` returns `listSessionsFromStore(...)`; `sessions.subscribe` adds the connection to the global session-event subscriber set; `sessions.messages.subscribe` canonicalizes the supplied key and subscribes that connection; `sessions.get` reads messages for the resolved key. None of these handlers consult caller-to-session ownership.
  - file: [src/gateway/server-methods/sessions.ts:579](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/src/gateway/server-methods/sessions.ts#L579)
