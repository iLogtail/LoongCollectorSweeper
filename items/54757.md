## Summary

Keep open. Current `main` adds a global `messages.suppressToolErrors` option, but it only suppresses non-mutating tool warnings. Mutating failures are still intentionally surfaced, and WhatsApp still has no channel-specific silent/error policy. `deliverWebReply` continues to send any text payload it receives to WhatsApp.

## Evidence

- **Mutating tool failures still bypass the global suppress flag:** `resolveToolErrorWarningPolicy` returns `showWarning: true` for mutating tool errors before it checks `messages.suppressToolErrors`, so write/message/edit-style failures are still emitted as warning payloads.
  - file: [src/agents/pi-embedded-runner/run/payloads.ts:107](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/agents/pi-embedded-runner/run/payloads.ts#L107)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Tests explicitly lock in the current behavior:** Current tests expect non-mutating errors to be suppressed by `messages.suppressToolErrors`, but also explicitly expect mutating errors to remain visible even when that flag is enabled.
  - file: [src/agents/pi-embedded-runner/run/payloads.errors.test.ts:286](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/agents/pi-embedded-runner/run/payloads.errors.test.ts#L286)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Public config only exposes a global suppress flag:** The user-facing config adds `messages.suppressToolErrors` as a global setting; it is not scoped to WhatsApp or other external channels.
  - file: [src/config/types.messages.ts:121](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/config/types.messages.ts#L121)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **WhatsApp config has no error-policy field:** The WhatsApp channel config types define DM/group/reply settings, but there is no WhatsApp `errorPolicy` or WhatsApp-local `suppressToolErrors` field comparable to Telegram's channel error policy.
  - file: [src/config/types.whatsapp.ts:53](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/config/types.whatsapp.ts#L53)
  - command: `rg -n "type WhatsApp|interface WhatsApp|errorPolicy" src/config/types.whatsapp.ts && nl -ba src/config/types.whatsapp.ts | sed -n '53,112p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **WhatsApp delivery still forwards reply text directly:** Once a warning payload reaches WhatsApp delivery, `deliverWebReply` sends each text chunk with `msg.reply(...)` and does not add a tool-error suppression layer there.
  - file: [extensions/whatsapp/src/auto-reply/deliver-reply.ts:85](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/whatsapp/src/auto-reply/deliver-reply.ts#L85)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)

## Risks / Open Questions

- The issue examples include mutating failures like `✉️ Message failed` and `📝 Edit ... failed`; those are still designed to surface on current `main`, even with the new global suppress flag.
- There is still no WhatsApp-specific or external-channel-specific error policy to make these warnings silent for customer-facing chats while preserving visibility elsewhere.
