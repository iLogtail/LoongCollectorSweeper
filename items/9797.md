## Summary

Keep #9797 open. Current main still lacks the requested agent-facing `queue_status` tool and also lacks a stable queue snapshot RPC/CLI backend that could satisfy the dispatcher use case. The existing alternatives expose agent/session metadata and coarse queue counters, but not per-lane queue depth, active-vs-max concurrency, or per-agent pending workload. The recent maintainer/Codex issue comment also explicitly kept this item open, and related PR #43618 is scoped to operator diagnostics rather than the in-agent tool requested here.

## Best Possible Solution

Implement a shared read-only queue snapshot provider in core, then expose it through the requested agent-facing `queue_status` tool with tool catalog entry, permission/policy handling, docs, and tests. If PR #43618 becomes the backend for operator diagnostics, reuse that backend and let #9797 track the remaining in-agent consumer until it ships.
