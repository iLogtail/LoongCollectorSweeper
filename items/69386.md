## Summary

Keep open. Current main has useful abort plumbing (`chat.abort`, `/stop`, queue interrupt/steer modes, and some denial-safe copy), but it does not implement the issue’s core safeguards: Control UI still misses the reported Chinese stop phrases, denied exec approvals still start an agent follow-up turn for normal sessions, and `runTaskInFlow` has no consecutive-denial threshold/cancel guard. The linked PR #69387 is open/unmerged and only proposes a narrow TaskFlow denial guard, so it does not supersede the issue.

## Best Possible Solution

Keep the issue open and fix it as a cross-surface reliability bug. The likely path is to centralize Control UI stop detection on the shared abort primitives, add the exact phrases `停下来` and `暂停`, treat stop/deny as priority interrupts that clear or bypass queued followups, suppress or convert denied exec completion followups into a non-agent status after repeated denials or a stop request, add a per-session/run consecutive-denial breaker, and cover the flow with focused tests in `ui/src/ui/app-chat.test.ts`, `src/agents/bash-tools.exec-approval-followup.test.ts`, and the task/run cancellation path.
