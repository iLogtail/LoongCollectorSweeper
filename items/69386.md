## Summary

Keep open. Current main has partial stop/abort and exec-denial handling, but it does not fully solve this report: the exact Chinese stop phrases from the issue are not recognized, unrecognized stop text is queued behind an active Control UI run, denied exec approvals still trigger an agent follow-up turn for normal sessions, and there is no consecutive-denial fuse in runTaskInFlow. The related fix PR #69387 was closed unmerged, and #69570 tracks a related but different approved-command cascade.

## Best Possible Solution

Keep #69386 open and fix it as queue/approval lifecycle hardening: extend stop phrase recognition across Control UI and gateway for the reported Chinese phrases, route stop text ahead of the busy chat queue, suppress or cancel denied exec followup turns when appropriate, add a consecutive-denial fuse for repeated approval denials, and cover the webchat busy-queue plus denied-approval paths with regression tests. Track shared async completion backlog behavior with #69570 rather than closing this issue as a duplicate.
