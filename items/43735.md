## Summary

Keep open. The report is recent, non-maintainer-authored, and still describes an actionable core skills-snapshot problem. Current main scans workspace skill roots, but the `<available_skills>` prompt is intentionally capped and the snapshot invalidation path is still version-counter based; a cached version-0 snapshot can still look current after a process restart. The linked #44898 is closed and #48089 was closed unmerged, so there is no open canonical replacement that safely supersedes this issue.

## Best Possible Solution

Keep #43735 open and resolve it in the skills subsystem. The strongest fix is to make skills snapshots restart-safe by invalidating against a deterministic inventory fingerprint or persisted mtime/hash state, then apply that check consistently in `src/agents/agent-command.ts`, `src/auto-reply/reply/session-updates.ts`, and cron snapshot resolution. Add regression coverage for an existing session with cached `skillsSnapshot.version === 0` after a gateway restart plus changed `workspace/skills`, and separately clarify prompt caps/gating diagnostics so users can distinguish invalid frontmatter, allowlists, environment gates, and prompt-budget truncation.
