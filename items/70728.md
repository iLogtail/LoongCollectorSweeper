## Summary

Keep open. Current main has meaningful partial coverage: channel startup is fault-isolated and channel `startAccount` work is handed off into tracked background tasks, so slow Telegram/Feishu network recovery should no longer abort later channel startup. However the issue's broader user-facing request is not fully implemented: startup logging still emits a generic `ready (...)` line before the sidecar/channel phase, there is no per-channel recovery timing summary, Feishu still has a 30s startup bot identity probe with background retry, and Telegram polling still awaits `deleteWebhook` cleanup/retry inside its polling task before it can connect. Related #63475 remains open for Feishu startup probe timeouts, but it does not cover the full Telegram plus startup UX scope here.

## Best Possible Solution

Keep this issue open as a focused product/engineering follow-up: add explicit startup UX that separates core gateway readiness from channel recovery, emit per-channel startup/recovery timings and degraded states, and either link or split Feishu-specific probe timeout work to #63475 while keeping Telegram cleanup/fallback visibility tracked here.
