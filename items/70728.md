## Summary

Keep open. Current main has partial mitigations for this report, including channel-start fault isolation, Feishu sequential startup probes, Telegram control-plane timeouts, and readiness checks. But the observable user problem is not fully implemented: channel tasks are still handed off before recovery is complete, Feishu identity recovery can still enter the 30s timeout plus background retry path, Telegram webhook cleanup remains a startup prerequisite for polling, and there is no clear per-channel startup/recovery timing summary that distinguishes gateway-ready from channels-fully-recovered.

## Best Possible Solution

Keep this open unless maintainers split it into narrower canonical issues. The best fix is a channel startup/recovery observability pass: report gateway listener readiness separately from per-channel recovered/degraded state, emit per-channel startup timing/recovery summaries, make Feishu identity recovery visibly degraded without long perceived startup ambiguity, and ensure Telegram cleanup/network fallback delays are surfaced as channel-specific recovery rather than generic startup slowness.
