## Summary

Keep #53314 open. Current main still conflates “blocked by pending descendants” with announce delivery failure: the announce flow returns `false`, lifecycle cleanup passes that boolean into the retry decision, and the second descendant-count check can increment the retry budget if descendants have just settled. The PR is not implemented on main, though its final head still needs maintainer review for the outstanding cleanup and expiry concerns.

## Best Possible Solution

Keep #53314 open and adapt it to current main with a typed announce result that distinguishes delivered, failed, and pending-descendants. The fix should defer pending-descendant cleanup without consuming `announceRetryCount`, preserve the 30-minute completion-message hard expiry and 5-minute non-completion expiry, reuse the normal finalize error recovery path, and add a regression test for the inner-count-positive / outer-count-zero race.
