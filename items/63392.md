## Summary

Keep this issue open. Current main still has instance-level `openclaw backup create` and `openclaw backup verify` only; there is no first-party restore command, no `--agent`/`--as` backup or restore surface, and the backup planner currently gathers the whole state directory plus all configured workspaces rather than a single agent slice. Related backup/restore items remain open implementation work, not a shipped or canonical duplicate that fully covers per-agent backup, restore, and clone.

## Best Possible Solution

Keep the issue open as a real core CLI feature request. The best path is to design it with the existing backup planner and agent-scope helpers: add `--agent <id>` to backup creation once the archive manifest can represent a single agent entry, workspace, `agents/<id>/agent`, and sessions; add restore only alongside the safe native restore work tracked by related backup/restore items; and support `--as <id>` with validated config/path rewrites, conflict checks, docs, and focused command plus archive tests.

## Risks / Open Questions

- The native restore path is state-mutating and needs explicit safety semantics before implementation: archive validation, conflict handling, dry-run behavior, path rewriting, and rollback/staging rules.
- Maintainers may choose to consolidate the work under a broader backup/restore tracker, but the per-agent granularity and clone behavior are not fully covered by a shipped implementation today.
