## Summary

Keep open. Current `main` still has no explicit cleanup path for already-enqueued main-session cron events or pending heartbeat wakes when a job is removed, and the reported `sessionTarget: "main"` update error for non-default agents still exists. I did not find evidence of an on-main fix strong enough to close this conservatively.

## Evidence

- **Main-session cron runs enqueue work before heartbeat delivery:** `executeMainSessionCronJob()` enqueues the system event for the target session, then may queue a heartbeat wake if the main lane is busy or if it is using `next-heartbeat`. That means a run can leave downstream queued work independent of the job record itself.
  - file: [src/cron/service/timer.ts:1206](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/cron/service/timer.ts#L1206)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '1188,1278p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Cron removal only deletes the stored job:** `remove()` filters the job out of `state.store.jobs`, persists, and re-arms the timer. It does not clear queued system events, pending wakes, or other downstream delivery state.
  - file: [src/cron/service/ops.ts:332](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/cron/service/ops.ts#L332)
  - command: `nl -ba src/cron/service/ops.ts | sed -n '332,352p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **System events are process-global in-memory queues:** `system-events.ts` keeps per-session queues in a global map. `enqueueSystemEvent()` appends to that queue, and the queue is only cleared by later drain/consume paths, not by cron job deletion.
  - file: [src/infra/system-events.ts:32](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/infra/system-events.ts#L32)
  - command: `nl -ba src/infra/system-events.ts | sed -n '1,120p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Heartbeat wakes are also global queued state with no job-specific cancel path:** `heartbeat-wake.ts` stores pending wakes in a module-global `Map`; `requestHeartbeatNow()` only queues and schedules them. I did not find a corresponding API that removes a wake because a cron job was deleted.
  - file: [src/infra/heartbeat-wake.ts:44](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/infra/heartbeat-wake.ts#L44)
  - command: `nl -ba src/infra/heartbeat-wake.ts | sed -n '1,180p'; nl -ba src/infra/heartbeat-wake.ts | sed -n '252,266p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Issue context around update failure is still valid on main:** `assertMainSessionAgentId()` still throws `sessionTarget "main" is only valid for the default agent` for non-default agents, which matches the reporter's inability to disable/update older jobs after the validation change.
  - file: [src/cron/service/jobs.ts:172](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/cron/service/jobs.ts#L172)

## Risks / Open Questions

- Static inspection does not prove the exact replay path that produced the repeated downstream deliveries.
- There may be more than one bug involved here: legacy invalid main-session jobs, queued system-event/wake cleanup, and possible heartbeat-side redelivery behavior.
