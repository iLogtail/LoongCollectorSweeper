## Summary

Keep #49251 open. Current main has adjacent model failover, auth cooldowns, an in-memory UI/run queue, and a durable restart-continuation queue, but it does not implement the requested behavior of preserving a prompt as a visible, durable pending item and automatically replaying it when every model/fallback is blocked by API limits. Ryce's comment correctly frames the remaining gap as durable queued prompt recovery, not just fallback classification.

## Best Possible Solution

Keep the issue open and implement a narrow durable pending-prompt queue in the core gateway/chat reply path. The queue should capture prompts that exhaust model/auth fallback due to rate-limit/capacity errors, expose visible pending status to UI/channel surfaces, persist enough session and delivery context to replay safely after restart, and drain when the relevant cooldown/capacity window expires. Reuse the existing session-delivery/recovery substrate only if it can be generalized without conflating restart continuations with normal user prompts; otherwise add a separate per-session prompt queue with focused tests for chat.send, auto-reply, fallback exhaustion, restart recovery, idempotency, and duplicate transcript prevention.

## Risks / Open Questions

- A durable replay design must avoid duplicating user transcript entries because `chat.send` already emits the accepted user turn early.
- Replay needs idempotency, session policy, delivery route, attachment/media, and abort semantics so queued prompts do not run in the wrong session or channel after recovery.
