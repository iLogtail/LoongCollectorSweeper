## Summary

Keep #70420 open. Current main still keeps `sessions.list` / `sessions_list` as a persisted session-store view, while the Discord health-monitor restart path only restarts the channel runtime and Discord only records session metadata on inbound message processing. The issue discussion already identifies this as a contract/documentation gap, and current docs point live channel checks at `channels status --probe` but do not explicitly say that `sessions_list` is not a Discord/channel liveness signal.

## Best Possible Solution

Keep this issue open for a narrow channel-health owner decision. The low-risk fix is a docs/contract update: document that `sessions_list` is a persisted conversation/session index, not a channel socket or provider liveness check, and direct Discord heartbeat/health automation to `openclaw channels status --probe`, `openclaw health --json`, or readiness/channel status surfaces. If maintainers want `sessions_list` to remain a liveness-compatible signal instead, implement explicit channel heartbeat/session materialization on successful channel start/restart with regression coverage for Discord reconnects.
