## Summary

Keep open. Current main still has the reported lifecycle split: `sessions.list` is backed by persisted session-store rows, while Discord health-monitor recovery only restarts the channel runtime and Discord lifecycle status only publishes channel status/transport activity. The supplied maintainer-style comment correctly identifies this as by-design behavior, but it also leaves a real docs/product decision open: either document that `sessions_list` is not a channel-liveness signal or deliberately materialize a channel heartbeat/session row after reconnect.

## Best Possible Solution

Keep #70420 open for channel-health/docs ownership. The lowest-risk fix is a docs/status clarification on the public health and session-tool pages that channel liveness should be checked via `openclaw status --deep`, `openclaw health --json/--verbose`, `channels status --probe`, or readiness/channel runtime status, not `sessions_list`. If maintainers want `sessions_list` to become a liveness signal, that should be an explicit product change that materializes a synthetic channel heartbeat/session row on successful `startChannel` or provider connected status, with cross-channel contract tests.
