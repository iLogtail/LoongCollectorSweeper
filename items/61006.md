## Summary

Keep #61006 open. Current main partially changed the old behavior from a synthetic-only retry prompt to a retry marker plus the original body, but it still injects non-user-authored retry text into the fallback prompt path. Existing tests assert that injected prefix for fallback retries with history, and the embedded runner path does not pass a separate transcriptPrompt here, even though the lower runner and docs already define a mechanism for keeping runtime context out of visible user transcript turns.

## Best Possible Solution

Keep the issue open. The best fix is to preserve the original user body as the transcript-visible prompt for fallback retries with history while carrying any retry hint through the existing hidden runtime-context/transcriptPrompt seam or a lifecycle/UI event. Update the fallback retry test to expect the original body for `sessionHasHistory: true`, and add coverage that persisted JSONL or chat.history does not expose the retry marker as user-authored content.

## Risks / Open Questions

- Current main partially mitigates task loss by including the original body, so the fix should preserve model-visible retry context through hidden runtime context, lifecycle state, or UI metadata rather than deleting useful recovery signal outright.
- Closing this now would leave the visible transcript contract unresolved: the fallback retry can still include text the user did not type in a role:user prompt path.
