## Summary

Keep PR #71843 open. Current main still lets Slack `already_reacted` bubble out of the agent-facing `react` path because `reactSlackMessage()` directly awaits `client.reactions.add()` and `handleSlackAction()` awaits that helper without suppression. The PR is active, targeted, and now has maintainer feedback asking for the fix to move to the shared helper boundary rather than be closed as stale or superseded.

## Best Possible Solution

Keep #71843 open for maintainer review and revision. The best fix is to move duplicate-reaction idempotency into `extensions/slack/src/actions.ts` in `reactSlackMessage()`, preferably checking Slack's structured `data.error === "already_reacted"` while preserving propagation for unrelated errors; then remove the duplicate status-adapter catch if the helper owns it and keep focused regression coverage. If maintainers want the broader cleanup, handle `no_reaction` in `removeSlackReaction()` in the same helper-level style and explicitly supersede #50881/#58242.
