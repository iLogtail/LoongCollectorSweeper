## Summary

Current main still appears to have the reported Control UI restore behavior. Plain reload/startup hydrates persisted per-gateway session selection from local settings, initializes the active UI session from that value, and gateway default normalization only maps main aliases to the real main session while preserving non-alias routed keys as explicit selections. The only reset-to-main path I found is for tokenized URLs without an explicit session, which is narrower than this issue.

## Best Possible Solution

Keep this issue open and implement a narrow Control UI startup-restore fix. The likely path is to distinguish persisted startup restore from explicit URL/user session selection; when no explicit session was requested and persisted sessionKey or lastActiveSessionKey matches a routed channel key such as ^agent:[^:]+:[^:]+:(direct|group):.+$, fall back to sessionDefaults.mainSessionKey and persist the normalized main selection. Add focused tests for plain reload/localStorage restore, plus tests that explicit session URLs and deliberate in-session user selection still work.
