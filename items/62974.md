## Summary

Keep open. Current main has related fallback and prompt-cache work, including OpenRouter routing metadata and transient auto fallback persistence, but it does not implement the requested cross-request sticky fallback window. In fact, auto-failover overrides are explicitly cleared on the next turn so the configured primary is retried immediately, which is the prompt-cache bouncing behavior this issue is about.

## Best Possible Solution

Implement a core/session failover policy that records successful auto fallback selection with an expiry, honors it across future turns until the sticky window ends, then probes the primary once. The config should probably live near auth/model fallback cooldown settings, support a fixed duration and optionally cacheRetention-derived TTL, preserve explicit user overrides, and include docs plus unit/e2e coverage for primary timeout -> fallback success -> subsequent turn starts on fallback -> expiry probes primary.
