## Summary

Keep open. Current `main` still clears auto-sourced failover overrides at the start of the next turn, retries the configured primary, and exposes no sticky fallback/cache-aware cooldown setting. The existing behavior matches the reported cross-provider prompt-cache bouncing rather than fixing it.

## Evidence

- **Auto failover is explicitly transient on the next turn:** `createModelSelectionState()` detects an auto-sourced fallback override, clears it, and resets in-memory selection back to the configured primary so the next request retries the primary instead of staying on the fallback.
  - file: [src/auto-reply/reply/model-selection.ts:173](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/auto-reply/reply/model-selection.ts#L173)
  - command: `nl -ba src/auto-reply/reply/model-selection.ts | sed -n '130,245p'`
  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **Tests lock in primary retry behavior:** Unit tests assert that an auto failover override is cleared and the configured primary is retried on the next turn, which is the opposite of the requested sticky fallback window.
  - file: [src/auto-reply/reply/model-selection.test.ts:635](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/auto-reply/reply/model-selection.test.ts#L635)
  - command: `nl -ba src/auto-reply/reply/model-selection.test.ts | sed -n '635,715p'`
  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **Fallback selection is persisted only as an auto override:** The runner writes a successful fallback candidate into session state, but only as `modelOverrideSource: "auto"`; combined with the next-turn clearing above, this persistence is not a stable cross-request stickiness mechanism.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:735](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/auto-reply/reply/agent-runner-execution.ts#L735)
  - command: `nl -ba src/auto-reply/reply/agent-runner-execution.ts | sed -n '720,780p'`
  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **Config surface has no sticky fallback control:** `AuthConfig.cooldowns` only exposes billing/auth/failure-window knobs plus `overloadedProfileRotations`, `overloadedBackoffMs`, and `rateLimitedProfileRotations`; there is no `stickyFallbackMs` or cache-TTL-derived fallback window.
  - file: [src/config/types.auth.ts:35](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/config/types.auth.ts#L35)
  - command: `nl -ba src/config/types.auth.ts | sed -n '35,65p'`
  - sha: [f896c3935ef8](https://github.com/openclaw/openclaw/commit/f896c3935ef8d706fe4193443c5c5eede5e64178)
- **Cross-turn tests still probe primary before fallback:** E2E tests cover the current design: when a provider is already in overloaded cooldown, OpenClaw still probes it again and then falls back, including across turns. That behavior would keep bouncing prompt caches between providers.
  - file: [src/agents/model-fallback.run-embedded.e2e.test.ts:582](https://github.com/openclaw/openclaw/blob/f896c3935ef8d706fe4193443c5c5eede5e64178/src/agents/model-fallback.run-embedded.e2e.test.ts#L582)
