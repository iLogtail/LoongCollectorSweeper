## Summary

Keep #33838 open. Current main still models node commands as string-only declarations and allowlists; it does not provide a node-declared input/output schema mechanism for gateway validation, routing metadata, or API discovery. The prior ClawSweeper keep-open comment remains accurate against b67d9bf7f06f4c655b1e0869395056c7d631abc2, and the provided related items cover different work rather than superseding this gateway protocol gap.

## Best Possible Solution

Keep #33838 open for maintainer/product design. A likely implementation is an additive gateway protocol extension that preserves existing `commands: string[]` behavior while allowing optional per-command metadata such as `inputSchema` and `outputSchema`, persists approved schema metadata with the node surface, exposes it through `node.list`/`node.describe` or a dedicated discovery method, and validates `node.invoke` params/results where appropriate.

## Risks / Open Questions

- The eventual fix is a protocol/API change spanning gateway schemas, node clients, pairing storage, docs, tests, and discovery surfaces, so it needs explicit additive design and backward compatibility for existing string-only command declarations.
- Maintainers still need to choose the exact schema shape and trust model, including whether schemas are validated at connect, pairing approval, invocation input, invocation output, or only exposed for discovery.
