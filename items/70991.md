## Summary

Keep #70991 open. Current main has partial abort support, including session-scoped `chat.abort` and queued abort after reconnect, but it still does not prove the reported Control UI failure is fixed: Stop visibility is derived only from local `chatRunId`, reconnect clears that state, a page reload loses any in-memory pending abort, and the long-task connection-drop side remains unresolved. The provided related context also says older #46376 was closed in favor of this issue, so #70991 is the canonical open tracker rather than a duplicate to close.

## Best Possible Solution

Keep #70991 open as the canonical Control UI/Gateway reliability bug. A maintainer-quality fix should recover or derive abortable active-run state after reconnect/reload, keep a session-scoped Stop affordance available when the selected session still has active runs, provide visible `/stop` feedback after a fresh reload, and separately verify whether protocol-level WebSocket ping keepalive is needed for the long-task connection drops. Add regression coverage for disconnect/reconnect/reload during an active run.
