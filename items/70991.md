## Summary

Keep open. Current main has partial abort improvements, including session-scoped `/stop`/`chat.abort`, but it still does not fully solve the reported loss-of-control path: the Control UI Stop affordance is derived from local `chatRunId`, reconnect clears that state, and reload/reconnect can still leave no Stop button for a server-side active run. The connection-drop side is also still plausibly open because Gateway uses application-level JSON `tick` broadcasts and has no Gateway WebSocket protocol ping/pong keepalive implementation.

## Best Possible Solution

Keep #70991 open as the canonical issue. The best fix is to expose or recover active abortable run state from the Gateway after reconnect/reload, make the Control UI Stop affordance depend on that server-side state instead of only local `chatRunId`, preserve `/stop`/`sessions.abort` as session-scoped fallbacks across canonical session keys, and address the transport stability side through #56668 or an equivalent protocol-level keepalive with focused regression coverage.
