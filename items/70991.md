## Summary

Keep open. Current main contains partial fixes for abort control, including `/stop`/`chat.abort` session-level aborts and queued Stop-button aborts across reconnect, but the reported Control UI loss-of-control path is not fully implemented: the Stop button is still driven only by local `chatRunId`, and reconnect clears that state, so a reload/reconnect can still leave the browser without a Stop button for a server-side active run.

## Best Possible Solution

Keep this issue open and use it to drive a focused fix: expose or recover active abortable run state from the Gateway after reconnect/reload, make the Control UI Stop affordance available from that server-side state rather than only local `chatRunId`, keep `/stop` working without a run id for both `chat.send` and `agent`-registered runs, and address the WebSocket stability gap with protocol-level keepalive or an equivalent proven reconnect strategy plus regression coverage.
