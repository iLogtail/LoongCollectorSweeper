## Summary

Keep open. Current main has a related OpenRouter fix for `Key limit exceeded` on 401/403, but I do not see source or test evidence that it covers the reporter's exact `403 API key budget limit exceeded (monthly limit)` wording. The scoped OpenRouter matcher requires adjacent `key limit`, the generic billing matcher does not include `budget limit`, and the 401/403 status path only bypasses generic auth when the message is already classified as billing. Existing regression tests cover `Key limit exceeded`, not `API key budget limit exceeded`, matching the reporter's follow-up concern.

## Best Possible Solution

Keep the issue open and add targeted coverage for the exact OpenRouter `401`/`403 API key budget limit exceeded (monthly limit)` message. The likely fix belongs in the failover classification layer: either extend the OpenRouter-scoped key-limit matcher to accept `key budget limit`, or add a high-confidence budget-exhaustion billing pattern, then prove it with unit tests and a fallback-chain test showing the next configured model is attempted.
