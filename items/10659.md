## Summary

Keep #10659 open. Current main has useful SecretRef infrastructure, supported credential surfaces, logging redaction heuristics, and plugin hooks that make the linked openclaw-secrets-hardening and openclaw-credential-vault mitigations credible today. It still does not ship the native masked-secrets behavior requested here: no `{{secret:NAME}}` command/runtime substitution, no `secrets.mask`/`maskedSecrets` config, no native block for `env` or `.env` reads, and no guaranteed redaction of raw exec/process output before the agent sees it. A same-day maintainer-associated Codex review already kept this issue open, and the related core PRs in the provided context (#9271 and #22744) are closed unmerged while #11829/#59165 remain open roadmap/RFC items rather than shipped behavior.

## Best Possible Solution

Keep #10659 open until OpenClaw ships a native masked-secret or credential-broker flow for agent-executed commands and process output, or maintainers explicitly supersede it with a canonical design such as #59165. Continue to acknowledge SecretRefs, openclaw-credential-vault, and jmkritt/openclaw-secrets-hardening as useful mitigations, but do not treat them as closing the core security/product request.
