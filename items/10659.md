## Summary

Keep #10659 open. Current main has useful SecretRef infrastructure, host exec environment sanitization, redaction-oriented logging/docs, and plugin hook surfaces that make the linked workaround/plugin approaches credible, but it still does not ship the native masked-secrets behavior requested here: no `{{secret:NAME}}` command/runtime substitution, no `secrets.mask` or `maskedSecrets` config surface, no native block for reading `.env` or arbitrary secret vars, and no configured secret-value redaction layer on exec/process output before the agent sees it. The issue discussion also includes a same-day maintainer-associated Codex review explicitly keeping this open while acknowledging `jmkritt/openclaw-secrets-hardening` and `openclaw-credential-vault` as mitigations rather than a core implementation.

## Best Possible Solution

Keep #10659 open until OpenClaw ships a native masked-secret or credential-broker path for agent-executed commands and process output, or maintainers explicitly supersede it with a canonical design such as #59165. Continue to acknowledge `jmkritt/openclaw-secrets-hardening` and `openclaw-credential-vault` as useful mitigations, but do not treat them as closing this core security/product request.
