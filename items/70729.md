## Summary

Current main still has the reported gap. Trusted-proxy auth extracts and returns the header identity, but the WebSocket client state does not persist that user, and `chat.send` builds `MsgContext.SenderId`/`SenderName` from the generic client metadata (`openclaw-control-ui` style) rather than the authenticated proxy user.

## Best Possible Solution

Keep this issue open and implement it in the core gateway/WebSocket path: persist the successful trusted-proxy `authResult.user` on `GatewayWsClient`, thread it into webchat-originated `chat.send` and initial `sessions.create`/`sessions.send` flows, populate `MsgContext` with either `SenderId`/`SenderName` or a deliberately named authenticated-user field, and add a trusted-proxy webchat regression test plus a docs note.

## Risks / Open Questions

- Changing `SenderId` semantics could affect command authorization, routing, and audit behavior; the fix should explicitly decide whether trusted-proxy identity maps to sender fields or to a distinct authenticated-user field.
- The implementation must only use the identity returned by the successful trusted-proxy auth decision, not arbitrary per-message headers or client-supplied frame metadata.
