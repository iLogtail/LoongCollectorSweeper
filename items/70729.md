## Summary

Keep #70729 open. Current main still authenticates trusted-proxy users but does not retain that identity on the WebSocket client or inject it into webchat `chat.send` message context. The agent currently sees sender fields derived from the static Control UI client identity, not `X-Forwarded-User`.

## Best Possible Solution

Keep the issue open. The best product path is a core gateway fix: persist the trusted-proxy authenticated user/source on the WebSocket client after handshake, thread it into webchat-originated `chat.send` and `sessions.create` initial-send context, choose and document whether it maps to `SenderId`/`SenderName` or a distinct authenticated-user field, and add a regression test for trusted-proxy webchat message context.

## Risks / Open Questions

- The fix should be explicit about semantics: overloading `SenderId`/`SenderName` with proxy-authenticated identity may affect downstream allowlists and audit behavior, while a distinct authenticated-user field would need agent/tool propagation and docs.
