## Summary

Current main still appears to drop the trusted-proxy subject before webchat messages reach the agent. The auth layer extracts and authorizes `X-Forwarded-User`, but the WebSocket client state and gateway request handler do not retain that user, and `chat.send` populates `SenderId`/`SenderName` from static Control UI client metadata instead. The provided related items do not supersede this: #23585 is a separate missing-header fallback request, while the other title matches are unrelated webchat/session UI issues.

## Best Possible Solution

Keep #70729 open and implement a core gateway fix: persist the trusted-proxy subject from the successful WebSocket handshake on the connection/client context, thread it into webchat `chat.send` message context as either `SenderId`/`SenderName` or an explicit authenticated-user field, and add a regression test for trusted-proxy webchat propagation. The optional session-index owner metadata can be included in the same patch or tracked as a narrower follow-up.

## Risks / Open Questions

- Mapping trusted-proxy identity directly into `SenderId`/`SenderName` may affect command, hook, and approval authorization semantics; maintainers should decide whether to use those existing fields or add a distinct authenticated-user field.
- The fix should only use the server-authenticated WebSocket handshake result, not any client-supplied RPC field or arbitrary `chat.send` header value.
