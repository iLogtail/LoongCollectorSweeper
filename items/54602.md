## Summary

Keep PR #54602 open. Current main has moved toward concrete OpenRouter `...:free` models and the `openrouter:free` compatibility alias, so the PR likely needs updating, but the underlying OpenRouter onboarding preservation bug is not clearly implemented on main: `applyOpenrouterConfig` still unconditionally sets `openrouter/auto`.

## Best Possible Solution

Keep the PR open but ask for a refresh against current main: preserve an existing OpenRouter primary in `applyOpenrouterConfig` or the provider-auth config patch path, include current refs such as `openrouter:free` and concrete `openrouter/...:free`, and add a focused regression test in `extensions/openrouter/onboard.test.ts` or provider-auth choice tests. Avoid adding `openrouter/free` as a new static pseudo-model unless maintainers explicitly want that compatibility surface.
