## Summary

Keep open. Current main still does not provide a per-instance Bonjour/mDNS service name config key. The gateway startup path only passes the machine display name and mDNS mode into discovery, the bundled Bonjour plugin always formats that machine display name into the service instance name, and the plugin manifest exposes no plugin config properties. The reported `OPENCLAW_MDNS_HOSTNAME` limitation also still matches the code: it is used for hostname/fallback behavior, not as an override when the bundled plugin supplies `opts.instanceName`. The related issues in the provided context cover interface conflicts, restart cleanup, and VPS crash handling, not this distinct same-host multi-gateway naming knob.

## Best Possible Solution

Add a real bundled Bonjour naming knob rather than closing this. The narrow path is a validated `plugins.entries.bonjour.config.instanceName` or documented `discovery.mdns.instanceName` setting, threaded through gateway startup into the discovery service context and used as `opts.instanceName`. Keep `OPENCLAW_MDNS_HOSTNAME` scoped to the advertised host name, update config schema/help/docs, and add tests proving two configured instances can advertise distinct service names on the same host.
