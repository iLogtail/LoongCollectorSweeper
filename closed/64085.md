## Summary

Keep open. Current main has useful model-fallback/auth-profile cooldown machinery, but it does not implement the requested provider quota circuit breaker: Gemini/RESOURCE_EXHAUSTED and HTTP 429 quota shapes still classify as `rate_limit`, rate-limit failures get short cooldowns rather than provider-wide disabled windows, same-provider fallback probes are still allowed, and there is no `providerHealth` config or `provider_tripped` hook/status surface. The related implementation PR #64127 is open and unmerged, so this is not shipped on main.

## Best Possible Solution

Keep #64085 open until the provider-failure classification and fallback policy distinguish persistent quota exhaustion from transient rate limits. The likely fix belongs in the failover classifier and auth-profile/model-fallback path: classify quota-exhausted daily/weekly/monthly provider errors into a persistent lane, mark the provider/profile disabled with a reset/cooldown policy, skip same-provider candidates while tripped, expose a plugin-observable hook/status signal if that is still desired, and add regression tests for Gemini 429 RESOURCE_EXHAUSTED plus fallback-chain behavior. PR #64127 is the current related implementation thread to review or replace.
