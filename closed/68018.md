## Summary

Current main and the latest shipped release already route subagent completion delivery from the stored/requester delivery context instead of falling back to a DM. The spawn path captures the parent session's channel/to/thread into `requesterOrigin`, the registry persists that origin, and announce coverage includes explicit direct-route delivery and Telegram thread/topic delivery.

## Evidence

- **Spawn captures parent delivery context for child runs:** `spawnSubagentDirect` resolves `requesterOrigin` from the parent session context (`agentChannel`, `agentAccountId`, `agentTo`, `agentThreadId`) and seeds `childSessionOrigin` from it, which is the routing state later used for completion delivery.
  - file: [src/agents/subagent-spawn.ts:653](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/agents/subagent-spawn.ts#L653)

  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Spawn registers requester origin for announce flow:** After starting the child run, `spawnSubagentDirect` registers the run with `requesterOrigin: childSessionOrigin`, so completion announce state carries the parent's routable destination instead of relying on an empty session origin object.
  - file: [src/agents/subagent-spawn.ts:1064](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/agents/subagent-spawn.ts#L1064)
