## Summary

Current main and the latest release already solve the PR’s user-visible problem by keeping status/health and read-only channel surfaces off the broad channel plugin runtime preload path. The PR’s exact `resolveChannelPluginIds()` refactor is not present, but `openclaw status`/`health` no longer use that broad channel-scoped registry load, and read-only channel discovery now respects disabled channel/plugin state without loading unused runtime plugins.

## Best Possible Solution

Close this PR as superseded by the shipped read-only status/health implementation. Keep the current cold status/health path; if maintainers want the lower-level `scope: "channels"` semantics hardened independently, open a narrow plugin-registry issue that is not tied to the already-fixed status hang.
