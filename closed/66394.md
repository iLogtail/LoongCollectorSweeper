## Summary

Current `main` does not implement automatic log rotation. It writes to a dated daily file, enforces a 500 MB per-file cap by suppressing further writes, and only prunes old rolling logs when the logger is built. Docs also tell operators to use external rotation, and the issue already cross-references an open implementation PR (#66457), so this should stay open rather than be closed as already fixed or obsolete.

## Evidence

- **File logging stops at cap instead of rotating:** `buildLogger()` tracks current file size and, once the next write would exceed `maxFileBytes`, emits a one-time warning and returns without writing more lines. There is no branch that creates a successor file.
  - file: [src/logging/logger.ts:439](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/logging/logger.ts#L439)
  - command: `nl -ba src/logging/logger.ts | sed -n '40,530p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Rolling behavior is date-based, not size-based rotation:** The default path is `openclaw-YYYY-MM-DD.log`. Old rolling logs are pruned by mtime cutoff (`24h`) in `pruneOldRollingLogs()`, and that pruning is only invoked when the logger is built for a rolling path.
  - file: [src/logging/logger.ts:604](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/logging/logger.ts#L604)
