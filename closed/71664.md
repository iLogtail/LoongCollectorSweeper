## Summary

#71664 is a duplicate of the still-open canonical bug #32868. Current main still has the same same-channel block-streaming delivery gap: tool-start flushing waits for the block-reply pipeline, but the WhatsApp same-channel callback only queues dispatcher delivery and does not wait for that dispatcher send before tool execution continues.

## Best Possible Solution

Close #71664 as a duplicate and keep #32868 as the canonical open bug. The eventual fix should make same-channel block reply flushing wait for actual dispatcher delivery, likely by returning/awaiting a delivery promise from sendBlockReply() or awaiting dispatcher.waitForIdle() at the block-reply flush boundary, with regression coverage for WhatsApp/same-channel blockStreamingBreak: "text_end" before tool execution.
