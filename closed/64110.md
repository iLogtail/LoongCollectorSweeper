## Summary

Keep PR #64110 open. Current main still has the core timer-side deletion race: `cron.remove` only removes the job from the persisted cron store, while `onTimer` can already have captured due job objects, persisted their `runningAtMs`, and later execute those captured objects outside the store lock. The existing post-run `applyOutcomeToStoredJob` guard only discards state writes after execution, so it does not prevent main-session events or isolated delivery side effects. The PR is not obsolete or superseded by the provided related items, but it still needs maintainer review because the PR discussion flags correctness issues in the attempted fix, including an uncaught pre-execution reload failure path.

## Best Possible Solution

Keep the item open and ask for a revised cron fix: add a safe pre-execution deletion check for captured timer due jobs and any reserved manual run window, handle existence-check reload errors as per-job outcomes inside the worker try/catch, and add a regression test that removes a job after it has been queued/captured but before execution starts and verifies no system event or isolated delivery occurs. Do not clear the whole shared cron command lane for one job unless the queue grows keyed cancellation.
