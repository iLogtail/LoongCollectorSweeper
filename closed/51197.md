## Summary

Keep open. Current `main` still matches the reported write-failure leak in both lock helpers, so this is not implemented or obsolete. The PR also picked up a credible unresolved review finding in its Go-side cleanup change, so it is not a safe candidate for conservative auto-close.

## Evidence

- **Gateway lock leak still exists on current main:** `acquireGatewayLock()` opens the lock file, writes the payload, and only closes the `FileHandle` inside the returned `release()` closure. If `handle.writeFile()` throws, control jumps to the outer catch and wraps the error as `GatewayLockError` without closing the handle or removing the partial lock file.
  - file: [src/infra/gateway-lock.ts:274](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/infra/gateway-lock.ts#L274)
  - command: `nl -ba src/infra/gateway-lock.ts | sed -n '272,292p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Plugin SDK file lock leak still exists on current main:** `acquireFileLock()` also opens the lock file and writes before storing the handle in `HELD_LOCKS`. If `writeFile()` throws, the catch rethrows non-`EEXIST` errors and the handle is never closed or tracked for later cleanup.
  - file: [src/plugin-sdk/file-lock.ts:185](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/plugin-sdk/file-lock.ts#L185)
  - command: `nl -ba src/plugin-sdk/file-lock.ts | sed -n '183,195p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Current tests do not cover the reported post-open write failure path:** `gateway-lock` tests currently cover `fs.open()` rejection (`EACCES`) rather than `handle.writeFile()` failure after a successful open, and `file-lock` tests cover retry timeout behavior only. I did not find coverage that would prove cleanup on the reported path.
