## Summary

Current main already contains Telegram polling fixes that match this report: 409 getUpdates conflicts now force a fresh transport/socket, stale transports are explicitly closed, and successful getUpdates calls now publish channel liveness so the health monitor does not falsely restart a healthy poller. The behavior is covered by Telegram regression tests and documented in the v2026.4.23 changelog.

## Evidence

- **409 conflict now forces a fresh polling transport:** Telegram polling treats a getUpdates 409 conflict as a restart condition and marks the transport dirty specifically to avoid reusing the old keep-alive TCP socket that would otherwise keep producing a 409 loop.
  - file: [extensions/telegram/src/polling-session.ts:366](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/telegram/src/polling-session.ts#L366)
  - command: `nl -ba extensions/telegram/src/polling-session.ts | sed -n '360,390p'`
- **Stale Telegram transports are closed on rebuild:** The polling transport state now closes the previous transport when a dirty rebuild happens, preventing abandoned ESTABLISHED sockets from lingering in-process across restart cycles.
  - file: [extensions/telegram/src/polling-transport-state.ts:31](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/telegram/src/polling-transport-state.ts#L31)

## Close Comment

Closing as implemented on current `main` after a Codex review. The Telegram polling path now handles `409` `getUpdates` conflicts by marking the polling transport dirty and retrying on a fresh TCP socket instead of reusing the old keep-alive connection (`extensions/telegram/src/polling-session.ts:366`). Stale transports are explicitly closed during rebuild/dispose (`extensions/telegram/src/polling-transport-state.ts:31`), and successful `getUpdates` calls now publish channel liveness so the health monitor does not falsely treat a healthy Telegram poller as stopped (`extensions/telegram/src/polling-status.ts:20`). That exact 409-conflict restart flow is covered by regression tests in `extensions/telegram/src/polling-session.test.ts:917` and `extensions/telegram/src/monitor.test.ts:790`. This shipped in `v2026.4.23` (`CHANGELOG.md:502`). If you still hit duplicate polling on `v2026.4.23` or newer, please open a fresh issue with current logs, `openclaw status --deep`, and the exact Telegram config/account layout.
